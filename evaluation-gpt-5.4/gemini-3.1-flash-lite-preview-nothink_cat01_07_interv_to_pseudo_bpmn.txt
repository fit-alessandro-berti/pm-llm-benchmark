## Why it is not higher

### 1. It collapses important steps and misassigns responsibility
The biggest issue is that it **over-compresses the process** and in doing so **distorts role ownership**.

From the interviews, the normal flow is closer to:

- Invoice received
- AP checks for PO
- If no PO: supplier contacted / exception handling
- If PO exists: AP logs invoice
- AP forwards to Purchasing
- Purchasing confirms receipt / checks against what was ordered
- Purchasing sends confirmation back to AP
- AP matches invoice line items against PO in system
- System triggers manager approval request
- Manager approves or rejects/holds
- AP schedules payment

The answer instead reduces this to:

- Log invoice
- Purchasing verifies receipt & match
- Manager review

That is too coarse. It **omits AP’s explicit matching step** after purchasing confirmation and effectively gives that matching responsibility to Purchasing alone, which is not a faithful synthesis of the interviews. Mary explicitly says **she** matches the invoice line items in the system after receiving confirmation.

### 2. It misses an important trigger dependency
Mary clearly states that manager approval is triggered **after both**:
- receipt of goods/services is confirmed, and
- the invoice matches the PO.

The answer does not model this dependency precisely. It jumps from “matching check” to manager review without clearly showing the dual prerequisite or the system-generated approval request.

### 3. The gateway structure is too vague to be safely converted into BPMN
The prompt asks for a description that another person could convert into BPMN **without ambiguity**. This answer is not precise enough.

Problems:
- “Verify Receipt & Match” is modeled as one task, but the evidence suggests **multiple checks and handoffs**.
- “Resolve Discrepancy” is used as a catch-all blob for many very different exception types:
  - missing PO
  - pricing discrepancy
  - extra units billed
  - corrected invoice request
  - retrospective PO creation
  - supplier contact
  - manager rejection/hold
- There is no explicit return flow from each exception to a specific point in the process.

A BPMN model needs clearer re-entry logic. For example:
- Missing PO  obtain PO or retrospective PO  log invoice / continue
- Discrepancy during purchasing review  corrected invoice / clarification  return to purchasing/AP match
- Manager reject/hold  investigation  back to manager review or rejection end

The answer does not specify these loops rigorously enough.

The prompt explicitly asks to consolidate conflicting accounts and **indicate uncertainty where necessary**.

A major uncertainty in the interviews is:

It only notes uncertainty around the manager reject workflow, which is not the only ambiguity.
