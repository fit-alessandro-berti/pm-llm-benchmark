## Major SQL problems

### 1. “Exact normal sequence” query is invalid
```sql
HAVING STRING_AGG(oel.activity ORDER BY oel.timestamp, oel.event_id) = ...
```
In PostgreSQL, `STRING_AGG` requires a separator argument, e.g.:
```sql
STRING_AGG(oel.activity, ',' ORDER BY ...)
```
So this is not valid as written.

### 2. Window function misused in `WHERE`
```sql
AND LAG(oel.activity) OVER (...) != 'Confirm Shipment'
```
PostgreSQL does **not** allow window functions directly in the `WHERE` clause. This query is invalid unless wrapped in a subquery/CTE.

### 3. Payment-before-invoice query has a severe logical bug
```sql
WHERE oel.activity IN ('Receive Payment', 'Issue Invoice')
...
HAVING ... OR COUNT(CASE WHEN oel.activity = 'Perform Credit Check' THEN 1 END) = 0
```
Because the `WHERE` clause already excludes `Perform Credit Check`, that count will always be zero.  
So the `OR` condition makes the query return cases far more broadly than intended. This is a major flaw.
