Why it is not higher under a hyper-strict rubric:

1. **Variant encoding is not fully robust**  
   It uses `string_agg(activity, '->' ...)` to represent a variant. If an activity name itself contains `->`, different sequences could collapse to the same string. A more rigorous answer would use a structural type such as `list(activity ORDER BY timestamp)` / `array_agg(...)` if supported for grouping and joining.

2. **It slightly changes the ordering rule**  
   The prompt defines ordering by `timestamp`. The answer uses `ORDER BY timestamp, activity`, which adds an extra tie-breaker not specified in the problem. That is pragmatic, but strictly speaking it changes the definition.

3. **K is hardcoded rather than parameterized**  
   It says “replace 3 with K,” which is acceptable, but not ideal in a benchmark answer that asks for top K.

4. **Top-K tie handling is not part of the main solution**  
   The main query uses `LIMIT 3`, which arbitrarily cuts ties at the K boundary. It does mention a `DENSE_RANK()` alternative later, which helps, but the primary answer is still not fully rigorous if tie semantics matter.
