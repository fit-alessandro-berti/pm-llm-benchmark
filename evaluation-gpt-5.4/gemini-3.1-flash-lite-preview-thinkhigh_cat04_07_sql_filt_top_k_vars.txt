     is **not a fully safe encoding** of a sequence.
   - If activity names themselves contain the separator string (`' -> '`), different activity sequences can collapse into the same textual variant.
   - A more robust DuckDB approach would use `list(activity ORDER BY timestamp)` rather than a concatenated string.
   - This is a real correctness issue, not just style.

2. **Top-K selection is not deterministic for ties**
   - The query uses:
     ```sql
     ORDER BY case_count DESC
     LIMIT :K
     ```
   - If multiple variants have the same `case_count`, the chosen top K is arbitrary unless a secondary sort key is added, e.g.:
     ```sql
     ORDER BY case_count DESC, variant
     ```
   - That makes the result unstable.

3. **The performance note is weak / partially misleading**
   - Advising to “ensure there is an index on `case_id`” is not a strong DuckDB-specific recommendation.
   - DuckDB is not typically used or tuned like a traditional OLTP engine where indexing this way is the main optimization lever.
