## Significant Flaws

1. **Repeated SQL bug — `EXTRACT(MINUTE FROM interval)`:** This appears in at least three queries. It extracts only the *minutes field* of an interval, not total elapsed minutes. A 2-hour-5-minute gap would return `5`. The correct approach is `EXTRACT(EPOCH FROM ...)/60`. This would silently produce wrong results, the worst kind of analytical error.
2. **Logic bug in Anomaly 6, Query 1:** The `WHERE` clause restricts rows to `activity = 'Perform Credit Check'`, so `MAX(CASE WHEN oel.activity = 'Receive Payment' ...)` (`payment_received`) is **always 0**, and `total_events` counts only credit-check events while being labeled "total_events." The query cannot answer the question it claims to answer.
3. **Factual inaccuracy in Anomaly 5:** It states Case 1004 "ends at Ship Goods with no record of completion" — but Receive Payment *was* logged (event 22, just early). It also references a "delivery confirmation" activity that does not exist in the defined process. This conflates "incomplete" with "out-of-order" and largely duplicates Anomaly 3.
4. **Brittle sequence-conformance query (Anomaly 2, Query 1):** Comparing `ROW_NUMBER()` against a fixed `seq_order` produces false positives whenever any activity is missing (every subsequent activity shifts), and assumes no repeated activities. This limitation is not acknowledged.
5. **Missed anomaly — `shipment_scheduled=N` in Case 1004:** The Confirm Shipment event explicitly says the shipment was *not* scheduled, yet goods shipped 25 minutes later. This is a directly visible red flag in the data and goes unmentioned. Similarly, the `attempted_early=Y` and `late_confirmation=Y` flags in `additional_info` are never queried, despite being obvious investigative levers.
6. **Minor issues:** Redundant `DISTINCT` + `GROUP BY` in Anomaly 1's first query; the Anomaly 5 query (CROSS JOIN + FILTER with correlated NOT EXISTS) is convoluted and fragile; the negative interval in Anomaly 2's second query (`oel1.timestamp - oel2.timestamp` when `oel1 < oel2`) yields confusing negative partial values; Case 1004's invoice-before-shipping is not explicitly called out as a sequence violation.
