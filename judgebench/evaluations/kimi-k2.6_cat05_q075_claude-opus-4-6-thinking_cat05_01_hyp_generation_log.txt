### SQL Query 1 Has a Fundamental Flaw
```sql
ROW_NUMBER() OVER (PARTITION BY oel.case_id ORDER BY oel.timestamp) AS actual_step
```
This calculates position by timestamp, but then compares to `expected_step`. If two activities have the same timestamp or if the problem is *missing* activities, this comparison becomes meaningless. More critically, it doesn't account for the fact that with missing activities, all subsequent activities will show deviation even if internally sequential—creating false positives. A better approach would compare pairwise precedence constraints, not absolute positions.

### SQL Query 7 Is Poorly Constructed
The `BOOL_OR` aggregations without `GROUP BY` case_id in the CTE will produce indeterminate results—actually this works due to `GROUP BY oel.case_id`, but the query then joins to `orders` without preserving that grouping properly. The `dc.has_credit_check` returns NULL/TRUE (boolean), yet is sorted with `ASC`—this is valid PostgreSQL but semantically awkward. More critically, the `early_flags` and `late_flags` filters on `additional_info` use `LIKE '%early%'` and `LIKE '%late%'` which are fragile pattern matches that would fail with minor text variations; explicit checks would be more robust.
