### Fatal Flaw: Misunderstanding Petri Net "AND-Joins" vs "XOR-Joins"
In Petri net semantics, a transition fires **only if all of its input places contain at least one token**. Multiple arcs entering a single transition act as an **AND-join** (synchronization). To create an **XOR-join** (multiple alternative paths leading to the same activity), you must route multiple transitions into a **single place**, which then feeds into the target transition. 

In your code for the Document Verification (`t_DV`) loop, you wrote:
```python
petri_utils.add_arc_from_to(p_after_CR, t_DV, net)
petri_utils.add_arc_from_to(p_before_DV, t_DV, net)
```
By giving `t_DV` two input places (`p_after_CR` and `p_before_DV`), you inadvertently modeled an AND-join. For Document Verification to execute, the network now requires a token in `p_after_CR` **and** a token in `p_before_DV` simultaneously. Because Claim Registration (`t_CR`) only deposits a token into `p_after_CR`, `t_DV` will never have the required tokens to fire. **Your Petri net deadlocks immediately after the first step.**

You repeated this exact same catastrophic error for the Fraud Check (`t_FC`) loop:
```python
petri_utils.add_arc_from_to(p_after_DV, t_FC, net)
petri_utils.add_arc_from_to(p_before_FC, t_FC, net)
```
This creates another AND-join, ensuring that even if the token somehow bypassed the first deadlock, it would inevitably deadlock here.

### How it should have been resolved:
