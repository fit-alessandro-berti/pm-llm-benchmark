#### 🔴 Critical SQL & Logical Flaws
1. **`EXTRACT(MINUTE FROM interval)` Misuse (Anomaly 1 & 4)**
   - **Issue:** In PostgreSQL, `EXTRACT(MINUTE FROM timestampdiff)` returns the *minute component* of an interval (0–59), not the total elapsed minutes. To calculate total minutes, you must use `EXTRACT(EPOCH FROM diff) / 60`.
   - **Impact:** Queries will consistently output values between 0 and 59 regardless of actual time gaps, making duration analysis completely invalid.

2. **Naïve Sequence Comparison via `ROW_NUMBER()` (Anomaly 2 SQL 1)**
   - **Issue:** Comparing chronological `ROW_NUMBER()` positions against a fixed `seq_order` assumes every case contains exactly 7 events in the same relative order. If an activity is skipped (e.g., Case 1003 missing `Validate Stock`), all subsequent position numbers shift, causing false positives for "OUT_OF_ORDER" and masking the true nature of the deviation.
   - **Impact:** Process mining requires alignment algorithms or sequence-based mining, not positional indexing. This approach is fundamentally incorrect for detecting out-of-sequence behavior in logs with missing events.

3. **Invalid `STRING_AGG` + `FILTER` Syntax (Anomaly 5)**
   - **Issue:** The query uses `STRING_AGG(DISTINCT ea.activity, ', ' ORDER BY ea.activity) FILTER (WHERE NOT EXISTS (...))`. Additionally, `FILTER` clauses do not accept `NOT EXISTS` subqueries in this manner; a `CASE WHEN` construct is required.
