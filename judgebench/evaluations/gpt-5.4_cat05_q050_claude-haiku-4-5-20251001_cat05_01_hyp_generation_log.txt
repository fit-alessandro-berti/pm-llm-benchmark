### SQL flaws

#### 4. The “out-of-sequence” query is conceptually weak
```sql
ROW_NUMBER() ... as actual_position
...
CASE WHEN actual_position != seq_order THEN 'OUT_OF_ORDER'
```
This is a crude method and can mislabel cases whenever activities are missing.  
If one step is skipped, later steps all shift positions and appear “out of order,” even if their relative order among observed events is fine. A better approach would test **specific precedence constraints**.

#### 5. One query description does not match what it does
The query labeled:
- “Compare time gaps in cases with vs without Validate Stock”

actually only returns cases **without** Validate Stock. There is no real comparison set.

#### 6. `EXTRACT(MINUTE FROM interval)` is sloppy
This extracts the **minute component**, not total elapsed minutes.  
If intervals exceed an hour, results become misleading. Strictly speaking, it should use something like:
```sql
EXTRACT(EPOCH FROM (...))/60
```

#### 7. A clear bug in the low-credit query
```sql
COUNT(oel.event_id) as total_events
FROM order_event_log oel
WHERE oel.activity = 'Perform Credit Check'
```
This will count only the credit-check rows, so `total_events` is effectively always 1 per case here. That is plainly wrong if the intention was total case events.

#### 8. Some queries are only loosely targeted
Examples:
- The “resource involvement” query mostly just lists who touched anomalous cases; it does not really investigate any stated hypothesis.
- The “process compliance score” query measures only **coverage**, not **conformance**, yet is framed as compliance.
