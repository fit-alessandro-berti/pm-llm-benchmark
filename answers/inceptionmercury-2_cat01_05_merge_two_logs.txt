## 1.  How the matching was performed  

| LogA event | LogB event (candidate) | Matching criteria used | Result |
|--------------|-------------------------|------------------------|--------|
| **Order Received** (10:00:00) | **OrderReceived** (09:59:58) | Same `order_id` **Order#1234** and timestamps differ by **2seconds** (2s tolerance) | **Merge** |
| **Order Validated** (10:01:30) | **OrderValidation** (10:01:29) | Same `order_id`, timestamp difference **1s** | **Merge** |
| **Payment Processed** (10:02:00) | **PaymentCheck** (10:02:05) | Same `order_id`, timestamp difference **5s**  **outside** the 2second tolerance, but the event names are clearly the same business step (payment). Because the gap is small and there is no other paymentrelated event in LogB, we treat it as a **probable match** and merge, noting the mismatch. |
| **Item Shipped** (10:05:00) | **Shipping** (10:05:02) | Same `order_id`, timestamp difference **2s** | **Merge** |
| **Item Delivered** (10:20:00) | – | No corresponding entry in LogB | **Keep asis (origin=A)** |
| – | **Quality Check** (10:03:00) | No counterpart in LogA (different activity) | **Keep asis (origin=B)** |

*Why the 2second tolerance?*  
Both systems record events with slightly different clock skews (secondslevel). A window of ±2s captures the usual network/processing delay while avoiding false positives.

*Why the Payment event was merged despite a 5second gap?*  
The only paymentrelated record in LogB is this one, and the business meaning (payment verification) matches LogA’s “Payment Processed”. The gap is still modest (<10s) and there is no competing candidate, so a merge is justified, but the original timestamps are both retained for traceability.

## 3.  Summary of decisions & conflict resolution  

| Situation | Decision | Rationale |
|-----------|----------|-----------|
| **Exact name match with 2s offset** (Received, Validated, Shipped) | Merge, keep LogA timestamp as primary | Timestamp difference well within tolerance; event semantics identical. |
| **Payment events** (5s offset) | Merge, keep LogA timestamp as primary, flag the offset in notes | No other paymentrelated entry in LogB; business meaning matches; offset is still small enough to be plausible. |
| **Event present only in LogB** (Quality Check) | Keep as separate record, origin=B | No counterpart in LogA, represents a distinct activity (inspection). |
| **Event present only in LogA** (Delivered) | Keep as separate record, origin=A | No counterpart in LogB; LogB simply does not record the final delivery step. |
| **Naming differences** (e.g., “Order Received” vs “OrderReceived”) | Normalise to a single canonical name (the LogA wording) | Provides a consistent view while preserving the original semantics. |
| **Conflicting timestamps** | Prefer LogA timestamp as the primary chronological key (LogA is the “primary timeline”). Both timestamps are retained in the audit trail (the notes column can record the secondary timestamp if needed). | Keeps the order dictated by the system that is considered the source of truth for timing. |
| **Missing attributes** | Leave blank / “–” and indicate origin | No information to infer; preserving the origin makes downstream analysis transparent. |
