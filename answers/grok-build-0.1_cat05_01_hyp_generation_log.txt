## Hypotheses for Root Causes

1. **Process Bypass for High-Value Orders**: Staff may be overriding standard controls for large orders (Case 1004 = $3000) to accelerate fulfillment, either due to management pressure or informal "VIP" handling procedures.

2. **System Integration Failures**: The event log may be capturing activities from multiple systems that are not properly synchronized, causing timestamps to reflect logging time rather than execution time.

3. **Fraudulent Activity**: The sequence in Case 1004 (payment received before validation) could indicate collusion between sales and finance to process unauthorized orders, or external payment fraud where funds are received but goods should not have been released.

4. **Training/Compliance Gaps**: Warehouse and logistics staff may not be trained to verify upstream approvals before executing physical shipment (evident in both Case 1002 and 1003).

5. **Data Quality Issues**: Critical events may be missing from the log due to system outages, manual workarounds not being captured, or selective logging that only records certain activity types.

## SQL Queries for Investigation

**Query 1: Reconstruct full process flows to identify sequence violations**
```sql
SELECT 
    case_id,
    STRING_AGG(activity || ' (' || TO_CHAR(timestamp, 'HH24:MI') || ')', '  ' ORDER BY timestamp) as actual_flow
FROM order_event_log
GROUP BY case_id
ORDER BY case_id;
```

**Query 2: Identify cases missing critical control activities**
```sql
SELECT 
    o.case_id,
    o.order_type,
    o.order_value,
    COUNT(CASE WHEN e.activity = 'Perform Credit Check' THEN 1 END) as has_credit_check,
    COUNT(CASE WHEN e.activity = 'Validate Stock' THEN 1 END) as has_stock_validation,
    COUNT(CASE WHEN e.activity = 'Confirm Shipment' THEN 1 END) as has_shipment_confirmation
FROM orders o
LEFT JOIN order_event_log e ON o.case_id = e.case_id
GROUP BY o.case_id, o.order_type, o.order_value
HAVING COUNT(CASE WHEN e.activity = 'Perform Credit Check' THEN 1 END) = 0
   OR COUNT(CASE WHEN e.activity = 'Validate Stock' THEN 1 END) = 0;
```

**Query 3: Detect payment received before credit approval**
```sql
WITH payment_times AS (
    SELECT case_id, MIN(timestamp) as payment_time
    FROM order_event_log 
    WHERE activity = 'Receive Payment'
    GROUP BY case_id
),
credit_times AS (
    SELECT case_id, MIN(timestamp) as credit_time
    FROM order_event_log 
    WHERE activity = 'Perform Credit Check'
    GROUP BY case_id
)
SELECT 
    p.case_id,
    p.payment_time,
    c.credit_time,
    EXTRACT(EPOCH FROM (p.payment_time - c.credit_time))/60 as minutes_between
FROM payment_times p
LEFT JOIN credit_times c ON p.case_id = c.case_id
WHERE c.credit_time IS NULL OR p.payment_time < c.credit_time;
```

**Query 4: Analyze activity sequences with previous activity context**
```sql
SELECT 
    case_id,
    activity,
    timestamp,
    LAG(activity) OVER (PARTITION BY case_id ORDER BY timestamp) as previous_activity,
    LAG(timestamp) OVER (PARTITION BY case_id ORDER BY timestamp) as previous_timestamp
FROM order_event_log
WHERE activity IN ('Ship Goods', 'Receive Payment', 'Confirm Shipment')
ORDER BY case_id, timestamp;
```

**Query 5: Cross-reference with resource roles to identify authorization violations**
```sql
SELECT 
    e.case_id,
    e.activity,
    e.resource,
    r.role,
    r.department,
    e.timestamp
FROM order_event_log e
JOIN resources r ON e.resource = r.resource_id
WHERE e.activity IN ('Ship Goods', 'Confirm Shipment')
ORDER BY e.case_id, e.timestamp;
```

These queries will help distinguish between data quality issues, systematic process violations, and potential fraudulent activity by revealing patterns across the entire dataset rather than isolated cases.
