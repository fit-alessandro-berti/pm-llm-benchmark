# Process Mining Event Log

## Transformation Logic (Case Strategy)

I grouped events into **cases based on the work artifact** (the specific document, email, spreadsheet, or PDF the user is acting upon), as identified by the `Window` title. This is the most coherent unit of "logical user work." Each artifact becomes one case, even when the user switches away and returns later (which is common in real multitasking sessions).

I then standardized raw action verbs (`FOCUS`, `TYPING`, `SAVE`, `SWITCH`, etc.) into higher-level **process activities** appropriate to each artifact type.

## Explanation of Logic

**1. Case Identification (Artifact-Based Grouping)**
Each case represents a single artifact the user worked on, keyed by the `Window` title:
- **CASE-001** — `Document1.docx` (the active working draft)
- **CASE-002** — the Annual Meeting email thread
- **CASE-003** — `Report_Draft.pdf` review
- **CASE-004** — `Budget_2024.xlsx`
- **CASE-005** — `Quarterly_Report.docx`

This approach captures real-world multitasking: notice that **CASE-001 and CASE-005 are interrupted and resumed** later (the user opens the Quarterly Report at 08:59, jumps into other work, and returns to it at 09:07). Grouping by artifact preserves the full lifecycle of each work item from open  edit  save  close.

**2. Handling SWITCH / FOCUS Events**
- `FOCUS` and `SWITCH` events that **land on** an artifact were normalized to **Open Document / Open Email / Open PDF / Resume Document**.
- I used **"Resume Document"** when a previously-opened case is revisited (CASE-005), to distinguish a fresh open from a return visit — useful for analyzing context-switching and rework.
- The `SWITCH` event's *FromApp* side was not logged as a separate event to avoid duplication; the transition is implicit in the timestamp ordering across cases.

**3. Activity Name Standardization**
Raw low-level verbs were mapped to consistent, analyst-friendly activity names tailored to the artifact type:

| Raw Action | Standardized Activity (context-dependent) |
|------------|-------------------------------------------|
| FOCUS / SWITCH (arriving) | Open / Resume Document, Open PDF, Open Spreadsheet |
| TYPING | Edit Content / Edit Spreadsheet / Compose Reply |
| SAVE | Save Document / Save Spreadsheet |
| CLOSE | Close Document |
| CLICK (email) | Open Email / Initiate Reply / Send Email |
| SCROLL | Review Email / Review PDF |
| HIGHLIGHT | Annotate PDF |

**4. Resulting Narrative**
The log tells a coherent story of a single morning work session: the user begins a quarterly reporting task, drafts a working document, pauses to confirm meeting logistics via email, reviews a source PDF, updates the supporting budget spreadsheet, returns to integrate the budget reference into the draft, and finally finalizes the executive summary in the Quarterly Report before closing out. This structure supports standard process-mining analyses such as **rework detection, context-switching frequency, and per-artifact cycle time**.
