# Data-Driven Scheduling Optimization for Precision Parts Inc.
## A Process Mining-Enabled Approach

---

## 1. Analyzing Historical Scheduling Performance and Dynamics

### 1.1 Reconstructing the Actual Process

The MES event log structure (Case ID = Job ID, Activity = Task, Timestamp, Resource) is directly compatible with process mining. I would build **two complementary logs**:

- **Job-centric log** (Case = JOB-ID): to reconstruct job routings, lead times, and flow patterns.
- **Resource-centric log** (Case = Machine-ID): to reconstruct the *sequence* of jobs on each machine, which is essential for setup-time analysis.

Using tools such as Celonis, PM4Py, or Disco, I would apply:

- **Inductive Miner / Heuristics Miner** to discover the as-is job-routing process model, revealing undocumented routing variants.
- **Directly-Follows Graphs (DFGs)** weighted by frequency and mean duration to visualize flow bottlenecks.
- **Performance Spectrum / Handover Analysis** to visualize the temporal density of job flows between machines.

### 1.2 Quantitative Metrics

| Dimension | Technique | Metric |
|---|---|---|
| **Flow time / Lead time** | Case-duration analysis from *Job Released* to *final Task End* | Mean, P50, P90, P95, CV of lead time; makespan per work batch |
| **Queue/wait time** | Delta between `Queue Entry` and `Setup Start` (or `Task Start`) | Wait time distribution per machine; queue length time-series |
| **Machine utilization** | Resource-centric log segmented into Setup / Run / Idle / Breakdown intervals | % productive, % setup, % idle, % down |
| **Operator utilization** | Group events by Operator ID | Active vs. idle time, multi-machine tending patterns |
| **Schedule adherence** | Compare actual End-Task to promised due date | Tardiness = max(0, C_j  d_j); lateness; % on-time |
| **Disruption impact** | Event-based intervention analysis | WIP, queue, tardiness pre/post breakdown |

### 1.3 Sequence-Dependent Setup Time Analysis

This is the most nuanced analysis. From the resource-centric log:

1. For each machine, sort events chronologically and extract consecutive (JOB_prev  JOB_curr) pairs.
2. For each pair, compute the **actual setup duration** = `Setup End  Setup Start`.
3. Enrich each job with *feature vectors* (material type, tool family, part geometry, coolant type, tolerance class).
4. Build a **setup-transition matrix** S[f_prev, f_curr] for each machine, populated with mean/variance of actual setup times grouped by *feature transitions* (not just JOB-IDs, which are unique).
5. Fit a regression (e.g., gradient-boosted tree) with features = (prev_job_attrs, curr_job_attrs, machine, operator)  actual setup time. The feature importance exposes *which* attributes drive setup duration (changeovers of material > tool > fixture, etc.).

### 1.4 Schedule Adherence and Tardiness

- Compute **tardiness T_j = max(0, C_j  d_j)**, **earliness**, and **flow factor** (lead time / pure processing time).
- Variant analysis: segment cases into *on-time* and *tardy* cohorts; compare their variant footprints, queue times at each step, and the sequence of resources visited.
- **Root cause decomposition**: for each tardy job, decompose delay into (waiting, setup overrun, processing overrun, disruption wait).

### 1.5 Disruption Impact

- Define breakdown periods from Resource events. For each breakdown, measure:
  - Number of jobs queued at the affected machine at t=breakdown.
  - Downstream propagation: time for WIP to normalize.
- For priority changes (JOB-7005 becomes Urgent), trace *preemption ripple*: which jobs were overtaken, and what delay they accumulated.

## 2. Diagnosing Scheduling Pathologies

Based on the patterns the analysis would surface, I expect and would explicitly test for the following pathologies:

### 2.1 Bottleneck Identification
Using **bottleneck analysis** (utilization > 85%, queue length persistently >3, wait time >> processing time), I would rank machines. Expect: CNC Milling and Heat Treatment as bottlenecks, evidenced by:
- Time-weighted queue length distributions.
- **Active-bottleneck detection** (Roy & Dalal-style): the resource whose idle time drives job waiting time at any moment.

### 2.2 Poor Prioritization
- Compare *tardiness by priority class*. If High-priority jobs show tardiness comparable to Medium, FCFS is overriding EDD.
- **Overtaking analysis**: count instances where a lower-priority job was started before a higher-priority queued job on the same machine.

### 2.3 Suboptimal Sequencing (Setup Waste)
- Compute the **setup ratio** =  setup time /  productive time per machine.
- Compare realized sequence setup cost against a *lower-bound TSP-style sequencing* of the same job set (using the setup-transition matrix). The gap quantifies sequencing waste.

### 2.4 Downstream Starvation
- Cross-correlate *upstream queue growth* with *downstream idle time*. Lagged correlation reveals starvation induced by poor release/dispatching upstream.

### 2.5 WIP Bullwhip
- Time-series of WIP per work center; compute **coefficient of variation amplification** from upstream to downstream. High amplification indicates scheduling-induced variability, not demand-driven.

### 2.6 Variant Analysis – On-Time vs. Late Cohorts
A powerful diagnostic: filter cases by lateness and compare DFGs. Typical findings:
- Tardy jobs revisit the bottleneck more often (rework).
- Tardy jobs experienced 1 setup > 2× the mean.
- Tardy jobs were queued during a breakdown event.

## 3. Root Cause Analysis

### 3.1 Limitations of Static Dispatching
FCFS ignores due dates; EDD ignores setup cost; neither considers downstream load. When mixed inconsistently across work centers, **locally rational decisions become globally suboptimal** (no coordinated objective).

### 3.2 Absence of Real-Time Visibility
Decisions made at each work center with no view of:
- Downstream queue depth  dispatches may aggravate congestion.
- Machine availability at the next step  WIP accumulates ahead of down machines.

### 3.3 Inaccurate Time Estimates
Comparing **Planned vs. Actual durations** from the log: systematic bias (e.g., actuals 10–20% longer) and operator-dependent variance. Planning that uses stale standards produces infeasible schedules.

### 3.4 Setup Time Mishandling
If the dispatching rule is setup-agnostic, the log will show frequent *adjacent-job attribute changes* (e.g., high-carbon steel  aluminum  high-carbon steel), each triggering a long setup. Process mining quantifies this directly.

### 3.5 Poor Coordination & Disruption Response
No mechanism to reschedule downstream when upstream is delayed; hot jobs are inserted ad hoc, causing preemption-cascade.

### 3.6 Scheduling Logic vs. Capacity vs. Variability

This is a critical diagnostic distinction. I would use:

- **Utilization-based decomposition**: If  > 95% on a machine, even an optimal scheduler cannot prevent queues  capacity issue.
- **Counterfactual simulation**: Replay the historical log under an optimal scheduler (e.g., offline MILP on the realized job set). If gap between actual and optimal performance is large  scheduling logic is the culprit. If small  capacity or variance dominates.
- **Variance attribution**: decompose task duration variance into (inherent process), (operator), (material), (setup predecessor). High residual variance  inherent; high explained variance by scheduling-controllable factors  schedulable.

## 4. Advanced Data-Driven Scheduling Strategies

### Strategy 1: Composite Dynamic Dispatching with Setup-Aware Scoring (CDS-SA)

**Core logic**: Replace FCFS/EDD with a *continuously recomputed priority score* at each dispatching decision:

```
Score(j, m, t) = w1 · Slack(j,t) 
              + w2 · PriorityWeight(j)
              + w3 · (1 / EstSetup(prev_m, j))
              + w4 · CriticalRatio(j,t)
              + w5 · DownstreamLoadPenalty(j)
              + w6 · BottleneckBias(j)
```

Where:
- **Slack(j,t)** = d_j  t   remaining processing (including expected queue time, derived from mined queue-length distributions).
- **EstSetup(prev_m, j)** predicted by the ML model from §1.3.
- **DownstreamLoadPenalty** = signals against dispatching to an already-congested next step.
- **BottleneckBias** = implements the Theory of Constraints: always feed the bottleneck.

**Process mining inputs**:
- Weights w1–w6 learned by **inverse optimization** against historically *successful* shifts (low tardiness days).
- Setup predictor from §1.3.
- Queue-time distributions per machine, conditioned on time-of-day and WIP.

**Pathologies addressed**: poor prioritization, setup waste, starvation, lack of downstream visibility.

**Expected KPI impact**: tardiness 20–35%, setup ratio 15–25%, lead time P90 15%.

---

### Strategy 2: Predictive Rolling-Horizon Scheduling with Stochastic Durations

**Core logic**: A rolling-horizon MILP (or CP-SAT) solver re-optimizes every *N* minutes or on triggering events (job release, breakdown, priority change). Objective: weighted tardiness + WIP holding + setup cost.

**Key innovation**: Use **probabilistic duration estimates** rather than point estimates. For each (activity, machine, operator, job-complexity) combination, fit a distribution (e.g., log-normal) from the historical log. Use the **P75 or P85 quantile** in the optimization model so schedules are robust to variability.

**Predictive maintenance integration**: If breakdown inter-arrival times and TBF/MTTR can be mined from Resource events (Breakdown Start/End), fit survival models per machine. Machines with elevated hazard get reduced planned loading or scheduled preventive intervals.

**Process mining inputs**:
- Full duration distributions, conditional on operator skill and job features.
- Breakdown hazard functions from event history.
- Routing probabilities for jobs with conditional paths.

**Pathologies addressed**: unpredictable lead times, disruption response, inaccurate estimates.

**Expected KPI impact**: due-date prediction error 40%, disruption recovery time 50%, on-time delivery  to 90%+.

---

### Strategy 3: Setup-Time-Optimized Bottleneck Sequencing with Intelligent Batching

**Core logic**: At bottleneck machines (identified in §2.1), treat sequencing as a **Sequence-Dependent Setup Time Travelling Salesman Problem with time windows (due dates)**. Solve with metaheuristics (simulated annealing, ALNS) or learned policies over a tactical horizon (e.g., next 4–8 hours of queued jobs).

**Intelligent batching**: Cluster queued jobs by setup-affinity features (from §1.3 model). Form batches that minimize intra-batch transitions while respecting due-date constraints. Use a **look-ahead window**: pull in jobs that will arrive within t to improve batching, if doing so does not violate upstream slack.

**Safeguard**: Bound the maximum "waiting cost" so that setup savings never tardify a high-priority job beyond a threshold.

**Process mining inputs**:
- The learned setup transition matrix.
- Historical evidence that "affinity-based batches" (mineable by clustering past sequences and scoring against actual setup times) reduce setup time.
- Job arrival rate at bottleneck (Poisson fit from event log) to size the look-ahead window.

**Pathologies addressed**: setup waste, bottleneck throughput.

**Expected KPI impact**: bottleneck setup time 30–50%, bottleneck throughput 10–20%, enabling corresponding lead-time reduction.

---

### Integration

The three strategies are complementary and layered:
- **Strategic layer**: Strategy 2 (rolling horizon) sets the plan.
- **Tactical layer**: Strategy 3 optimizes sequencing at bottlenecks inside the plan.
- **Operational layer**: Strategy 1 handles real-time dispatching and deviations between re-plans.

## 5. Simulation, Evaluation, and Continuous Improvement

### 5.1 Discrete-Event Simulation Design

Build a DES model (SimPy, AnyLogic, or Arena) parameterized entirely from process mining outputs:

| Simulation Parameter | Source from Process Mining |
|---|---|
| Job arrival process | Inter-arrival distribution from "Job Released" events |
| Routing | Transition probabilities discovered per job-type |
| Task durations | Fitted distributions per (activity, machine, operator) |
| Setup times | Learned matrix from §1.3 |
| Breakdowns | MTBF/MTTR distributions per machine |
| Priority mix & hot-job frequency | Empirical from log |

**Validation**: Replay the historical period and verify the simulator reproduces observed KPIs (tardiness, WIP, utilization) within ±5%. This builds trust before policy testing.

### 5.2 Test Scenarios

1. **Baseline reproduction** – current FCFS/EDD mix.
2. **Nominal load** – each proposed strategy under average conditions.
3. **High load (+25% arrival rate)** – stress test bottleneck and tardiness behavior.
4. **High disruption** – double the breakdown rate; measure recovery time.
5. **Hot-job surge** – 3× the rate of urgent priority changes.
6. **Operator variability** – simulate skill-mix changes.
7. **Ablation** – each strategy individually vs. combined, to isolate contributions.

**Evaluation framework**: Multi-KPI dashboard with statistical significance (paired replications, CRN—common random numbers), Pareto frontier visualization (tardiness vs. WIP vs. utilization).

### 5.3 Continuous Monitoring & Adaptation Framework

Deploy the chosen strategy with a **closed-loop process mining layer**:

**(a) Real-time KPI cockpit**: Streaming event log ingestion (e.g., via Kafka) feeding live dashboards: tardiness rate, WIP per cell, bottleneck utilization, setup ratio, schedule adherence.

**(b) Drift detection**:
- **Concept drift monitors** on duration distributions (KS-test weekly), setup matrix (PSI on feature transitions), routing (variant frequency shifts).
- **Control charts** (EWMA) on KPIs; alert on out-of-control signals.
- **Conformance checking** between the scheduler's plan and execution: high deviation flags policy misfit.

**(c) Model retraining cadence**:
- Duration & setup models: weekly incremental updates.
- Dispatching weights (Strategy 1): monthly re-optimization via inverse learning.
- Structural model revisions (new machines, new part families): triggered by variant analysis.

**(d) A/B / Champion-Challenger**: Periodically run a challenger policy on a subset of jobs/shifts; compare via causal-effect methods (difference-in-differences on matched job pairs).

**(e) Feedback to humans**: Scheduler transparency — for every dispatch decision, log the score components. This supports planner trust, auditability, and root-cause analysis when something goes wrong.

**(f) Governance loop**: Quarterly review of (i) strategic KPI trajectory, (ii) identified new pathologies, (iii) re-evaluation of whether the strategy mix is still Pareto-optimal given evolving business conditions (new products, demand mix shifts).

---

### Summary

Process mining transforms the raw MES log into a quantitative diagnosis (bottlenecks, setup-waste, prioritization failures, variability sources), a calibrated simulation, and a continuous feedback mechanism. Layered onto this, a composite of dynamic dispatching (Strategy 1), predictive rolling-horizon optimization (Strategy 2), and setup-aware bottleneck sequencing (Strategy 3) directly attacks Precision Parts' tardiness, WIP, and unpredictability — while embedding the adaptive capability needed for a disruption-prone job-shop environment.
