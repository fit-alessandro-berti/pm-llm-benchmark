## 1. Process Discovery and Conformance Checking

### 1.1 Data preprocessing and integration into a mining-ready event log

To analyze Speedy Parcels effectively, I would first build a **unified event log** from the four source systems. In logistics, this is the most critical step because the quality of process mining results depends heavily on proper event correlation and timestamp consistency.

### A. Define process perspective and case notion

For this scenario, I would not rely on only one case notion. I would create at least **three linked process views**:

1. **Vehicle-day case**  
   - Case ID: `Vehicle ID + Date`  
   - Purpose: analyze end-to-end daily route execution, depot departure/return, delays, deviations, maintenance interruptions, overtime.

2. **Package case**  
   - Case ID: `Package ID`  
   - Purpose: analyze parcel lifecycle, failed deliveries, repeated attempts, time-window compliance.

3. **Stop case / customer-stop case**  
   - Case ID: `Vehicle-day + Stop sequence` or inferred stop cluster  
   - Purpose: analyze arrival-service-departure behavior, service time variability, parking/search time, handover duration.

This multi-level design is important because one case notion alone will hide key insights. Vehicle-day shows route behavior; package-level shows customer outcome and rework.

### B. Standardize and harmonize source data

#### GPS tracker data
From GPS logs I would derive:
- trip segments
- motion states: moving, idle, parked
- dwell times
- route traces
- traffic-delay indicators
- unscheduled stops
- speed patterns
- possible harsh driving proxies if available

Raw GPS pings are too granular for direct process mining, so I would transform them into **higher-level events**, such as:
- `Depart Depot`
- `Travel Segment Start`
- `Traffic Delay Start`
- `Traffic Delay End`
- `Unscheduled Stop Start`
- `Unscheduled Stop End`
- `Arrive Customer Area`
- `Arrive Depot`

#### Driver scanner data
These already contain process events:
- `Scan at Depot`
- `Arrive Customer`
- `Delivery Success`
- `Delivery Failed`
- `Depart Customer`

These should be normalized into a consistent activity taxonomy. For example:
- `Customer Arrival`
- `Delivery Completed`
- `Delivery Failed - Not Home`
- `Customer Departure`

#### Dispatch system
This provides the **planned process model** and reference attributes:
- planned route
- planned stop sequence
- package assignment
- expected route start/end
- customer time windows
- capacity constraints
- driver assignment

I would load these as:
- planning events, such as `Route Assigned`
- reference tables for conformance analysis
- expected timestamps or expected stop order

#### Maintenance logs
These need to be aligned with operational execution:
- `Scheduled Maintenance Start/End`
- `Repair Start/End`
- events overlapping with route execution
- vehicle availability calendar

This allows detection of:
- mid-shift failures
- routes operated by vehicles with recent repair history
- downtime impact

### C. Build the unified schema

A mining-ready event log would contain, at minimum:

- **Case ID**
- **Timestamp**
- **Activity**
- **Event source**
- **Vehicle ID**
- **Driver ID**
- **Package ID** if applicable
- **Stop ID / Customer ID** if available
- **Location**
- **Planned sequence number**
- **Actual sequence number**
- **Planned time window start/end**
- **Vehicle type / capacity**
- **Derived metrics**:
  - leg distance
  - leg travel time
  - dwell time
  - delay vs plan
  - idle duration
  - route deviation indicator
  - maintenance overlap indicator

### D. Event correlation and sequencing

A major task is linking events from different systems into coherent traces:
- match scanner stop events to nearby GPS positions and times
- match package events to assigned vehicle-day
- infer actual stop order when sequence is not explicit
- attach planned stop order from dispatch
- identify maintenance episodes affecting route execution

### E. Key challenges

#### 1. Timestamp synchronization
Different devices may have clock drift or timezone inconsistencies. Even a few minutes of misalignment can distort arrival/departure and service time calculations.

#### 2. Granularity mismatch
GPS gives high-frequency telemetry; scanner events are sparse; dispatch gives planned states. These must be transformed to the same abstraction level.

#### 3. Missing or noisy data
Examples:
- missed scans
- GPS signal loss in urban canyons
- duplicate events
- maintenance events recorded late
- incorrect driver login

#### 4. Case correlation complexity
A package can be reassigned, a vehicle can perform multiple loops, and multiple packages can be delivered at one stop. Correlation rules must handle such one-to-many relations.

#### 5. Activity inference
Some operational steps are not explicitly recorded, such as:
- parking search
- walking to customer
- loading reorganization in van
- waiting for customer response

These may need to be inferred from GPS and scanner time gaps.

### 1.2 Process discovery of the actual delivery process

Once the event log is prepared, I would use process discovery to reconstruct the real flow of operations.

### A. Discovery objective

The aim is to visualize the **actual end-to-end route execution**, for example:

`Start Shift  Route Assigned  Load/Scan at Depot  Depart Depot  Travel  Arrive Customer  Delivery Success/Failed  Depart Customer  Travel ...  Unscheduled Stop / Delay / Breakdown  Continue Route  Arrive Depot  End Shift`

### B. Suitable discovery techniques

For this domain, I would combine several views:

#### 1. Directly-Follows Graph (DFG)
Useful for an initial understanding of the dominant flow and frequent deviations.

#### 2. Inductive Miner
Good when we want a sound process model with understandable branching:
- successful deliveries
- failed delivery loops
- return-to-depot
- interruption handling

#### 3. Heuristics Miner
Useful in noisy logistics data because it highlights frequent paths and suppresses less meaningful noise.

#### 4. Variant analysis
Very important in last-mile delivery because the process is inherently variable. I would identify major route execution variants such as:
- smooth route, all deliveries successful
- route with heavy traffic delay
- route with failed deliveries and reattempts
- route interrupted by breakdown
- route with overtime return

### C. Add spatial and temporal views

In transportation, process maps alone are not enough. I would enrich process discovery with:
- geo-process maps: process steps plotted on road network or zones
- heatmaps of dwell time
- time-annotated process models
- route progression timelines

This helps distinguish whether delays happen:
- in transit
- at specific customer clusters
- before first delivery
- near depot return
- at recurring hotspots

### 1.3 Conformance checking against planned routes

Conformance checking compares the **planned route and expected schedule** with actual execution.

### A. Reference model

The dispatch system provides the planned model:
- planned stop sequence
- expected departure and arrival timing
- assigned packages
- time windows
- vehicle/driver assignment

This becomes the normative model for each vehicle-day.

### B. Conformance techniques

#### 1. Sequence conformance
Check whether stops were visited in planned order or in acceptable reordered clusters.

Look for:
- skipped planned stops
- out-of-sequence execution
- extra stops
- route truncation
- repeated visits

#### 2. Temporal conformance
Compare actual vs planned timing:
- departure delay from depot
- arrival delay at each stop
- service time overrun
- depot return delay
- total route duration variance

#### 3. Spatial conformance
Compare actual GPS path with planned route corridor:
- detours
- unauthorized route deviations
- excessive backtracking
- high deadhead mileage

#### 4. Resource conformance
Check whether:
- assigned driver changed
- wrong vehicle used
- vehicle capacity constraints were breached indirectly
- deliveries occurred during maintenance-excluded periods

### C. Deviations to look for

Key deviations include:

- **Sequence deviations**
  - customer stops executed in non-planned order
  - skipped deliveries
  - unnecessary revisits
  - route fragmentation

- **Unplanned stops**
  - fuel stops
  - breaks outside planned windows
  - breakdowns
  - unexplained idle periods
  - return to depot mid-route

- **Significant timing differences**
  - late route start
  - chronic underestimation of travel time
  - long service time at specific stop types
  - repeated late arrivals to time-window customers

- **Operational compliance deviations**
  - working time overruns
  - route extension beyond scheduled shift
  - unplanned reassignment

Conformance checking would quantify the deviation rate and identify where deviations are benign adaptations versus true inefficiencies.

## 2. Performance Analysis and Bottleneck Identification

### 2.1 KPIs relevant to Speedy Parcels

Below are the most important KPIs and how to derive them from the event log.

### A. Service and punctuality KPIs

#### 1. On-Time Delivery Rate
Percentage of deliveries completed within customer time windows.

**Calculation:**  
For each package:
- compare `Delivery Success timestamp` to planned time-window end
- optionally check arrival within window if that is the contractual criterion

Formula:
`On-Time Delivery Rate = on-time delivered packages / total delivered packages`

#### 2. First-Attempt Delivery Success Rate
Measures failed delivery avoidance.

Formula:
`successful first attempts / total delivery attempts`

From scanner events:
- detect whether first customer visit resulted in success or failure

#### 3. Late Stop Rate
Share of stops where actual arrival exceeds planned arrival or window threshold.

#### 4. Average Route Delay
Difference between actual route completion time and planned route completion time.

---

### B. Operational efficiency KPIs

#### 5. Average Time per Delivery Stop
Includes arrival to departure at customer location.

Formula:
`Depart Customer - Arrive Customer`

Can be segmented by:
- zone
- package type
- building type
- driver
- time of day

#### 6. Travel Time vs Service Time Ratio
Indicates whether inefficiency lies more in driving or stop execution.

Formula:
`total moving time / total customer service time`

#### 7. Stops per Hour
`completed stops / active shift hours`

#### 8. Route Adherence Index
Composite KPI measuring stop-order conformance, timing conformance, and extra detour distance.

---

### C. Cost-related KPIs

#### 9. Fuel Consumption per km
If fuel data exists, directly:
`fuel used / km traveled`

If not, estimate using proxies:
- idling duration
- stop-and-go patterns
- aggressive acceleration proxies
- route length and congestion exposure

#### 10. Fuel Consumption per Delivered Package
`estimated fuel / successful deliveries`

#### 11. Maintenance Incident Rate
`unscheduled repair events / 1,000 vehicle-km` or `/ vehicle-month`

#### 12. Breakdown Impact Time
Total route disruption duration caused by vehicle issues.

---

### D. Asset and labor utilization KPIs

#### 13. Vehicle Utilization Rate
Time vehicle is actively used for routes versus available time.

#### 14. Driver Utilization Rate
`productive time / shift time`, where productive time includes driving and service, excluding excessive waiting/idling.

#### 15. Overtime Rate
`routes ending after planned shift / total routes`

---

### E. Delay and disruption KPIs

#### 16. Frequency and Duration of Traffic Delays
Using low-speed events, GPS dwell during movement corridors, or traffic tags:
- number of delay episodes per route
- average delay duration
- delay minutes per 100 km

#### 17. Unplanned Stop Frequency
Stops not linked to customer service, depot activity, or planned breaks.

#### 18. Re-delivery Rate
Packages requiring additional attempt(s), which directly drives extra cost.

---

### 2.2 Techniques to identify bottlenecks

### A. Performance-annotated process maps
Overlay average and median waiting/travel/service times on the discovered process. This shows where time accumulates:
- before first stop
- between stop 4 and stop 5
- at failed delivery handling
- at return-to-depot

### B. Throughput time decomposition
Split total route duration into:
- depot preparation time
- travel time
- stop service time
- idle/waiting time
- disruption time
- return time

This reveals whether delays are predominantly road-related, customer-related, or internal.

### C. Bottleneck segmentation
Analyze KPI distributions by:
- route
- territory/zone
- postal code
- time of day
- day of week
- driver
- vehicle type/age
- customer/building category

### D. Geo-spatial hotspot analysis
Map:
- recurring traffic delay locations
- long dwell stop locations
- failed delivery clusters
- breakdown locations

### E. Variant and cohort analysis
Compare:
- top 20% best-performing routes vs bottom 20%
- high-performing drivers vs low-performing
- old vehicles vs newer vehicles
- urban dense routes vs suburban routes

### F. Quantifying impact
For each bottleneck, estimate:
- delay minutes added per route
- extra km
- extra fuel cost
- number of late deliveries caused
- overtime cost
- re-delivery cost

Example:
- A recurring traffic hotspot between 08:00–09:00 adds 18 minutes on average to 23% of routes from Depot A.
- Failed first attempts in Zone C add 0.7 extra stops and 14 km per affected package.
- Vehicle V12’s warning-light disruptions account for 11 lost route hours per month.

## 3. Root Cause Analysis for Inefficiencies

Identifying bottlenecks is not enough; we need to validate why they occur.

### 3.1 Potential root causes and how to test them

### A. Suboptimal route planning

#### Potential issue
Dispatch may use mostly static planning, ignoring real execution patterns, traffic peaks, parking difficulty, and stop service variability.

#### How to validate
- compare planned sequence with actual frequent successful variants
- detect recurring beneficial reordering by drivers
- measure whether routes with high conformance are actually better or worse than adaptive routes
- identify routes that are structurally overloaded

If drivers repeatedly deviate from the plan but perform better, the planning model may be unrealistic rather than drivers being non-compliant.

---

### B. Inaccurate travel time estimation

#### Potential issue
Planned ETAs may rely on average road speed, ignoring congestion patterns by hour/day/zone.

#### How to validate
- compare planned leg times with actual leg times by route segment and time bucket
- calculate systematic bias:
  - e.g. downtown legs at 08:00 underestimated by 25%
- correlate late deliveries with underestimated travel segments

---

### C. Traffic congestion patterns

#### Potential issue
Specific corridors or neighborhoods cause repeated delays at peak times.

#### How to validate
- mine GPS low-speed episodes spatially and temporally
- link hotspots to late stops downstream
- create delay heatmaps by hour/day/location
- compare route variants that avoid these hotspots

---

### D. High service time variability at customer locations

#### Potential issue
Large apartments, offices, difficult parking, gated communities, or high parcel counts increase dwell time.

#### How to validate
- analyze `Arrive Customer  Depart Customer` duration distribution
- segment by location type, package count, zone, building category, driver
- separate likely parking/search time from customer interaction time where possible

A stop with 3-minute expected service but 11-minute actual median is a major route design issue.

---

### E. Vehicle breakdowns and maintenance impact

#### Potential issue
Aging vehicles or deferred maintenance create route interruptions, overtime, failed deliveries, and extra fleet imbalance.

#### How to validate
- connect unscheduled stops and warning events to maintenance history
- compare route punctuality for vehicles with recent repair incidents
- calculate disruption time and downstream missed deliveries
- identify whether certain vehicle models or mileage bands have higher disruption probability

---

### F. Driver behavior or skill differences

#### Potential issue
Different drivers may show different adherence, idling, service speed, parking efficiency, and failed-delivery handling behavior.

#### How to validate
- variant analysis by driver controlling for route difficulty
- compare:
  - stop service time
  - idle time
  - route adherence
  - first-attempt success
  - overtime rate
- identify whether some drivers systematically outperform peers on similar territories

This must be handled carefully and fairly, adjusting for route complexity and customer mix.

---

### G. Failed delivery attempts

#### Potential issue
Poor time window design, weak customer communication, or suboptimal stop timing causes avoidable failures and rework.

#### How to validate
- analyze failed delivery reasons by time of day, customer segment, and route sequence position
- compare failed-attempt rates for narrow vs broad time windows
- examine whether customers are visited outside historically successful time bands
- estimate re-delivery burden

---

### 3.2 Specific process mining analyses for root cause validation

### A. Variant analysis: high-performing vs low-performing routes
Compare process variants with:
- highest on-time rate
- lowest fuel per package
- lowest overtime
- highest stop productivity

Questions:
- Do high performers leave depot earlier?
- Do they avoid certain road segments at peak?
- Do they cluster nearby deliveries better?
- Do they have shorter service times at similar stops?

### B. Driver comparison controlling for route type
Cluster routes by complexity first, then compare drivers within cluster. This avoids unfair conclusions.

### C. Dwell-time analysis
Break stop duration into:
- parking/search proxy
- service interaction
- post-service delay

Use GPS and scanner timestamps to isolate where customer-stop inefficiency lies.

### D. Correlation with traffic patterns
Merge external or inferred traffic data with actual route performance:
- delay per corridor by hour
- lateness propagation downstream
- route-start time sensitivity

### E. Rework loop analysis
Mine the lifecycle of failed deliveries:
- first attempt fail
- requeue
- second attempt
- depot return
- customer service exception

This shows the hidden cost of failure loops.

### F. Maintenance process overlay
Overlay vehicle health/disruption events with route execution:
- whether warning lights tend to occur after long idle periods or specific mileage
- whether routes using certain vehicles have lower conformance

## 4. Data-Driven Optimization Strategies

Below are five concrete strategies; at least three would likely be implemented in phases.

---

### Strategy 1: Dynamic routing and departure-time optimization

#### Targets
- traffic-related delays
- late deliveries
- excess fuel from congestion and stop-and-go driving

#### Root cause addressed
- static route plans
- inaccurate travel time assumptions
- recurring congestion hotspots

#### Process mining support
From mined logs we can identify:
- road segments with recurring low-speed events
- hour-specific travel time inflation
- routes where lateness starts after a known hotspot
- beneficial driver deviations that outperform planned order

#### Action
- incorporate time-dependent travel times into route planning
- allow controlled dynamic re-sequencing during the day
- adjust route departure times by territory to avoid recurrent congestion windows
- define exception-based rerouting when delay threshold is breached

#### Expected KPI impact
- improved On-Time Delivery Rate
- reduced Average Route Delay
- lower fuel per km/package
- lower overtime rate
- improved route adherence to realistic plans

---

### Strategy 2: Territory and stop-sequence redesign using historical variants

#### Targets
- inefficient route geometry
- excess backtracking
- overloaded routes
- poor balance across vehicles/drivers

#### Root cause addressed
- route design based on static assumptions rather than actual execution performance
- poor clustering of stops
- unrealistic stop count assignments for dense urban zones

#### Process mining support
Using six months of actual traces:
- identify frequent actual stop clusters
- detect repeated manual reordering by drivers
- find routes with chronic lateness despite good execution
- compare route productivity by territory

#### Action
- redesign delivery territories based on actual travel and service behavior
- rebalance stop density and parcel volume across vans
- adjust expected route lengths and stop counts by zone complexity
- use historical best-performing sequence patterns as planning templates

#### Expected KPI impact
- lower travel time vs service time ratio
- more stops per hour
- fewer late deliveries
- reduced km per package
- more balanced vehicle and driver utilization

---

### Strategy 3: Customer time-window and communication optimization to reduce failed deliveries

#### Targets
- failed delivery attempts
- re-delivery cost
- wasted travel and service time

#### Root cause addressed
- customers unavailable at attempted times
- poorly chosen time windows
- insufficient pre-arrival communication

#### Process mining support
Logs can show:
- when failed attempts occur most often
- which customer segments or locations have high fail rates
- whether failures correlate with early/late visits versus requested windows
- cost of reattempt loops

#### Action
- refine time-window promises using historical success probabilities
- introduce proactive customer notifications with narrower ETA updates
- prioritize high-risk recipients later/earlier based on historical availability
- enable customer rescheduling or pickup alternatives before failed attempt occurs

#### Expected KPI impact
- higher first-attempt success rate
- lower re-delivery rate
- improved on-time rate for all customers due to less rework
- lower fuel and labor cost per delivered package

---

### Strategy 4: Predictive maintenance scheduling and vehicle risk management

#### Targets
- mid-route breakdowns
- unscheduled stops
- overtime and missed deliveries caused by vehicle issues
- high maintenance cost from reactive repairs

#### Root cause addressed
- maintenance done too late or without considering route risk/vehicle usage pattern
- problematic vehicles repeatedly assigned to demanding routes

#### Process mining support
By linking route events with maintenance logs:
- identify vehicles with repeated warning-light stops
- estimate disruption patterns by vehicle, mileage, route type
- quantify delivery impact of mechanical incidents

#### Action
- create risk scores for vehicles based on usage, breakdown history, idle patterns, warning events
- schedule preventive maintenance before high-risk periods
- avoid assigning high-risk vehicles to long or time-critical routes
- redesign spare vehicle allocation

#### Expected KPI impact
- lower maintenance incident rate
- reduced route disruption time
- improved on-time delivery
- lower overtime and emergency repair cost
- higher vehicle availability

---

### Strategy 5: Targeted driver coaching and best-practice standardization

#### Targets
- excessive idling
- long service times
- low route adherence where harmful
- inconsistent productivity

#### Root cause addressed
- skill differences
- local knowledge gaps
- inefficient stop execution habits
- avoidable fuel-wasting behavior

#### Process mining support
Comparative analysis can identify drivers who:
- achieve lower dwell times on similar routes
- have better first-attempt success
- avoid unnecessary detours
- show lower idling duration

#### Action
- use peer-benchmarking adjusted for route difficulty
- coach on parking approach, stop preparation, scanning discipline, eco-driving
- codify best-performing route tactics into SOPs and planning rules
- provide route-specific onboarding for substitute drivers

#### Expected KPI impact
- improved stops per hour
- lower average service time
- lower fuel consumption
- lower overtime rate
- more consistent service performance

## 5. Operational Constraints and Continuous Monitoring

### 5.1 Accounting for operational constraints

Any optimization must respect operational realities.

### A. Driver working hours
Dynamic routing or territory redesign must ensure:
- legal maximum driving/working time
- required breaks
- overtime minimization
- feasible route completion

In process mining dashboards, I would track route plans and actual execution against shift limits and break compliance.

### B. Vehicle capacity
Route redesign must consider:
- parcel volume and weight
- loading sequence
- special parcel requirements
- need to avoid reloads or mid-route depot returns

Conformance analysis should flag routes where actual stop sequence causes inefficient unloading or capacity stress.

### C. Customer time windows
Optimization cannot simply minimize travel time if it causes time-window violations. The planning model should explicitly balance:
- travel efficiency
- stop sequence
- probability of successful handover
- SLA priority

### D. Maintenance availability
Predictive maintenance scheduling must not reduce fleet capacity below daily demand. The decision needs a fleet-wide balancing view.

### E. Service fairness and route complexity
Driver comparisons should normalize for:
- urban density
- high-rise delivery complexity
- parking restrictions
- package count per stop
- customer accessibility

This avoids false conclusions and unfair interventions.

### D. Closed-loop improvement approach

I would recommend a continuous cycle:

1. **Discover** actual operations
2. **Check conformance** vs plan
3. **Diagnose** bottlenecks and root causes
4. **Implement** targeted interventions
5. **Monitor** process and KPI changes
6. **Refine** planning rules and operating procedures

This is especially important in last-mile logistics because traffic, demand patterns, customer behavior, and fleet conditions all change over time.

## Final recommendation summary

For Speedy Parcels, process mining should be used not just to visualize routes, but to create a **multi-level operational intelligence system** across route execution, parcel outcomes, and vehicle reliability.

The highest-value opportunities are likely to come from:

1. **Making route planning reality-based** using mined actual travel and stop performance.
2. **Reducing failed delivery rework** through smarter time windows and customer communication.
3. **Lowering disruption and cost** via predictive maintenance and targeted driver/route interventions.

If implemented well, this approach should improve:
- on-time delivery performance
- fuel efficiency
- first-attempt success
- route productivity
- maintenance-related uptime
- overtime control

while staying within key constraints like driver hours, capacity, and customer SLAs.
