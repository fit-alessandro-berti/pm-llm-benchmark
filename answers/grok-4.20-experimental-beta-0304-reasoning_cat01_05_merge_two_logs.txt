**Merged Event Log**

**Primary Timestamp Rule**: 
- For matched events, Log A’s timestamp is used as the primary timestamp (as Log A is designated the primary timeline). 
- Log B’s timestamp is retained for reference.
- Events are sorted chronologically by the primary timestamp.

**Matching Rules Applied**:
- Matches based on `order_id` + semantic similarity of event names + time delta  5 seconds (the suggested <2s tolerance was used as a guideline; 5s was accepted for Payment because the events occur in the same logical sequence position with clearly related naming).
- Event name normalized to Log A’s convention when a match is made.
- All Log B attributes (`user_id`, `resource_id`, `notes`) are carried over for matched events.
- Unmatched events are kept with `N/A` in missing fields and their original source indicated.

### Summary of Decisions
- **Strong matches** (Order Received, Order Validated, Item Shipped): time deltas of 1–2 seconds + very similar naming  confidently merged.
- **Payment events**: 5-second offset is slightly outside the strict 2s example but is the only payment-related activity in both logs and appears at the expected point in the sequence  merged with note.
- **Quality Check**: clearly belongs to the process but has no temporal or naming counterpart in Log A  kept as a B-only event.
- **Item Delivered**: appears only in Log A  retained as-is.
- No events were discarded. All attributes from both systems are preserved where available.

This merged log provides a single source of truth with enriched context from System B while respecting the primary timeline of System A.
