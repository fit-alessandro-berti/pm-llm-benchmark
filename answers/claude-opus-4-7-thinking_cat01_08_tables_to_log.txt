# Comprehensive Event Log

Below is the integrated event log derived from the provided ERP tables. Each row represents a single event in the lifecycle of an order (Case ID = `order_id`). Events are sorted chronologically within each case.

## Assumptions
1. **Case ID** = `order_id`. Events related to invoices and payments are linked back to the order via `invoice_id  order_id`.
2. Each **picked item** produces a separate "Items Picked" event (one per order line) to preserve granularity.
3. Each **delivery record** produces two events: "Order Dispatched" and "Order Arrived".
4. Each **invoice** and **payment** produces its own event (supporting the fact that orders may have multiple invoices/payments).
5. **Resource names** have been resolved from the Resources table for readability; IDs are kept as additional attributes.
6. Additional attributes (quantity, item description, carrier, amount, method, etc.) are included where meaningful.

## Notes / Observations

- **Case 1004** is **incomplete**: it has an order-creation event and one "Items Picked" event, but **no delivery, invoice, or payment**. This likely represents an open/in-progress case (may be relevant during process-mining conformance checking).
- **Case 1002** is a good example of a **split delivery** scenario (two dispatches, two arrivals) and a **partial invoicing** case (two invoices totaling 500.00, each separately paid).
- **Payments** do not have a resource attribute in the source data, so the "Resource" column for those events is marked as *(System)*. If payment processor / clerk data becomes available, this can be enriched.
- Events are sorted chronologically per case; when two events share the same timestamp (none here), the logical order would still be enforced (e.g., Dispatched before Arrived).
- All resource IDs have been resolved to human-readable names via the Resources table for analyst-friendly reading, while the structured IDs can be reintroduced easily for automated tooling.
