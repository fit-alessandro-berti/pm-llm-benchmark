Here is a complete, production-ready DuckDB SQL query that fulfills all your requirements:

```sql
WITH CaseVariants AS (
    -- 1. Build the ordered activity sequence per case
    SELECT 
        case_id,
        list_agg(activity ORDER BY timestamp) AS variant_sequence
    FROM event_log
    GROUP BY case_id
),
VariantCounts AS (
    -- 2 & 3. Count how many cases follow each unique variant
    SELECT 
        variant_sequence,
        COUNT(*) AS case_count
    FROM CaseVariants
    GROUP BY variant_sequence
),
TopKVariants AS (
    -- 3. Keep only the top K most frequent variants
    SELECT variant_sequence
    FROM VariantCounts
    ORDER BY case_count DESC
    LIMIT :K  -- Replace :K with your target integer (e.g., 10)
),
TopKCases AS (
    -- 4. Map the top variants back to their corresponding case_ids
    SELECT cv.case_id
    FROM CaseVariants cv
    JOIN TopKVariants tkv ON cv.variant_sequence = tkv.variant_sequence
)
-- Final: Return all original events belonging to cases in the top K variants
SELECT e.*
FROM event_log e
JOIN TopKCases tkc ON e.case_id = tkc.case_id
ORDER BY e.case_id, e.timestamp;
```

###  Step-by-Step Breakdown
| CTE / Clause | Purpose |
|--------------|---------|
| `CaseVariants` | Groups by `case_id` and uses `list_agg()` to create an ordered array of activities per case. DuckDB natively preserves the `ORDER BY timestamp` inside the aggregation. |
| `VariantCounts` | Groups identical `variant_sequence` arrays together and counts how many cases share each sequence. |
| `TopKVariants` | Sorts variants by frequency descending and slices the result to the top `K`. |
| `TopKCases` | Joins back to `CaseVariants` to extract only the `case_id`s that match the top `K` sequences. |
| Final `SELECT` | Filters the original `event_log` to only rows belonging to those `case_id`s, returning the full event details in chronological order. |

###  DuckDB-Specific Notes
1. **`list_agg()`**: Returns a native `LIST` type. DuckDB supports direct equality comparison between lists (`cv.variant_sequence = tkv.variant_sequence`), making variant matching both safe and efficient.
2. **Parameterization**: `:K` is a named parameter placeholder. In DuckDB CLI, Python, or R drivers, you can pass it safely (e.g., `con.execute(query, {"K": 5})`). If running interactively, just replace `:K` with a literal like `10`.
3. **Tie-Breaking**: If multiple events share the exact same `timestamp` within a case, their relative order in the variant sequence is non-deterministic. If your data has ties, consider adding a secondary sort key: `list_agg(activity ORDER BY timestamp, activity)` or include a primary key like `event_id` if available.
4. **Performance**: For very large logs, `list_agg` + list comparison is highly optimized in DuckDB. If you encounter memory constraints, swapping `list_agg()` for `string_agg(activity, '' ORDER BY timestamp)` and comparing strings can reduce overhead slightly.
