This pseudo-BPMN describes a **customer request handling process** that starts when a request is received and ends when the customer is either:

- **rejected early** for an infeasible custom request, or
- **approved, invoiced, and formally confirmed**.

## 1) Overall business meaning of the process

At a high level, the process does four things:

1. **Intake the request**
2. **Determine whether it is standard or custom**
3. **Do the appropriate checks and preparation**
4. **Apply approvals if needed, then finalize and confirm**

It is essentially a mix of **order validation, feasibility assessment, commercial preparation, approval, and customer communication**.

---

## 2) Step-by-step explanation of each part

### Start Event
The process begins when a customer request enters the organization.

### Task A: “Receive Customer Request”
This is the intake step. The business likely:

- records the request,
- captures customer details,
- identifies products/services requested,
- notes deadlines or delivery expectations,
- determines whether the request appears standard or customized.

This task creates the case that the rest of the process works on.

---

## 3) First decision: standard vs custom request

### Gateway (XOR): “Check Request Type”
This is an **exclusive gateway**. That means **only one path is chosen**.

- If the request is **Standard**, it goes to the standard-processing branch.
- If the request is **Custom**, it goes to the custom-processing branch.

### Meaning of XOR here
An XOR gateway represents a **decision based on conditions**. It does **not** split work into parallel tasks. It selects exactly one route.

---

# 4) Standard request path

A **standard request** usually means the customer is asking for something already defined in the normal catalog, service list, or operating model.

### Task B1: “Perform Standard Validation”
Purpose:

- verify the request is complete,
- confirm it matches an existing standard offering,
- check customer/order data,
- ensure normal business rules are met.

This is a routine validation step for non-custom work.

---

### Gateway (AND): “Run Parallel Checks”
This is a **parallel gateway**. It starts two tasks at the same time:

- **Task C1: Credit Check**
- **Task C2: Inventory Check**

### Meaning of AND here
An AND gateway means **both branches happen in parallel**. It is used to save time when tasks can be performed simultaneously.

---

### Task C1: “Credit Check”
Purpose:

- confirm the customer is financially acceptable,
- verify credit limit or payment reliability,
- assess financial risk before moving forward.

---

### Task C2: “Inventory Check”
Purpose:

- confirm the requested item is available,
- check stock or production capacity,
- see whether the organization can fulfill the request from current resources.

---

### “All Parallel Checks Completed” (Join)
This is the **parallel join**. It waits until **both** the credit check and inventory check are finished.

Important meaning:
- The process **cannot continue** after the parallel split until **all required parallel tasks** are complete.

---

### Task D: “Calculate Delivery Date”
Once validation and checks are done, the business determines a realistic delivery commitment.

Purpose:

- estimate when the order can be fulfilled,
- use stock/capacity information,
- set a customer-facing delivery promise.

So for standard requests, the key output is usually:
- **a validated order**
- **and a delivery date**

---

# 5) Custom request path

A **custom request** means the customer wants something outside the normal standard offering.

### Task B2: “Perform Custom Feasibility Analysis”
This is different from standard validation. Instead of checking a predefined offering, the business asks:

- Can we actually do this?
- Is it technically possible?
- Is it operationally manageable?
- Do we have the resources, capability, or time?

This is a more specialized assessment than the standard path.

---

### Gateway (XOR): “Is Customization Feasible?”
This is another exclusive decision.

- **Yes**  continue with preparing a custom offer
- **No**  reject the request

---

### If Yes  Task E1: “Prepare Custom Quotation”
If the custom work is feasible, the business prepares a tailored commercial proposal.

This usually includes:

- custom pricing,
- scope,
- assumptions,
- lead time,
- special terms,
- possibly a proposed delivery schedule.

For the custom branch, the main output is not just a delivery promise, but a **custom quotation**.

---

### If No  Task E2: “Send Rejection Notice”  End Event
If the customization is not feasible, the request is ended immediately.

Purpose:

- inform the customer the requested customization cannot be provided,
- close the case cleanly.

Important point:
- This branch **does not rejoin** the main flow.
- It ends the process right there.

---

# 6) Where the branches converge

After the successful part of either branch:

- **Standard path** reaches completion after **Task D**
- **Custom path** reaches completion after **Task E1**

Then both move to the next decision:

### “After Standard or Custom Path Tasks Completed”
This is effectively a **merge point**.

Business meaning:
- if the standard branch succeeded, continue;
- if the custom branch succeeded, continue;
- if custom was infeasible and rejected, the process has already ended and does not come here.

### BPMN interpretation
Because only one of those branches is active at a time, this is conceptually an **exclusive merge** rather than a parallel join.

---

# 7) Approval logic

### Gateway (XOR): “Is Approval Needed?”
This is a policy/control decision.

It asks whether the request must be approved by a manager before finalization.

Possible reasons approval may be needed include:

- high order value,
- unusual pricing,
- low margin,
- special delivery commitments,
- custom work,
- risk concerns,
- exception to normal policy.

The pseudo-BPMN does not define the exact rule, but this gateway clearly represents a business rule check.

---

## If approval is needed

### Task F: “Obtain Manager Approval”
A manager reviews the request, quote, delivery commitment, or conditions.

What is being approved depends on the branch:

- for **standard requests**, likely the delivery/date/terms,
- for **custom requests**, likely the custom quotation and associated conditions.

---

### Gateway (XOR): “Is Approval Granted?”
Again, exactly one outcome:

- **Yes**  proceed
- **No**  rework is required

---

### If Yes  Task G: “Generate Final Invoice”
The process can now finalize the commercial document.

Business meaning:
- the request is internally cleared,
- a final billing or formal commercial output is generated.

---

### If No  Task H: “Re-evaluate Conditions”
This is the **rework step**.

Purpose:
- revise the terms so they become acceptable,
- adjust commercial or operational conditions,
- address whatever caused the approval rejection.

This is not a full restart of the process. It is a targeted correction step.

---

# 8) How the rework loop functions

After **Task H**, the process loops back differently depending on the original path:

- **Custom request**  loop back to **Task E1: Prepare Custom Quotation**
- **Standard request**  loop back to **Task D: Calculate Delivery Date**

## What that means in practice

### For standard requests
If approval is denied, the business does not go all the way back to request intake.  
Instead, it reworks the fulfillment conditions, most likely by revising:

- delivery timing,
- order terms,
- availability assumptions,
- possibly commercial conditions tied to the standard order.

Then it moves forward again.

### For custom requests
If approval is denied, the custom proposal is revised, such as:

- price,
- scope,
- assumptions,
- delivery commitment,
- terms of the custom offering.

Then it is resubmitted through the downstream approval logic.

---

## Important loop behavior
Because the loop returns to **Task D or Task E1**, and from there the flow goes again to **“Is Approval Needed?”**, the revised case may:

- still require approval, or
- no longer require approval if the reworked conditions now fall within normal policy.

So the model allows **re-evaluation and resubmission**.

---

# 9) If no approval is needed

If the gateway says **No Approval Needed**, the process bypasses manager review and goes directly to:

### Task G: “Generate Final Invoice”

This means the request is within normal authority and can be finalized immediately.

---

# 10) Final customer communication

### Task I: “Send Confirmation to Customer”
After invoice generation, the customer is formally notified.

This likely includes:

- confirmation that the request is accepted,
- final terms,
- delivery commitment,
- invoice or commercial details.

This is the final external communication step.

### End Event
The process is complete.

---

# 11) Key difference between standard and custom requests

## Standard request path
Used when the request matches existing offerings.

Main characteristics:

- routine validation,
- parallel operational/financial checks,
- delivery-date calculation,
- typically less uncertainty.

Key tasks:
- B1 Standard Validation
- C1 Credit Check
- C2 Inventory Check
- D Calculate Delivery Date

## Custom request path
Used when the request requires something non-standard.

Main characteristics:

- feasibility must be proven first,
- higher uncertainty,
- tailored quote preparation,
- may be more likely to need approval.

Key tasks:
- B2 Custom Feasibility Analysis
- Feasibility decision
- E1 Custom Quotation
- or E2 Rejection if not feasible

## Core difference
- **Standard** = validate and fulfill an existing offer.
- **Custom** = determine if a special offer can even be created, then price and propose it.

---

# 12) When approvals and rework are triggered

## Approval is triggered when:
The process reaches **“Is Approval Needed?”** and the business rule says approval is required.

Typical triggers might be:

- custom work,
- exceptional discounts,
- high-risk customer,
- large order value,
- unusual delivery commitment,
- policy exception.

## Rework is triggered when:
Approval was requested **and not granted**.

That causes:
- **Task H: Re-evaluate Conditions**
- then a loop back to revise the relevant output:
  - **Task D** for standard
  - **Task E1** for custom

## Rejection is triggered when:
A **customization is not feasible**.

That leads to:
- **Task E2: Send Rejection Notice**
- **End Event**

That is not rework; it is termination.

---

# 13) Meaning of the gateways, summarized

## XOR gateways
Used for decisions where **only one path** is taken:

- Check Request Type
- Is Customization Feasible?
- Is Approval Needed?
- Is Approval Granted?

## AND gateway
Used for parallel processing where **all branches must complete**:

- Run Parallel Checks
- All Parallel Checks Completed (join)

---

# 14) A few BPMN review notes

The logic is clear, but there are a few implicit modeling assumptions:

1. **The merge after standard/custom completion is implicit**  
   In formal BPMN, this would typically be shown as an XOR merge or simple merge.

2. **The process remembers whether the case was standard or custom**  
   That is necessary so the rework loop knows whether to return to **D** or **E1**.

3. **Credit and inventory exceptions are not explicitly branched**  
   The diagram shows those checks happening, but does not show what happens if they fail. So the model appears to assume their outcomes are absorbed into later decisions or operational adjustments.

4. **Repeated approval rejection could create a loop**  
   As written, the process can keep cycling through re-evaluation and approval until acceptable conditions are found. A real BPMN model might add an explicit stop/rejection path after repeated denials.

---

## In one sentence
This process handles customer requests by first determining whether they are standard or custom, performing the appropriate validation or feasibility work, optionally seeking managerial approval, reworking the request if approval is denied, and then finalizing and confirming the request unless a custom request is found infeasible and rejected early.
