```sql
WITH ordered_events AS (
    SELECT
        case_id,
        activity,
        timestamp,
        -- Look at the next event and the one after it (direct succession)
        LEAD(activity, 1) OVER (PARTITION BY case_id ORDER BY timestamp) AS next_activity,
        LEAD(activity, 2) OVER (PARTITION BY case_id ORDER BY timestamp) AS next_next_activity,
        LEAD(timestamp, 1) OVER (PARTITION BY case_id ORDER BY timestamp) AS next_timestamp,
        LEAD(timestamp, 2) OVER (PARTITION BY case_id ORDER BY timestamp) AS next_next_timestamp
    FROM event_log
),

-- Cases that contain the exact direct-succession sequence
-- 'Approve Purchase Requisition' -> 'Create Order' -> 'Confirm Order'
-- AND where Confirm Order occurs more than 5 days after Create Order
cases_to_exclude AS (
    SELECT DISTINCT case_id
    FROM ordered_events
    WHERE activity            = 'Approve Purchase Requisition'
      AND next_activity       = 'Create Order'
      AND next_next_activity  = 'Confirm Order'
      -- Time between 'Create Order' (next_timestamp) and 'Confirm Order' (next_next_timestamp)
      AND next_next_timestamp > next_timestamp + INTERVAL 5 DAY
)

-- Return all events EXCEPT those belonging to the excluded cases
SELECT
    el.case_id,
    el.activity,
    el.timestamp
FROM event_log AS el
WHERE el.case_id NOT IN (SELECT case_id FROM cases_to_exclude)
ORDER BY el.case_id, el.timestamp;
```

### Explanation

1. **`ordered_events`** – Uses the `LEAD` window function (partitioned by `case_id`, ordered by `timestamp`) to attach each event's **next** and **next-next** activity and timestamp. This is what enforces *direct succession*: the three activities must be consecutive within the case, with no event in between.

2. **`cases_to_exclude`** – Filters to rows where:
   - the current activity is `'Approve Purchase Requisition'`,
   - the immediately following activity is `'Create Order'`,
   - the activity two steps ahead is `'Confirm Order'`, and
   - the gap between the `'Create Order'` timestamp (`next_timestamp`) and the `'Confirm Order'` timestamp (`next_next_timestamp`) is **more than 5 days** (`INTERVAL 5 DAY`).
   
   `DISTINCT` collapses this to a clean list of offending `case_id`s.

3. **Final query** – Selects all events from `event_log` whose `case_id` is **not** in the exclusion list, ordered for readability.

### Notes / Variations
- If your DuckDB version prefers it, `next_timestamp + INTERVAL 5 DAY` can also be written as `next_timestamp + INTERVAL '5' DAY`.
- To be robust against `NULL` handling with `NOT IN`, you could swap the final filter to an anti-join:

```sql
SELECT el.case_id, el.activity, el.timestamp
FROM event_log el
LEFT JOIN cases_to_exclude x ON el.case_id = x.case_id
WHERE x.case_id IS NULL
ORDER BY el.case_id, el.timestamp;
```

This avoids any edge cases where `NULL` values in the exclusion set could affect `NOT IN`.
