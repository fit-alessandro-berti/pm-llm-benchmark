**BPMNlike textual process model for the Invoice Approval Workflow**

```text
Start
  |
  v
1. Receive Invoice (Electronic Email)
  |
  v
2. Check PO Number (Accounts Payable Clerk – Mary)
      |
      |-- If PO Missing  Send Request to Supplier (Task 2a)
      |                     |
      |                     v
      |                2a. Supplier Provides PO or Details
      |                     |
      |                     v
      |                  Return to Step 2 (recheck)
      |
      |-- If PO Present  Log Invoice in Accounting System (Task 2b)
      |
      v
3. Forward to Purchasing for Confirmation of Goods/Services (Task 3)
      |
      v
4. Purchasing Officer – Dan – Verify Invoice against PO (Task 4)
      |-- If Discrepancy detected  Communicate to Accounts Payable (Task 4a)
      |                               |
      |                               v
      |                        4a. Resolve Issue (e.g., supplier clarification)
      |                               |
      |                               v
      |                          Loop back to Step 3 or Step 2b as needed
      |
      v
5. Match Invoice Line Items with PO in System (Task 5)
      |
      |-- If Match OK  Send to Department Manager for Approval (Task 6)
      |                     |
      |                     v
      |                6. Department Manager – Carlos – Review & Approve (Task 6)
      |                     |
      |                     |-- If Issue / Suspicion detected  Reject or Hold (Task 6a)
      |                     |                               |
      |                     |                               v
      |                     |                        6a. Investigate Further (e.g., supervisor input)
      |                     |                               |
      |                     |                               v
      |                     |                          Loop back to Step 5
      |                     |
      |                     v
      |                6b. Approval Granted  Proceed to Next Step
      |
      v
7. Exception Handling / Escalation (Accounts Payable Supervisor – Rita)
      |-- Triggered when Mary cannot resolve a matching or POmissing issue
      |        Investigate Supplier, Create Retrospective PO, or Reject
      |       |
      |       v
      |    7a. Issue Resolved  Return to normal flow (Step 5)
      |       |
      |       v
      |    7b. Issue Unresolved  Escalate to Rita’s review (already captured)
      |

      |       |
      |       v
      |    8a. Supplier Supplies Correct PO or Corrected Invoice
      |       |
      |       v
      |    8b. Pricing Verified  Invoice moves forward
      |
```

### Process Elements Explained

| Element | Description |
|---------|-------------|
| **Activities / Tasks** | 1Receive Invoice, 2Check PO, 2aSupply PO, 2bLog Invoice, 3Forward to Purchasing, 4Dan verifies PO, 4aResolve discrepancy, 5Match invoice, 6Manager approval (6areject/hold, 6bapprove), 7Supervisor exception handling, 8Karen assists, 9Schedule payment |
| **Sequence Flow** | Linear progression from receiving the invoice  PO check  purchasing confirmation  invoice matching  manager approval  payment scheduling. Alternate flows (2a, 4a, 6a, 7, 8) are conditional branches triggered by exceptions. |
| **Gateways / Decision Points** | • Decision after PO check: *PO missing?* (leads to 2a vs. continue) <br> • Decision after Dan’s verification: *Discrepancy?* (leads to 4a) <br> • Decision after matching: *Match OK?* (leads to manager approval) <br> • Decision after manager review: *Issue detected?* (leads to 6a) |
| **Roles / Participants** | • **Accounts Payable Clerk (Mary)** – initial receipt, PO check, logging, payment scheduling <br> • **Purchasing Officer (Dan)** – confirms PO match <br> • **Department Manager (Carlos)** – final approval <br> • **Accounts Payable Supervisor (Rita)** – exception escalation & resolution <br> • **Supplier Relationship Manager (Karen)** – assists with POmissing or pricing issues |
| **Artifacts / Inputs / Outputs** | • **Input**: Email invoice (document) <br> • **Intermediate Artifact**: PO number, confirmation email from Dan, systemgenerated matching report <br> • **Output**: Approved invoice ready for payment schedule |
| **Exceptions / Variations** | • PO missing  escalation to supplier (Task2a) <br> • Discrepancies in invoice vs. PO  communication loop (Task4a) <br> • Manager flags suspicious invoice  possible rejection or hold (Task6a) <br> • Complex exceptions requiring retrospective PO or rejection  handled by Supervisor (Task7) and Supplier Manager (Task8) |
| **Uncertainties / Note** | The interviews slightly differ on trigger timing of the manager’s approval (Mary says system autoemails; Dan assumes Mary moves it after confirmation). The model captures both: the system autoemail (implicit trigger) and the manual forwarding step, indicating the process may use either depending on configuration. |

This textual model can be directly translated into a BPMN diagram (e.g., using a modeling tool) by mapping each numbered task as an **Activity**, each decision as a **Gateway**, and linking them with **Sequence Flows** as described. The model reflects the consensus of the five stakeholder interviews while explicitly noting conditional branches for exceptions identified by Rita and Karen.
