## Rationale for Each Added Constraint

| Constraint | Type | Bias-Mitigation Effect |
|-----------|------|------------------------|
| `existence: BiasMitigationCheck` | unary | Guarantees every case includes a fairness review—no trace can skip oversight. |
| `absence: AutoReject_Sensitive` | unary | Categorically forbids any automated reject path triggered solely by a sensitive attribute. |
| `exactly_one: BiasMitigationCheck` | unary | Ensures the review happens once and is auditable (no omission, no dilution). |
| `responded_existence: Check*  ManualReview` | binary | Any inspection of a sensitive attribute mandates a human ManualReview somewhere in the trace. |
| `coexistence: Approve/Reject_Minority  ManualReview` | binary | Decisions on sensitive-demographic applicants must be accompanied by manual oversight. |
| `coexistence: FinalDecision  BiasMitigationCheck` | binary | A final outcome cannot exist without an accompanying fairness check. |
| `response: Check*  BiasMitigationCheck` | binary | After any sensitive-attribute event, a fairness check is eventually guaranteed. |
| `precedence: Approve/Reject  BiasMitigationCheck` | binary | A decision is only valid if a fairness check preceded it. |
| `succession: BiasMitigationCheck  FinalDecision` | binary | Tightly couples the check and the decision in both directions. |
| `altprecedence / chainprecedence: Reject  BiasMitigationCheck` | binary | A reject must be *immediately* (or in strict alternation) preceded by the fairness check, eliminating any intervening biased trigger. |
| `chainresponse: CheckApplicantRace  BiasMitigationCheck` | binary | The very next step after viewing race must be the fairness check—never a decision. |
| `noncoexistence: Reject_Minority  AutoReject_Sensitive` | binary | Prevents biased and automated reject labels from co-occurring. |
| `nonsuccession: Check*  Reject` | binary | Forbids any path where a sensitive check is later followed by a reject without intervening review. |
| `nonchainsuccession: Check*  Decision` | binary | Most direct guard: a decision can never *immediately* follow a sensitive-attribute event. |

## Summary

These constraints enforce a **mandatory fairness checkpoint (`BiasMitigationCheck` / `ManualReview`) between any disclosure of a sensitive attribute and any decision**. The combination of:

- **chain/non-chain rules** (blocking *immediate* biased transitions),
- **response/precedence rules** (guaranteeing review *eventually* surrounds decisions), and
- **coexistence/existence rules** (guaranteeing oversight is *present* in every relevant trace),

makes it structurally impossible for a sensitive attribute to flow directly into an Approve/Reject outcome. This reduces both **disparate treatment** (a sensitive attribute directly causing a decision) and **disparate impact** (systematically different paths for sensitive demographics), while keeping every fairness step auditable via the `exactly_one` and `existence` guarantees.
