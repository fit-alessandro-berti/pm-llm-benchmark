Below is a readytorun DuckDBSQL solution that returns **all rows** from `event_log` whose `case_id` belongs to a case that contains **six events or fewer**.  
Feel free to pick the style you like best – the three versions below are logically identical and will all produce the same result.

---

## 1  CTE+JOIN (most explicit)

```sql
WITH case_counts AS (
    -- count events per case and keep only those 6
    SELECT case_id,
           COUNT(*) AS event_cnt
    FROM   event_log
    GROUP BY case_id
    HAVING COUNT(*) <= 6
)

SELECT e.*
FROM   event_log e
JOIN   case_counts c
       ON e.case_id = c.case_id;
```

**Why it works**

* The CTE `case_counts` aggregates the table and throws away any `case_id` that has more than six rows.
* The subsequent `JOIN` brings back every row of the original table **only** for the surviving cases.

---

## 2  Subquery with `IN` (concise)

```sql
SELECT *
FROM   event_log
WHERE  case_id IN (
    SELECT case_id
    FROM   event_log
    GROUP BY case_id
    HAVING COUNT(*) <= 6
);
```

**Why it works**

* The inner query returns the set of `case_id`s that satisfy the 6 condition.
* The outer query keeps every row whose `case_id` belongs to that set.

---

## 3  Windowfunction + `QUALIFY` (singlescan, DuckDBspecific)

```sql
SELECT  case_id,
        activity,
        timestamp,
        cnt
FROM (
    SELECT *,
           COUNT(*) OVER (PARTITION BY case_id) AS cnt
    FROM   event_log
) AS with_cnt
QUALIFY cnt <= 6;
```

**Why it works**

* The `COUNT(*) OVER (…)` computes the size of each case onthefly without an extra aggregation step.
* `QUALIFY` acts like a `HAVING` clause for window functions, keeping only rows whose windowaggregated count 6.

> **Tip:** If you don’t need the extra `cnt` column in the final result, simply omit it from the `SELECT` list (e.g., `SELECT case_id, activity, timestamp …`).

---

### Which version to use?

| Version | Pros | Cons |
|---------|------|------|
| **CTE+JOIN** | Easy to read, clearly separates “countfilter” from “output”. | Twopass (one for the CTE, one for the join). |
| **INsubquery** | Very compact, onepass for the inner aggregation. | Might be slightly slower on very large tables because the outer query cannot push the filter down before the subquery runs. |
| **Window+QUALIFY** | Single table scan (if DuckDB can push the window through the storage), no extra join. | Slightly more advanced syntax; relies on DuckDB’s `QUALIFY` support. |

All three will work in DuckDB0.8+ and are fully standardSQLcompatible (the `QUALIFY` clause is the only nonstandard extension). Choose the one that best matches your team’s coding style and performance needs.
