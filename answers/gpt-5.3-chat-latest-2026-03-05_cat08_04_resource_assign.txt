1) Analyzing resource behavior and assignment patterns

Using the event log, I would first enrich the dataset into a resource-centric process mining model where each case is a ticket and each event includes attributes such as agent, tier, skill, priority, and category. The goal is to understand how work actually flows through agents and tiers.

Key performance and behavior metrics

Agent-level metrics  
• Workload distribution: number of tickets handled per agent per day/week  
• Average handling time per activity (Work L1, Work L2, Work L3)  
• Average queue waiting time before the agent starts work  
• Escalation rate per agent (tickets escalated after agent handling)  
• Reassignment rate per agent  
• SLA compliance rate for tickets handled by that agent  
• Skill utilization rate (percentage of tasks aligned with declared skills)

Tier-level metrics  
• Average resolution time by tier  
• Percentage of tickets resolved within tier (e.g., L1 first-call resolution)  
• Escalation rate from L1  L2 and L2  L3  
• Queue waiting times before work begins at each tier  
• Workload balance across tiers

Ticket-type handling metrics  
• Resolution time per category and required skill  
• Escalation likelihood by category  
• Reassignment frequency by category or skill

First-call resolution (L1) analysis  
• Percentage of tickets closed by L1 without escalation  
• Resolution success by category and priority  
• Activities performed by successful L1 cases vs escalated ones

Process mining techniques

Process discovery  
Build the real incident-handling model showing the typical flow:
Ticket Created  Assign L1  Work L1  Escalate  Assign L2  Work L2  etc.

This reveals whether tickets follow the intended tiered process or deviate (e.g., skipping tiers, repeated reassignments).

Resource interaction analysis  
By examining handovers between agents, we identify patterns such as:
L1 agent  specific L2 specialist clusters  
Frequent handovers between particular agents

Social network analysis (handover-of-work)  
Construct graphs showing who transfers work to whom. This can reveal:

• Bottleneck agents who receive excessive escalations  
• Informal expert hubs  
• Reassignment loops (A  B  C  B)

Role discovery  
Process mining can cluster agents based on behavior instead of declared tier. For example:

Cluster 1: mainly networking tickets  
Cluster 2: CRM/database issues  
Cluster 3: general troubleshooting

This may reveal that actual roles differ from formal tier design.

Comparing intended vs actual assignment logic

Intended logic:  
Round-robin assignment within tiers and manual escalation.

Actual behavior discovered may show:  
• Some agents receive disproportionate assignments  
• Dispatcher overrides round-robin for specific agents  
• Tickets circulate between multiple L2 agents before resolution  
• Certain skills cause repeated reassignment loops

Skill utilization analysis

Using the "Required Skill" and "Agent Skills" attributes:

• Skill match rate = percentage of assignments where agent skill matches required skill  
• Skill mismatch rate leading to reassignment  
• Specialist utilization rate (time spent on specialized tasks vs generic tasks)

Example insights:  
• DB-SQL specialists spending 40% of time on generic application tickets  
• Networking tickets often first assigned to non-network specialists causing reassignment  
• Some skills rarely used despite existing in agent profiles

This highlights whether expertise is being allocated efficiently.


2) Identifying resource-related bottlenecks and issues

Using performance overlays on the discovered process model, we can pinpoint delays and inefficiencies.

Skill availability bottlenecks

Example indicators:  
• Long waiting time between "Assign L2" and "Work L2 Start" for certain skills  
• Queue buildup for Networking-Firewall tickets

Metric example:  
Average waiting time for Networking-Firewall skill = 85 minutes vs 20 minutes for other skills.

This indicates insufficient capacity for that skill.

Reassignment delays

Using case analysis:

Average delay introduced per reassignment  
= Time between reassignment and next work start.

Example finding:  
Average delay per reassignment = 1.6 hours.

If 35% of tickets have at least one reassignment, the operational impact is significant.

Escalation inefficiency

Measure:  
• Tickets escalated from L1 but later resolved with basic actions

Example finding:  
25% of escalated tickets only required actions within L1 capability.

This suggests avoidable escalations.

Incorrect initial assignment

Compare:

Required Skill vs first assigned agent skill.

Example finding:  
40% of DB-SQL tickets initially assigned to agents without DB expertise.

These tickets experience longer resolution times and more reassignments.

Agent overload vs underutilization

Workload heatmaps reveal imbalance.

Example pattern:  
• 5 L2 specialists handle 60% of complex tickets  
• Several agents handle far fewer cases

This often happens when dispatchers informally route tickets to “known experts”.

Correlation with SLA breaches

Using case-level analysis:

Compare SLA breaches vs:

• number of reassignments  
• waiting time before skilled agent  
• escalation count

Example results:

• Tickets with 2 reassignments have 3.4× higher SLA breach probability  
• Tickets waiting >1 hour for skill match breach SLA 48% of the time

These metrics quantify operational impact.


3) Root cause analysis for assignment inefficiencies

Potential root causes

Skill-agnostic assignment rules  
Round-robin assignment ignores skill requirements and complexity.

Result:  
Tickets initially routed to agents who cannot resolve them.

Inaccurate skill profiles  
Agent skill databases may be outdated.

Example:  
Agent B12 listed as App-CRM but actually specializes in DB issues.

Poor ticket classification

If category or required skill is incorrect, routing decisions become flawed.

Example:  
CRM ticket incorrectly categorized as generic Software-App instead of DB-SQL.

Limited visibility into workload

Dispatchers or automated systems may not consider:

• current queue length  
• ongoing workload  
• agent availability

This causes uneven work distribution.

Insufficient L1 capability

If L1 agents lack training or knowledge articles, escalation rates increase unnecessarily.

Variant analysis

Compare two case groups:

Efficient cases  
• resolved with 1 assignment  
• no reassignment  
• SLA compliant

Problematic cases  
• multiple reassignments  
• long queue delays  
• SLA breaches

Process mining can identify distinguishing attributes such as:

• specific categories  
• certain dispatchers  
• certain agents  
• specific skills

Decision mining

By analyzing decision points like:

Escalate vs resolve at L1  
Assign to agent X vs Y

We can train decision models using historical data to identify which factors influence outcomes.

Example factors discovered:

• priority  
• ticket category  
• keywords in description  
• previous similar cases  
• agent workload

This helps identify why poor assignment decisions occur.


4) Data-driven resource assignment strategies

Strategy 1: Skill-based intelligent routing

Issue addressed  
Incorrect initial assignment and frequent reassignments.

Approach  
Use ticket attributes (category, required skill, keywords) to route directly to agents with matching skills.

Enhancements  
• skill proficiency levels  
• fallback routing when specialists unavailable

Required data  
• accurate agent skill profiles  
• historical ticket-skill mappings  
• category and keyword data

Expected benefits  
• fewer reassignments  
• faster resolution times  
• better specialist utilization

Example improvement:  
Skill match rate increases from 60% to 90%.


Strategy 2: Workload-aware dynamic assignment

Issue addressed  
Uneven workload distribution and queue delays.

Approach  
Assignment algorithm considers:

• current workload per agent  
• number of active tickets  
• average handling time  
• skill match

Routing score example:

Assignment score = skill match + workload availability + priority weighting.

Required data  
• real-time agent workload  
• ticket queues  
• historical handling times

Expected benefits  
• balanced workload  
• shorter waiting times before work begins  
• reduced SLA breaches


Strategy 3: Predictive ticket skill classification

Issue addressed  
Poor initial categorization and wrong skill estimation.

Approach  
Use machine learning or NLP on ticket descriptions to predict:

• required skill  
• likely resolution tier  
• complexity

For example, keywords like “SQL error”, “database timeout” automatically predict DB-SQL skill.

Required data  
• historical tickets and resolutions  
• description text  
• category corrections

Expected benefits  
• more accurate routing  
• fewer escalations  
• faster expert engagement


Strategy 4: Data-driven L1 empowerment

Issue addressed  
Excessive escalations.

Approach  
Use process mining insights to identify:

• ticket types frequently resolved by L2 but technically simple  
• tasks repeatedly escalated unnecessarily

Actions:

• create knowledge articles  
• train L1 agents  
• expand L1 tool access

Expected benefits  
• higher first-call resolution  
• reduced L2 workload  
• faster service.


5) Simulation, implementation, and monitoring

Using business process simulation

The mined process model can be enriched with:

• activity durations  
• queue waiting times  
• resource availability  
• escalation probabilities

We simulate scenarios such as:

Scenario A: current round-robin system  
Scenario B: skill-based routing  
Scenario C: skill + workload-aware routing

Simulation outputs:

• predicted SLA compliance  
• average resolution time  
• resource utilization  
• queue lengths

This allows evaluation of strategies before implementation.

Example outcome:

Skill-based routing simulation shows  
• 22% reduction in resolution time  
• 35% fewer reassignments  
• 15% SLA improvement.
