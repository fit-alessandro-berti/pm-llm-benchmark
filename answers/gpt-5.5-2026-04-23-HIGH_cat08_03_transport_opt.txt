## 1. Process Discovery and Conformance Checking

### 1.1 Data preprocessing and integration

Speedy Parcels has data from GPS trackers, handheld scanners, dispatch planning, and maintenance systems. Before applying process mining, these sources need to be converted into a unified event log.

A recommended event log schema would include:

| Field | Example | Purpose |
|---|---:|---|
| Case ID | `V12-20241205` | Vehicle-day operational case |
| Activity | `Depart Depot`, `Arrive Customer`, `Delivery Success` | Process step |
| Timestamp | `2024-12-05 08:25:10` | Event ordering and duration calculation |
| Vehicle ID | `V12` | Resource/fleet analysis |
| Driver ID | `D105` | Driver performance analysis |
| Package ID | `P9876` | Package-level lifecycle tracking |
| Route ID | `R-20241205-V12` | Planned route comparison |
| Location | Lat/lon or geofence | Geo/process analysis |
| Speed/status | Moving, idle, ignition off | Travel, idling, delay detection |
| Planned attributes | Planned stop sequence, ETA, time window | Conformance checking |
| Maintenance attributes | Repair type, downtime | Breakdown/root-cause analysis |

Because delivery operations involve multiple objects — vehicles, drivers, packages, stops, routes — I would recommend using both:

1. **Vehicle-day case view**: one case per vehicle per day, useful for route execution analysis.
2. **Package-level case view**: one case per package, useful for delivery success, failed attempts, and customer SLA analysis.
3. **Object-centric process mining view**: linking vehicle, driver, package, route, stop, and maintenance events together without forcing everything into one flat case structure.

### Key preprocessing steps

#### a. Timestamp harmonization

All systems must use a common timezone and timestamp format. Scanner devices, GPS trackers, and dispatch systems may have clock drift, so events need to be synchronized.

Example issue:

- GPS says the vehicle reached a location at 08:22.
- Scanner says “Arrive Customer” at 08:25.
- The difference may reflect walking/parking time, but it could also be scanner delay.

#### b. Identity resolution

The same delivery operation must be linked across systems:

- Vehicle ID from GPS.
- Driver ID from shift assignment.
- Route ID from dispatch.
- Package IDs from scanner.
- Maintenance events from fleet system.

This allows analysis such as: “Vehicle V12 on route R45 with driver D105 had a late delivery after an engine-warning stop.”

#### c. GPS event abstraction

Raw GPS data is too granular for process mining if every ping is treated as an event. GPS data should be transformed into meaningful logistics activities, for example:

- `Depart Depot`
- `Travel to Stop`
- `Low Speed / Traffic Delay`
- `Idle`
- `Arrive Near Customer Geofence`
- `Unplanned Stop`
- `Route Deviation`
- `Arrive Depot`

Map matching and geofencing should be used to detect whether the vehicle was at the depot, customer location, maintenance site, fuel station, or an unplanned location.

#### d. Scanner event normalization

Scanner events such as:

- `Scan at Depot`
- `Arrive Customer`
- `Delivery Success`
- `Delivery Failed`
- `Depart Customer`

should be aligned with GPS-derived stop events. If a driver forgets to scan immediately, GPS can help infer the real arrival/departure time.

#### e. Planned versus actual enrichment

Each actual event should be enriched with dispatch data:

- Planned stop number.
- Planned ETA.
- Customer time window.
- Expected travel time.
- Expected service time.
- Package priority.
- Planned route sequence.

This enables conformance checking and performance analysis.

#### f. Maintenance integration

Maintenance events should be joined to the operational timeline:

- Scheduled maintenance before/after route.
- Unscheduled repair during shift.
- Warning events during route.
- Vehicle downtime.
- Repeated breakdown patterns.

### Likely data challenges

Common challenges include:

- Missing scanner events.
- Out-of-order timestamps.
- GPS noise in dense urban areas.
- Long GPS gaps due to connectivity loss.
- Ambiguous stops, e.g., traffic light versus delivery stop.
- Different location formats across systems.
- Drivers using the scanner late or inconsistently.
- Multiple packages delivered at one location.
- A vehicle-day case containing many package-level subprocesses.
- Privacy and labor-relations concerns when analyzing driver behavior.

These issues should be handled with data quality rules, confidence scores, and validation with operational staff.

### 1.2 Process discovery: visualizing the actual delivery process

After preparing the event log, process discovery can reveal how deliveries actually happen, not how they are assumed to happen.

I would begin with a high-level discovered process model for a vehicle-day case:

1. `Start Shift`
2. `Route Assigned`
3. `Load / Scan at Depot`
4. `Depart Depot`
5. Repeating loop:
   - `Travel to Customer`
   - `Traffic Delay` or `Route Deviation`, if applicable
   - `Arrive Customer`
   - `Delivery Success` or `Delivery Failed`
   - `Depart Customer`
6. Optional deviations:
   - `Unplanned Stop`
   - `Refuel`
   - `Vehicle Warning`
   - `Maintenance / Repair`
   - `Return to Depot Mid-route`
7. `Arrive Depot`
8. `End Shift`

Suitable process discovery techniques include:

- **Directly-Follows Graphs** for an initial overview of common paths.
- **Fuzzy Miner** to simplify complex “spaghetti” logistics processes.
- **Inductive Miner** to generate structured process models.
- **Variant analysis** to compare common route-execution patterns.
- **Dotted charts** to visualize events over time by vehicle, driver, or route.
- **Geo-process mining** to overlay process events on maps.

The output should include both a process map and a geographic visualization. For example:

- Process map shows that many late routes include `Low Speed Detected  Unplanned Stop  Delivery Failed`.
- Geo-map shows that low-speed delays cluster around a specific ring road between 08:00 and 09:30.

### 1.3 Conformance checking against planned routes

The dispatch system provides the planned route, planned stop sequence, ETAs, capacity assignments, and customer time windows. This planned model becomes the reference for conformance checking.

For each vehicle-day case, we compare:

| Planned | Actual |
|---|---|
| Planned depot departure time | Actual depot departure time |
| Planned stop sequence | Actual stop sequence |
| Planned ETA per stop | Actual arrival time |
| Planned customer time window | Actual delivery timestamp |
| Planned route distance | Actual GPS distance |
| Planned service duration | Actual dwell time |
| Planned return time | Actual depot arrival |
| No breakdown expected | Actual maintenance or warning events |

### Deviations to look for

#### Sequence deviations

Examples:

- Stop 5 performed before Stop 3.
- Customer skipped.
- Same customer visited twice.
- Redelivery added mid-route.
- Driver chose a different stop order than dispatch plan.

Some sequence deviations may be harmful, but others may be beneficial if drivers are adapting to traffic or access conditions. These should be analyzed rather than automatically penalized.

#### Timing deviations

Examples:

- Late departure from depot.
- Excessive travel time between two stops.
- Service time much longer than planned.
- Early arrival before customer time window.
- Late arrival after promised window.
- Excessive end-of-route overtime.

#### Location and route deviations

Examples:

- GPS path significantly differs from planned path.
- Long detour around congestion.
- Unplanned stop at non-customer location.
- Repeated idle time in certain areas.
- Parking search pattern near dense urban addresses.

#### Operational deviations

Examples:

- Delivery failed despite customer time window.
- Maintenance interruption during route.
- Vehicle returns to depot before completing route.
- Driver makes unscheduled refueling stop.
- Package assigned to route but never scanned at delivery.

Conformance results should be quantified using alignment costs, deviation frequency, delay minutes, extra kilometers, and SLA impact.

## 2. Performance Analysis and Bottleneck Identification

### 2.1 Relevant KPIs

The KPIs should directly reflect Speedy Parcels’ goals: punctuality, cost reduction, fleet reliability, and operational efficiency.

| KPI | Calculation from event log | Why it matters |
|---|---|---|
| On-Time Delivery Rate | Deliveries completed within customer time window / total deliveries | Measures customer promise fulfillment |
| Average Lateness | Actual delivery time minus promised latest time | Quantifies SLA severity |
| First-Attempt Delivery Success Rate | Successful first attempts / total first attempts | Failed deliveries create redelivery cost |
| Failed Delivery Rate | Failed deliveries / total attempted deliveries | Identifies avoidable rework |
| Average Service Time per Stop | `Depart Customer - Arrive Customer` | Measures customer-location efficiency |
| Average Travel Time between Stops | `Arrive Stop N+1 - Depart Stop N` | Measures routing and traffic impact |
| Travel Time vs. Service Time Ratio | Total travel time / total service time | Shows whether inefficiency is road-based or stop-based |
| Total Route Duration | `Arrive Depot - Depart Depot` | Measures operational day efficiency |
| Depot Delay | `Depart Depot - Route Assigned` or `Depart Depot - Start Shift` | Identifies loading/dispatch delays |
| Overtime Rate | Routes ending after planned shift / total routes | Links inefficiency to labor cost |
| Vehicle Utilization Rate | Loaded packages or volume / vehicle capacity | Identifies underused or overloaded vehicles |
| Stop Density | Stops per kilometer or stops per hour | Measures route compactness |
| Kilometers per Delivered Package | Actual GPS km / successful deliveries | Strong cost and fuel proxy |
| Fuel Consumption per km/package | Fuel used or estimated fuel / km or packages | Tracks cost and emissions |
| Idling Time Ratio | Idle time / route duration | Indicates fuel waste |
| Traffic Delay Duration | Actual travel time minus expected free-flow or historical baseline | Identifies congestion cost |
| Route Adherence | Actual sequence/path compliance versus plan | Measures plan execution quality |
| Maintenance Downtime | Repair duration during operating hours | Measures fleet availability loss |
| Breakdowns per 1,000 km | Unscheduled maintenance events / distance | Measures vehicle reliability |

If direct fuel data is unavailable, fuel consumption can be estimated using vehicle type, distance, speed profiles, acceleration, and idling duration.

---

### 2.2 Process mining techniques for bottleneck identification

#### a. Performance-annotated process maps

A process map can be enriched with average and percentile durations.

Example findings:

- `Depart Depot  Stop 1` takes 45 minutes on average, much higher than planned.
- `Arrive Customer  Depart Customer` is highly variable in apartment-heavy zones.
- `Delivery Failed  Reattempt Delivery` causes significant rework.

This helps distinguish travel bottlenecks from service bottlenecks.

#### b. Variant analysis

Compare high-performing and low-performing route variants.

Example:

- High-performing variant: `Depart Depot  Travel  Delivery Success` repeated, no failed deliveries, no unplanned stops.
- Low-performing variant: includes repeated `Low Speed`, `Unplanned Stop`, `Delivery Failed`, and overtime.

Variant analysis helps identify the behavioral or contextual patterns associated with delays and costs.

#### c. Dotted charts and timeline analysis

A dotted chart by vehicle-day can show:

- Late depot departures.
- Midday idle clusters.
- Routes consistently ending after shift.
- Dense traffic delay periods.
- Long gaps between scanner events.

This is useful for spotting temporal bottlenecks.

#### d. Geo-spatial process mining

By mapping events geographically, Speedy Parcels can identify:

- Traffic hotspots.
- Parking difficulty zones.
- Areas with high failed-delivery rates.
- Neighborhoods with long service times.
- Road segments causing repeated delay.

#### e. Resource and driver analysis

Performance can be sliced by:

- Driver.
- Vehicle.
- Route.
- Territory.
- Shift start time.
- Vehicle type.
- Package type.
- Customer type.

The goal is not to blame drivers but to identify training needs, planning issues, equipment problems, or route unfairness.

#### f. Quantifying bottleneck impact

Each bottleneck should be quantified in operational and financial terms:

- Extra minutes per route.
- Extra kilometers per route.
- Additional fuel cost.
- Additional overtime cost.
- Number of late deliveries caused.
- Number of redelivery attempts caused.
- Lost vehicle availability.
- Customer SLA penalty or complaint risk.

For example:

> “Routes passing through Zone A between 08:00 and 09:30 experience an average of 18 additional travel minutes, causing 12% more late deliveries and increasing fuel consumption by 7% due to idling.”

## 3. Root Cause Analysis for Inefficiencies

Identifying delay points is not enough. Process mining should be used to determine why they occur.

### 3.1 Suboptimal route planning

Potential root cause:

- Routes are planned statically and do not account for time-dependent traffic, parking difficulty, or actual service-time patterns.

Process mining validation:

- Compare planned stop sequence versus actual sequence.
- Identify where drivers frequently deviate from the plan.
- Check whether deviations improve or worsen punctuality.
- Compare planned distance with actual GPS distance.
- Analyze whether certain planned routes consistently produce overtime.

Example insight:

> Drivers on Route Cluster 4 deviate from the planned sequence on 65% of days. The deviations reduce travel time by 12%, suggesting the dispatch plan is not optimized for real traffic conditions.

---

### 3.2 Inaccurate travel time estimations

Potential root cause:

- Dispatch uses average travel times that do not reflect rush-hour congestion, school zones, construction, or urban density.

Process mining validation:

- Compare planned travel time between stop pairs with actual travel time.
- Segment by time of day, day of week, and road area.
- Build origin-destination travel-time distributions.
- Detect systematic underestimation.

Example insight:

> Travel from the depot to the first urban delivery zone is planned at 22 minutes but takes 38 minutes at the 80th percentile on weekdays.

---

### 3.3 Traffic congestion patterns

Potential root cause:

- Vehicles enter known congestion hotspots at predictable times.

Process mining validation:

- Use low-speed GPS events and travel-time deviations.
- Overlay them on maps.
- Correlate delays with time of day and route sequence.
- Compare routes that pass through the same area at different times.

Example insight:

> Vehicles entering the central district after 08:15 lose an average of 21 minutes. Routes reaching the same area before 07:55 have much better on-time performance.

---

### 3.4 High variability in customer service time

Potential root cause:

- Some stops require more time due to apartments, business receptions, security access, signatures, cash-on-delivery, bulky parcels, or parking difficulty.

Process mining validation:

- Analyze dwell time: `Depart Customer - Arrive Customer`.
- Compare service time by customer type, address type, package size, and delivery outcome.
- Detect outliers and recurring long-service customers.
- Separate parking/walking time from actual customer interaction if GPS and scanner timing allow.

Example insight:

> Apartment buildings in Zone B have median service time of 7 minutes versus 3 minutes for suburban houses, but the dispatch model assumes 4 minutes for all stops.

---

### 3.5 Vehicle breakdowns and maintenance needs

Potential root cause:

- Older or heavily used vehicles are more likely to experience warning events, unscheduled stops, and mid-route breakdowns.

Process mining validation:

- Link maintenance logs to route execution.
- Analyze breakdown frequency per vehicle, vehicle age, mileage, and recent maintenance history.
- Measure delay caused by vehicle-warning events.
- Identify recurring vehicles or components associated with disruptions.

Example insight:

> Vehicles with more than 180,000 km account for 42% of unscheduled maintenance stops but only 18% of total distance driven.

---

### 3.6 Driver behavior or skill differences

Potential root cause:

- Differences in route familiarity, scanning discipline, parking strategy, idling habits, or driving style.

Process mining validation:

- Compare driver performance while controlling for territory, route difficulty, vehicle type, and package count.
- Analyze service time, idle time, route adherence, failed scan frequency, and overtime.
- Identify best-practice variants from high-performing drivers.

Example insight:

> Driver D105 has normal travel time but above-average customer dwell time, while Driver D122 has similar dwell time but much higher idling and fuel use.

---

### 3.7 Failed delivery attempts

Potential root cause:

- Poor customer availability prediction, inaccurate time windows, insufficient communication, or delivery attempts at unsuitable times.

Process mining validation:

- Analyze failed deliveries by customer, area, day of week, time of day, package type, and time-window type.
- Identify repeat failed-delivery customers.
- Measure redelivery cost and delay impact.
- Compare routes with high failure rates versus low failure rates.

Example insight:

> Residential deliveries attempted between 10:00 and 14:00 have a 19% failure rate, while the same neighborhoods after 17:00 have a 7% failure rate.

## 4. Data-Driven Optimization Strategies

### Strategy 1: Dynamic route adjustment using real-time and historical traffic data

#### Inefficiency targeted

- Late deliveries caused by congestion, static routing, and inaccurate travel-time assumptions.

#### Root cause addressed

- Dispatch plans are not sufficiently time-dependent or adaptive to actual traffic conditions.

#### Process mining support

Process mining can reveal:

- Repeated traffic bottlenecks.
- Planned versus actual travel-time gaps.
- Stop sequences that drivers frequently override.
- Time periods where certain zones should be avoided.

#### Proposed action

Implement dynamic routing that:

- Uses real-time traffic data.
- Uses historical travel-time distributions discovered from process logs.
- Re-sequences remaining stops during the route when delays occur.
- Alerts dispatchers when a time-window violation is likely.
- Suggests alternative routes or stop orders.

#### Expected KPI impact

- Higher On-Time Delivery Rate.
- Lower average lateness.
- Reduced travel time.
- Reduced idling and fuel consumption.
- Lower overtime rate.

---

### Strategy 2: Redesign delivery territories and route sequences based on historical performance

#### Inefficiency targeted

- Long routes, excessive kilometers per package, low stop density, and recurring overtime.

#### Root cause addressed

- Existing territories or route assignments may not reflect actual delivery density, traffic zones, service-time variability, or capacity constraints.

#### Process mining support

Process mining can identify:

- Routes with consistently poor performance.
- Geographic clusters with high service time.
- Zones that should not be combined in the same route.
- Routes where planned workload exceeds realistic shift capacity.
- Actual versus planned route duration and distance.

#### Proposed action

Use historical process data to redesign:

- Delivery zones.
- Stop clustering.
- Route start times.
- Vehicle assignment by area.
- Sequence rules for urban versus suburban stops.
- Buffer times for high-variability zones.

For example, dense urban routes may need fewer stops but more service-time allowance, while suburban routes may support more stops but longer drive distances.

#### Expected KPI impact

- Lower kilometers per package.
- Higher vehicle utilization without overloading.
- Reduced overtime.
- Improved route adherence.
- More predictable route duration.
- Better on-time performance.

---

### Strategy 3: Improve time-window management and customer communication to reduce failed deliveries

#### Inefficiency targeted

- Failed delivery attempts, redelivery cost, wasted travel time, and customer dissatisfaction.

#### Root cause addressed

- Customers may not be available at planned delivery times, and the system may not use historical availability patterns.

#### Process mining support

Process mining can show:

- Failure rates by time of day, customer type, area, and package type.
- Repeat failed-delivery customers.
- Routes where failed deliveries cause downstream lateness.
- Cost impact of redelivery loops.

#### Proposed action

Implement customer-focused interventions:

- Predict high-risk failed deliveries.
- Send proactive ETA notifications.
- Offer narrower live delivery windows.
- Allow customers to reschedule before the van arrives.
- Encourage parcel-locker or pickup-point redirection for high-risk addresses.
- Prioritize business deliveries during business hours and residential deliveries outside typical working hours.

#### Expected KPI impact

- Higher First-Attempt Delivery Success Rate.
- Lower failed delivery rate.
- Fewer redelivery kilometers.
- Lower cost per successful delivery.
- Better customer satisfaction.
- Reduced route disruption.

---

### Strategy 4: Predictive maintenance and smarter vehicle assignment

#### Inefficiency targeted

- Unscheduled stops, breakdowns during routes, repair downtime, and maintenance-driven late deliveries.

#### Root cause addressed

- Maintenance may be reactive rather than based on actual vehicle usage, route stress, warning events, and breakdown patterns.

#### Process mining support

Process mining can identify:

- Vehicles with recurring warning events.
- Routes or driving conditions associated with high mechanical stress.
- Maintenance events that occur soon after certain GPS or vehicle-status patterns.
- Delay impact of vehicle breakdowns.
- Vehicle reliability differences.

#### Proposed action

Develop predictive maintenance rules using:

- Mileage.
- Engine warning events.
- Idle time.
- Stop-start frequency.
- Route terrain or congestion intensity.
- Past repair history.
- Breakdown frequency.

Assign more reliable vehicles to critical routes with strict time windows, and schedule maintenance during low-demand periods.

#### Expected KPI impact

- Reduced unscheduled maintenance events.
- Lower route disruption.
- Higher vehicle availability.
- Reduced repair cost.
- Improved On-Time Delivery Rate.
- Lower overtime caused by vehicle issues.

---

### Strategy 5: Targeted driver coaching and operational best-practice sharing

#### Inefficiency targeted

- Excessive idling, long dwell times, route deviations, inconsistent scanning, and high fuel consumption.

#### Root cause addressed

- Drivers may differ in route familiarity, parking strategy, customer interaction handling, scanning discipline, and eco-driving behavior.

#### Process mining support

Process mining can compare drivers while controlling for route difficulty:

- Service time per stop.
- Idling ratio.
- Fuel consumption per km.
- Route adherence.
- Failed scan frequency.
- Overtime frequency.
- Delivery success rate.

The goal should be improvement and standardization, not punitive monitoring.

#### Proposed action

Create targeted coaching programs:

- Eco-driving training for high fuel/idling drivers.
- Parking and stop-handling best practices in dense areas.
- Scanner process compliance training.
- Route familiarization for difficult territories.
- Sharing process variants from high-performing drivers.

#### Expected KPI impact

- Lower fuel consumption.
- Reduced idling.
- Shorter service times.
- Better data quality.
- Improved route predictability.
- Lower maintenance wear from harsh driving or excessive idling.

## 5. Operational Constraints and Continuous Monitoring

### 5.1 Accounting for operational constraints

Optimization must respect real-world constraints. Recommendations should be embedded into a vehicle-routing and dispatch-planning framework that considers:

#### Driver working hours

- Maximum shift duration.
- Legal break requirements.
- Overtime limits.
- Start and end depot times.
- Driver availability and skills.

#### Vehicle capacity

- Package count.
- Volume.
- Weight.
- Special handling requirements.
- Vehicle type restrictions.
- Electric vehicle range, if applicable.

#### Customer time windows

- Promised delivery windows.
- Business opening hours.
- Customer preferences.
- Priority customers.
- Same-day or express commitments.

#### Maintenance constraints

- Vehicles unavailable during scheduled maintenance.
- Vehicles with warning indicators assigned to lower-risk routes or removed from service.
- Spare vehicle availability.

#### Route feasibility

- Urban access restrictions.
- Parking constraints.
- Low-emission zones.
- Road closures.
- Traffic patterns.
- Depot loading capacity.

Dynamic route changes should also be constrained. For example, the system should not re-sequence a route in a way that violates customer time windows or causes the driver to exceed legal working hours.

### 5.2 Continuous monitoring with process mining dashboards

After implementing improvements, Speedy Parcels should establish continuous process monitoring. The dashboard should compare current performance to the historical baseline and detect emerging issues early.

### Recommended dashboard views

#### Executive KPI dashboard

Track:

- On-Time Delivery Rate.
- Average lateness.
- Failed delivery rate.
- Cost per delivered package.
- Fuel consumption per km/package.
- Overtime rate.
- Vehicle utilization.
- Maintenance downtime.
- Customer complaints.

#### Process map dashboard

Show the current discovered process model with:

- Most frequent variants.
- New or increasing deviations.
- Average duration per activity.
- Bottlenecks highlighted.
- Rework loops such as failed delivery and redelivery.

#### Conformance dashboard

Track:

- Planned versus actual route sequence.
- Planned versus actual arrival times.
- Route adherence.
- Time-window violations.
- Unplanned stops.
- Missed scans.
- Depot departure delays.

#### Geo-operational dashboard

Show:

- Traffic-delay hotspots.
- High failed-delivery zones.
- Long service-time locations.
- Excessive idling zones.
- Frequent unplanned-stop areas.
- Parking difficulty clusters.

#### Fleet reliability dashboard

Track:

- Warning events.
- Breakdown frequency.
- Maintenance backlog.
- Downtime by vehicle.
- Repair repeat rate.
- Vehicle reliability score.

#### Driver and route fairness dashboard

Track driver performance while normalizing for:

- Route difficulty.
- Package count.
- Territory.
- Vehicle type.
- Weather or traffic conditions.

This prevents unfair comparisons and helps identify genuine training opportunities.

---

### 5.3 Monitoring cycle

A practical monitoring cycle would be:

1. **Daily operational monitoring**
   - Detect late routes, breakdowns, missed scans, and severe deviations.

2. **Weekly process review**
   - Review bottlenecks, route variants, driver/vehicle issues, failed delivery patterns.

3. **Monthly optimization review**
   - Update travel-time estimates, service-time assumptions, territory design, and routing rules.

4. **Quarterly strategic review**
   - Assess fleet replacement needs, depot capacity, customer time-window policy, and staffing levels.

5. **A/B testing of interventions**
   - Test dynamic routing, customer notifications, or new territories on selected routes.
   - Compare against control routes using process mining KPIs.

---

## Summary

Using process mining, Speedy Parcels can move from anecdotal explanations of lateness and cost overruns to evidence-based operational improvement.

The core approach is:

1. Integrate GPS, scanner, dispatch, and maintenance data into a unified event log.
2. Discover the actual delivery process and its common variants.
3. Compare actual execution against planned routes and time windows.
4. Identify bottlenecks by route, driver, vehicle, geography, and time of day.
5. Validate root causes such as traffic, poor planning assumptions, service-time variability, vehicle reliability, and failed deliveries.
6. Implement targeted optimizations such as dynamic routing, territory redesign, customer communication, predictive maintenance, and driver coaching.
7. Monitor the operation continuously with process mining dashboards.

The expected benefits include higher delivery punctuality, fewer failed attempts, lower fuel consumption, reduced overtime, better fleet reliability, and a more predictable last-mile delivery operation.
