## Explanation of Transformation Logic

### Case Identification

I identified **five cases**, each representing a coherent unit of user work centered on a specific **document or task object** (rather than the application alone):

1. **C1_Document1** – Drafting and editing `Document1.docx`. Notably, this case is *interrupted* (user switches to email, PDF, and Excel) and *resumed* at 09:06:00, demonstrating an interleaved process — a common and analytically valuable pattern in knowledge-worker process mining.
2. **C2_QuarterlyReport** – Work on `Quarterly_Report.docx`. The initial brief FOCUS at 08:59:50 and the later editing session belong to the same case, since the case object (the document) is identical.
3. **C3_EmailAnnualMeeting** – Handling a single email thread ("Annual Meeting"): open  read  reply  send. This forms a complete, self-contained mini-process.
4. **C4_PDFReview** – Reviewing `Report_Draft.pdf` (open, read, highlight). The case has no explicit close event — it ends implicitly when focus moves away.
5. **C5_BudgetUpdate** – Updating `Budget_2024.xlsx` figures and saving.

The **case notion = document/work-item**, because grouping by application or by time window would merge unrelated work (e.g., two distinct Word documents) and break the analyst-friendly narrative.

### Activity Naming

Raw low-level actions were lifted to standardized, process-level activities:

| Raw Action | Standardized Activity |
|-----------|----------------------|
| FOCUS / SWITCH-to (first time) | Open Document / Open Spreadsheet / Open Email Client |
| FOCUS / SWITCH-to (returning) | Resume Document Editing |
| TYPING | Edit Document Content / Edit Spreadsheet Data / Compose Reply |
| SAVE | Save Document / Save Spreadsheet |
| SCROLL | Read Document / Read Email |
| HIGHLIGHT | Annotate Document |
| CLICK (semantic) | Open Email / Start Reply / Send Email |
| CLOSE | Close Document |

Context (application + the semantic `Action=` payload of CLICK events) determined which standardized name applied. SWITCH events were attributed to the **target** case, since they mark the start/resumption of work on that case.

### Analytical Value

This log supports standard process mining analyses: discovering typical document lifecycles (Open  Edit  Save  Close), measuring interruption/resumption patterns (C1 and C2 both exhibit suspend-resume behavior), and quantifying time spent per case via timestamp differences.
