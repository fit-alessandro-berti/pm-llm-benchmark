# DuckDB Query: Top-K Process Variant Filtering

## Complete SQL Query

```sql
-- Replace :K with your desired integer (e.g., 5 for the top 5 variants)

WITH case_variants AS (
    -- Step 1: Build the ordered activity sequence (variant) for each case
    SELECT
        case_id,
        list(activity ORDER BY timestamp) AS variant
    FROM event_log
    GROUP BY case_id
),

variant_counts AS (
    -- Step 2 & 3: Count cases per variant and keep only the top K
    SELECT
        variant,
        count(*) AS case_count
    FROM case_variants
    GROUP BY variant
    ORDER BY case_count DESC
    LIMIT :K
),

top_k_cases AS (
    -- Step 4a: Identify which case_ids belong to the top K variants
    SELECT cv.case_id
    FROM case_variants cv
    INNER JOIN variant_counts vc
        ON cv.variant = vc.variant
)

-- Step 4b: Return all original events for those filtered cases
SELECT e.*
FROM event_log e
INNER JOIN top_k_cases tkc
    ON e.case_id = tkc.case_id
ORDER BY e.case_id, e.timestamp;
```

---

## How It Works — Step by Step

### CTE 1: `case_variants`
| Purpose | Mechanism |
|---------|-----------|
| Constructs the process variant per case | `list(activity ORDER BY timestamp)` aggregates all activities within a `case_id` into an **ordered list** (DuckDB's native `LIST` type), sorted chronologically. |

**Example output:**
| case_id | variant |
|---------|---------|
| C1 | `[Register, Review, Approve]` |
| C2 | `[Register, Review, Reject]` |
| C3 | `[Register, Review, Approve]` |

### CTE 2: `variant_counts`
| Purpose | Mechanism |
|---------|-----------|
| Groups identical variants and ranks them | Groups by the `LIST` value (DuckDB supports equality/hashing on `LIST` types), counts occurrences, and applies `LIMIT :K` to retain only the **K most frequent** variants. |

**Example output (K=1):**
| variant | case_count |
|---------|------------|
| `[Register, Review, Approve]` | 2 |

### CTE 3: `top_k_cases`
| Purpose | Mechanism |
|---------|-----------|
| Maps top variants back to their case IDs | Joins `case_variants` with `variant_counts` on the `variant` list to extract only the `case_id`s that follow a top-K variant. |

### Final SELECT
Joins the filtered `case_id`s back to the **original `event_log`** to return every raw event (all columns) belonging to those cases, ordered for readability.

## Alternative: String-Based Variant Representation

If you prefer a human-readable string over a `LIST` (useful for exporting or debugging), swap `list()` for `string_agg()`:

```sql
string_agg(activity, '  ' ORDER BY timestamp) AS variant
```

This produces variants like `"Register  Review  Approve"`. The rest of the query remains identical.

## Key Design Decisions & Edge Cases

| Consideration | Handling |
|---|---|
| **Tied timestamps** | If two events in the same case share an identical `timestamp`, their relative order is non-deterministic. Add a tiebreaker column (e.g., `event_id`) to the `ORDER BY`: `list(activity ORDER BY timestamp, event_id)` |
| **Ties at the K-th boundary** | `LIMIT :K` arbitrarily breaks ties. If you want **all** variants tied at the K-th rank, replace `LIMIT :K` with a `DENSE_RANK()` window function and filter on `rank <= :K`. |
| **LIST equality in DuckDB** | DuckDB natively supports `=` comparison and hashing on `LIST` types, so joining and grouping on the `variant` column works correctly without serialization. |
| **Performance on large logs** | The most expensive step is the initial aggregation. Ensure an index or sort on `(case_id, timestamp)` if the table is very large. |
