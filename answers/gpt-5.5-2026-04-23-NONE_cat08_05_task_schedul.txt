## 1. Analyzing Historical Scheduling Performance and Dynamics

Precision Parts Inc. should treat the MES event log as the factual record of how the shop actually operates, not how it is assumed to operate. The first objective is to reconstruct the real production flow, resource behavior, queues, setups, disruptions, and schedule outcomes from the event data.

The analysis should be performed at two complementary levels:

1. **Job-level process mining**: reconstructing the end-to-end lifecycle of each customer order.
2. **Resource-level process mining**: reconstructing the chronological execution sequence on each machine and operator.

Both views are necessary because a job shop is not a single linear process. Jobs compete for shared resources, and the schedule emerges from interactions between routings, queues, setup dependencies, resource calendars, disruptions, and priority changes.

---

### 1.1 Event Log Preparation and Process Reconstruction

The MES log should be transformed into an enriched process mining event log with at least the following core fields:

| Field | Purpose |
|---|---|
| Case ID / Job ID | Tracks each job end-to-end |
| Activity | Job released, queue entry, setup start, setup end, task start, task end, inspection, rework, completion |
| Timestamp | Enables sequencing and duration calculations |
| Resource / Machine | Enables machine-level analysis |
| Operator | Enables labor-capacity and skill analysis |
| Work center | Groups equivalent machines |
| Operation sequence number | Needed for routing reconstruction |
| Order priority | Supports priority/tardiness analysis |
| Due date | Required for lateness and tardiness measures |
| Planned processing time | Baseline for schedule adherence |
| Actual processing time | Performance measurement |
| Setup flag and setup duration | Setup analysis |
| Previous job / previous part family | Sequence-dependent setup modeling |
| Disruption events | Breakdown, repair, priority change, hot job insertion |

The first process mining task is to build a **digital process model** of actual job movement through the shop.

Techniques include:

- **Process discovery**
  - Inductive Miner, Heuristics Miner, or Directly-Follows Graphs to identify the dominant routing patterns.
  - Useful for high-mix environments where there are many variants and no single fixed process.
- **Variant analysis**
  - Group jobs by actual routing, product family, customer, material type, priority, or lateness outcome.
- **Conformance checking**
  - Compare actual routings and timing against planned routings, standard operation sequences, dispatching rules, or due-date commitments.
- **Performance mining**
  - Overlay waiting times, processing times, setup times, and delays on the discovered process model.
- **Resource mining**
  - Build machine-centric and operator-centric views of actual utilization, handoffs, queues, and contention.

---

### 1.2 Reconstructing the Actual Flow of Jobs

For each job, the reconstructed lifecycle would include:

1. Job release.
2. Queue entry at first work center.
3. Setup start and end, if required.
4. Task start and end.
5. Queue entry at next operation.
6. Repeat until completion.
7. Shipment or final job close.

From this, the actual routing sequence can be reconstructed, for example:

> JOB-7001: Release  Cutting Queue  Cutting Setup  Cutting  Milling Queue  Milling  Grinding Queue  Grinding  Inspection  Complete

This allows Precision Parts to answer questions such as:

- Which routings are most common?
- Which routing variants have the longest lead times?
- Which operations create the most waiting?
- Which machines are shared by the most competing job classes?
- Which deviations from planned routing cause lateness?

---

### 1.3 Reconstructing Execution Sequence on Each Machine

In addition to job-centric traces, the event log should be re-indexed by machine. For each machine, create a chronological sequence of events:

> CUT-01: JOB-6998 processing  setup for JOB-7001  JOB-7001 processing  idle  JOB-7006 setup  JOB-7006 processing  breakdown  repair  restart

This resource-centric trace enables direct analysis of:

- The actual sequence of jobs processed on each machine.
- Idle intervals.
- Setup intervals.
- Breakdown intervals.
- Periods where the machine was blocked, starved, or overloaded.
- Priority interruptions and schedule disruptions.

This view is essential for sequence-dependent setup analysis because setup time depends on the previous job processed on the same resource.

---

## 1.4 Metrics and Process Mining Techniques

### A. Job Flow Times, Lead Times, and Makespan Distributions

Important metrics:

| Metric | Definition |
|---|---|
| Job lead time | Completion time minus release time |
| Manufacturing flow time | Time from first queue entry to final operation completion |
| Processing time | Sum of actual task durations |
| Waiting time | Sum of all queue times |
| Setup time | Sum of all setup durations |
| Rework time | Time spent on repeated or corrective operations |
| Makespan | Time to complete a group of jobs, batch, or schedule horizon |
| Value-added ratio | Processing time divided by total lead time |
| Flow efficiency | Productive time divided by total time in system |

For each job:

\[
LeadTime_j = CompletionTime_j - ReleaseTime_j
\]

\[
TotalWaiting_j = \sum_{operations} (TaskStart - QueueEntry)
\]

\[
TotalProcessing_j = \sum_{operations} (TaskEnd - TaskStart)
\]

\[
TotalSetup_j = \sum_{operations} (SetupEnd - SetupStart)
\]

Process mining techniques:

- **Performance-annotated process maps** showing average and percentile time spent between activities.
- **Variant performance analysis** comparing common job routings.
- **Cohort analysis** by priority, product family, customer, complexity, due-date tightness, and routing.
- **Distributional analysis** using median, P75, P90, P95, and tail behavior, not only averages.

For a job shop, average lead time is often misleading. Tardiness and customer dissatisfaction are usually driven by long-tail delays.

---

### B. Task Waiting Times at Each Work Center or Machine

Queue time is calculated as:

\[
QueueTime_{j,o} = StartTime_{j,o} - QueueEntryTime_{j,o}
\]

If setup is modeled separately:

\[
QueueTime_{j,o} = SetupStartTime_{j,o} - QueueEntryTime_{j,o}
\]

or, if no setup:

\[
QueueTime_{j,o} = TaskStartTime_{j,o} - QueueEntryTime_{j,o}
\]

Metrics:

- Average queue time per work center.
- Median and percentile queue times.
- Queue time by shift, day of week, product family, priority, and machine.
- Queue length over time.
- WIP accumulation before each machine.
- Waiting-time contribution to total lead time.
- Ratio of waiting time to processing time.

Process mining techniques:

- **Bottleneck analysis** using performance overlays.
- **Queue mining** from queue entry and task start events.
- **Heat maps** showing queue time by resource and time period.
- **Temporal performance slicing** to identify recurring congestion windows.
- **Social/resource network analysis** to see operator-resource dependencies.

---

### C. Resource Utilization: Productive Time, Idle Time, Setup Time

For each machine:

\[
AvailableTime = CalendarTime - PlannedDowntime
\]

\[
ProductiveUtilization = \frac{ProcessingTime}{AvailableTime}
\]

\[
SetupUtilization = \frac{SetupTime}{AvailableTime}
\]

\[
BreakdownFraction = \frac{UnplannedDowntime}{AvailableTime}
\]

\[
IdleFraction = \frac{IdleTime}{AvailableTime}
\]

\[
TotalBusyUtilization = \frac{ProcessingTime + SetupTime}{AvailableTime}
\]

The machine time states should be classified as:

1. Productive processing.
2. Setup.
3. Idle but available.
4. Down due to breakdown.
5. Planned maintenance.
6. Blocked due to downstream congestion, if detectable.
7. Starved due to no upstream jobs, if detectable.
8. Waiting for operator, tooling, material, or inspection.

Process mining/resource mining can generate machine Gantt charts from event logs and calculate:

- Productive time.
- Setup time.
- Idle time.
- Downtime.
- Changeover burden.
- Utilization variability by shift.
- Operator-dependent performance differences.
- Shared operator constraints causing machine idle time.

Important: very high utilization at a bottleneck machine may be desirable, but very high utilization everywhere usually creates long queues and unstable lead times. The target is not simply “maximize utilization”; it is to increase throughput and due-date performance while controlling WIP.

---

### D. Sequence-Dependent Setup Times

Sequence-dependent setup analysis requires reconstructing, for each machine, the exact order of jobs processed.

For machine \(m\), suppose the sequence is:

\[
..., Job_i, Job_j, Job_k, ...
\]

Then the setup before \(Job_j\) depends on properties of \(Job_i\) and \(Job_j\):

\[
SetupTime_{i,j,m} = SetupEnd_{j,m} - SetupStart_{j,m}
\]

The analysis should enrich each job with attributes such as:

- Material type.
- Part family.
- Geometry class.
- Tooling family.
- Fixture type.
- Customer.
- Tolerance class.
- Heat treatment requirement.
- Surface finish requirement.
- Program family.
- Batch/similarity group.
- Previous operation state.

Then model setup time as a function of transition:

\[
SetupTime = f(machine, previous\_job\_attributes, current\_job\_attributes, operator, shift, tooling, fixture, material)
\]

Useful analyses:

1. **Setup matrix estimation**
   - Estimate average, median, and P90 setup time for transitions between job families.
   - Example:

| From / To | Aluminum family A | Steel family B | Titanium family C |
|---|---:|---:|---:|
| Aluminum family A | 10 min | 35 min | 50 min |
| Steel family B | 30 min | 15 min | 60 min |
| Titanium family C | 45 min | 55 min | 20 min |

2. **Transition mining**
   - Identify frequently occurring job-to-job or family-to-family transitions.
   - Identify high-penalty transitions that occur often.

3. **Regression or machine learning setup models**
   - Predict setup duration based on transition attributes.
   - Models may include random forest, gradient boosting, Bayesian hierarchical models, or generalized additive models.

4. **Operator effect analysis**
   - Determine whether setup time depends heavily on operator skill or shift.

5. **Setup conformance**
   - Compare actual setup time to standard setup time.
   - Identify jobs or transitions with systematically underestimated setups.

Key metrics:

- Total setup time per machine per week.
- Setup time as percentage of available capacity.
- Setup frequency.
- Average setup by transition type.
- Setup variability.
- Avoidable setup time under alternative sequencing.
- Setup-induced bottleneck load.

---

### E. Schedule Adherence and Tardiness

For each job:

\[
Lateness_j = CompletionTime_j - DueDate_j
\]

Lateness can be negative if early.

\[
Tardiness_j = max(0, CompletionTime_j - DueDate_j)
\]

Additional measures:

| Metric | Meaning |
|---|---|
| On-time delivery rate | Percentage of jobs completed by due date |
| Mean tardiness | Average delay among all jobs |
| Conditional mean tardiness | Average delay among late jobs only |
| Maximum tardiness | Worst-case delay |
| P90/P95 tardiness | Tail-risk measure |
| Weighted tardiness | Tardiness weighted by priority/customer importance |
| Earliness | Jobs completed too early, potentially increasing finished goods inventory |
| Due-date reliability | Difference between promised and actual completion |
| Schedule adherence | Difference between planned task start/end and actual task start/end |

Weighted tardiness:

\[
WeightedTardiness = \sum_j w_j \cdot max(0, C_j - D_j)
\]

where \(w_j\) reflects order priority, customer criticality, or penalty cost.

Process mining analyses:

- Compare on-time versus late job variants.
- Identify which activities and waits most strongly predict tardiness.
- Create predictive process monitoring models estimating lateness risk while a job is still in process.
- Analyze whether high-priority jobs truly receive faster service.
- Measure time from priority change to actual scheduling response.

---

### F. Impact of Disruptions

Disruptions include:

- Machine breakdowns.
- Repairs and maintenance.
- Hot job insertion.
- Priority changes.
- Operator absence.
- Material shortages.
- Quality failures and rework.
- Tooling unavailability.

Each disruption should be represented as an event with time, affected resource, duration, and affected jobs.

Metrics:

| Metric | Definition |
|---|---|
| MTBF | Mean time between failures |
| MTTR | Mean time to repair |
| Breakdown impact window | Time period over which queues and delays remain elevated |
| Lost capacity | Breakdown duration multiplied by bottleneck criticality |
| Jobs affected | Jobs queued or scheduled on the affected machine during disruption |
| Delay propagation | Downstream lateness caused by upstream disruption |
| Recovery time | Time until WIP and queue times return to baseline |
| Hot job displacement cost | Additional delay imposed on other jobs by urgent insertion |

Process mining techniques:

- **Temporal slicing before and after disruptions**
  - Compare queue lengths, lead times, tardiness, and utilization before/during/after breakdowns.
- **Causal impact analysis**
  - Estimate the effect of disruptions on KPIs compared to normal periods.
- **Event correlation**
  - Link breakdowns to downstream queue starvation or upstream WIP buildup.
- **Predictive monitoring**
  - Forecast which jobs will become late after a disruption.
- **Conformance analysis after hot job insertion**
  - Identify which planned jobs were displaced and by how much.

---

# 2. Diagnosing Scheduling Pathologies

After reconstructing actual behavior, the next step is to identify systemic scheduling pathologies. The goal is not merely to say “the shop is busy,” but to determine where the scheduling system causes avoidable lateness, WIP, and instability.

---

## 2.1 Bottleneck Resources

A bottleneck is not simply the machine with the highest utilization. It is the resource that most constrains system throughput and due-date performance.

Evidence from process mining:

- Persistent high utilization.
- Long and growing queues.
- High queue time contribution to total job lead time.
- Strong correlation between waiting time at the resource and job tardiness.
- Downstream starvation after bottleneck disruptions.
- Large backlog recovery time after breakdowns.
- High setup burden consuming effective capacity.

Metrics:

\[
BottleneckLoad_m = ProcessingTime_m + SetupTime_m + ReworkTime_m
\]

\[
LoadRatio_m = \frac{RequiredCapacity_m}{AvailableCapacity_m}
\]

\[
QueueContribution_m = \frac{\sum QueueTime_m}{\sum TotalLeadTime}
\]

If MILL-03, for example, shows 92% busy utilization, 18% setup load, average queue time of 14 hours, and 70% of late jobs waited there, it is a strong candidate bottleneck.

Process mining tools:

- Bottleneck analysis on process maps.
- Resource utilization heat maps.
- Queue length timelines.
- Job lateness correlation analysis.
- Variant comparison for jobs using versus not using the bottleneck.

---

## 2.2 Poor Task Prioritization

The current FCFS/EDD mix applied locally may produce global inefficiencies. A job that appears urgent at one machine may be less critical than another job whose downstream bottleneck queue is about to close or whose remaining routing is much longer.

Evidence:

- High-priority jobs spending excessive time in queues.
- Jobs with near due dates not being expedited until too late.
- Low-priority jobs consuming bottleneck capacity while urgent jobs wait.
- Priority changes not translating quickly into machine sequencing.
- Local EDD decisions causing setup explosions or downstream congestion.

Process mining analyses:

- Compare queue time by priority class.
- Measure rank position of high-priority jobs in queues.
- Determine whether urgent jobs are processed faster after priority change.
- Analyze dispatch decisions: when multiple jobs were waiting, which one was selected and why?
- Compare selected job against alternatives available at that time.

A useful metric is **dispatch regret**:

\[
DispatchRegret = KPI(ActualChoice) - KPI(BestAvailableChoice)
\]

where the “best available choice” is determined retrospectively using due-date, setup, and downstream impact.

---

## 2.3 Suboptimal Sequencing Increasing Setup Time

In a high-mix job shop, poor sequencing can consume enormous capacity through avoidable setups.

Evidence:

- Frequent alternation between dissimilar part families.
- High setup time percentage on bottleneck machines.
- Long setup transitions occurring repeatedly.
- Jobs split into sequences that maximize urgency but ignore setup similarity.
- Setup estimates systematically below actuals.

Process mining analyses:

- Mine transition matrix by machine and job family.
- Identify repeated high-cost transitions.
- Compare actual sequence setup time against heuristic alternatives.
- Quantify lost capacity due to avoidable setups.
- Detect whether urgent jobs frequently trigger expensive changeovers and delay many others.

Example:

If a bottleneck CNC machine spends 25% of available time in setup, and historical sequence analysis shows that grouping by fixture family could reduce setup time by 30%, then the current dispatching rules are directly reducing capacity.

---

## 2.4 Starvation of Downstream Resources

Downstream machines may appear underutilized not because they have excess capacity, but because upstream sequencing fails to release work in a stable pattern.

Evidence:

- Downstream idle periods despite large total WIP elsewhere.
- Queue starvation after upstream bottleneck congestion.
- Long gaps between arrivals at downstream resources.
- Bursty arrivals causing alternating starvation and overload.
- Operators waiting for jobs while upstream machines process less urgent work.

Process mining analyses:

- Arrival process mining at each work center.
- Inter-arrival time variability.
- WIP heat maps by location.
- Blocking/starvation state classification.
- Cross-correlation between upstream completion events and downstream idle periods.

---

## 2.5 WIP Bullwhip and Flow Instability

The bullwhip effect in a job shop appears as unstable WIP accumulation and release waves caused by local decisions and disruptions.

Evidence:

- WIP surges before bottlenecks.
- Batch releases creating artificial congestion.
- Hot jobs displacing normal jobs and causing queue resequencing.
- Recovery after breakdown causing downstream overload.
- Highly variable queue lengths despite stable demand.

Metrics:

- WIP by work center over time.
- Coefficient of variation of queue length.
- Flow time variability.
- Arrival variability versus processing variability.
- Number of jobs waiting more than threshold time.

Process mining techniques:

- Time-series analysis of WIP.
- Token-based replay to estimate WIP accumulation in the process model.
- Animated process maps showing congestion movement.
- Comparison of stable versus unstable operating periods.

---

## 2.6 Variant Analysis: On-Time Versus Late Jobs

A powerful diagnostic is to compare process variants of on-time jobs against late jobs.

Questions:

- Do late jobs use particular routings?
- Do they visit specific machines?
- Do they experience more rework?
- Do they wait longer at specific bottlenecks?
- Are they more often interrupted by hot jobs?
- Are their planned durations less accurate?
- Are they released too late relative to due date?

Process mining approach:

1. Split cases into on-time and late.
2. Discover separate process maps.
3. Overlay performance metrics.
4. Compare:
   - Average queue time by operation.
   - Number of operations.
   - Rework frequency.
   - Setup burden.
   - Priority changes.
   - Breakdown exposure.
   - Resource assignment.

This provides evidence-based root-cause prioritization.

---

# 3. Root Cause Analysis of Scheduling Ineffectiveness

Once pathologies are identified, the next question is why they occur. In Precision Parts’ case, several root causes are likely interacting.

---

## 3.1 Limitations of Static Dispatching Rules

FCFS and EDD are simple and transparent, but they are not adequate for complex high-mix, low-volume manufacturing.

Problems:

- FCFS ignores due dates, priorities, setup times, and downstream bottlenecks.
- EDD ignores processing time, setup burden, and resource contention.
- Local rules optimize local queues, not global due-date performance.
- Static rules cannot adapt to disruptions or changing bottlenecks.
- They do not account for remaining routing complexity.
- They can increase setup frequency by switching between dissimilar jobs.
- They fail when due dates are tight but routings differ significantly.

Example:

A job with the earliest due date but only one short remaining operation may not be as urgent as a job due slightly later with four remaining operations including a bottleneck machine.

---

## 3.2 Lack of Real-Time Visibility

Without real-time visibility, dispatchers cannot make globally informed decisions.

Missing or underused information may include:

- Current queue lengths at all machines.
- Actual machine states.
- Breakdown status and expected repair time.
- Jobs at risk of tardiness.
- Remaining processing time for each job.
- Downstream bottleneck load.
- Operator availability.
- Tooling availability.
- Setup implications of the next job selected.
- Impact of hot job insertion on existing commitments.

Process mining can expose this by showing cases where decisions were made while critical information existed in the MES but was not incorporated into scheduling.

---

## 3.3 Inaccurate Task Duration and Setup Estimates

If planned task durations are biased or too deterministic, schedules will be unreliable.

Analysis:

\[
EstimationError = ActualDuration - PlannedDuration
\]

Evaluate by:

- Operation type.
- Machine.
- Operator.
- Part family.
- Material.
- Shift.
- Customer.
- Complexity.
- Lot size.
- Setup transition.

Key questions:

- Are processing times systematically underestimated?
- Are setup times ignored or averaged too crudely?
- Is variability high enough that deterministic planning is inappropriate?
- Are certain operators or machines consistently slower/faster?
- Are urgent jobs given unrealistic completion promises?

Process mining can quantify whether tardiness is due to bad sequencing or simply incorrect duration assumptions.

---

## 3.4 Ineffective Handling of Sequence-Dependent Setups

If the current rules do not account for previous job attributes, they may create excessive changeovers.

Root causes:

- No setup matrix exists.
- Setup is treated as a fixed standard allowance.
- Dispatching ignores family similarity.
- Hot jobs trigger costly setups.
- Similar jobs are not grouped.
- Bottleneck machines are sequenced using the same rule as non-bottlenecks.
- Operators choose next jobs based on availability rather than setup impact.

Process mining directly supports this diagnosis by estimating actual setup penalties for transitions and calculating the capacity lost to avoidable setups.

---

## 3.5 Poor Coordination Between Work Centers

Local optimization can damage global performance.

Examples:

- Cutting releases many jobs to Milling without considering Milling capacity.
- Milling processes jobs that later wait for Heat Treatment, creating WIP.
- Inspection is underplanned and becomes a late-stage bottleneck.
- Bottleneck machines are starved because upstream machines process the wrong jobs.
- Downstream machines receive work in bursts, creating unstable queues.

Process mining can identify coordination failures by analyzing:

- Inter-work-center waiting times.
- Arrival variability.
- Upstream completion patterns.
- Downstream queue lengths.
- Jobs delayed because the wrong upstream tasks were prioritized.

---

## 3.6 Inadequate Response to Breakdowns and Urgent Orders

A dynamic job shop must reschedule intelligently when disruptions occur.

Common problems:

- No rapid rescheduling after machine breakdown.
- Jobs remain assigned to unavailable machines.
- Hot jobs are inserted without quantifying displacement cost.
- Recovery rules are ad hoc.
- Bottleneck capacity is consumed by expedites, worsening total tardiness.
- Operators manually improvise without global optimization.

Process mining can analyze:

- How quickly the system responds to breakdowns.
- Which jobs are delayed by disruption events.
- How recovery differs by machine and shift.
- Whether hot jobs improve priority performance at excessive cost to other jobs.
- Whether rescheduling decisions reduce or amplify WIP instability.

---

## 3.7 Differentiating Scheduling Logic Problems from Capacity Problems

This distinction is critical. A better schedule cannot fix a severe capacity shortage, but poor scheduling can make adequate capacity appear insufficient.

Process mining helps distinguish causes through several analyses.

### A. Capacity Load Analysis

If required workload consistently exceeds available capacity at a resource:

\[
RequiredLoad > AvailableCapacity
\]

then the issue is structural capacity.

If required workload is below capacity but queues still grow, then the issue may be sequencing, batching, release timing, or coordination.

---

### B. Counterfactual Simulation

Using process-mined parameters, simulate alternative scheduling rules under the same demand and capacity.

- If better rules substantially improve KPIs, scheduling logic is a major cause.
- If all rules perform poorly under the same workload, capacity is likely insufficient.
- If performance collapses only during breakdown-heavy periods, resilience and contingency planning are weak.

---

### C. Variability Decomposition

Separate delay causes into:

- Waiting due to queue congestion.
- Processing overruns.
- Setup overruns.
- Breakdown downtime.
- Rework.
- Late release.
- Material/tooling shortages.
- Priority preemption.

This creates a delay Pareto chart.

---

### D. Bottleneck Stability Analysis

If the bottleneck shifts constantly because of poor release and dispatching, the problem may be scheduling instability.

If the same resource is overloaded across most periods, there may be a true capacity constraint.

---

### E. Conformance to Available Information

If jobs were late even though there was sufficient capacity and the information needed to prioritize them was available, this indicates poor scheduling logic.

If jobs were late because no feasible sequence could meet all due dates, this indicates either capacity shortage, unrealistic due dates, or excessive demand load.

---

# 4. Developing Advanced Data-Driven Scheduling Strategies

Precision Parts should move from static local dispatching to adaptive, predictive, and globally informed scheduling. The following three strategies can be implemented progressively and tested through simulation before deployment.

---

## Strategy 1: Dynamic Multi-Factor Dispatching Rules

### Core Logic

Replace simple FCFS/EDD with a dynamic priority score for each job waiting at a machine. At every dispatching decision point, the system scores all eligible jobs and selects the one with the best score.

A possible score:

\[
PriorityScore_j =
\alpha_1 Urgency_j
+ \alpha_2 PriorityWeight_j
+ \alpha_3 BottleneckProtection_j
+ \alpha_4 SetupEfficiency_j
+ \alpha_5 DownstreamLoad_j
+ \alpha_6 RemainingWork_j
+ \alpha_7 TardinessRisk_j
\]

Where:

- **Urgency**: based on slack time.
- **Priority weight**: customer/order importance.
- **Bottleneck protection**: prioritizes jobs feeding or using bottlenecks appropriately.
- **Setup efficiency**: favors jobs with low setup time from current machine state.
- **Downstream load**: avoids sending work into overloaded downstream queues unless necessary.
- **Remaining work**: considers total remaining route complexity.
- **Tardiness risk**: predicted probability that job will miss due date.

A useful slack metric:

\[
Slack_j = DueDate_j - CurrentTime - RemainingProcessingTime_j - ExpectedRemainingSetup_j
\]

Critical ratio:

\[
CR_j = \frac{DueDate_j - CurrentTime}{RemainingProcessingTime_j}
\]

Jobs with low or negative slack become urgent.

Setup-adjusted dispatching should use:

\[
EffectiveProcessingTime_{j,m} = ProcessingTime_{j,m} + EstimatedSetupTime_{previous,j,m}
\]

A possible ranking objective:

\[
Score_j =
w_p Priority_j
- w_s Slack_j
- w_q DownstreamQueuePenalty_j
- w_{setup} EstimatedSetupTime_{previous,j,m}
+ w_b BottleneckFeedingValue_j
+ w_r TardinessRisk_j
\]

The exact formulation should be calibrated using historical and simulated performance.

---

### Use of Process Mining Data

Process mining provides:

- Historical waiting time by work center.
- Estimated processing time distributions.
- Remaining route patterns.
- Setup time matrix.
- Bottleneck identification.
- Historical dispatch decisions and their outcomes.
- Tardiness predictors.
- Downstream queue behavior.
- Priority response effectiveness.

The weights in the dispatching rule can be learned or tuned through:

- Regression against historical tardiness.
- Simulation optimization.
- Reinforcement learning in a simulated environment.
- Multi-objective optimization balancing tardiness, WIP, setup time, and utilization.

---

### Pathologies Addressed

This strategy addresses:

- Poor prioritization.
- Local optimization.
- Excessive setup from blind sequencing.
- Jobs becoming late due to ignored remaining work.
- Downstream congestion.
- Bottleneck starvation.
- Lack of real-time responsiveness.

---

### Expected KPI Impact

Expected effects:

- Reduced weighted tardiness.
- Improved on-time delivery.
- Reduced queue time for high-risk jobs.
- Better bottleneck utilization.
- Lower WIP through downstream-load awareness.
- More predictable lead times.
- Some increase in rule complexity, requiring operator trust and clear explanations.

---

## Strategy 2: Predictive and Proactive Scheduling

### Core Logic

Use process-mined historical data to predict task durations, setup durations, breakdown risk, and lateness risk. Then generate schedules using realistic probabilistic estimates rather than deterministic standards.

This can be implemented as a rolling-horizon predictive scheduling system:

1. At regular intervals or events, collect current shop status.
2. Predict remaining processing and setup times for each job.
3. Predict machine availability and breakdown risk.
4. Estimate tardiness risk for each job.
5. Generate a feasible schedule for the next horizon.
6. Re-optimize when disruptions occur.

The scheduling horizon may be:

- Short-term: next 4–8 hours for dispatching.
- Medium-term: next 1–3 days for bottleneck scheduling.
- Long-term: weekly capacity planning.

---

### Predictive Models

#### A. Processing Time Prediction

Model:

\[
ActualDuration = f(activity, machine, operator, material, complexity, lot size, shift, priority, historical performance)
\]

Possible models:

- Gradient boosting.
- Random forest.
- Quantile regression.
- Bayesian hierarchical models.
- Survival models for operation completion.
- Neural networks if data volume is sufficient.

Use prediction intervals, not just point estimates.

Example output:

> Milling operation for JOB-7001 on MILL-03: expected 72 min, P80 88 min, P95 110 min.

---

#### B. Setup Time Prediction

Model:

\[
SetupDuration = f(machine, previous\_family, current\_family, fixture, tooling, operator, shift)
\]

This supports sequence-aware scheduling.

---

#### C. Tardiness Risk Prediction

At each event, predict:

\[
P(Job_j \text{ will be late} | current state)
\]

Features:

- Current slack.
- Remaining operations.
- Remaining bottleneck visits.
- Current queue positions.
- Historical queue times.
- Due date tightness.
- Priority.
- Current WIP.
- Machine availability.
- Breakdown status.
- Setup requirements.

This enables proactive intervention before lateness becomes unavoidable.

---

#### D. Breakdown and Availability Prediction

If MES logs include breakdowns and maintenance events, estimate:

- MTBF and MTTR by machine.
- Failure probability over next shift/day.
- Downtime distributions.
- Post-breakdown recovery time.
- Maintenance-related capacity loss.

If sensor data exists, integrate predictive maintenance signals.

---

### Scheduling Method

Use predictive inputs in one of the following:

- Constraint programming.
- Mixed-integer optimization for bottleneck scheduling.
- Rolling-horizon finite-capacity scheduling.
- Simulation-based optimization.
- Robust scheduling with buffers.
- Stochastic scheduling using duration distributions.
- Hybrid: optimize bottlenecks centrally, dispatch dynamically elsewhere.

A practical approach is:

1. Use optimization to create a schedule for bottleneck machines.
2. Use dynamic dispatching for non-bottleneck machines.
3. Recompute the schedule whenever:
   - A bottleneck breaks down.
   - A hot job arrives.
   - A job becomes high lateness risk.
   - Actual execution deviates significantly from prediction.

## Strategy 3: Sequence-Dependent Setup Time Optimization

### Core Logic

Explicitly minimize setup losses, especially at bottleneck machines, while respecting due dates and priorities.

This strategy can operate at two levels:

1. **Bottleneck machine sequencing**
   - Optimize the order of jobs waiting for or approaching bottleneck machines.
2. **Family-based batching windows**
   - Temporarily group similar jobs when due-date slack allows.

The objective is not simply to minimize setups at all costs. Excessive batching can increase tardiness. The correct objective balances setup savings against due-date risk.

A possible objective:

\[
Minimize \sum_j w_j Tardiness_j + \lambda \sum_m \sum_{i,j} SetupTime_{i,j,m} x_{i,j,m}
\]

Where:

- \(x_{i,j,m}=1\) if job \(j\) follows job \(i\) on machine \(m\).
- \(w_j\) is priority weight.
- \(\lambda\) controls setup minimization strength.

---

### Use of Historical Setup Patterns

Process mining supports this strategy by providing:

- Setup time matrices by machine and family.
- High-cost transition identification.
- Setup variability.
- Operator-specific setup performance.
- Transition frequencies.
- Bottleneck setup burden.
- Jobs whose setup time is avoidable through resequencing.

---

### Possible Implementation Approaches

#### A. Setup-Aware Dispatching

At each machine, when several jobs are available, favor jobs that:

- Have low setup time from current machine state.
- Are in the same part family or fixture family.
- Have enough urgency to justify selection.
- Do not starve downstream bottlenecks.
- Do not cause excessive delay to urgent jobs.

This is easier to implement than full optimization.

---

#### B. Rolling-Horizon Sequencing at Bottlenecks

For bottleneck machines, solve a small sequencing problem every few hours.

Inputs:

- Jobs currently queued.
- Jobs arriving soon from upstream.
- Due dates and priorities.
- Setup matrix.
- Processing time predictions.
- Machine availability.
- Breakdown risk.

Output:

- Recommended sequence for the next several jobs.
- Expected completion times.
- Setup time savings.
- Jobs at risk of lateness.

---

#### C. Controlled Batching

Group jobs by setup family within limited time windows.

Example:

- Process all steel-family parts requiring Fixture A within a 4-hour window if doing so does not cause high-priority tardiness.
- Avoid switching from titanium to aluminum and back to titanium within the same shift unless required by due-date risk.

Batching should be constrained by:

- Maximum allowable waiting time.
- Due-date slack.
- Priority class.
- WIP limits.
- Bottleneck capacity.

---

### Pathologies Addressed

This strategy addresses:

- Excessive setup load.
- Lost bottleneck capacity.
- Suboptimal job-to-job transitions.
- Frequent switching caused by local dispatching.
- Instability caused by hot job insertions.

---

### Expected KPI Impact

Expected effects:

- Reduced total setup time.
- Increased effective bottleneck capacity.
- Higher throughput.
- Reduced average lead time when setup savings dominate.
- Lower WIP before bottlenecks.
- Improved schedule feasibility.
- Potential risk: if batching is too aggressive, some urgent jobs may wait too long. Therefore, setup optimization must be due-date aware.

## Additional Advanced Strategy: WIP-Controlled Release and Bottleneck-Oriented Scheduling

Although only three strategies are required, Precision Parts would benefit strongly from a fourth complementary strategy.

### Core Logic

Control job release to the shop based on real-time WIP and bottleneck capacity rather than releasing jobs immediately.

Methods:

- CONWIP.
- Workload control.
- Drum-Buffer-Rope.
- Bottleneck-based release.
- Dynamic WIP caps by work center.

The bottleneck becomes the “drum.” Upstream release is paced to ensure the bottleneck is never starved but also not flooded.

# 5. Simulation, Evaluation, and Continuous Improvement

Before live deployment, Precision Parts should test candidate scheduling strategies in a realistic, data-driven discrete-event simulation environment.

- Machines.
- Operators.
- Tooling or fixtures, if constraining.
- Material availability.
- Maintenance resources, if relevant.

### Events

- Job release.

- Setup start/end.

- Repair start/end.

- Job completion.

### Resources

- Machine capacities.
- Operator skill matrices.

### Routing

- Actual routings mined from logs.
- Routing probabilities for uncertain flows.
- Rework loops and inspection failures.

### Time Distributions

1. Baseline: current FCFS/EDD mix.

## 5.3 Scenarios to Test

The simulation should include multiple realistic and stress-test scenarios.

### Scenario 1: Normal Demand

Use historical average job arrivals, routings, and due-date mix.

Purpose:

- Validate that the simulation reproduces historical baseline KPIs.
- Compare strategies under typical conditions.

---

### Scenario 2: High Load

Increase job arrivals or workload by 10%, 20%, and 30%.

Purpose:

- Test capacity limits.
- Identify when each strategy breaks down.
- Determine whether bottlenecks are structural.

---

### Scenario 3: Frequent Breakdowns

Increase breakdown frequency or duration for critical machines.

Purpose:

- Test disruption resilience.
- Evaluate dynamic rescheduling and predictive maintenance policies.
- Measure recovery time.

---

### Scenario 4: Hot Job Surge

Inject urgent jobs with tight due dates.

Purpose:

- Evaluate impact on normal jobs.
- Quantify displacement cost.
- Test whether priority handling is controlled or chaotic.

---

### Scenario 5: Setup-Intensive Product Mix

Increase product variety or dissimilar transitions.

Purpose:

- Test setup-aware sequencing.
- Quantify capacity lost to changeovers.
- Evaluate batching policies.

---

### Scenario 6: Operator Constraint

Reduce availability of skilled operators or introduce skill-specific constraints.

Purpose:

- Determine whether apparent machine bottlenecks are actually labor bottlenecks.
- Test cross-training value.

---

### Scenario 7: Due-Date Tightening

Reduce average due-date allowance.

Purpose:

- Test feasibility of customer promises.
- Improve due-date quoting policies.
- Identify when lateness is unavoidable.

---

### Scenario 8: Machine Addition or Outsourcing

Add capacity to a bottleneck or outsource selected operations.

Purpose:

- Compare scheduling improvements against capital investment.
- Determine economic value of extra capacity.

---

## 5.4 Simulation Calibration and Validation

The simulation must first reproduce historical behavior before being trusted.

Validation steps:

1. Run the simulation using the baseline dispatching logic.
2. Feed historical job arrivals and routings.
3. Compare simulated KPIs with actual historical KPIs:
   - Lead time distribution.
   - Queue time by machine.
   - Utilization.
   - Tardiness.
   - WIP.
   - Setup time.
4. Adjust model assumptions until baseline behavior matches reality.
5. Then test alternative strategies.

This avoids the common mistake of optimizing a model that does not represent the real shop.

### D. Adaptive Scheduling Logic

Scheduling rules should be periodically recalibrated.

Possible mechanisms:

1. **Monthly weight tuning**
   - Adjust multi-factor dispatching weights using recent KPI performance.

2. **Rolling retraining of predictive models**
   - Update duration, setup, and tardiness prediction models.

3. **Automatic bottleneck re-identification**
   - If the bottleneck shifts, modify bottleneck-protection logic.

4. **Setup matrix refresh**
   - Recalculate transition times based on recent job mix and operator performance.

5. **Simulation-based policy review**
   - Before making major scheduling logic changes, test them in the digital twin.

6. **Exception learning**
   - Analyze manual overrides and determine whether the scheduling logic should be changed.

### E. Human-in-the-Loop Deployment

Because job shops contain exceptions and tacit knowledge, the system should support human planners rather than blindly replace them.

Recommended features:

- Explainable dispatch recommendations.
- “Why this job next?” reasoning.
- Predicted impact of choosing another job.
- Alert when a manual override may cause tardiness.
- Scenario comparison for planners.
- Feedback capture from supervisors and operators.

Example recommendation:

> Recommended next job on MILL-03: JOB-7012.  
> Reason: high lateness risk, due in 18 hours, requires same fixture as current job, estimated setup 8 minutes versus 42 minutes for JOB-7009, and downstream inspection queue is currently low.

This builds trust and improves adoption.

# Conclusion

Precision Parts Inc. should use process mining to transform its MES logs into a detailed, factual model of how jobs, machines, operators, setups, queues, and disruptions interact. The analysis should quantify lead times, waiting times, machine utilization, setup dependencies, tardiness, and disruption impacts. It should then diagnose bottlenecks, poor prioritization, setup inefficiencies, starvation, and WIP instability.

The likely root cause is not a single issue but a combination of local static dispatching, poor visibility, inaccurate time estimates, ignored sequence-dependent setups, weak cross-work-center coordination, and inadequate disruption response.

A strong improvement program would combine:

1. **Dynamic multi-factor dispatching** using due dates, priority, slack, setup time, downstream load, bottleneck status, and tardiness risk.
2. **Predictive scheduling** using mined duration distributions, setup models, breakdown patterns, and lateness predictions.
3. **Setup-aware sequencing and batching** at bottleneck machines to reduce avoidable changeovers.
4. Optionally, **WIP-controlled release and bottleneck-oriented scheduling** to stabilize flow.

Before deployment, these strategies should be tested in a process-mining-calibrated discrete-event simulation model under normal, high-load, disruption-heavy, hot-job, and setup-intensive scenarios. After deployment, ongoing process mining should continuously monitor KPIs, detect process drift, recalibrate models, and support adaptive scheduling decisions.

The end state is a closed-loop scheduling system: the MES records reality, process mining extracts insight, predictive models forecast risks, scheduling logic recommends actions, simulation validates changes, and continuous monitoring drives ongoing improvement.
