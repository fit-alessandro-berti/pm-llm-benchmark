### **1. Process Discovery and Conformance Checking**

**Data Preprocessing and Integration**

To construct a unified event log suitable for process mining, the following preprocessing steps are essential:

- **Event Log Normalization**: All event sources (GPS, scanner, dispatch, maintenance) must be harmonized into a common schema with mandatory fields: *Timestamp, Event Type, Case ID, Activity, Resource (Driver/Vehicle), Activity Attributes (e.g., Package ID, Location, Speed, Notes)*. The *Case ID* is defined as `VehicleID-Date` (e.g., V12-20241205), enabling tracking of daily operational cycles.

- **Cross-System Correlation**: Key linking attributes must be resolved:
  - *Package ID* from scanner events is matched with dispatch system records to determine planned delivery sequence and time windows.
  - *Location coordinates* from GPS are enriched with geospatial metadata (e.g., street name, traffic zone, proximity to known congestion points) using reverse geocoding and traffic API overlays (e.g., Google Maps or HERE Technologies).
  - *Unscheduled stops* (e.g., “Engine Warning Light”) from GPS are matched with maintenance logs using timestamp proximity and vehicle ID to classify as “maintenance-related delay”.
  - Driver ID links scanner and GPS events to human performance analysis.

- **Challenges Encountered**:
  - **Temporal Misalignment**: GPS data is high-frequency (every 30s), while scanner events are sparse. Interpolation is needed to estimate vehicle status (e.g., “moving” or “idle”) between scanner events.
  - **Missing/Inconsistent Data**: Some scanner events may be skipped (e.g., driver forgets to scan “Depart Customer”). Imputation using GPS speed thresholds (e.g., speed > 2km/h for 2+ mins = departure) can help.
  - **Ambiguous Events**: “Low Speed Detected” (GPS) vs. “Arrive Customer” (scanner) — must be aligned using spatial clustering (e.g., within 100m radius for 5+ mins = arrival).
  - **Data Silos**: Integration requires ETL pipelines (e.g., Apache Airflow) and a data warehouse (e.g., PostgreSQL with spatial extensions) to join relational dispatch data with event streams.

**Process Discovery**

Using **Alpha Miner**, **Heuristic Miner**, or **Inductive Miner** algorithms (via tools like Celonis, Disco, or ProM), we discover the *actual* process model from the integrated event log. The resulting model will reveal:

- The typical sequence: *Depot Departure  Travel  Arrive Customer  Delivery Success/Failure  Depart Customer  ...  Arrive Depot*.
- Common deviations: e.g., *Arrive Customer  Unscheduled Stop (Maintenance)  Return to Customer*.
- Parallel paths: e.g., multiple failed deliveries on a single route leading to re-attempts.
- Rare but costly paths: e.g., *Depot Departure  Traffic Jam (1hr+)  Unscheduled Stop  Late Delivery  Overtime*.

The model will visualize **frequent paths** (highly frequent variants) and **rare deviations** (low-frequency but costly paths), highlighting where actual operations diverge from ideal.

**Conformance Checking**

We compare the discovered *actual* process model against the *planned* route model from the dispatch system (encoded as a BPMN or Petri net of planned stops in sequence with time windows). Using **conformance checking algorithms** (e.g., token-based replay, alignment-based analysis):

- **Sequence Deviations**: Driver skips Stop 3, delivers Stop 5 before Stop 4  violates planned sequence.
- **Unplanned Stops**: Maintenance stops or detours not in dispatch plan  flagged as non-conformance.
- **Timing Deviations**: Actual arrival at Stop 7 is 45 mins late vs. planned window (e.g., 10:00–11:00)  conformance violation.
- **Omission/Redundancy**: Driver visits 38 stops instead of planned 35  over-delivery due to re-attempts or misrouting.

*Key deviations to target*:
- >15% of routes have >2 unplanned stops (often maintenance or failed delivery re-visits).
- >30% of deliveries arrive outside time windows.
- 20% of “Arrive Customer” events occur more than 10 mins after GPS indicates vehicle stopped (suggesting driver delay in walking to door).

This reveals systemic misalignment between planning and execution — not just driver error, but flawed planning assumptions.

### **2. Performance Analysis and Bottleneck Identification**

**Key Performance Indicators (KPIs)**

| KPI | Definition | Calculation from Event Log |
|-----|------------|-----------------------------|
| **On-Time Delivery Rate (OTDR)** | % of deliveries within customer time window | (# Deliveries within window) / (# Total Deliveries) |
| **Average Time per Delivery Stop** | Total service time / # deliveries | Sum of (Depart Customer – Arrive Customer) / Total Deliveries |
| **Travel Time vs. Service Time Ratio** | Efficiency of movement vs. activity | Total GPS “moving” time / Total “service” time (arrive to depart) |
| **Fuel Consumption per km/package** | Cost efficiency | Estimated fuel use (from GPS idle/moving time × vehicle mpg) / (km driven / # packages delivered) |
| **Vehicle Utilization Rate** | % of shift time spent actively delivering | (Total moving + service time) / (Shift duration) |
| **Frequency/Duration of Traffic Delays** | # and avg. duration of speed < 10 km/h for >5 mins | Count of GPS events with speed <10km/h for >5 mins; avg. duration per route |
| **Failed Delivery Rate (FDR)** | % of delivery attempts that fail | (# Delivery Failed) / (# Delivery Attempts) |

*Example*: From the snippet, Stop 1: Service time = 8:28:30 – 8:25:05 = 3.4 mins. Traffic delay: 08:10–08:25 (15 mins). FDR = 1/35 = 2.9% for this day.

**Bottleneck Identification Techniques**

- **Performance Spectrum**: Visualize average duration of each activity (e.g., “Arrive Customer” to “Depart Customer”) across all cases. If “Arrive Customer” takes 8 mins on average but 15 mins on Route X, that location is a bottleneck.
- **Petri Net-based Bottleneck Analysis**: Identify places with high token backlog (e.g., “Waiting for Customer Response” after “Arrive Customer”).
- **Multi-dimensional Filters**: Slice data by:
  - **Route**: Route 7 has 3x higher average service time than Route 1.
  - **Time of Day**: 11:00–13:00 has 40% longer delays due to lunchtime traffic.
  - **Driver**: Driver D105 has 2x higher FDR than D102.
  - **Vehicle Type**: Light vans (V1–V10) have 25% more maintenance stops than newer models (V11–V20).
  - **Location**: “High Street District” shows 50% of traffic delays occur here.

*Quantifying Impact*:  
- If average service time drops from 8.2 to 6.0 mins per stop, and 5000 deliveries/month occur  **1100 saved minutes/month**  potential for 3–4 additional deliveries per driver/day.  
- If FDR drops from 5% to 2%, 150 failed deliveries/month avoided  saves 300 extra km and 12 hours of overtime.

### **3. Root Cause Analysis for Inefficiencies**

Process mining enables deep root cause analysis by correlating anomalies with contextual attributes:

- **Suboptimal Route Planning**:  
  *Evidence*: Conformance checking shows 40% of routes violate planned stop sequence. *Root Cause*: Dispatch system uses static, non-adaptive algorithms (e.g., TSP without real-time traffic or priority zones).  
  *Validation*: Variant analysis shows high-performing routes (OTDR >90%) follow spatially clustered clusters (e.g., 5 stops in same building complex), while low-performing routes zigzag across districts.

- **Inaccurate Travel Time Estimations**:  
  *Evidence*: Planned travel time between Stop 2 and 3 = 8 mins; actual = 22 mins. Correlated with GPS data showing congestion on “Main Avenue” during 9–10 AM.  
  *Validation*: Heatmaps of delay hotspots show consistent congestion at 5 intersections during peak hours — not accounted for in planning.

- **High Variability in Service Time**:  
  *Evidence*: Service time ranges from 2 mins (apartment building with intercom) to 18 mins (residential home with no doorbell).  
  *Validation*: Cluster analysis of dwell times reveals 3 customer types: “High Interaction” (e.g., elderly, complex instructions), “Low Interaction” (e.g., secure lockers), “Unreachable” (FDR >50%).  

- **Vehicle Breakdowns During Shifts**:  
  *Evidence*: 15% of unscheduled stops coincide with “Engine Warning Light” events. Maintenance logs show these vehicles have >4000 engine hours and no predictive maintenance.  
  *Validation*: Correlation between vehicle age and unscheduled stops (r = 0.78).

- **Driver Behavior Differences**:  
  *Evidence*: Driver D105 has 2x more failed deliveries than D102, despite similar routes. D105’s average dwell time is 7 mins longer.  
  *Validation*: Replay of process variants shows D105 often waits at curb instead of walking to door — visible via GPS location drift (stays at vehicle, scanner scan delayed).

- **Failed Delivery Re-attempts**:  
  *Evidence*: 60% of failed deliveries occur between 12–4 PM. Customer time window is 10 AM–6 PM — no optimization for “optimal retry time.”  
  *Validation*: Variant analysis shows re-attempts on same day increase total route length by 18% and fuel cost by 12%.

*Conclusion*: Root causes are systemic — not driver laziness, but poor planning, static routing, lack of customer segmentation, and reactive maintenance.

### **4. Data-Driven Optimization Strategies**

**Strategy 1: Dynamic Route Optimization with Real-Time Traffic and Failed Delivery Prediction**

- **Targets**: Traffic delays, failed deliveries, overtime.  
- **Root Cause Addressed**: Static routing ignores real-time congestion and customer availability patterns.  
- **Process Mining Support**:  
  - Heatmaps from GPS traffic delays + FDR hotspots (e.g., “Maple Street” has 20% FDR + 30-min delays at 2 PM) are fed into a dynamic routing engine.  
  - ML model (trained on historical data) predicts failure probability per customer using features: past FDR, time window, parcel size, location density.  
- **Implementation**:  
  - Dispatch system dynamically re-orders stops every 30 mins based on:  
    (a) Real-time traffic (via API),  
    (b) Predicted failure likelihood (high-risk customers moved to AM slots),  
    (c) Proximity clustering (group nearby high-probability-success stops).  
- **Expected Impact**:  
  - OTDR  from 82%  92%  
  - Average route duration  15%  
  - Failed deliveries  30%  
  - Fuel cost  8–12%

**Strategy 2: Territory Rebalancing and Time Window Optimization Based on Historical Dwell Patterns**

- **Targets**: High variability in service time, inefficient routing.  
- **Root Cause Addressed**: One-size-fits-all time windows and unoptimized territories.  
- **Process Mining Support**:  
  - Cluster customers into 3 types: “Fast” (dwell <5 mins), “Slow” (dwell >10 mins), “Unreachable” (FDR >40%).  
  - Group customers geographically into “micro-territories” (5–7 stops) using DBSCAN on location clusters.  
  - Assign “Fast” zones to AM, “Slow” to late AM/early PM, “Unreachable” to early AM with SMS reminders.  
- **Implementation**:  
  - Dispatch system assigns routes per micro-territory + optimized time windows (e.g., “Unreachable” customers get 8:00–9:30 window, with automated SMS reminder 2 hrs prior).  
  - “Slow” customers flagged for priority delivery (e.g., larger packages, elderly) to avoid rush hour.  
- **Expected Impact**:  
  - Avg. service time  from 8.2  6.1 mins  
  - FDR  25%  
  - Driver satisfaction  (predictable workload)

**Strategy 3: Predictive Maintenance Scheduling Based on Usage Patterns**

- **Targets**: Unscheduled stops, maintenance costs, vehicle downtime.  
- **Root Cause Addressed**: Maintenance scheduled by mileage/interval, not actual stress (e.g., frequent stop-start cycles in urban traffic).  
- **Process Mining Support**:  
  - Extract vehicle usage features: # stop-start cycles/day, avg. idle time, # unscheduled stops, max speed variance.  
  - Train survival model (Cox regression) to predict time-to-failure.  
  - Low-performing vehicles (e.g., V12: 120 stop-starts/day, 1.5 hrs idle/day) show 4x higher failure risk.  
- **Implementation**:  
  - Replace fixed maintenance schedule with **predictive alerts**: “V12 at 85% risk of engine failure in next 72 hrs — schedule for PM on Wed.”  
  - Integrate with maintenance log to refine model weekly.  
- **Expected Impact**:  
  - Unscheduled stops  50%  
  - Maintenance cost  18% (preventive > reactive)  
  - Vehicle utilization  (less downtime)

### **5. Operational Constraints and Continuous Monitoring**

**Accounting for Operational Constraints**

All strategies are designed with constraints in mind:
- **Driver Hours**: Dynamic routing respects legal driving limits (e.g., EU/US regulations) — if re-routing exceeds 10 hrs, system blocks and alerts dispatcher.
- **Vehicle Capacity**: Route optimization ensures package count  capacity (enforced in dispatch engine).
- **Time Windows**: Customer windows are hard constraints — no delivery outside window unless customer opts in (via app).
- **Fallback Rules**: If dynamic rerouting fails to resolve delay, system suggests “defer to next day” for low-priority packages.

**Continuous Monitoring via Process Mining Dashboards**

A real-time **Process Mining Operations Dashboard** (e.g., in Celonis or Power BI + ProM) should track:

| Dashboard View | Key Metrics | Purpose |
|----------------|-----------|---------|
| **Real-Time Route Health** | % of active routes on-time, # traffic delays, # unscheduled stops | Immediate intervention (e.g., reroute V12 now) |
| **Bottleneck Heatmap** | Avg. dwell time, traffic delay frequency by location | Identify new congestion zones or customer clusters needing adjustment |
| **Driver Performance Radar** | OTDR, FDR, Avg. Service Time, Utilization Rate per driver | Identify coaching needs (e.g., D105 needs training on dwell reduction) |
| **Vehicle Health Tracker** | # unscheduled stops, idle time, predicted failure risk per vehicle | Prioritize maintenance scheduling |
| **Cost & Efficiency Trends** | Fuel per km, cost per delivery, OTDR monthly trend | Measure ROI of optimizations |

**Monitoring Plan**:
- **Weekly**: Automated conformance reports comparing new process model vs. optimized baseline.
- **Monthly**: KPI trend analysis + driver/vehicle ranking (top/bottom 10%).
- **Trigger Alerts**:  
  - If FDR rises >4% for 3 consecutive days  investigate customer communication system.  
  - If unscheduled stops spike in V11–V15  trigger maintenance audit.  
  - If OTDR drops below 85%  re-run route optimization model.

This creates a **closed-loop optimization system**: Process mining  insights  action  monitoring  re-learning. Speedy Parcels transitions from reactive firefighting to proactive, data-driven logistics excellence.
