### 1. **Analyzing Resource Behavior and Assignment Patterns**

**Approach:**
To understand how agents and tiers (L1, L2, L3) perform and interact with tickets, I would apply **process mining** techniques to the event log. The goal is to extract, visualize, and quantify the actual flow of tickets through the service desk, comparing it to the intended workflow.

**Key Metrics to Track:**
- **Agent-Level Metrics:**
  - *Average handling time per ticket* (from ticket creation to closure).
  - *First-call resolution (FCR) rate* for L1 agents.
  - *Number of reassignments per agent* (to identify instability or skill mismatches).
  - *Ticket volume per agent* (to assess workload balance).
  - *Skill utilization rate* (how often an agent’s skills match the ticket’s required skills).
- **Tier-Level Metrics:**
  - *Escalation frequency* (how often L1 tickets escalate to L2/L3).
  - *Average time to escalation* (from L1 start to escalation).
  - *Reassignment rate* (tickets reassigned between tiers or agents).
  - *Ticket type distribution per tier* (to see if L2/L3 are handling tickets they’re not optimized for).
- **Process-Level Metrics:**
  - *Process conformance* (how closely actual flows match the defined SOPs).
  - *Process variance* (deviations like early reassignments or late escalations).
  - *Bottleneck identification* (stages where tickets pile up).

**Process Mining Techniques:**
- **Resource Interaction Analysis:** Maps which agents handle which tickets, revealing actual assignment patterns versus intended rules (e.g., round-robin assignment).
- **Social Network Analysis (SNA):** Models handovers between agents/dispatchers, identifying frequent handoff points and potential delays.
- **Role Discovery:** Automatically infers roles or clusters (e.g., “L1 specialists in Networking”) based on activity sequences.
- **Performance Mining:** Measures time-based performance indicators (e.g., time between ticket creation and first action).

**Insights Gained:**
- Reveals whether L1 agents are overloaded or underutilized.
- Shows if escalations are due to skill mismatches or poor initial assignment.
- Highlights skill gaps (e.g., L2 agents handling Database-SQL tickets without the right skills).
- Quantifies how often tickets are reassigned unnecessarily, indicating process friction.

### 2. **Identifying Resource-Related Bottlenecks and Issues**

**Bottleneck Identification:**
- **Skill Shortages:** If many L2 tickets require Database-SQL skills but are assigned to agents without those skills, delays and escalations increase.
- **Reassignment Delays:** Frequent reassignments (e.g., INC-1001 reassigned twice) indicate poor initial routing or unclear ticket categorization.
- **Escalation Overuse:** High escalation rates for P2/P3 tickets suggest L1 agents lack confidence or capability for complex issues.
- **Workload Imbalance:** Some agents consistently handle more tickets than others, leading to burnout or idle time.
- **SLA Correlation:** Tickets with specific attributes (e.g., P2, Network category) may consistently breach SLAs due to resource constraints.

**Quantification:**
- Calculate *average delay per reassignment* (e.g., 15 minutes per reassignment).
- Determine *percentage of SLA breaches linked to skill mismatch* (e.g., 40% of P2 tickets missed SLAs due to incorrect assignment).
- Measure *escalation rate per ticket type* (e.g., 30% of P2 tickets escalated).
- Track *agent utilization variance* (e.g., top 20% agents handle 60% of tickets).

---

### 3. **Root Cause Analysis for Assignment Inefficiencies**

**Potential Root Causes:**
- **Flawed Assignment Rules:** Purely round-robin assignment ignores skills and current workload.
- **Incomplete Skill Profiles:** Agents may lack updated or accurate skill tags, leading to mismatches.
- **Poor Ticket Categorization:** If the system misclassifies tickets (e.g., labeling a Database-SQL issue as Software-App), the wrong agent is assigned.
- **Lack of Real-Time Visibility:** Without dashboards showing agent availability, tickets may be assigned to offline or overloaded agents.
- **Insufficient L1 Empowerment:** If L1 agents lack authority or training to resolve certain issues, they escalate unnecessarily.

**Analysis Techniques:**
- **Variant Analysis:** Compare cases with smooth resolution (few reassignments) vs. problematic ones to identify decision patterns.
- **Decision Mining:** Model decision points (e.g., “If ticket is Network, assign to L2”) to find where rules fail.
- **Root Cause Trees:** Map failure paths (e.g., L1  escalation  L2  reassignment  L3) to pinpoint where breakdowns occur.

### 4. **Developing Data-Driven Resource Assignment Strategies**

**Strategy 1: Skill-Based Routing with Proficiency Weighting**
- **Issue Addressed:** Skill mismatches causing delays and escalations.
- **Insight Used:** Process mining shows frequent skill mismatches and escalations.
- **Data Required:** Agent skill profiles, ticket skill requirements, historical resolution success.
- **Benefits:** Routes tickets to agents with matching or higher proficiency, reducing escalations and improving FCR.

**Strategy 2: Workload-Aware Dynamic Assignment**
- **Issue Addressed:** Uneven workload distribution and bottlenecks.
- **Insight Used:** Resource interaction analysis reveals overloaded agents and idle periods.
- **Data Required:** Real-time queue lengths, agent availability, historical handling times.
- **Benefits:** Balances load, reduces wait times, and prevents burnout.

**Strategy 3: Predictive Assignment Based on Ticket Attributes**
- **Issue Addressed:** Unnecessary reassignments due to poor initial routing.
- **Insight Used:** Social network analysis identifies common handoff patterns.
- **Data Required:** Ticket category, keywords, historical assignment paths.
- **Benefits:** Anticipates required skills early, minimizing reassignments and SLA breaches.

- Regularly re-run process mining to adapt to changing patterns.
- Update skill profiles and rules based on new data.
