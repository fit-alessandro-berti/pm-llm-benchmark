# Visualization Description

This image depicts a **business process model** (workflow diagram), likely using a notation similar to YAWL (Yet Another Workflow Language). It represents a **request handling process**, commonly used as a teaching example in process mining and workflow management.

## Main Process Flow

The diagram shows the following activities (tasks), labeled with letters and descriptions:

- **a** – register request (starting point)
- **b** – examine thoroughly
- **c** – examine casually
- **d** – check ticket
- **e** – decide
- **f** – reinitiate request
- **g** – pay compensation
- **h** – reject request
- **z** – end (terminating point)

## Process Logic

The workflow proceeds as follows:
1. After **registering a request (a)**, the process splits to perform examination tasks and a ticket check.
2. The examination can be done either **thoroughly (b)** or **casually (c)**, alongside **checking the ticket (d)**.
3. These feed into the **decide (e)** task.
4. After deciding, three paths are possible:
   - **Reinitiate the request (f)** — loops back into examination/checking
   - **Pay compensation (g)** — leads to the end
   - **Reject the request (h)** — leads to the end
5. The process concludes at **end (z)**.

## Legend (Gray Box)

The shaded box at the bottom-right explains the **routing/connector semantics** using black dots and arrows:

**Splits** (one input, multiple outputs):
- **XOR-split** – exactly one outgoing path is chosen
- **AND-split** – all outgoing paths are taken (parallel)
- **OR-split** – one or more paths are taken

**Joins** (multiple inputs, one output):
- **XOR-join** – proceeds when any one path arrives
- **AND-join** – waits for all paths (synchronization)
- **OR-join** – waits for all active paths

The **black dots** on the connections in the main diagram correspond to these split/join behaviors, indicating how control flow branches and merges throughout the process.
