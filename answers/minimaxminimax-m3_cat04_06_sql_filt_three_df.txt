# DuckDB SQL Solution

Here's the query that identifies cases with the specified sequence and timing condition, then excludes all events from those cases:

```sql
WITH sequence_check AS (
    -- Look ahead 1 and 2 events within each case (ordered by timestamp)
    SELECT 
        case_id,
        activity,
        timestamp,
        LEAD(activity, 1) OVER (PARTITION BY case_id ORDER BY timestamp) AS next_activity,
        LEAD(timestamp, 1) OVER (PARTITION BY case_id ORDER BY timestamp) AS next_timestamp,
        LEAD(activity, 2) OVER (PARTITION BY case_id ORDER BY timestamp) AS next_next_activity,
        LEAD(timestamp, 2) OVER (PARTITION BY case_id ORDER BY timestamp) AS next_next_timestamp
    FROM event_log
),
cases_to_exclude AS (
    -- Identify cases with the exact sequence and timing condition
    SELECT DISTINCT case_id
    FROM sequence_check
    WHERE activity = 'Approve Purchase Requisition'
      AND next_activity = 'Create Order'
      AND next_next_activity = 'Confirm Order'
      AND (next_next_timestamp - next_timestamp) > INTERVAL '5' DAY
)
-- Return all events that do NOT belong to the excluded cases
SELECT e.*
FROM event_log e
WHERE e.case_id NOT IN (SELECT case_id FROM cases_to_exclude)
ORDER BY e.case_id, e.timestamp;
```

## How It Works

**Step 1 – `sequence_check` CTE:**
Uses the `LEAD` window function twice (`LEAD(activity, 1)` and `LEAD(activity, 2)`) to look at the **next two events** for every row, partitioned by `case_id` and ordered by `timestamp`. This guarantees "direct succession" — there cannot be any other event in between.

**Step 2 – `cases_to_exclude` CTE:**
Filters down to rows where:
- Current activity = `'Approve Purchase Requisition'`
- Next activity = `'Create Order'`
- Activity after that = `'Confirm Order'`
- The time between `next_timestamp` (Create Order) and `next_next_timestamp` (Confirm Order) exceeds **5 days**.

`DISTINCT case_id` ensures a case is only flagged once even if the sequence repeats.

**Step 3 – Final `SELECT`:**
Returns every event from `event_log` whose `case_id` is **not** in the exclusion list, ordered sensibly for downstream inspection.

## Optional Variant (using `EXCEPT` / set difference)

If you prefer a cleaner separation, you can also write it as:

```sql
SELECT *
FROM event_log
WHERE case_id NOT IN (
    SELECT DISTINCT case_id
    FROM (
        SELECT 
            case_id,
            activity,
            timestamp,
            LEAD(activity, 1) OVER (PARTITION BY case_id ORDER BY timestamp) AS n1,
            LEAD(timestamp, 1) OVER (PARTITION BY case_id ORDER BY timestamp) AS t1,
            LEAD(activity, 2) OVER (PARTITION BY case_id ORDER BY timestamp) AS n2,
            LEAD(timestamp, 2) OVER (PARTITION BY case_id ORDER BY timestamp) AS t2
        FROM event_log
    )
    WHERE activity = 'Approve Purchase Requisition'
      AND n1 = 'Create Order'
      AND n2 = 'Confirm Order'
      AND (t2 - t1) > INTERVAL '5' DAY
)
ORDER BY case_id, timestamp;
```

Both versions produce the same result — the first is more readable with named CTEs, the second is a single-statement alternative.
