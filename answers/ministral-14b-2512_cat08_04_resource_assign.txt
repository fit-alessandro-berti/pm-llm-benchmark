# **Comprehensive Process Mining-Driven Optimization of TechSolve’s IT Service Desk Resource Assignment**

## **1. Analyzing Resource Behavior and Assignment Patterns**

### **Key Metrics to Extract from Event Log Data**
To assess agent and tier performance, we would analyze the following metrics using process mining techniques:

#### **Agent-Level Metrics:**
- **Workload Distribution:**
  - Number of tickets handled per agent (volume).
  - Time spent per ticket (processing time).
  - Idle time vs. active time (utilization rate).
- **Resolution Efficiency:**
  - First-call resolution (FCR) rate for L1 agents.
  - Escalation rate (percentage of tickets escalated to L2/L3).
  - Reassignment frequency (how often tickets are moved between agents).
- **Skill Utilization:**
  - Percentage of time agents spend on tasks matching their documented skills.
  - Overqualification (agents with high skills handling low-complexity tickets).
  - Underutilization (specialists idle due to lack of matching tickets).

#### **Tier-Level Metrics:**
- **Escalation Patterns:**
  - Average escalation time (time from L1 assignment to L2/L3 handoff).
  - Escalation success rate (percentage of escalated tickets resolved at the next tier).
- **SLA Compliance:**
  - Percentage of P2/P3 tickets meeting SLAs.
  - Correlation between escalation delays and SLA breaches.
- **Queue Dynamics:**
  - Average wait time in queues before assignment.
  - Bottlenecks in specific skill areas (e.g., long waits for "Networking-Firewall" experts).

#### **Process Mining Techniques for Analysis:**
- **Resource Interaction Analysis:**
  - Visualize agent handoffs (e.g., L1  L2  L3) to identify unnecessary escalations.
  - Detect "handshake" patterns (e.g., frequent reassignments between the same agents).
- **Social Network Analysis (SNA):**
  - Map agent collaboration networks to identify silos or over-reliance on specific specialists.
  - Highlight agents who frequently reassign tickets to others (potential skill gaps).
- **Role Discovery:**
  - Automatically infer agent roles (e.g., "Firewall Specialist," "CRM Troubleshooter") based on ticket handling behavior.
  - Compare inferred roles with documented skill profiles to identify mismatches.
- **Conformance Checking:**
  - Compare actual assignment paths (e.g., L1  L2  L3) against intended workflows to detect deviations.

### **Skill Utilization Analysis**
- **Skill Matching Efficiency:**
  - For each ticket, check if the assigned agent’s skills match the required skill.
  - Calculate the percentage of tickets where the first assignment was optimal vs. required reassignment.
- **Specialist Underutilization:**
  - Identify L2/L3 agents whose skills are rarely used (e.g., a "Database-SQL" expert handling mostly L1-level tickets).
  - Detect cases where L1 agents escalate tickets unnecessarily because they lack basic troubleshooting skills.

## **3. Root Cause Analysis for Assignment Inefficiencies**

### **Potential Root Causes**
| **Root Cause**                          | **Evidence from Process Mining**                                                                 | **Supporting Analysis**                                                                 |
|-----------------------------------------|--------------------------------------------------------------------------------------------------|----------------------------------------------------------------------------------------|
| **Round-Robin Ignoring Skills**         | Agents assigned tickets outside their skill set (e.g., L1 handling "Networking-Firewall").   | **Social network analysis** shows frequent reassignments due to skill mismatches.      |
| **Inaccurate Skill Profiles**          | Agents marked as "App-CRM" but never handle CRM tickets.                                       | **Role discovery** reveals discrepancies between documented and inferred skills.        |
| **Poor Ticket Categorization**         | Tickets labeled "Software-App" but requiring "Database-SQL" skills.                           | **Variant analysis** shows high reassignment rates for tickets with ambiguous categories.|
| **Lack of Real-Time Workload Visibility** | Agents overloaded while others are idle (e.g., L1 queue backlog while L2 specialists wait).  | **Conformance checking** reveals deviations from optimal workload balancing.           |
| **Excessive Escalations from L1**       | L1 agents escalating tickets that could be resolved with better training.                     | **Decision mining** shows L1 agents escalate when they lack confidence, not skills.    |

### **Variant Analysis for Poor Assignments**
- **Good Assignments (Low Reassignments):**
  - Tickets resolved in **1 tier** with **no reassignments**.
  - Common in cases where:
    - L1 agents have **basic troubleshooting** skills matching the ticket.
    - Required skills are **clearly defined** upfront.
- **Bad Assignments (High Reassignments):**
  - Tickets reassigned **2x** or escalated **>1 tier**.
  - Common in cases where:
    - **Skill requirements are ambiguous** (e.g., "Software-App" vs. "Database-SQL").
    - **L1 agents lack empowerment** to diagnose deeper issues.
    - **Specialists are overbooked**, causing delays.

### **Decision Mining Insights**
- **Escalation Triggers:**
  - L1 agents escalate when:
    - Ticket **priority  P2** (even if resolvable).
    - They **lack confidence** (e.g., no recent experience with the category).
  - **Action:** Refine escalation criteria based on **historical L1 resolution rates** per category.

## **4. Data-Driven Resource Assignment Strategies**

### **Strategy 1: Skill-Based Routing with Proficiency Weighting**
**Problem Addressed:**
- Current round-robin assignments lead to **skill mismatches** and **reassignments**.

**Implementation:**
- **Dynamic Skill Matching:**
  - Assign tickets to agents whose **documented skills** best match the **required skill**.
  - Weight assignments by **proficiency level** (e.g., an agent with "App-CRM (Advanced)" gets priority over "App-CRM (Basic)").
- **Fallback Mechanisms:**
  - If no exact match, assign to the **closest skill** (e.g., "App-CRM"  "App-Salesforce" if no CRM expert is available).
  - Escalate only if **no viable agent** is found.

**Data Required:**
- Agent skill profiles (with proficiency levels).
- Historical resolution success rates per skill combination.

**Expected Benefits:**
- **30-50% reassignments** (better first-time assignments).
- **20-40% L1 FCR** (agents handle more tickets within their skill set).
- **15-25% SLA breaches** (faster resolution due to fewer handoffs).

---

### **Strategy 2: Workload-Aware Dynamic Assignment**
**Problem Addressed:**
- **Uneven workload distribution** (some agents overloaded, others idle).

**Implementation:**
- **Real-Time Workload Balancing:**
  - Use **queue length + agent availability** to assign tickets.
  - Prioritize **high-priority tickets (P1/P2)** to agents with **low current workload**.
  - Avoid assigning **low-priority (P4) tickets** to overloaded agents.
- **Predictive Load Forecasting:**
  - Use **historical demand patterns** (e.g., "Networking-Firewall" spikes on Fridays) to pre-assign specialists.

**Data Required:**
- Agent availability (logged hours, breaks).
- Historical ticket volume by **time of day, day of week, category**.
- Current queue lengths per tier.

**Expected Benefits:**
- **20-30% agent idle time** (better utilization).
- **10-20% resolution time** (faster assignment to available agents).
- **SLA compliance** (critical tickets handled by least busy agents).

---

### **Strategy 3: Predictive Assignment Based on Ticket Characteristics**
**Problem Addressed:**
- **Poor initial categorization** leads to **unnecessary escalations**.

**Implementation:**
- **Natural Language Processing (NLP) for Ticket Classification:**
  - Analyze **ticket titles, descriptions, and keywords** to predict **required skills**.
  - Example:
    - Keywords like **"firewall," "port," "VPN"**  **Networking-Firewall**.
    - **"CRM," "Salesforce," "account sync"**  **App-CRM**.
- **Machine Learning Model Training:**
  - Train a classifier on **historical tickets** where the **required skill was later corrected**.
  - Use **process mining** to identify **misclassified tickets** that caused reassignments.

**Data Required:**
- Full text of ticket descriptions (from web portal/email).
- Historical corrections (e.g., tickets reassigned due to **incorrect initial skill tagging**).

**Expected Benefits:**
- **40-60% incorrect initial assignments** (better skill prediction).
- **25-40% escalations** (L1 handles more tickets correctly).
- **Agent productivity** (fewer reassignments = more time per ticket).

## **5. Simulation, Implementation, and Monitoring**

### **Business Process Simulation**
Before implementing changes, we would:
1. **Model Current Process:**
   - Use **Discrete Event Simulation (DES)** to replicate the current assignment logic.
   - Simulate **10,000+ tickets** based on historical data.
2. **Test Proposed Strategies:**
   - Run simulations with:
     - **Skill-based routing** (vs. round-robin).
     - **Workload-aware assignment** (vs. FIFO).
     - **Predictive classification** (vs. manual tagging).
   - Measure **SLA compliance, resolution time, and agent utilization**.
3. **Optimize Parameters:**
   - Adjust **skill weighting thresholds** (e.g., how strict to be on exact matches).
   - Fine-tune **workload balancing rules** (e.g., how much to prioritize high-priority tickets).
