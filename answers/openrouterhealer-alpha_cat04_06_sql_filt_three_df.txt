```sql
WITH numbered_events AS (
    SELECT 
        case_id,
        activity,
        timestamp,
        ROW_NUMBER() OVER (PARTITION BY case_id ORDER BY timestamp) as seq_num
    FROM event_log
),

sequence_starts AS (
    SELECT 
        n1.case_id,
        n1.timestamp as approve_timestamp,
        n2.timestamp as create_timestamp,
        n3.timestamp as confirm_timestamp,
        DATE_DIFF('day', n2.timestamp, n3.timestamp) as days_between
    FROM numbered_events n1
    JOIN numbered_events n2 ON n1.case_id = n2.case_id 
        AND n2.seq_num = n1.seq_num + 1
    JOIN numbered_events n3 ON n1.case_id = n3.case_id 
        AND n3.seq_num = n1.seq_num + 2
    WHERE n1.activity = 'Approve Purchase Requisition'
        AND n2.activity = 'Create Order'
        AND n3.activity = 'Confirm Order'
        AND DATE_DIFF('day', n2.timestamp, n3.timestamp) > 5
),

cases_to_exclude AS (
    SELECT DISTINCT case_id 
    FROM sequence_starts
)

SELECT *
FROM event_log
WHERE case_id NOT IN (SELECT case_id FROM cases_to_exclude)
ORDER BY case_id, timestamp;
```

## Design Decisions and Explanation

This query uses a multi-CTE approach to handle the sequence detection and filtering requirements:

### 1. **Sequence Detection Strategy**
- **`numbered_events`**: First, I assign a sequential row number to each event within a case, ordered by timestamp. This creates a deterministic sequence for each case.
- **`sequence_starts`**: Using this numbered sequence, I join three consecutive events (current, next, and next-next) to find the exact activity sequence. The join condition `seq_num + 1` and `seq_num + 2` ensures direct succession with no intervening events.

### 2. **Timing Condition**
- Within the `sequence_starts` CTE, I calculate `DATE_DIFF('day', n2.timestamp, n3.timestamp)` between 'Create Order' and 'Confirm Order' events.
- The `WHERE` clause includes `> 5` to only include cases where this duration exceeds 5 days.

### 3. **Exclusion Logic**
- **`cases_to_exclude`**: Collects the distinct `case_id`s that meet both the sequence and timing criteria.
- The final `SELECT` filters out events from these cases using `NOT IN`.

### 4. **Performance Considerations**
- The window function approach with `ROW_NUMBER()` is efficient in DuckDB for partitioning and ordering.
- The joins on consecutive sequence numbers are direct and avoid recursive CTEs for better performance.
- The query structure allows DuckDB to optimize execution, particularly with its vectorized processing.

### 5. **Edge Cases Handled**
- Cases with multiple occurrences of the sequence are handled correctly (any occurrence triggers exclusion).
- Cases with fewer than 3 events naturally won't match the sequence condition.
- Ties in timestamps are resolved deterministically by the `ROW_NUMBER()` ordering.

### 6. **DuckDB-Specific Considerations**
- Uses `DATE_DIFF('day', ...)` syntax which is DuckDB's interval function.
- The query is compatible with DuckDB's timestamp handling and window function implementation.
- No external dependencies or UDFs are used, ensuring portability within DuckDB environments.

The solution efficiently identifies and excludes the target cases while returning all other events in a clean, ordered result set.
