This pseudoBPMN describes an order/request handling process that adapts depending on whether the customer request is **standard** or **custom**, performs checks, and may require **manager approval** before issuing the final invoice and confirmation.

1. Start and request intake  
The process begins when a **customer request is received** (Task A). This could be an order, service request, or product inquiry. The system or staff records the request and prepares it for evaluation.

2. Request type decision (XOR Gateway)  
An **exclusive gateway (XOR)** checks the **type of request**. Only one path is chosen:

Ģ If the request is **Standard**, it follows the standard processing path.  
Ģ If the request is **Custom**, it follows the customization analysis path.

XOR means **only one branch is taken** based on the condition.

3. Standard request path  
If the request is standard:

Ģ Task B1: Perform Standard Validation  
  Basic checks ensure the request is complete, valid, and compliant with normal business rules.

Ģ Parallel checks (AND Gateway)  
  An **AND gateway** starts two tasks at the same time:

  - Task C1: Credit Check  
    Verifies the customer's credit status or payment reliability.

  - Task C2: Inventory Check  
    Confirms that the requested items or resources are available.

These tasks run **in parallel** to reduce processing time.

Ģ Parallel join  
The process waits until **both checks finish**. This is the synchronization point of the AND gateway.

Ģ Task D: Calculate Delivery Date  
Using inventory status and other logistics information, the system determines the expected delivery or fulfillment date.

4. Custom request path  
If the request is custom:

Ģ Task B2: Perform Custom Feasibility Analysis  
  Technical, operational, or cost feasibility is assessed to determine if the customization is possible.

Ģ XOR Gateway: Is Customization Feasible?

Two possible outcomes:

- If **Yes**  Task E1: Prepare Custom Quotation  
  A tailored quote is created based on the customization requirements.

- If **No**  Task E2: Send Rejection Notice  
  The customer is informed that the requested customization cannot be delivered, and the process **ends** here.

5. Convergence of paths  
If the request was **standard** (after delivery date calculation) or **custom and feasible** (after quotation preparation), both paths converge into a common stage where the system determines if managerial approval is required.

6. Approval requirement decision (XOR Gateway)  
The gateway checks **whether approval is needed**. Reasons might include high order value, unusual terms, or special customization.

Ģ If **No approval is needed**  
  The process proceeds directly to **Task G: Generate Final Invoice**.

Ģ If **Approval is needed**  
  The process goes to managerial review.

7. Manager approval step  
Ģ Task F: Obtain Manager Approval  
  A manager evaluates pricing, risk, feasibility, or special conditions.

Ģ XOR Gateway: Is Approval Granted?

Two outcomes:

- If **Yes**  
   Task G: Generate Final Invoice.

- If **No**  
   Task H: Reevaluate Conditions.

8. Rework loop  
If approval is denied:

Ģ Task H: Reevaluate Conditions  
  The request is reconsidered and adjusted (e.g., pricing changes, delivery adjustments, or scope reduction).

After re-evaluation, the process **loops back** to the appropriate step depending on the request type:

- For **Custom requests**  back to **Task E1: Prepare Custom Quotation** (quote is revised).  
- For **Standard requests**  back to **Task D: Calculate Delivery Date** (delivery or terms may change).

The process then moves again through the approval check.

9. Finalization  
Once approval is granted (or not required):

Ģ Task G: Generate Final Invoice  
  The final billing document is created.

Ģ Task I: Send Confirmation to Customer  
  The customer receives confirmation including invoice and delivery details.

10. Process end  
After confirmation is sent, the process reaches the **End Event**, indicating completion.

Summary of gateway meanings  
Ģ XOR Gateway: A decision point where **only one path** is taken based on conditions.  
Ģ AND Gateway: Used to **run tasks in parallel** and later synchronize them.

Key behavioral characteristics of the process  
Ģ Two main processing modes: **Standard vs Custom requests**.  
Ģ **Parallel operational checks** for standard orders.  
Ģ **Feasibility evaluation** for custom requests.  
Ģ Optional **manager approval** depending on business rules.  
Ģ **Rework loop** when approval is denied, allowing adjustments rather than terminating the request.  
Ģ Both request types ultimately converge into a **common invoicing and confirmation stage** if they proceed successfully.
