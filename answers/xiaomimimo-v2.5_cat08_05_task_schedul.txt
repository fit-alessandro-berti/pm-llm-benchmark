# Data-Driven Scheduling Optimization for Precision Parts Inc.: A Process Mining and Advanced Scheduling Approach

## 1. Analyzing Historical Scheduling Performance and Dynamics

Using process mining on the MES event logs, we can reconstruct and analyze the actual operational flow with granular precision.

**Reconstruction and Analysis:**
- **Event Log Alignment:** First, we standardize the event log into a process-ready format. Using process mining tools (e.g., Celonis, ProM, or Python libraries like PM4Py), we connect events via `Case ID` (Job ID) and `Resource` (Machine ID). This creates two primary perspectives:
    - **Job-centric process model:** Shows the actual sequence of operations (tasks) each job undergoes across machines, including wait states (queue entries). This reveals the *actual* routing versus the *planned* routing.
    - **Resource-centric process model:** Shows the sequence of jobs processed on each machine, including setup tasks and idle periods. This is crucial for analyzing machine utilization and sequence-dependent effects.

**Quantification of Key Metrics:**
- **Flow/Lead Time & Makespan:** By calculating the time difference between the first event of a job (e.g., `Job Released`) and its final event (e.g., `Task End (Quality Inspection)`), we derive the actual flow time for each job. Aggregating these gives lead time distributions. Makespan is derived for the entire shop or subsets (e.g., per week) by analyzing the span of all active jobs. Process mining visualizations (e.g., directly-follows graphs) will highlight long queues or loops that inflate lead times.
- **Task Waiting Times:** We calculate the duration between `Queue Entry (Task X)` and `Task Start (Task X)` for each activity. Aggregating this by machine or work center quantifies waiting time bottlenecks. Process mining's "Queue Time Analysis" plugin or custom SQL queries on timestamped events provide this directly.
- **Resource Utilization:** We segment a machine's timeline into productive time (`Task Start` to `Task End`), setup time (`Setup Start` to `Setup End`), and idle time (gaps between events where the machine is available but not processing). Utilization is productive time / total time. We can further break down setup time into planned vs. actual to assess estimation accuracy.
- **Sequence-Dependent Setup Times:** This requires advanced analysis. We parse the `Resource`-centric log for each machine. For each setup event, we record the `Case ID` of the *current* job and the `Case ID` of the *previous* job (often noted in `Notes` or inferred by the preceding completed job). By grouping setups by (Previous Job Type  Current Job Type) pairs, we can build a historical **Setup Time Matrix** for each machine, showing average, median, and variance of setup durations for every sequence pair. This quantifies the sequencing problem.
- **Schedule Adherence & Tardiness:** We compare the `Task End` timestamp of the final task for a job against its `Order Due Date`. Tardiness is max(0, Completion Time - Due Date). We calculate metrics like Average Tardiness, % of Tardy Jobs, and Tardiness Severity (total tardy days).
- **Impact of Disruptions:** We filter events for `Breakdown Start`/`End` and `Priority Change` (hot jobs). By correlating these with subsequent events on affected machines and jobs, we measure the ripple effect: how breakdowns increased queue times downstream, how hot jobs displaced other work, and the overall deviation in planned vs. actual completion for jobs processed during these events.

## 2. Diagnosing Scheduling Pathologies

Process mining diagnostics will reveal core inefficiencies:

- **Bottleneck Identification:** Throughput analysis (jobs completed per time unit) per machine, combined with queue time data, will pinpoint bottlenecks. Resources with high utilization (>85-90%), long and variable waiting times for jobs in front of them, and where a large percentage of total shop lead time is spent as waiting time in their queue, are clear candidates. The "Performance Spectrum" analysis can show where jobs are delayed most between consecutive tasks.
- **Poor Prioritization Evidence:** We can perform a **variant analysis** comparing "on-time" vs. "late" job process flows. A common pathology: late jobs often have identical or similar routings and are processed on the same machines as on-time jobs, but their tasks are scheduled *after* lower-priority jobs that arrived later, due to simple FCFS rules. We can quantify the percentage of tardy jobs that were behind a "less urgent" job at a critical resource.
- **Suboptimal Sequencing & Excessive Setups:** By applying the Setup Time Matrix analysis, we can identify machine shifts where jobs were sequenced in a way that incurred very high total setup time (e.g., ABA instead of AABB). We can calculate the "Setup Time Overhead" – the difference between the actual total setup time and the theoretical minimum if jobs were optimally sequenced (solved via a Traveling Salesman Problem approximation on the setup matrix).
- **Starvation & WIP Imbalances:** Process mining models (like Petri nets) will show where downstream machines frequently enter idle states ("starvation") while upstream machines have large queues. This visualizes poor coordination. We can measure WIP levels over time per machine queue. High, variable WIP in certain areas coupled with starvation elsewhere indicates a classic "bullwhip" effect from inconsistent scheduling pushes.
- **Resource Contention:** Analyzing the load on flexible vs. specialized machines will show if specialized "bottleneck" machines are chronically overloaded while flexible machines are underutilized, due to rigid routing rules that don't allow for rerouting.

## 3. Root Cause Analysis of Scheduling Ineffectiveness

Process mining helps dissect the "why" behind the pathologies:

- **Static Rules vs. Dynamic Reality:** The log evidence will show that simple rules like FCFS or EDD fail to account for critical dynamic factors: current machine queue lengths, downstream conditions, and crucially, **sequence-dependent setup costs**. A high-priority job that requires a long setup on a congested machine might be better off waiting if a similar job is about to finish, a decision static rules cannot make.
- **Lack of Real-Time Visibility:** The event logs themselves prove this root cause. Scheduling decisions (queue entries, task starts) are made based on local, instantaneous triggers (job arrival, previous task completion), not a holistic view of shop state. The high variability in actual task times vs. planned (seen in log data) further decouples static plans from reality.
- **Inaccurate Estimations:** By comparing `Task Duration (Planned)` vs. `Task Duration (Actual)` across all tasks, we can quantify estimation error. Poor estimates lead to unrealistic schedules. For setups, the `Setup Required` flag being `TRUE` doesn't guarantee an accurate setup time forecast. Our Setup Time Matrix is the missing predictive model.
- **Poor Coordination & Reactive Disruption Handling:** The log shows jobs moving between independent work centers with no regard for overall flow. Breakdown events are followed by chaotic replanning, evident from sudden shifts in queue lengths and job priorities (hot jobs). There is no evidence of predictive rescheduling or load balancing triggered by these events.

**Distinguishing Logic vs. Capacity/Variability:**
- **Logic Issues:** Indicated by poor sequencing decisions on machines with spare capacity, or systematic mis-prioritization even when jobs are not behind a hard bottleneck.
- **Capacity Issues:** Indicated by chronic, high utilization at a machine where jobs queue for long times *regardless* of sequence or priority. The root cause is insufficient resource for the load.
- **Variability Issues:** Indicated by high variance in task and setup times for identical job types. Even optimal scheduling will struggle here; the solution may need to include buffers or robustness strategies.

## 4. Developing Advanced Data-Driven Scheduling Strategies

### Strategy 1: Dynamic Setup-Aware Dispatching Rules (DSDR)
- **Core Logic:** At each machine, when selecting the next job from the queue, use a weighted scoring function:
  `Score = w1*(Urgency) + w2*(Flow) + w3*(DownstreamLoad) - w4*(Est. Setup Time)`
    - *Urgency:* Based on due date proximity and priority.
    - *Flow:* Jobs that have experienced more delays get a boost to recover flow.
    - *DownstreamLoad:* Prefer jobs whose next resource is less loaded, to prevent future blocking.
    - *Est. Setup Time:* The critical negative term. Uses the historical **Setup Time Matrix** to predict the setup time for the candidate job given the *last job completed on this machine*. This incentivizes sequencing similar jobs together.
- **Data-Driven Insight:** Process mining provides the weights (`w1..w4`) by analyzing which factors were most correlated with on-time completion in historical data. The Setup Time Matrix provides the predictive input. This directly addresses pathologies of poor sequencing and mis-prioritization.
- **Expected Impact:** Reduced average setup times, lower tardiness for high-priority jobs, smoother flow due to downstream consideration, and a reduction in overall WIP as flow becomes more predictable.

### Strategy 2: Predictive & Resilient Scheduling with Stochastic Modeling
- **Core Logic:** Instead of a single static schedule, generate a **predictive schedule** using stochastic task and setup duration distributions (mined from logs, e.g., log-normal for machining, triangular for setups). Use Monte Carlo simulation to build a schedule that has a high probability of completion by due dates.
- **Process Mining Data Use:**
    - Build a probabilistic model of task durations per machine/job type.
    - Use breakdown event logs to estimate MTBF and MTTR per machine to schedule predictive maintenance windows or build buffers.
    - The schedule includes explicit "reschedule points" (e.g., after each shift or after a significant disruption event).
- **Addressing Pathologies:** Tackles unpredictability and disruption impact. It proactively identifies bottlenecks *before* they form by simulating forward. Allows for proactive adjustment when disruptions occur, rather than reactive scrambling.
- **Expected Impact:** Significant reduction in tardiness variance (more predictable lead times), lower WIP due to better flow control, and improved resilience to breakdowns.

### Strategy 3: Setup-Time-Optimized Batching and Sequencing (STOBS)
- **Core Logic:** This strategy operates at a slightly longer horizon (daily/shift level) for bottleneck machines. It performs:
    1. **Dynamic Batching:** Groups incoming jobs with similar characteristics (material, tooling, setup requirements) that are due around the same time. Batches are processed sequentially to minimize setups.
    2. **Optimal Sequencing:** For each batch and for standalone jobs, solves a **Traveling Salesman Problem (TSP)** variant using the historical Setup Time Matrix to find the sequence that minimizes total setup time on the bottleneck machine.
- **Process Mining Data Use:** The entire Setup Time Matrix is the core dataset. The algorithm clusters jobs based on historical setup compatibility derived from the matrix.
- **Addressing Pathologies:** Directly attacks the "excessive setup time" pathology on bottlenecks. It transforms sequencing from a local, real-time decision into a planned, optimized one, freeing up productive capacity.
- **Expected Impact:** Increased effective capacity of bottleneck machines (less time lost to setup), reduced makespan, and lower tardiness for jobs processed on these critical resources.

## 5. Simulation, Evaluation, and Continuous Improvement

**Discrete-Event Simulation (DES) Framework:**
We will build a DES model of the entire job shop. The model will be parameterized directly from the process mining analysis:
- **Routing Logic:** Derived from the most common process variants.
- **Task Time Distributions:** Fitted to the actual task time data (e.g., Weibull distributions).
- **Setup Time Model:** The sequence-dependent setup time matrix for each machine.
- **Disruption Model:** Breakdowns (MTBF/MTTR) and hot job arrival rates and patterns, derived from historical event frequencies.
- **Scheduling Rules:** The baseline (FCFS/EDD) and each proposed strategy (DSDR, Predictive, STOBS) are implemented as scheduling plugins within the simulation.

**Evaluation Scenarios:**
1. **Base Case:** Simulate with average historical load and disruption rates.
2. **High Load Scenario:** Increase arrival rate by 20% to test system stress and bottleneck management.
3. **High Disruption Scenario:** Double the frequency of machine breakdowns and urgent jobs to test resilience.
4. **"What-if" Analysis:** Test capacity investments (e.g., adding a CNC) or different job mixes.

**Evaluation Metrics:** We compare strategies on the same KPIs: Average Tardiness, % Tardy Jobs, Average WIP, Average Lead Time, and Resource Utilization. The goal is to identify the strategy that offers the best trade-off (e.g., a small sacrifice in utilization for a large drop in tardiness).

**Continuous Improvement & Monitoring Framework:**
1. **Online Process Mining Dashboard:** Implement a real-time dashboard using the same MES data, visualizing the KPIs and key process models (bottleneck queues, job flows).
2. **Automated Anomaly Detection:** Use statistical process control (SPC) or machine learning on the live event stream to detect deviations: e.g., a machine's average queue time suddenly increases, or the frequency of high-setup sequences rises. This triggers alerts.
3. **Strategy Re-tuning Cycle:** On a quarterly basis, re-run the process mining analysis on the latest data. Check for:
    - **Concept Drift:** Has the optimal dispatching weight changed? Are new job types introducing new setup patterns?
    - **New Bottlenecks:** Has shifting demand created new constraints?
    - **KPI Degradation:** Are tardiness or WIP creeping up despite the new strategy?
4. **Feedback Loop:** The insights from monitoring feed back into updating the simulation model and refining the scheduling algorithms. This creates a closed-loop system for perpetual optimization.

By integrating deep process mining diagnostics with advanced, data-informed scheduling strategies and validating them in a realistic simulation, Precision Parts Inc. can transition from a reactive, rule-based shop floor to a predictive, optimized, and resilient manufacturing operation.
