# **Process Mining-Driven Optimization for Speedy Parcels: A Comprehensive Approach**

Speedy Parcels’ challenges—late deliveries and rising operational costs—are classic symptoms of inefficiencies in last-mile logistics. By leveraging **process mining**, we can transform raw event data (GPS, scanners, dispatch, maintenance) into actionable insights to diagnose inefficiencies, validate root causes, and implement data-driven optimizations. Below is a structured approach addressing each of the five key areas.

---

---

## **1. Process Discovery and Conformance Checking**

### **1.1 Data Preprocessing and Integration**
**Objective:** Consolidate heterogeneous data sources into a unified event log suitable for process mining (e.g., XES format).

#### **Steps:**
1. **Data Alignment by Case ID:**
   - Define a **case** as a *vehicle-day* (e.g., `V12-20241205`), representing the operational cycle of a van from depot departure to return.
   - Link sub-processes (e.g., package deliveries) to the main case via `Package ID` or `Stop Number`.

2. **Event Log Construction:**
   - **Timestamp:** Standardize time zones and formats (e.g., ISO 8601).
   - **Activity Labeling:** Normalize event types (e.g., "Arrive Customer"  `Customer Arrival`; "Low Speed Detected"  `Traffic Delay`).
   - **Resource Mapping:** Associate `Vehicle ID`, `Driver ID`, and `Package ID` with each event.
   - **Location Data:** Attach GPS coordinates to movement events (e.g., `Depart Depot`, `Low Speed Detected`).

3. **Handling Missing/Noisy Data:**
   - **Gaps in GPS Data:** Use interpolation for short missing intervals (e.g., linear interpolation for speed/location).
   - **Inconsistent Scanner Events:** Flag cases where `Arrive Customer` lacks a corresponding `Depart Customer` or `Delivery Success/Failure`.
   - **Maintenance Overlaps:** Ensure maintenance events are correctly linked to the vehicle-day case (e.g., if a repair starts at 11:00, all subsequent events for that vehicle should reflect the downtime).

**Challenges:**
- **Temporal Misalignment:** GPS and scanner events may have slight timestamp discrepancies (e.g., GPS records movement 30 seconds before the scanner logs `Arrive Customer`).
  *Solution:* Use a tolerance window (e.g., ±2 minutes) to merge related events.
- **Semantic Heterogeneity:** Different systems use different terms for the same activity (e.g., "Ignition On" vs. "Start Shift").
  *Solution:* Create a controlled vocabulary (ontology) for event types.
- **Sparse Data:** Some events (e.g., unscheduled stops) may lack detailed notes.
  *Solution:* Use clustering (e.g., DBSCAN) on GPS data to detect anomalous stops not logged in scanners.

---

### **1.2 Process Discovery**
**Objective:** Visualize the *actual* end-to-end delivery process, including deviations from the planned workflow.

#### **Techniques:**
- **Alpha Miner / Inductive Miner:**
  - Generate a **process model** (e.g., Petri net or BPMN) from the event log to show the sequence of activities (e.g., `Start Shift  Route Assigned  Depart Depot  [Traffic Delay/Stop]  Arrive Customer  Delivery Success/Failure  ...  Arrive Depot  End Shift`).
  - **Key Insight:** Identify frequent paths (e.g., most deliveries follow `Arrive  Success  Depart`), as well as rare but costly deviations (e.g., `Arrive  Failed  Depart  Reattempt Later`).

- **Heuristics Miner:**
  - Useful for noisy logs (e.g., GPS pings every few seconds). Filter to key milestones (e.g., depot events, customer stops, maintenance) to avoid overfitting.

- **Fuzzy Miner:**
  - Aggregate low-level events (e.g., GPS pings) into higher-level clusters (e.g., "Travel," "Service," "Idle") to simplify the model.

#### **Expected Output:**
A **process map** highlighting:
- **Main Flow:** Typical delivery sequence (e.g., 80% of cases follow `Depart Depot  Stop 1  Stop 2  ...  Return`).
- **Deviations:**
  - Unplanned stops (e.g., engine warning, traffic).
  - Failed deliveries requiring reattempts.
  - Loops (e.g., returning to a customer after a failed attempt).

---

### **1.3 Conformance Checking**
**Objective:** Compare the discovered model against the **planned routes** from the dispatch system.

#### **Approach:**
1. **Extract Planned Process:**
   - From the dispatch system, derive the *intended* sequence (e.g., `Depart Depot  [Stop 1, Stop 2, ..., Stop 35]  Return Depot`).

2. **Alignment-Based Conformance Checking:**
   - Use **alignments** (e.g., in ProM or Celonis) to map actual traces to planned traces, identifying:
     - **Sequence Deviations:** E.g., Stop 3 was visited before Stop 2.
     - **Missing/Extra Activities:** E.g., unplanned maintenance stop; skipped customer.
     - **Timing Deviations:** E.g., time between `Depart Depot` and `Arrive Stop 1` exceeds planned duration by 30%.

3. **Fitness and Precision Metrics:**
   - **Fitness:** % of actual traces that can be replayed by the planned model (low fitness  frequent deviations).
   - **Precision:** % of planned behavior observed in reality (low precision  planned model allows unrealistic paths).

**Key Deviations to Investigate:**
| Deviation Type          | Example                          | Potential Cause                     |
|-------------------------|----------------------------------|-------------------------------------|
| **Sequence Swap**       | Stop 5 before Stop 4              | Driver took a suboptimal route     |
| **Unplanned Stop**      | Engine warning at 11:05          | Vehicle maintenance issue           |
| **Timing Overrun**      | Stop 10 took 20 mins (planned: 5)| Traffic or long customer interaction|
| **Skipped Stop**        | Stop 15 not visited              | Driver error or time pressure       |
| **Reattempt Loop**      | Stop 20  Failed  Stop 20 again | Customer unavailable                |

**Tools:** ProM, Celonis, Disco, or Python libraries (`pm4py`).

## **2. Performance Analysis and Bottleneck Identification**

### **2.1 Key Performance Indicators (KPIs)**
**Definition and Calculation from Event Log:**

| KPI                          | Formula                                                                 | Data Sources                          | Business Relevance                          |
|------------------------------|-------------------------------------------------------------------------|---------------------------------------|---------------------------------------------|
| **On-Time Delivery Rate**    | `(# Deliveries within time window) / (Total Deliveries)` × 100          | Scanner (Delivery Success + Timestamp) | Customer satisfaction                       |
| **Average Time per Stop**    | `Sum(Depart Customer - Arrive Customer) / Total Stops`                 | Scanner events                        | Efficiency of service time                  |
| **Travel Time Ratio**        | `Total Travel Time / Total Service Time`                               | GPS (movement) + Scanner (stops)      | Balance between driving and delivering     |
| **Fuel Consumption/km**      | `Total Fuel Used / Total Distance` (from GPS)                           | GPS (speed, distance) + Fuel logs     | Cost efficiency                             |
| **Vehicle Utilization Rate** | `(Total Active Time) / (Shift Duration)` × 100                          | GPS (ignition on/off)                  | Productivity                                |
| **Failed Delivery Rate**     | `(# Failed Deliveries) / (Total Attempts)` × 100                        | Scanner (Delivery Failed)             | Rework cost (reattempts)                    |
| **Idle Time %**              | `(Total Idle Time) / (Shift Duration)` × 100                            | GPS (speed = 0, not at customer)      | Wasted time (traffic, parking)              |
| **Maintenance Downtime**     | `Sum(Repair Duration)` per vehicle/day                                  | Maintenance Logs                      | Reliability and cost                        |

---

### **2.2 Bottleneck Identification Techniques**
**Process Mining Tools for Bottlenecks:**
1. **Performance Spectra (ProM):**
   - Visualize **time distributions** for each activity (e.g., `Arrive Customer` to `Depart Customer`).
   - Identify outliers (e.g., stops taking >15 minutes when average is 5).

2. **Dotted Chart Analysis:**
   - Plot events over time to spot:
     - **Clusters of Delays:** E.g., frequent `Low Speed Detected` between 8:00–9:00 AM in a specific area (traffic hotspot).
     - **Long Service Times:** E.g., certain drivers consistently take longer at stops.

3. **Bottleneck Analysis (Celonis):**
   - Rank activities by **waiting time** or **cycle time** (e.g., "Finding Parking" adds 10 mins per stop in downtown areas).

4. **Variant Analysis:**
   - Compare **fast vs. slow routes**:
     - *Fast Variant:* `Depart Depot  Stop 1 (3 mins)  Stop 2 (4 mins)  ...`
     - *Slow Variant:* `Depart Depot  Traffic (20 mins)  Stop 1 (10 mins)  ...`
   - Identify **common patterns** in slow variants (e.g., traffic, failed deliveries).

5. **Spatial Analysis:**
   - Overlay GPS data on a map to detect:
     - **Traffic Bottlenecks:** Areas with frequent low-speed events.
     - **Inefficient Detours:** Routes deviating from the shortest path (use **Haversine distance** to compare actual vs. planned distance).

**Quantifying Impact:**
- **Cost of Delays:** Estimate fuel wasted during idle time (e.g., 10 mins idle = 0.2L fuel × $1.50/L = $0.30 per stop).
- **Reattempt Costs:** Each failed delivery adds ~$5 in fuel, driver time, and customer dissatisfaction.

**Example Findings:**
| Bottleneck               | Metric Affected          | Root Cause Hypothesis          | Impact                     |
|--------------------------|--------------------------|--------------------------------|----------------------------|
| Downtown Stops (8–10 AM) | Avg. Stop Time = 12 mins | Parking difficulty             | +$2.40/stop in labor cost   |
| Vehicle V12              | 15% Idle Time             | Frequent engine warnings       | +$50/day in fuel/maintenance|
| Driver D105              | 20% Failed Deliveries    | Poor customer communication   | +$100/day in reattempts    |

## **3. Root Cause Analysis for Inefficiencies**

### **3.1 Potential Root Causes**
| **Category**               | **Specific Issue**                          | **Evidence in Data**                                                                 |
|----------------------------|---------------------------------------------|--------------------------------------------------------------------------------------|
| **Route Planning**         | Static routes ignore real-time traffic     | High `Travel Time Ratio` in areas with frequent `Low Speed Detected` events.        |
| **Time Estimations**       | Underestimated service time per stop       | `Avg. Time per Stop` consistently exceeds planned duration.                          |
| **Traffic Patterns**       | Rush-hour congestion in urban areas         | `Low Speed Detected` events cluster during 7–9 AM and 4–6 PM in city center.        |
| **Service Time Variability** | Long customer interactions                 | High variance in `Depart Customer - Arrive Customer` time across stops.             |
| **Vehicle Reliability**    | Frequent unscheduled maintenance            | `Unscheduled Stop` events correlated with `Engine Warning Light` in maintenance logs.|
| **Driver Behavior**        | Some drivers take longer routes             | Variant analysis shows Driver D105’s routes are 20% longer than average.           |
| **Failed Deliveries**      | High reattempt rate due to missed windows  | 15% of deliveries are `Delivery Failed`, with 60% reattempted the next day.          |

---

### **3.2 Process Mining Techniques for Validation**
1. **Variant Analysis:**
   - Compare **high-performing routes** (on-time, low cost) vs. **low-performing routes**:
     - *Finding:* Fast routes avoid a specific highway during peak hours.
     - *Action:* Adjust dynamic routing to avoid this highway.

2. **Correlation Analysis:**
   - **Traffic vs. Delays:** Overlay `Low Speed Detected` events with external traffic data (e.g., Google Maps API) to confirm congestion as a root cause.
   - **Maintenance vs. Breakdowns:** Check if vehicles with frequent `Unscheduled Stop` events have higher maintenance logs.

3. **Dwell Time Analysis:**
   - **At Customer Locations:** If `Depart Customer - Arrive Customer` > 10 mins for 30% of stops, investigate:
     - Are these high-rise buildings (elevator wait times)?
     - Are certain drivers slower at scanning packages?

4. **Causal Inference (Advanced):**
   - Use **Granger Causality** or **Bayesian Networks** to test hypotheses:
     - *Hypothesis:* "Failed deliveries cause higher fuel consumption due to reattempts."
     - *Test:* Does `Delivery Failed` predict increased `Fuel Consumption/km` in subsequent days?

**Example Validation:**
- **Hypothesis:** "Traffic in Zone A causes 40% of delays."
  - *Method:* Filter event log for `Low Speed Detected` in Zone A during 8–10 AM.
  - *Result:* 80% of delays in Zone A occur in this window  Confirmed.

## **4. Data-Driven Optimization Strategies**

### **Strategy 1: Dynamic Routing with Real-Time Traffic Integration**
**Targeted Inefficiency:** Static routes leading to traffic delays.
**Root Cause:** Planned routes do not account for real-time congestion.
**Process Mining Insight:**
- `Low Speed Detected` events cluster in specific areas/time windows.
- Variants with detours around these areas have 15% faster delivery times.

**Proposal:**
- Integrate **real-time traffic APIs** (e.g., HERE, TomTom) into the dispatch system.
- Use **reinforcement learning** or **ant colony optimization** to adjust routes dynamically based on:
  - Historical traffic patterns (from GPS logs).
  - Live traffic data.
  - Predicted service times (from scanner data).

**Expected Impact:**
| KPI                     | Before | After  | Improvement |
|-------------------------|--------|--------|--------------|
| On-Time Delivery Rate  | 75%    | 85%    | +10%         |
| Travel Time Ratio       | 1.8    | 1.5    | -17%         |
| Fuel Consumption/km     | 0.12L  | 0.10L  | -17%         |

**Constraints Addressed:**
- Driver working hours: Ensure dynamic routes do not exceed legal shift limits.
- Customer time windows: Prioritize time-sensitive deliveries in routing adjustments.

---

### **Strategy 2: Predictive Maintenance Based on Usage Patterns**
**Targeted Inefficiency:** Unscheduled vehicle stops due to breakdowns.
**Root Cause:** Reactive maintenance leads to downtime and higher repair costs.
**Process Mining Insight:**
- Vehicles with >5 `Unscheduled Stop` events/month have 30% higher maintenance costs.
- `Engine Warning Light` events precede 80% of breakdowns by <24 hours.

**Proposal:**
- Develop a **predictive maintenance model** using:
  - **Features:** Vehicle age, mileage, frequency of `Engine Warning Light`, idle time, speed patterns.
  - **Target:** Probability of unscheduled stop in the next 7 days.
- **Actions:**
  - Schedule maintenance proactively for high-risk vehicles.
  - Replace or repair vehicles with recurring issues.

**Expected Impact:**
| KPI                     | Before | After  | Improvement |
|-------------------------|--------|--------|--------------|
| Maintenance Downtime    | 2.5 h/week | 0.5 h/week | -80%       |
| Vehicle Utilization Rate| 85%    | 92%    | +7%          |
| Fuel Consumption/km     | 0.12L  | 0.11L  | -8%          |

**Constraints Addressed:**
- Vehicle capacity: Ensure maintenance scheduling does not disrupt peak delivery days.
- Driver assignments: Reassign drivers to other vehicles during maintenance.

---

### **Strategy 3: Failed Delivery Reduction via Smart Time Windows**
**Targeted Inefficiency:** 15% failed delivery rate due to missed customer availability.
**Root Cause:** Static time windows do not account for customer behavior or driver delays.
**Process Mining Insight:**
- 60% of failed deliveries occur in the last 2 hours of the shift (driver rushing).
- Customers in suburban areas are 2x more likely to be unavailable during 9–11 AM.

**Proposal:**
1. **Dynamic Time Windows:**
   - Use **machine learning** to predict customer availability based on:
     - Historical `Delivery Success/Failure` patterns (e.g., Customer X is home 90% of the time before 10 AM).
     - Driver’s current progress (e.g., if behind schedule, adjust subsequent windows).
   - Offer **flexible slots** (e.g., "Morning," "Afternoon") instead of fixed 1-hour windows.

2. **Proactive Customer Communication:**
   - Send **SMS notifications** 30 mins before arrival with a **live tracking link** (using GPS data).
   - Allow customers to **confirm availability** or reschedule via a mobile app.

3. **Reattempt Optimization:**
   - Group failed deliveries by **geographic cluster** and assign to the next day’s route for the same area.

**Expected Impact:**
| KPI                     | Before | After  | Improvement |
|-------------------------|--------|--------|--------------|
| Failed Delivery Rate    | 15%    | 8%     | -7%          |
| On-Time Delivery Rate   | 75%    | 82%    | +7%          |
| Reattempt Costs         | $100/day | $40/day | -60%       |

**Constraints Addressed:**
- Driver working hours: Reduce time spent on reattempts by clustering them efficiently.
- Vehicle capacity: Ensure reattempts do not overload the next day’s route.

## **5. Operational Constraints and Continuous Monitoring**

### **5.1 Accounting for Constraints**
| **Constraint**               | **Impact on Strategies**                                                                 | **Mitigation Approach**                                                                 |
|------------------------------|-----------------------------------------------------------------------------------------|-----------------------------------------------------------------------------------------|
| **Driver Working Hours**     | Dynamic routing must not extend shifts beyond legal limits (e.g., 8 hours).            | Cap route adjustments to stay within shift duration; use overtime only for critical cases. |
| **Vehicle Capacity**         | Predictive maintenance may reduce fleet availability.                                  | Stagger maintenance; use rental vehicles during peak periods.                          |
| **Customer Time Windows**    | Dynamic time windows must respect customer preferences.                              | Allow customers to opt into flexible slots; penalize last-minute changes.              |
| **Traffic Regulations**      | Some routes may be restricted (e.g., no left turns, truck restrictions).              | Integrate traffic rules into the routing algorithm.                                    |
| **Union/Contract Rules**     | Drivers may resist changes to routes or schedules.                                     | Involve drivers in pilot testing; highlight efficiency gains (e.g., less idle time).     |

---

   - **Current State:** Live process map showing active vehicle-day cases.
   - **Deviation Alerts:** Flag unplanned stops, failed deliveries, or excessive idle time.

2. **KPI Tracking:**

   |--------------------------|----------------------|---------------------------------------|-------------------------------------|
   | On-Time Delivery Rate    | Daily                | <80%                                  | Investigate root causes for delays. |

   | Maintenance Downtime     | Weekly               | >1% of fleet time                     | Schedule proactive maintenance.     |

3. **Drill-Down Capabilities:**
   - **By Route:** Identify which routes have the highest `Travel Time Ratio`.
   - **By Driver:** Compare performance metrics across drivers (e.g., `Avg. Time per Stop`).
   - **By Vehicle:** Track maintenance trends per vehicle.
   - **By Time/Location:** Heatmaps of `Low Speed Detected` events to spot congestion.

4. **Predictive Analytics:**
   - **Forecast Delays:** Use historical data to predict delays (e.g., "Route X will likely be 30 mins late due to traffic").
   - **Anomaly Detection:** Flag unusual patterns (e.g., a driver suddenly taking 50% longer per stop).

5. **Feedback Loop:**
   - **Driver Input:** Allow drivers to report issues (e.g., "Traffic jam at Location Y") via the handheld scanner.
   - **Customer Feedback:** Integrate delivery satisfaction ratings to correlate with process metrics.

# **Conclusion**
By applying **process mining** to Speedy Parcels’ event data, we can:
1. **Diagnose** inefficiencies (e.g., traffic delays, failed deliveries, maintenance issues).
2. **Validate** root causes through variant analysis, correlation, and dwell time studies.
3. **Optimize** with dynamic routing, predictive maintenance, and smart time windows.
4. **Monitor** continuously to ensure improvements are sustained.

The proposed strategies are **data-driven**, **actionable**, and **aligned with last-mile delivery constraints**, offering a clear path to **higher punctuality and lower costs**. Implementation should start with a **pilot on high-impact routes** (e.g., those with the most delays) and scale based on results.
