Here is the DuckDB SQL query designed to meet your requirements. It uses Common Table Expressions (CTEs) to break down the logic into clear steps: generating variants, ranking them by frequency, filtering for the top K, and finally selecting the relevant events.

```sql
WITH 
-- Step 1: Construct the ordered activity sequence for each case_id
case_variants AS (
    SELECT 
        case_id,
        -- Aggregates activities into a string, maintaining timestamp order
        STRING_AGG(activity, ' -> ' ORDER BY timestamp) AS variant_sequence
    FROM event_log
    GROUP BY case_id
),

-- Step 2: Count cases per variant and rank them by frequency
variant_counts AS (
    SELECT 
        variant_sequence,
        COUNT(*) AS case_count,
        -- RANK() handles ties in count appropriately
        RANK() OVER (ORDER BY COUNT(*) DESC) AS frequency_rank
    FROM case_variants
    GROUP BY variant_sequence
),

-- Step 3: Filter to the Top K variants (e.g., K=5)
top_k_variants AS (
    SELECT 
        variant_sequence
    FROM variant_counts
    WHERE frequency_rank <= 5 -- Adjust this number to change K
),

-- Step 4: Identify all case_ids belonging to the top K variants
valid_cases AS (
    SELECT 
        cv.case_id
    FROM case_variants cv
    JOIN top_k_variants tkv ON cv.variant_sequence = tkv.variant_sequence
)

-- Step 5: Return original events only for cases in the valid_cases list
SELECT 
    e.case_id,
    e.activity,
    e.timestamp
FROM event_log e
JOIN valid_cases vc ON e.case_id = vc.case_id
ORDER BY e.case_id, e.timestamp;
```

### Explanation of the Query Logic:

1.  **`case_variants` CTE**: 
    *   Groups events by `case_id`.
    *   Uses `STRING_AGG(... ORDER BY timestamp)` to create a unique fingerprint for each process variant (e.g., `"Login -> Search -> Buy"`). This preserves the temporal order defined in your schema.
2.  **`variant_counts` CTE**: 
    *   Groups by the generated sequence string.
    *   Calculates how many cases follow that specific path (`COUNT(*)`).
    *   Uses `RANK()` to assign a frequency rank (1 being the most common).
3.  **`top_k_variants` CTE**: 
    *   Filters the variants to keep only those with a `frequency_rank` less than or equal to your chosen K (currently set to 5).
4.  **`valid_cases` CTE**: 
    *   Joins the original variant sequences back to their `case_id`s where the sequence is in the top list. This creates a whitelist of allowed `case_ids`.
5.  **Final SELECT**: 
    *   Selects rows from the original `event_log` table only if the `case_id` exists in the `valid_cases` list.
    *   The final result includes all events (not just the summary) for those filtered cases, ordered by their original timestamp.

### Note on Parameter K:
To change the number of top variants, simply modify the line `WHERE frequency_rank <= 5`. Since DuckDB does not support direct variable substitution in standard SQL scripts, you must hardcode this integer or wrap the query in a stored procedure/function if dynamic parameters are required.
