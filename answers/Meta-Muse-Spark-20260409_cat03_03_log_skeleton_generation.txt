Got it — we need to turn that order fulfillment flow into a Log Skeleton dictionary. Based on standard warehouse logic, here’s a reasonable constraint model you can use as a starting point:

```python
log_skeleton = {
    'equivalence': {
        ('RO', 'IC'),      # Every order received must be invoiced exactly once
        ('GSL', 'D')       # A label is generated iff the package is dispatched
    },
    
    'always_before': {
        ('CA', 'RO'),      # Availability is only checked after order is received
        ('RI', 'CA'),      # Can only reserve after checking availability
        ('PI', 'RI'),      # Picking happens only after items are reserved
        ('QC', 'PI'),      # Quality check happens after picking
        ('PO', 'QC'),      # Pack only after quality check passes
        ('GSL', 'PO'),     # Label generated after packing
        ('D', 'GSL'),      # Dispatch after label is attached
        ('IC', 'D'),       # Invoice after dispatch in this model
        ('RP', 'IC')       # Payment recorded only after invoice exists
    },
    
    'always_after': {
        ('RO', 'CA'),      # Receiving an order means availability must be checked later
        ('CA', 'RI'),     # If we check availability, reservation must follow
        ('RI', 'PI'),     # Reserved items must be picked
        ('PI', 'QC'),     # Picked items must be quality checked
        ('QC', 'PO'),     # Items that pass QC must be packed
        ('PO', 'GSL'),    # Packed orders must get a label
        ('GSL', 'D'),     # Labeled packages must be dispatched
        ('D', 'IC')       # Dispatched orders must be invoiced
    },
    
    'never_together': set(),  # No activities are mutually exclusive in normal flow
    
    'activ_freq': {
        'RO': {1},        # Exactly one order received per case
        'CA': {1},        # Check availability once per order
        'RI': {0, 1},     # May skip reservation if out of stock
        'PI': {0, 1},     # Pick only if items were reserved
        'QC': {0, 1},     # QC only if items were picked
        'PO': {0, 1},     # Pack only if QC passed
        'GSL': {0, 1},    # Label only if packed
        'D': {0, 1},      # Dispatch only if labeled
        'IC': {1},        # Every case generates an invoice, even if cancelled
        'RP': {0, 1}      # Payment may not be received in same case
    },
    
    'directly_follows': {
        ('RO', 'CA'),     # First step after receive is check availability
        ('QC', 'PO'),     # Items go straight from QC to packing
        ('GSL', 'D')      # Dispatch happens immediately after labeling
    }
}
```

### Key design choices explained

| Constraint type | Why these pairs |
| --- | --- |
| **equivalence** | `RO` and `IC` are 1:1 because every order triggers billing. `GSL` and `D` are paired because you don't create a label unless you ship, and you don't ship without a label. |
| **always_before** | Captures the main sequence: you can't reserve before checking stock, can't pack before QC, etc. |
| **always_after** | Ensures the process doesn't stall. If `RO` happens, we must at least try `CA`. If items are picked `PI`, they must be checked `QC`. |
| **never_together** | Left empty because in a standard case none of these are forbidden together. You could add something like `('RI', 'Cancel')` if you had a cancellation activity. |
| **activ_freq** | `RO`, `CA`, `IC` occur exactly once. Downstream activities `RI` through `RP` are `{0,1}` to allow for out-of-stock cases where the flow stops early. |
| **directly_follows** | Only used where the business rule is strict. `QC` → `PO` is usually immediate. You wouldn't want other activities inserted there. |
