### Data Preprocessing and Integration

#### 1. Defining the Case ID
In last-mile logistics, a single-perspective case ID often obscures critical operational realities. We establish a **primary multi-perspective schema**:
*   **Case ID (Operational View):** `Vehicle-Day` (e.g., `V12-20241205`). This captures the end-to-end journey of a single asset and driver from depot departure to return.
*   **Nested/Sub-Case ID (Item View):** `Package ID` (e.g., `P9876`). This allows us to track individual parcel lifecycles within the broader vehicle journey, resolving issues like multi-package drops at single locations.

#### 2. Schema Mapping Table

| Source System | Raw Attribute | Event Log Mapping | Transformation / Processing Rule |
| :--- | :--- | :--- | :--- |
| **GPS Tracker** | Timestamp | `Timestamp` | Standardize to UTC ISO 8601. |
| **GPS Tracker** | Coordinates | `Location (Lat/Lon)`| Spatial-temporal clustering to define "Stops" vs. "Transit". |
| **GPS Tracker** | Speed / Ignition | `Activity / Status` | Maps to `Ignition On`, `Ignition Off`, `Moving`, `Idle`. |
| **Handheld Scanner**| Scan Timestamp | `Timestamp` | Match to the nearest GPS coordinate to validate physical presence. |
| **Handheld Scanner**| Event Code | `Activity` | Map codes to standard activities: `Arrive Customer`, `Delivery Success`. |
| **Handheld Scanner**| Package ID | `Package_ID` | Map as a transactional attribute linked to the operational case. |
| **Dispatch System** | Planned Stop Sequence | `Planned_Sequence` | Inject as context/metadata at Case initialization (`Route Assigned`). |
| **Dispatch System** | Promised Window | `Time_Window` | Inject as metadata constraint to calculate conformance. |
| **Maintenance Log** | Repair Down Times | `Vehicle_Status` | Correlate with Case IDs using `Vehicle_ID` and active dates. |

#### 3. Data Quality Challenges & Mitigation Strategies
*   **GPS Drift and Noise:** When a vehicle is parked near tall buildings, GPS coordinates "drift," falsely indicating low-speed movement. 
    *   *Mitigation:* Apply a **Kalman filter** to smooth coordinate streams. Use spatial-temporal clustering (e.g., **ST-DBSCAN**) with a minimum duration threshold of 120 seconds and a maximum radius of 15 meters to declare a true "Unplanned Stop" or "Customer Stop," filtering out momentary traffic stops.
*   **Asynchronous Clocks across Devices:** Scanners, GPS units, and dispatch servers may experience clock drift, creating physically impossible sequences (e.g., `Delivery Success` registered before GPS `Arrive Customer`).
    *   *Mitigation:* Establish the GPS unit as the ground-truth clock for spatial transitions and the scanner for user-action sequences. Use **event correlation heuristics** to auto-align scanner events to the closest spatial stop window where $\Delta t < 5 \text{ minutes}$.
*   **Missing Package/Stop Links:** Drivers occasionally bypass scanner steps (e.g., delivering a package without scanning it at the door).
    *   *Mitigation:* Infer missing activities. If a package is marked `Scan at Depot` and later `Arrive Depot` (returned), but has no delivery attempt event, and GPS shows a stop near the customer address, classify the event sequence as `Inferred Attempt (Unscanned)` with a low-confidence flag for audit.

#### Algorithm Selection
We bypass the *Alpha Miner* due to its extreme sensitivity to noise and complex loops. Instead, we use a hybrid approach:
1.  **Heuristic Miner:** Ideal for discovering the overall flow because it accounts for event frequencies and dependency thresholds, filtering out rare, non-representative "noise" paths (e.g., a rare engine warning light event).
2.  **Inductive Miner (Infrequently used for visual maps, primarily for formal modeling):** Guarantees "soundness" (no deadlocks) and produces block-structured process trees that are highly effective for formal conformance checking.

#### Visualizing the End-to-End Journey
The discovered model displays a primary pipeline of expected behavior:

$$\text{Start Shift} \rightarrow \text{Route Assigned} \rightarrow \text{Depart Depot} \rightarrow [\text{Transit} \rightarrow \text{Arrive Customer} \rightarrow \text{Delivery Success/Failed} \rightarrow \text{Depart Customer}]^* \rightarrow \text{Arrive Depot} \rightarrow \text{End Shift}$$

However, the discovery tool will project **deviant paths** visually highlighted by thickness (frequency) and color (duration):
*   **Unplanned Maintenance Loops:** A transition from `Transit` directly to `Unscheduled Stop (Engine Warning)` followed by long idle states.
*   **Re-routing Loops:** Instances where a single location is visited multiple times within one Case ID, showing repetitive `Transit` paths between distant stops.

### Conformance Checking Methodology

Conformance checking maps actual execution paths against the nominal paths prescribed by the dispatch system.

```
       Planned Route Sequence: [Stop A] -> [Stop B] -> [Stop C] -> [Stop D]
                                  |          |          |          |
                                  v          x          v          v
       Actual Route Sequence:  [Stop A] -> [Stop C] -> [Stop B] -> [Stop D]
                                             \_________________/
                                              Sequence Deviation
```

#### Constructing the Normative Model
For every `Case ID`, the dispatch system contains a planned sequence of stops (e.g., $\text{Depot} \rightarrow S_1 \rightarrow S_2 \rightarrow \dots \rightarrow S_n \rightarrow \text{Depot}$). We dynamically generate a matching normative process model (a sequence transition graph) for each specific case.

#### Alignment-Based Conformance Checking
We use state-of-the-art **alignment algorithms**. An alignment maps events in the observed log to transitions in the normative model, identifying where they deviate. Cost metrics are assigned to deviations:
*   $\text{Cost}(\text{Skip Planned Stop}) = 10$
*   $\text{Cost}(\text{Unplanned Stop}) = 5$
*   $\text{Cost}(\text{Sequence Violation}) = 3$

#### Specific Deviations to Detect
1.  **Sequence Violations (Out-of-Order Deliveries):** The model planned $S_1 \rightarrow S_2 \rightarrow S_3$, but the driver executed $S_1 \rightarrow S_3 \rightarrow S_2$. This suggests the driver ignored the dispatch system's routing engine, likely due to local knowledge or perceived traffic issues.
2.  **Unplanned Stops / Geofence Deviations:** GPS records a stop ($\text{speed} = 0$, $\text{duration} > 5 \text{ mins}$) at a coordinate not matching any planned customer or fuel depot. This highlights unauthorized driver breaks or undocumented traffic hotspots.
3.  **Bypassed Stops:** A package planned for delivery is returned to the depot (`Delivery Failed` or no scan event) without a corresponding GPS-validated dwell event near the customer's coordinates. This indicates a "phantom delivery attempt" where the driver marked a package undeliverable without actually visiting the address.
4.  **Temporal Violations (SLA Deviations):** Comparing the actual timestamp of `Arrive Customer` against the dispatch target time window.

## 2. Performance Analysis and Bottleneck Identification

Operational cost control and customer satisfaction rely heavily on identifying where, when, and why time and fuel are lost.

### Key Performance Indicators (KPIs) Derived from the Event Log

#### 1. On-Time Delivery Rate (OTDR)
The percentage of successful deliveries completed within the committed customer time window.

$$\text{OTDR} = \left( \frac{\sum_{i=1}^{N} \mathbb{I}\left( \text{TS}_{\text{Actual\_Delivery}, i} \le \text{TW}_{\text{End}, i} \right) \times \mathbb{I}\left( \text{Status}_i = \text{Success} \right)}{\text{Total Planned Deliveries}} \right) \times 100\%$$

*Where:*
*   $\text{TS}_{\text{Actual\_Delivery}, i}$ is the timestamp of the `Delivery Success` event for package $i$.
*   $\text{TW}_{\text{End}, i}$ is the upper boundary of the committed customer time window.
*   $\mathbb{I}(\cdot)$ is the indicator function (equals 1 if true, 0 if false).

#### 2. Service Time vs. Travel Time Ratio (ST:TT)
Identifies whether vehicles are spending more time moving or serving customers. High travel ratios indicate inefficient routing.

$$\text{ST:TT} = \frac{\sum_{j=1}^{M} \left( T_{\text{Depart Customer}, j} - T_{\text{Arrive Customer}, j} \right)}{\sum_{j=1}^{M} \left( T_{\text{Arrive Customer}, j} - T_{\text{Depart Depot/Prior Customer}, j} \right)}$$

*Where:*
*   The numerator represents cumulative Service Time ($ST$) across all $M$ successful stops.
*   The denominator represents cumulative Travel Time ($TT$) between stops.

#### 3. Idle Fuel Consumption Metric (IFCM)
Estimates fuel wasted during unnecessary idle time.

$$\text{IFCM (Liters)} = \sum_{k=1}^{P} \left( T_{\text{Ignition On, Speed=0}, k} \times \text{Idle Consumption Rate}_v \right)$$

*Where:*
*   $T_{\text{Ignition On, Speed=0}, k$ is the duration of idle state $k$ where speed is zero but ignition is active (excluding stoplights, based on duration > 3 minutes).
*   $\text{Idle Consumption Rate}_v$ is the fuel use per minute for vehicle type $v$ (typically $0.02 - 0.04 \text{ liters/min}$ for light commercial vehicles).

#### 4. Vehicle Capacity Utilization (VCU)
Measures physical volume/weight efficiency.

$$\text{VCU} = \frac{\sum \text{Volume of Loaded Packages at Departure}}{\text{Total Usable Cargo Volume of Vehicle}} \times 100\%$$

#### 5. First-Time-Right (FTR) Delivery Rate
Measures delivery accuracy, directly correlating with fuel consumption and customer satisfaction.

$$\text{FTR} = \frac{\sum \text{Delivery Success Events}}{\sum \text{Delivery Success Events} + \sum \text{Delivery Failed Events}} \times 100\%$$

2.  **Social Network Analysis (SNA) for Depot Handover Bottlenecks:**
    By mapping interactions between the "Dispatch Resource" (Depot operators) and "Driver Resource," we analyze the duration of the `Route Assigned` $\rightarrow$ `Depart Depot` transition.     We isolate the `Arrive Customer` $\rightarrow$ `Delivery Success/Failure` sub-process. High dwell times (e.g., >10 minutes) with zero speed indicate issues like:
    *   High-rise building deliveries without adequate parking access.

4.  **Multi-Dimensional Filtering:**
    We filter the event log across several parameters:
    *   *Driver Experience:* Veteran vs. newly hired drivers.
    *   *Vehicle Class:* Large box vans vs. small electric vans.
    *   *Time-of-Day:* Morning peak (08:00–10:00) vs. midday off-peak.
    *   *Spatial Hotspots:* Intersections and corridors with repeated `Low Speed Detected` events.

### Root Cause Scenarios & Verification Methods

#### Scenario A: Suboptimal Route Planning (Static vs. Dynamic)
*   **Hypothesis:** The dispatch system plans static routes based on nominal geographic distance, ignoring real-time traffic and time-of-day dynamics, leading to severe delays during peak hours.
*   **Verification via Process Mining:** 
    *   *Variant Analysis:* Group cases into two cohorts: Cohort A (Routes completed within SLA) and Cohort B (Routes with >20% SLA violations).
    *   *Insight:* Variant mapping reveals that Cohort B frequently shows sequence changes where drivers manually override the planned order to avoid known traffic bottlenecks. However, this manual rescheduling often increases total travel distance, proving that static route planning forces drivers into inefficient ad-hoc routing decisions.

#### Scenario B: Inaccurate Service/Dwell Time Estimates
*   **Hypothesis:** The dispatch system allocates a flat 3-minute delivery window for every stop, regardless of whether it is a suburban house or a 15th-floor downtown office, leading to cumulative delays.
*   **Verification via Process Mining:**
    *   *Correlation Analysis:* Plot the actual duration of the activity `Arrive Customer` $\rightarrow$ `Depart Customer` against the customer type and geographic location.
    *   *Insight:* The analysis shows that high-density commercial zones have an average dwell time of 14.5 minutes (with high variance), while suburban residential zones average 2.4 minutes. Applying a flat 3-minute planning assumption guarantees that urban routes fall behind schedule by mid-morning.

#### Scenario C: Driver Behavior and Inefficient Practices
*   **Hypothesis:** Significant variations in delivery efficiency are driven by individual driver habits (e.g., poor load organizing, frequent off-route stops) rather than external traffic conditions.
*   **Verification via Process Mining:**
    *   *Resource Profiling:* Filter the event log by `Driver_ID` while controlling for identical routes and vehicle types.
    *   *Insight:* Under identical external conditions, some drivers consistently exhibit 30% higher transition times from `Arrive Customer` to `Delivery Success`. This indicates time lost searching for packages inside the vehicle. Additionally, their logs show more frequent, brief `Unplanned Stops` near convenience stores.

#### Scenario D: Failed Deliveries and the Re-delivery Loop
*   **Hypothesis:** First-time delivery failures generate high-cost process loops, forcing vehicles to return to the same neighborhoods on subsequent days, compounding fuel and labor costs.
*   **Verification via Process Mining:**
    *   *Loop/Rework Analysis:* Measure the frequency of transition sequences containing: 

$$[\text{Delivery Failed}] \rightarrow [\text{Return to Depot}] \rightarrow [\text{Redeliver Day+1}]$$

*   *Insight:* The log reveals that failed deliveries account for 18% of all stop attempts on suburban routes. Analyzing these failures indicates they are highly concentrated between 13:00 and 16:00 (when customers are typically at work). Each failed attempt costs approximately \$6.50 in direct operational overhead, and the subsequent re-delivery attempt adds duplicate mileage to the route.

#### Scenario E: Unscheduled Maintenance & Mid-Route Breakdowns
*   **Hypothesis:** Delayed preventive maintenance leads to mid-route breakdowns, which disrupt schedules and trigger high tow and recovery costs.
*   **Verification via Process Mining:**
    *   *Cross-Log Correlation:* Join the process mining event log with vehicle maintenance logs.
    *   *Insight:* Vehicles with a history of deferred fluid or brake service show a high frequency of mid-route `Unscheduled Stop` events followed by a `Route Aborted` event. The average recovery cost and SLA penalty for a mid-route breakdown is **8.2 times higher** than the cost of a scheduled, off-shift maintenance intervention.

This decision tree reveals that:

2.  **Depot Departure Delay:** If the transition `Route Assigned` $\rightarrow$ `Depart Depot` exceeds 15 minutes, the probability of late delivery for subsequent PM-window packages increases to **81%** due to traffic congestion later in the shift.

```

                                                               |
```

3.  **Implement Real-Time Route Adjustments:** Integrate historical travel times between zones into the dispatch planning engine. This ensures routes are sequenced to avoid known bottleneck areas during peak congestion hours.

#### Expected Impact on KPIs
*   **ST:TT Ratio:** Improve by **20–25%** as transit times decrease.
*   **On-Time Delivery Rate (OTDR):** Increase from an average of 84% to **over 93%**.

Process optimization must prioritize driver safety and regulatory compliance (e.g., maximum driving times and mandatory break periods).
*   *Process Constraints:* Define compliance rules in the alignment checking model. For example, if driving time exceeds 4.5 hours without a 45-minute break activity, flag the case as a **critical regulatory compliance exception**.
*   *Optimization Balancing:* Ensure the routing engine limits assigned stop volumes to fit within standard 8-hour shifts. This reduces the need for expensive overtime while ensuring drivers can complete all deliveries safely and within legal limits.

#### 2. Physical Vehicle Capacity Constraints
To avoid overloading vans, we link the dispatch planning system directly to vehicle payload limits.
*   *Process Constraints:* When allocating packages to cases, the system cross-references package volume and weight against the vehicle’s maximum load limits:
    
$$\sum \text{Volume}_{\text{Packages}} \le 85\% \times \text{Volume}_{\text{Max Vehicle Capacity}}$$

    Maintaining a 15% safety buffer ensures drivers have enough room to safely access and organize packages inside the vehicle.

#### 3. Process Drift Detection and Feedback Loops
*   **Detecting Process Drift:** Use **Concept Drift Detection** to identify when system behaviors change over time. For example, if average service times in a specific sector increase from 4 minutes to 9 minutes over a two-week period, the system flags a "Process Drift Alert."
*   **Closing the Feedback Loop:** Once drift is detected, the system analyzes the root cause (such as new road construction or seasonal volume spikes) and automatically updates the routing engine. This updates travel and service time assumptions to match the new reality on the ground.
*   **Sustaining Operational Discipline:** By continuously measuring actual performance against planned routes, fleet managers can identify when drivers consistently deviate from optimized paths. This data can be used to provide targeted coaching and reinforce best practices, ensuring long-term operational efficiency.
