## 3. Verification Queries Using the Underlying Database

The following SQL queries use `claim_events` as the primary log, joined with `claims` for context. They are designed to find concrete evidence for the hypotheses.

### Query 1: Identify Claims with Premature or Skipped Closure (Direct Test for Anomaly 3)
**Logic:** Find claims where `C` (Close) occurs without a preceding `P` (Approve) or `N` (Notify). This reveals the effect of the erroneous `A  C` edge and the missing `xor  C` order.

```sql
SELECT 
    c.claim_id,
    c.claim_type,
    c.submission_date,
    ce_c.timestamp as close_time,
    ce_a.timestamp as assign_time,
    -- Check for the presence of key events after close
    MAX(CASE WHEN ce.activity = 'P' THEN 1 ELSE 0 END) as has_approval,
    MAX(CASE WHEN ce.activity = 'N' THEN 1 ELSE 0 END) as has_notification,
    MAX(CASE WHEN ce.activity = 'E' THEN 1 ELSE 0 END) as has_evaluation
FROM claims c
JOIN claim_events ce ON c.claim_id = ce.claim_id
JOIN claim_events ce_c ON c.claim_id = ce_c.claim_id AND ce_c.activity = 'C'
JOIN claim_events ce_a ON c.claim_id = ce_a.claim_id AND ce_a.activity = 'A'
WHERE ce.timestamp <= ce_c.timestamp -- Events up to and including close
GROUP BY c.claim_id, c.claim_type, c.submission_date, ce_c.timestamp, ce_a.timestamp
HAVING 
    -- Close event is missing prior 'P' (Approve)
    MAX(CASE WHEN ce.activity = 'P' AND ce.timestamp < ce_c.timestamp THEN 1 ELSE 0 END) = 0
    OR
    -- Close event is missing prior 'N' (Notify) if 'P' exists before close
    (MAX(CASE WHEN ce.activity = 'P' AND ce.timestamp < ce_c.timestamp THEN 1 ELSE 0 END) = 1
     AND MAX(CASE WHEN ce.activity = 'N' AND ce.timestamp < ce_c.timestamp THEN 1 ELSE 0 END) = 0)
ORDER BY c.submission_date DESC;
```

### Query 2: Identify Potential Abuse of the Evaluate-Approve Loop (Direct Test for Anomaly 1)
**Logic:** Count the number of `E` and `P` events per claim. Claims with more than 2 of either (or an imbalance where P happens without a preceding E) indicate an anomalous looping pattern.

```sql
WITH event_counts AS (
    SELECT 
        claim_id,
        SUM(CASE WHEN activity = 'E' THEN 1 ELSE 0 END) as eval_count,
        SUM(CASE WHEN activity = 'P' THEN 1 ELSE 0 END) as approve_count
    FROM claim_events
    WHERE activity IN ('E', 'P')
    GROUP BY claim_id
)
SELECT 
    claim_id,
    eval_count,
    approve_count,
    -- A proper flow should have 1 E, 1 P. A loop might have >1.
    -- But also check for P without E (imbalance).
    CASE 
        WHEN eval_count != 1 OR approve_count != 1 THEN 'Anomalous Loop Pattern'
        ELSE 'Normal Pattern'
    END as status
FROM event_counts
WHERE eval_count != 1 OR approve_count != 1
ORDER BY eval_count DESC, approve_count DESC;
```

### Query 3: Frequency of Skipped Notification (Direct Test for Anomaly 2)
**Logic:** Check what fraction of claims that reach approval (`P`) are never followed by a notification (`N`).

```sql
WITH approved_claims AS (
    SELECT DISTINCT claim_id, MAX(timestamp) as approve_time
    FROM claim_events WHERE activity = 'P'
    GROUP BY claim_id
),
notified_claims AS (
    SELECT DISTINCT claim_id
    FROM claim_events WHERE activity = 'N'
)
SELECT 
    COUNT(DISTINCT ac.claim_id) as total_approved_claims,
    COUNT(DISTINCT nc.claim_id) as notified_claims,
    COUNT(DISTINCT ac.claim_id) - COUNT(DISTINCT nc.claim_id) as skipped_notifications,
    ROUND(100.0 * (COUNT(DISTINCT ac.claim_id) - COUNT(DISTINCT nc.claim_id)) / COUNT(DISTINCT ac.claim_id), 2) as pct_skipped
FROM approved_claims ac
LEFT JOIN notified_claims nc ON ac.claim_id = nc.claim_id;
```
