# **Comprehensive Process Mining Approach for Optimizing TechSolve’s Resource Assignment Practices**

This response outlines a structured, data-driven methodology to analyze and improve TechSolve’s IT service desk resource assignment processes using process mining. The approach leverages event log data to diagnose inefficiencies, identify root causes, and propose actionable optimization strategies.

---

## **1. Analyzing Resource Behavior and Assignment Patterns**

### **1.1 Key Metrics for Agent and Tier Performance**
To assess resource behavior, we analyze the following metrics from the event log:

| **Metric**                          | **Description**                                                                 | **Process Mining Technique**                          |
|-------------------------------------|---------------------------------------------------------------------------------|-------------------------------------------------------|
| **Workload Distribution**           | Number of tickets handled per agent/tier, normalized by working hours.          | Resource utilization analysis (frequency, duration).   |
| **Activity Processing Time**        | Time taken per activity (e.g., "Work L1 Start" to "Work L1 End") per agent/tier. | Performance spectrum, box plots, flow time analysis.   |
| **First-Call Resolution (FCR) Rate** | % of tickets resolved by L1 without escalation.                                | Variant analysis (filtering for non-escalated cases). |
| **Skill Utilization**               | Frequency of tickets requiring specific skills handled by agents with those skills. | Skill-to-agent mapping, heatmaps.                     |
| **Escalation Rate**                 | % of tickets escalated from L1  L2, L2  L3.                                   | Path analysis, handover frequency.                    |
| **Reassignment Rate**               | % of tickets reassigned (e.g., due to skill mismatch).                          | Social network analysis (handover patterns).          |
| **SLA Compliance**                  | % of tickets resolved within SLA targets (P1-P4).                               | Performance mining, SLA breach analysis.              |
| **Idle Time**                       | Time agents spend waiting for new tickets (queue time).                         | Resource idle time analysis.                          |

### **1.2 Process Mining Techniques for Assignment Patterns**
#### **A. Resource Interaction & Social Network Analysis (SNA)**
- **Objective:** Visualize *actual* assignment flows, escalations, and reassignments.
- **Method:**
  - **Handover of Work (SNA):** Construct a social network where nodes = agents/tiers, edges = frequency of ticket handovers. Highlights:
    - Overloaded agents (high in-degree).
    - Agents frequently receiving escalated tickets (L2/L3 bottlenecks).
    - Unnecessary reassignments (e.g., L1  L2  L1).
  - **Working Together Metrics:** Identify agents who frequently collaborate (e.g., L1 agents escalating to the same L2 specialist).
- **Comparison to Intended Logic:**
  - The current round-robin assignment should distribute tickets evenly, but SNA may reveal:
    - Certain agents consistently overloaded (e.g., Agent B12 in the snippet).
    - Escalations bypassing intended tiers (e.g., L1  L3 directly).

#### **B. Role Discovery**
- **Objective:** Identify *de facto* roles based on actual behavior (vs. documented tiers).
- **Method:**
  - Cluster agents based on:
    - Types of tickets handled (category, required skill).
    - Processing times.
    - Escalation/reassignment patterns.
  - Example findings:
    - Some L1 agents handle P2 tickets effectively (potential for upskilling).
    - L2 agents frequently resolve "L1-level" tickets (skill underutilization).

#### **C. Skill Utilization Analysis**
- **Objective:** Assess whether specialized skills are used efficiently.
- **Method:**
  - **Skill-Ticket Matching:** For each ticket, compare `Required Skill` vs. `Agent Skills`.
  - **Heatmaps:** Show frequency of skill usage across agents/tiers.
  - **Over/Under-Qualification Metrics:**
    - % of tickets assigned to agents with *higher* skills than required (e.g., L3 agent handling a P3 "Basic-Troubleshoot" ticket).
    - % of tickets assigned to agents *lacking* the required skill (leading to reassignments).

## **2. Identifying Resource-Related Bottlenecks and Issues**

### **2.1 Pinpointing Specific Problems**
Using the metrics and techniques above, we identify:

| **Issue**                          | **Process Mining Evidence**                                                                 | **Quantifiable Impact**                                                                 |
|------------------------------------|--------------------------------------------------------------------------------------------|----------------------------------------------------------------------------------------|
| **Skill Mismatches**               | High reassignment rate (e.g., 30% of tickets reassigned due to wrong skill).               | +20% resolution time per reassignment; 15% of SLA breaches linked to skill mismatches. |
| **L1 Escalation Overload**         | 60% of P3 tickets escalated to L2 (vs. target 30%).                                         | L2 agents spend 40% of time on L1-level tickets.                                       |
| **Specialist Underutilization**    | L3 agents handle 20% "Basic-Troubleshoot" tickets.                                          | 25% of L3 capacity wasted on low-complexity work.                                      |
| **Uneven Workload Distribution**   | Top 10% of agents handle 40% of tickets; bottom 10% handle 2%.                              | Overloaded agents have 30% longer resolution times.                                    |
| **Dispatcher Bottlenecks**         | "Assign L2" activities take 15+ minutes on average (manual routing).                       | 10% of SLA breaches due to assignment delays.                                          |
| **Incorrect Initial Assignments**  | 25% of tickets reassigned within 1 hour of L1 assignment.                                  | +15% resolution time for reassigned tickets.                                           |

### **2.2 Root Cause Hypotheses**
- **Skill Mismatches:** Dispatchers lack real-time visibility into agent skills/workload.
- **High Escalations:** L1 agents lack training or confidence to resolve P2/P3 tickets.
- **Specialist Underutilization:** Round-robin assignment ignores skill levels.
- **Uneven Workload:** No dynamic load balancing; agents cherry-pick "easy" tickets.

## **3. Root Cause Analysis for Assignment Inefficiencies**

### **3.1 Deficiencies in Current Assignment Logic**
- **Round-Robin Limitations:**
  - Ignores agent skills, workload, and ticket complexity.
  - Example: A P2 "Networking-Firewall" ticket assigned to an L1 agent with only "Basic-Troubleshoot" skills (as in INC-1002).
- **Manual Escalation Decisions:**
  - L1 agents escalate based on gut feeling, not data-driven criteria.
  - No standardized escalation rules (e.g., "Escalate if resolution time > 30 mins for P2").
- **Static Skill Profiles:**
  - Agent skills are documented but not dynamically updated (e.g., new certifications).
  - No proficiency levels (e.g., "App-CRM: Beginner/Intermediate/Expert").

### **3.2 Variant Analysis for Poor Assignments**
- **Method:**
  - Compare "smooth" tickets (resolved in 1 assignment) vs. "problematic" tickets (multiple reassignments).
  - **Decision Mining:** Use decision trees to identify factors leading to reassignments (e.g., ticket category, agent skill mismatch, time of day).
- **Example Findings:**
  - **Ticket Category:** "Software-App" tickets are 2x more likely to be reassigned than "Hardware."
  - **Agent Tier:** Tickets assigned to L1 agents with <1 year experience have 40% higher escalation rates.
  - **Time of Day:** Assignments during shift changes have 25% more reassignments.

### **3.3 Key Root Causes**
1. **Lack of Skill-Based Routing:** Assignments ignore required skills.
2. **Poor Initial Ticket Categorization:** Misclassified tickets lead to wrong skill requirements.
3. **No Workload Awareness:** Agents are assigned tickets even when overloaded.
4. **Insufficient L1 Empowerment:** L1 agents escalate too quickly due to lack of training or incentives.
5. **Static Assignment Rules:** No adaptation to real-time demand (e.g., spikes in "Network" tickets).

## **4. Data-Driven Resource Assignment Strategies**

### **Strategy 1: Skill-Based Routing with Proficiency Levels**
**Issue Addressed:** Skill mismatches leading to reassignments and SLA breaches.

**Implementation:**
- **Data Required:**
  - Agent skill profiles (with proficiency levels: Beginner/Intermediate/Expert).
  - Ticket required skills (from categorization or NLP on descriptions).
  - Historical resolution success rates (e.g., "Agent X resolves 'App-CRM' tickets in 20 mins on average").
- **Algorithm:**
  - For each ticket, match `Required Skill` to agents with the skill, prioritizing:
    1. **Proficiency Level:** Expert > Intermediate > Beginner.
    2. **Current Workload:** Agents with <3 active tickets.
    3. **Historical Performance:** Agents with fastest resolution times for the skill.
  - Example: A P2 "Networking-Firewall" ticket is assigned to an L2 agent with "Expert" proficiency and <2 active tickets.
- **Expected Benefits:**
  - **30% reduction in reassignments** (fewer skill mismatches).
  - **15% faster resolution times** (agents better suited to tasks).
  - **20% improvement in SLA compliance** for P2/P3 tickets.

**Process Mining Insight:**
- Heatmaps of skill utilization show that 40% of "App-CRM" tickets are handled by agents with "Beginner" proficiency, leading to longer resolution times.

---

### **Strategy 2: Workload-Aware Dynamic Assignment**
**Issue Addressed:** Uneven workload distribution and agent overload.

**Implementation:**
- **Data Required:**
  - Real-time agent workload (number of active tickets, estimated completion time).
  - Ticket priority and SLA targets.
  - Agent availability (shift schedules, breaks).
- **Algorithm:**
  - **Load Balancing:** Assign tickets to the least busy agent with the required skills.
  - **Priority Queuing:** P1 tickets jump the queue; P4 tickets are deferred if agents are overloaded.
  - **Dynamic Thresholds:** Adjust assignment rules based on time of day (e.g., fewer assignments during shift changes).
  - Example: If Agent A05 has 5 active tickets and Agent A06 has 1, assign the next ticket to A06.
- **Expected Benefits:**
  - **25% reduction in agent overload** (fewer tickets per agent).
  - **10% faster resolution times** (less multitasking).
  - **5% improvement in SLA compliance** (prioritization of high-priority tickets).

**Process Mining Insight:**
- Social network analysis shows that 10% of agents handle 50% of tickets, while 30% are underutilized.

---

### **Strategy 3: Predictive Assignment with NLP and Machine Learning**
**Issue Addressed:** Incorrect initial assignments and escalations due to poor ticket categorization.

**Implementation:**
- **Data Required:**
  - Historical ticket descriptions and resolutions.
  - Agent performance data (resolution time, success rate).
  - Ticket metadata (category, priority, channel).
- **Algorithm:**
  - **NLP for Skill Prediction:** Use text mining on ticket descriptions to predict required skills (e.g., "CRM login error"  "App-CRM").
  - **Complexity Scoring:** Classify tickets as "L1/L2/L3" based on historical resolution patterns.
  - **Predictive Routing:** Assign tickets to the agent/tier most likely to resolve them quickly.
  - Example: A ticket with description "Firewall blocking VPN" is predicted to require "Networking-Firewall" and assigned to an L2 agent.
- **Expected Benefits:**
  - **40% reduction in escalations** (better initial assignments).
  - **20% faster resolution times** (agents receive tickets they can resolve).
  - **10% improvement in FCR rate** (L1 agents handle more tickets successfully).

**Process Mining Insight:**
- Variant analysis shows that 30% of escalations occur because the initial assignment was to the wrong tier.

## **5. Simulation, Implementation, and Monitoring**

### **5.1 Pre-Implementation Simulation**
- **Objective:** Evaluate the impact of proposed strategies before deployment.
- **Method:**
  - **Discrete-Event Simulation (DES):**
    - Model the current process using the event log (arrival rates, processing times, resource pools).
    - Simulate the proposed strategies (e.g., skill-based routing, workload-aware assignment).
    - Compare KPIs (resolution time, SLA compliance, reassignments) between current and simulated states.
  - **What-If Scenarios:**
    - What if we reduce L1 escalations by 20%?
    - What if we reallocate 10% of L3 agents to L2?
  - **Tools:** Celonis, ProM, or custom Python (SimPy, PM4Py).

## **Conclusion**
TechSolve’s resource assignment inefficiencies stem from static, skill-agnostic routing and lack of real-time workload awareness. By leveraging process mining to analyze agent behavior, skill utilization, and assignment patterns, we can implement **skill-based routing**, **workload-aware assignment**, and **predictive routing** to reduce reassignments, improve SLA compliance, and optimize agent utilization.
