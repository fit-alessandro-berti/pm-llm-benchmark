# Optimizing Resource Assignment at TechSolve Solutions: A Process Mining Approach

## Preliminary Note: Event Log Preparation

Before analysis, I would ensure the log is properly structured for resource-oriented process mining. Key preparation steps:

- **Compute activity durations** by pairing START/COMPLETE timestamp types per Case ID + Activity (e.g., `Work L1 End  Work L1 Start` = handling time; the gap between `Assign L2` and `Work L2 Start` = queue/waiting time).
- **Enrich with derived attributes**: skill-match flag (does `Agent Skills` contain `Required Skill`?), tier-of-resolution, number of handovers per case, channel, and SLA target per priority (e.g., P1=4h, P2=8h, P3=24h, P4=48h).
- **Build a resource dimension table** linking Agent ID  Tier  documented Skills, and a separate "actual skills demonstrated" view derived from tickets they successfully resolved.

This separation of **waiting time vs. processing time** is critical — much of TechSolve's SLA pain (e.g., INC-1001 waited ~25 min in the L2 queue) is in waiting, not work.

## 1. Analyzing Resource Behavior and Assignment Patterns

### 1.1 Agent- and Tier-Level Performance Metrics

I would build a **resource performance profile** computed from the log, aggregated by agent and by tier:

| Metric | How it's computed | What it reveals |
|---|---|---|
| **Workload distribution** | Count of cases/activities per agent per period; active-handling hours | Over/under-utilization (validates the "uneven workload" suspicion) |
| **Activity processing time** | Median/p90 of (Work End  Work Start) per agent, per ticket category | Speed differences; outliers; skill-fit effects |
| **First-Contact Resolution (FCR) rate (L1)** | % of L1-touched cases closed without `Escalate` | L1 effectiveness; over-escalation |
| **Escalation rate & target** | % escalated, and to which tier/skill | Whether L1 is bypassed or over-relied upon |
| **Reassignment frequency** | Count of `Reassign` per case, per agent | Identifies agents/skill areas generating churn |
| **Rework / ping-pong** | Cases returning to a previously visiting resource/tier | Misrouting |
| **Skill-match rate** | % of assignments where required skill  agent skills | Core assignment-quality metric |
| **SLA compliance per agent/tier** | % of cases meeting target | Links behavior to outcomes |

I'd compute these **per priority and category** because a P2-Network profile behaves very differently from a P4-Access ticket.

### 1.2 Discovering Actual Assignment Patterns (vs. Intended)

The intended logic is "round-robin within tier + manual escalation." Process mining lets me reveal what *actually* happens:

- **Social Network Analysis (SNA) — Handover-of-Work network**: Build a graph where nodes = agents/tiers and edges = ticket handovers (e.g., A05  B12  B15 for INC-1001). Edge weights = frequency and average delay. This exposes:
  - **Handover hotspots** (specific L1L2 paths that are heavy).
  - **Ping-pong subgraphs** (cyclic handovers indicating misrouting).
  - **De-facto specialization** — e.g., a particular L2 agent receiving nearly all CRM tickets regardless of round-robin intent.
- **Role/Organizational Mining**: Cluster agents by the activities and ticket types they actually perform. If discovered roles diverge from documented tiers (e.g., an "L1" agent frequently resolving App-CRM issues that "should" be L2), the formal structure is misaligned with reality.
- **Resource interaction / co-occurrence analysis**: Which agents/skills consistently appear together on the same cases — reveals dependency chains.
- **Conformance checking**: Model the *intended* routing (round-robin, escalate-by-skill) as a reference, then replay the log. Deviations (self-assignments like INC-1002's A02, skipped dispatch steps, escalations to wrong-skill agents) are quantified as **fitness gaps**.

**Comparison output**: A side-by-side "intended vs. discovered" map showing, e.g., "Round-robin assumed; in reality 60% of CRM tickets funnel to 2 agents, creating a queue bottleneck."

### 1.3 Skill Utilization Analysis

I would cross-tabulate **agent skill level vs. task complexity/required skill**:

- **Skill-down assignments**: Frequency of high-tier specialists (L2/L3) working tickets whose `Required Skill` = `Basic-Troubleshoot` or whose priority is low (P4). This directly tests the "specialists doing L1 work" complaint.
- **Skill-up / mismatch assignments**: Cases where required skill  assigned agent's skills  forces a reassignment (INC-1001: B12 lacked DB-SQL  reassigned to B15). Quantify reassignment-due-to-skill-gap.
- **Skill scarcity heatmap**: For each required skill, compare **demand** (ticket volume + arrival times) vs. **supply** (agents possessing it, their availability). Skills with high demand-to-supply ratios and long queue times are scarcity bottlenecks (e.g., Database-SQL, Networking-Firewall).
- **Skill ROI view**: % of a specialist's time spent on tasks matching their highest skill vs. below it.

## 2. Identifying Resource-Related Bottlenecks and Issues

Using the above, I'd pinpoint and **quantify** specific problems:

**a) Skill-availability bottlenecks.** Measure **queue waiting time per required skill** (gap between `Assign L2` and `Work L2 Start`). In INC-1001 this was ~25 minutes ("Delay due to queue"). Aggregated: *"P2-Network tickets wait on average X hours for a Firewall-skilled L2; this skill is held by only 3 agents and is the single largest contributor to P2 SLA breaches."*

**b) Reassignment/escalation delays.** Compute **average added cycle time per reassignment**. INC-1001's reassignment (B12B15) injected delay because work restarted in a new queue. Metric: *"Cases with 1 reassignment have median cycle time 2.4× higher; reassignments add an average of Y hours each; Z% of all reassignments are skill-driven (avoidable with better initial routing)."*

**c) Incorrect initial assignment.** Correlate first-assignment skill-match flag with downstream reassignment/escalation. *"When the dispatcher's first L2 assignment matches required skill, reassignment rate is 5%; when it doesn't, 70%."* This isolates dispatcher/L1 routing quality as a cause.

**d) Over-/under-loaded agents.** From workload distribution + utilization, flag agents above (say) p90 of concurrent active tickets vs. those idle. Cross-reference with processing-time degradation (overloaded agents slow down  SLA risk).

**e) SLA breach correlation.** Build a breach-attribution analysis: for each breached case, decompose total cycle time into waiting (queue) vs. processing vs. reassignment overhead, and tag the dominant factor. Output: *"Of P2/P3 breaches, A% driven by skill-queue waiting, B% by reassignment churn, C% by L1 over-escalation, D% by overloaded specialists."* This converts vague complaints into a ranked, quantified problem list.

**f) Unnecessary escalations.** Identify ticket types that L1 *sometimes* resolves but *usually* escalates — a sign of inconsistent L1 capability/empowerment rather than genuine complexity.

## 3. Root Cause Analysis for Assignment Inefficiencies

For each quantified issue, I'd trace candidate root causes:

| Symptom | Likely root cause | How to confirm with data |
|---|---|---|
| High reassignment from correct tier but wrong agent | Round-robin ignores **skills & workload** | Reassignments occur even when a skill-matched agent was idle (compare against availability at assignment time) |
| Skill-driven reassignments (B12B15) | **Inaccurate/incomplete skill profiles** OR poor matching at dispatch | High mismatch rate despite documented skills  routing logic ignores skills |
| Many escalations from L1 | **Insufficient L1 training/empowerment** | Same simple ticket type resolved by some L1s, escalated by others (variant analysis by agent) |
| Long queues for specific skills | **Capacity/skill scarcity** + no load balancing | Demand-supply heatmap; queue forms only for scarce skills |
| Wrong initial routing | **Poor ticket categorization / required-skill identification** at creation | Cases where `Required Skill` changes mid-flow (INC-1001 App-CRMDatabase-SQL) |
| No rebalancing | **Lack of real-time workload visibility** for the dispatcher | Assignments to busy agents while idle skilled agents exist |

### Analytical techniques to confirm causes

- **Variant analysis**: Partition cases into "happy path" (created  L1 resolve OR clean single escalation  resolve, no reassignment) vs. "problematic" (multiple reassignments/escalations). Then compare attribute distributions (priority, category, channel, initial agent, first skill-match flag) across variants. Features that differ sharply are causal candidates — e.g., "problematic variants are 80% web-portal-created with auto-categorization, vs. phone-created cases categorized by an agent."
- **Decision mining**: At each decision point (e.g., "Escalate L2 vs. resolve at L1", "which L2 agent"), train an interpretable model (decision tree) on case/resource attributes to learn the *implicit* decision rule actually used, and where it produces poor outcomes. This reveals, e.g., "escalation decisions are driven by individual L1 agent identity, not ticket complexity" — pointing to inconsistent empowerment.
- **Root-cause correlation on the categorization itself**: If `Required Skill` frequently changes mid-case, the upstream categorization (especially auto-categorized web tickets) is a root cause feeding every downstream problem.

## 4. Data-Driven Resource Assignment Strategies

I propose four concrete strategies (3 required).

### Strategy 1 — Skill-Based Routing Weighted by Proficiency

- **Issue addressed**: Skill mismatches and skill-driven reassignments (INC-1001 type).
- **Leverages mining insights**: The skill-match analysis and decision mining showed round-robin ignores skills. Replace it with routing that requires `Required Skill  agent skills`, ranked by a **proficiency score** derived from the log itself — each agent's historical FCR rate and processing speed *per skill* (an evidence-based skill profile, correcting inaccurate documented profiles).
- **Data required**: Validated required-skill per ticket; mined per-agent-per-skill proficiency (from historical resolution outcomes); agent availability.
- **Expected benefits**: Sharp drop in skill-driven reassignments (the analysis quantified these as avoidable); faster processing as best-fit agents handle tickets; better FCR.

### Strategy 2 — Workload-Aware (Load-Balancing) Assignment

- **Issue addressed**: Uneven workload; overloaded agents slowing down; queues forming despite idle capacity.
- **Leverages mining insights**: Workload distribution and queue-waiting metrics identified overloaded vs. idle agents and which skill queues build up. Among skill-eligible agents, assign to the one with the **lowest current active load / shortest projected queue**, using mined processing-time distributions to estimate completion.
- **Data required**: Real-time count of in-progress tickets per agent; mined service-time distributions per skill/priority; agent shift/availability calendar.
- **Expected benefits**: Reduced p90 waiting time, smoother utilization, fewer SLA breaches attributable to queueing.

### Strategy 3 — Predictive / Triage-Based Routing at Intake

- **Issue addressed**: Poor initial categorization and required-skill identification (the upstream root cause; mid-case skill changes), and unnecessary L1 escalations.
- **Leverages mining insights**: Variant/decision mining linked misrouting to weak (especially auto-) categorization. Train a model on historical tickets (category, channel, description keywords  actual required skill, complexity, eventual resolving tier) to **predict required skill and likely resolution tier at creation**. Tickets predicted to need specialist skills bypass L1 (skip-level routing); tickets predicted L1-resolvable go to L1 with confidence.
- **Data required**: Historical ticket text/attributes + ground-truth resolving skill/tier; NLP features; model retraining pipeline.
- **Expected benefits**: Fewer mid-flow skill corrections, less ping-pong, reduced unnecessary escalation hops (cuts cycle time for cases that currently waste an L1 stage).

### Strategy 4 — Data-Refined Escalation Criteria & L1 Empowerment

- **Issue addressed**: Excessive/inconsistent L1 escalations; specialists doing low-level work.
- **Leverages mining insights**: Analysis identified ticket types that some L1s resolve but others escalate, and specialists' skill-down time. Define **evidence-based escalation rules** (escalate only when ticket type historically has <X% L1 FCR), and conversely **reserve specialists for high-complexity tickets** by routing simple ones away from them. Pair with targeted L1 training on the high-volume "wrongly escalated" categories.
- **Data required**: Per-category L1 FCR rates; per-agent skill-vs-task-level history; training records.
- **Expected benefits**: Higher FCR, freed specialist capacity for genuinely complex work (relieving scarce-skill bottlenecks from Strategy 1), lower escalation volume.

> **Synergy**: 3 routes correctly at intake  1 picks the right-skilled agent  2 balances load among eligible agents  4 ensures the right *tier* and protects scarce specialists. Together they attack waiting time, reassignment churn, and skill misuse simultaneously.

## 5. Simulation, Implementation, and Monitoring

### 5.1 Pre-Implementation Validation via Simulation

I would build a **data-driven Discrete-Event Simulation (DES)** (or use a process-mining tool's simulation module) calibrated entirely from the mined model:

- **Inputs from the log**: ticket arrival patterns (by priority/category/channel, with time-of-day/day-of-week seasonality), mined activity-duration distributions per skill, the discovered process map, resource pools with mined skill/proficiency profiles, and current routing rules.
- **Baseline validation**: Run the as-is simulation and confirm it reproduces observed KPIs (cycle time, SLA rate, reassignment frequency) within tolerance — this proves the model is trustworthy.
- **Scenario testing**: Encode each proposed strategy as a routing policy and run **what-if scenarios**: skill-based routing alone, +load-balancing, +predictive triage, and combinations. Also stress-test capacity questions — "how many additional Firewall/SQL-skilled agents eliminate the P2 queue bottleneck?" and "what if demand grows 20%?"
- **Outputs compared**: predicted SLA compliance, mean/p90 cycle time, reassignment rate, specialist utilization, and waiting-time reduction per scenario. This quantifies expected ROI and de-risks rollout before touching production. Recommend phased rollout (e.g., pilot skill-based routing on one category first).

### 5.2 Post-Implementation Monitoring (Process Mining Dashboards)

Establish **continuous, near-real-time process mining dashboards** (re-ingesting the live event log) to verify benefits and catch regressions:

**Core resource KPIs to track:**
- Skill-match rate at first assignment (target  toward ~95%)
- Reassignment frequency per case & % skill-driven (target )
- L1 FCR rate (target )
- SLA compliance by priority (P2/P3 focus) with breach root-cause decomposition (waiting vs. processing vs. churn)
- Per-skill queue waiting time & demand-supply heatmap (scarcity monitoring)
- Workload distribution / utilization variance across agents (fairness)
- Specialist skill-down time (specialists on low-level tasks, target )

**Process views:**
- **Conformance dashboard**: live deviations from the new intended routing model (e.g., manual overrides bypassing skill-based routing) — measures adoption.
- **Handover/SNA network** refreshed periodically to confirm ping-pong subgraphs have dissolved.
- **Variant trend**: share of "happy-path" vs. problematic variants over time.
- **Model-performance monitor** (for Strategy 3): prediction accuracy and drift, triggering retraining.

**Governance loop**: Set thresholds/alerts on each KPI; review monthly; feed observations back into routing rules and proficiency profiles — making resource assignment a **continuously learning, closed-loop system** rather than a one-time fix.

### Summary

By (1) profiling actual resource behavior, (2) quantifying skill-, queue-, and churn-driven delays, (3) tracing them to round-robin logic, weak skill profiles, and poor intake categorization via variant/decision mining, (4) deploying skill-based + workload-aware + predictive routing with refined escalation rules, and (5) validating through simulation and sustaining through dashboards, TechSolve can convert a reactive, round-robin assignment process into an evidence-driven, skill- and load-optimized one — directly targeting the P2/P3 SLA breaches, reassignment delays, and specialist misuse it currently faces.
