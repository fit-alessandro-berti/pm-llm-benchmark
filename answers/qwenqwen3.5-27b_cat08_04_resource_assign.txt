# Process Mining-Driven Resource Optimization Plan for TechSolve Solutions

## 1. Analyzing Resource Behavior and Assignment Patterns

To understand the current state of TechSolve𠏋 resource management, we must move beyond static reports and utilize the dynamic capabilities of process mining on the provided event log. The goal is to visualize the "As-Is" reality of how tickets flow through the human resource network.

### Metrics and Performance Analysis
We will extract the following metrics directly from the event log, filtering by `Resource`, `Agent Tier`, and `Activity`:
*   **Workload Distribution:** Calculate the volume of tickets handled per agent per hour/day. We will look for variance (standard deviation) to identify overloaded vs. underutilized agents.
*   **Activity Processing Time:** Measure the duration of `Work L1 Start` to `Work L1 End` (and L2/L3 equivalents) per agent. This identifies efficiency outliers (e.g., Agent A05 takes 20 mins on average, Agent A02 takes 45 mins for the same category).
*   **First-Call Resolution (FCR) Rate:** Filter for cases where `Ticket Created` -> `Assign L1` -> `Work L1 End` (Resolved) without an `Escalate` or `Reassign` activity.
*   **Skill Match Rate:** Cross-reference `Ticket Required Skill` with `Agent Skills`. Calculate the percentage of assignments where the agent possesses the explicitly required skill versus those where they do not.

### Process Mining Techniques for Assignment Patterns
*   **Resource Interaction Analysis (Social Network Analysis):** We will construct a handover network where nodes are Agents and edges represent ticket escalations or reassignments.
    *   *Insight:* This reveals "Shadow Escalation Paths." For example, the official process may say L1 escalates to L2 Pool, but the SNA might show L1 Agent A05 consistently escalating directly to L2 Agent B12 (a specific individual), indicating a dependency or a "go-to" expert bottleneck.
*   **Role Discovery:** Using clustering algorithms on the activity logs, we can group agents based on their actual behavior rather than their official `Agent Tier`.
    *   *Insight:* An agent officially labeled L1 might be performing complex `Database-SQL` tasks frequently. This suggests a misclassification of roles and potential over-qualification of the L1 pool.
*   **Conformance Checking:** We will define the "To-Be" model (e.g., P1 -> L2 immediately, P4 -> L1) and compare it against the event log.
    *   *Insight:* This quantifies deviation. If 30% of P1 tickets go to L1 first (violating the rule), it highlights a failure in the initial triage logic.

### Skill Utilization Analysis
We will perform a **Skill Heatmap Analysis**.
*   **X-Axis:** Ticket Categories/Required Skills (e.g., Networking-Firewall, App-CRM).
*   **Y-Axis:** Agent Skill Levels.
*   **Color Intensity:** Frequency of assignment.
*   *Goal:* Identify "Skill Silos." Are L3 specialists (e.g., Agent B15 with `Database-SQL`) handling `Basic-Troubleshoot` tickets? If the heatmap shows high intensity of low-complexity tickets on high-skill agents, it confirms the "underutilization of specialists" problem.

## 3. Root Cause Analysis for Assignment Inefficiencies

Once bottlenecks are identified, we must determine *why* the current logic fails.

### Potential Root Causes
1.  **Deficient Assignment Rules:** The current "Round-Robin" logic likely ignores the `Required Skill` attribute. It treats all L1 agents as interchangeable, which is false given the skill variance.
2.  **Inaccurate Ticket Tagging:** If `Required Skill` is often missing or incorrect in the `Ticket Created` phase, the system cannot route correctly.
3.  **Triage Failure:** L1 agents may be accepting tickets they cannot resolve (`Work L1 Start`) without verifying skills, leading to immediate escalation.
4.  **Lack of Visibility:** Dispatchers may not see real-time `Agent Status` (Available vs. Busy), leading to assignments to overloaded agents.

### Process Mining Techniques for Root Cause
*   **Variant Analysis:**
    *   We will split the event log into two variants: **Variant A (Efficient)** = 1 Assignment, Resolved within SLA; **Variant B (Inefficient)** = 3+ Assignments, SLA Breach.
    *   We compare the attributes of these variants.
    *   *Finding:* Variant B might show a high correlation with `Category = Network` and `Priority = P2`. This suggests the current logic specifically fails for Network P2 tickets.
*   **Decision Mining:**
    *   We will build a decision tree model based on the `Assign` activity.
    *   *Input:* Ticket Priority, Category, Required Skill, Time of Day.
    *   *Output:* Assigned Agent Tier/Skill.
    *   *Insight:* This reveals the implicit rules used by the Dispatcher. For example, the tree might show that for P3 tickets, the Dispatcher ignores `Required Skill` 80% of the time, confirming the "blind round-robin" suspicion.

## 4. Developing Data-Driven Resource Assignment Strategies

Based on the analysis, I propose three distinct strategies to optimize the assignment logic.

### Strategy 1: Skill-Based Routing (SBR) with Proficiency Weighting
*   **Issue Addressed:** Skill mismatches, frequent reassignments, specialists doing low-level work.
*   **Logic:**
    *   Implement an algorithm that matches `Ticket Required Skill` to `Agent Skills`.
    *   **Weighting:** If multiple agents have the skill, prioritize based on proficiency (e.g., Years of experience or historical resolution rate for that skill).
    *   **Tier Guardrails:** P1/P2 tickets require L2/L3 skills; P3/P4 can go to L1 if skills match.
*   **Data Required:** Accurate `Agent Skills` matrix, `Ticket Required Skill` tags, Historical resolution success rates per skill.
*   **Expected Benefits:** Reduction in reassignments by 40%, increase in First-Contact Resolution (FCR) for L1 by 15%, reduced SLA breaches due to skill mismatch.

### Strategy 2: Dynamic Workload-Aware Balancing
*   **Issue Addressed:** Uneven workload distribution, overloaded agents causing delays.
*   **Logic:**
    *   Instead of pure Round-Robin, use **Weighted Round-Robin**.
    *   The system calculates a "Load Score" for each agent: `(Current Active Tickets * Complexity Weight) / Agent Capacity`.
    *   Tickets are assigned to the qualified agent with the lowest Load Score.
    *   **Thresholds:** If an agent's Load Score exceeds 80%, they are temporarily removed from the assignment pool for 30 minutes to focus on clearing existing work.
*   **Data Required:** Real-time queue depth, Agent status (Active/Idle), Ticket complexity scores (derived from historical resolution times).
*   **Expected Benefits:** Smoother workload distribution (reduced variance), reduced burnout, consistent response times across the team.

### Strategy 3: Predictive Triage and Auto-Escalation
*   **Issue Addressed:** L1 agents wasting time on complex tickets, delayed escalations.
*   **Logic:**
    *   Use **Predictive Process Mining** (Machine Learning) on the ticket description and category at `Ticket Created`.
    *   Predict the probability of L1 Resolution.
    *   **Rule:** If `Probability(L1 Resolution)` < 30%, bypass L1 and route directly to the appropriate L2/L3 skill pool.
    *   **Rule:** If `Probability(L1 Resolution)` > 70%, route to L1 with a suggested knowledge base article.
*   **Data Required:** Historical ticket text/descriptions, Historical resolution outcomes (Resolved vs. Escalated).
*   **Expected Benefits:** Reduced L1 cycle time, immediate access to specialists for complex issues, improved SLA compliance for high-priority tickets.
