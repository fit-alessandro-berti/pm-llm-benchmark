# Log Skeleton — Order Fulfillment Process

## Assumptions about process behavior

- Every case starts with **RO** (exactly once).
- **CA** may need to be repeated (e.g., re-check after a stock issue)  1 or 2 times.
- **PI** and **QC** may repeat once if quality check fails and items must be re-picked  1 or 2 times; every pick is *immediately* followed by a quality check.
- **RI, PO, GSL, D, IC** happen exactly once per order.
- **RP** is optional within the case (customer may not have paid yet)  0 or 1 times.
- No activities are mutually exclusive in this process, so `never_together` is empty.

## Python dictionary

```python
log_skeleton = {
    'equivalence': {
        # All the "exactly once" activities mirror each other's frequency
        ('RO', 'RI'), ('RI', 'RO'),
        ('RO', 'PO'), ('PO', 'RO'),
        ('RO', 'GSL'), ('GSL', 'RO'),
        ('RO', 'D'), ('D', 'RO'),
        ('RO', 'IC'), ('IC', 'RO'),
        ('PO', 'GSL'), ('GSL', 'PO'),
        ('GSL', 'D'), ('D', 'GSL'),
        ('D', 'IC'), ('IC', 'D'),
        # Every pick is checked: PI and QC always occur equally often
        ('PI', 'QC'), ('QC', 'PI')
    },

    'always_before': {
        ('CA', 'RO'),    # availability check only after order received
        ('RI', 'CA'),    # reservation only after availability confirmed
        ('PI', 'RI'),    # picking only after reservation
        ('QC', 'PI'),    # quality check only after picking
        ('PO', 'QC'),    # packing only after quality check
        ('GSL', 'PO'),   # label only after packing
        ('D', 'GSL'),    # dispatch only after label
        ('IC', 'D'),     # invoice only after dispatch
        ('RP', 'IC')     # payment recorded only after invoicing
    },

    'always_after': {
        ('RO', 'CA'),    # an order is always checked for availability
        ('CA', 'RI'),    # availability check eventually leads to reservation
        ('RI', 'PI'),    # reserved items get picked
        ('PI', 'QC'),    # picked items get quality-checked
        ('QC', 'PO'),    # quality check eventually leads to packing
        ('PO', 'GSL'),   # packed order gets a shipping label
        ('GSL', 'D'),    # labeled package gets dispatched
        ('D', 'IC'),     # dispatch is always followed by invoicing
        ('RO', 'D')      # every received order is eventually dispatched
    },

    'never_together': set(),   # no mutually exclusive activities in this process

    'activ_freq': {
        'RO':  {1},        # exactly once per case
        'CA':  {1, 2},     # may be re-checked once
        'RI':  {1},        # exactly once
        'PI':  {1, 2},     # may be re-picked after failed QC
        'QC':  {1, 2},     # one check per pick
        'PO':  {1},        # exactly once
        'GSL': {1},        # exactly once
        'D':   {1},        # exactly once
        'IC':  {1},        # exactly once
        'RP':  {0, 1}      # optional (payment may be pending)
    },

    'directly_follows': {
        ('RO', 'CA'),    # availability is checked immediately upon receipt
        ('PI', 'QC'),    # quality check happens immediately after picking
        ('PO', 'GSL'),   # label generated immediately after packing
        ('GSL', 'D')     # dispatch happens immediately after labeling
    }
}
```

## Rationale highlights

| Constraint | Key intuition |
|---|---|
| **Equivalence** | All "happens-exactly-once" activities (RO, RI, PO, GSL, D, IC) co-occur with equal frequency; PI  QC are tied since each pick triggers exactly one check. |
| **Always before** | Encodes the strict precedence chain RO  CA  RI  PI  QC  PO  GSL  D  IC  RP. |
| **Always after** | The forward guarantees of the chain — note RP is **not** included as a target of `(IC, RP)` since payment may not occur within the case. |
| **Never together** | Empty — the scenario contains no exclusive branches. |
| **Activity occurrences** | Reflects mandatory single execution, allowed loops (CA, PI, QC), and the optional RP. |
| **Directly-follows** | Only the tightest hand-offs (no other activity can intervene). |
