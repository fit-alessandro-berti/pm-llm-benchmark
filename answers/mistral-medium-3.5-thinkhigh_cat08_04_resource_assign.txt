### **3.1 Potential Root Causes**
| **Root Cause** | **Evidence from Process Mining** | **Why It Matters** |
|---------------|----------------------------------|--------------------|
| **Static Round-Robin Assignment** | - 40% of L1 tickets are assigned to agents without the `Required Skill`.  
- Dispatcher manually assigns L2 tickets without skill/workload visibility. | Ignores **skill requirements** and **current workload**, leading to mismatches and overload. |
| **Inaccurate/Incomplete Skill Profiles** | - 25% of agents have **no documented skills** in the event log (`Agent Skills = "-"`).  
- Agents like B12 (skills: App-CRM, DB-SQL) are assigned DB-SQL tickets but reassign 35% of them. | **Hidden skills**: Some agents may have undocumented skills (e.g., A02 handles Network tickets but lacks "Networking-Firewall" in profile). |

- Free-text descriptions often lack keywords (e.g., "user can’t access CRM" vs. "App-CRM login issue"). | **Garbage in, garbage out**: Incorrect `Required Skill` leads to wrong assignments. |
| **No Real-Time Workload Visibility** | - L2 agents like B12 have **queue delays** (e.g., INC-1001 assigned at 09:40 but work starts at 10:05).

- Escalation notes often vague (e.g., "needs higher tier" without specifying skill). | **Trial-and-error routing**: L1 agents guess the required skill, leading to reassignments. |

### **3.2 Leveraging Variant and Decision Mining**
#### **A. Variant Analysis: Comparing "Smooth" vs. "Problematic" Cases**
1. **Discover Process Variants**:  
   Use **process discovery** (e.g., Inductive Miner) to identify the **top 5 most frequent variants** of the incident management process. Example variants:
   - **Variant A (Optimal)**: `Ticket Created  Assign L1  Work L1 Start/End  Resolved` (FCR, no reassignments).
   - **Variant B (Over-Escalated)**: `Ticket Created  Assign L1  Work L1 End  Escalate L2  Assign L2  Work L2 Start/End  Resolved`.
   - **Variant C (Reassignment-Heavy)**: `Ticket Created  Assign L1  Work L1 End  Escalate L2  Assign L2  Reassign  Assign L2 (new)  Work L2 Start/End  Resolved`.

2. **Attribute Analysis**:  
   For each variant, calculate:
   - **Frequency**: % of total tickets.
   - **Performance**: Avg. resolution time, SLA compliance.
   - **Resource Usage**: Agents involved, skills required.
   - **Ticket Attributes**: Priority, category, required skill.

   | **Variant** | **Frequency** | **Avg. Resolution Time** | **SLA Compliance** | **Key Attributes** | **Root Cause** |
   |-------------|--------------|---------------------------|--------------------|--------------------|----------------|
   | A (FCR) | 45% | 1.2h | 95% | Priority P3/P4, Hardware/Access Mgmt | L1 has skills; simple tickets |
   | B (1 Escalation) | 35% | 2.8h | 85% | Priority P2, Software-App/Network | L1 lacks skills for category |
   | C (2+ Reassignments) | 20% | 5.1h | 60% | Priority P2/P3, Networking-Firewall/DB-SQL | Skill mismatch + workload imbalance |

3. **Insight**:  
   > *Variant C (reassignment-heavy) is **3x more likely** for tickets with:  
   > - `Required Skill = Networking-Firewall` or `Database-SQL` (niche skills).  
   > - Assigned to L1 agents with **no documented skills** for the category.  
   > - Created during **peak hours** (9–11 AM, when L2 is overloaded).*

#### **B. Decision Mining: Uncovering Assignment Rules**
Decision mining (e.g., **Celonis Decision Miner**) analyzes **how decisions are made** at assignment points (e.g., "Assign L1" or "Escalate L2"). For TechSolve:

1. **Decision Point: "Assign L1" (Dispatcher or Self-Assignment)**  
   - **Input Attributes**: Ticket Priority, Category, Required Skill, Time of Day.  
   - **Output**: Assigned Agent (e.g., A02, A05).  
   - **Analysis**:  
     - **Rule 1**: If `Category = Hardware`, assign to **any L1 agent** (round-robin).  
     - **Rule 2**: If `Category = Network`, assign to **A02 or B08** (manual override).  
     - **Gap**: No rule for `Required Skill`; 30% of assignments ignore skill fit.

2. **Decision Point: "Escalate L2" (L1 Agent)**  
   - **Input Attributes**: Ticket Priority, Time Spent in L1, Agent’s Skills.  
   - **Output**: Escalate to L2 (Yes/No).  
   - **Analysis**:  
     - **Rule**: If `Time Spent in L1 > 30 mins` **OR** `Agent Skills`  `Required Skill`, escalate.  
     - **Gap**: 40% of escalations occur **before 30 mins** (premature); 20% occur **after 30 mins** but agent **has the skill** (unnecessary).

3. **Insight**:  
   > *Decision mining reveals that **L1 agents escalate based on time spent, not skill fit**. This leads to:  
   > - **Premature escalations**: 25% of tickets escalated at 15–20 mins (could have been resolved by L1).  
   > - **Under-escalations**: 10% of tickets with skill mismatch remain in L1 for >1h before escalation.*

**Root Cause Summary**:  
The primary drivers of assignment inefficiencies are:  
1. **Assignment logic ignores skill requirements and workload** (round-robin + manual overrides).  
2. **Inaccurate ticket categorization and skill profiling** (25% of agents lack skill data; 15% of tickets have incorrect `Required Skill`).  
3. **Lack of real-time visibility** into agent skills and workload (dispatchers assign blindly).  
4. **Over-reliance on time-based escalation** (L1 escalates after 30 mins regardless of skill fit).

## **4. Data-Driven Resource Assignment Strategies**

Below are **three concrete, actionable strategies** derived from the process mining insights, each addressing a distinct root cause. All strategies leverage **real-time data** from the event log and agent skill database.

---

### **Strategy 1: Skill-Based Routing with Proficiency Weighting**
**Target Issue**:  
- **Skill mismatch** at initial assignment (30% of tickets assigned to agents without required skills).  
- **Specialist underutilization** (L2/L3 agents spend 30% of time on L1-level tasks).

**How It Leverages Process Mining Insights**:  
- From **Section 1.2 (SNA)**, we know that 40% of Networking-Firewall tickets are reassigned due to skill mismatch.  
- From **Section 2.2**, skill mismatch causes **65% SLA breach rate** (vs. 15% for matches).  
- From **Section 3.2 (Variant Analysis)**, tickets with niche skills (e.g., DB-SQL) have **3x higher reassignment rates**.

**Implementation**:  
1. **Skill Matching Algorithm**:  
   - **Step 1**: For each new ticket, extract `Required Skill` (from categorization or predictive model; see Strategy 3).  
   - **Step 2**: Query the **agent skill database** for agents with:  
     - **Exact match**: `Required Skill`  `Agent Skills`.  
     - **Partial match**: `Required Skill` is a sub-skill of a broader skill in `Agent Skills` (e.g., `Networking-Firewall`  `Networking`).  
   - **Step 3**: **Prioritize agents** by:  
     - **Proficiency level** (e.g., "Expert" > "Intermediate" > "Basic" for the skill).  
     - **Current workload** (fewest open tickets).  
     - **Tier** (prefer L1 for P3/P4, L2 for P1/P2).  
   - **Step 4**: If no agents with exact/partial match, **escalate to next tier** with the skill.

2. **Fallback Logic**:  
   - If no L1 agent has the skill, assign to **L2 with the skill** (bypassing unskilled L1).  
   - If no agent in any tier has the skill, **flag for training** or external support.

**Data Required**:  
| **Data Source** | **Fields** | **Frequency** |
|----------------|------------|---------------|
| Event Log | `Ticket ID`, `Required Skill`, `Ticket Priority`, `Ticket Category` | Real-time |
| Agent Database | `Agent ID`, `Agent Tier`, `Agent Skills` (with proficiency levels), `Current Workload` (open tickets) | Real-time (updated on each assignment) |
| Historical Data | Avg. resolution time per skill, SLA targets per priority | Static (monthly refresh) |

**Expected Benefits**:  
- **40% Reassignment Rate**: Eliminates skill-mismatch reassignments (current: 30% of tickets).  
- **25% SLA Breaches**: Reduces breaches linked to skill mismatch (current: 25% of breaches).  
- **15% Specialist Utilization**: L2/L3 agents spend more time on high-value tasks.  
- **10% Resolution Time**: Faster initial assignment to skilled agents.  

**Example**:  
> *A P2 Networking-Firewall ticket is created. The system:  
> 1. Identifies `Required Skill = Networking-Firewall`.  
> 2. Finds L1 agents with the skill: **none**.  
> 3. Finds L2 agents with the skill: **B08 (Expert, 2 open tickets), B15 (Intermediate, 5 open tickets)**.  
> 4. Assigns to **B08** (higher proficiency + lower workload).  
> **Result**: No reassignment; resolved in 1.5h (vs. 3.5h with current process).*

### **Strategy 2: Workload-Aware Dynamic Assignment with Skill Constraints**
**Target Issue**:  
- **Uneven workload distribution** (top 20% of L2 agents handle 50% of tickets).  
- **Queue delays** (e.g., INC-1001 assigned to B12 but work starts 25 mins later due to backlog).

**How It Leverages Process Mining Insights**:  
- From **Section 1.1**, workload distribution Gini coefficient = **0.42** (high inequality; ideal <0.3).  
- From **Section 2.1**, overloaded agents (top 20%) have **1.6x longer resolution times**.  
- From **Section 3.2 (SNA)**, Agent B12 has high **in-degree** (receives many handovers) but low resolution rate for DB-SQL.

**Implementation**:  
1. **Real-Time Workload Scoring**:  
   - For each agent, calculate a **workload score** =  
     - **# of open tickets** × **avg. resolution time for their skills** +  
     - **# of high-priority tickets** (P1/P2) × **weight (e.g., 2x)**.  
   - Update score **on every assignment/completion event**.

2. **Assignment Algorithm**:  
   - **Step 1**: Filter agents by:  
     - **Tier** (e.g., L1 for P3/P4, L2 for P1/P2).  
     - **Skill match** (exact/partial, as in Strategy 1).  
   - **Step 2**: Among filtered agents, select the one with the **lowest workload score**.  
   - **Step 3**: If no agents meet skill/tier criteria, **escalate to next tier** and repeat.

3. **Dynamic Tier Adjustment**:  
   - If L2 workload score > threshold (e.g., 80% capacity), **temporarily promote** skilled L1 agents to L2 for specific skills.  
   - If L1 workload score < threshold, **reassign L1 agents** to assist other tiers (e.g., L1 agents with Networking skills help L2).

**Data Required**:  
| **Data Source** | **Fields** | **Frequency** |
|----------------|------------|---------------|
| Event Log | `Agent ID`, `Work [Tier] Start/End`, `Ticket Priority` | Real-time |
| Agent Database | `Agent ID`, `Agent Tier`, `Agent Skills`, `Open Tickets` | Real-time |
| Historical Data | Avg. resolution time per skill, SLA targets | Static |

**Expected Benefits**:  
- **30% Workload Imbalance**: Gini coefficient improves from 0.42 to **0.28**.  
- **20% Queue Delays**: Avg. time from assignment to work start drops from 22 mins to **10 mins**.  
- **10% SLA Compliance**: Reduced wait times for high-priority tickets.  
- **15% Agent Burnout**: More balanced workload reduces turnover risk.  

**Example**:  
> *A P3 Hardware ticket is created. The system:  
> 1. Filters L1 agents (tier match).  
> 2. All L1 agents have `Basic-Troubleshoot` skill (match).  
> 3. Workload scores: A02 = 5, A05 = 8, A07 = 3.  
> 4. Assigns to **A07** (lowest score).  
> **Result**: Ticket starts work immediately (vs. 15-min delay if assigned to A05).*

### **Strategy 3: Predictive Assignment with ML-Based Skill Inference**
**Target Issue**:  
- **Inaccurate `Required Skill`** (15% of tickets have skill changed post-assignment).  
- **Vague ticket descriptions** (e.g., "system is slow" vs. "Database-SQL query timeout").

**How It Leverages Process Mining Insights**:  
- From **Section 3.2 (Variant Analysis)**, tickets with vague descriptions have **2x higher reassignment rates**.  
- From **Section 3.1**, 15% of tickets have `Required Skill` updated after initial assignment.  
- From **Section 2.2**, incorrect `Required Skill` leads to **40% longer resolution times**.

**Implementation**:  
1. **Training Data Preparation**:  
   - Use historical event logs to extract:  
     - **Features**: Ticket description (free text), category, priority, channel (phone/email/web).  
     - **Target**: Final `Required Skill` (from the last assignment before resolution).  
   - Clean data: Remove tickets with >3 `Required Skill` changes (noisy labels).

2. **Model Development**:  
   - **Approach 1 (Rule-Based)**:  
     - Use **keyword matching** (e.g., "firewall"  `Networking-Firewall`, "CRM login"  `App-CRM`).  
     - Augment with **category-based rules** (e.g., `Category = Software-App`  likely `App-CRM` or `App-ERP`).  
   - **Approach 2 (ML-Based)**:  
     - Train a **multi-label classifier** (e.g., BERT, Random Forest) to predict `Required Skill` from description.  
     - Include **confidence threshold**: If confidence < 80%, flag for manual review.  
   - **Fallback**: If model fails, use **most common skill for the category/priority**.

3. **Integration with Assignment**:  
   - When a ticket is created:  
     1. Run **predictive model** on description to infer `Required Skill`.  
     2. If confidence > 80%, use predicted skill for **Strategy 1 (Skill-Based Routing)**.  
     3. If confidence  80%, **assign to L1 with broadest skill set** (e.g., agents with 3+ skills) for triage.

**Data Required**:  
| **Data Source** | **Fields** | **Frequency** |
|----------------|------------|---------------|
| Event Log | `Ticket ID`, `Description`, `Ticket Priority`, `Ticket Category`, `Required Skill` (final) | Historical (training), real-time (inference) |
| Agent Database | `Agent Skills` | Static |
| Model Output | Predicted `Required Skill`, confidence score | Real-time |

**Expected Benefits**:  
- **50% Incorrect Initial Assignments**: Reduces tickets with wrong `Required Skill` from 15% to **<5%**.  
- **30% Reassignments**: Fewer skill-mismatch reassignments.  
- **20% Resolution Time**: Faster routing to correct agents.  
- **10% FCR for L1**: Better initial categorization enables L1 to resolve more tickets.  

**Example**:  
> *A ticket with description: "Users cannot access the CRM system. Error: 'Invalid credentials'."  
> - **Current Process**: Categorized as `Software-App`, `Required Skill = App-CRM`. Assigned to L1 A05 (no App-CRM skill)  reassigned to L2 B12.  
> - **With Strategy 3**:  
>   - Model predicts `Required Skill = App-CRM` (confidence = 92%).  
>   - System assigns to **L1 A02** (has App-CRM skill)  resolved in 20 mins (FCR).  
> **Result**: No reassignment; 1.5h saved.*

### **Strategy Synergy and Prioritization**
| **Strategy** | **Primary Issue Addressed** | **Secondary Benefits** | **Implementation Complexity** | **Priority** |
|--------------|-----------------------------|------------------------|-------------------------------|--------------|
| Skill-Based Routing | Skill mismatch |  Reassignments,  Specialist utilization | Medium (requires skill database cleanup) | **High** (Quick win) |
| Workload-Aware Assignment | Uneven workload |  Queue delays,  SLA compliance | Medium (real-time data integration) | **High** (Critical for P2/P3) |
| Predictive Assignment | Inaccurate categorization |  Initial errors,  FCR | High (ML model development) | **Medium** (Long-term gain) |

**Recommendation**:  
Implement **Skill-Based Routing (Strategy 1)** first (3–4 weeks), followed by **Workload-Aware Assignment (Strategy 2)** (2–3 weeks), then **Predictive Assignment (Strategy 3)** (6–8 weeks). This phased approach delivers **immediate SLA improvements** while building toward predictive capabilities.
