### Justification of Naming Changes

Only activities that repeated *within the same case* were renamed. The new names were derived directly from the timestamp sequence and the "Additional Context" field:

- **C001 – Verify Documents**:  
  The first instance (09:20) occurs right after initiation with no prior submissions, so it was renamed **Initial Identity Document Verification**.  
  The second instance (10:30) occurs *after* the customer’s “Submit Additional” event and explicitly references re-checking the newly provided ID scan, so it was renamed **Re-verification of Supplemented Documents**.

- **C002 – Call Customer**:  
  The first call (09:00) happens immediately after initial verification and is described as addressing an “account discrepancy,” so it was renamed **Outbound Call for Account Discrepancy Clarification**.  
  The second call (09:50) occurs *after* the internal “Adjust Account” step and is described as a confirmation of those changes, so it was renamed **Follow-up Call for Adjustment Confirmation**.

- **C003 – Verify Documents**:  
  The first instance (11:25) is performed at a branch by a BranchOfficer on a walk-in customer and is described as “Physical verification of ID,” so it was renamed **On-site Physical Document Verification**.  
  The second instance (12:20) occurs *after* the “Adjust Account” correction and is explicitly labeled “Re-verify docs after account adjustment,” so it was renamed **Post-Adjustment Document Re-verification**.

Activities that did not repeat within any single case (“Initiate Request”, “Adjust Account”, “Call Customer” in C001 and C003, “Submit Additional”, and “Approve Request”) were left unchanged. This preserves the original activity vocabulary where no contextual disambiguation was required within a case.
