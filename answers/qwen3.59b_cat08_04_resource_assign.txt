# Process Mining Optimization Plan: TechSolve Solutions Resource Assignment

## 1. Analyzing Resource Behavior and Assignment Patterns

To understand how TechSolve currently utilizes its human capital, we must move beyond aggregate metrics and inspect the granular flow of work through the agents. Using the provided event log, we will apply specific process mining techniques to extract insights into resource behavior.

### A. Performance and Behavior Metrics
We will calculate agent-specific performance indicators by filtering the event log for activities where `Resource` equals an Agent ID. Key metrics include:

*   **Workload Distribution (Load Balancing):** We will count the number of "Work Start" activities per agent per day. By plotting this against ticket priority, we can identify if high-priority tickets are disproportionately assigned to underperforming agents or if L1 agents are overloaded with complex P2/P3 tickets they cannot handle.
*   **Activity Processing Time (APT):** Calculate the duration between `Assign` and `Work Start` for each agent. High variance here indicates inconsistency in how agents manage their queues or engage with incoming work.
*   **First-Call Resolution (FCR) Rate:** For L1, we will calculate the percentage of tickets where `Work L1 End` is followed by a successful closure (or immediate escalation to L2/L3). We will correlate this with agent skills. An agent with "Basic-Troubleshoot" handling "App-CRM" tickets that require "App-CRM" skill likely has a lower FCR rate than one handling "Hardware" tickets.
*   **Skill Utilization Matrix:** Cross-tabulate `Agent Skills` against `Ticket Required Skill`. We will calculate the **Skill Match Score** for every assignment. A high match score indicates effective assignment; a low score (e.g., Agent has no SQL skill but handles SQL tickets) indicates a potential bottleneck or misassignment.

### B. Process Mining Techniques for Assignment Patterns
*   **Resource Interaction Analysis:** By mapping the sequence of events, we can visualize the "Handover Chains." For example, `INC-1001` shows a handover from L1 (Agent A05) to L2 (Agent B12). We will map these paths to see if tickets are bouncing between specific agents (e.g., Agent A05 -> Agent B12 -> Agent B15) rather than resolving.
*   **Role Discovery:** Using clustering algorithms on the `Activity` and `Resource` columns, we can identify implicit roles. If a "Dispatcher" is rarely involved but assignments happen automatically, we might discover that the "System" or an unlisted automation script is the actual decision-maker.
*   **Conformance Checking:** We will define the *intended* logic (e.g., "Round-robin within tier," "Escalate if L1 cannot resolve"). We then overlay the event log to see deviations. If the rule says "L2 only for P2 Network," but we see P3 Software-App tickets going to L2, this reveals a deviation from policy that needs investigation.

### C. Skill Utilization Analysis
We will analyze the overlap between `Agent Skills` and `Ticket Required Skill`.
*   **Specialist Efficiency:** We will check if L2/L3 agents are assigned tasks requiring only "Basic-Troubleshoot." If Agent B15 (who has SQL skills) is assigned a simple P4 Login issue, this indicates an under-utilization of high-value skills.
*   **Skill Mismatch Cost:** We will calculate the time spent on tickets where `Agent Skills` do not contain `Required Skill`. This quantifies the "learning cost" or "guessing cost" for agents working outside their competency.

## 2. Identifying Resource-Related Bottlenecks and Issues

Based on the analysis above, we can pinpoint specific operational failures that are driving SLA breaches and inefficiencies.

### A. Pinpointing Specific Problems
*   **Skill-Based Bottlenecks:** If `Ticket Category` is "Network" but only Agent B08 has "Networking-Firewall," and the queue for this agent grows significantly, we identify a skill scarcity bottleneck. We can quantify this by measuring the average wait time before assignment when that specific skill is required vs. generic skills.
*   **Reassignment Penalties:** We will calculate the **"Reassignment Delay"**: `(Timestamp of Reassign Activity) - (Timestamp of Previous Work End)`. Summing these across all tickets will reveal the average cost of a reassignment in minutes/hours. If this value is high, it indicates that agents are constantly swapping tickets or dispatchers are frequently changing assignments.
*   **Escalation Inefficiency:** We will compare the time taken to resolve a ticket at L1 if assigned directly vs. the time taken after an escalation. If direct assignment to L2 resolves P3 tickets faster than L1 -> L2, it suggests the initial L1 assignment was incorrect (wrong skill set).
*   **SLA Breach Correlation:** We will correlate `Timestamp Type` = "COMPLETE" with SLA targets. By filtering for cases where `Priority` is P2/P3 and `SLA Status` is Breached, we can check if these breaches are clustered around specific agents or specific ticket categories (e.g., all "Database-SQL" P2 tickets breach SLA).
*   **Underperformance/Overload:** We will identify agents with a high ratio of "Work Start" to "Ticket Created" where `Agent Tier` is L1, but the `Required Skill` is complex. These are agents being forced to work outside their tier's capability, leading to delays.

### B. Quantifying Impact
*   **Reassignment Cost:** *Example:* If 20% of tickets require a reassignment and each reassignment adds an average of 15 minutes (dispatch time + agent context switch), this represents significant lost productivity.
*   **Skill Mismatch SLA Impact:** *Example:* "35% of P2 Network SLA breaches occur on tickets assigned to agents without 'Networking-Firewall' skills." This directly links resource capability to compliance failure.

## 3. Root Cause Analysis for Assignment Inefficiencies

Process mining allows us to move from symptoms (SLA breaches) to causes (assignment logic).

### A. Deficiencies in Current Rules
*   **Round-Robin vs. Skill-Based:** Decision mining on the `Assign` activity will reveal if the system is strictly round-robin (assigning to Agent X, then Y, then Z regardless of skills) or if it attempts skill matching. If the log shows L1 agents without specific skills handling tickets requiring those skills, the root cause is a **rule deficiency**.
*   **Skill Profile Accuracy:** We will compare `Agent Skills` in the system against their actual resolution history. If an agent claims "Database-SQL" but has never resolved a SQL ticket successfully or has high reassignment rates on SQL tickets, the skill profile is inaccurate.

### B. Categorization and Escalation Issues
*   **Initial Categorization:** We will analyze the time difference between `Ticket Created` and `Assign`. If this gap is large for specific categories, it suggests the initial categorization (e.g., "Software-App") was too vague, requiring manual intervention by a Dispatcher to assign the right agent.
*   **Escalation Triggers:** We will analyze cases where L1 fails to resolve. Is the failure due to complexity or lack of skill? If 80% of escalations happen because the L1 agent lacked the specific skill required, the root cause is **insufficient L1 empowerment** or **poor initial routing**.

### C. Decision Mining and Variant Analysis
*   **Variant Analysis:** We will split the event log into two variants:
    1.  *Efficient:* Tickets resolved at L1 or with minimal escalations.
    2.  *Inefficient:* Tickets with reassignments or SLA breaches.
    We will compare the `Assign` activity attributes (Time of day, Agent Tier, Queue Length) between these two variants. If "Inefficient" tickets are consistently assigned during peak hours to overloaded agents, we identify **load imbalance** as a root cause.
*   **Correlation Analysis:** We will check for correlations between `Agent Workload` and `Assignment Time`. If agents with high workload get fewer new assignments despite high skill availability, there is a lack of **real-time visibility** in the dispatch system.

## 4. Developing Data-Driven Resource Assignment Strategies

Based on the insights derived from the mining analysis, we propose three distinct strategies to optimize TechSolve's resource assignment logic.

### Strategy 1: Skill-Based Intelligent Routing (SBIR)
*   **Issue Addressed:** Frequent reassignments due to skill mismatches and L2/L3 specialists handling tasks below their level.
*   **Leveraging Insights:** Uses the **Skill Utilization Matrix** from Section 1. We will assign tickets based on a "Skill Match Score" (Agent Skill vs. Required Skill). If an exact match exists in L1, route there. If not, route to L2 with the specific skill.
*   **Data Required:** Accurate `Agent Skills` and `Ticket Required Skill` columns.
*   **Expected Benefit:** Reduces reassignment rates by 30-50%. Increases First Call Resolution (FCR) at L1 for standard issues. Frees up L2/L3 specialists for complex tasks, improving SLA adherence on P1/P2 tickets.

### Strategy 2: Dynamic Workload-Aware Assignment
*   **Issue Addressed:** Uneven workload distribution and some agents being consistently overloaded while others are underutilized.
*   **Leveraging Insights:** Uses **APT (Activity Processing Time)** and **Workload Distribution** metrics from Section 1. Instead of round-robin, the system calculates a "Capacity Score" for each agent: `(Max Throughput - Current Queue) / Average Resolution Time`.
*   **Data Required:** Real-time `Ticket Created` logs, current queue lengths per agent, and historical resolution velocity per agent.
*   **Expected Benefit:** Balances load across agents, preventing burnout on high-volume agents and reducing idle time on underutilized ones. Reduces average ticket wait time by smoothing the intake flow.

### Strategy 3: Predictive Escalation & Pre-Assignment
*   **Issue Addressed:** Delays introduced by frequent or unnecessary escalations and L1 inability to handle complexity.
*   **Leveraging Insights:** Uses **Decision Mining** on `Ticket Created` vs. `Assign`. If historical data shows that "Network-Firewall" P2 tickets have a 90% chance of failing at L1, the system should bypass L1 for these specific high-complexity categories or assign them to an L2 specialist with a "shadow ticket" workflow.
*   **Data Required:** Historical resolution rates per `Ticket Category` and `Priority`, combined with current agent skill sets.
*   **Expected Benefit:** Reduces the cycle time for complex tickets by skipping ineffective tiers. Improves SLA compliance on P2/P3 tickets by ensuring they are handled by agents capable of resolving them from the start (or immediate escalation).

## 5. Simulation, Implementation, and Monitoring

To ensure these strategies work without disrupting live operations, we must validate them first and monitor continuously.

### A. Business Process Simulation
*   **Modeling:** We will construct a **Petri Net** or **Discrete Event Simulation (DES)** model of the current ITSM process using the mined event log data.
*   **Scenario Testing:** In the simulation, we will replace the "Round-Robin" logic with the proposed "Skill-Based Routing."
    *   *Metric to Watch:* Does the average cycle time for P2 tickets decrease?
    *   *Metric to Watch:* Do L2 agents become underutilized (idle)?
    *   *Metric to Watch:* What is the impact on SLA breach rates during peak load periods?
*   **Validation:** We run the simulation with historical data to see if the proposed strategy would have prevented past breaches. If the simulation shows negative impacts (e.g., queue buildup), we refine the algorithm before deployment.

### B. Implementation Plan
1.  **Data Cleaning:** Ensure `Agent Skills` and `Ticket Required Skill` are standardized in the ITSM tool.
2.  **Pilot Phase:** Implement Strategy 1 (SBIR) for a specific category (e.g., Software-App) to test without affecting critical infrastructure tickets.
3.  **Rollout:** Gradually apply workload-aware logic across all tiers.

### C. Continuous Monitoring and Dashboards
Post-implementation, we will deploy Process Mining dashboards to track:
*   **Reassignment Frequency (per Ticket):** Target < 10% of total tickets.
*   **Skill Match Rate:** Percentage of assigned tickets where Agent Skill == Required Skill. Target > 95%.
*   **Agent Utilization Heatmap:** Visualizing who is over/under loaded in real-time.
*   **SLA Breach by Resource:** Identifying if specific agents are consistently causing breaches (indicating training or reassignment need).
*   **Process Variance:** Monitoring the "Happy Path" vs. "Exception Path." If the exception path lengthens, we investigate why the new logic isn't working as expected.

By continuously feeding the process mining engine with new logs, TechSolve can adapt its assignment rules dynamically, ensuring that resource allocation remains efficient as ticket volumes and agent skills evolve.
