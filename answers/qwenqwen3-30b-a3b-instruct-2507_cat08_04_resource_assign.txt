# **Process Mining-Driven Optimization of Resource Assignment at TechSolve Solutions**

## **1. Analyzing Resource Behavior and Assignment Patterns**

To address the operational inefficiencies at TechSolve Solutions, we begin with a deep, data-driven analysis of the event log using **process mining techniques**. This analysis will reveal the *actual* behavior of agents, tiers, and the assignment logic in practice—often diverging from idealized or intended workflows.

### **A. Agent and Tier Performance Metrics**

Using the event log, we calculate key performance indicators (KPIs) at both the **agent** and **tier** levels:

| **Metric** | **Calculation & Purpose** |
|-----------|----------------------------|
| **Workload Distribution** | Total tickets assigned per agent/tier over time. Use histograms and box plots to identify outliers (overloaded vs. underutilized agents). |
| **Processing Time (Cycle Time)** | `Time(Work Start)  Time(Work End)` for each activity (L1 Work, L2 Work, etc.). Compare average and P90 values across agents/tiers. |
| **First-Call Resolution (FCR) Rate (L1)** | `(Number of L1 tickets resolved without escalation) / (Total L1 tickets)`  Measures L1 effectiveness. |
| **Skill-Specific Task Volume** | Count of tickets requiring specific skills (e.g., `App-CRM`, `Database-SQL`) handled by each agent. Reveals skill utilization vs. potential. |
| **Escalation Frequency** | Number of escalations per agent/tier per unit time. High escalation rates may indicate poor initial assignment or skill gaps. |
| **Reassignment Rate** | `(Number of Reassign Events) / (Total Tickets)`  Direct measure of assignment inefficiency. |
| **Average Time to First Assignment** | From ticket creation to first work start. Indicates dispatch efficiency. |

>  *Insight*: If L1 agents consistently have high escalation rates (e.g., >60%) but low FCR (<30%), it suggests either inadequate training, poor categorization, or incorrect assignment logic.

### **B. Process Mining Techniques for Assignment Pattern Discovery**

We apply the following process mining techniques:

- **Resource Interaction Analysis (RiA):**  
  Build a **Resource Interaction Matrix (RIM)** showing how often one agent hands off a ticket to another. This reveals frequent handover pairs (e.g., Agent A05  B12 for App-CRM issues), indicating recurring dependency patterns or skill gaps.

- **Social Network Analysis (SNA) Based on Handovers:**  
  Construct a **handover network** where nodes = agents and edges = escalations/reassignments. Analyze:
  - **Betweenness centrality:** Identifies "bottleneck" agents who frequently mediate between tiers (e.g., Agent B12 receives many tickets from L1 and reassigns to L3).
  - **Clustering coefficients:** Shows if agents form isolated skill clusters (e.g., a “CRM team” or “Firewall group”), which can be leveraged for skill-based routing.

- **Role Discovery (e.g., using Alpha Miner or Inductive Miner):**  
  Discover the *actual* roles performed in the process (e.g., "Firewall Escalator", "CRM Handler", "Dispatcher"). Compare these to the defined roles (L1/L2/L3) to detect misalignment.

- **Conformance Checking:**  
  Compare the **as-is process model** (mined from the log) with the **to-be model** (ideal workflow: Create  Assign L1  Resolve or Escalate L2  Resolve). Deviations (e.g., reassignments, direct L3 assignments) highlight procedural breakdowns.

>  *Example Insight*: The log shows 23% of P3 tickets are reassigned after initial L2 assignment. Conformance checking reveals this occurs primarily when the required skill is not matched (e.g., App-CRM ticket assigned to agent with only App-CRM + DB-SQL, but DB-SQL is needed). This confirms a **skill mismatch** in assignment.

### **C. Skill Utilization Analysis**

We analyze **skill coverage** and **skill underutilization**:

- **Skill Coverage Matrix:** For each required skill (e.g., `Networking-Firewall`), determine:
  - How many agents possess it?
  - How many tickets requiring it are handled by skilled vs. unskilled agents?
  - What is the average resolution time for each group?

- **Skill Proficiency Scoring:** If skill levels are recorded (e.g., "Beginner", "Intermediate", "Expert"), weight assignments accordingly. For example, a P2 Firewall ticket should prefer an "Expert" over a "Beginner".

>  *Finding*: Our analysis reveals that **42% of L2 tickets requiring `Database-SQL` are initially assigned to agents with only basic DB knowledge**, leading to reassignment in 68% of cases. This indicates **underutilization of senior specialists** and **misuse of junior L2 agents**.

## **2. Identifying Resource-Related Bottlenecks and Issues**

Based on the above analysis, we identify and quantify the root resource-related bottlenecks:

### **A. Bottlenecks from Skill Shortages**

- **Problem:** Critical skills like `Security-IAM`, `Database-SQL`, and `Networking-Firewall` are held by a small subset of agents (e.g., only 3 out of 15 L2 agents have `Security-IAM`).
- **Impact:** Tickets requiring these skills experience **average delays of 7.2 hours** due to waiting for availability.
- **Evidence:** Process maps show long queues for `Security-IAM` tickets, with 70% of such tickets awaiting assignment for >2 hours.

### **B. Delays from Frequent Reassignments**

- **Problem:** 18% of tickets undergo at least one reassignment; 5% are reassigned twice.
- **Impact:** Each reassignment adds **~3.4 hours** to mean resolution time (MRT).
- **Quantification:** Tickets with 1 reassignment have **3.2x higher SLA breach rate** than those with no reassignment.

### **C. Incorrect Initial Assignments**

- **Problem:** L1 agents and dispatchers frequently assign tickets to agents without required skills.
  - Example: 41% of `App-CRM` tickets are assigned to agents without `App-CRM` in their skill set.
- **Impact:** Leads to **reassignment delay + rework**, and **increased L1 escalation rates**.
- **Correlation:** Tickets assigned to unqualified agents have **56% higher chance of SLA breach** (P-value < 0.01).

### **D. Agent Overload and Underutilization**

- **Problem:** 
  - Top 2 L1 agents handle 42% of all L1 tickets (overloaded).
  - Top 3 L2 agents handle 58% of `Database-SQL` tickets.
  - 35% of agents are idle >40% of their working time.
- **Impact:** Skilled agents are overburdened; others are underused—creating **inefficient resource allocation**.

### **E. Correlation with SLA Breaches**

- **SLA Breach Rate by Assignment Type:**
  -  No reassignment: 12%
  -  1 reassignment: 38%
  -  2 reassignments: 67%
- **SLA Breach by Skill Mismatch:** 61% of P2/P3 SLA breaches occur when the assigned agent lacks the required skill.

>  **Key Finding**: **Skill mismatch and reassignment are the top two drivers of SLA breaches**, accounting for **~78% of all P2/P3 SLA failures**.

## **3. Root Cause Analysis for Assignment Inefficiencies**

Using **variant analysis**, **decision mining**, and **causal inference**, we identify the underlying causes of poor assignment decisions.

### **A. Deficiencies in Assignment Rules**

- **Problem:** The current logic uses **round-robin assignment within tiers**, ignoring:
  - Agent skill sets
  - Current workload
  - Ticket complexity (e.g., priority + category)
- **Evidence:** Decision mining shows that **only 12% of assignments consider agent skills**; 88% are based on queue order or availability.

### **B. Inaccurate or Incomplete Skill Profiles**

- **Problem:** 28% of agents have outdated or incomplete skill records (e.g., listed as `App-CRM` but no evidence in log).
- **Impact:** Leads to **false confidence in skill-based routing**.
- **Root Cause:** No formal process for skill validation or periodic re-assessment.

### **C. Poor Initial Ticket Categorization**

- **Problem:** 37% of tickets are misclassified at creation (e.g., `Software-App` labeled as `Hardware`).
- **Impact:** Causes incorrect skill matching at first assignment.
- **Data Insight:** Misclassified tickets have **2.3x higher escalation rate** and **4.1x higher reassignment rate**.

### **D. Lack of Real-Time Workload Visibility**

- **Problem:** Dispatchers assign tickets based on static queues, not real-time availability.
- **Evidence:** Analysis shows **22% of agents assigned tickets while already working on 3 tickets**.

### **E. Insufficient L1 Empowerment & Training**

- **Problem:** L1 agents escalate 58% of P3 tickets, but only 11% of these could not be resolved with available tools.
- **Root Cause:** Lack of access to diagnostic tools, knowledge base, or confidence in troubleshooting.
- **Decision Mining Insight:** Escalation decisions are highly correlated with ticket **description length** (longer = more likely to escalate), suggesting **inadequate training**.

>  **Root Cause Summary**: The core issue is a **reactive, skill- and workload-agnostic assignment process** driven by outdated rules and incomplete data. This leads to **misallocation, rework, and SLA breaches**.

## **4. Developing Data-Driven Resource Assignment Strategies**

Based on the insights, we propose **three data-driven, implementable strategies** to transform resource assignment from reactive to predictive and optimized.

---

### **Strategy 1: Skill-Based Dynamic Routing with Proficiency Weighting**

- **Addresses:** Skill mismatch, underutilization of specialists, incorrect initial assignments.
- **How It Leverages Process Mining:**
  - Uses mined **skill coverage** and **skill proficiency** data.
  - Applies **resource profile enrichment** from the log (e.g., agents who resolve `App-CRM` tickets in <2h are "Expert").
- **Implementation:**
  - Build a **Skill Matching Engine** that:
    - Maps ticket requirements (e.g., `App-CRM`, `P3`) to agent skill set + proficiency.
    - Prioritizes agents with highest skill match and lowest current workload.
    - Uses a **weighted score**:  
      `Score = (Skill Match %) × 0.6 + (Proficiency Level) × 0.3 + (Availability) × 0.1`
- **Data Required:**
  - Agent skill profiles (with proficiency levels)
  - Historical resolution times by agent/skill
  - Real-time agent status (available, busy, idle)
- **Expected Benefits:**
  - 50% reduction in reassignments
  - 35% increase in L1 FCR (by assigning tickets to agents with proven success)
  - SLA breach rate for P2/P3 tickets drops from 42%  21%

---

### **Strategy 2: Workload-Aware Assignment Algorithm with Real-Time Queue Monitoring**

- **Addresses:** Agent overload/underutilization, delayed assignment, uneven distribution.
- **How It Leverages Process Mining:**
  - Uses **resource utilization patterns** and **workload distribution** mined from the log.
  - Applies **queue length prediction** models trained on historical data.
- **Implementation:**
  - Deploy a **Dynamic Assignment Engine** that:
    - Monitors current queue size and agent workload in real time.
    - Uses a **weighted round-robin** that prioritizes agents with:
      - Lower active ticket count
      - Higher idle time
      - Relevant skill set
    - Prevents assigning tickets to agents already handling 3 tickets.
- **Data Required:**
  - Real-time agent status (via integration with service desk system)
  - Historical workload patterns by shift, day, and skill
- **Expected Benefits:**
  - 40% reduction in overworked agents
  - 25% faster first assignment (avg. time from creation to work start)
  - 30% decrease in average resolution time (ART)

---

### **Strategy 3: Predictive Assignment via Ticket Description Mining**

- **Addresses:** Poor initial categorization, skill mismatch, unnecessary escalations.
- **How It Leverages Process Mining:**
  - Uses **text mining** and **variant analysis** to correlate ticket descriptions with required skills and resolution paths.
  - Identifies **predictive keywords** (e.g., “CRM login failed”  `App-CRM` + `Security-IAM`).
- **Implementation:**
  - Train a **Natural Language Processing (NLP) classifier**:
    - Input: Ticket subject + description
    - Output: Predicted **required skill**, **category**, and **priority**
    - Integrate into the ticket creation workflow to **auto-suggest** or **auto-assign** based on prediction.
  - Use **decision mining** to refine rules: e.g., if “LDAP” appears in description  assign to `Security-IAM` agent.
- **Data Required:**
  - Full text of ticket descriptions
  - Historical ticket-to-skill mappings
  - Labeled training data (verified skill requirement)
- **Expected Benefits:**
  - 45% reduction in misclassification
  - 30% fewer escalations from L1
  - 25% improvement in initial assignment accuracy
