### 3.1.Detect cases where the *logical* activity order deviates from the reference model

```sql
WITH ordered_events AS (
    SELECT
        case_id,
        activity,
        event_id,
        timestamp,
        LAG(activity) OVER (PARTITION BY case_id ORDER BY event_id, timestamp) AS prev_activity,
        LEAD(activity) OVER (PARTITION BY case_id ORDER BY event_id, timestamp) AS next_activity
    FROM order_event_log
),
-- Define the allowed predecessor for each activity (based on the “normal” flow)
allowed_pred AS (
    SELECT *
    FROM (VALUES
        ('Register Order',                     NULL),                         -- first step, no predecessor
        ('Perform Credit Check',              'Register Order'),
        ('Validate Stock',                    'Perform Credit Check'),
        ('Confirm Shipment',                  'Validate Stock'),
        ('Ship Goods',                        'Confirm Shipment'),
        ('Issue Invoice',                     'Ship Goods'),
        ('Receive Payment',                   'Issue Invoice')
    ) AS v(activity, allowed_prev)
)
SELECT
    oe.case_id,
    oe.event_id,
    oe.activity,
    oe.prev_activity,
    ap.allowed_prev
FROM ordered_events oe
LEFT JOIN allowed_pred ap
       ON oe.activity = ap.activity
WHERE oe.prev_activity IS DISTINCT FROM ap.allowed_prev
   OR (oe.prev_activity IS NULL AND ap.allowed_prev IS NOT NULL)   -- missing first step
ORDER BY oe.case_id, oe.event_id;
```

*What it shows*: every event whose immediate predecessor is not the one prescribed by the reference flow (including missing first step).  
*Relevance*: catches H1, H2, H3, H6 (rolebased violations will often appear as a wrong predecessor).

### 3.2.Find cases where a **gatekeeping** activity (Credit Check, Validate Stock) is missing entirely

```sql
SELECT
    oel.case_id
FROM orders o
JOIN (
    SELECT DISTINCT case_id
    FROM order_event_log
) oel ON o.case_id = oel.case_id
LEFT JOIN (
    SELECT case_id
    FROM order_event_log
    WHERE activity IN ('Perform Credit Check','Validate Stock')
    GROUP BY case_id
) gate ON oel.case_id = gate.case_id
WHERE gate.case_id IS NULL;
```

*What it shows*: orders that never had a credit check or stock validation.  
*Relevance*: supports H2 (priority bypass), H3 (manual override), H1 (missing gateway).

### 3.3.Spot **paymentbeforeinvoice** or **paymentbeforeshipment** patterns

```sql
SELECT
    oel.case_id,
    MIN(CASE WHEN oel.activity = 'Receive Payment' THEN oel.timestamp END) AS pay_ts,
    MIN(CASE WHEN oel.activity = 'Issue Invoice'   THEN oel.timestamp END) AS inv_ts,
    MIN(CASE WHEN oel.activity = 'Ship Goods'      THEN oel.timestamp END) AS ship_ts
FROM order_event_log oel
WHERE oel.activity IN ('Receive Payment','Issue Invoice','Ship Goods')
GROUP BY oel.case_id
HAVING
    (pay_ts < inv_ts) OR               -- paid before invoice
    (pay_ts < ship_ts);                -- paid before goods shipped
```

*What it shows*: any case where the payment timestamp precedes either the invoice or the shipment.  
*Relevance*: directly targets the anomaly in case1004 and can reveal H5 (asynchronous logging) or H2/H3 (policy overrides).

### 3.4.Check **resourcerole mismatches** (e.g., a FinanceTeam performing a logistics activity)

```sql
SELECT
    oel.case_id,
    oel.event_id,
    oel.activity,
    oel.resource,
    r.role,
    r.department
FROM order_event_log oel
JOIN resources r
  ON oel.resource = r.resource_id
WHERE
    (oel.activity IN ('Confirm Shipment','Ship Goods','Validate Stock')
     AND r.department <> 'Logistics')
    OR
    (oel.activity = 'Perform Credit Check' AND r.department <> 'Finance')
    OR
    (oel.activity IN ('Issue Invoice','Receive Payment') AND r.department <> 'Finance')
    OR
    (oel.activity = 'Register Order' AND r.department <> 'Sales')
ORDER BY oel.case_id, oel.event_id;
```

*What it shows*: events where the resource’s department does not match the expected department for that activity.  
*Relevance*: tests H6 (insufficient RBAC) and can also surface dataentry errors where the wrong `resource` ID was typed.

### 3.5.Inspect **suspicious flag values** that should block progression

```sql
SELECT

    oel.additional_info

    oel.activity = 'Confirm Shipment'

```

*What it shows*: shipment confirmations that were explicitly flagged as *not scheduled* or *late* yet still appear in the log.  
*Relevance*: speaks to H7 (flag ignored) and possibly H2/H3 (override of safety checks).

### 3.6.Optional: check **timestamp monotonicity** per case (to surface H4/H5)

```sql
SELECT
    case_id,
    event_id,
    activity,
    timestamp,
    LAG(timestamp) OVER (PARTITION BY case_id ORDER BY event_id) AS prev_ts,
    EXTRACT(EPOCH FROM (timestamp - LAG(timestamp) OVER (PARTITION BY case_id ORDER BY event_id))) AS diff_seconds
FROM order_event_log
WHERE
    LAG(timestamp) OVER (PARTITION BY case_id ORDER BY event_id) IS NOT NULL
    AND timestamp < LAG(timestamp) OVER (PARTITION BY case_id ORDER BY event_id)
ORDER BY case_id, event_id;
```

*What it shows*: any event whose timestamp is **earlier** than the previous event in the same case (possible clock skew or outoforder logging).  
*Relevance*: helps differentiate true process deviations (H1H3) from mere timestamp noise (H4/H5).

### 3.7.Combine several signals into a **caselevel risk score** (illustrative)

```sql
WITH flags AS (
    SELECT
        case_id,
        -- 1 if any predecessor rule broken
        bool_or(prev_activity IS DISTINCT FROM allowed_prev OR
                (prev_activity IS NULL AND allowed_prev IS NOT NULL)) AS bad_pred,
        -- 1 if missing credit/stock
        bool_or(gate.case_id IS NULL) AS missing_gate,
        -- 1 if payment before invoice/ship
        bool_or(pay_ts < inv_ts OR pay_ts < ship_ts) AS pay_early,
        -- 1 if wrong role/dept
        bool_or(
            (activity IN ('Confirm Shipment','Ship Goods','Validate Stock') AND r.department <> 'Logistics') OR
            (activity = 'Perform Credit Check' AND r.department <> 'Finance') OR
            (activity IN ('Issue Invoice','Receive Payment') AND r.department <> 'Finance') OR
            (activity = 'Register Order' AND r.department <> 'Sales')
        ) AS wrong_role,
        -- 1 if suspicious flag
        bool_or(
            (activity = 'Confirm Shipment' AND additional_info ILIKE '%shipment_scheduled=N%') OR
            (activity = 'Confirm Shipment' AND additional_info ILIKE '%late_confirmation=Y%')
        ) AS bad_flag
    FROM order_event_log oel
    LEFT JOIN (
        SELECT case_id
        FROM order_event_log
        WHERE activity IN ('Perform Credit Check','Validate Stock')
        GROUP BY case_id
    ) gate ON oel.case_id = gate.case_id
    LEFT JOIN resources r ON oel.resource = r.resource_id
    LEFT JOIN (
        SELECT
            case_id,
            MIN(CASE WHEN activity = 'Receive Payment' THEN timestamp END) AS pay_ts,
            MIN(CASE WHEN activity = 'Issue Invoice'   THEN timestamp END) AS inv_ts,
            MIN(CASE WHEN activity = 'Ship Goods'      THEN timestamp END) AS ship_ts
        FROM order_event_log
        WHERE activity IN ('Receive Payment','Issue Invoice','Ship Goods')
        GROUP BY case_id
    ) times ON oel.case_id = times.case_id
    GROUP BY oel.case_id
)
SELECT
    case_id,
    (bad_pred::int +
     missing_gate::int +
     pay_early::int +
     wrong_role::int +
     bad_flag::int) AS risk_score,
    bad_pred, missing_gate, pay_early, wrong_role, bad_flag
FROM flags
ORDER BY risk_score DESC, case_id;
```

*What it shows*: a simple additive score (05) that flags cases with multiple independent signals.  
*Relevance*: great for prioritising further investigation (e.g., manual review, audit, or rootcause workshops).
