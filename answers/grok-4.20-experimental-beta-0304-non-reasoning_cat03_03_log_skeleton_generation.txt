**Log Skeleton for the Order Fulfillment Process**

```python
log_skeleton = {
    'equivalence': {
        ('RO', 'IC'),      # Every order that is received must be invoiced
        ('CA', 'RI'),      # Every availability check is followed by a reservation (and vice versa)
        ('GSL', 'D')       # Every shipping label is followed by dispatch (and vice versa)
    },
    'always_before': {
        ('CA', 'RO'),      # Check Availability only after Receive Order
        ('RI', 'CA'),      # Reserve Items only after Check Availability
        ('PI', 'RI'),      # Pick Items only after Reserve Items
        ('QC', 'PI'),      # Quality Check only after Pick Items
        ('PO', 'QC'),      # Pack Order only after Quality Check
        ('GSL', 'PO'),     # Generate Shipping Label only after Pack Order
        ('D', 'GSL'),      # Dispatch only after Generate Shipping Label
        ('IC', 'PO'),      # Invoice Customer only after Pack Order (billing after packing)
        ('RP', 'IC')       # Record Payment only after Invoice Customer
    },
    'always_after': {
        ('RO', 'CA'),      # After receiving order, availability must be checked
        ('RI', 'PI'),      # After reserving, items must be picked
        ('PI', 'QC'),      # After picking, quality check must occur
        ('QC', 'PO'),      # After quality check, order must be packed
        ('PO', 'GSL'),     # After packing, shipping label must be generated
        ('GSL', 'D'),      # After generating label, dispatch must occur
        ('PO', 'IC'),      # After packing, customer must be invoiced
        ('IC', 'RP')       # After invoicing, payment must be recorded
    },
    'never_together': {
        # No mutually exclusive activities in this linear fulfillment flow
    },
    'activ_freq': {
        'RO': {1},      # Exactly once per case
        'CA': {1},
        'RI': {1},
        'PI': {1},
        'QC': {1},
        'PO': {1},
        'GSL': {1},
        'D': {1},
        'IC': {1},
        'RP': {0, 1}    # Payment may not have been received yet at log time
    },
    'directly_follows': {
        ('RO', 'CA'),   # Receive Order is directly followed by Check Availability
        ('CA', 'RI'),   # Check Availability  Reserve Items
        ('RI', 'PI'),   # Reserve Items  Pick Items
        ('PI', 'QC'),   # Pick Items  Quality Check
        ('QC', 'PO'),   # Quality Check  Pack Order
        ('PO', 'GSL'),  # Pack Order  Generate Shipping Label
        ('GSL', 'D'),   # Generate Shipping Label  Dispatch
        ('PO', 'IC'),   # Pack Order can also directly lead to invoicing (parallel branch)
        ('IC', 'RP')    # Invoice Customer  Record Payment
    }
}
```

**Explanation of design choices:**

- The main flow is mostly sequential, hence many `always_before` / `always_after` and several `directly_follows` constraints.
- `equivalence` is used for activities that must occur the same number of times (RO蜂C, CA乏I, GSL縫).
- `RP` is allowed to occur 0 or 1 time because payment may not have been received when the case is observed.
- No `never_together` constraints because all activities are part of the same overall process and can co-exist in a case.
