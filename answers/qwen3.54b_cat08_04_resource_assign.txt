# Process Mining Optimization Plan for TechSolve Solutions IT Service Desk

As a Process Improvement Consultant specializing in ITSM and Process Mining, I have analyzed the operational context of TechSolve Solutions. The provided event log snippet reveals critical inefficiencies in the resource assignment logic, particularly regarding skill matching and escalation timing. The following plan outlines a data-driven approach to diagnose current behaviors, identify bottlenecks, root-cause the inefficiencies, and implement optimized strategies.

## 1. Analyzing Resource Behavior and Assignment Patterns

To move beyond anecdotal evidence (e.g., "management suspects the logic is inefficient"), we must ground our analysis in the event log data using process mining techniques tailored for resource-constrained environments.

### Data Extraction and Metrics
We will extract specific metrics from the `Resource`, `Agent Tier`, `Agent Skills`, and `Ticket` attributes to build a baseline performance model:

*   **Individual Agent Performance:**
    *   **Throughput:** Number of tickets resolved per agent per hour (based on `Work Start` to `Work End`).
    *   **Utilization:** Ratio of active `Work Start` events to total assigned tickets.
    *   **Error Rate:** Frequency of tickets requiring reassignment after initial assignment.
*   **Tier-Level Metrics:**
    *   **First Call Resolution (FCR):** Percentage of tickets where `Work End` occurs within the same tier (L1) without an `Escalate` event.
    *   **Mean Time To Resolve (MTTR):** Average time from `Ticket Created` to `Work End`.
*   **Skill Utilization Analysis:**
    *   **Skill Coverage Rate:** Percentage of tickets assigned where the Agent's Skill Set overlaps with the Ticket's Required Skill.
    *   **Specialist Overuse:** Frequency of L2/L3 agents handling low-complexity (P4) tickets that could be handled by L1 or generalists.

### Process Mining Techniques for Assignment Patterns
*   **Resource Interaction Analysis:** We will analyze the sequence of events involving a specific `Agent ID` across different tiers. This reveals "social network" patterns—e.g., does Agent A05 frequently handover to Agent B12? Is this due to skill complementarity or confusion?
*   **Decision Mining (Process Decision Trees):** By comparing cases where the assignment was correct vs. incorrect, we can identify the decision points in the process. For example, looking at `Assign L1` events, we can compare tickets assigned to "Basic-Troubleshoot" agents vs. those requiring specific skills.
*   **Conformance Checking:** We will validate the actual flow against an ideal model (e.g., "If Required Skill = 'DB-SQL', Assign Tier = L2 or L3"). Deviations from this rule indicate assignment logic flaws.

**Application to Log Snippet:**
Analyzing **INC-1001**, we observe:
1.  `Ticket Created`: Requires `App-CRM`.
2.  `Assign L1`: Assigned to Agent A05 (`Basic-Troubleshoot`).
3.  **Finding:** There is a skill mismatch. The agent did not have the specific skill required for the ticket category. This likely caused the escalation to L2 later.

### Skill Utilization Analysis
We will calculate a **"Skill Match Score"** for every assignment event.
*   *Metric:* `(Number of Assigned Tickets with Perfect Skill Match) / (Total Assigned Tickets)`
*   *Insight:* If this score is low for L1, it indicates L1 agents are being assigned tickets beyond their capability, leading to the "escalation needed" notes seen in INC-1002.

## 2. Identifying Resource-Related Bottlenecks and Issues

Based on the analysis of resource behavior, we can pinpoint specific structural problems that are driving SLA breaches.

### Skill Scarcity and Mismatch Bottlenecks
*   **Issue:** The log shows INC-1001 being assigned to an L1 agent lacking `App-CRM` skills. This forces a reassignment to L2 (Agent B12).
*   **Identification:** We will calculate the **"Reassignment Risk Score"** for each skill category. If a ticket requires `Database-SQL` but is routed through L1 agents with no DB skills, it triggers a high-risk path.
*   **Quantification:** Determine the average delay introduced per reassignment (Time from `Assign L2` to `Work Start`). If this averages >30 minutes across P2/P3 tickets, it directly impacts SLA compliance.

### Delay from Unnecessary Reassignments
*   **Issue:** INC-1001 shows a reassignment at 11:15:00 due to "Needs different skill (DB)".
*   **Identification:** Use the `Reassign` activity type in the log. Correlate this with the priority (`P2`, `P3`).
*   **Quantification:** Calculate **"Mean Time To Reassignment"**. If P2 tickets spend >1 hour in "Queue/Assign" states, the bottleneck is not processing speed but assignment accuracy.

### Impact of Incorrect Initial Assignments
*   **Issue:** INC-1002 was assigned to A02 (`Basic-Troubleshoot`) for a `Networking-Firewall` ticket (INC-1002). While A02 eventually handled it, the log shows an escalation later.
*   **Identification:** Compare cases where the Dispatcher assigned based on Tier vs. Skill.
*   **Quantification:** Measure the **"L1 Resolution Efficiency"**. If L1 handles 40% of P3 tickets but only resolves 5%, the initial assignment logic is too aggressive for L1 capability.

### Underperforming or Overloaded Agents
*   **Issue:** Some agents appear overloaded while others are underutilized (e.g., A05 vs. B12 in the snippet).
*   **Identification:** Use **"Workload Distribution Analysis"**. Plot tickets assigned per hour against agent capacity.
*   **Quantification:** Identify agents with high "Ticket Age" (time since last resolution) or low "Active Work Start" counts despite having available slots.

### Correlation with SLA Breaches
*   **Issue:** High reassignment rates correlate with P2/P3 SLA breaches.
*   **Identification:** Join the Event Log with SLA data (Resolution Time vs. Deadline).
*   **Quantification:** Calculate **"SLA Breach Attribution Rate"**. E.g., "40% of P3 SLA breaches occurred because the ticket was reassigned at least once."

## 3. Root Cause Analysis for Assignment Inefficiencies

To fix the problem, we must determine *why* the current logic fails. We will use **Variant Analysis** and **Decision Mining**.

### Deficiencies in Current Assignment Rules
*   **Observation:** The log implies a mix of "Round-Robin within tiers" and "Manual escalation."
*   **Root Cause:** A rule like `IF Priority == P3 THEN Assign L1` exists, but it ignores `Required Skill`.
*   **Verification:** Decision Mining will show that the decision node for assignment often defaults to "Available Agent in Tier X" rather than "Agent with Skill Y".

### Inaccurate or Incomplete Agent Skill Profiles
*   **Observation:** INC-1001 was assigned to A05. A05's skill is `Basic-Troubleshoot`. Ticket needs `App-CRM`.
*   **Root Cause:** The agent profile might be outdated, or the dispatcher did not query the skill database before assigning.
*   **Verification:** Analyze cases where a ticket has `Required Skill` that matches *no one's* profile in L1. If this is common, the system lacks granularity.

### Poor Initial Ticket Categorization
*   **Observation:** INC-1001 Category is `Software-App`, Required Skill `App-CRM`.
*   **Root Cause:** The ticket might not be tagged correctly at creation. If the "Required Skill" field was empty or generic, the dispatcher couldn't route it properly.
*   **Verification:** Analyze cases where `Required Skill` is missing vs. present.

### Lack of Real-Time Visibility
*   **Observation:** INC-1001 had a delay ("Delay due to queue").
*   **Root Cause:** The dispatcher does not have access to real-time agent availability or queue depth, leading to sub-optimal assignment (assigning A05 instead of B12 who might be free).
*   **Verification:** Compare timestamps between `Assign` and `Work Start`. High variance suggests lack of real-time status updates.

### Insufficient L1 Empowerment
*   **Observation:** INC-1001 escalated after 20 minutes of work (`Work L1 End`).
*   **Root Cause:** L1 agents may not be trained to handle complex CRM issues, or the escalation criteria are too loose (e.g., "Any error" triggers escalation).
*   **Verification:** Compare cases where `Escalate` happened immediately vs. after prolonged troubleshooting (`Work Start` duration).

## 4. Developing Data-Driven Resource Assignment Strategies

Based on the analysis, we propose three concrete strategies to replace the current inefficient logic.

### Strategy 1: Intelligent Skill-Based Routing (Hard Skills First)
*   **Problem Addressed:** Skill mismatch causing reassignments and SLA delays (e.g., INC-1001).
*   **Insight Leveraged:** The analysis showed L1 agents often lack specific skills (`App-CRM`, `DB-SQL`).
*   **Data Required:** Real-time Agent Skill Set, Ticket Required Skills, Agent Availability.
*   **Implementation:**
    *   Modify the Dispatcher Logic to prioritize "Perfect Match" before "Tier Match".
    *   Rule: `IF Required Skill IN [Specialized Skills] THEN Assign L2/L3 OR Assign L1 with Training`.
    *   Fallback: Only assign L1 if no specialized agent is available and the task complexity allows (e.g., P4).
*   **Expected Benefit:** Reduction in reassignments by ~60%, improved First Contact Resolution for complex tickets.

### Strategy 2: Workload-Aware Assignment with Skill Constraints
*   **Problem Addressed:** Uneven workload distribution and queue delays.
*   **Insight Leveraged:** Analysis of agent utilization showed some agents (A05) were overused or underutilized due to lack of visibility.
*   **Data Required:** Current Queue Length per Agent, Ticket Type Distribution, Agent Workload History.
*   **Implementation:**
    *   Implement a **Weighted Round-Robin** algorithm.
    *   Weighting Factors: `Skill Match Score` + `Current Load`.
    *   Logic: If Agent A has 0 pending tickets and Skill Match = 1, assign to A. If Agent B has 2 pending tickets, skip B even if Skill Match = 1 (to prevent overload).
*   **Expected Benefit:** Reduced "Assign" to "Work Start" latency, more balanced team capacity.

### Strategy 3: Predictive Escalation and Fallback Routing
*   **Problem Addressed:** Excessive escalations for simple issues or delays in escalation (e.g., INC-1002).
*   **Insight Leveraged:** Historical data shows P3 tickets often require L2 intervention, but L1 resolution attempts are often short-lived if skills don't match.
*   **Data Required:** Ticket Resolution History, Time-to-Resolution per Skill Category.
*   **Implementation:**
    *   **Smart Escalation Trigger:** Instead of waiting for `Work End`, trigger escalation if:
        1.  No progress after 30 minutes (for P2/P3).
        2.  Ticket is reassigned to an agent without the required skill within the first hour.
    *   **Predictive Routing:** Use NLP on ticket descriptions (if available) or Category + Priority to pre-assign skills before assignment (e.g., `Software-App` + `P3` = Auto-tag `App-CRM`).
*   **Expected Benefit:** Faster resolution for complex tickets, reduced "Queue" time.

## 5. Simulation, Implementation, and Monitoring

### Process Business Simulation (Pre-Implementation)
Before rolling out new assignment logic, we must validate its impact using **Discrete Event Simulation (DES)**.

1.  **Model Construction:** Build a process model in the event log data that represents current rules. Add "Simulated Rules" for Strategy 1 and 2.
2.  **Parameterization:** Input historical agent capacity curves and ticket arrival rates.
3.  **Simulation Run:**
    *   Run the simulation on the past year of data (e.g., Q1-Q4 2024).
    *   Compare metrics: Total Tickets Processed, SLA Breach Count, Average Reassignment Time.
4.  **Outcome:** If Strategy 1 predicts a 5% increase in L2 workload but a 15% decrease in reassignments, it is viable. We can tune thresholds (e.g., only escalate if no match found).

### Implementation Plan
*   **Phase 1 (Data Cleaning):** Ensure the `Agent Skills` and `Required Skill` fields are populated accurately for all historical tickets.
*   **Phase 2 (Logic Update):** Deploy a rule engine or middleware script to replace the current "Dispatcher" logic with the new Weighted Skill-Based algorithm.
*   **Phase 3 (Training):** Train L1 agents on handling P3 tickets where they have partial skill overlap, reducing unnecessary escalations.

### Monitoring and Continuous Improvement (Post-Implementation)
We will maintain a **Process Mining Dashboard** to track effectiveness continuously.

**Key Resource-Related KPIs:**
1.  **Skill Match Rate:** % of assigned tickets where Agent Skill  Required Skill. Target: >90%.
2.  **Reassignment Frequency:** Count of `Reassign` events per ticket. Target: <5% for P3, <1% for P4.
3.  **Tier Resolution Efficiency:** % of tickets resolved within the assigned tier (FCR).
4.  **Agent Utilization Variance:** Standard deviation of workload across agents. Target: Low variance (<0.2).

**Process Views to Track:**
*   **Timeline View:** To check if SLA breaches are reducing over time.
*   **Resource Interaction Map:** To visualize if new routing creates "choke points" where specific agents (e.g., B15) become the bottleneck for DB skills.
*   **Decision Trace:** To ensure the new logic is being followed (e.g., did we skip L1 due to skill mismatch?).

By continuously mining the log against these KPIs, TechSolve can iterate on the assignment logic dynamically, ensuring resources are always allocated where they add maximum value.
