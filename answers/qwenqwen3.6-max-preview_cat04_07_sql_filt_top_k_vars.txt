Here is a complete, production-ready DuckDB SQL query that fulfills all requirements. It uses CTEs for clarity and leverages DuckDB's native support for ordered aggregation and nested types.

```sql
WITH case_sequences AS (
    -- 1. Build the ordered activity sequence per case
    SELECT 
        case_id,
        array_agg(activity ORDER BY timestamp, activity) AS variant_seq
    FROM event_log
    GROUP BY case_id
),
top_k_variants AS (
    -- 2. Count cases per variant and select the top K by frequency
    SELECT 
        variant_seq,
        COUNT(*) AS case_count
    FROM case_sequences
    GROUP BY variant_seq
    ORDER BY case_count DESC, variant_seq
    LIMIT K  --  Replace K with your desired number (e.g., 5)
),
qualifying_cases AS (
    -- 3. Extract case_ids that belong to the top K variants
    SELECT cs.case_id
    FROM case_sequences cs
    INNER JOIN top_k_variants tv ON cs.variant_seq = tv.variant_seq
)
-- 4. Return all original events for the qualifying cases
SELECT e.*
FROM event_log e
INNER JOIN qualifying_cases q ON e.case_id = q.case_id
ORDER BY e.case_id, e.timestamp;
```

###  Step-by-Step Breakdown
| CTE / Step | Purpose | DuckDB Feature Used |

| `case_sequences` | Aggregates activities per `case_id` in chronological order. Secondary sort by `activity` ensures deterministic ordering when timestamps tie. | `array_agg(... ORDER BY ...)` |
| `top_k_variants` | Groups cases by their exact sequence, counts frequencies, and keeps only the top `K`. | Native `GROUP BY` on arrays/lists |
| `qualifying_cases` | Maps the top `K` sequences back to the original `case_id`s. | Standard `INNER JOIN` on nested types |
| Final `SELECT` | Filters the raw event log to only include cases matching the top variants, preserving original event granularity. | Join pushdown & efficient filtering |

###  DuckDB-Specific Optimizations & Notes
1. **Native List/Array Grouping**: DuckDB treats arrays as first-class citizens. `GROUP BY variant_seq` works efficiently without needing string concatenation or hashing.
2. **Deterministic Ordering**: `ORDER BY timestamp, activity` inside `array_agg` prevents non-deterministic variant sequences when multiple events share the exact same timestamp.
3. **Performance Tip**: For very large logs, replace the final `INNER JOIN` with DuckDB's `SEMI JOIN`:
   ```sql
   SELECT e.* FROM event_log e SEMI JOIN qualifying_cases q ON e.case_id = q.case_id
   ```
   `SEMI JOIN` avoids duplicate row generation and is often faster in DuckDB's execution engine.
4. **Parameterizing `K`**: In DuckDB, you can make `K` dynamic without string replacement:
   ```sql

   -- Then use: LIMIT getvariable('k_val')
   ```
5. **NULL Handling**: If `activity` can contain `NULL`s, wrap it with `COALESCE(activity, '<<NULL>>')` inside `array_agg` to ensure consistent variant matching.
