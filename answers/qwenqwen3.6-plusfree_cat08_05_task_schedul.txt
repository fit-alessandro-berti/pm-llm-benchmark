# Advanced Process Mining & Data-Driven Scheduling Framework for Precision Parts Inc.

## 1. Analyzing Historical Scheduling Performance and Dynamics

To reconstruct and quantify shop-floor dynamics, I will implement a **multi-perspective process mining pipeline** that transforms raw MES event logs into structured, analyzable process models.

### Reconstruction of Actual Job & Task Flows
- **Case-Centric Reconstruction:** Group events by `Case ID (Job ID)` sorted by `Timestamp`. Extract the actual routing sequence by parsing `Queue Entry`  `Setup Start/End`  `Task Start/End` transitions. Handle deviations (skips, rework loops, ad-hoc routing) using **flexible process discovery algorithms** (e.g., Heuristic Miner or Inductive Miner with noise thresholds).
- **Resource-Centric Reconstruction:** Group events by `Resource (Machine ID)` to reconstruct execution sequences per workstation. This reveals the true chronological order of jobs processed on each machine, essential for analyzing contention, starvation, and setup dependencies.

### Specific Techniques & Metrics
| KPI / Dynamic | Process Mining Technique | Quantification Method |
|:---|:---|:---|
| **Job Flow/Lead/Makespan** | Performance overlay on Directly-Follows Graphs (DFGs); Case duration extraction | `Lead Time = Timestamp(Task End, Final Step) - Timestamp(Job Released)`. Analyze distributions (P50, P90, CV), not just averages. |
| **Task Waiting (Queue) Times** | Resource-centric log filtering; Timestamp differencing | `Queue Time = Timestamp(Setup Start or Task Start) - Timestamp(Queue Entry)`. Aggregate per resource and job type. |
| **Resource Utilization** | Resource perspective Petri nets; Gantt reconstruction from logs | `Utilization = (Actual Setup + Actual Task Duration) / Available Shift Time`. Segment into productive, idle/starvation, and setup time. |
| **Sequence-Dependent Setups** | Sequence mining; Transition matrix construction | Parse `Notes` or `Previous Job` fields to build `(Job_A  Job_B)` pairs. Calculate empirical setup durations per pair. Use Markov transition probabilities to model setup likelihoods. |
| **Schedule Adherence & Tardiness** | Conformance checking against due dates; Temporal alignment | `Tardiness = max(0, Completion Time - Due Date)`. Compute % on-time, mean absolute tardiness, and tardiness distribution by priority/routing. |
| **Disruption Impact** | Event correlation analysis; Causal process mining | Tag `Breakdown` and `Priority Change` events. Measure downstream queue inflation, delay propagation depth (how many subsequent jobs are pushed), and recovery time to baseline throughput. |

**Supporting PM Techniques:**

- **Conformance Checking:** Align actual traces against idealized routing templates to quantify deviations, rework loops, and unauthorized bypasses.

## 3. Root Cause Analysis of Scheduling Ineffectiveness

Process mining enables rigorous separation of **scheduling logic failures** from **capacity constraints** and **process variability**.

### Potential Root Causes
1. **Static Dispatching Limitations:** FCFS/EDD ignore real-time queue states, downstream bottlenecks, and setup penalties.
2. **Lack of Real-Time Visibility:** Schedulers react to outdated snapshots; MES logs show delayed priority updates and breakdowns propagating unchecked.
3. **Inaccurate Estimations:** `Task Duration (Planned)` vs `(Actual)` shows systematic bias. Variance correlates with `Operator ID` or material properties, not accounted for in planning.
4. **Ineffective Setup Handling:** No sequencing logic minimizes changeovers. High-mix environment treated as random arrivals.
5. **Poor Disruption Response:** `Priority Change` and `Breakdown Start` events trigger ad-hoc resequencing, causing schedule nervousness and WIP spikes.

### Differentiating Scheduling vs. Capacity/Variability Issues
| Diagnostic | Scheduling Logic Issue | Capacity/Process Variability Issue |
|:---|:---|:---|
| **Utilization vs. Queue Time** | Low/Moderate utilization but high queue times (poor prioritization/flow) | High utilization (>90%) + long queues (true capacity constraint) |
| **Duration Variance** | Actual  Planned, but completion misses due dates (sequencing/timing logic) | Actual >> Planned, high operator/machine-specific variance (process variability) |
| **Setup Patterns** | High setup times despite similar jobs queued (ignores sequence dependency) | Setup times inherently long/unpredictable regardless of sequence (tooling/process limit) |
| **Conformance to Routing** | High deviation from planned routing (expediting, rework loops) | Routing stable, but timing slips (variability or capacity) |

**PM Approach:** Use **feature importance modeling** (e.g., Random Forest trained on log-derived features predicting tardiness). If `Queue Position`, `Dispatch Rule Tag`, and `Setup Sequence` dominate, it's scheduling. If `Breakdown Frequency`, `Operator Skill`, and `Material Type` dominate, it's capacity/variability. **Queueing theory validation** (Little's Law: `WIP = Throughput × Lead Time`) further isolates whether delays stem from flow control or physical constraints.

- **Expected Impact:**  Schedule reliability (P90 lead time  25%),  Hot job integration without cascading delays,  Tardiness variance, improved customer quoting accuracy.
