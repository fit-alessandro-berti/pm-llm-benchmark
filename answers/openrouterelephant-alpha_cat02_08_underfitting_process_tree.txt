The process tree is:  

```
SEQUENCE
 Receive_Order
 Check_Inventory

    PARALLEL (Authorize_Payment, Package_Items, Ship_Order)
    SEQUENCE (Cancel_Order, Close_Order)

```

LetÆs analyze the possible traces.

### 1. Identify a problematic trace

The **XOR** means either:
- **Branch A**: all three of Authorize_Payment, Package_Items, Ship_Order happen (in parallel, so order among them not strictly enforced by the process tree, but they can occur in any interleaving as long as all three occur), **or**
- **Branch B**: Cancel_Order then Close_Order.

After the XOR, there is another **Close_Order** in the sequence.

So possible traces:

**From Branch A path**:  
Receive_Order  Check_Inventory  (any interleaving of Authorize_Payment, Package_Items, Ship_Order)  Close_Order  

**From Branch B path**:  
Receive_Order  Check_Inventory  Cancel_Order  Close_Order  Close_Order  

Wait, note: in Branch B, after Cancel_Order and Close_Order (from the XOR), the sequence continues with another Close_Order. That means Close_Order appears **twice** in the trace if the cancel branch is taken.  

But maybe the model intends that after the XOR, the final Close_Order is only executed if not already done? However, in process mining, a trace is a sequence of activities, and if Close_Order appears twice, itÆs still a valid trace under the model.

LetÆs pick a trace from Branch A that is problematic.

Consider:  
**Receive_Order, Check_Inventory, Ship_Order, Close_Order**

This trace has Ship_Order without Package_Items and without Authorize_Payment.  

But is it allowed? In the parallel XOR subtree, all three must occur, but the process tree parallel allows any interleaving, and in process trees, a parallel means all children must complete, but they can happen in any order and possibly with other things in between? Actually, in process trees, the parallel operator means all children must be executed, but not necessarily at the same time Ś they can be interleaved. However, in terms of a trace, a **visible trace** from a parallel is any sequence that contains all the labels of the children, in any order, with possible *tau* (invisible) steps, but here all are visible.  

But wait Ś in PM4PyÆs process tree semantics, a parallel means the activities can happen in any order, but **all must happen**. So a trace missing one of them is **not** allowed.  

So my earlier thought about Ship_Order alone is wrong Ś the model actually forces all three (Authorize_Payment, Package_Items, Ship_Order) to occur if the successful branch is taken.  

So maybe the problematic trace is one where the **order is wrong** even though all three are present.  

But the model doesnÆt enforce order among the three in the parallel Ś so from domain perspective, you cannot ship before authorizing payment or packaging.  

So a trace like:  
**Receive_Order, Check_Inventory, Ship_Order, Authorize_Payment, Package_Items, Close_Order**  

This trace contains all three activities of the parallel, so the model accepts it (since parallel only cares about set of activities, not order). But in reality, you cannot ship before authorizing payment or packaging.  

This is a good example of underfitting Ś the model allows an invalid order.

---

### 2. Why is this problematic?

From domain knowledge in Order-to-Cash:  
- **Authorize Payment** must happen before **Ship Order** (you canÆt ship without payment approval).  
- **Package Items** must happen before **Ship Order** (you canÆt ship unpackaged items).  

The process treeÆs parallel operator does not enforce these ordering constraints, so the model permits traces where Ship_Order occurs before Authorize_Payment or before Package_Items, which are logically invalid in the real world.  

This is the underfitting problem: the model is too permissive and allows behavior that violates business rules.
