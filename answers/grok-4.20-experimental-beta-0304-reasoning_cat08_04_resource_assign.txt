**Process Mining-Driven Optimization of Resource Assignment in TechSolve's IT Service Desk**

### 1. Analyzing Resource Behavior and Assignment Patterns

I would begin by importing the full year-long event log into a process mining tool (e.g., Celonis, Disco, PM4Py, or Apromore) and enriching it with derived attributes: skill match (binary or weighted score between "Required Skill" and "Agent Skills"), reassignment count per case, escalation depth, and resolution tier.

**Key metrics to extract per agent and tier:**
- **Workload distribution**: Cases handled, active time (sum of Work Start to Work End durations), and concurrent load (using dotted chart or interval analysis to detect overlapping activities).
- **Processing time**: Median and 90th percentile duration of "Work X Start" to "Work X End" segmented by tier, priority, and required skill. This reveals whether certain L1 agents are faster on P3 Software-App tickets but slower on Network tickets.
- **First-Contact Resolution (FCR) rate for L1**: Percentage of tickets resolved without escalation, broken down by category and required skill.
- **Skill utilization effectiveness**: Frequency each agent handles their documented skills vs. "below-grade" tasks. For example, how often Agent B12 (App-CRM, DB-SQL) is assigned basic troubleshooting versus specialists being pulled into L1-type work.
- **Escalation and reassignment frequency**: Percentage of tickets escalated from L1 to L2/L3 and intra-tier reassignments, correlated with initial assignment method (round-robin, self-assigned, dispatcher).

**Process mining techniques:**
- **Resource interaction analysis and social network analysis (SNA)**: Generate a handover network showing frequency of transfers between specific agents (L1  L2 handovers). High-density edges between certain L1 agents and specific L2 specialists indicate informal "favorite" routing rather than systematic logic.
- **Role discovery**: Apply clustering (e.g., based on activity profiles or skill usage vectors) to discover *de facto* roles. This often reveals that the official L1/L2/L3 tiers do not match reality耀ome "L1" agents behave like L2 specialists, while some L2 agents are frequently assigned low-complexity tickets.
- **Performance analysis on discovered process maps**: Filter the log by resource and priority to create variant-specific models. Compare the *intended* model (round-robin within tier + manual escalation on skill mismatch) against the *actual* model using conformance checking. Deviations will highlight systematic bypassing of the dispatcher, self-assignment of high-priority tickets by certain L1 agents, and recurring reassignment loops (e.g., L2  L2 when skill mismatch is identified late).

Skill utilization analysis would use a skill matrix heatmap: rows as agents, columns as required skills, cells showing frequency and average resolution time. This typically exposes that highly skilled L2/L3 agents spend 25-40% of their time on tickets that historically could have been resolved by specific L1 agents with partial skill overlap.

### 2. Identifying Resource-Related Bottlenecks and Issues

Using the metrics above, I would perform root-cause filtering and correlation analysis:

- **Skill availability bottlenecks**: Calculate "skill wait time" as the duration between "Escalate/Assign" and "Work Start" when the required skill is rare. Cross-reference with agent availability (derived from active timestamps). Bottlenecks appear as long queues for "Networking-Firewall" or "Security-IAM" specialists.
- **Reassignment and escalation delays**: Measure the average additional cycle time per reassignment (typically 2-6 hours based on similar ITSM logs). Identify "reassignment loops" where the same ticket bounces between 2-3 L2 agents.
- **Incorrect initial assignment impact**: Compare cases where the initial L1 assignment matched the required skill vs. those that did not. Skill-mismatched initial assignments show significantly higher escalation rates and SLA breach probability.
- **Overloaded vs. underutilized agents**: Use workload histograms and Gini coefficient for workload inequality across the agent pool. Identify "hero agents" (top 20% handling 50%+ of high-priority tickets) and "idle specialists."
- **SLA correlation**: Perform statistical tests (e.g., chi-square or logistic regression) between assignment patterns and SLA breaches. In similar projects, 35-55% of P2/P3 SLA breaches are directly attributable to either skill mismatch on initial assignment or excessive handovers.

Quantification examples from typical analysis of such logs:
- Average delay per reassignment: +4.2 hours
- Percentage of SLA breaches linked to skill mismatch on first assignment: 42%
- Percentage of L2 specialist time spent on tasks solvable by top 30% of L1 agents: 31%

### 3. Root Cause Analysis for Assignment Inefficiencies

Potential root causes, validated through process mining:

- **Deficient assignment rules**: Round-robin ignores both skill and current workload, leading to frequent reassignments. Decision mining (using attributes like Ticket Priority, Category, Required Skill, Agent Current Load, Time of Day) typically shows that the current dispatcher logic has very low predictive power for successful first assignment.
- **Inaccurate skill profiles**: Many agents have outdated or incomplete skill registers. By comparing "Agent Skills" field against actual successful resolutions, we can quantify skill profile drift.
- **Poor initial categorization**: Ticket Category and Required Skill are often set generically at creation ("Software-App" instead of "App-CRM" or "Database-SQL"), causing downstream mismatch. Variant analysis of "smooth" cases (no reassignment, SLA compliant) vs. "problematic" cases (multiple reassignments) reveals distinct patterns in initial attributes.
- **Lack of real-time visibility**: Dispatchers and L1 agents have no dashboard showing current agent workload or queue per skill.
- **L1 capability gap**: Low FCR rate for specific categories indicates insufficient training or decision support for L1 agents.

**Analytical techniques**:
- **Variant analysis**: Extract the top 10 variants with highest reassignment counts and compare them to the top 10 "happy path" variants. Differences in ticket attributes and resource paths reveal decision points that lead to failure.
- **Decision mining**: Build decision trees at the "Assign L1" and "Escalate" activities using case attributes. This often identifies that certain categories are almost always escalated by specific L1 agents regardless of ticket complexity用ointing to training or confidence issues rather than technical difficulty.

### 4. Developing Data-Driven Resource Assignment Strategies

**Strategy 1: Skill- and Proficiency-Based Routing (Addresses skill mismatch and specialist overuse)**  
Leverage the skill utilization heatmap and historical success rates to create a weighted matching score (skill match ﾗ proficiency level ﾗ historical resolution speed for that skill). Replace pure round-robin with a priority queue that scores available agents in the appropriate tier.  
*Data required*: Maintained skill matrix (with proficiency scores 1-5 derived from historical performance), real-time agent availability.  
*Expected benefits*: 25-40% reduction in reassignments, 15-20% improvement in L1 FCR for matched skills, reduced L2/L3 workload on basic tickets.

**Strategy 2: Workload-Aware + Skill-Aware Assignment Algorithm (Addresses uneven distribution and bottlenecks)**  
Implement a composite scoring function: `Score = (Skill_Match ﾗ 0.6) + (Workload_Balance_Factor ﾗ 0.3) + (Urgency_Weight ﾗ 0.1)`. Use current queue length per skill and individual agent utilization (calculated over last 4 hours). Introduce "skill pools" with dynamic thresholds.  
*Data required*: Real-time event data feed for workload calculation, skill requirements predicted at ticket creation.  
*Expected benefits*: More even workload distribution (target Gini < 0.35), reduction in peak overload situations, 10-18% faster resolution for P2 tickets.

**Strategy 3: Predictive Skill and Complexity Assignment at Creation (Addresses poor initial categorization)**  
Use historical data to train a model (random forest or transformer on ticket description + category) that predicts required skill(s), complexity tier, and probability of L1 resolution at ticket creation time. Route directly to the best available agent or enriched L1 pool.  
*Data required*: Ticket description text, historical resolution outcomes, agent performance vectors.  
*Expected benefits*: Earlier correct routing (reducing average time to correct agent by 1-3 hours), improved L1 empowerment through "suggested next step" based on similar past cases, 20-30% reduction in unnecessary escalations.

### 5. Simulation, Implementation, and Monitoring

**Simulation**: I would discover a comprehensive process model with resource characteristics (processing time distributions per activity-skill-resource combination, arrival rates by priority/category, and current assignment rules). Using discrete-event simulation (e.g., in ProM, Simod, or Arena with process mining extensions), I would create "as-is" and "to-be" scenarios for the three strategies. Key inputs include actual inter-arrival times, skill distributions, and agent calendars extracted from the log. Output metrics would include SLA compliance rate, average resolution time, reassignment rate, and resource utilization. This typically shows 18-35% improvement in SLA adherence with the combined strategies, allowing prioritization and risk mitigation before rollout.

**Post-Implementation Monitoring Plan**:
Deploy a process mining dashboard with continuous log ingestion. Track these KPIs in real time and weekly trend views:
- Primary: Reassignment rate, SLA compliance by priority and category, overall FCR rate by tier.
- Resource-specific: Skill utilization heatmap (updated weekly), workload balance index, top overloaded/underutilized agents, average wait time for critical skills.
- Process views: Conformance to new assignment model, variant comparison (new vs. old), handover network evolution, bottleneck drill-down by skill.
- Leading indicators: Skill match rate on first assignment, predictive model accuracy, L1 resolution probability score distribution.

Review gates would be set at 30, 60, and 90 days, with automated alerts for deviation >10% from simulated benefits. The dashboard would also include agent-specific scorecards to support coaching and skill profile updates.

This approach transforms the current reactive, rule-of-thumb assignment logic into a continuously improving, data-driven system that respects agent capabilities while maximizing SLA performance and operational efficiency.
