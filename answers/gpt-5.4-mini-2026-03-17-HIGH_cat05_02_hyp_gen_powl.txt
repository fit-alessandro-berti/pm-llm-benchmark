## 1) Anomalies in the POWL model

### A. Loop around `E` and `P`
The model places **Evaluate (`E`)** and **Approve (`P`)** inside a loop.

**Why this is anomalous:**
- In a normal claim process, evaluation and approval are usually **single-step** activities, or at most evaluation may repeat after rework.
- A loop between `E` and `P` suggests:
  - repeated approvals,
  - repeated re-evaluations,
  - or a rework cycle that may not be intended in the ideal process.

### B. XOR between `N` and `skip`
The model allows either:
- **Notify Customer (`N`)**, or
- **do nothing** (`skip`).

**Why this is anomalous:**
- In the intended process, notification is a standard step.
- Making it optional means customer notification can be omitted entirely, which is a business-rule deviation unless there is a legitimate exception.

### C. Partial order allows premature closure `C`
The model only enforces:
- `R -> A`
- `A -> loop`
- `loop -> xor`
- `A -> C`

There is **no strict order from the evaluation/approval/notification branch to closing**.

**Why this is anomalous:**
- Since `C` depends only on `A`, the claim may be closed **before**:
  - evaluation finishes,
  - approval occurs,
  - notification occurs.
- This is the biggest process-control anomaly in the model.

## 2) Hypotheses for why these anomalies exist

### Hypothesis 1: Business rules changed, but the model was only partially updated
Examples:
- Rework was introduced for complex claims, but the final closure constraint was not updated.
- Notification became optional for some claim types, but the process model still treats it as part of the general flow.

### Hypothesis 2: Miscommunication between teams
Examples:
- The operations team believes notification is mandatory.
- The process/design team encoded it as optional due to incomplete requirements.

### Hypothesis 3: Workflow system or modeler allowed unsafe partial orders
Examples:
- The modeling tool may permit overly loose ordering constraints.
- A bug or incorrect translation from BPMN/tree semantics to POWL may have introduced concurrency that should not exist.

### Hypothesis 4: The loop is a workaround for real operational rework
Examples:
- Claims may be sent back for re-evaluation if documents are missing.
- Approvals may need to be repeated after escalation.
- The loop may reflect a real exception-handling process, but it should probably be modeled more explicitly.

### Hypothesis 5: Controls were intentionally relaxed for speed
Examples:
- Claims can be closed early in special cases.
- Notification may be skipped for internal closures, rejected claims, or fraud cases.
- This could be a policy exception rather than a modeling error.

## 3) How to verify these hypotheses using the database

You can use `claim_events` to reconstruct the real traces and compare them to the ideal flow.

### A. Reconstruct full event traces per claim
This helps you see whether the actual paths contain the anomalies.

```sql
SELECT
    claim_id,
    string_agg(activity, ' -> ' ORDER BY "timestamp", event_id) AS trace
FROM claim_events
GROUP BY claim_id
ORDER BY claim_id;
```

---

### B. Find claims closed without evaluation or approval
This checks whether `C` occurs when `E` or `P` is missing, or whether closure happened before those steps.

```sql
WITH first_events AS (
    SELECT
        claim_id,
        MIN("timestamp") FILTER (WHERE activity = 'E') AS eval_ts,
        MIN("timestamp") FILTER (WHERE activity = 'P') AS approve_ts,
        MIN("timestamp") FILTER (WHERE activity = 'N') AS notify_ts,
        MIN("timestamp") FILTER (WHERE activity = 'C') AS close_ts
    FROM claim_events
    GROUP BY claim_id
)
SELECT
    c.claim_id,
    c.claim_type,
    f.eval_ts,
    f.approve_ts,
    f.notify_ts,
    f.close_ts
FROM claims c
JOIN first_events f USING (claim_id)
WHERE f.close_ts IS NOT NULL
  AND (
       f.eval_ts IS NULL
    OR f.approve_ts IS NULL
    OR f.close_ts < f.eval_ts
    OR f.close_ts < f.approve_ts
  )
ORDER BY f.close_ts;
```

### What this tells you
- If many claims are closed without `E` or `P`, the premature-close anomaly is real.
- If this happens mostly after a certain date, it may support a business-rule change or system rollout issue.

---

### C. Find claims with multiple approvals
This tests whether the loop is actually occurring in practice.

```sql
SELECT
    claim_id,
    COUNT(*) AS approval_count,
    array_agg("timestamp" ORDER BY "timestamp", event_id) AS approval_times
FROM claim_events
WHERE activity = 'P'
GROUP BY claim_id
HAVING COUNT(*) > 1
ORDER BY approval_count DESC, claim_id;
```

### What this tells you
- Multiple approvals could indicate re-approval, escalation, or an unintended loop.
- If repeated approvals happen mostly for specific claim types, that may suggest a real rework rule.

---

### D. Check whether customer notification is skipped
This verifies the XOR/skip behavior.

```sql
WITH notif AS (
    SELECT
        claim_id,
        MIN("timestamp") FILTER (WHERE activity = 'N') AS notify_ts,
        MIN("timestamp") FILTER (WHERE activity = 'C') AS close_ts
    FROM claim_events
    GROUP BY claim_id
)
SELECT
    c.claim_type,
    COUNT(*) AS closed_claims,
    COUNT(*) FILTER (WHERE n.notify_ts IS NULL) AS closed_without_notification,
    COUNT(*) FILTER (WHERE n.notify_ts IS NOT NULL AND n.close_ts < n.notify_ts) AS notified_after_close,
    ROUND(
        100.0 * COUNT(*) FILTER (WHERE n.notify_ts IS NULL) / NULLIF(COUNT(*), 0),
        2
    ) AS pct_without_notification
FROM claims c
JOIN notif n USING (claim_id)
WHERE n.close_ts IS NOT NULL
GROUP BY c.claim_type
ORDER BY closed_claims DESC;
```

### What this tells you
- If a large fraction of closed claims have no `N`, then notification is frequently skipped in practice.
- If notification happens after closure, that may indicate a system timing issue or poor sequencing.

---

### E. Look for claims where closure happens before notification
This is a more direct sequencing check.

```sql
WITH first_events AS (
    SELECT
        claim_id,
        MIN("timestamp") FILTER (WHERE activity = 'N') AS notify_ts,
        MIN("timestamp") FILTER (WHERE activity = 'C') AS close_ts
    FROM claim_events
    GROUP BY claim_id
)
SELECT *
FROM first_events
WHERE close_ts IS NOT NULL
  AND (notify_ts IS NULL OR close_ts < notify_ts);
```

### F. Segment anomalies by claim type, date, or region
This helps test whether the anomalies are tied to a business-rule change or operational group.

#### By claim type
```sql
WITH anomaly_flags AS (
    SELECT
        claim_id,
        MIN("timestamp") FILTER (WHERE activity = 'E') AS eval_ts,
        MIN("timestamp") FILTER (WHERE activity = 'P') AS approve_ts,
        MIN("timestamp") FILTER (WHERE activity = 'N') AS notify_ts,
        MIN("timestamp") FILTER (WHERE activity = 'C') AS close_ts
    FROM claim_events
    GROUP BY claim_id
)
SELECT
    c.claim_type,
    COUNT(*) AS claims,
    COUNT(*) FILTER (
        WHERE close_ts IS NOT NULL
          AND (eval_ts IS NULL OR approve_ts IS NULL OR close_ts < eval_ts OR close_ts < approve_ts)
    ) AS closure_anomalies,
    COUNT(*) FILTER (
        WHERE close_ts IS NOT NULL
          AND notify_ts IS NULL
    ) AS missing_notifications
FROM claims c
JOIN anomaly_flags a USING (claim_id)
GROUP BY c.claim_type
ORDER BY claims DESC;
```

#### By adjuster region or specialization
If `claim_events.resource` stores adjuster names, you can join to `adjusters`:

```sql
SELECT
    a.region,
    a.specialization,
    COUNT(DISTINCT ce.claim_id) AS claims_handled
FROM claim_events ce
JOIN adjusters a
  ON ce.resource = a.name
WHERE ce.activity IN ('E', 'P')
GROUP BY a.region, a.specialization
ORDER BY claims_handled DESC;
```

If anomalies cluster in one region or specialization, that supports a training/process issue rather than a global model issue.

- **Claims closed before `E` or `P`**  the partial order is too permissive, or the system is allowing invalid execution.
- **Anomalies concentrated after a certain date**  likely a business-rule change or system deployment issue.
