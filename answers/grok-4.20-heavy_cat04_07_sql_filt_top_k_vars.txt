```sql
WITH case_variants AS (
    SELECT 
        case_id,
        list_agg(activity ORDER BY "timestamp") AS variant
    FROM event_log
    GROUP BY case_id
),
variant_frequencies AS (
    SELECT 
        variant,
        COUNT(*) AS frequency
    FROM case_variants
    GROUP BY variant
),
top_k_variants AS (
    SELECT variant
    FROM variant_frequencies
    ORDER BY frequency DESC
    LIMIT ?  -- bind/parameter for K (or replace with number)
)
SELECT 
    el.*
FROM event_log el
INNER JOIN case_variants cv 
    ON el.case_id = cv.case_id
WHERE cv.variant IN (SELECT variant FROM top_k_variants)
ORDER BY el.case_id, el."timestamp";
```

**Explanation of the query**:
- `case_variants` CTE builds the exact ordered activity sequence per `case_id` using `list_agg(... ORDER BY "timestamp")` (DuckDB preserves order in the aggregate).
- `variant_frequencies` groups identical sequences (i.e., variants) and counts their case frequency.
- `top_k_variants` selects only the top K most frequent variants (via `LIMIT ?` — the `?` is a bind parameter for K in prepared statements; replace with a literal like `5` if needed).
- Final `SELECT` joins back to keep *all original events* (all columns) only for cases whose variant is among the top K; results are ordered for readability.
- Uses `"timestamp"` (quoted) because `timestamp` is a reserved word/datatype in SQL. Lists are natively supported in DuckDB for grouping/joining/comparison. This satisfies all requirements in a single query.
