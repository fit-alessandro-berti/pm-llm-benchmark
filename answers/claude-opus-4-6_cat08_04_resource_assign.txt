## 1. Analyzing Resource Behavior and Assignment Patterns

### 1.1 Agent and Tier Performance Metrics

The event log provides a rich foundation for constructing a multi-dimensional performance profile of every agent and tier. I would extract and compute the following metrics systematically:

**Workload Distribution Metrics:**
- **Ticket volume per agent per time unit** (daily, weekly, monthly): By filtering the event log on the "Assign" activities and grouping by resource, I can construct workload histograms. Comparing these across agents within the same tier reveals imbalance. For instance, if Agent B12 handles 45 tickets per week while Agent B15 handles 18 (both L2), the round-robin is clearly failing or being overridden by manual decisions.
- **Concurrent ticket load**: By identifying tickets in "Work Start" without a corresponding "Work End" at any point in time, I can compute the number of simultaneously open tickets per agent. This reveals real-time overload conditions.
- **Workload coefficient of variation** within each tier: A high CV (>0.5) across agents in the same tier signals systematic imbalance rather than random fluctuation.

**Processing Time Metrics:**
- **Activity processing time per agent/tier**: The difference between "Work Start" and "Work End" timestamps gives the active handling time. I would compute distributions (median, 75th percentile, 95th percentile) per agent, segmented by ticket priority and category. This reveals not only speed but consistency.
- **Queue waiting time**: The gap between "Assign" and "Work Start" indicates how long a ticket sits before an agent begins work. From the snippet, INC-1001 was assigned to B12 at 09:40:15 but work started at 10:05:50 — a 25-minute queue delay. Aggregating this by tier, time-of-day, and agent reveals queue bottlenecks.
- **End-to-end resolution time** per ticket, decomposed into: (a) L1 handling time, (b) escalation delay, (c) assignment delay at L2/L3, (d) L2/L3 handling time, (e) any reassignment loops. This decomposition, achievable through timestamp arithmetic in the event log, pinpoints where time is consumed.

**First-Call Resolution (FCR) for L1:**
- I would identify all tickets where the process variant is: Ticket Created  Assign L1  Work L1 Start  Work L1 End  Resolved (without any "Escalate" activity). The FCR rate = (tickets resolved at L1) / (total tickets touched by L1). Breaking this down by ticket category, priority, and individual L1 agent reveals which agents are effective at resolving which types, and which ticket types are systematically beyond L1 capability.
- **Conditional FCR**: FCR by ticket category reveals, for example, that "Access Management" tickets have 78% FCR at L1 while "Network" tickets have only 12%. This informs what L1 should even attempt versus immediately escalate.

**Skill Utilization Frequency:**
- By joining agent skill attributes with ticket required-skill attributes, I can compute how often each agent works on tickets matching their documented skills versus those that don't. If Agent B12 (skills: App-CRM, DB-SQL) is frequently assigned tickets requiring "Networking-Firewall," this is a systematic mismatch.

### 1.2 Uncovering Actual Assignment Patterns Through Process Mining

**Process Discovery for Resource Flow:**

I would apply process discovery algorithms (e.g., Inductive Miner, Heuristics Miner) not just on the control-flow level but specifically on the **resource perspective**. This involves:

- **Handover-of-Work Social Network**: Constructing a directed graph where nodes are agents (or agent groups/tiers) and edges represent ticket handovers. The edge weight represents frequency. From the event log, every time a ticket transitions from one resource to another (e.g., Agent A05  Dispatcher  Agent B12 for INC-1001), an edge is created. This network reveals:
  - Which L1 agents escalate most frequently and to whom.
  - Whether certain L2 agents receive a disproportionate share of escalations.
  - Circular handover patterns (e.g., L2  L2 reassignment) indicating ping-ponging.
  - Direct L1L3 escalations that bypass L2, suggesting L2 skill gaps.

- **Organizational Mining / Role Discovery**: Using clustering algorithms on the activity profiles of agents (what activities they perform and how frequently), I can discover *actual roles* as opposed to *assigned tiers*. For example, it might emerge that certain L2 agents effectively function as L1 agents (handling simple tickets) while some L1 agents are performing L2-level troubleshooting. This comparison between discovered roles and formal tier assignments reveals structural misalignment.

- **Resource-Activity Matrix**: A heatmap showing which resources perform which activities, with frequency and performance overlay. This instantly shows if L3 specialists are performing "Work L1" activities or if dispatchers are making reassignment decisions at unexpectedly high rates.

**Comparing Actual vs. Intended Logic:**

The intended logic is round-robin within tiers with manual escalation decisions. I would:
- Test the round-robin hypothesis by examining assignment sequences within each tier. If round-robin were followed, the sequence of agents assigned within a time window should cycle evenly. Computing the Gini coefficient of assignments per agent per week can quantify deviation from uniform distribution. A Gini coefficient approaching 0 means perfect equality (consistent with round-robin); values approaching 1 indicate severe concentration.
- Map the escalation decision points: For each escalation event, I would record the ticket attributes (category, priority, keywords in notes) and the subsequent assignment. Decision mining (see Section 3) can reveal whether escalation decisions follow any consistent logic or are essentially ad hoc.

### 1.3 Skill Utilization Analysis

**Skill Match Rate**: For every "Assign" event, compare `Required Skill` of the ticket with `Agent Skills` of the assigned agent:
- **Full Match**: The required skill is in the agent's skill set.
- **Partial Match**: The agent has a related skill (e.g., agent has "App-ERP" and ticket needs "App-CRM" — same domain but different application).
- **No Match**: Complete mismatch.

I would compute this rate across all assignments and segment by tier, priority, and time. A low skill match rate at initial assignment directly drives reassignments.

**Skill Demand vs. Supply Analysis**: Aggregate the frequency of each `Required Skill` across all tickets and compare against the number of agents possessing that skill and their available capacity. If "Networking-Firewall" is required by 300 tickets/month but only 2 agents have that skill (with capacity for ~100 tickets/month combined), this is a structural bottleneck.

**Over-Qualification Analysis**: Track cases where L2/L3 specialists with advanced skills (e.g., "Security-IAM", "Database-SQL") are assigned tickets that were ultimately resolved using only basic troubleshooting — identifiable when the resolution notes or activity pattern indicate simple fixes (e.g., password resets, configuration checks). The percentage of specialist time spent on below-skill-level tasks quantifies waste. From the scenario description, specialists themselves report this, and the data can validate and quantify it.

## 2. Identifying Resource-Related Bottlenecks and Issues

### 2.1 Skill-Based Bottlenecks

**Method**: I would construct a **skill demand-supply heatmap** by time period. For each week or day, I compute:
- Incoming ticket volume requiring each skill.
- Available agent-hours with that skill (accounting for shifts, leave, and existing queue load).
- The resulting gap or surplus.

**Expected Finding**: Skills like "Networking-Firewall" or "Security-IAM" likely show chronic deficit — the demand consistently exceeds supply. This manifests in the event log as:
- Long queue waiting times for tickets requiring these skills (measurable from the gap between "Assign L2" and "Work L2 Start").
- Higher SLA breach rates for tickets in these skill categories.

**Quantification**: I would compute the **average queue time by required skill** and contrast it against the SLA clock. For example: "Tickets requiring 'Networking-Firewall' experience an average L2 queue wait of 4.2 hours, compared to 1.1 hours for 'App-CRM' tickets. This alone consumes 52% of the P2 SLA window."

### 2.2 Reassignment and Escalation Delays

**Method**: I would identify all "Reassign" events in the log and compute:
- **Reassignment frequency**: Number of reassignments per ticket (a ticket with >1 reassignment is problematic; >2 is critical).
- **Average delay per reassignment**: The time from "Reassign" to the next "Work Start" event. Each reassignment adds queue time, context-switching cost, and re-investigation time.
- **Reassignment root cause classification**: By examining the notes and the skill/tier change at reassignment, I can categorize reassignments as:
  - **Skill mismatch** (ticket required skill  agent skill — e.g., INC-1001 where B12 realized it needed DB, not CRM).
  - **Workload overflow** (agent too busy, ticket bounced).
  - **Incorrect categorization** (ticket re-categorized after initial assessment).
  - **Tier mismatch** (complexity was over- or under-estimated).

**Expected Finding and Quantification**: 
- "28% of all tickets experience at least one reassignment. Each reassignment adds an average of 2.3 hours to resolution time."
- "Of reassignments, 45% are due to skill mismatch at initial assignment, 25% due to incorrect categorization, 20% due to workload overflow, and 10% due to other reasons."
- "Tickets with 2+ reassignments have a 73% SLA breach rate versus 15% for tickets with zero reassignments."

### 2.3 Impact of Incorrect Initial Assignments

**Method**: I would trace the path of every ticket from "Assign L1" (or "Assign L2" after escalation) and determine whether the first assignment was "correct" — defined as the ticket being resolved by the first assigned agent at that tier without reassignment. The **first-assignment accuracy rate** can be computed overall and segmented by dispatcher, ticket category, and time of day.

**Expected Finding**: "The first-assignment accuracy rate at L2 is only 62%, meaning 38% of L2 assignments result in at least one reassignment. Accuracy is lowest for 'Software-App' tickets (48%) where the required skill often shifts during investigation (e.g., from App-CRM to Database-SQL)."

### 2.4 Agent Overload and Underutilization

**Method**: Using concurrent workload analysis and total ticket volume, I would:
- Compute a **utilization rate** per agent: (total active working time on tickets) / (total available working time). Agents above 90% utilization are at burnout risk and likely creating queues; agents below 50% are underutilized.
- Plot utilization distributions within each tier. A wide bimodal distribution (some agents at 95%, others at 40%) confirms the suspected imbalance.

**Expected Finding**: "Within L2, utilization ranges from 38% to 97%. Agents B08 and B12 are consistently above 90% (they possess high-demand skills: Networking-Firewall and App-CRM), while agents B03 and B21 average 45% utilization (skills in lower-demand areas like Hardware-Printer)."

### 2.5 Correlation Between Resource Patterns and SLA Breaches

**Method**: I would perform a systematic correlation analysis:
- For each SLA-breached ticket, decompose the timeline to identify which phase consumed the most time (L1 handling, escalation wait, L2 queue, L2 handling, reassignment loops).
- Build a logistic regression or decision tree with SLA breach (yes/no) as the dependent variable and features including: number of reassignments, skill match at first assignment, agent utilization at time of assignment, ticket priority, ticket category, time of day, required skill availability.

**Expected Finding**: "The strongest predictors of SLA breach are: (1) number of reassignments (each additional reassignment increases breach probability by 35%), (2) initial skill mismatch (2.1x increase in breach odds), and (3) assignment to an agent with >85% utilization at time of assignment (1.8x increase)."

## 3. Root Cause Analysis for Assignment Inefficiencies

### 3.1 Deficiencies in Current Assignment Rules

The round-robin approach treats all agents within a tier as interchangeable, completely ignoring:
- **Skill differentiation**: An L2 agent with Database-SQL skills receives networking tickets equally with one who has Networking-Firewall skills.
- **Current workload**: An agent already handling 8 open P2 tickets receives the same assignment probability as one handling 2 P4 tickets.
- **Ticket-agent affinity**: Historical data may show that certain agents resolve certain ticket types much faster, but this knowledge isn't leveraged.

The event log can prove this by showing that the assignment pattern within each tier approximates uniform distribution across agents regardless of their skill profiles — while the reassignment pattern then "corrects" these misassignments after the fact, at the cost of delay.

### 3.2 Inaccurate or Incomplete Skill Profiles

If the documented agent skills don't reflect actual capabilities (either understating or overstating them), any skill-based routing will misfire. I would validate skill profiles by comparing:
- **Documented skills** vs. **tickets successfully resolved by skill category**. If Agent B12 is documented as having "App-CRM, DB-SQL" but successfully resolves 15% of their tickets in the "Network" category without reassignment, their profile may be incomplete.
- Conversely, if an agent documented with a skill consistently escalates or reassigns tickets requiring that skill, their proficiency may be overstated.

### 3.3 Poor Initial Ticket Categorization

From the event log, I can identify tickets where the `Required Skill` field changed during the lifecycle (e.g., INC-1001 changed from "App-CRM" to "Database-SQL"). The rate of re-categorization directly measures initial classification accuracy.

**Method**: Compute the **categorization stability rate**: percentage of tickets where the initial category and required skill remain unchanged through resolution. Low stability (e.g., <70%) indicates a systemic intake problem — likely because:
- Web portal self-service categorization options are confusing to users.
- L1 agents lack diagnostic frameworks to correctly identify the underlying issue.
- Ticket templates don't capture enough information for accurate routing.

### 3.4 Lack of Real-Time Visibility

The event log can indirectly reveal this: if assignments consistently target agents who are already overloaded (high concurrent ticket counts at time of assignment), it suggests dispatchers lack visibility into agent queues. I would compute the **assignment-to-overloaded-agent rate**: the percentage of assignments made to agents whose concurrent ticket count exceeds a threshold (e.g., >5 open tickets).

### 3.5 Insufficient L1 Empowerment

I would analyze the L1 escalation pattern in detail:
- **Escalation rate by ticket category**: If L1 escalates 95% of "Software-App" tickets but data shows that 30% of those are eventually resolved with basic steps (password resets, configuration changes), L1 could have handled them with better training or knowledge base access.
- **L1 handling time before escalation**: Very short L1 handling times (<5 minutes) before escalation suggest agents are triaging and routing rather than attempting resolution — a sign of either insufficient knowledge or a cultural norm of "escalate early."
- **Compare high-FCR L1 agents vs. low-FCR L1 agents**: What differentiates them? Longer handling time (indicating they try harder)? Specific ticket categories? This reveals training opportunities.

### 3.6 Variant Analysis and Decision Mining

**Variant Analysis**: I would segment all cases into two groups:
- **Smooth cases**: Resolved with 1 assignment, no reassignment, within SLA.
- **Problematic cases**: Multiple reassignments, SLA breach, or both.

Comparing the process variants, ticket attributes, and resource attributes between these groups reveals distinguishing factors. For example:
- Smooth cases may disproportionately involve tickets with clear, well-defined required skills (e.g., "Access Management" with required skill "Security-IAM") where skill-matching is straightforward.
- Problematic cases may involve ambiguous categories (e.g., "Software-App" which could require App-CRM, DB-SQL, or OS-WindowsServer skills).

**Decision Mining**: At each "Escalate" or "Assign" decision point, I would apply decision tree induction with the decision outcome (e.g., which tier/agent was selected, whether the ticket was subsequently reassigned) as the target variable and ticket attributes (priority, category, keywords, channel) as input features. This reveals:
- What factors *actually* drive escalation decisions (versus what the process documentation says).
- Whether dispatchers implicitly use heuristics that could be formalized and improved.
- Decision paths that consistently lead to reassignments (e.g., "When category = Software-App AND priority = P2, dispatcher assigns to B12, but 60% of these get reassigned to B15 because they need DB skills").

## 4. Developing Data-Driven Resource Assignment Strategies

### Strategy 1: Proficiency-Weighted Skill-Based Routing

**Issue Addressed**: Skill mismatches at assignment, leading to reassignments and delays. The current round-robin completely ignores skill requirements.

**How It Leverages Process Mining Insights**: 
The skill match analysis (Section 1.3) quantifies the mismatch problem, and the reassignment root cause analysis (Section 2.2) proves that ~45% of reassignments stem from skill mismatch. The skill demand-supply analysis identifies which skills are bottlenecks.

**Mechanism**:
For each incoming ticket, the assignment algorithm would:
1. Identify the required skill(s) from the ticket attributes.
2. Filter the agent pool to those possessing the required skill.
3. Within the filtered pool, rank agents by a **proficiency score** derived from historical data:
   - **Resolution rate**: Percentage of tickets with that skill resolved without reassignment.
   - **Average handling time**: Faster resolution for that skill type = higher proficiency.
   - **Customer satisfaction scores** (if available) for that ticket type.
   - The proficiency score could be: `P(agent, skill) = w1 × ResolutionRate + w2 × (1/NormalizedAHT) + w3 × CSAT`, with weights calibrated from historical data.
4. Assign to the highest-proficiency available agent, with a secondary sort by current workload (see Strategy 2).

**Data Required**:
- Complete agent skill profiles (validated against historical resolution patterns).
- Historical resolution outcomes per agent per skill category (from event log).
- Real-time agent availability status.

**Expected Benefits**:
- Reduction in skill-mismatch reassignments by 60-70% (from 45% of all reassignments to ~15%).
- Reduction in average resolution time by 15-25% due to tickets reaching the right expert faster.
- Improved SLA compliance for P2/P3 tickets, particularly in high-demand skill areas.

### Strategy 2: Workload-Aware Dynamic Assignment with Real-Time Queue Balancing

**Issue Addressed**: Uneven workload distribution, agent overload causing queue delays, and the disconnect between assignment decisions and actual agent capacity.

**How It Leverages Process Mining Insights**:
The utilization analysis (Section 2.4) reveals extreme variation in agent load. The correlation analysis (Section 2.5) proves that assigning to overloaded agents increases SLA breach probability by 1.8x. Queue time analysis quantifies the delay caused by agent overload.

**Mechanism**:
Integrate a **weighted workload score** into the assignment algorithm:
1. For each candidate agent (filtered by skill from Strategy 1), compute a real-time workload score:
   - `W(agent) =  (Priority_weight(ticket_i) × Estimated_remaining_time(ticket_i))` for all currently assigned open tickets.
   - Priority weights: P1=4, P2=3, P3=2, P4=1 (reflecting the effort and urgency of higher-priority tickets).
   - Estimated remaining time: Derived from historical average handling times for that ticket type and the elapsed time already spent.
2. Normalize W(agent) across all eligible agents and use it as an **inverse preference factor** — agents with lower workload scores are preferred.
3. Combine with proficiency: `FinalScore =  × Proficiency(agent, skill) +  × (1 - NormalizedWorkload(agent))`, where  and  are tunable parameters (initially =0.6, =0.4, adjustable based on whether skill match or load balance is more critical).

Additionally, implement **dynamic threshold-based overflow routing**:
- If all agents with a required skill exceed a workload threshold (e.g., >80% utilization), the system triggers:
  - Temporary cross-skill routing to agents with related skills (identified from the process mining organizational analysis showing agents who successfully handle adjacent skill areas).
  - Escalation of the queue condition to a supervisor for potential temporary reallocation.

**Data Required**:
- Real-time ticket assignment and status data (open, in-progress, waiting).
- Historical handling time distributions by ticket type, priority, and skill (from event log).
- Real-time agent status (available, on-call, on-break, shift schedule).

**Expected Benefits**:
- Reduction in workload Gini coefficient within tiers from ~0.45 to <0.20.
- Reduction in average queue wait time by 30-40%.
- Reduction in SLA breaches attributable to agent overload by 50%.
- Reduced agent burnout and improved morale for previously overloaded specialists.

### Strategy 3: Predictive Ticket Classification and Intelligent Triage with L1 Empowerment

**Issue Addressed**: Poor initial categorization leading to incorrect routing, excessive L1 escalations for tickets that could be resolved at L1, and specialists handling below-skill-level tasks.

**How It Leverages Process Mining Insights**:
The categorization stability analysis (Section 3.3) reveals that a significant percentage of tickets are re-categorized during handling. The L1 escalation analysis (Section 3.5) shows that many escalated tickets are ultimately resolved with basic procedures. Decision mining (Section 3.6) identifies the ticket attributes that predict complexity and required skill.

**Mechanism — Three Sub-Components**:

**(A) Predictive Skill and Complexity Classification:**
- Train a machine learning model on historical ticket data (from the event log enriched with ticket descriptions, initial categorization, and final resolution details):
  - **Input features**: Ticket category, priority, channel, description keywords (NLP-extracted), reporting user's department, time of day, historical frequency of that issue type.
  - **Output 1**: Predicted required skill (corrected, not just the initial categorization).
  - **Output 2**: Predicted complexity level (L1-resolvable, L2-required, L3-required) based on the actual resolution tier in historical data.
- At ticket creation, the model predicts the true required skill and complexity. If the prediction differs from the submitter's categorization (with high confidence), the system suggests or automatically corrects the routing.
- This directly reduces the re-categorization problem that causes mid-process reassignments.

**(B) Intelligent L1 Triage with Decision Support:**
- Using the decision mining results, create a **structured triage decision tree** for L1 agents:
  - For ticket types historically resolved at L1 >40% of the time, provide the L1 agent with a guided diagnostic checklist and relevant knowledge base articles.
  - For ticket types historically requiring L2/L3 >90% of the time, enable **fast-track escalation** — skip L1 troubleshooting entirely and route directly to the predicted specialist. This saves the 15-30 minutes of L1 handling that adds no value.
- Define explicit **escalation criteria** per ticket category based on data: "If troubleshooting steps A, B, C have been completed and the issue persists, escalate to L2 with skill X. Do not escalate before completing these steps."

**(C) L1 Skill Expansion and Deflection:**
- Identify the top 5-10 ticket types currently escalated to L2 that have straightforward resolution procedures (identifiable from short L2 handling times and simple resolution notes in the event log).
- Develop L1 training programs and runbooks for these specific ticket types.
- Implement a **phased transfer**: Initially, L1 handles these under L2 mentorship (co-assignment), then independently, with monitoring.

**Data Required**:
- Historical ticket descriptions and resolution notes (for NLP model training).
- Complete ticket lifecycle data from event log (for complexity prediction).
- Knowledge base articles mapped to ticket categories and resolution procedures.
- Escalation outcomes (was the escalation necessary? Did L2/L3 use basic steps to resolve?).

**Expected Benefits**:
- Improvement in first-assignment accuracy from ~62% to >85%.
- Increase in L1 FCR rate by 10-15 percentage points (e.g., from 30% to 42%).
- Reduction in unnecessary escalations by 20-30%.
- Recovery of 15-20% of L2/L3 specialist capacity currently spent on below-skill-level tasks.
- Reduction in SLA breaches linked to incorrect categorization by 50-60%.

## 5. Simulation, Implementation, and Monitoring

### 5.1 Business Process Simulation for Pre-Implementation Evaluation

Before deploying any strategy, I would use **business process simulation** to model the expected impact:

**Model Construction:**
1. **Baseline Model**: Use the mined process model (control flow, branching probabilities, activity durations, resource pools) as the simulation baseline. This model should faithfully reproduce the current KPIs (average resolution time, SLA breach rates, reassignment rates) when simulated — this is the validation step.
2. **Resource Parameterization**: Each agent is modeled with their tier, skill set, proficiency scores, shift schedule, and historical processing time distributions (not just averages — use fitted distributions like lognormal or Weibull).
3. **Arrival Process**: Model ticket arrivals using the empirical inter-arrival time distribution from the event log, segmented by priority, category, and time-of-day patterns.

**Scenario Modeling:**
For each proposed strategy:

- **Strategy 1 (Skill-Based Routing)**: Modify the assignment rules in the simulation to filter by skill match and rank by proficiency. Remove the round-robin logic. Run 1,000 simulation replications and compare:
  - Reassignment rate reduction.
  - SLA compliance improvement.
  - Impact on workload distribution (does skill-based routing concentrate load on a few highly skilled agents?).

- **Strategy 2 (Workload-Aware Assignment)**: Add the workload scoring mechanism. Test different / weight combinations. Evaluate:
  - Queue time reduction.
  - Workload CV within tiers.
  - Whether there are scenarios where workload balancing degrades resolution quality (routing to a less proficient but less busy agent).

- **Strategy 3 (Predictive Triage + L1 Empowerment)**: Modify the L1 branching probabilities to reflect higher FCR rates and more accurate initial routing. Test:
  - Impact on L2/L3 queue lengths.
  - Overall throughput improvement.
  - Sensitivity to prediction accuracy (what if the ML model is only 75% accurate vs. 90%?).

- **Combined Strategies**: The greatest value likely comes from implementing all three together. Simulate the combined effect, watching for emergent interactions (e.g., skill-based routing + workload awareness may require careful calibration to avoid contradicting each other).

**Sensitivity Analysis:**
- Test scenarios with increased ticket volume (10%, 20%, 30% growth) to assess scalability.
- Test with agent attrition (remove high-skill agents) to assess resilience.
- Test with varying accuracy of the predictive model (Strategy 3) to establish minimum required accuracy thresholds.

**Output**: A comparative table of scenarios showing expected KPI improvements with confidence intervals, enabling management to make an informed investment decision.

### 5.2 Implementation Plan

**Phase 1 (Weeks 1-4): Foundation**
- Validate and update agent skill profiles using the historical resolution analysis.
- Deploy the workload visibility dashboard so dispatchers can see real-time agent queues.
- Implement basic skill-based filtering on the existing ticketing system (stop assigning network tickets to agents without networking skills).

**Phase 2 (Weeks 5-10): Core Strategy Deployment**
- Deploy the proficiency-weighted skill-based routing algorithm (Strategy 1) with workload awareness (Strategy 2).
- Run in **shadow mode** for 2 weeks: the algorithm recommends assignments, but dispatchers can override. Track recommendation acceptance rate and outcomes.
- Go live with automated assignment, with dispatcher override capability retained.

**Phase 3 (Weeks 11-16): Predictive Triage**
- Train and validate the predictive classification model on historical data.
- Deploy prediction as a **suggestion** to L1 agents during triage (not mandatory).
- Begin L1 training program for identified deflectable ticket types.

**Phase 4 (Ongoing): Optimization and Learning**
- Continuously retrain the predictive model with new data.
- Adjust proficiency scores monthly based on rolling performance.
- Review and adjust / weights in the combined scoring formula quarterly.

### 5.3 Continuous Monitoring with Process Mining Dashboards

Post-implementation, I would establish a real-time process mining monitoring framework with the following dashboards and KPIs:

**Dashboard 1: SLA Compliance Monitor**
- **KPIs**: SLA compliance rate by priority level (P1-P4), trend over time, compliance by ticket category and required skill.
- **Alert**: If weekly SLA compliance for any priority drops below target by >5%, trigger investigation.
- **Process View**: Dotted chart showing ticket lifecycles colored by SLA status (green = met, red = breached), enabling visual identification of breach patterns by time of day or day of week.

**Dashboard 2: Assignment Quality Monitor**
- **KPIs**:
  - First-assignment accuracy rate (overall and by dispatcher/algorithm).
  - Reassignment rate and reassignment count distribution.
  - Skill match rate at assignment.
  - Average delay per reassignment.
- **Process View**: Resource handover social network, refreshed weekly, with edge thickness indicating handover frequency and color indicating whether handovers are planned escalations (acceptable) or unplanned reassignments (problematic).
- **Alert**: If reassignment rate exceeds baseline+2 for any week, investigate.

**Dashboard 3: Resource Utilization and Balance Monitor**
- **KPIs**:
  - Agent utilization rate by tier and individual (with red/yellow/green zones: <40% underutilized, 40-85% optimal, >85% overloaded).
  - Workload Gini coefficient within each tier (target: <0.20).
  - Queue wait time by skill area.
  - L1 FCR rate (overall and by category).
- **Process View**: Heatmap of agent × skill showing volume handled and performance, enabling identification of emerging skill gaps or demand shifts.

**Dashboard 4: Process Conformance and Variant Monitor**
- **KPIs**:
  - Conformance rate to the designed process (are agents following the new triage procedures?).
  - Distribution of process variants (tracking whether the proportion of "clean" variants increases over time).
  - Dispatcher override rate of algorithm recommendations (and outcomes of overrides vs. accepted recommendations).
- **Process View**: Variant explorer showing the top 20 most frequent process variants, with trends indicating whether problematic variants (multiple reassignments, circular escalations) are decreasing.

**Dashboard 5: Predictive Model Performance**
- **KPIs**:
  - Prediction accuracy for required skill and complexity level.
  - False escalation rate (predicted L2/L3 but resolved at L1).
  - Missed escalation rate (predicted L1-resolvable but actually required L2/L3).
- **Trigger**: If prediction accuracy drops below 80%, initiate model retraining with recent data.

**Review Cadence:**
- **Daily**: Automated alerts for SLA breaches and extreme workload imbalances.
- **Weekly**: Review of assignment quality and reassignment trends by team leads.
- **Monthly**: Comprehensive review of all dashboards with management; proficiency score recalculation; identification of new training needs.
- **Quarterly**: Strategic review of resource allocation, skill investment priorities (hiring/training), and algorithm parameter tuning.

## Summary

This approach transforms TechSolve's resource assignment from a reactive, manual, skill-blind process into a data-driven, continuously optimized system. The three strategies work synergistically:

| Strategy | Primary Problem Solved | Expected SLA Improvement |
|----------|----------------------|--------------------------|
| Skill-Based Routing | Skill mismatch  reassignments | +15-20% compliance |
| Workload-Aware Assignment | Uneven load  queue delays | +10-15% compliance |
| Predictive Triage + L1 Empowerment | Poor categorization + over-escalation | +10-15% compliance |
| **Combined** | **All of the above** | **+25-35% compliance** |

The process mining foundation ensures that every recommendation is evidence-based, every implementation is measurable, and every adjustment is informed by actual process behavior rather than assumptions. The continuous monitoring framework ensures the improvements are sustained and adapted as conditions change.
