Here is a DuckDB SQL query that implements the logic step-by-step using window functions to inspect direct succession within each case, and then filters out the matching cases.

```sql
WITH ordered_events AS (
    /* Step 1: Look ahead within each case to find the next two activities and their timestamps.
       This lets us check for direct succession (no other events in between). */
    SELECT
        case_id,
        activity,
        timestamp,
        LEAD(activity, 1) OVER (PARTITION BY case_id ORDER BY timestamp)   AS next_activity_1,
        LEAD(activity, 2) OVER (PARTITION BY case_id ORDER BY timestamp)   AS next_activity_2,
        LEAD(timestamp, 1) OVER (PARTITION BY case_id ORDER BY timestamp)  AS next_timestamp_1,
        LEAD(timestamp, 2) OVER (PARTITION BY case_id ORDER BY timestamp)  AS next_timestamp_2
    FROM event_log
),
cases_to_exclude AS (
    /* Step 2: Identify cases that have the exact sequence AND where the gap between
       'Create Order' and 'Confirm Order' is strictly more than 5 days. */
    SELECT DISTINCT case_id
    FROM ordered_events
    WHERE activity = 'Approve Purchase Requisition'
      AND next_activity_1 = 'Create Order'
      AND next_activity_2 = 'Confirm Order'
      AND next_timestamp_2 - next_timestamp_1 > INTERVAL '5 days'
)
/* Step 3: Return all events from cases that do NOT meet the criteria. */
SELECT e.*
FROM event_log e
WHERE NOT EXISTS (
    SELECT 1
    FROM cases_to_exclude c
    WHERE c.case_id = e.case_id
)
ORDER BY e.case_id, e.timestamp;
```

### How it works

1. **Direct succession check**: `LEAD(..., 1)` and `LEAD(..., 2)` fetch the immediately following two events within the same `case_id` (ordered by `timestamp`). If they are `'Create Order'` and `'Confirm Order'` respectively, the three events are in direct succession.
2. **Timing condition**: `next_timestamp_2 - next_timestamp_1 > INTERVAL '5 days'` verifies that the `'Confirm Order'` timestamp is more than 5 days after the `'Create Order'` timestamp for that specific sequence.
3. **Exclusion**: The `cases_to_exclude` CTE collects every `case_id` that satisfies **both** conditions. The final `SELECT` uses `NOT EXISTS` to return every row from `event_log` except those belonging to an excluded case.

> **Note on ordering**: If your data contains events with identical timestamps within the same case, you should add a secondary tie-breaker to the `ORDER BY` inside the window functions (e.g., `ORDER BY timestamp, rowid` or a dedicated event ID column) to make the direct succession deterministic.
