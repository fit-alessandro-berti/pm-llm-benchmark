Here is a complete, production-ready DuckDB SQL query that fulfills all your requirements:

```sql
WITH case_sequences AS (
    -- Step 1 & 2: Build the ordered activity sequence per case
    SELECT 
        case_id,
        ARRAY_AGG(activity ORDER BY timestamp) AS variant_sequence
    FROM event_log
    GROUP BY case_id
),
variant_counts AS (
    -- Step 3: Count how many cases follow each unique variant
    SELECT 
        variant_sequence,
        COUNT(*) AS variant_frequency
    FROM case_sequences
    GROUP BY variant_sequence
),
top_k_case_ids AS (
    -- Step 4: Identify the top K variants and retrieve their case IDs
    SELECT cs.case_id
    FROM case_sequences cs
    JOIN variant_counts vc 
      ON cs.variant_sequence = vc.variant_sequence
    ORDER BY vc.variant_frequency DESC, cs.case_id
    LIMIT K  --  Replace K with your desired number (e.g., 5, 10)
)
-- Step 5: Return all original events belonging only to the top K cases
SELECT e.*
FROM event_log e
WHERE e.case_id IN (SELECT case_id FROM top_k_case_ids)
ORDER BY e.case_id, e.timestamp;
```

###  How It Meets Your Requirements

| Requirement | Implementation |
|-------------|----------------|
| **Extract ordered sequence per case** | `ARRAY_AGG(activity ORDER BY timestamp)` groups events per `case_id` while preserving chronological order. |
| **Group by complete sequence & count** | `variant_counts` CTE groups by the generated `variant_sequence` and uses `COUNT(*)` to tally cases per variant. |
| **Determine top K variants** | `top_k_case_ids` joins sequences with their frequencies, orders descending, and applies `LIMIT K`. Tie-breaking uses `case_id` for deterministic results. |
| **Return only events from top K cases** | The final `SELECT` filters the original `event_log` using a `WHERE ... IN (...)` clause, ensuring non-matching cases are excluded. |

###  DuckDB-Specific Notes
1. **Array Grouping**: DuckDB natively supports grouping and joining on array types, so `variant_sequence` (an array) can be safely used in `GROUP BY` and `JOIN` without converting to strings.
2. **Parameterization**: Replace `K` with an integer literal, or use DuckDB's parameter binding: `LIMIT ?` or `LIMIT $1` when executing via drivers (Python, JDBC, etc.).
3. **Performance**: For very large logs, consider adding an index on `event_log(case_id, timestamp)` to speed up the `ARRAY_AGG` window/grouping operation.
