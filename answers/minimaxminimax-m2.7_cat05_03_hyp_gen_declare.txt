### 3.1.  Verify the *existence* of **C** (every claim must be closed)

```sql
-- Claims that never contain a Close event
SELECT c.claim_id,
       c.submission_date,
       c.claim_type
FROM   claims c
WHERE  NOT EXISTS (
       SELECT 1
       FROM   claim_events ce
       WHERE  ce.claim_id = c.claim_id
         AND  ce.activity = 'C'
       );
```

*Interpretation*: Any row returned violates the *existence* rule.  

---

### 3.2.  Detect the *noncoexistence* violation (E+C in the same trace)

```sql
-- Claims that have both an Evaluate and a Close event
SELECT DISTINCT ce1.claim_id
FROM   claim_events ce1
JOIN   claim_events ce2
  ON   ce1.claim_id = ce2.claim_id
WHERE  ce1.activity = 'E'
  AND  ce2.activity = 'C';
```

*Interpretation*: This query flags the exact violation of the *noncoexistence* rule.  

---

### 3.3.  Check *respondedexistence* of **E** (E must follow A)

```sql
-- Claims where an Evaluate event exists but no prior Assign event
SELECT e.claim_id,
       e.timestamp AS eval_time,
       e.activity  AS eval_activity
FROM   claim_events e
WHERE  e.activity = 'E'
  AND  NOT EXISTS (
       SELECT 1
       FROM   claim_events a
       WHERE  a.claim_id = e.claim_id
         AND  a.activity = 'A'
         AND  a.timestamp < e.timestamp
       );
```

*Interpretation*: Returns cases where evaluation occurs without a preceding assignment – a breach of the respondedexistence constraint.

### 3.4.  Detect *assignment* without evaluation (optional but revealing)

```sql

       a.timestamp AS assign_time
FROM   claim_events a
WHERE  a.activity = 'A'
  AND  NOT EXISTS (
       SELECT 1
       FROM   claim_events e
       WHERE  e.claim_id = a.claim_id
         AND  e.activity = 'E'
       );
```

*Interpretation*: Shows the “dangling” assignments that would become illegal if the *respondedexistence* rule were strictly enforced together with the *noncoexistence* rule.

### 3.5.  Verify *precedence* of **C** after **R**

```sql
-- Claims where Close appears before (or without) a preceding Receive
SELECT ce.claim_id,
       ce.timestamp AS close_time,
       r.timestamp  AS receive_time
FROM   claim_events ce
LEFT   JOIN claim_events r
       ON ce.claim_id = r.claim_id
      AND r.activity = 'R'
      AND r.timestamp < ce.timestamp
WHERE  ce.activity = 'C'
  AND  (r.timestamp IS NULL OR r.timestamp > ce.timestamp);
```

*Interpretation*: Flags cases that violate the *precedence* rule CR.

### 3.7.  Verify the *init* rule (first activity must be **R**)

```sql
-- Claims whose first logged activity is not Receive
WITH first_event AS (
    SELECT claim_id,
           activity,
           ROW_NUMBER() OVER (PARTITION BY claim_id ORDER BY timestamp) AS rn
    FROM   claim_events

SELECT fe.claim_id,
       fe.activity AS first_activity
FROM   first_event fe
WHERE  fe.rn = 1
  AND  fe.activity <> 'R';
```

*Interpretation*: Any result breaks the *init* constraint.

### 3.8.  Full trace reconstruction & pattern check (for complex violations)

```sql
-- Build a chronological trace for each claim and search for illegal patterns
WITH traces AS (
    SELECT claim_id,
           STRING_AGG(activity, ',' ORDER BY timestamp) AS trace

```

*Interpretation*: Onestop view of every claim’s trace together with a flag for each known anomaly.
