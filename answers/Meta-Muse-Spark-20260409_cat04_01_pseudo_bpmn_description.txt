This process handles customer requests from intake to final confirmation, with different workflows depending on whether the request is **Standard** or **Custom**. Here’s how it all fits together:

### **End-to-End Flow Overview**
1. **Start → Receive Request**: Every process instance begins when a customer submits a request.
2. **Early Branch by Request Type**: The very first decision splits the flow into two distinct tracks.
3. **Late Re-Convergence**: Both tracks eventually meet at the “Is Approval Needed?” check, then share the final invoice + confirmation steps.

---

### **Step-by-Step Purpose**

| Step | Purpose | Notes |
| --- | --- | --- |
| **Task A: Receive Customer Request** | Entry point. Capture all customer data, items, specs, deadlines. | Triggers the entire process |
| **Task B1: Perform Standard Validation** | For catalog/off-the-shelf items. Quick checks: Is the product valid, SKU correct, basic eligibility met? | Standard path only |
| **Task C1: Credit Check + Task C2: Inventory Check** | Run simultaneously to save time. C1 = can customer pay / is credit limit OK? C2 = do we have stock / can we fulfill by date? | Standard path only |
| **AND Gateway: Run Parallel Checks** | Means both C1 and C2 must complete before moving on. No dependency between them. | Speeds up Standard path |
| **Join after C1/C2** | Synchronization point. Process waits until *both* checks are done | BPMN rule: AND-split must have AND-join |
| **Task D: Calculate Delivery Date** | Based on inventory + logistics, set ETA for standard items | Standard path only |
| **Task B2: Perform Custom Feasibility Analysis** | For non-catalog requests. Engineering/ops reviews if we *can* build it: technical, cost, resource, lead time | Custom path only |
| **Task E1: Prepare Custom Quotation** | If feasible, price the custom work + propose terms and delivery | Custom path only |
| **Task E2: Send Rejection Notice → End** | If not feasible, politely decline and terminate process here | Custom path dead-end |
| **Task F: Obtain Manager Approval** | Human sign-off for risk, margin, or policy reasons | Triggered by “Is Approval Needed?” gateway |
| **Task G: Generate Final Invoice** | Create binding commercial document with price, terms, delivery | Reached from multiple paths |
| **Task H: Re-evaluate Conditions** | If manager rejects, revise terms, price, or scope to address concerns | Triggers a loop |
| **Task I: Send Confirmation to Customer** | Final step: communicate invoice, ETA, next steps. Customer gets closure | All successful paths end here |

---

### **Gateway Meanings & Logic**

**1. XOR Gateway: Check Request Type**  
*Meaning*: Exclusive choice - only one path taken.  
- **Standard**: High-volume, low-variance requests. We use automation + parallel checks.  
- **Custom**: Bespoke requests. Need human/technical assessment first. No credit/inventory check yet because we don’t know what we’re building.

**2. AND Gateway: Run Parallel Checks**  
*Meaning*: Split into concurrent tasks. Both Credit Check and Inventory Check start at same time. The following “Join” means the process pauses until *all* branches complete. This is only used for Standard to reduce cycle time.

**3. XOR Gateway: Is Customization Feasible?**  
*Meaning*: Go/No-Go decision after analysis.  
- **Yes** → Continue to quote.  
- **No** → Immediate rejection and End Event. This is the only mid-process termination point.

**4. XOR Gateway: Is Approval Needed?**  
*Meaning*: Policy-based routing. Both Standard and Custom paths hit this.  
*Typical triggers*: Order value > $X, discount > Y%, new customer, high-risk custom work, margin below threshold. If “No”, skip straight to invoice.

**5. XOR Gateway: Is Approval Granted?**  
*Meaning*: Manager’s decision.  
- **Yes** → Generate invoice.  
- **No** → Don’t end. Instead go to “Re-evaluate Conditions” to fix issues and loop back.

---

### **How Paths Converge & Loop**

**Convergence points:**
1. **After Standard D or Custom E1** → Both feed into “Is Approval Needed?” This is where the two tracks reunite.  
2. **After G: Generate Final Invoice** → Both approval paths + “no approval needed” path all lead to Task I.

**Loop back logic:**  
Only happens on manager rejection via Task H: Re-evaluate Conditions.  
- **For Custom path**: Loop goes to E1 “Prepare Custom Quotation”. Idea: adjust price/scope and re-quote, then try approval again.  
- **For Standard path**: Loop goes to D “Calculate Delivery Date”. Idea: maybe delivery terms or pricing caused rejection, so recalc and retry.  
This prevents endless loops if rejection is final, but the model implies rework is allowed.

---

### **Standard vs Custom: Key Differences**

| Aspect | Standard Path | Custom Path |
| --- | --- | --- |
| **Validation style** | Fast, rule-based: SKU, credit, stock | Analytical: Can we technically do this? |
| **Parallelism** | Yes: credit + inventory run at same time | No: Feasibility is sequential gatekeeper |
| **Early exit** | No early rejection shown | Yes: E2 ends process if infeasible |
| **Output before approval** | Delivery date calculated | Custom quotation prepared |
| **Risk focus** | Financial + fulfillment risk | Technical + commercial risk |
| **Rework target** | Delivery/terms via Task D | Quotation via Task E1 |

---

### **When Approvals or Rework Are Triggered**

**Approval Needed if:**  
Business rules apply after initial processing. Common examples: order > $50k, custom margin < 20%, first-time customer, non-standard payment terms, expedited shipping requested. The model doesn’t specify the rule, but this gateway is the enforcement point for both paths.

**Rework triggered if:**  
Manager says “No” at “Is Approval Granted?”. Instead of killing the deal, Task H lets a sales/engineer revise the offer. This acknowledges that many rejections are conditional: “not at this price” or “not with this timeline.” The loop ensures we only end at E2 or after customer confirmation, not on internal rejection.

One edge case: If a Standard request loops from H → D, it will hit “Is Approval Needed?” again. So repeated rejections could cycle. In practice you’d add a loop counter to avoid infinite cycles.
