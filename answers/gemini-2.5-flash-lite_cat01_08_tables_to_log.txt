Here is the integrated event log for the ERP data, formatted as a Markdown table.

**Assumptions Made:**

1.  **Picking Events:** For each order, two picking events are created: "Picking Started" and "Picking Completed".
    *   "Picking Started" uses the earliest `item_picking_date` recorded for any item in that order. The `picker_id` associated with that earliest date is used as the resource.
    *   "Picking Completed" uses the latest `item_picking_date` recorded for any item in that order. The `picker_id` associated with that latest date is used as the resource.
    *   If an order has only one item line, "Picking Started" and "Picking Completed" will have the same timestamp and resource.
    *   If an order has no line items, no picking events are generated.
2.  **Resource Mapping:** IDs from various tables (e.g., `sales_person_id`, `picker_id`, `carrier_id`, `billing_person_id`) are mapped to human-readable names using the `Resources` table.
3.  **Payment Resource:** The `Payments` table does not directly link a payment event to a specific person or system actor. Therefore, the 'Resource' column for "Payment Received" events is left blank.
4.  **Attribute Inclusion:** Relevant attributes from the source tables are included in the "Other Attributes" columns for each event type to provide richer context. Columns like `item_id` and `quantity` are specific to picking events, `delivery_note` to dispatch/arrival events, `invoice_amount` to invoice events, and `payment_method`/`payment_reference` to payment events. Other attributes (`customer_id`, `customer_type`, `order_priority`) are included with the "Order Created" event.
5.  **Chronological Order:** All events within each `Case ID` are strictly ordered by their `Timestamp`.
