### Assumptions, Observations, and Design Choices

1. **Granularity of "Item Picked"**: Instead of aggregating picking into a single "Order Picked" event, I retained individual "Item Picked" events. This is highly valuable in process mining as it reveals parallel work, bottlenecks, and specific resource assignments (e.g., Order 1001 was picked by two different warehouse staff members sequentially).
2. **Split Shipments (Order 1002)**: Order 1002 has two distinct `Order Dispatched` and `Order Arrived` events. This accurately reflects the real-world process of partial backorders and split shipments, which is a critical variant to detect in process mining.
3. **Incomplete Process Instance (Order 1004)**: Order 1004 only contains "Order Created" and "Item Picked" events. There are no corresponding delivery, invoice, or payment records. In process mining, this is a valid and important "drop-off" or "stuck" case (e.g., order canceled after picking, or data missing from the ERP). It has been left in the log to allow for conformance checking and bottleneck analysis.
4. **Resource Mapping**: Resource IDs were mapped to human-readable names and roles using the provided `Resources` table to make the log interpretable. For payments, the resource is marked as "External Payer / System" since the specific actor is not tracked in the payments table, only the method and reference.
5. **Chronological Sorting**: The table is strictly sorted first by `Case ID` and then by `Timestamp` ascending, which is the mandatory format for importing into process mining tools like Celonis, ProM, or Disco.
6. **Attribute Formatting**: Additional context (like quantities, carrier notes, and invoice amounts) is packed into a single `Attributes` column using key-value pairs. Most process mining tools can parse this into separate columns upon import, keeping the core event log clean while preserving rich data.
