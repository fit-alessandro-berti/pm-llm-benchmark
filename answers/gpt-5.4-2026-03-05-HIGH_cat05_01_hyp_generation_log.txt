Below is a process-conformance read of the sample log, using the assumed normal flow as the benchmark.

## 1) What looks anomalous

### Case 1001
Looks normal and can serve as the reference path:

`Register Order -> Perform Credit Check -> Validate Stock -> Confirm Shipment -> Ship Goods -> Issue Invoice -> Receive Payment`

### Case 1002
**Anomalies**
- `Confirm Shipment` happens **before** `Perform Credit Check`
- `Ship Goods` happens **before** both `Perform Credit Check` and `Validate Stock`
- `additional_info = attempted_early=Y` explicitly suggests an early/shadow attempt

**Why this is bad**
- Shipping occurred before key control gates
- Customer credit risk and stock availability were checked only after fulfillment had already started
- Since this is a **priority** order, it may indicate operational pressure causing gate bypass

---

### Case 1003
**Anomalies**
- `Ship Goods` occurs **before** `Confirm Shipment`
- `Validate Stock` is **missing entirely**
- `Confirm Shipment` carries `late_confirmation=Y`, suggesting it may have been back-posted

**Why this is bad**
- Possible retroactive approval
- Missing stock validation could mean either a real control failure or missing event capture
- Invoice was issued before shipment was formally confirmed

---

### Case 1004
**Anomalies**
- `Receive Payment` occurs **before** `Issue Invoice`
- `Issue Invoice` occurs **before** `Ship Goods`
- `Perform Credit Check` is **missing**
- `Validate Stock` is **missing**
- `Confirm Shipment` says `shipment_scheduled=N`, yet `Ship Goods` still happens later

**Why this is bad**
- Strongest anomaly in the sample
- Could indicate:
  - an unmodeled **prepayment** variant, or
  - broken controls / contradictory statuses / manual overrides

---

## 2) Likely explanations

These anomalies could come from several sources:

1. **Operational bypass / policy violations**
   - Staff expedite orders, especially priority ones, by skipping controls

2. **System integration or timestamp issues**
   - Events may come from multiple systems with asynchronous logging
   - A confirmation may be recorded late even if operationally done earlier

3. **Manual back-posting**
   - `late_confirmation=Y` strongly suggests retrospective entry

4. **Unmodeled process variants**
   - `Receive Payment` before invoicing may be legitimate for prepayment orders
   - If so, the model should explicitly include a separate variant

5. **Ambiguous event semantics**
   - `Confirm Shipment` may actually log both successful confirmations and failed attempts
   - Example: `attempted_early=Y` and `shipment_scheduled=N`

6. **Training or workflow enforcement gaps**
   - Users may not understand mandatory sequencing
   - Or systems may not technically block invalid next steps

A noteworthy pattern: **LogisticsMgr_2** appears in both an early confirmation case and a late confirmation case. That is worth investigating.

## 3) SQL to investigate further

## A. Reconstruct each case trace
Good first step to see the observed variants.

```sql
SELECT
    o.case_id,
    o.order_type,
    o.order_value,
    string_agg(e.activity, ' -> ' ORDER BY e."timestamp", e.event_id) AS trace
FROM orders o
JOIN order_event_log e USING (case_id)
GROUP BY o.case_id, o.order_type, o.order_value
ORDER BY o.case_id;
```

## B. Find cases with missing mandatory activities
This will flag missing controls like case 1003 and 1004.

```sql
WITH required(activity) AS (
    VALUES
        ('Register Order'),
        ('Perform Credit Check'),
        ('Validate Stock'),
        ('Confirm Shipment'),
        ('Ship Goods'),
        ('Issue Invoice'),
        ('Receive Payment')
),
cases AS (
    SELECT DISTINCT case_id
    FROM order_event_log
),
case_acts AS (
    SELECT DISTINCT case_id, activity
    FROM order_event_log
)
SELECT
    c.case_id,
    array_agg(r.activity ORDER BY r.activity) FILTER (WHERE ca.activity IS NULL) AS missing_activities
FROM cases c
CROSS JOIN required r
LEFT JOIN case_acts ca
    ON ca.case_id = c.case_id
   AND ca.activity = r.activity
GROUP BY c.case_id
HAVING COUNT(ca.activity) < (SELECT COUNT(*) FROM required)
ORDER BY c.case_id;
```

## C. Case-level rule violations
This gives a compact anomaly summary per case.

```sql
WITH per_case AS (
    SELECT
        case_id,
        MIN("timestamp") FILTER (WHERE activity = 'Register Order') AS register_ts,
        MIN("timestamp") FILTER (WHERE activity = 'Perform Credit Check') AS credit_ts,
        MIN("timestamp") FILTER (WHERE activity = 'Validate Stock') AS stock_ts,
        MIN("timestamp") FILTER (WHERE activity = 'Confirm Shipment') AS confirm_ts,
        MIN("timestamp") FILTER (WHERE activity = 'Ship Goods') AS ship_ts,
        MIN("timestamp") FILTER (WHERE activity = 'Issue Invoice') AS invoice_ts,
        MIN("timestamp") FILTER (WHERE activity = 'Receive Payment') AS payment_ts
    FROM order_event_log
    GROUP BY case_id
)
SELECT
    p.case_id,
    array_remove(ARRAY[
        CASE WHEN p.credit_ts IS NULL THEN 'Missing Credit Check' END,
        CASE WHEN p.stock_ts IS NULL THEN 'Missing Validate Stock' END,
        CASE WHEN p.confirm_ts IS NULL THEN 'Missing Confirm Shipment' END,
        CASE WHEN p.ship_ts IS NOT NULL AND (p.credit_ts IS NULL OR p.ship_ts < p.credit_ts)
            THEN 'Shipped before Credit Check' END,
        CASE WHEN p.ship_ts IS NOT NULL AND (p.stock_ts IS NULL OR p.ship_ts < p.stock_ts)
            THEN 'Shipped before Validate Stock' END,
        CASE WHEN p.ship_ts IS NOT NULL AND (p.confirm_ts IS NULL OR p.ship_ts < p.confirm_ts)
            THEN 'Shipped before Confirm Shipment' END,
        CASE WHEN p.invoice_ts IS NOT NULL AND (p.ship_ts IS NULL OR p.invoice_ts < p.ship_ts)
            THEN 'Invoice before Ship Goods' END,
        CASE WHEN p.payment_ts IS NOT NULL AND (p.invoice_ts IS NULL OR p.payment_ts < p.invoice_ts)
            THEN 'Payment before Invoice' END
    ], NULL) AS anomalies
FROM per_case p
WHERE
       p.credit_ts IS NULL
    OR p.stock_ts IS NULL
    OR p.confirm_ts IS NULL
    OR (p.ship_ts IS NOT NULL AND (
           p.credit_ts IS NULL OR p.ship_ts < p.credit_ts
        OR p.stock_ts IS NULL OR p.ship_ts < p.stock_ts
        OR p.confirm_ts IS NULL OR p.ship_ts < p.confirm_ts
    ))
    OR (p.invoice_ts IS NOT NULL AND (p.ship_ts IS NULL OR p.invoice_ts < p.ship_ts))
    OR (p.payment_ts IS NOT NULL AND (p.invoice_ts IS NULL OR p.payment_ts < p.invoice_ts))
ORDER BY p.case_id;
```

## D. Find shipments executed before prerequisite controls
This isolates the most operationally risky pattern.

```sql
WITH per_case AS (
    SELECT
        case_id,
        MIN("timestamp") FILTER (WHERE activity = 'Perform Credit Check') AS credit_ts,
        MIN("timestamp") FILTER (WHERE activity = 'Validate Stock') AS stock_ts,
        MIN("timestamp") FILTER (WHERE activity = 'Confirm Shipment') AS confirm_ts,
        MIN("timestamp") FILTER (WHERE activity = 'Ship Goods') AS ship_ts
    FROM order_event_log
    GROUP BY case_id
)
SELECT
    case_id,
    ship_ts,
    credit_ts,
    stock_ts,
    confirm_ts
FROM per_case
WHERE ship_ts IS NOT NULL
  AND (
       credit_ts IS NULL OR ship_ts < credit_ts
    OR stock_ts IS NULL OR ship_ts < stock_ts
    OR confirm_ts IS NULL OR ship_ts < confirm_ts
  )
ORDER BY case_id;
```

Expected in sample: **1002, 1003, 1004**

## E. Find confirmations entered after shipment
Useful for testing the “back-posted confirmation” hypothesis.

```sql
WITH first_ship AS (

        MIN("timestamp") AS first_ship_ts
```

## F. Find invoicing/payment order violations
Useful for testing whether 1004 is a true anomaly or an unmodeled prepayment process.

```sql
WITH per_case AS (
    SELECT
        case_id,
        MIN("timestamp") FILTER (WHERE activity = 'Ship Goods') AS ship_ts,
        MIN("timestamp") FILTER (WHERE activity = 'Issue Invoice') AS invoice_ts,
        MIN("timestamp") FILTER (WHERE activity = 'Receive Payment') AS payment_ts
    FROM order_event_log
    GROUP BY case_id
)
SELECT
    case_id,
    ship_ts,
    invoice_ts,
    payment_ts,
    (invoice_ts IS NOT NULL AND (ship_ts IS NULL OR invoice_ts < ship_ts)) AS invoice_before_shipping,

WHERE (invoice_ts IS NOT NULL AND (ship_ts IS NULL OR invoice_ts < ship_ts))
   OR (payment_ts IS NOT NULL AND (invoice_ts IS NULL OR payment_ts < invoice_ts))
ORDER BY case_id;
```

## H. Unexpected direct-follow transitions
This is a good process-mining style check.

```sql
WITH allowed(from_act, to_act) AS (
    VALUES
        ('Register Order', 'Perform Credit Check'),
        ('Perform Credit Check', 'Validate Stock'),
        ('Validate Stock', 'Confirm Shipment'),
        ('Confirm Shipment', 'Ship Goods'),
        ('Ship Goods', 'Issue Invoice'),
        ('Issue Invoice', 'Receive Payment')
),
ordered AS (
    SELECT
        case_id,
        activity AS from_act,
        LEAD(activity) OVER (PARTITION BY case_id ORDER BY "timestamp", event_id) AS to_act
    FROM order_event_log
)
SELECT
    case_id,
    from_act,
    to_act
FROM ordered o
WHERE to_act IS NOT NULL
  AND NOT EXISTS (
      SELECT 1
      FROM allowed a
      WHERE a.from_act = o.from_act
        AND a.to_act = o.to_act
  )
ORDER BY case_id, from_act, to_act;
```

This should reveal transitions like:
- `Register Order -> Confirm Shipment`
- `Perform Credit Check -> Ship Goods`
- `Register Order -> Receive Payment`

## I. Check whether anomalies are concentrated in certain order types
Tests the “priority orders are bypassing controls” hypothesis.

```sql
WITH per_case AS (
    SELECT
        case_id,
        MIN("timestamp") FILTER (WHERE activity='Perform Credit Check') AS credit_ts,
        MIN("timestamp") FILTER (WHERE activity='Validate Stock') AS stock_ts,
        MIN("timestamp") FILTER (WHERE activity='Confirm Shipment') AS confirm_ts,
        MIN("timestamp") FILTER (WHERE activity='Ship Goods') AS ship_ts,
        MIN("timestamp") FILTER (WHERE activity='Issue Invoice') AS invoice_ts,
        MIN("timestamp") FILTER (WHERE activity='Receive Payment') AS payment_ts
    FROM order_event_log
    GROUP BY case_id
),
flags AS (
    SELECT
        p.case_id,
        (p.credit_ts IS NULL OR p.stock_ts IS NULL OR p.confirm_ts IS NULL) AS missing_control,
        (p.ship_ts IS NOT NULL AND (
            p.credit_ts IS NULL OR p.ship_ts < p.credit_ts
         OR p.stock_ts IS NULL OR p.ship_ts < p.stock_ts
         OR p.confirm_ts IS NULL OR p.ship_ts < p.confirm_ts
        )) AS ship_gate_violation,
        (p.invoice_ts IS NOT NULL AND (p.ship_ts IS NULL OR p.invoice_ts < p.ship_ts)) AS invoice_before_ship,
        (p.payment_ts IS NOT NULL AND (p.invoice_ts IS NULL OR p.payment_ts < p.invoice_ts)) AS pay_before_invoice
    FROM per_case p
)
SELECT
    o.order_type,
    COUNT(*) AS total_cases,
    SUM((missing_control OR ship_gate_violation OR invoice_before_ship OR pay_before_invoice)::int) AS anomalous_cases,
    ROUND(100.0 * AVG((missing_control OR ship_gate_violation OR invoice_before_ship OR pay_before_invoice)::int), 2) AS pct_anomalous,
    AVG(o.order_value) AS avg_order_value
FROM flags f
JOIN orders o USING (case_id)
GROUP BY o.order_type
ORDER BY o.order_type;
```

## J. See which resources are repeatedly involved in suspicious events
Useful for training / override / role-misuse hypotheses.

```sql
WITH per_case AS (
    SELECT
        case_id,
        MIN("timestamp") FILTER (WHERE activity='Perform Credit Check') AS credit_ts,
        MIN("timestamp") FILTER (WHERE activity='Validate Stock') AS stock_ts,
        MIN("timestamp") FILTER (WHERE activity='Confirm Shipment') AS confirm_ts,
        MIN("timestamp") FILTER (WHERE activity='Ship Goods') AS ship_ts
    FROM order_event_log
    GROUP BY case_id
),
violations AS (
    SELECT
        e.case_id,
        e.resource,
        'premature_ship' AS violation_type
    FROM order_event_log e
    JOIN per_case p USING (case_id)
    WHERE e.activity = 'Ship Goods'
      AND (
           p.credit_ts IS NULL OR p.ship_ts < p.credit_ts
        OR p.stock_ts IS NULL OR p.ship_ts < p.stock_ts
        OR p.confirm_ts IS NULL OR p.ship_ts < p.confirm_ts
      )

    UNION ALL

    SELECT
        e.case_id,
        e.resource,
        'suspicious_confirmation' AS violation_type
    FROM order_event_log e
    WHERE e.activity = 'Confirm Shipment'
      AND (
           e.additional_info LIKE '%attempted_early=Y%'
        OR e.additional_info LIKE '%late_confirmation=Y%'
        OR e.additional_info LIKE '%shipment_scheduled=N%'
      )
)
SELECT
    v.violation_type,
    v.resource,
    r.role,
    r.department,
    COUNT(*) AS events,
    COUNT(DISTINCT v.case_id) AS cases
FROM violations v
LEFT JOIN resources r
    ON r.resource_id = v.resource
GROUP BY v.violation_type, v.resource, r.role, r.department
ORDER BY v.violation_type, cases DESC, events DESC;
```

On the sample, this should make **LogisticsMgr_2** stand out for suspicious confirmations.

## K. Check activity-vs-department mismatches
Not obvious in the sample, but important on the full dataset.

```sql
WITH expected(activity, expected_department) AS (
    VALUES
        ('Register Order', 'Sales'),
        ('Perform Credit Check', 'Finance'),
        ('Validate Stock', 'Logistics'),
        ('Confirm Shipment', 'Logistics'),
        ('Ship Goods', 'Logistics'),
        ('Issue Invoice', 'Finance'),
        ('Receive Payment', 'Finance')
)
SELECT
    e.case_id,
    e.event_id,
    e.activity,
    e.resource,
    r.department,
    ex.expected_department
FROM order_event_log e
LEFT JOIN resources r
    ON r.resource_id = e.resource
JOIN expected ex
    ON ex.activity = e.activity
WHERE r.department IS DISTINCT FROM ex.expected_department
   OR r.resource_id IS NULL
ORDER BY e.case_id, e."timestamp";
```

## L. Test whether low credit-score cases are being shipped early
Useful if the business suspects risky orders are being rushed through.

```sql
WITH credit AS (
    SELECT
        case_id,
        substring(additional_info from 'credit_score=([0-9]+)')::int AS credit_score,
        "timestamp" AS credit_ts
    FROM order_event_log
    WHERE activity = 'Perform Credit Check'
),
ship AS (
    SELECT
        case_id,
        MIN("timestamp") AS ship_ts
    FROM order_event_log
    WHERE activity = 'Ship Goods'
    GROUP BY case_id
)
SELECT
    s.case_id,
    c.credit_score,
    s.ship_ts,
    c.credit_ts,
    (c.credit_ts IS NULL OR s.ship_ts < c.credit_ts) AS shipped_before_credit_result
FROM ship s
LEFT JOIN credit c USING (case_id)
WHERE c.credit_ts IS NULL OR s.ship_ts < c.credit_ts
ORDER BY c.credit_score NULLS FIRST, s.case_id;
```
