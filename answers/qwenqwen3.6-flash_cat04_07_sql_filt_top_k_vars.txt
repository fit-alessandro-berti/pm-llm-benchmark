Here is a complete, production-ready DuckDB SQL query that fulfills all your requirements:

```sql
WITH case_variants AS (
    -- Step 1: Extract the ordered activity sequence per case
    SELECT 
        case_id,
        STRING_AGG(activity, '' ORDER BY timestamp) AS variant
    FROM event_log
    GROUP BY case_id
),
variant_counts AS (
    -- Step 2 & 3: Group by variant, count cases, and determine top K
    SELECT 
        variant,
        COUNT(*) AS case_count
    FROM case_variants
    GROUP BY variant
    ORDER BY case_count DESC
    LIMIT 5 -- Replace 5 with your desired K
),
top_k_case_ids AS (
    -- Step 4: Identify case_ids belonging to the top K variants
    SELECT cv.case_id
    FROM case_variants cv
    JOIN variant_counts vc ON cv.variant = vc.variant
)
-- Final step: Return all original events for the filtered cases
SELECT e.*
FROM event_log e
WHERE e.case_id IN (SELECT case_id FROM top_k_case_ids);
```

###  Query Breakdown

| CTE / Clause | Purpose |
|--------------|---------|
| `case_variants` | Uses `STRING_AGG(... ORDER BY timestamp)` to build a deterministic string representation of each case's activity sequence. Groups by `case_id` to ensure one row per process instance. |
| `variant_counts` | Aggregates the `case_variants` CTE by `variant`, counts how many cases share each sequence, sorts by frequency descending, and limits to the top `K` variants. |
| `top_k_case_ids` | Joins back to `case_variants` to extract only the `case_id`s that belong to the top `K` sequences. This avoids scanning the entire log multiple times. |
| Final `SELECT` | Filters the original `event_log` to return **all events** (original columns) exclusively for cases in the top `K` variants. |

###  DuckDB-Specific Notes
1. **Parameterization**: Replace `LIMIT 5` with `LIMIT :K` if you're running this via a client that supports bind parameters (e.g., `duckdb-cli`, Python `duckdb` package, or BI tools).
2. **Separator Choice**: `STRING_AGG(activity, '')` uses an empty string as a delimiter. If your activity names contain characters that could cause false matches (e.g., `A`+`B` vs `AB`), use a safe delimiter like `'->'` or `'|'` and adjust accordingly:  
   `STRING_AGG(activity, '->' ORDER BY timestamp)`
3. **Timestamp Ties**: If multiple events share the exact same `timestamp`, add a secondary sort key for determinism:  
   `ORDER BY timestamp, activity, event_id` (replace `event_id` with your actual primary key if available).
4. **Performance**: DuckDB handles this pattern efficiently. The `IN` subquery is optimized internally, and CTEs are materialized only when referenced multiple times. For multi-million row logs, ensure `timestamp` is indexed or sorted in the source data for faster aggregation.
