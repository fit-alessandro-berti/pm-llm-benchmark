# Invoice Approval Process Model

Based on the stakeholder interviews, here is a structured textual representation of the invoice approval process:

## 1. Participants/Roles
- **Accounts Payable (AP) Clerk** (Mary): Primary process owner, handles initial invoice processing
- **Purchasing Officer** (Dan): Confirms receipt and matches against PO
- **Department Manager** (Carlos): Final approval authority
- **AP Supervisor** (Rita): Exception handler for complex discrepancies
- **Supplier Relationship Manager** (Karen): External liaison for supplier communication

## 2. Process Flow

```
START EVENT: Invoice received via email
   
TASK 1: AP Clerk checks for PO number
   
EXCLUSIVE GATEWAY 1: PO present?
    NO: TASK 1a: Return invoice to supplier requesting PO
          WAIT for corrected invoice  LOOP BACK to TASK 1
    YES: TASK 2: Log invoice in accounting system
            
TASK 3: Forward invoice to Purchasing for receipt confirmation
   
TASK 4: Purchasing Officer verifies invoice against PO
   
EXCLUSIVE GATEWAY 2: Invoice matches PO?
    NO: TASK 4a: Contact supplier for correction OR
           instruct AP to hold invoice
            WAIT for corrected invoice  LOOP BACK to TASK 4
   
         ADDITIONAL CONDITION: If PO number missing/invalid despite
         initial check, Purchasing requests AP to obtain correct PO
          LOOP BACK to TASK 1
   
    YES: TASK 4b: Purchasing sends confirmation to AP
            
TASK 5: AP matches invoice line items against PO in system
   
EXCLUSIVE GATEWAY 3: Match successful?
    NO: TASK 5a: Email Purchasing to clarify discrepancy
           
         EXCLUSIVE GATEWAY 4: Discrepancy resolved?
            YES:  Continue to TASK 6
            NO: TASK 5b: Escalate to AP Supervisor
                    
                  TASK 5c: AP Supervisor investigates
                  (contacts supplier, coordinates with departments)
                    
                  [For invoices without PO: AP Supervisor works with
                   department heads to create retrospective PO or reject]
                    
                  TASK 5d: After resolution, return to normal flow
                   LOOP BACK to TASK 4 (Purchasing re-confirmation)
   
    YES:  Continue to TASK 6
            
TASK 6: AP sends invoice to Department Manager for approval
   
TASK 7: Department Manager reviews invoice, PO, and confirmation
   
EXCLUSIVE GATEWAY 5: Manager approves?
    NO: TASK 7a: Manager rejects or requests investigation
            May involve AP Supervisor and/or Supplier Relationship Manager
            After resolution, LOOP BACK to TASK 7 or END with rejection
   
    YES: TASK 8: AP schedules invoice for next payment cycle
            
END EVENT: Invoice processed for payment
```

## 3. Data Objects/Documents
- Invoice (received via email)
- Purchase Order (PO)
- Accounting system entry
- Confirmation email from Purchasing
- Payment schedule entry

## 4. Exception Paths & Variations

**Primary Exceptions:**
1. **Missing PO**: Invoice returned to supplier  corrected invoice received  re-enters at TASK 1
2. **Invoice/PO Mismatch**: 
   - Minor discrepancies: Supplier contacted  corrected invoice  re-enters at TASK 4
   - Major discrepancies: Escalated to AP Supervisor  investigation  re-enters at TASK 4
3. **Informal Contracts (no PO)**: AP Supervisor works with department heads  retrospective PO created or invoice rejected
4. **Manager Concerns**: Investigation required  may involve Supplier Relationship Manager  re-evaluation

**Uncertainty Note:** 
There is ambiguity about whether corrected invoices after Purchasing verification re-enter at TASK 1 or TASK 4. The model assumes they re-enter at TASK 4 if the PO is already logged, but some stakeholders suggest they might need full reprocessing from TASK 1.

## 5. System Interactions
- Accounting system: Used for logging and matching
- Email system: Primary communication channel
- Approval notification system: Automatic manager alerts
- Payment scheduling system: Final processing

This model represents the consolidated "happy path" with noted exceptions. The process is primarily sequential with decision points at each verification stage, and includes multiple feedback loops for exception resolution.
