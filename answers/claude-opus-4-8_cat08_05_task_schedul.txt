# Data-Driven Scheduling Optimization for Precision Parts Inc.

## A Process Mining-Based Approach to Job Shop Scheduling

---

## 1. Analyzing Historical Scheduling Performance and Dynamics

### 1.1 Reconstructing Actual Job and Task Flows

The foundation of this analysis is transforming the MES event log into a process-mining-ready format. The key mapping is:

- **Case ID**  Job ID (JOB-7001)
- **Activity**  Task type + lifecycle transition (e.g., "Cutting:Queue Entry", "Cutting:Setup Start", "Cutting:Task Start", "Cutting:Task End")
- **Timestamp**  event time
- **Resource**  Machine ID + Operator ID
- **Attributes**  Priority, Due Date, Setup Required, Planned/Actual Durations, Previous Job

Because each task carries multiple lifecycle events (Queue Entry  Setup Start  Setup End  Task Start  Task End), I would use an **interval/lifecycle-aware event log** rather than treating each event as atomic. This allows precise decomposition of time into **queue / setup / processing** segments — critical for a setup-dominated job shop.

**Process discovery techniques:**

- **Directly-Follows Graphs (DFG)** with frequency and performance overlays to visualize the actual routing network and dominant flows between machines.
- **Inductive Miner (infrequent variant)** to produce a sound process model robust to the noise inherent in high-mix routings.
- **Fuzzy Miner** to handle the "spaghetti" nature of a high-mix, low-volume shop where routings vary per job.
- **Variant analysis** — clustering jobs by their routing signature (sequence of operations), since "unique routings" actually collapse into a manageable number of *routing families*.

A crucial complement is the **resource/machine perspective**: building a per-machine timeline (a "resource-centric log") where each machine is treated as the case. This reveals the *actual execution sequence* of jobs on each machine — essential for sequence-dependent setup analysis and bottleneck diagnosis.

### 1.2 Quantifying Key Metrics

**Flow times, lead times, makespan:**
- Lead time per job = (Last Task End)  (Job Released). Compute the full distribution (not just means) — report median, P90, P95, and coefficient of variation (CV) to capture predictability.
- Flow time decomposition: For each job, partition lead time into **value-added time** (Task processing), **setup time**, and **non-value-added waiting** (queue time). Typical job shops show 80–95% of lead time as waiting — quantifying this is the single most powerful insight.
- Makespan analyzed per planning horizon (daily/weekly buckets) to track shop-level throughput.

**Task waiting times (queue analysis):**
- Queue time per task = (Setup Start or Task Start)  (Queue Entry). 
- Aggregate per machine and per routing family. Build **waiting-time heatmaps** by machine × time-of-day × day-of-week to expose congestion patterns.
- Correlate queue length (number of jobs in "Queue Entry" but not yet started at a machine, derived by event-time reconstruction) with downstream delay.

**Resource utilization:**
- For each machine, sum over the horizon: Productive time (Task StartEnd), Setup time (Setup StartEnd), Idle time (gaps), and Down time (Breakdown intervals).
- Compute Utilization = (Productive + Setup) / Available. Distinguish **productive utilization** from **setup-inflated utilization** — a machine 90% "busy" but 30% in setup is a red flag.
- Operator utilization computed separately, since operator availability can constrain machine usage independent of machine availability.

**Sequence-dependent setup times:**
This is the analytical centerpiece. The log records `Setup Required`, `Previous Job`, actual setup duration, and current job. I would:
1. Build a transition dataset: for each machine, tuples of *(prev_job_attributes, current_job_attributes, actual_setup_duration)*.
2. Since job-to-job pairs are sparse in high-mix shops, **abstract jobs into setup-relevant feature classes** (e.g., material type, fixture type, tolerance class, geometry family) derived from job attributes.
3. Construct an empirical **setup time matrix S[machine][class_from][class_to]** with distributions, not just means.
4. Use regression / tree-based models (e.g., gradient-boosted trees) to predict setup duration from feature transitions, quantifying which attribute changes drive long setups. This becomes a reusable *setup prediction model* for later scheduling strategies.

**Schedule adherence and tardiness:**
- Tardiness T = max(0, Completion  DueDate); Lateness L = Completion  DueDate.
- Report % tardy jobs, mean/max tardiness, total weighted tardiness (weighted by priority), and the tardiness distribution.
- Stratify by routing family and priority to see *which* jobs are systematically late.

**Impact of disruptions:**
- Treat `Breakdown Start/End` and `Priority Change` as marked events. Use **interventional / before-after analysis**: compare KPIs in time windows containing disruptions vs. baseline windows.
- Quantify breakdown propagation: trace which jobs were queued at/routed to a failed machine and measure their incremental delay (cascade effect).
- For hot jobs, measure the *collateral delay* imposed on other jobs that were preempted or deprioritized.

## 2. Diagnosing Scheduling Pathologies

Using the metrics above, I would surface and **evidence** the following pathologies:

### 2.1 Bottleneck Identification
- **Evidence:** Machines with simultaneously high utilization, long and persistent queues (high average WIP-in-front), and high "synchronization" — i.e., downstream machines wait for them. Process-mining **performance DFGs** with edge-coloring by waiting time reveal these visually.
- Quantify bottleneck impact via **CONWIP/Little's Law**: WIP = Throughput × Lead Time. A machine whose queue dominates total lead time is the throughput constraint (Theory of Constraints lens).

### 2.2 Poor Task Prioritization
- **Variant analysis comparing on-time vs. late jobs:** segment cases into "on-time" and "tardy" cohorts and compare their queue-time profiles. If high-priority/near-due jobs show *long queue times behind low-priority jobs*, the FCFS/local-EDD logic is failing globally.
- **Evidence:** Detect "overtaking" events where a later-arriving, lower-priority job starts processing before an earlier high-priority job at the same machine — directly visible in the resource-centric log.

### 2.3 Suboptimal Sequencing Inflating Setups
- Using the setup matrix, compute **actual total setup time** vs. a **counterfactual minimum** (the setup time achievable under an optimized sequence for the same job set — solvable as a TSP-like sequencing problem per machine per period). The gap quantifies avoidable setup waste.
- **Evidence:** FCFS sequencing scatters similar jobs, causing frequent high-cost class transitions.

### 2.4 Downstream Starvation
- **Evidence:** Periods where a downstream machine is idle while upstream WIP exists but is "trapped" behind the bottleneck or mis-sequenced. Detected by correlating idle intervals on machine B with queue buildup at upstream machine A.

### 2.5 WIP Bullwhip
- Plot WIP-over-time per work center (reconstructed from Queue Entry/Task End events). Amplifying oscillations downstream indicate scheduling-induced variability rather than demand variability — confirmed by comparing input variance vs. WIP variance across stages.

## 3. Root Cause Analysis

| Pathology | Candidate Root Causes |
|---|---|
| High tardiness | Local dispatching rules optimize per-station, not globally; no due-date-aware sequencing at bottleneck |
| Excess setups | No setup-awareness in sequencing; FCFS ignores job similarity |
| Starvation/blocking | No coordination/look-ahead between work centers |
| Disruption fragility | No reactive rescheduling protocol; ad-hoc hot-job insertion |
| Unpredictable lead times | Planning uses planned durations, but actuals deviate systematically (66.4 vs 60 min) |

### 3.1 Differentiating Scheduling Logic vs. Capacity vs. Variability

This distinction is critical to avoid solving the wrong problem:

- **Capacity-limited (cannot be scheduled away):** If a bottleneck's **productive utilization alone** (excluding setup and idle) approaches 100% over sustained periods, the problem is capacity, not scheduling. No sequencing rule creates capacity — this requires investment or offloading.
- **Scheduling-logic-driven (recoverable):** If the bottleneck shows significant **idle + setup** time while queues build, OR if the counterfactual setup-optimized schedule shows large gains, the loss is recoverable through better scheduling.
- **Inherent variability:** Quantify CV of actual task/setup durations. High CV with low mean utilization but high queue times points to variability-induced queuing (per Kingman's equation: waiting  utilization/(1utilization) × variability). Mitigated by variability reduction, not pure sequencing.

Process mining enables this decomposition by separately measuring productive/setup/idle/down time and the variability of each — letting us attribute lost throughput to its true source.

## 4. Advanced Data-Driven Scheduling Strategies

### Strategy 1: Multi-Factor Dynamic Dispatching (Composite Priority Index)

**Core logic:** Replace static FCFS/EDD with a real-time composite score computed for every job in a machine's queue at each dispatch decision:

```
Score(job j on machine m) = 
    w · Slack-based urgency        [ (DueDate  Now  RemainingWork) ]
  + w · Priority weight
  + w · Estimated_Setup(prev_on_m  j)   [from mined setup model, negative weight]
  + w · Downstream_Congestion(next machine of j)  [favor jobs whose next station is starved]
  + w · Critical-Ratio term
```

The job with the best score is selected next. This is effectively a learned, setup- and look-ahead-aware ATC (Apparent Tardiness Cost) rule extended with setup.

**Process mining inputs:** 
- Setup prediction model (Section 1.2) supplies w term.
- Remaining work computed from mined actual-duration distributions per routing family.
- Downstream congestion from real-time queue reconstruction (validated against mined patterns).
- Weights w **tuned via the simulation** (Section 5), not guessed.

**Pathologies addressed:** Poor prioritization (slack + priority terms), setup waste (w), starvation (downstream term).

**Expected KPI impact:** Tardiness  25–40%, setup time  15–30%, reduced starvation, smoother WIP.

### Strategy 2: Predictive / Proactive Scheduling

**Core logic:** Build a rolling-horizon predictive schedule using:
- **Mined task duration distributions** stratified by operator skill, job complexity, and machine — replacing optimistic planned durations with realistic (and risk-buffered, e.g., P75) estimates.
- **Predictive breakdown model**: from historical breakdown events, estimate per-machine failure hazard / MTBF and build expected-availability profiles; flag machines at elevated risk.
- **Bottleneck forecasting**: project queue evolution forward to predict which machines will become bottlenecks in the next shift and which jobs are at-risk of tardiness, triggering pre-emptive action (expediting, alternative routing).

**Process mining inputs:** duration distributions, breakdown frequency/timing models, routing-family probabilities.

**Pathologies addressed:** Unpredictable lead times (realistic estimates), disruption fragility (anticipated availability), reactive firefighting (proactive at-risk flagging).

**Expected KPI impact:** Lead-time predictability (P90 quote accuracy)  substantially, fewer surprise tardiness events.

### Strategy 3: Setup-Aware Batching & Bottleneck Sequence Optimization

**Core logic:** At the bottleneck machine(s), instead of greedy dispatch, periodically (every planning window) solve a **constrained sequencing optimization** that minimizes total weighted tardiness + total setup time, using the mined setup matrix as the cost structure. Group setup-compatible jobs (same material/fixture class) into **dynamic batches** — but cap batch size by a due-date constraint so batching never starves urgent jobs (avoiding the classic "batch too big, hurt due dates" trap).

Formulated as a weighted sequencing problem (sequence-dependent setup, akin to a TSP with due-date penalties), solved heuristically (e.g., metaheuristics / dispatch-then-improve) within the dispatch cadence.

**Process mining inputs:** setup matrix S[m][from][to], job similarity classes, tardiness weights.

**Pathologies addressed:** Setup inflation (primary), bottleneck throughput loss.

**Expected KPI impact:** Setup time at bottleneck  30–50%, recovered effective capacity  throughput , tardiness .

## 5. Simulation, Evaluation, and Continuous Improvement

### 5.1 Discrete-Event Simulation (DES) Validation

I would build a **DES digital twin** of the shop floor, parameterized *entirely* from process-mined data:
- Routing-family probabilities and operation sequences.
- Task duration distributions (fitted per machine/operator/complexity).
- Sequence-dependent setup model (the mined matrix).
- Breakdown processes (mined MTBF/MTTR distributions).
- Order arrival and priority-change processes.

**Validation step:** First replay history through the simulator under the *baseline* dispatching rules and confirm it reproduces observed KPIs (lead time, tardiness, utilization) within tolerance — establishing credibility before testing new strategies.

**Experimental design — scenarios to test:**
- **Baseline load** (representative period).
- **High-load / surge** (peak order arrival).
- **High-disruption** (elevated breakdown frequency + hot-job injection).
- **Setup-heavy job mix** (many class transitions).
- **Combined stress** (surge + disruption simultaneously).

Each strategy (and combinations) is run with **multiple replications** for statistical confidence; compare via mean weighted tardiness, WIP, lead-time P90, utilization, and robustness (variance across replications). Use this environment to **tune the dispatching weights w** and batch-size caps via optimization (e.g., response surface / Bayesian optimization). Only strategies that win on both performance *and* robustness proceed to a phased live pilot.

### 5.2 Continuous Monitoring & Adaptation

Deploy a **closed-loop process-mining monitoring layer**:
- **Live KPI dashboards** recomputed from streaming MES events: tardiness, WIP, utilization, setup ratio, queue lengths.
- **Concept-drift detection:** statistically monitor task/setup duration distributions and routing mix for drift (e.g., new product families, machine degradation shifting durations). Trigger alerts when drift exceeds control limits.
- **Conformance checking:** compare actual execution against the scheduler's intended plan; large/systematic deviations indicate either model staleness or unmodeled disruptions.
- **Periodic re-mining & re-calibration:** on a scheduled cadence (or drift-triggered), refresh setup matrices, duration distributions, and breakdown models, then re-tune dispatching weights in the simulator before pushing updates.
- **A/B / champion-challenger evaluation:** test refined rules on a subset of work centers or in shadow mode before full rollout.

This creates a self-improving loop: **mine  diagnose  design  simulate  deploy  monitor  re-mine**, ensuring the scheduling logic adapts as the shop, product mix, and equipment evolve.

### Summary

The approach moves Precision Parts Inc. from reactive, local, static dispatching to a **globally-aware, setup-conscious, predictive, and continuously-adapting** scheduling system — grounded at every step in evidence extracted from their own MES event logs, validated in simulation, and maintained through ongoing process mining.
