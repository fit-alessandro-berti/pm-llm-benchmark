Below is a selfcontained DuckDB SQL solution.  
It uses a window function (`LEAD`) to look ahead two rows for every event inside a case, checks that the three activities appear **consecutively**, and that the interval between *Create Order* and *Confirm Order* exceeds 5days.  
All caseids that satisfy this condition are collected in a CTE (`bad_cases`) and the final `SELECT` returns every event **except** those belonging to any of those caseids.

```sql
/* -------------------------------------------------------------
   1  Find every case that contains the exact threestep pattern
       “Approve Purchase Requisition  Create Order  Confirm Order”
       with no intervening events and a gap > 5 days between the
       second and third step.
   2  Return all events from the table that do NOT belong to any
       of those cases.
   ------------------------------------------------------------- */
WITH
-- Step1: add two “lookahead” columns for each event
event_with_leads AS (
    SELECT
        el.case_id,
        el.activity,
        el.timestamp,
        LEAD(el.activity)   OVER (PARTITION BY el.case_id
                                 ORDER BY el.timestamp)               AS next_activity,
        LEAD(el.activity,2) OVER (PARTITION BY el.case_id
                                 ORDER BY el.timestamp)               AS next_next_activity,
        LEAD(el.timestamp)  OVER (PARTITION BY el.case_id
                                 ORDER BY el.timestamp)               AS next_timestamp,
        LEAD(el.timestamp,2) OVER (PARTITION BY el.case_id
                                 ORDER BY el.timestamp)               AS next_next_timestamp
    FROM event_log AS el
),

-- Step2: keep only the rows that start the required threestep pattern
--         and satisfy the >5day gap condition.
--         The DISTINCT eliminates duplicate caseids (a case might contain
--         the pattern more than once).
bad_cases AS (
    SELECT DISTINCT case_id
    FROM event_with_leads
    WHERE activity          = 'Approve Purchase Requisition'
      AND next_activity     = 'Create Order'
      AND next_next_activity = 'Confirm Order'
      AND (next_next_timestamp - next_timestamp) > INTERVAL '5 days'
)

-- Final result: all events from cases that are NOT in the “bad” set
SELECT
    el.case_id,
    el.activity,
    el.timestamp
FROM event_log AS el
WHERE NOT EXISTS (
    SELECT 1
    FROM bad_cases AS bc
    WHERE bc.case_id = el.case_id
)
ORDER BY el.case_id, el.timestamp;
```

### How it works  

| Clause | Purpose |
|--------|---------|
| **`event_with_leads`** | For every row we look ahead one (`next_activity`, `next_timestamp`) and two rows (`next_next_activity`, `next_next_timestamp`). This guarantees the three activities are consecutive in the chronological order of the case. |
| **`bad_cases`** | Filters the windowed rows to those that start with *Approve Purchase Requisition* followed by *Create Order* followed by *Confirm Order* and where the time between the second and third activity (`next_next_timestamp - next_timestamp`) is greater than 5days. `DISTINCT case_id` ensures each offending case appears only once. |
| **Main `SELECT`** | Returns every row from `event_log` whose `case_id` is **not** present in `bad_cases`. The `NOT EXISTS` subquery efficiently discards all rows belonging to any case that met the patternplustiming condition. |

The query is fully ANSISQL compatible and runs on DuckDB (which supports window functions, interval arithmetic, and `INTERVAL` literals). Adjust the literal `'5 days'` if you need a different threshold (e.g., `'120 hours'`).
