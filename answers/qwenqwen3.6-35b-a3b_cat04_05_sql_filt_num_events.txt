Here are two clean, production-ready approaches for DuckDB. The first leverages DuckDB's native `QUALIFY` clause (recommended for conciseness and performance), and the second uses standard SQL with a `JOIN`/`CTE` for broader dialect compatibility.

###  Recommended: DuckDB `QUALIFY` Approach
```sql
SELECT *
FROM event_log
QUALIFY COUNT(*) OVER (PARTITION BY case_id) <= 6;
```
**How it works:**
- `COUNT(*) OVER (PARTITION BY case_id)` computes the total event count per case for every row.
- `QUALIFY` filters the result set **after** window functions are evaluated, keeping only rows where that count is `<= 6`.
- Returns all columns and all events from qualifying cases.

---

###  Alternative: Standard SQL `JOIN` Approach
```sql
SELECT e.*
FROM event_log e
JOIN (
    SELECT case_id
    FROM event_log
    GROUP BY case_id
    HAVING COUNT(*) <= 6
) valid_cases ON e.case_id = valid_cases.case_id;
```
**How it works:**
- The subquery identifies `case_id`s with 6 or fewer events.
- The `JOIN` filters the main table to only include rows matching those valid cases.
- Fully compatible with PostgreSQL, MySQL, SQL Server, etc.

---

###  Performance Tip for DuckDB
If your `event_log` table is large, ensure you have an index or use DuckDB's automatic vectorized filtering. For this specific query, DuckDB will typically optimize either approach efficiently. If you frequently filter by case size, consider adding a materialized view or precomputing case counts:
```sql
CREATE VIEW case_event_counts AS
SELECT case_id, COUNT(*) as event_count
FROM event_log
GROUP BY case_id;
```
Then join against `case_event_counts` instead of scanning the raw table repeatedly.
