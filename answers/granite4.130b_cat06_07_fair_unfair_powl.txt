Below are two **POWL** models that capture the hiring workflow described for **GlobalTeqSolutions**.  
Both models share the same core activities (application receipt, datacompleteness loop, skill assessment, managerial review and final decision). They differ only in how the *culturalfit* step is modelled – one includes a biased “CommunityAffiliationCheck” branch (the source of unfairness), while the other forces every candidate through an identical culturalfit evaluation.

---

## 1 POWL Model **with Potential Unfairness**

In this version the workflow contains an **XOR** (`Operator.XOR`) after the skill assessment.  
One leg is a normal `CulturalFitCheck`; the other leg executes `CommunityAffiliationCheck`, which implicitly boosts the score of applicants who belong to a local group (e.g., “Global Business Leaders Club” or are flagged as local residents). This creates a subtle, preferential path and therefore a source of bias.

```python
import pm4py
from pm4py.objects.powl.obj import (
    StrictPartialOrder,
    OperatorPOWL,
    Transition,
    SilentTransition
)
from pm4py.objects.process_tree.obj import Operator

# ----- Activities (as simple labelled transitions) -----
ReceiveApplication      = Transition(label="ReceiveApplication")
DataCompletenessCheck   = Transition(label="DataCompletenessCheck")
RequestMoreInfo         = Transition(label="RequestMoreInfo")   # loopback edge
SkillAssessment         = Transition(label="SkillAssessment")

# ----- Culturalfit branch (fair) -----
CulturalFitCheck        = Transition(label="CulturalFitCheck")

# ----- Biased communityaffiliation branch -----
CommunityAffiliationCheck = Transition(label="CommunityAffiliationCheck")

ManagerialReview       = Transition(label="ManagerialReview")
FinalDecision          = Transition(label="FinalDecision")

# -------------------------------------------------------
# 1. Datacompleteness loop
#    * Execute DataCompletenessCheck  if data missing, go to RequestMoreInfo,
#      then rerun DataCompletenessCheck (loop).
LoopData = OperatorPOWL(
    operator=Operator.LOOP,
    children=[DataCompletenessCheck, RequestMoreInfo]
)

# 2. XOR choice after skill assessment
#    Either a standard cultural fit check **or** the communityaffiliation path.
XorCulturalOrAffiliated = OperatorPOWL(
    operator=Operator.XOR,
    children=[CulturalFitCheck, CommunityAffiliationCheck]
)

# 3. Main workflow (partial order)
Workflow = StrictPartialOrder(nodes=[
    ReceiveApplication,
    LoopData,                     # loop until data is complete
    SkillAssessment,
    XorCulturalOrAffiliated,
    ManagerialReview,
    FinalDecision
])

# Execution ordering (edges) -------------------------------------------------
# Application  Datacompleteness loop  Skill assessment  cultural choice  review  decision
Workflow.order.add_edge(ReceiveApplication, LoopData)
Workflow.order.add_edge(LoopData,          SkillAssessment)
Workflow.order.add_edge(SkillAssessment,   XorCulturalOrAffiliated)
Workflow.order.add_edge(XorCulturalOrAffiliated, ManagerialReview)
Workflow.order.add_edge(ManagerialReview,  FinalDecision)

# Internal loop edges (already captured by the LOOP operator; shown for clarity) 
# DataCompletenessCheck  RequestMoreInfo  back to DataCompletenessCheck
```

**Where unfairness appears:**  
The **XOR** node `XorCulturalOrAffiliated` offers two mutually exclusive paths. The branch `CommunityAffiliationCheck` applies an *implicit* uplift for certain applicants, biasing the downstream managerial review and final decision.

---

## 2 POWL Model **Without Unfairness**

In this version every candidate follows exactly the same culturalfit path; there is no XOR that can steer a privileged subset toward a boosted evaluation. The only looping construct remains the datacompleteness check, ensuring fairness throughout.

```python
import pm4py
from pm4py.objects.powl.obj import (
    StrictPartialOrder,
    OperatorPOWL,
    Transition,
    SilentTransition
)
from pm4py.objects.process_tree.obj import Operator

# ----- Activities (as simple labelled transitions) -----
ReceiveApplication      = Transition(label="ReceiveApplication")
DataCompletenessCheck   = Transition(label="DataCompletenessCheck")
RequestMoreInfo         = Transition(label="RequestMoreInfo")   # loopback edge
SkillAssessment         = Transition(label="SkillAssessment")

# ----- Uniform culturalfit evaluation (no bias) -----
CulturalFitCheck        = Transition(label="CulturalFitCheck")

ManagerialReview       = Transition(label="ManagerialReview")
FinalDecision          = Transition(label="FinalDecision")

# -------------------------------------------------------
# 1. Datacompleteness loop
LoopData = OperatorPOWL(
    operator=Operator.LOOP,
    children=[DataCompletenessCheck, RequestMoreInfo]
)

# 2. Main workflow (partial order) – all candidates go through the same CulturalFitCheck
WorkflowFair = StrictPartialOrder(nodes=[
    ReceiveApplication,
    LoopData,                     # loop until data is complete
    SkillAssessment,
    CulturalFitCheck,            # **single** culturalfit activity for everybody
    ManagerialReview,
    FinalDecision
])

# Execution ordering (edges) -------------------------------------------------
WorkflowFair.order.add_edge(ReceiveApplication, LoopData)
WorkflowFair.order.add_edge(LoopData,          SkillAssessment)
WorkflowFair.order.add_edge(SkillAssessment,   CulturalFitCheck)
WorkflowFair.order.add_edge(CulturalFitCheck,  ManagerialReview)
WorkflowFair.order.add_edge(ManagerialReview,  FinalDecision)

# Internal loop edges (captured by the LOOP operator) 
# DataCompletenessCheck  RequestMoreInfo  back to DataCompletenessCheck
```

**Why this model is unbiased:**  
All applicants traverse **exactly the same sequence of activities** after their resumes are verified. The only conditional construct (`*`loop for data completeness) does not favor any subgroup, and there is no XOR that could route a privileged subset through a “boosted” evaluation.

---

### Summary

| Feature | Model1 (with potential unfairness) | Model2 (without unfairness) |
|---------|-------------------------------------|------------------------------|
| **Loop** | Datacompleteness `*` loop (`LOOP`) | Same datacompleteness `*` loop |
| **Choice after skill test** | **XOR**: `CulturalFitCheck` **or** `CommunityAffiliationCheck` (biased path) | No XOR – every candidate goes to a single `CulturalFitCheck` |
| **Bias source** | Implicit uplift via `CommunityAffiliationCheck` branch | None – uniform treatment |
| **Overall structure** | `Receive  LoopData  SkillAssessment  XorChoice  Review  Decision` | `Receive  LoopData  SkillAssessment  CulturalFitCheck  Review  Decision` |

These two POWL specifications make the hidden bias explicit (Model1) and demonstrate how removing the XOR branch eliminates it (Model2).
