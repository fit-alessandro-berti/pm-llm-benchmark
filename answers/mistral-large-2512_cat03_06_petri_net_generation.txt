# Accepting Petri Net for Insurance Claim Management Process

Here's the Python code to construct an accepting Petri net for the insurance claim management process using pm4py:

```python
from pm4py.objects.petri_net.obj import PetriNet, Marking
from pm4py.objects.petri_net.utils import petri_utils

def create_insurance_claim_petri_net():
    # Create the Petri net
    net = PetriNet(name="Insurance Claim Management Process")

    # Create places
    start = PetriNet.Place(name="start")
    p_cr = PetriNet.Place(name="after_registration")
    p_dv = PetriNet.Place(name="after_document_verification")
    p_rds = PetriNet.Place(name="after_re_document_submission")
    p_fc = PetriNet.Place(name="after_fraud_check")
    p_ii = PetriNet.Place(name="after_in_depth_investigation")
    p_ca = PetriNet.Place(name="after_claim_assessment")
    p_ap = PetriNet.Place(name="after_approval")
    p_p = PetriNet.Place(name="after_payment")
    p_n = PetriNet.Place(name="after_notification")
    end = PetriNet.Place(name="end")

    # Add places to the net
    net.places.update([start, p_cr, p_dv, p_rds, p_fc, p_ii, p_ca, p_ap, p_p, p_n, end])

    # Create transitions
    t_c = PetriNet.Transition(name="C", label="Insurance Claim")
    t_cr = PetriNet.Transition(name="CR", label="Claim Registration")
    t_dv = PetriNet.Transition(name="DV", label="Document Verification")
    t_rds = PetriNet.Transition(name="RDS", label="Re-Document Submission")
    t_fc = PetriNet.Transition(name="FC", label="Fraud Check")
    t_ii = PetriNet.Transition(name="II", label="In-depth Investigation")
    t_ca = PetriNet.Transition(name="CA", label="Claim Assessment")
    t_ap = PetriNet.Transition(name="AP", label="Approval")
    t_p = PetriNet.Transition(name="P", label="Payment")
    t_n = PetriNet.Transition(name="N", label="Notification")
    t_cl = PetriNet.Transition(name="CL", label="Closure")

    # Add transitions to the net
    net.transitions.update([t_c, t_cr, t_dv, t_rds, t_fc, t_ii, t_ca, t_ap, t_p, t_n, t_cl])

    # Add arcs for the main flow
    petri_utils.add_arc_from_to(start, t_c, net)
    petri_utils.add_arc_from_to(t_c, p_cr, net)
    petri_utils.add_arc_from_to(p_cr, t_cr, net)
    petri_utils.add_arc_from_to(t_cr, p_dv, net)
    petri_utils.add_arc_from_to(p_dv, t_dv, net)
    petri_utils.add_arc_from_to(t_dv, p_fc, net)
    petri_utils.add_arc_from_to(p_fc, t_fc, net)
    petri_utils.add_arc_from_to(t_fc, p_ca, net)
    petri_utils.add_arc_from_to(p_ca, t_ca, net)
    petri_utils.add_arc_from_to(t_ca, p_ap, net)
    petri_utils.add_arc_from_to(p_ap, t_ap, net)
    petri_utils.add_arc_from_to(t_ap, p_p, net)
    petri_utils.add_arc_from_to(p_p, t_p, net)
    petri_utils.add_arc_from_to(t_p, p_n, net)
    petri_utils.add_arc_from_to(p_n, t_n, net)
    petri_utils.add_arc_from_to(t_n, end, net)
    petri_utils.add_arc_from_to(end, t_cl, net)

    # Add arcs for document re-submission loop
    petri_utils.add_arc_from_to(t_dv, p_rds, net)  # Documents need re-submission
    petri_utils.add_arc_from_to(p_rds, t_rds, net)
    petri_utils.add_arc_from_to(t_rds, p_dv, net)  # Back to document verification

    # Add arcs for in-depth investigation loop
    petri_utils.add_arc_from_to(t_fc, p_ii, net)   # Fraud check raises doubts
    petri_utils.add_arc_from_to(p_ii, t_ii, net)
    petri_utils.add_arc_from_to(t_ii, p_fc, net)   # Back to fraud check

    # Add arcs for approval (conditional path)
    petri_utils.add_arc_from_to(t_ca, p_p, net)    # Skip approval if amount is below threshold

    # Create initial and final markings
    im = Marking()
    im[start] = 1

    fm = Marking()
    fm[end] = 1

    return net, im, fm

# Create the accepting Petri net
net, im, fm = create_insurance_claim_petri_net()
```

## Explanation:

1. **Places**: Represent the states between activities in the process:
   - `start`: Initial state when a claim is filed
   - Intermediate states after each activity (e.g., `after_registration`, `after_document_verification`)
   - `end`: Final state when the claim is closed

2. **Transitions**: Represent the activities in the process with their labels:
   - Main activities (Claim Registration, Document Verification, etc.)
   - Exception handling activities (Re-Document Submission, In-depth Investigation)

3. **Arcs**:
   - Main flow connects all activities in sequence
   - Loops for document re-submission (DV  RDS  DV)
   - Loops for in-depth investigation (FC  II  FC)
   - Conditional path for approval (CA can go directly to P if amount is below threshold)

4. **Markings**:
   - Initial marking (`im`) has a token in the `start` place
   - Final marking (`fm`) expects a token in the `end` place

This Petri net models the complete insurance claim process with all possible paths, including the loops for document re-submission and in-depth investigation.
