## 1. Analyzing Resource Behavior and Assignment Patterns

To gain a data-driven understanding of TechSolve’s resource utilization, I would leverage the event log to construct a comprehensive performance and behavioral profile for each agent and tier. The key metrics and techniques are:

### Metrics per Agent / Tier
- **Workload distribution**: Total number of tickets handled, total active working time (sum of work intervals), and the number of concurrent tickets (if overlapping). This reveals overloaded vs. underutilized agents.
- **Activity processing times**: Average duration of “Work L1 Start” to “Work L1 End” for each L1 agent; similarly for L2/L3. Compare with tier averages and ticket complexity.
- **Firstcall resolution (FCR) rate per L1 agent**: Proportion of tickets that end at L1 without escalation. High variation indicates skill or empowerment gaps.
- **Frequency of handling specific ticket types / required skills**: For each agent, count of tickets by `Required Skill`. This shows whether agents work on tickets matching their documented skills or are assigned mismatched tasks.
- **Escalation and reassignment frequency**: For L1 agents, how often they escalate; for L2/L3, how often they reassign to another tier or to another specialist within the same tier.

### Process Mining Techniques

- **Resource interaction analysis (handover graph)**: Build a directed graph where nodes are agents (or roles) and edges represent handovers (e.g., “Work L1 End”  “Assign L2”). Edge weight = count of handovers. This reveals the *actual* escalation and reassignment paths—many handovers between certain agents may indicate a bottleneck or an excessive chain.
- **Social network analysis (SNA)**: Calculate centrality measures (e.g., betweenness, degree) to identify agents who act as key connectors (often causing delays) or isolated agents seldom receiving tickets. Comparing with the intended logic (L1L2L3, minimal crosstier reassignments) highlights deviations.
- **Role discovery**: Cluster agents based on their activity profiles (e.g., types of tickets handled, escalation behavior, typical processing times). This can reveal “hidden roles” – e.g., some L1 agents actually perform L2like work, or certain L2 agents frequently handle L1 tasks.
- **Skill utilization analysis**: For each agent, compute a “skill match ratio” = (number of tickets where `Required Skill` is a subset of `Agent Skills`) / (total tickets handled). Low ratios indicate specialists are assigned to tasks below their skill level (or lacking necessary skills). Conversely, high ratios with long handling times might indicate overspecialization causing delays.

**Contrast with intended logic**: The intended design likely assumes a strict L1firsttriage, then escalation to L2/L3 based on skill requirements. The actual process, as mined, will show deviations such as selfassignments by L2 agents to L1eligible tickets, or multiple reassignments within L2 caused by missing skills.

## 2. Identifying ResourceRelated Bottlenecks and Issues

Using the mined metrics and patterns, I would pinpoint specific problems:

### Bottleneck: Insufficient availability of agents with a required skill
- **Detection**: For the most common `Required Skill` (e.g., `Networking-Firewall`), compare the total workload (active time) of agents possessing that skill versus the total time demanded (sum of work durations for tickets requiring that skill). If demand > supply, delays will accumulate.  
- **Quantification**: Average waiting time between “Assign L2” and “Work L2 Start” for tickets with that skill. This is the queue delay attributable to skill shortage.

### Delay due to frequent reassignments
- **Detection**: Identify case variants containing “Reassign” activity. Compute the average additional idle time per reassignment (the interval between the previous work end and the new work start). Multiply by the number of reassignments to estimate total delay.  
- **Quantification**: Example: “INC-1001” had a reassignment causing ~70 minutes idle time. Aggregating all such delays reveals the contribution to SLA breaches.

### Impact of incorrect initial assignment (L1 or dispatcher)
- **Detection**: Compare `Required Skill` with the actual skill set of the first assigned agent. If mismatch, the ticket is likely to be escalated or reassigned. Use decision mining to see which ticket attributes (e.g., Category, Priority) correlate with correct first assignment.  
- **Quantification**: Percentage of tickets where the first assigned L1 agent lacked the required skill, leading to immediate escalation. Correlate with SLA breach probability.

### Overloaded vs. underutilized agents
- **Detection**: Plot each agent’s total active time vs. the average. Flags: agents in the top 20% of workload with aboveaverage handling times (overloaded, possibly rushing); agents in the bottom 20% with low handling times (underutilized).  
- **Quantification**: For overloaded agents, calculate how many of their tickets experienced SLA breaches. For underutilized, estimate the lost capacity that could be redirected.

### Correlation between assignment patterns and SLA breaches
- **Approach**: Build a logistic regression or decision tree using mined features: number of reassignments, first assignment skill match, tier escalation count, queue wait time. The model’s coefficients show the relative impact of each factor on SLA breach probability. For instance, each additional reassignment may increase breach probability by 15%.

## 3. Root Cause Analysis for Assignment Inefficiencies

Potential root causes, investigated through variant analysis and decision mining:

- **Roundrobin ignoring skills/workload**: If the `Assign L1` activity does not consider `Required Skill` or current workload, we will see mismatches. Compare variants: cases assigned via roundrobin (no skill consideration) vs. those where the dispatcher manually chose an agent. The latter may have lower reassignment rates, proving the rule’s deficiency.
- **Inaccurate or incomplete agent skill profiles**: If an agent’s documented skills (e.g., `App-CRM`) are missing from the log, but the agent frequently resolves `App-CRM` tickets, then the log is correct – but the HR system may be outdated. Conversely, if the agent lacks a skill but is assigned anyway, it suggests a data quality issue.
- **Poor initial ticket categorization**: The `Ticket Category` and `Required Skill` may be inferred from freetext descriptions. If the initial categorization is wrong, the dispatcher might assign an underskilled agent. Use **decision mining** on the first activity (`Ticket Created`) to see how `Category` and `Priority` influence the first assignment. If many mismatched assignments come from ambiguous categories (e.g., “Software-App” could require `App-CRM` or `Database-SQL`), then the root cause is categorization.
- **Lack of realtime workload visibility**: Agents selfassign (e.g., Agent A02 in the log) or dispatchers assign without seeing current queues. The result is overloaded agents picking up new tickets while others idle. Compare the workload entropy (uneven distribution) across shifts – high entropy suggests lack of visibility.
- **Insufficient L1 empowerment**: L1 agents may escalate too quickly (low FCR) due to lack of knowledge or authority. Variant analysis: For similar tickets (same Category & Required Skill) handled by different L1 agents, compare resolution time and escalation rate. If certain agents consistently escalate, they may need training.

**Variant analysis approach**: Cluster cases by the number of reassignments/escalations (e.g., “smooth” = 0–1, “bumpy” = 2+). For each cluster, compare distributions of `Ticket Category`, `Required Skill`, first assigned agent tier, and time of day. Statistically significant differences (e.g., bumpy cases have a higher proportion of `Network` tickets, or were assigned during peak hours) point to root causes.

## 4. Developing DataDriven Resource Assignment Strategies

I propose three concrete strategies, each leveraging process mining insights:

### Strategy A: SkillWeighted Routing with Proficiency Levels
- **Issue addressed**: Frequent reassignments due to skill mismatch; specialists handling lowlevel tasks.
- **Insight from mining**: The event log reveals which skills truly resolve which ticket types. For each `Required Skill`, compute a proficiency score per agent based on average resolution time and success rate (no escalation). Use this score as a weight.
- **Data required**: Agent skill tags (from HR) enriched with mined performance scores; historical mapping of `Required Skill` to proficient agents.
- **Implementation**: When a new ticket arrives, the dispatcher (or automated routing system) selects the lowesttier agent whose skill proficiency for the required skill exceeds a threshold (e.g., 80th percentile of handling time). If none, escalate to the next tier. This ensures L1 handles what they can, and specialists handle complex cases.
- **Expected benefits**: Reduction of reassignments by 3050%, improved FCR, and better skill utilization.

### Strategy B: WorkloadAware Dynamic Assignment
- **Issue addressed**: Uneven workload, overloaded agents causing SLA breaches.
- **Insight from mining**: Historical workload entropy and queue wait times show that roundrobin or skillonly routing ignores current load. The log can estimate each agent’s current backlog (number of open tickets or total remaining active time).
- **Data required**: Realtime agent status (busy/idle), estimated remaining work time per active ticket (using average handling time per skill), and queue length.
- **Implementation**: A scoring formula: `Score(agent) =  * skill_match(agent, ticket) -  * current_load(agent)`. The agent with the highest score gets the ticket.  and  are tuned via simulation (see section 5) to balance skill use and load.
- **Expected benefits**: 2030% reduction in SLA breaches for P2/P3, plus improved agent satisfaction (fair load).

### Strategy C: Predictive Assignment via Machine Learning
- **Issue addressed**: Poor initial categorization leading to misassignments; dispatcher heuristic bias.
- **Insight from mining**: Decision mining shows that freetext description words (e.g., “error”, “crash”, “login”) and Category are strong predictors of `Required Skill` and complexity. Historical cases with smooth resolution provide training data.
- **Data required**: Full ticket attributes (description, category, priority) and outcome (e.g., time to resolution, number of reassignments). Use a classification model to predict the required skill and optimal tier.
- **Implementation**: At ticket creation, the model outputs a predicted `Required Skill` and a confidence score. If confidence > 90%, automatically assign the highestscoring agent (per Strategy A/B). Otherwise, route to a dispatcher with suggested skills.
- **Expected benefits**: Almost eliminates initial misassignments; reduces average firstcontact time and reassignment rate further. Could yield 4050% fewer reassignments.
