# Process Mining-Driven Optimization of Resource Assignment at TechSolve Solutions

## 1. Analyzing Resource Behavior and Assignment Patterns

### 1.1 Agent and Tier Performance Metrics

The event log provides a rich foundation for computing per-resource and per-tier performance indicators. I would compute the following metrics, slicing the log by `Resource`, `Agent Tier`, and `Agent Skills`:

**Per-Agent Workload Metrics**
| Metric | Computation | Insight |
|---|---|---|
| Ticket Volume | Count of distinct `Case ID`s per agent | Raw workload distribution |
| Active Processing Time | Sum of durations from `Work * Start` to `Work * End` events per agent | True capacity consumed |
| Idle Time | Total logged time minus active processing time | Underutilization |
| Queue Waiting Time | Time between `Assign L*` and the corresponding `Work * Start` | Queue saturation for that agent/skill |

**Per-Agent Performance Quality**
| Metric | Computation | Insight |
|---|---|---|
| First-Call Resolution (FCR) – L1 only | % of L1-handled cases closed without escalation | L1 effectiveness |
| Escalation Rate | % of L1 cases that produced an `Escalate` event | L1 capability gaps |
| Reassignment Rate | # of `Reassign` events per case per agent | Assignment quality |
| Average Resolution Time | End-to-end case duration by `Resource` who completed it | Productivity |
| SLA Breach Rate | % of cases where resolution time exceeded the SLA threshold, sliced by owning agent | Compliance contribution |

**Per-Tier Metrics**
- Throughput (cases completed/day), average waiting time before work start, average active processing time, and hand-off delay to the next tier. From the snippet, INC-1001 sat in L2 queue from 09:40:15 (Assign L2) to 10:05:50 (Work L2 Start) — a **25 min 35 s queue delay** — pointing to L2 capacity constraints for `App-CRM`.

**Per-Skill Metrics**
- Demand vs. supply: count of cases per `Required Skill` (demand) vs. count of agents possessing that skill (supply).
- Skill match rate: % of `Work Start` events where the assigned agent's `Agent Skills` intersect the `Required Skill`.
- Skill proficiency proxy: average resolution time per agent per skill (lower = higher proficiency).

### 1.2 Process Mining Techniques for Assignment Patterns

**Resource Activity Matrix and Handover Analysis**
A *resource activity matrix* (Resource on rows, Activity on columns, frequency as cell values) reveals who actually performs which activities. From the snippet, we already see that Agent B12 (L2) handled an `App-CRM` ticket despite also having `DB-SQL` — and then had to reassign it because the *actual* requirement was `DB-SQL`. This pattern is invisible without a matrix view.

**Social Network Analysis (SNA) – Handover of Work Network**
Build a directed weighted network where nodes are resources and an edge AB exists when A performed an activity immediately before B on the same case. The edge weight = # of handovers. From the snippet:
- Dispatcher  A05 (L1)  B12 (L2)  Dispatcher  B15 (L2) for INC-1001
- Agent A02 (L1)  Dispatcher  B08 (L2) for INC-1002

Aggregating across the year will reveal:
- The *de facto* routing topology.
- Whether the Dispatcher is always the routing bottleneck.
- Whether informal "back-channels" exist (e.g., A05 always hands to B12).

**Role Discovery (Clustering)**
Apply a role-discovery algorithm (e.g., the one in ProM / Celonis) that clusters resources based on their activity profiles. We can expect the discovery to confirm an L1/L2/L3 split, but it may also surface:
- "L1-plus" agents who frequently perform L2 activities (specialists mis-deployed as L1).
- "Generalist" L2 agents handling many skills vs. "niche" L2 agents handling only one.
- A "Dispatcher-plus" role if certain L2 agents also perform dispatching.

Comparing **discovered roles vs. intended roles** quantifies the gap between design and reality — a key diagnostic.

**Reassignment and Escalation Path Analysis**
Filter the log to cases with  2 `Assign` events. Build a process map showing the most frequent reassignment chains (e.g., `Assign L2 (B12)  Reassign  Assign L2 (B15)  Resolve`). The control-flow variants involving many reassignments are the prime targets for optimization.

### 1.3 Skill Utilization Analysis

For every `Work Start` event, compute:
- `Skill Match = 1` if `Required Skill`  `Agent Skills`, else 0.
- `Skill Excess` = number of agent skills not required by the ticket (a proxy for over-qualification).

The snippet shows Agent B12 (skills: `App-CRM, DB-SQL`) being assigned a ticket with `Required Skill = App-CRM`. The match is 1 but the case is later reassigned when DB-SQL turns out to be needed. Two questions arise:
1. Was the `Required Skill` tag wrong at intake?
2. Did B12 have an outdated view of what the ticket required?

Across the full log, I would compute:
- **% of L1 starts where Skill Match = 0** (L1 only has `Basic-Troubleshoot` so this is expected — but the question is whether L1 escalations correlate with required-skill categories that L1 can *never* match).
- **% of L2/L3 starts where Skill Excess > 0** — a high value indicates specialists are being underutilized (e.g., a DB-SQL expert doing only App-CRM work).
- **Skill coverage gap**: required skills for which < 3 agents exist (fragility).

## 2. Identifying Resource-Related Bottlenecks and Issues

### 2.1 Bottleneck Identification

**Skill-Specific Queue Bottlenecks**
For each `(Required Skill, Tier)` combination, compute the average and 90th-percentile time from `Assign L*` to `Work * Start`. Skills with consistently high queue times are bottlenecked. From the snippet, `App-CRM` already shows a 25-min L2 queue for INC-1001; if this pattern is pervasive, the L2 pool for `App-CRM` is under-sized.

**Activity Processing-Time Outliers**
Box-plots of `Work Start  Work End` duration per `(Activity, Resource)` surface agents who are systematically slower — potentially overloaded, undertrained, or working on tasks beyond their skill set.

**Throughput Imbalance**
Compute the Gini coefficient of tickets handled per agent per month. A Gini > 0.4 indicates significant workload imbalance, consistent with the "some overloaded, some underutilized" observation.

### 2.2 Reassignment and Escalation Delays

Define:
- `Reassignment Delay = Time(Reassign  next Work Start)` — the idle time caused by handoff.
- `Escalation Delay = Time(Escalate  Work at next Tier Start)` — the L2/L3 pickup delay.

Average these across all cases. From the snippet, INC-1001 incurred two delays:
- L1L2 escalation: 09:36:00 (Escalate)  10:05:50 (Work L2 Start) = **~30 min**.
- L2 internal reassignment: 11:15:00 (Reassign)  unknown (but the Dispatcher step at 11:16:10 and the eventual new assignment add further minutes).

If the average reassignment adds, say, 45 minutes and 18% of cases are reassigned, **reassignments alone could add ~8 minutes to the mean resolution time** — a quantifiable target for improvement.

### 2.3 Impact of Initial Assignment Quality

Compute the **conditional mean resolution time** for:
- Cases with 0 reassignments vs. 1, 2, 3+ reassignments.
- Cases with correct L1 skill match vs. cases that required an escalation L1 could not have solved.

A monotonically increasing relationship confirms the cost of poor initial routing.

### 2.4 Identifying Overloaded vs. Underutilized Agents

For each agent, plot:
- **% of logged time active** (occupancy).
- **# of concurrent open cases** (work-in-progress, derived from open `Work Start` events without matching `End`).

An agent with 95% occupancy and a long WIP list is overloaded; one with 20% occupancy and a short WIP list is underutilized. Correlate this with SLA breach rate per agent.

### 2.5 SLA Breach Correlation

Slice the cases by:
- Owning tier when breach occurred.
- Number of reassignments before breach.
- Skill match of the agent who finally resolved vs. skill match of intermediate agents.

The slice that shows the highest breach concentration reveals the *causal correlate* (not necessarily causation, but a strong signal). For example, if **70% of P2 breaches occur in cases with  2 reassignments**, the linkage is clear.

## 3. Root Cause Analysis for Assignment Inefficiencies

### 3.1 Hypothesized Root Causes

| # | Hypothesis | How to Test with the Log |
|---|---|---|
| RC1 | Round-robin routing ignores skills | Compute correlation between assignment order and skill match; cluster agents by skill profile to see if all skills get the same throughput |
| RC2 | Skill profiles are outdated/over-stated | Compare `Agent Skills` field with the actual skills on tickets the agent successfully resolves; compute precision/recall per agent |
| RC3 | Initial ticket categorization is wrong | Compare `Ticket Category` declared at `Ticket Created` with the `Required Skill` that actually drove resolution; high divergence = mis-categorization |
| RC4 | No real-time workload visibility | Look for self-assignment patterns (e.g., INC-1002 "Self-assigned") and see if agents who self-assign are less overloaded than those who wait |
| RC5 | L1 training is insufficient | Compute FCR by `Required Skill` and by `Ticket Category`; low FCR for certain categories indicates training gaps |
| RC6 | Escalation criteria are too loose | Compare L1 `Escalate` notes with subsequent L2 actions; if L2 often just "did the same thing L1 could have done," criteria are loose |

### 3.2 Variant Analysis

Partition cases into cohorts:
- **Happy path**: 0 reassignments, resolved within SLA.
- **Pain path**:  2 reassignments OR breached SLA.

Build process maps for each. Differences in the *attributes* (priority, category, required skill, initial agent, hour-of-day, channel) reveal the discriminators. A decision tree (e.g., in Celonis Action Engine) trained on these features can output rules like:
- *IF* `Priority = P2` *AND* `Category = Network` *AND* `Required Skill = Networking-Firewall` *AND* `L1 Agent has skill = Basic-Troubleshoot only` *THEN* P(reassignment) = 0.78.

### 3.3 Decision Mining

Apply decision mining to the points in the process where a branching choice exists (escalate vs. resolve, reassign vs. continue, assign-to-which-agent). For each branching point, train a classifier (e.g., random forest) on case attributes up to that point. The resulting feature importances surface which data the decision-maker *should* have used — and the gap between the model's recommendations and the actual decisions is the **assignment inefficiency**.

For example, if the decision tree shows that `Required Skill` is the most predictive feature for successful L1 resolution, but the actual dispatcher ignores it (round-robin), the conflict is identified.

### 3.4 L1 Empowerment Audit

Compute, per L1 agent, the **cohort FCR**: the FCR of other L1 agents on the same `(Category, Required Skill)` combination. If an L1 agent's FCR is much lower than their cohort's, they are individually underperforming; if the cohort FCR is low, the entire L1 pool is under-trained on that category.

## 4. Data-Driven Resource Assignment Strategies

### Strategy A — Skill-Weighted Routing with Proficiency Scoring

**Issue addressed:** Skill mismatches at assignment, and the over-/under-utilization of specialized skills (RC1, RC2).

**Insight from mining:** The handover network and role-discovery show that DispatcherB12B15 chains occur when the wrong skill is chosen. Skill utilization analysis shows B12 (with `App-CRM, DB-SQL`) gets tagged to `App-CRM` work it can't fully resolve.

**Mechanism:** Replace round-robin with a scoring function for each `(case, agent)` pair:
```
Score = w1 · SkillMatch(case, agent) 
      + w2 · HistoricalResolutionRate(agent, category)
      + w3 · (1 / CurrentWIP(agent))   // load component (Strategy B)
      + w4 · ProficiencyProxy(agent, skill)
      - w5 · SkillExcess(agent, case)  // penalize over-qualification
```
Weights `w1..w5` are learned by regression against historical SLA outcomes.

**Data required:**
- Master agent skill list (validated, with proficiency rating 1–5).
- Per-(agent, category) historical resolution rate and average resolution time.
- Real-time WIP per agent.

**Expected benefits:** 30–50% reduction in intra-tier reassignments; better use of L1 for genuinely simple cases; faster initial pickup for specialized L2/L3 skills.

### Strategy B — Workload-Aware, Real-Time Assignment Engine

**Issue addressed:** Uneven workload (some agents overloaded, some idle), queue bottlenecks for specific skills.

**Insight from mining:** Throughput-imbalance (Gini), per-skill queue-time distributions, and self-assignment patterns show that the current system has no load balancing.

**Mechanism:** A continuous-assignment engine that monitors:
- Per-agent WIP and active-processing-time budget remaining in the shift.
- Per-skill queue length.
- Per-priority SLA countdown.

When a new ticket arrives, the engine computes the eligible agent pool (matching skill) and selects the one with the **lowest projected SLA-breach risk** — a function of remaining SLA time, current queue, and historical processing time for that ticket type.

**Data required:**
- Live status feed (Available / Busy / On Break) — minimal: a heartbeat from the agent's console.
- Per-agent shift schedule.
- Historical activity processing time per `(Activity, Ticket Category)` — directly from the event log.

**Expected benefits:** Reduction in average queue waiting time, fewer SLA breaches, lower stress on overloaded agents. Expected ~20% improvement in mean end-to-end resolution time for P2 tickets.

### Strategy C — Predictive Required-Skill and Complexity Inference

**Issue addressed:** Wrong `Required Skill` at intake, wrong initial L1/L2 routing (RC3), and excessive escalations from mis-categorization.

**Insight from mining:** Decision mining reveals that the *correct* required skill is often identifiable from the ticket description and category. Variant analysis of cases like INC-1001 shows that the initial `App-CRM` tag was wrong — it should have been `DB-SQL` from the start.

**Mechanism:**
1. Train a text classifier (e.g., on the `Notes` / description field, or a free-text field derivable from the ticket) that predicts `Required Skill` and `Estimated Complexity Tier` (L1-resolvable vs. needs L2+).
2. At `Ticket Created`, the classifier produces:
   - `Predicted_Required_Skill`
   - `Predicted_Tier`
   - `Confidence`
3. If `Confidence > threshold`, the assignment engine (Strategy A/B) routes directly to that tier.
4. If low confidence, default to L1 with a knowledge-base hint.

**Data required:**
- Historical tickets with `Notes`/`description` text and final `Required Skill` after resolution.
- Labeled examples of L1-resolvable cases (cases with no escalation).
- Channel and requester metadata as auxiliary features.

**Expected benefits:** 15–25% reduction in L1L2 escalations caused by mis-categorization; faster routing of genuinely specialized tickets; lower L1 handling time on tickets they were never going to resolve.

### Strategy D — Refined Escalation Criteria with L1 Empowerment Playbooks

**Issue addressed:** Excessive L1L2 escalations (RC5, RC6).

**Insight from mining:** Variant analysis identifies categories where L1 *does* successfully resolve (e.g., password resets, basic printer troubleshooting). Cohort FCR analysis identifies the specific `Required Skill` × `Category` combinations where L1 could resolve but currently escalates.

**Mechanism:**
1. From the event log, build a "L1 Playbook" table mapping `(Category, simple keyword in description)  L1 resolution steps known to work historically`.
2. Embed the playbook into the L1 console as guided troubleshooting.
3. Modify the escalation rule: L1 can only escalate after the playbook is exhausted (or after a time threshold).
4. Track compliance via process mining — cases that escalate without playing the relevant playbook are flagged.

**Data required:**
- Historical sequence of activities inside successful L1 resolutions (per category).
- Time-to-resolve for L1-only cases vs. escalated cases.
- Notes / resolution codes from completed cases.

**Expected benefits:** 10–20% improvement in L1 FCR; significant reduction in L2/L3 queue load; more meaningful work for L2/L3 specialists.

### Strategy E — Dynamic Tier and Skill Capacity Planning

**Issue addressed:** Static tier staffing that doesn't track temporal demand patterns (a *resource planning* lever that complements the real-time assignment engine of Strategy B).

**Insight from mining:** A heatmap of ticket volume by `(Required Skill, hour-of-day, day-of-week)` from the log identifies demand cycles (e.g., `Networking-Firewall` peaks at 10:00 and 14:00; `App-CRM` peaks on Monday mornings). Mining also reveals that inter-tier loans (L1 covering L2 queue when L2 is overloaded) currently happen ad hoc.

**Mechanism:** Build a daily capacity plan that:
- Predicts the per-skill demand curve for the day.
- Pre-positions agents (including trained cross-skilling) to match the demand curve.
- Allows real-time inter-tier loans when a skill's queue exceeds a threshold.

**Data required:**
- Historical temporal demand patterns (extracted from the log).
- Agent cross-skilling matrix and training records.
- Real-time queue length feeds.

**Expected benefits:** Smoother demand-supply matching; reduced peak-hour SLA breaches; better agent utilization through planned cross-skilling.

## 5. Simulation, Implementation, and Monitoring

### 5.1 Business Process Simulation for Strategy Evaluation

Before deploying any strategy, build a **discrete-event simulation (DES)** model using the mined process model as the backbone. The simulation should be parameterized by:

**Process parameters (from the mined model)**
- Activity duration distributions per `(Activity, Required Skill, Priority)` — fit to the log's `Work Start`  `Work End` intervals.
- Branching probabilities at each decision point (e.g., probability of L1 escalation given attributes) — derived from variant analysis.
- Arrival process of new tickets — fit a non-homogeneous Poisson process to the inter-arrival times per hour-of-day.
- Required-skill distribution — empirical from the log.

**Resource parameters (from the log)**
- Number of agents per tier, per skill profile.
- Shift schedules.
- Current WIP handling rules.
- Skill match probability distribution (to model the inefficiencies of the current system).

**Scenarios to simulate**
1. **Baseline:** current round-robin + manual escalation.
2. **Strategy A only:** skill-weighted routing.
3. **Strategy A + B:** skill-weighted + workload-aware.
4. **Strategy A + B + C:** all three, including the predictive classifier.
5. **Full bundle (A + B + C + D + E).**

**Outputs to compare (over many replications)**
- Mean and 95th percentile of end-to-end resolution time per priority.
- SLA breach rate per priority.
- Mean queue waiting time per skill.
- Mean # of reassignments per case.
- Agent utilization (Gini and per-agent occupancy).
- Throughput per shift.

The DES output is a *risk-adjusted forecast* of each strategy's impact — much more credible than anecdotal estimates.

### 5.3 Post-Implementation Monitoring – Process Mining Dashboards

A continuous process-mining dashboard (e.g., in Celonis, Minit, or a custom Power BI + process-mining library) should track the following:

**Real-time operational KPIs**
- **SLA Compliance %** by priority, broken down by `(Tier, Required Skill, # of Reassignments)`.
- **Mean Reassignment Delay** and **% of cases with  2 reassignments** (target: drop from baseline, e.g., 18% to < 8%).
- **L1 First-Call Resolution %** by category (target: +10–20 pp).
- **Queue Wait Time** per `(Required Skill, Tier)` (target: 90th percentile < 5 min for P1/P2).

**Resource health KPIs**
- **Per-agent WIP** heatmap — early warning of overload.
- **Per-agent occupancy** (active processing / shift hours) — target band 70–85%.
- **Skill match rate** at every `Work Start` event (target: > 95% for L2/L3).
- **Skill excess rate** for L2/L3 — proxy for misallocation of specialists.

**Conformance & drift monitoring**
- Compare the *actual* process map monthly against the *expected* (modeled) map. New variants — e.g., an emerging pattern of "Reassign to B15" — are flagged for investigation.
- **Conformance score** for the assignment rules (e.g., did the engine's top recommendation match the actual assignment?).

**Outcome-quality KPIs**
- **Resolution time vs. SLA** for cases that were routed by the new engine vs. legacy routing — direct A/B comparison.
- **Customer satisfaction (CSAT)** if available, correlated with the # of reassignments (a strong driver of dissatisfaction).

### 5.4 Feedback Loop

The monitoring layer feeds three downstream activities:
1. **Quarterly recalibration** of the scoring function weights in Strategy A using the latest resolution outcomes.
2. **Quarterly retraining** of the Strategy C classifier with the newest labeled cases.
3. **Annual re-simulation** to validate that the assumed resource model still matches reality and to identify the next wave of optimizations (e.g., a previously hidden skill bottleneck may now be the top issue).

## Summary

By treating the event log as the primary source of truth and applying process mining techniques — resource activity matrices, handover networks, role discovery, variant analysis, and decision mining — TechSolve can move from anecdotal complaints ("we have too many reassignments") to quantified root causes ("18% of cases are reassigned, adding 45 minutes each; 70% of those reassignments are caused by L1 mis-categorization of Network tickets"). A continuous process-mining dashboard with the right resource-centric KPIs then ensures the improvements are sustained and the next round of optimization is data-driven.
