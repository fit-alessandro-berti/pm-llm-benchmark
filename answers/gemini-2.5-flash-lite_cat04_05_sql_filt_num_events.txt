Here is a DuckDB SQL query that accomplishes the task using a window function. This approach is generally efficient and concise for this type of filtering.

```sql
SELECT
    case_id,
    activity,
    timestamp
FROM (
    SELECT
        case_id,
        activity,
        timestamp,
        -- Calculate the total number of events for each case using a window function
        COUNT(*) OVER (PARTITION BY case_id) AS event_count_per_case
    FROM
        event_log
) AS LogWithCounts
WHERE
    event_count_per_case <= 6;
```

**Explanation:**

1.  **Inner Query (Subquery `LogWithCounts`):**
    *   `SELECT *, COUNT(*) OVER (PARTITION BY case_id) AS event_count_per_case FROM event_log`: This part iterates through the `event_log` table. For each row, it calculates the total number of rows (`COUNT(*)`) that share the same `case_id` using the `PARTITION BY case_id` clause of the `OVER()` window function. This calculated count is added as a new column, `event_count_per_case`, to each row.

2.  **Outer Query:**
    *   `SELECT case_id, activity, timestamp FROM ...`: This selects the original columns (`case_id`, `activity`, `timestamp`) from the results of the inner query.
    *   `WHERE event_count_per_case <= 6`: This clause filters the rows. It keeps only those rows where the `event_count_per_case` (calculated in the subquery) is less than or equal to six.

This effectively achieves the goal: any `case_id` that has more than six events will have its `event_count_per_case` greater than 6, and thus all its associated rows will be filtered out. Conversely, all events from `case_id`s with six or fewer events will be retained.
