This process represents a **Customer Order Processing Workflow**. It is designed to handle both standard inventory items (standard requests) and bespoke, made-to-order products (custom requests). The process is divided into four main phases: **Initiation**, **Validation**, **Decision & Approval**, and **Finalization**.

#### **Phase 1: Initiation & Classification**
*   **Start Event / Task A:** The process begins when a customer request is received.
*   **Gateway (XOR): Check Request Type:** The system first classifies the request.

    *   **Custom Path:** If the request requires new manufacturing, engineering, or unique specifications.

#### **Phase 2: Validation & Feasibility**
*   **Standard Path (Task B1):** "Perform Standard Validation." Here, the system checks if the order follows company rules (e.g., minimum order quantity, standard pricing tiers).
*   **Gateway (AND): Run Parallel Checks:**
    *   This gateway splits the process into two parallel threads: **Credit Check** and **Inventory Check**.
    *   Both must complete before the process can proceed. This ensures that there is sufficient stock and the customer has available credit simultaneously.
*   **Task D:** "Calculate Delivery Date." Once parallel checks are complete, the system calculates the ship date based on inventory availability and credit status.

*   **Custom Path (Task B2):** "Perform Custom Feasibility Analysis." This is a human-intensive task involving engineering or procurement teams to estimate the cost and time required to build a non-standard item.
*   **Gateway (XOR): Is Customization Feasible?:**
    *   **If YES:** Proceeds to prepare a quotation.
    *   **If NO:** Sends a rejection notice directly to the end event.

#### **Phase 3: Approval Process (The Hub)**

### **3. Explanation of Gateways**

*   **XOR Gateway (Exclusive Gateway):**
    *   **Meaning:** The process flows through exactly one path (Exclusive OR).
    *   **Usage:** Used for binary decisions, such as "Standard vs. Custom," "Feasible vs. Not Feasible," and "Approval Granted vs. Denied." Only one output path will be taken.
*   **AND Gateway (Parallel Gateway):**
    *   **Meaning:** The process splits into multiple concurrent paths and waits for **all** of them to complete before the main process resumes.
    *   **Usage:** Used in the Standard path for Credit and Inventory checks. The workflow does not proceed to calculate the delivery date until *both* checks pass.

The process clearly differentiates between pre-defined items and unique requests:

*   **Standard Requests:**
    *   **Validation:** Automated and rule-based.

*   **Custom Requests:**

    *   **Approval:** Always requires manager sign-off if feasible, as the risk/cost is higher.
