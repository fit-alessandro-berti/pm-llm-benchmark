# Integrated Event Log for Process Mining

Below is the comprehensive event log integrating all tables. Each row represents a single event, sorted by **Case ID** and then **Timestamp** to maintain chronological correctness.

## Assumptions & Notes

1. **Case ID** = `order_id` (as instructed).
2. **Resource names** were resolved using the Resources table (e.g., `SP-001`  Alice Brown).
3. **"Items Picked"** events are produced **one per line item**, since each has its own timestamp, picker, and item — this preserves granularity useful for process mining.
4. **"Order Dispatched"** and **"Order Arrived"** are produced **per delivery record** (orders may be split into multiple shipments, e.g., Order 1002).
5. **"Invoice Issued"** is produced **per invoice** (orders may have multiple invoices, e.g., Order 1002).
6. **"Payment Received"** events are linked from Payments  Invoice  Order via the `invoice_id`  `order_id` chain.
7. **Order 1004** has no Delivery, Invoice, or Payment records — only Order Created and Items Picked events exist. This is documented as an **incomplete case** (potential process deviation/bottleneck).
8. Product descriptions/categories enriched into the "Items Picked" rows as additional attributes.
9. All timestamps are in UTC (ISO 8601, as provided).

3. **Payment lead times** range from ~2 to ~5 days after invoicing, useful for **cycle-time analysis**.
