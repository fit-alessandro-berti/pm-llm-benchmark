### Explanation of Transformation Logic

To transform the raw Windows system logs into a high-quality event log, I applied the following logic:

#### 1. Case Identification (Object-Centric Approach)
I identified the **"Case"** based on the specific work object being manipulated (the file name). 
*   **Document1.docx** acts as a single case because the user performs a continuous workflow around it珦tarting the document, switching to other apps (Email, PDF, Excel) to gather information, and then returning to the document to finalize it.
*   **Quarterly_Report.docx** is treated as a separate case. Although the user's activity is interleaved with the Document1 workflow, the "work unit" for the Quarterly Report begins with an initial focus and concludes with the final drafting and closing.

#### 2. Activity Naming (Semantic Abstraction)
Raw system actions like `TYPING`, `CLICK`, `SCROLL`, and `HIGHLIGHT` are "micro-actions" that create too much noise for process mining. I abstracted these into "macro-activities" that represent business-meaningful steps:
*   **Edit Document**: Combined `TYPING` and `FOCUS` actions.
*   **Handle Email**: Aggregated `SWITCH` to Chrome, `CLICK` (open/reply), `TYPING` (the message), and `CLICK` (send). This represents the complete "Communication" sub-process.
*   **Review Reference PDF**: Combined `SWITCH` to Adobe, `SCROLL`, and `HIGHLIGHT`. This captures the intent of the user (reviewing/reading) rather than just the mechanical movement.
*   **Update Budget Spreadsheet**: Combined the `SWITCH` to Excel with the `TYPING` and `SAVE` actions.

#### 3. Handling Context Switching
In Windows logs, a `SWITCH` event is a transition. In this event log, I used the `SWITCH` events as the temporal boundaries to end one activity and start another. This allows a process mining tool to accurately model "Context Switching" as a pattern (e.g., seeing how many times a user leaves `Document1.docx` before returning to it).

#### 4. Narrative Summary
The resulting log tells a story of a user working on a larger reporting task. They start by drafting a preliminary document (`Document1.docx`), which requires them to context-switch through an email exchange, a PDF review, and a budget update. Once the information is gathered, they return to `Document1.docx` to integrate the findings. Finally, they return to the primary goal, the `Quarterly_Report.docx`, to complete the Executive Summary.
