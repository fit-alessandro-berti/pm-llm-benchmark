To solve this problem in DuckDB, the most efficient and readable approach is to use a **Window Function** (specifically `COUNT() OVER(...)`) or a **Common Table Expression (CTE)**.

Here is the SQL query using a CTE, which is generally the best practice for readability and performance:

```sql
WITH CaseCounts AS (
    -- Calculate the total number of events for each case_id
    SELECT 
        *,
        COUNT(*) OVER(PARTITION BY case_id) as event_count
    FROM event_log
)
-- Filter to keep only cases with 6 or fewer events
SELECT 
    case_id, 
    activity, 
    timestamp
FROM CaseCounts
WHERE event_count <= 6;
```

### Explanation:
1.  **`WITH CaseCounts AS (...)`**: This creates a temporary result set that includes all original columns plus a new column, `event_count`.
2.  **`COUNT(*) OVER(PARTITION BY case_id)`**: This window function calculates the total number of rows associated with each `case_id` without collapsing the rows into a single summary line (as a `GROUP BY` would do).
3.  **`WHERE event_count <= 6`**: This final filter removes any rows belonging to a case that exceeded the threshold.

### Alternative Approach (Using `IN`):
If you prefer a more standard subquery approach, you can identify the valid `case_id`s first and then filter the table:

```sql
SELECT *
FROM event_log
WHERE case_id IN (
    SELECT case_id
    FROM event_log
    GROUP BY case_id
    HAVING COUNT(*) <= 6
);
```

**Which one should you use?**
*   The **CTE approach** is usually faster in DuckDB for large datasets because it performs the window calculation in a single pass over the data.
*   The **`IN` approach** is often easier to understand for beginners, but it requires scanning the table twice (once to group/count and once to filter).
