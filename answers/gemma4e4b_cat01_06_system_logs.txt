## Explanation and Methodology

### 1. Case Identification Logic (The "Why")

The primary mechanism for identifying cases is tracking the *user's goal* and observing significant temporal breaks or clear shifts in the work product.

*   **Case A (Drafting and Coordination):** All activities from 09:00 to 09:07 are grouped under **Case A**. Although the user bounces between three different applications (Word, Chrome, Acrobat, Excel), they are all working toward a single, overarching goal: creating and coordinating the content for the Quarterly Report (documented in `Document1.docx`). The activity sequence is a single, multi-stage process of drafting, gathering external data (email/PDF), and updating metrics (Excel).
*   **Case B (Finalization):** The switch to `Quarterly_Report.docx` at 09:07:15, coupled with the fact that Case A ended by closing `Document1.docx`, signals a distinct shift in task scope. The user is no longer drafting content (which was done in Document1) but is now polishing and finalizing the main report itself. This separation creates a more accurate representation of the workflow stages.

### 2. Activity Naming and Standardization Logic (The "What")

The raw system log uses highly granular, technical verbs (e.g., `TYPING`, `FOCUS`, `SWITCH`, `SCROLL`). These were mapped to standardized, process-friendly activity names that reflect the **user's intent** rather than the technical action.
