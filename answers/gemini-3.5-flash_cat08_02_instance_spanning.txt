### 1. Identifying Instance-Spanning Constraints and Their Impact

To diagnose and optimize the warehouse’s operations, we must shift from traditional, single-instance process mining (which treats each order as an isolated path) to **multi-perspective and object-centric process mining (OCPM)**. Traditional process mining struggles with "instance-spanning" constraints because it assumes cases do not interact. 

In this system, cases compete for resources, are grouped into batches, preempt one another, and are subject to global system-capacity limits.

```

       [Quality Check] < (Hazmat Concurrency Limit:  10)
                           
                           
    [Batching Buffer]

```

#### A. Discovery & Quantification Methodologies

##### Shared Cold-Packing Stations (Resource Contention)
* **Log Analysis:** Filter for activities where `Activity = 'Packing'` and `Requires Cold Packing = TRUE`. 
* **State Reconstruction:** Build a step-function timeline of active Cold-Packing processes. For any timestamp $t$, the number of active cold-pack stations $A_{cold}(t)$ is:
  $$A_{cold}(t) = |\{ \text{case} \mid \text{Start}(t) \le t < \text{Complete}(t) \land \text{Requires Cold Packing} = \text{TRUE} \}|$$
* **Impact Isolation:** When $A_{cold}(t) = 5$, the resource pool is saturated. For any order $i$ arriving during this saturation window, the queue time $T_{queue}$ is calculated as:
  $$T_{queue} = \text{Start}_{i}(\text{Packing}) - \text{Complete}_{i-1}(\text{Item Picking})$$
  Any $T_{queue} > 0$ during saturation is flagged as a resource contention delay.

##### Shipping Batches (Batching Delay)
* **Log Analysis:** Identify instances where `Activity = 'Shipping Label Gen.'` and extract the batch ID from the Resource field (e.g., `System (Batch B1)`). Group cases by `Destination Region` and `Batch ID`.
* **State Reconstruction:** For each batch $B$, identify the boundary timestamps:
  * $t_{first\_ready}$: The timestamp when the first order in batch $B$ completed its `Quality Check`.
  * $t_{last\_ready}$: The timestamp when the last order in batch $B$ completed its `Quality Check`.
  * $t_{batch\_process}$: The start timestamp of `Shipping Label Gen.` for the batch.
* **Impact Isolation:** The waiting-for-batch time ($T_{batch\_wait}$) for any order $i$ in batch $B$ is:
  $$T_{batch\_wait, i} = t_{batch\_process} - \text{Complete}_{i}(\text{Quality Check})$$
  This captures the systematic delay introduced by waiting for other orders in the same geographic region to catch up.

```
Order 1 (Ready)  [Quality Check Complete] 
                                                                         
Order 2 (Ready)  [Quality Check Complete]  [Batch Shipping Label Gen]
                                                                         
Order 3 (Ready) 
                     T_batch_wait 
```

##### Priority Order Handling (Preemption)
* **Log Analysis:** Identify "suspect" Standard orders where `Order Type = 'Standard'` and `Activity = 'Packing'` (or picking/QC) exhibits unusually long durations or is interrupted.
* **State Reconstruction:** Look for overlapping execution windows on the same resource ID. If `Resource = 'Station S7'` is processing standard order $A$, and the log records a `START` event for Express order $B$ on `Station S7` *before* order $A$ registers a `COMPLETE` event, preemption or suspension has occurred.
* **Impact Isolation:** Calculate the cycle-time inflation of standard orders when interrupted.
  $$T_{preempt\_overhead} = \text{Duration}_{\text{interrupted}} - \text{Median Duration}_{\text{uninterrupted}}$$

##### Regulatory Compliance (Hazardous Material Limits)
* **Log Analysis:** Track cases where `Hazardous Material = TRUE`.
* **State Reconstruction:** Calculate the running concurrency of hazardous materials in the Packing and Quality Check steps:
  $$C_{haz}(t) = |\{ \text{case} \mid t \in [\text{Start}_{case}(\text{Packing}), \text{Complete}_{case}(\text{Quality Check})] \land \text{Hazardous} = \text{TRUE} \}|$$
* **Impact Isolation:** Identify throttling periods when $C_{haz}(t) = 10$. If a hazardous order completes `Item Picking` when $C_{haz}(t) = 10$, measure the delay before its `Packing` step begins. This delay is directly attributed to regulatory throttling:
  $$T_{throttle} = \text{Start}(\text{Packing}) - \text{Complete}(\text{Item Picking})$$

| Constraint Type | Key Metric | Formula / Definition | Target Threshold |
| :--- | :--- | :--- | :--- |
| **Shared Cold-Packing** | Cold-Packing Queue Delay ($T_{wait\_cold}$) | Time from completion of Picking to start of Packing for cold items. | $< 10$ mins |
| **Shipping Batches** | Batch Accumulation Overhead ($T_{batch\_wait}$) | $\sum (t_{batch\_process} - t_{order\_ready}) / N$ | $< 30$ mins |
| **Priority Handling** | Standard Order Preemption Delay ($T_{preempt}$) | Total idle time forced on standard cases due to express insertion. | $< 15$ mins per event |
| **Regulatory Limits** | Hazmat Throttling Delay ($T_{throttle}$) | Wait time at the boundary of Packing when Hazmat concurrency $= 10$. | $< 5$ mins |

#### C. Differentiating Within-Instance vs. Between-Instance Waiting Time

To isolate whether a delay is caused by an individual order's characteristics (within-instance) or system-wide constraints (between-instance), we apply a **Resource State Reconstruction** algorithm:

```
               Is the order ready for the next activity?
                                  
                                  
               Is the designated resource free?
               
                YES                                  NO
                       
      Within-Instance                     Between-Instance 
       Execution Time                      Queue Delay     
                       
```

1. **Calculate Handoff Time ($T_{handoff}$):** 
   $$T_{handoff} = \text{Start}_{i}(\text{Activity } K) - \text{Complete}_{i}(\text{Activity } K-1)$$
2. **Determine Resource Availability:** At $\text{Complete}_{i}(\text{Activity } K-1)$, check if the required resource (e.g., Station, Picker, or QC staff) was idle or busy with another case.
   * If the resource was **idle**, but the activity did not start, the delay is **Within-Instance** (e.g., transport delay, worker fatigue, or missing paperwork for that specific order).
   * If the resource was **busy** processing another case, or if a global constraint prevented the start (e.g., Hazmat limit reached), the delay is **Between-Instance** (system congestion, batching, or priority interruption).

### 2. Analyzing Constraint Interactions

Instance-spanning constraints do not operate in isolation. They form a complex, feedback-driven system where a bottleneck in one area creates cascading delays across the entire warehouse.

```
               
                                                              (Cascades queue)
              
  Express Cold-Pack Order      Standard Cold-Pack Delayed   
              
                (Preempts resource)                          (Delays batch completion)
                                                            
              
   Cold Station Saturated                    Whole Shipping Batch Stalled 
              
```

#### A. Key Interaction Scenarios

##### Interaction 1: Priority Express Orders vs. Cold-Packing Capacity
Express orders needing cold-packing will preempt standard cold-packing orders. 
* **The Interaction:** Because cold-packing capacity is highly constrained (only 5 stations), a cluster of Express Cold orders can monopolize these stations. Standard cold orders are pushed back in the queue. 
* **The Cascade:** Because cold items are perishable, extended delays in standard queues can violate product safety limits, requiring those orders to be scrapped and re-picked. This wastes picking capacity and inflates overall system cycle times.

##### Interaction 2: Shipping Batching vs. Hazardous Material Limits
* **The Interaction:** Suppose a shipping batch requires 20 orders to be completed before a batch is closed and shipped to the "South" region, and 12 of these orders contain hazardous materials.
* **The Cascade:** Because safety regulations allow no more than 10 hazmat orders to undergo Packing or Quality Check simultaneously across the entire facility, the 12 hazmat orders cannot be processed together. They must be throttled and run in two separate blocks. 
* **The Ripple Effect:** The 8 non-hazardous orders in that same batch must sit in the shipping buffer, waiting for the remaining hazmat orders to clear the throttled packing and QC steps. This unnecessarily consumes physical floor space and delays delivery times for non-hazardous goods.

##### Interaction 3: Express Preemption vs. Batching Cohesion
* **The Interaction:** When an Express order preempts a Standard order at a Packing station, the Standard order's processing is paused.
* **The Cascade:** If that paused Standard order is the final "missing piece" of an regional shipping batch, the entire batch is held up at the shipping dock. 
* **The Ripple Effect:** By prioritizing a single Express order, the system inadvertently delays a batch of 20 Standard orders, reducing overall shipping throughput and increasing average transit times.

---

#### B. Systemic Impact of Interactions

Failing to understand these interactions leads to **sub-optimization**. For example, adding more pickers to resolve an "Item Picking bottleneck" will only push more orders toward the Packing step, worsening the Cold-Packing queue and exceeding Hazmat limits more frequently. 

Analyses must evaluate the process as an interconnected network, where changing the control parameters of one constraint changes the arrival patterns and queue behaviors of all downline activities.

### 3. Constraint-Aware Optimization Strategies

To resolve these systemic bottlenecks, we propose three target, data-driven optimization strategies.

```
      RESERVATION            DYNAMIC BATCHING           HAZMAT QUEUE

 • Prevents preemption    • Releases early based   • Pulls orders based on 
 • Reduces scrap rates      on SLA thresholds        batch completion priority

```

---

#### Strategy 1: Dynamic Capacity Reservation & Threshold-Based Preemption for Cold-Packing

##### Constraints Addressed
Shared Cold-Packing Stations (1) and Priority Order Handling (3).

##### Specific Changes Proposed
* Stop the practice of hard preemption, which pauses standard orders mid-pack (wasting packing materials and setup time).
* Implement a **hybrid reservation rule**: Of the 5 cold-packing stations, 4 are designated as "General Access" (Standard or Express), and 1 is "Express Reserved." 
* Introduce a **soft preemption threshold**: If an Express order arrives and all 5 stations are busy, preemption is only allowed if a Standard order on a General Access station is in its first $15\%$ of processing time. Otherwise, the Express order must wait for the next station to finish.

##### Data/Analysis Leverage
Analyze historical event log data to determine the arrival patterns of Express Cold orders. If the historical probability of an Express Cold arrival within any 30-minute window during peak season is $< 40\%$, keep the 5th station open for Standard orders, but flag it to clear out when an Express order is detected at the picking stage.

##### Expected Outcomes
* Eliminates wasted setup time and packaging spoilage from interrupted standard pack runs.
* Reduces standard cold-packing cycle times by smoothing out resource access.
* Keeps Express Cold-Pack processing times predictable.

---

#### Strategy 2: Predictive Dynamic Shipping Batching

##### Constraints Addressed
Batching for Shipping (2) and Priority Order Handling (3).

##### Specific Changes Proposed
* Replace static batching rules (which wait for a fixed batch size of $N$ orders or a fixed cutoff time) with a **dynamic batch release trigger**.
* Integrate real-time queue visibility: If a batch is waiting for 2 outstanding orders, and those orders are flagged as "Express," the batching window shrinks to trigger immediate release. 
* If the system detects a high volume of orders for a specific region, it automatically splits large batches into smaller, sub-batches to prevent long queue wait times.

##### Data/Analysis Leverage
Using historical logs, calculate the **Batch Accumulation Decay Function** ($f(t_{wait})$), which models how long orders wait relative to the time of day and regional demand. 

```
Batch Release Decision:
Is Batch Size  N?  YES  RELEASE BATCH
     
      NO
Is Max Wait Time  T_sla?  YES  RELEASE BATCH
     
      NO
Predictive Check:  High probability of no matching orders  RELEASE BATCH
                            in next 15 mins?
```

The system uses this function to calculate whether waiting for another order to join the batch will violate delivery SLAs for the orders already in the queue.

##### Expected Outcomes
* Reduces average shipping buffer wait times ($T_{batch\_wait}$) during off-peak hours by up to $35\%$.
* Improves space utilization at the shipping dock by maintaining steady inventory flow.

---

#### Strategy 3: Token-Based Virtual Queueing for Hazardous Materials

##### Constraints Addressed
Regulatory Compliance (4) and Shipping Batching (2).

##### Specific Changes Proposed
* Implement a **virtual Kanban token system** for Hazardous Material orders. The warehouse system issues exactly 10 "Hazmat Tokens" for the entire facility.
* An order cannot start its `Packing` step unless it acquires a Hazmat Token. The token is released only when the order completes its `Quality Check` and exits the restricted zone.
* Allocate these tokens using a **priority scoring rule**:
  $$\text{Priority Score} = w_1 \cdot (\text{Express Flag}) + w_2 \cdot (\text{Time in Queue}) + w_3 \cdot (\text{Batch Affinity})$$
  *Batch Affinity* prioritizes hazmat orders that belong to shipping batches that are nearly complete, pulling them through the regulated zone first.

##### Data/Analysis Leverage
Use event logs to calculate the average processing times for Hazmat Packing ($T_{haz\_pack}$) and Hazmat QC ($T_{haz\_qc}$). This data is used to dynamically adjust the token allocation algorithm, preventing the system from releasing too many tokens to Packing when the Quality Check station is backed up.

##### Expected Outcomes
* Guarantees 100% compliance with safety regulations (never exceeding 10 concurrent cases).
* Minimizes total waiting times by prioritizing hazmat orders that are holding up larger shipping batches.

### 4. Simulation and Validation

Before deploying these strategies in the live warehouse, we must validate them using **Discrete Event Simulation (DES)**. Traditional simulation models treat processes as sequential flows of independent transactions, which fails to capture instance-spanning constraints. We will construct a **Constraint-Aware Discrete Event Simulation** using tools like SimPy or Arena, calibrated by the event log data.

```
                           SIMULATION CALIBRATION ENGINE
               
       HISTORICAL EVENT LOG                           SIMULATION ENVIRONMENT     
               
 • Timestamp distributions                       • Shared Resource Pools         
 • Service time parameters        • Preemption Rules Engine       
 • Case arrival distributions                    • Global Concurrency Limit (10) 
                • Batch-Join Nodes              
                                                  
                                                                   
                                                                   
                                                  
                                                        VALIDATION COMPARISON      
                                                       Historical vs. Simulated    
                                                  
```

---

#### A. Key Modeling Elements

##### Shared Resource Pools with Stateful Attributes
The simulation must model the 5 Cold-Packing stations as a single, shared resource pool. Each station must maintain its current state (Idle, Busy-Standard, Busy-Express) and support preemptive-resume logic.

##### Batch-Join Nodes
The `Shipping Label Gen.` step must be modeled as a join node with a buffer. This node holds transactions grouped by `Destination Region` attributes and releases them only when the dynamic batching rules are satisfied.

##### Global Concurrency Monitors
Implement a global state variable `Active_Hazmat_Count` in the simulation engine. Before a simulated case with `Hazardous = TRUE` enters the `Packing` or `Quality Check` activities, it must pass a conditional gate:
$$\text{If } \text{Active\_Hazmat\_Count} < 10 \text{ then Proceed, else Block/Queue}$$

##### Inter-arrival and Process Time Distributions
Fit probability distributions to the historical event log data for each activity (e.g., log-normal distributions for `Item Picking` durations, Weibull distributions for packing times), split by attributes like order type and size.

---

#### B. Calibration and Validation Process

1. **Historical Replay (Trace-Driven Simulation):**
   Configure the simulation model using the exact case arrival times and attributes from the first half of the historical event log.
2. **Behavioral Alignment:**
   Run the simulation and compare its outputs against the second half of the actual event log. Verify that the simulated bottlenecks, resource utilization rates, and queue build-ups match the patterns seen in the real-world data.
3. **Scenario Stress-Testing:**
   Run predictive scenarios to test the proposed strategies:
   * **Scenario A:** Baseline (current state).
   * **Scenario B:** Baseline + Dynamic Cold-Pack Reservation.
   * **Scenario C:** Baseline + Predictive Batching.
   * **Scenario D:** Baseline + Token-Based Hazmat Queueing.
   * **Scenario E:** Combined implementation of all three strategies.
4. **KPI Impact Evaluation:**
   Measure the impact of each scenario on key performance indicators (SLA compliance rates, average cycle times, resource utilization, and peak queue sizes) to ensure the changes improve operations without causing unintended bottlenecks elsewhere.

#### A. Key Performance Indicators (KPIs)

##### 1. Average Cold-Packing Queue Time ($T_{wait\_cold}$)
* **Metric Definition:** The time from picking completion to the start of packing for cold orders.
* **Why it matters:** Confirms that the capacity reservation strategy is successfully reducing queue times.

##### 2. Standard Order Cycle Time Inflation Factor ($I_{std\_order}$)
* **Metric Definition:**
  $$I_{std\_order} = \frac{\text{Mean Cycle Time of Preempted Standard Orders}}{\text{Mean Cycle Time of Non-Preempted Standard Orders}}$$
* **Why it matters:** Measures whether the soft-preemption threshold is successfully protecting standard orders from excessive delays.

##### 3. Batch Accumulation Efficiency ($\eta_{batch}$)
* **Metric Definition:**
  $$\eta_{batch} = \frac{\text{Time spent processing shipping labels}}{\text{Total time spent in batching buffer}}$$
* **Why it matters:** A higher ratio means orders are spending less idle time waiting for batches to form.

##### 4. Hazmat Concurrency Violation Count ($V_{haz}$)
* **Metric Definition:** The number of times the running active hazmat count exceeds the safety limit:
  $$\text{Count where } C_{haz}(t) > 10$$
* **Why it matters:** Verifies compliance with safety regulations. The target is exactly zero violations.

##### 5. Hazmat Queue Starvation Rate ($S_{haz}$)
* **Metric Definition:** The percentage of time where $C_{haz}(t) < 7$ while there are hazardous orders waiting in the picking queue.
* **Why it matters:** Ensures the virtual token system is pulling hazmat orders through efficiently, keeping the packing line steadily supplied without exceeding safety limits.
