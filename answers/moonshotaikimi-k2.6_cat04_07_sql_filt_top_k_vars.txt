Here is a self-contained DuckDB query that fulfills all four requirements.  
It uses `STRING_AGG` to materialise the ordered activity sequence per case, groups by that sequence to rank variants, and finally joins back to the original log to retrieve every event belonging to a topK case.

```sql
-- ============================================================
-- 1. Parameter: define K (change this value as needed)
-- ============================================================
WITH params AS (
    SELECT 5 AS k
),

-- ============================================================
-- 2. Build the ordered activity sequence for each case
-- ============================================================
case_variants AS (
    SELECT
        case_id,
        STRING_AGG(activity, ' -> ' ORDER BY timestamp, activity) AS variant
    FROM event_log
    GROUP BY case_id
),

-- ============================================================
-- 3. Count cases per variant and keep the top K
-- ============================================================
top_k_variants AS (
    SELECT
        variant,
        COUNT(*) AS case_count
    FROM case_variants
    GROUP BY variant
    ORDER BY case_count DESC
    LIMIT (SELECT k FROM params)
)

-- ============================================================
-- 4. Return every original event whose case belongs to a top-K variant
-- ============================================================
SELECT
    e.*
FROM event_log e
INNER JOIN case_variants cv
    ON e.case_id = cv.case_id
WHERE cv.variant IN (SELECT variant FROM top_k_variants)
ORDER BY
    e.case_id,
    e.timestamp;
```

### How it works

| Step | CTE | What it does |
|------|-----|--------------|
| **1** | `params` | Defines the cutoff `K` in one place so it is easy to adjust. |
| **2** | `case_variants` | Aggregates all activities for a `case_id` into a single string, preserving chronology with `ORDER BY timestamp` (and `activity` as a tiebreaker). This string is the **process variant** fingerprint. |
| **3** | `top_k_variants` | Groups the `case_variants` by their fingerprint, counts how many cases instantiate each variant, and uses `ORDER BY case_count DESC LIMIT K` to retain only the most frequent ones. |
| **4** | Final `SELECT` | Joins the original `event_log` back to `case_variants` on `case_id`, keeps only those rows whose fingerprint appears in `top_k_variants`, and returns the untouched original columns. |

### Notes

* **Tie handling** – The query uses a simple `LIMIT K`. If you want to include all variants that tie for the Kth rank, replace the `LIMIT` with a window function:
  ```sql
  SELECT variant
  FROM (
      SELECT variant,
             RANK() OVER (ORDER BY case_count DESC) AS rnk
      FROM variant_counts
  ) ranked
  WHERE rnk <= (SELECT k FROM params)
  ```
* **Separator choice** – `' -> '` is used for readability; any delimiter (or even `LIST(activity ORDER BY timestamp)` if you prefer a DuckDB list type) works as long as it uniquely identifies the sequence.
* **Ordering** – The final `ORDER BY e.case_id, e.timestamp` is for presentation only and does not affect the logic.
