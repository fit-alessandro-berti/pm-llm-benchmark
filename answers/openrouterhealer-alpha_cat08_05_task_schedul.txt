# Precision Parts Inc. — Data-Driven Scheduling Optimization Framework

## A Process Mining Approach to Manufacturing Excellence

---

## 1. Analyzing Historical Scheduling Performance and Dynamics

### Reconstructing the Actual Process from Event Logs

Process mining begins with the fundamental assumption that event logs are digital footprints of reality. The MES logs from Precision Parts Inc. contain precisely the artifacts needed: case identifiers (Job IDs), activities (tasks), timestamps, resources (machines, operators), and contextual attributes (priority, due dates, setup requirements). My approach transforms these raw records into a living model of shop floor behavior.

**Log Preparation and Enhancement**

Before analysis, the event log requires careful preprocessing:

- **Case identification**: Each `JOB-XXXX` represents a case instance. However, the *true* case for scheduling purposes is not the job alone—it is the `Job × Resource` pair. A job passing through Cutting and then Milling represents two distinct *task instances*, each with its own lifecycle. I would reconstruct a flat event log where each row represents a `(Case_ID, Activity, Resource, Timestamp)` tuple, enriched with job-level attributes (priority, due date) and resource-level attributes (machine type, capability set).

- **Event correlation**: `Setup Start`, `Task Start`, `Task End`, and `Setup End` events must be correlated into logical task instances. Using temporal proximity and resource/case matching, I would construct derived events representing complete task executions:

```
Task_Instance = {
    job_id, 
    activity, 
    resource, 
    queue_entry_time,
    setup_start_time,
    setup_end_time,       // or setup_start + planned_setup
    task_start_time,
    task_end_time,        // actual completion
    planned_duration,
    actual_duration,
    planned_setup_time,
    actual_setup_time,
    order_priority,
    due_date
}
```

- **Handling censored data**: Some events may be missing (e.g., a job starts processing before the log begins, or a breakdown event lacks a corresponding "Breakdown End"). I would flag these and apply conservative assumptions (e.g., for unfinished tasks, use current timestamp as a lower bound on completion time).

### Process Mining Techniques and Metrics

**A. Reconstructing Process Models and Flow Analysis**

Using a process mining tool (Celonis, ProM, or PM4Py in Python), I would:

1. **Generate a Directly-Follows Graph (DFG)** at both the job level and the work-center level. The job-level DFG reveals the *actual* routing variations across jobs—critical because "high-mix" implies significant routing divergence. The work-center-level DFG reveals the *sequence* of jobs processed on each machine.

2. **Apply the Heuristic Miner or Inductive Miner** to discover the process model, which handles the noise and concurrency inherent in job shop logs better than the alpha miner. This model exposes:
   - Parallel activities (jobs being processed simultaneously on different machines)
   - Choice points (where jobs diverge to different machine alternatives)
   - Loops (rework, re-entrant flows)

3. **Perform Variant Analysis**: Group jobs by their process variant (unique sequence of activities/machines). In a high-mix shop, I expect dozens to hundreds of variants. The key insight is not in the rarest variants but in the *most frequent* ones and their deviation patterns.

**B. Specific Metrics and How to Compute Them**

| Metric | Process Mining Technique | Computation |
|--------|-------------------------|-------------|
| **Job Flow Time** | Case-level timestamp analysis | `flow_time = last_task_end_time - first_task_start_time` for each job. Compute distributions (mean, median, P95) and segment by priority, routing variant, and entry period. |
| **Job Lead Time** | Case-level timestamp analysis | `lead_time = last_task_end_time - job_release_timestamp`. Compare against quoted lead times to assess estimation accuracy. |
| **Makespan** | Resource-level timeline reconstruction | For the entire planning period: `makespan = max(all_task_end_times) - min(all_task_start_times)`. For individual machines: calculate the active window. |
| **Queue/Waiting Time** | Task-instance timestamp gaps | `queue_time = task_start_time - queue_entry_time`. Aggregate by resource, time period, and preceding activity. This is *the* critical metric for identifying where WIP accumulates. |
| **Resource Utilization** | Resource-level activity timeline | For each resource over a time window: `utilization = total_task_processing_time / total_available_time`. Decompose into productive processing time, setup time, and idle (gap) time. |
| **Sequence-Dependent Setup Time** | Resource-level job sequence + setup events | For each resource, construct the sequence of jobs. For each pair `(Job_A  Job_B)`, extract the setup duration when B follows A. Build a **setup time matrix** `S[i][j]` for each machine. |
| **Schedule Adherence / Tardiness** | Case-level due date comparison | `tardiness = max(0, job_completion_time - due_date)`. Compute: mean tardiness, percent tardy, maximum tardiness, and total weighted tardiness (weighted by priority). |
| **Disruption Impact** | Event correlation around disruption events | For each breakdown/priority change event, compute the "disruption radius": time delta between event and the completion of all affected tasks (directly and indirectly). Quantify schedule deviation (actual vs. planned completion times) for jobs processed before, during, and after the disruption. |

**C. Detailed Setup Time Analysis**

This deserves special attention given the sequence-dependent nature. My approach:

1. **Extract job sequences per machine**: From the log, reconstruct the chronological order of jobs on each resource.
2. **Compute pairwise setup durations**: When Job B begins setup on Machine M immediately after Job A completes, the setup time is `setup_start(B) - task_end(A)`. This captures the *true* transition cost.
3. **Build setup time matrices**: For each machine, create a matrix `S_M[A][B]` = average setup time when B follows A. Where data is sparse, apply hierarchical clustering on job properties (material type, part family, tooling requirements) to generalize.
4. **Identify high-cost transitions**: Flag the top 10% most expensive transitions and correlate with job characteristics. This directly informs sequencing strategy.

**D. Disruption Impact Analysis**

1. **Identify disruption events**: Filter for `Breakdown Start`, `Breakdown End`, `Priority Change`, and any `Notes` field containing disruption keywords.
2. **Construct timeline segments**: Partition the event log into "pre-disruption," "disruption window," and "recovery" periods.
3. **Compare KPIs across segments**: Compute flow times, queue times, and tardiness for jobs active during each segment. The delta quantifies disruption severity.
4. **Network propagation analysis**: Use the process model to trace how a breakdown at Machine X propagates to downstream machines and subsequent jobs. This reveals the *systemic* cost of disruptions beyond the directly affected machine.

## 2. Diagnosing Scheduling Pathologies

With the metrics established, I now diagnose the *systemic* inefficiencies. Process mining provides the evidence.

### Pathology 1: Bottleneck Resources and Their Cascading Impact

**Diagnostic Approach:**

- **Bottleneck identification via utilization and queue analysis**: A true bottleneck shows simultaneously high utilization (>85-90%) *and* growing queue lengths over time. I would plot queue length at each work center as a function of time. Resources where queue length trends upward are capacity-constrained.

- **Waiting time decomposition using the process model**: For each job, decompose total flow time into processing time and waiting time. Then attribute waiting time to specific resources. A resource that accounts for disproportionately large waiting time across many jobs is either a bottleneck or a poorly scheduled resource.

- **Throughput analysis at each work center**: Compare the *output rate* (jobs completed per day) of each machine. Machines with lower throughput despite high utilization are genuine bottlenecks. Machines with high utilization but low throughput indicate excessive setup times or poor sequencing.

**Expected Finding**: In a typical job shop with specialized machines, I expect 2-3 bottleneck resources (likely CNC Milling or Heat Treatment) that dictate overall system throughput. The process model will show that jobs queue disproportionately before these resources, and that disruption at these points causes the longest recovery times.

### Pathology 2: Poor Task Prioritization

**Diagnostic Approach:**

- **Priority-adherence analysis**: For each job, compare its `Order Priority` attribute against its actual *scheduling position* in machine queues. Do High-priority jobs actually get processed first? I would compute:

```
Priority_Position_Ratio = (actual queue position) / (expected queue position if strictly priority-ordered)
```

A ratio >> 1 for high-priority jobs indicates priority rules are being overridden or ignored.

- **Variant analysis: On-time vs. Late jobs**: Using the process mining platform, I would perform a *comparative variant analysis*:
  - Split cases into "on-time" (tardiness = 0) and "late" (tardiness > 0) cohorts.
  - Compare the two cohorts on: job priority distribution, routing variants, resource contention levels at the time of job entry, and the specific machines where waiting time was excessive.
  - If late jobs are not systematically lower-priority, the dispatching rules are failing to correlate priority with outcomes.

- **Due-date sensitivity analysis**: Plot tardiness against the ratio of processing time remaining to time until due date (the "urgency ratio") at the point of each scheduling decision. If jobs with high urgency ratios are not being prioritized, the current rules are blind to due-date pressure.

**Expected Finding**: The "mix of FCFS and EDD" likely means that at some work centers, jobs are processed in arrival order regardless of urgency, while at others, EDD creates a rigid sequence that ignores setup time consequences. Neither approach dynamically adapts to the changing shop state.

### Pathology 3: Excessive Setup Times from Poor Sequencing

**Diagnostic Approach:**

- **Setup time distribution analysis**: For each machine, compute the distribution of setup times. If the distribution is wide (high variance) and the mean is significantly higher than the minimum observed setup time, suboptimal sequencing is adding unnecessary cost.

- **Sequencing pattern analysis**: For bottleneck machines, analyze the actual job sequence against the setup time matrix. Compute the *total setup cost of the actual sequence* versus the *optimal sequence* (solved as a Traveling Salesman Problem variant on the setup time matrix). The gap quantifies the cost of poor sequencing.

- **Part-family clustering**: If jobs that share similar properties (material, geometry, tooling) are frequently separated in the schedule, this indicates a failure to batch similar work. Process mining can reveal this by clustering jobs on machine M by their setup time when following each other—similar jobs will have low mutual setup times.

**Expected Finding**: I expect to find that bottleneck machines process jobs in an order that maximizes setup time (e.g., alternating between dissimilar part families) because the dispatching rules are agnostic to setup consequences.

### Pathology 4: Downstream Starvation and WIP Accumulation

**Diagnostic Approach:**

- **Queue length correlation across machines**: Compute the correlation between queue lengths at upstream and downstream machines. In a healthy system, downstream machines should have a steady supply. If downstream machines experience frequent idle periods (queue length = 0) while upstream queues are long, the bottleneck is starving the downstream.

- **Work-in-process tracking via the event log**: At any timestamp `t`, the WIP is the count of jobs that have started but not completed. I would compute WIP over time, segmented by location (which machines/stations hold the WIP). Growing WIP before a specific machine confirms it as a congestion point.

- **Flow time variability analysis**: Compute the coefficient of variation (CV) of flow times. A CV > 1 indicates high variability, often caused by unpredictable waiting times at bottleneck resources. High flow time variability makes lead time estimation unreliable.

**Expected Finding**: WIP likely accumulates before bottleneck machines and in buffer zones between work centers, while downstream flexible machines experience starvation. This creates a paradox of high WIP (costing working capital) alongside low throughput.

### Pathology 5: Schedule Fragility to Disruptions

**Diagnostic Approach:**

- **Disruption frequency and duration analysis**: From the logs, compute the frequency, duration, and type of disruptions per machine and per time period.

- **Recovery time analysis**: For each disruption event, compute how long it takes for the affected machine and downstream machines to return to pre-disruption queue lengths and throughput rates. Long recovery times indicate insufficient buffer capacity or poor rescheduling capability.

- **Comparison of disruption-affected vs. unaffected jobs**: For jobs that were in the system during a disruption, compare their actual flow time to the average flow time of similar jobs (same routing, similar complexity) processed during disruption-free periods. The delta quantifies the "disruption penalty."

**Expected Finding**: The current reactive dispatching rules likely have no mechanism for absorbing disruptions—no buffer time, no pre-emptive rescheduling, no dynamic re-prioritization when urgent jobs arrive or machines break down.

## 3. Root Cause Analysis of Scheduling Ineffectiveness

### Root Cause 1: Static Dispatching Rules in a Dynamic Environment

**The fundamental mismatch**: The shop floor state (queue lengths, machine availability, job urgency) changes continuously. Static rules like FCFS and EDD compute priorities once at job arrival and never revisit those decisions. A job that was "low priority" at arrival may become critically urgent after a breakdown delays downstream capacity. A job that seemed efficient to schedule may create a massive setup cost for the next job.

**Process mining evidence**: By overlaying dispatching decisions on the dynamic shop state, I would demonstrate that:
- High-priority jobs arriving during high-utilization periods are delayed as much as low-priority jobs arriving during low-utilization periods—because the rules don't account for current load.
- Jobs dispatched during bottleneck congestion create worse outcomes than identical jobs dispatched during low congestion, but the rules treat them identically.

### Root Cause 2: Lack of Real-Time Visibility and Predictive Capability

**Current state**: The MES logs events but doesn't feed insights back into scheduling decisions in real-time. Dispatchers likely make decisions based on local observation (what they can see at their work center) rather than global shop state.

**Process mining evidence**: By reconstructing what information was *available* at each scheduling decision point (from the log timestamps), I can demonstrate that critical information existed in the system but was not used:
- A machine's breakdown history (available from past logs) could predict future breakdown probability, but the current schedule doesn't build in buffer time.
- A job's actual processing time on a previous operation (logged as completed) could update the remaining processing time estimate, but the current plan uses static estimates.

### Root Cause 3: Inaccurate Duration and Setup Time Estimates

**Current state**: Task durations are estimated statically. Setup times may be estimated or ignored. Neither is updated based on actual performance.

**Process mining evidence**: By comparing `Task Duration (Planned)` against `Task Duration (Actual)` in the log, I can quantify estimation error. My hypothesis:
- Planned durations are based on engineering estimates or historical averages, but actual durations show significant variance (possibly CV > 0.3).
- Setup times show even higher variance, often exceeding planned times, because the plan doesn't account for the specific preceding job (sequence dependency).
- The combination of underestimation creates compounding delays: each task runs slightly longer than planned, and the schedule has no margin to absorb this.

### Root Cause 4: Ineffective Handling of Sequence-Dependent Setups

**Current state**: The dispatching rules treat each job's processing time as independent of what came before. Setup time is either added as a fixed constant per operation or ignored entirely in the sequencing decision.

**Process mining evidence**: The setup time matrix analysis (from Section 2, Pathology 3) will show that:
- The cheapest and most expensive transitions on each machine can differ by 5-10x.
- The current sequencing often results in near-worst-case setup transitions because the dispatching rules are agnostic to setup consequences.
- Quantitatively: `Total actual setup time / Total optimal setup time` likely exceeds 1.5-2.0 for bottleneck machines.

### Root Cause 5: Poor Coordination Between Work Centers

**Current state**: Each work center schedules locally without considering downstream consequences. A work center may process a batch of jobs quickly, creating a surge of WIP that overwhelms the next station.

**Process mining evidence**: By analyzing the *inter-work-center arrival patterns*, I would show that:
- Arrivals at downstream machines are "bursty" (large batches arrive together followed by long gaps) rather than smooth.
- This burstiness creates alternating periods of overload and starvation at downstream machines.
- The coordination failure is structural: there is no mechanism to level the release of work across the shop.

### Root Cause 6: Inadequate Disruption Response

**Current state**: When a breakdown occurs or an urgent job arrives, the response is ad hoc—likely manual re-prioritization by a supervisor with incomplete information.

**Process mining evidence**: The disruption impact analysis (from Section 2, Pathology 5) will show that:
- Recovery times are long and highly variable, depending on the human response rather than a systematic protocol.
- Pre-disruption jobs suffer disproportionately because they are "older" in the queue but may not be re-prioritized.
- Urgent jobs may be inserted into the schedule in a way that maximizes disruption to existing jobs, because there's no optimization of insertion point.

### Differentiating Root Causes: Scheduling Logic vs. Capacity vs. Variability

Process mining helps distinguish these through the following analysis:

| Root Cause Type | Diagnostic Evidence from Process Mining |
|----------------|----------------------------------------|
| **Poor scheduling logic** | Underutilization exists at the same time as long queues. Jobs are processed in orders that maximize setup time. Priority jobs are not processed first. |
| **Capacity limitations** | Bottleneck machines are at >90% utilization, queues grow monotonically, and *no* sequencing improvement can eliminate the queue—it's a pure throughput problem. |
| **Process variability** | Same jobs on same machines show high variance in processing times (even controlling for operator). Machine breakdowns follow no predictable pattern. |

In reality, all three contribute, but process mining quantifies the relative contribution of each, enabling targeted intervention.

## 4. Developing Advanced Data-Driven Scheduling Strategies

### Strategy 1: Dynamic Multi-Criteria Dispatching with Contextual Priority Scoring

**Core Logic:**

Replace static FCFS/EDD rules with a dynamic priority score computed *at each scheduling decision point*, incorporating multiple factors that reflect the current shop state and job characteristics. The score is recalculated continuously, not just at job arrival.

**Priority Score Formula:**

```
P(job, machine, time) = w·URGENCY(job, time) + w·DOWNSTREAM_LOAD(job, time) + w·SETUP_COST(job, machine, time) + w·PRIORITY_TIER(job) + w·FLOW_TIME_RISK(job, time)
```

Where:

- **URGENCY**: `max(0, 1 - (time_until_due - remaining_processing_time) / time_until_due)`. This captures how "tight" the schedule is for this job. Negative values (plenty of slack) are clamped to zero.

- **DOWNSTREAM_LOAD**: From the process model, the expected waiting time at the *next* machine in the job's routing. A job heading toward a lightly-loaded machine should be dispatched sooner to maintain flow. This is computed from real-time queue data.

- **SETUP_COST**: From the historical setup time matrix, the estimated setup duration if this job follows the job most recently completed on this machine. High setup costs *reduce* the priority (to encourage batching of similar jobs).

- **PRIORITY_TIER**: The customer-assigned order priority (High/Medium/Low), normalized.

- **FLOW_TIME_RISK**: `1 - (elapsed_flow_time / expected_flow_time)`. Jobs that have already consumed a large fraction of their expected flow time with much processing remaining are at high risk of tardiness.

**How Process Mining Informs This Strategy:**

- **Setup cost weights**: The pairwise setup time matrix (from historical logs) provides the `SETUP_COST` values. For sparse matrix entries, use clustering on job properties to impute.
- **Factor weights (w through w)**: These are *not* guessed. I would use historical log data to train a regression model predicting tardiness from these five factors (and potentially others) at each historical scheduling decision point. The regression coefficients become the weights. This is a *learned* dispatching rule, not a designed one.
- **DOWNSTREAM_LOAD estimation**: Process mining reveals the distribution of waiting times at each machine as a function of queue length and job mix. These distributions parameterize the downstream load estimator.

**How It Addresses Identified Pathologies:**

- **Poor prioritization** (Pathology 2): Dynamic urgency and flow-time risk directly respond to changing job criticality.
- **Excessive setup times** (Pathology 3): The setup cost factor encourages similar-job batching at bottleneck machines.
- **Downstream starvation** (Pathology 4): The downstream load factor promotes flow balancing across work centers.

**Expected Impact:**

| KPI | Expected Improvement |
|-----|---------------------|
| Mean tardiness | 30-45% reduction |
| High-priority job tardiness | 50-60% reduction |
| Average setup time per task | 15-25% reduction |
| Mean flow time | 10-20% reduction |

### Strategy 2: Predictive Scheduling with Stochastic Task Duration Models

**Core Logic:**

Move from deterministic scheduling (assuming planned durations are correct) to stochastic scheduling that explicitly models task duration uncertainty and proactively identifies future bottlenecks.

**Component 1: Task Duration Distribution Models**

From the MES logs, for each `(Activity, Resource, Job_Family)` combination, fit a probability distribution to actual task durations. Common patterns in manufacturing:

- **Log-Normal distribution**: Most task durations, with a right tail for delays.
- **Bimodal distributions**: Indicating two distinct operating modes (e.g., "normal" and "problem" states).
- **Operator-dependent distributions**: If the same task on the same machine shows different distributions for different operators, include operator skill as a factor.

Each model provides: expected duration, variance, and tail probabilities (e.g., "P(duration > 1.5 × planned) = 12%").

**Component 2: Forward-Looking Schedule Simulation**

Given a current shop state (jobs in queue at each machine, jobs in progress, machine availability), simulate the *forward schedule* using Monte Carlo sampling from the task duration distributions. Run N simulations (e.g., N=1000) and compute:

- **P(job completion before due date)** for each job.
- **Expected queue length** at each machine over the next horizon.
- **Identified risk periods**: Time windows where queue lengths are predicted to exceed capacity.

**Component 3: Predictive Interventions**

Based on the simulation results:

- **Early warning system**: If P(job_j tardy) exceeds a threshold (e.g., 20%), flag the job for expedited handling *before* it becomes critically late.
- **Proactive bottleneck management**: If the simulation predicts a queue buildup at Machine X starting at time T, pre-emptively increase the priority of jobs at upstream machines that feed into Machine X, or schedule maintenance/setup work on Machine X *before* the predicted congestion.
- **Buffer time allocation**: For jobs routed through high-variance tasks, add stochastic buffer time proportional to the predicted tail risk. This replaces the current practice of using fixed (and often inadequate) safety margins.

**How Process Mining Informs This Strategy:**

- **Distribution fitting**: Entirely data-driven, derived from historical `Task Duration (Actual)` values grouped by context.
- **Routing probabilities**: For jobs with alternative routings (multiple machines capable of the same task), the process model reveals the historical routing choice probabilities, informed by machine availability at decision time.
- **Disruption frequency models**: From historical breakdown data, fit distributions for time-between-failures and time-to-repair for each machine. Incorporate these into the Monte Carlo simulation to model disruption scenarios.

**How It Addresses Identified Pathologies:**

- **Unpredictable lead times**: Stochastic models directly quantify uncertainty, enabling accurate lead time quotes (e.g., "We are 95% confident this job will be done by April 28").
- **Schedule fragility** (Pathology 5): Predictive simulation catches emerging problems before they materialize.
- **Inaccurate duration estimates** (Root Cause 3): The models are built on actual performance data, not engineering estimates.

**Expected Impact:**

| KPI | Expected Improvement |
|-----|---------------------|
| Lead time estimation accuracy (within ±1 day) | 60%  85% of jobs |
| Mean flow time (with reduced variability) | 10-15% reduction |
| Disruption recovery time | 20-30% reduction |
| Planner confidence in schedule | Qualitative but significant |

### Strategy 3: Setup Time Optimization through Intelligent Batch Sequencing

**Core Logic:**

At bottleneck machines where setup time is a significant fraction of available capacity, explicitly optimize the job sequence to minimize total setup time, using techniques from combinatorial optimization informed by the historical setup time matrix.

**Algorithm:**

1. **Batch formation**: Group jobs waiting at a bottleneck machine into batches based on similarity. Define similarity as:
   - Same or compatible material type
   - Same or compatible tooling requirements  
   - Same part family (from part number taxonomy)
   - Low pairwise setup times (from the setup time matrix)

2. **Within-batch sequencing**: For each batch, solve a Traveling Salesman Problem (TSP) variant on the setup time matrix to find the minimum-setup sequence. Use nearest-neighbor heuristic for real-time feasibility, or branch-and-bound for near-optimal solutions when the batch size allows.

3. **Between-batch sequencing**: Sequence batches to minimize the transition setup time between the last job of batch *k* and the first job of batch *k+1*. This is itself a TSP on the "batch centroids" in setup-time space.

4. **Dynamic re-sequencing**: When a new job arrives or a disruption occurs, re-run the local sequencing optimization for the affected machine, considering the current queue.

**How Process Mining Informs This Strategy:**

- **Setup time matrix**: The historical data provides the pairwise setup times. For pairs with no historical data, use a predictive model: `S(AB) = f(material_A, material_B, tooling_A, tooling_B, ...)` trained on the available pairwise data.
- **Batch size sensitivity**: Process mining reveals the relationship between batch size and setup time savings. If the data shows diminishing returns beyond batch size 5, cap the batches accordingly to avoid excessive flow time for jobs waiting in large batches.
- **Interaction with priority**: Not all jobs can be batched regardless of priority. The sequencing optimization includes a constraint: *high-priority jobs must be processed within X hours of arrival*, limiting the maximum batch formation time.

**Integration with Strategy 1:**

The setup cost factor in the dynamic dispatching rule (Strategy 1) acts as a "pull" mechanism that naturally creates favorable sequences. Strategy 3 provides an explicit "push" mechanism at bottleneck machines where setup optimization has the highest impact. The two strategies work synergistically: the dispatching rule creates good incoming order, and the batch sequencing optimizes locally.

**How It Addresses Identified Pathologies:**

- **Excessive setup times** (Pathology 3): Directly targeted. The optimization explicitly minimizes the metric that current dispatching rules ignore.
- **Resource utilization** (general inefficiency): By reducing setup time, productive capacity increases. A 20% reduction in setup time at a bottleneck machine translates directly to 20% more available production time.

**Expected Impact:**

| KPI | Expected Improvement |
|-----|---------------------|
| Total setup time at bottleneck machines | 25-40% reduction |
| Throughput at bottleneck machines | 10-20% increase |
| Mean flow time (due to reduced waiting) | 15-25% reduction |
| WIP (due to faster throughput) | 15-20% reduction |

## 5. Simulation, Evaluation, and Continuous Improvement

### Discrete-Event Simulation for Strategy Validation

Before deploying any scheduling strategy on the live shop floor, rigorous simulation testing is essential. The simulation environment serves as a "digital twin" where strategies can be compared safely.

**Simulation Model Construction:**

Using a discrete-event simulation framework (SimPy in Python, AnyLogic, or Arena), I would build a model parameterized entirely by process-mined data:

| Model Component | Data Source from Process Mining |
|----------------|--------------------------------|
| **Machine pool** | Resource list with capabilities (from unique Resource IDs in the log) |
| **Job arrival process** | Inter-arrival time distribution (fitted to `Job Released` timestamps), routing probabilities per job type, processing time distributions per operation |
| **Task durations** | Context-specific distributions (Activity × Resource × Job_Family), as fitted in Strategy 2 |
| **Setup times** | Sequence-dependent setup time matrix per machine, with fallback predictive model for unseen transitions |
| **Machine availability** | Time-between-failures and time-to-repair distributions per machine (from breakdown events) |
| **Priority distribution** | Historical frequency of High/Medium/Low priority orders, with time-of-week patterns if present |
| **Operator assignment** | Operator-to-machine assignment patterns and operator-specific speed factors (from Operator ID in the log) |

**Simulation Scenarios:**

I would design the following test scenarios, representing the range of conditions Precision Parts Inc. faces:

| Scenario | Description | Purpose |
|----------|-------------|---------|
| **Baseline (Current Rules)** | FCFS/EDD dispatching as currently implemented | Establishes the benchmark |
| **Normal Load** | Job arrival rate at historical average | Tests strategy under typical conditions |
| **High Load** | Job arrival rate 20% above historical average | Tests strategy under stress |
| **Disruption-Heavy** | Machine breakdowns at 2× historical frequency, 2 urgent jobs per week | Tests resilience |
| **New Product Introduction** | 15% of jobs have no historical data (new part families) | Tests generalization to novel jobs |
| **Seasonal Peak** | Arrival rate varies by day-of-week matching observed patterns | Tests time-varying strategy adaptation |

**Evaluation Metrics:**

For each strategy × scenario combination, I would compute:

1. **Primary KPIs**: Mean tardiness, percent tardy, mean flow time, WIP (average and peak), bottleneck utilization, total setup time.
2. **Distribution metrics**: P90 and P95 of flow time and tardiness (critical for worst-case performance).
3. **Robustness metrics**: Variance of KPIs across simulation replications (lower is better = more predictable).
4. **Trade-off metrics**: Plot Pareto fronts (e.g., tardiness vs. WIP, or setup time vs. flow time) to visualize strategy trade-offs.

**Strategy Comparison Framework:**

I would run each strategy through all scenarios (100 replications per scenario for statistical significance) and produce:

- **Summary tables**: Mean KPI values with 95% confidence intervals.
- **Radar charts**: Showing each strategy's performance across KPIs (normalized to the baseline).
- **Sensitivity analysis**: How performance degrades as load increases or disruption frequency rises. The strategy that degrades most gracefully is preferred.
- **Cost-benefit analysis**: Quantifying the financial impact (reduced tardiness penalties, lower WIP carrying costs, improved throughput  more revenue capacity).

**Decision Rule**: Select the strategy that provides the best weighted combination of KPIs, with weights determined by management's priorities. Typically, tardiness reduction receives the highest weight for customer-facing operations.

### Continuous Monitoring and Adaptation Framework

After deployment, the system must continuously learn and adapt. A static strategy will degrade as conditions change.

**Real-Time Process Mining Pipeline:**

```
MES Events  Event Log  Real-Time Process Model  KPI Dashboard  Alert Engine  Strategy Adjustment
```

**Component 1: KPI Monitoring Dashboard**

Continuously compute and visualize:
- **Flow time trend**: Rolling 7-day average of job flow times, segmented by job type and routing.
- **Queue length heat map**: Real-time queue lengths at all work centers, color-coded by severity.
- **Setup time tracking**: Actual setup times compared to the historical matrix—flagging transitions where actual exceeds expected by >2 standard deviations (indicating process drift).
- **Tardiness tracker**: Rolling tally of tardy jobs, with root cause tagging (was it a bottleneck delay? A disruption? An inaccurate estimate?).

**Component 2: Automated Drift Detection**

Statistical process control applied to key metrics:
- **CUSUM (Cumulative Sum) control charts** on flow times and setup times to detect gradual shifts in process performance.
- **Change point detection** (e.g., Bayesian Online Change Point Detection) on task duration distributions to identify when a machine or operator's performance has shifted, triggering re-fitting of duration models.
- **Anomaly detection** on disruption frequency to detect whether the environment is becoming more volatile.

**Component 3: Strategy Adaptation Triggers**

| Trigger Condition | Adaptation Action |
|-------------------|-------------------|
| Flow time P90 increases by >15% over 4 weeks | Re-run bottleneck analysis; adjust dispatching rule weights |
| Setup time at a machine increases by >10% | Investigate cause (new part family? tooling degradation?); update setup time matrix |
| New job routing variant appears >5 times | Add new variant to the routing probability model |
| Machine breakdown frequency increases | Update reliability model; consider proactive maintenance scheduling |
| Priority distribution shifts (e.g., more urgent jobs) | Re-train tardiness prediction model with recent data; adjust urgency weight in dispatching rule |

**Component 4: Periodic Re-Optimization**

Quarterly:
1. Re-mine the full event log (now containing additional months of data).
