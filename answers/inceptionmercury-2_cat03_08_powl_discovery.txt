**POWL model that captures the observed manufacturingdistribution process**

Below is a **complete, selfcontained description** of the POWL model that reproduces the behaviour that can be seen in the five cases of the event log.  
The model is expressed with the same classes that are used in *pm4py* (`Transition`, `SilentTransition`, `OperatorPOWL`, `StrictPartialOrder`) and the two operators that are required (`XOR` for a choice, `LOOP` for a repeatable fragment).

---

### 1.  Atomic activities (transitions)

| Variable | Class | Meaning (label) |
|----------|-------|-----------------|
| `or_`    | `Transition(label="OR")` | Order Received |
| `ms`     | `Transition(label="MS")` | Material Sourcing |
| `qs`     | `Transition(label="QS")` | Quality Screening |
| `as_`    | `Transition(label="AS")` | Assembly |
| `iqc`    | `Transition(label="IQC")` | Inline Quality Check (may be repeated) |
| `tst`    | `Transition(label="TST")` | Testing |
| `rt`     | `Transition(label="RT")` | ReTesting (if the test fails) |
| `pk`     | `Transition(label="PK")` | Packaging |
| `lb`     | `Transition(label="LB")` | Labeling (optional) |
| `dp`     | `Transition(label="DP")` | Distribution Planning |
| `d`      | `Transition(label="D")`  | Dispatch |
| `skip`   | `SilentTransition()`      | “donothing” – used to model the *skip* of the optional labeling step |

---

### 2.  Structured operators

| Variable | Class (operator) | Children (order inside the operator) |
|----------|------------------|--------------------------------------|
| `loop_iqc` | `OperatorPOWL(operator=Operator.LOOP, children=[as_, iqc])` | **A** = `as_` (execute once)  **B** = `iqc` (optional repeat).  The semantics are: *execute Assembly, then either exit the loop or execute IQC and go back to Assembly again*. |
| `loop_rt`  | `OperatorPOWL(operator=Operator.LOOP, children=[tst, rt])` | **A** = `tst`  **B** = `rt`.  *Execute Testing, if it fails execute ReTesting and go back to Testing; otherwise exit*. |
| `xor_lb`   | `OperatorPOWL(operator=Operator.XOR, children=[lb, skip])` | *Exclusive choice between “Labeling” and “skip”.  The log shows that some cases omit the labeling step, which is modelled by the silent transition*. |

> **Why a loop for IQC?**  
> In cases1,3 and5 the pattern *AS  IQC  AS  …* appears several times.  The `LOOP` operator captures the “repeatuntilnomoreIQC” behaviour.  
> **Why a loop for RT?**  
> In cases1,3 and5 the pattern *TST  RT  TST* occurs when a product fails the first test.  The `LOOP` operator models the “retestuntilpass” behaviour.  

---

### 3.  Global partial order (the “workflow” skeleton)

All fragments above are placed inside a **single `StrictPartialOrder`** that defines the *causal* dependencies between the major steps of the process.  
The nodes of the order are the **toplevel objects** (atomic transitions or operatornodes).  
The order edges are:

```
or_    ms
ms     qs
qs     loop_iqc
loop_iqc  loop_rt
loop_rt    pk
pk         xor_lb
xor_lb     dp
dp         d
```

In codelike notation (the nodes are immutable after construction, therefore they are listed once in the constructor):

```python
# -----------------------------------------------------------------
# 1. atomic transitions (already defined above)
# -----------------------------------------------------------------
# or_, ms, qs, as_, iqc, tst, rt, pk, lb, dp, d, skip

# -----------------------------------------------------------------
# 2. operator nodes
# -----------------------------------------------------------------
loop_iqc = OperatorPOWL(operator=Operator.LOOP,
                        children=[as_, iqc])          # * (AS , IQC)

loop_rt  = OperatorPOWL(operator=Operator.LOOP,
                        children=[tst, rt])           # * (TST , RT)

xor_lb   = OperatorPOWL(operator=Operator.XOR,
                        children=[lb, skip])          # X ( LB ,  )

# -----------------------------------------------------------------
# 3. partialorder skeleton
# -----------------------------------------------------------------
root = StrictPartialOrder(
        nodes=[or_, ms, qs, loop_iqc, loop_rt, pk, xor_lb, dp, d]
)

# add the causal edges (the order is irreflexive, transitive, asymmetric)
root.order.add_edge(or_, ms)
root.order.add_edge(ms, qs)
root.order.add_edge(qs, loop_iqc)
root.order.add_edge(loop_iqc, loop_rt)
root.order.add_edge(loop_rt, pk)
root.order.add_edge(pk, xor_lb)
root.order.add_edge(xor_lb, dp)
root.order.add_edge(dp, d)
```

> **Concurrency** – The log never shows two activities happening truly in parallel, therefore the partial order is *totally* ordered (i.e. a linear chain).  If in a later dataset some activities could be performed concurrently (e.g. packaging and labeling), you would simply **omit** the edge between the corresponding nodes; the `StrictPartialOrder` would then treat them as *concurrent*.

### 4.  Summary of the final POWL structure (textual view)

```
OR  MS  QS  * ( AS , IQC )  * ( TST , RT )  PK  X ( LB ,  )  DP  D
```

* `* (A , B)` denotes a **loop** (execute A, then optionally B and go back to A).  
* `X ( A , B )` denotes an **exclusive choice** (either A or B).  
* `` is a **silent transition** (the “skip” of the optional labeling step).

This POWL model reproduces every trace observed in the five cases:

| Case | Path taken in the model |
|------|--------------------------|
| 1    | ORMSQS* (ASIQC)  * (TSTRT)  PKLBDPD |
| 2    | ORMSQS* (ASIQC)  * (TST)  PKDPD |
| 3    | ORMSQS* (ASIQC)  * (TSTRT)  PKLBDPD |
| 4    | ORMSQS* (ASIQC) (no IQC, loop exits immediately)  * (TST)  PKLBDPD |
| 5    | ORMSQS* (ASIQC)  * (TSTRT) (multiple RT cycles)  PKLBDPD |

All the observed **rework loops** (IQC and RT) and the **optional labeling** are captured by the `LOOP` and `XOR` operators, respectively, while the `StrictPartialOrder` enforces the required sequential dependencies.
