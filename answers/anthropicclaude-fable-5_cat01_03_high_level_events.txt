# Event Abstraction: From Low-Level Events to High-Level Process Stages

## 1. Analysis Approach

To group low-level events, I applied three abstraction heuristics:

1. **Temporal proximity** — events occurring within seconds of each other likely belong to the same work phase (note the ~40s gap between "Preheat" and "Pick up welding tool" — a natural phase boundary).
2. **Resource continuity** — consecutive events by the same resource (or functionally related resources, e.g., Operator B + welding tools) typically constitute one task.
3. **Functional/semantic cohesion** — events that contribute to a single physical outcome (preparing material, joining parts, finishing the surface, verifying quality) belong together.

Boundaries between high-level steps were placed where **resource handoffs** coincide with **time gaps** and a **change in functional purpose**.

---

## 2. Proposed High-Level Activities

### **HL-1: Material Preparation**
| Low-Level Activities | Resources |
|---|---|
| Retrieve raw metal sheet | Operator A |
| Scan barcode | Operator A |
| Place metal sheet on table | Operator A |
| Align metal sheet | Robot Arm #2 |
| Preheat metal sheet | Heating Unit #1 |

**Rationale:** All five events occur within a ~20-second window and serve a single purpose: getting the raw material identified, positioned, and conditioned before any value-adding transformation. Although three different resources are involved (Operator A, Robot Arm #2, Heating Unit #1), they form a tightly chained handoff sequence dedicated to one component. The barcode scan is included here because it is part of intake/registration, not inspection. The step ends when the material is ready and a ~40-second gap precedes the next resource (Operator B).

---

### **HL-2: Welding / Assembly**
| Low-Level Activities | Resources |
|---|---|
| Pick up welding tool | Operator B |
| Weld corner A | Operator B |
| Weld corner B | Operator B |

**Rationale:** A new resource (Operator B) begins working, marking a phase transition. "Pick up welding tool" is a **setup action** that exists only to enable the welds that immediately follow — tool acquisition + tool use is a classic pattern indicating a single task instance. All three events span only ~10 seconds, performed by one resource, transforming the part by joining (the core value-adding step).

---

### **HL-3: Weld Quality Inspection**
| Low-Level Activities | Resources |
|---|---|
| Measure weld integrity | Quality Sensor #1 |

**Rationale:** Even though it's a single event, it deserves its own high-level step because: (a) it is performed by a dedicated quality resource, (b) it is a verification activity rather than transformation, and (c) it acts as a **decision point** — the integrity score (95, 93) plausibly gates whether the case proceeds to coating or is reworked. Merging it into Assembly would hide this control point in the process model.

---

### **HL-4: Surface Finishing (Coating)**
| Low-Level Activities | Resources |
|---|---|
| Apply protective coating | Coating Unit #3 |
| Dry coating | Heating Unit #2 |

**Rationale:** "Apply" and "Dry" are inseparable sub-steps of one treatment — drying has no meaning without a prior coating application. They occur in immediate sequence and jointly produce one outcome: a finished protective surface. Note: Heating Unit #2 is grouped here (not with HL-1's Heating Unit #1) — demonstrating that **functional purpose outranks resource similarity** when grouping.

---

### **HL-5: Final Quality Inspection**
| Low-Level Activities | Resources |
|---|---|
| Visual check | Operator C |

**Rationale:** Performed by a new resource (Operator C) after all transformation steps are complete; it is the end-of-line acceptance check (Check: Passed) that releases the product. Kept separate from HL-3 because the two inspections occur at different process stages, check different attributes (weld vs. overall appearance), and use different resource types (sensor vs. human).

## 5. Generalizable Rules for the Full Log

To apply this abstraction at scale:

1. **Map each low-level Activity label to a high-level group** using the dictionary above (deterministic mapping works here because the pattern is consistent across cases).
2. **Handle variants:** If the full log contains rework loops (e.g., a failed `Measure weld integrity` followed by re-welding), allow high-level activities to repeat — the abstraction should produce traces like `...  Welding  Inspection  Welding  Inspection  ...`.
3. **Use gap thresholds as a safety net:** if an unmapped low-level event appears, assign it to the group of its temporally nearest neighbor by the same resource, or flag it for manual review.
4. **Preserve key attributes when aggregating:** carry forward decision-relevant data (IntegrityScore, Check result, PartID) as attributes of the high-level event so the abstracted log remains useful for conformance and performance analysis.
