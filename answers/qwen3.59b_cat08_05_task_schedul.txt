## 1. Analyzing Historical Scheduling Performance and Dynamics

To transition from reactive dispatching to data-driven optimization, we must first reconstruct the "As-Is" process model from the MES event logs. This involves transforming raw timestamps into a structured process flow that captures the reality of shop floor dynamics.

### Process Reconstruction & Mining Techniques
Using the provided event log snippet as a foundation, we will apply **Process Discovery** algorithms (e.g., Heuristic Miner or Inductive Miner) to generate a probabilistic Petri Net or Event Log model. This model will not just show the ideal flow, but the *actual* deviations caused by breakdowns and priority changes.

### Key Metrics & Quantification Strategies
We will extract specific metrics to quantify performance bottlenecks:

*   **Job Flow Times & Lead Time:**
    *   **Calculation:** $\text{LeadTime} = \text{Timestamp(Task End)} - \text{Timestamp(Job Released)}$.
    *   **Analysis:** We will plot the distribution of lead times using Kernel Density Estimation (KDE). This reveals if lead times are bimodal (indicating two distinct process paths) or heavily skewed right (indicating frequent breakdowns or starvation).
*   **Task Waiting Times (Queue Time):**
    *   **Calculation:** $\text{QueueTime} = \text{Timestamp(Task Start)} - \text{Timestamp(Previous Task End on Same Machine)}$.
    *   **Analysis:** We will calculate the average queue time per machine. High variance in queue times indicates instability. Specifically, we will analyze the "Queue Entry" events (e.g., `JOB-7001` waiting for `MILL-03`) to see if jobs sit idle unnecessarily.
*   **Resource Utilization:**
    *   **Calculation:** $\text{Utilization} = \frac{\sum(\text{Task Duration} + \text{Setup Duration})}{\text{Total Available Time}}$.
    *   **Analysis:** We will segment utilization by "Productive" (processing) vs. "Non-Productive" (setup, breakdown). High setup time on `CUT-01` might appear as low utilization but is actually masking inefficiency in sequencing.
*   **Sequence-Dependent Setup Times:**
    *   **Methodology:** We will join the log table with itself to identify pairs of jobs $(J_{prev}, J_{curr})$ processed on the same resource consecutively.
    *   **Formula:** $\text{SetupCost}(J_{prev}, J_{curr}) = \text{Duration(Setup Start)} - \text{Duration(Setup End)}$.
    *   **Clustering:** We will use K-Means clustering to group job pairs based on setup duration. This allows us to identify "High Friction" transitions (e.g., switching from Lathing to Cutting) versus "Low Friction" transitions, quantifying the penalty of changing material types or tolerances.
*   **Schedule Adherence & Tardiness:**
    *   **Metric:** $\text{Tardiness} = \max(0, \text{Actual Completion Date} - \text{Due Date})$.
    *   **Analysis:** We will calculate the "Tardiness Rate" (percentage of jobs missing due dates). Crucially, we will perform a **Conformance Check** against the planned schedule to see where the deviation occurred (e.g., was it a delay at `CUT-01` or did the job simply wait too long in queue?).
*   **Impact of Disruptions:**
    *   **Methodology:** We will correlate timestamps of "Breakdown Start" events with the subsequent lead time of jobs queued on that machine.
    *   **Metric:** $\text{Disruption Impact} = \sum(\text{Delay caused by event})$. This isolates the cost of unplanned maintenance vs. inherent process variability.

## 2. Diagnosing Scheduling Pathologies

Based on the analysis in Section 1, we will identify specific inefficiencies ("pathologies") that explain the high WIP and tardiness.

### Key Pathologies & Evidence
*   **Bottleneck Resource Identification:**
    *   **Pathology:** A machine (e.g., `MILL-02`) is consistently overloaded while others starve.
    *   **Evidence:** The process model will show a "siphon" or bottleneck where jobs accumulate upstream of this resource. If utilization > 85% and queue times are high, it is a true bottleneck. If utilization is low but throughput is low, the issue is likely starvation (upstream dependency).
*   **Priority Inversion & Dispatching Logic Failure:**
    *   **Pathology:** High-priority "Hot Jobs" (like `JOB-7005`) are delayed by lower-priority jobs.
    *   **Evidence:** Variant analysis comparing "On-Time" vs. "Late" cases. We will filter logs where a High Priority job entered the queue but was processed after Medium Priority jobs. This proves the dispatching rule (likely FCFS) is ignoring priority weights.
*   **Setup Time Bloat:**
    *   **Pathology:** Suboptimal sequencing increases total lead time.
    *   **Evidence:** Analysis of setup logs will show that switching between dissimilar materials (e.g., Steel to Aluminum) causes disproportionately long setups compared to similar material switches. The current schedule ignores this, treating all setups as equal or static.
*   **Bullwhip Effect in WIP:**
    *   **Pathology:** Small variations in task duration cause large fluctuations in inventory levels between workstations.
    *   **Evidence:** Little's Law analysis ($\text{WIP} = \text{Throughput} \times \text{LeadTime}$). If Lead Time variance is high (due to setups/breakdowns) but Throughput is stable, WIP must spike. We will visualize the "Inventory Accumulation Points" in Sankey diagrams where flow slows down.

### Process Mining Evidence
*   **Bottleneck Analysis:** Using the "Miner's Algorithm" to find resources with the longest average residence time for jobs.
*   **Resource Contention:** Mapping the overlap of tasks on shared resources (e.g., if `CUT-01` and `MILL-03` require a specific operator).
*   **On-Time vs. Late Comparison:** Overlaying process maps for late jobs to see if they share common delays (e.g., always getting stuck at the same machine) versus early jobs.

## 3. Root Cause Analysis of Scheduling Ineffectiveness

Understanding *why* these pathologies exist is critical before applying fixes. We must distinguish between "Capacity Constraints" and "Logic Flaws."

### Differentiating Root Causes
*   **Static Rules vs. Dynamic Environment:**
    *   **Issue:** The current system uses static rules (FCFS/EDD) applied at the start of a shift.
    *   **Root Cause:** These rules cannot account for dynamic changes like `JOB-7005`'s priority change occurring mid-shift (as seen in the log). The scheduler lacks a "Re-planning" trigger based on event logs.
*   **Inaccurate Duration Estimation:**
    *   **Issue:** Planned duration $\neq$ Actual duration (e.g., `CUT-01` planned 60 min, actual 66.4 min).
    *   **Root Cause:** The planning engine uses average times, not distributions. Variability is ignored. If a machine takes longer on a specific material batch, the schedule fails immediately.
*   **Sequence-Dependent Setup Ignorance:**
    *   **Issue:** Scheduling assumes setup time is constant or independent of the previous job.
    *   **Root Cause:** The MES log shows `Setup Required: TRUE` and `Previous Job: JOB-6998`. The current scheduler likely treats this as a generic "Changeover" time rather than a specific $S_{ij}$ function. This leads to scheduling dissimilar jobs consecutively, maximizing setup waste.
*   **Lack of Real-Time Visibility:**
    *   **Issue:** Operators or planners do not know `MILL-02` is down until the breakdown event logs in.
    *   **Root Cause:** No predictive maintenance integration. The system reacts to events rather than anticipating them.

### Mining for Logic vs. Capacity
*   **Simulation of Variability:** By introducing stochastic noise (based on actual variance) into a simulation of the current logic, we can see if delays are inevitable (capacity issue) or avoidable (logic issue).
*   **Constraint Checking:** If the system *could* theoretically schedule jobs faster but doesn't due to "setup rules," that is a logic constraint. If it *cannot* even meet capacity, that is a resource constraint.

## 4. Developing Advanced Data-Driven Scheduling Strategies

Based on the mining insights, we will implement three distinct strategies. These move beyond simple rules to dynamic, predictive approaches.

### Strategy 1: Dynamic Weighted Dispatching Rules (WDR)
*   **Core Logic:** Replace static FCFS/EDD with a dynamic index score calculated in real-time for every job waiting in a queue.
*   **Formula:** $\text{PriorityScore} = \frac{\alpha}{\text{DueDate}} + \beta \times \text{RemainingProcessingTime} - \gamma \times \text{DownstreamLoad}$.
*   **Process Mining Input:**
    *   The mining analysis will determine the optimal weights ($\alpha, \beta, \gamma$). For example, if data shows that ignoring downstream load causes starvation at `MILL-03`, $\gamma$ is increased.
    *   If "Hot Jobs" consistently get delayed by 4 hours when queued behind Medium jobs, we increase $\alpha$ (Due Date urgency) significantly for High Priority items.
*   **Impact:** Reduces tardiness by prioritizing jobs that are truly at risk of missing deadlines, rather than just those with the earliest date.

### Strategy 2: Predictive Scheduling & Duration Modeling
*   **Core Logic:** Use Machine Learning (e.g., Gradient Boosting) to predict actual task durations before they start, and predict machine availability.
*   **Process Mining Input:**
    *   Train models on historical logs where features include `Job Material`, `Operator ID`, `Machine ID`, and `Previous Job Type`.
    *   Output: A probability distribution for Task Duration (e.g., "This Cutting task has a 90% chance of taking between 65-70 mins").
*   **Application:** When creating the schedule, we use the *predicted* duration + safety margin derived from historical variance, rather than the static planned time.
*   **Impact:** Improves Lead Time Accuracy and reduces WIP because jobs are not over-scheduled (causing queue starvation) or under-scheduled (causing lateness).

### Strategy 3: Sequence-Dependent Setup Optimization (SDO)
*   **Core Logic:** Optimize the sequence of jobs on bottleneck machines to minimize total setup time. This is a variation of the Traveling Salesman Problem (TSP) adapted for job shops.
*   **Process Mining Input:**
    *   Identify "High Friction" transitions from Section 1 (e.g., `CUT-01` switching from `JOB-6998` to `JOB-7001` takes 23 mins vs. 5 mins for similar materials).
    *   Cluster jobs into families based on material/process requirements.
*   **Application:** The scheduler will attempt to group "Family A" jobs together before switching to "Family B," even if it means slightly altering the release order, to reduce total changeover time.
*   **Impact:** Significantly reduces non-value-added setup time, increasing effective machine capacity and reducing WIP waiting for setups.

## 5. Simulation, Evaluation, and Continuous Improvement

Before deploying these strategies live, we must validate them against the current system using a rigorous testing framework.

### Discrete-Event Simulation (DES)
*   **Modeling:** We will build a DES model in software like FlexSim or AnyLogic.
*   **Parameterization:** We will feed the model with mined distributions:
    *   Task Durations (Beta Distribution derived from logs).
    *   Breakdown Frequencies (Poisson Process based on historical downtime).
    *   Setup Times (Function of $Job_{prev}, Job_{curr}$).
*   **Scenarios:**
    1.  **Baseline:** Current FCFS/EDD rules with static setup times.
    2.  **Scenario A:** Strategy 1 (Dynamic WDR) applied only to bottleneck machines.
    3.  **Scenario B:** Strategy 3 (Setup Optimization) applied to all machines.
    4.  **Scenario C:** Hybrid of A and B + Predictive Maintenance alerts.
*   **Evaluation Metrics:** Compare Tardiness %, WIP Level, and Throughput across 10,000 simulation replications per scenario.

### Continuous Improvement Framework
Once deployed, the MES logs will continue to feed the analysis loop:
*   **Drift Detection:** Monitor the distribution of "Planned vs. Actual" times. If a new material type is introduced that consistently exceeds predicted times, the model must be retrained.
*   **KPI Monitoring Dashboard:** Real-time dashboards showing "Schedule Adherence Rate" and "Bottleneck Utilization."
*   **Automatic Adjustment:** Set thresholds (e.g., if Tardiness > 5% for 3 consecutive days). Trigger an alert to the Operations Manager to review the Dispatching Rule weights.
*   **Feedback Loop:** The continuous flow of new event logs updates the Process Model, refining the predictive accuracy over time.

By leveraging Process Mining, Precision Parts Inc. moves from a reactive, rule-based scheduling system to a proactive, data-driven operation.
