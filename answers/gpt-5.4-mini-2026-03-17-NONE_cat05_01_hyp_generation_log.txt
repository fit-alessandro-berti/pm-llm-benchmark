## Prompt for the Target LLM

You are given a PostgreSQL event log for an order processing workflow, along with related `orders` and `resources` tables.

### Schema

#### `order_event_log`
- `case_id` (INTEGER): identifies a specific order/process instance
- `event_id` (INTEGER): unique ID of each event
- `activity` (VARCHAR): activity name
- `timestamp` (TIMESTAMP): event time
- `resource` (VARCHAR): actor performing the activity
- `additional_info` (VARCHAR): extra context

#### `orders`
- `case_id` (INTEGER)
- `customer_id` (INTEGER)
- `order_type` (VARCHAR)
- `order_value` (DECIMAL)

#### `resources`
- `resource_id` (VARCHAR)
- `role` (VARCHAR)
- `department` (VARCHAR)

### Expected Process Flow
A typical order should follow this sequence:
1. Register Order
2. Perform Credit Check
3. Validate Stock
4. Confirm Shipment
5. Ship Goods
6. Issue Invoice
7. Receive Payment

### Task

Analyze the event log data and identify any anomalies, deviations, or undesirable behaviors in the process execution.

For each anomaly you find:
1. Explain what is unusual about it.
2. Hypothesize plausible reasons it could have occurred, such as:
   - system or integration errors
   - process non-compliance
   - manual workarounds
   - training issues
   - data quality issues
   - policy violations
3. Suggest SQL queries that could be used to investigate the issue further.

### Important Instructions
- Do **not** assume the process always follows the expected sequence.
- Look for issues such as:
  - missing activities
  - activities occurring out of order
  - repeated or duplicated steps
  - implausible timestamps
  - suspicious resource usage
  - inconsistent activity-resource-department combinations
  - abnormal case durations
  - inconsistencies between the event log and `orders`
  - unusual patterns in `additional_info`
- Use the schema and data patterns to reason about the log.
- When proposing SQL, make it concrete and relevant to the anomaly.
- Prefer PostgreSQL-compatible SQL.
- You may use joins with `orders` and `resources` where useful.

### Output Format
Provide your answer in the following structure:

#### 1. Observed Anomalies
List each anomaly with:
- `case_id` or affected scope
- description
- evidence from the log
- likely impact

#### 2. Hypotheses
For each anomaly, provide one or more plausible explanations.

#### 3. Suggested SQL Investigations
For each hypothesis or anomaly, provide SQL queries that could help validate it.

#### 4. Summary
Briefly summarize the overall process quality and main risks.

### Notes
- Focus on practical investigative reasoning rather than generic commentary.
- The goal is to uncover both obvious and subtle process problems.
- If multiple anomalies are related, group them together.
