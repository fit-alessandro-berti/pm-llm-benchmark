The visualization is a **BPMN-style process diagram** for preparing and dispatching goods in a **Computer Hardware Shop**. It shows the workflow across three swimlanes: **Logistics**, **Secretary**, and **Warehouse**.

### Overall purpose

The process starts when goods need to be shipped and ends when the shipment is prepared and moved to the pickup area. The diagram shows that two major streams happen in parallel:

1. **Administrative/shipping preparation**, handled mainly by the Secretary and Logistics.
2. **Physical packaging**, handled by the Warehouse.

These streams later join before the final shipment paperwork and pickup-area handoff.

---

## Step-by-step flow

### 1. Shipment process starts

The process begins with the start event labeled **“Ship goods.”**

Immediately after that, a **parallel gateway** marked with a **plus sign** splits the process into two simultaneous paths:

* One path goes to the **Secretary** for shipment-method clarification.
* The other path goes to the **Warehouse** for packaging the goods.

This means shipment planning and packaging can begin at the same time.

---

## Secretary / administrative path

### 2. Clarify shipment method

The Secretary first performs the task:

**“Clarify shipment method.”**

After that, the process reaches a decision point:

**“Special handling?”**

This is an exclusive gateway, meaning the process chooses one of two alternatives.

---

### 3A. If special handling is required

If the answer is **yes**, the Secretary:

1. **Gets 3 offers from logistics companies**
2. **Selects a logistics company and places the order**

This path is for shipments that require special arrangements, such as fragile items, oversized goods, express delivery, hazardous materials, or other special logistics requirements.

---

### 3B. If no special handling is required

If the answer is **no**, the process goes to:

**“Check if insurance is required.”**

This task uses supporting information:

* **Order details**
* **Customer information**

The Secretary checks whether the parcel should be insured.

Then the process reaches another decision area:

* If insurance is necessary, the parcel is sent to **Logistics** for insurance.
* Regardless of whether insurance is needed, the Secretary still proceeds to **write the package label**.

So this part has two possible activities:

**“Insure parcel”**
and
**“Write package label.”**

The label is always created, while insurance happens only when required.

These paths are then merged back together before continuing.

---

## Warehouse / packaging path

### 4. Package goods

At the same time as the administrative path, the Warehouse performs:

**“Package goods.”**

There is a timer event attached to this task labeled:

**“48 hours.”**

This means the process monitors the packaging duration. If packaging takes too long, specifically more than 48 hours, an exception path is triggered.

---

### 5. Delay notification

If the 48-hour timer is triggered, the Warehouse performs:

**“Notify customer about delay.”**

After notifying the customer, the flow rejoins the normal packaging path.

The intention is: packaging should proceed normally, but if it is delayed, the customer must be informed before the shipment preparation continues.

---

## Joining the paths

### 6. Administrative path and packaging path synchronize

Once both major streams are complete:

* the shipment method / insurance / labeling / logistics-order side, and
* the packaging side,

they meet at another **parallel gateway** marked with a **plus sign**.

This gateway acts as a synchronization point. The process cannot continue until both sides are ready.

---

## Final shipment preparation

### 7. Fill out shipment paperwork

After synchronization, the Warehouse performs:

**“Fill out Shipment paperwork.”**

This likely includes shipping documents, carrier forms, delivery notes, customs documents if needed, or internal dispatch records.

---

### 8. Move package to pickup area

Then the package is moved to the pickup area:

**“Move package to pickup area.”**

---

### 9. Shipment prepared

The process ends with the end event:

**“Shipment prepared.”**

At this point, the goods are packed, labeled, documented, and ready for collection or dispatch.

---

## Meaning of the main symbols

The diagram uses common BPMN symbols:

| Symbol                 | Meaning in this diagram                                                   |
| ---------------------- | ------------------------------------------------------------------------- |
| Circle at the start    | Start event: the shipping process begins                                  |
| Circle at the end      | End event: shipment is prepared                                           |
| Rounded rectangle      | Task or activity                                                          |
| Diamond with plus sign | Parallel gateway: paths split or join and happen in parallel              |
| Diamond with X         | Exclusive gateway: choose one path based on a condition                   |
| Diamond with circle    | Inclusive-style branching/merging, used where one or more paths may apply |
| Document icon          | Data object, such as order details                                        |
| Cylinder icon          | Data store, such as customer information                                  |
| Timer attached to task | Time-based exception, here after 48 hours                                 |

---

In plain language: **the shop starts preparing a shipment by arranging the shipping details and packaging the goods at the same time. Depending on the shipment type, it may request logistics offers, buy insurance, and create labels. If packaging is delayed beyond 48 hours, the customer is notified. Once everything is ready, the paperwork is completed, the package is moved to the pickup area, and the shipment is considered prepared.**
