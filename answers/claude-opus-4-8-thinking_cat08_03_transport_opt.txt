# Process Mining Analysis for Speedy Parcels: A Comprehensive Approach

## Section 1: Process Discovery and Conformance Checking

### 1.1 Data Preprocessing and Integration

The foundation of any successful process mining initiative is a clean, well-structured event log. For Speedy Parcels, this requires integrating four heterogeneous data sources into a unified log adhering to the core event log structure: **Case ID, Activity, Timestamp**, plus relevant attributes.

**Integration Approach:**

**a) Defining the Case Notion (Critical Decision):**
The choice of Case ID fundamentally shapes the analysis. I would create **multiple log perspectives** rather than committing to one:

- **Vehicle-Day perspective** (`V12-20241205`): Ideal for analyzing the operational flow of a route, vehicle utilization, and shift-level efficiency.
- **Package-centric perspective** (`P9876`): Essential for tracking individual parcel journeys, calculating delivery lead times, and analyzing failed-delivery/re-delivery loops that span multiple days.
- **Driver perspective**: For behavioral comparison.

Using **Object-Centric Process Mining (OCPM)** is strongly recommended here, since one event (e.g., "Arrive Customer") relates to multiple objects (vehicle, driver, package). OCPM avoids the "convergence/divergence" problems of flattening multi-object data into a single case notion.

**b) Timestamp Harmonization:**
- Normalize all timestamps to a single timezone and consistent granularity.
- GPS data is high-frequency (sub-minute); scanner/dispatch data is event-based. I would **resample/aggregate** GPS streams into meaningful activities rather than treating every ping as an event.

**c) Activity Abstraction from GPS Data:**
Raw GPS pings must be transformed into semantic activities. For example:
- Speed = 0 + ignition on + duration > threshold near customer  "Service/Dwell"
- Speed = 0 + not near a planned stop + sustained  "Unplanned Idle/Traffic Delay"
- Continuous movement between stops  "Travel Segment"

This **event abstraction** step uses geofencing (matching coordinates to depot/customer addresses) and speed-state classification.

**d) Cross-Source Correlation:**
- Link scanner Package IDs to dispatch's planned stop sequence.
- Join maintenance logs to vehicle-days via Vehicle ID + time overlap.
- Enrich each event with dispatch attributes (planned arrival window, planned sequence position, capacity).

**Anticipated Challenges:**

| Challenge | Description | Mitigation |
|-----------|-------------|------------|
| **Timestamp misalignment** | Clock drift between scanner devices and GPS units | Cross-validate using co-located events (e.g., "Arrive Customer" scan vs. GPS speed=0) |
| **GPS noise/gaps** | Urban canyons, tunnels cause signal loss | Interpolation, dead-reckoning; flag low-confidence segments |
| **Event abstraction ambiguity** | Distinguishing "stuck in traffic" from "stopped for delivery" | Combine speed, location proximity, and scanner context |
| **Missing data / incomplete cases** | A scan missed by driver | Use GPS dwell as proxy; flag incomplete traces |
| **Geocoding accuracy** | Customer addresses vs. actual GPS stop location | Geofence with tolerance radius |
| **Volume** | Six months × fleet × high-frequency GPS = massive data | Aggregate GPS pre-ingestion; use scalable PM tooling |

### 1.2 Process Discovery

I would apply discovery algorithms to visualize the **actual** as-is process:

- **Start with the Inductive Miner** (infrequent variant) to produce a sound, hierarchically-structured model that handles the noise and concurrency typical in delivery operations and guarantees a replayable, block-structured model.
- Use **Directly-Follows Graphs (DFG)** with frequency and performance overlays for intuitive stakeholder communication — showing the dominant flow: `Start Shift  Route Assigned  Depart Depot  [Travel  Arrive Customer  Service  Depart Customer]*  Arrive Depot  End Shift`.
- Apply **Fuzzy Miner** for the spaghetti-like reality of full operations to filter low-frequency paths.

The discovered model would reveal **emergent behaviors not in the plan**: maintenance stops mid-route, failed-delivery loops, re-sequencing, extended idle periods, and overtime patterns.

### 1.3 Conformance Checking

To compare actual execution against the **planned routes** from dispatch:

**Approach:**
- Construct a **normative model per route** from the dispatch system (the planned stop sequence with time windows).
- Apply **alignment-based conformance checking** to replay actual traces against the planned model, computing fitness and identifying precise deviation points.

**Deviation Types to Detect:**

1. **Sequence deviations** — Stops visited out of planned order (re-sequencing by drivers).
2. **Unplanned activities (insertions)** — Maintenance stops, unscheduled idle, personal stops, traffic delays.
3. **Skipped activities** — Planned stops not visited (deferred to next day).
4. **Temporal deviations** — Arrival outside the customer's requested time window; service taking far longer than allotted.
5. **Resource deviations** — Driver/vehicle substitutions vs. plan.

I would quantify deviations using **fitness scores** per route/driver and produce a **deviation heatmap** highlighting where and how often the reality diverges from the plan — this directly exposes whether planning is unrealistic or execution is undisciplined.

## Section 2: Performance Analysis and Bottleneck Identification

### 2.1 Key Performance Indicators

| KPI | Definition | Calculation from Event Log |
|-----|------------|---------------------------|
| **On-Time Delivery Rate** | % deliveries within requested window | Compare "Delivery Success" timestamp vs. dispatch time window |
| **Avg. Time per Stop (Service Time)** | Dwell at customer | "Depart Customer"  "Arrive Customer" |
| **Travel-to-Service Ratio** | Productive vs. transit time | (travel segments) / (service times) |
| **Failed Delivery Rate** | % attempts failing | Count("Delivery Failed") / total attempts |
| **Vehicle Utilization** | Active work vs. shift length | (Depart Depot  Arrive Depot active time) / (Start  End Shift) |
| **Traffic Delay Frequency/Duration** | Low-speed/idle events | Count & sum duration of "Low Speed/Unplanned Idle" |
| **Fuel per km/package** | Cost efficiency | Telematics fuel data / distance or packages delivered |
| **Route Completion Time** | Total cycle | "Arrive Depot"  "Depart Depot" |
| **Overtime Rate** | Shift overruns | "End Shift" beyond planned shift end |
| **Re-delivery Cost** | Repeat visits | Package traces with multiple "Arrive Customer" events |

### 2.2 Bottleneck Identification Techniques

**Techniques:**

1. **Performance-annotated process maps**: Overlay average/median waiting and service times on the DFG. Edges with high duration (e.g., long travel between specific stops) and nodes with high dwell (e.g., service time) immediately surface bottlenecks.

2. **Dotted Chart Analysis**: Plots all events over time, revealing time-of-day patterns — e.g., delays clustering during morning rush or lunchtime parking congestion.

3. **Variant Analysis**: Group cases by route/driver/region and compare performance distributions.

4. **Bottleneck quantification via waiting-time analysis**: Identify activities where cases queue (e.g., extended idle before "Arrive Customer" indicating parking search).

5. **Geospatial hotspot analysis**: Aggregate "Low Speed Detected" / idle events by location to map **traffic congestion hotspots**.

**Dimensions to slice bottlenecks by:**
- **Route/territory** — which geographic zones consistently overrun.
- **Time of day** — peak congestion windows.
- **Driver** — performance variance across drivers on similar routes.
- **Vehicle type** — older vans with more maintenance stops.
- **Activity** — is the bottleneck travel, parking, customer interaction, or scanning?

**Quantifying impact:** For each bottleneck, calculate total time/cost contribution = frequency × average impact × cost rate. E.g., "Parking search adds avg 4 min × 30 stops × 250 days = X hours of lost productivity / overtime cost annually." This prioritizes improvement effort by ROI.

## Section 3: Root Cause Analysis

Identifying *where* delays occur is insufficient; we must establish *why*. Process mining enables hypothesis-driven root cause validation.

### Potential Root Causes & Validation Methods

**a) Suboptimal/Static Route Planning**
- *Hypothesis:* Routes are planned statically, ignoring real-time conditions.
- *Validation:* High conformance deviation in sequence (drivers constantly re-sequence) suggests the plan doesn't match optimal real-world order. Compare planned vs. actual travel times — systematic underestimation indicates static assumptions.

**b) Inaccurate Travel Time Estimations**
- *Validation:* Compute the delta between planned segment durations and actual GPS-derived travel times. If actual consistently exceeds plan on certain segments/times, estimations are flawed. Correlate with time-of-day.

**c) Traffic Congestion Patterns**
- *Validation:* Correlate "Low Speed/Idle" events with external traffic data overlaid by location and time. Geospatial heatmaps confirm recurring congestion hotspots vs. random delays.

**d) High Service-Time Variability**
- *Validation:* Analyze the **distribution** (not just mean) of dwell times per customer type/location. High variance points to address-finding issues, building access (apartments), or signature requirements. Use **dwell-time analysis** segmented by customer category.

**e) Vehicle Breakdowns/Maintenance**
- *Validation:* Correlate "Unscheduled Stop / Engine Warning" frequency with vehicle age/mileage from maintenance logs. Identify whether breakdowns cluster on specific vehicles  reactive vs. predictive maintenance gap.

**f) Driver Behavior/Skill**
- *Validation:* **Comparative variant analysis** of high- vs. low-performing drivers on *similar routes*. Controlling for route difficulty, if Driver A consistently achieves lower service times and higher on-time rates, the difference is behavioral/skill-based (e.g., parking strategy, customer handling).

**g) Failed Delivery Attempts**
- *Validation:* Analyze failed-delivery traces by **time of day** and **customer location**. If failures cluster in working hours (customer not home), root cause is poor time-window matching. Quantify re-delivery cost via package traces with repeat visits.

**Key technique — Multi-dimensional correlation:** By combining variant analysis, dwell-time distribution analysis, and external data correlation, we move from *symptom* (late delivery) to *validated root cause* (e.g., "lateness in Zone 3 caused by chronic 11 AM congestion + static routing that doesn't account for it").

## Section 4: Data-Driven Optimization Strategies

### Strategy 1: Dynamic, Traffic-Aware Route Sequencing

- **Targets:** Bottleneck of excessive travel time and on-time delivery failures.
- **Root cause addressed:** Static route planning + inaccurate travel time estimations + ignored congestion patterns.
- **Data support:** Process mining revealed (i) drivers frequently re-sequence stops (conformance deviations), (ii) actual travel times exceed planned on specific segments, (iii) congestion hotspots map to recurring time windows. These insights feed a dynamic routing engine that re-optimizes sequence using historically-validated, time-dependent travel times.
- **Expected KPI impact:**  On-Time Delivery Rate,  Route Completion Time,  Travel-to-Service Ratio,  Fuel/km,  Overtime.

### Strategy 2: Predictive Maintenance Scheduling

- **Targets:** Unscheduled mid-shift breakdowns disrupting routes.
- **Root cause addressed:** Reactive maintenance regime; breakdowns clustering on high-usage/older vehicles.
- **Data support:** Correlation of "Engine Warning/Unscheduled Stop" events with vehicle usage patterns (mileage, idle time, route intensity) from integrated GPS + maintenance logs. Build usage-based maintenance triggers rather than purely calendar-based.
- **Expected KPI impact:**  Frequency/duration of unscheduled stops,  Vehicle Utilization,  Maintenance cost (planned cheaper than emergency),  On-Time Delivery (fewer disruptions).

### Strategy 3: Intelligent Time-Window Management & Customer Communication

- **Targets:** High failed-delivery rate and costly re-deliveries.
- **Root cause addressed:** Mismatch between delivery timing and customer availability; poor communication.
- **Data support:** Failed-delivery analysis showed failures cluster at specific times/locations (customer not home). Process mining of package re-delivery loops quantifies wasted trips. Solution: proactive ETA notifications, customer-selectable windows, and clustering deliveries to a location by historical success windows.
- **Expected KPI impact:**  Failed Delivery Rate,  Re-delivery cost,  First-attempt success,  Fuel/package.

### Strategy 4 (Additional): Targeted Driver Coaching

- **Targets:** Service-time variability and route execution inefficiency.
- **Root cause addressed:** Skill/behavior differences validated via comparative variant analysis.
- **Data support:** High-performing drivers' practices (parking strategy, efficient sequencing) are identified and codified into training for lower-performing peers. Crucially, comparison controls for route difficulty to ensure fairness.
- **Expected KPI impact:**  Avg & variance of service time,  On-Time Rate, reduced performance dispersion across fleet.

## Section 5: Operational Constraints and Continuous Monitoring

### 5.1 Accounting for Operational Constraints

Optimization strategies must respect hard constraints, embedded as rules in the routing/scheduling engine:

- **Driver Working Hours:** Dynamic routing must enforce legal/contractual shift limits and break requirements. Process mining's overtime analysis ("End Shift" overruns) feeds realistic shift-capacity planning, preventing over-assignment.
- **Vehicle Capacity:** Package assignment respects volume/weight limits (from dispatch data). Route optimization solves a Capacitated Vehicle Routing Problem (CVRP).
- **Customer Time Windows:** The routing engine treats time windows as constraints, not just objectives — balancing on-time commitments against travel efficiency (VRPTW). Time-window management (Strategy 3) widens flexibility where customers permit.

These create a **constrained optimization** problem; process mining provides the realistic parameters (true travel/service times) that make constraint-handling accurate rather than based on optimistic plans.

### 5.2 Continuous Monitoring Plan

Optimization is not one-off. I would establish **operational process mining dashboards** for sustained value:

**Dashboard Components:**

1. **KPI Trend Monitor:** On-Time Rate, Failed Delivery Rate, Travel-to-Service Ratio, Fuel/package, Overtime — tracked over time with **before/after** comparison to validate improvement.

2. **Live Conformance Dashboard:** Continuous comparison of actual execution vs. (newly optimized) plans. Alerts when fitness drops, signalling new deviations.

3. **Geospatial Hotspot Map:** Refreshed congestion/delay hotspots — detects *emerging* traffic patterns or problem zones early.

4. **Driver/Route Scorecards:** Ongoing comparative performance to sustain coaching gains and flag regression.

5. **Predictive Maintenance Alerts:** Vehicle health indicators trending toward thresholds.

**Sustainability Mechanisms:**
- **Statistical control limits** on KPIs to distinguish normal variation from genuine degradation.
- **Concept drift detection:** Process mining can detect when the *process itself* shifts (e.g., seasonal volume changes, new urban construction) prompting re-optimization.
- **Feedback loop:** Monitoring insights feed back into route plans and travel-time models monthly — creating a **continuous improvement cycle** rather than static optimization.

**Key views to track:** the discovered process model (is it converging toward the optimized plan?), variant distribution (are inefficient variants shrinking?), and bottleneck evolution (are old bottlenecks resolved without creating new ones downstream?).

### Summary

This approach moves Speedy Parcels from intuition-based suspicion of inefficiency to **evidence-based optimization**: integrating multi-source data into object-centric event logs, discovering the true process, validating root causes through targeted analyses, implementing constraint-aware data-driven strategies, and sustaining gains through continuous process monitoring — directly addressing both punctuality and operational cost objectives.
