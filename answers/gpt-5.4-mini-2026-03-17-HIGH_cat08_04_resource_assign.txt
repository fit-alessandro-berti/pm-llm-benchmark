Below is a structured, process-mining-driven approach to diagnose and optimize TechSolve Solutions’ resource assignment practices.

---

## 1. Analyzing Resource Behavior and Assignment Patterns

### 1.1 Prepare the event log for resource analysis
Before analysis, I would enrich the log with derived attributes such as:

- **Case lifecycle times**: total resolution time, waiting time between steps, time to first touch, time between escalation and reassignment
- **Resource metrics**: number of tickets handled, active processing time, idle/queue time, number of handovers caused
- **Assignment quality metrics**: whether the first assigned agent had the required skill, number of reassignments, escalation count
- **Ticket context**: priority, category, required skill, channel, complexity proxy, SLA deadline

This creates the basis for comparing actual performance by **agent**, **tier**, **skill**, **priority**, and **category**.

---

### 1.2 Metrics to analyze agent and tier performance

#### By individual agent
I would calculate, at minimum:

- **Workload volume**
  - tickets opened / worked / completed
  - tickets per shift or per day
- **Throughput**
  - average tickets resolved per hour/day
- **Processing times**
  - average handling time per activity
  - average time to start work after assignment
- **Reassignment rate**
  - how often the agent’s tickets are later reassigned
- **Escalation rate**
  - share of tickets escalated from that agent
- **First-contact resolution / first-call resolution (FCR)**
  - percentage resolved without escalation or transfer
- **Skill-specific success rate**
  - resolution success for tickets requiring each skill
- **SLA breach contribution**
  - percentage of breached cases handled by the agent before breach
- **Overload indicator**
  - actual workload versus capacity / shift availability

#### By tier (L1, L2, L3)
At tier level, I would compare:

- **Total intake and completion volume**
- **Average wait time before work begins**
- **Average resolution time**
- **Escalation inflow/outflow**
- **Containment rate**
  - especially for L1: share of tickets resolved without escalation
- **Backlog growth**
- **SLA breach rate by tier**
- **Skill coverage**
  - which skills each tier is actually servicing most often

This will show whether L1 is truly performing triage and simple resolution, or whether L2/L3 are compensating for weak front-line assignment.

---

### 1.3 Process mining techniques to reveal actual assignment behavior

#### Resource interaction analysis
This helps identify how tickets move across resources and tiers. I would analyze:

- **handover frequency** between agents
- **handover frequency between tiers** (L1  L2, L2  L3)
- **rework loops** where cases bounce back and forth
- **self-assignment patterns**
- **reassignment chains**

A high number of repeated handovers or loops usually indicates poor initial routing, unclear ownership, or missing expertise.

#### Social network analysis based on handovers
Using handover-of-work networks, I would build a graph where:

- nodes = agents or tiers
- edges = ticket handovers

From this, we can identify:

- **central “hub” agents** who receive many reassignments
- **bottleneck specialists**
- **informal routing patterns**
- **cross-tier dependencies**
- **isolated agents** or underused specialists

This often exposes the *real* organizational structure, which is not always the same as the formal tier design.

#### Role discovery / organizational mining
Role discovery groups agents by the type of work they actually do, not just their job title. This is useful for detecting:

- L1 agents who behave like junior L2s
- L2 specialists who spend too much time on generic tickets
- L3 staff pulled into issues that should have been contained earlier
- agents with similar operational profiles but different formal roles

This shows whether the actual process design matches the intended support model.

---

### 1.4 Compare actual assignment patterns to intended logic
The intended logic sounds like:

- **L1** receives the ticket first
- **L1** resolves basic issues
- **L2/L3** handle more complex or skill-specific incidents
- assignments should depend on **required skill + tier + availability**

Process mining can reveal whether the reality is different, for example:

- tickets are assigned round-robin rather than skill-based
- a specialist receives tickets late after multiple handovers
- L1 escalates too quickly due to uncertainty
- tickets are reassigned because the first assignee does not match the required skill
- some agents are overloaded while others remain idle

A discovered process model and handover matrix will make these deviations visible.

---

### 1.5 Analyze utilization of specific skills
To understand skill usage, I would create a **skill-to-ticket matrix**:

- rows = agents / skills
- columns = ticket skill requirements
- values = volume, resolution time, success rate, and reassignment rate

Key indicators:

- **Skill match rate**
  - % of cases first assigned to someone with the required skill
- **Specialist utilization**
  - % of a specialist’s time spent on cases that truly require their expertise
- **Skill spillover**
  - specialists handling low-complexity or generic work
- **Skill gap frequency**
  - tickets requiring a skill for which capacity is insufficient
- **Skill proficiency performance**
  - resolution time and breach rate by skill-holder

This will tell you whether specialists are being used effectively or are being consumed by work that could be handled at a lower tier.

---

## 2. Identifying Resource-Related Bottlenecks and Issues

### 2.1 Bottlenecks caused by insufficient skill availability
I would look for:

- long queue times for specific required skills
- repeated assignment delays for the same skill group
- disproportionate SLA breaches for certain categories, e.g. Network or Security
- congestion around a few expert agents

Typical signs:
- many tickets waiting for `Networking-Firewall`
- L2/L3 agents with `Database-SQL` overloaded
- low-volume but high-delay skill groups

#### How to quantify
- average waiting time by required skill
- backlog aging by skill
- queue length over time by skill
- SLA breach rate for cases requiring scarce skills

---

### 2.2 Delays introduced by reassignments and escalations
Frequent reassignment is usually a major source of delay.

I would quantify:

- **average extra delay per reassignment**
  - compare cases with 0, 1, 2, 3+ reassignments
- **incremental resolution time**
  - resolution time increase associated with each additional handover
- **handover-to-start lag**
  - time between reassignment and next work start
- **reopen/escalation loops**
  - tickets escalated then returned or reassigned again

A simple and useful measure:

- **reassignment penalty**
  - average(resolution time of cases with reassignments) minus average(resolution time of direct-resolution cases)

You can also model this more rigorously using regression or survival analysis to isolate the effect of reassignments after controlling for priority and category.

---

### 2.3 Impact of incorrect initial assignments by L1 or dispatcher
Misrouting at the start is often the highest-impact issue because it creates avoidable downstream work.

Signals include:

- first assignee lacks the required skill
- cases routed to a resource whose historical success rate for that ticket type is low
- high first-touch escalation rate
- high reassignment rate after L1 start

#### Quantify the impact
- **skill mismatch rate**
  - % of cases first assigned to an agent without the required skill
- **breach rate for mismatched cases**
  - compare against matched cases
- **average extra time due to wrong first assignment**

A useful KPI is:

- **first assignment accuracy**
  - % of tickets assigned correctly on the first attempt

If this is low, the dispatch logic is a major root cause.

---

### 2.4 Overloaded vs underperforming agents/teams
To detect uneven workload, I would compare:

- tickets per agent
- active time versus capacity
- queue aging handled by each agent
- service times and output quality
- frequency of late starts or overdue work

Indicators of overload:
- high utilization
- rising backlog
- increased breaches for tickets assigned to the same agents
- increasing service times under peak demand

Indicators of underuse:
- low ticket volume despite skill availability
- low complexity work assigned to highly skilled staff
- idle time while queues remain open

A good way to summarize this is with a **workload balance index** or Gini coefficient across agents.

---

### 2.5 Correlation between assignment patterns and SLA breaches
I would build a breach analysis model with variables such as:

- number of reassignments
- first-assignment skill match
- time to first touch
- tier transitions
- current agent workload
- required skill scarcity
- ticket priority and category

This can be analyzed with:

- cross-tabs and cohort comparison
- logistic regression for breach probability
- survival analysis for time-to-resolution risk
- process mining conformance checks against SLA thresholds

#### Example quantifications
- % of SLA breaches linked to skill mismatch
- % of breaches after 2+ reassignments
- breach probability for tickets that start in the wrong tier
- average breach uplift for overloaded agents

## 3. Root Cause Analysis for Assignment Inefficiencies

### 3.1 Likely root causes

#### 1) Assignment rules are too simplistic
If routing is mostly round-robin within tiers, then it ignores:

- skill fit
- current workload
- ticket complexity
- SLA urgency
- agent availability

This often causes mismatches and reassignments.

#### 2) Skill profiles are outdated or incomplete
The system may think an agent has a skill they no longer practice, or fail to recognize hidden expertise.

Signs:
- documented skill does not match observed performance
- some agents repeatedly resolve certain issue types despite not being tagged for them
- specialist queues behave inconsistently across shifts

#### 3) Ticket classification is poor at intake
If category or required skill is wrong at creation, the ticket is routed incorrectly from the start.

Indicators:
- many cases are reclassified before resolution
- early reassignment follows a corrected diagnosis
- certain channels, such as email, have worse categorization accuracy

#### 4) No real-time workload visibility
Dispatchers may not see:

- active queue length
- backlog age
- current agent occupancy
- shift overlap
- time remaining before SLA breach

So tickets get assigned to the “next” person instead of the right person.

#### 5) L1 lacks empowerment or training
If L1 resolves a low proportion of tickets and escalates too early, the process overuses L2/L3.

Indicators:
- low L1 FCR
- high L1 escalation rate for repetitive patterns
- variance in performance among L1 agents
- cases escalated despite having known, simple resolution paths

---

### 3.2 How variant analysis helps
Variant analysis compares the “smooth” cases to the problematic ones.

#### Example comparison
- **Good variants**
  - single assignment, no escalation, low resolution time, SLA met
- **Bad variants**
  - multiple reassignments, tier bouncing, long waits, SLA breach

By comparing these variants, you can identify:

- which ticket types are most likely to drift into rework
- which agents or tiers are associated with the good vs bad paths
- whether delays begin at the first assignment or after escalation

This is useful for isolating patterns such as:
- “Network incidents with channel = email and low-quality categorization produce more reassignments”
- “P3 software tickets routed to L1 without App-CRM expertise often require L2 correction”

---

### 3.3 How decision mining helps
Decision mining models the actual decisions taken at assignment points.

For example, it can learn rules like:

- if priority = P2 and required skill = Network and backlog is high, assign to B08
- if ticket arrives via phone and L1 load is low, keep at L1
- if skill match is absent, escalate immediately

This lets you compare:
- **what should happen**
vs
- **what actually happens**

It can reveal whether decisions are driven by:
- skill and workload
- or simple sequence/round-robin behavior

Decision mining is particularly useful for identifying bad rules hidden in routine dispatch behavior.

## 4. Developing Data-Driven Resource Assignment Strategies

Below are four concrete strategies.

---

### Strategy 1: Skill-based routing with proficiency weighting
#### What it addresses
- skill mismatch
- repeated reassignments
- specialist overload caused by poor routing

#### How it works
Assign tickets based on a **fit score** using:
- required skill match
- agent proficiency in that skill
- historical success rate
- current workload
- SLA urgency

#### How process mining supports it
Process mining identifies:
- which agent-skill combinations resolve tickets fastest
- where mismatches occur most often
- which specialists are overused or underused

#### Data required
- agent skill matrix
- ticket required skill
- historical resolution success by skill
- handling time and breach data
- workload and shift calendars

#### Expected benefits
- fewer reassignments
- better first-time resolution
- lower SLA breach rate
- better specialist utilization

---

### Strategy 2: Workload-aware dynamic dispatch
#### What it addresses
- uneven workload
- overloaded agents
- long waiting times in queue

#### How it works
Routing considers:
- current queue length
- agent availability
- active WIP
- predicted handling time
- remaining SLA time

This is more intelligent than round-robin because it balances load while still respecting skill needs.

#### How process mining supports it
It reveals:
- which queues become congested
- which agents are bottlenecks
- how delay grows when utilization exceeds a threshold

#### Data required
- live workload data
- agent calendars/availability
- queue aging
- predicted service times
- ticket priority/SLA deadline

#### Expected benefits
- reduced waiting times
- smoother workload distribution
- fewer overload-driven breaches
- less idle time for underused agents

---

### Strategy 3: Predictive triage and skill prediction at intake
#### What it addresses
- wrong initial assignment
- poor categorization
- slow escalation due to uncertainty

#### How it works
Use historical ticket data and text features to predict:
- likely required skill
- likely tier of resolution
- probability of escalation
- expected resolution time

Then route more accurately at first touch.

#### How process mining supports it
Process mining identifies:
- which ticket patterns often result in rework
- which categories are misrouted most often
- which variants are associated with successful L1 containment

#### Data required
- ticket category
- description text / keywords
- channel
- historical final resolution skill/tier
- escalation and reassignment history

#### Expected benefits
- better first-assignment accuracy
- fewer early escalations
- shorter resolution cycles
- improved L1 containment

---

### Strategy 4: Refined escalation policy and L1 empowerment
#### What it addresses
- excessive escalation
- specialist overload
- low L1 first-contact resolution

#### How it works
Define escalation rules based on:
- ticket pattern
- known resolution playbooks
- threshold of attempts/time
- skill confidence score

Also train L1 agents on high-frequency recurring variants that they can resolve without escalation.

#### How process mining supports it
It identifies:
- common cases that L1 could have resolved
- escalation points where unnecessary handoff occurs
- tickets that succeed at L1 after minor delays versus those that truly need specialist care

#### Data required
- L1 success history
- knowledge base usage
- ticket variants
- escalation reasons
- agent performance by issue type

#### Expected benefits
- higher L1 resolution rate
- less pressure on L2/L3
- lower resolution cost
- faster response for complex tickets

---

### Optional Strategy 5: Cross-skilling and demand-based staffing
#### What it addresses
- skill shortages
- demand peaks by time/day
- specialist bottlenecks

#### How it works
Use demand heatmaps to:
- schedule more agents with scarce skills during peak hours
- cross-train agents in high-demand skills
- reallocate capacity across tiers dynamically

#### Data required
- demand by skill over time
- shift calendars
- training records
- historical backlog by time period

#### Expected benefits
- better skill coverage
- fewer queue buildups
- improved resilience during peaks

---

## 5. Simulation, Implementation, and Monitoring

### 5.1 Use simulation before implementation
I would build a **discrete-event simulation** or process simulation model using:

- arrival rates by ticket type and priority
- discovered routing probabilities
- service-time distributions by tier/skill/agent
- escalation and reassignment probabilities
- agent calendars and capacity constraints
- SLA deadlines

#### What to test
- current round-robin routing
- skill-based routing
- workload-aware routing
- hybrid routing rules
- different staffing or cross-skilling scenarios

#### What to compare
- average resolution time
- SLA breach rate
- average number of reassignments
- queue length and backlog aging
- specialist utilization
- L1 containment rate

Simulation is essential because it lets TechSolve test policies before changing live operations.

---

### 5.2 Implementation approach
A practical rollout would be:

1. **Pilot** on one ticket category or one region
2. Compare old vs new routing in parallel
3. Use a fallback rule for high-priority tickets
4. Refine the model weekly based on actual performance
5. Expand gradually to all incident types

This reduces operational risk and builds trust in the new routing logic.

---

### 5.3 Post-implementation monitoring with process mining dashboards
I would monitor the new strategy with dashboards that update regularly from the event log.

#### Core KPIs to track
- **First-assignment accuracy**
- **First-contact resolution rate**
- **Reassignment rate per ticket**
- **Escalation rate**
- **Average resolution time**
- **Average waiting time to first touch**
- **SLA breach rate by priority**
- **Workload balance across agents**
- **Utilization by tier and skill**
- **Specialist spillover into low-complexity work**
- **Backlog aging**
- **Skill-specific queue times**

#### Process views to track
- **directly-follows graphs** for flow changes
- **bottleneck heatmaps** by step and skill
- **handover networks** to detect unnecessary transfers
- **variant analysis** to track the share of “good” versus “bad” paths
- **conformance checking** against the new routing policy
- **time trend charts** for SLA and reassignment reduction
- **resource performance comparisons** by agent, team, and tier

#### Ongoing governance
- set alert thresholds for overload and breach risk
- review exceptions weekly
- refresh the model monthly or quarterly
- recalibrate skill scores based on actual performance

---

## Bottom line
Process mining can turn TechSolve’s incident log into a precise picture of how tickets really move, where assignment logic fails, and which resources or skills are being over- or under-used. The strongest improvement path is to combine:

- **skill-based routing**
- **workload-aware dispatch**
- **predictive triage**
- **refined escalation rules**
- backed by **simulation** and **continuous monitoring**

That combination should reduce SLA breaches, lower reassignments, balance workload, and use L2/L3 specialists where they add the most value.
