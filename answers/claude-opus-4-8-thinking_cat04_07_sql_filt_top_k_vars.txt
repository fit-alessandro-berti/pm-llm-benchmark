## DuckDB SQL Query

Here's a complete, well-structured query using CTEs. I'll use a parameter `K = 10` (adjust as needed).

```sql
WITH
-- 1. Build the ordered activity sequence (variant) for each case.
--    list() respects ORDER BY in DuckDB, and we join into a readable string.
case_variants AS (
    SELECT
        case_id,
        string_agg(activity, ' -> ' ORDER BY timestamp) AS variant
    FROM event_log
    GROUP BY case_id
),

-- 2 & 3. Count how many cases follow each variant, then rank them.
variant_frequency AS (
    SELECT
        variant,
        COUNT(*) AS case_count          -- one row per case in case_variants, so COUNT(*) = #cases
    FROM case_variants
    GROUP BY variant
),

ranked_variants AS (
    SELECT
        variant,
        case_count,
        ROW_NUMBER() OVER (
            ORDER BY case_count DESC, variant ASC   -- tie-break for determinism
        ) AS variant_rank
    FROM variant_frequency
),

-- Keep only the top K variants.
top_k_variants AS (
    SELECT variant
    FROM ranked_variants
    WHERE variant_rank <= 10            -- <-- K here
),

-- 4. Determine which cases belong to those top K variants.
top_k_cases AS (
    SELECT cv.case_id
    FROM case_variants cv
    JOIN top_k_variants tkv
      ON cv.variant = tkv.variant
)

-- Return all original events for cases in the top K variants.
SELECT el.*
FROM event_log el
JOIN top_k_cases tkc
  ON el.case_id = tkc.case_id
ORDER BY el.case_id, el.timestamp;
```

## How it works

| Step | CTE | Purpose |
|------|-----|---------|
| 1 | `case_variants` | Collapses each case into a single ordered sequence string using `string_agg(... ORDER BY timestamp)`. |
| 2 | `variant_frequency` | Groups identical sequences and counts how many cases follow each (`COUNT(*)` because each row in `case_variants` represents exactly one case). |
| 3 | `ranked_variants` + `top_k_variants` | Ranks variants by descending frequency and selects the top **K**. |
| 4 | `top_k_cases`  final `SELECT` | Maps top variants back to their cases, then returns **all original events** for those cases. |

## Notes & Variations

**Avoid delimiter collisions:** If an `activity` name could contain `->`, prefer a true list instead of a string:

```sql
case_variants AS (
    SELECT
        case_id,
        list(activity ORDER BY timestamp) AS variant   -- LIST<VARCHAR>
    FROM event_log
    GROUP BY case_id
)
```
DuckDB can `GROUP BY` and join on `LIST` values directly, so the rest of the query works unchanged.

**Parameterizing K:** Replace the literal `10` with a prepared-statement parameter:

```sql
WHERE variant_rank <= $K
```

**Handling ties at the K boundary:** `ROW_NUMBER` gives an exact cut of K variants. If you want to *include all variants tied* at the cutoff frequency, use `RANK()` instead of `ROW_NUMBER()` (this may return more than K variants).

**Duplicate timestamps within a case:** If two events in the same case share a timestamp, ordering is non-deterministic. Add a stable secondary key (e.g., an event id or `activity`) to the `ORDER BY` inside `string_agg`/`list` to make variants reproducible.
