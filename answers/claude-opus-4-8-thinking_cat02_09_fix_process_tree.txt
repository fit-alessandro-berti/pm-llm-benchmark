# 1. Issues with the Current Model

The core problem is the **`PARALLEL` (AND) operator** wrapped around five activities:

```python
parallel_block = PARALLEL(create_po, receive_goods, receive_invoice, match_invoice, pay_invoice)
```

The `PARALLEL` operator allows these activities to occur in **any interleaving order**, which permits domain-invalid traces such as:

- **Pay_Invoice** before **Receive_Invoice** (paying for an invoice you never received)
- **Match_Invoice_to_PO** before **Create_Purchase_Order** (matching against a PO that doesn't exist)
- **Receive_Goods** before **Create_Purchase_Order** (receiving goods never ordered)
- **Pay_Invoice** before **Match_Invoice_to_PO** (paying without verification)

**What the domain actually requires:**
Every one of these eight steps has a strict causal dependency on its predecessor. There is **no genuine concurrency** and **no optionality** in this simplified process — it is a fully linear chain. Therefore the entire process should be a single `SEQUENCE`, with **no `PARALLEL`** and **no `XOR`** operators.

---

# 2. Corrected Process Tree Model

```python
from pm4py.objects.process_tree.obj import Operator, ProcessTree

# Define activities
create_req      = ProcessTree(label="Create_Purchase_Requisition")
approve_req     = ProcessTree(label="Approve_Purchase_Requisition")
create_po       = ProcessTree(label="Create_Purchase_Order")
receive_goods   = ProcessTree(label="Receive_Goods")
receive_invoice = ProcessTree(label="Receive_Invoice")
match_invoice   = ProcessTree(label="Match_Invoice_to_PO")
pay_invoice     = ProcessTree(label="Pay_Invoice")
close_case      = ProcessTree(label="Close_Case")

# Corrected structure: one strict SEQUENCE enforcing the full causal order.
root = ProcessTree(operator=Operator.SEQUENCE, children=[
    create_req,
    approve_req,
    create_po,
    receive_goods,
    receive_invoice,
    match_invoice,
    pay_invoice,
    close_case
])

# Re-parent children so the tree is well-formed
for child in root.children:
    child.parent = root
```

This yields the single allowed trace:

```
Create_Purchase_Requisition 
Approve_Purchase_Requisition 
Create_Purchase_Order 
Receive_Goods 
Receive_Invoice 
Match_Invoice_to_PO 
Pay_Invoice 
Close_Case
```

> **Optional refinement (real-world variant):** In practice, *Receive_Goods* and *Receive_Invoice* may legitimately arrive concurrently (goods and invoice from different channels), but **both must precede** *Match_Invoice_to_PO*. If your domain allows that, you can model just those two in parallel while keeping everything else sequential:
> ```python
> goods_and_invoice = ProcessTree(operator=Operator.PARALLEL,
>                                 children=[receive_goods, receive_invoice])
> root = ProcessTree(operator=Operator.SEQUENCE, children=[
>     create_req, approve_req, create_po,
>     goods_and_invoice,
>     match_invoice, pay_invoice, close_case
> ])
> ```
> Use the fully-sequential version if the specification demands strict ordering; use this variant only if genuine concurrency exists.

---

# 3. Why the Updated Model Enforces Domain Logic

| Mechanism | Effect |
|-----------|--------|
| **Single `SEQUENCE` root** | A `SEQUENCE` operator forces its children to execute **strictly left-to-right**. No child can start before its predecessor completes. |
| **Removal of `PARALLEL`** | Eliminates all interleavings, so the model can no longer produce out-of-order traces like *Pay before Receive*. |
| **No `XOR`** | Every activity is mandatory and occurs exactly once — no step can be skipped, matching the "happy path" of the process. |

**Conformance guarantees produced by this structure:**

1. **Causal correctness** — *Create_Purchase_Order* always precedes *Receive_Goods*; *Receive_Invoice* always precedes *Match_Invoice_to_PO*; *Match_Invoice_to_PO* always precedes *Pay_Invoice*. The forbidden behaviors listed in Section 1 become **unreachable** in the model's language.
2. **Tightened (less underfitting) behavior** — The original model allowed `5! = 120` permutations of the middle block. The corrected model allows exactly **one** trace, dramatically reducing over-generalization and raising **precision** while preserving **fitness** for the standard process.
3. **Domain integrity** — Financial controls (you cannot pay an invoice that wasn't received and matched against a PO) are now structurally guaranteed by the model rather than left to chance.

In short, by replacing the permissive `PARALLEL` block with a strict `SEQUENCE`, the model transitions from *"any order is acceptable"* to *"only the prescribed Procure-to-Pay order is acceptable,"* fixing the underfitting problem.
