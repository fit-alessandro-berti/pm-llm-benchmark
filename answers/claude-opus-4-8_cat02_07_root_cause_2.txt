## 2. Root Cause Analysis

### A) Complexity is the strongest predictor

| Complexity | Cases | Avg Duration | Doc Requests |
|------------|-------|--------------|--------------|
| Low | 2001, 2004 | ~1.5 hours | 0 |
| Medium | 2002 | ~26 hours | 1 |
| High | 2003, 2005 | ~62 hours | 2–3 |

**Clear correlation:** Higher complexity  more "Request Additional Documents" loops  longer duration. Each documentation request introduces a long waiting period (often overnight or multi-day) while the insurer awaits the claimant's response.

### B) The "Request Additional Documents" rework loop is the primary bottleneck

Examining where time is actually lost:

- **Case 2005:** Three sequential doc requests span 04-01 11:30  04-04 10:00 (~3 days waiting). This single segment accounts for almost the entire delay.
- **Case 2003:** Two doc requests (04-01 11:00  04-02 16:00) consume ~29 hours before approval.
- **Case 2002:** One doc request (04-01 14:00  04-02 10:00) adds ~20 hours of overnight wait.

The repeated looping suggests **incomplete or piecemeal information gathering** — documents are requested one at a time rather than all at once.

### C) Region — weak/inconclusive signal

| Region | Cases | Notes |
|--------|-------|-------|
| A | 2001 (fast), 2003 (slow) | Mixed |
| B | 2002, 2004 (fast), 2005 (slow) | Mixed |

Region does **not** independently explain delays. Both regions have fast and slow cases; the difference tracks with complexity, not geography.

### D) Resource — secondary contributing signal

- **Manager_Bill** approves both high-complexity slow cases (2003, 2005), while **Manager_Ann** handles the faster Low/Medium cases. This may reflect either:
  - Bill being assigned the hardest cases (selection effect), OR
  - Bill being slower at approvals.
- **Adjuster_Lisa** handled Case 2005 with 3 doc requests — worth examining whether her documentation-gathering process is less thorough (causing repeated loops).

 **Caution:** With only 5 cases, resource-level conclusions are statistically weak. Bill/Lisa appearing on slow cases is confounded by complexity.

---

## 3. Explanations & Mitigation Recommendations

### Why these attributes drive delays

1. **Complexity drives rework:** High-complexity claims need more validation, triggering multiple documentation requests, each with a long external wait (customer turnaround).
2. **Sequential requesting:** Requesting documents one at a time (as in Case 2005) multiplies wait cycles instead of consolidating them.
3. **Hand-off / waiting time, not work time:** The actual activity durations (15–45 min) are short. The lost time is **idle waiting** between Request  Approve.

### Mitigation Suggestions

| Root Cause | Recommendation |
|------------|----------------|
| Multiple doc-request loops | **Consolidate document requests** — use a complexity-based checklist so all needed docs are requested in one batch upfront |
| Long external wait times | Implement **SLA timers and automated reminders** to claimants; set a documentation deadline |
| High-complexity bottleneck | Create a **specialized fast-track team** trained for complex claims to reduce per-loop time |
| Resource assignment effects | Conduct a **larger-sample analysis** to confirm whether Bill/Lisa are genuinely slower or just assigned harder cases; provide targeted coaching if confirmed |
| Overnight delays | Adjust **shift scheduling / queue prioritization** so high-complexity claims don't sit idle overnight |

### Key Takeaway
The dominant root cause is **claim complexity manifesting through repeated "Request Additional Documents" loops and associated customer wait time** — not region or resource. Reducing the *number of documentation iterations* (by requesting everything at once and enforcing response SLAs) would yield the largest improvement, especially for high-complexity claims like 2003 and 2005.

*Note: This analysis is based on only 5 cases. Conclusions about resources and regions should be validated against a larger dataset before operational changes.*
