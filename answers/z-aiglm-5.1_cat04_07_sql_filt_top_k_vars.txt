Here is the DuckDB SQL query that accomplishes your goal. It uses Common Table Expressions (CTEs) to break the problem down into logical steps: constructing the variants, finding the top K, and filtering the original event log.

```sql
-- Replace 3 with your desired K value
WITH case_variants AS (
    -- Step 1: Construct the ordered sequence of activities for each case_id
    SELECT 
        case_id,
        string_agg(activity, '->' ORDER BY timestamp) AS variant
    FROM event_log
    GROUP BY case_id
),
variant_counts AS (
    -- Step 2 & 3: Group cases by variant, count them, and find the Top K
    SELECT 
        variant,
        COUNT(case_id) AS case_count
    FROM case_variants
    GROUP BY variant
    ORDER BY case_count DESC
    LIMIT 3 -- <--- Set your K value here
),
top_k_cases AS (
    -- Step 4a: Identify the case_ids that belong to the Top K variants
    SELECT case_id
    FROM case_variants
    WHERE variant IN (SELECT variant FROM variant_counts)
)
-- Step 4b: Return all original events for only those filtered cases
SELECT 
    e.*
FROM event_log e
JOIN top_k_cases t 
  ON e.case_id = t.case_id
ORDER BY 
    e.case_id, 
    e.timestamp;
```

### How it works:
1. **`case_variants`**: We group the event log by `case_id` and use DuckDB's `string_agg()` with an `ORDER BY timestamp` clause. This guarantees the activities are concatenated in the exact chronological order they occurred, creating a readable variant string (e.g., `Start->Process->End`).
2. **`variant_counts`**: We take the constructed variants, group by the variant string, and count the number of cases for each. By ordering in descending order and applying `LIMIT K`, we isolate the most frequent process variants.
3. **`top_k_cases`**: We filter the `case_variants` CTE to only return the `case_id`s that match one of our Top K variants. 
4. **Final `SELECT`**: We join the original `event_log` table against the `top_k_cases` CTE. This efficiently filters out any events belonging to non-top-K cases while preserving the complete event history for the cases that remain.

### DuckDB-Specific Alternative (Using `LIST`):
If your activity names contain characters that might interfere with string delimiters (like `->`), DuckDB natively supports arrays/lists. You can use `list()` instead of `string_agg()` to group by an actual array object rather than a string:

```sql
WITH case_variants AS (
    SELECT 
        case_id,
        list(activity ORDER BY timestamp) AS variant -- Returns a LIST object instead of a string
    FROM event_log
    GROUP BY case_id
),
variant_counts AS (
    SELECT 
        variant,
        COUNT(case_id) AS case_count
    FROM case_variants
    GROUP BY variant
    ORDER BY case_count DESC
    LIMIT 3 
)
SELECT e.*
FROM event_log e
WHERE e.case_id IN (
    SELECT case_id 
    FROM case_variants 
    WHERE variant IN (SELECT variant FROM variant_counts)
)
ORDER BY e.case_id, e.timestamp;
```
*Note: DuckDB fully supports grouping and joining on `LIST` types, making this a robust alternative if string delimiters are a concern.*
