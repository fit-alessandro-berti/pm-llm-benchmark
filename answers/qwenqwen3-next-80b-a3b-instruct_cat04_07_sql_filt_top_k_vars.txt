To solve this problem in DuckDB SQL, we need to:

1. **Group by `case_id`** and aggregate the activities in timestamp order to form the **process variant** (i.e., the ordered sequence of activities).
2. **Count the number of cases** per variant.
3. **Identify the top K variants** by frequency.
4. **Filter the original `event_log`** to return only events from cases that belong to these top K variants.

We'll assume `K` is provided as a parameter. Since DuckDB doesn't support direct parameter binding in all contexts (like in pure SQL without external binding), we'll write the query using a **CTE-based approach** that can be easily adapted by substituting `K` with a numeric value (e.g., `K = 5`).

---

###  Final DuckDB SQL Query

```sql
WITH case_variants AS (
    SELECT
        case_id,
        STRING_AGG(activity, ' > ' ORDER BY timestamp) AS variant_sequence
    FROM event_log
    GROUP BY case_id
),
variant_counts AS (
    SELECT
        variant_sequence,
        COUNT(*) AS case_count
    FROM case_variants
    GROUP BY variant_sequence
),
top_k_variants AS (
    SELECT variant_sequence
    FROM variant_counts
    ORDER BY case_count DESC
    LIMIT K  -- Replace K with desired number, e.g., 5
),
filtered_cases AS (
    SELECT cv.case_id
    FROM case_variants cv
    INNER JOIN top_k_variants tkv
        ON cv.variant_sequence = tkv.variant_sequence
)
SELECT el.*
FROM event_log el
INNER JOIN filtered_cases fc
    ON el.case_id = fc.case_id
ORDER BY el.case_id, el.timestamp;
```

---

###  Explanation

| Step | Description |
|------|-------------|
| `case_variants` | Groups events by `case_id`, aggregates activities in `timestamp` order into a string sequence (e.g., `"A > B > C"`). |
| `variant_counts` | Counts how many `case_id`s have each unique variant sequence. |
| `top_k_variants` | Selects the top `K` most frequent variants (by case count), ordered descending. |
| `filtered_cases` | Finds all `case_id`s that belong to one of the top K variants. |
| Final `SELECT` | Joins back to original `event_log` to return **all events** from those filtered cases, ordered by `case_id` and `timestamp` to preserve sequence. |

---

###  Usage Note

To use this query, replace `K` with your desired number (e.g., `LIMIT 5` for top 5 variants). In DuckDB CLI or applications, you can bind `K` dynamically:

```sql
-- Example: Top 3 variants
LIMIT 3
```

If you're using a scripting language (Python, etc.) to generate the query, you can safely interpolate `K` as a variable.

###  Example Input/Output

**Input (`event_log`):**

| case_id | activity | timestamp           |
|---------|----------|---------------------|
| 1       | A        | 2023-01-01 10:00:00 |
| 1       | B        | 2023-01-01 11:00:00 |
| 2       | A        | 2023-01-01 09:00:00 |
| 2       | B        | 2023-01-01 10:00:00 |
| 3       | A        | 2023-01-01 08:00:00 |
| 3       | C        | 2023-01-01 09:00:00 |
| 4       | A        | 2023-01-01 10:00:00 |
| 4       | B        | 2023-01-01 11:00:00 |
| 5       | A        | 2023-01-01 07:00:00 |
| 5       | C        | 2023-01-01 08:00:00 |

**Variant Counts:**

- `"A > B"`  2 cases (case_id 1, 4)
- `"A > C"`  2 cases (case_id 3, 5)
- `"A > B"` and `"A > C"` are tied for top 2  if `K=2`, both are included.

**Output:** All events from `case_id` 1, 3, 4, 5 (excluding case 2 if it were different, but in this example case 2 is same as 1).

>  All events from cases belonging to top K variants are returned, ordered by case and timestamp.
