# Log Skeleton for the Order Fulfillment Process

Looking at the scenario, the process follows a largely sequential flow. Each activity occurs exactly once per case. The main physical-fulfillment chain is:

**RO  CA  RI  PI  QC  PO  GSL  D**

with an associated billing flow **IC  RP**.

```python
log_skeleton = {

    # ----------------------------------------------------------------
    # EQUIVALENCE: activities that always share the same # of occurrences
    # (every order that starts is fully handled and billed -> all run once)
    # ----------------------------------------------------------------
    'equivalence': {
        ('RO', 'D'),     # every received order is eventually dispatched
        ('CA', 'RI'),    # checking availability is matched by reserving items
        ('PI', 'QC'),    # everything picked is quality-checked
        ('PO', 'GSL'),   # every packed order gets a shipping label
        ('IC', 'RP'),    # every invoice issued is eventually paid/recorded
    },

    # ----------------------------------------------------------------
    # ALWAYS BEFORE: (X, Y) => if X occurs, Y must have occurred earlier
    # ----------------------------------------------------------------
    'always_before': {
        ('CA', 'RO'),    # availability checked only after order received
        ('RI', 'CA'),    # items reserved only after availability checked
        ('PI', 'RI'),    # picking after reservation
        ('QC', 'PI'),    # quality check after picking
        ('PO', 'QC'),    # packing after quality check
        ('GSL', 'PO'),   # label after packing
        ('D', 'GSL'),    # dispatch after label generated
        ('RP', 'IC'),    # payment recorded only after invoicing
    },

    # ----------------------------------------------------------------
    # ALWAYS AFTER: (X, Y) => if X occurs, Y is executed afterwards
    # ----------------------------------------------------------------
    'always_after': {
        ('RO', 'CA'),
        ('CA', 'RI'),
        ('RI', 'PI'),
        ('PI', 'QC'),
        ('QC', 'PO'),
        ('PO', 'GSL'),
        ('GSL', 'D'),
        ('IC', 'RP'),
    },

    # ----------------------------------------------------------------
    # NEVER TOGETHER: mutually exclusive activities
    # (none in this scenario — all activities co-exist in a normal case)
    # ----------------------------------------------------------------
    'never_together': set(),

    # ----------------------------------------------------------------
    # ACTIVITY OCCURRENCES: each activity happens exactly once per case
    # ----------------------------------------------------------------
    'activ_freq': {
        'RO':  {1},
        'CA':  {1},
        'RI':  {1},
        'PI':  {1},
        'QC':  {1},
        'PO':  {1},
        'GSL': {1},
        'D':   {1},
        'IC':  {1},
        'RP':  {1},
    },

    # ----------------------------------------------------------------
    # DIRECTLY-FOLLOWS: (X, Y) => Y occurs immediately after X
    # ----------------------------------------------------------------
    'directly_follows': {
        ('RO', 'CA'),
        ('CA', 'RI'),
        ('RI', 'PI'),
        ('PI', 'QC'),
        ('QC', 'PO'),
        ('PO', 'GSL'),
        ('GSL', 'D'),
        ('IC', 'RP'),
    },
}
```

## Modeling Rationale

| Constraint | Reasoning |
|------------|-----------|
| **Equivalence** | Because the process runs to completion once per order, paired activities (e.g., `RO/D`, `CA/RI`, `IC/RP`) always occur the same number of times. |
| **Always Before / Always After** | Encode the strict ordering of the fulfillment chain (an item can't be picked before it's reserved, can't be dispatched before a label exists, etc.) and the billing chain (`IC  RP`). |
| **Directly-Follows** | The physical handling steps proceed immediately one after another in the warehouse, plus the billing step `IC  RP`. |
| **Activity Occurrences** | A standard order triggers each task exactly once, so each maps to `{1}`. |
| **Never Together** | Empty — there are no mutually-exclusive activities in this single, happy-path process. |

> **Note:** The billing branch (`IC`, `RP`) is modeled as ordered relative to itself but kept independent of the exact interleaving with the physical chain (it may run in parallel), which is why no `always_before/after` links tie it rigidly into the picking/packing sequence. If your event log shows invoicing always occurring at a fixed point (e.g., right after `D`), you would add the corresponding `always_before`/`directly_follows` pairs.
