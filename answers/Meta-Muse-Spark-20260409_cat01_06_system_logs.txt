### **Explanation of Transformation Logic**

#### **1. Case Identification Logic**
I used a "document-centric" case definition because it produces the most coherent narrative for knowledge work. Each case represents a logical work item tied to a specific artifact:

- **CASE-001**: `Quarterly_Report.docx` - All interactions with this Word file are grouped, even though the user left and returned at 09:07:15. This captures the full lifecycle of that document task.
- **CASE-002**: `Document1.docx` - Grouped all create, edit, cross-reference, and close events for this draft. The switch to Excel and back at 09:06:00 is treated as a task interruption, not a new case, because the user returned to the same document.
- **CASE-003**: `Email about Annual Meeting` - Bounded by the SWITCH to Chrome through the "Send Email" action. Represents completing one email communication cycle.
- **CASE-004**: `Report_Draft.pdf` - Short review/annotation session in Acrobat. No save/close, so it ends when focus shifts to Excel.
- **CASE-005**: `Budget_2024.xlsx` - All Excel interactions with this specific budget file.

This approach prioritizes "what is the user working on" over "what app is active." Alternative would be session-based cases, but that would fragment related work on Document1.docx into two cases, hiding the fact that the Excel work was used to update the Word doc.

#### **2. Activity Naming Strategy**
Raw system actions like `FOCUS`, `TYPING`, `CLICK` are too granular for process analysis. I mapped them to business-meaningful steps using temporal and contextual rules:

| Raw Action Pattern | Standardized Activity | Rationale |
| --- | --- | --- |
| `FOCUS` + new window | `Open Document` / `Open Email Client` / `Review PDF Document` | Marks start of interaction with an artifact |
| `FOCUS` + returning to prior window | `Resume Document Editing` | Indicates task continuation after interruption |
| `TYPING` in Word | `Author Content`, `Cross-Reference Data`, `Author Executive Summary` | Inferred from `Keys` attribute + sequence context |
| `TYPING` in Excel | `Update Financial Data` | Domain-specific verb for spreadsheet work |
| `SAVE` | `Save Document` / `Save Spreadsheet` | Standardized across apps |
| `CLICK` + `Reply`/`Send` | `Reply to Email` / `Send Email` | Captures email workflow milestones |
| `HIGHLIGHT` in PDF | `Annotate Document` | Business term for review actions |
| `SCROLL` | `Review Email Content` / `Scan Document` | Indicates consumption, not just navigation |
| `CLOSE` | `Close Document` | Explicit completion marker |

#### **3. Temporal and Application Context Rules**
1. **Task switching ≠ new case**: `SWITCH` events were used to add `Task_Switch` attributes but didn't create cases unless the target was a new artifact. Example: 09:06:00 switch returns to CASE-002.
2. **Sequencing matters**: Multiple `TYPING` events in sequence within the same app/document were kept as separate events but named consistently as `Author Content` or `Update Financial Data`. This preserves duration data for performance analysis while avoiding noise.
3. **Completion inference**: Cases end with explicit `CLOSE` or `Send Email`. CASE-004 has no close, which is realistic for quick PDF reviews and useful for analysts to see.

#### **4. Why This Log is Analyst-Friendly**
- **Traceable**: You can follow CASE-002 and see the user drafted → saved → interrupted by email/PDF/Excel → returned to cross-reference budget data → saved → closed. That’s a complete work pattern.
- **Measurable**: Durations between `Open Document` and `Close Document` give cycle time. `Resume Document Editing` events flag rework/interruption cost.
- **Comparable**: Standardized names like `Save Document` work across Word and Excel, enabling cross-artifact process comparison in tools like Celonis or Disco.
- **Enrichable**: Added `Document / Context` and `Derived Attributes` columns to support filtering by artifact type, content type, or completion status without losing the raw trace.
