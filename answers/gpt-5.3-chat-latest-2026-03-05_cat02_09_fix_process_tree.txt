1. Issues with the current model

The problem is the PARALLEL block containing:

- Create_Purchase_Order  
- Receive_Goods  
- Receive_Invoice  
- Match_Invoice_to_PO  
- Pay_Invoice  

A parallel (AND) operator means all activities can occur in any order and even interleave. This allows invalid behaviors such as:

- Pay_Invoice before Receive_Invoice
- Pay_Invoice before Match_Invoice_to_PO
- Receive_Goods before Create_Purchase_Order
- Match_Invoice_to_PO before Receive_Invoice
- Create_Purchase_Order after receiving goods or invoice

In a real Procure-to-Pay process, these dependencies are strict:

Required order:

- Create_Purchase_Requisition  Approve_Purchase_Requisition  
- Approve_Purchase_Requisition  Create_Purchase_Order  
- Create_Purchase_Order  Receive_Goods  
- Create_Purchase_Order  Receive_Invoice  
- Receive_Goods + Receive_Invoice  Match_Invoice_to_PO  
- Match_Invoice_to_PO  Pay_Invoice  
- Pay_Invoice  Close_Case

Additionally, goods and invoice can arrive in any order, but both must occur before matching.

2. Corrected Process Tree Model

The fix is:

- Replace the large parallel block with a structured sequence
- Keep a small parallel block only for Receive_Goods and Receive_Invoice
- Force matching and payment after both are completed

Corrected pseudocode:

python
from pm4py.objects.process_tree.obj import Operator, ProcessTree

# Define activities
create_req = ProcessTree(label="Create_Purchase_Requisition")
approve_req = ProcessTree(label="Approve_Purchase_Requisition")
create_po = ProcessTree(label="Create_Purchase_Order")
receive_goods = ProcessTree(label="Receive_Goods")
receive_invoice = ProcessTree(label="Receive_Invoice")
match_invoice = ProcessTree(label="Match_Invoice_to_PO")
pay_invoice = ProcessTree(label="Pay_Invoice")
close_case = ProcessTree(label="Close_Case")

# Goods and invoice can arrive in any order but both must happen
delivery_and_invoice = ProcessTree(
    operator=Operator.PARALLEL,
    children=[
        receive_goods,
        receive_invoice
    ]
)

# Matching and payment must follow both
financial_processing = ProcessTree(
    operator=Operator.SEQUENCE,
    children=[
        delivery_and_invoice,
        match_invoice,
        pay_invoice
    ]
)

# Full corrected process
root = ProcessTree(
    operator=Operator.SEQUENCE,
    children=[
        create_req,
        approve_req,
        create_po,
        financial_processing,
        close_case
    ]
)

3. Why this corrected model works

The revised tree enforces the business logic through sequencing and controlled parallelism.

Strict sequential control

The sequence operator enforces the core lifecycle:

Create_Purchase_Requisition  
 Approve_Purchase_Requisition  
 Create_Purchase_Order  

This guarantees that procurement cannot start before approval and that downstream steps cannot happen before a purchase order exists.

Controlled concurrency

A parallel block is used only for:

Receive_Goods  
Receive_Invoice

This reflects reality: the invoice may arrive before or after the goods. The AND operator requires both events to occur before the process can continue.

Dependency enforcement

Because Match_Invoice_to_PO is placed after the parallel block, it cannot occur until:

- goods are received  
- invoice is received  

This correctly models the threeway matching logic.

Payment control

Pay_Invoice is placed after Match_Invoice_to_PO, ensuring payment cannot occur before verification.

Case completion

Close_Case is the final step in the outer sequence, ensuring the case only closes after payment.

Result

The updated model prevents:

- payment before invoice receipt
- matching before goods or invoice exist
- goods or invoice before a purchase order
- out-of-order financial processing

while still allowing the only realistic concurrency: goods and invoice arrival.
