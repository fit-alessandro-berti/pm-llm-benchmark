# 1. Analyzing Resource Behavior and Assignment Patterns

## 1.1 Prepare the event log for resource-focused process mining

Before analysis, I would structure the log at three levels:

- **Case level:** Ticket ID, priority, category, channel, SLA target, final outcome, closure code, number of assignments/reassignments/escalations, total resolution time.
- **Event level:** Activity, timestamp, resource, tier, timestamp type, queue wait, work duration, handover-from/handover-to, assignment reason, required skill.
- **Resource level:** Agent ID, tier, documented skills, shifts/availability, team, tenure, historical performance.

I would derive additional fields such as:

- **Assignment delay:** time from `Assign Lx` to `Work Lx Start`
- **Work duration:** time from `Work Start` to `Work End`
- **Queue delay after escalation:** time from `Escalate Lx` to `Work Lx Start`
- **Reassignment count per case**
- **Skill match flag:** whether assigned agent had the required skill
- **Overqualification flag:** assigned agent had significantly higher skill level than needed
- **Touches per ticket:** number of distinct agents involved
- **Tier bounce:** e.g., L1  L2  L2 reassignment, or L2  L1  L2
- **SLA breach flag** and breach magnitude

This enables both classic process mining and resource mining.

---

## 1.2 Metrics for individual agents and tiers

### A. Workload distribution
To understand whether work is balanced:

- Tickets assigned per agent/day/week/month
- Active work time per agent
- Queue size inherited by each agent
- Share of P1/P2/P3/P4 tickets per agent
- Share of complex categories/required skills per agent
- Distinct ticket types handled by each agent
- Number of concurrent tickets handled

**Purpose:** Identify overload, underutilization, unfair assignment patterns, and hidden specialists taking disproportionate load.

### B. Processing and waiting times
Per agent and tier:

- Average and median:
  - first response time
  - assignment-to-start time
  - active handling time
  - escalation delay
  - resolution time
- Percentiles, especially P80/P90/P95
- Time by priority/category/required skill
- Time by channel (phone, email, portal)

**Purpose:** Distinguish whether delays come from queueing, slow work, or misassignment.

### C. L1 effectiveness
For L1 specifically:

- **First-call / first-touch resolution rate**
- Escalation rate by category and required skill
- Reopen rate after L1 closure
- Percentage of tickets correctly categorized at intake
- Share of tickets solved without reassignment
- Average troubleshooting time before escalation

**Purpose:** Determine whether L1 is resolving enough, escalating too much, or escalating too late.

### D. Agent assignment accuracy
Metrics include:

- % of tickets assigned to an agent with matching required skill
- % of cases later reassigned due to skill mismatch
- % of same-tier reassignment vs cross-tier escalation
- Number of handoffs before resolution
- Reassignment rate by dispatcher, by source agent, by category

**Purpose:** Surface whether the assignment logic is choosing the right person initially.

### E. Tier-level performance
Compare L1/L2/L3 on:

- inbound volume
- assignment delays
- active handling times
- backlog
- escalation in/out rates
- reassignment rates
- SLA attainment
- skill coverage vs actual demand

**Purpose:** See whether bottlenecks are tier-specific or skill-specific.

---

## 1.3 Process mining techniques to reveal actual assignment behavior

### A. Resource interaction / handover analysis
Using handover-of-work mining, I would build networks showing:

- which agents frequently hand work to which others
- which tiers hand over to which tiers
- common reassignment loops
- high-friction paths like:
  - L1 Agent A  L2 Agent B  L2 Agent C
  - Dispatcher  wrong L2 specialist  corrected assignment

This reveals the **actual collaboration and routing structure**, which often differs from formal policy.

### B. Social network analysis (SNA)
Useful SNA views:

- **Handover network:** who transfers work to whom
- **Working-together network:** which agents repeatedly appear on same tickets
- **Subcontracting network:** one agent repeatedly routes difficult work to certain specialists
- **Betweenness centrality:** agents/dispatchers acting as routing hubs
- **In-degree concentration:** specialists receiving disproportionate escalations

**Insight:** If a few L2 agents are central bottlenecks, assignment logic may be over-concentrating work.

### C. Role discovery
Rather than assuming official roles equal actual roles, role mining can infer clusters of behavior based on:

- activities executed
- ticket types handled
- skills actually used
- escalation patterns

This can reveal, for example:

- some L1 agents acting as quasi-L2 for certain software categories
- some L2 agents mostly performing routine work
- hidden specialist groups around CRM, firewall, IAM, SQL

This helps redesign work allocation around **actual capability**, not org chart labels.

### D. Conformance checking against intended assignment logic
The intended logic appears to be roughly:

- initial L1 handling
- round-robin assignment within tier
- manual escalation by complexity/skill need

I would compare event log behavior against this design:

- Are tickets really assigned round-robin?
- Are priority rules followed?
- Are required skills considered at initial assignment?
- Are escalations routed to proper skill pools?
- Are some categories bypassing L1 in practice?
- How often are tickets reassigned after “successful” assignment?

This provides a direct gap between **designed process** and **executed process**.

---

## 1.4 Analyze skill utilization across the agent pool

I would create a **skill-demand vs skill-supply matrix**.

### Demand side
For each required skill:

- number of tickets
- priority mix
- arrival patterns by hour/day/month
- average resolution time
- SLA breach rate
- reassignment rate
- share initially assigned correctly

### Supply side
For each skill:

- number of agents with that skill
- tier location of those agents
- effective available capacity
- utilization rate
- queue load
- performance outcomes

### Skill utilization questions
This analysis answers:

- Are firewall, SQL, CRM, IAM skills in shortage?
- Are some specialists overloaded due to rare skills?
- Are highly skilled L2/L3 agents spending time on low-complexity tasks?
- Could some recurring specialist tasks be shifted to L1 through training or runbooks?
- Are documented skill profiles aligned with observed successful resolution behavior?

A very useful metric is:

- **Skill utilization efficiency**
  - % of tickets requiring advanced skill actually worked by matching specialists
  - % of specialist time spent on tickets not requiring specialist skill
  - % of tickets handled successfully by lower-tier staff despite advanced required-skill labels

This is crucial for detecting overuse of expensive resources.

# 2. Identifying Resource-Related Bottlenecks and Issues

## 2.1 Pinpoint bottlenecks caused by scarce skills

Using queue and waiting-time analysis by required skill, I would identify skills where:

- assignment-to-start time is consistently high
- backlog accumulates before a small subset of agents
- SLA breaches cluster around the same required skill
- escalations wait longer than active work itself

Example indicators:

- `Networking-Firewall` tickets have median queue delay of 3.2 hours vs 25 minutes overall
- only 2 L2 agents handle 78% of firewall cases
- these cases breach P2 SLA 31% of the time

This strongly indicates a skill-capacity bottleneck.

---

## 2.2 Detect delays from reassignments and escalations

For each reassignment/escalation type, I would measure:

- average added waiting time
- average added total cycle time
- breach probability before vs after reassignment
- whether reassignment was same-tier or cross-tier
- whether reassignment was caused by skill mismatch, capacity issue, or category correction

Important cuts:

- L1  L2 direct escalation
- L2  L2 reassignment
- L2  L3 escalation
- repeated same-tier bouncing
- reassignment within first 30 minutes after assignment

Typical quantifications:

- Average delay per reassignment
- % of tickets with >2 assignments
- SLA breach likelihood by number of handoffs
- extra queue time created by “wrong first assignment”

A strong practical metric is:
- **Incremental delay per handoff**
  - e.g., each additional handoff adds 1.8 hours median resolution time

---

## 2.3 Assess incorrect initial assignments

I would define “incorrect initial assignment” using operational proxies such as:

- reassigned before any meaningful work
- reassigned due to “wrong skill” note
- assigned to agent lacking required skill
- category or required skill changed after first touch
- L1 work duration very short followed by escalation for obvious specialist need

Then I would measure:

- % of tickets incorrectly assigned initially
- by channel, category, priority, dispatcher, shift, and ticket source
- impact on SLA, total time, and customer wait

This can reveal whether portal tickets are miscategorized more often than phone tickets, or whether certain dispatch patterns are error-prone.

---

## 2.4 Identify overloaded or underperforming agents/teams

This needs care: process mining should distinguish **performance issues** from **assignment context**.

I would evaluate agents on risk-adjusted measures:

- workload intensity
- complexity-adjusted handling time
- queue load inherited
- reassignment rate after their assignment
- success rate for categories they usually handle
- SLA attainment for tickets in their care
- after controlling for priority/skill complexity

This avoids unfairly flagging specialists who receive the hardest tickets.

Indicators of overload:

- sustained high queue
- rising assignment-to-start times
- increased reassignment out
- longer active work times late in shifts
- disproportionate case count in scarce skills

Indicators of underutilization:

- low assignment volume
- low occupancy despite nearby SLA breaches in same category
- documented skills rarely used
- narrow routing concentration excluding them

---

## 2.5 Correlate assignment patterns with SLA breaches

I would perform breach analysis across assignment variables:

- number of handoffs
- skill-match quality
- agent tier of first meaningful work
- queue delay after escalation
- time spent in wrong tier
- dispatcher vs self-assigned vs automatic assignment
- specific agent/team/skill pool involved

Useful statistical methods:

- logistic regression for SLA breach probability
- survival analysis for time to resolution
- decision trees to identify combinations leading to breach
- SHAP-style feature importance if using predictive models

Typical outputs:

- tickets with skill mismatch at first assignment are 2.4x more likely to breach
- each same-tier reassignment adds 18 percentage points to P3 breach risk
- P2 firewall tickets assigned outside the top-matching skill pool breach 35% more often

These quantified links make the case for redesign.

# 3. Root Cause Analysis for Assignment Inefficiencies

## 3.1 Likely root causes

### A. Round-robin assignment ignores skill and workload
This is probably the biggest structural issue. Round-robin can distribute volume fairly but not capability fit. In a tiered support environment, this often causes:

- assignment to available but unsuitable agents
- later correction through reassignment
- specialists receiving work only after delay
- uneven actual effort despite equal ticket counts

### B. Skill profiles may be incomplete or static
Documented skills often lag reality. Process mining may reveal:

- agents resolving categories they are not officially tagged for
- officially skilled agents rarely handling that skill successfully
- skill depth differences not captured
- no distinction between proficiency levels

### C. Initial categorization / skill requirement identification may be weak
If tickets are poorly classified at intake, even perfect routing rules will fail. Common signs:

- required skill changes after first touch
- portal tickets miscategorized more often
- same symptom routed differently by channel
- free-text descriptions contain the true signal, but structured fields do not

### D. No real-time workload visibility in assignment decisions
Manual dispatch or simple round-robin often ignores:

- current queue depth
- who is already in active work
- who is absent or near shift end
- who is handling many urgent tickets

This creates overload on visible specialists while other capable agents remain idle.

### E. L1 training and empowerment may be insufficient
If L1 agents escalate too frequently or too late:

- specialists get flooded with routine work
- customers wait longer for solvable issues
- L1 may perform unnecessary troubleshooting on clearly specialist cases
- escalations may be based on confidence rather than data-driven criteria

---

## 3.2 Using variant analysis

Variant analysis would compare:

### Smooth cases
Cases with:
- one assignment
- no reassignment
- fast resolution
- SLA met

versus

### Problematic cases
Cases with:
- multiple handoffs
- category changes
- same-tier reassignment
- prolonged queueing
- SLA breach

I would compare these groups on:

- ticket category and required skill
- creation channel
- time of day/day of week
- initial agent/tier
- initial categorization quality
- first assignment skill match
- dispatcher/agent involved
- wording patterns in ticket description

This can isolate what differentiates smooth routing from problematic routing.

Example findings:
- smooth CRM incidents are initially tagged correctly and sent directly to a small skill pool
- problematic CRM incidents start as generic “Software-App” and bounce between generalists before reaching DB-SQL support

---

## 3.3 Using decision mining

Decision mining can infer what actually drives routing choices at key points such as:

- initial assignment
- escalation from L1
- reassignment within L2
- escalation to L3

Input features could include:

- priority
- category
- required skill
- channel
- free-text features from description
- current backlog
- agent availability
- prior failed action
- customer group or service affected

This reveals:

- which variables are currently influencing decisions
- whether decisions are consistent
- whether irrelevant variables dominate
- what factors best predict successful assignment

Example:
- current L2 assignment may be driven mostly by recent round-robin position and not by required skill
- successful assignments may be best predicted by matching category + required skill + current workload + historical success rate

This directly informs redesign of routing logic.

# 4. Developing Data-Driven Resource Assignment Strategies

Below are four concrete strategies.

---

## Strategy 1: Skill-based routing with proficiency weighting

### Issue addressed
Wrong initial assignments, excessive reassignments, specialist bottlenecks hidden inside tier-based routing.

### How it leverages process mining insights
From the mining analysis, we know:

- which skills are actually required for which ticket types
- which agents successfully resolve which categories/skills
- which assignments lead to minimal handoffs and good SLA outcomes
- where official and actual skill profiles differ

### How it would work
Replace pure tier/round-robin routing with a routing score such as:

**Assignment score = skill match × proficiency × historical success × SLA urgency**

A ticket would be routed first to agents with the strongest demonstrated fit for the required skill/category, not merely the next person in the tier.

Potential refinements:
- mandatory exact match for scarce specialist skills
- proficiency levels (basic, intermediate, advanced)
- reserve top specialists for high-complexity/high-priority cases
- route low-complexity tasks to less senior but qualified agents

### Data required
- historical ticket-to-skill mappings
- agent skills and proficiency levels
- actual success/resolution history by agent and category
- reassignment history
- ticket metadata and text-derived classification features

### Expected benefits
- fewer wrong assignments
- lower reassignment rate
- shorter time to first effective work
- better specialist utilization
- improved SLA performance for P2/P3 tickets

---

## Strategy 2: Workload-aware dynamic assignment

### Issue addressed
Uneven workload, overloaded agents, long queue delays after assignment, avoidable SLA breaches due to waiting.

### How it leverages process mining insights
The event log reveals:

- actual queue times by agent/team
- variation in occupancy and backlog over time
- where assignment-to-start delay is the real bottleneck
- which skills have capacity pressure at which hours/days

### How it would work
Use real-time assignment that considers:

- current open tickets assigned but not started
- active workload
- agent availability/shift status
- ticket priority and remaining SLA time
- skill suitability
- concurrency thresholds

Instead of “next in round-robin,” the engine chooses the **best-fit available** agent under load balancing rules.

Possible routing score:

**Final score = skill fit + availability + low queue load + urgency compatibility**

It can also include:
- aging rules to prioritize tickets near SLA breach
- caps to prevent overloading top performers
- delayed assignment to reserve scarce specialists for matching work

### Data required
- real-time queue and assignment data
- shift schedules and status
- active workload metrics
- remaining SLA clocks
- skill fit and proficiency

### Expected benefits
- better workload distribution
- reduced assignment-to-start time
- fewer overloaded specialists
- improved queue stability
- lower breach rate due to waiting rather than work time

---

## Strategy 3: Predictive assignment and early skill inference

### Issue addressed
Poor initial categorization, hidden complexity, delayed routing to correct specialist pool.

### How it leverages process mining insights
Mining shows which combinations of:
- category
- priority
- channel
- keywords in description
- customer/service context

predict:
- required skill
- escalation likelihood
- probability of reassignment
- likely resolution tier

### How it would work
Build models that, at ticket creation, predict:

- likely required skill
- likely resolution tier
- complexity score
- probability L1 can resolve

Then:
- directly route obvious specialist tickets to correct L2/L3 queue
- send likely L1-resolvable tickets to stronger L1 agents or self-service/runbooks
- flag ambiguous tickets for enhanced triage instead of blind assignment

For example:
- “VPN blocked after firewall policy update” may predict `Networking-Firewall`
- “CRM timeout after report export” may predict `App-CRM + DB-SQL`

### Data required
- historical event log
- structured ticket fields
- ticket text/notes
- resolved skill and final resolver
- escalation and reassignment outcomes

### Expected benefits
- improved first assignment accuracy
- reduced same-tier rework
- faster path to effective resolver
- better SLA compliance, especially for P2/P3
- more consistent triage across channels

---

## Strategy 4: Data-driven L1 triage and escalation redesign

### Issue addressed
L2/L3 specialists doing routine work; excessive or delayed escalation from L1.

### How it leverages process mining insights
Process mining identifies:

- ticket types L1 resolves successfully
- ticket types L1 almost always escalates
- agents at L1 with superior outcomes in specific categories
- time wasted in low-value L1 handling before inevitable escalation

### How it would work
Create evidence-based triage rules:

- **Direct-to-specialist rules** for ticket patterns with near-certain escalation
- **L1 resolution playbooks** for ticket types with high L1 success rates
- **Targeted L1 upskilling** for categories where L1 is close to success but escalates often
- **Escalation thresholds** based on elapsed troubleshooting time or failed checks

Examples:
- If historical data shows 90% of `Database-SQL` incidents escalate from L1 after 20+ minutes, route directly to L2 SQL for qualifying patterns
- If some `Access Management` tickets are routinely solved by trained L1 agents, route them to a specialized L1 micro-pool

### Data required
- L1 first-touch resolution rates by category/skill
- troubleshooting duration before escalation
- knowledge article usage
- category-level resolution success by agent
- escalation reasons

### Expected benefits
- fewer unnecessary specialist interruptions
- less wasted handling time at L1
- higher L1 productivity
- reduced total resolution time
- better role clarity across tiers

# 5. Simulation, Implementation, and Monitoring

## 5.1 Using business process simulation before implementation

Simulation is highly valuable because assignment changes affect queueing dynamics in non-obvious ways.

### Build a simulation model using mined process characteristics
From the event log, estimate:

- arrival rates by hour/day, category, priority
- branching probabilities:
  - L1 resolution
  - L1 escalation
  - L2 reassignment
  - L3 escalation
- service-time distributions by activity, skill, tier
- resource pools and capacities
- shift calendars and availability
- queue disciplines and SLA targets

### Simulate baseline
First simulate the current state to validate that the model reproduces:

- current SLA rates
- average resolution time
- reassignment frequency
- queue lengths by tier/skill

### Test proposed strategies
Run scenarios such as:

1. skill-based routing only
2. workload-aware routing only
3. predictive triage only
4. combined strategy
5. selective L1 upskilling + direct specialist routing
6. adding one firewall-skilled agent vs changing assignment rules only

### Evaluate outputs
For each scenario, compare:

- SLA attainment by priority
- median and P90 resolution time
- assignment-to-start delay
- number of handoffs per case
- specialist utilization
- backlog size by skill
- L1 first-contact resolution
- cost per resolved ticket

This allows management to choose the best intervention based on both performance and staffing cost.

---

## 5.2 Implementation plan

A practical rollout would be phased.

### Phase 1: Diagnostic and design
- Clean and enrich event log
- Validate skill taxonomy
- Build baseline dashboards
- Confirm root causes with team leads

### Phase 2: Pilot
- Choose 1–2 categories with high reassignment/SLA pain, e.g. Network and Software-App
- Pilot skill-based + workload-aware routing
- Add reason codes for reassignment and escalation
- Update agent skill profiles

### Phase 3: Expand and automate
- Introduce predictive triage for selected categories
- Formalize direct-routing rules
- Add L1 playbooks and targeted training
- Integrate routing engine with service desk platform

### Phase 4: Continuous optimization
- Periodically refresh routing models
- recalibrate skill scores from actual outcomes
- detect drift in demand patterns
- refine staffing and training plans

---

## 5.3 Post-implementation monitoring with process mining dashboards

I would build dashboards around both process and resource views.

### Core resource-related KPIs
- SLA attainment by priority/category/skill
- first-touch and first-call resolution rate
- average and P90 resolution time
- assignment-to-start time
- escalation rate from L1
- reassignment rate
- average handoffs per ticket
- % correct first assignment
- % tickets routed to skill-matched agent
- specialist time spent on low-complexity work
- workload balance index across agents/teams
- queue length by skill pool
- utilization by agent and by skill

### Process views to track continuously
#### A. Process map with performance overlays
Show where delays concentrate:
- after assignment
- after escalation
- during same-tier reassignments

#### B. Handover network
Track whether handoffs reduce after redesign and whether bottleneck hubs disappear.

#### C. Variant dashboard
Compare top variants:
- one-touch resolved
- L1L2 smooth path
- reassign-heavy path
- breach-heavy path

#### D. Skill heatmaps
View demand, capacity, queueing, and SLA outcomes by skill.

#### E. Conformance view
Measure adherence to the new routing rules:
- was the recommended skill pool used?
- was the direct-route rule followed?
- was an avoidable reassignment created?

### Review cadence
- Daily: operational queue and SLA dashboard
- Weekly: skill bottleneck and workload balancing review
- Monthly: routing effectiveness, conformance, and retraining needs
- Quarterly: update skill profiles, routing rules, and simulation assumptions

# Conclusion

Using process mining, TechSolve can move from a largely tier-and-round-robin assignment model to a **fact-based resource assignment model** that aligns tickets with the right skill, at the right time, under the right workload conditions.

The most important insights will likely come from:
- measuring first-assignment accuracy
- quantifying queue delays by skill
- mapping handoffs and reassignments
- comparing actual vs intended routing behavior
- identifying where specialist capacity is wasted on low-value work

The strongest improvement levers are:
1. **skill-based routing**
2. **workload-aware dynamic assignment**
3. **predictive triage and early skill inference**
4. **data-driven redesign of L1 escalation rules**

Combined with simulation before rollout and continuous process-mining monitoring afterward, these changes can reduce reassignment loops, improve SLA compliance for P2/P3 tickets, and make far better use of L1, L2, and L3 capability.
