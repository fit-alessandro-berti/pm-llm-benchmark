### 1. Process Discovery and Conformance Checking

**Data Preprocessing & Integration**
The raw data spans four systems with different granularities and identifiers. To build a process-mining-ready event log, I would implement a multi-step pipeline:
- **Case Definition & Hierarchy:** Traditional process mining assumes a single `Case ID`. Given the multi-entity nature of the data, I would adopt **Object-Centric Process Mining (OCPM)** or a hierarchical case structure: `Vehicle-Day` as the primary case, with `Package ID`, `Driver ID`, and `Maintenance Event` as secondary objects linked via relationships.
- **Temporal & Spatial Alignment:** GPS pings (e.g., every 10-30s) are too granular for direct mining. I would apply **activity abstraction** using speed and dwell-time thresholds:
  - `Speed > 15 km/h` + continuous movement  `Traveling`
  - `Speed < 5 km/h` for >3 mins between stops  `Idling/Traffic Delay`
  - `Speed = 0` + geofence match to customer address  `At Customer`
- **Event Merging & Standardization:** Scanner events (`Arrive`, `Success`, `Failed`, `Depart`) are merged with GPS-derived `Traveling/Idling` states. Dispatch and maintenance events are mapped as control-flow and resource attributes. Missing timestamps or GPS drift are handled via interpolation or spatial buffering (e.g., snapping to nearest road segment).
- **Challenges:** 
  - *Scanner non-compliance* (drivers forgetting to scan on arrival/departure) requires GPS fallback logic.
  - *Parallelism* (multiple packages delivered at one stop, or multiple vehicles in the same area) complicates case boundaries.
  - *Data quality inconsistencies* across legacy systems (e.g., mismatched time zones, unsynced clocks) require rigorous validation and normalization.

**Process Discovery**
Using algorithms like the **Inductive Miner** or **Heuristics Miner** (chosen for robustness to noise and high-frequency events), I would generate:
- A **Directly-Follows Graph (DFG)** at the macro level: `Start Shift  Route Assigned  Depart Depot  [Travel/Stop Loop]  Arrive Depot  End Shift`
- A **micro-level model** for stop execution: `Arrive Customer  [Wait/Access]  Delivery Success/Failed  Depart Customer`
- **Object-Centric Discovery** to visualize interactions: how package delivery states, driver actions, and vehicle telemetry co-occur. This reveals branching paths (e.g., successful delivery vs. failed attempt  re-route or return-to-depot).

**Conformance Checking**
The dispatch system provides the *prescriptive* process (planned sequence, time windows, route topology). I would use **alignment-based conformance checking** to map actual executions to the planned model:
- **Sequence Deviations:** Stops visited out of planned order (detected via token-based replay mismatches).
- **Timing Deviations:** Early/late arrivals vs. customer time windows, or extended travel/service times beyond dispatch estimates.
- **Unplanned Events:** Maintenance stops, traffic idling, driver breaks, or route diversions not in the plan.
- **Skipped/Added Stops:** Deliveries omitted or ad-hoc pickups added mid-route.
Conformance metrics (fitness, precision, alignment cost) would quantify the "distance" between planned and actual operations, highlighting where routing assumptions break down in reality.

### 2. Performance Analysis and Bottleneck Identification

**Key Performance Indicators (KPIs) & Calculation**
| KPI | Calculation from Event Log |
|-----|----------------------------|
| **On-Time Delivery Rate (OTDR)** | `% of "Delivery Success" events where arrival timestamp  customer time window end` |
| **Average Time per Delivery Stop** | `AVG(Depart Timestamp - Arrive Timestamp)` per stop |
| **Travel vs. Service Time Ratio** | `SUM(Travel Durations) / SUM(Service Durations)` where travel = `Depart  Next Arrive`, service = `Arrive  Depart` |
| **Fuel Consumption Proxy per km/package** | `Distance traveled (from GPS) ﾗ Idle Time Penalty Factor / Packages Delivered` (or integrate telematics fuel flow if available) |
| **Vehicle Utilization Rate** | `Active Driving Time / Total Shift Time` |
| **Traffic Delay Frequency/Duration** | `Count/Duration of "Low Speed/Idle" events >3 mins between planned stops` |
| **Failed Delivery Rate** | `# "Delivery Failed" / (# "Delivery Success" + # "Delivery Failed")` |

**Bottleneck Identification Techniques**
- **Performance-Overlay DFG:** Color-code arcs with median/90th percentile durations. Thick, red arcs indicate high-frequency delays (e.g., `Traveling  Arrive Customer` during 08:00-09:30).
- **Throughput Time Decomposition:** Break total shift time into travel, service, idle, and deviation time. Identify which component dominates delays.
- **Resource & Spatial Analysis:** Cross-reference bottlenecks with `Driver ID`, `Vehicle ID`, `Location`, and `Time of Day`. Use heatmaps to identify traffic hotspots, depot congestion zones, or specific routes with chronic delays.
- **Critical Path & Queueing Approximation:** Identify the longest sequential path in the DFG. If multiple stops cluster in one zone, apply queueing models to estimate parking/search time bottlenecks.
- **Impact Quantification:** Calculate the **contribution to delay** (e.g., "Traffic idling accounts for 18% of total shift time, adding 1.2 hours daily per vehicle, translating to ~15% OTDR loss").

### 3. Root Cause Analysis for Inefficiencies

Beyond locating delays, process mining enables causal validation through advanced analytical techniques:

| Suspected Root Cause | Process Mining Validation Approach |
|----------------------|------------------------------------|
| **Suboptimal/Static Routing** | Compare planned vs. actual sequences via conformance deviation costs. If out-of-order stops consistently reduce backtracking distance, the original routing algorithm is flawed. |
| **Inaccurate Travel/Service Time Estimates** | Decision mining on `Travel Duration` and `Service Duration` attributes vs. dispatch estimates. High residual errors in specific zones or times indicate systemic estimation bias. |
| **Traffic Congestion Patterns** | Correlate `Low Speed Detected` events with time-of-day, day-of-week, and location clusters. Use temporal process mining to identify recurring congestion windows. |
| **High Service Time Variability** | Cluster dwell times (`Arrive  Depart`) by location type, package size, and driver. High variance suggests inconsistent access procedures, customer availability, or parking constraints. |
| **Vehicle Breakdowns/Maintenance** | Align `Engine Warning Light` or `Unscheduled Stop` events with prior telemetry (harsh braking, excessive idle, high RPM). Survival analysis on `Vehicle-Day` cases can predict time-to-failure. |
| **Driver Behavior/Skill Differences** | Variant analysis: Compare top vs. bottom quartile drivers. Look for differences in idle-to-travel ratios, scanning compliance, route adherence, and stop sequencing. |
| **Failed Deliveries & Re-delivery Costs** | Analyze `Delivery Failed` variants. Correlate with time windows, location type, and driver notes. High failure rates in specific windows indicate mismatched customer expectations. |

**Analytical Tools:** 
- **Decision Tree Mining** on process attributes to predict delays or failures.
- **Multi-Perspective Conformance** to see how driver, vehicle, and package objects interact during bottlenecks.
- **Causal Process Mining** to test hypotheses (e.g., "Does idling >5 mins before a stop increase failure rate?").
