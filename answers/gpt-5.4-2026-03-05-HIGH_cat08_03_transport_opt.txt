## 1. Process Discovery and Conformance Checking

### 1.1 Preprocess and integrate the data into a usable event log

For this use case, I would **not rely on a single simple case ID only**. A “vehicle-day” case is useful, but delivery operations are inherently multi-object:

- **Vehicle-day / route execution**: how a van’s day unfolded
- **Package**: whether the package was delivered, failed, reattempted
- **Vehicle**: maintenance and reliability history
- **Driver**: resource behavior and performance
- **Stop / customer location**: dwell time, parking difficulty, failure patterns

So the best design is an **object-centric event log (OCEL)**, or at minimum several synchronized logs:

1. **Route execution log**: one case per vehicle-shift/day  
2. **Package lifecycle log**: one case per package  
3. **Vehicle reliability log**: one case per vehicle over time  

### 1.2 Build a canonical event model

Each event should be normalized into a common structure such as:

- `timestamp`
- `activity`
- `source system` (GPS/scanner/dispatch/maintenance)
- `vehicle_id`
- `driver_id`
- `route_id` or `vehicle-day`
- `package_id` (if applicable)
- `stop_id/customer_id` (if applicable)
- `location`
- `planned stop sequence`
- `planned ETA / customer time window`
- `speed / status`
- `exception flag`
- `event confidence`

### 1.3 Convert raw GPS into process-relevant events

Raw GPS pings are too granular for direct mining. They must be **abstracted into meaningful logistics events**, for example:

- Depart Depot
- Arrive Stop Area
- Parking / curb dwell start
- Arrive Customer
- Depart Customer
- Traffic Delay Episode
- Unplanned Stop
- Route Deviation
- Return to Depot
- Ignition Off / End Shift

This requires:

- **Map-matching** GPS points to the road network
- **Geofencing** depot and customer locations
- **Episode detection** from continuous pings:
  - low-speed episode
  - stationary period
  - stop outside planned locations
  - repeated circling near customer
- **Trip segmentation** between stops

### 1.4 Join dispatch and maintenance data

Dispatch data adds the **reference plan**:

- planned route sequence
- assigned packages
- vehicle capacity
- driver assignment
- customer delivery windows
- planned route start/end

Maintenance logs add:

- scheduled downtime
- unscheduled repair intervals
- overlap between route execution and vehicle issues

### 1.5 Main preprocessing challenges

Key challenges will be:

#### a) Timestamp synchronization
Different systems often have clock drift or different time granularities. A 2–5 minute mismatch can distort arrival, service, or delay analysis.

#### b) GPS noise and urban canyon effects
Dense urban areas can cause location jumps, false stops, or inaccurate route deviations.

#### c) Mixed granularity
GPS is continuous, scanner events are discrete, dispatch is planned state, maintenance is interval-based.

#### d) Missing or inconsistent scans
Drivers may forget to scan “Arrive Customer” or “Depart Customer.” Missing milestones may need to be inferred from GPS.

#### e) Multiple packages at one stop
A physical customer stop may involve several packages. Stop-level and package-level views must be separated.

#### f) Route plan changes during the day
If dispatchers reassign or resequence stops mid-route, conformance should compare actual execution to the **latest approved plan**, not only the morning plan.

#### g) Spaghetti risk
If every GPS-derived micro-event is included, discovery becomes unreadable. The event log must be **hierarchically abstracted**.

---

### 1.6 Process discovery: reveal the actual delivery process

I would discover the process at **three levels**.

#### Level 1: Route-day lifecycle
Typical discovered model:

Start Shift  
 Route Assigned  
 Load / Scan at Depot  
 Depart Depot  
 repeat:
- travel
- delay/traffic episode (optional)
- arrive customer
- delivery success or failed attempt
- depart customer
- possible unplanned stop / break / vehicle issue
 Return Depot  
 End Shift

This gives the real operational flow including:
- failed deliveries
- unplanned stops
- maintenance interruptions
- overtime patterns
- repeat visits or loops

#### Level 2: Stop-level micro-process
For a stop, the actual sequence may be:

Enter stop area  
 parking/search time  
 Arrive Customer scan  
 customer interaction / handover / failed attempt  
 Depart Customer

This helps isolate whether delays occur in:
- approach travel
- parking/finding address
- customer interaction
- proof-of-delivery activity

#### Level 3: Package lifecycle
Assigned  
 Loaded  
 Out for delivery  
 Attempted  
 Delivered / Failed  
 Reattempted / returned / exception

This is important because failed first attempts create **rework**, which increases route load the next day.

### 1.7 Discovery algorithms and views

Because delivery processes are variable, I would use a combination of:

- **Inductive Miner** for a clean, interpretable baseline model
- **Heuristics or Fuzzy Miner** for high-variability route behavior
- **Directly-Follows Graphs with performance overlays** for timing bottlenecks
- **Object-centric discovery** if supported, to preserve relations among vehicles, packages, drivers, and stops

I would also supplement classic process models with:

- **geospatial route heatmaps**
- **stop-cluster views**
- **variant explorer**
- **time-of-day performance overlays**

---

### 1.8 Conformance checking against the planned routes

The dispatch plan is the reference model. For each vehicle-day, I would compare actual execution to plan using:

#### a) Sequence conformance
Did the driver visit the planned stops in the planned order?

Measures:
- skipped planned stops
- extra unplanned stops
- reordered stop sequence
- repeated visits
- depot returns mid-route

Useful methods:
- alignment-based conformance
- edit distance / Levenshtein distance on stop sequences
- route adherence score

#### b) Temporal conformance
Did execution match planned timings?

Measures:
- late route departure
- late arrival per stop
- service outside customer time window
- excessive dwell at stops
- overtime at route end
- segment travel time far above planned

#### c) Spatial conformance
Did the vehicle follow the planned path or corridor?

Measures:
- actual km vs planned km
- off-corridor distance
- detour frequency
- extra looping

#### d) Resource conformance
Was the assigned vehicle/driver actually used? Did maintenance overlap with the route?

#### e) Exception conformance
Were there unplanned vehicle warning stops, breakdowns, or unusually long idle periods?

### 1.9 Deviations to specifically look for

- Late departures from depot
- Stops visited out of planned order
- Too many unplanned stops
- Excessive idling between planned stops
- Significant actual-vs-planned travel time gaps
- Delivery attempts outside promised windows
- Capacity-related route overloading
- Vehicle issues mid-route
- High failed-delivery loops
- Long dwell at specific neighborhoods or customer types

## 2. Performance Analysis and Bottleneck Identification

### 2.1 KPIs to calculate from the event log

Here are the most relevant KPIs.

| KPI | How to calculate from the log | Why it matters |
|---|---|---|
| **On-Time Delivery Rate** | `# deliveries with Delivery Success timestamp within promised window / total due deliveries` | Core punctuality metric |
| **First-Attempt Success Rate** | `# packages delivered without prior failed attempt / total attempted packages` | Reduces costly rework |
| **Average Service Time per Stop** | `Depart Customer - Arrive Customer` | Indicates stop efficiency |
| **Average Inter-stop Travel Time** | `Arrive next stop - Depart previous stop` | Captures travel performance |
| **Travel Time / Service Time Ratio** | `Total travel time / total service time` | Shows whether inefficiency is mostly on-road or at-stop |
| **Traffic Delay Minutes** | `Actual segment time - expected segment baseline`, when positive | Quantifies congestion impact |
| **Idle Time Ratio** | `Ignition on + speed=0 outside legitimate service time / route duration` | Strong fuel and productivity indicator |
| **Unplanned Stop Rate** | `# unplanned stops above threshold / route or per 100 stops` | Reveals exceptions and waste |
| **Overtime Rate / Minutes** | `Actual shift end - scheduled shift end`, if positive | Labor cost and planning signal |
| **Vehicle Time Utilization** | `(travel + service + productive movement) / available shift time` | Measures asset productivity |
| **Capacity Utilization** | `loaded parcels/volume/weight / vehicle capacity` | Reveals under/over-loaded routes |
| **Maintenance Interruption Rate** | `# unscheduled repair events or stop minutes / 1000 km` | Reliability and cost |
| **Failed Delivery Rate** | `# failed attempts / total attempts` | Driver productivity and customer experience |
| **Actual vs Planned Distance** | `Actual km - planned km` or `% over plan` | Fuel and route efficiency |
| **Fuel per km / package** | Actual liters if available; otherwise estimated from km, idling, speed profile, vehicle type | Direct operating cost |

**Important note:** fuel is not explicitly in the described log, so I would either:
- integrate telematics/fuel-card data if available, or
- build a **fuel-consumption proxy model** using distance, idling, speed profile, stop-and-go intensity, and vehicle type.

---

### 2.2 Process mining techniques to identify bottlenecks

#### a) Performance overlays on discovered models
Overlay average and median durations on the process graph to see where time accumulates:
- travel between certain stop types
- specific exception paths
- failed-delivery branches
- return-to-depot bottlenecks

#### b) Variant analysis
Compare:
- high-performing vs low-performing route variants
- on-time vs late route cases
- routes with vs without breakdowns
- routes with high vs low failed-delivery rates

#### c) Process cubes / slicing
Slice the process by:
- route/territory
- urban vs suburban areas
- time of day
- weekday
- driver
- vehicle type
- customer time-window type

#### d) Geo-process mining
Map delays and deviations onto geography:
- road-segment delay hotspots
- parking/search hotspots
- customer clusters with long dwell
- areas with high failed-delivery rates

#### e) Resource analysis
Benchmark drivers and vehicles, but **normalize for route difficulty**. Otherwise you may blame drivers for inherently difficult territories.

#### f) Rework analysis
Track packages that go through failed attempt  reattempt loops. This quantifies the operational cost of first-attempt failure.

---

### 2.3 How to quantify bottleneck impact

For each bottleneck, I would quantify:

- **extra minutes per route**
- **extra km per route**
- **extra overtime minutes**
- **share of late deliveries affected**
- **estimated fuel impact**
- **rework generated**
- **cost-to-serve increase**

Example:
- “Segment A between 08:00–09:30 adds a median 14 minutes vs plan and appears in 38% of late routes.”
- “Apartment-cluster stops in Zone 3 add 4.5 extra minutes of service time per stop.”
- “Vehicle V12’s warning-related stops add 52 minutes average route delay and 19% more overtime.”

## 3. Root Cause Analysis for Inefficiencies

Finding bottlenecks is not enough; the key is validating **why** they happen.

### 3.1 Potential root causes and how process mining would test them

| Suspected root cause | How to validate with process mining | Evidence that would confirm it |
|---|---|---|
| **Suboptimal route planning** | Compare planned stop sequence with actual high-performing variants | Drivers frequently deviate from plan and those deviations improve on-time performance |
| **Inaccurate travel time estimates** | Compare planned vs actual inter-stop times by zone/time-of-day | Repeated underestimation on specific arcs or peak periods |
| **Traffic congestion patterns** | Correlate low-speed GPS episodes and delayed arcs with lateness | Persistent hotspots causing lateness spikes at certain times |
| **High service-time variability** | Analyze stop dwell-time distributions by customer, area, parcel count, stop type | Some stop categories have very high variance and push routes late |
| **Parking/finding-address issues** | Measure time from entering customer geofence to scanner “Arrive Customer” | Dense urban stops show long pre-arrival search/parking time |
| **Vehicle breakdowns/maintenance needs** | Link unscheduled stops and repair intervals to route disruption and vehicle history | Specific vehicles repeatedly create delay and overtime |
| **Driver behavior differences** | Compare drivers on similar routes after controlling for territory/time-window mix | Some drivers show more idling, detours, longer dwell, or poor sequence adherence |
| **Failed deliveries and re-delivery loops** | Analyze package variants with failed attempts and next-day rework | Certain time windows/customers/areas consistently generate failed attempts |

### 3.2 Specific analyses I would run

#### a) High-performer vs low-performer variant analysis
Compare routes with:
- high on-time rate vs low on-time rate
- low cost vs high cost
- low failed-delivery rates vs high failed-delivery rates

Look for structural differences:
- route order
- departure time
- idle behavior
- stop clustering quality
- failure loops
- breakdown exposure

#### b) Planned vs actual segment analysis
For every planned leg:
- planned duration
- actual duration
- actual speed profile
- time of day
- zone type

This shows whether the issue is poor execution or poor planning assumptions.

#### c) Dwell time decomposition
Break stop time into:
- approach/parking/search
- on-site service time
- post-service delay before departure

That helps distinguish:
- urban parking pain
- customer interaction complexity
- scanner/admin inefficiency

#### d) Traffic correlation
Using GPS low-speed events and optionally historical traffic data, identify whether lateness is:
- predictable recurring congestion
- random incident-based
- localized to certain corridors

#### e) Vehicle reliability analysis
Create a chain:
warning light / unscheduled stop  
 route interruption  
 missed windows  
 overtime / partial route completion

If certain vehicles repeatedly follow this pattern, maintenance is a root cause, not random bad luck.

#### f) Driver normalization analysis
Benchmark drivers only after controlling for:
- route density
- number of stops
- parcel volume
- time-window tightness
- urban/suburban mix

Otherwise comparisons are misleading.

## 4. Data-Driven Optimization Strategies

Below are **five concrete strategies**. Speedy Parcels only needs three to start, but these are the strongest candidates.

### Strategy 1: Dynamic route resequencing and ETA recalculation

**Targets**
- late deliveries
- overtime
- traffic-driven inefficiency
- excess idling in congestion

**Root cause addressed**
- static route planning
- poor travel-time assumptions
- failure to adapt to live traffic conditions

**How process mining supports it**
- conformance shows where actual execution systematically diverges from plan
- traffic hotspot analysis reveals recurring peak-period problem arcs
- high-performing variants may already demonstrate better stop sequences than the planned ones

**What to do**
- use historical actual travel times by segment, time-of-day, and area type
- recalculate ETAs during the route
- allow controlled resequencing of remaining stops when predicted lateness exceeds a threshold
- freeze only high-priority or tight-window stops

**Expected KPI impact**
- higher On-Time Delivery Rate
- lower Traffic Delay Minutes
- lower Overtime
- lower Actual vs Planned Distance
- lower fuel per route/package

---

### Strategy 2: Redesign delivery territories and route templates

**Targets**
- overloaded or unbalanced routes
- route duration variability
- excessive travel between clusters
- capacity mismatch

**Root cause addressed**
- poor territory design
- route plans based on geography alone rather than actual service/travel behavior
- underestimation of urban stop difficulty

**How process mining supports it**
- route-level variants show which territories consistently run late
- service-time analysis reveals areas with longer dwell due to parking/building access
- capacity utilization and overtime patterns show imbalance between routes

**What to do**
- re-cluster customers into micro-territories based on real execution data
- balance routes using:
  - historical travel time
  - historical service time
  - parcel density
  - time-window density
  - vehicle capacity
- separate difficult urban cores from suburban routes if needed
- build standard route templates for recurring zones

**Expected KPI impact**
- lower route duration variance
- better Vehicle Time Utilization and Capacity Utilization
- lower overtime
- better On-Time Delivery Rate
- fewer extreme “bad routes”

---

### Strategy 3: Failed-delivery reduction program

**Targets**
- failed attempts
- re-delivery cost
- wasted travel and service time
- customer dissatisfaction

**Root cause addressed**
- wrong attempt timing
- poor customer availability alignment
- address/access issues
- insufficient communication before arrival

**How process mining supports it**
- package lifecycle mining shows the volume and cost of rework
- failed attempts can be clustered by:
  - time window
  - customer/location
  - area
  - driver
  - stop time
- process variants show how failed deliveries cascade into late routes and next-day overload

**What to do**
- build a risk score for likely failed deliveries
- proactively message customers with ETA windows
- require confirmation for high-risk deliveries
- route high-risk residential parcels to lockers/PUDO points where possible
- for recurring failure addresses, use different time bands or call-before-arrival policy

**Expected KPI impact**
- higher First-Attempt Success Rate
- lower Failed Delivery Rate
- lower km per successful delivery
- better On-Time Delivery Rate
- lower next-day route load and rework

---

### Strategy 4: Predictive maintenance and vehicle assignment optimization

**Targets**
- unscheduled stops
- route disruption from vehicle issues
- maintenance cost
- fuel inefficiency from poor vehicle condition

**Root cause addressed**
- reactive maintenance
- repeated use of unreliable vehicles on critical routes
- vehicle deterioration linked to stop-and-go urban usage

**How process mining supports it**
- vehicle timelines show recurring warning-light or unscheduled stop patterns
- alignment of route delays with maintenance events quantifies operational impact
- some vehicles may be overrepresented in overtime, route failure, and idling patterns

**What to do**
- create vehicle risk profiles based on:
  - mileage
  - stop-start intensity
  - idle time
  - warning events
  - repair history
- schedule maintenance before failure
- avoid assigning high-risk vehicles to long or time-window-tight routes
- use more reliable vans on denser urban routes if they are most vulnerable to failure impact

**Expected KPI impact**
- lower Maintenance Interruption Rate
- lower route failure and overtime
- higher On-Time Delivery Rate
- lower maintenance cost volatility
- potentially lower fuel consumption

---

### Strategy 5: Targeted driver coaching and SOP standardization

**Targets**
- excess idle time
- inconsistent stop execution
- unnecessary detours
- avoidable dwell variation

**Root cause addressed**
- driver practice differences
- inefficient stop routines
- poor parking/search habits
- inconsistent failed-delivery handling

**How process mining supports it**
- driver comparisons on similar route types reveal performance gaps
- best-practice variants can be extracted from top performers
- stop-level analysis shows whether issues occur before arrival scan, during service, or after completion

**What to do**
- coach drivers using evidence, not generic instruction:
  - idle reduction
  - efficient stop sequence adherence
  - better customer approach/parking routines
  - faster scanning/proof-of-delivery handling
  - better failed-attempt protocol
- create route-type-specific SOPs rather than one generic SOP

**Expected KPI impact**
- lower Average Service Time per Stop
- lower Idle Time Ratio
- lower fuel use
- better route adherence
- improved punctuality consistency

## 5. Operational Constraints and Continuous Monitoring

### 5.1 How to respect operational constraints

Any optimization must be implemented as **constrained optimization**, not pure cost minimization.

#### Driver working hours
- treat legal hours and breaks as hard constraints
- monitor overtime risk in real time
- dynamic resequencing must not push routes beyond legal limits

#### Vehicle capacity
- route redesign must respect parcel count/volume/weight
- if urban routes have many bulky parcels, stop count alone is misleading

#### Customer time windows
- dynamic routing may reorder stops, but only within promised service windows
- use time-window adherence as a top-level constraint, not an afterthought

#### Vehicle availability and maintenance
- predictive maintenance scheduling must align with route demand and spare-fleet availability

#### Operational practicality
- dispatchers need explainable recommendations
- drivers need stable enough routes to execute safely
- some deviations are good adaptations, not non-compliance

A good rollout approach is:
1. baseline measurement  
2. pilot on selected routes/territories  
3. compare before/after with control group  
4. scale only if KPI gains are sustained

Process mining should become an ongoing control mechanism, not a one-off study.

#### A. Executive dashboard
Track weekly and monthly:
- On-Time Delivery Rate
- First-Attempt Success Rate
- Overtime Minutes
- Actual vs Planned Distance
- Fuel per km/package
- Cost per successful delivery
- Unscheduled maintenance interruptions

#### B. Conformance dashboard
Track:
- route adherence score
- stop sequence deviation
- time-window violations
- unplanned stop rate
- late depot departure rate
- actual vs planned route duration

#### C. Bottleneck dashboard
Track:
- top delayed road segments
- zones with highest stop dwell
- customer clusters with high failed-delivery rate
- routes with the highest travel/service imbalance
- recurring parking/search hotspots

#### D. Resource dashboard
Track by driver and vehicle, normalized for route difficulty:
- idle time ratio
- average service time per stop
- failed-delivery rate
- deviation frequency
- warning/repair frequency

#### E. Process views to keep active
- discovered route process with performance overlay
- geo heatmap of delay hotspots
- variant explorer by territory and time of day
- package rework loop view
- vehicle reliability timeline
- before/after comparison view for each optimization initiative

#### F. Alerting and drift detection
Set alerts for:
- sudden drop in on-time performance in a zone
- new traffic hotspot
- spike in failed deliveries in a customer segment
- emerging vehicle failure pattern
- rising overtime on a route family

Also use **process drift monitoring** so management can detect:
- seasonal changes
- operational degradation after initial gains
- new bottlenecks created by old fixes

4. **Quantify delay, failure, idle, and maintenance bottlenecks**
