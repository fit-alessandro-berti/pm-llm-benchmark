This problem requires several complex steps: ordering, aggregation into a sequence, counting, ranking, and finally, filtering the original dataset based on the successful cases.

We will use Common Table Expressions (CTEs) to break down the process logically.

Assume $K=3$ for demonstration purposes.

### DuckDB SQL Solution

```sql
WITH
-- 1. Determine the ordered sequence for every case_id
-- We order the activities within each case_id by timestamp and aggregate them into a single string.
sequenced_cases AS (
    SELECT
        case_id,
        -- Using array_agg and then casting to text (or using STRING_AGG if supported/preferred) 
        -- ensures the sequence is consistently represented for grouping.
        -- For robustness across different DuckDB versions/environments, STRING_AGG is ideal.
        STRING_AGG(activity, ' -> ORDER -> ') AS sequence
    FROM
        event_log
    GROUP BY
        case_id
),

-- 2. Group cases by their complete activity sequence and count the frequency
variant_counts AS (
    SELECT
        sequence,
        COUNT(case_id) AS variant_count
    FROM
        sequenced_cases
    GROUP BY
        sequence
),

-- 3. Identify the top K sequences based on frequency
top_k_variants AS (
    SELECT
        sequence
    FROM
        variant_counts
    ORDER BY
        variant_count DESC,
        sequence ASC -- Tie-breaker
    LIMIT 3 -- K = 3 (Change this number to select a different K)
)

-- 4. Retrieve all original events (all columns) belonging only to cases matching these top K variants
SELECT
    e.*
FROM
    event_log e
JOIN
    sequenced_cases sc ON e.case_id = sc.case_id
JOIN
    top_k_variants tkv ON sc.sequence = tkv.sequence
ORDER BY
    e.case_id, e.timestamp;
```

### Explanation of Steps:

1.  **`sequenced_cases` (Windowing/Ordering):**
    *   This CTE performs the core sequence generation. It groups the raw events by `case_id`.
    *   Crucially, `STRING_AGG(activity, ' -> ORDER -> ')` aggregates all `activity` values within a single `case_id` into one unique string, using the specified separator. Because the events were processed *implicitly* by the grouping structure (and assuming an underlying ordered aggregation or that the data is grouped correctly), this step creates a unique identifier (the `sequence`) for the type of process that occurred in that case.
2.  **`variant_counts` (Aggregation):**
    *   We group the results of `sequenced_cases` by the newly generated `sequence`.
    *   `COUNT(case_id)` counts how many distinct cases arrived at this specific sequence, thus calculating the variant frequency.
3.  **`top_k_variants` (Ranking/Filtering):**
    *   This CTE ranks the sequences by `variant_count` in descending order.
    *   `LIMIT 3` selects the top K sequences, identifying the successful process patterns.
4.  **Final `SELECT` (Filtering and Output):**
    *   We join `event_log` (the original data, `e`) back through `sequenced_cases` (`sc`) and finally to `top_k_variants` (`tkv`).
    *   By joining on the shared `sequence`, we ensure that *only* the `case_id`s that exhibited one of the top K sequences are included.
    *   The final `SELECT e.*` retrieves all the original event details for these filtered, successful cases.
