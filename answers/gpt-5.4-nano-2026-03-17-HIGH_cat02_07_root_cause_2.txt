## 2) Root-cause analysis from correlations with Resource / Region / Complexity

### A) **Complexity is the dominant driver**
A clear pattern appears:

- **Low complexity (2001, 2004)**  ~**1–1.5 hours**
- **Medium complexity (2002)**  ~**26 hours**
- **High complexity (2003, 2005)**  ~**48–77 hours**

So longer lead times strongly correlate with **higher Complexity**.

**Why this would happen:** High-complexity claims typically require more verification and more evidence, which leads to waiting for additional documents and rework.

---

### B) Complexity likely increases “Request Additional Documents” churn (and waiting)
Number of **Request Additional Documents** events:

- **2001 (Low):** 0  
- **2004 (Low):** 0  
- **2002 (Medium):** 1 request
- **2003 (High):** **2 requests**
- **2005 (High):** **3 requests**

And the cases with the most requests are exactly the longest-duration ones (**2003, 2005**).

Additionally, in the high-duration cases there are large gaps between document requests and later approvals, e.g.:
- **2003:** last request (04-01 17:00)  approval (04-02 16:00): **~23h**
- **2005:** last request (04-03 15:00)  approval (04-04 10:00): **~19h**

**Why this would happen:** repeated document requests extend time, but even after the final request, the process still waits for (a) claimants to respond and (b) internal review/queue time to approve.

---

### C) **Region seems weaker as a root cause**
- **Region A**: 2001 (fast, Low), 2003 (slow, High)
- **Region B**: 2002 (medium), 2004 (fast, Low), 2005 (slow, High)

So both regions include both fast and slow outcomes. This suggests **region isn’t the primary bottleneck** in this dataset.

---

### D) **Resource assignment is not consistently predictive**, but capacity/queue may matter
- Adjusters handling the “Request Additional Documents” include **Adjuster_Lisa** (2002, 2005) and **Adjuster_Mike** (2003).
- Approvals are done by **Manager_Ann** (2001, 2002, 2004) and **Manager_Bill** (2003, 2005).
- Despite different resources, the slow cases align far more with **High complexity** and **more document requests** than with any single named resource.

**Most plausible interpretation:** resources contribute indirectly (review capacity/queue), but the main structural driver is **complexity-driven documentation cycles**.

## 3) Mitigation suggestions (to reduce long case durations)

### 1) Reduce avoidable additional-document cycles (biggest lever)
- Create/standardize **claim-specific document checklists** for each complexity level.
- At **Evaluate Claim**, proactively request the *complete* set of documents (avoid multiple rounds).
- Use **document completeness validation** (e.g., automated checks) before issuing further requests.

### 2) Add SLAs and escalation for the “waiting” portion
Since long gaps occur around/after document requests:
- Set **response SLAs** for claimants and internal reviewers.
- Implement **automated reminders + escalation** if documents aren’t received within a defined window.
- Consider **parallel processing**: start review of what’s already received while waiting for missing items.

### 3) Capacity planning for high-complexity workflows
- Flag high-complexity claims early and route them to a **dedicated queue** with enough adjuster/manager coverage.
- Ensure managers handling approval (especially for high-complexity cases) have predictable throughput and fewer bottlenecks.

### 4) Improve “single request” strategy
For cases like **2005** (3 requests), aim for:
- **One consolidated request** with clear instructions and acceptable evidence formats.
- Better claimant guidance (templates, examples, common failure points).

---

### Bottom line
The longest-running cases (**2003, 2005**) are best explained by **High complexity** leading to **multiple rounds of additional document requests** and subsequent waiting/queue time. **Region and individual resource names show weaker correlation** than complexity-driven documentation effort.
