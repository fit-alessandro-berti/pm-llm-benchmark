### Critique of Your Draft
*   **Pro:** It leaves no ambiguity about the schema or the business logic.
*   **Con (for benchmarking):** The second paragraph **"gives away the answer"** by outlining the exact step-by-step logic (`GROUP BY`, `COUNT`, `FILTER`, `JOIN`/Subselect). If you want to test an LLM's reasoning and SQL knowledge, it is usually better to state the *business rule* and let the LLM figure out the *technical execution*.

Here are three ways to refine this prompt based on what kind of benchmark you are running:

### Option 1: The "Balanced" Version (Recommended)
*Use this if you want a clean, standard benchmark that measures general SQL competence without hand-holding.*

> "Assume you have an event log stored in a DuckDB table named `event_log`. Each row represents a single event. The table has at least the following columns: `case_id`, `activity`, and `timestamp` (where `case_id` groups events that belong to a single process instance). 
> 
> Write a DuckDB SQL query that returns all columns for all events, but completely excludes any cases that contain more than six total events."

### Option 2: The "Idiomatic DuckDB" Version (Harder step)
*Use this if you want to see if the LLM actually knows DuckDB's specific syntax, rather than just writing generic standard SQL.*

> "Assume you have an event log stored in a DuckDB table named `event_log`, with at least the following columns: `case_id`, `activity`, and `timestamp`. 
> 
> Write a DuckDB SQL query that returns all events, but filters out any cases containing more than six events. **Constraint:** Please write the most concise, idiomatic DuckDB query possible, avoiding Common Table Expressions (CTEs) or subqueries if DuckDB's advanced SQL features allow for a simpler approach."

### Option 3: The "Polished Original" (If you *want* to guide the logic)
*Use this if you are testing lower-tier/smaller open-source models that tend to get confused without explicit step-by-step instructions.*

> "Assume you have an event log stored in a DuckDB table named `event_log`, with each row representing a single event. The table has at least the following columns: `case_id`, `activity`, and `timestamp`. Each `case_id` groups events that belong to a single process instance (or 'case').
> 
> Your task: Please craft a DuckDB SQL query that returns all columns for the events from cases that have six or fewer events in total. To do this, your query should identify the total count of events per `case_id`, exclude any `case_id` where this count is greater than six, and return the underlying event rows for the remaining cases."

**Level 2: Intermediate (Pass)**
