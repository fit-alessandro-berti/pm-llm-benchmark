## 1. Process Discovery and Conformance Checking

### 1.1 Building a cohesive event log

For Speedy Parcels, I would not treat the raw GPS, scanner, dispatch, and maintenance data as one event log immediately. The first step is to build a clean, analysis-ready process mining log with a consistent case concept, timestamp standard, activity taxonomy, and event hierarchy.

The most practical primary case ID is:

> **Vehicle-Day**: one operational shift for one vehicle, for example `V12-20241205`.

This case captures the end-to-end delivery route from shift start, depot departure, customer stops, failed deliveries, unplanned stops, return to depot, and shift end.

However, Speedy Parcels also needs package-level analysis. A single vehicle-day contains many package-level subprocesses, so I would create either:

1. **A hierarchical log**
   Primary case: vehicle-day.
   Subcase: package delivery attempt.

2. **An object-centric event log**
   Objects: vehicle, driver, route, package, depot, customer stop, maintenance job.
   This is useful because one route contains many packages, one vehicle is used across many days, and one failed package may appear in multiple delivery attempts.

A recommended event log schema would include:

| Field                           | Description                                                                            |
| ------------------------------- | -------------------------------------------------------------------------------------- |
| Case ID                         | Vehicle-day, e.g. `V12-20241205`                                                       |
| Package ID                      | Present for package-specific events                                                    |
| Route ID                        | Planned route from dispatch system                                                     |
| Activity                        | Standardized activity name, e.g. `Depart Depot`, `Arrive Customer`, `Delivery Success` |
| Timestamp start/end             | Event timestamp or interval for duration activities                                    |
| Lifecycle                       | Start, complete, or instantaneous                                                      |
| Vehicle ID                      | Vehicle resource                                                                       |
| Driver ID                       | Human resource                                                                         |
| Planned stop number             | From dispatch                                                                          |
| Actual stop number              | Derived from scanner/GPS sequence                                                      |
| Planned arrival time            | From dispatch system                                                                   |
| Actual arrival time             | From GPS/scanner                                                                       |
| Customer time window            | Requested delivery window                                                              |
| Location                        | Lat/lon or geofence ID                                                                 |
| Distance since previous event   | Derived from GPS trace                                                                 |
| Travel time since previous stop | Derived                                                                                |
| Service time                    | `Depart Customer - Arrive Customer`                                                    |
| Idle time                       | Derived from GPS moving/idle status                                                    |
| Delay category                  | Traffic, customer service, failed delivery, maintenance, unexplained idle, depot delay |
| Maintenance flag                | Whether vehicle had planned or unplanned maintenance                                   |
| Outcome                         | Success, failed attempt, returned package, re-delivery needed                          |

The GPS data needs event abstraction. Raw GPS pings are too granular for process mining, so I would transform them into meaningful business events, for example:

| Raw GPS pattern                                             | Process event                          |
| ----------------------------------------------------------- | -------------------------------------- |
| Vehicle exits depot geofence                                | `Depart Depot`                         |
| Vehicle enters customer geofence and speed becomes 0        | `Arrive Customer`                      |
| Vehicle stays stopped outside planned delivery location     | `Unplanned Stop`                       |
| Speed below threshold for sustained period                  | `Traffic Delay` or `Low Speed Segment` |
| Ignition off during route                                   | `Vehicle Stopped`                      |
| Vehicle enters maintenance location or warning code appears | `Maintenance/Breakdown Event`          |
| Vehicle re-enters depot geofence                            | `Arrive Depot`                         |

The scanner data provides the operational truth for package outcomes. GPS confirms where and when the vehicle actually moved. Dispatch gives the intended plan. Maintenance logs explain vehicle-related interruptions.

### 1.2 Data integration challenges

Several challenges are likely.

**Timestamp synchronization** is critical. GPS units, scanners, dispatch systems, and maintenance systems may use different clocks, time zones, or recording delays. All timestamps should be normalized to one timezone and audited for clock drift.

**Activity naming inconsistencies** are common. For example, `Delivery Failed`, `Failed Attempt`, and `Customer Not Home` may need to be mapped into one standardized activity with a failure reason attribute.

**Different event granularity** must be reconciled. GPS data may have thousands of points per route, while dispatch and scanner data have only milestone events. The GPS trace must be compressed into route segments, stop events, idle periods, congestion events, and deviation events.

**Missing or delayed scans** can distort the process model. A driver may forget to scan `Arrive Customer` but scan `Delivery Success`; the arrival time may then need to be inferred from GPS geofence entry.

**Location quality issues** can occur in dense urban areas. GPS drift between tall buildings may make it look like a vehicle stopped at the wrong address. Map matching and customer-location geofencing are needed.

**Ambiguous stop attribution** is another issue. A vehicle may stop near multiple customer addresses. The system must decide which stop the event belongs to by combining planned stop order, package scan, location proximity, and timestamp.

**Multi-object relationships** matter. One vehicle-day contains many packages; one failed package can reappear on another day; one vehicle can have repeated maintenance issues. This is why a purely single-case log may not be sufficient for all analyses.

**Privacy and labor relations** must be handled carefully. Driver performance analysis should be used for operational improvement and coaching, not simplistic blame assignment. External factors such as route difficulty, traffic, parking availability, package mix, and vehicle condition must be controlled for.

### 1.3 Process discovery: visualizing the actual delivery process

After preprocessing, I would use process discovery to reveal the actual process followed by vehicles and drivers, not the process assumed by dispatch.

At a high level, the discovered process may look like:

> `Start Shift → Route Assigned → Load/Scan at Depot → Depart Depot → Travel to Stop → Arrive Customer → Delivery Success/Failed → Depart Customer → … repeated stops … → Return Depot → End Shift`

But the actual model will likely show additional behavior:

> `Traffic Delay`, `Unplanned Stop`, `Long Idle`, `Re-sequenced Stop`, `Missed Time Window`, `Breakdown`, `Maintenance Stop`, `Package Returned`, `Overtime`, `Failed Delivery Reattempt`

I would use multiple discovery views rather than one model only.

**Directly-Follows Graphs** would show the most common activity transitions, such as `Depart Customer → Travel → Arrive Customer`, and reveal frequent unexpected transitions, such as `Travel → Unplanned Stop → Travel` or `Delivery Failed → Return Depot`.

**Inductive Miner** would be useful for creating a structured model that separates normal loops, optional activities, and exceptional paths. It is especially helpful for showing the repeated stop cycle in a clean way.

**Heuristics Miner** would help handle noisy logistics data by focusing on frequent behavior and filtering rare GPS anomalies.

**Fuzzy Miner or process maps with frequency sliders** would be useful for management dashboards because delivery operations can have many variants. A fuzzy model can hide low-frequency noise while preserving important bottlenecks.

**Variant analysis** would show the most common route execution patterns, for example:

| Variant   | Pattern                                                            | Interpretation              |
| --------- | ------------------------------------------------------------------ | --------------------------- |
| Variant A | Depart depot → mostly successful stops → return depot on time      | Healthy route               |
| Variant B | Early traffic delay → late first stops → cascading late deliveries | Traffic-sensitive route     |
| Variant C | Many failed deliveries → returned packages → re-delivery next day  | Customer availability issue |
| Variant D | Unplanned maintenance stop → route incomplete → overtime           | Vehicle reliability issue   |
| Variant E | Long depot loading time → late departure → late deliveries         | Depot dispatch issue        |

**Geospatial process mining** would overlay process events on a map. This is essential in transportation. The same process map should be viewable by route corridor, stop cluster, traffic hotspot, depot zone, and parking-problem area.

### 1.4 Conformance checking against planned routes

The dispatch system provides the planned route: stop sequence, expected arrival/departure times, package assignments, capacity assumptions, driver assignment, and customer time windows.

Conformance checking compares this planned model with actual execution.

I would perform conformance checking at four levels.

#### A. Sequence conformance

This checks whether stops were visited in the planned order.

Examples of deviations:

| Planned                                   | Actual                            | Deviation         |
| ----------------------------------------- | --------------------------------- | ----------------- |
| Stop 1 → Stop 2 → Stop 3                  | Stop 1 → Stop 3 → Stop 2          | Re-sequencing     |
| Stop 1 → Stop 2 → Stop 3                  | Stop 1 → Stop 3                   | Skipped stop      |
| Stop 1 → Stop 2                           | Stop 1 → Unplanned Stop → Stop 2  | Extra stop        |
| Planned package P9879 delivered at Stop 5 | Failed attempt, returned to depot | Outcome deviation |

Re-sequencing is not always bad. A driver may adapt intelligently to traffic or parking. The question is whether re-sequencing improves or worsens punctuality, distance, fuel use, and overtime.

#### B. Time conformance

This checks whether events occurred within planned timing expectations.

Examples:

| Planned timing          | Actual result       | Deviation                                  |
| ----------------------- | ------------------- | ------------------------------------------ |
| Depart depot 07:30      | Depart depot 07:45  | Late departure                             |
| Arrive Stop 1 08:05     | Arrive Stop 1 08:25 | Travel-time underestimate or traffic delay |
| Service time 3 min      | Service time 12 min | Long customer dwell time                   |
| Return depot 16:30      | Return depot 16:50  | Late return                                |
| Time window 09:00–11:00 | Arrival 11:20       | Missed delivery window                     |

Time deviations should be split into categories: depot delay, travel delay, customer service delay, idle delay, failed delivery delay, maintenance delay, and unexplained delay.

#### C. Location conformance

This checks whether the vehicle physically followed the planned route corridor and visited the intended geofences.

Examples:

| Expected                             | Actual                        | Possible cause                                     |
| ------------------------------------ | ----------------------------- | -------------------------------------------------- |
| Vehicle follows planned corridor     | Vehicle takes long detour     | Road closure, traffic, driver choice, bad routing  |
| Stop near customer geofence          | Stop far from customer        | Parking issue, GPS error, wrong address            |
| No stop between customers            | 20-minute stop away from plan | Break, vehicle issue, personal stop, loading issue |
| Return to depot after all deliveries | Early depot return            | Failed route, vehicle issue, missing packages      |

#### D. Resource and capacity conformance

This checks whether the assigned vehicle, driver, and package load matched the plan.

Examples:

| Planned                      | Actual                     | Risk                       |
| ---------------------------- | -------------------------- | -------------------------- |
| Vehicle V12 assigned         | Vehicle V15 used           | Vehicle availability issue |
| 35 stops                     | 44 actual package attempts | Overloaded route           |
| Driver D105                  | Substitute driver          | Skill/familiarity effect   |
| Vehicle capacity 100 parcels | Loaded 118 parcels         | Capacity violation         |
| Shift ends 17:00             | End shift 17:40            | Overtime and cost issue    |

For formal process mining, alignments or token-based replay can quantify the fitness of actual traces against planned models. In practical logistics terms, the result should also be translated into business-friendly deviation metrics:

| Metric                           | Meaning                                        |
| -------------------------------- | ---------------------------------------------- |
| Route adherence score            | How closely actual stop sequence followed plan |
| Planned vs actual distance gap   | Extra kilometers driven                        |
| Planned vs actual duration gap   | Extra time consumed                            |
| Stop order deviation count       | Number of sequence changes                     |
| Unplanned stop count/duration    | Frequency and severity of unexplained stops    |
| Time-window violation count      | Customer punctuality failure                   |
| Late departure contribution      | How much delay started at depot                |
| Maintenance interruption minutes | Direct vehicle downtime during route           |

## 2. Performance Analysis and Bottleneck Identification

### 2.1 Key KPIs and how to calculate them

Speedy Parcels’ KPIs should connect directly to its two goals: improving delivery punctuality and reducing operating cost.

#### Delivery punctuality KPIs

| KPI                                     | Calculation from event log                                                             |
| --------------------------------------- | -------------------------------------------------------------------------------------- |
| **On-Time Delivery Rate**               | Successful deliveries within customer time window ÷ total successful deliveries        |
| **First-Attempt Delivery Success Rate** | Packages delivered successfully on first attempt ÷ total first attempts                |
| **Late Delivery Rate**                  | Deliveries completed after requested or promised window ÷ total deliveries             |
| **Average Lateness**                    | Average of `actual delivery timestamp - promised latest timestamp` for late deliveries |
| **Route Completion Rate**               | Completed planned stops ÷ assigned planned stops                                       |
| **Time Window Compliance by Route**     | On-time stops per route ÷ stops with time windows                                      |
| **Cascading Delay Index**               | Increase in lateness from early stops to later stops on same route                     |

#### Productivity KPIs

| KPI                                | Calculation                                               |
| ---------------------------------- | --------------------------------------------------------- |
| **Average Time per Delivery Stop** | `(Depart Customer - Arrive Customer)` averaged over stops |
| **Median Service Time per Stop**   | Median customer dwell time, less sensitive to outliers    |
| **Travel Time per Stop**           | `Arrive current stop - Depart previous stop`              |
| **Stops per Driver Hour**          | Completed stops ÷ shift duration                          |
| **Packages per Vehicle Hour**      | Delivered packages ÷ vehicle active time                  |
| **Depot Departure Delay**          | `Depart Depot actual - planned depart time`               |
| **Return-to-Depot Delay**          | `Arrive Depot actual - planned return time`               |
| **Overtime Minutes**               | `End Shift - scheduled shift end`, when positive          |

#### Cost and fuel KPIs

If direct fuel sensor data is unavailable, fuel can be estimated using distance, speed profile, idling duration, vehicle type, and known fuel curves.

| KPI                              | Calculation                                                                        |
| -------------------------------- | ---------------------------------------------------------------------------------- |
| **Distance per Package**         | Actual route kilometers ÷ delivered packages                                       |
| **Distance per Stop**            | Actual route kilometers ÷ completed stops                                          |
| **Fuel Consumption per km**      | Fuel used ÷ actual route kilometers                                                |
| **Fuel Consumption per Package** | Fuel used ÷ delivered packages                                                     |
| **Idle Time Ratio**              | Idle minutes ÷ route active minutes                                                |
| **Low-Speed Driving Ratio**      | Minutes below congestion threshold ÷ driving minutes                               |
| **Extra Distance vs Plan**       | Actual kilometers - planned kilometers                                             |
| **Extra Cost per Route**         | Extra distance cost + overtime cost + re-delivery cost + maintenance downtime cost |

#### Vehicle and capacity KPIs

| KPI                                     | Calculation                                              |
| --------------------------------------- | -------------------------------------------------------- |
| **Vehicle Utilization Rate**            | Loaded volume or weight ÷ vehicle capacity               |
| **Route Load Balance**                  | Variation in stops/packages across vehicles and drivers  |
| **Maintenance Downtime Rate**           | Maintenance or repair time ÷ available vehicle time      |
| **Breakdown Rate per 1,000 km**         | Unscheduled repair events ÷ kilometers driven × 1,000    |
| **Routes Interrupted by Vehicle Issue** | Routes with breakdown/maintenance event ÷ total routes   |
| **Mean Time Between Failures**          | Operating time or kilometers between unscheduled repairs |

#### Delivery failure KPIs

| KPI                         | Calculation                                                                     |
| --------------------------- | ------------------------------------------------------------------------------- |
| **Failed Delivery Rate**    | Failed attempts ÷ total attempts                                                |
| **Failure Reason Mix**      | Failures grouped by reason, e.g. customer not home, wrong address, access issue |
| **Re-delivery Burden**      | Re-delivery attempts ÷ total packages                                           |
| **Cost of Failed Delivery** | Additional distance/time from reattempts + customer service cost                |
| **Failure Hotspot Rate**    | Failed attempts by customer zone/building type/time window                      |

### 2.2 Bottleneck identification using process mining

Bottlenecks should be analyzed across time, space, route design, driver behavior, customer characteristics, and vehicle condition.

#### A. Performance overlays on process maps

A discovered process model can be enriched with average and percentile durations.

For example:

| Process segment                          | Metric to overlay        |
| ---------------------------------------- | ------------------------ |
| `Depart Depot → Arrive Stop 1`           | First-leg travel time    |
| `Arrive Customer → Delivery Success`     | Customer service time    |
| `Delivery Success → Depart Customer`     | Post-delivery dwell time |
| `Depart Customer → Arrive Next Customer` | Inter-stop travel time   |
| `Travel → Low Speed Detected → Travel`   | Traffic delay duration   |
| `Vehicle Issue → Continue Route`         | Breakdown delay          |
| `Arrive Depot → End Shift`               | End-of-day admin time    |

This can reveal where the longest delays accumulate.

#### B. Variant comparison

Routes can be grouped into high-performing and low-performing variants.

For example:

| Group                   | Criteria                                                              |
| ----------------------- | --------------------------------------------------------------------- |
| High-performing routes  | On-time rate above 95%, low extra kilometers, no overtime             |
| Low-performing routes   | On-time rate below 85%, high fuel per package, frequent re-sequencing |
| Breakdown routes        | Unscheduled maintenance event during shift                            |
| Failure-heavy routes    | Failed delivery rate above threshold                                  |
| Congestion-heavy routes | High low-speed ratio or traffic delay minutes                         |

Then the process models for each group are compared. If low-performing routes show frequent `Late Depot Departure`, `Low Speed Detected`, `Unplanned Stop`, and `Long Customer Dwell`, the bottlenecks become visible.

#### C. Waiting-time and dwell-time analysis

Dwell time at customer locations should be separated from travel time.

A stop may have:

> `Arrive Customer → Delivery Success → Depart Customer`

The total stop dwell time is:

> `Depart Customer - Arrive Customer`

This should be split where possible into:

| Component                                    | Possible interpretation                                     |
| -------------------------------------------- | ----------------------------------------------------------- |
| Time from arrival to scan                    | Parking, walking to building, access delay                  |
| Time from scan to delivery success           | Customer interaction, signature, parcel handover            |
| Time after delivery success before departure | Admin, sorting next package, break, congestion leaving area |

Long dwell times may indicate apartment buildings, access-code problems, commercial receiving delays, parking difficulty, or driver process issues.

#### D. Traffic and geospatial bottleneck analysis

GPS-derived low-speed segments should be mapped by time of day and location.

This can identify:

| Pattern                                                | Interpretation                                |
| ------------------------------------------------------ | --------------------------------------------- |
| Repeated low-speed events on same corridor 08:00–09:30 | Morning congestion                            |
| Long idle near city center stops                       | Parking/search issue                          |
| Detours around specific roads                          | Roadworks, restrictions, routing system issue |
| Frequent late first stops                              | Departure timing or first-leg traffic problem |
| Same route late only on Fridays                        | day-specific traffic/customer pattern         |

#### E. Resource analysis

Driver, vehicle, and route performance should be compared carefully.

Useful dimensions:

| Dimension   | Analysis                                                        |
| ----------- | --------------------------------------------------------------- |
| Driver      | Stops/hour, service time, scan compliance, route adherence      |
| Vehicle     | Fuel/km, breakdown rate, idle ratio, maintenance downtime       |
| Route       | On-time rate, distance/package, congestion exposure             |
| Territory   | failed delivery rate, parking delay, average dwell time         |
| Time of day | travel time reliability, service time variability               |
| Package mix | bulky parcels, signature required, commercial/residential split |

Driver comparison must control for route difficulty. A driver on dense urban routes with apartment buildings should not be compared directly with a driver on suburban single-family routes without adjustment.

---

### 2.3 Quantifying bottleneck impact

Each bottleneck should be translated into business impact.

Examples:

| Bottleneck            | Impact calculation                                                 |
| --------------------- | ------------------------------------------------------------------ |
| Late depot departure  | Minutes late at first stop; percentage of route lateness explained |
| Traffic hotspot       | Added travel minutes and fuel per route passing through hotspot    |
| Long service time     | Extra dwell minutes × stops affected                               |
| Failed delivery       | Extra future route kilometers and labor minutes for reattempt      |
| Vehicle breakdown     | Lost route minutes, missed deliveries, overtime, repair cost       |
| Poor route sequencing | Extra kilometers vs optimized sequence                             |
| Idle time             | Fuel wasted, driver time lost, emissions impact                    |
| Overloaded route      | Relationship between planned stop count and late return/overtime   |

The goal is to move from “Route V12 was late” to “Route V12 was late because it departed 15 minutes late, lost 22 minutes in a known traffic corridor, spent 35 minutes above normal dwell time in apartment buildings, and had one unplanned vehicle stop.”

## 3. Root Cause Analysis for Inefficiencies

### 3.1 Suboptimal route planning

A key hypothesis is that planned routes are too static and do not reflect actual urban delivery conditions.

Evidence to look for:

| Process mining evidence                                                 | Root cause implication                                              |
| ----------------------------------------------------------------------- | ------------------------------------------------------------------- |
| Frequent stop re-sequencing by experienced drivers improves performance | Dispatch sequence is suboptimal                                     |
| Planned distance much lower than actual distance                        | Route plan ignores access restrictions, parking, or road conditions |
| Certain planned first stops are often late                              | First-leg travel estimate is unrealistic                            |
| Same planned corridor repeatedly shows low-speed GPS events             | Static routing ignores predictable congestion                       |
| High route adherence but poor punctuality                               | Plan itself is unrealistic, not driver deviation                    |

Variant analysis can compare routes where drivers follow the plan exactly against routes where drivers deviate. If deviations improve punctuality or reduce distance, the routing algorithm should learn from those actual patterns.

---

### 3.2 Inaccurate travel time estimations

The dispatch system may underestimate travel time between stops, especially during peak periods.

Analysis:

| Test                                     | Method                                                                           |
| ---------------------------------------- | -------------------------------------------------------------------------------- |
| Planned vs actual travel time by segment | Compare dispatch estimates with GPS-derived inter-stop times                     |
| Time-of-day reliability                  | Calculate median and 90th percentile travel time by road segment and time bucket |
| Driver-independent delays                | Check whether all drivers are delayed on same segment                            |
| Day-of-week effect                       | Compare Monday–Friday patterns                                                   |
| Weather/event sensitivity                | Add external factors if available                                                |

The important distinction is between average travel time and reliable travel time. A route may look feasible using average times but fail regularly because the 90th percentile travel time is much higher during peak hours.

---

### 3.3 Traffic congestion patterns

Traffic root causes can be validated by combining GPS low-speed events with route timing.

Evidence:

| Pattern                                                         | Likely root cause                            |
| --------------------------------------------------------------- | -------------------------------------------- |
| Repeated low-speed events in same location and same time window | Predictable congestion                       |
| Late deliveries mostly after early congestion                   | Cascading delays                             |
| Drivers detour around same area                                 | Known traffic bottleneck or road restriction |
| High fuel/km in congestion-heavy routes                         | Stop-and-go driving and idling               |
| Overtime correlated with congestion exposure                    | Underestimated route duration                |

This supports dynamic routing, time-of-day-aware route planning, and revised customer time windows.

---

### 3.4 High variability in service time at customer locations

Customer service time is often a hidden last-mile bottleneck. The delivery itself may take 2 minutes for a house but 10–15 minutes for an apartment building, office tower, locked building, school, hospital, or commercial dock.

Analysis:

| Analysis                                | Insight                                       |
| --------------------------------------- | --------------------------------------------- |
| Dwell time by customer type             | Residential vs commercial vs apartment        |
| Dwell time by building/geofence         | Repeated difficult locations                  |
| Dwell time by parcel type               | Bulky, signature-required, fragile            |
| Time from arrival to first scan         | Parking/walking/access delay                  |
| Time from delivery success to departure | Admin or package sorting delay                |
| Long-tail dwell distribution            | A few stops may consume disproportionate time |

Root causes may include poor address instructions, lack of access codes, parking difficulty, recipient not ready, signature requirements, or driver searching for packages inside the van.

---

### 3.5 Vehicle breakdowns and maintenance needs

Maintenance logs should be linked to vehicle-day cases.

Analysis:

| Process mining view                                | Purpose                             |
| -------------------------------------------------- | ----------------------------------- |
| Routes with `Unscheduled Stop` or `Engine Warning` | Identify interruption patterns      |
| Vehicle-level breakdown frequency                  | Detect unreliable vehicles          |
| Breakdown rate after high-mileage periods          | Support predictive maintenance      |
| Maintenance event before/after route lateness      | Quantify service impact             |
| Repair type vs recurrence                          | Find unresolved mechanical problems |
| Fuel/km trend by vehicle over time                 | Detect deteriorating efficiency     |

A vehicle that has rising fuel consumption, longer idle issues, repeated warning events, and higher repair frequency should be prioritized for maintenance before it disrupts routes.

---

### 3.6 Driver behavior and skill differences

Driver behavior can affect fuel use, route adherence, service time, scan compliance, and customer interaction time. However, this analysis must adjust for route complexity.

Useful analyses:

| Driver-related metric                        | Interpretation                            |
| -------------------------------------------- | ----------------------------------------- |
| Service time after controlling for stop type | Customer handling efficiency              |
| Route adherence vs outcomes                  | Whether deviations are helpful or harmful |
| Idle time ratio                              | Potential fuel waste or operational delay |
| Harsh acceleration/braking, if available     | Fuel and maintenance impact               |
| Scan timing compliance                       | Data quality and process discipline       |
| Stops/hour normalized by route difficulty    | Productivity comparison                   |

The aim should be targeted coaching, not punitive ranking. For example, a driver with long post-delivery dwell time may need training in van organization or scanner workflow. A driver with low fuel efficiency may need eco-driving coaching.

---

### 3.7 Failed delivery attempts

Failed deliveries are expensive because they create rework. Process mining can follow a package across multiple attempts.

Analysis:

| Question                                             | Method                                           |
| ---------------------------------------------------- | ------------------------------------------------ |
| Which packages fail most often?                      | Package-level process variants                   |
| Which zones have high failure rates?                 | Geospatial failed-delivery heatmap               |
| Which time windows have low success?                 | Failure rate by hour and customer type           |
| Are failures clustered by customer segment?          | Residential apartments, offices, rural addresses |
| How much rework do failures create?                  | Link failed attempt to later successful attempt  |
| Do failed deliveries cause lateness for later stops? | Compare downstream delay after failed attempts   |

Potential root causes include poor customer availability, inaccurate delivery windows, no pre-delivery notification, wrong address data, access restrictions, and unsuitable route timing.

## 4. Data-Driven Optimization Strategies

Below are six concrete strategies. Speedy Parcels could implement them in phases, starting with the highest-impact route groups.

---

### Strategy 1: Dynamic, time-of-day-aware routing

**Targets:** Late deliveries, traffic delays, extra kilometers, overtime, fuel consumption.

**Root cause addressed:** Static route plans do not reflect actual congestion, road restrictions, and time-of-day travel variability.

**Process mining evidence:**
If the event log shows frequent `Low Speed Detected` events on specific corridors, high planned-vs-actual travel time gaps, repeated first-stop delays, and better outcomes when drivers deviate from the planned sequence, the dispatch routing logic is not aligned with reality.

**Recommended action:**
Use historical GPS and scanner data to build time-dependent travel time profiles by road segment, zone, and time bucket. Replace static average travel times with dynamic estimates using median, 80th percentile, or 90th percentile travel times depending on service-level requirements.

Routes should be recalculated or adjusted when:

| Trigger                         | Action                                |
| ------------------------------- | ------------------------------------- |
| Traffic delay exceeds threshold | Re-sequence remaining stops           |
| Customer time window at risk    | Prioritize time-sensitive stops       |
| Vehicle breakdown occurs        | Reassign remaining packages           |
| Driver falls behind plan        | Dispatch recommends recovery sequence |
| Major road congestion appears   | Avoid known hotspot                   |

**Expected KPI impact:**

| KPI                    | Expected direction                                                        |
| ---------------------- | ------------------------------------------------------------------------- |
| On-Time Delivery Rate  | Increase                                                                  |
| Average lateness       | Decrease                                                                  |
| Extra distance vs plan | Decrease                                                                  |
| Fuel per package       | Decrease                                                                  |
| Overtime minutes       | Decrease                                                                  |
| Route adherence        | May decrease initially, but controlled deviations should improve outcomes |

Important point: the goal is not blind adherence to the original route. The goal is better controlled execution against customer and cost objectives.

---

### Strategy 2: Redesign delivery territories and route sequences using historical performance

**Targets:** Uneven workload, overloaded routes, long inter-stop travel, high overtime, low vehicle utilization.

**Root cause addressed:** Territories or route clusters may have been designed geographically but not operationally. Two routes with the same number of stops may have very different service times, congestion exposure, parking difficulty, and failure rates.

**Process mining evidence:**
The event log may show that some routes consistently have higher dwell time, higher failed delivery rates, more traffic delay, or more overtime even when stop count is similar. Variant analysis may show that low-performing routes share certain zones, customer types, or road corridors.

**Recommended action:**
Rebalance delivery territories using actual historical workload, not just map distance.

Route planning should incorporate:

| Factor                                | Why it matters                                  |
| ------------------------------------- | ----------------------------------------------- |
| Actual service time by stop type      | Prevents underestimating dense apartment routes |
| Historical travel time by time of day | Avoids unrealistic sequences                    |
| Failed delivery probability           | Helps decide timing and notification strategy   |
| Parking/access difficulty             | Accounts for urban stop complexity              |
| Vehicle capacity and parcel volume    | Prevents overloaded vans                        |
| Driver shift limits                   | Reduces planned overtime                        |
| Customer time windows                 | Preserves punctuality                           |

Instead of assigning “35 stops” uniformly, route workload should be measured as expected route minutes:

> Expected route workload = depot time + travel time + service time + buffer + expected failure/rework time

This allows a dense urban route with 28 difficult stops to be treated as equivalent to a suburban route with 42 easier stops.

**Expected KPI impact:**

| KPI                       | Expected direction          |
| ------------------------- | --------------------------- |
| Route completion rate     | Increase                    |
| Overtime minutes          | Decrease                    |
| Stops per driver hour     | Stabilize                   |
| On-time rate by territory | More balanced               |
| Distance per package      | Decrease                    |
| Vehicle utilization       | Improve without overloading |

---

### Strategy 3: Improve time-window management and customer communication

**Targets:** Failed deliveries, missed time windows, re-delivery cost, customer dissatisfaction.

**Root cause addressed:** Customers may not be available when drivers arrive; delivery windows may be too broad, unrealistic, or poorly communicated; customer access information may be missing.

**Process mining evidence:**
Package-level process mining may show repeated variants such as:

> `Arrive Customer → Delivery Failed → Return Depot → Reassigned Next Day → Delivery Success`

If failed deliveries are concentrated in certain zones, times of day, customer types, or buildings, Speedy Parcels can act specifically instead of treating all failures equally.

**Recommended action:**
Use historical failed-delivery patterns to adjust delivery communication and time-window planning.

Concrete actions:

| Finding                                                  | Action                                                     |
| -------------------------------------------------------- | ---------------------------------------------------------- |
| High failure rate in residential areas during work hours | Offer evening or pickup-point options                      |
| High failure rate in apartment buildings                 | Request access code or safe-place instruction before route |
| Repeated failures for same customer                      | Require confirmation before dispatch                       |
| Commercial stops fail after business hours               | Schedule earlier in route                                  |
| High-value/signature parcels fail frequently             | Send tighter ETA notifications and confirmation links      |

Introduce customer notifications based on actual route progress:

> “Your parcel is expected between 10:40 and 11:10. Please confirm availability or choose an alternative.”

For at-risk deliveries, allow customers to redirect to lockers, pickup points, neighbors, or rescheduled windows before the driver arrives.

**Expected KPI impact:**

| KPI                          | Expected direction |
| ---------------------------- | ------------------ |
| Failed delivery rate         | Decrease           |
| First-attempt success rate   | Increase           |
| Re-delivery burden           | Decrease           |
| Cost per delivered package   | Decrease           |
| Customer satisfaction        | Increase           |
| Route completion reliability | Increase           |

### Strategy 4: Predictive maintenance based on route execution and vehicle behavior

**Targets:** Unscheduled stops, breakdown-related delays, high repair cost, fuel inefficiency.

**Root cause addressed:** Maintenance may currently be calendar-based or reactive instead of usage- and condition-based.

**Process mining evidence:**
If certain vehicles show rising fuel/km, repeated warning-light events, longer idle anomalies, more unscheduled stops, or recurrent repairs after similar mileage patterns, the process log can identify leading indicators of failure.

**Recommended action:**
Create a vehicle health model combining:

| Data                       | Use                                              |
| -------------------------- | ------------------------------------------------ |
| Kilometers driven          | Maintenance exposure                             |
| Engine hours               | Better than mileage for urban stop-start driving |
| Idle time                  | Wear and fuel impact                             |
| Low-speed/stop-start ratio | Urban stress indicator                           |
| Warning-light events       | Early failure signal                             |
| Repair history             | Recurring defect detection                       |
| Route interruption events  | Operational impact                               |
| Fuel/km trend              | Efficiency degradation                           |

Vehicles with high predicted failure risk should be scheduled for maintenance outside delivery peaks. High-risk vehicles should not be assigned to long or time-critical routes.

**Expected KPI impact:**

| KPI                                 | Expected direction |
| ----------------------------------- | ------------------ |
| Routes interrupted by vehicle issue | Decrease           |
| Maintenance downtime during shifts  | Decrease           |
| On-time delivery rate               | Increase           |
| Fuel consumption per km             | Decrease           |
| Repair cost volatility              | Decrease           |
| Fleet availability                  | Increase           |

---

### Strategy 5: Targeted driver coaching using normalized process performance

**Targets:** Long dwell times, high idle time, inefficient scan behavior, fuel consumption, avoidable route deviations.

**Root cause addressed:** Some inefficiencies may come from inconsistent work practices, unfamiliarity with routes, poor van organization, excessive idling, or inefficient customer-handling routines.

**Process mining evidence:**
Resource analysis may reveal that, after controlling for route type and package mix, some drivers have consistently longer customer dwell time, higher idle ratios, lower scan compliance, or higher fuel per kilometer.

**Recommended action:**
Create driver coaching profiles based on fair, normalized comparisons.

Examples:

| Observed pattern                                  | Coaching intervention                                 |
| ------------------------------------------------- | ----------------------------------------------------- |
| Long time from arrival to first scan              | Improve parking/access approach or parcel pre-sorting |
| Long time after delivery success before departure | Scanner/admin workflow training                       |
| High idling at stops                              | Eco-driving and engine-off policy                     |
| Frequent harmful route deviations                 | Route plan adherence coaching                         |
| Experienced drivers outperform plan               | Capture best practices and update dispatch logic      |
| Low scan compliance                               | Scanner process training                              |

The best drivers should be studied as positive deviants. Their successful route adaptations can become standard operating procedures.

**Expected KPI impact:**

| KPI                           | Expected direction |
| ----------------------------- | ------------------ |
| Average service time per stop | Decrease           |
| Idle time ratio               | Decrease           |
| Fuel per package              | Decrease           |
| Scan data completeness        | Increase           |
| Stops per driver hour         | Increase           |
| On-time rate                  | Increase           |

---

### Strategy 6: Depot departure and loading process optimization

**Targets:** Late first stops, cascading route delays, overtime, missed time windows.

**Root cause addressed:** Routes may start late because loading, package sorting, scanner preparation, vehicle availability, or dispatch assignment is delayed.

**Process mining evidence:**
If many late routes already show delay before `Depart Depot`, then downstream routing changes alone will not solve the problem. A common pattern may be:

> `Start Shift → Route Assigned → Long Depot Wait → Depart Depot Late → Late Stop 1 → Cascading Late Deliveries`

**Recommended action:**
Analyze the depot subprocess separately:

> `Route Assigned → Vehicle Available → Packages Loaded → Scanner Ready → Driver Ready → Depart Depot`

Then identify whether late departure comes from package sorting, route assignment timing, vehicle availability, driver readiness, or scanner/device issues.

Concrete improvements could include:

| Finding                                  | Action                                      |
| ---------------------------------------- | ------------------------------------------- |
| Packages not ready when driver starts    | Pre-sort by route before shift              |
| Drivers spend time searching parcels     | Load by stop sequence                       |
| Route assignment changes last minute     | Freeze route earlier or automate exceptions |
| Scanner/device problems                  | Pre-shift device check                      |
| Vehicle not available due to maintenance | Better fleet readiness planning             |

**Expected KPI impact:**

| KPI                   | Expected direction |
| --------------------- | ------------------ |
| Depot departure delay | Decrease           |
| First-stop lateness   | Decrease           |
| Cascading delay index | Decrease           |
| Overtime minutes      | Decrease           |
| On-time delivery rate | Increase           |

## 5. Operational Constraints and Continuous Monitoring

### 5.1 Accounting for operational constraints

The proposed strategies must be embedded into dispatch planning with real-world constraints. Otherwise, recommendations may be analytically attractive but operationally unusable.

#### Driver working hours

Routes should be planned using legal and contractual working limits, including breaks and maximum shift duration.

Planning should include:

| Constraint                | Process mining use                                    |
| ------------------------- | ----------------------------------------------------- |
| Maximum shift length      | Predict whether route will exceed shift               |
| Required breaks           | Insert breaks without violating time windows          |
| Overtime thresholds       | Flag high-risk route plans before dispatch            |
| Driver start/end location | Ensure feasible route start and return                |
| Skill/familiarity         | Assign difficult zones to trained drivers when needed |

A route should not be considered feasible unless it fits within working hours under realistic travel and service-time estimates, not just optimistic averages.

#### Vehicle capacity

Vehicle capacity should include parcel count, volume, weight, and special handling requirements.

Planning should consider:

| Constraint                                | Process mining use                                |
| ----------------------------------------- | ------------------------------------------------- |
| Parcel volume                             | Prevent overloaded vans                           |
| Parcel weight                             | Avoid unsafe or inefficient loading               |
| Stop sequence                             | Load van by delivery order                        |
| Vehicle type                              | Match vehicle to urban/suburban route             |
| Fuel range/electric charging, if relevant | Avoid route interruption                          |
| Maintenance status                        | Avoid assigning risky vehicles to critical routes |

Historical logs can show whether high load factors correlate with longer service times, late departures, or failed route completion.

#### Customer time windows

Time windows should be treated as hard or soft constraints depending on service promise.

Planning should consider:

| Constraint                       | Process mining use            |
| -------------------------------- | ----------------------------- |
| Requested delivery window        | Sequence stops by urgency     |
| Historical customer availability | Offer better delivery windows |
| Commercial opening hours         | Avoid failed attempts         |
| Residential availability         | Use evening or pickup options |
| High-priority customers          | Reserve buffer time           |
| Re-delivery risk                 | Confirm before dispatch       |

The route optimizer should use realistic travel-time percentiles to protect time-window commitments.

#### Maintenance and fleet availability

Vehicles with open maintenance risks should be excluded from long or critical routes.

Planning should consider:

| Constraint                       | Process mining use                              |
| -------------------------------- | ----------------------------------------------- |
| Scheduled maintenance            | Remove vehicle from dispatch pool               |
| Predicted failure risk           | Assign to shorter/non-critical route or inspect |
| Repair history                   | Avoid repeated disruption                       |
| Vehicle type and fuel efficiency | Match route profile                             |
| Spare vehicle availability       | Prepare contingency plan                        |

### 5.3 Monitoring governance and improvement cycle

Speedy Parcels should use a structured improvement cycle:

1. **Baseline current process performance**
   Measure current KPIs and process variants over the last six months.

2. **Identify top loss drivers**
   For example: traffic corridors, depot delays, failed delivery rework, vehicle interruptions.

3. **Prioritize by business impact**
   Estimate minutes, kilometers, fuel, overtime, and customer failures caused by each issue.

4. **Pilot optimization strategies**
   Test dynamic routing or territory redesign on selected routes before full rollout.

5. **Compare pilot vs control group**
   Use process mining to compare route variants and KPI changes.

6. **Standardize successful changes**
   Update dispatch rules, driver procedures, maintenance schedules, and customer communication workflows.

7. **Monitor continuously**
   Track whether improvements persist and whether new bottlenecks emerge.

## Summary of the recommended approach

Speedy Parcels should use process mining to connect planned operations with actual execution. The strongest approach is to build a multi-source event log that links vehicle-day routes, package-level delivery attempts, GPS-derived movement events, dispatch plans, and maintenance records.

The analysis should answer five practical questions:

| Question                                  | Process mining answer                                                                                                           |
| ----------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------- |
| What actually happens on delivery routes? | Process discovery and variant analysis                                                                                          |
| Where do delays and costs occur?          | Performance overlays, dwell analysis, geospatial bottlenecks                                                                    |
| Why do they occur?                        | Root cause analysis by route, driver, vehicle, customer, time, and location                                                     |
| What should be changed?                   | Dynamic routing, territory redesign, better time-window management, predictive maintenance, driver coaching, depot optimization |
| Are improvements working?                 | Continuous conformance and KPI dashboards                                                                                       |

The most important principle is to move beyond average route performance. Last-mile delivery is highly variable. A route’s success depends on time of day, customer availability, parking, traffic, vehicle reliability, driver process, package mix, and depot readiness. Process mining gives Speedy Parcels the evidence needed to make these factors visible, quantify their impact, and convert operational data into targeted improvements.
