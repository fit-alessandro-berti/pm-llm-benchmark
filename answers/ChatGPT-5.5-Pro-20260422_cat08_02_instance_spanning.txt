## 1. Identifying Instance-Spanning Constraints and Their Impact

The first step is to transform the event log from a case-centric order history into a **constraint-aware interval log**. The current log already contains the most important ingredients: case ID, activity, start and complete timestamps, resource/station, order priority, region, cold-packing requirement, hazardous-material flag, and lifecycle state.

For every order and activity, I would derive:

```text
activity_duration = COMPLETE timestamp - START timestamp
waiting_before_activity = START timestamp of activity - COMPLETE timestamp of previous activity
end_to_end_time = final completion timestamp - order received timestamp
```

Then I would enrich each event with the state of the rest of the system at that moment:

```text
cold stations busy at time t
standard stations busy at time t
hazardous orders currently in Packing or Quality Check
orders waiting for the same region batch
express orders waiting for or occupying the same resource
queue length by activity, resource type, order type, and region
```

This turns the log into something suitable for **object-centric process mining**, because the relevant objects are not only orders, but also packing stations, batches, regions, and regulatory capacity.

### 1.1 Shared Cold-Packing Stations

Cold-packing is an instance-spanning constraint because one order’s access to a cold station depends on other orders currently occupying those stations.

For each cold order, I would calculate:

```text
cold_packing_wait = Packing START - Item Picking COMPLETE
```

or, more generally:

```text
ready_for_packing_time = completion of previous required step
cold_packing_start_time = Packing START
cold_resource_wait = cold_packing_start_time - ready_for_packing_time
```

Then I would reconstruct the timeline of the five cold-packing stations:

```text
C1 occupied from order A Packing START to COMPLETE
C2 occupied from order B Packing START to COMPLETE
...
```

At every point in time, I would compute:

```text
cold_station_occupancy(t) = number of active cold-packing intervals at time t
cold_station_utilization(t) = cold_station_occupancy(t) / 5
```

A cold order’s wait can be attributed to cold-station contention when the order is ready for packing and all compatible cold stations are busy.

Key metrics:

| Metric                                                                  | Purpose                                                           |
| ----------------------------------------------------------------------- | ----------------------------------------------------------------- |
| Average, median, P90, P95 cold-packing wait                             | Measures typical and tail delay caused by cold-station contention |
| % of cold orders waiting while all cold stations are occupied           | Identifies whether the capacity limit is truly binding            |
| Cold-station utilization by hour/day                                    | Shows peak-period saturation                                      |
| Cold queue length and queue age                                         | Identifies backlog risk                                           |
| SLA miss contribution from cold-packing wait                            | Quantifies business impact                                        |
| Express cold order wait vs. standard cold order wait                    | Shows whether priority rules are working                          |
| Number of standard orders delayed or interrupted by express cold orders | Quantifies priority side effects                                  |

A useful diagnostic is to compare cold and non-cold packing waits after controlling for priority, arrival hour, order size, and seasonality. If cold orders have materially higher queue times while cold-station utilization is near 100%, the constraint is a true bottleneck.

---

### 1.2 Shipping Batches

Shipping batching creates a synchronization dependency between orders. An order may be ready for label generation, but the system may intentionally hold it until other orders for the same region are ready.

For each order, I would calculate:

```text
batch_wait = Shipping Label Generation COMPLETE - Quality Check COMPLETE
```

If the log contains batch identifiers, such as `System (Batch B1)`, I would explicitly reconstruct each batch:

```text
batch_first_ready_time = earliest QC COMPLETE among orders in batch
batch_last_ready_time = latest QC COMPLETE among orders in batch
batch_label_time = Shipping Label Generation timestamp
batch_formation_time = batch_last_ready_time - batch_first_ready_time
post_batch_processing_time = batch_label_time - batch_last_ready_time
```

For an individual order:

```text
individual_batch_wait = batch_label_time - order_QC_complete_time
```

If explicit batch IDs are missing, I would infer batches using destination region, close label-generation timestamps, shipment wave timing, and resource identifiers.

Key metrics:

| Metric                                                   | Purpose                                                             |
| -------------------------------------------------------- | ------------------------------------------------------------------- |
| Average, median, P90 batch wait by region                | Shows which regions create the most delay                           |
| Batch size distribution                                  | Identifies whether the company waits too long to fill large batches |
| Batch formation time                                     | Measures synchronization delay                                      |
| First-ready-to-label time                                | Captures how long early orders are held                             |
| Last-ready-to-label time                                 | Separates batching delay from system processing delay               |
| % of SLA misses where batch wait was a major contributor | Quantifies business impact                                          |
| Express orders included in standard batches              | Identifies priority-policy violations                               |
| Batch fill rate vs. delivery-time performance            | Measures route-efficiency tradeoff                                  |

This is especially important because shipping-label generation may appear slow in a normal process map, but the real issue may not be the label-generation activity itself. The delay may be the wait for other orders to join the batch.

---

### 1.3 Priority Handling for Express Orders

Priority handling creates dependencies because an express order may overtake, delay, or interrupt a standard order.

I would analyze this in three ways.

First, I would compare queue times:

```text
express_wait_before_activity
standard_wait_before_activity
```

by activity and resource type. A healthy priority system should reduce express waiting time without creating excessive standard-order tail delays.

Second, I would identify overtaking behavior. For two orders waiting for the same activity and compatible resource:

```text
Order A ready before Order B
Order B starts before Order A
```

If Order B is express and Order A is standard, this is a priority overtaking event.

Third, I would detect interruptions or preemptions. If the log has explicit pause/resume events, this is straightforward. If not, I would infer possible preemption from resource-level timelines. For example:

```text
Standard order starts Packing on Station S7
Express order later uses S7 before the standard order completes
Standard order completes only after the express order finishes
```

That pattern suggests a preemption or resource conflict, although it should be validated against operational knowledge.

Key metrics:

| Metric                                               | Purpose                                                 |
| ---------------------------------------------------- | ------------------------------------------------------- |
| Express queue time by activity                       | Measures whether priority orders are actually expedited |
| Standard queue time after express arrivals           | Measures side effects                                   |
| Number of overtaking events                          | Quantifies priority intervention frequency              |
| Number and duration of interruptions                 | Measures operational disruption                         |
| Standard-order delay attributable to express orders  | Quantifies fairness and throughput impact               |
| Express SLA improvement vs. standard SLA degradation | Measures net business value                             |
| Preemption recovery/setup time                       | Captures hidden cost of interrupting work               |

Priority handling should not be evaluated only by express-order performance. The key question is whether the express gains justify the delay, disruption, and variability imposed on standard orders.

---

### 1.4 Hazardous Material Regulatory Limit

The hazardous-material rule is a facility-wide capacity constraint:

```text
No more than 10 hazardous orders may be in Packing or Quality Check simultaneously.
```

For every hazardous order, I would create active intervals:

```text
hazardous active interval = 
    Packing START to Packing COMPLETE
    Quality Check START to Quality Check COMPLETE
```

Then I would compute:

```text
hazardous_active_count(t) =
    count of hazardous orders active in Packing or Quality Check at time t
```

The process is compliant when:

```text
hazardous_active_count(t) <= 10
```

The constraint is binding when:

```text
hazardous_active_count(t) = 10
```

and additional hazardous orders are ready to enter Packing or Quality Check but cannot start.

Key metrics:

| Metric                                                | Purpose                                               |
| ----------------------------------------------------- | ----------------------------------------------------- |
| % of time hazardous count is 10                       | Measures how often the regulatory limit is saturated  |
| Number and duration of compliance violations          | Confirms whether the rule is being respected          |
| Hazardous queue time before Packing and QC            | Measures operational impact                           |
| Cap-blocked waiting time                              | Separates regulatory delay from normal resource delay |
| Hazardous throughput per hour/day                     | Shows capacity impact                                 |
| Non-hazardous resource idle time caused by hazmat cap | Detects inefficient blocking                          |
| SLA miss contribution from hazardous cap waits        | Quantifies business impact                            |

A hazardous order may wait because no station is available, because no QC staff is available, because the hazardous cap is full, or because several of these are true at once. The analysis should therefore classify waiting by the actual blocking condition at each time interval.

---

### 1.5 Differentiating Within-Instance and Between-Instance Waiting

The central analytical challenge is distinguishing delay caused by the order itself from delay caused by other orders.

I would decompose each order’s lead time as follows:

```text
End-to-end time =
    active processing time
  + waiting for next activity
  + batching/synchronization waiting
  + regulatory/capacity blocking
  + interruption/recovery time
  + calendar/non-working time
  + unexplained delay
```

For each transition:

```text
previous activity COMPLETE -> next activity START
```

the order is not actively being processed. This is where most between-instance dependencies appear.

Examples:

| Delay Type                      | Signal in the Log                                                               |
| ------------------------------- | ------------------------------------------------------------------------------- |
| Within-instance long processing | Long START-to-COMPLETE duration for a single activity                           |
| Shared-resource wait            | Order is ready, but compatible resource is occupied by other orders             |
| Cold-station contention         | Cold order is ready, all cold stations are occupied                             |
| Batch wait                      | Order completed QC but label generation waits for other same-region orders      |
| Priority displacement           | Standard order ready earlier but express order starts first                     |
| Preemption                      | Standard order’s activity is paused or effectively extended by express work     |
| Hazardous cap wait              | Hazardous order ready for Packing/QC while hazardous active count is already 10 |

For intervals where multiple blockers apply, I would avoid assigning the whole delay to only one cause. Instead, I would time-slice the wait interval and label each segment:

```text
10:00-10:05 cold station unavailable
10:05-10:08 cold station available but hazmat cap full
10:08-10:11 waiting due to express priority
```

When this is not possible directly from the log, I would use counterfactual simulation:

```text
What would the order’s start time have been if the cold-station limit were relaxed?
What would it have been if batching were disabled?
What would it have been if express orders did not preempt standard orders?
```

That gives a more robust estimate of each constraint’s marginal impact.

## 2. Analyzing Constraint Interactions

The constraints should not be optimized independently. They interact in ways that can amplify delays or move bottlenecks from one part of the process to another.

### 2.1 Express Orders and Cold-Packing

An express order requiring cold-packing competes for the scarcest packing resource. If it is allowed to jump the queue, standard cold orders wait longer. This may be correct from an SLA perspective, but it can create a long cold queue, especially during peak demand.

Possible interaction pattern:

```text
Cold stations already near 100% utilization
Express cold orders arrive
Express orders overtake standard cold orders
Standard cold queue grows
Some standard orders miss SLA
Operators increase preemption
Preemption creates additional setup/recovery loss
Effective cold capacity falls further
```

This means a simple “always prioritize express” rule may be harmful. A better rule would prioritize express orders only when the SLA benefit exceeds the cost imposed on the queue.

---

### 2.2 Express Priority and Hazardous-Material Limits

If an express order is hazardous, priority handling cannot override the regulatory limit. The order may be urgent, but it still cannot enter Packing or QC when the hazardous active count is already 10.

This creates a hierarchy of constraints:

```text
Regulatory compliance > safety/resource compatibility > priority > batching efficiency
```

A dangerous optimization mistake would be to treat express priority as absolute. The scheduling rule must explicitly say:

```text
Express hazardous orders may be prioritized only among legally admissible starts.
```

---

### 2.3 Batching and Express Shipping

Batching improves route efficiency, but it can harm express-order performance if express orders are forced to wait for regional batch completion.

For example:

```text
Express order completes QC at 10:36
Region West batch is not closed until 10:55
Express order misses target despite fast Picking, Packing, and QC
```

In that situation, the bottleneck is not warehouse execution. It is the batching policy.

Express orders may need one of the following:

```text
batch bypass
smaller express-only batches
maximum wait cap
earlier label generation with later route assignment
```

---

### 2.4 Batching and Hazardous Orders

Hazardous orders going to the same region can create synchronization problems. Suppose several hazardous orders for the South region are intended to be shipped together. If some are delayed upstream by the hazardous cap, non-hazardous orders in the same regional batch may wait unnecessarily.

Interaction example:

```text
South region batch expects 12 orders
8 are ready after QC
4 are hazardous and blocked before QC by the hazmat cap
The 8 ready orders wait for the 4 blocked orders
Batch wait increases for all 12 orders
```

The batching rule should therefore consider the predicted readiness of remaining orders, not just the target batch size.

---

### 2.5 Cold-Packing and Hazardous-Material Limits

A cold hazardous order is constrained by both:

```text
cold station availability
hazardous active count
```

This creates dual-resource blocking. A cold station may be available, but the hazardous cap may be full. Or the hazardous cap may have room, but all cold stations may be occupied.

This matters because adding another cold station may not help if the hazardous cap is the real binding constraint during peak windows. Conversely, improving hazmat scheduling may not help if cold-station capacity remains saturated.

---

### 2.6 Why Interaction Analysis Matters

If the company optimizes one constraint in isolation, it may only shift the bottleneck.

Examples:

```text
Adding cold stations may reduce packing wait but increase QC congestion.
Speeding up QC may increase batch waiting.
Prioritizing express orders may improve express SLA but damage standard-order tail performance.
Larger batches may reduce shipping cost but increase end-to-end lead time.
Strict hazmat admission control may maintain compliance but leave cold stations idle.
```

Therefore, the goal should be system-level optimization:

```text
Minimize SLA misses and excessive waiting
while respecting resource capacity, priority rules, batching logic, and regulatory limits.
```

## 3. Developing Constraint-Aware Optimization Strategies

### Strategy 1: Dynamic Constraint-Aware Dispatching for Packing and QC

**Primary constraints addressed:**
Cold-packing capacity, express priority, hazardous-material limit, standard-order delay.

**Proposed change:**
Replace simple first-in-first-out or manual prioritization with a dynamic dispatching rule that decides which order should start next based on:

```text
order priority
time remaining until SLA deadline
required station type
hazardous-material status
current hazardous active count
cold-station availability
batch impact
predicted remaining processing time
queue age
```

A practical dispatching score could look like:

```text
dispatch_score =
    SLA urgency
  + express priority weight
  + queue aging weight
  + batch completion benefit
  - preemption penalty
  - regulatory infeasibility penalty
```

The rule should also include hard constraints:

```text
Do not start hazardous Packing/QC if hazardous active count would exceed 10.
Do not assign cold orders to standard stations.
Do not preempt a standard order unless the express SLA benefit exceeds the expected disruption cost.
Do not preempt an order that is almost finished unless absolutely necessary.
```

**How it uses data:**
The historical event log can estimate:

```text
expected processing time by order type and activity
expected wait by queue state
express SLA risk
standard-order delay caused by preemption
hazardous cap saturation patterns
cold-station demand by hour/day
```

The dispatching rule can then be tested using simulation before implementation.

**Expected outcomes:**

```text
Lower P90/P95 cold-packing wait
Reduced unnecessary standard-order interruptions
Better express SLA adherence
Maintained hazardous-material compliance
Lower variance in end-to-end fulfillment time
```

This strategy explicitly handles interdependencies because it does not prioritize express orders blindly. It prioritizes them subject to cold-station capacity, hazardous limits, and disruption cost.

---

### Strategy 2: Dynamic Shipping Batch Formation with Maximum Wait Caps

**Primary constraints addressed:**
Shipping batching delay, express-order delay, regional synchronization.

**Proposed change:**
Replace fixed or overly rigid regional batching rules with dynamic triggers.

A batch should close when one of the following is true:

```text
target batch size reached
maximum wait time reached
scheduled carrier cutoff is approaching
an express order in the batch is close to SLA risk
predicted next-order arrival is too late to justify waiting
```

For example:

```text
North standard batch:
    close at 20 orders or 15 minutes, whichever comes first

West express batch:
    close at 5 orders or 3 minutes, whichever comes first

Low-volume region:
    close at 8 orders or 10 minutes, whichever comes first
```

Express orders should either bypass standard batching or use smaller express-only micro-batches.

The company should also consider splitting the current “Shipping Label Generation” step into two logical steps:

```text
1. Generate order-level label immediately after QC
2. Assign order to route/shipment batch later
```

This decouples label generation from route optimization. Orders are no longer blocked unnecessarily at the label step while the system waits for a perfect batch.

**How it uses data:**
The log can be used to estimate:

```text
arrival rate of QC-completed orders by region
batch-size distribution
historical batch wait by region and hour
probability that another order for the same region will be ready soon
SLA risk caused by waiting for additional orders
shipping efficiency tradeoff by batch size
```

A predictive batching model can decide whether waiting is worth it.

**Expected outcomes:**

```text
Reduced batch waiting time
Fewer orders held after QC
Improved express SLA performance
Better control of the tradeoff between route efficiency and delivery speed
Lower end-to-end lead-time variability
```

This strategy directly addresses the fact that batch waiting is not caused by the individual order. It is caused by synchronization with other orders.

---

### Strategy 3: Cold-Packing Capacity Smoothing and Flexible Capacity

**Primary constraints addressed:**
Cold-packing resource contention, express cold-order delays, peak-season congestion.

**Proposed change:**
Use the process-mining analysis to identify when cold-packing demand exceeds capacity, then introduce flexible capacity during those periods.

Possible actions:

```text
Add temporary cold-packing stations during peak windows.
Convert selected standard stations into cold-capable stations during predictable peaks.
Pre-stage cold-pack materials before expected surges.
Cross-train staff to operate cold-packing stations.
Reserve one cold station dynamically for express cold orders only when express demand is high.
Create appointment-like release windows from Picking into Cold Packing to avoid flooding the queue.
```

The key is that capacity should not be added blindly. It should be based on observed saturation and predicted demand.

**How it uses data:**
From the log, I would identify:

```text
hours where all five cold stations are occupied
cold queue length by time of day
cold orders arriving by hour/day
express cold arrivals
cold station utilization
cold wait contribution to SLA misses
```

Then I would run capacity scenarios:

```text
current: 5 cold stations
scenario A: 6 cold stations during peak hours
scenario B: 5 stations + one flexible conversion station
scenario C: 5 stations + express reservation policy
scenario D: 6 stations + revised dispatching
```

**Expected outcomes:**

```text
Reduced cold-packing queue length
Lower tail waiting time for cold orders
Fewer express cold-order escalations
Lower need for disruptive preemption
Higher peak-season throughput
```

This strategy should be evaluated together with the hazardous-material rule. If many cold orders are also hazardous, adding cold stations will help only when the hazmat cap is not the binding constraint.

---

### Strategy 4: Hazardous-Material Token Control and Flow Smoothing

**Primary constraints addressed:**
Hazardous-material concurrency limit, compliance risk, resource idle time caused by regulatory blocking.

**Proposed change:**
Introduce a digital “hazmat token” system. There are exactly 10 tokens. A hazardous order must acquire a token before entering Packing or Quality Check and releases the token after completing Quality Check.

Rules:

```text
Hazardous order cannot start Packing without a token.
Token remains assigned through Packing and QC, or at least during the regulated active intervals.
Token is released immediately after the order exits the regulated zone.
Dispatching system prioritizes hazardous orders based on SLA risk and queue age.
```

The company should also smooth hazardous flow upstream. For example:

```text
avoid releasing too many hazardous orders from Picking into Packing at the same time
create controlled hazmat waves
pre-check documentation before the order occupies regulated capacity
prioritize short hazardous QC tasks when the cap is saturated
```

**How it uses data:**
The log can reveal:

```text
times when hazardous_active_count = 10
hazmat orders waiting while the cap is full
hazmat queue age
hazmat service durations in Packing and QC
whether QC or Packing consumes most regulated time
resources idle while hazmat orders are blocked
```

**Expected outcomes:**

```text
Zero regulatory violations
Reduced cap-blocked waiting time
Better predictability of hazardous-order flow
Lower chance of large hazardous backlogs
Improved utilization of Packing and QC resources
```

This strategy recognizes that regulatory capacity is a shared system-level resource, just like a station or worker.

---

### Strategy 5: Smarter Preemption Policy for Express Orders

**Primary constraints addressed:**
Priority handling, standard-order disruption, cold-resource contention.

**Proposed change:**
Formalize when express orders are allowed to interrupt standard orders.

Instead of:

```text
Always pause standard work when express arrives
```

use:

```text
Preempt only when express order is at material SLA risk
and the standard order is not close to completion
and the expected recovery/setup cost is acceptable
and the resource is truly the bottleneck.
```

Example preemption rules:

```text
Do not preempt if the standard order has less than 2 minutes of expected work remaining.
Do not preempt if another compatible resource will free within 3 minutes.
Do not preempt the same standard order more than once.
Do not preempt hazardous work unless safety rules explicitly allow it.
Prioritize express orders at queue boundaries rather than mid-activity when possible.
```

The event log should also be enhanced to capture:

```text
PAUSE
RESUME
PREEMPTED_BY
REASON_FOR_PREEMPTION
```

Without these lifecycle events, preemption analysis will remain partly inferential.

**How it uses data:**
The log can quantify:

```text
how much express orders gain from preemption
how much standard orders lose
how often interrupted orders later miss SLA
how much capacity is lost to setup/recovery
which activities are most sensitive to interruption
```

**Expected outcomes:**

```text
Express orders remain fast
Standard-order tail delays fall
Less operational churn
More stable station utilization
Fewer avoidable interruptions
```

This strategy is especially important for cold-packing, where preemption can have a high opportunity cost because the resource is already scarce.

## 4. Simulation and Validation

Before implementing changes, I would build a **constraint-aware discrete-event simulation** calibrated from the process-mining analysis.

The simulation should reproduce the current process first, then test alternative policies.

### 4.1 Baseline Simulation Model

The baseline model should include:

```text
order arrivals by time of day, weekday, season, and order type
activity sequences and routing probabilities
processing-time distributions by activity and order attributes
resource calendars and capacities
standard packing stations
cold-packing stations
QC staff capacity
shipping-label system behavior
regional batch formation rules
express priority rules
hazardous-material concurrency limit
```

Each simulated order should carry attributes:

```text
Order Type: Standard or Express
Destination Region
Requires Cold Packing: true/false
Hazardous Material: true/false
SLA deadline
```

### 4.2 Capturing the Instance-Spanning Constraints

The simulation must explicitly model the dependencies between orders.

For cold-packing:

```text
Cold orders can use only cold stations.
Only five cold stations are available unless a scenario changes capacity.
Orders queue when all compatible stations are occupied.
```

For batching:

```text
Orders wait in region-specific batch queues after QC.
Batch closes according to current or proposed policy.
Individual orders experience batch wait until the batch is released.
```

For priority:

```text
Express orders may overtake standard orders.
Preemption rules define whether an in-process standard order can be paused.
Preemption creates interruption and recovery/setup time if applicable.
```

For hazardous materials:

```text
A global counter tracks hazardous orders in Packing and QC.
A hazardous order cannot enter a regulated activity if the counter is already 10.
The counter is decremented when the order exits the regulated activity.
```

This is the most important part: the simulation must not treat each order independently. It must model shared resources, shared queues, shared batches, and shared regulatory capacity.

### 4.3 Calibration and Validation

I would calibrate the simulation using part of the three-month log and validate it against a holdout period.

Validation targets should include:

```text
end-to-end lead-time distribution
activity duration distributions
waiting-time distributions by activity
cold-station utilization
cold queue length
batch size distribution
batch wait distribution by region
express vs standard SLA performance
hazardous active count over time
frequency and duration of hazmat cap saturation
throughput per shift/day
```

The baseline simulation is credible only if it reproduces the actual observed bottlenecks. For example, if the real log shows cold-packing P95 wait of 28 minutes during peak periods, but the simulation shows only 5 minutes, then the cold-station model is missing something.

### 4.4 Scenario Testing

I would test each strategy independently and in combination.

Example scenarios:

```text
Scenario 0: Current process baseline
Scenario 1: Dynamic dispatching only
Scenario 2: Dynamic batching only
Scenario 3: Add one flexible cold station during peak hours
Scenario 4: Hazmat token control with flow smoothing
Scenario 5: Smarter express preemption policy
Scenario 6: Combined dispatching + dynamic batching + hazmat token control
Scenario 7: Combined strategy plus extra peak cold capacity
```

KPIs to compare:

```text
average end-to-end time
P90/P95 end-to-end time
SLA compliance rate
express SLA compliance
standard SLA compliance
throughput per day
cold-packing wait
batch wait
hazmat cap-blocked wait
resource utilization
queue length and queue age
number of preemptions
regulatory violations
shipping batch size/fill rate
```

The best strategy may not be the one that minimizes average lead time. The company should optimize for a balanced objective, such as:

```text
maximize SLA compliance
minimize P95 lead time
maintain route efficiency
avoid regulatory violations
avoid excessive standard-order degradation
```

### 4.5 Stress Testing

Because peak seasons are the main pain point, the simulation should include stress tests:

```text
20% higher order volume
higher express-order share
higher cold-order share
higher hazardous-order share
regional demand spikes
staff absenteeism
cold-station downtime
carrier cutoff compression
```

This prevents the company from adopting a policy that works in normal weeks but fails during peaks.

process-mining diagnostic dashboard
```

| WIP by process stage                   | Shows where orders accumulate                      |
| Waiting time vs. processing time split | Shows whether delays are operational or structural |

fewer repeated interruptions of the same order
```

The goal is not to eliminate priority handling. The goal is to make it economically and operationally justified.

## Final Recommendation

The warehouse should not treat the fulfillment process as a simple sequence of independent order cases. The main delays are caused by **shared constraints across multiple orders**: cold-station contention, batch synchronization, express-order priority effects, and the hazardous-material concurrency limit.

The recommended approach is:

```text
1. Reconstruct resource, batch, priority, and regulatory states from the event log.
2. Attribute waiting time to specific between-instance constraints.
3. Quantify each constraint’s impact on SLA, throughput, and tail latency.
4. Analyze interactions between constraints rather than optimizing them separately.
5. Test changes with constraint-aware discrete-event simulation.
6. Implement dynamic dispatching, dynamic batching, hazmat token control, smarter preemption, and targeted cold-capacity flexibility.
7. Monitor post-implementation with dashboards focused on both overall KPIs and constraint-specific behavior.
```

The biggest practical improvement will likely come from combining three changes: **dynamic dispatching**, **dynamic batch closure rules**, and **explicit hazardous-cap control**. Additional cold-packing capacity should be considered if simulation confirms that cold stations remain binding after better scheduling and batching policies are introduced.
