Below is a practical process-mining approach I would use for TechSolve.

**First, I would enrich the event log before any analysis.**  
The raw log is good, but to make resource assignment analysis actionable I would derive an **assignment episode table** from it. For each ticket and each assignment/reassignment, I would calculate:

- **Time to first touch** = first `Work Start` after creation
- **Queue delay after assignment** = `Work Start`  latest `Assign/Reassign`
- **Touch time** = `Work End`  `Work Start`
- **Escalation delay** = next `Assign/Work Start`  `Escalate`
- **# of reassignments**, **# of escalations**, **tiers involved**
- **Initial assignee**, **final resolver**, **final resolving tier**
- **SLA target and breach flag**
- **Assignment mode**: dispatcher / self-assigned / auto-assigned
- **Skill-fit flag**: exact match / adjacent match / mismatch / overqualified
- **Current workload at assignment time**: open tickets, active work, queue length, if available
- **Case complexity proxy**: priority, category, required skill, channel, time-of-day, number of steps, text features

If skill profiles change over time, I would use **effective-dated skills**, not a single static skill list.

## 1) Analyzing resource behavior and assignment patterns

### A. Analyze performance and behavior of agents and tiers

I would analyze at **three levels**: agent, tier, and skill pool.

#### Agent-level metrics
For each agent, I would measure:

- **Assigned volume**
  - tickets assigned
  - tickets actually touched
  - tickets resolved
- **Workload distribution**
  - average concurrent tickets
  - open assigned backlog
  - utilization = touch time / staffed time
  - queue ownership time
- **Speed**
  - median and P90 time to first touch
  - median touch time
  - median total resolution time
  - median delay after reassignment
- **Outcome quality**
  - SLA attainment %
  - reopen rate
  - escalation-out rate
  - reassignment-out rate
  - reassignment-in rate
- **L1-specific**
  - **first-call / first-tier resolution rate**  
    = tickets resolved by L1 without escalation / tickets first handled by L1
- **Case mix**
  - share of P1/P2/P3/P4
  - share by category
  - share by required skill

Important: I would **case-mix adjust** agent comparisons. A DB or firewall specialist should not be judged directly against an L1 generalist on raw handling time alone.

#### Tier-level metrics
For L1, L2, L3 I would measure:

- inbound ticket volume
- tickets resolved within tier
- escalation-in / escalation-out rate
- intra-tier reassignment rate
- median assign-to-start delay
- median touch time
- backlog age
- SLA attainment by priority and category
- work mix by required skill

For L1 specifically, I would focus on:

- first-tier resolution by category/skill/channel
- escalation rate by agent and by category
- whether L1 is escalating too early or too often
- whether some L1 agents already perform like an “L1+” specialist tier

### B. Reveal the *actual* assignment patterns using process mining

This is where **organizational process mining** becomes critical.

#### 1. Process discovery with performance overlays
I would discover the actual incident flow and color it by:

- waiting time
- touch time
- frequency
- SLA breach rate

This will show where time is lost:
- `Assign L1 -> Work L1 Start`
- `Escalate L2 -> Assign L2`
- `Assign L2 -> Work L2 Start`
- `Assign L2 -> Reassign -> Assign L2`

In your example, **INC-1001** clearly shows:
- a queue delay after L2 assignment
- then a later skill-based reassignment from App-CRM to Database-SQL

At scale, process mining will tell us whether that pattern is common or just anecdotal.

#### 2. Resource interaction analysis / social network analysis
I would build several resource networks:

- **Handover-of-work network**  
  Node = agent or tier; edge = one resource hands work to another  
  Weight edges by:
  - number of handovers
  - median waiting time after handover
  - SLA breach rate after that handover

- **Subcontracting network**  
  Reveals patterns like:
  - certain L1 agents always escalate to the same L2 specialist
  - certain L2 agents repeatedly pass specific categories to a small expert set

- **Working-together network**  
  Helps identify informal collaboration clusters or dependency on “go-to” people

These networks often reveal hidden operational realities:
- some specialists become bottlenecks
- certain dispatchers prefer a subset of agents
- some tickets “ping-pong” inside L2 before reaching the right person

#### 3. Role discovery
Formal tiers may not match real behavior. I would use **role discovery** to cluster agents based on actual work patterns:

- activities performed
- ticket categories handled
- required skills handled
- resolution success
- handover behavior

This may reveal actual roles like:
- “L2 CRM + DB troubleshooters”
- “L2 firewall specialists”
- “L1 access-management high performers”
- “L3 security escalation hub”

If the discovered roles differ from L1/L2/L3 structure, the current routing design is too coarse.

### C. Compare actual behavior with intended assignment logic

The intended logic appears to be:

- round-robin inside tiers
- manual escalation based on skill/complexity
- L1 handles initial triage/basic troubleshooting

I would test conformance against these design assumptions.

#### Conformance questions
- Are assignments truly round-robin within the eligible pool?
- Do tickets escalate only after meaningful L1 work?
- Are tickets assigned to agents who actually have the required skill?
- Are there many intra-tier reassignments that the designed logic never intended?
- Are self-assigned phone tickets behaving differently from dispatcher-assigned portal/email tickets?

#### Useful conformance metrics
- **Round-robin deviation**  
  actual assignments / expected assignments for each eligible agent
- **Initial routing accuracy**  
  % of tickets where first assignee had appropriate skill and no skill-related correction followed
- **Escalation rule adherence**  
  % of escalations preceded by expected L1 troubleshooting steps
- **Organizational conformance**  
  % of work executed by intended tier/role/skill pool

If “round-robin” is not happening, that could mean:
- manual bias,
- hidden specialization,
- or valid local workarounds that should be formalized.

### D. Analyze skill utilization across the agent pool

I would build a **required-skill x assigned-agent-skill matrix** and a **required-skill x tier** matrix.

#### Skill utilization metrics
For each skill (e.g., Networking-Firewall, Database-SQL, App-CRM, Security-IAM):

- ticket demand volume
- touch hours required
- queue hours accumulated
- SLA breach rate
- exact-match assignment rate
- adjacent-skill assignment rate
- mismatch rate
- number of distinct agents using that skill
- top-3-agent concentration for that skill

#### Are specialists used effectively?
I would quantify:

- **Specialist spillover / overqualification**
  - % of L2/L3 touch time spent on tickets historically resolvable by L1 or generic L2
- **Underqualification**
  - % of assignments where assignee lacked the required skill and ticket later bounced
- **Skill concentration risk**
  - If one or two people handle most of a skill’s volume, that is a single-point-of-failure

A strong way to detect “below skill level” work is to estimate the **historical minimal sufficient tier** for each ticket cluster.  
Example: if similar P3 Access Management tickets are usually resolved by L1, then L2/L3 time spent on them is likely capacity leakage.

## 2) Identifying resource-related bottlenecks and issues

Once the current-state patterns are visible, I would pinpoint issues by decomposing ticket delay into **assignment-related components**.

### A. Bottlenecks caused by scarce skills

For each required skill, I would compare:

- demand (tickets / touch hours)
- available capacity (qualified staffed hours)
- queue delay after assignment
- utilization of qualified agents
- breach rate

A skill bottleneck exists when you see a pattern like:

- high queue delay after assignment
- high utilization of qualified agents
- repeated overflow to non-specialists
- elevated SLA breaches for tickets needing that skill

Likely examples in TechSolve:
- Networking-Firewall
- Database-SQL
- Security-IAM
- niche application skills

### B. Delays introduced by reassignments and escalations

Not every handoff is bad. I would distinguish:

1. **Legitimate diagnostic progression**
   - prior assignee did meaningful work and discovered a deeper issue
2. **Routing correction**
   - reassignment happened with little/no real work because the ticket was assigned to the wrong person

That distinction matters.

#### Metrics
- % of tickets with 0 / 1 / 2+ reassignments
- median added delay per reassignment
- median added delay per escalation
- breach rate by number of handovers
- intra-tier vs inter-tier reassignment rate

A useful formula:

- **Added delay of reassignment**  
  = median(`next Work Start - Reassign timestamp`)

And also:

- **Impact score**  
  = frequency × added delay × SLA sensitivity

That highlights high-volume moderate problems, not just rare extreme ones.

### C. Impact of incorrect initial assignments

I would analyze:

- tickets whose initial assignee lacked the needed skill
- tickets whose required skill later changed
- tickets that end up resolved by a very different role than the initial assignee

For example, if many **Software-App / App-CRM** tickets later move to **Database-SQL**, that suggests the initial skill identification is too coarse.

#### Metrics
- initial routing accuracy %
- eventual-skill-switch %
- % of breached tickets with initial skill mismatch
- % of reassigned tickets where mismatch was already inferable from intake data

### D. Overloaded vs underutilized agents and teams

I would identify overload using a combination of:

- utilization
- active WIP
- queue backlog
- inbound handovers
- handover centrality in the resource network

And underutilization using:

- low touch time
- low assigned volume
- available qualified capacity while tickets wait elsewhere

#### Workload balance metrics
- coefficient of variation or Gini index across eligible agents
- actual-to-expected assignment ratio under round-robin
- “queued tickets while qualified agents idle” events

### E. Correlation between assignment patterns and SLA breaches

This is one of the most important analyses.

I would compare **breached vs non-breached tickets** on:

- initial skill match
- number of handovers
- queue delay after assignment
- escalation path
- assignee utilization at assignment time
- category / required skill
- channel
- shift/day/time
- assignment mode (dispatcher vs self-assigned)

#### Quantify the impact
Examples of measures I would report:

- **% of breaches linked to at least one reassignment**
- **% of breaches linked to initial skill mismatch**
- **breach uplift**  
  `P(breach | reassignment)` minus `P(breach | no reassignment)`
- **breach odds ratio** for tickets assigned when target skill pool utilization is above threshold
- **share of total cycle time spent waiting between assignments and starts**

One caution: exclude or separately tag delays unrelated to assignment, such as waiting on customer, vendor, or approvals, so assignment problems are not overstated.

## 3) Root cause analysis for assignment inefficiencies

After the symptoms are quantified, I would look for root causes in four broad categories.

### A. Deficiencies in current assignment rules

This is likely the first major cause.

A pure or semi-pure **round-robin by tier** tends to ignore:

- skill fit
- actual workload
- estimated start time
- SLA urgency
- historical success rate
- time-of-day coverage

Typical consequences:
- tickets land with the wrong specialist
- specialists get general work
- queueing occurs behind overloaded agents while other eligible agents are free

### B. Inaccurate or incomplete skill profiles

Some organizations have documented skills that are stale or too generic.

Symptoms:
- hidden experts receive many manual assignments despite not being tagged for that skill
- tickets bounce because the “official” assignee pool is wrong
- specialists are overloaded because only a few people are formally recognized

A good practice is to compare:
- **declared skills**
with
- **empirically demonstrated skills from the log**

Then validate with team leads.

### C. Poor ticket categorization / weak skill requirement identification

This is probably another major root cause.

Examples:
- category = Software-App, but the true issue is DB-related
- category = Network, but the real skill needed is Firewall/VPN
- category = Access Management, but issue is IAM workflow/configuration

In your example, **INC-1001** looks exactly like a case where the initial requirement may have been too broad.

Channel also matters:
- web/email tickets may have incomplete data
- phone tickets may get better live triage but may also be self-assigned inconsistently

### D. Lack of real-time visibility into workload and availability

Even good skill logic fails if assignment does not consider:

- who is already overloaded
- who is actually on shift
- who has paused work
- who is handling a long-running case
- queue length for each skill pool

This creates invisible bottlenecks and explains why some agents are overloaded while others look idle.

### E. Insufficient L1 training / empowerment / diagnostic discipline

If L1 lacks:
- troubleshooting scripts
- knowledge articles
- authority to resolve common cases
- skill-specific triage guidance

then L1 will over-escalate.

A related issue is **immature escalations**:
- L1 escalates before collecting enough diagnostics
- L2 then repeats basic troubleshooting
- specialists spend time on work that should have been done earlier

### F. How variant analysis helps

I would explicitly compare:

#### Smooth variants
Example:
`Ticket Created -> Assign L1 -> Work L1 -> Resolved`

versus

#### Friction variants
Example:
`Ticket Created -> Assign L1 -> Work L1 -> Escalate L2 -> Assign L2 -> Work L2 -> Reassign -> Assign L2 -> Work L2 -> Close`

Then I would compare these groups on:

- priority
- category
- required skill
- channel
- time/day/shift
- initial assignee
- dispatcher
- assignment mode
- text features from ticket notes
- workload at assignment time

This tells us what differentiates “clean” from “problematic” routing.

### G. How decision mining helps

I would apply **decision mining** at the key routing points:

1. **After ticket creation**  
   What factors lead to assignment to a given tier/agent/pool?
2. **After L1 work**  
   What drives resolve vs escalate?
3. **After L2 assignment**  
   What predicts a later reassignment?
4. **Before SLA breach**  
   What assignment conditions most strongly predict late resolution?

Decision trees or rule mining can expose practical drivers such as:

- category + keyword pattern predicts later DB reassignment
- tickets assigned during certain shifts are more likely to wait
- certain dispatchers have higher misrouting
- high workload at assignment time predicts delayed start
- some L1 agents escalate categories that peers resolve successfully

That gives you evidence for whether the problem is:
- routing logic,
- data quality,
- staffing/capacity,
- or capability/training.

## 4) Developing data-driven resource assignment strategies

The target design should be:

> **Route to the lowest sufficient skill level, with the best expected start time, under SLA constraints.**

These strategies are complementary.

### Strategy 1: Skill- and proficiency-based routing

**Issue addressed**  
Incorrect initial assignments, frequent within-tier reassignments, and specialists receiving mismatched work.

**How it uses mining insights**  
Use the mined **required-skill vs actual-resolver** patterns and role discovery results to define **virtual skill pools** rather than broad tier pools.  
Example: instead of a generic L2 pool, create pools such as:
- L2 Network-Firewall
- L2 App-CRM
- L2 Database-SQL
- L2 Security-IAM

Use **proficiency-weighted eligibility**:
- exact skill match first
- adjacent backup skill second
- overqualified specialist only when needed by SLA risk or no qualified alternative exists

Round-robin should become only a **final tie-breaker**, not the main rule.

**Data required**
- cleaned skill taxonomy
- agent skill profiles and proficiency levels
- historical resolution success by skill/category
- ticket category, required skill, text-derived features
- tier eligibility rules

**Expected benefits**
- fewer reassignment loops
- faster start after assignment
- better specialist utilization
- improved SLA for P2/P3
- more consistent routing behavior

---

### Strategy 2: Workload-aware, SLA-aware dispatching

**Issue addressed**  
Uneven workload, overloaded experts, delayed work starts, and mid-priority tickets aging in queue.

**How it uses mining insights**  
Use event-log-derived workload measures to assign based on:
- skill fit
- current WIP
- estimated start time
- current queue length
- SLA remaining time
- historical success for that ticket type
- overqualification penalty

This is better than assigning to the “next” person in a round-robin sequence.

I would also add **dynamic cross-pool rebalancing**:
- when specific skill queues spike, temporarily flex cross-trained agents
- reserve scarce specialist capacity for high-risk/high-value cases
- prevent P3 starvation when P1/P2 bursts occur

**Data required**
- real-time or near-real-time workload
- shift calendars and availability
- backlog by skill pool
- SLA clocks
- expected handling times from historical data
- cross-skill eligibility matrix

**Expected benefits**
- reduced assign-to-start delay
- more even workload distribution
- fewer artificial bottlenecks
- better P2/P3 SLA compliance
- less overload on “go-to” specialists

---

### Strategy 3: Predictive triage at intake

**Issue addressed**  
Poor initial categorization and late discovery of true required skill/complexity.

**How it uses mining insights**  
Train a model using historical ticket data to predict:
- likely required skill
- likely resolving tier
- probability of L1 resolution
- expected complexity / handling time
- risk of reassignment or breach

Use structured attributes plus text mining on:
- subject/description
- notes
- category/subcategory
- channel
- affected application/CI
- prior similar incidents

Where model confidence is low, ask more triage questions at intake.  
Example: if “CRM” tickets often later require DB-SQL, add intake questions about query errors, integrations, or database connectivity.

**Data required**
- historical tickets with final resolving skill/tier
- description/notes text
- channel and category metadata
- outcomes: resolved tier, reassignments, breaches
- linked system/application data if available

**Expected benefits**
- more accurate first assignment
- fewer skill-correction reassignments
- faster routing for complex tickets
- better use of L1 for genuinely resolvable cases
- improved consistency across phone/email/portal channels

---

### Strategy 4: Evidence-based escalation redesign and L1/L1+ enablement

**Issue addressed**  
Excessive escalations, low-quality handoffs, and specialists doing basic work.

**How it uses mining insights**  
From historical data, identify:
- categories with high L1 success potential
- L1 agents with strong performance in certain domains
- escalations that add little value before specialist involvement
- diagnostic steps that are often skipped before escalation

Then redesign escalation rules:

- create category-specific **escalation criteria**
- require minimum diagnostic completeness for escalation
- create **L1+ pods** for high-volume, low-to-medium complexity specialties
- target training where one L1 group escalates cases that peers resolve

Also track whether escalated tickets are resolved in the next tier without further bounce; that is a good handoff quality metric.

**Data required**
- L1 resolution rates by category/skill/agent
- escalation reasons and notes
- knowledge article usage if available
- troubleshooting steps performed
- training/certification data
- reopen and post-escalation reassignment data

**Expected benefits**
- higher L1 first-tier resolution
- lower L2/L3 workload on routine cases
- better handoff quality
- fewer “specialist redoes basic checks” situations
- lower cost per resolved ticket

## 5) Simulation, implementation, and monitoring

### A. Use business process simulation before implementation

I would build a **discrete-event simulation / digital twin** from the mined process.

#### Simulation inputs
- discovered control-flow model
- case arrival patterns by hour/day/channel/priority/category
- service time distributions by activity/tier/skill
- escalation and reassignment probabilities
- resource calendars and shift patterns
- skill pools and cross-skill eligibility
- current and proposed assignment rules
- SLA policies

#### Scenarios to test
- current baseline
- skill-based routing only
- workload-aware dispatch only
- predictive triage + skill routing
- L1+ enablement
- adding/removing capacity in scarce skill pools
- peak-demand scenarios
- absenteeism / reduced availability scenarios

#### What to evaluate
- SLA attainment by priority
- median and P90/P95 resolution time
- assign-to-start delay
- reassignment rate
- L1 first-tier resolution
- utilization by tier and skill pool
- specialist time spent on low-level work
- queue lengths and backlog aging

Simulation is valuable because some changes can shift the bottleneck.  
Example: improving L1 resolution may help L2, but could overload L1 unless staffing and routing are adjusted.

### B. Practical implementation plan

I would implement in stages:

1. **Data cleanup and baseline**
   - standardize skill taxonomy
   - validate effective skill profiles
   - build current-state dashboards
   - establish baseline KPIs

2. **Shadow mode**
   - run the new routing logic in parallel
   - compare recommended assignee vs actual assignee
   - capture override reasons

3. **Pilot**
   - choose 1–2 high-impact domains, such as
     - Network/Firewall
     - App/CRM + DB
   - focus on P2/P3 where pain is highest

4. **Controlled rollout**
   - expand by category/skill/shift
   - compare with control groups or prior matched periods to avoid seasonality bias

5. **Governance**
   - monthly skill-profile review
   - quarterly routing-rule review
   - retrain predictive models as patterns drift

### C. Post-implementation monitoring with process mining dashboards

I would monitor at three levels: **operational, tactical, strategic**.

#### Core KPIs to track continuously

**Outcome KPIs**
- SLA attainment % by priority/category/skill/tier
- median and P90 resolution time
- time to first touch
- backlog aging

**Routing quality KPIs**
- initial routing accuracy
- exact skill-match rate
- reassignment rate
- avoidable reassignment rate
- escalation rate by L1/category/agent
- average delay introduced by reassignment/escalation
- % of tickets resolved in next tier after escalation

**Resource KPIs**
- utilization by agent/tier/skill pool
- workload balance index across eligible agents
- queued tickets while qualified capacity exists
- inbound handover load per agent
- queue length by skill and hour

**Skill utilization KPIs**
- demand vs capacity by skill
- skill concentration risk
- specialist spillover rate
- minimal-sufficient-tier mismatch rate

**Conformance/adoption KPIs**
- adherence to new routing rules
- manual override rate
- outcome of overrides vs system recommendation
- prediction accuracy / drift for triage model

#### Process mining views I would include
- **performance-colored process map**
- **variant explorer** for breach vs non-breach paths
- **handover network / social network graph**
- **required-skill x assigned-skill heatmap**
- **tier transition Sankey**
- **backlog and delay heatmaps by hour/day**
- **agent/tier workload heatmaps**
- **breach drill-down view** from SLA breach to exact path and assignee history

#### Monitoring cadence
- **Daily operations:** backlog, queue delay, hot skill pools, overloaded agents
- **Weekly tactical review:** reassignment patterns, L1 escalation quality, routing accuracy
- **Monthly improvement review:** structural bottlenecks, skill gaps, training priorities, rule tuning

## Closing recommendation

TechSolve should move away from a mostly **tier-based round-robin model** and toward a **skill-, workload-, and SLA-aware routing model**.

The event log is rich enough to answer four critical questions with evidence:

1. **Who is actually doing what?**
2. **Where are tickets waiting because of assignment decisions?**
3. **Which reassignments are necessary vs avoidable?**
4. **How should routing change so the lowest sufficient skill handles each ticket at the right time?**

If done well, this will directly reduce:
- P2/P3 SLA breaches
- reassignments and queue hopping
- unnecessary specialist workload
- workload imbalance across the desk

while increasing:
- L1 first-tier resolution
- specialist effectiveness
- routing consistency
- end-to-end incident throughput.
