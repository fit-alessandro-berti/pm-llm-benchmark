# Temporal Profile Analysis: Insurance Claim Handling Process

## 1. Identified Anomalies

After reviewing the temporal profile model, the following pairs display suspicious timing characteristics:

| Activity Pair | Avg Time | STDEV | Observation |
|---------------|----------|-------|-------------|
| **R  P** | ~25 hours | 1 hour | Very low STDEV relative to the mean — unnaturally consistent |
| **P  N** | 7 days | 2 days | Excessive average delay with high variability |
| **A  C** | 2 hours | 1 hour | Closure is suspiciously fast after assignment, bypassing intermediate activities |
| **E  N** | 5 minutes | 1 minute | Transition is too quick — little room for approval step in between |
| **E  C** | 1 hour | 50 min | Short span that may skip Notify step |
| **R  A** | 1 hour | 10 min | Tight STDEV could reflect automated/batched assignment |

---

## 2. Hypotheses on Root Causes

### a) R  P: Artificial/Scheduled Approvals
- Approvals may be run as **batch jobs** at fixed intervals (e.g., nightly), producing near-constant elapsed times.
- An automated rule-based approval engine could be bypassing human review for certain claim types.
- A compliance SLA may force approvals within a strict window, causing timestamps to cluster.

### b) P  N: Notification Bottleneck
- A **backlog in the notification queue** (e.g., mail-out batches, customer service workload).
- Manual intervention required before notification (e.g., supervisor sign-off).
- Resource constraints — insufficient staff to dispatch notifications on approved claims.
- Possible system integration issue with the notification subsystem (email/SMS gateway).

### c) A  C: Premature Closures
- Claims closed without being fully evaluated/approved — possibly **rejections, duplicates, or fraud flags**.
- An adjuster shortcut where certain low-value claims are auto-closed.
- Data entry errors where close events are logged in error.

### d) E  N: Skipped Approval Step
- The Approval (P) step may be bypassed for specific claim types.
- Evaluator may possess delegated approval authority, collapsing E, P, and N into one action.
- Possible automation that fires N immediately after E without awaiting P.

### e) R  A: Automated Assignment
- Auto-assignment rules (round-robin or region-based) produce near-constant elapsed times.

## 3. Verification SQL Queries

### Query 1 — Identify claims with unusual R  P intervals
```sql
WITH pairs AS (
  SELECT r.claim_id,
         r.timestamp AS r_ts,
         p.timestamp AS p_ts,
         EXTRACT(EPOCH FROM (p.timestamp - r.timestamp)) AS secs
  FROM claim_events r
  JOIN claim_events p
    ON r.claim_id = p.claim_id
   AND r.activity = 'R' AND p.activity = 'P'
)
SELECT claim_id, r_ts, p_ts, secs
FROM pairs
WHERE ABS(secs - 90000) > 3 * 3600   -- Beyond ZETA=3 deviation
ORDER BY secs DESC;
```

### Query 2 — Detect claims closed immediately after assignment (possible premature closure)
```sql
SELECT a.claim_id,
       a.resource  AS assigner,
       c.resource  AS closer,
       EXTRACT(EPOCH FROM (c.timestamp - a.timestamp)) AS secs_to_close
FROM claim_events a
JOIN claim_events c
  ON a.claim_id = c.claim_id
 AND a.activity = 'A' AND c.activity = 'C'
WHERE NOT EXISTS (
    SELECT 1 FROM claim_events e
    WHERE e.claim_id = a.claim_id
      AND e.activity IN ('E','P')
      AND e.timestamp BETWEEN a.timestamp AND c.timestamp
)
ORDER BY secs_to_close ASC;
```

### Query 3 — Find excessive P  N delays and correlate with resources
```sql
SELECT p.claim_id,
       p.resource AS approver,
       n.resource AS notifier,
       EXTRACT(EPOCH FROM (n.timestamp - p.timestamp))/86400 AS days_delay
FROM claim_events p
JOIN claim_events n
  ON p.claim_id = n.claim_id
 AND p.activity = 'P' AND n.activity = 'N'
WHERE EXTRACT(EPOCH FROM (n.timestamp - p.timestamp)) > 604800 + 3*172800
ORDER BY days_delay DESC;
```

### Query 4 — E  N too quick (possibly skipping P)
```sql
SELECT e.claim_id,
       EXTRACT(EPOCH FROM (n.timestamp - e.timestamp)) AS secs,
       EXISTS (
         SELECT 1 FROM claim_events p
         WHERE p.claim_id = e.claim_id
           AND p.activity = 'P'
           AND p.timestamp BETWEEN e.timestamp AND n.timestamp
       ) AS has_approval_between
FROM claim_events e
JOIN claim_events n
  ON e.claim_id = n.claim_id
 AND e.activity = 'E' AND n.activity = 'N'
WHERE EXTRACT(EPOCH FROM (n.timestamp - e.timestamp)) < 60
ORDER BY secs ASC;
```

### Query 5 — Correlate anomalies with adjusters, regions, and claim types
```sql
WITH anomaly_claims AS (
  SELECT p.claim_id,
         EXTRACT(EPOCH FROM (n.timestamp - p.timestamp)) AS pn_gap
  FROM claim_events p
  JOIN claim_events n
    ON p.claim_id = n.claim_id
   AND p.activity = 'P' AND n.activity = 'N'
  WHERE EXTRACT(EPOCH FROM (n.timestamp - p.timestamp)) > 604800 + 3*172800
)
SELECT c.claim_type,
       a.region,
       a.specialization,
       COUNT(*) AS anomaly_count,
       AVG(ac.pn_gap)/86400 AS avg_days_delay
FROM anomaly_claims ac
JOIN claims c          ON c.claim_id = ac.claim_id
JOIN claim_events ae   ON ae.claim_id = ac.claim_id AND ae.activity = 'A'
JOIN adjusters  a      ON a.adjuster_id::text = ae.resource
GROUP BY c.claim_type, a.region, a.specialization
ORDER BY anomaly_count DESC;
```

### Query 6 — Detect batch-processed approvals (clustered timestamps)
```sql
SELECT DATE_TRUNC('hour', timestamp) AS approval_hour,
       COUNT(*) AS approvals,
       COUNT(DISTINCT resource) AS distinct_resources
FROM claim_events
WHERE activity = 'P'
GROUP BY DATE_TRUNC('hour', timestamp)
HAVING COUNT(*) > 20
ORDER BY approvals DESC;
```

### Query 7 — Resource concentration across anomalous transitions
```sql
SELECT resource, activity, COUNT(*) AS fast_transitions
FROM claim_events ce
WHERE EXISTS (
   SELECT 1 FROM claim_events n
   WHERE n.claim_id = ce.claim_id
     AND ce.activity = 'E' AND n.activity = 'N'
     AND EXTRACT(EPOCH FROM (n.timestamp - ce.timestamp)) < 60
)
GROUP BY resource, activity
ORDER BY fast_transitions DESC;
```

## Summary

The model exhibits two opposing symptoms: **over-regularity** (RP, EN, RA) suggesting automation or batch scheduling, and **over-dispersion** (PN) suggesting a resource bottleneck. Additionally, **AC** and **EN** pairs suggest steps are being skipped. The proposed queries let analysts pinpoint offending claims, attribute anomalies to specific adjusters/regions/claim types, and distinguish automated from manual effects.
