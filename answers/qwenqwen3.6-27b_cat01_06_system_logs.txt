###  Explanation of Transformation Logic

#### 1. Case Identification Logic
Cases were grouped by **primary work artifact** (document, email thread, or spreadsheet) rather than by continuous time blocks. This approach aligns with standard process mining practice where a "case" represents a distinct task or business object being processed. By assigning each document/email to a unique `Case ID`, we capture the true lifecycle of individual work items despite the user's interleaved multitasking:
- **Case_001**: `Document1.docx`  Represents the drafting and finalization of a specific document.
- **Case_002**: `Email - Inbox`  Represents handling a single email thread about the Annual Meeting.
- **Case_003**: `Report_Draft.pdf`  Represents reviewing and annotating a PDF report.
- **Case_004**: `Budget_2024.xlsx`  Represents updating a financial spreadsheet.
- **Case_005**: `Quarterly_Report.docx`  Represents opening, editing, and closing the main quarterly report.

This grouping allows process mining algorithms to reconstruct the full lifecycle of each artifact, measure cycle times, and identify how tasks are sequenced or paused in favor of other work items.

#### 2. Activity Naming & Standardization
Low-level system actions (`FOCUS`, `TYPING`, `CLICK`, `SCROLL`, `SAVE`, `CLOSE`) were mapped to **higher-level process steps** that reflect user intent and workflow phases:
- **Document/Spreadsheet Cases**: `Open Draft/Report/Spreadsheet`  `Draft Content/Update Data`  `Save Draft/Spreadsheet`  `Close Draft/Report`
- **Email Case**: `Open Email`  `Review Email`  `Draft Reply`  `Send Reply`
- **PDF Case**: `Open Report`  `Review & Annotate`

This abstraction removes noise from repetitive UI interactions and highlights meaningful process milestones. For example, multiple `TYPING` or `SCROLL` events were consolidated into single activity instances (`Draft Content`, `Update Data`, `Review Email`) to reflect the logical step rather than granular keystrokes or mouse movements.

#### 3. Handling of `SWITCH` Events
Context-switch events were **excluded** from the event log. In process mining, switches represent navigation between concurrent cases rather than progress within a case. Including them as activities would fragment process flows and distort performance metrics. Instead, the temporal gaps between events naturally capture multitasking behavior, which can be analyzed using **interleaving-aware process discovery** or **resource utilization metrics** (e.g., how long a document was left open before returning to it).

#### 4. Coherent Narrative & Analyst Utility
The transformed log tells a clear story of a **report preparation workflow** with parallel task handling:
1. The user opens the main quarterly report (`Case_005`) but immediately pivots to draft a supporting document (`Case_001`).
2. After an initial save, they handle an email (`Case_002`), review a reference PDF (`Case_003`), and update financial data (`Case_004`).
3. They return to `Document1.docx` to incorporate new information, save, and close it.
4. Finally, they reopen the quarterly report, draft an executive summary, save, and close it, completing the session.

This structure enables analysts to:
- Measure **cycle time** per document type.
- Identify **process variants** (e.g., drafting  saving  switching  returning  finalizing).
- Analyze **multitasking patterns** and **context-switching costs** using timestamps.
- Map the log directly into tools like **Celonis, Disco, or ProM** for conformance checking, bottleneck detection, or resource allocation analysis.

The event log preserves temporal fidelity while abstracting raw system actions into business-meaningful steps, making it ready for standard process mining workflows.
