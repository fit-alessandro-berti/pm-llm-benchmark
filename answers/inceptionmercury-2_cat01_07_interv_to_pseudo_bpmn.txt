**BPMNlike Textual Model – “InvoicetoPayment” Process**  

Below is a stepbystep, roleoriented description that can be copied directly into a BPMNtool (e.g., Camunda Modeler, Signavio) without ambiguity.  
Each line shows **[Role]Task**, the **sequence flow arrow ()**, and any **gateway** (diamond) with its condition.  
Artifacts that are created, read or passed are listed in **italics**.

### 2. Artifacts (Data Objects)

- **Invoice** (received by email)  
- **Purchase Order (PO)** (reference number)  
- **Receipt Confirmation** (goods/services received)  
- **Matching Report** (invoicePO comparison)  
- **Discrepancy Notification** (email / system message)  
- **Approval Request** (automatic email from system)  
- **Approved Invoice** (status = “Approved”)  
- **Payment Schedule** (batch run)

### 3. Process Flow (Numbered for reference)

| # | Flow (BPMNstyle) |
|---|-------------------|
| **Start** | **Start Event** “Invoice Received (email)” |
| 1 | **[Mary]** *Check PO number on Invoice*  |
| 2 | **GatewayG1** – *PO present?*  <br>• **Yes**  3  <br>• **No**  4 |
| 3 | **[Mary]** *Log Invoice into Accounting System*  |
| 4 | **[Mary]** *Send “Missing PO” request to Supplier*  |
| 5 | **[Mary]** *Wait for Supplier to resend Invoice (loop back to 1)*  |
| 6 | **[Mary]** *Forward logged Invoice to Purchasing for receipt confirmation*  |
| 7 | **[Purchasing – Dan]** *Validate Goods/Services Receipt*  |
| 8 | **GatewayG2** – *Receipt confirmed?*  <br>• **Yes**  9  <br>• **No**  10 |
| 9 | **[Mary]** *Match Invoice line items against PO (systemgenerated Matching Report)*  |
| 10 | **[Dan]** *If receipt not confirmed, request proof from supplier / hold invoice*  |
| 11 | **GatewayG3** – *Matching Report OK?*  <br>• **Yes**  12  <br>• **No**  13 |
| 12 | **[Mary]** *Trigger automatic Approval Request to Department Manager (system email)*  |
| 13 | **[Mary]** *Email Purchasing team about discrepancy (Discrepancy Notification)*  |
| 14 | **[Dan]** *Investigate discrepancy – either: <br>a) Contact Supplier (escalate to Karen) <br>b) Hold invoice for supplier correction <br>c) Escalate to Supervisor (Rita) if cannot resolve* |
| 15 | **[Rita]** *Supervisor Review – resolve “hard” discrepancies, create retrospective PO, or reject*  |
| 16 | **[Mary]** *After resolution, rerun steps 912 (loop back to G3)*  |
| 17 | **[Carlos]** *Receive Approval Request – review Invoice, PO, Matching Report*  |
| 18 | **GatewayG4** – *Manager approves?*  <br>• **Yes**  19  <br>• **No**  20 |
| 19 | **[Carlos]** *Click “Approve”  system marks Invoice as “Approved”*  |
| 20 | **[Carlos]** *Reject or request hold – send Discrepancy Notification back to Mary (loop to 13)*  |
| 21 | **[Mary]** *Schedule Approved Invoice for next Payment Run (create Payment Schedule)*  |
| 22 | **End Event** “Invoice Paid (batch run)” |

---

### 4. Gateways (Decision Points)

| Gateway | Condition | Outgoing Paths |
|---------|-----------|----------------|
| **G1** – *PO present?* | Invoice contains a valid PO number | **Yes**  Log Invoice (3)  <br> **No**  Request PO from Supplier (4) |
| **G2** – *Receipt confirmed?* | Purchasing confirms goods/services received | **Yes**  Match Invoice (9)  <br> **No**  Request proof / hold (10) |
| **G3** – *Matching Report OK?* | All lineitem quantities, prices, taxes match PO | **Yes**  Manager Approval Request (12)  <br> **No**  Discrepancy handling (1316) |
| **G4** – *Manager approves?* | Manager deems invoice correct & budgetcompliant | **Yes**  Schedule payment (21)  <br> **No**  Return to Mary for investigation (20) |

---

### 5. Exceptions / Variations (Explicitly Modeled)

| Situation | Path |
|-----------|------|
| **Missing PO** | G1No  4  Supplier resends  loop to 1. |
| **Goods not received** | G2No  10  Supplier proof  back to G2 (once receipt confirmed). |
| **Pricing / quantity discrepancy** | G3No  13  14 (Dan)  15 (Rita) if needed  16 (rematch). |
| **Supplieronly contracts (no PO)** | Rita may create a *retrospective PO* before matching; after creation the flow rejoins at step9. |
| **Manager rejects** | G4No  20  13  14/15  16  back to G3. |
| **Urgent supplier escalation** | Dan may directly involve **Karen** (outside the core flow) to obtain corrected invoice; after correction the flow returns to step9. |
| **Parallel processing** | While Mary is waiting for receipt confirmation, Dan can preemptively review the PO for completeness (no impact on sequence). |

---

### 6. Ambiguities / “Insomecases” Notes

- **Direct jump from “Log Invoice” to “Manager Approval”** – In a few informal interviews (not shown here) some clerks reported that when a PO is *prevalidated* they skip the explicit “Receipt Confirmation” step and send the invoice straight to the manager. In the model above we keep the receiptconfirmation activity because it is mentioned by both Mary and Dan as the standard practice.  

- **Timing of “Payment Scheduling”** – Mary said she *schedules* the invoice after manager approval, but Dan mentioned that the invoice “goes back to Accounts Payable for the next payment run”. The model reflects this as a single task (21) that can be executed either immediately or at the next batch run, depending on the organization’s paymentrun cadence.  

- **Role of the Accounting System** – The system automatically sends the managerapproval email (step12) and marks the invoice as “Approved” (step19). Those automated actions are shown as systemgenerated events but are not assigned to a human lane.

### 7. QuickCopy BPMN PseudoCode (for import)

```
StartEvent(InvoiceReceived) -> 
Task(Mary, CheckPONumber) -> 
ExclusiveGateway(G1) 
   [PO present] -> Task(Mary, LogInvoice) -> Task(Mary, ForwardToPurchasing) -> 
   Task(Dan, ValidateReceipt) -> ExclusiveGateway(G2) 
        [Receipt OK] -> Task(Mary, MatchInvoice) -> ExclusiveGateway(G3) 
            [Match OK] -> Task(System, SendApprovalRequest) -> Task(Carlos, ReviewAndApprove) -> ExclusiveGateway(G4) 
                [Approve] -> Task(Mary, SchedulePayment) -> EndEvent(PaymentRun) 
                [Reject] -> Task(Mary, SendDiscrepancy) -> Task(Dan, Investigate) -> 
                    ExclusiveGateway(Resolution) 
                        [CanResolve] -> Task(Mary, ReMatch) -> (loop to G3) 
                        [Escalate] -> Task(Rita, SupervisorReview) -> Task(Mary, ReMatch) -> (loop to G3) 
            [Match NOT OK] -> Task(Mary, SendDiscrepancy) -> Task(Dan, Investigate) -> 
                (same resolution subflow as above) 
        [Receipt NOT OK] -> Task(Dan, RequestProof) -> (loop to G2) 
   [PO missing] -> Task(Mary, RequestPOFromSupplier) -> (loop to CheckPONumber) 
```

*The pseudocode mirrors the numbered flow above and can be pasted into most BPMNmodeling tools that accept a textual DSL (e.g., bpmn-js, Camunda Modeler “BPMNXML” generator).*
