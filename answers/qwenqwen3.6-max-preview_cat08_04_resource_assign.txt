### Process Mining Analysis & Optimization Plan: TechSolve Solutions Resource Assignment

**To:** TechSolve Solutions Management  
**From:** Process Improvement Consultant  
**Subject:** Data-Driven Optimization of ITSM Resource Assignment via Process Mining

Based on the provided scenario and event log structure, the following comprehensive approach leverages process mining to diagnose resource inefficiencies, identify root causes, and implement optimized assignment strategies. The analysis focuses on transforming the current suspicion-based understanding into fact-based insights derived from the event log.

---

### 1. Analyzing Resource Behavior and Assignment Patterns

To understand the current state, we must move beyond aggregate statistics and analyze the granular behavior of resources and the flow of work between them using the event log.

**A. Performance and Behavior Metrics:**
We will calculate the following metrics per Agent, Tier, and Skill Group using the `Timestamp`, `Resource`, `Agent Tier`, and `Activity` columns:
*   **Workload Distribution:** Count of `Work Start` events per agent over time. We will calculate the Gini coefficient to quantify workload inequality. A high coefficient confirms the suspicion of uneven distribution.
*   **Processing Efficiency:** Average duration (`Work End` - `Work Start`) per agent/tier, segmented by `Ticket Category` and `Priority`. This identifies if specific agents are slower on certain ticket types or if L2/L3 are processing simple tickets faster than L1 (indicating potential misrouting).
*   **First-Call Resolution (FCR) Rate:** Percentage of tickets where `Work L1 End` is followed by `Ticket Resolved` rather than `Escalate L2`. We will segment this by `Agent ID` to identify high-performing L1 agents and those who over-escalate.
*   **Skill Utilization Ratio:** For each agent, we compare `Agent Skills` against the `Required Skill` of handled tickets.
    *   *Over-qualification Index:* % of tickets handled where the agent's skill level exceeds the requirement (e.g., L3 agent handling `Basic-Troubleshoot`).
    *   *Skill Match Rate:* % of assignments where `Required Skill` $\in$ `Agent Skills`.

**B. Process Mining Techniques for Assignment Patterns:**
*   **Social Network Analysis (SNA) / Handover of Work:** We will construct a resource interaction network where nodes are agents and edges represent ticket handovers (e.g., `Escalate L1` $\to$ `Assign L2`).
    *   *Insight:* This reveals "cliques" where tickets circulate among a small group, or "bottleneck nodes" where many agents hand over to a single specialist. It visualizes the *actual* escalation paths versus the intended L1$\to$L2$\to$L3 hierarchy.
*   **Role Discovery:** Using algorithmic role discovery, we cluster agents based on the activities they perform and the ticket attributes they handle.
    *   *Insight:* This may reveal that some L1 agents functionally act as L2 for specific categories (shadow experts) or that some L2 agents are predominantly doing L1 work, validating the complaint of specialists handling low-level tasks.
*   **Variant Analysis with Resource Overlay:** We will filter the process map by variants.
    *   *Happy Path:* `Created` $\to$ `Assign L1` $\to$ `Work L1` $\to$ `Resolved`.
    *   *Reassignment Loops:* Variants containing `Reassign` or multiple `Assign` activities. We will quantify the frequency and volume of these loops.

**C. Skill Utilization Analysis:**
*   We will map the `Required Skill` attribute against the `Agent Skills` attribute at every assignment event.
*   *Analysis:* Calculate the frequency of "Skill Gaps" (assignment made where agent lacks required skill) leading to subsequent `Reassign` events. Conversely, identify "Skill Waste" where agents with rare skills (e.g., `Security-IAM`) spend significant time on generic categories.

### 2. Identifying Resource-Related Bottlenecks and Issues

Using the metrics and patterns above, we will pinpoint specific operational failures and quantify their impact.

**A. Pinpointing Problems:**
*   **Skill Scarcity Bottlenecks:** By analyzing queue times (`Assign L2` timestamp minus `Escalate L2` timestamp) grouped by `Required Skill`, we can identify skills with consistently high wait times.
    *   *Example:* If `Networking-Firewall` tickets wait 4 hours on average for assignment while `OS-Windows` waits 15 minutes, we have a capacity gap in Firewall skills.
*   **Reassignment Impact:** We will correlate the number of `Reassign` activities per case with the total cycle time.
    *   *Quantification:* Calculate `Avg(Resolution Time | Reassignments > 0)` vs. `Avg(Resolution Time | Reassignments = 0)`. The delta represents the average delay cost per reassignment.
*   **Ping-Pong Effects:** Identify cases with sequences like `L2` $\to$ `Reassign` $\to$ `L2` (different agent) or `L2` $\to$ `Reassign` $\to` L1`.
    *   *Issue:* This indicates poor initial diagnosis or unclear responsibility boundaries.
*   **Overload vs. Underutilization:** Using the workload distribution analysis, we will flag agents in the top 10% of workload and compare their SLA breach rates to the bottom 10%. Overloaded agents often have higher breach rates due to context switching, while underutilized agents represent wasted capacity.
*   **SLA Breach Correlation:** We will perform a root-cause correlation analysis on the `SLA Breach` flag (derived from Priority and Resolution Timestamp).
    *   *Quantification:* "X% of P2 SLA breaches are associated with tickets that had >1 reassignment." "Y% of breaches occur when the initial assignment was to an agent lacking the `Required Skill`."

**B. Quantification Examples:**
*   *Reassignment Cost:* "Each reassignment adds an average of 3.5 hours to resolution time."
*   *Skill Mismatch:* "22% of L2 assignments involve a skill mismatch, resulting in a 40% higher probability of SLA breach."
*   *Specialist Misuse:* "L3 agents spend 30% of their active work time on P3/P4 tickets requiring only `Basic-Troubleshoot` skills."

### 3. Root Cause Analysis for Assignment Inefficiencies

We will investigate *why* the patterns in Section 2 exist using advanced process mining techniques.

**A. Potential Root Causes & Validation:**
*   **Deficient Assignment Rules:**
    *   *Hypothesis:* Round-robin ignores skills.
    *   *Validation:* Check assignments made by `Dispatcher`. If the distribution of tickets to agents is uniform regardless of `Required Skill`, round-robin is confirmed as the cause of skill mismatches.
*   **Inaccurate Skill Profiles:**
    *   *Hypothesis:* Agents have skills not documented, or lack documented skills.
    *   *Validation:* Identify cases where an agent successfully resolves a ticket requiring a skill *not* listed in their `Agent Skills`. This suggests the skill matrix is outdated. Conversely, frequent reassignments by an agent possessing the required skill may indicate a proficiency issue or data error.
*   **Poor Initial Categorization:**
    *   *Hypothesis:* `Required Skill` changes during the process.
    *   *Validation:* Analyze cases where `Required Skill` at `Ticket Created` differs from `Required Skill` at `Work L2 Start`. A high frequency indicates L1 or the portal is misdiagnosing the skill need, leading to wrong routing.
*   **Lack of Visibility/Empowerment:**
    *   *Hypothesis:* L1 escalates too quickly or Dispatchers assign to busy agents.
    *   *Validation:*
        *   *L1 Behavior:* Analyze `Work L1` duration for escalated tickets. If duration is consistently very short (e.g., <5 mins), L1 may be "rubber-stamping" escalations without genuine troubleshooting.
        *   *Dispatcher Behavior:* Correlate assignment decisions with agent queue length at the timestamp of assignment. If dispatchers assign to agents with high active workloads while others are idle, it confirms a lack of real-time visibility.

**B. Decision Mining:**
*   We will apply decision mining at the `Escalate L2` decision point. The algorithm will analyze attributes (`Category`, `Priority`, `Agent ID`, `Time of Day`, `Keywords`) to predict the decision.
*   *Insight:* This may reveal rules like "Agent A05 always escalates `Software-App` tickets" or "P2 tickets created after 4 PM are 80% more likely to be reassigned," pointing to training gaps or shift-handover issues.

### 4. Developing Data-Driven Resource Assignment Strategies

Based on the analysis, I propose three concrete strategies to optimize assignment logic.

#### Strategy 1: Skill-Based Routing with Proficiency Weighting
*   **Issue Addressed:** High reassignment rates, skill mismatches, and SLA breaches due to tickets landing with agents lacking the right expertise.
*   **Process Mining Insight:** SNA and Skill Match analysis show that mismatches drive reassignments and delays. Role discovery identifies "shadow experts" who resolve specific skills efficiently.
*   **Mechanism:** Replace round-robin with a routing engine that matches `Required Skill` to `Agent Skills`.
    *   *Weighting:* Assign a "Proficiency Score" to each agent-skill pair based on historical data (e.g., inverse of average resolution time for that skill, weighted by success rate). The router assigns to the available agent with the highest proficiency score for the required skill.
*   **Data Required:** Real-time `Agent Skills` matrix, `Required Skill` from ticket, Historical resolution performance per agent/skill, Agent availability status.
*   **Expected Benefits:**
    *   Reduction in reassignments by 30-50%.
    *   Improved FCR at L2/L3.
    *   Direct improvement in SLA compliance for skill-sensitive categories.

#### Strategy 2: Workload-Aware Dynamic Assignment & Tier Bypass
*   **Issue Addressed:** Uneven workload, bottlenecks at specific tiers, and specialists doing low-level work.
*   **Process Mining Insight:** Workload analysis shows high variance in agent utilization. Variant analysis shows P2/P3 tickets often wait in L1 queues unnecessarily before escalation, contributing to SLA breaches.
*   **Mechanism:**
    *   *Workload Balancing:* The assignment algorithm incorporates real-time queue length and active work count. If the best-skilled agent is overloaded (threshold exceeded), the ticket routes to the next best-skilled agent or holds in a priority queue rather than adding to the bottleneck.
    *   *Tier Bypass:* For tickets with high confidence skill requirements (e.g., `Security-IAM` or `Database-SQL`) and Priority P1/P2, implement rules to bypass L1 and route directly to the appropriate L2/L3 skill pool. This is validated by decision mining showing L1 adds no value (immediate escalation) for these specific patterns.
*   **Data Required:** Real-time agent workload counts, Ticket Priority, Confidence score of `Required Skill` classification, Tier bypass rules derived from historical L1 value-add analysis.
*   **Expected Benefits:**
    *   Balanced utilization across the agent pool (lower Gini coefficient).
    *   Reduced cycle time for P2/P3 tickets by eliminating non-value-added L1 touchpoints.
    *   L1 focus shifted to true triage and basic resolution, reducing specialist waste.

#### Strategy 3: Predictive Triage and Skill Tagging Enhancement
*   **Issue Addressed:** Poor initial categorization leading to wrong routing, and L1 over-escalation due to lack of knowledge.
*   **Process Mining Insight:** Root cause analysis reveals `Required Skill` often changes post-assignment. Decision mining shows L1 escalates tickets that historically have high L1 resolution rates for similar attributes.
*   **Mechanism:**
    *   *Predictive Skill Tagging:* Implement an ML model trained on historical resolved tickets (using `Category`, `Description`, and final `Required Skill`) to predict the `Required Skill` at the moment of `Ticket Created`. This ensures the routing engine (Strategy 1) receives accurate skill requirements immediately.
    *   *Next-Best-Action for L1:* When L1 picks up a ticket, the system suggests "Likely Resolution Steps" or "Recommended Skill" based on similar past cases. If the model predicts high L1 resolvability, the UI prompts the agent to attempt resolution before escalating, potentially blocking immediate escalation for "easy" patterns.
*   **Data Required:** Ticket text/description (NLP), Historical case attributes and outcomes, Real-time inference engine integration with ITSM tool.
*   **Expected Benefits:**
    *   Higher accuracy of initial routing, reducing "Ping-Pong."
    *   Increased L1 FCR by empowering agents with data-driven guidance.
    *   Reduction in "Skill Waste" by ensuring specialists only receive tickets that truly require their expertise.
