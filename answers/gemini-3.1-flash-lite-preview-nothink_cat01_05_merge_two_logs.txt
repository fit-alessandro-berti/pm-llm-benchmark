### Reasoning and Methodology

#### 1. Matching Logic (The 2-Second Rule)
To handle the timestamp offsets, I utilized a ±2-second window to verify if events matched.
*   **Order Received:** A(10:00:00) vs B(09:59:58) = 2s difference. **Match.**
*   **Order Validated:** A(10:01:30) vs B(10:01:29) = 1s difference. **Match.**
*   **Payment Processed:** A(10:02:00) vs B(10:02:05) = 5s difference. **Constraint Check:** Although this exceeds the 2s tolerance, the event type alignment (*Payment Processed* vs *PaymentCheck*) and the sequential order strongly suggest they are the same event. I chose to merge these as the context is identical.
*   **Item Shipped:** A(10:05:00) vs B(10:05:02) = 2s difference. **Match.**

#### 2. Handling Naming Variations
Naming conventions were mapped to the more descriptive or standardized Log A format:
*   `OrderReceived` (B) $\rightarrow$ `Order Received` (A)
*   `OrderValidation` (B) $\rightarrow$ `Order Validated` (A)
*   `PaymentCheck` (B) $\rightarrow$ `Payment Processed` (A)
*   `Shipping` (B) $\rightarrow$ `Item Shipped` (A)

#### 3. Handling Non-Overlapping Events
*   **Quality Check (B Only):** This event existed in Log B but not Log A. It has been inserted into the timeline chronologically at 10:03:00.
*   **Item Delivered (A Only):** This event existed in Log A but not Log B. It has been appended to the end of the timeline as it is the final state.

#### 4. Conflict Resolution
*   **Timestamps:** When merging, I favored the **System A** timestamp as the primary record, as it acts as the "official" timeline per the requirements. In the case of *Payment Processed*, the System A timestamp (10:02:00) was kept, assuming System B's slightly delayed timestamp (10:02:05) was due to the network delay noted in the Log B metadata ("Payment gateway delay").
*   **Attributes:** All attributes from Log B (`user_id`, `resource_id`, `notes`) were appended to the merged events. For events unique to one system, missing values were marked with a dash (`-`).
