# Log Skeleton for Order Fulfillment Process

## Analysis

The process follows a **linear sequential flow**:

**RO  CA  RI  PI  QC  PO  GSL  D  IC  RP**

Every order that enters the system proceeds through all ten activities exactly once.

## Python Dictionary

```python
log_skeleton = {
    'equivalence': {

        # Listed along the process chain and anchored to RO

        ('CA', 'RO'),

        ('IC', 'RO'), ('IC', 'CA'), ('IC', 'RI'), ('IC', 'PI'), ('IC', 'QC'), ('IC', 'PO'), ('IC', 'GSL'), ('IC', 'D'),
        ('RP', 'RO'), ('RP', 'CA'), ('RP', 'RI'), ('RP', 'PI'), ('RP', 'QC'), ('RP', 'PO'), ('RP', 'GSL'), ('RP', 'D'), ('RP', 'IC')

    'always_after': {
        # (A, B) means: if A occurs, B must occur after A
        ('RO', 'CA'), ('RO', 'RI'), ('RO', 'PI'), ('RO', 'QC'), ('RO', 'PO'), ('RO', 'GSL'), ('RO', 'D'), ('RO', 'IC'), ('RO', 'RP'),
        ('CA', 'RI'), ('CA', 'PI'), ('CA', 'QC'), ('CA', 'PO'), ('CA', 'GSL'), ('CA', 'D'), ('CA', 'IC'), ('CA', 'RP'),
        ('RI', 'PI'), ('RI', 'QC'), ('RI', 'PO'), ('RI', 'GSL'), ('RI', 'D'), ('RI', 'IC'), ('RI', 'RP'),
        ('PI', 'QC'), ('PI', 'PO'), ('PI', 'GSL'), ('PI', 'D'), ('PI', 'IC'), ('PI', 'RP'),
        ('QC', 'PO'), ('QC', 'GSL'), ('QC', 'D'), ('QC', 'IC'), ('QC', 'RP'),

    },

    'never_together': set(),
    # All 10 activities co-exist in every case — no exclusions

    'activ_freq': {
        'RO':  {1},
        'CA':  {1},
        'RI':  {1},
        'PI':  {1},
        'QC':  {1},
        'PO':  {1},
        'GSL': {1},

        ('RO', 'CA'),    # Order received  check stock
        ('CA', 'RI'),    # Availability confirmed  reserve items
        ('RI', 'PI'),    # Items reserved  pick from shelves
        ('PI', 'QC'),    # Items picked  quality inspection
        ('QC', 'PO'),    # Quality passed  pack the order
        ('PO', 'GSL'),   # Packed  generate shipping label
        ('GSL', 'D'),    # Label attached  dispatch to carrier
        ('D', 'IC'),     # Dispatched  invoice the customer
        ('IC', 'RP')     # Invoice sent  record payment
    }
}
```

## Rationale

| Constraint | Justification |
|---|---|
| **Equivalence** | Every activity occurs exactly once per case, so all are pairwise equivalent. Key chain and anchor pairs are listed. |
| **Always Before** | Full transitive closure of the linear ordering — e.g., before *Quality Check* can happen, *Receive Order*, *Check Availability*, *Reserve Items*, and *Pick Items* must all have occurred. |
| **Always After** | Mirror of *always_before* — e.g., after *Receive Order*, every subsequent activity is guaranteed to follow. |
| **Never Together** | Empty — all activities always co-occur within every case. |
| **Activity Freq** | Each activity occurs **exactly once** per order (`{1}`). |
| **Directly-Follows** | Strict linear handoff: each activity is immediately followed by the next step with no interleaving. |
