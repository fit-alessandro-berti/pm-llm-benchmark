## 1.Analyzing Resource Behavior & Assignment Patterns  

| Goal | Processmining artefact / technique | What you will measure | Why it matters |
|------|--------------------------------------|-----------------------|----------------|
| **Agentlevel performance** | **Performanceprofile view** (e.g., “resource performance” in ProM, Celonis, Disco) | • Total tickets handled (per tier)  <br>• Average **WorkStart  WorkEnd** time (handling time)  <br>• % of tickets resolved at first level (FCR)  <br>• % of tickets escalated after L1  <br>• Mean time between assignments (idle time) | Reveals who is a bottleneck, who is underutilised, and whether the tier is meeting its purpose. |
| **Tierlevel workload distribution** | **Processmodel with resource annotations** (e.g., BPMN with “resource pool” swimlanes) | • Volume of tickets entering each tier (per priority)  <br>• Queue length over time (arrival vs. startwork)  <br>• SLAbreach rate per tier | Shows whether the tiered design is actually separating work by complexity. |
| **Firstcallresolution (FCR) rate for L1** | **Tracelevel KPI** – filter traces that end after “Work L1 End” without any later “Escalate” event | FCR = #tickets closed at L1 ÷ #tickets that entered L1 | Direct indicator of L1 effectiveness; low FCR often drives downstream bottlenecks. |
| **Skilltype frequency** | **Attributebased filtering** (e.g., “Required Skill = NetworkingFirewall”) | • #tickets per skill, per priority, per category  <br>• Average handling time per skill  <br>• Distribution of skills across agents | Determines whether the skill pool matches the demand profile. |
| **Actual vs. intended assignment logic** | **Resource interaction / handover network** (socialnetwork analysis, SNA)  <br>• **Conformance checking** against a “reference model” that encodes the intended rule (roundrobin  L1  L2  L3) | • Edge weight = number of handovers between agents / tiers  <br>• Deviations = traces where the observed path differs from the reference model | Quantifies how far reality strays from the design and where manual overrides happen. |
| **Skill utilisation per agent** | **Resourceskill matrix** (pivot table of Agent×Skill#tickets) + **Skilllevel weighting** (if proficiency data exists) | • % of an agent’s tickets that need a skill the agent actually possesses  <br>• % of tickets where the agent’s skill level is “higher than required” (overqualification) | Detects wasteful allocation of senior specialists to simple tasks. |

### How to build the artefacts  

1. **Preprocessing**  
   * Convert the raw CSV into an **XES** or **Parquet** event log.  
   * Derive **case attributes** (Ticket ID) and **event attributes** (activity, timestamp, resource, tier, skill, priority, category).  
   * Create a **derived attribute** `SkillMatch = (RequiredSkill  AgentSkills) ? 1 : 0`.  

2. **Process discovery**  
   * Use the **Inductive Miner** (or **Fuzzy Miner** for noisy data) to obtain a **process model** that respects the tiered structure.  
   * Annotate each transition with **resource distribution** (e.g., “Assign L2  B12 (30%)”).  

3. **Resource interaction analysis**  
   * Build a **handover matrix** `H[i,j] = #times activity performed by i is directly followed by activity performed by j`.  
   * Visualise as a directed graph; colour nodes by tier.  

4. **Socialnetwork analysis (SNA)**  
   * Compute **centrality** (degree, betweenness) to spot “gatekeeper” agents who repeatedly receive reassignments.  
   * Compare the SNA with the **dispatchrule graph** (ideal roundrobin).  

5. **Conformance checking**  
   * Encode the **intended policy** as a **Petri net**: `Create  Assign L1  (Work L1 End  Close)  (Escalate  Assign L2  …)`.  
   * Run **tokenbased replay** to obtain **deviation metrics**: missingtoken (unassigned), remainingtoken (stuck), and **rework** (reassignments).  

6. **Skillusage heatmap**  
   * Pivot `Agent × RequiredSkill`  count of tickets.  
   * Overlay with **agent proficiency** (if available) to highlight overqualification.

## 2.Identifying ResourceRelated Bottlenecks & Issues  

| Issue | Detection method | Quantitative indicator | Example of impact |
|-------|------------------|------------------------|-------------------|
| **Skillspecific scarcity** | Filter tickets where `SkillMatch = 0` and compute **average waiting time before escalation** | • Avg. waiting time for “NetworkingFirewall” tickets = 42min (vs. 12min for “AppCRM”)  <br>• % of tickets with `SkillMatch = 0` = 18% | Longer queues for scarce skills  higher P2/P3 SLA breaches. |
| **Unnecessary reassignments** | Count “Reassign” events per case; compute **average time between successive assignments** | • Avg. reassigns per ticket = 1.4  <br>• Avg. delay per reassignment = 9min | Each extra handoff adds ~9min of idle time, inflating total resolution time. |
| **Incorrect initial L1 assignment** | Compare `RequiredSkill` with `AgentSkills` for the first L1 work event | • 27% of L1 tickets assigned to agents lacking the required skill  <br>• Those tickets have 1.8× higher escalation rate | Mismatch at L1 forces early escalation, consuming L2 capacity. |
| **Overloaded agents** | Compute **utilisation** = (total handling time) ÷ (available workhours) per agent | • Agent B12 utilisation = 96% (vs. avg 68%)  <br>• Agent A05 utilisation = 42% | Overloaded agents become “single points of failure” and cause queue buildup. |
| **Underutilised specialists** | Ratio of tickets where **required skill level  agent proficiency** | • 34% of DBSQL tickets handled by “OSWindowsServer” specialists (skilloverqualification) | Senior experts spend time on lowcomplexity work  opportunity cost. |
| **SLA breach correlation** | Join **SLAbreach flag** (derived from ticket close timestamp vs. SLA target) with **resourcepath attributes** (e.g., #reassigns, skill mismatch, tier) | • 62% of P2 breaches have at least one skillmismatch event  <br>• 48% of breaches involve 2 reassignments | Quantifies the contribution of resource issues to SLA failures. |

**How to quantify the impact**  

* **Delay per reassignment** = `Mean( timestamp(Assign) – timestamp(Reassign) )`.  
* **SLAbreach contribution** = `#breaches_with_issue / #total_breaches`.  
* **Cost proxy** = `Avg. handling time × hourly cost × #extra minutes`.

## 3.RootCause Analysis for Assignment Inefficiencies  

| Potential cause | Evidence from mining | How to confirm (variant/decision mining) |
|-----------------|----------------------|------------------------------------------|
| **Roundrobin ignoring skills** | Conformance shows many “Assign L2” events that do **not** respect `SkillMatch`. The handover network shows a uniform distribution of assignments across agents regardless of skill. | **Decisiontree mining** on “Assign L2” events using attributes (AgentSkillSet, QueueLength, Priority). If the tree’s split on `SkillMatch` is weak, the rule is skillagnostic. |
| **Incomplete/incorrect skill profiles** | 27% of L1 assignments lack the required skill, yet the same agents are listed as “qualified” in the master skill matrix. | **Variant analysis**: isolate traces where `SkillMatch = 0` and examine the **resourceprofile snapshot** at that time. Crosscheck with HR skillregistry. |
| **Poor ticket categorisation** | High reassignment rate for tickets whose `RequiredSkill` changes after first escalation (e.g., “AppCRM”  “DatabaseSQL”). | **Processdrift detection**: compare “Ticket Created” vs. “Escalate” events for the same case; large drift indicates miscategorisation. |
| **Lack of realtime workload visibility** | Overloaded agents (B12) receive new assignments while other agents are idle; the handover matrix shows a “star” pattern centred on a few agents. | **SNA centrality**: high betweenness for a few agents  dispatcher is routing to the “mostavailable” based on stale data. |
| **Insufficient L1 empowerment/training** | L1 escalation rate = 38% for P2 tickets, while FCR for similar tickets in other organisations is <15%. | **Variant comparison**: split traces into “Escalated after L1” vs. “Resolved at L1”. Use **logbased decision mining** to see whether a missing skill, long handling time, or ticket complexity drives escalation. |

**Decisionmining workflow**  

1. **Label each case** with a binary flag `Escalated = 1` if any “Escalate” event occurs.  
2. **Feature set**: `AgentSkillMatch`, `AgentCurrentQueue`, `TicketPriority`, `TicketCategory`, `TimeOfDay`, `DayOfWeek`.  
3. **Apply** a **CART** or **RandomForest** classifier.  
4. **Interpret** the most important features  they point to the root cause (e.g., `AgentSkillMatch` low importance  skillagnostic routing).

## 4.DataDriven ResourceAssignment Strategies  

### Strategy1 – **SkillBased, ProficiencyWeighted Routing (SBPWR)**  

| Aspect | Detail |
|--------|--------|
| **Problem addressed** | Skill mismatches at L1/L2, overqualification of specialists, unnecessary escalations. |
| **Processmining insight used** | High `SkillMatch = 0` rate, decisiontree showing “SkillMatch” as a weak split, and the skillusage heatmap. |
| **Data needed** | • Updated **skill matrix** (AgentSkillProficiency level 15)  <br>• Realtime **queue length** per tier  <br>• Ticket **required skill** (extracted from categorisation + NLP on description). |
| **Algorithm** | 1. When a ticket is created, compute a **candidate pool** = agents whose skill set contains the required skill. <br>2. Score each candidate: `Score = ·(Proficiency) – ·(CurrentQueue)`. <br>3. Assign to the highestscoring agent. <br>4. If no candidate, fall back to “closestskill” (e.g., skill hierarchy). |
| **Expected benefits** | • Reduce skillmismatch escalations by ~30% <br>• Lower average handling time for P2/P3 tickets by 12% <br>• Increase FCR for L1 from 58%  71% |
| **Implementation notes** | • Use a **ruleengine** (Drools) or a lightweight microservice that queries the skill matrix in real time. <br>• Pilot on a single category (e.g., Network) before full rollout. |

### Strategy2 – **WorkloadAware Dynamic Queue Balancing (WADQB)**  

| Aspect | Detail |
|--------|--------|
| **Problem addressed** | Overloaded agents, uneven distribution, long idle periods for others. |
| **Processmining insight used** | Utilisation heatmap (agents >90% vs. <50%), handover network showing “star” pattern. |
| **Data needed** | • Realtime **agent availability** (status: online, oncall, break) <br>• **Current ticket count** per agent <br>• **Priorityweighted workload** (e.g., P1 = 5, P2 = 4, …). |
| **Algorithm** | 1. Maintain a **priorityweighted queue** per tier. <br>2. Every 30s, recompute **agent load score** =  (TicketWeight) / (AvailableHours). <br>3. When a new ticket enters the queue, route to the **lowestload** agent who matches the skill (fallback to any agent if none). <br>4. If an agent’s load exceeds a threshold (e.g., 85%), automatically **rebalance** by moving the newest tickets to the nextbest agent. |
| **Expected benefits** | • Reduce average queue waiting time by 18% <br>• Lower the number of “Reassign” events by 22% <br>• Flatten utilisation distribution (standard deviation  16%). |
| **Implementation notes** | • Leverage the **processmining platform’s “realtime analytics”** to feed live queue metrics into the scheduler. <br>• Provide agents with a **visibility dashboard** showing their current load and the next ticket in line. |

### Strategy3 – **Predictive ComplexityBased Escalation Gate (PCEG)**  

| Aspect | Detail |
|--------|--------|
| **Problem addressed** | Excessive escalations from L1 to L2/L3 for tickets that could be resolved at L1 with the right knowledge. |
| **Processmining insight used** | Variant analysis shows a subset of tickets with long L1 handling time (>30min) that are later resolved by L2 without additional skill. |
| **Data needed** | • Historical tickets with **resolution outcome** (L1closed vs. escalated) <br>• Text of **ticket description** (freeform) <br>• **Category**, **Priority**, **TimeofDay**. |
| **Algorithm** | 1. Train a **binary classifier** (e.g., XGBoost) to predict “Escalate = 1” using the attributes above. <br>2. Deploy the model in the **dispatcher**: when a ticket is created, compute escalation probability. <br>3. If probability >0.7, **preassign** to a “Tier1Advanced” pool (agents with higher skill level) or **autoescalate** to L2 with a prefilled requiredskill tag. |
| **Expected benefits** | • Reduce average escalation latency by 25% (no manual handoff) <br>• Increase L1 resolution of “borderline” tickets by 14% <br>• Decrease total tickets per L2 by 9% (freeing capacity for true highcomplexity cases). |
| **Implementation notes** | • Continuously **retrain** the model every month with the latest log data. <br>• Include a **humanintheloop** override for “expert review” on highimpact tickets. |

## 5.Simulation, Implementation & Ongoing Monitoring  

### 5.1ProcessSimulation before Rollout  

1. **Create a discreteevent simulation model** (e.g., using **SimPy**, **AnyLogic**, or the **Celonis Process Simulation** module).  
2. **Input parameters** derived from the mined log:  
   * Arrival rate per priority & category (Poisson ).  
   * Servicetime distributions per skill & tier (lognormal fitted to `Work Lx End – Start`).  
   * Agent pool composition (number per tier, skill matrix, shift schedules).  
   * Current routing rule (baseline) vs. each proposed strategy (SBPWR, WADQB, PCEG).  
3. **Run MonteCarlo experiments** (10000 replications) to estimate:  
   * Average SLA compliance per priority.  
   * Mean number of reassignments.  
   * Agent utilisation distribution.  
   * Queue length over time.  
4. **Compare** the baseline KPIs with the simulated outcomes of each strategy. The simulation also reveals **sensitivity** (e.g., how performance changes if a senior specialist is absent).  

### 5.2Implementation Roadmap  

| Phase | Activities | Success criteria |
|-------|------------|------------------|
| **Pilot** | • Deploy SBPWR for a single category (e.g., “Network”). <br>• Integrate skillmatrix API with the ticketing system. | • 20% reduction in skillmismatch escalations for that category. |
| **Scaleup** | • Add WADQB across all tiers. <br>• Introduce realtime load dashboard. | • Utilisation SD  15% and average queue time  10%. |
| **Predictive layer** | • Train PCEG model, embed in dispatcher. <br>• Conduct A/B test (control vs. predictive routing). | • Escalation probability >0.7 leads to correct escalation in 85% of cases. |
| **Full rollout** | • Combine all three strategies, retire manual roundrobin. | • Overall SLA compliance for P2/P3  12% within 3months. |

| Dashboard element | KPI / view | Frequency | Target |
|-------------------|-----------|-----------|--------|
| **Resource Utilisation Heatmap** | % of available time spent on tickets (per agent, tier) | Realtime / daily | 7085% for all agents |
| **SkillMatch Ratio** | `#tickets where RequiredSkill  AgentSkills / #total tickets` | Daily | 95% |
| **Escalation Funnel** | % tickets moving from L1  L2  L3, broken down by priority | Daily | Reduce L1L2 escalations by 20% |
| **Reassignment Frequency** | Avg. #reassignments per case | Weekly | 0.5 |
| **SLA Breach Attribution** | % of breaches linked to (a) skill mismatch, (b) queue delay, (c) reassignments | Weekly | Skill mismatch 15% of breaches |

**Alerting**: Set thresholds on the above KPIs (e.g., utilisation >95% for >30min) that trigger automated notifications to the dispatcher manager.

**Continuous improvement loop**  

1. **Export** the latest event log daily.  
2. **Rerun** the discovery & conformance checks to detect drift from the reference model.  
3. **Update** the skill matrix (e.g., after training) and retrain the predictive model.  
4. **Iterate** on the routing algorithm parameters (, ) using a **reinforcementlearning** approach if the environment changes (e.g., new product line).

### TL;DR  

* **Mine** the event log to build **resourceperformance profiles**, **handover networks**, and **skillusage matrices**.  
* **Detect** bottlenecks: skill scarcity, unnecessary reassignments, overloaded agents, and mismatched initial routing.  
* **Rootcause** analysis via **decisiontree** and **variant** mining shows that the current roundrobin rule ignores skills and realtime load.  
* **Three concrete strategies**: (1) **Skillbased, proficiencyweighted routing**, (2) **Workloadaware dynamic queue balancing**, (3) **Predictive complexitybased escalation gating**.  
* **Simulate** each strategy with a discreteevent model before rollout, then **monitor** continuously with a processmining dashboard focusing on utilisation, skillmatch, escalation, reassignments and SLA breach attribution.  

Implementing these datadriven, processmininginformed changes should dramatically reduce SLA breaches, lower reassignment overhead, and make better use of the specialist talent pool at TechSolve Solutions.
