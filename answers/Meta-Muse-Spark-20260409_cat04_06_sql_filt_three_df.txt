To filter out cases with the exact sequence `Approve Purchase Requisition` → `Create Order` → `Confirm Order` where `Create Order` to `Confirm Order` > 5 days, you need to:

1. Order events per `case_id` by `timestamp` 
2. Find the 3-event window that matches the sequence with no gaps
3. Check the time difference between activity 2 and 3
4. Exclude all events from cases that match both conditions

DuckDB supports `LEAD()` and `QUALIFY`, which makes this clean.

### DuckDB query

```sql
WITH ordered AS (
    SELECT
        case_id,
        activity,
        timestamp,
        -- Look ahead 1 and 2 events within each case
        LEAD(activity, 1) OVER (PARTITION BY case_id ORDER BY timestamp, event_id) AS next_act_1,
        LEAD(activity, 2) OVER (PARTITION BY case_id ORDER BY timestamp, event_id) AS next_act_2,
        LEAD(timestamp, 1) OVER (PARTITION BY case_id ORDER BY timestamp, event_id) AS next_ts_1,
        LEAD(timestamp, 2) OVER (PARTITION BY case_id ORDER BY timestamp, event_id) AS next_ts_2
    FROM event_log
),
bad_cases AS (
    SELECT DISTINCT case_id
    FROM ordered
    WHERE activity = 'Approve Purchase Requisition'
      AND next_act_1 = 'Create Order'
      AND next_act_2 = 'Confirm Order'
      AND next_ts_2 > next_ts_1 + INTERVAL '5 days'  -- more than 5 days between Create and Confirm
)
SELECT e.*
FROM event_log AS e
WHERE e.case_id NOT IN (SELECT case_id FROM bad_cases)
ORDER BY e.case_id, e.timestamp;
```

### Why this works

**1. Identifying the direct sequence**  
`LEAD(activity, 1)` and `LEAD(activity, 2)` look at the next two events for the same `case_id` when ordered by `timestamp`. If they match `'Create Order'` and `'Confirm Order'`, we know the three activities occur in direct succession with no other events between.

**2. Checking the timing condition**  
`next_ts_2 > next_ts_1 + INTERVAL '5 days'` ensures the gap between `'Create Order'` and `'Confirm Order'` is strictly more than 5 days. Use `>=` if you want "5 days or more".

**3. Excluding the cases**  
`bad_cases` collects all `case_id` that have at least one occurrence of the sequence + timing violation. The final `NOT IN` returns all events from every other case.

**Notes** 
- **Tie handling**: I added `event_id` to the `ORDER BY` as a secondary sort key. If your table has duplicate timestamps, replace `event_id` with your primary key or another deterministic tiebreaker. Without it, `LEAD()` order is not guaranteed for identical timestamps.
- **Multiple occurrences**: If a case has the bad sequence more than once, it’s still excluded entirely. If you only wanted to exclude the specific 3 events and keep the rest of the case, the logic would need to flag event ranges instead.
- **Case sensitivity**: DuckDB string comparison is case-sensitive by default. Make sure your `activity` values match exactly.
