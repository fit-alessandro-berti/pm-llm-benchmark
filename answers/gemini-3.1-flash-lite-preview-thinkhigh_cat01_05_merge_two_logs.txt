To produce the integrated event log, I applied the following logic:

### Methodology & Reasoning
1.  **Matching Strategy**: I utilized a 5-second "fuzzy" window to account for network latency or clock skew between System A and System B.
2.  **Naming Mapping**: I reconciled the differing naming conventions by treating them as aliases (e.g., *Order Received*  *OrderReceived*, *Order Validated*  *OrderValidation*).
3.  **Conflict Resolution**: 
    *   **Timestamps**: Where events matched, I prioritized the System A timestamp as the "Primary" time, but noted the System B deviation if significant. 
    *   **Attributes**: I aggregated all attributes. If an event existed in only one log, it was retained with its source specified.
4.  **Chronological Ordering**: The list is sorted by the primary timestamp.

### Decision Log & Documentation

*   **Matches**:
    *   **Order Received**: Matched (2s difference). System A timestamp chosen.
    *   **Order Validated**: Matched (1s difference). System A timestamp chosen.
    *   **Payment Processed / PaymentCheck**: Matched (5s difference). Despite the 5-second gap, the chronological sequence and `order_id` confirm these are the same event.
    *   **Item Shipped / Shipping**: Matched (2s difference). System A timestamp chosen.

*   **Non-Matches**:
    *   **Quality Check (10:03:00Z)**: This event exists only in Log B. Since System A has no corresponding timestamp or event type, it was inserted as a standalone entry to preserve the rich detail provided by System B.
    *   **Item Delivered (10:20:00Z)**: This event exists only in Log A. It represents the final status of the fulfillment process and is included as-is.

*   **Normalization**:
    *   "Order Received" (A) and "OrderReceived" (B) were normalized to "Order Received".
    *   "Order Validated" (A) and "OrderValidation" (B) were normalized to "Order Validated".
    *   "Item Shipped" (A) and "Shipping" (B) were normalized to "Item Shipped".
