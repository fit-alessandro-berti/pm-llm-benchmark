### Proposed SQL Queries for Investigation

The following SQL queries are designed to investigate the identified anomalies, test hypotheses, and detect similar patterns across the dataset.

#### 1. Detect Sequence Violations
Identifies cases where activities do not follow the expected chronological order.

```sql
WITH case_activity_order AS (
    SELECT 
        case_id,
        activity,
        MIN(timestamp) AS first_occurrence
    FROM order_event_log
    GROUP BY case_id, activity
),
ordered_sequence AS (
    SELECT 
        case_id,
        STRING_AGG(activity, ',' ORDER BY first_occurrence) AS actual_sequence
    FROM case_activity_order
    GROUP BY case_id
)
SELECT 
    o.case_id,
    o.order_type,
    os.actual_sequence,
    CASE 
        WHEN os.actual_sequence LIKE '%Ship Goods%Confirm Shipment%' THEN 'Ship before Confirm'
        WHEN os.actual_sequence LIKE '%Receive Payment%Issue Invoice%' THEN 'Payment before Invoice'
        WHEN os.actual_sequence LIKE '%Receive Payment%Register Order%' OR 
             os.actual_sequence LIKE '%Receive Payment%Perform Credit Check%' THEN 'Payment too early'
        ELSE 'Potential Order Deviation'
    END AS anomaly_type
FROM orders o
JOIN ordered_sequence os ON o.case_id = os.case_id
WHERE 
    os.actual_sequence NOT LIKE 'Register Order,Perform Credit Check,Validate Stock,Confirm Shipment,Ship Goods,Issue Invoice,Receive Payment'
    AND os.actual_sequence NOT LIKE 'Register Order,Confirm Shipment,Ship Goods,%'; -- Allow priority-like deviations for detection
```

#### 2. Identify Missing Activities per Case
Finds cases that do not contain all expected activities.

```sql
WITH expected_activities AS (
    UNNEST(ARRAY['Register Order', 'Perform Credit Check', 'Validate Stock', 'Confirm Shipment', 'Ship Goods', 'Issue Invoice', 'Receive Payment']) AS activity
),
case_activity_counts AS (
    SELECT 
        case_id,
        COUNT(DISTINCT activity) AS activity_count
    FROM order_event_log
    GROUP BY case_id
)
SELECT 
    cac.case_id,
    o.order_type,
    cac.activity_count,
    7 - cac.activity_count AS missing_count
FROM case_activity_counts cac
JOIN orders o ON cac.case_id = o.case_id
WHERE cac.activity_count < 7;
```

#### 3. Investigate Credit Check Timing vs. Order Type
Tests the hypothesis that priority orders bypass or delay credit checks.

```sql
SELECT 
    o.case_id,
    o.order_type,
    o.order_value,
    MAX(CASE WHEN e.activity = 'Perform Credit Check' THEN e.timestamp END) AS credit_check_time,
    MAX(CASE WHEN e.activity = 'Ship Goods' THEN e.timestamp END) AS ship_time,
    MAX(CASE WHEN e.activity = 'Perform Credit Check' THEN e.additional_info END) AS credit_info
FROM orders o
JOIN order_event_log e ON o.case_id = e.case_id
GROUP BY o.case_id, o.order_type, o.order_value
HAVING 
    MAX(CASE WHEN e.activity = 'Perform Credit Check' THEN e.timestamp END) > 
    MAX(CASE WHEN e.activity = 'Ship Goods' THEN e.timestamp END)
    OR 
    MAX(CASE WHEN e.activity = 'Perform Credit Check' THEN e.timestamp END) IS NULL;
```

#### 4. Analyze Payment and Invoice Timing
Detects cases where payment occurs before invoicing or unusually early.

```sql
WITH payment_invoice_times AS (
    SELECT 
        case_id,
        MAX(CASE WHEN activity = 'Receive Payment' THEN timestamp END) AS payment_time,
        MAX(CASE WHEN activity = 'Issue Invoice' THEN timestamp END) AS invoice_time,
        MAX(CASE WHEN activity = 'Register Order' THEN timestamp END) AS order_time
    FROM order_event_log
    GROUP BY case_id
)
SELECT 
    pit.case_id,
    pit.payment_time,
    pit.invoice_time,
    pit.order_time,
    pit.payment_time - pit.order_time AS time_to_payment,
    CASE 
        WHEN pit.payment_time < pit.invoice_time THEN 'Payment before Invoice'
        WHEN pit.payment_time < pit.order_time + INTERVAL '1 hour' THEN 'Extremely Early Payment'
    END AS payment_anomaly
FROM payment_invoice_times pit
WHERE 
    pit.payment_time < pit.invoice_time 
    OR pit.payment_time < pit.order_time + INTERVAL '1 hour';
```

#### 5. Extract Explicit Anomaly Flags
Queries `additional_info` for known flags indicating deviations.

```sql
SELECT 
    case_id,
    event_id,
    activity,
    timestamp,
    resource,
    additional_info
FROM order_event_log
WHERE additional_info ~* '(late_confirmation|attempted_early|skipped|override)';
```

#### 6. Correlate Anomalies with Resources and Departments
Investigates if specific roles or departments are associated with deviations.

```sql
SELECT 
    o.case_id,
    o.order_type,
    e.activity,
    e.resource,
    r.role,
    r.department,
    e.timestamp,
    e.additional_info
FROM order_event_log e
JOIN resources r ON e.resource = r.resource_id
JOIN orders o ON e.case_id = o.case_id
WHERE o.case_id IN (
    -- List of anomalous cases identified above
    SELECT case_id FROM ordered_sequence 
    WHERE actual_sequence NOT LIKE 'Register Order,Perform Credit Check,Validate Stock,Confirm Shipment,Ship Goods,Issue Invoice,Receive Payment'
)
ORDER BY o.case_id, e.timestamp;
```
