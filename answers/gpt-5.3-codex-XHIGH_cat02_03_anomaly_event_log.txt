A typical **P2P “happy path”** is:

**Create PR  Approve PR  Issue PO  Receive Goods  Receive Invoice  Match (PO/GR/Invoice)  Release Payment  Close**

Using that as baseline, here are the anomalies:

| Case | Anomalous sequence | Why this is problematic |
|------|---------------------|--------------------------|
| **1001** | No clear anomaly | Sequence is consistent with standard controls. |
| **1002** | **PO issued before PR approval** | Commitment to vendor without authorization (approval control bypassed). |
|  | **Payment released before PR approval** | Very serious: money leaves before internal authorization is complete. |
|  | **Payment before goods receipt** | Risk of paying for items not yet delivered/accepted. |
|  | **Payment before invoice matching** (match happens later) | Three-way match control is ineffective if done after payment. |
|  | Invoice received before goods | Can happen, but combined with early payment it increases risk. |
| **1003** | **Payment released before invoice received** | Payment without invoice is a major AP control/compliance red flag. |
|  | **Payment before goods receipt** | Prepayment risk; no proof of delivery before disbursement. |
|  | **Match performed after payment** | Preventive control becomes a post-fact check, too late to stop bad payment. |
| **1004** | **Goods received before PO issued** | Indicates possible maverick/unauthorized buying or retroactive PO creation (“after-the-fact PO”). Weakens procurement control and audit trail. |

## Overall assessment
- **Most critical anomalies:** Cases **1002** and **1003** (payments released before core controls).
- **Moderate anomaly:** Case **1004** (PO timing issue suggesting process non-compliance).
- **Conforming case:** **1001**.

If these are not intentional exceptions (e.g., approved prepayments/emergency buys), they represent significant control weaknesses and audit risk.
