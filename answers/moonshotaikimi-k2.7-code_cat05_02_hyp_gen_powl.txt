## 1. Anomalies in the POWL model

The intended trace is:

```text
R  A  E  P  N  C
```

The POWL model you posted deviates from this in three main ways.

| # | Structure in the model | Why it is anomalous |
|---|------------------------|---------------------|
| **A** | `loop = OperatorPOWL(Operator.LOOP, children=[E, P])` | In PM4Py, a 2-child loop has the semantics **E (P E)\***. That means: do `E`, then either stop **or** do `P` followed by another `E`, repeatedly. Consequences:<br>• The loop is **unbounded** – a claim can be evaluated/approved indefinitely.<br>• It can exit **after the first `E`**, so a claim can be closed **without any approval (`P`)**.<br>• Every occurrence of `P` must be followed by another `E`, so an approval is always followed by a re-evaluation. |
| **B** | `xor = OperatorPOWL(Operator.XOR, children=[N, skip])` | The XOR lets the model choose between notifying the customer (`N`) and a silent skip. In the intended process, notification is mandatory, so this makes **skipping notification a valid path**. |
| **C** | Partial order `RAloopxor` **and** `AC`, but **no edge `xorC`** | `C` only waits for `A`. It is unordered with respect to `loop` and `xor`. Therefore the model allows `C` to fire **right after assignment**, concurrently with or even before evaluation, approval and notification. This enables **closing a claim before it has been properly evaluated, approved or notified**. |

In summary, the model permits traces such as:

```text
R A E C                 -- closed without approval or notification
R A E P E P E C         -- repeated evaluate/approve cycles
R A E P E N C           -- approval followed by re-evaluation (rework loop)
R A C N                 -- closed before notification
```

All of these violate the ideal end-to-end flow.

## 2. Hypotheses for why these anomalies exist

### H1 – Iterative review was intended but not constrained
The business may require complex claims to go through multiple “evaluate  approve  re-evaluate” cycles (e.g., when new evidence is submitted). The analyst correctly added a loop but forgot to bound it to a maximum number of iterations or to distinguish an optional rework path from the standard one-time path.

### H2 – Customer notification happens outside the system
Notification may be performed by an external channel (email gateway, customer portal) and not logged as `N`. To make the model fit the observed logs, the designer added an XOR with a silent skip, inadvertently making notification optional in the model even though the business rule still requires it.

### H3 – A legacy “fast-track / close-after-assignment” path still exists
Originally, claims might have been closable immediately after assignment (e.g., for invalid or withdrawn claims). When evaluation/approval was later introduced, the old edge `A  C` was never removed, so the model still permits closure before the new mandatory steps.

### H4 – BPMN / POWL import or drawing error
The missing ordering edge between the XOR and `C` could be a technical artifact: a sequence flow was forgotten in the drawing tool, or a BPMN-to-POWL conversion dropped the edge that should connect `N` to `C`.

### H5 – Weak model-validation rules in the process-modeling tool
The modeling tool may not have checked that every end activity has all mandatory predecessors, or that a loop between two non-silent activities is bounded. Without such controls, incomplete or incorrect models can be saved and executed.

### H6 – Departmental miscommunication
The compliance/legal team may require `E`, `P`, `N`, and `C` in strict order, while operations sees evaluation/approval as iterative and notification as conditional. The model is a compromise that captures both views but enforces neither correctly.

```sql
    COUNT(*) AS approval_count
FROM claim_events
WHERE activity = 'P'
GROUP BY claim_id
HAVING COUNT(*) > 1
ORDER BY approval_count DESC;
```

If many claims have `P` more than once, the unbounded loop/rework hypothesis (**H1**) is supported.

### 3.4 Approvals followed by another evaluation (rework cycles)

```sql
SELECT
    p.claim_id,
    p."timestamp"  AS approved_at,
    MIN(e."timestamp") AS re_evaluated_at
FROM claim_events p
JOIN claim_events e
  ON e.claim_id = p.claim_id
 AND e.activity = 'E'
 AND e."timestamp" > p."timestamp"
WHERE p.activity = 'P'
GROUP BY p.claim_id, p."timestamp"
ORDER BY p.claim_id, approved_at;
```

This finds the `P  E` pattern implied by the loop. A high count suggests genuine rework (**H1**) rather than a one-off modeling mistake.

### 3.5 Claims where notification was skipped

```sql
SELECT
    c.claim_id,
    c."timestamp" AS closed_at
FROM claim_events c
WHERE c.activity = 'C'
  AND NOT EXISTS (
    SELECT 1 FROM claim_events n
    WHERE n.claim_id = c.claim_id
      AND n.activity = 'N'
  )
ORDER BY c."timestamp";
```

A large result set supports **H2** (notification happens outside the system or is deliberately skipped).

### 3.6 Notification occurring *after* closure

```sql
SELECT DISTINCT
    c.claim_id,
    c."timestamp" AS closed_at,
    n."timestamp" AS notified_at
FROM claim_events c
JOIN claim_events n
  ON n.claim_id = c.claim_id
 AND n.activity = 'N'
WHERE c.activity = 'C'
  AND n."timestamp" > c."timestamp"
ORDER BY c.claim_id;
```

If notifications routinely happen after closure, it confirms the missing `xor  C` ordering (**H4**).

### 3.7 Claims closed directly after assignment, bypassing evaluation/approval

```sql
SELECT
    a.claim_id,
    a."timestamp" AS assigned_at,
    c."timestamp" AS closed_at
FROM claim_events a
JOIN claim_events c
  ON c.claim_id = a.claim_id
 AND c.activity = 'C'
WHERE a.activity = 'A'
  AND NOT EXISTS (
    SELECT 1 FROM claim_events e
    WHERE e.claim_id = a.claim_id
      AND e.activity IN ('E','P')
  )
ORDER BY c."timestamp";
```

This directly tests the `A  C` bypass anomaly.

### 3.8 Do anomalies cluster by adjuster, specialization or region?

If `claim_events.resource` stores an adjuster identifier or name, you can join to the `adjusters` table to see whether certain teams are associated with the anomalies.

```sql
WITH anomaly_claims AS (
    -- Closed without E or P
    SELECT DISTINCT c.claim_id
    FROM claim_events c
    WHERE c.activity = 'C'
      AND NOT EXISTS (
        SELECT 1 FROM claim_events e
        WHERE e.claim_id = c.claim_id
          AND e.activity IN ('E','P')
      )

    UNION

    -- Multiple approvals
    SELECT claim_id
    FROM claim_events
    WHERE activity = 'P'
    GROUP BY claim_id
    HAVING COUNT(*) > 1

    UNION

    -- No notification
    SELECT DISTINCT c.claim_id
    FROM claim_events c
    WHERE c.activity = 'C'
      AND NOT EXISTS (
        SELECT 1 FROM claim_events n
        WHERE n.claim_id = c.claim_id
          AND n.activity = 'N'
      )
)
SELECT
    COALESCE(adj.name, ce.resource) AS resource,
    adj.specialization,
    adj.region,
    COUNT(DISTINCT ac.claim_id) AS anomaly_count
FROM anomaly_claims ac
JOIN claim_events ce
  ON ce.claim_id = ac.claim_id
LEFT JOIN adjusters adj
  ON adj.adjuster_id::text = ce.resource
  OR adj.name = ce.resource
GROUP BY
    COALESCE(adj.name, ce.resource),
    adj.specialization,
    adj.region
ORDER BY anomaly_count DESC;
```

If certain adjusters/regions dominate, the issue may be **training, local practice or a technical configuration** rather than a global business rule.

### 3.9 Optional sanity check: activity label coverage

Before running the queries, confirm that the event log actually uses the same labels as the model:

```sql
SELECT activity, COUNT(*) AS freq
FROM claim_events
GROUP BY activity
ORDER BY freq DESC;
```

If the database uses full names such as `Receive Claim` instead of `R`, adapt the queries accordingly.

---

### How to interpret the results

| Query | Empty result | Non-empty result |
|-------|--------------|------------------|
| Closed without E/P | Model over-approximates; the anomaly is theoretically allowed but not exercised. | **H1 / H3 supported** – claims are really bypassing mandatory steps. |
| Closed before E/P/N | Timestamps respect the intended order; partial-order anomaly is latent. | **H3 / H4 supported** – closure actually occurs out of order. |
| Multiple approvals | Loop not used in practice. | **H1 supported** – rework/iteration is real. |
| Notification skipped | Notification is always recorded. | **H2 supported** – notification is omitted or logged elsewhere. |
| Clustering by adjuster | Anomalies are spread evenly. | Anomalies may be caused by specific teams, tools or regional rules. |

By combining these SQL checks with stakeholder interviews, you can determine whether the POWL model needs to be tightened (remove `AC`, add `xorC`, bound or replace the loop, make `N` mandatory) or whether the real process genuinely contains exceptions that should be modeled explicitly.
