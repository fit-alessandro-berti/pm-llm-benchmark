# Data-Driven Queue Mining Approach for Outpatient Clinic Optimization

## 1. Queue Identification and Characterization

### Defining Waiting Time
In this event log, waiting time (queue time) for an activity *A* in a case is calculated as:

**Waiting Time (A) = Start Timestamp (A)  Complete Timestamp (Previous Activity in Case)**

Since the log captures both START and COMPLETE events per activity, I can cleanly separate:
- **Service time** = Complete(A)  Start(A)
- **Waiting time** = Start(A)  Complete(Predecessor) *(non-value-adding idle time from the patient's perspective)*
- **Sojourn time** = Waiting time + Service time

For the first activity (Registration), waiting time could be approximated using the scheduled appointment time vs. actual Registration Start, if available, or patient arrival timestamps.

### Key Queue Metrics
For each transition (e.g., Registration  Nurse Assessment, Nurse Assessment  Doctor Consultation, etc.), I would compute:

| Metric | Purpose |
|---|---|
| **Mean waiting time** | Overall magnitude |
| **Median (P50)** | Central tendency robust to outliers |
| **P90 / P95 waiting time** | Tail behavior — drives patient dissatisfaction |
| **Max waiting time** | Worst-case experience |
| **Std. dev. / CV (coefficient of variation)** | Variability; high CV  unpredictability |
| **Queue frequency** | How often this wait occurs (per day/week) |
| **% cases exceeding SLA threshold** (e.g., >20 min) | Compliance |
| **Queue length over time** | # cases waiting simultaneously (derived by counting cases whose predecessor completed but next start hasn't happened) |
| **Waiting time share of total sojourn** | Lost time ratio |

### Identifying the Most Critical Queues
I would rank queues using a **composite impact score**:

**Impact = (P90 Waiting Time) × (Frequency) × (Patient-experience weight)**

Criteria to justify "critical":
1. **High P90, not just mean** — tail waits drive complaints.
2. **High frequency** — affects many patients.
3. **Large share of total visit duration** — a queue taking 40% of visit time is the biggest lever.
4. **Disproportionate impact on priority groups** — e.g., Urgent patients waiting longer than Normal ones would be a red flag, regardless of absolute numbers.
5. **Correlation with drop-outs/complaints** if such data is available.

A **Pareto (80/20) analysis** on waiting time typically reveals 2–3 transitions causing most of the pain — often NurseDoctor and DoctorDiagnostic Test based on common clinic patterns.

## 2. Root Cause Analysis

Identifying *where* queues occur is only half the picture. Root causes typically fall into these categories, each investigable with specific mining techniques:

### Candidate Root Causes and Diagnostic Techniques

**(a) Resource Bottlenecks**
- Compute **resource utilization** =  service time / availability window for each staff/room.
- If Dr. Smith's utilization > 90% but Dr. Jones's is 55%, we have imbalance.
- **Social network / handover-of-work analysis** reveals overloaded resources.

**(b) Activity Dependencies & Handovers**
- **Handover matrices** between resources/roles show where handoffs cause delays (e.g., Nurse completes assessment but doctor isn't notified).
- Check whether waiting correlates with resource pool switches.

**(c) Service Time Variability**
- Compute **CV of service times** per activity per resource. High variability (e.g., doctor consults ranging 5–40 min) cascades downstream queues.
- Compare service times across resources performing the same activity — variance there signals training or process-standardization issues.

**(d) Arrival Patterns & Scheduling**
- Plot **arrival rate vs. time of day / day of week**. Clustering at 9–11 AM creates morning pile-ups.
- Compare scheduled vs. actual registration start to detect over-booking.
- **Little's Law** (L = W) can validate whether observed queue length is consistent with arrival rate and waiting time.

**(e) Patient Type / Urgency Effects**
- **Variant analysis**: Segment cases by Patient Type (New vs. Follow-up) and Urgency. New patients may require longer nurse assessments, creating downstream cascades.
- **Performance dotted charts** stratified by segment highlight differential treatment.

**(f) Process Variants and Rework**
- Conformance checking against a reference process may reveal loops (e.g., patient sent back for additional tests) that amplify waits.

### Integrating Techniques
I would build a **multi-dimensional bottleneck heatmap** crossing:
- Activity transition (where)
- Resource (who)
- Time of day (when)
- Patient segment (who for)

This pinpoints, for example: "NurseCardiology Doctor wait is 35 min on avg, but specifically 55 min for New patients on Monday/Tuesday mornings because only one cardiologist is scheduled against an overbooked new-patient block."

## 3. Data-Driven Optimization Strategies

### Strategy 1: Resource Re-Allocation Based on Temporal Demand Profiling
- **Target queue**: Registration  Nurse Assessment and Nurse  Doctor (morning peaks).
- **Root cause**: Mismatch between arrival rate curve and staffing curve.
- **Data support**: Arrival-rate histograms by 30-min intervals cross-referenced with resource utilization. If arrivals peak at 9:00–10:30 but staffing is flat, we have quantified evidence.
- **Action**: Shift one clerk's and one nurse's schedule to cover 8:30–12:30 instead of 9:00–17:00; introduce staggered breaks.
- **Expected impact**: Using queueing theory (M/M/c approximations), adding one nurse during the peak 90 minutes typically cuts morning wait by 40–60%. Target: reduce P90 Nurse Assessment wait from, say, 25 min to 12 min.

### Strategy 2: Smart Appointment Scheduling (Segmented & Smoothed)
- **Target queue**: Nurse  Doctor Consultation.
- **Root cause**: Over-booking of New patients (longer consults) in the same time block, combined with service-time variability.
- **Data support**: Mined service-time distributions per patient type (New: µ=28 min; Follow-up: µ=12 min). Current scheduling treats slots uniformly.
- **Action**: Implement **segmented scheduling** — alternate New-patient slots with Follow-up slots and match slot length to actual distribution (e.g., 30-min slots for New, 15-min for Follow-up). Introduce a **small buffer slot every 4 appointments** to absorb variability.
- **Expected impact**: Reduced doctor-side queue accumulation; 20–35% reduction in Doctor-Consultation waiting time; smoother downstream flow to tests.

### Strategy 3: Parallelization of Nurse Assessment and Pre-Consultation Data Collection
- **Target queue**: Registration  Nurse Assessment, and pre-consultation waits.
- **Root cause**: Sequential dependency; nurses collect vitals/history that could be partially done via self-service kiosks or pre-visit digital forms.
- **Data support**: Nurse Assessment service time analysis reveals ~40% is data-entry (history, forms), ~60% clinical measurement. Decompose the activity.
- **Action**: Introduce a **pre-visit digital intake** (tablet or mobile app) for history-taking. Nurse focuses on clinical vitals only.
- **Expected impact**: Nurse Assessment service time cut ~30%, freeing capacity and reducing the downstream wait cascade. Net visit duration reduction of 10–15 minutes on average.

### Strategy 4 (Bonus): Real-Time Flow Coordination Dashboard
- **Target queue**: All intermediate handovers, especially Doctor  Diagnostic Tests.
- **Root cause**: Poor visibility — doctors don't know current test-room queue; patients wander or wait blindly.
- **Action**: Real-time dashboard using the same event stream showing live queue lengths and estimated waits. Doctors route patients to the least-loaded diagnostic resource; patients get SMS wait estimates.
- **Expected impact**: Improved load balancing across equivalent resources (e.g., multiple ECG techs); 15–25% reduction in test-waiting variance; substantial patient-experience lift.

## 4. Trade-offs and Constraints

| Strategy | Potential Trade-off / Risk | Mitigation |
|---|---|---|
| Shifted staffing | Coverage gap in late afternoon; potential union/contract issues | Start with voluntary shift offers; monitor afternoon KPIs |
| Segmented scheduling | Reduced daily slot count for doctors  revenue concern | Offset with reduction in unused "slack" from overruns; measure net throughput |
| Pre-visit digital intake | Digital exclusion of elderly patients; data-quality concerns | Hybrid: kiosks with staff assistance; fallback to nurse completion |
| Parallelization | Possible care-quality concern if nurse review is rushed | Maintain clinical validation step; audit periodically |
| Real-time dashboard | IT investment; change-management resistance from staff | Phased rollout; involve clinicians in design |

### Bottleneck Displacement
A critical mining principle: **reducing one queue often shifts the bottleneck**. If we speed up Nurse Assessment, the Doctor queue may lengthen. I would run **what-if simulations** on the discovered process model (using tools like ProM's CPN simulator or Disco/Celonis simulation) to pre-empt this, and stage changes sequentially.

### Balancing Objectives
Use a **weighted scorecard** combining:
- Patient wait reduction (40%)
- Cost neutrality (25%)
- Care-quality indicators — readmissions, missed-diagnosis flags (25%)
- Staff workload equity (10%)

Optimization decisions are accepted only if no dimension degrades beyond a defined threshold.

## 5. Measuring Success

### Primary KPIs
| KPI | Target Direction |
|---|---|
| Mean & P90 waiting time per transition |  |
| Total visit sojourn time (end-to-end) |  |
| % visits exceeding 2-hour total duration |  |
| Waiting-to-service time ratio |  toward <1 |
| Resource utilization balance (Gini coefficient across resources) |  |
| Patient satisfaction (NPS / CSAT survey) |  |
| Throughput (patients completed/day) |  or stable |
| On-time start rate of appointments |  |

### Secondary / Guardrail KPIs
- Staff overtime hours (cost control).
- Clinical quality indicators (avoid quality erosion): revisit rate within 14 days, test-order error rate.
- Variance of service times (stability).

### Ongoing Monitoring Framework
1. **Automated event-log ingestion pipeline** feeding a process-mining tool (Celonis, Disco, PM4Py) daily.
2. **Control-chart dashboards** with upper/lower control limits on key waits; automatic alerts when P90 drifts.
3. **Monthly variant analysis** to detect emerging process patterns that may introduce new queues.
4. **Pre-/post comparison using Mann-Whitney U tests** on waiting-time distributions to confirm statistically significant improvement, not just averages.
5. **Quarterly root-cause review** — repeat the diagnostic analysis to catch new bottlenecks that emerge as old ones are resolved.
6. **Continuous feedback loop**: Link patient-satisfaction data back to case-level log data to discover which specific waits most influence experience — refining the composite impact score for future prioritization.

### Closing the Loop
Success is not just lower waits but **sustained, predictable flow**. If P90 drops but variance rises, patients' experiences remain inconsistent. Hence I would emphasize **both central tendency and variability metrics**, and treat queue mining as a continuous practice — not a one-off project — embedded in the clinic's operational governance.
