**1. What is wrong with the current model?**  

| Activity | Desired position in the “ProcuretoPay” flow | Problem caused by the current parallel block |
|----------|---------------------------------------------|----------------------------------------------|
| **CreatePurchaseRequisition** | First step | OK – it is placed at the start of the sequence. |
| **ApprovePurchaseRequisition** | Immediately after the requisition is created | OK – it follows the first step. |
| **CreatePurchaseOrder** | After the requisition is approved | OK – it is inside the parallel block, but the block allows it to be executed *anytime* relative to the other children. |
| **ReceiveGoods** | After the PO is created, before the invoice can be paid | In the parallel block it can be executed after the invoice, after the payment, or even never – which is not allowed. |
| **ReceiveInvoice** | After the PO is created (often after goods are received) | Same problem – the parallel block lets it happen in any order. |
| **MatchInvoicetoPO** | After both the goods and the invoice have been received | The parallel block does **not** enforce that both predecessor activities have finished first; the model permits matching before receiving either. |
| **PayInvoice** | After the invoice has been matched to the PO | Because the block is parallel, payment may occur before matching, before goods are received, or even before the invoice is received. |
| **CloseCase** | The very last step, after payment | This is already after the parallel block, but the block’s looseness means the case can be closed while many earlier steps are still pending. |

In short, the **ANDparallel** node treats the five activities (`Create_PO`, `Receive_Goods`, `Receive_Invoice`, `Match_Invoice_to_PO`, `Pay_Invoice`) as *independent* and can be performed in any order (or even simultaneously). That is why the model is “underfitting” – it allows many traces that are illegal in a real ProcuretoPay process (e.g., paying before goods are received, or matching an invoice that has never been received).

---

**2. Corrected processtree model**

Below is a minimal change that restores the intended **strict sequential** flow while keeping the same set of activities and the same overall “rootsequence” shape.

```python
from pm4py.objects.process_tree.obj import Operator, ProcessTree

# ------------------------------------------------------------------
# 1. Define the leaf activities (unchanged)
# ------------------------------------------------------------------
create_req      = ProcessTree(label="Create_Purchase_Requisition")
approve_req     = ProcessTree(label="Approve_Purchase_Requisition")
create_po       = ProcessTree(label="Create_Purchase_Order")
receive_goods   = ProcessTree(label="Receive_Goods")
receive_invoice = ProcessTree(label="Receive_Invoice")
match_invoice   = ProcessTree(label="Match_Invoice_to_PO")
pay_invoice     = ProcessTree(label="Pay_Invoice")
close_case      = ProcessTree(label="Close_Case")

# ------------------------------------------------------------------
# 2. Build a *single* SEQUENCE that respects the domain order
# ------------------------------------------------------------------
root = ProcessTree(
    operator=Operator.SEQUENCE,
    children=[
        create_req,          # 1. Create requisition
        approve_req,         # 2. Approve requisition
        create_po,           # 3. Create PO
        receive_goods,       # 4. Receive goods
        receive_invoice,     # 5. Receive invoice
        match_invoice,       # 6. Match invoice to PO
        pay_invoice,         # 7. Pay invoice
        close_case           # 8. Close case
    ]
)
```

*If you still want to keep a tiny amount of flexibility (e.g., the invoice may arrive **before** the goods but payment must wait for both), you could replace the two “receive” nodes with a **parallel** subtree that is *followed* by the matchingandpayment sequence. The key is that **matching and payment stay inside a strict sequence after the parallel block**. The puresequence version above is the simplest and safest way to guarantee conformance.*

---

**3. Why the new tree enforces the correct logic**

| Step in the tree | What the operator guarantees | How it blocks illegal behavior |
|------------------|------------------------------|--------------------------------|
| `Create_Purchase_Requisition  Approve_Purchase_Requisition` | `SEQUENCE` forces *Approve* to occur **only after** *Create*. | You cannot approve a nonexistent requisition. |
| `Approve_Purchase_Requisition  Create_Purchase_Order` | `SEQUENCE` forces PO creation **after** approval. | No PO can be created before the requisition is approved. |
| `Create_Purchase_Order  Receive_Goods` | `SEQUENCE` forces goods receipt **after** PO creation. | Goods cannot be received before a PO exists. |
| `Receive_Goods  Receive_Invoice` | `SEQUENCE` forces the invoice to be received **after** goods (or at least after the PO). | Prevents “invoicefirst” scenarios that would break the usual flow. |
| `Receive_Invoice  Match_Invoice_to_PO` | `SEQUENCE` forces matching **after** the invoice is present. | You cannot match an invoice that you have not yet received. |
| `Match_Invoice_to_PO  Pay_Invoice` | `SEQUENCE` forces payment **after** matching. | Guarantees that payment is only possible when the invoice has been verified against the PO. |
| `Pay_Invoice  Close_Case` | `SEQUENCE` forces case closure **after** payment. | The case cannot be closed while the invoice is still unpaid. |

Because **every node is a child of a single `SEQUENCE` operator**, the process engine (or any conformancechecking tool) will only accept traces that follow the exact order listed. Any trace that attempts to:

* Pay before matching,
* Match before the invoice is received,
* Receive goods after payment,
* Close the case while earlier steps are pending,

will be **rejected** as nonconformant.  

Thus the corrected tree eliminates the underfitting behaviour of the original parallel block and faithfully represents the canonical ProcuretoPay lifecycle while preserving the same activity set.
