import pm4py
from pm4py.objects.powl.obj import StrictPartialOrder, OperatorPOWL, Transition, SilentTransition
from pm4py.objects.process_tree.obj import Operator

# ==============================================================================
# 1. Define all process activities as Transitions
# ==============================================================================
SR = Transition(label="SR")
CS = Transition(label="CS")
DA = Transition(label="DA")
ACI = Transition(label="ACI")
BNC = Transition(label="BNC")
DR = Transition(label="D&R")

MD = Transition(label="MD")
QA = Transition(label="QA")
SA = Transition(label="SA")
GLA = Transition(label="GLA")

# ==============================================================================

#    Executes BNC, then optionally D&R and repeats BNC until stable.
# ==============================================================================
config_loop = OperatorPOWL(operator=Operator.LOOP, children=[BNC, DR])

# ==============================================================================
# 4. Deployment Choice Phase: X ( CD, MD )
#    Exclusive choice between Containerized and Monolithic deployment.
# ==============================================================================
deploy_xor = OperatorPOWL(operator=Operator.XOR, children=[CD, MD])

# ==============================================================================
# 5 & 6. Final Phase: QA and SA are concurrent, both must precede GLA
#    Encapsulated in a sub-partial order for modularity and clarity.
# ==============================================================================
end_phase = StrictPartialOrder(nodes=[QA, SA, GLA])
end_phase.order.add_edge(QA, GLA)
end_phase.order.add_edge(SA, GLA)

# ==============================================================================
# Main Flow Construction: Chain all components in sequence
# ==============================================================================
main_nodes = [SR, CS, DA, ACI, config_loop, deploy_xor, end_phase]

# Create the root partial order
root = StrictPartialOrder(nodes=main_nodes)

# Enforce execution order constraints according to the scenario
root.order.add_edge(SR, CS)                     # 1. SR verified -> CS starts
root.order.add_edge(CS, DA)                     # 2. Contract done -> DA starts
root.order.add_edge(CS, ACI)                    # 2. Contract done -> ACI starts
root.order.add_edge(DA, config_loop)            # 3. DA & ACI done -> Loop starts
root.order.add_edge(ACI, config_loop)           # 3. DA & ACI done -> Loop starts
root.order.add_edge(config_loop, deploy_xor)    # 4. Loop stable -> Deployment choice
root.order.add_edge(deploy_xor, end_phase)      # 5. Deployment done -> Final phase

# The constructed model is now fully defined in the `root` variable.
# It can be exported, visualized, or used for conformance checking within pm4py.

### Key Modeling Decisions:
1. **Hierarchy & Nesting**: POWL supports nesting partial orders and operators. The final phase (`QA`, `SA`, `GLA`) is grouped into an `end_phase` partial order to explicitly model their concurrency and dependency on `GLA`, keeping the root clean.
2. **Loop Semantics**: `Operator.LOOP` with `children=[BNC, DR]` directly maps to `* ( BNC, D&R )`, allowing repeated execution of BNC with optional D&R interventions until the loop is exited.
3. **Exclusive Choice**: `Operator.XOR` with `children=[CD, MD]` maps to `X ( CD, MD )`, ensuring only one deployment path is taken.
4. **Concurrency**: Unconnected nodes in a `StrictPartialOrder` are inherently concurrent. This is leveraged for `DA`/`ACI` (parallel after `CS`) and `QA`/`SA` (parallel before `GLA`).
5. **Silent Transitions**: Not required for this scenario since every step has an explicit business label, but `SilentTransition` is imported as per the prompt's specification for completeness.
