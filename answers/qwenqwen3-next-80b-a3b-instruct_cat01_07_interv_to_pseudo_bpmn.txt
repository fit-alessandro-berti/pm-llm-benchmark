### **BPMN-Like Textual Process Model: Invoice Approval Process**

#### **1. Start Event**
- **Event:** Invoice received (via email or electronic submission)

#### **2. Activities & Participants**
| Activity | Role | Description |
|---------|------|-------------|
| **A1: Validate PO Number** | Accounts Payable Clerk (Mary) | Check if invoice contains a valid Purchase Order (PO) number. |
| **A2: Request Missing PO** | Accounts Payable Clerk (Mary) | If no PO is present, email supplier (via Supplier Relationship Manager Karen if needed) to request PO or revised invoice. |
| **A3: Log Invoice in System** | Accounts Payable Clerk (Mary) | Enter invoice into accounting system with PO number, amount, vendor, date. |
| **A4: Forward for Receipt Confirmation** | Accounts Payable Clerk (Mary) | Send invoice to Purchasing Department (Dan) to confirm goods/services were received. |
| **A5: Verify PO Match** | Purchasing Officer (Dan) | Compare invoice line items (quantities, prices, charges) against PO. |
| **A6: Report Discrepancy** | Purchasing Officer (Dan) | If mismatch found, notify AP Clerk (Mary) to hold invoice or request corrected invoice from supplier. |
| **A7: Confirm Match** | Purchasing Officer (Dan) | If PO and invoice match, send confirmation email to AP Clerk. |
| **A8: Match Invoice to PO in System** | Accounts Payable Clerk (Mary) | Systematically cross-check invoice line items with PO in accounting system. |
| **A9: Flag Discrepancy (Post-Match)** | Accounts Payable Clerk (Mary) | If system mismatch detected (e.g., pricing error, missing items), notify Purchasing to resolve. |
| **A10: Escalate to Supervisor** | Accounts Payable Clerk (Mary) | If discrepancy cannot be resolved by Purchasing, escalate to AP Supervisor (Rita). |
| **A11: Notify Manager for Approval** | Accounts Payable Clerk (Mary) | System auto-sends notification to Department Manager (Carlos) after PO match and receipt confirmation are confirmed. |
| **A12: Review & Approve Invoice** | Department Manager (Carlos) | Review invoice, PO, and Purchasing confirmation in system. Approve if amounts, items, and budget align. |
| **A13: Reject or Hold for Investigation** | Department Manager (Carlos) | If invoice appears suspicious (e.g., budget overage, unexpected items), reject or request AP to investigate. |
| **A14: Resolve Exceptional Cases** | AP Supervisor (Rita) | Handle invoices without PO (e.g., informal services), create retrospective PO, or coordinate with department heads to validate. |
| **A15: Coordinate with Supplier** | Supplier Relationship Manager (Karen) | Upon request from AP or Purchasing, contact supplier to correct PO, pricing, or invoice details. |
| **A16: Schedule for Payment** | Accounts Payable Clerk (Mary) | Add approved invoice to next payment run in accounting system. |

#### **3. Gateways (Decision Points)**

| Gateway ID | Type | Condition | Outcome |
|-----------|------|-----------|---------|
| **G1** | Exclusive | Is PO number present on invoice? | **Yes**  Proceed to A3<br>**No**  Go to A2 (Request Missing PO) |
| **G2** | Exclusive | Does Purchasing confirm receipt and match? | **Yes**  Proceed to A8<br>**No**  Go to A6 (Report Discrepancy)  then loop back to A1 or A14 if unresolved |
| **G3** | Exclusive | Does system match invoice to PO? | **Yes**  Proceed to A11 (Notify Manager)<br>**No**  Go to A9 (Flag Discrepancy)  then A10 (Escalate) if unresolved |
| **G4** | Exclusive | Is manager approval granted? | **Yes**  Proceed to A16 (Schedule Payment)<br>**No**  Go to A13 (Reject/Hold)  then loop to A14 (Supervisor Resolution) or A2/A15 (Supplier Correction) |
| **G5** | Exclusive | Can discrepancy be resolved by Purchasing? | **Yes**  Return to A5/A8<br>**No**  Go to A10 (Escalate to Rita) |
| **G6** | Exclusive | Is invoice associated with a formal PO? | **Yes**  Continue normal flow<br>**No**  Go to A14 (Rita creates retrospective PO or decides rejection) |

> **Note:** Gateways G1, G2, G3, and G4 are sequential and nested. G5 and G6 are sub-decisions within exception paths.

#### **4. Sequence Flow (Standard Path)**

1. **Start**  A1  G1  
2. G1  **Yes**  A3  A4  A5  G2  
3. G2  **Yes**  A8  G3  
4. G3  **Yes**  A11  A12  G4  
5. G4  **Yes**  A16  **End**  
6. G4  **No**  A13  (loop to A14 or A15)  (if resolved)  re-enter at A4 or A8

#### **5. Exception Paths (Alternate Flows)**

- **Path E1: Missing PO**  
  A1  G1  **No**  A2  (supplier contacted via Karen)  if PO received  loop to A1  A3  ...

- **Path E2: Discrepancy Detected by Purchasing**  
  A5  G2  **No**  A6  (supplier contacted via Karen)  corrected invoice received  loop to A1 or A3  continue normal flow

- **Path E3: System Mismatch After PO Match**  
  A8  G3  **No**  A9  A7 (if Purchasing re-confirms)  loop to A8  
  If unresolved  A10  A14  (Rita resolves: retrospective PO or rejection)  if resolved  re-enter at A4 or A8

- **Path E4: Manager Rejects Invoice**  
  A12  G4  **No**  A13  A14 (Rita investigates)  if resolved  re-enter at A4 or A8  
  If rejected permanently  **End (Rejected)**

- **Path E5: Non-PO Invoice (Informal Services)**  
  A1  G1  **No**  A2  Karen contacted  supplier confirms service was rendered  Rita creates retrospective PO  A3  continue normal flow  
  *If no retrospective PO approved  A14  Rejection  **End (Rejected)**

#### **6. Artifacts / Inputs / Outputs**

| Artifact | Description | Origin | Destination |
|---------|-------------|--------|-------------|
| **Invoice** | Electronic or PDF document with vendor, amount, line items, PO# | Supplier | A1  A16 |
| **Purchase Order (PO)** | System-recorded order document | Procurement System | A1, A5, A8, A12 |
| **Confirmation Email** | From Purchasing to AP confirming receipt & match | Purchasing (Dan) | AP Clerk (Mary) |
| **Discrepancy Report** | Internal note/email detailing mismatch | AP or Purchasing | Supplier (via Karen) or Supervisor (Rita) |
| **Retrospective PO** | Manually created PO for non-PO invoices | AP Supervisor (Rita) + Dept Head | System (A3) |
| **Approval Notification** | Auto-generated system alert | Accounting System | Manager (Carlos) |
| **Payment Run Entry** | Final record scheduled for disbursement | AP Clerk (Mary) | Finance System |

#### **7. Roles Summary**

| Role | Responsibilities |
|------|------------------|
| **Accounts Payable Clerk (Mary)** | Initial validation, logging, matching, escalation, scheduling. Primary process executor. |
| **Purchasing Officer (Dan)** | Confirm receipt and PO match. Flag mismatches. |
| **Department Manager (Carlos)** | Final approval based on budget, reasonableness, and system data. |
| **AP Supervisor (Rita)** | Resolve exceptions: non-PO invoices, unresolved discrepancies, coordination. No final approval. |
| **Supplier Relationship Manager (Karen)** | Liaise with suppliers to correct invoices or POs. No system access. |
| **Accounting System** | Auto-notifies manager, logs data, performs system matching, triggers payment scheduling. |

#### **8. Key Observations & Ambiguities Noted**

- **Ambiguity 1:** It is unclear whether the “system” performs the invoice-to-PO matching automatically (A8) or if Mary does it manually. *Assumed manual based on Mary’s description of “matching in the system,” but system may assist. Recommend validation via system logs.*
- **Ambiguity 2:** The exact trigger for manager notification is stated as “automatic,” but it’s unclear if this occurs after A5 (Purchasing confirmation) or after A8 (system match). *Assumed after both are confirmed (logical sequence).*
- **Ambiguity 3:** No explicit end state is defined for rejected invoices. *Model includes “End (Rejected)” as a valid terminal state.*
- **Consensus Note:** All stakeholders agree that **no invoice proceeds without a PO** — even if created retrospectively. This is a hard constraint.
- **Exception Frequency:** Rita and Carlos estimate exceptions occur in 5–10% of cases. The model reflects this as optional parallel branches, not the main path.
