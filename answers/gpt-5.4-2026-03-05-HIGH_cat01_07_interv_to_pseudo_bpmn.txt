## BPMN-like textual process model: Invoice Approval Process

### 1) Process scope
This model covers the process from **invoice receipt** to **invoice scheduled for payment**.  
The actual payment execution is **out of scope**, because the interviews only describe scheduling/adding the invoice to the next payment cycle.

---

## 2) Participants / lanes

### Primary lanes
1. **Accounts Payable (AP) Clerk / AP Team**  
   - Receives invoice  
   - Checks PO presence  
   - Logs invoice  
   - Performs matching  
   - Coordinates clarifications  
   - Schedules payment

2. **Purchasing Department / Purchasing Officer**  
   - Verifies PO-related details  
   - Confirms goods/services receipt and/or order alignment  
   - Flags discrepancies

3. **Department Manager**  
   - Performs final approval or rejection/hold decision

4. **Accounting System**  
   - Stores logged invoice  
   - Sends automatic approval notification to manager  
   - Displays invoice, PO, and confirmation for review

### Exception/support lanes
5. **Accounts Payable Supervisor**  
   - Resolves unresolved exceptions  
   - Coordinates across departments/suppliers  
   - May arrange retrospective PO creation or rejection

6. **Supplier Relationship Manager**  
   - Contacts supplier for missing PO, revised invoice, or pricing clarification  
   - Checks contract terms

7. **Department Head** *(only mentioned in exception cases)*  
   - Works with AP Supervisor on retrospective PO creation for informal/unplanned services

### External participant
8. **Supplier**  
   - Sends invoice  
   - May send corrected invoice / missing PO information

---

## 3) Main artifacts / data objects

- **Invoice**
- **Purchase Order (PO)**
- **Corrected / revised invoice**
- **PO reference / missing details**
- **Purchasing confirmation email**
- **Clarification correspondence**
- **Contract / agreed supplier terms**
- **Retrospective PO** *(exception cases)*
- **Manager approval request / notification**
- **Manager approval or rejection**
- **Payment schedule / next payment cycle entry**

---

## 4) BPMN-like flow

### Start event
**Message Start Event:** Invoice is received by Accounts Payable, usually by email.

---

## 5) Normal flow with gateways and exception branches

### Step 1
**AP Task:** Receive invoice.

### Step 2
**AP Task:** Check whether the invoice contains a valid PO number/reference.

### Gateway G1 — “Valid PO present?”
#### If YES:
Proceed to normal processing.

#### If NO:
Go to missing-PO exception flow.

---

# A. Normal processing path

### Step 3
**AP Task:** Log invoice into the accounting system.

**Data:** Invoice record created in system.

### Step 4
**AP Task:** Forward invoice to Purchasing for confirmation.

### Step 5
**Purchasing Task:** Review invoice against the PO and confirm that goods/services were received and that invoice details align with what was ordered.  
This includes checking, where applicable:
- quantities
- prices
- delivery charges
- general alignment with the PO

### Gateway G2 — “Purchasing confirms invoice is acceptable?”
#### If YES:
Proceed.

#### If NO:
Go to discrepancy resolution flow.

---

### Step 6
**Purchasing Task:** Send confirmation email/message back to AP.

**Data:** Purchasing confirmation.

### Step 7
**AP Task:** Match invoice line items against the PO in the system.

### Gateway G3 — “AP match successful?”
#### If YES:
Proceed to manager approval.

#### If NO:
Go to discrepancy resolution flow.

---

### Step 8
**System Task:** Automatically send approval request / notification to the Department Manager.

**Trigger condition:**  
This occurs after:
- Purchasing confirmation is available, and
- AP has matched the invoice to the PO.

### Step 9
**Manager Task:** Review invoice, PO, purchasing confirmation, and any notes in the system.

### Gateway G4 — “Manager approves?”
#### If YES:
Proceed to payment scheduling.

#### If NO:
Go to manager rejection/hold flow.

---

### Step 10
**AP Task:** Schedule invoice for payment / add invoice to next payment cycle.

### End event E1
**End Event:** Invoice scheduled for payment.

---

# B. Missing-PO exception flow

This branch starts from **G1 = NO**.

### Step B1
**AP Task:** Return invoice to supplier / request missing PO number or additional details.

### Optional Step B2
**Supplier Relationship Manager Task:** Contact supplier to obtain PO information or request a revised invoice.

### Gateway B1 — “PO/reference obtained?”
#### If YES:
Return to **Step 2: Check valid PO**  
(or directly to **Step 3: Log invoice**, if the revised invoice now clearly contains the valid PO).

#### If NO:
Proceed to supervisor exception handling.

---

### Step B3
**AP Task:** Escalate to AP Supervisor.

### Step B4
**AP Supervisor Task:** Investigate exception.  
Possible actions:
- contact supplier
- coordinate with AP and Purchasing
- work with department heads
- determine whether a **retrospective PO** can be created
- determine whether the invoice must be rejected

### Gateway B2 — “Can a retrospective PO or other acceptable resolution be created?”
#### If YES:
**Data:** Retrospective PO created/confirmed.  
Return to **Step 3: Log invoice** (or Step 2 if revalidation is needed).

#### If NO:
### End event E2
**End Event:** Invoice rejected / process terminated.

---

# C. Discrepancy resolution flow

This branch can start from:
- **G2 = NO** (Purchasing found discrepancy), or
- **G3 = NO** (AP found mismatch)

Typical discrepancy examples mentioned:
- incorrect pricing
- extra units billed
- missing items
- invoice details not aligned with PO
- missing information

### Step C1
**AP and/or Purchasing Task:** Put invoice on hold and seek clarification.

Possible operational variants mentioned:
- Purchasing contacts supplier directly
- Purchasing asks AP to hold invoice until corrected invoice arrives
- AP emails Purchasing for clarification
- AP contacts supplier for more details

### Optional Step C2
**Supplier Relationship Manager Task:**  
If discrepancy involves supplier terms, missing PO, or pricing:
- contact supplier
- review contract/agreed pricing
- communicate correct terms back to AP or Purchasing

### Gateway C1 — “Issue resolved quickly?”
#### If YES:
Proceed based on the type of resolution:
- If a corrected invoice or new PO info is received, return to **Step 2** or **Step 5/7** as appropriate.
- If clarification confirms the original invoice is acceptable, return to **Step 6** or **Step 7**.

#### If NO:
Escalate to AP Supervisor.

---

### Step C3
**AP Task:** Escalate unresolved discrepancy to AP Supervisor.

### Step C4
**AP Supervisor Task:** Coordinate resolution across AP, Purchasing, supplier, and possibly department heads.

Possible outcomes:
- supplier provides correction
- pricing/terms are clarified
- retrospective PO is arranged
- invoice remains invalid and is rejected

### Gateway C2 — “Supervisor resolves issue?”
#### If YES:
Return to the normal flow at the appropriate control point:
- usually **Purchasing confirmation** step if ordering/receipt questions were involved
- or **AP match** step if the issue was a matching error
- or **PO validation** if a PO had to be created/confirmed

#### If NO:
### End event E2
**End Event:** Invoice rejected / process terminated.

---

# D. Manager rejection / hold flow

This branch starts from **G4 = NO**.

### Step D1
**Manager Task:** Reject invoice or request that AP hold it for investigation.

### Step D2
**AP Task:** Place invoice on hold and investigate.

### Optional Step D3
**AP Supervisor and/or Supplier Relationship Manager Task:** Assist with investigation if needed.

### Gateway D1 — “Issue resolved after manager concern?”
#### If YES:
Return to the appropriate earlier step:
- **Manager review** again if the concern was explanatory only
- **Purchasing confirmation** and/or **AP match** again if invoice content changed
- **PO validation** if the issue involved PO legitimacy

#### If NO:
### End event E2
**End Event:** Invoice rejected / process terminated.

---

## 6) Condensed sequence notation

A converter could also render the process from this compact form:

**Start**  
 AP receives invoice  
 AP checks valid PO  
 **XOR: PO present?**

**If yes:**  
 AP logs invoice in system  
 AP forwards to Purchasing  
 Purchasing verifies receipt/order alignment  
 **XOR: discrepancy found?**  
- If no  Purchasing sends confirmation  AP matches invoice to PO  **XOR: match OK?**  
  - If yes  System sends manager approval request  Manager reviews  **XOR: approve?**  
    - If yes  AP schedules payment  **End: scheduled for payment**  
    - If no  AP holds/investigates  resolve?  loop back / reject  
  - If no  discrepancy resolution  resolve?  loop back / reject
- If yes  discrepancy resolution  resolve?  loop back / reject

**If no:**  
 AP requests missing PO/details from supplier  
 optional Supplier Relationship Manager follow-up  
 **XOR: PO obtained?**  
- If yes  return to PO validation/logging  
- If no  escalate to AP Supervisor  
 retrospective PO possible?  
- If yes  return to normal flow  
- If no  **End: rejected**

## 7) Noted variations / ambiguities from the interviews

1. **Overlap between Purchasing review and AP matching**  
   - Mary says AP matches invoice line items to the PO **after** Purchasing confirms receipt.  
   - Dan says Purchasing also compares quantities, prices, and charges against the PO.  
   - To reconcile this, the model treats these as **two sequential control tasks**:
     - Purchasing performs operational/business confirmation
     - AP performs formal system-based invoice-to-PO match

2. **Missing-PO invoices may occasionally reach Purchasing**  
   - AP says missing-PO invoices are usually screened before logging/forwarding.  
   - Purchasing says sometimes he receives invoices lacking a PO and sends them back.  
   - Model assumption: **normal flow screens for PO in AP first**, but if Purchasing receives a missing-PO invoice, it should route back to the **PO validation step**.

3. **Resolution path is not always the same**  
   - Discrepancies may be handled by AP, Purchasing, Supplier Relationship Manager, or AP Supervisor, depending on severity and ease of resolution.  
   - Therefore, the exception flow should be modeled as a **branch with optional supporting tasks**, not a single rigid path.

4. **Retrospective PO creation is an exception-only path**  
   - This appears mainly for informal service purchases without an existing PO.  
   - It should be modeled as an **exception branch**, not part of the standard path.

5. **Manager rejection is relatively infrequent**  
   - Carlos says it happens only occasionally.  
   - It is still modeled as a valid exception branch because it changes process flow materially.

## 8) Recommended BPMN interpretation

If converted into BPMN, the model would likely include:

- **Pool:** Organization  
- **External Pool:** Supplier
- **Lanes:** AP, Purchasing, Manager, AP Supervisor, Supplier Relationship Manager, Accounting System, optional Department Head
- **Main XOR gateways:**  
  1. Valid PO present?  
  2. Purchasing confirms?  
  3. AP match successful?  
  4. Manager approves?  
  5. Exception resolved?
- **Message flows:**  
  - Supplier  AP: invoice  
  - AP/Purchasing/Karen  Supplier: request clarification/corrected invoice  
  - Supplier  AP/Purchasing: revised invoice / PO details  
  - Purchasing  AP: confirmation  
  - System  Manager: approval request
