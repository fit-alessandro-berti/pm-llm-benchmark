### 1. Identify the Issues with the Current Model

The current model is underfitting because it uses a `PARALLEL` (AND) operator for the core Procure-to-Pay activities (`Create_PO`, `Receive_Goods`, `Receive_Invoice`, `Match_Invoice_to_PO`, `Pay_Invoice`). 

In a process tree, the `PARALLEL` operator dictates that all children must execute, but **they can execute in any order**. This allows severely out-of-sequence and illogical behavior, such as:
* **Paying an invoice before receiving it or matching it to a PO.**
* **Matching an invoice to a PO before the PO or the invoice is even created.**
* **Receiving goods before a Purchase Order is created.**

**Strict Sequence Dependencies Required by Domain Logic:**
* You must create a requisition before approving it.
* You must approve a requisition before creating a PO.
* You must create a PO before receiving goods or invoices against it.
* You must receive the goods and the invoice before you can perform a 3-way match.
* You must match the invoice to the PO before issuing payment.
* You must pay the invoice before closing the case.

Because each step logically depends on the completion of the prior step, these activities must occur strictly in sequence.

### 2. Corrected Process Tree Model

To fix the underfitting model, we must replace the `PARALLEL` block with a `SEQUENCE` operator. This ensures the activities can only be executed in the exact order dictated by Procure-to-Pay domain logic.

from pm4py.objects.process_tree.obj import Operator, ProcessTree

create_req = ProcessTree(label="Create_Purchase_Requisition")

create_po = ProcessTree(label="Create_Purchase_Order")
