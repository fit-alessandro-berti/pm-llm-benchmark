# Process Mining Approach for Speedy Parcels Last-Mile Delivery Optimization

## 1. Process Discovery and Conformance Checking

### 1.1 Data Preprocessing and Event Log Integration

Process mining requires a single cohesive **event log** in which each row represents one event, ordered by timestamp, and grouped into **cases**. For Speedy Parcels, the case concept is most naturally the **Vehicle-Day** (e.g., `V12-20241205`) for high-level route analysis, but a **Package/Delivery-ID** case notion may also be used for parcel-level analytics, and a **Driver-Shift** case notion for workforce analysis. I would construct a unified log as follows:

| Event ID | Timestamp | Activity | Case ID (Vehicle-Day) | Case ID (Package) | Vehicle ID | Driver ID | Package ID | Location (Lat/Lon) | Speed | Event Source | Planned vs Actual | Attributes |
|---|---|---|---|---|---|---|---|---|---|---|---|---|
| ... | ... | Depart Depot | V12-20241205 | - | V12 | D105 | - | Depot | 15 | GPS/Dispatch | Planned Stop 0 | Route=R1, PlannedDep=07:45 |
| ... | ... | Arrive Customer | V12-20241205 | P9876 | V12 | D105 | P9876 | 50.81N, 6.12E | 0 | Scanner | Stop 1 | TW=08:00-10:00 |
| ... | ... | Delivery Success | V12-20241205 | P9876 | V12 | D105 | P9876 | 50.81N, 6.12E | 0 | Scanner | - | Outcome=Success |

**Preprocessing steps:**

1. **Entity resolution / ID harmonization.** GPS, scanners, dispatch, and maintenance systems may use different identifiers for the same vehicle or driver. A master mapping table must be built to align `Vehicle ID`, `Driver ID`, and `Package ID`.
2. **Timestamp synchronization.** Ensure all systems share a common clock/time zone, including daylight-saving transitions. GPS trackers usually record UTC; dispatch may store local time.
3. **Geospatial matching.** Map raw lat/lon coordinates to semantic locations: depot, customer address, traffic junctions, parking zones. This enables creation of activity labels like `Arrive Customer`, `Depart Customer`, `Idle at Intersection`, `Unscheduled Stop`.
4. **Activity abstraction.** Convert low-level GPS events into process activities. For example:
   - Speed < 5 km/h for >2 minutes outside a customer location  `Idle / Traffic Delay`
   - GPS stationary at non-depot/non-customer location  `Unscheduled Stop`
   - Scanner events  `Arrive Customer`, `Delivery Success`, `Delivery Failed`
   - Maintenance logs  `Maintenance Start`, `Maintenance End`
5. **Event correlation.** A scanner `Arrive Customer` event must be correlated to the nearest planned customer stop using spatial proximity and temporal window matching. Package-level events are nested inside vehicle-day cases.
6. **Missing event handling.** Some drivers may forget to scan; GPS data can infer the `Arrive`/`Depart` events to avoid incomplete cases.
7. **Outlier/noise filtering.** Remove GPS jumps caused by signal loss; cap unrealistic speeds; discard duplicate scanner entries.

**Key challenges:**

- **Heterogeneous granularity:** GPS emits events every few seconds, whereas scanner events are sparse. Aggregation is required to avoid a noisy, unreadable model.
- **Multi-perspective data:** Combining spatial, temporal, resource, and case-level attributes requires enrichment before mining.
- **Synchronization errors:** Clock drift between systems can create impossible sequences.
- **Ambiguous case notion:** A single package case may cross multiple vehicle-days if re-delivery occurs; this must be modeled carefully.

### 1.2 Process Discovery

Once the log is prepared, I would apply several discovery algorithms:

- **Directly-Follows Graph (DFG) / Fuzzy Miner:** Best for visualizing the end-to-end flow of a vehicle-day. Edges show how frequently one activity follows another, and edge thickness/color can represent average time between activities. This quickly reveals common paths such as `Depart Depot  Travel  Arrive Customer  Delivery Success  Depart Customer  Travel …  Arrive Depot`.
- **Heuristic Miner / Inductive Miner:** Used to derive a more structured process model (Petri net or BPMN) that handles concurrency and loops, for example the loop caused by `Delivery Failed  Re-attempt` or `Return to Depot`.
- **Dotted Chart / Performance Spectrum:** To visualize the temporal behavior of many cases simultaneously and spot deviations, delays, and seasonality.

The discovered model would include not only the happy path but also deviations such as:

- `Unscheduled Stop` due to maintenance
- `Delivery Failed` followed by `Return to Depot` or `Re-attempt`
- `Idle / Traffic Delay` between travel segments
- `Overtime` at end of shift

### 1.3 Conformance Checking Against Planned Routes

Conformance checking compares the discovered **actual** process with the **planned** routes from the dispatch system. Techniques include:

- **Token-based replay** or **alignments** on a Petri net model built from the planned route sequence.
- **Route-sequence matching:** For each Vehicle-Day, compare the ordered list of actual customer visits with the planned list using metrics such as sequence similarity, Levenshtein distance, or longest common subsequence.
- **Geospatial conformance:** Compare actual GPS traces to planned polyline routes using map-matching; detect deviations from the planned road network.

**Deviations to detect:**

| Deviation Type | Description | Data Required |
|---|---|---|
| Sequence deviation | Customer visits occur in a different order than planned. | Planned route vs. scanner/GPS order |
| Unplanned stop | Vehicle stops at a location not on the route (fuel, food, breakdown). | GPS stationary events + dispatch data |
| Missing stop | Planned customer not visited or visited next day. | Planned stops vs. scanner events |
| Timing deviation | Large gaps between planned and actual arrival times. | Planned ETA vs. actual `Arrive Customer` |
| Route geometry deviation | Driver takes a different road than planned. | GPS trace vs. planned route polyline |
| Capacity deviation | Vehicle exceeds weight/volume at any point. | Load per stop + vehicle capacity |
| Maintenance interruption | Unscheduled maintenance forces route abortion. | Maintenance logs + GPS |

By quantifying these deviations, Speedy Parcels can measure **plan adherence rate** and identify whether delays stem from non-compliance or from unrealistic planning.

## 2. Performance Analysis and Bottleneck Identification

### 2.1 Key Performance Indicators (KPIs)

The following KPIs align directly with Speedy Parcels’ punctuality and cost objectives and can be calculated from the unified event log:

| KPI | Definition | Calculation from Event Log |
|---|---|---|
| **On-Time Delivery Rate (OTD)** | % of deliveries completed within the customer time window. | Count `Delivery Success` with `Actual Arrival` inside requested TW / Total `Delivery Success` |
| **On-Time Return Rate** | % of vehicles returning to depot within planned shift end. | Count `Arrive Depot` before planned shift end / Total shifts |
| **Average Time per Delivery Stop** | Mean time from `Arrive Customer` to `Depart Customer`. | Average of `Depart Customer`  `Arrive Customer` per stop |
| **Travel Time vs. Service Time Ratio** | Proportion of shift spent driving vs. serving customers. | Sum `Travel` segments / Sum `Service` segments per Vehicle-Day |
| **Failed Delivery Rate** | % of attempted deliveries that fail. | `Delivery Failed` / (`Delivery Failed` + `Delivery Success`) |
| **Re-delivery Rate** | % of failed packages requiring a second attempt. | Packages with >1 `Delivery Failed` or second `Arrive Customer` event / Total packages |
| **Fuel Consumption per km / per package** | Fuel used normalized by distance or throughput. | Fuel sensor data (if available) or proxy: idle time + distance / packages delivered |
| **Vehicle Utilization Rate** | % of capacity (weight/volume) used during route. | Load at departure / vehicle capacity from dispatch |
| **Idle / Traffic Delay Frequency & Duration** | Number and length of low-speed/idle events outside customer locations. | Count and sum durations of events labeled `Low Speed Detected` / `Idle` |
| **Overtime Frequency** | % of shifts exceeding contracted hours. | Count shifts with `End Shift`  `Start Shift` > contracted hours |
| **Maintenance Downtime** | Time vehicles are unavailable due to repair. | Sum `Maintenance End`  `Maintenance Start` per vehicle |
| **Cost per Delivery** | Total operating cost allocated per successful delivery. | (Fuel + maintenance + labor cost) / successful deliveries |

These KPIs can be aggregated by **vehicle, driver, route, day-of-week, time-of-day, customer zone, and vehicle type** to enable drill-down analysis.

### 2.2 Bottleneck Identification Techniques

To locate bottlenecks, I would apply:

1. **Performance-annotated DFG.** Color edges by average waiting/transition time. Thick, red edges indicate where most time is lost.
2. **Critical path analysis.** On the process model, identify the longest chain of activities from `Depart Depot` to `Arrive Depot`; the activities with the highest cumulative waiting time are prime bottleneck candidates.
3. **Bottleneck pattern detection.** Use event log filtering to find patterns such as:
   - Long `Travel` segments with low speed  traffic congestion
   - Long `Arrive Customer`  `Delivery Success` gaps  difficulty finding parking or customer interaction
   - Repeated `Idle` events near customer locations  parking/search time
4. **Variant analysis.** Compare high-performing cases (OTD > 95%, low fuel) with low-performing cases to identify structural differences.
5. **Dwell time analysis.** Compute time spent at each customer location, geospatially clustered to detect areas with consistently high service times.
6. **Resource/role analysis.** Use social network / resource interaction mining to see if certain drivers or vehicles are consistently associated with delays.
7. **Heatmaps.** Overlay delay events on a map to reveal traffic hotspots or problematic delivery zones.

**Quantifying impact:**

For each bottleneck, calculate:

- **Frequency:** How often the bottleneck occurs.
- **Duration:** Average additional time consumed per occurrence.
- **Cost impact:** Convert time into cost (e.g., extra labor, fuel) using standard rates.
- **Service impact:** Correlation between bottleneck occurrence and OTD, failed deliveries, or customer complaints.

## 3. Root Cause Analysis for Inefficiencies

Identifying bottlenecks is not enough; we must understand *why* they occur. The following root causes are common in last-mile delivery and can be validated with process mining.

### 3.1 Suboptimal Route Planning (Static vs. Dynamic)

If conformance checking shows drivers frequently deviate from planned sequences *and* those deviations correlate with better on-time performance, the static routing algorithm is likely suboptimal. **Variant analysis** can compare cases where drivers followed the plan versus cases with deviations, controlling for traffic and load. If the deviating cases are consistently faster, the planning engine underestimates local road conditions or customer service variability.

### 3.2 Inaccurate Travel Time Estimations

Compare planned segment durations with actual GPS-derived travel times. A **statistical deviation analysis** by road segment and time-of-day reveals systematic under- or over-estimation. If planned ETAs are frequently missed during morning rush hours, the planning model lacks time-dependent travel speeds.

### 3.3 Traffic Congestion Patterns

By correlating GPS `Low Speed Detected` events with time-of-day, day-of-week, and road segments, Speedy Parcels can build **congestion heatmaps**. A process-enhanced map showing average speed and idle duration per road segment highlights chronic traffic hotspots that static routing may ignore.

### 3.4 High Variability in Service Time

**Dwell time analysis** computes the distribution of `Arrive Customer`  `Delivery Success` and `Depart Customer` intervals. High variance suggests:

- Difficulty locating buildings or apartments
- Lack of parking
- Extended customer interactions
- Cash-on-delivery or proof-of-signature requirements

Clustering stops by location type (residential, office, apartment, gated community) can validate these hypotheses.

### 3.5 Vehicle Breakdowns and Maintenance

Join GPS usage data (mileage, harsh braking/acceleration, idle time) with maintenance logs. **Survival analysis** can estimate time-to-failure as a function of usage patterns. Vehicles with frequent `Unscheduled Stop` events or `Engine Warning Light` occurrences can be flagged before breakdown.

### 3.6 Driver Behavior or Skill Differences

Resource-based process mining can profile each driver:

- Average speed, harsh braking events, idle time
- Adherence to planned route
- Failed delivery rate
- Overtime frequency

Statistical comparison (e.g., ANOVA or decision-tree learning) can isolate driver-specific effects from route and traffic effects. If certain drivers consistently outperform peers on the same route, their practices can be documented and shared.

### 3.7 Failed Delivery Attempts

Failed deliveries create re-delivery loops in the process model. By analyzing the **variants** ending in `Delivery Failed`, we can identify common precursors:

- Delivery attempted outside customer time window
- High-rise buildings without access codes
- Repeated absence patterns by customer or neighborhood
- Insufficient advance notification

## 5. Operational Constraints and Continuous Monitoring

### 5.1 Accounting for Operational Constraints

Any optimization must respect hard constraints:

- **Driver working hours and breaks.** Dynamic routing must include legal/contractual driving limits, mandatory breaks, and overtime thresholds. The event log already records `Start Shift` and `End Shift`; optimization should not propose schedules that violate these.
- **Vehicle capacities.** Route sequencing and territory design must respect weight and volume limits. The unified log includes planned load per stop and vehicle capacity; capacity utilization can be checked at each point in the route.
- **Customer time windows.** Dynamic rerouting can only reschedule stops within acceptable customer windows. Conformance checking flags violations.
- **Vehicle compatibility.** Some parcels require refrigerated, large, or lift-gate vehicles; maintenance scheduling must consider vehicle class.
- **Service-level agreements.** Same-day, next-day, or priority deliveries must be preserved in re-optimized routes.

Process mining helps validate that optimized plans remain feasible: by replaying optimized routes over historical logs, Speedy Parcels can simulate whether proposed routes would have violated constraints.

### 5.2 Continuous Monitoring Plan

After implementation, Speedy Parcels should deploy a **process mining-based control tower** with the following components:

#### Dashboard Metrics

- **Real-time OTD** by route, driver, zone.
- **Plan adherence rate** (sequence and timing deviations).
- **Average delay per activity** (travel, service, idle).
- **Fuel and cost per delivery** trends.
- **Failed and re-delivery rates**.
- **Vehicle health scores** from predictive maintenance.
- **Driver performance league tables** (fair, anonymized where appropriate).

#### Process Views

- **Live DFG / Performance Spectrum** updated daily to visualize current operational flow.
- **Geospatial bottleneck map** refreshed weekly to detect emerging traffic hotspots.
- **Variant explorer** to spot new deviations introduced by changes (e.g., a new depot gate procedure).
- **Conformance dashboard** comparing actual vs. optimized planned routes.

#### Monitoring Cadence

| Frequency | Activity |
|---|---|
| Real-time | Alerts for missed time windows, route abortions, breakdown predictions, overtime breaches. |
| Daily | Review previous day’s KPIs, conformance deviations, and failed deliveries. |
| Weekly | Update congestion heatmaps, driver coaching targets, maintenance schedule. |
| Monthly | Re-run territory and route-sequence optimization models with latest data. |
| Quarterly | Strategic review of cost trends, SLA performance, and model drift. |

#### Detecting New Issues

- **Process drift detection:** Statistical tests (e.g., concept drift detectors) on case durations and variant distributions to detect when the process changes unexpectedly.
- **Control charts:** Track KPIs with upper/lower control limits; flag special-cause variation.
- **A/B testing:** Compare routes/drivers using new strategies against control groups using historical baselines.
- **Feedback loops:** Feed monitoring insights back into route planning, maintenance, and training systems.

By embedding process mining into daily operations, Speedy Parcels can move from reactive firefighting to **proactive, continuous improvement**, ensuring that delivery punctuality rises and operational costs fall sustainably.
