Here are two clean, production-ready approaches. The first uses standard SQL (highly readable), and the second leverages DuckDB's native `QUALIFY` clause for maximum conciseness and performance.

###  Option 1: Standard SQL (CTE + JOIN)
```sql
WITH valid_cases AS (
    SELECT case_id
    FROM event_log
    GROUP BY case_id
    HAVING COUNT(*) <= 6
)
SELECT e.*
FROM event_log e
JOIN valid_cases v ON e.case_id = v.case_id;
```

**How it works:**
1. `valid_cases` groups the log by `case_id` and filters out any group with more than 6 events using `HAVING COUNT(*) <= 6`.
2. The main query joins the original `event_log` back to this filtered list, returning all original columns for only the qualifying cases.

---

###  Option 2: DuckDB-Native (Window Function + `QUALIFY`)
```sql
SELECT *
FROM event_log
QUALIFY COUNT(*) OVER (PARTITION BY case_id) <= 6;
```

**Why this shines in DuckDB:**
- `QUALIFY` filters on window function results directly, avoiding a subquery or join.
- DuckDB's vectorized execution engine processes this in a single pass, making it highly efficient for large event logs.
- If you only want the specific columns mentioned in the prompt, replace `SELECT *` with `SELECT case_id, activity, timestamp`.

###  Performance Tip
If your `event_log` table is large, ensure you have an index or at least a sort on `case_id`. DuckDB will automatically leverage partition pruning and sort-based aggregation, but explicitly sorting or creating a `CASE INDEX` (if applicable) can speed up the `PARTITION BY` or `GROUP BY` operations.

Both queries are fully compatible with DuckDB and will correctly exclude any case with >6 events while preserving all original event rows for the remaining cases. Choose the one that best fits your team's SQL style preferences.
