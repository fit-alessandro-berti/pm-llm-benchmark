Below is an example **DECLARE** model expressed as a Python dictionary that captures plausible declarative constraints for the multidepartment **product design and launch** process you described.  
The keys follow exactly the naming convention used by **pm4py**, and each rule’s value is a small innerdictionary containing:

* `support` – set to **1.0** (the rule is assumed to hold in all observed cases)  
* `confidence` – an illustrative confidence score (between0 and1) that reflects how strongly the organization believes the rule must be respected.

Feel free to adjust the confidence numbers or add/remove rules as your realworld data dictates.

```python
declare_model = {

    # 1 Existence / Absence / Exactlyone / Initiation constraints
    "existence": {
        # Every product idea **must** go through Idea Generation first.
        "IG": {"support": 1.0, "confidence": 0.99}

    "absence": {
        # Certain activities should never appear together in the same trace,

    },

    "exactly_one": {
        # Exactly one approval gate is allowed per product (no duplicate approvals).
        "AG": {"support": 1.0, "confidence": 0.98}
    },

    "init": {
        # The process **starts** with Idea Generation.
        "IG": {"support": 1.0, "confidence": 0.99}
    },


    # 2 Conditional / Temporal constraints
    "responded_existence": {
        # If a design draft is created (DD), then Technical Feasibility Check must follow it.
        "TFC": {"support": 1.0, "confidence": 0.96}

    "coexistence": {
        # Design Draft and Cost Evaluation can occur concurrently but not without each other once the idea exists.
        "DD": {"support": 1.0, "confidence": 0.93},
        "CE": {"support": 1.0, "confidence": 0.92}

    "response": {
        # After Technical Feasibility Check passes, a prototype can be built.
        "PC": {"support": 1.0, "confidence": 0.95}

        "DD": {"successor": "TFC", "support": 1.0, "confidence": 0.97},

        "CE":  {"successor": "PC",          "support": 1.0, "confidence": 0.95},
```

### How to read the dictionary

| Key (DECLARE type)                | Meaning in this process                                                                                                                                       |
|-----------------------------------|----------------------------------------------------------------------------------------------------------------------------------------------------------------|
| **existence**                     | `IG` must appear at least once (the start of any product).                                                                                                   |
| **absence**                       | Activities like `TFC` and `CE` are *not* allowed to occur without a preceding design draft (`DD`).                                                           |
| **exactly_one**                   | Only one approval gate (`AG`) may be recorded per product.                                                                                                    |
| **init**                          | The trace must begin with `IG`.                                                                                                                               |
| **responded_existence**           | After `DD` appears, `TFC` is required.                                                                                                                        |
| **coexistence**                   | `DD` and `CE` may cooccur once the idea exists, but not earlier.                                                                                            |
| **response / precedence / succession** | Classic “must follow” relationships (e.g., `DD  TFC`, `TFC  CE`, …).                                                                                      |
| **altresponse / altprecedence / altsuccession** | Allow alternative paths (`LT` *or* `UT` before approval, fasttrack launch).                                                                                  |
| **chain…**                        | Full linear (happypath) constraints ensuring the entire sequence proceeds without missing steps.                                                             |
| **noncoexistence / nonsuccession / nonchainsuccession** | Negative rules that forbid certain unwanted patterns (e.g., skipping approval, jumping straight from `IG` to launch).                                        |

You can feed this dictionary directly into pm4py’s DECLARErelated functions (e.g., `pm4py.discover.declare(...)`), adjust the confidence values with real data mining results, or extend it with additional activities as your process evolves.
