## 1. Analyzing Resource Behavior and Assignment Patterns

### 1.1 Prepare and enrich the event log

Before analysis, I would transform the incident event log into a process-mining-ready dataset with at least the following:

- **Case ID:** Ticket ID.
- **Activity:** Ticket Created, Assign L1, Work L1 Start, Work L1 End, Escalate L2, Assign L2, Reassign, Resolve, Close, etc.
- **Timestamp and lifecycle:** Start/Complete where available, to calculate waiting and processing times.
- **Resource:** Agent or dispatcher performing the activity.
- **Resource attributes:** Tier, team, skills, proficiency level if available, shift/calendar.
- **Ticket attributes:** Priority, category, channel, required skill, SLA target, business service/application.
- **Outcome attributes:** Resolved, escalated, reassigned, reopened, SLA breached, customer satisfaction if available.

This enriched log allows analysis at three levels:

1. **Case level:** How each incident flows through the process.
2. **Activity level:** How long each process step takes and who performs it.
3. **Resource level:** How agents, tiers, and skill groups behave in practice.

---

### 1.2 Metrics for individual agents and support tiers

I would analyze performance and behavior by agent, tier, team, skill, priority, and ticket type.

#### Workload and utilization metrics

Key metrics:

- **Number of tickets assigned per agent/tier**
- **Number of tickets actively worked**
- **Work-in-progress per agent**
- **Queue length before each agent or skill group**
- **Average and peak workload by hour/day/week**
- **Utilization rate**

Example:

> Utilization = total active work time / available working time

This helps identify agents who are overloaded versus underutilized.

For example:

- Agent B08 may have 90% utilization on Network tickets.
- Agent B15 may have 45% utilization but receives few Database-SQL tickets.
- L2 Network specialists may have large queues while L2 application specialists are idle.

#### Processing time metrics

For each agent and tier, I would calculate:

- **Average work duration**
- **Median work duration**
- **90th percentile processing time**
- **Time from assignment to work start**
- **Time from work start to work end**
- **Total ticket resolution time**
- **Waiting time versus active handling time**

Important distinction:

- **Processing time:** Time an agent actively works on a ticket.
- **Waiting time:** Time a ticket sits in a queue or waits for assignment.
- **Cycle time:** Total time from ticket creation to resolution or closure.

This is important because SLA breaches may not be caused by slow technical work, but by queueing delays before the right person starts working.

#### First-call resolution rate for L1

For L1, I would measure:

> First-call resolution rate = tickets resolved by L1 without escalation / total tickets handled by L1

This should be analyzed by:

- Ticket category
- Priority
- Channel
- Required skill
- Individual L1 agent
- Time of day
- Knowledge base article usage, if available

Example insights:

- L1 resolves 75% of Access Management tickets but only 12% of Software-App CRM tickets.
- Some L1 agents resolve significantly more Windows desktop issues than others.
- P2 Network tickets almost always escalate, suggesting they should perhaps be routed directly to L2 Network.

#### Escalation and reassignment metrics

For each agent, tier, and category:

- Escalation rate from L1 to L2
- Escalation rate from L2 to L3
- Reassignment rate within the same tier
- Reassignment rate across tiers
- Number of handovers per ticket
- Average time lost per handover
- Reassignment reasons, if captured
- “Ping-pong” cases where tickets move back and forth between teams

Example:

> Software-App tickets requiring Database-SQL may be assigned first to CRM specialists, then reassigned to Database specialists, causing an average delay of 2.3 hours.

#### Skill-specific handling metrics

For each required skill, I would measure:

- Number of tickets requiring that skill
- Number of agents with that skill
- Average queue time before first qualified agent starts work
- Average resolution time by skill
- SLA breach rate by skill
- Reassignment rate by skill
- Skill mismatch rate

Example:

> 38% of Networking-Firewall tickets are first assigned to agents without Networking-Firewall skill, and those tickets breach SLA twice as often as correctly assigned tickets.

### 1.3 Discovering actual assignment patterns using process mining

Process mining can reveal what actually happens, not just what the documented process says should happen.

#### Process discovery

Using process discovery, I would create separate process maps for:

- All tickets
- P1, P2, P3, P4 tickets
- Each category
- Each required skill
- SLA-breached versus non-breached tickets
- Reassigned versus non-reassigned tickets

This would reveal common paths such as:

- Ticket Created  Assign L1  Work L1  Resolve
- Ticket Created  Assign L1  Work L1  Escalate L2  Assign L2  Work L2  Resolve
- Ticket Created  Assign L1  Work L1  Escalate L2  Assign L2  Reassign  Assign L2  Work L2  Escalate L3

The process map would show where tickets wait, where loops happen, and where reassignments are concentrated.

#### Conformance checking

I would compare the actual process against intended assignment rules.

Examples of intended rules may include:

- P1 and P2 Security incidents should go directly to Security specialists.
- L1 should not hold P2 tickets for more than a defined threshold before escalation.
- Tickets requiring Database-SQL should be assigned only to agents with that skill.
- L3 should only handle tickets that require deep expertise.

Conformance checking can identify deviations such as:

- P2 Network tickets assigned to generic L1 agents without network skills.
- L2 specialists receiving low-complexity password reset tickets.
- Tickets reassigned multiple times before reaching a qualified agent.
- Manual dispatcher decisions bypassing routing rules.

#### Social network analysis and handover analysis

Using resource interaction analysis, I would build a handover-of-work network.

Nodes represent agents or teams. Edges represent ticket handovers.

This helps identify:

- Which agents frequently hand work to each other.
- Which teams act as bottlenecks.
- Which agents are central hubs in the process.
- Whether tickets frequently bounce between specific teams.
- Whether certain dispatchers create more reassignment loops.

Example findings:

- L1 Agent A02 frequently escalates Network tickets to L2 Agent B08.
- B08 often reassigns Firewall tickets to L3 Security Network specialists.
- This indicates that A02 may not be identifying the correct required skill at intake, or B08 is not the correct L2 target for Firewall tickets.

#### Role discovery

Role discovery clusters resources based on what they actually do.

This can reveal whether the formal tier structure matches reality.

For example:

- Some L1 agents may behave like Access Management specialists.
- Some L2 agents may spend most of their time on basic L1-type tasks.
- Some L3 specialists may be used as general overflow capacity.
- Dispatchers may assign tickets based on availability rather than skill.

This allows TechSolve to compare the documented operating model with actual behavior.

### 1.4 Analyzing utilization of specialized skills

I would create a **skill demand versus skill capacity matrix**.

For each skill:

| Required Skill | Ticket Volume | Agents with Skill | Avg Queue Time | SLA Breach Rate | Reassignment Rate | Utilization |
|---|---:|---:|---:|---:|---:|---:|
| Networking-Firewall | High | Low | High | High | High | Very high |
| Database-SQL | Medium | Medium | Medium | Medium | High | Uneven |
| App-CRM | High | High | Low | Low | Medium | Balanced |
| Security-IAM | High | Low | High | High | High | Overloaded |

This helps answer:

- Are there enough agents with each required skill?
- Are skilled agents being used on the right tickets?
- Are specialists overloaded with specialized work?
- Are specialists spending time on low-complexity work?
- Are less skilled agents being assigned to tickets they cannot resolve?

I would also analyze **skill-level mismatch**.

Examples:

- Ticket requires Networking-Firewall, but assigned agent has only Basic-Troubleshoot.
- Ticket requires Database-SQL, but assigned agent has App-CRM only.
- Ticket is a simple Access Management request, but assigned to L3 Security-IAM specialist.

A useful metric is:

> Skill match rate = tickets assigned to an agent with the required skill / total assigned tickets

Another useful metric is:

> Specialist low-complexity utilization = time specialists spend on tickets that could historically be resolved by L1 or L2 / total specialist work time

This can identify whether expensive specialist capacity is being wasted.

## 2. Identifying Resource-Related Bottlenecks and Issues

### 2.1 Bottlenecks caused by limited availability of specific skills

To identify skill bottlenecks, I would analyze:

- Queue time by required skill
- Number of open tickets waiting for each skill
- SLA breach rate by required skill
- Utilization of agents with that skill
- Time of day/week when queues grow
- Shift coverage gaps

A bottleneck may look like this:

- Tickets requiring Networking-Firewall wait 4 hours on average before L2 starts work.
- Firewall specialists have utilization above 90%.
- P2 Network tickets breach SLA at a much higher rate than other P2 tickets.
- Reassignment is frequent because tickets are first sent to general Network agents instead of Firewall specialists.

This would indicate insufficient capacity, poor routing, or both.

---

### 2.2 Delays caused by reassignments and escalations

I would quantify the impact of reassignments using several measures:

- Average number of reassignments per ticket
- Percentage of tickets reassigned at least once
- Average additional delay per reassignment
- Average resolution time for tickets with zero, one, two, or more reassignments
- SLA breach rate by reassignment count

Example structure:

| Reassignment Count | Avg Resolution Time | SLA Breach Rate |
|---:|---:|---:|
| 0 | 5.2 hours | 8% |
| 1 | 8.7 hours | 19% |
| 2 | 13.4 hours | 35% |
| 3+ | 21.1 hours | 58% |

To avoid misleading conclusions, I would compare similar tickets: same priority, category, required skill, and complexity level.

A useful impact metric:

> Average delay per reassignment = average resolution time for reassigned comparable tickets - average resolution time for non-reassigned comparable tickets

If the data shows that each reassignment adds, for example, 1.5 hours of queue and coordination delay, that becomes a direct business case for better initial routing.

---

### 2.3 Impact of incorrect initial assignments

Incorrect initial assignment can be detected using indicators such as:

- Assigned agent does not have the required skill.
- Ticket is reassigned shortly after work starts.
- Ticket is escalated after minimal L1 handling time.
- Reassignment note says “wrong team,” “needs DB,” “needs firewall,” or similar.
- Ticket category changes after initial triage.
- Required skill changes after L1 review.

Metrics:

- Initial skill mismatch rate
- Percentage of incorrect initial assignments by dispatcher or channel
- Average delay caused by incorrect initial assignment
- SLA breach rate for initially mismatched tickets
- Categories with highest first-assignment error

Example:

> 27% of P2 Network tickets are initially assigned to agents without Firewall skill. These tickets have a 42% SLA breach rate compared with 18% for correctly routed tickets.

This would strongly indicate that initial routing logic is contributing to SLA breaches.

---

### 2.4 Identification of overloaded or underutilized agents and teams

I would compare workload and performance across agents using normalized metrics.

Important: agents should not be compared only on average resolution time, because some handle more complex tickets. The comparison should be adjusted by:

- Priority
- Category
- Required skill
- Ticket complexity
- Shift
- Escalation level
- Reopen rate

Metrics:

- Utilization rate
- Assigned ticket count
- Active work time
- Queue size
- Average ticket age in agent queue
- SLA risk exposure
- Complexity-adjusted processing time
- Reassignment rate from that agent
- Escalation quality
- Reopen rate

Examples of findings:

- Agent B08 is overloaded with P2 Firewall tickets and has a growing backlog.
- Agent B15 has Database-SQL skill but receives few database tickets.
- L3 Security specialists spend 30% of their time on IAM requests that L1 could handle with better tooling.
- Some L1 agents escalate 80% of Software-App tickets, while peers escalate only 45% with similar outcomes.

---

### 2.5 Correlation between resource assignment patterns and SLA breaches

I would analyze SLA breaches using process mining combined with statistical analysis or machine learning.

Potential predictors:

- Number of reassignments
- Initial skill mismatch
- Queue time before first work start
- Tier path, e.g., L1  L2  L3
- Required skill
- Assigned team utilization at time of assignment
- Time remaining until SLA deadline at assignment
- Ticket priority
- Ticket category
- Channel
- Agent workload at assignment time
- Agent proficiency level
- After-hours or weekend creation

Techniques:

- Decision trees
- Logistic regression
- Survival analysis
- Root-cause mining
- Variant comparison
- Conformance checking against routing rules

Example metrics:

- Percentage of SLA breaches with at least one reassignment
- Percentage of SLA breaches with initial skill mismatch
- Average waiting time for breached versus non-breached tickets
- Odds ratio of breach when assigned to an overloaded queue
- SLA breach probability by routing path

Example insight:

> P3 Software-App tickets following the path L1  L2 CRM  Reassign L2 Database have a 31% SLA breach rate, while tickets routed directly to L2 Database have a 12% breach rate.

That would provide a concrete target for improving assignment logic.

## 3. Root Cause Analysis for Assignment Inefficiencies

### 3.1 Deficiencies in current assignment rules

The current logic appears to rely on round-robin assignment and manual escalation. This can cause problems if routing ignores:

- Required skill
- Agent proficiency
- Current workload
- SLA urgency
- Ticket complexity
- Historical success patterns
- Agent availability
- Shift schedules
- Queue aging

Round-robin is fair in a simple environment, but not effective when ticket requirements and agent capabilities vary significantly.

Example issue:

> A P2 Network Firewall ticket may be assigned to the next available L1 or L2 agent even if that agent lacks Firewall expertise. This creates avoidable reassignment and SLA risk.

---

### 3.2 Inaccurate or incomplete skill profiles

Assignment quality depends on reliable agent skill data.

Potential problems:

- Skill profiles are outdated.
- Skills are listed as binary yes/no instead of proficiency levels.
- Agents have undocumented expertise.
- Agents are marked skilled but rarely handle that type of ticket successfully.
- New training is not reflected in the ITSM system.
- Certifications do not match practical capability.

Process mining can validate skill profiles by comparing documented skills with actual performance.

Example:

- Agent B12 is listed as App-CRM and DB-SQL.
- However, DB-SQL tickets assigned to B12 are frequently reassigned to B15.
- This suggests B12’s DB-SQL skill level may be insufficient for certain database incidents.

---

### 3.3 Poor initial categorization or skill requirement identification

Many assignment problems start at intake.

Possible root causes:

- Ticket category selected incorrectly by users.
- Web portal categories are too broad.
- Phone agents manually choose categories inconsistently.
- Required skill is inferred too late in the process.
- Ticket descriptions are unstructured and not analyzed automatically.
- L1 agents lack diagnostic scripts to identify the right skill.

Example:

> A ticket is created as Software-App / App-CRM, but after L2 investigation it is changed to Database-SQL. The initial App-CRM assignment creates delay and reassignment.

This can be detected by analyzing category changes, required skill changes, and reassignment reasons.

---

### 3.4 Lack of real-time workload and availability visibility

Dispatchers and agents may make poor assignment decisions if they cannot see:

- Current workload per agent
- Ticket age in queue
- SLA deadlines
- Agent shift end time
- Agent availability
- Existing high-priority work
- Skill-specific queue pressure

For example, an agent may appear available but already have several P2 tickets nearing breach. Assigning another P2 ticket to that agent increases SLA risk.

---

### 3.5 Insufficient L1 training or empowerment

Excessive escalation may indicate that L1 lacks:

- Knowledge articles
- Diagnostic scripts
- Tool permissions
- Training on common categories
- Authority to perform standard fixes
- Access to automation scripts

Process mining can identify ticket types that are frequently escalated but later resolved quickly by L2.

Example:

> 60% of password synchronization tickets are escalated to L2, but L2 resolves them in under 10 minutes using a standard procedure. This suggests L1 could handle them with training and access.

---

### 3.6 Variant analysis to identify poor assignment patterns

Variant analysis compares different process paths.

I would define two groups:

#### Smooth assignment cases

Examples:

- No reassignment
- Resolved within SLA
- Correct skill assigned first time
- Low waiting time
- No unnecessary escalation

#### Problematic cases

Examples:

- Two or more reassignments
- SLA breached
- Skill mismatch
- Long queue time
- L1  L2  L3 escalation path
- Reopened after closure

Then I would compare attributes:

- Priority
- Category
- Required skill
- Channel
- Initial assignee
- Dispatcher
- Time of day
- Agent workload at assignment
- Skill match
- Ticket description keywords
- Team queue length

Example insight:

> Smooth Database-SQL tickets are usually created through the web portal with correct category selection and assigned directly to B15 or B16. Problematic Database-SQL tickets are often created by phone as Software-App and first assigned to App-CRM agents.

This would identify intake and routing rules that need improvement.

---

### 3.7 Decision mining to explain assignment choices

Decision mining examines choice points in the process, such as:

- Assign L1
- Escalate or resolve at L1
- Assign L2
- Reassign or continue
- Escalate L3

At each decision point, I would analyze which attributes influenced the decision and whether those decisions led to good outcomes.

For example, at “Assign L2,” decision mining may reveal that tickets are assigned mostly based on tier and queue rotation, not required skill or workload.

A decision tree might show:

- If category = Network and priority = P2, assign to L2 Network pool.
- But if required skill = Networking-Firewall, 45% of assignments still go to non-Firewall agents.
- Those cases have higher reassignment and SLA breach rates.

This supports changing the routing decision logic.

## 4. Developing Data-Driven Resource Assignment Strategies

Below are five concrete strategies. TechSolve could implement them incrementally, starting with the highest-impact routing problems.

---

### Strategy 1: Skill-based routing with proficiency levels

#### Issue addressed

This addresses:

- Incorrect initial assignments
- Skill mismatch
- Excessive reassignments
- Specialists receiving the wrong work
- SLA breaches due to tickets waiting for the right expert

#### How it uses process mining insights

Process mining identifies:

- Which ticket categories require which skills.
- Which agents actually resolve those tickets successfully.
- Which skills have high reassignment rates.
- Which assignments are associated with SLA breaches.
- Where formal skill profiles differ from actual performance.

Instead of assigning based on round-robin within a tier, tickets should be routed to agents whose skills match the ticket requirement.

#### Assignment logic

A basic rule:

> Assign the ticket to an available agent whose documented and validated skills match the required skill and whose tier is appropriate for the ticket complexity.

A more advanced scoring model:

> Assignment score = skill match + proficiency + historical success rate + availability + SLA urgency - current workload

Example:

- P2 Network ticket requiring Networking-Firewall should go to an agent with Networking-Firewall skill, not just any L2 Network agent.
- P3 App-CRM ticket with suspected database error should go to an agent with both App-CRM and Database-SQL or to the correct specialist queue.

#### Data required

- Required skill per ticket
- Agent skill profile
- Skill proficiency level
- Agent availability and shift schedule
- Historical resolution performance by skill
- Ticket priority and SLA deadline
- Current workload per agent
- Reassignment history

#### Expected benefits

- Fewer reassignments
- Shorter queue times
- Higher first-time-right assignment rate
- Improved SLA compliance
- Better use of specialist capacity
- Reduced frustration for agents and customers

---

### Strategy 2: Workload-aware and SLA-aware assignment

#### Issue addressed

This addresses:

- Uneven workload distribution
- Overloaded agents
- Underutilized agents
- SLA breaches caused by queue delays
- Round-robin assignment that ignores real-time capacity

#### How it uses process mining insights

Process mining reveals:

- Queue waiting times by agent and team.
- Utilization patterns.
- Peak workload periods.
- Which queues cause SLA breaches.
- How much time tickets spend waiting after assignment.

Instead of giving the next ticket to the next agent in rotation, the system should consider workload and SLA risk.

#### Assignment logic

For each candidate agent, calculate expected completion risk.

Factors:

- Current assigned WIP
- Estimated remaining work
- Skill match
- Priority
- SLA deadline
- Agent availability
- Shift end time
- Historical processing time for similar tickets
- Current queue length

Possible rule:

> Assign to the qualified agent with the lowest expected SLA risk, not simply the next agent in the queue.

For P2/P3 tickets, the algorithm should prioritize tickets based on due date and risk.

Example:

- Agent B08 has the right skill but 12 open tickets, including three P2 tickets close to breach.
- Agent B14 has the same skill and only three open tickets.
- The system assigns the new ticket to B14, even if B08 is next in round-robin.

#### Data required

- Current open tickets per agent
- Estimated processing time per ticket type
- Agent calendars and availability
- SLA deadline per ticket
- Skill profile
- Historical handling times
- Priority and category

#### Expected benefits

- Reduced queue delays
- Better workload balance
- Lower SLA breach rate
- Fewer overloaded specialists
- More transparent assignment decisions

---

### Strategy 3: Predictive triage and predictive assignment

#### Issue addressed

This addresses:

- Poor categorization
- Incorrect required skill identification
- Delayed escalation
- Wrong initial assignment
- Excessive L1 handling of tickets that clearly require L2/L3

#### How it uses process mining insights

Historical event logs show which ticket attributes predict:

- Required skill
- Needed tier
- Likelihood of L1 resolution
- Likelihood of reassignment
- SLA breach risk
- Expected resolution time

A predictive model can learn from past tickets and recommend the best initial route.

#### Assignment logic

At ticket creation, use ticket data to predict:

- Correct category
- Required skill
- Complexity level
- Best support tier
- Probability of L1 resolution
- SLA breach risk
- Recommended agent or queue

Inputs may include:

- Ticket title
- Description keywords
- Category selected by user
- Configuration item
- Service/application affected
- Priority
- Channel
- Customer/business unit
- Historical similar tickets

Example:

- Ticket description contains “CRM slow,” “SQL timeout,” and “database connection.”
- User selected Software-App / App-CRM.
- Model predicts Required Skill = Database-SQL with high confidence.
- Ticket is routed directly to L2 Database-SQL rather than first going to App-CRM L2.

#### Data required

- Historical ticket text
- Historical category and final resolved category
- Final required skill
- Resolution team
- Assignment path
- SLA outcome
- Reassignment history
- Knowledge article used, if available
- Configuration item data

#### Expected benefits

- Better first-time routing
- Reduced manual dispatcher effort
- Fewer reassignments
- Shorter resolution time
- Earlier identification of complex tickets
- Improved P2/P3 SLA performance

### Strategy 4: Refined L1 resolution and escalation rules

#### Issue addressed

This addresses:

- Excessive unnecessary escalations
- L1 underutilization
- L2/L3 specialists spending time on simple tasks
- Inconsistent L1 performance
- Slow escalation when L1 cannot resolve

#### How it uses process mining insights

Process mining can identify:

- Ticket types frequently escalated but quickly resolved by L2.
- L1 agents with high first-call resolution rates.
- Categories where L1 can be trained to resolve more.
- Categories where L1 involvement adds no value and causes delay.
- Optimal escalation timing thresholds.

#### Assignment logic

Create two types of rules:

1. **L1 empowerment rules**
   - Tickets historically resolvable by L1 should stay with L1.
   - Provide scripts, access, and knowledge articles.
   - Route to L1 agents with demonstrated capability.

2. **Fast-track escalation rules**
   - Tickets that L1 almost never resolves should skip L1 or have strict time-boxed triage.
   - For example, P2 Firewall incidents may go directly to L2/L3 Network Security.

Example:

- Access Management password reset issues should remain with trained L1 agents.
- P2 Firewall outage tickets should bypass L1 troubleshooting and go directly to L2/L3 Firewall specialists.
- L1 should spend no more than 15 minutes on certain P2 ticket types before escalation.

#### Data required

- L1 resolution rate by category and required skill
- L1 processing time before escalation
- L2 resolution time for escalated tickets
- Knowledge article usage
- Agent training records
- Tool access rights
- SLA deadlines
- Ticket complexity labels

#### Expected benefits

- Higher L1 first-call resolution
- Reduced L2/L3 workload
- Faster escalation for complex tickets
- Better use of specialist time
- Improved customer response time

---

### Strategy 5: Dynamic capacity management and cross-training

#### Issue addressed

This addresses:

- Skill bottlenecks
- Uneven workload across teams
- Recurring queues for specific skills
- Overdependence on a small number of specialists
- After-hours coverage gaps

#### How it uses process mining insights

Process mining shows:

- Demand patterns by time, category, priority, and required skill.
- Skills with chronic overload.
- Agents who could be cross-trained.
- Periods where certain queues accumulate.
- Tickets that could be handled by adjacent skill groups.

#### Assignment logic

Use demand and capacity forecasts to dynamically adjust staffing and routing.

Examples:

- Temporarily assign cross-trained L1 agents to Access Management backlog.
- Move L2 App specialists into CRM ticket handling during peak hours.
- Schedule more Firewall specialists during known high-demand windows.
- Create backup pools for Database-SQL or Security-IAM tickets.
- Train selected L1 agents on high-volume, low-complexity tasks currently handled by L2.

#### Data required

- Ticket arrival patterns
- Skill-specific demand forecasts
- Agent schedules
- Cross-training records
- Historical utilization by skill
- Backlog trends
- SLA risk by queue

#### Expected benefits

- Reduced skill bottlenecks
- Better coverage during peak periods
- Lower dependency on individual experts
- Reduced specialist overload
- Improved resilience of the service desk

## 5. Simulation, Implementation, and Monitoring

### 5.1 Using business process simulation before implementation

Before changing the assignment logic in production, I would use business process simulation based on the mined process model.

The simulation should represent:

- Ticket arrival rates by priority, category, channel, and time of day
- Routing paths between L1, L2, and L3
- Assignment rules
- Agent skills and proficiency
- Agent working calendars
- Processing time distributions
- Queueing behavior
- Escalation probabilities
- Reassignment probabilities
- SLA targets
- Resolution and closure patterns

#### Baseline simulation

First, simulate the current process to confirm that the model reproduces actual performance.

Baseline outputs should match historical patterns such as:

- Current P2/P3 SLA breach rate
- Average resolution time
- Average reassignment count
- Agent utilization
- Queue times
- First-call resolution rate
- Skill-specific bottlenecks

If the simulation approximates reality, it can be used to test alternative scenarios.

#### Scenario testing

I would simulate scenarios such as:

1. **Current round-robin model**
2. **Skill-based routing**
3. **Skill-based plus workload-aware routing**
4. **Predictive triage with direct-to-L2 routing**
5. **Improved L1 empowerment**
6. **Additional staffing for bottleneck skills**
7. **Cross-training selected L1 or L2 agents**
8. **Combination scenario**

For each scenario, measure:

- SLA breach rate by priority
- Average and median resolution time
- 90th percentile resolution time
- Reassignment count
- Queue waiting time by skill
- Agent utilization
- Specialist workload
- Cost impact
- Customer response time
- Number of tickets resolved at L1

Example simulation question:

> If Networking-Firewall tickets are routed directly to qualified L2/L3 agents and one additional cross-trained agent is added during peak hours, how much would P2 SLA breaches decrease?

Simulation allows TechSolve to evaluate improvement options before investing in tooling, staffing, or process changes.

### 5.2 Implementation roadmap

A practical implementation approach would be:

#### Phase 1: Diagnostic baseline

- Clean and enrich the event log.
- Build current process and resource dashboards.
- Identify top bottleneck categories and skills.
- Quantify reassignment and SLA impact.
- Validate findings with service desk managers and agents.

#### Phase 2: Pilot improved routing

Start with one high-impact area, for example:

- P2 Network / Firewall tickets
- P3 Software-App / Database-SQL tickets
- Access Management tickets frequently escalated unnecessarily

Implement:

- Skill-based routing
- Updated required skill mapping
- Workload-aware assignment
- L1 escalation thresholds
- Dashboard monitoring

#### Phase 3: Expand and automate

- Integrate routing logic into the ITSM platform.
- Add predictive triage using historical ticket data.
- Automate skill matching and workload checks.
- Create exception handling for urgent or ambiguous tickets.
- Update agent skill profiles continuously.

#### Phase 4: Continuous improvement

- Re-mine the process monthly or weekly.
- Monitor routing accuracy.
- Update predictive models.
- Adjust staffing and training plans.
- Use feedback from agents and customers.

## Summary

TechSolve should use process mining to move from intuition-based resource assignment to evidence-based resource management. The event log can reveal the actual flow of tickets, actual handover patterns, real agent workloads, skill mismatches, reassignment loops, and their impact on SLA breaches.

The recommended improvement direction is to replace simple round-robin and manual escalation with a combination of:

1. **Skill-based routing**
2. **Workload-aware and SLA-aware assignment**
3. **Predictive triage and assignment**
4. **Data-driven L1 empowerment and escalation rules**
5. **Dynamic capacity management and cross-training**

These strategies should be tested through simulation before full deployment and then monitored continuously with process mining dashboards. The goal is to increase first-time-right assignment, reduce reassignment delays, improve P2/P3 SLA performance, and ensure that scarce L2/L3 expertise is used where it creates the most value.
