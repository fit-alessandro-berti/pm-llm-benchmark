## 1) Process Discovery and Conformance Checking

### 1.1 Preprocess and integrate the data into a cohesive event log

Because the company has multiple sources with different granularities, the first step is to build a **harmonized, analysis-ready event log**.

#### A. Define the right case notions
A single “case” is not enough for last-mile logistics. I would recommend at least:

- **Vehicle-day case**: best for analyzing the end-to-end route execution of a van on a given day.
- **Package case**: best for analyzing delivery success/failure, time windows, and re-attempts.
- **Route/dispatch case**: best for comparing planned vs actual route execution.
- **Maintenance case**: for maintenance-specific performance and downtime.

In practice, this is often best handled via **object-centric process mining** or by maintaining **linked sublogs**.

#### B. Standardize and align the event data
From each source:

- **GPS trackers**
  - Convert raw pings into meaningful events such as:
    - Depart Depot
    - Arrive Stop
    - Depart Stop
    - Idle/Stop
    - Low Speed / Traffic Delay
    - Unscheduled Stop
  - Use **geofencing** around depots and customer locations to infer arrivals/departures.
  - Smooth noisy GPS and remove impossible jumps.

- **Driver handheld scanners**
  - Map milestones into a consistent lifecycle:
    - Scan at Depot
    - Arrive Customer
    - Delivery Success
    - Delivery Failed
    - Depart Customer
  - Ensure package IDs, stop IDs, and timestamps are preserved.

- **Dispatch system**
  - Convert planned routes into structured constraints:
    - expected stop order
    - planned stops/packages per vehicle
    - customer time windows
    - capacity constraints
    - driver assignment
  - This is not a process log yet, but a **reference model** for conformance checking.

- **Maintenance logs**
  - Convert maintenance start/end into:
    - Scheduled Maintenance Start/End
    - Unscheduled Repair Start/End
  - Link to vehicle and date, and treat these as interruptions to operational flow.

#### C. Create a common event schema
Normalize the following fields across sources:

- Timestamp
- Case ID(s)
- Activity name
- Resource attributes: vehicle, driver
- Object attributes: package, route, maintenance job
- Location coordinates / geofence
- Status / lifecycle
- Service duration / travel duration
- Planned vs actual time window
- Notes / reason codes

#### D. Key preprocessing challenges
Typical issues in this environment:

- **Different granularity**: GPS pings every few seconds, scanner events only at milestones, maintenance events only occasionally.
- **Multiple objects per event**: one route day contains many packages and one vehicle.
- **Missing events**: e.g., no scan on arrival, no explicit depart event.
- **Noisy coordinates**: urban canyons can distort GPS.
- **Timestamp misalignment**: device clocks may drift.
- **Ambiguous activity mapping**: “idle” could mean parking, waiting for access, or traffic.
- **Overlapping processes**: a vehicle can be delivering while also entering maintenance due to a warning light.
- **Case fragmentation**: a route may span midnight or be split across shifts.

A strong practical approach is to build:
1. a **vehicle-day log** for route execution,
2. a **package log** for service outcomes,
3. a **maintenance log**, and
4. linked attributes so you can analyze them together.

### 1.2 Use process discovery to visualize the actual delivery process

Once cleaned, use process discovery to obtain the real process, not the intended one.

#### Recommended discovery techniques
- **Directly-Follows Graph (DFG)** with performance overlays
  - Very useful as a first view.
  - Shows frequent transitions and average waiting/processing times.
- **Inductive Miner**
  - Produces a clean, structured model.
  - Good when you want a stable process model suitable for interpretation.
- **Heuristics Miner**
  - Useful when logs are noisy and highly variable.
- **Performance-enhanced process maps**
  - Highlight bottlenecks by color-coding time spent, waiting, or frequency.

#### What the discovered model should show
For Speedy Parcels, the “actual” end-to-end process should typically resemble:

1. Start shift / scan at depot  
2. Route assigned / loaded  
3. Depart depot  
4. Travel between stops  
5. Arrive customer  
6. Deliver success or delivery failed  
7. Depart customer  
8. Possible unplanned stop / traffic delay / maintenance interruption  
9. Return to depot  
10. End shift

The discovered model should expose:
- repeated detours
- idle periods
- failed delivery loops
- maintenance interruptions
- deviations by route, driver, or vehicle type

### 1.3 Conformance checking against planned routes

The dispatch system provides the **reference behavior**; the event log provides the **actual behavior**.

#### How to compare them
Use conformance checking methods such as:

- **Alignments**
  - Compare actual traces to the planned sequence.
  - Identify the cheapest set of deviations.
- **Token-based replay**
  - Useful for quantifying how well actual events fit the planned model.
- **Temporal conformance**
  - Compare actual arrival times and service durations against expected ones.
- **Data conformance**
  - Compare package/vehicle assignments, capacity usage, and time-window compliance.

#### Deviations to look for
- **Sequence deviations**
  - Stops visited in a different order than planned
  - skipped stops
  - repeated stops
  - extra stops not in the route plan

- **Temporal deviations**
  - late depot departure
  - delayed arrival at customer
  - excessive dwell time at stops
  - late return to depot
  - missed delivery windows

- **Operational deviations**
  - unplanned maintenance stop
  - unscheduled idle time
  - route reassignment mid-shift
  - capacity overloading
  - driver/vehicle mismatch

- **Outcome deviations**
  - delivery failed instead of successful
  - re-delivery required
  - customer not home patterns

A useful output is a **deviation dashboard** showing which route variants, vehicles, and drivers diverge most from the plan.

## 2) Performance Analysis and Bottleneck Identification

### 2.1 Key KPIs and how to calculate them

Below are the most relevant KPIs for Speedy Parcels.

| KPI | How to calculate from the event log | Why it matters |
|---|---|---|
| **On-Time Delivery Rate** | `# deliveries with delivery success within promised window / total deliveries` | Direct measure of customer service |
| **Average Time per Delivery Stop** | Mean of `(Depart Customer - Arrive Customer)` for each stop | Measures service efficiency |
| **Travel Time vs. Service Time Ratio** | Total travel time between stops / total customer service time | Helps identify whether delays are due to driving or stop handling |
| **Vehicle Utilization Rate** | Productive vehicle time / available shift time | Indicates how efficiently vehicles are used |
| **Failed Delivery Rate** | `# failed delivery attempts / total attempts` | Reveals issues with time windows, access, or customer availability |
| **Traffic Delay Frequency** | Count of low-speed/stop-and-go events above threshold | Identifies congestion exposure |
| **Traffic Delay Duration** | Total time spent in low-speed / anomalous travel states | Quantifies congestion cost |
| **Fuel Consumption per km** | Fuel used / distance traveled | Key cost metric; requires fuel telematics or fuel-card data |
| **Fuel Consumption per Package** | Fuel used / delivered packages | Useful for route-level cost comparison |
| **Maintenance Downtime Rate** | Maintenance time / total scheduled vehicle availability | Measures fleet reliability |
| **Overtime Rate** | Overtime hours / total shift hours | Indicates route overload or inefficiency |
| **Depot Dwell Time** | Time between shift start and depot departure | Captures loading or dispatch delays |

#### Important note on fuel
Your current source list does not explicitly include fuel readings. If fuel sensors or fuel-card data are available, integrate them. If not, use proxies such as:
- idle time
- stop-and-go frequency
- harsh acceleration/braking
- route distance per delivered package

Those proxies are often highly predictive of fuel consumption in urban delivery.

### 2.2 Process mining techniques for bottleneck identification

#### A. Performance-aware process maps
Overlay the discovered process with:
- average duration per activity
- waiting times
- frequency of occurrence
- delay contribution

This quickly reveals whether bottlenecks sit in:
- depot loading
- travel between certain zones
- customer stops
- failed delivery handling
- return-to-depot times

#### B. Variant analysis
Compare high-performing vs low-performing route variants:
- same area, different sequence
- same vehicle type, different driver
- same driver, different time of day

This reveals which process paths are slow or unreliable.

#### C. Time-of-day and day-of-week slicing
Bottlenecks may be concentrated:
- morning peak
- school dismissal time
- lunchtime urban congestion
- Friday afternoon
- month-end volume spikes

#### D. Spatiotemporal hotspot analysis
Map:
- low-speed events
- long dwell times
- failed deliveries
- unplanned stops

This identifies geographic bottlenecks such as:
- congested corridors
- hard-to-park zones
- gated communities
- dense apartment areas

#### E. Resource analysis
Check whether bottlenecks are associated with:
- specific drivers
- specific vehicle types
- specific depots
- certain customer clusters

#### F. Quantifying the impact
For each bottleneck, quantify:
- extra minutes per stop
- extra minutes per route
- overtime hours
- missed SLA count
- extra fuel use
- increase in reattempts
- cost per route/day/week

The most useful metric is often:
> **Total cost of the bottleneck = (delay minutes × labor cost) + extra fuel + maintenance impact + missed-delivery cost**

## 3) Root Cause Analysis for Inefficiencies

Process mining tells you **where** the problem is; root-cause analysis tells you **why**.

### 3.1 Likely root causes in Speedy Parcels’ context

#### A. Suboptimal route planning
Symptoms:
- frequent detours
- high travel time compared with route length
- repeated late returns to depot
- routes that look efficient on paper but perform poorly in practice

Possible cause:
- routes are static and not updated for real traffic, customer density, or stop-time variability.

#### B. Inaccurate travel time estimation
Symptoms:
- consistent lateness on the same road segments
- repeated underestimation on certain time windows

Possible cause:
- planning uses average travel times instead of time-of-day dependent travel times.

#### C. Traffic congestion patterns
Symptoms:
- low-speed events concentrated in specific corridors and hours
- delay spikes in morning and evening peaks

Possible cause:
- routes not adapted to local congestion patterns.

#### D. High variability in service time at customer locations
Symptoms:
- some stops are extremely short, others much longer
- certain customers or building types cause long dwell times

Possible cause:
- parking difficulty, access control, customer unavailability, building layout, or parcel handoff complexity.

#### E. Vehicle breakdowns or maintenance needs during shifts
Symptoms:
- warning light followed by unscheduled stop
- route interruptions
- overtime and missed deliveries

Possible cause:
- predictive maintenance not being used; vehicles are kept in service too long.

#### F. Driver behavior or skill differences
Symptoms:
- different delivery times for similar routes
- variation in idle time, stop time, or route adherence

Possible cause:
- differences in experience, local knowledge, parking skill, or driving style.

#### G. Failed delivery attempts
Symptoms:
- reattempts, late deliveries, extra route loops
- high failure rate in specific customer segments

Possible cause:
- poor time-window alignment, low customer availability, bad address data, or insufficient customer communication.

---

### 3.2 How to validate root causes with process mining analyses

#### Variant analysis
Compare:
- on-time vs late routes
- successful vs failed delivery cases
- low-cost vs high-cost drivers/vehicles

Look for differences in:
- route sequence
- dwell time
- travel patterns
- unplanned stops
- maintenance interruptions

#### Performance clustering
Cluster route-days by behavior:
- fast, stable routes
- congestion-heavy routes
- failed-delivery-heavy routes
- maintenance-disrupted routes

This often reveals hidden operational archetypes.

#### Correlation with external data
Enrich the log with:
- traffic data
- weather
- public events
- roadworks
- school hours
- neighborhood density

Then examine whether delays systematically increase with congestion or weather.

#### Dwell-time analysis
For each stop:
- compare arrival-to-departure times
- segment by customer type, building type, zone, or driver

This helps identify whether delay is due to the route or the customer stop itself.

#### Sequence/precedence analysis
Look for patterns such as:
- low-speed event  late arrival  failed delivery
- unplanned stop  missed time window  overtime
- repeated customer failures  route rework

#### Driver-level comparison with controls
Compare drivers on similar routes and time windows, not just raw averages. This avoids blaming drivers for route complexity.

A good approach is:
- normalize by route difficulty
- then compare performance by driver

#### Predictive modeling on process features
Use process-derived features as inputs:
- number of stops before delay
- average dwell time
- time since departure
- congestion indicators
- maintenance history
- failed delivery history

Then apply:
- decision trees
- logistic regression
- gradient boosting
- explainable AI tools

This helps confirm which factors most strongly predict lateness or failure.

### Strategy 1: Dynamic route re-optimization using live traffic and current execution data

**Targets**
- traffic delays
- late deliveries
- excessive travel time
- route deviations caused by congestion

**Root cause addressed**
- static route plans that ignore live congestion and actual stop durations

**Process mining support**
- discovery shows recurring congestion-prone segments

- performance analysis reveals which areas/time slots consistently cause delay

**What to do**
- use process-mining insights to identify unstable route segments

**Expected KPI impact**
- higher on-time delivery rate
- lower travel time
- lower overtime
- reduced fuel consumption
- fewer missed time windows

**Targets**
- inefficient stop ordering

## 5) Considering Operational Constraints and Monitoring

### 5.1 How to account for operational constraints

Any optimization must respect real-world constraints, including:

- **Driver working hours**
  - labor rules, breaks, overtime limits
- **Vehicle capacities**
  - weight, volume, stop count, package mix
- **Customer time windows**
  - promised delivery windows
- **Maintenance windows**
  - vehicle availability and workshop scheduling
- **Skill/vehicle compatibility**
  - some drivers may be trained for certain vehicle types
- **Depot cutoffs**
  - loading and dispatch deadlines
- **Priority packages**
  - same-day, express, or fragile items

In practice, process mining should feed into a **constraint-based route optimization layer** such as VRP/VRPTW with:
- live traffic inputs
- historical stop-time distributions
- maintenance availability
- driver-hour constraints
- package priority rules

This ensures the optimization is not just “fast” but also operationally feasible.
