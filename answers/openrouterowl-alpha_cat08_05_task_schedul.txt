## 1. Analyzing Historical Scheduling Performance and Dynamics

### 1.1 Reconstructing the Actual Flow of Jobs and Task Execution Sequences

The foundational step is transforming the raw MES event log into an analyzable event log conforming to standardized process mining schema. Each **Case ID** (e.g., JOB-7001) represents a process instance, and each recorded **Activity/Task** (e.g., Setup Start, Task Start (Cutting), Queue Entry (Milling)) constitutes an event, enriched with attributes (Resource, Timestamp, Duration, Priority, etc.).

**Process Discovery:**
I would apply **Inductive Miner** and **Heuristic Miner** algorithms (using tools like PM4Py in Python, or Celonite/ProM) to discover the actual process model(s) from the event log. This reveals:
- **All observed job routing variants** — the actual sequence of tasks each job underwent, including rework loops, skipped steps, or unexpected routing changes.
- **Parallelisms and synchronizations** — instances where tasks for a single job or multiple jobs overlapped on different resources.
- **Informal flow patterns** that may not exist in the nominal routing sheets but occurred in reality (e.g., re-inspection loops after quality failures).

**Performance-Annotated Process Models:**
By overlaying timing data onto the discovered model, I would generate **performance-conannotated BPMN or Directly-Follows Graph (DFGs)** where:
- Each arc (transition between activities) is annotated with **mean/median waiting time, processing time, and their distributions** (not just averages, since manufacturing times are typically right-skewed).
- Each node (activity/resource) is annotated with **frequency, sojourn time, and processing time**.
- **Bottleneck waiting times** become immediately visually apparent as arcs with the highest queue durations.

**Enrichment with Sequence Context:**
To support sequence-dependent analysis, I would construct the **"preceding job" attribute** for each event by joining the log with itself on the same machine, ordered by timestamp. This creates enriched events showing:
- Previous Job ID  Current Job ID on Resource X
- This enables setup time analysis conditioned on the transition type.

### 1.2 Specific Metrics and Techniques

#### A. Job Flow Times, Lead Times, and Makespan Distributions

- **Flow Time (Manufacturing Lead Time):** For each Case ID, `Flow Time = Task End timestamp of final operation  Job Released timestamp`. I would compute the empirical distribution (CDF), median, mean, standard deviation, and 90th/95th percentiles across all jobs. Stratification by **Order Priority** and **Job Complexity** (number of operations) would reveal disparities.

- **Makespan:** Time from release of the first job in a scheduling period to the completion of the last. Computed per scheduling interval (weekly/monthly) to understand system throughput capacity.

- **Cycle Time per Task:** `Task End  Task Start` per activity, capturing the actual touch time for each operation type.

- **Inter-Arrival Analysis:** Arrival rate of new jobs over time to understand temporal demand patterns, seasonality, and clustering.

#### B. Task Waiting Times (Queue Times) at Each Work Center

- **Queue Time per Resource:** `Task Start  Queue Entry` for each resource-machine pair. Compute distributions stratified by:
  - Time of day / shift
  - Job priority
  - Day of week (to detect shift-change delays)
  - Machine identity

- **Sojourn Time:** `Task End  Queue Entry` = Queue Time + Setup Time + Processing Time. This is the total time a job spends at a work center, including productive time.

- **Work-in-Progress (WIP) Estimation via Little's Law:** Using `WIP = Throughput Rate × Flow Time`, I can estimate average WIP and correlate it with measured WIP at different periods. Deviations indicate accumulation or starvation phases.

- **DFG Waiting Time Heatmaps:** Visualizing the Directly-Follows Graph with edge weights representing queue times to immediately identify which transitions suffer the longest waits.

#### C. Resource Utilization Analysis

| Utilization Category | Calculation Method |
|---|---|
| **Productive Processing Time** | Sum of (`Task End  Task Start`) per resource per time window / Total window duration |
| **Setup Time** | Sum of (`Setup End  Setup Start`) per resource per time window / Total window duration |
| **Idle/Available Time** | Total time machine is available but no job is queued or being processed |
| **Unavailable/Breakdown Time** | Time in `Breakdown Start  Repair End` states |
| **Overtime/Extended Shifts** | Any processing beyond standard shift hours |

**Key Distinction:** I would differentiate between:
- **Technical Availability** = 1 — (Breakdown Time / Calendar Time)
- **Operational Utilization** = Productive Time / Available Time
- **Overall Equipment Effectiveness (OEE)** component = Productive Time / Calendar Time

This reveals whether low utilization is due to breakdowns (maintenance problem), lack of demand (scheduling/planning problem), or excessive setups (sequencing problem).

#### D. Sequence-Dependent Setup Time Analysis

This is a critical analysis. For each machine:

1. **Extract Setup Events** by joining each "Setup Start/End" event with the immediately preceding job completion on the same machine. This creates a transition table:

   | From Job Type | To Job Type | Setup Duration (min) | Count |
   |---|---|---|---|
   | Material: SS304, Op: Cutting | Material: Al7075, Op: Cutting | 35 | 12 |
   | Material: SS304, Op: Cutting | Material: SS304, Op: Cutting | 8 | 28 |

2. **Construct a Setup Time Matrix** for each machine group, where rows = "from" job characteristics and columns = "to" job characteristics, cells = average setup duration.

3. **Cluster Jobs by Setup Similarity:** Using job attributes (material type, geometry complexity, fixture type, tooling requirements, tolerance class), apply hierarchical clustering or similarity grouping to identify natural "families" of jobs requiring minimal intra-group setup.

4. **Quantify Total Daily/Weekly Setup Overhead:** Sum all setup durations and express as a percentage of productive time per machine. Identify machines where setups are the dominant productivity drain.

5. **Correlation with Sequencing Decisions:** Compare actual setup transition frequencies against a random/uniform sequence baseline. If setup times are purely sequence-dependent and random sequencing yields high setup times, this quantifies the optimization opportunity.

#### E. Schedule Adherence and Tardiness

- **Tardiness per Job:** `Tardiness = max(0, Actual Completion Date  Due Date)`. Earliness can similarly be measured: `Earliness = max(0, Due Date  Actual Completion Date)`.

- **Delivery Performance Metrics:**
  - **On-Time Rate:** Percentage of jobs with Tardiness = 0
  - **Average Tardiness** (mean and distribution) across all jobs
  - **Maximum Tardiness** per period
  - **Lateness Distribution** (histogram of `Actual  Due`, including negative values for early)
  - **Conditional Tardiness:** Average tardiness *given* a job is late

- **Critical Ratio Analysis:** Compute `Critical Ratio = (Due Date  Current Time) / Remaining Processing Time` historically at each decision point. Track how often jobs with low critical ratios received attention.

- **Due Date Attainment by Priority:** Cross-tabulate On-Time Rate by priority level — if high-priority jobs have low on-time rates, prioritization is broken.

#### F. Impact of Disruptions on Schedules and KPIs

- **Breakdown Impact Analysis:**
  - For each breakdown event, identify all jobs **actively being processed** on that machine (immediate disruption), all jobs **in queue** for that machine (delayed start), and all **downstream operations** of affected jobs (cascading delay).
  - Compute **Time-to-Recovery**: Time from breakdown occurrence until the delayed job queue returns to its pre-breakdown level.
  - Measure **Knock-on Tardiness Attributable to Breakdowns:** Compare actual tardiness with a counterfactual "no-breakdown" scenario estimated via simulation.

- **Priority Change / Hot Job Impact:**
  - For each job experiencing a priority escalation, measure the **displacement effect** — which other jobs were delayed as a result? Quantify the additional waiting time injected into other jobs.
  - Compute **Priority Change Frequency and Timing:** Are hot jobs introduced early enough, or typically at the last moment?

## 2. Diagnosing Scheduling Pathologies

Based on the above analyses, I would systematically identify the following key pathologies:

### Pathology 1: Chronic Bottleneck Resources with Cascading Delays

**Evidence from Process Mining:**
- The **enriched DFG** would reveal nodes (machines) with disproportionately long incoming queue times — likely MILL-03, HEAT-01 (furnace), or QC-02 (CMM) based on the scenario's mention of "specialized bottleneck machines."
- **Resource utilization analysis** shows these machines at 90%+ productive utilization while others sit at 40-60%.
- **Variant analysis**: Comparing the sub-log of on-time jobs vs. late jobs would show that late jobs consistently spent disproportionately more time queuing at the same one or two machines.
- **Throughput analysis**: Applying **bottleneck detection algorithms** (e.g., the technique by Rojas et al. using resource contention periods, or the "bottleness score" from the inductive miner with performance data) would formally identify which resource(s) constrain overall system throughput.

**Quantified Impact:**
- "Jobs queuing at MILL-03 wait an average of X hours ( = Y), accounting for Z% of total flow time."
- "Reducing MILL-03 processing time by 10% would reduce average flow time by W% (per Little's Law and bottleneck sensitivity analysis)."

### Pathology 2: Ineffective Task Prioritization

**Evidence from Process Mining:**
- **Critical ratio analysis**: At decision points when a machine became available, the job selected did NOT have the lowest critical ratio. Plotting the critical ratio distribution of selected jobs vs. available jobs shows non-optimal selection.
- **Cross-tabulation**: High-priority jobs show on-time rates no better than or even worse than low-priority jobs, suggesting priority is ignored or undermined.
- **Due date proximity vs. sequence**: Jobs with imminent due dates are not preferentially sequenced. The correlation between `Due Date  Current Time` and queue position should be strongly negative but is weak or positive.
- **Hot job disruption analysis**: When urgent jobs are injected, they preempt lower-priority jobs BUT the preemption logic doesn't account for how far along the preempted job is (wasting partial progress).

### Pathology 3: Suboptimal Sequencing Leading to Excessive Setup Times

**Evidence from Process Mining:**
- **Setup transition matrix analysis**: Reveal that the most frequent transitions are between "dissimilar" job families requiring long setups. For example, if a machine alternates between aluminum and steel jobs (high setup) when sequencing similar materials together would halve setup time.
- **Setup time as % of productive time**: If setup times constitute 15-25%+ of total available time at any machine, sequencing optimization can recover significant capacity.
- **Comparison with optimal setup scenario**: Comparing actual cumulative setup time against a lower bound (theoretical minimum = all jobs of same family grouped perfectly) quantifies the waste.

### Pathology 4: Resource Starvation Downstream of Bottlenecks

**Evidence from Process Mining:**
- **Idle time coincidence analysis**: Periods of idle time at downstream machines align temporally with bottleneck congestion periods — when the bottleneck finally releases a batch, downstream is overwhelmed, then starves.
- **From-to flow analysis**: The flow between a bottleneck and its downstream resource is intermittent rather than steady, confirming irregular output from the bottleneck.
- **Operator utilization analysis**: If operators alternate between waiting and rushing, this indicates poor flow smoothing.

### Pathology 5: Bullwhip Effect in WIP

**Evidence from Process Mining:**
- **WIP time series analysis**: Plotting WIP level over time shows high variance and occasional "WIP explosions" — periods where WIP spikes dramatically.
- **Release pattern analysis**: If new job releases are batch-driven (e.g., many jobs released Monday morning) rather than paced, this creates periodic WIP surges, especially dangerous when the system is near bottleneck capacity.
- **Quarantine WIP**: Jobs that are early but wait excessively for a no-wait operation (e.g., heat treatment that must occur immediately after a previous step) create temporary WIP accumulation.
- **Correlation between release rate and WIP**: Computing the cross-correlation would reveal if WIP spikes follow release batching.

### Pathology 6: Breakdowns Causing Asymmetric Damage

**Evidence from Process Mining:**
- **Breakdown impact asymmetry**: Not all breakdowns are equal. Breakdowns at bottleneck machines during high-load periods cause 5-10× more tardiness than breakdowns at non-bottlenecks during low-load periods.
- **Lack of buffer management**: Jobs critical to on-time delivery are not processed on bottleneck machines near the end of a maintenance-risk window.
- **Recovery time analysis**: Post-breakdown recovery times are longer than necessary, suggesting no pre-planned job reassignment strategy existed.

## 3. Root Cause Analysis of Scheduling Ineffectiveness

### 3.1 Locking Root Causes Using Process Mining Evidence

#### Root Cause 1: Limitation of Static Dispatching Rules in a Dynamic Environment

**The Logic:** First-Come-First-Served (FCFS) and Earliest Due Date (EDD) are static, single-factor rules. They do not adapt to:
- Fluctuating shop floor load (the "best" rule depends on system utilization level)
- Sequence-dependent setup opportunities
- Downstream resource status
- Job priority changes

**Mining-Based Evidence:**
- **Contextual performance comparison**: Segment the historical log into periods of low, medium, and high load (measured by WIP level at decision time). Compute average tardiness under the current rules for each load level. If EDD works at low load but fails catastrophically at high load, the root cause is confirmed — the rule set lacks adaptive capability.
- **What-if via replay**: Replay the historical job arrivals using different rules at each decision point (conformance checking / replay approach), comparing simulated outcomes with actual. For example, if applying a "shortest processing time" rule at MILL-03 in the replay reduces average flow time by 15% in the simulation but was never actually used, the current rule choice is suboptimal.
- **Decision point analysis**: At each machine decision point (machine becomes idle, which job from the queue to process?), analyze the actual decision criteria used vs. optimal criteria. If FCFS was used but the selected job had a later due date than other queued jobs, this is a direct observation of rule deficiency.

#### Root Cause 2: Lack of Real-Time Visibility

**The Logic:** Operators and supervisors make scheduling decisions with incomplete information — they don't know downstream queue lengths, upcoming breakdowns, or the current state of critical jobs elsewhere.

**Mining-Based Evidence:**
- **Reinforcement from the event log timestamps**: If job completions at upstream machines are recorded minutes/hours before the job appears in the downstream queue, there is a physical/logistical delay, but MES data latency (if any) would compound this.
- **Conformance checking**: Comparing the actual job sequence with what an "ideal" omniscient scheduler would have chosen (given all information available at that timestamp). The gap between actual and ideal measures the cost of suboptimal information.
- **Decision blindness patterns**: Frequent instances where a machine was idle (starved) because a queued job was "reserved" for a different purpose, or where two machines duplicate work neither knew the other had started.

#### Root Cause 3: Inaccurate Task Duration Estimations

**The Logic:** Schedules (if they exist) use planned durations that don't match reality, or no time estimates are used at all (pure reactive dispatching).

**Mining-Based Evidence:**
- **Planned vs. Actual duration correlation**: Compute `Actual Duration / Planned Duration` for each task. If the ratio has high variance (e.g., CV > 0.5) or systematic bias (mean ratio  1.0), the estimation is poor.
- **Operator-specific duration analysis**: Group tasks by operator and compare. If OP-105's actual durations are consistently 20% above plan while OP-110's are at plan, this suggests either skill differences or that plans don't account for operator variability.
- **Contextual duration variability**: Actual durations may depend on material, time of day (fatigue), or prior machine state (warm vs. cold). Regression analysis on the event log attributes identifies significant predictors.
- **Impact on Scheduling**: If the scheduler uses point estimates (means) but actual distributions are wide, the schedule will be resource-overloaded because it doesn't account for variability.

#### Root Cause 4: Ineffective Handling of Sequence-Dependent Setups

**The Logic:** FFCS/EDD ignore setup times entirely, so a job that's next in FCFS order might require 40 minutes of setup when an alternative job needs only 5 minutes.

**Mining-Based Evidence:**
- **Setup-to-processing-time ratio**: If setup times are comparable to or longer than processing times for some setup transitions, then ignoring them is devastating.
- **Opportunity cost analysis**: For each actual setup transition, compute the "opportunity cost" = actual setup time — minimum possible setup time from the previous job type. Sum over a period to quantify total wasted setup time.
- **Setup transition frequency**: How often does the system incur the "worst" setup (longest duration transition)? Is this random or patterned (suggesting a fixable sequencing flaw)?

#### Root Cause 5: Inadequate Recovery Strategies for Disruptions

**The Logic:** When breakdowns occur, the response is ad hoc — jobs pile up at the broken machine, and there's no systematic reassignment to alternate machines or resequencing plan.

**Mining-Based Evidence:**
- **Post-breakdown flow analysis**: After a breakdown, trace the affected jobs. Did they return to the same machine? Were alternate machines (e.g., MILL-01, MILL-02 as fallback for MILL-03) used? If alternates were available but not used, this is a process gap.
- **Recovery time benchmarking**: Compare time-to-first-job-resumed after breakdowns of the same type. High variance suggests no standardized recovery protocol.
- **Knock-on tardiness attribution**: How many jobs became late specifically due to a breakdown that occurred more than 48 hours before their due date (suggesting recovery was too slow)?

### Differentiating: Poor Scheduling Logic vs. Resource Constraints vs. Inherent Variability

This is a critical diagnosis:

| Diagnosis | Process Mining Method |
|---|---|
| **Resource capacity limitation** | Consistently high utilization (90%+) across all days, regardless of scheduling rule. Simulated improvement from scheduling changes is minimal. Throughput cannot exceed a hard ceiling. |
| **Poor scheduling logic** | Simulation replay shows that an alternative dispatching policy (applied to the same historical demand and resource availability) achieves 20%+ improvement in KPIs without adding capacity. Utilization patterns are uneven when they should be balanced. |
| **Inherent process variability** | Even with the ideal simulated schedule, tardiness and flow time have a high irreducible floor due to stochastic processing times, unpredictable breakdowns, etc. The P90/P50 ratio for task durations is very high. |

**Combined Diagnosis for Precision Parts Inc.:**
The high-mix, low-volume nature suggests that both logic and variability contribute. The **relative magnitude** of each effect can be quantified by:
1. Replaying the historical log through an optimized scheduling strategy (if viability constraint only  minimal improvement expected).
2. Feeding a deterministic simulation with mean task durations (improvement from this vs. stochastic simulation quantifies the variability penalty).
3. Comparing simulated KPI improvements vs. observed KPIs (gap = improvement opportunity from better scheduling logic).

## 4. Advanced Data-Driven Scheduling Strategies

### Strategy 1: Dynamic Multi-Factor Dispatching Rule (DMF-DR)

**Core Logic:**
At each decision point (machine becomes available), a **composite priority index** is computed for every job in the queue based on multiple weighted factors. The job with the highest index is selected. This index is defined as:

```
Priority Index(j) = w1 × f_CR(j) + w2 × f_PRIO(j) + w3 × f_SETUP(j) + w4 × f_DS(j) + w5 × f_RPT(j)
```

Where:
- **f_CR(j)** — Critical Ratio factor: `(Due Date  Now) / Remaining Processing Time`. Values < 1.0 are already late; near 0 are critical. Nonlinearly scaled so deeply negative CR values get very high priority.
- **f_PRIO(j)** — Order priority score: numeric encoding of (High=3, Medium=2, Low=1), with escalating weight based on the historical finding that priority was poorly respected.
- **f_SETUP(j)** — Setup time consideration: Estimated sequence-dependent setup time from the CURRENT machine's last job TO job j, using the Setup Time Matrix from mining. Lower setup cost  higher factor value. Predicted from material type, fixture group, tooling class.
- **f_DS(j)** — Downstream load factor: Measures the total queued + remaining processing time at the NEXT machine(s) in job j's routing. If the downstream machine is heavily loaded, processing j now will just add to downstream WIP. This prevents bottleneck flooding.
- **f_RPT(j)** — Remaining Processing Time factor: Shorter remaining total processing time gets moderate weight (SPT effect — clears jobs faster, reducing WIP).

**Weights (w1-w5)** are tuned through the simulation (see Section 5) to minimize the weighted objective:
```
Minimize:  × Mean Tardiness +  × 90th Percentile Tardiness +  × Mean Flow Time +  × WIP
```

**Process Mining Informed Elements:**
- The setup time matrix directly provides f_SETUP lookup values.
- Historical regression on actual durations by operator/material informs the Remaining Processing Time estimates.
- Downstream queue lengths are monitored in real-time via MES event firing.
- The F_CR factor's scaling function is calibrated from the historical critical ratio distribution — at what CR threshold did tardiness become almost certain in the past?

**Addresses Pathologies:**
- **Pathology 2 (Prioritization):** f_CR and f_PRIO directly address due date and priority ignorance.
- **Pathology 3 (Setup waste):** f_SETUP drives setup-aware sequencing even at non-bottleneck machines.
- **Pathology 5 (Bullwhip/WIP):** f_DS prevents flooding downstream bottlenecks, promoting flow smoothing.
- **Pathology 6 (Breakdowns):** When a machine breaks down, its queued jobs' CR values will degrade, triggering priority escalation when reassigned.

**Expected Impact:**
| KPI | Expected Improvement |
|---|---|
| Mean Tardiness | -25 to -40% |
| On-Time Rate | +10 to +20 percentage points |
| Mean WIP | -10 to -20% |
| Setup Time Overhead | -15 to -25% |

### Strategy 2: Predictive Scheduling Engine (PSE)

**Core Logic:**
Instead of purely reactive dispatching, build a **rolling horizon scheduling model** that:
1. **Forecasts task durations** using historical distributions from process mining (not point estimates).
2. **Predicts available capacity** by integrating probabilistic breakdown models.
3. **Generates a schedule for the next N hours** (e.g., 24-48 hour rolling horizon) that optimizes a multi-objective function considering tardiness, WIP, and setup costs.

**Component A: Predictive Duration Modeling**

From the process mining analysis, build a **task duration prediction model** for each `(Task, Resource, Job_Attribute)` combination:
```
E[Duration(Task k on Machine m for Job j)] = f(Material, Geometry_Complexity, Operator, Time_of_Day, Last_Maintenance_Date)
```
Using the historical data, train:
- **Regression models** (e.g., Gradient Boosted Trees) for point predictions of duration.
- **Distributional predictions** — fit a distribution (e.g., log-normal, Gamma) to historical durations for each context, parameterized by the covariates. This captures the inherent uncertainty.

The prediction uncertainty is critical — the scheduler should know not just "this task will take 60 minutes" but "there's a 20% chance it exceeds 80 minutes."

**Component B: Predictive Breakdown Modeling**

Analyze historical breakdown events to build a **survival/reliability model** per machine:
- Time-between-breakdowns fitted to a Weibull distribution (standard for reliability engineering).
- Breakdown probability as a function of machine age, cumulative operating hours, last maintenance time, and current load.
- Required for the machine:  P(Machine m is available in the next H hours) = ...

This enables **predictive maintenance scheduling** during low-load windows and **schedule risk assessment** ("given a 30% chance of MILL-02 breakdown in 8 hours, we'll pre-identify alternative routings").

**Component C: Rolling Horizon Optimization Model**

Every H hours (or upon a disruptive event), solve:

```
Decision Variables:
  x_{j,m,t} = 1 if job j is assigned to machine m starting at time t
  C_j = completion time of job j
  
Objective (weighted):
  Minimize: 1 ×  max(0, C_j  DD_j) + 2 ×  SetupTime(prev, j, m) + 3 × Makespan
  
Constraints:
  Each machine processes at most one job at a time
  Precedence constraints (job routing)
  Sequence-dependent setup times modeled as transition times
      (if job j follows job k on machine m: gap  SetupTime(k,j,m))
  Machine availability windows (from breakdown prediction model)
  Job release dates respected
```

This is a **Job Shop Scheduling Problem with Sequence-Dependent Setup Times (JSP-SDST)**, which is NP-hard. For the practical size of Precision Parts Inc.:
- Use **Constraint Programming (CP)** solvers (e.g., Google OR-Tools) for exact solutions on short horizons.
- **Decompose** by bottleneck vs. non-bottleneck machines: solve the bottleneck machine scheduling exactly (it's typically a small number of machines with few operations each), and use heuristic dispatching rules elsewhere.
- **Metaheuristic methods** (Genetic Algorithms, Tabu Search) for medium-term planning.
- Incorporate **Monte Carlo sampling** of task durations and breakdown events to generate **robust schedules** rather than single-point schedules.

**Process Mining Informed Elements:**
- All input distributions (task durations, breakdown probabilities, setup times) come directly from process mining of historical logs.
- Routing probabilities (including rework loops) inform which machines a job will visit.
- The weights 1, 2, 3 in the objective function are informed by which KPI has the largest improvement opportunity (from the gap analysis in Section 3).

**Addresses Pathologies:**
- **Pathology 1 (Bottlenecks):** Explicit bottleneck-aware scheduling; the optimization model identifies and sequences bottleneck machines optimally.
- **Pathology 4 (Starvation):** Downstream starvation is detected in the model and addressed by pace-matching release and bottleneck schedules.
- **Pathology 6 (Breakdowns):** Predictive breakdown model enables **proactive rescheduling** before breakdowns occur, not just reactive scrambling after.

**Expected Impact:**
| KPI | Expected Improvement |
|---|---|
| 90th Percentile Tardiness | -30 to -50% (tail risk reduction) |
| Lead Time Predictability (forecast accuracy) | ±10% accuracy vs. ±40% currently |
| Effective Capacity (via predictive maintenance) | +5 to +10% |
| Makespan | -10 to -15% |

### Strategy 3: Intelligent Setup Time Optimization via Job Batching and Sequencing (ITOBS)

**Core Logic:**
This strategy specifically targets minimizing sequence-dependent setup times, modeled after the **Travelling Salesman Problem (TSP)** structure inherent in setup minimization: each "city" is a job, each "distance" is the setup time between two jobs, and the goal is to find the minimum-total-distance tour (sequence) of jobs through each machine.

**Component A: Setup Time Opportunity Quantification**

From the setup transition matrix (Section 1.2D), compute:
- **Total setup waste per period** = Actual cumulative setup time — Theoretical minimum setup time.
- If total setup waste exceeds 10% of productive time, this strategy has major potential.

**Component B: Dynamic Job Grouping and Sequencing**

When a machine completes a job:

1. **Look ahead at the queue** (plus near-future arrivals, using the predictive model from Strategy 2).

2. **Identify the "setup-optimal" next job:** Among all candidate jobs in the queue, select the one with the **minimum setup time from the current job**. If the minimum setup time is small ('family' job), process it immediately. If no family job exists, evaluate whether **waiting** for a family job (if one is expected within a computed wait threshold) is beneficial.

3. **Optimal sequencing for batch formation:**

   Formally, given N queued jobs and the setup time matrix S[i][j], find the sequence  that minimizes:
   ```
   Total_Setup() = S[last_completed][(1)] +  S[(i)][(i+1)] for i=1..N-1
   ```
   
   For N < 15: solve exactly via dynamic programming (Held-Karp algorithm) or CP-SAT.
   For N  15: use nearest-neighbor heuristic with 2-opt improvement, or simulated annealing.

4. **Batching Policy:**
   
   Allow a **limited queue buffering** at setup-prone machines:
   - Set a **minimum batch threshold**: Don't start a different-family job until at least K same-family jobs have accumulated, OR a maximum wait time W has elapsed (to prevent excessive waiting for a "perfect" sequence that never comes).
   - K and W are set by analyzing the historical trade-off:
     ```
     Marginal Setup Saving from Waiting = Setup_Reduction  (Waiting_Cost of delayed jobs)
     ```
     Find the point where marginal saving becomes negative.

5. **Bottleneck Priority:**
   
   This strategy is **prioritized at bottleneck machines** because:
   - Setup time savings at a bottleneck directly increase system throughput (Theory of Constraints).
   - Setup time savings at a non-bottleneck just increases idle time (less impactful).

**Component C: Pre-Sequencing Compatibility Matrix**

Based on jobs' material, tolerance, fixture, and tooling requirements, build a **compatibility/similarity score** between any two jobs (even if they've never been sequenced together historically). This generalizes the setup matrix and handles new job types:
```
Compatibility(Material_A, Material_B, Tolerance_A, Tolerance_B, Fixture_A, Fixture_B, Tool_A, Tool_B)
 Predicted Setup Time
```
Learned via regression on the historical setup time matrix. Enables **proactive batching decisions** for new incoming jobs.

**Process Mining Informed Elements:**
- The entire setup transition matrix is mined from historical data.
- Setup time distribution models exist per transition type (not just means), accounting for variability.
- The wait-time vs. batching trade-off is calibrated from historical patterns of what wait times were "too long" (jobs became late) vs. "acceptable."
- Operator skill matrix (some operators do setups faster) can be incorporated.

**Addresses Pathologies:**
- **Pathology 3 (Setup waste):** Directly and aggressively targets this.
- **Pathology 1 (Bottleneck):** Capacity freed by setup reduction at bottleneck machines is additional system capacity — equivalent to adding a machine.
- **Pathology 5 (Bullwhip/WIP):** Batched processing at setup-intensive machines creates slightly more WIP in those queues but reduces batch arrival variance downstream.

**Expected Impact:**
| KPI | Expected Improvement |
|---|---|
| Total Setup Time | -30 to -50% |
| Capacity at Bottleneck Machines | +10 to +15% (effective, without capital expenditure) |
| Mean Flow Time | -10 to -15% (secondary, from freed bottleneck capacity) |
| Throughput | +8 to -12% |

## 5. Simulation, Evaluation, and Continuous Improvement

### 5.1 Discrete-Event Simulation Framework

**Model Construction:**

Build a **discrete-event simulation (DES)** model using simulation software (e.g., Siemens Tecnomatix Plant Simulation, AnyLogic, Simio, or Python's SimPy for custom models) parameterized entirely from process mining:

| Simulation Parameter | Source from Process Mining |
|---|---|
| Job arrival process | Inter-arrival time distribution, release pattern, seasonal demand |
| Job routing | Variant frequencies (probability of each routing path) from process discovery |
| Task duration distributions | Empirical distributions per (task type, resource, material, operator) from event log analysis |
| Setup time matrix | Setup transition matrix, conditioned on job similarity attributes |
| Breakdown model | Time-to-failure (Weibull parameters) per machine from breakdown event analysis |
| Operator availability | Shift schedules, skill levels, learning curves from historical data |
| Priority assignment distribution | Historical priority frequencies |
| Hot job injection pattern | Frequency and timing of priority changes from event log |
| Rework probability | Conformance checking reveals non-conforming paths and their frequencies |

**Validation:**
- Run the simulation model with the **actual historical dispatching rules** (FCFS/EDD) and input the **actual historical job sequence**.
- Compare simulated KPIs (flow time, tardiness, WIP, utilization) with **observed KPIs** from the historical log.
- Calibrate until the simulation accurately reproduces historical performance metrics within ±5% (face validity and statistical validation).

**Testing Proposed Strategies:**

Apply each of the three strategies (and combinations) to the validated simulation, using:
- **Same historical job stream** (controlled comparison across strategies).
- **Multiple replications** (30+ runs per scenario, each with different random number seeds) to compute confidence intervals on KPIs.

**Specific Scenarios to Test:**

| Scenario | Purpose | Parameters |
|---|---|---|
| **Baseline** | Benchmark | Historical arrival pattern, current dispatching rules |
| **High Load Stress Test** | System robustness | 120%, 150%, 180% of current arrival rate |
| **Frequent Disruption** | Resilience | 3x breakdown frequency, 2x hot job injection rate |
| **Mixed Disruption Types** | Combined stress | High load + frequent breakdowns + hot jobs |
| **Demand Variability** | Handling uncertainty | Alternating high/low demand weeks (2x swing) |
| **New Product Introduction** | Generalization | 30% of jobs have no historical analogs (use similarity-based estimation) |
| **Operator Absence** | Resource reduction | One critical operator unavailable |
| **Bottleneck Removal** | Capacity sensitivity | Adding one more bottleneck machine (to compare cost of capital vs. cost of scheduling optimization) |

**Metrics Tracked per Scenario:**
- Mean, median, P75, P90, P95 of job flow time
- Mean tardiness, maximum tardiness, on-time percentage
- WIP distribution over time
- Machine utilization breakdown (productive/setup/idle/breakdown)
- Schedule makespan
- Total setup time
- Earliness (to detect over-correction where jobs finish too early, consuming storage)

### 5.2 Comparison and Selection

Construct a **Pareto frontier analysis** plotting strategies against multiple objectives (tardiness vs. WIP vs. utilization). The chosen strategy (or parametric variant) lies on the frontier and is acceptable to stakeholders on all metrics.

### 5.3 Continuous Monitoring and Adaptation Framework

**Real-Time KPI Dashboard (Process Control):**

Deploy a **real-time process monitoring dashboard** that tears the incoming event log stream into a continuous performance measurement system:

```
KPI Dashboard (Refreshed Every 15 Minutes):
 WIP Level: Current vs. Target Band
 Machine Status: [Real-time Gantt chart]
 Tardiness Risk: Jobs at risk of being late (CR < 1.2)
 Queue Alert: Queues exceeding normal range
 Setup Efficiency: Actual vs. Predicted setup times
 Schedule Adherence: % of jobs on schedule for next 24 hrs
```

**Statistical Process Control (SPC) for KPIs:**

Apply **control charts** (e.g., X-bar and R charts, or individuals charts for non-grouped data) to key metrics:
- **Flow time per job** (plotted on an individuals chart weekly)
- **Queue time at bottleneck** (updated per job completion)
- **Actual vs. Predicted setup time** (to detect drift in setup model accuracy)
- **Tardiness** (weekly aggregate)

**Rule:** If a KPI exceeds its UCL/LCL (Upper/Lower Control Limit), or shows a sustained shift (e.g., 7 consecutive points above centerline), an **automatic alert** is generated for the production planning team.

**Drift Detection and Model Retraining:**

Manufacturing environments evolve — new products, operator turnover, machine aging, supplier changes. Process mining enables **continuous model refresh:**

1. **Weekly:** Update task duration models with last week's data. Check if distribution parameters have shifted (using change-point detection algorithms — e.g., Bayesian Online Change Point Detection, or Page-Hinkley test).

2. **Monthly:** Re-run the full setup time matrix analysis. Check for new job types or changed setup patterns.

3. **Quarterly:** Re-validate the simulation model against the latest 3 months of performance. Are simulated predictions still matching reality? If not, recalibrate.

4. **Annually:** Full re-evaluation of the scheduling strategy weights (w1-w5 in DMF-DR, batching parameters in ITOBS) using the latest year's data. Re-run simulation with updated distributions.

**Adaptive Scheduling Logic:**

Implement a **self-tuning mechanism:**
```
If (Measured Tardiness > Target × 1.2 for 2 consecutive weeks):
   Increase w1 (Critical Ratio weight) in DMF-DR
   Re-run simulation to verify improvement
   Deploy updated weights

If (Bottleneck utilization < 80% for 1 month):
   This suggests demand reduction OR that previous optimization recovered capacity
   Reassess constraint — has the bottleneck shifted to a different machine?
   Re-center optimization focus on new bottleneck

If (Actual flow time consistently below predicted 90th percentile):
   Scheduling is conservative — duration models may be biased high
   Relax duration estimates slightly, allowing more aggressive scheduling
```

**Closed-Loop Feedback Architecture:**

```
        
  Scheduling    Shop Floor     MES Event Log

                                         Process Mining

```

This creates a **learning manufacturing system** where scheduling decisions are continuously informed by actual performance, and the performance measurement is continuously refined by process mining of operational data.

This approach transforms scheduling from a reactive, firefighting mode into a **proactive, data-driven, continuously self-improving system** — turning Precision Parts Inc.'s MES data from a passive record into an active competitive advantage.
