We used the following **matching strategy**:

- **Timestamp Selection**: When merging, we **preserved the original timestamp** from both logs, but used **Log B’s timestamp** as the primary anchor when it was earlier, as it reflects a system closer to the physical process (warehouse). ##  **Conflict Resolution & Attribute Integration Strategy**

| **Event name differences** | Mapped semantically: `OrderReceived`  `Order Received`, `OrderValidation`  `Order Validated`, `Shipping`  `Item Shipped` |
| **Timestamp discrepancies** | Used ±2 seconds as tolerance; resolved by selecting the **earlier timestamp** where events were matched (to reflect actual event timing), but **both timestamps are preserved in context**. |
| **Missing events in one log** | Events not matched are **included separately**, tagged by source. |
| **Attribute enrichment** | When merging, included **all attributes** from both logs. For non-matched events, used only the available attributes. |
| **Chronological ordering** | All events sorted by **original timestamp** (from either log), ensuring temporal accuracy. |
