## 1. POWL model with potential unfairness

This model includes:
- a loop for data completeness,
- sequential progression to skill assessment,
- an XOR after skill assessment,
- one XOR branch with a standard cultural fit check,
- another XOR branch with a `CommunityAffiliationCheck`, which can introduce unfairness,
- then managerial review and final decision.

### Structural idea
- `ReceiveApplication`
- loop: `DataCompletenessCheck` with `RequestMoreInfo`
- `SkillAssessment`
- XOR:
  - `CulturalFitCheck`
  - `CommunityAffiliationCheck`
- `ManagerialReview`
- `FinalDecision`

### POWL expression
```text
PO = (
  nodes = {
    ReceiveApplication,
    * ( DataCompletenessCheck, RequestMoreInfo ),
    SkillAssessment,
    X ( CulturalFitCheck, CommunityAffiliationCheck ),
    ManagerialReview,
    FinalDecision
  },
  order = {
    ReceiveApplication --> * ( DataCompletenessCheck, RequestMoreInfo ),
    * ( DataCompletenessCheck, RequestMoreInfo ) --> SkillAssessment,
    SkillAssessment --> X ( CulturalFitCheck, CommunityAffiliationCheck ),
    X ( CulturalFitCheck, CommunityAffiliationCheck ) --> ManagerialReview,
    ManagerialReview --> FinalDecision
  }
)
```

### pm4py-style construction
```python
import pm4py
from pm4py.objects.powl.obj import StrictPartialOrder, OperatorPOWL, Transition
from pm4py.objects.process_tree.obj import Operator

ReceiveApplication = Transition(label="ReceiveApplication")
DataCompletenessCheck = Transition(label="DataCompletenessCheck")
RequestMoreInfo = Transition(label="RequestMoreInfo")
SkillAssessment = Transition(label="SkillAssessment")
CulturalFitCheck = Transition(label="CulturalFitCheck")
CommunityAffiliationCheck = Transition(label="CommunityAffiliationCheck")
ManagerialReview = Transition(label="ManagerialReview")
FinalDecision = Transition(label="FinalDecision")

data_loop = OperatorPOWL(
    operator=Operator.LOOP,
    children=[DataCompletenessCheck, RequestMoreInfo]
)

culture_xor_unfair = OperatorPOWL(
    operator=Operator.XOR,
    children=[CulturalFitCheck, CommunityAffiliationCheck]
)

root_unfair = StrictPartialOrder(
    nodes=[
        ReceiveApplication,
        data_loop,
        SkillAssessment,
        culture_xor_unfair,
        ManagerialReview,
        FinalDecision
    ]
)

root_unfair.order.add_edge(ReceiveApplication, data_loop)
root_unfair.order.add_edge(data_loop, SkillAssessment)
root_unfair.order.add_edge(SkillAssessment, culture_xor_unfair)
root_unfair.order.add_edge(culture_xor_unfair, ManagerialReview)
root_unfair.order.add_edge(ManagerialReview, FinalDecision)
```

### Why this model is potentially unfair
The unfairness is represented by:
```text
X ( CulturalFitCheck, CommunityAffiliationCheck )
```
This means some applicants may go through a different evaluative path based on affiliation/local cues, which can give a subtle advantage.

## 2. POWL model without unfairness

This model keeps:
- application receipt,
- data completeness loop,
- skill assessment,
- one single cultural fit evaluation for everyone,
- managerial review,
- final decision.

There is **no XOR branch** for affiliation/community-based treatment.

### Structural idea
- `ReceiveApplication`
- loop: `DataCompletenessCheck` with `RequestMoreInfo`
- `SkillAssessment`
- `CulturalFitCheck`
- `ManagerialReview`
- `FinalDecision`

### POWL expression
```text
PO = (
  nodes = {
    ReceiveApplication,
    * ( DataCompletenessCheck, RequestMoreInfo ),
    SkillAssessment,
    CulturalFitCheck,
    ManagerialReview,
    FinalDecision
  },
  order = {
    ReceiveApplication --> * ( DataCompletenessCheck, RequestMoreInfo ),
    * ( DataCompletenessCheck, RequestMoreInfo ) --> SkillAssessment,
    SkillAssessment --> CulturalFitCheck,
    CulturalFitCheck --> ManagerialReview,
    ManagerialReview --> FinalDecision
  }
)
```

### pm4py-style construction
```python

from pm4py.objects.process_tree.obj import Operator
```

### Why this model removes the unfairness
All applicants follow the same path:
```text
SkillAssessment --> CulturalFitCheck --> ManagerialReview
```
There is no special branch such as `CommunityAffiliationCheck`, so the process no longer structurally encodes selective advantage based on community affiliation or local ties.
