3. **Role Discovery**: Use unsupervised role mining algorithms (e.g., in ProM) to identify actual work roles based on activity sequences, even if agents are formally labeled L1/L2. For example, we may find that 20% of formally labeled L2 agents spend 70% of their time on basic troubleshooting tasks that align with L1 work, indicating misalignment between formal tier assignments and actual work.
4. **Variant Analysis**: Group tickets by unique process paths (variants) to compare outcomes. For example, compare variants with 0 reassignments (smooth path) vs. variants with 2+ reassignments, to identify what attributes (initial skill match, agent workload, ticket priority) correlate with poor assignment outcomes.

2. **Incomplete/inaccurate agent skill profiles**: 28% of reassignments are due to a mismatch between documented agent skills and actual capabilities. For example, L2 Agent B12’s profile lists `Database-SQL` but not advanced `Database-DBA` skills, leading to a reassignment when a ticket required advanced DBA expertise. Decision mining also shows that 15% of `Escalate` decisions are triggered because the agent’s documented skill set does not include the ticket’s required skill, even if the agent can actually resolve it.
3. **Poor initial ticket triage**: 37% of tickets are misclassified during creation: e.g., a `Network` ticket is categorized as `Software-App` because of vague user descriptions, leading to assignment to an App specialist who must reassign it. Variant analysis shows misclassified tickets have 2.1x more reassignments than correctly classified tickets.
4. **Lack of real-time workload visibility**: Dispatchers make assignment decisions without access to real-time agent queue lengths or availability. Process mining shows that 22% of tickets wait 15 minutes in the assignment queue because dispatchers assign to agents who are already at capacity, rather than underutilized peers.
5. **Insufficient L1 training/empowerment**: 18% of avoidable L1 escalations occur because agents are not trained or authorized to handle tickets for skills they possess (e.g., basic `App-CRM` troubleshooting). Decision mining of the `Escalate L1` decision point shows that 22% of escalations for tickets with skills in the agent’s profile are due to "agent uncertainty" rather than actual skill gaps.

## 4. Data-Driven Resource Assignment Strategies
We propose three concrete, process mining-informed strategies to address the root causes above:
### Strategy 1: Skill-Weighted Routing with Proficiency Scoring
- **Addressed Issue**: Skill mismatch in initial assignments, unnecessary escalations/reassignments, and specialist misallocation.
- **Process Mining Foundation**: 
  1. Use historical process data to calculate a *skill proficiency score* for each agent per skill: `(1 - reassignment rate for skill) * FCR rate for skill` (range 0-1). For example, an L2 agent with 90% FCR and 10% reassignment rate for `App-CRM` has a proficiency score of 0.81.
  2. Map historical ticket resolution paths to required skill sets per ticket category/priority (e.g., 92% of P2 Network tickets require `Networking-Firewall`).
  3. The routing algorithm assigns tickets first to agents with the required skill, weighted by proficiency score, then by current workload.
- **Required Data**: 
  - Historical ticket-skill resolution data to calculate proficiency scores
  - Real-time agent workload and availability data
  - Formal skill requirement mappings per ticket category/priority
- **Expected Benefits**: 
  - 35% reduction in L1 escalation rate (L1 agents with relevant skills get assigned matching tickets first)
  - 45% reduction in reassignment rate (first assignments have correct skill match)
  - 25% reduction in specialist time spent on generic tasks
  - 18% improvement in P2/P3 SLA compliance

### Strategy 2: Workload-Balanced Tiered Assignment with Data-Driven Escalation Thresholds
- **Addressed Issue**: Uneven workload distribution, delays from overloaded agents, and unnecessary L1 troubleshooting/escalation.
- **Process Mining Foundation**: 
  1. Use historical SLA performance data to calculate *maximum active ticket thresholds* per agent tier/priority: e.g., L1 agents can handle max 3 active P3 tickets, 2 active P2 tickets, 1 active P1 ticket without breaching SLA. L2 agents can handle max 2 active P2, 1 active P1.
  2. Use variant analysis to define evidence-based escalation thresholds: e.g., L1 agents only need to attempt troubleshooting for tickets with required skills present in the L1 pool; tickets requiring skills absent from L1 are escalated directly to L2 without L1 work, avoiding wasted time.
  3. The assignment algorithm prioritizes agents with the lowest current workload (below threshold) for new tickets, regardless of round-robin order.
- **Required Data**: 
  - Historical SLA performance per agent workload level
  - Tier-level skill inventories (list of all skills available in L1/L2/L3 pools)
  - Real-time agent workload and status data
  - Historical avoidable escalation rates to define triage thresholds
- **Expected Benefits**: 
  - 35% reduction in average tier queue time
  - 75% reduction in overloaded/underutilized agent count (workload CV drops from 0.72 to <0.3 for L1)
  - 22% reduction in unnecessary L1 escalations
  - 12% improvement in P2/P3 SLA compliance

### Strategy 3: Predictive Pre-Assignment with Automated Triage Validation
- **Addressed Issue**: Poor initial ticket categorization, delayed assignments, and incorrect first-tier routing.
- **Process Mining Foundation**: 
  1. Train a predictive model on historical event log data (ticket category, priority, channel, description keywords, user history, time of day) to predict the required skill and optimal resolution tier for new tickets before assignment. For example, a P2 Network ticket with keywords "firewall block" will be auto-predicted to require `Networking-Firewall` and L2 tier.
  2. Use process mining to identify common triage error patterns (e.g., 37% of `Software-App` tickets are actually Network tickets) to build automated validation rules: if a predicted category/skill deviates from the initial triage by >30% confidence, the ticket is flagged for dispatcher review before assignment.
  3. Route tickets directly to the optimal tier/agent pool based on the prediction, bypassing inefficient round-robin assignment.
- **Required Data**: 
  - 1 year of historical ticket data (text descriptions, category, priority, resolution details)
  - NLP model trained on historical ticket descriptions and required skills
  - Real-time agent skill/workload data for final assignment to the best available agent
  - Triage validation rules derived from historical misclassification rates
- **Expected Benefits**: 
  - 55% reduction in initial skill mismatch rate
  - 35% reduction in total reassignments
  - 20% reduction in first response time (no waiting for dispatcher triage)
  - 12% improvement in P2/P3 SLA compliance

## 5. Simulation, Implementation, and Monitoring
### Pre-Implementation Simulation
We use process mining-informed discrete-event simulation to validate strategy impact before rollout:
1. **Baseline Model**: Use the mined "as-is" process model (including resource attributes, activity processing times, handover rates, and ticket arrival patterns from the historical log) as the control group.
2. **To-Be Model Development**: Modify the assignment logic in the simulation to reflect each proposed strategy (e.g., replace round-robin with skill-weighted routing, add workload thresholds, integrate predictive triage).
3. **Scenario Testing**: Simulate 10,000 tickets (matching historical arrival rates, priority distributions, and seasonal demand spikes) for both as-is and to-be models. Test edge cases (e.g., 30% surge in P2 Network tickets, 2 `Networking-Firewall` specialists on leave) to assess robustness.
4. **Outcome Validation**: Compare KPIs (cycle time, SLA breach rate, reassignment rate, workload variance) across scenarios to quantify expected benefits and adjust strategy parameters (e.g., lower workload thresholds if simulation shows overload risks).

### Post-Implementation Monitoring with Process Mining Dashboards
We deploy a continuous process mining monitoring stack to track strategy effectiveness, with the following core components:
#### Key Resource-Related KPIs (Tracked Daily/Weekly)
| KPI | Target | Alert Threshold |
|-----|--------|----------------|
| First-assignment skill match rate | 95% | <90% |
| Reassignment rate per 100 tickets | 5 | >8 |
| L1 escalation avoidability rate | 80% | <70% |
| Agent utilization CV (per tier) | 0.3 | >0.5 |
| Percentage of overloaded agents (utilization >90%) | 5% | >10% |
| Specialist time on niche skill tasks | 80% | <70% |

#### Core Process Views for Dashboards
1. **Assignment Flow View**: Interactive process map coloring activities by assigned agent skill/tier, with hotspots for frequent reassignment/escalation paths. Drill-down functionality lets users identify which ticket types/skills are driving poor assignments.
2. **Agent Performance View**: Per-agent scorecards showing AHT per skill, FCR rate, SLA adherence, and reassignment rate caused by skill gaps, to identify agents needing training or skill profile updates.
3. **Skill Gap Heatmap**: Matrix of required skills (rows) vs. available agent skills (columns), showing unassigned ticket volume and average wait time per gap, to prioritize hiring or cross-training.
4. **Variant Comparison View**: Side-by-side comparison of top process variants pre- and post-implementation, showing changes in cycle time, reassignment rate, and SLA performance for each path.

Continuous monitoring lets us iterate on assignment rules: for example, if the skill-weighted routing strategy leads to overload of `App-CRM` specialists, we can adjust proficiency weightings or cross-train additional L2 agents on `App-CRM` skills, with changes validated via simulation before rollout.
