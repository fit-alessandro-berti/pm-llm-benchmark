# Comprehensive Process Mining Analysis for Speedy Parcels

## Executive Summary

This report outlines a detailed process mining approach to analyze Speedy Parcels' last-mile delivery operations, identify root causes of inefficiencies, and propose data-driven optimization strategies. By leveraging the rich event data from GPS trackers, driver scanners, dispatch systems, and maintenance logs, we can uncover operational insights that directly address delivery punctuality challenges and escalating operational costs.

## 1. Process Discovery and Conformance Checking

### 1.1 Data Preprocessing and Integration

#### Integration Architecture

Creating a cohesive event log from four disparate data sources requires a systematic approach to data harmonization:

**Step 1: Define a Unified Case Notion**

The case concept is foundational to process mining. For Speedy Parcels, I recommend a **hierarchical case structure**:

- **Primary Case ID**: Vehicle-Day (e.g., V12-20241205) - Captures the complete operational journey
- **Secondary Case ID**: Package-Journey (e.g., P9876-20241205) - Tracks individual package lifecycle
- **Tertiary Case ID**: Driver-Shift (e.g., D105-20241205) - Enables driver performance analysis

```
Case Hierarchy:
 Vehicle-Day (V12-20241205)
     Package 1 (P9876) journey events
     Package 2 (P9877) journey events
     ... Package N events
     Vehicle movement events
     Maintenance events (if applicable)
```

**Step 2: Timestamp Synchronization and Alignment**

| Data Source | Timestamp Precision | Synchronization Challenge | Resolution Strategy |
|-------------|--------------------|--------------------------|--------------------|
| GPS Trackers | ~1-5 seconds | Clock drift across devices | NTP synchronization; establish ±3 second tolerance |
| Driver Scanners | ~1 second | Manual trigger delays | Accept as ground truth for delivery activities |
| Dispatch System | Minute-level | Planning timestamps vs. execution | Treat as reference; don't merge with execution timeline |
| Maintenance Logs | Variable | Manual entry delays | Use as supplementary attribute data |

**Step 3: Event Harmonization Schema**

```sql
CREATE TABLE unified_event_log (
    event_id VARCHAR PRIMARY KEY,
    timestamp TIMESTAMP NOT NULL,
    case_id_vehicle_day VARCHAR NOT NULL,
    case_id_package VARCHAR,
    activity VARCHAR NOT NULL,
    event_source VARCHAR NOT NULL,  -- 'GPS', 'Scanner', 'Dispatch', 'Maintenance'
    vehicle_id VARCHAR NOT NULL,
    driver_id VARCHAR,
    location_lat DECIMAL(10,8),
    location_lon DECIMAL(11,8),
    speed_kmh DECIMAL(5,2),
    planned_sequence_position INT,
    actual_sequence_position INT,
    customer_time_window_start TIMESTAMP,
    customer_time_window_end TIMESTAMP,
    activity_duration_seconds INT,
    distance_from_previous_km DECIMAL(8,3),
    fuel_consumption_liters DECIMAL(6,3),
    notes TEXT
);
```

**Step 4: Activity Abstraction Levels**

GPS data presents a particular challenge due to its high granularity. I propose a multi-level activity abstraction:

| Raw GPS Events | Abstracted Activities | Business Activities |
|---------------|----------------------|---------------------|
| Continuous location pings | "Traveling," "Stopped," "Idling" | "En-route to Stop X" |
| Speed < 5 km/h for > 30 sec | "Traffic Delay" | "Congestion-Related Delay" |
| Ignition on, speed = 0 > 2 min | "Extended Idle" | "Unproductive Wait Time" |
| Location matches customer address | "At Customer Location" | "Service Time" |

#### Key Integration Challenges and Solutions

**Challenge 1: GPS Data Volume and Noise**

*Problem*: GPS trackers generate events every few seconds, creating millions of records that can overwhelm process mining tools and obscure meaningful process patterns.

*Solution*: Implement intelligent event abstraction:
- Apply trajectory segmentation algorithms (e.g., stay-point detection)
- Aggregate micro-movements into meaningful travel segments
- Define location-based triggers (geofencing) to create discrete events when vehicles enter/exit customer zones or traffic hotspots
- Use speed thresholds to differentiate active travel from delays

**Challenge 2: Event Correlation Across Sources**

*Problem*: Linking a GPS "vehicle stopped" event to a scanner "delivery success" event for the same physical occurrence.

*Solution*: 
- Implement spatial-temporal correlation with configurable tolerances (e.g., events within 50 meters and 2 minutes are potentially related)
- Use package-location mappings from dispatch data to validate correlations
- Create composite events that merge correlated source events

**Challenge 3: Missing or Inconsistent Data**

*Problem*: Scanner events might be missed if a driver forgets to scan; GPS signals may drop in urban canyons.

*Solution*:
- Implement data imputation rules based on surrounding context
- Flag imputed events for transparency in analysis
- Use GPS trail to infer missing scanner events when location/time align with planned stops
- Calculate data quality metrics to assess reliability by vehicle/driver

**Challenge 4: Defining Process Start and End Points**

*Problem*: Determining exact boundaries of the operational process.

*Solution*: Define clear boundary events:
- Process Start: "Route Assigned" or "Depart Depot" (whichever is chosen as the trigger)
- Process End: "Arrive Depot" at end of day
- Handle multi-day cases or overnight operations with explicit shift boundaries

### 1.2 Process Discovery

#### Discovery Algorithm Selection

For Speedy Parcels' delivery operations, I recommend a **multi-algorithm approach** to capture different perspectives:

**Algorithm 1: Directly-Follows Graph (DFG) for Initial Exploration**

The DFG provides an intuitive visualization of activity sequences and their frequencies. For delivery operations:

```
[Start Shift]  [Route Assigned]  [Depart Depot]  [Travel to Stop]  
[Arrive Customer]  [Delivery Success/Failed]  [Depart Customer]  
[Travel to Stop]...  [Arrive Depot]  [End Shift]
```

This reveals:
- Most common process flows
- Frequency of path variants
- Unexpected transitions (e.g., "Travel"  "Unscheduled Stop"  "Maintenance")

**Algorithm 2: Inductive Miner for Sound Process Models**

The Inductive Miner generates process trees that guarantee soundness, capturing:
- Concurrent activities (e.g., packages delivered in parallel at multi-stop locations)
- Loops (re-delivery attempts, returns to previous areas)
- Choices (successful delivery vs. failed delivery paths)

**Algorithm 3: Fuzzy Miner for Handling Complexity**

Given the potential for hundreds of delivery stops and numerous variants, Fuzzy Mining helps by:
- Abstracting less significant paths
- Highlighting dominant process flows
- Managing "spaghetti-like" processes common in logistics

### 1.3 Conformance Checking

#### Establishing the Normative Model

The dispatch system provides the "as-planned" process that serves as the normative model for conformance checking:

**Planned Route Structure:**
```
Planned_Route = {
    vehicle_id: "V12",
    date: "2024-12-05",
    planned_departure: "07:45:00",
    planned_stops: [
        {sequence: 1, package: "P9876", address: "...", eta: "08:20", time_window: "08:00-12:00"},
        {sequence: 2, package: "P9877", address: "...", eta: "08:45", time_window: "08:00-12:00"},
        ...
        {sequence: 35, package: "P9910", address: "...", eta: "16:30", time_window: "14:00-18:00"}
    ],
    planned_return: "17:00:00"
}
```

#### Conformance Checking Techniques

**Technique 1: Token Replay Analysis**

Token replay tracks how well the actual event traces fit the planned process:

| Conformance Metric | Description | Target |
|-------------------|-------------|--------|
| Fitness | Proportion of events that can be replayed on the model | > 95% |
| Precision | How much the model allows behavior not in the log | > 85% |
| Generalization | Model's ability to handle unseen but valid behavior | > 80% |

**Technique 2: Alignment-Based Conformance**

This provides detailed deviation diagnostics by computing optimal alignments between planned and actual sequences:

```
Planned Sequence: Stop1  Stop2  Stop3  Stop4  Stop5
Actual Sequence:  Stop1  Stop2  Stop4  Stop3  Stop5
                                            
                              Sequence Deviation (swap)
```

#### Types of Deviations to Detect

**1. Sequence Deviations**

| Deviation Type | Detection Method | Business Impact |
|---------------|------------------|-----------------|
| Stop order changes | Alignment comparison with planned sequence | May improve or degrade efficiency; indicates route planning issues |
| Skipped stops | Missing expected activities | Failed deliveries, customer complaints |
| Unplanned stops | Activities not in planned route | Personal stops, efficiency issues, or legitimate needs |
| Repeated stops | Loops in actual trace | Re-delivery attempts, forgotten items |

**2. Temporal Deviations**

```
Temporal Conformance Analysis:
 Departure Time Variance: Actual vs. Planned depot departure
 Arrival Time Variance: Per-stop ETA vs. actual arrival
 Service Time Variance: Expected vs. actual time at customer
 Return Time Variance: End-of-day actual vs. planned
 Cumulative Delay Propagation: How early delays compound
```

**Analysis Query Example:**
```sql
SELECT 
    case_id,
    stop_sequence,
    planned_arrival,
    actual_arrival,
    TIMESTAMPDIFF(MINUTE, planned_arrival, actual_arrival) as arrival_variance_minutes,
    CASE 
        WHEN actual_arrival > customer_time_window_end THEN 'LATE'
        WHEN actual_arrival < customer_time_window_start THEN 'EARLY'
        ELSE 'ON_TIME'
    END as delivery_status
FROM unified_event_log
WHERE activity = 'Arrive Customer';
```

**3. Geographic Deviations**

Using GPS data to identify:
- Route deviations from planned paths
- Unexpected geographic patterns (e.g., backtracking)
- Detours and their causes (traffic, construction, driver preference)

**4. Duration Deviations**

| Activity | Planned Duration | Threshold for Deviation Flag |
|----------|-----------------|------------------------------|
| Travel between stops | Based on distance/speed model | > 150% of expected |
| Service time at customer | 3-5 minutes average | > 10 minutes |
| Depot loading | 30 minutes | > 45 minutes |
| Break time | Regulated limits | Exceeds allowance |

| **Customer Time Window Compliance** | Deliveries within window / Total with windows | Scanner + Dispatch | > 92% |

**Calculation Example - On-Time Delivery Rate:**
```sql
SELECT 
    DATE(timestamp) as delivery_date,
    vehicle_id,
    COUNT(CASE 
        WHEN activity = 'Delivery Success' 
        AND timestamp BETWEEN customer_time_window_start AND customer_time_window_end 
        THEN 1 
    END) as on_time_deliveries,
    COUNT(CASE WHEN activity = 'Delivery Success' THEN 1 END) as total_deliveries,
    ROUND(
        COUNT(CASE 
            WHEN activity = 'Delivery Success' 
            AND timestamp BETWEEN customer_time_window_start AND customer_time_window_end 
            THEN 1 
        END) * 100.0 / 
        NULLIF(COUNT(CASE WHEN activity = 'Delivery Success' THEN 1 END), 0), 2
    ) as otdr_percentage
FROM unified_event_log
GROUP BY DATE(timestamp), vehicle_id;
```

#### Root Cause Category 5: Vehicle Reliability Issues

**Hypothesis**: Vehicle maintenance patterns correlate with operational disruptions.

**Maintenance Pattern Analysis:**

```sql
-- Correlate breakdowns with vehicle characteristics
SELECT 
    v.vehicle_age_years,
    v.vehicle_type,
    AVG(v.total_km) as avg_odometer,
    COUNT(m.maintenance_id) as unscheduled_repairs,
    SUM(m.duration_hours) as total_downtime_hours,
    AVG(DATEDIFF(m.repair_date, m.last_scheduled_maintenance)) as days_since_scheduled
FROM vehicles v
LEFT JOIN maintenance_logs m ON v.vehicle_id = m.vehicle_id 
    AND m.maintenance_type = 'unscheduled'
WHERE m.repair_date >= DATE_SUB(CURRENT_DATE, INTERVAL 6 MONTH)
GROUP BY v.vehicle_age_years, v.vehicle_type;
```

**Pattern Analysis: Pre-Breakdown Indicators**

```sql
-- Identify vehicles showing pre-breakdown patterns
WITH vehicle_metrics AS (
    SELECT 
        vehicle_id,
        DATE(timestamp) as metric_date,
        AVG(fuel_consumption) as daily_fuel_efficiency,
        SUM(CASE WHEN activity = 'Extended Idle' THEN duration ELSE 0 END) as idle_time,
        COUNT(CASE WHEN activity = 'Low Speed Detected' THEN 1 END) as speed_issues,
        MAX(CASE WHEN notes LIKE '%Warning%' THEN 1 ELSE 0 END) as warning_flag
    FROM unified_event_log
    GROUP BY vehicle_id, DATE(timestamp)
)
SELECT 
    vehicle_id,
    metric_date,
    daily_fuel_efficiency,
    LAG(daily_fuel_efficiency, 7) OVER (PARTITION BY vehicle_id ORDER BY metric_date) as fuel_7d_ago,
    (daily_fuel_efficiency - LAG(daily_fuel_efficiency, 7) OVER (PARTITION BY vehicle_id ORDER BY metric_date)) 
        / LAG(daily_fuel_efficiency, 7) OVER (PARTITION BY vehicle_id ORDER BY metric_date) as fuel_trend,
    idle_time,
    CASE 
        WHEN fuel_trend > 0.15 OR idle_time > threshold OR warning_flag = 1 
        THEN 'MAINTENANCE_RECOMMENDED' 
        ELSE 'NORMAL' 
    END as maintenance_alert
FROM vehicle_metrics;
```

| Driver-reported issues | Scanner notes | Medium |
| Vehicle age | Fleet records | Medium |
| Route intensity (stops/km) | Dispatch | Low-Medium |

## 5. Operational Constraints and Continuous Monitoring

### 5.1 Accounting for Operational Constraints

#### Constraint 1: Driver Working Hours Regulations

**Regulatory Requirements:**
- Maximum 10-hour driving day
- Mandatory 30-minute break after 5 hours
- Weekly hour limits

**How Strategies Account for This:**

| Strategy | Constraint Consideration | Safeguard |
|----------|-------------------------|-----------|
| Dynamic Routing | Routes must complete within shift limits | Hard constraint in optimization algorithm; trigger alerts at 85% of limit |
| Failed Delivery Prevention | Reduce re-delivery burden | Fewer re-deliveries = more buffer for delays without overtime |
| Driver Performance | No incentives that encourage hour violations | Performance metrics normalized by hours worked |

**Implementation in Route Optimization:**
```
Route Optimization Constraints:
 Hard Constraints (must satisfy):
    Max shift duration: 10 hours
    Break requirement: 30 min after 5 hours
    Route must complete within shift
    Overtime approval threshold
 Soft Constraints (optimize for):
    Minimize total route time
    Maximize on-time deliveries
    Balance workload across drivers
 Adaptive behavior:
     If route infeasible  reduce stops or split route
     If running late  prioritize time-window deliveries
```

#### Constraint 2: Vehicle Capacity

**Capacity Considerations:**
- Volume capacity (m³)
- Weight capacity (kg)
- Package count limits
- Special handling requirements

**How Strategies Account for This:**

| Strategy | Constraint Consideration |
|----------|-------------------------|
| Dynamic Routing | Capacity verified before route changes; no additions beyond capacity |
| Failed Delivery Prevention | Alternative delivery options reduce need for re-delivery trips |
| Fleet Management | Predictive maintenance ensures reliable capacity availability |

**Capacity Optimization Approach:**
```
Vehicle-Package Matching Algorithm:
 Inputs:
    Package dimensions and weights
    Vehicle available space
    Delivery sequence (accessibility needs)
    Special requirements (refrigeration, fragile, etc.)
 Constraints:
    Total weight  vehicle weight capacity
    Total volume  vehicle volume capacity
    Load sequence supports delivery sequence
    Special handling requirements met
 Optimization:
     Maximize packages per vehicle while meeting constraints
```

#### Constraint 3: Customer Time Windows

**Time Window Types:**
- Hard windows (contractual, must meet)
- Soft windows (preferred, should try to meet)
- Flexible windows (no specific preference)

**How Strategies Account for This:**

```
Time Window Management in Routing:
 Classification:
    Priority 1: Hard time windows (SLA commitments)
    Priority 2: Soft windows (customer preference)
    Priority 3: Flexible (optimize for efficiency)
 Routing Logic:
    Hard windows: Must be scheduled within window
    Soft windows: Attempt to meet, re-sequence if needed
    Flexible: Fill in around fixed appointments
 Dynamic Re-routing Rules:
     Never sacrifice hard windows for optimization
     Can adjust soft windows with customer notification
     Flexible stops can be freely re-sequenced
```

**Trade-off Management:**
```
When route optimization conflicts with constraints:
 Scenario: Traffic delay threatens time window
    Option 1: Skip earlier flexible stops, prioritize window
    Option 2: Contact customer to adjust window
    Option 3: Hand off to nearby driver (rare)
 Decision hierarchy:
    1. Protect customer time windows
    2. Complete route within driver hours
    3. Maximize deliveries
    4. Minimize costs
```
