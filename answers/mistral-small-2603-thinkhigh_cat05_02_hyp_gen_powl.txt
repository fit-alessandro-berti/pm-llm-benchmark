### **Query 3: Identify Premature or Incorrect Closures (Partial Ordering Anomaly)**
**Hypothesis:** The `A  C` path allows claims to be closed before proper evaluation/approval.

```sql
WITH required_steps AS (
    SELECT
        claim_id,
        MAX(CASE WHEN activity = 'R' THEN timestamp END) AS receive_time,
        MAX(CASE WHEN activity = 'A' THEN timestamp END) AS assign_time,
        MAX(CASE WHEN activity = 'E' THEN timestamp END) AS evaluate_time,
        MAX(CASE WHEN activity = 'P' THEN timestamp END) AS approve_time,
        MAX(CASE WHEN activity = 'N' THEN timestamp END) AS notify_time,
        MAX(CASE WHEN activity = 'C' THEN timestamp END) AS close_time
    FROM claim_events
    GROUP BY claim_id
),
claim_status AS (
    SELECT
        claim_id,
        receive_time,
        assign_time,
        evaluate_time,
        approve_time,
        notify_time,
        close_time,
        -- Check if closure happened before any critical step
        CASE
            WHEN close_time < assign_time THEN 'Closed before adjuster assigned'
            WHEN close_time < evaluate_time THEN 'Closed before evaluation'
            WHEN close_time < approve_time THEN 'Closed before approval'
            WHEN close_time < notify_time THEN 'Closed before notification'
            ELSE 'Closed after all steps'
        END AS closure_status,
        -- Check if adjuster was assigned but claim was closed without evaluation
        CASE
            WHEN assign_time IS NOT NULL AND evaluate_time IS NULL THEN 'Closed without evaluation'
            WHEN assign_time IS NOT NULL AND approve_time IS NULL THEN 'Closed without approval'
            ELSE 'Properly processed'
        END AS processing_status
    FROM required_steps
    WHERE close_time IS NOT NULL  -- Only claims that were closed
)

SELECT
    c.claim_id,
    c.claim_amount,
    c.submission_date,
    c.claim_type,
    a.name AS adjuster_name,
    a.specialization,
    a.region,
    cs.closure_status,
    cs.processing_status,
    cs.close_time
FROM claim_status cs
JOIN claims c ON cs.claim_id = c.claim_id
JOIN adjusters a ON c.customer_id = a.adjuster_id  -- Again, likely incorrect; adjusters are assigned to claims.
-- Correction: Link adjusters via claim_events where activity='A' and resource=adjuster_id.
WHERE cs.closure_status LIKE '%before%'
   OR cs.processing_status IN ('Closed without evaluation', 'Closed without approval')
ORDER BY cs.close_time DESC;
```

**What This Tests:**
- Identifies claims closed **before** critical steps (`A`, `E`, `P`, `N`).
- Flags claims closed **without evaluation** or **without approval**.
- Includes adjuster details to see if certain adjusters are more likely to close claims prematurely.

### **Query 5: Detect Out-of-Sequence Events (System-Level Issues)**
**Hypothesis:** The partial ordering anomalies are due to **system-generated events** or **timestamp errors**.

```sql
SELECT
    ce1.claim_id,
    ce1.activity AS event1,
    ce1.timestamp AS time1,
    ce2.activity AS event2,
    ce2.timestamp AS time2,
    ce1.timestamp > ce2.timestamp AS is_out_of_sequence,
    ce1.additional_info,
    ce2.additional_info

    )
ORDER BY ce1.claim_id, time1;
```

**What This Tests:**
- Finds **out-of-sequence events** that violate the intended process flow.
- Focuses on:
  - `C` (Close) before `E`/`P` (evaluation/approval).

  - `N` (Notify) after `C`/`P` (closure/approval).
