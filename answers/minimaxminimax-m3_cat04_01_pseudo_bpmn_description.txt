# Business Process Explanation: Customer Request Handling

## 1. Process Overview

This process represents an **order-to-confirmation workflow** for a company that handles both **standard product requests** and **custom product/solution requests**. The process begins when a customer submits a request and ends when the customer receives a confirmation (or, in some failure scenarios, a rejection notice). The process is designed to:

- Validate the customer's request according to its type
- Run the necessary background checks (credit, inventory, feasibility)
- Calculate delivery timelines or prepare quotations
- Route cases through managerial approval when required
- Generate the final invoice and confirm the order to the customer

---

## 2. Step-by-Step Walkthrough

### **Start Event**
Marks the entry point of the process.

### **Task A — Receive Customer Request**
The system or a sales/customer service representative captures the customer's incoming request, including relevant details such as product type, quantities, customization specifications, and customer identity.

### **XOR Gateway — "Check Request Type"**
An **XOR (Exclusive OR) gateway** means **only one path** is taken based on the evaluation of a condition. Here, the process determines whether the request is **Standard** or **Custom** and routes it accordingly.

---

## 3. The Standard Request Path

### **Task B1 — Perform Standard Validation**
For standard requests, the company performs routine validation against existing product catalog rules (e.g., pricing, availability rules, eligibility).

### **AND Gateway — "Run Parallel Checks"**
An **AND (Parallel) gateway** splits the flow into **multiple concurrent branches** that all execute simultaneously. Both checks must be completed before the process can continue.

- **Task C1 — Credit Check**: Verifies the customer's creditworthiness.
- **Task C2 — Inventory Check**: Confirms product availability and stock levels.

### **Join — All Parallel Checks Completed**
The AND join synchronizes the parallel branches. The process proceeds only after **both** C1 and C2 are finished.

### **Task D — Calculate Delivery Date**
Using the inventory and validation results, the system computes a realistic delivery date based on stock and logistics.

---

## 4. The Custom Request Path

### **Task B2 — Perform Custom Feasibility Analysis**
For custom requests, the company performs a deeper analysis to determine whether the requested customization can be delivered (resource, technical, time constraints).

### **XOR Gateway — "Is Customization Feasible?"**
Another XOR gateway decides between two outcomes:

- **If Yes  Task E1 — Prepare Custom Quotation**
  A tailored quotation is generated based on cost, timeline, and feasibility analysis.

- **If No  Task E2 — Send Rejection Notice  End Event**
  The process terminates early with a rejection sent to the customer. No further steps are executed.

---

## 5. Convergence Point

After either the Standard path (Task D) or the Custom path (Task E1) is completed, the process **merges back into a single flow** at the "After Standard or Custom Path Tasks Completed" point. From here, the workflow is identical regardless of request type.

---

## 6. Approval and Rework Sub-Process

### **XOR Gateway — "Is Approval Needed?"**
A conditional check (e.g., order value exceeds threshold, custom request, new customer) determines whether managerial approval is required.

- **If No Approval Needed**  Go directly to **Task G — Generate Final Invoice**.
- **If Approval Needed**  Proceed to **Task F — Obtain Manager Approval**.

### **Task F — Obtain Manager Approval**
The request is escalated to a manager for review.

### **XOR Gateway — "Is Approval Granted?"**
The manager's decision splits the flow again:

- **If Yes**  **Task G — Generate Final Invoice** (the process continues normally).
- **If No**  **Task H — Re-evaluate Conditions**.

### **Task H — Re-evaluate Conditions (Rework Loop)**
This is a **loop-back step** where the original request is revisited to adjust parameters (e.g., reduce scope, change price, modify terms). After re-evaluation, the flow **loops back** to:

- **Task E1 (Prepare Custom Quotation)** if it was a custom request, **OR**
- **Task D (Calculate Delivery Date)** if it was a standard request.

This loop allows the company to attempt fulfilling the request again with modified conditions, eventually returning to the approval gate.

---

## 7. Final Steps

### **Task G — Generate Final Invoice**
Once approval (if needed) is secured, the final invoice is created.

### **Task I — Send Confirmation to Customer**
The customer is notified of the order confirmation, including the invoice and (if applicable) the delivery date.

### **End Event**
The process terminates.

---

## 8. Summary of Gateway Types Used

| Gateway | Type | Meaning |
|---|---|---|
| Check Request Type | XOR | Splits flow based on request classification (Standard vs. Custom) |
| Run Parallel Checks | AND | Executes Credit Check and Inventory Check simultaneously |
| Is Customization Feasible? | XOR | Routes to quotation or rejection path |
| Is Approval Needed? | XOR | Decides whether escalation is required |
| Is Approval Granted? | XOR | Routes to invoice or rework loop |

---

## 9. Key Differences: Standard vs. Custom Requests

| Aspect | Standard Path | Custom Path |
|---|---|---|
| Validation | Lightweight standard validation (B1) | Deep feasibility analysis (B2) |
| Checks | Parallel credit + inventory checks | No parallel checks; feasibility is the main gate |
| Output | Calculated delivery date (D) | Custom quotation (E1) or rejection (E2) |
| Early Termination | Possible only via approval rework | Possible via feasibility failure (Task E2) |
| Rework Target | Loops back to Task D | Loops back to Task E1 |

---

## 10. When Approvals and Rework Are Triggered

- **Approval is required** when business rules deem it necessary (e.g., high-value orders, custom requests, or risky customers).
- **Rework (Task H)** is triggered **only when a manager denies approval**. This is the only loop in the process, and it always re-enters the workflow just before the approval gate — at Task D (Standard) or Task E1 (Custom) — depending on the original request type.

This design ensures that rejected approvals do not silently end the process; instead, the order is given a chance to be **revised and re-submitted**, preserving customer relationships and operational flexibility.
