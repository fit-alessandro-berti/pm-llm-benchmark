### 1. Process Discovery and Conformance Checking

#### Data Preprocessing and Integration
To apply process mining effectively, the disparate data sources must be normalized into a single, high-quality **Event Log**.

*   **Case Identification:** The fundamental unit of analysis in logistics is usually a specific delivery task or a vehicle shift. Based on the provided log, the `Case ID` should be constructed as `{Vehicle_ID} + {Date}` (e.g., `V12-20241205`). This allows us to analyze the full lifecycle of a driver's shift end-to-end.
*   **Schema Alignment:** We must map all sources to a unified schema:
    *   **Timestamp:** Normalize all timestamps to UTC or local time with timezone offsets to ensure synchronization between GPS and Dispatch logs.
    *   **Activity Definition:** Map raw events (e.g., `Delivery Success`) into standardized process steps (e.g., `Stop Completion`). For GPS data, we need to convert location coordinates into logical nodes (e.g., "Traffic Congestion," "Parking Search").
    *   **Missing Data Handling:** If a driver stops the scanner but the GPS shows movement, or vice versa, we must decide whether to impute or treat as an anomaly. We will assume that `Scanner` events define the logical steps and `GPS` data defines the physical reality (travel vs. idle).
*   **Challenge:** The primary challenge is **temporal alignment**. A "Low Speed" event from GPS might not align perfectly with a "Delivery Failed" scanner event if there is a latency in reporting. Additionally, defining the "Start" and "End" of a process instance can be tricky (e.g., does the process end when the vehicle returns to the depot, or after the shift ends?).

#### Process Discovery
Using **Process Discovery Algorithms** (such as Heuristic Miner or Inductive Miner), we will generate a visual map of the *actual* delivery process.

*   **Visualization:** The resulting model will show paths from `Start Shift` -> `Depart Depot` -> `Travel to Customer 1` -> `Arrive/Leave` -> ... -> `Return to Depot`.
*   **Identifying Loops and Branches:** Crucially, this will reveal branches for "Delivery Failed" leading to a "Re-Attempt" or "Reschedule." It will also highlight loops caused by traffic jams (detected via GPS speed drops) or maintenance stops.
*   **Frequency Analysis:** The model will show which routes are taken most frequently and how often deviations occur.

#### Conformance Checking
We will compare the discovered *actual* process against the *planned* routes from the Dispatch System.

*   **Deviation Types to Monitor:**
    1.  **Sequence Deviations:** Did the driver visit Stop 5 before Stop 4? (e.g., due to a missed appointment).
    2.  **Timing Deviations:** Did a specific stop take significantly longer than the average duration for that route type?
    3.  **Unplanned Stops:** The log shows "Unscheduled Stop" with an "Engine Warning Light." Conformance checking will flag these against the "No Maintenance" plan.
    4.  **Route Variance:** Did the vehicle take a different physical path (GPS coordinates) than the optimized route assigned in the Dispatch System?

### 2. Performance Analysis and Bottleneck Identification

#### Key Performance Indicators (KPIs)
To address punctuality and cost, we will define and calculate the following KPIs directly from the event log:

1.  **On-Time Delivery (OTD) Rate:**
    *   *Calculation:* (Count of `Delivery Success` within Time Window / Total `Delivery Success`) * 100.
2.  **Average Service Time per Stop:**
    *   *Calculation:* `(Sum of (Depart Customer Timestamp - Arrive Customer Timestamp)) / Number of Stops`.
3.  **Travel vs. Service Ratio:**
    *   *Calculation:* `Total Travel Time` (derived from GPS speed > 0) / `Total Service Time` (derived from Scanner events). High ratios indicate inefficient travel; low ratios may indicate excessive time spent at customers or searching for parking.
4.  **Fuel Efficiency Proxy (Cost per Package):**
    *   *Calculation:* `(Distance Traveled * Fuel Consumption Rate) / Number of Successful Deliveries`. If fuel data isn't directly in the log, we can correlate `Idle Time` + `Stop Duration` with known vehicle maintenance logs to estimate efficiency losses.
5.  **Failed Delivery Rate:**
    *   *Calculation:* (Count of `Delivery Failed` events) / (Total Stops Planned).
6.  **Traffic Delay Frequency:**
    *   *Calculation:* Count of events where `Speed < 10 km/h` for > 5 minutes without a scanner event indicating a stop.

#### Bottleneck Identification Techniques
Using **Performance Miner** or **Variant Analysis**, we will identify where the process slows down.

*   **Activity Duration Analysis:** We will plot histograms for each activity (e.g., `Arrive Customer` to `Depart Customer`). Long tails indicate bottlenecks (e.g., difficulty finding parking).
*   **Path Analysis:** We will analyze specific route variants. If Route A consistently has higher delays than Route B, the bottleneck is likely a geographic or traffic issue rather than a driver issue.
*   **Time of Day Correlation:** By overlaying timestamps, we can identify if bottlenecks occur during peak rush hour (e.g., 8:00-9:30) specifically on certain corridors.
*   **Driver Clustering:** We will analyze if specific drivers consistently have higher `Service Time` or more `Failed Deliveries`. This helps distinguish between a systemic issue and individual performance issues.

### 3. Root Cause Analysis for Inefficiencies

Identifying *where* the problem is not enough; we must determine *why*. Process mining allows us to correlate events to find causality.

*   **Suboptimal Route Planning:**
    *   *Analysis:* Compare the Dispatch System's planned route order against GPS coordinates. If drivers frequently deviate to take a "shortcut," it suggests the planned route is inefficient or ignores real-time traffic data.
*   **Inaccurate Travel Time Estimations:**
    *   *Analysis:* Correlate `Distance` with `Actual Travel Time`. If actual time consistently exceeds estimated time for specific segments, the estimation logic in the Dispatch System is flawed (likely ignoring historical congestion).
*   **Impact of Traffic Congestion:**
    *   *Analysis:* Use GPS speed data to cluster locations. Identify "Hotspots" where speed drops below a threshold repeatedly. Overlay these with `Delivery Failed` events. If failures occur at specific traffic hotspots, the root cause is external infrastructure, not operational.
*   **Service Time Variability:**
    *   *Analysis:* Analyze the duration between `Arrive Customer` and `Depart Customer`. High variance suggests inconsistency (e.g., some drivers get signatures quickly, others struggle). This could point to customer behavior or driver skill differences.
*   **Vehicle Breakdowns:**
    *   *Analysis:* Correlate `Unscheduled Stop` events with Maintenance Logs. If breakdowns happen frequently after a certain mileage or specific route types (e.g., hilly terrain), the root cause is vehicle strain or maintenance scheduling failure.
*   **Failed Delivery Attempts:**
    *   *Analysis:* Link `Delivery Failed` events to subsequent `Re-attempt` events. If a high percentage of failures occur at 14:00, the root cause might be "afternoon customer absence," suggesting better time window management is needed.

### 4. Data-Driven Optimization Strategies

Based on the analysis above, here are three distinct strategies to improve punctuality and reduce costs.

#### Strategy 1: Dynamic Route Adjustment & Traffic-Aware Routing
*   **Target Inefficiency:** High travel times due to traffic congestion and static route planning.
*   **Root Cause:** Dispatch system uses historical data or static maps that do not reflect real-time conditions (e.g., accidents, roadworks).
*   **Process Mining Support:** Use GPS speed data to create a "Traffic Heatmap" of the city. Identify segments where `Speed < 10 km/h` is frequent.
*   **Action:** Implement an algorithm that allows drivers to deviate from the planned route if real-time traffic (inferred from live GPS) suggests a delay > X minutes. Prioritize stops in low-traffic zones during peak hours.
*   **Expected Impact:** Reduced `Travel Time`, lower fuel consumption, improved OTD Rate.

#### Strategy 2: "Parking & Access" Optimization for Service Time Reduction
*   **Target Inefficiency:** High variability and duration in `Arrive Customer` to `Depart Customer`.
*   **Root Cause:** Drivers spending excessive time searching for parking or waiting for customers inside buildings (e.g., apartment complexes).
*   **Process Mining Support:** Analyze the `Location` data during `Service Time`. Identify locations where the vehicle is stationary (`Speed = 0`) but no scanner event occurs. This indicates "Parking Search."
*   **Action:** Redesign delivery territories to prioritize customers with known parking availability or instruct drivers on preferred drop-off zones (e.g., side streets vs. main roads). Communicate specific access instructions to customers to reduce waiting time.
*   **Expected Impact:** Reduced `Average Service Time`, higher vehicle utilization, lower fuel waste from idling while searching for parking.

#### Strategy 3: Predictive Maintenance & Driver Behavior Coaching
*   **Target Inefficiency:** Unscheduled repairs and high fuel consumption due to poor driving habits.
*   **Root Cause:** Reactive maintenance (fixing issues after they happen) and inefficient driving (excessive acceleration/braking/idling).
*   **Process Mining Support:** Correlate `Unscheduled Stop` events with `Vehicle ID`. Identify if specific vehicles break down more often. Analyze GPS data for "Hard Braking" or excessive idling (`Speed = 0` while Ignition On).
*   **Action:**
    *   **Maintenance:** Schedule service based on accumulated mileage and usage patterns (from the log) rather than fixed intervals to prevent breakdowns mid-shift.
    *   **Training:** Provide feedback reports to drivers with high "Hard Braking" or "Idling" incidents.
*   **Expected Impact:** Reduced `Maintenance Costs`, fewer delays due to breakdowns, improved fuel efficiency per package.

### 5. Considering Operational Constraints and Monitoring

#### Handling Operational Constraints
Any optimization strategy must respect the physical and regulatory limits of the business:
*   **Driver Working Hours:** Route adjustments (Strategy 1) cannot push drivers beyond legal driving limits or cause them to miss mandatory rest periods. The process model must include "Shift End" as a hard stop.
*   **Vehicle Capacity:** Re-routing must ensure that vehicle load capacity is not exceeded when consolidating deliveries.
*   **Customer Time Windows:** Strategies must prioritize high-value time windows. If a delivery fails (Strategy 2), the system should automatically re-schedule it for a window where the customer is available, rather than forcing an immediate re-attempt which might violate service level agreements.

#### Continuous Monitoring Plan
To ensure sustainability, we will implement a **Process Mining Dashboard** that runs continuously:

1.  **Real-Time Alerts:** Trigger alerts when `Speed = 0` for > 30 mins without scanner activity (potential breakdown or lost driver). Alert if `Arrive Time` exceeds the planned buffer by > 15%.
2.  **KPI Tracking Dashboard:** Visualize OTD Rate, Cost per Delivery, and Failed Delivery Rate daily. Use trend lines to detect degradation over time.
3.  **Variant Monitoring:** Monitor specific "Problem Paths" identified in Phase 1. If a new path emerges (e.g., a new route that consistently causes delays), investigate immediately.
4.  **Feedback Loop:** Monthly review meetings using the process models to validate if implemented changes (like Strategy 1) are reducing delay variance.

By continuously mining this event log, Speedy Parcels can move from reactive firefighting to proactive optimization, ensuring their fleet remains efficient and reliable despite urban challenges.
