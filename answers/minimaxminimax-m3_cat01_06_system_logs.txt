## Explanation of Logic

### 1. Case Identification Strategy

I identified **5 distinct cases**, where each case represents a logical unit of user work centered on a specific artifact (document, email, spreadsheet, or PDF). The cases are:

| Case ID | Type | Rationale |
|---------|------|-----------|
| **Document1.docx** | Word document | All activity on this document is grouped together, even across two separate work sessions separated by other tasks |
| **Quarterly_Report.docx** | Word document | Includes both the initial brief focus and the later active editing session |
| **Annual Meeting Email** | Email task | All activities related to reading and replying to this specific email |
| **Report_Draft.pdf** | PDF review | All activities related to reviewing this specific PDF |
| **Budget_2024.xlsx** | Excel spreadsheet | All activities related to updating this specific budget file |

The key principle is that a **case follows the artifact**, not the time window. This allows Document1.docx to be interrupted and resumed naturally, and captures the full lifecycle of Quarterly_Report.docx (which was briefly opened, abandoned, then resumed later).

| TYPING (Word) | Type Content | Generic composition activity |
| TYPING (Email) | Type Reply | Specific to the email context |

### 3. Handling Multi-Session Cases

**Document1.docx** has two distinct work sessions:
- Session 1: 09:00:00 – 09:01:15 (initial drafting and saving)
- Session 2: 09:06:00 – 09:07:00 (adding budget reference, saving, closing)

Both sessions belong to the same case, since they relate to the same artifact and the user is building on prior work.

**Quarterly_Report.docx** also has an interrupted lifecycle:
- 08:59:50: Brief initial focus (no actual work done)
- 09:07:15 – 09:08:15: Active editing session

The brief 08:59:50 focus is preserved as the case start, since it represents an actual interaction with the artifact, even if immediately abandoned.

### 4. Special Considerations

- **SWITCH events** were converted to "Open [Artifact]" activities belonging to the destination case, representing the start of work on the new artifact. The original "From" information is captured in the Details column.
- **FOCUS events** on already-active cases (e.g., Document1.docx at 09:06:00) are treated as "Open Document" events to maintain a consistent vocabulary, while the Details column notes the context ("Resumed from Excel").
- **No explicit "Resume" activity** is used; instead, multiple "Open Document" events in the same case naturally represent the interrupted workflow pattern.
- The **Details column** preserves important context from the raw log (typed content, scroll direction, highlighted text) that adds analytical value without cluttering the activity names.
