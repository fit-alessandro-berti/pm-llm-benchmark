### **Optimized Draft Prompt (Recommended)**

> "Assume you have an event log stored in a DuckDB table named `event_log`, with each row representing a single event. The table has at least the following columns: `case_id`, `activity`, and `timestamp`. Each `case_id` groups events that belong to a single process instance (or 'case').
> 
> Your task: Please craft a DuckDB SQL query that filters out any cases containing more than six events, and returns all columns for the events from cases that have six or fewer events in total. 
> 
> Please ensure the final output is ordered by `case_id` and then by `timestamp` ascending. Consider utilizing DuckDB-specific SQL features if they make the query more concise."

### **Ideal Benchmark Solutions**

When grading the benchmark, here are the two main ways LLMs will solve this. 

#### **1. The DuckDB Idiomatic Solution (Using `QUALIFY`)**
DuckDB supports the `QUALIFY` clause, which allows filtering the results of window functions directly without needing a subquery or CTE. This is the most concise and modern way to write this query:

```sql
SELECT *
FROM event_log
QUALIFY COUNT(*) OVER (PARTITION BY case_id) <= 6
ORDER BY case_id, timestamp;
```
*Why this is great for a benchmark:* Testing if an LLM uses `QUALIFY` is a fantastic way to measure its specific knowledge of modern SQL dialects (like DuckDB/BigQuery) versus generic SQL-92.

#### **2. The Standard SQL Solution (Using Subquery or CTE)**
A more standard SQL approach that LLMs trained on older data might generate:

```sql
WITH case_counts AS (
    SELECT case_id, COUNT(*) AS event_count
    FROM event_log
    GROUP BY case_id
)
SELECT e.*
FROM event_log e
JOIN case_counts c ON e.case_id = c.case_id
WHERE c.event_count <= 6
ORDER BY e.case_id, e.timestamp;
```
*(Or a standard window function inside a subquery)*:
```sql
SELECT case_id, activity, timestamp
FROM (
    SELECT *, COUNT(*) OVER (PARTITION BY case_id) as case_size
    FROM event_log
)
WHERE case_size <= 6
ORDER BY case_id, timestamp;
```
