## BPMN-like textual process model (Accounts Payable – Invoice Approval)

### Participants / Pools
- **AP Clerk (Mary)**
- **Purchasing Officer (Dan)**
- **Department Manager (Carlos)**
- **AP Supervisor (Rita)**
- **Supplier / Supplier Relationship Manager (Karen)** (involved for corrections/contract checks)
- **Accounting/ERP System** (logging, matching views, automated notifications/emails, approval in system, scheduling triggers)

---

### Key Data Objects / Artifacts
- **Invoice (from supplier via email)**
- **Purchase Order (PO)**
- **Goods Receipt / Receipt confirmation** (used to confirm items/services were received)
- **Corrected/Revised Invoice** (when discrepancies exist)
- **Contract terms / price reference** (used when pricing conflicts arise)
- **Retrospective PO** (created for services without an existing PO)
- **Manager approval / sign-off request & decision**
- **Payment scheduling / payment run entry**

---

## Process Flow

### Start
**[Start]** Invoice arrives (typically via email)  **Task A1: Check PO reference (Mary)**

---

## Main Flow

### 1) PO existence check
**Task A1: Check if Invoice has a valid PO number (Mary)**  
- Input: *Invoice*
- Output: decision to proceed or request PO info

**Gateway G1: PO number present and valid?**
- **No**  **Task A2: Request PO details / corrected invoice from supplier (Mary)**  
  - Mary sends back to supplier requesting more details / PO reference.
  - (In some cases Dan also instructs Mary to obtain PO before proceeding.)
  - **Flow back** to **Task A1** when revised invoice/PO info is received.
- **Yes**  **Task A3: Log invoice in accounting system (Mary)**  
  - Output artifact: *Logged Invoice record*

---

### 2) Purchasing confirmation (quantities/prices/charges vs PO)
**Task A4: Forward logged invoice / request PO check (Mary  Dan)**

**Task A5: Compare invoice vs PO (quantities, prices, delivery charges) (Dan)**  
- Input: *PO*, *Invoice*
- Output: match/mismatch decision

**Gateway G2: Invoice line items match PO (incl. billing amounts)?**
- **No (discrepancy detected)**  **Task A6: Resolve discrepancy (Dan + Supplier)**
  - Dan either:
    - **Contacts supplier directly** for corrected terms/invoice, **or**
    - **Asks Mary to hold the invoice** until corrected version is received.
  - **Flow back** to **Task A5** after *Corrected/Revised Invoice* is obtained.
- **Yes**  **Task A7: Send confirmation to AP (Dan  Mary/AP)**  
  - Output artifact: *Confirmation email/message*

---

### 3) Goods receipt + invoice-to-PO matching
**Task A8: Confirm goods/services received + match invoice to PO (System view used by Mary) (Mary)**  
- Mary checks receipt confirmation and matches line items against the PO.

**Gateway G3: Any remaining discrepancies (pricing, missing items, suspicious notes)?**
- **No**  proceed to manager approval.
- **Yes**  **Task A9: Clarify discrepancy (Mary  Dan)**  
  - Mary emails purchasing team for clarification.
  - **Gateway G4: Discrepancy resolved quickly?**
    - **Yes**  return to **Task A8**
    - **No**  **Task A10: Escalate to AP Supervisor for exception handling (Mary  Rita)**

---

## Exception Handling (Supervisor / No-PO / Hard-to-resolve cases)

**Task A10: Exception resolution (Rita)**  
Rita ensures problem invoices don’t get stuck indefinitely and coordinates between suppliers/departments.

**Gateway G5: Invoice matches any PO?**
- **No (e.g., services contracted informally / no matching PO)**   
  **Task A11: Work with department heads to create retrospective PO or decide rejection (Rita + Dept heads)**  
  - Option 1: Create **Retrospective PO**
    - **Flow back** to **Task A4** (send for purchasing confirmation under the new PO)
  - Option 2: Reject invoice
    - **Task A12: Reject invoice (communicate decision) (Rita/AP/Mary)**  **[End: Invoice Rejected]**
- **Yes (PO exists but reconciliation failed)**   
  **Task A13: Contact supplier / coordinate resolution (Rita + Karen/Supplier)**  
  - Use contract info or request corrected invoice/terms as needed.
  - **Flow back** to **Task A5** (or **Task A8** depending on where resolution completes)

---

## Manager approval

### Trigger and sign-off
**Task A14: System sends manager sign-off request after match/confirmation (System  Carlos)**  
- Mary/Carlos note that manager approval is triggered after:
  - receipt confirmation + PO/invoice match, and/or
  - purchasing confirmation is present.

**Task A15: Manager review and approve/reject in system (Carlos)**  
- The system shows invoice details, PO, and purchasing confirmation.

**Gateway G6: Manager approves?**
- **Approve**  **Task A16: Send approved invoice back to AP / mark ready for payment (Carlos  System  AP)**  
   **Task A17: Schedule payment for next payment cycle/run (Mary)**  **[End: Payment Scheduled]**
- **Reject / request hold for investigation**  **Task A18: Ask Mary to hold & investigate further (Carlos  Mary + possibly Dan/Rita)**  
  - Loop back to **Task A9** (clarify discrepancy) and/or **Task A10** (escalate) depending on severity.

---

## Notes on uncertainty / variations reflected in interviews
- **PO missing case**: both Mary and Dan reference the need for a PO before proceeding; model assumes a loop where supplier sends a revised invoice/PO reference.
- **Where discrepancies are detected**: Mary performs invoice-vs-PO matching after receiving purchasing confirmation; Dan already checks invoice details against PO. The model keeps both checks as separate tasks, allowing for “remaining discrepancies.”
- **Exception escalation frequency**: Rita notes escalation happens when discrepancies can’t be resolved; the model represents this as a conditional escalation gateway.
- **No-matching-PO invoices**: handled as an exception with a possible **retrospective PO creation** or **rejection**.
