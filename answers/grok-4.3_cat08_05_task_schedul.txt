**1. Analyzing Historical Scheduling Performance and Dynamics**

Process mining reconstructs the actual execution of jobs from the MES event log by treating each Job ID (Case ID) as a unique process instance. Since routings are highly customized, standard process discovery (e.g., Inductive Miner or Heuristics Miner) would first be applied at an aggregated level to identify common task sequences and frequent variants, while preserving case-level granularity for performance analysis. The log’s rich attributes (timestamp, Activity, Resource, Setup Required, Previous Job reference in Notes, Planned/Actual Durations, Priority, Due Date) enable direct mapping of the control-flow, data, and resource perspectives.

Specific techniques and metrics include:
- **Job flow times, lead times, and makespan distributions**: Compute case duration as the timestamp difference between “Job Released” and final “Task End” (or implicit completion). Lead time is compared against due date. Use performance analysis to derive empirical distributions, histograms, and percentiles (e.g., 90th percentile makespan). Dotted charts visualize timestamp patterns across machines.
- **Task waiting times (queue times)**: Measure the interval between “Queue Entry” and “Task Start” (or “Setup Start”) per work center. Aggregate by machine to identify high-wait resources. Use waiting-time heatmaps or queue-length reconstructions via resource event correlation.
- **Resource utilization**: For each machine/operator, calculate productive time (sum of actual task durations + setups), idle time (gaps between consecutive tasks), and setup proportion. Utilization = (productive time / total observation window) × 100. Separate setup time from processing time using the “Setup Required = TRUE” flag and actual duration differences.
- **Sequence-dependent setup times**: Filter events where “Setup Required = TRUE” and link to the prior job on the same machine via timestamp ordering and Notes (“Previous job: JOB-XXXX”). Create derived attributes such as “setup_from_type” and “setup_to_type” (or relevant job properties like material, dimensions if available). Compute conditional statistics: average, variance, and distribution of setup duration grouped by (previous activity + current activity) pairs per machine. This yields a setup time matrix for each resource.
- **Schedule adherence and tardiness**: For each completed job, calculate tardiness = max(0, actual completion timestamp  due date). Derive metrics such as percentage of tardy jobs, average tardiness, maximum tardiness, and on-time delivery rate. Use conformance checking to quantify deviation from any nominal due-date-driven flow.
- **Impact of disruptions**: Isolate breakdown events (“Breakdown Start”) and priority changes. Measure propagation by comparing pre- and post-event task delays, queue lengths, and downstream flow times. Use causal analysis (e.g., via Directly-Follows Graphs filtered around disruption timestamps) to quantify delay cascades.

These analyses are performed in tools such as ProM, Celonis, or Python libraries (pm4py) with custom attribute extraction for setup matrices and disruption tagging.

**2. Diagnosing Scheduling Pathologies**

Process mining reveals several core inefficiencies. Bottleneck identification relies on combined high utilization (>85%) and elevated waiting times at specific machines (e.g., MILL-03 or specialized Heat Treatment units), with downstream starvation visible as low utilization and long idle periods. Variant analysis comparing on-time versus late jobs shows that late jobs disproportionately route through high-setup-transition machines or experience priority inversions. Suboptimal sequencing appears as frequent high-setup transitions in the Directly-Follows Graph for bottleneck resources, where dissimilar jobs (different materials or geometries) follow one another, inflating total setup time by 20–40% versus similar-job clustering.

Starvation is diagnosed via resource performance analysis showing periods where downstream machines wait despite upstream jobs being completed, indicating poor release timing. The bullwhip effect in WIP is quantified by reconstructing queue lengths over time and observing high variance in arrival rates to downstream centers caused by upstream variability and disruption propagation. Process mining evidence includes:
- Bottleneck analysis using waiting-time and utilization overlays on the process map.
- Variant analysis (on-time vs. late cohorts) revealing systematic differences in routing or resource contention.
- Resource contention periods identified by overlapping task intervals on the same machine or operator.

**3. Root Cause Analysis of Scheduling Ineffectiveness**

The primary root causes are the limitations of static local dispatching rules (FCFS + EDD mix) in a high-variability environment. These rules ignore global state (downstream load, sequence-dependent setup estimates, and real-time queue lengths), leading to myopic decisions. Lack of real-time visibility is evident because dispatching occurs without mined predictive models of duration or setup. Inaccurate planning manifests as large planned-vs-actual duration deviations, which process mining quantifies per activity and context (e.g., by operator or job complexity). Sequence-dependent setups are not modeled in dispatching, so transitions incur unnecessary time. Poor inter-work-center coordination appears as uncoordinated release policies that create upstream queues while starving downstream resources. Disruption handling is purely reactive, with no lookahead or rescheduling logic.

Process mining differentiates scheduling logic issues from capacity problems by comparing conformance metrics: high deviation from an idealized due-date or setup-optimized model indicates logic flaws, while persistently high utilization with low waiting-time variance points to capacity constraints. Inherent variability is isolated by examining residual time distributions after controlling for known factors (machine, operator, prior job).

**4. Developing Advanced Data-Driven Scheduling Strategies**

Three distinct strategies are proposed, each directly informed by the mined insights.

**Strategy 1: Multi-factor Dynamic Dispatching with Setup Estimation**  
At each work center, replace static rules with a composite priority score:  
Score = w·(due date slack) + w·(1 / estimated setup time) + w·(downstream load factor) + w·(priority weight) + w·(remaining processing time).  
Weights are calibrated from process mining (e.g., regression on historical tardiness vs. setup contribution). Setup time is looked up from the mined conditional distribution for the (last job properties, current job properties) pair. Downstream load is estimated from current queue lengths and average processing times. This directly addresses poor prioritization and suboptimal sequencing by penalizing high-setup transitions and incorporating global visibility. Expected impact: 15–25% reduction in average tardiness, 10–20% lower total setup time, and improved on-time delivery without increasing WIP.

**Strategy 2: Predictive Scheduling with Duration and Disruption Forecasting**  
Train machine-learning models (e.g., gradient boosting or LSTM on sequential features) on the event log to predict task durations and setup times using attributes such as job complexity proxies, operator ID, time-of-day, and prior job characteristics. Breakdown prediction uses survival analysis or recurrent models on resource event sequences to estimate remaining useful life. These predictions feed a rolling-horizon scheduler that simulates short-term future states and selects the next task that minimizes expected tardiness or a composite cost function. Process mining supplies the training features and validation distributions. This tackles inaccurate estimates and reactive disruption handling. Expected impact: 20–30% improvement in lead-time predictability, proactive bottleneck avoidance, and 10–15% lower WIP through better release timing.

**Strategy 3: Setup-Time-Optimized Sequencing via Job Clustering and Lookahead**  
At bottleneck machines, periodically cluster queued jobs using features that influence setup (material, geometry, tolerances) derived from historical setup matrix analysis. Within clusters, apply a heuristic or small-scale optimization (e.g., nearest-neighbor on the mined setup distance matrix) to sequence jobs for minimal total setup. Between clusters, insert “changeover” jobs or batch releases. This is triggered when queue length exceeds a mined threshold. It directly mitigates excessive setup times and bottleneck overload. Expected impact: 25–40% reduction in setup time at critical resources, higher effective capacity, and downstream stabilization that lowers overall lead times and tardiness.

**5. Simulation, Evaluation, and Continuous Improvement**

A discrete-event simulation model is parameterized directly from process mining outputs: empirical distributions for task and setup durations (conditioned on context), routing probabilities from the discovered process map, breakdown inter-arrival and repair times from resource events, and arrival patterns of new jobs. The simulator replays historical job mixes under high-load, high-disruption, and mixed-priority scenarios to compare the three strategies against the baseline (current dispatching rules) on KPIs including tardiness distribution, average WIP, machine utilization, total setup time, and makespan.

Validation uses hold-out log periods; statistical tests (e.g., paired t-tests on KPI differences) determine superiority. Before live deployment, stress tests include 20% demand surge, doubled breakdown frequency, and introduction of novel job variants.

For continuous improvement, an online process mining framework runs in parallel with the MES. Key KPIs are tracked in real time via dashboards. Drift detection (e.g., statistical process control on setup time distributions or arrival rates) triggers alerts. When drift exceeds thresholds, models are retrained on recent logs, weights or clustering parameters are re-optimized, and the simulation is re-run to validate adjustments. This creates a closed-loop system where scheduling logic evolves with the shop floor dynamics.
