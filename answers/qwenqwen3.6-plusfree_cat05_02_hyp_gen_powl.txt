####  Query 1: Detect Premature Closure (`A  C` bypass)
*Checks if claims were closed without passing through Evaluation or Approval, or if `C` occurred immediately after `A`.*
```sql
WITH event_timeline AS (
    SELECT 
        claim_id, 
        activity, 
        timestamp,
        LAG(activity) OVER (PARTITION BY claim_id ORDER BY timestamp) AS prev_activity
    FROM claim_events
    WHERE activity IN ('A', 'C', 'E', 'P', 'N')
)
SELECT 
    claim_id,
    timestamp AS closure_time,
    prev_activity AS preceding_step
FROM event_timeline
WHERE activity = 'C'
  AND (
      -- Case 1: Closed immediately after assignment
      prev_activity = 'A'
      OR
      -- Case 2: Closed without any E or P ever occurring
      NOT EXISTS (
          SELECT 1 FROM claim_events ce 
          WHERE ce.claim_id = event_timeline.claim_id 
            AND ce.activity IN ('E', 'P')
      )
  )
ORDER BY closure_time DESC;
```
**Interpretation:** High frequency suggests systemic exception handling or adjuster bypass. Cross-reference `additional_info` or `resource` to see if specific adjusters or claim types drive this.

####  Query 2: Detect Repeated Evaluation/Approval (Loop Anomaly)
*Identifies claims with multiple `E` or `P` events, and specifically flags `P  E` sequences.*
```sql
WITH activity_counts AS (
    SELECT 
        claim_id,
        SUM(CASE WHEN activity = 'E' THEN 1 ELSE 0 END) AS eval_count,
        SUM(CASE WHEN activity = 'P' THEN 1 ELSE 0 END) AS approve_count
    FROM claim_events
    WHERE activity IN ('E', 'P')
    GROUP BY claim_id
    HAVING SUM(CASE WHEN activity = 'E' THEN 1 ELSE 0 END) > 1 
        OR SUM(CASE WHEN activity = 'P' THEN 1 ELSE 0 END) > 1
),
p_followed_by_e AS (
    SELECT claim_id, timestamp AS approval_time
    FROM (
        SELECT claim_id, activity, timestamp,
               LEAD(activity) OVER (PARTITION BY claim_id ORDER BY timestamp) AS next_activity
        FROM claim_events
        WHERE activity IN ('E', 'P')
    ) seq
    WHERE activity = 'P' AND next_activity = 'E'
)
SELECT 
    ac.claim_id, 
    ac.eval_count, 
    ac.approve_count,
    CASE WHEN pfe.claim_id IS NOT NULL THEN 'Yes' ELSE 'No' END AS has_p_then_e_sequence
FROM activity_counts ac
LEFT JOIN p_followed_by_e pfe ON ac.claim_id = pfe.claim_id
ORDER BY ac.approve_count DESC, ac.eval_count DESC;
```
**Interpretation:** If `has_p_then_e_sequence = 'Yes'` is common, the loop is actively used for rework. If counts are high but sequence is absent, it may indicate duplicate logging or system retries.

####  Query 3: Detect Skipped Customer Notification
*Finds closed claims that never triggered the `N` activity.*
```sql
SELECT 
    c.claim_id,
    c.claim_type,
    c.claim_amount,
    c.submission_date,
    MAX(ce.timestamp) AS closure_timestamp
FROM claims c
JOIN claim_events ce ON c.claim_id = ce.claim_id AND ce.activity = 'C'
LEFT JOIN claim_events ne ON c.claim_id = ne.claim_id AND ne.activity = 'N'
WHERE ne.event_id IS NULL
GROUP BY c.claim_id, c.claim_type, c.claim_amount, c.submission_date
ORDER BY closure_timestamp DESC;
```
**Interpretation:** A high volume of skipped notifications, especially for high-value claims, indicates a compliance risk. If `skip` correlates with specific `claim_type`s, it may reflect an undocumented policy exception.
