**Process Mining Approach for Optimizing Resource Assignment at TechSolve Solutions**

### 1. Analyzing Resource Behavior and Assignment Patterns

Using the event log (with attributes for Resource, Agent Tier, Agent Skills, Required Skill, Ticket Category, Priority, and activity types like Assign/ Work Start/End/ Reassign/ Escalate), resource behavior is analyzed through **resource-centric process mining** techniques.

- **Individual Agent and Tier Performance Metrics**:
  - Workload distribution: Count of active cases per agent/tier over time (using START/COMPLETE timestamps to compute concurrency).
  - Activity processing times: Average handling time (AHT) per agent/tier for specific activities (e.g., Work L1 End duration), broken down by Ticket Category and Required Skill.
  - First-call resolution (FCR) rate for L1: Percentage of tickets resolved at L1 without escalation (filter cases where "Work L1 End" is followed by resolution rather than "Escalate L2").
  - Skill utilization: Frequency with which an agent's documented skills match the Required Skill at assignment vs. actual work performed. Calculate mismatch rate (e.g., agent assigned to "Networking-Firewall" but lacking the skill).
  - Escalation and reassignment rates: Per-agent and per-tier counts of "Escalate" and "Reassign" activities, normalized by ticket volume and priority.

- **Process Mining Techniques for Actual Patterns**:
  - **Resource interaction and handover analysis**: Build a handover matrix or social network from activities where one resource (e.g., Dispatcher or L1 agent) hands off to another via "Assign" or "Reassign". Use tools like ProM or Celonis to visualize flows (e.g., L1  L2 handovers).
  - **Social network analysis (SNA)**: Apply SNA metrics (betweenness centrality, density) on handover graphs to identify key connectors or bottlenecks in reassignments.
  - **Role discovery and conformance checking**: Discover actual roles from the log (e.g., via fuzzy mining or inductive miner with resource perspectives) and compare against the normative model (round-robin within tiers + manual escalation). Deviations appear as frequent "Reassign" paths or L1 agents performing L2-level work.
  - **Performance analysis**: Heatmaps of waiting times between "Escalate L2" and "Assign L2", or between "Reassign" and next assignment.

- **Skill Utilization Analysis**:
  Aggregate cases by Required Skill and cross-reference with assigned agent's skills. Calculate "skill underutilization" (specialists handling basic tickets) and "over-assignment" (L1 agents on complex skills). In the example log, INC-1001 shows initial assignment to App-CRM specialist followed by reassignment for Database-SQL, indicating poor initial skill matching.

This reveals gaps between intended round-robin/manual logic and actual behavior (e.g., frequent reassignments due to skill mismatches).

### 2. Identifying Resource-Related Bottlenecks and Issues

From the analyses above, specific problems are pinpointed as follows:

- **Skill Availability Bottlenecks**: Cases where "Assign L2" is delayed or followed by "Reassign" because no agent with the exact Required Skill is available (measure as queue time after escalation). In the log, INC-1001's delay from 09:36 to 10:05 before L2 work start, plus reassignment at 11:15, indicates this.
- **Reassignment and Escalation Delays**: Count "Reassign" occurrences and compute average added cycle time (time between Reassign timestamp and resolution). Frequent reassignments (as in INC-1001) introduce 1-2 hour delays.
- **Incorrect Initial Assignments**: Track cases where L1 or Dispatcher assigns without matching skills, leading to immediate escalation. Correlate with ticket attributes (e.g., P2 Network tickets escalated by L1 agents lacking Networking-Firewall skills).
- **Overloaded/Underutilized Agents**: Use concurrency metrics to identify agents with >150% average workload vs. those below 50%. Overlay with SLA breach indicators (inferred from resolution times exceeding priority-based thresholds).
- **SLA Correlation**: Use correlation analysis (e.g., regression or decision trees) between assignment patterns and SLA breaches for P2/P3 tickets. Quantify: e.g., "Reassignment increases breach probability by 35%"; "Skill mismatch adds average 2.3 hours to resolution."

These are quantified via performance dashboards showing % of cases with >1 reassignment, average delay per reassignment, and % of SLA breaches linked to specific patterns (e.g., L1-to-L2 escalation without skill filter).

### 3. Root Cause Analysis for Assignment Inefficiencies

Potential root causes, validated through the log:

- **Deficient Assignment Rules**: Round-robin ignores skills and real-time workload, causing mismatches (evident in INC-1001's initial App-CRM assignment despite later DB need).
- **Inaccurate Skill Profiles**: Agents' documented skills (e.g., "App-CRM, DB-SQL") do not align with actual work or ticket requirements.
- **Poor Initial Categorization**: Required Skill changes mid-case (as in INC-1001), indicating weak front-end triage.
- **Lack of Real-Time Visibility**: No data-driven view of availability leads to queueing and manual reassignments.
- **L1 Empowerment Gaps**: High escalation rates suggest insufficient training or decision support for L1 agents.

**Advanced Techniques**:
- **Variant Analysis**: Cluster cases into "smooth" (0-1 reassignments, quick resolution) vs. "problematic" (multiple reassigns/escalations) using trace clustering. Compare attributes (priority, category, initial assignee) to isolate triggers.
- **Decision Mining**: Mine rules (e.g., via decision trees) on features like Ticket Category, Priority, and Agent Tier that precede "Escalate" or "Reassign". Example rule: "If Required Skill = Database-SQL and assigned to L1 with only Basic-Troubleshoot  80% escalation probability."
- **Conformance Checking**: Measure fitness of actual paths against intended model to quantify rule deviations.

This identifies whether issues stem from rules, data quality, or human factors.

### 4. Developing Data-Driven Resource Assignment Strategies

Three concrete strategies, derived from the mined insights:

**Strategy 1: Skill-Based Routing with Proficiency Weighting**  
Addresses: Skill mismatches and underutilization of specialists.  
Leverages: Resource profiles and skill utilization analysis showing frequent reassignments for specific Required Skills.  
Data Required: Historical matches between Required Skill and Agent Skills, plus proficiency scores (inferred from resolution success rates per skill).  
Implementation: Replace round-robin with a matching algorithm (e.g., cosine similarity on skill vectors) that prioritizes exact matches, weighted by historical success.  
Expected Benefits: 25-40% reduction in reassignments, 15% faster resolution for complex tickets, improved specialist utilization (from ~40% effective use to 70%+).

**Strategy 2: Workload-Aware Assignment with Real-Time Queue Visibility**  
Addresses: Uneven workload distribution and delays from overloaded agents.  
Leverages: Concurrency and workload metrics identifying overloaded vs. idle agents/tiers.  
Data Required: Live open case counts per agent (from event log timestamps), current queue lengths, and agent availability flags.  
Implementation: Integrate a dispatcher algorithm that assigns to the least-loaded qualified agent (e.g., using shortest queue + skill filter). Add "pull" capability for L1 agents during low-volume periods.  
Expected Benefits: Balanced utilization (variance reduced by 50%), 20% drop in SLA breaches for P2/P3, fewer handoff delays.

**Strategy 3: Predictive Assignment and Escalation Engine**  
Addresses: Excessive/unnecessary escalations and poor initial categorization.  
Leverages: Variant analysis and decision mining on factors predicting successful L1 resolution vs. escalation needs.  
Data Required: Ticket attributes (Category, Priority, historical resolution outcomes), plus optional text features from descriptions for ML models (e.g., random forest or logistic regression trained on past cases).  
Implementation: Pre-assign or auto-escalate based on predicted complexity/skill needs at creation. Refine L1 guidelines (e.g., "Resolve if Category=Software-App and no DB skill needed").  
Expected Benefits: 30% increase in L1 FCR, 25% fewer escalations, proactive skill matching reducing average handling time by 18%.

### 5. Simulation, Implementation, and Monitoring

**Simulation for Pre-Implementation Evaluation**:
- Import the discovered process model (via inductive miner) and resource profiles (utilization, AHT distributions, skill mappings) into a simulation tool (e.g., ProM's simulation plugin or AnyLogic with process mining export).
- Run "what-if" scenarios: Apply each new assignment strategy as a routing policy and simulate 6-12 months of synthetic cases based on historical arrival rates and ticket characteristics.
- Measure KPIs: Resolution time, SLA compliance, reassignment count, and resource utilization under different loads. Compare against baseline (current round-robin) to quantify risk and ROI before rollout.

**Post-Implementation Monitoring Plan**:
- Deploy continuous process mining (e.g., via Celonis or Power BI connected to the ticketing system) on the live event log.
- Key Resource-Related KPIs and Views:
  - Reassignment rate (% of cases with 1 Reassign).
  - Skill match accuracy (% of assignments where Agent Skills  Required Skill).
  - Agent utilization heatmaps and workload variance.
  - SLA breach rate by tier and priority, correlated with assignment patterns.
  - FCR rate by L1 agent and ticket type.
  - Handover network diagrams updated weekly.
  - Decision rule dashboards showing adherence to new predictive rules.
- Review cadence: Weekly operational dashboards + monthly deep-dive conformance checks. Trigger alerts for >10% deviation from targets (e.g., rising reassignments).

This data-driven, iterative approach—rooted in the event log—enables TechSolve to transition from reactive, inefficient assignment to proactive, skill-optimized operations while continuously validating improvements through process mining.
