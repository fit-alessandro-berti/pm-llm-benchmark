# Optimizing Resource Assignment at TechSolve Solutions: A Process Mining Approach

## 1. Analyzing Resource Behavior and Assignment Patterns

### 1.1 Agent and Tier Performance Analysis

Using the event log, I would construct a **resource-centric view** by pivoting activities on the `Resource` attribute rather than just the `Case ID`. The core metrics I would compute per agent and per tier are:

**Workload & Capacity Metrics:**
- **Ticket volume per agent per period** (daily/weekly), segmented by Priority and Category
- **Concurrent ticket load** — number of tickets in "Work Start" state without a matching "Work End" at any point in time (measured via event stream replay)
- **Occupancy rate** — ratio of active working time to available shift time
- **Queue-waiting time contribution** — time between "Assign" and corresponding "Work Start" per agent

**Processing Performance Metrics:**
- **Activity processing time** (Work Start  Work End) per agent, benchmarked against tier median for the same ticket category/skill
- **First-Call Resolution (FCR) rate for L1** = tickets where the sequence contains "Work L1 End" followed by "Ticket Resolved" (no "Escalate L2") / total L1 tickets
- **Rework rate** — frequency of "Reassign" or "Escalate" events originating from the agent
- **SLA compliance rate per agent** — proportion of the agent's tickets closed within priority-based SLA thresholds

**Skill-Ticket Matching Metrics:**
- **Skill match ratio** = (cases where `Required Skill`  `Agent Skills`) / (total cases handled by agent)
- **Activity mix per agent** — distribution of ticket categories handled vs. primary skill set

### 1.2 Revealing Actual Assignment Patterns

**Social Network Analysis (Handover-of-Work):**
I would build a directed handover graph where nodes are agents and edges represent a ticket passing from one resource to another (via Escalate/Reassign/Assign events). Edge weights represent frequency. This reveals:
- **Central "hub" agents** receiving disproportionate escalations (overloaded specialists)
- **Ping-pong patterns** between specific L2 agents indicating skill boundary confusion
- **Isolated nodes** — underutilized agents
- **Tier-crossing density** — actual vs. expected L1L2L3 flow

**Role Discovery (using organizational mining, e.g., similarity-based clustering):**
Cluster agents by the activities and ticket types they actually perform. If the discovered clusters don't align with the documented tier/skill structure, there's an organizational drift — e.g., some "L2" agents may be behaving functionally like L1.

**Process Variant Analysis:**
Compute the frequency of each process variant (sequence of activities per case). Compare the "golden path" (L1 Created  L1 Work  Resolved) with variants containing multiple escalations/reassignments. The deviation volume quantifies how often reality diverges from the intended flow.

**Conformance Checking:**
Model the *intended* assignment logic (e.g., "P1 should skip L1", "skill X tickets should route to skill-X-capable agents") as a reference process and replay the log to measure fitness — exposing systematic policy violations.

### 1.3 Skill Utilization Analysis

- Build a **skill-activity matrix**: rows are skills, columns are activity categories, cells contain frequency and median duration.
- For each specialist skill (e.g., Database-SQL), compute the **task complexity distribution** — what proportion of tickets handled by SQL experts are genuinely SQL-complex vs. tasks that any agent could perform?
- Calculate a **skill-over-qualification index**: tickets resolved by L3 agents where the required skill is also held by available L2 agents (and L3's specialized skill was unused) / total L3 work.
- Identify **skill bottleneck points**: tickets waiting in a queue because only 1-2 agents have the required rare skill (measurable via queue-length time series at the skill level).

## 2. Identifying Resource-Related Bottlenecks and Issues

### 2.1 Specific Bottleneck Identification

| Issue | Detection Method | Example Metric |
|---|---|---|
| **Skill scarcity bottleneck** | Queue length time series by `Required Skill`; waiting time distribution | "Networking-Firewall tickets wait avg. 2.4 hr before Work Start — 4× the category median" |
| **Reassignment delays** | Count of Reassign events per case × avg. delay per Reassign | "18% of P3 tickets have 1 reassignment; avg. added cycle time = 3.8 hr" |
| **Incorrect initial routing** | Cases where L1 escalates within <10 min of Work L1 Start with no substantive work | "32% of L1 tickets escalate immediately — pure routing waste" |
| **Overloaded agents** | Concurrent ticket count > 95th percentile of tier | "Agent B08 consistently carries 15+ concurrent tickets vs. L2 median of 7" |
| **Underutilization** | Occupancy rate < 40% | "Agent A11: 28% occupancy but same shift schedule" |
| **SLA-breach correlation** | Statistical test (e.g., logistic regression) linking SLA breach to reassignment count, initial assignment wait, skill mismatch | "Tickets with 2 reassignments have 4.7× higher SLA breach odds" |

### 2.2 Quantification Framework

Build a **Pareto analysis** of SLA breaches by contributing factor:
- % breaches attributable to **dispatcher wait time** (time Ticket Created  Assign L1)
- % breaches attributable to **escalation lag** (Escalate  Work L2 Start)
- % breaches attributable to **reassignment cycles**
- % breaches attributable to **skill mismatch** (cases where `Required Skill`  `Agent Skills` at assignment time)

Example expected finding: *"Skill mismatches occur in 22% of escalations but account for 41% of SLA breaches in P2 tickets, adding an average of 94 minutes to resolution time."*

## 3. Root Cause Analysis for Assignment Inefficiencies

### 3.1 Candidate Root Causes

1. **Round-robin dispatch** ignores skill and load — evidenced by the fact that Reassign events cluster on specific ticket-agent combinations where the initial assignment violated skill requirements.
2. **Stale skill profiles** — detectable when agents routinely resolve tickets requiring skills not listed in their documented skill set (implicit skill), or never touch documented skills (phantom skill).
3. **Weak ticket categorization at intake** — identified when the `Required Skill` field changes between Assign L1 and Assign L2 (as in INC-1001 where the skill changed from App-CRM to Database-SQL).
4. **No real-time workload visibility** — evidenced by simultaneous assignment of multiple tickets to already-busy agents while peers are idle.
5. **L1 under-empowerment** — evidenced by high escalation rates for ticket types that L1 *sometimes* resolves successfully (indicating capability exists but isn't consistently applied).

### 3.2 Variant and Decision Mining

**Variant analysis** — Split cases into two cohorts:
- **"Smooth" cohort**: single assignment, resolved within SLA
- **"Turbulent" cohort**: 2 reassignments or escalations

Compare their attribute distributions: channel of origin, initial categorizer, time-of-day, assigning dispatcher, ticket priority, initial assigned agent. Significant differences pinpoint structural causes (e.g., "tickets created via phone during shift changeover are 3× more likely to be turbulent").

**Decision mining** on the assignment decision point(s) (Assign L1, Escalate, Assign L2): train a decision tree with target = assignment outcome (smooth vs. reassigned) and features = ticket priority, category, required skill, current queue depth, assigning dispatcher, agent skill match, agent current load. The learned rules expose *why* poor assignments are made — often showing that skill attributes carry almost no weight in the current decision logic.

## 4. Data-Driven Resource Assignment Strategies

### Strategy 1: Skill-Weighted, Proficiency-Based Routing

- **Issue addressed:** Skill mismatches, reassignments due to wrong specialist, overuse of senior specialists for easy tasks.
- **Process mining insight leveraged:** The skill-activity matrix and historical processing times reveal each agent's **empirical proficiency** per skill — not just binary "has skill" but a proficiency score (e.g., median processing time and success rate for that skill vs. peers).
- **Mechanism:** When a ticket arrives, score each available agent as `score = proficiency(skill) × availability_factor × tier_match`. Assign to highest score. Tier match demotes over-qualified agents for simpler tickets.
- **Data required:** Historical event log for proficiency computation; current agent status (available/busy); ticket's required skill and complexity estimate.
- **Expected benefits:** 30–50% reduction in reassignments; specialists freed up ~20% for truly complex work; faster average resolution.

### Strategy 2: Workload-Aware Dispatch with Queue-Depth Signals

- **Issue addressed:** Uneven workload distribution; idle agents alongside overloaded ones; queue build-up for specific skills.
- **Process mining insight leveraged:** Real-time replay of the event stream produces live queue length and concurrent-load metrics per agent and skill. Historical analysis identifies saturation thresholds beyond which processing times spike (e.g., >10 concurrent tickets  60% slower).
- **Mechanism:** Dispatch algorithm uses a cost function combining skill match AND current agent load, with a soft cap at the saturation threshold. Tickets overflow to the next-best-qualified agent below threshold.
- **Data required:** Live event stream; per-agent saturation curves derived from historical data; shift/availability schedules.
- **Expected benefits:** Workload variance (Gini coefficient across agents) reduced by ~40%; queue-wait time reduced for high-demand skills; fewer "stuck" tickets.

### Strategy 3: Predictive Triage and Smart Escalation

- **Issue addressed:** Incorrect initial categorization; unnecessary L1 touches that add delay; excessive escalation of resolvable tickets.
- **Process mining insight leveraged:** Decision mining on historical tickets learns patterns predicting (a) the *true* required skill from initial ticket fields/text, and (b) the probability that L1 can resolve it.
- **Mechanism:** A classifier evaluates every new ticket:
  - If `P(L1 resolution) > threshold`, route to L1 with matching secondary skill
  - If low probability and clear specialist skill, **skip L1 entirely** and route directly to appropriate L2/L3
  - Flag ambiguous tickets for dispatcher review
- **Data required:** Historical tickets with outcomes (resolved at which tier); ticket text/category/channel; resolution paths; agent skill graph.
- **Expected benefits:** 15–25% of currently-escalated tickets bypass L1 (saving ~45 min each); L1 FCR improves as they handle only resolvable tickets; P2 SLA compliance rises materially.

### Strategy 4 (Bonus): Dynamic Tier Flexing Based on Demand Forecast

- **Issue addressed:** Rigid tier boundaries cause backlog when demand spikes for specific skills.
- **Process mining insight leveraged:** Time-series analysis of ticket arrival by category reveals predictable demand patterns (e.g., Monday morning access-management spike; end-of-quarter CRM issues).
- **Mechanism:** Forecast demand; temporarily re-designate cross-trained agents to the tier/skill pool in highest need during predicted peaks.
- **Data required:** Multi-month arrival patterns; agent cross-skill matrix; demand forecasting model.
- **Expected benefits:** Flattens peak-driven SLA breaches; higher overall throughput without added headcount.

## 5. Simulation, Implementation, and Monitoring

### 5.1 Pre-Implementation Simulation

Using the **discovered process model** (from directly-follows graph or Petri net mining) as the structural foundation, I would build a **discrete-event simulation** enriched with mined parameters:

- Ticket arrival distributions (inter-arrival times) by priority/category — derived from event log
- Empirical activity duration distributions per (activity, agent tier, skill match) — sampled from the log
- Agent shift calendars and availability
- Current assignment logic encoded as the baseline

**Scenarios to simulate:**
| Scenario | Change |
|---|---|
| S0 (Baseline) | Current round-robin |
| S1 | Skill-weighted routing (Strategy 1) |
| S2 | Workload-aware dispatch (Strategy 2) |
| S3 | Predictive triage (Strategy 3) |
| S4 | Combined S1+S2+S3 |

Run each with Monte Carlo (e.g., 500 replications) and compare distributions (not just means) of: avg. resolution time, SLA compliance %, reassignment rate, specialist utilization, agent load variance. Also perform **stress tests** (2× arrival rate) to check robustness.

This quantifies expected gains *and* surfaces unintended consequences (e.g., a new bottleneck shifting to L3) before committing.

### 5.2 Post-Implementation Monitoring Plan

Deploy a **live process mining dashboard** fed by the ITSM system's event stream, with these components:

**Operational KPIs (Real-Time):**
- SLA compliance rate by priority (rolling 24 hours)
- Average queue-wait time per skill
- Active ticket count vs. agent capacity (load heatmap)
- Reassignment/escalation rate (rolling 7 days)
- FCR rate at L1

**Resource-Focused Views:**
- Agent occupancy heatmap (green/yellow/red zones)
- Skill utilization chart vs. historical baseline
- Handover social network visualization — highlight deviations from expected flow
- Agent proficiency drift (is the skill model still accurate?)

**Strategic KPIs (Weekly/Monthly):**
- Mean cycle time per ticket category
- Percentage of tickets following the "smooth" variant
- Conformance score of actual flow vs. new assignment policy
- Cost per resolved ticket (FTE time × resolution time)

**Continuous Feedback Loop:**
- Monthly refresh of agent proficiency scores from new log data
- Quarterly decision-mining re-run to refresh predictive triage model
- Alerting on KPI drift (e.g., SLA compliance drops >5% week-over-week triggers root cause drill-down)
- **A/B testing** for further refinements: route a small % of tickets through alternative algorithms and compare outcomes in the live environment

This closed-loop monitoring ensures that the assignment logic remains data-driven and adapts as the workforce, skills, and ticket mix evolve — preventing regression to the inefficiencies that prompted this engagement.
