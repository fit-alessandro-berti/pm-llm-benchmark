## 1. Analyzing Historical Scheduling Performance and Dynamics

To reconstruct and analyze the actual flow of jobs and task execution, I would apply **process mining** on the MES event logs. The logs contain timestamped events per job (Case ID) for each activity/task, including resource allocation, setup times, durations, and disruption events. The key steps and techniques are:

- **Event Log Preprocessing & Filtering**  
  Convert raw logs into a standard format (e.g., XES or CSV with case, activity, timestamp, resource, and additional attributes like priority, due date, planned/actual duration, setup flag). Remove incomplete or erroneous records. Identify the process model of the job shop (routing per job type) using **discovery algorithms** (e.g., heuristic mining, inductive miner) to capture the typical sequence of operations per product family.

- **Performance Metrics Mining**  
  Use **performance spectrum** and **dotted chart** analyses to visualize job progress over time. Extract:
  - *Job flow time* = time from job release to completion (end of last task).  
  - *Lead time* = time from order placement (or release) to due date or delivery – compare actual vs. planned.  
  - *Makespan* per job or per batch – distribution across jobs.
  - *Task waiting time* = time between queue entry and task start for each work center. Aggregate by resource to get average, median, and tail quantiles.
  - *Resource utilization*: For each machine, compute % time spent on:
    - *Productive* (task duration actual)
    - *Setup* (setup actual)
    - *Idle* (no job queued or waiting for operator)
    - *Breakdown* (from resource events)
    - *Blocked* (downstream queue full? – infer from waiting times).

- **Sequence-Dependent Setup Times**  
  Extract pairs of consecutive jobs on the same machine (using resource and timestamps). For each pair (previous job, current job), calculate actual setup time from setup start to setup end. Build a **setup time matrix** per machine, grouped by job attributes (e.g., material, geometry, required tooling). Use **decision tree** or **clustering** on historical data to model setup time as a function of job properties and sequence.

- **Schedule Adherence & Tardiness**  
  For each job, compute tardiness = max(0, actual end time – due date). Plot distribution of tardiness per job, per priority class, per resource. Use **conformance checking** to compare planned schedule (if any) against actual execution; measure deviation in start times and sequence.

- **Impact of Disruptions**  
  Tag jobs that experienced a breakdown or priority change. Compare their flow time, waiting time, and tardiness against jobs without disruption using **statistical tests** (e.g., Mann-Whitney). Use **process variant analysis** (e.g., compare “on-time” vs “late” jobs) to identify patterns like longer setups or additional rework.

## 3. Root Cause Analysis of Scheduling Ineffectiveness

Process mining helps distinguish between causes:

- **Static Dispatching Rules vs. Dynamic Environment**  
  Compare the actual sequence of job execution with the sequence that would be produced by EDD or FCFS (simulated from log). High deviation indicates that local rules were overridden by expediting – symptom of lack of global visibility. Root cause: rules ignore downstream load, setup dependencies, and real-time urgency.

- **Lack of Real-Time Visibility**  
  Logs show operators start setups with outdated info (e.g., a machine idle because a tool is missing). Process mining reveals long gaps between task end and next setup start – inefficiency due to lack of a central dispatch.

- **Inaccurate Duration Estimations**  
  Compare planned vs actual task duration per machine/operator. If actual duration has high variance (e.g., CNC milling 60 min plan, actual range 30-120 min), the deterministic planning is poor. Process mining can extract **duration distribution** per combination of job type, operator, machine, and time-of-day to reveal root causes (e.g., operator skill, complexity).

- **Ineffective Handling of Setup Dependencies**  
  The setup time matrix from mining shows that disregarding sequence leads to excessive setups. Root cause: the current rules (EDD) do not consider setup similarity.

- **Poor Coordination & Disruption Response**  
  When breakdown occurs, logs show that downstream jobs are not reprioritized – they continue waiting long times. Cause: no dynamic rescheduling triggered; operators simply continue FIFO after repair.

- **Capacity vs. Scheduling**  
  If even under optimal sequencing the bottleneck utilization is >95%, then root cause is capacity shortage, not just scheduling. Process mining can simulate perfect sequencing and compare to observed throughput – if theoretical throughput is still below demand, capacity expansion is needed.

## 4. Developing Advanced Data-Driven Scheduling Strategies

### Strategy 1: Dynamic Multi-Criteria Dispatching with Setup Awareness
- **Core Logic**: At each machine, when a resource becomes free, select the next job from its queue using a weighted sum of:
  - *Urgency*: (due date – current time) / remaining total processing time (critical ratio)
  - *Priority*: numeric score (High=10, Medium=5, Low=1)
  - *Downstream load index*: average queue length at subsequent work centers (from live MES)
  - *Setup cost*: estimated setup time for the candidate job given previous job (from mined setup matrix), normalized to a cost penalty
  - *Idle time risk*: if job is almost ready but not yet arrived (e.g., still in upstream queue) – we avoid selecting jobs that would cause idle wait.
- **Use of Process Mining**: Mining provides the setup time model, load patterns, and critical ratio distributions. The weights can be tuned via simulation using historical logs (e.g., optimize for minimum weighted tardiness).
- **Addresses Pathologies**: Reduces tardiness by considering urgency and priority; reduces setup overhead by penalizing long-setup sequences; balances downstream load to prevent starvation.
- **Expected Impact**: 20-30% reduction in tardiness, 15% reduction in WIP, 10% improvement in utilization.

### Strategy 2: Predictive Scheduling with Duration Distributions and Maintenance Forecast
- **Core Logic**: Generate a proactive schedule using a **simulation/heuristic** (e.g., shifting bottleneck heuristic) that uses:
  - **Probabilistic task durations** – sample from empirical distributions (per resource, job type, operator) mined from event logs, not fixed plans.
  - **Predictive maintenance model** – if logs show breakdown patterns (e.g., after X hours runtime), forecast likely next breakdown and insert buffer time or schedule preventive maintenance.
  - **Setup time distributions** – use sequence-dependent means (or full distribution) to create robust sequences.
  - The schedule is re-evaluated every shift or upon disruption using **rolling horizon** (e.g., 2-day horizon re-optimized with updated status).
- **Use of Process Mining**: Extract duration distributions (log-normal, truncated), breakdown frequencies (Poisson or Weibull), setup time family matrices. These parameterize the scheduling engine.
- **Addresses Pathologies**: Realistic durations prevent over-optimistic promises; buffer against breakdowns reduces disruption impact; setup-aware sequencing decreases total setup time.
- **Expected Impact**: 30% improvement in schedule adherence (tardiness reduction), more predictable lead times, 10-15% reduction in WIP due to smoother flow.

### Strategy 3: Setup-Oriented Batching and Sequencing at Bottleneck
- **Core Logic**: Focus on the primary bottleneck machine(s) identified from process mining. Implement a **two-level approach**:
  - **Batching**: Group jobs with identical or very similar setup properties (e.g., same material, same tool) into “families”. Release them together from upstream stations (using a hold-for-family queue) but within due-date constraints.
  - **Sequencing**: For jobs waiting at the bottleneck, apply a **setup-optimized sequencing algorithm** (e.g., traveling salesman heuristic using setup time as distance) while respecting due date windows (e.g., constraint: job must complete before its due date minus downstream processing time).
  - Non-bottleneck machines are driven by the bottleneck schedule (Drum-Buffer-Rope logic) to avoid excess WIP.
- **Use of Process Mining**: The setup time matrix (from mining) defines distances. Also, the routing variants show which jobs share similar downstream paths (to avoid splitting families).
- **Addresses Pathologies**: Drastically reduces non-productive setup time at bottleneck (up to 50% reduction in total setup), which increases effective capacity, reduces WIP, and improves throughput without capital investment.
- **Expected Impact**: 20% increase in bottleneck throughput, 25% reduction in average flow time, lower tardiness due to more capacity.
