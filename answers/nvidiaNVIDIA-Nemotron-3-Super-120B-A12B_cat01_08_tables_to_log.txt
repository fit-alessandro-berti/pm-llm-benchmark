**Assumptions / Notes**

1. **Case ID** – taken directly from `order_id`.  
2. **Resource/Actor** – mapped using the optional *Resources* table where IDs exist (sales person, warehouse staff, carrier, billing clerk). For payment events no explicit performer is recorded, so the column is left blank; payment method and reference are placed in *Other Attributes*.  
3. **Item Picked** – each line in `Order Lines` creates a separate event to preserve detail (item id, quantity, picker).  
4. **Delivery** – each shipment yields both a dispatch and an arrival event.  
5. **Chronology** – events are sorted by timestamp within each case; where timestamps are identical (none in this dataset) the order follows the natural process (creation  picking  dispatch  arrival  invoicing  payment).  
6. **Missing data** – orders 1004 has no delivery, invoice, or payment records; only the available events are included.  
7. **Other Attributes** – represented as a simple `key=value; key=value` string for readability.  

This table can now be loaded into processmining tools (e.g., ProM, Disco, Celonis) for further analysis.
