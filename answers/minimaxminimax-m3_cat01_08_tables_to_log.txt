# Comprehensive Event Log for Process Mining

## Approach & Methodology

I integrated data from all five tables (`Orders`, `Order Lines`, `Delivery`, `Invoice`, `Payments`) and enriched it using the `Resources` table. Each row in the resulting event log represents a single activity in the order-to-cash process.

### Activity Derivation Rules
| Source Table | Activity | Timestamp Field | Resource Field |
|---|---|---|---|
| Orders | Order Created | `order_date` | `sales_person_id` |
| Order Lines | Item Picked (one per line) | `item_picking_date` | `picker_id` |
| Delivery | Order Dispatched | `delivery_dispatch_date` | `carrier_id` |
| Delivery | Order Arrived | `delivery_arrival_date` | `carrier_id` |
| Invoice | Invoice Issued | `invoice_issue_date` | `billing_person_id` |
| Payments | Payment Received | `payment_date` | (Customer / External) |

### Assumptions & Notes
1. **Case ID** = `order_id` (5 cases: 1001–1005).
2. **One "Item Picked" event per line item** (not aggregated), since pickers differ and the picking timestamps may differ — this preserves real process granularity.
3. **Multiple deliveries per order** produce multiple Dispatch/Arrive event pairs (e.g., Order 1002 is split into two shipments).
4. **Multiple invoices per order** produce multiple Invoice Issued events (e.g., Order 1002).
5. **Payment Received** is linked to an order through its `invoice_id`  `order_id` relationship.
6. **Customer (External)** is used as the resource for payment events, as the data does not identify an internal actor for receiving payments.
7. **Order 1004 is an incomplete case** — only Order Created and Item Picked events exist; no delivery, invoice, or payment occurred in the dataset (this is preserved as-is).
8. All events within a case are **sorted chronologically** to reflect actual process flow.
