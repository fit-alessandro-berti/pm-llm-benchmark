We are given several tables and we need to create an event log for process mining.
 Steps:
 1. Identify cases: use `order_id` as the case identifier.
 2. Derive activities from each table.
 3. Map timestamps: each event should have a timestamp. When multiple timestamps exist for one activity (like multiple picking events for one order), we may create separate events if they are distinct.
 4. Combine data: link events by `order_id` and ensure chronological order.
 5. Enrich with resources and attributes.

 We have the following tables:
   - Orders: gives the order creation event.
   - Order Lines: gives picking events for each item. Note that an order may have multiple items, and each item is picked at a different time by different pickers.
   - Delivery: gives dispatch and arrival events. Note that an order may have multiple deliveries (split shipments).
   - Invoice: gives invoice issued events. An order may have multiple invoices.
   - Payments: gives payment received events for each invoice.

 We also have optional tables for Products and Resources, which we can use for additional attributes.

 Let's define the activities we can create:

 From Orders:
   - "Order Created": timestamp = order_date, resource = sales_person_id (or from Resources table, we can get the name)

 From Order Lines:
   - For each line item (each row in Order Lines) we can create an event "Item Picked". However, note that multiple items in the same order may be picked at the same time? In this data, they are at different times. We'll create one event per item.
   - Alternatively, if we want to model the picking of all items in an order as one event, we would need to aggregate. But the problem says: "If multiple timestamps are available (e.g., `item_picking_date` for multiple items), produce one event per activity if it𠏋 meaningful, or summarize if appropriate." 
   - Since the picking of each item is a separate activity, we will create one event per line item.

 From Delivery:
   - We have two events per delivery: dispatch and arrival. So we can create two events per delivery record: "Order Dispatched" and "Order Arrived".
   - Note: an order may have multiple deliveries, so we may have multiple dispatch and arrival events for the same order.

 From Invoice:
   - "Invoice Issued": one event per invoice.

 From Payments:
   - "Payment Received": one event per payment.

 We also note that we have a Products table that can be joined with Order Lines to get item details, and a Resources table that can be joined to get the name of the resource (for sales_person, picker, carrier, billing_person).

**Assumptions made:**
1. **Order ID as Case ID:** Each process instance is uniquely identified by `order_id`.
2. **Activity Mapping:** 
   - "Order Created" from `Orders.order_date`.
   - "Item Picked" from each row in `Order Lines` (one event per line item).
   - "Order Dispatched" from `Delivery.delivery_dispatch_date`.
   - "Order Arrived" from `Delivery.delivery_arrival_date`.
   - "Invoice Issued" from `Invoice.invoice_issue_date`.
   - "Payment Received" from `Payments.payment_date`.
3. **Resource Mapping:** 
   - `Orders.sales_person_id`  "Order Created" resource.
   - `Order Lines.picker_id`  "Item Picked" resource.
   - `Delivery.carrier_id`  "Order Dispatched" and "Order Arrived" resource.
   - `Invoice.billing_person_id`  "Invoice Issued" resource.
   - Payments have no explicit resource; left blank.
4. **Attributes:** Joined `Products` table to enrich "Item Picked" events with `item_description` and `item_category`.
5. **Chronological Order:** Events for each case are sorted by timestamp, reflecting the actual process flow.
6. **Multiple Deliveries/Invoices:** For orders with multiple deliveries or invoices (e.g., order 1002), separate events were created for each, as they represent distinct activities.
7. **Order 1004:** Only the "Order Created" event is present, as no further data exists for this order in the provided tables.
