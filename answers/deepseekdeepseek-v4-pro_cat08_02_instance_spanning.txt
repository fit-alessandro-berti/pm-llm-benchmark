## 1. Identifying Instance-Spanning Constraints and Their Impact

The event log records both start and complete timestamps for each activity, along with resource, order type, destination, coldpacking requirement, and hazardous material flag. Using process mining, we can isolate waiting times caused by interinstance dependencies from those originating within a single order.

### Mapping Each Constraint to Log Evidence and Metrics

**Shared ColdPacking Stations (limited to 5)**
- **Detection:** Filter orders with `Requires Cold Packing = TRUE`. Compute the time between “Packing – START” and the preceding “Item Picking – COMPLETE” (the “waitingforcoldpacking” period). Plot a dotted chart with time on the xaxis and coldpacking stations on the yaxis; orders waiting for the same station will stack, exposing contention.  
- **Metrics:**  
  - *Waiting time due to coldpacking contention*: time from picking completion until packing start for cold orders.  
  - *Queue length* over time at the pool of 5 stations.  
  - *Service time* at cold stations vs. standard stations to see if cold packing is inherently longer.  
  - *Throughput impact*: compare cycle times of cold vs. noncold orders.
- **Differentiating within vs. betweeninstance waiting:** If a cold order’s packing start is delayed even though a station is idle at picking completion, it’s withininstance (e.g., operator travel). If delay correlates with all stations busy on other orders, it’s betweeninstance. We use a resourceaware process model (e.g., social network analysis) to see which orders held the station in the preceding window.

**Batching for Shipping**
- **Detection:** The log shows a “Shipping Label Gen.” activity with a batch identifier (e.g., Batch B1). The completion of “Quality Check” (or sometimes Packing if QC is skipped) marks when the order is ready to ship. The shipping label timestamp reveals batch formation delay.  
- **Metrics:**  
  - *Batch waiting time*: time from orderready timestamp (e.g., QC COMPLETE) to “Shipping Label Gen. – COMPLETE”.  
  - *Batch size and composition*: number of orders per batch, distribution of destinations.  
  - *Batch formation lag*: time between the first ready order in a batch and the last one that triggers label generation.  
  - *Impact on endtoend time*: total cycle time split into prebatch and batching portions.
- **Differentiation:** If an order waits while its batch mates are still being processed, the delay is betweeninstance. Withininstance delay would show a gap between QC complete and shipping label start even when the batch is already full; this could be a system scheduling delay rather than waiting for other orders.

**Priority Handling (Express orders)**
- **Detection:** Express orders (`Order Type = Express`) may interrupt standard orders—the log will show an express order starting an activity on a resource while a standard order’s activity is paused (we need start/complete timestamp pairs and resource IDs). We reconstruct resource usage timelines: if a standard order’s activity ends later than expected because the same resource handled an express order in the middle, we attribute the extra interval to preemption. We can also flag cases where an express order starts on a resource that was occupied by a standard order at that moment (the standard order’s activity would be in an “interrupted” state if the log captured suspensions; if not, we infer preemption from overlapping resource assignments).  
- **Metrics:**  
  - *Preemption frequency*: number of times a standard order’s activity is paused/elongated by an express order.  
  - *Delay imposed on standard orders*: additional waiting time beyond pure service time due to express interruptions.  
  - *Express order waiting time reduction*: compare waiting times for express vs. standard orders at shared resources to quantify the speedup.  
  - *Resource reallocation rate*: share of time a resource spends on express vs. standard work.
- **Differentiation:** Withininstance delay is the normal processing time; betweeninstance is the extra time a standard order spends waiting while the resource serves express orders. By comparing standard order activity durations with and without express interruptions (using control charts), we isolate interinstance impact.

**Hazardous Material Regulatory Limit (10 in Packing + QC simultaneously)**

## 2. Analyzing Constraint Interactions

The constraints do not operate in isolation—they interact in ways that can amplify delays. Understanding these interactions is critical for tuning optimization strategies.

**Priority + ColdPacking:** An express order that also requires coldpacking immediately jumps to the front of the coldstation queue. This pushes back standard cold orders, extending their wait. During peak season, the coldpacking queue may already be long; express preemptions can cascade, causing standard cold orders to miss shipping batch windows and adding variability to batch formation.

**Batching + Hazardous Material Limit:** If several hazardous orders are destined for the same region, they tend to be grouped in the same shipping batch. While they wait for the batch to fill, some may be blocked from entering Packing/QC because the facilitywide hazardous count is already 10. This doubleblocking (batch delay + regulatory delay) can drastically increase total flow time. Worse, the hazardous batch itself may be held up by a single slow order, while other nonhazardous batches for the same region move ahead—wasting transportation consolidation opportunities.

**Priority + Hazardous Limit:** Express hazardous orders are still subject to the regulatory limit. If 10 hazardous orders are already in Packing/QC, even an express order must wait. This erodes the effectiveness of the priority policy for hazardous express orders and can cause them to miss delivery promises. Conversely, if we give express hazardous orders absolute priority and exceed the limit, compliance is violated.

**ColdPacking + Hazardous Limit:** Coldpacking stations may be a subset of packing stations. A hazardous order that needs coldpacking competes for both the cold station and the global hazardous slot. If the cold stations are already fully occupied with nonhazardous orders, a hazardous cold order may be forced to wait even though the hazardous limit is not reached, and vice versa.

**Batching + Priority:** Express orders are often shipped individually or in very small, timesensitive batches. Their presence can disrupt the formation of larger standard batches if the system holds standard orders waiting for a batch that never fills because express orders are removed from the pool. Alternatively, if express orders are forced into standard batches, they may incur batching delay, undermining their express status.

A process mining "spaghetti" analysis that overlays all constraints on a timeline reveals these interactions quantitatively. For example, we can create a combined resourceconstraint heatmap showing the state of cold stations, hazardous counter, and batch queues simultaneously. This insight helps us avoid fixes that improve one constraint at the expense of another (e.g., strictly prioritizing express orders could starve standard orders and block batching, ultimately lowering overall throughput).

## 3. Developing Constraint-Aware Optimization Strategies

### Strategy 1: Dynamic ColdPacking Capacity Partitioning with Predictive Priority Gating

*Primary constraints addressed:* Shared coldpacking station contention, priority handling, and interactions with hazardous material limits.

**Proposed changes:** Instead of treating all 5 coldpacking stations as a single firstinfirstout queue with express preemption, we implement a virtual partitioning:
- Reserve 1 station exclusively for express orders (or for express + hazardous-combined) during predicted peak express arrival windows.
- The remaining 4 stations serve standard orders, but can “borrow” back from the express station if the express queue is empty and standard queue exceeds a threshold.
- Use a hazardousaware admission rule: a hazardous order requiring coldpacking is allowed to use any station, but the coldpacking step is only started if the global hazardous counter is <10. If approaching the limit, the order is set to a “prepack staging” state where it holds its position in the cold queue without consuming a hazardous slot.

**Data/analysis used:**
- Historical event log to build a timedependent demand forecast for coldpacking orders by type (express vs. standard, hazardous vs. nonhazardous) per hour.
- Queuing analysis to determine optimal partition (e.g., 1 fixed express station reduces standard waiting time variance while maintaining express SLA).
- Hazardous counter trace to set dynamic threshold warnings (e.g., start refusing hazardous entry when counter reaches 8).

**Expected outcomes:**
- Express cold orders meet their delivery targets by avoiding long queue waits.
- Standard cold orders suffer less unpredictability because the express station absorbs preemptions.
- Hazardous limit compliance is maintained without idle stations: the prepack staging ensures a hazardous order is first in line when a slot opens.
- Overall coldpacking throughput increases because stations are continuously busy with appropriate work.

### Strategy 2: Smart Dynamic Batch Formation with HazardousAware Grouping

*Primary constraints addressed:* Shipping batching, hazardous material limit, and interaction with priority handling.

**Proposed changes:**
- Replace fixed size/time batch triggers with a costbased dynamic model. The decision to close a batch and generate labels considers:
  - The total estimated shipping cost saving from consolidating more orders (diminishing returns based on historical volumetocost curves).
  - The waiting time penalty already incurred by the orders in the batch, weighted by order priority (express orders incur higher penalty per hour).
  - The hazardous material status: to avoid hitting the regulatory limit, hazardous orders destined for the same region are not forced to wait for a full batch if the hazardous counter is high. Instead, we create a separate “hazardousfirst” batch that can ship with fewer orders, or merge them with nonhazardous orders earlier.
- Implement a “timeoutandrelease” rule: if any order in the batch has waited beyond a configurable threshold (shorter for express, longer for standard), the batch is closed regardless of size.

**Data/analysis used:**
- Historical shipping cost data per region and batch size to model the consolidation benefit function.
- Logderived distributions of order arrival intertimes, readytimes, and destination mix to simulate batch formation under different policies.
- Hazardous limit impact analysis: identify periods where hazardous orders accumulated in the Packing/QC queue; use those patterns to set dynamic hazardous batch size limits.

**Expected outcomes:**
- Reduced average batch waiting time, especially for timesensitive orders, without disproportionate increase in shipping costs.
- Hazardous orders move through Packing/QC more smoothly because they are not artificially delayed waiting for batch mates while the regulatory limit is available.
- Overall cycle time drops and deliverytime variance shrinks, directly improving ontime performance.

### Strategy 3: Global HazardousAware Sequencer and Decoupled QC for NonHazardous Orders

*Primary constraints addressed:* Hazardous material regulatory limit, interaction with coldpacking and batching.

**Proposed changes:**
- Introduce a central “Hazardous Flow Controller” that governs admission of hazardous orders into Packing and Quality Check. Before a hazardous order starts Packing, it must obtain a token from a global pool of 10. The token is released only when the order completes both Packing and QC (i.e., the entire hazardous window).
- To minimize idle capacity, nonhazardous orders can proceed through Packing and QC without token restriction. Where possible, we decouple Quality Check for nonhazardous orders into a separate, parallel lane so that hazardous orders in QC do not block nonhazardous flow in Packing.
- For hazardous cold orders, integrate the token request with the coldpacking queue: they only join the cold station queue after securing a token, avoiding occupation of a cold station while waiting for a hazardous slot.

**Data/analysis used:**
- Process discovery and simulation to measure the proportion of time the hazardous limit acts as a bottleneck vs. resource bottlenecks.
- Queue network analysis to size the separate QC lane capacity needed to prevent spillover delays.
- Event log replay to verify that hazardous orders currently spend long periods in idle “waiting for token” state.

**Expected outcomes:**
- Hazardous limit compliance is enforced systematically, eliminating manual oversight.
- Throughput of nonhazardous orders increases because they are not throttled by a full hazardous counter blocking shared Packing/QC resources.
- Hazardous orders experience shorter tokenwaiting times because the controller can prioritize them based on attributes (express, coldrequired) while maintaining the 10order cap.
- Combined with Strategy 1 and 2, the whole system becomes more predictable and scalable during seasonal peaks.

## 4. Simulation and Validation

Before deploying any changes, we build a discreteevent simulation model that mirrors the real fulfillment center. Process mining provides the foundation:

**Model components:**
- **Order arrival:** timevarying Poisson processes fitted from the event log, with attributes (express/standard, cold, hazardous, region) sampled from empirical distributions.
- **Activity durations:** probability distributions per activity conditioned on resource and order type, derived from historical durations (e.g., lognormal for picking).
- **Resource pools:** Staff pickers, QC staff, standard packing stations (unlimited in number), and the 5 coldpacking stations with their specific availability.
- **Constraints engine:**
  - *Coldpacking*: orders requiring cold packing route to the 5station pool; priority rules (FIFO, preemptive priority) are coded.
  - *Batching*: specific batch formation logic that groups orders by region after they are ready to ship. Rules for triggering label generation are parameterized (size, timeout, or the dynamic model from Strategy 2).
  - *Preemption*: express orders can interrupt standard orders at any resource; we model the suspension and resumption with correct remaining time.
  - *Hazardous limit:* a global counter that increments when a hazardous order starts Packing and decrements when it finishes QC. If the counter is 10, new hazardous orders cannot start Packing until a slot frees.

**Validation:**
- Run the simulation with current (asis) parameters and compare KPIs (waiting times, throughput, order cycle times, batch sizes, hazardous queue length) against the historical log. Finetune until the simulation reproduces observed performance within acceptable error margins.

**Experimentation:**
- Implement each proposed strategy (and combinations) in the model, then stresstest with historical peakseason load and projected order growth.
- Key metrics to watch: mean and 95th percentile of total order cycle time, coldpacking queue length, batch waiting time, hazardous token waiting time, utilization of coldpacking stations, and ontime delivery rate for express vs. standard.
- Sensitivity analysis on parameter choices (e.g., partition ratio, batch timeout thresholds, token release policy) to identify robust settings.

Focus on ensuring the simulation faithfully captures the interinstance locking (e.g., a hazardous order holding a cold station while waiting for a batch to fill could block another cold order even if the limit is not reached—the model must reflect such deadlocks). Only with this fidelity can we trust the simulation results to recommend operational changes.

## 5. Monitoring Post-Implementation

After changes go live, continuous monitoring via process mining dashboards ensures that the instancespanning constraints are being managed effectively and that the intended improvements materialize.

**Key metrics and dashboards:**

1. **Shared resource contention dashboard**  
   - Realtime queue lengths and waiting times for coldpacking stations.  
   - Trend chart comparing before/after deploying dynamic partition.  
   - Alert when coldpacking queue exceeds a threshold or express wait time degrades.

2. **Batching performance monitor**  
   - Average batch formation time (time from first ready to label generation) per region.  
   - Distribution of batch sizes; percentage of batches closed by timeout vs. size.  
   - Endshipment cost per order (tracked against baseline) to ensure shipping efficiency is not sacrificed.

3. **Hazardous material compliance and flow**  
   - Continuous count of hazardous orders in Packing + QC, with a compliance gauge (must never exceed 10).  
   - Time spent awaiting hazardous token for orders that are ready for Packing.  
   - Throughput rate of hazardous vs. nonhazardous orders, showing that nonhazardous flow improved.

4. **Priority handling effectiveness**  
   - Number of preemptions per day, and delay added to standard orders.  
   - Ontime delivery rate for express orders, broken down by whether they required cold packing or were hazardous.  
   - Scatterplot of express order waiting time at each constrained resource.

5. **Overall process health**  
   - Endtoend cycle time distribution, split by order type and region.  
   - A performance spectrum that highlights remaining bottlenecks—if a constraint simply moved, the dashboard reveals the new pain point.

**Instancespanning specific tracking:**
- For shared resources, we monitor the “resource contention ratio” (time orders spend waiting for a resource vs. being served) and set targets to keep it below historical peaks.
- For batching, we track the wavefront of batches: the time lag between when a batch could theoretically ship (all orders ready) and when it actually ships; reduction means better synchronization.
- For the hazardous limit, a “hazardous slot utilization” metric shows what fraction of the 10slot capacity is genuinely blocking orders, helping to decide if the limit itself should be renegotiated with regulators if permanently constraining.

The dashboards are configured to run on the live event stream, enabling nearrealtime detection of constraint violations or regressions. Periodic process mining conformance checks verify that the new rules (e.g., token admission, priority gating) are being followed accurately. This closedloop approach ensures that the optimization strategies remain effective as order volumes and patterns evolve.
