#### **Initiation**
*   **Start Event:** The process triggers when a **Customer Request** is received.

#### **Routing Logic (The First Decision)**
*   **Gateway (XOR): Check Request Type:**
    *   **Meaning:** An **Exclusive OR (XOR)** gateway means the process can take *only one* of the outgoing paths. The system must categorize the request immediately.
    *   **Path 1 (Standard):** The request fits predefined criteria (e.g., off-the-shelf product, standard service).
    *   **Path 2 (Custom):** The request requires modifications or non-standard specifications.

#### **Path 1: Standard Request Flow**
1.  **Task B1 ("Perform Standard Validation"):** Automated checks to ensure the customer and order details meet basic requirements (e.g., valid customer ID, correct formatting).
2.  **Gateway (AND): Run Parallel Checks:**
    *   **Meaning:** An **AND** gateway splits the flow into multiple tasks that happen **simultaneously**. The process does not wait for one to finish before starting the other.
3.  **Task C1 ("Credit Check") & Task C2 ("Inventory Check"):**
    *   These run in parallel. One checks if the customer has credit limits available; the other checks if stock is available.
4.  **Join (Implicit in "All Parallel Checks Completed"):** The process waits until **both** C1 and C2 are finished before proceeding.
5.  **Task D ("Calculate Delivery Date"):** Based on the inventory and credit status, the system determines when the customer can receive the goods.

#### **Path 2: Custom Request Flow**

2.  **Gateway (XOR): Is Customization Feasible?**

### 3. Key Logic Clarifications

#### **Gateway Meanings**
*   **XOR Gateway (Exclusive Choice):** Used for decision points where only one path can be taken (e.g., Standard vs. Custom, Yes vs. No).
*   **AND Gateway (Parallel Split/Join):** Used specifically in the Standard path. It ensures that both Credit and Inventory are checked independently and simultaneously to save time, but the process cannot move forward until *both* checks are complete.

#### **Standard vs. Custom Differences**
| Feature | Standard Path | Custom Path |
| :--- | :--- | :--- |
| **Complexity** | Low (Automated) | High (Analytical) |
| **Key Tasks** | Validation, Credit, Inventory, Delivery Date | Feasibility Analysis, Quotation |
| **Parallelism** | Yes (Credit & Inventory run together) | No (Sequential analysis) |
| **Failure Point** | Rare (Only if Credit/Inventory fails, though not explicitly modeled as an end event) | High (Feasibility can lead to immediate Rejection) |
| **Rework Target** | Loops to Delivery Date Calculation | Loops to Quotation Preparation |

#### **Approval & Rework Triggers**
*   **Approval Trigger:** Determined at the "Is Approval Needed?" gateway. This is likely based on thresholds (e.g., Order Value > $10,000 or Custom Discount > 15%).
*   **Rework Trigger:** Occurs only if **Manager Approval is Denied**.
*   **Loop Behavior:** The loop is **conditional on the original request type**. The system must retain the "context" of whether the order was Standard or Custom to know whether to loop back to Task D or Task E1.

### 4. Summary of Process Health
This process is designed for **efficiency and risk management**:
1.  **Efficiency:** By using the **AND Gateway** for Standard orders, Credit and Inventory checks happen in parallel, reducing lead time.
2.  **Risk Control:** The **Approval Gateway** ensures high-risk orders are vetted by management.
3.  **Flexibility:** The **Loop** mechanism allows for negotiation and rework rather than immediate rejection, increasing the chance of closing the sale.
4.  **Exit Strategy:** The **Custom Feasibility Check** provides an early exit for impossible requests, saving resources on validation and invoicing.
