## 3) Brief justification for the naming choices

### `Verify Documents`
- Renamed to distinguish **initial verification** from **re-verification**.
- Key clues:
  - It appears **before** and **after** a corrective event.
  - In **C001**, the second check follows `Submit Additional` / missing ID upload.
  - In **C003**, the second check follows `Adjust Account`, so it is verification after correction.
  - In **C003**, the resource/context also shows this is **in-person branch verification**.

### `Call Customer`
- Renamed by the **purpose of the call**.
- Key clues:
  - **C001:** call is to request missing documents.
  - **C002 first:** call happens before `Adjust Account`, so it is a clarification call.
  - **C002 second:** call happens after `Adjust Account`, so it is a confirmation/follow-up call.
  - **C003:** call is to have the customer return to the branch.

### `Adjust Account`
- Renamed based on why the adjustment happened.
- Key clues:
  - **C002:** adjustment follows a discrepancy clarification call.
  - **C003:** adjustment is described as an internal correction due to missing information.

### `Initiate Request`
- Renamed by intake channel.
- Key clues:
  - **C001/C002:** submitted via `WebPortal`  online request submission.
  - **C003:** handled by `BranchReception`  branch walk-in registration.

### `Submit Additional`
- Renamed to explicitly describe what was submitted.
- Key clue:
  - Context says customer uploaded a requested ID scan.

### `Approve Request`
- Standardized as `Supervisor Final Approval`.
- Key clue:
  - Same role, same process position, same meaning in all cases.
