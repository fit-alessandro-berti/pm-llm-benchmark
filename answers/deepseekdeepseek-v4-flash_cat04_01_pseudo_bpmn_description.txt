## Detailed Explanation of the Pseudo-BPMN Process

The provided pseudo-BPMN diagram models a **customer request handling process** for a company that offers both standard and custom products or services. The process guides a request from initial receipt through validation, feasibility analysis, parallel checks, approval, invoicing, and final confirmation. Below is a stepbystep breakdown, including the role of each gateway, path differentiation, and loop mechanisms.

---

### 1. Start Event and Initial Task

- **Start Event** ¢ The process begins when a customer request is received.
- **Task A: ōReceive Customer Requestö** ¢ The request is logged and basic information (e.g., customer details, product/service requested) is captured. This task is the single entry point for all subsequent steps.

---

### 2. First XOR Gateway ¢ Request Type

- **Gateway: ōCheck Request Typeö** ¢ An exclusive (XOR) gateway that splits the flow into two mutually exclusive paths based on the nature of the request.
  - **If Type = Standard**  the request follows a **standard handling** path.
  - **If Type = Custom**  the request follows a **custom handling** path.

#### 2a. Standard Path

- **Task B1: ōPerform Standard Validationö** ¢ Basic checks are executed to ensure the request meets predefined criteria (e.g., completeness, product availability, pricing rules). This is a single, sequential step before parallel work.
- **AND Gateway: ōRun Parallel Checksö** ¢ An **AND split** initiates two activities that must be carried out simultaneously:
  - **Task C1: ōCredit Checkö** ¢ Verifies the customerÆs creditworthiness (e.g., payment history, credit limit).
  - **Task C2: ōInventory Checkö** ¢ Confirms that required standard items are in stock or can be sourced.
- **Join (AND) ¢ All Parallel Checks Completed** ¢ After both Credit Check and Inventory Check have finished, the flow merges.
- **Task D: ōCalculate Delivery Dateö** ¢ Based on the results of the parallel checks (e.g., inventory availability, credit approval), a realistic delivery date is computed.

#### 2b. Custom Path

- **Task B2: ōPerform Custom Feasibility Analysisö** ¢ A dedicated assessment is made to determine whether the requested customization is technically and commercially viable (e.g., engineering review, cost estimation).
- **XOR Gateway: ōIs Customization Feasible?ö** ¢ An exclusive decision point:
  - **If Yes**  **Task E1: ōPrepare Custom Quotationö** ¢ A detailed quote with pricing, delivery terms, and specifications is generated.
  - **If No**  **Task E2: ōSend Rejection Noticeö** ¢ The customer is informed that the customization cannot be fulfilled, and the process reaches an **End Event** (no further steps for that request).

---

### 3. PostStandard/Custom Paths ¢ Convergence

After either the Standard path (Task D) or the Custom path (Task E1) completes, the flow converges into a common step:

- **XOR Gateway: ōIs Approval Needed?ö** ¢ This gateway evaluates whether the request requires managerial approval. The criterion could be based on factors like total value, risk level, or company policy.
  - **If Yes**  proceed to **Task F: ōObtain Manager Approvalö**.
  - **If No**  skip directly to **Task G: ōGenerate Final Invoiceö**.

---

### 4. Approval Subprocess

- **Task F: ōObtain Manager Approvalö** ¢ The manager reviews the request (including validation results, quotation, delivery date) and makes a decision.
- **XOR Gateway: ōIs Approval Granted?ö** ¢ An exclusive split based on the managerÆs decision:
  - **If Yes**  continue to **Task G: ōGenerate Final Invoiceö**.
  - **If No**  trigger a **rework loop**:
    - **Task H: ōReevaluate Conditionsö** ¢ The request is sent back for adjustments. Depending on the originating path:
      - **For Standard Path**  the loop returns to **Task D: ōCalculate Delivery Dateö** (so conditions like inventory or credit are reconsidered, potentially recalculating the delivery date).
      - **For Custom Path**  the loop returns to **Task E1: ōPrepare Custom Quotationö** (so the quotation is revised, e.g., different pricing, altered scope, or new feasibility options).
    - This rework loop can repeat until approval is granted or the request is abandoned (not shown, but could be modelled with a maximum retry).

---

### 5. Final Steps

- **Task G: ōGenerate Final Invoiceö** ¢ Once either no approval is needed or approval is granted, the official invoice is created with final amounts, taxes, and payment terms.
- **Task I: ōSend Confirmation to Customerö** ¢ The invoice and a confirmation of the order (and delivery date) are dispatched to the customer.
- **End Event** ¢ The process terminates successfully.

## Meaning of the Gateways

| Gateway Type | Symbol | Purpose in this Process |
|--------------|--------|-------------------------|
| **XOR (Exclusive)** | Diamond with æXÆ | Splits the flow into exactly one outgoing branch based on a condition. Used for: request type check, feasibility decision, approval needed check, approval granted check. |
| **AND (Parallel)** | Diamond with æ+Æ | Splits the flow into multiple concurrent activities that must all be completed before proceeding. Used for running Credit Check and Inventory Check in parallel. |
| **Join (AND)** | Diamond with æ+Æ (merging) | Synchronizes parallel branches ¢ waits for all incoming flows to arrive before continuing. Used after parallel checks. |
| **Loopback** | Implicit (dashed arrow) | Not a BPMN gateway per se, but a reentrant flow from Task H back to Task D or E1, effectively creating a rework cycle. |

## How Tasks Differ Based on Standard vs. Custom

| Aspect | Standard Request | Custom Request |
|--------|------------------|----------------|
| **Initial validation** | Simple, rulebased (Task B1) | Complex feasibility study (Task B2) |
| **Parallel activities** | Two parallel tasks (credit + inventory) | No parallel tasks (single sequential analysis) |
| **Outcome after checks** | Delivery date is calculated (Task D) | If feasible  quotation prepared (Task E1); if not  rejection (Task E2) and process ends |
| **Rework loop start** | Recalculate delivery date (Task D) | Reprepare quotation (Task E1) |
| **Approval necessity** | May be needed based on value/risk (same gateway for both paths) | Same ¢ approval gateway applies equally after both paths converge |

## Conditions for Approvals and Rework Steps

- **Approval is triggered** when the **ōIs Approval Needed?ö** gateway evaluates to **Yes**. The exact criterion (e.g., order value above $10k, new customer, high risk) is not defined in the pseudoBPMN but would be decided by business rules.
- **Rework (loop)** is triggered **only** when approval is needed **and** the manager **denies** the request. The loop sends the process back to the last major decision point in the respective path:
  - **Standard path**: back to **Calculate Delivery Date** ¢ implying that conditions such as inventory availability, credit status, or delivery options may be adjusted (e.g., alternative shipping, partial shipment, or renegotiating credit terms).
  - **Custom path**: back to **Prepare Custom Quotation** ¢ implying that the quotation must be revised (e.g., different pricing, reduced scope, or alternative customization) to address the managerÆs concerns.
- **No loop** occurs if approval is not needed, or if approval is granted. The process then proceeds directly to invoicing and confirmation.

## Summary

The process efficiently separates standard and custom requests, uses parallel work for faster standard handling, includes a feasibility gate for custom requests (with an early exit for infeasible ones), and incorporates a conditional approval step with a rework loop to allow iterative refinement. The final confirmation ensures that the customer receives both the invoice and a clear order status, closing the loop in a structured manner.
