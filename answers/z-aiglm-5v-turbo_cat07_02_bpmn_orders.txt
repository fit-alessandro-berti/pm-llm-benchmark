This is a **BPMN (Business Process Model and Notation) diagram** depicting an end-to-end order-to-cash process spanning **four functional departments**: Sales, Production, Procurement, and Warehouse. Here's a detailed breakdown:

## **1. Sales Lane (Top)**
- **Trigger**: Process starts every *Monday morning*
- **Core Activity**: "Check customer's order" (subprocess containing):
  - Verify order completeness
  - Check if customer exists in database
  - **Decision branch**: If new customer  Add to database; If existing  proceed
- **Output**: Send order details to downstream departments
- **Final Step**: Wait for "Product sent" confirmation  Draft invoice  End

## **2. Production Lane**
- **Input**: Receives order details from Sales
- **Flow**: 
  - Plan production
  - **Decision gateway**: Check if materials are faulty
  - Manufacture product
- **Output**: Signal "Product complete" to Warehouse

## **3. Procurement Lane**
- **Input**: Receives order details (parallel to Production)
- **Key Subprocess**: "Material Compensation Subprocess"
  - Order materials
  - Handle returns (with timer/wait state)
- **Quality Gate**: Check materials
  - **Exception path**: If issues detected  Trigger order cancellation

## **4. Warehouse Lane**
- **Trigger**: Receives "Product complete" signal from Production
- **Activity**: Send product to customer
- **Output**: Emit "Product sent" signal back to Sales (completing the loop)

---

### **Key Characteristics:**
- **Message Flows**: Dashed lines show asynchronous communication between pools (e.g., Order details, Product sent, Product complete)
- **Parallel Processing**: Production and Procurement work simultaneously after receiving order details
- **Error Handling**: Procurement has compensation logic for material returns/cancellations
- **Data Store**: Customer database icon in Sales lane shows persistent storage

The process represents a typical **manufacturing order fulfillment workflow** with cross-functional handoffs and exception handling mechanisms.
