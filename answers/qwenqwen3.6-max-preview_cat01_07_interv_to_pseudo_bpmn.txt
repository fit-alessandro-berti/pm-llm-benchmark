# BPMN-Like Textual Process Model: Invoice Approval & Payment Scheduling

## 1. Participants (BPMN Lanes)
| Role/Participant | BPMN Lane | Responsibility |
|------------------|-----------|----------------|
| **Accounts Payable (AP) Clerk** | `AP_Clerk` | Initial validation, system logging, routing, system matching, payment scheduling |
| **Purchasing Officer** | `Purchasing` | Receipt confirmation, line-item/price verification, supplier liaison for discrepancies |
| **Department Manager** | `Dept_Manager` | Final budgetary & compliance approval |
| **AP Supervisor** | `AP_Supervisor` | Exception resolution, cross-department coordination, retrospective PO creation |
| **Supplier Relationship Manager** | `SRM_Support` | Ad-hoc contract verification & supplier communication (support role, no system approvals) |
| **Accounting System** | `System_Auto` | Automated logging, matching validation, approval notifications |
| **Supplier** | `External_Supplier` | Invoice submission, corrections, PO provision |

---

## 2. Data Objects & Artifacts
- `Invoice` (Email/PDF)
- `Purchase Order (PO)`
- `Receipt Confirmation`
- `Match Report` (System-generated)
- `Approval Notification` (System email)
- `Corrected Invoice` (Supplier-provided)
- `Retrospective PO` (Exception artifact)
- `Payment Schedule Entry`

---

## 3. Main Process Flow (Sequence & Gateways)
*(Note: All gateways are Exclusive (XOR) unless stated otherwise. Flow proceeds top-to-bottom.)*

1. `[Start Event]` Invoice received via email
2. `[User Task: AP_Clerk]` Check invoice for valid PO number
3. `[Exclusive Gateway]` **Valid PO Present?**
   - **No**  `[Send Task: AP_Clerk]` Return to Supplier requesting PO/details  `[Intermediate Event]` Await response  Loop to Step 2
   - **Yes**  `[User Task: AP_Clerk]` Log invoice into Accounting System
4. `[Send Task: AP_Clerk]` Forward invoice to Purchasing for receipt verification
5. `[User Task: Purchasing]` Confirm goods/services received & compare invoice vs. PO (quantities, prices, delivery charges)
6. `[Exclusive Gateway]` **Purchasing Verification Successful?**
   - **No**  Route to *Exception Path A* (Purchasing Discrepancy)
   - **Yes**  `[Send Task: Purchasing]` Send confirmation email/record to AP
7. `[User Task: AP_Clerk]` Perform system line-item match (Invoice vs. PO)
8. `[Exclusive Gateway]` **System Match Successful?**
   - **No**  Route to *Exception Path B* (AP Matching Discrepancy)
   - **Yes**  `[Service Task: System_Auto]` Trigger approval notification to Dept Manager
9. `[User Task: Dept_Manager]` Review invoice, PO, match data, & purchasing confirmation
10. `[Exclusive Gateway]` **Manager Approval Granted?**
    - **No**  Route to *Exception Path C* (Manager Rejection/Hold)
    - **Yes**  `[User Task: AP_Clerk]` Schedule invoice for payment run
11. `[End Event]` Invoice added to payment cycle

---

## 4. Exception Handling & Rework Loops

###  Exception Path A: Purchasing Discrepancy (Step 6  No)
- `[User Task: Purchasing]` Identify mismatch (extra units, pricing, charges)
- `[Exclusive Gateway]` **Resolution Approach?**
  - Contact Supplier directly  Await `Corrected Invoice`  Loop to Step 5
  - Request AP to hold invoice  `[User Task: AP_Clerk]` Place on hold  Await resolution  Loop to Step 5
- *Support Trigger:* May loop in `SRM_Support` to verify contract terms & communicate correct pricing to AP/Purchasing.

###  Exception Path B: AP Matching Discrepancy (Step 8  No)
- `[User Task: AP_Clerk]` Email Purchasing for clarification
- `[Exclusive Gateway]` **Issue Resolved?**
  - **Yes**  Update records  Loop to Step 7
  - **No**  `[User Task: AP_Clerk]` Escalate to `AP_Supervisor`
- `[User Task: AP_Supervisor]` Coordinate between departments/suppliers, resolve discrepancy  Return to Step 7

###  Exception Path C: Manager Rejection/Hold (Step 10  No)
- `[User Task: Dept_Manager]` Reject or place on hold (e.g., budget variance, unexpected items)
- `[Send Task: System_Auto]` Notify AP of rejection/hold
- `[User Task: AP_Clerk / AP_Supervisor]` Investigate, request corrections, or adjust coding
- `[Exclusive Gateway]` **Correctable?**
  - **Yes**  Update documentation  Loop to Step 7 (or Step 2 if PO/data fundamentally changed)
  - **No**  `[End Event]` Invoice formally rejected & archived

###  Special Variation: No PO / Informal Services
- Triggered when invoice lacks PO and cannot be returned to supplier.
- `[User Task: AP_Supervisor]` Collaborate with Department Head to either:
  - Create `Retrospective PO`  Return to Step 3
  - Formally reject invoice  `[End Event]` Rejection

## 6. BPMN Diagram Conversion Guide
| BPMN Element | Mapping Instruction |
|--------------|---------------------|
| **Pools** | `Internal Organization` (contains all lanes) | `External Supplier` (separate pool, message flows only) |
| **Lanes** | `AP_Clerk`, `Purchasing`, `Dept_Manager`, `AP_Supervisor`, `System_Auto` |
| **Start Event** | Message Event (Invoice Email) |
| **End Events** | Message/None Event (Payment Scheduled) & Termination Event (Invoice Rejected) |
| **Gateways** | All decision points are **Exclusive (XOR)**. Label outgoing flows with conditions (e.g., "PO Missing", "Match OK", "Hold/Reject"). |
| **Tasks** | `User Task` (manual steps), `Service Task` (system auto-email), `Send/Receive Task` (external comms), `Manual Task` (filing/archiving if added) |
| **Data Objects** | Attach `Invoice`, `PO`, `Match Report`, `Approval Notification` to relevant tasks with data association arrows. |
| **Exception Flows** | Model as explicit XOR branches leading to rework loops. Use **Boundary Intermediate Events** (e.g., Error/Escalation) on Tasks 5, 7, 9 if your BPMN tool supports attached exception handlers. |
| **Message Flows** | Use dashed arrows for all Supplier communications and SRM support interactions crossing pool/lane boundaries. |
