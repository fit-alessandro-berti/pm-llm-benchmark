## BPMN-like Textual Process Model (Accounts Payable Invoice Approval)

### Participants / Roles
- **Supplier** (sends invoices)
- **Accounts Payable Clerk (Mary)** (receives, logs, matches, forwards/escalates)
- **Purchasing Officer (Dan)** (validates invoice vs PO, holds invoices if missing/incorrect PO)
- **Department Manager (Carlos)** (approves/rejects in system; may request further investigation)
- **Accounts Payable Supervisor (Rita)** (handles escalated exceptions; ensures closure)
- **Supplier Relationship Manager (Karen)** (indirect support; contacts suppliers / verifies contract pricing)
- **Accounting System / Workflow System** (logging, matching, automated email notifications, approvals)

### Data / Artifacts
- **Invoice** (received via email)
- **Purchase Order (PO) reference / PO** (must exist to proceed)
- **PO line items** (quantities, prices, delivery charges)
- **Invoice line items** (quantities, prices, items/services)
- **Manager approval notification / sign-off request** (automatic email from system)
- **Correction / revised invoice** (requested from supplier when needed)
- **Retrospective PO** (created for services contracted informally, as an exception outcome)
- **Payment schedule / payment run** (next payment cycle preparation)

## Main Process (Typical / “happy path”)

**Start Event:** *Invoice received (email) by Accounts Payable*

1. **Task AP1 – Check PO reference on invoice**
   - **Actor:** Mary (Accounts Payable Clerk)
   - **Input:** Invoice
   - **Rule:** Check whether invoice contains a **valid PO number**
   - **Output:** Valid/invalid PO status

2. **Gateway G1 – PO number present & valid?**
   - **Condition:** PO number exists and is usable in system

   **If NO (missing/invalid PO):**
   2a. **Task AP1b – Request PO / details from supplier via escalation**
       - **Actor:** Mary
       - **Action:** Send back to supplier requesting more details (per Mary)
       - **Note:** In some cases Dan also asks Mary to obtain PO info before proceeding (Interviews 1–2)
   - Then return to **Task AP1** when invoice is updated/PO becomes available.

   **If YES:**
   2b. **Task AP2 – Log invoice into accounting system**
       - **Actor:** Mary
       - **Output:** Invoice record in system tied to PO

3. **Task AP3 – Forward invoice to Purchasing for receipt/confirmation**
   - **Actor:** Mary  Dan (Purchasing Officer)
   - **Goal:** Confirm goods/services were actually received (Mary) / Dan checks against PO terms (Dan)

4. **Task PUR1 – Validate invoice vs PO in purchasing**
   - **Actor:** Dan
   - **Checks:** quantities, prices, delivery charges, line item alignment
   - **Output:** “Match OK” vs “Discrepancy”

5. **Gateway G2 – Invoice matches PO terms?**
   - **If OK:**
     5a. **Task PUR2 – Send confirmation email back to Accounts Payable**
         - **Actor:** Dan
         - **Output:** Confirmation status
     5b. **Task AP4 – Match invoice line items against PO (in system)**
         - **Actor:** Mary
         - **Output:** “Ready for manager approval” (implied)

     5c. **Task SYS1 – System sends manager sign-off request**
         - **Actor:** System
         - **Trigger:** After receipt confirmation + PO match

     5d. **Task MGR1 – Manager approves in system**
         - **Actor:** Carlos
         - **Condition:** quantities/pricing/special notes look good
         - **Output:** Approval complete

     5e. **Task AP5 – Schedule for payment / add to next payment cycle**
         - **Actor:** Mary (Accounts Payable) after approval returns
         - **Output:** Scheduled payment

   - **If NOT OK (discrepancy):**
     5f. **Task PUR3 – Handle discrepancy**
         - **Actor:** Dan
         - **Action (two mentioned approaches):**
           - Contact supplier directly **or**
           - Ask Mary to hold the invoice until a corrected version is provided
     5g. **Task AP6 – Email Purchasing team / clarify discrepancy**
         - **Actor:** Mary
         - **Action:** Email purchasing to clarify (Mary’s account)
     5h. **Gateway G3 – Discrepancy resolvable quickly?**
         - **If YES:**
           - Supplier/corrected invoice arrives  return to **Task AP4 / Task PUR1** (re-check match)
         - **If NO (cannot be resolved easily):**
           - **Task AP7 – Escalate to AP Supervisor**
             - **Actor:** Mary  Rita
             - **Action:** Escalation when mismatch cannot be resolved easily (Mary)
           - Then return to resolution path via **Task SUP4** / **Task MISC (exceptions)** (see Exception section)

**End Event:** *Invoice scheduled for payment*

## Exception / Variation Paths

### Exception 1: PO-related hold / missing PO (formal gating)
**When PO number missing/invalid:**
- Follow **G1 = PO number present? NO** loop:
  - Mary requests PO/details from supplier (and/or Dan requires PO before proceeding)
  - Once PO is obtained and invoice can be logged/linked  continue to **AP2 / PUR1**

### Exception 2: Discrepancy requires clarification or correction
**When quantities/prices/items don’t match:**
- **G2 = matches PO? NO** triggers:
  - Dan contacts supplier or requests hold
  - Mary and purchasing clarify via email
  - If corrected invoice is received  re-check match and proceed to manager approval

### Exception 3: Suspicious totals or unexpected items  manager intervention
**During manager review:**
- **Task MGR1** includes a decision:
  - **Gateway G4 – Manager finds something suspicious?**
    - **If YES:**
      - **Task MGR2 – Reject or request hold/investigation**
        - **Actor:** Carlos
        - **Action:** Ask Mary to hold and investigate (Carlos)
      - **Task AP8 – Investigate with AP/Purchasing/Supplier contacts**
        - **Actor:** Mary + Dan; may involve Karen (below)
      - Then return to **PUR1** or **AP4** depending on what is corrected
    - **If NO:**
      - proceed to approval and payment scheduling

### Exception 4: Invoices without any PO (informally contracted services)
**When invoice doesn’t match any PO:**
- **Task SUPV1 – AP Supervisor handles exception**
  - **Actor:** Rita
  - **Cases:**
    - Invoice doesn’t match any PO because it may be for services contracted informally
  - **Gateway G5 – Create retrospective PO or reject?**
    - **If retrospective PO created:**
      - **Task SUPV2 – Create retrospective PO**
        - **Actor:** Rita with department heads (per Rita)
      - Then route back to **Purchasing validation / manager approval** (re-enter normal flow)
    - **If should be rejected:**
      - **Task SUPV3 – Reject invoice**
        - **Actor:** Rita + department heads involvement (implied)
      - **End (exception) / No payment scheduled**

### Exception 5: Supplier contract verification for pricing discrepancies (indirect loop)
**When pricing discrepancies occur:**
- Karen is looped in (indirectly) to resolve supplier-side correctness:
  - **Task K1 – Karen contacts supplier / verifies contract pricing**
    - **Actor:** Karen
    - **Output:** Correct price terms or supplier provides revised invoice/PO info
  - Then continue to **PUR1** / **AP4** for re-matching.

## Overall Sequence Flow (Consolidated)
1. **Start:** Invoice received (email)   
2. **AP1:** Check PO on invoice   
3. **G1 PO present?**  
   - If NO  request PO/details  back to **AP1**  
   - If YES   
4. **AP2:** Log invoice to accounting system   
5. **AP3:** Forward to Purchasing   
6. **PUR1:** Validate invoice vs PO   
7. **G2 Match OK?**  
   - If NO  discrepancy handling (hold/correct/escalate)  back to **PUR1/AP4** after resolution  
   - If YES   
8. **PUR2:** Send confirmation to AP   
9. **AP4:** Match invoice line items in system   
10. **SYS1:** System requests manager approval   
11. **MGR1:** Manager approves (or triggers hold/reject if suspicious)   
12. **G4 Suspicious?**  
   - If YES  manager requests investigation/hold  back to discrepancy resolution  then re-approve  
   - If NO   
13. **AP5:** Schedule payment in next payment cycle   
14. **End:** Payment scheduled

### Noted Ambiguities (explicitly uncertain due to interviews)
- **Where the “matching” step is performed exactly**: Mary describes matching after receiving purchasing confirmation; Dan also checks invoice vs PO and would hold if discrepancies exist. The model assumes **both may perform checks**, but places Dan’s validation as the key purchasing gate (**PUR1**) and Mary’s as the system-ready matching (**AP4**).
- **Escalation trigger details**: Mary mentions escalation to supervisor when discrepancy can’t be resolved easily; Carlos may also ask Mary to hold/investigate. The model treats both as leading to the exception resolution layer handled by Rita (and possibly Karen).
- **Responsibility for initiating retrospective PO**: explicitly attributed to Rita (with department heads). The re-entry point into purchasing/manager approval is assumed but not described step-by-step.
