

# Advanced Process Mining-Driven Scheduling Optimization for Precision Parts Inc.

## Executive Summary

This report presents a comprehensive, data-driven approach to diagnosing and resolving the scheduling challenges at Precision Parts Inc. By leveraging process mining on the MES event logs, we will reconstruct the actual shop floor dynamics, identify systemic pathologies, trace their root causes, and design three sophisticated scheduling strategies. These strategies will be validated through discrete-event simulation before deployment, with a continuous improvement framework ensuring sustained performance gains.

## 1. Analyzing Historical Scheduling Performance and Dynamics

### 1.1 Reconstructing Actual Job Flows

The MES event logs provide the raw material for reconstructing the actual execution reality on the shop floor. The approach proceeds in layers:

**Event Log Preparation and Case-Centric Reconstruction:**

First, we parse the event log treating each `Case ID (Job ID)` as a process instance (case). For each job, we reconstruct the complete lifecycle by ordering its events chronologically. The event types (Job Released, Queue Entry, Setup Start, Setup End, Task Start, Task End) are mapped to a unified activity lifecycle model. This yields a complete trace per job:

```
JOB-7001: Job Released  Queue Entry (Cutting)  Setup Start (CUT-01)  Setup End (CUT-01)  
Task Start (Cutting, CUT-01)  Task End (Cutting, CUT-01)  Queue Entry (Milling)  ...
```

**Process Discovery:** Using discovery algorithms (e.g., Inductive Miner, Split Miner), we generate process models that capture:
- The **actual routing variants** — which sequences of operations jobs actually followed, including rework loops, skipped steps, or deviations from the planned routing.
- **Variant frequency analysis** — identifying the most common routing patterns and rare but potentially problematic deviations.
- **Directly-Follows Graphs (DFGs)** enriched with frequency and performance overlays to visualize flow intensity and time accumulation at each transition.

**Resource-Centric Reconstruction:**

In parallel, we construct a **resource-centric view** by pivoting the log on `Resource (Machine ID)`. For each machine, we reconstruct the chronological sequence of jobs processed, capturing:
- The exact sequence: ..., JOB-6998, JOB-7001, JOB-7003, ... on CUT-01
- Idle gaps between consecutive jobs
- Setup durations between consecutive jobs
- Breakdown periods

This dual perspective (case-centric and resource-centric) is essential for understanding both the job experience and the machine utilization patterns.

### 1.2 Quantifying Specific Metrics

#### Job Flow Times, Lead Times, and Makespan Distributions

**Flow Time** for each job is computed as: `Timestamp(last Task End or Quality Inspection complete)  Timestamp(Job Released)`

We decompose flow time into:
- **Value-added time**: Sum of actual task processing durations
- **Setup time**: Sum of actual setup durations
- **Queue/wait time**: Sum of time spent in queue entries (from Queue Entry to Setup Start or Task Start)
- **Blocking time**: Time a completed task cannot release the part because the downstream buffer is full (detectable when Task End is followed by an unusually long gap before the next Queue Entry)
- **Transport time**: Gaps between Task End at one machine and Queue Entry at the next

Using performance-enhanced process models (e.g., in tools like Celonis, Disco, or ProM), we overlay these time components on the process model. Statistical distributions (not just means) are generated for each metric using histograms, box plots, and kernel density estimates, segmented by:
- Job type/routing
- Priority level
- Time period (to detect temporal trends)
- Machine assignments

**Makespan analysis** at the shop level examines the total time to complete a batch/set of jobs, useful for capacity planning.

#### Task Waiting Times (Queue Times)

For each work center, queue time is calculated as:

```
Queue Time = Timestamp(Setup Start | Task Start)  Timestamp(Queue Entry)
```

We analyze:
- **Distribution of queue times per work center** — identifying which machines consistently have the longest queues
- **Queue time as a function of time-of-day and day-of-week** — revealing temporal patterns in congestion
- **Queue length over time** — by counting the number of concurrent "Queue Entry" events without a corresponding "Setup Start" for each machine at any given moment, reconstructing the queue length time series
- **Correlation between queue length and waiting time** — to model the queuing behavior (fitting M/G/1 or G/G/c queuing models where appropriate)

#### Resource Utilization

For each machine, we construct a **timeline** categorizing every time interval into one of:
- **Productive (processing)**: Between Task Start and Task End
- **Setup**: Between Setup Start and Setup End
- **Idle**: Time the machine is available but no job is being processed or set up
- **Down (breakdown)**: Between Breakdown Start and Breakdown End (or repair completion)
- **Blocked**: Machine has completed a task but cannot release (if detectable)

Utilization metrics:
- **Overall Equipment Effectiveness (OEE)** = Availability × Performance × Quality
- **Productive utilization** = Productive time / Available time
- **Setup ratio** = Setup time / (Productive time + Setup time)
- **Idle ratio** and **breakdown ratio** similarly

For operators, we correlate `Operator ID` with task durations to assess:
- Operator-specific processing time distributions (skill/efficiency differences)
- Operator availability patterns and multi-machine assignments

#### Sequence-Dependent Setup Times

This is a critical analysis. We extract setup data by joining consecutive jobs on each machine:

For each machine $m$, we identify the ordered sequence of jobs processed. For each transition $(JOB_i \rightarrow JOB_j)$ on machine $m$, we record:
- **Setup duration**: `Timestamp(Setup End)  Timestamp(Setup Start)` for $JOB_j$
- **Attributes of $JOB_i$** (the preceding job): material type, part dimensions, tooling requirements, product family
- **Attributes of $JOB_j$** (the incoming job): same attribute set

We then build a **setup time matrix** $S(a_i, a_j)$ where $a_i$ and $a_j$ are attribute categories (e.g., product families or material types). This matrix captures the mean and variance of setup time for every pair of job categories.

**Analytical techniques:**
- **Regression modeling**: Setup duration as a function of attribute differences between consecutive jobs (e.g., material change flag, tooling change complexity score, dimensional difference)
- **Clustering**: Group jobs into "setup families" where within-family setups are short and between-family setups are long, using hierarchical clustering on the setup time matrix
- **Decision tree/random forest**: To identify which specific attribute changes (material, tool, fixture) are the primary drivers of long setups

#### Schedule Adherence and Tardiness

For each completed job:
- **Lateness** = Actual Completion Date  Due Date (positive = late, negative = early)
- **Tardiness** = max(0, Lateness) — only penalizes late jobs
- **Earliness** = max(0, Lateness) — for understanding premature completions and WIP costs

Aggregate metrics:
- **Percentage of jobs tardy**
- **Mean/median tardiness**
- **Maximum tardiness** (worst case)
- **Weighted tardiness** (weighted by priority or order value)
- **Lateness distribution** by job type, routing, priority, and time period

We also compute **milestone adherence** — at each operation step, comparing actual completion against planned/expected completion, to identify where in the routing jobs first begin falling behind schedule.

#### Impact of Disruptions

**Machine Breakdowns:**
- Frequency and duration distributions per machine (time between failures, time to repair)
- Impact tracing: For each breakdown event, identify all jobs that were:
  - Currently being processed (directly affected — processing interrupted)
  - In the queue (indirectly affected — increased wait time)
  - At upstream or downstream stations (cascade effects)
- Compare the tardiness and flow time of disruption-affected jobs versus unaffected jobs using statistical testing (e.g., Mann-Whitney U test)

**Priority Changes / Hot Jobs:**
- Track every "Priority Change" event and trace forward: what happened to the job that was preempted or delayed?
- Measure the "disruption cost" — the total additional tardiness and queue time imposed on other jobs when a hot job is injected
- Frequency of hot jobs and their correlation with periods of high overall tardiness

## 2. Diagnosing Scheduling Pathologies

### 2.1 Bottleneck Identification and Quantification

**Bottleneck Mining Approach:**

Using the resource utilization timelines constructed in Section 1, we apply multiple bottleneck detection methods:

1. **Utilization-based**: Machines with the highest productive utilization (approaching 100% when including setup) are candidate bottlenecks. However, high utilization alone is insufficient — a machine might be 95% utilized but processing jobs quickly.

2. **Queue-based**: The definitive signal is persistent, long queue times. We compute the **time-weighted average queue length** for each machine. The machine where WIP accumulates most persistently is the active bottleneck.

3. **Shifting bottleneck detection**: Bottlenecks in job shops are dynamic. We apply the **Active Period Method** — for each time window, the machine that is continuously busy (never idle) for the longest uninterrupted stretch is the active bottleneck in that period. By plotting this over time, we identify:
   - Permanent bottlenecks (consistently the constraining resource)
   - Shifting bottlenecks (the constraint moves depending on the job mix)

4. **Flow-time contribution analysis**: By decomposing each job's flow time into per-station components, we determine which stations contribute the most waiting time on average. The station with the highest average wait time contribution is the practical bottleneck.

**Quantification of bottleneck impact:**
- **Throughput correlation**: Regression of daily throughput on bottleneck machine availability/uptime
- **Starvation analysis**: Counting idle time at downstream machines that directly follows a bottleneck queue buildup
- **Sensitivity analysis**: If the bottleneck machine processed 10% faster, what would be the estimated improvement in throughput (using Little's Law: WIP = Throughput × Lead Time)?

### 2.2 Poor Task Prioritization Evidence

**Variant Analysis (On-Time vs. Late Jobs):**

We segment all completed jobs into two groups: on-time and late. Using variant analysis:
- Compare the routing variants between groups — do late jobs follow longer or more complex routes?
- Compare waiting times at each station — do late jobs experience disproportionately longer waits?
- Critically, examine **queue position data**: Were near-due-date jobs stuck behind lower-priority jobs in queues? This is detectable by correlating queue time with the difference between remaining time to due date (slack) and remaining processing time at the moment of queue entry.

**Priority Inversion Detection:**

For each queue, we reconstruct the queue state at each moment and identify instances of **priority inversion** — where a job with less slack or higher priority entered the queue before another job but was processed after it, or where FCFS ordering caused a job with an imminent due date to wait behind a job with substantial slack.

Metric: **Priority inversion rate** = (Number of queue departures violating priority order) / (Total queue departures), measured for each queue, using priority definitions such as weighted remaining slack.

### 2.3 Suboptimal Sequencing and Excessive Setup Times

Using the setup time matrix $S(a_i, a_j)$ from Section 1.2, we compute:

- **Actual total setup time** over a period for each machine
- **Optimal setup time**: Using a Travelling Salesman Problem (TSP) or similar sequencing algorithm on the same set of jobs that were processed, determine the minimum total setup time achievable through better sequencing
- **Setup time gap** = Actual  Optimal: This quantifies the wasted setup time due to suboptimal sequencing

For bottleneck machines, this gap is particularly costly because every minute of setup time on a bottleneck directly reduces throughput.

**Pattern detection**: Identify specific sequences that repeatedly cause long setups (e.g., switching from aluminum to steel on MILL-03 consistently requires 45+ minutes, yet this transition occurs 3× daily when batching could reduce it to once).

### 2.4 Downstream Starvation

By aligning the resource timelines of machines that are sequentially connected in common routings, we identify **starvation events**: periods where a downstream machine is idle while its input queue is empty AND the upstream machine has a long queue or is in an extended setup/breakdown.

**Causal chain reconstruction**: For each starvation event on machine $m_j$, trace back to identify whether the upstream machine $m_i$ was:
- Processing slowly (task duration exceeded plan)
- In a long setup
- Broken down
- Starved itself (cascading starvation)

This reveals the propagation structure of delays through the shop.

### 2.5 WIP Bullwhip Effect

By plotting WIP levels (jobs in process or queue) over time for each work center, and computing the coefficient of variation of WIP, we assess:
- **WIP variability** at each station
- **Amplification ratio**: Does WIP variability increase as we move through the routing? This would indicate a bullwhip-like effect where scheduling variability at early stages compounds downstream.
- **Batch-and-queue behavior**: Do we see cyclical WIP patterns suggesting that the dispatching rules create artificial batching?

**Correlation analysis**: Cross-correlate WIP time series between stations with appropriate time lags to identify causal propagation of congestion waves.

## 3. Root Cause Analysis of Scheduling Ineffectiveness

### 3.1 Limitations of Static Dispatching Rules

The current FCFS/EDD mix suffers from fundamental limitations in a complex job shop:

**FCFS is myopic and priority-blind**: It ignores job urgency, remaining processing time, and downstream conditions. Process mining evidence: Priority inversion rates will be high, and the correlation between queue arrival order and job urgency will be near zero.

**EDD is locally optimal but globally subcoordinated**: While EDD at each station considers due dates, it doesn't account for:
- Remaining processing time (a job with a later due date but much more remaining work may actually be more urgent — this is where the **Critical Ratio** or **Slack-based rules** outperform EDD)
- Setup time consequences (EDD may create expensive setup sequences)
- Downstream conditions (sending a job urgently to the next station where it will sit in a long queue anyway)

**Verification through process mining:**
- Compare the actual dispatching order at each machine to what various rules (SPT, EDD, CR, ATC) would have dictated. For each rule, compute the percentage of decisions that matched the actual decision. This reveals which implicit logic the current practice follows.
- Apply counterfactual analysis: For completed jobs, retrospectively apply different dispatching rules to the same arrival sequences and compare theoretical outcomes.

### 3.2 Lack of Real-Time Visibility

Even if operators wanted to make intelligent decisions, the current system provides no:
- Real-time queue length information at downstream machines
- Visibility into which machines are down or about to come up
- Awareness of incoming hot jobs that haven't yet arrived

**Process mining evidence**: We can detect "information blind spots" by identifying cases where:
- A job was dispatched to a machine that broke down shortly after (information lag on breakdown status)
- Jobs were sent to the longer of two parallel queues when a shorter alternative existed
- Urgent jobs experienced delays because their priority change wasn't communicated to upstream stations

### 3.3 Inaccurate Duration and Setup Time Estimations

By comparing planned vs. actual durations across all tasks:

```
Estimation Error = (Actual Duration  Planned Duration) / Planned Duration
```

We characterize:
- **Systematic bias**: Are planned durations consistently too optimistic or pessimistic? (Mean error  0)
- **Variance**: Even if unbiased on average, high variance means individual estimates are unreliable
- **Conditional bias**: Are estimates worse for certain job types, machines, or operators?

**Setup time estimation**: The current system likely uses a single estimated setup time per operation type (e.g., 20 min for cutting setup), ignoring sequence dependence. The actual setup time varies from 5 to 50+ minutes depending on the predecessor. This systematic error undermines any planning that relies on estimated completion times.

### 3.4 Ineffective Disruption Response

When breakdowns occur, the current approach likely involves:
- Waiting for repair (no rerouting to alternative machines)
- Manually reassigning jobs with no systematic reprioritization

**Process mining evidence**:
- After breakdown events, measure the time until production resumes (response lag)
- Track whether jobs were rerouted and how quickly
- Measure the "disruption recovery time" — how long until KPIs return to pre-disruption levels

For hot jobs: the current practice of simply boosting priority via FCFS/EDD mix likely causes a **ripple effect** where the hot job disrupts sequences at every station it visits, increasing setup times and delaying other jobs.

### 3.5 Differentiating Scheduling Logic Issues from Capacity Limitations

This is a critical distinction:

**Capacity-limited scenarios**: If the bottleneck machine is at 98% utilization even with perfect scheduling, no dispatching rule will solve the tardiness problem — additional capacity is needed.

**Scheduling-limited scenarios**: If the bottleneck machine spends 25% of its time on setups that could be reduced to 10% through better sequencing, or if 15% of its productive time processes low-priority jobs while high-priority jobs wait, then scheduling improvements can yield significant gains.

**Analytical approach:**
1. Compute the **theoretical minimum flow time** for each job (sum of processing times plus minimum possible setup times with no waiting) — this is the capacity-constrained lower bound
2. Compare to actual flow time — the gap represents scheduling-recoverable time
3. Decompose the gap: How much is due to waiting (scheduling-driven)? How much is due to excess setup time (sequencing-driven)? How much is due to breakdowns (reliability-driven)?

If average flow time is 5× the theoretical minimum, there is substantial room for scheduling improvement regardless of capacity constraints.

## 4. Developing Advanced Data-Driven Scheduling Strategies

### Strategy 1: Multi-Factor Dynamic Dispatching with Process-Mining-Calibrated Weights

#### Core Logic

Replace the current FCFS/EDD mix with a composite scoring rule that evaluates every job in a machine's queue at the moment of dispatch decision. Each job $j$ at machine $m$ receives a priority score:

$$\pi_j = w_1 \cdot U_j + w_2 \cdot S_j + w_3 \cdot D_j + w_4 \cdot P_j + w_5 \cdot C_j$$

Where:
- $U_j$ = **Urgency factor** (normalized): Based on the critical ratio $CR_j = \frac{\text{Time remaining to due date}}{\text{Estimated remaining processing time}}$. Lower CR  higher urgency.
- $S_j$ = **Setup efficiency factor** (normalized): The estimated setup time if job $j$ follows the current job on machine $m$, derived from the mined setup time matrix $S(a_{\text{current}}, a_j)$. Shorter expected setup  higher score.
- $D_j$ = **Downstream load factor** (normalized): The current queue length at the next machine in job $j$'s routing. If the next machine is heavily loaded, there's less benefit in rushing this job through — it will just wait downstream. Conversely, if the next machine is starving, prioritize feeding it.
- $P_j$ = **Priority/customer weight** (normalized): Customer importance, contractual penalty rates, order priority class.
- $C_j$ = **Contribution to bottleneck** (normalized): If this machine is a bottleneck, minimize time spent on setup (increase weight $w_2$). If not a bottleneck, prioritize urgency.

The weights $w_1, ..., w_5$ are **not static** — they are calibrated through process mining analysis and adjusted dynamically:

#### How Process Mining Informs This Strategy

1. **Weight calibration**: Using the historical log, we perform a **retrospective optimization**: for each queue decision point in the past, evaluate what score each rule combination would have produced, and select weights that minimize total weighted tardiness over the historical dataset. This can be done via simulation-based optimization or genetic algorithm tuning.

2. **Setup time estimation**: The mined setup time matrix provides the $S_j$ estimates. For novel job pairs not seen historically, regression models extrapolate based on job attributes.

3. **Remaining processing time estimation**: Instead of using planned durations (shown to be inaccurate), we use the **mined duration distributions** conditioned on job type, machine, and operator. The expected remaining processing time uses median actual durations.

4. **Dynamic weight adjustment**: In periods of high load (detectable via real-time queue metrics), the urgency weight $w_1$ increases. When the bottleneck machine is being scheduled, the setup weight $w_2$ increases. These adjustment rules are derived from analyzing which factor was most predictive of tardiness under different load conditions in the historical data.

#### Pathologies Addressed
- **Poor prioritization**  Multi-factor urgency assessment prevents priority inversions
- **Excessive setup time**  Setup factor encourages grouping of similar jobs (soft batching)
- **Downstream starvation**  Downstream load factor coordinates across stations
- **Hot job disruption**  Priority factor absorbs urgent jobs without completely destabilizing sequences

#### Expected KPI Impact
- **Tardiness**: 30–50% reduction through better urgency-based prioritization
- **Setup time on bottleneck**: 15–25% reduction through setup-aware sequencing
- **WIP**: 10–20% reduction through downstream-aware dispatching
- **Lead time predictability**: Improved through more consistent, rational dispatching

### Strategy 2: Predictive Scheduling with Digital Twin Integration

#### Core Logic

This strategy moves from reactive dispatching to proactive scheduling by using a continuously updated predictive model of the shop floor — a "digital twin" parameterized by process mining data.

**Two-tier architecture:**

**Tier 1 — Predictive Forward Schedule (rolling horizon):**
At regular intervals (e.g., every 30 minutes or triggered by significant events), the system generates a forward-looking schedule for the next 4–8 hours using:
- Current shop floor state (jobs in progress, queue contents, machine status)
- Expected new arrivals (from order backlog and release schedule)
- **Stochastic task duration models**: For each task type × machine × operator combination, process mining provides not just a point estimate but a full distribution (e.g., log-normal with mined parameters $\mu$ and $\sigma$). The schedule uses percentile estimates (e.g., 75th percentile for planning buffer).
- **Predictive breakdown models**: From breakdown event history, fit reliability models (e.g., Weibull distribution for time between failures per machine). If a machine's time since last maintenance is approaching the hazard zone, the schedule proactively reduces load on that machine or front-loads critical jobs before the predicted failure window.
- **Setup time models**: Sequence-dependent setup times are explicitly modeled in the schedule.

The forward schedule is generated via a constructive heuristic (modified shifting bottleneck) or a MILP solver for the bottleneck machines, with dispatch rules for non-critical machines.

**Tier 2 — Anomaly Detection and Re-Scheduling Triggers:**
Continuous monitoring compares actual progress against the predicted schedule. When deviations exceed thresholds (e.g., a task takes >150% of the predicted time, a breakdown occurs, a hot job arrives), a re-scheduling event is triggered, and Tier 1 regenerates the forward schedule from the current state.

#### How Process Mining Informs This Strategy

1. **Stochastic duration models**: Mining the log for actual task durations, segmented by covariates (machine ID, operator ID, job complexity class, material type, shift), and fitting parametric distributions. For example:
   - CNC Milling on MILL-03 by OP-112 for aluminum parts: LogNormal(=3.8, =0.25) minutes (in log scale)
   - Same operation by OP-107: LogNormal(=4.0, =0.35) — slower with more variance

2. **Breakdown prediction**: Mining breakdown events to build:
   - Time-between-failure distributions per machine
   - Predictive features (e.g., increasing setup times or processing times as precursors to breakdown — detectable via trend analysis in the logs)
   - Seasonal patterns (breakdowns more common on Monday mornings due to cold starts)

3. **Routing probability models**: For jobs with optional or alternative routings, mining the actual routing frequencies and the conditions under which alternatives were chosen

4. **Hot job frequency model**: Mining the arrival pattern of priority changes to model their occurrence as a stochastic process, allowing the schedule to maintain buffer capacity for expected disruptions

#### Pathologies Addressed
- **Unpredictable lead times**  The predictive schedule provides realistic completion time estimates (with confidence intervals) for customer communication
- **Bottleneck overload**  Proactive identification of impending bottleneck congestion allows preemptive load balancing
- **Breakdown disruption**  Predictive maintenance integration reduces surprise breakdowns; when they occur, rapid re-scheduling minimizes cascade
- **Inaccurate planning**  Stochastic models replace point estimates, producing more robust schedules

#### Expected KPI Impact
- **Lead time predictability**: 40–60% improvement in estimation accuracy (measured by MAPE of quoted vs. actual completion)
- **Tardiness**: 25–40% reduction through proactive bottleneck avoidance and realistic buffering
- **Throughput**: 5–15% increase through better breakdown anticipation and load smoothing
- **WIP**: 15–25% reduction through tighter schedule control and reduced variability amplification

### Strategy 3: Setup Time Optimization through Intelligent Job Family Batching and Sequencing

#### Core Logic

This strategy specifically targets the sequence-dependent setup time problem, which process mining analysis will likely reveal as a major source of lost capacity, especially at bottleneck machines.

**Component A — Job Family Classification:**

Using the setup time matrix mined from historical data, apply clustering to create **setup families** — groups of jobs where intra-group setup times are minimal. The clustering algorithm (e.g., hierarchical agglomerative clustering using the setup time matrix as a distance matrix) produces $k$ families for each machine or work center.

Example outcome for CNC Milling:
- Family A: Aluminum aerospace brackets (5-min setup within family)
- Family B: Steel automotive shafts (8-min setup within family)
- Family C: Titanium medical components (3-min setup within family)
- Cross-family setup: 25–45 minutes

**Component B — Time-Windowed Family Batching at Bottleneck Machines:**

At bottleneck machines, instead of processing jobs in pure arrival order or due-date order, implement a **campaign scheduling** approach:
1. Define a **time window** $T_w$ (calibrated via simulation — e.g., 2–4 hours)
2. Within each window, group queued jobs by family
3. Process all jobs of one family before switching to the next
4. **Within each family**, sequence by urgency (critical ratio)
5. **Family ordering**: Choose the family sequence that minimizes total inter-family setup time (TSP on families) while respecting urgency constraints — a family containing any job with CR < threshold is processed first

**Component C — Coordinated Release with Look-Ahead:**

At the job release level, use a **controlled release** mechanism:
- Before releasing a job to the shop floor, check the current and near-future job mix at the bottleneck machine
- If a job belongs to a family that is currently being processed at the bottleneck (or will be within the next window), release it promptly
- If a job belongs to a family not scheduled until later, delay release slightly (if slack allows) to create better batching opportunities

This creates a gentle form of **CONWIP** (Constant WIP) that reduces WIP while improving setup efficiency.

#### How Process Mining Informs This Strategy

1. **Setup family discovery**: Entirely data-driven — the families emerge from the mined setup time patterns rather than being defined by engineering assumptions (which may be outdated)

2. **Time window calibration**: Process mining reveals the distribution of inter-arrival times for jobs of each family. If Family A jobs arrive every 30 minutes on average, a 2-hour window captures ~4 jobs for batching. If Family B arrives every 3 hours, a longer window or smaller batch is needed. The window size balances batching benefit against urgency risk.

3. **Setup time savings estimation**: The mined setup time matrix directly quantifies the savings: if the bottleneck processes 50 jobs/day with an average of 25 minutes setup per job (random sequencing) versus 10 minutes (family-batched), we save 50 × 15 = 750 minutes/day = 12.5 hours on a machine that was the constraint.

4. **Urgency risk calibration**: Historical analysis of how much delay jobs can tolerate before becoming tardy determines the maximum acceptable batching delay — ensuring setup optimization doesn't create new tardiness.

5. **Dynamic family updates**: As product mix evolves, periodic re-mining and re-clustering ensures families remain valid.

#### Pathologies Addressed
- **Excessive setup time**  Direct attack on the root cause, potentially recovering 15–30% of bottleneck capacity
- **Bottleneck throughput**  Reduced setup time translates directly to increased productive capacity
- **High WIP**  Controlled release prevents premature job injection
- **Downstream starvation**  By increasing bottleneck throughput, downstream machines receive more consistent flow

#### Expected KPI Impact
- **Setup time at bottleneck**: 40–60% reduction
- **Bottleneck throughput**: 10–20% increase (depending on current setup ratio)
- **Overall tardiness**: 20–35% reduction through increased effective capacity
- **WIP**: 15–25% reduction through controlled release
- **Lead time**: 15–25% reduction through faster bottleneck processing

---

### Strategy Integration

In practice, these three strategies are complementary, not competing:
- **Strategy 3** (setup optimization) operates at the job release and bottleneck sequencing level
- **Strategy 2** (predictive scheduling) provides the forward-looking framework and triggers re-scheduling
- **Strategy 1** (multi-factor dispatching) handles real-time decisions at non-bottleneck machines and resolves conflicts not covered by the higher-level strategies

The integrated system forms a hierarchical scheduling architecture:
1. **Release control** (Strategy 3) gates job entry to the shop floor
2. **Predictive schedule** (Strategy 2) generates a rolling plan for bottleneck machines
3. **Dynamic dispatching** (Strategy 1) executes decisions in real time, adapting to deviations from the plan

## 5. Simulation, Evaluation, and Continuous Improvement

### 5.1 Discrete-Event Simulation Framework

**Model Construction:**

A discrete-event simulation (DES) model of the job shop is constructed, parameterized entirely from process mining data:

| Model Component | Data Source (from Process Mining) |
|---|---|
| Machine set and capabilities | Resource log: which machines perform which operations |
| Job routing (probabilistic) | Variant analysis: routing frequencies and conditional routing probabilities |
| Job arrival process | Inter-arrival time distributions from job release events, segmented by job type |
| Task processing times | Per-task, per-machine, per-operator duration distributions (fitted from actuals) |
| Setup times | Sequence-dependent setup time matrix $S(a_i, a_j)$ with stochastic variation |
| Machine breakdown | Time-between-failure and time-to-repair distributions per machine |
| Hot job arrivals | Priority change event frequency and timing distributions |
| Operator availability | Shift patterns and operator-machine assignment rules from the log |
| Due date assignment | Distribution of lead time allowances (due date  release date) from historical jobs |

**Validation:** Before using the simulation for comparison, validate it against historical data:
- Run the simulation with the current dispatching rules (FCFS/EDD mix)
- Compare output distributions (throughput, WIP, tardiness, utilization) against the process-mining-measured actuals
- Calibrate until the model reproduces historical KPIs within acceptable error margins (e.g., <10% deviation on key metrics)

**Strategy Testing Protocol:**

Each strategy is implemented as an alternative scheduling module within the simulation. We test:

1. **Baseline**: Current FCFS/EDD rules (validated)
2. **Strategy 1**: Multi-factor dynamic dispatching
3. **Strategy 2**: Predictive scheduling
4. **Strategy 3**: Setup optimization with controlled release
5. **Integrated (1+2+3)**: Combined hierarchical approach

**Scenarios to Test:**

| Scenario | Description | Purpose |
|---|---|---|
| Normal operations | Average load, typical breakdown rate | Baseline comparison |
| High load (110–120% of average) | Increased job arrival rate | Stress test — does the strategy degrade gracefully? |
| Frequent breakdowns (2× historical rate) | Doubled breakdown frequency | Resilience testing |
| Hot job surge (3× historical rate) | Frequent priority changes | Disruption handling |
| Product mix shift | Different distribution of job families | Robustness to demand change |
| Seasonal peak | Combination of high load and rush orders | Worst-case realistic scenario |
| Single bottleneck failure | Extended outage of primary bottleneck | Catastrophic disruption recovery |

**Comparison Metrics:**

For each scenario × strategy combination, we collect:
- Mean and variance of: flow time, tardiness, WIP, throughput
- Percentage of jobs tardy
- Maximum tardiness (worst case)
- Bottleneck utilization and setup ratio
- Robustness index: ratio of performance degradation to stress increase

**Statistical Rigor:**
- Each scenario runs for a sufficient duration (e.g., 100 simulated days) with warm-up period removed
- Multiple replications (30+) with different random seeds
- Paired t-tests or Wilcoxon signed-rank tests to determine statistically significant differences between strategies
- Confidence intervals on all reported metrics

### 5.2 Continuous Monitoring and Adaptation Framework

After deploying the selected strategy, we establish a closed-loop improvement system:

**Layer 1 — Real-Time KPI Dashboard:**

Process mining on the live event stream (streaming process mining) provides:
- **Current KPIs**: Rolling average tardiness, WIP levels, queue lengths, utilization, setup ratios — updated every 15 minutes
- **Predictive KPIs**: Based on the forward schedule, projected tardiness for jobs in progress
- **Alert system**: Automatic alerts when any KPI deviates beyond control limits (established from the simulation's predicted performance range)

**Layer 2 — Concept Drift Detection:**

The scheduling system's parameters (duration distributions, setup time matrix, breakdown rates, arrival patterns) were calibrated from historical data. Over time, these will drift:
- **New product types** not represented in the original data
- **Equipment aging** changing breakdown patterns
- **Workforce changes** altering operator performance distributions
- **Customer mix evolution** changing job family proportions

Detection mechanism:
- **Kolmogorov-Smirnov tests** on rolling windows of task durations, setup times, and inter-arrival times, compared against the baseline distributions used to parameterize the scheduling system
- **CUSUM charts** on key process metrics to detect gradual shifts
- **Conformance checking**: Comparing actual process execution against the expected model — a drop in fitness score indicates process change

**Layer 3 — Automated Re-Calibration:**

When drift is detected:
1. **Parameter update**: Re-mine the affected distributions from the most recent 3–6 months of data
2. **Setup family refresh**: Re-cluster the setup time matrix
3. **Weight re-optimization**: Re-tune Strategy 1 weights on recent data
4. **Simulation re-validation**: Run the updated model through the scenario battery to confirm improved or maintained performance
5. **Deploy updated parameters**: Push to the live scheduling system

**Layer 4 — Periodic Deep Analysis (Monthly/Quarterly):**

Full process mining re-analysis:
- Comprehensive conformance checking
- New bottleneck analysis (has the bottleneck shifted due to improved scheduling?)
- Root cause analysis on any persistent tardiness
- Identification of new pathologies
- Assessment of whether the scheduling strategy itself has become the constraint (e.g., controlled release holding jobs too long, or batching windows causing unnecessary delays for specific job types)

**Layer 5 — Strategy Evolution:**

As the system matures and more data accumulates:
- Train machine learning models (e.g., reinforcement learning agents) that learn dispatching policies directly from the simulation environment, potentially discovering non-obvious strategies
- Integrate with APS (Advanced Planning and Scheduling) systems for longer-horizon planning
- Extend to multi-objective optimization (tardiness vs. WIP vs. setup cost vs. energy consumption) using Pareto-optimal scheduling frontiers
