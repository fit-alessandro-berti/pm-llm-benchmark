# Process Mining Analysis for Speedy Parcels: Last-Mile Delivery Optimization

## 1. Process Discovery and Conformance Checking

### 1.1 Data Preprocessing and Integration

To construct a cohesive event log, I would follow an **Extract-Transform-Load (ETL)** pipeline designed around process mining standards (XES/OCEL format):

**Integration Approach:**
- **Case ID design:** Use a hierarchical case notion. The primary case ID would be `Vehicle-Day` (e.g., V12-20241205) for route-level analysis, but I would also build **Object-Centric Event Logs (OCEL)** linking three object types: *Vehicle*, *Route*, and *Package*. This allows analysis from multiple perspectives (per-package vs. per-route).
- **Timestamp normalization:** Unify all time zones and resolve sub-second granularity differences between GPS (every few seconds) and scanner events (discrete).
- **Activity abstraction from GPS:** Raw GPS points must be abstracted into meaningful activities. Using **stay-point detection** (e.g., speed < 5 km/h for > 2 minutes within a 50m radius), continuous location streams become discrete events like "Travel," "Idle," "Stop at Location X."
- **Enrichment:** Join scanner events (delivery milestones) with GPS-derived stops using spatial-temporal matching, and overlay maintenance windows from the maintenance log.
- **Context enrichment:** Add external data (traffic, weather, time-of-day bucket) as event attributes.

**Challenges:**
- **Event granularity mismatch:** GPS generates thousands of events; scanners generate dozens. Aggregation/filtering is essential to avoid noise.
- **Missing events:** Scanners may be forgotten ("Depart Customer" not scanned), requiring imputation using GPS movement.
- **Ambiguous stops:** Distinguishing a customer stop from a traffic stop, toilet break, or refueling requires combining GPS location, duration thresholds, and scanner correlation.
- **Identity linking:** A vehicle may be driven by different drivers; packages may be reassigned mid-route.
- **GPS noise:** Urban canyons cause drift—requires smoothing (Kalman filters).
- **Data volume:** Six months × fleet × frequent GPS = massive. Use columnar storage (Parquet) and tools like PM4Py/Celonis.

### 1.2 Process Discovery

I would apply multiple discovery algorithms matched to the analytical purpose:

- **Inductive Miner (infrequent variant):** For a clean, structured baseline model of the "normal" delivery process (Start Shift  Depart Depot  [Travel  Arrive Customer  Service  Depart Customer]*  Arrive Depot  End Shift).
- **Heuristics Miner:** Captures frequencies and handles noise—useful for seeing main traffic flows.
- **Directly-Follows Graphs (DFG) with performance overlay:** Highlight where time is spent between activities.
- **Trace clustering:** Group similar route executions to reveal distinct behavioral patterns (e.g., urban dense routes vs. suburban sparse routes).

The discovered model should expose the *real* process—including deviations such as unscheduled maintenance stops, re-attempted deliveries, and long idle periods not visible in aggregate KPIs.

### 1.3 Conformance Checking

Using the dispatch system's planned routes as the **normative model**, I would apply:

- **Token-based replay** and **alignment-based conformance** to compare planned vs. actual sequences.
- **Deviation categories to detect:**
  - **Sequence deviations:** Stops visited out of planned order (suggesting driver improvisation).
  - **Skipped stops:** Planned deliveries not attempted.
  - **Unplanned stops:** Personal detours, unscheduled refueling, breakdowns.
  - **Timing deviations:** Arrival outside customer time window; travel legs significantly longer than estimated.
  - **Activity substitutions:** "Delivery Failed" instead of "Delivery Success."
- **Fitness, precision, and generalization** metrics quantify the gap between plan and reality per driver, per route, and per time period.

## 2. Performance Analysis and Bottleneck Identification

### 2.1 Key Performance Indicators

| KPI | Calculation from Event Log |
|---|---|
| **On-Time Delivery Rate** | % of "Delivery Success" events with timestamp within planned window (from dispatch) |
| **Average Service Time per Stop** | Mean(`Depart Customer`  `Arrive Customer`) |
| **Travel-to-Service Ratio** | (inter-stop travel) / (service time) per route |
| **Fuel Consumption per Package/km** | Fuel data joined with GPS distance and delivery count |
| **Vehicle Utilization Rate** | (Active moving + servicing time) / total shift duration |
| **Traffic Delay Frequency/Duration** | Count & duration of "Low Speed Detected" events |
| **Failed Delivery Rate** | Count("Delivery Failed") / Count(attempts) |
| **Stops-per-Hour** | Deliveries completed per active hour |
| **Overtime Rate** | % shifts where End Shift > planned end |

### 2.2 Bottleneck Identification Techniques

- **Performance-annotated DFG:** Color-code edges by average waiting/travel time. Thick red edges between "Arrive Customer" and "Delivery Success" pinpoint long on-site dwell.
- **Variant analysis:** Compare top 10% fastest vs. bottom 10% slowest vehicle-days. Look for activity patterns differentiating them.
- **Dotted chart analysis:** Plot events on a time-of-day axis to visually identify peak-hour congestion windows.
- **Geographic heat maps:** Overlay stop dwell times on a map to expose traffic/parking hotspots.
- **Resource/driver perspective:** Social network mining and handover-of-work analysis applied to driver performance distributions.
- **Sojourn-time analysis** per activity class quantifies each bottleneck's contribution to total cycle time.

Bottlenecks can thus be localized by **route, time, driver, or activity type**, with quantified impact (e.g., "driver parking search in Zone 3 between 11:00–13:00 adds avg. 4.2 min/stop").

## 3. Root Cause Analysis

### 3.1 Potential Root Causes

1. **Static route planning** that ignores real-time and historical traffic.
2. **Inaccurate ETAs**: Planned travel times uniformly underestimated by 15–25% during peak.
3. **Congestion hot spots**: Specific corridors consistently slow.
4. **Service time variability**: Some customer locations (e.g., apartment buildings requiring concierge interaction) systematically exceed 10 min/stop.
5. **Vehicle reliability**: Older vans show more unscheduled stops.
6. **Driver behavior variance**: Differences in stop efficiency, route-following, or customer handling.
7. **Failed deliveries**: Absent customers, wrong addresses—causing re-deliveries and cascading delays.

### 3.2 Validation via Process Mining

- **Variant analysis:** Compare high-performing vs. low-performing drivers on identical routes to isolate behavior effects from route effects.
- **Correlation analysis:** Cross-reference traffic conditions (as case attributes) with travel-leg durations to quantify congestion impact.
- **Dwell time decomposition:** Distinguish travel vs. waiting vs. service components to locate variability source.
- **Maintenance event correlation:** Link pre-maintenance performance degradation with vehicle age and mileage—detect predictive signals.
- **Failure pattern mining:** Cluster "Delivery Failed" events by time, address type, and customer segment.
- **Decision mining:** Identify data conditions (weather, route density, driver, zone) that predict failures.

## 4. Data-Driven Optimization Strategies

### Strategy 1: Dynamic, Traffic-Aware Route Re-Sequencing

- **Targets:** On-time delivery rate, travel time, fuel consumption.
- **Root cause addressed:** Static routing ignorant of congestion patterns.
- **Process mining support:** Discovered routes consistently show 20–30% longer travel legs in corridors X, Y during 08:00–10:00. Conformance checking reveals experienced drivers already deviate from planned sequences—and those deviations correlate with better on-time rates.
- **Implementation:** Feed process-mined travel-time distributions (by corridor × time-of-day) into a dynamic VRP solver; re-sequence stops at shift start and in real time when GPS detects major slowdowns.
- **Expected impact:** +8–15% on-time rate; 10% fuel use; reduced overtime.

### Strategy 2: Predictive Maintenance Based on Operational Signatures

- **Targets:** Unscheduled breakdowns, maintenance cost, shift disruption.
- **Root cause addressed:** Reactive maintenance causing mid-shift failures.
- **Process mining support:** Event log shows patterns preceding breakdowns—e.g., increasing "Low Speed/Idle" anomalies, unusual RPM patterns, or engine warnings weeks before failure. Correlate these precursors with maintenance log entries.
- **Implementation:** Build a predictive model using features derived from process mining (deviation frequency, idle-event clustering, mileage since service) and schedule preventive maintenance during off-hours.
- **Expected impact:** ~40% reduction in unscheduled stops; better vehicle utilization; lower repair cost.

### Strategy 3: Time-Window & Customer Communication Optimization

- **Targets:** Failed delivery rate, re-delivery cost, service time variability.
- **Root cause addressed:** Customers absent at delivery; inflexible windows.
- **Process mining support:** Analysis reveals failure rates spike for specific address types/zones at certain hours. Variant analysis shows failed deliveries add avg. 1.8 hours of re-delivery work per week per vehicle.
- **Implementation:** Use historical success probabilities per zone/time to propose personalized time windows; trigger proactive SMS/ETA notifications (with pre-arrival confirmation); offer alternative drop locations for high-risk cases.
- **Expected impact:** Failed delivery rate 30%; stops-per-hour +5%; customer satisfaction improvement.

### (Optional) Strategy 4: Targeted Driver Coaching

- **Targets:** Service time variability, driver overtime.
- **Root cause addressed:** Skill/behavior variance.
- **Support:** Performance analysis across drivers with identical route profiles reveals outliers. Coach on specific behaviors (e.g., parking strategy, scanner usage, customer interaction).
- **Expected impact:** Narrowed service-time distribution; reduced overtime.

## 5. Operational Constraints and Continuous Monitoring

### 5.1 Accounting for Constraints

- **Driver hours:** Route optimization models must respect shift-length limits and mandated breaks (encoded as hard constraints in the VRP).
- **Vehicle capacity:** Optimization must track volume/weight per van; process mining can validate historical capacity utilization to detect over/underloading.
- **Customer time windows:** Dispatch-defined windows are hard constraints; dynamic re-routing must maintain feasibility, re-checking upon every adjustment.
- **Regulatory/union rules:** Integrate into optimization as constraint layers.
- **Fairness:** Ensure driver workload balance across optimized routes (analyzed via resource perspective).

### 5.2 Continuous Monitoring Plan

Deploy a **live process mining dashboard** (Celonis, Disco, or custom PM4Py + BI) with:

**Operational (daily) views:**
- On-time delivery % by route, zone, driver (trend vs. baseline).
- Conformance score against the optimized plan.
- Real-time deviations map (stops off-plan, unusual idle).
- Failed delivery heatmap.

**Tactical (weekly) views:**
- Bottleneck drift analysis: are congestion hotspots shifting?
- Fuel and service-time KPI evolution.
- Driver performance distribution (are outliers shrinking?).
- Predictive maintenance alerts triggered vs. actual breakdowns.

**Strategic (monthly) views:**
- Before/after comparison of KPIs, with statistical significance.
- Variant analysis to detect emerging process patterns.
- ROI tracking per optimization strategy.

**Alerting:**
- Automatic alerts when conformance drops below thresholds, or when new variants emerge that weren't in historical patterns (**concept drift detection**).

This closed-loop approach ensures that the optimizations remain effective as urban traffic patterns, customer behavior, and fleet composition evolve—turning Speedy Parcels' operations into a **continuously learning logistics system**.
