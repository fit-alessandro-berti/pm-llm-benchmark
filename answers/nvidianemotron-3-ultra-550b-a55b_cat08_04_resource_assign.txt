# Process Mining-Driven Resource Assignment Optimization for TechSolve Solutions

## 1. Analyzing Resource Behavior and Assignment Patterns

### 1.1 Individual Agent & Tier Performance Profiling
Using the event log, I would construct **Resource Profiles** for every agent and aggregate views for tiers (L1, L2, L3) focusing on these key metrics:

| Metric Category | Specific Metrics | Process Mining Technique |
| :--- | :--- | :--- |
| **Workload & Utilization** | *Tickets handled per period (day/week/shift)*; *Active working time vs. idle time*; *Concurrency (parallel tickets)*; *Overtime patterns*. | **Resource Utilization Analysis**; Dotted Chart analysis colored by Resource. |
| **Processing Efficiency** | *Mean/median throughput time per activity (Work Start  Work End)*; *Touch time vs. Wait time ratio*; *First-Call Resolution (FCR) Rate for L1* (Tickets closed by L1 without Escalation/Reassign). | **Performance Spectrum**; **Variant Analysis** (filtering "Happy Path" vs. Escalated paths). |
| **Quality & Rework** | *Reassignment Rate* (Outgoing/Incoming); *Escalation Rate* (L1L2, L2L3); *Reopen Rate* (Post-closure); *SLA Breach Rate per agent*. | **Conformance Checking** against SLA model; **Root Cause Analysis** on "Reassign" activity. |
| **Skill Alignment** | *Ticket Category/Skill Frequency Matrix* (Which agents handle which categories?); *Skill Match Score* (Agent Skills  Ticket Required Skill  ); *Proficiency Proxy* (Resolution time for specific skill vs. peer median). | **Resource-Activity Matrix**; **Social Network Analysis (SNA)** on handovers filtered by Skill/Category. |

**L1 Specific Focus:** Calculate **FCR Rate** per agent. Low FCR + High Escalation Rate indicates either training gaps or overly complex tickets routed to L1. High FCR but long handling time suggests L1 "holding" tickets too long before escalating (SLA risk).

### 1.2 Revealing Actual Assignment Patterns vs. Intended Logic
The "intended logic" (Round-Robin + Manual Escalation) rarely matches reality. Process mining reveals the **"As-Is"** flow:

*   **Resource Interaction Analysis (Handover Network):**
    *   Construct a **Handover-of-Work Network** where nodes are Agents/Tiers and edges represent ticket transitions (Assign  Reassign  Escalate).
    *   *Edge Weight:* Frequency of handover. *Edge Attribute:* Avg. time between handover, % SLA breaches originating here.
    *   *Insight:* Identify "Super-Hubs" (agents receiving disproportionate escalations) and "Dead Ends" (agents frequently reassigning out). Visualize if L1L2 handovers bypass Dispatcher (shadow IT process).

*   **Role Discovery & Organizational Mining:**
    *   Apply **Role Mining algorithms** (e.g., based on activity clusters) to discover *actual* roles. Do "L2 Agents" cluster into distinct sub-roles (e.g., "DB Specialists", "Network Firewall Experts", "Generalist L2")?
    *   Compare discovered roles against HR-defined roles. Mismatch = inefficient assignment logic.

*   **Decision Point Analysis (Dispatcher/Assignment Logic):**
    *   Treat every `Assign L1`, `Assign L2`, `Reassign` event as a decision point.
    *   Extract context features at decision timestamp: *Ticket Priority, Category, Required Skill, Current Queue Length per Agent, Agent Shift Status, Agent Current Load.*
    *   Train a **Decision Tree / Rule Learner** to mimic the Dispatcher's logic.
    *   *Comparison:* "IF Priority=P2 THEN Assign Least Busy" vs. Actual: "IF Priority=P2 AND Category=Network THEN Assign Agent B08 (Specialist) ELSE Round Robin". This quantifies the "Manual Override" rate.

### 1.3 Skill Utilization Effectiveness Analysis
*   **Skill Coverage Heatmap:** Matrix (Agents x Skills) colored by *Frequency of Use* vs. *Proficiency (Speed/Quality)*.
    *   *Red Flag:* High proficiency, low usage (Underutilized Specialist - e.g., Agent B12 has `DB-SQL` but only used it 2 times in 6 months).
    *   *Red Flag:* High usage, low proficiency (Stretched Generalist - e.g., Agent B08 handling `Security-IAM` tickets slowly with high reassign rate).
*   **"Over-Qualification" Metric:** For every ticket, calculate `Max(Proficiency_Level_Assigned_Agent) - Required_Proficiency_Level`.
    *   High average over-qualification = Expensive L3 resources doing L2 work (Cost inefficiency).
*   **Skill Gap Analysis:** Tickets where `Required Skill  Assigned Agent Skills` at `Work Start`. Correlate these tickets directly with `Reassign` events and SLA breaches.

## 2. Identifying Resource-Related Bottlenecks and Issues

### 2.1 Bottleneck Identification Techniques
1.  **Resource-Constrained Waiting Time Analysis:**
    *   Calculate *Waiting Time* = `Assign Timestamp` - `Work Start Timestamp` (Queue time).
    *   Segment by **Required Skill**. Skills with highest median queue time = **Capacity Bottlenecks** (e.g., `Networking-Firewall` avg wait 45 mins vs `OS-Windows` 5 mins).
    *   Cross-reference with Agent Availability calendars (if available) or Shift patterns in log.

2.  **Reassignment/Escalation Impact Quantification:**
    *   **Process Variant Comparison:** Compare throughput time of Variant A (Create  Assign L1  Work L1  Close) vs. Variant B (Create  Assign L1  Work L1  Escalate  Assign L2  Reassign  Work L2  Close).
    *   *Quantification:* "Each Reassignment adds **X hours** median resolution time." "Tickets with 1 Reassignment have **Y% higher SLA breach rate**."
    *   **Ping-Pong Detection:** Identify tickets bouncing L1L2 or L2aL2b > 2 times. These are "Hot Potatoes" caused by skill ambiguity.

3.  **Initial Assignment Accuracy:**
    *   Define "Correct Initial Assignment": Ticket Assigned to Agent possessing `Required Skill` at first `Assign L1/L2` event.
    *   *Metric:* **Initial Assignment Accuracy Rate**.
    *   *Impact:* Tickets with *Incorrect Initial Assignment*  3.2x higher Reassignment rate, 2.5x SLA breach rate (Hypothetical finding based on log pattern).

4.  **Workload Imbalance (Gini Coefficient / Coefficient of Variation):**
    *   Calculate tickets handled per agent per shift.
    *   *Finding:* Top 20% L2 agents handle 60% of `App-CRM` tickets (Burnout risk); Bottom 30% handle < 5% (Underutilization).

### 2.2 Correlation with SLA Breaches
*   Build a **Classification Model (e.g., Random Forest)** predicting `SLA_Breach (True/False)` using features available at Ticket Creation + First Assignment:
    *   Features: Priority, Category, Required Skill, Initial Assigned Tier, Initial Agent Load, Time of Day, Initial Assignment Accuracy (Boolean).
*   *Key Output:* **Feature Importance** ranking. Likely top drivers: "Initial Assignment Accuracy", "Required Skill Queue Length", "Priority P2/P3 during Peak Hours".

## 4. Developing Data-Driven Resource Assignment Strategies

### Strategy 1: Context-Aware Skill-Based Routing (SBR) with Proficiency Weighting
*   **Issue Addressed:** Incorrect initial assignments, frequent reassignments, specialist misuse.
*   **Process Mining Insight:** Handover Network shows 35% of L1L2 escalations are pure skill mismatches. Skill Coverage Heatmap shows specific agents (e.g., B12, B15) are *only* efficient resolvers for `App-CRM`/`DB-SQL`.
*   **Logic:**
    1.  At Ticket Creation: NLP Classifier predicts `Required Skill Set` (Primary + Secondary) from Description/Subject.
    2.  Routing Engine queries **Real-Time Agent Registry**: `(Agent Skills  Required Skills) AND (Agent Status = Available) AND (Current Load < Threshold)`.
    3.  **Scoring Function:** `Score = Skill_Proficiency_Weight * W1 + (1 / Current_Load) * W2 + Affinity_Score * W3`.
        *   *Proficiency Weight:* Derived from historical median resolution time per skill per agent (Mined from log).
        *   *Affinity:* "Sticky assignment" for recurring user/issues (reduces context switching).
*   **Data Required:** Real-time Agent Status API (Available/Busy/Offline), Updated Skill Matrix with Proficiency Scores (derived from log), Ticket Text Stream for NLP.
*   **Expected Benefit:** 40% reduction in Reassignments; 25% reduction in L1L2 Escalations for skill-mismatch reasons; 15% improvement in P2/P3 SLA compliance.

### Strategy 2: Workload-Aware Dynamic Load Balancing with "Virtual Queues"
*   **Issue Addressed:** Uneven workload (Burnout vs. Boredom), Blind Assignment causing Queue Delays, SLA breaches due to hidden congestion.
*   **Process Mining Insight:** Dotted Charts show "Burst Arrivals" overwhelming specific agents while peers idle. Resource Utilization Analysis shows Coefficient of Variation (CoV) of workload > 0.8 for L2.
*   **Logic:**
    1.  Replace Agent-Specific Queues with **Skill-Based Virtual Queues** (e.g., Queue: `Networking-Firewall`, Queue: `App-CRM`).
    2.  Agents "Pull" from queues they are qualified for based on **Dynamic Capacity Score**: `Capacity = (Shift_Remaining_Hours * Efficiency_Factor) - Current_Active_Tickets_Weight`.
    3.  **Priority Preemption:** P1/P2 tickets jump queue; System calculates *Projected SLA Risk* for waiting tickets. If Risk > Threshold, trigger "Overflow Routing" to next-best-skilled tier (e.g., L3 helps L2 queue) or alert Team Lead.
*   **Data Required:** Real-time Ticket Queue State, Agent Shift Schedules, Real-time Agent Activity (Heartbeat/API), SLA Targets per Priority.
*   **Expected Benefit:** 50% reduction in Workload CoV (Balanced utilization); 30% reduction in Avg. Queue Time for bottleneck skills; Elimination of "Agent Cherry-Picking" easy tickets.

### Strategy 3: Predictive L1 Empowerment & Intelligent Escalation Gating
*   **Issue Addressed:** Excessive/Late Escalations, L1 "Thrashing" (working too long before escalating), Specialists doing L1 work.
*   **Process Mining Insight:** Variant Analysis shows tickets escalated *after* 30 mins L1 work have 80% SLA breach rate. Decision Mining identifies L1 agents with low FCR on specific categories due to missing *procedural knowledge* (not technical skill).
*   **Logic:**
    1.  **Predictive Escalation Recommendation:** At `Work L1 Start`, model predicts `P(Resolve_L1 | Ticket_Features, Agent_Profile, Time_Spent)`.
    2.  **Smart Guidance:** If `P(Resolve_L1) > 80%`  Show "Recommended KB Article / Runbook / Diagnostic Script" (Mined from successful past variants).
    3.  **Escalation Gate:** If `P(Resolve_L1) < 20%` OR `Time_Spent > Optimal_Threshold` (mined per Category)  **Auto-suggest Escalation** with *Pre-populated Required Skill* and *Context Summary* for L2. Prevents "Blind Escalation".
    4.  **Micro-Learning Loop:** Flag tickets where L1 failed but `P(Resolve_L1)` was high  Targeted Training Recommendation for that Agent/Category.
*   **Data Required:** Historical Resolution Paths (Variants), Knowledge Base Mapping (Ticket Category  Solution Pattern), Real-time Agent Timer.
*   **Expected Benefit:** 20% Increase in L1 FCR Rate; 35% Reduction in "Late Escalations" (>30 mins); 10% Reduction in L2 workload volume (filtering noise); Faster L2 Start (Context passed automatically).
