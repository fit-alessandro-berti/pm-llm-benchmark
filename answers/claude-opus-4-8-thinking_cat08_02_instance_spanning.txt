# Optimizing E-Commerce Fulfillment: A Constraint-Aware Process Mining Strategy

## Preliminary Note on Instance-Spanning Constraints (ISCs)

The defining challenge here is that traditional process mining treats each case (order) in isolation—measuring its trace, throughput, and bottlenecks independently. But the four documented phenomena are **instance-spanning constraints**: the behavior of one case directly determines what another case can do. This requires augmenting standard case-centric analysis with **resource-centric, time-window, and batch-centric perspectives**. My entire strategy is built on this dual lens.

---

## Section 1: Identifying Instance-Spanning Constraints and Quantifying Their Impact

### 1.1 General Approach: From Case View to System State View

The first analytical move is to transform the event log from a case-oriented format into a **time-indexed system-state log**. By pairing every START/COMPLETE event, I can reconstruct, for any timestamp, the complete "state of the world":

- Which orders occupy which resources
- How many hazardous orders are simultaneously in Packing/QC
- Which batches are forming and how many orders are waiting in each
- Which express orders are present and what they preempted

This is achieved by building an **activity-interval table** (one row per case-activity with start_ts, complete_ts, resource, attributes) and then computing rolling concurrency counts. This state reconstruction is the foundation for measuring every ISC.

### 1.2 Constraint-by-Constraint Detection and Metrics

**(a) Shared Cold-Packing Stations (5 stations)**

- *Detection:* Filter `Requires Cold Packing = TRUE`. Identify Cold-Packing station resources (e.g., C-prefix). For each timestamp, count concurrent cold-packing occupancy. Whenever an order's Item Picking completes but its Packing START is delayed *while all 5 cold stations are occupied*, that delay is attributable to resource contention.
- *Key metrics:*
  - **Cold-station utilization %** (occupied station-minutes ÷ available station-minutes) — values approaching 100% during peaks signal a hard bottleneck.
  - **Contention waiting time** = time between "ready for cold-packing" (Picking complete) and "cold-packing START," conditioned on all stations busy.
  - **Queue length over time** for cold-packing (number of orders waiting), via a virtual queue reconstructed from the state log.
  - **Probability of blocking** = fraction of cold orders that experienced any queue.

**(b) Shipping Batches**

- *Detection:* For each order, compute the gap between **QC COMPLETE** and **Shipping Label Generation COMPLETE**. Group by Destination Region and batch ID. The "(Waited for batch)" annotation and the shared `System (Batch Bx)` resource identify batched orders.
- *Key metrics:*
  - **Batch waiting time** per order = Shipping-label time  QC-complete time.
  - **Batch dwell / formation time** = time from first order entering a batch to batch release.
  - **Batch fill rate** (actual size vs. target size) and **straggler-induced delay** (how long early-arriving orders wait for late ones).
  - Correlation between batch size and per-order waiting time.

**(c) Priority (Express) Handling**

- *Detection:* Look for **preemption signatures** — a standard order with a START event whose corresponding COMPLETE is unusually delayed, where an express order's START on the *same resource* falls within that interval. A pause/resume pattern (gap in the standard order's resource occupancy coinciding with express activity) is the fingerprint.
- *Key metrics:*
  - **Preemption count** and **standard-order delay due to express** = added time standard orders spend because of express insertion.
  - **Express order flow time** vs. standard flow time (the benefit side).
  - **Resource displacement events** at each shared step.
  - **Net trade-off ratio**: express time saved per unit of standard time lost.

**(d) Hazardous Material Limit (max 10 concurrent in Packing OR QC)**

- *Detection:* Using the state log, compute a **rolling count of hazardous orders simultaneously in Packing+QC**. Identify timestamps where this count = 10 *and* another hazardous order was ready (Picking complete / prior step complete) but its Packing/QC START was deferred.
- *Key metrics:*
  - **Time at the hazardous cap** (minutes per day at count = 10).
  - **Regulatory-induced waiting time** for hazardous orders blocked at the cap.
  - **Throughput reduction**: hazardous orders processed per hour during capped vs. uncapped periods.
  - **Compliance verification**: confirm the count never exceeds 10 (also an audit/validation function).

### 1.3 Differentiating Within-Instance vs. Between-Instance Waiting

This is the crux. For each waiting period (gap between an activity's COMPLETE and the next activity's START), I classify the cause using the **system state at the moment of waiting**:

| Waiting Type | Diagnostic Test |
|---|---|
| **Within-instance** (e.g., slow staff, large order) | Long *active processing duration* (STARTCOMPLETE); occurs even when required resources/batches are available. Identified when waiting is short but the activity itself is long. |
| **Between-instance — resource contention** | The next required resource exists in the facility but is 100% occupied by *other cases* at the waiting timestamp. Confirmed by checking concurrent occupancy of that resource pool. |
| **Between-instance — batching** | Order is QC-complete but Shipping label deferred while batch is incomplete; resource is free, only the batch trigger is unmet. |
| **Between-instance — priority** | Resource occupancy shows an express case inserted; standard case's resource was displaced. |
| **Between-instance — regulatory** | Order ready, required resource (packing/QC staff) *available*, but hazardous concurrency = 10. Resource is free yet START is blocked — the signature of a regulatory ISC. |

Practically, I implement this as a **decision rule applied to every inter-activity gap**, attributing each minute of waiting to exactly one cause. Aggregating this produces a **waiting-time decomposition** — the single most valuable diagnostic output, showing leadership precisely how much delay stems from *between-instance dependencies* versus simple slow processing.

## Section 2: Analyzing Constraint Interactions

The constraints are not independent; their interactions often produce non-linear, compounding delays. Standard single-bottleneck analysis would miss these.

### 2.1 Priority × Cold-Packing Contention
An express order requiring cold-packing arrives at an already-saturated 5-station resource. Two cascade effects:
- It **preempts a standard cold order**, but cold-packing may not be cleanly interruptible (perishable integrity), so the preemption either fails or extends the standard order's spoilage-sensitive duration.
- It **jumps the cold queue**, pushing *all* waiting standard cold orders back by one slot. Because cold stations are the scarcest resource, the marginal impact of each express insertion is amplified compared to express insertion at abundant standard stations. **Express priority is most damaging precisely where capacity is tightest.**

### 2.2 Batching × Hazardous Limits
Suppose several hazardous orders are bound for the same region (e.g., South). Batching logic encourages holding them together for Shipping. But the hazardous cap restricts how many can be in Packing/QC *simultaneously*. The result:
- Hazardous orders for one batch are **serialized through Packing/QC** (cap-limited), then must **wait again at the batch** for siblings still queued behind the cap. This is **double waiting** — a regulatory delay *feeding into* a batching delay. The batch cannot release until all members clear QC, but the cap throttles QC entry, so the batch's slowest member is regulatory-bound.

### 2.3 Priority × Hazardous Limits
An express **hazardous** order arrives when the cap (10) is reached. Priority says expedite; regulation says no entry. Here regulation **overrides** priority — express status cannot be honored. This reveals a class of orders ("express + hazardous") whose SLA is structurally at risk, invisible unless we analyze constraint co-occurrence.

### 2.4 Cold-Packing × Batching
Cold orders cleared through scarce stations may then wait at a regional batch for non-cold orders. The hard-won cold-station throughput is **squandered in batch waiting**, meaning investing in more cold stations yields little benefit unless batching is also fixed.

### 2.5 Why Interaction Analysis Is Essential
Optimizing one constraint in isolation can **shift or worsen** another (a classic bottleneck-migration problem). For example, speeding cold-packing simply moves cold orders into batch-wait faster. Adding express priority aggressively starves standard orders. Therefore optimization must be evaluated **system-wide**, which is exactly why simulation (Section 4) is mandatory. I would quantify interactions by computing **conditional waiting metrics** (e.g., batch waiting time *given* hazardous content and regional concentration) and building a **correlation/dependency matrix** across constraint-attributed delays.

## Section 3: Constraint-Aware Optimization Strategies

### Strategy 1: Predictive Dynamic Allocation & Reservation for Cold-Packing Stations
- **Primary constraint addressed:** Shared Cold-Packing (interacting with Priority).
- **Proposed changes:**
  - Introduce a **demand-forecasting model** trained on the event log (arrival patterns of cold orders by hour/day/season) to predict cold-station demand 1–2 hours ahead.
  - Implement a **slot-reservation policy**: a small number of cold stations (e.g., 1 of 5) are **reserved for express cold orders** during predicted peak windows, so express orders rarely need to preempt — eliminating the most damaging preemptions.
  - Add a **"cold-eligibility smoothing"** rule: when forecast shows imminent cold saturation, release upstream Picking for cold orders in a staggered manner to avoid synchronized arrival surges.
- **Data leverage:** Historical inter-arrival distributions, utilization curves, and contention-waiting metrics from Section 1 calibrate the reservation count and trigger thresholds.
- **Expected outcome:** Reduced cold-station queue length, fewer express preemptions of standard cold orders, lower contention waiting time — directly attacking the scarcest-resource bottleneck.

### Strategy 2: Dynamic, Deadline-Aware Batch Formation
- **Primary constraint addressed:** Shipping Batches (interacting with Hazardous limits and Cold).
- **Proposed changes:**
  - Replace static "wait for full batch" logic with a **dual-trigger release rule**: release a batch when EITHER (a) target size is reached OR (b) a **max-wait threshold** is hit OR (c) the **oldest order in the batch approaches its SLA deadline**.
  - For regions with **hazardous concentration**, decouple hazardous batch members so the batch doesn't stall waiting for cap-throttled siblings — allow **partial batch release** with a follow-up sub-batch.
  - Optimize **max-wait thresholds per region** using historical data balancing delivery-route savings against accumulated batch-waiting cost.
- **Data leverage:** Batch dwell-time distributions, straggler analysis, and per-region order-arrival rates determine optimal thresholds; the regulatory-delay-feeding-batch-delay interaction (Section 2.2) identifies which regions need decoupling.
- **Expected outcome:** Lower batch waiting time, prevention of SLA breaches from stragglers, and breaking the double-waiting cascade between hazardous limits and batching.

### Strategy 3: Constraint-Aware Scheduling for Hazardous + Priority Orders
- **Primary constraint addressed:** Hazardous Material Limit (interacting with Priority).
- **Proposed changes:**
  - Implement a **hazardous-slot scheduler** that treats the 10-concurrent cap as a managed token pool. Orders entering Packing/QC acquire a token; the scheduler **prioritizes express-hazardous orders** within the cap (since regulation can't be relaxed, the only lever is *ordering* within the cap).
  - Introduce **look-ahead admission control**: when the cap is near, sequence non-express hazardous orders to maximize cap turnover (e.g., admit shorter-duration hazardous orders first to free tokens faster — a shortest-processing-time discipline *within* the regulatory constraint).
  - Smooth hazardous order release upstream so they don't all arrive at Packing simultaneously, reducing time spent at the cap.
- **Data leverage:** Hazardous processing-duration distributions and "time-at-cap" metrics identify when and how to sequence; express-hazardous co-occurrence frequency justifies prioritization rules.
- **Expected outcome:** Maintained 100% compliance while increasing hazardous throughput, reduced regulatory-induced waiting, and protected express-hazardous SLAs.

### (Optional Strategy 4: Targeted Capacity Adjustment, Data-Justified)
Only *after* Strategies 1–3 — and validated by simulation — would I recommend adding a cold station or QC staff, because Section 2 shows capacity additions can be wasted if downstream batching isn't fixed first. The business case would be built on the simulated marginal flow-time improvement per added unit.

## Section 4: Simulation and Validation

### 4.1 Why Simulation Is Non-Negotiable Here
Because the constraints interact non-linearly and create bottleneck migration, analytical "what-if" reasoning is unreliable. I would build a **discrete-event simulation (DES)** model, calibrated directly from the mined event log (a "**digital twin**" of the fulfillment center).

### 4.2 Model Construction from Process Mining Outputs
- **Process structure** from the discovered process model (e.g., via inductive miner) provides the activity flow.
- **Activity duration distributions** fitted per activity (and per order type) from STARTCOMPLETE intervals.
- **Arrival processes** modeled from historical inter-arrival times, segmented by order type, hazardous flag, cold requirement, and region — preserving the *attribute mix* that drives ISCs.

### 4.3 Critical: Modeling the Instance-Spanning Constraints Explicitly
A naive DES models resources as simple servers and would **miss the ISCs**. I would explicitly encode:
- **Cold-packing as a finite resource pool (capacity 5)** with the reservation policy from Strategy 1 as a configurable parameter.
- **Batching as a synchronization/assembly node** — orders accumulate per region and release on the dual-trigger logic (Strategy 2), tunable.
- **Priority as a preemptive/priority queue discipline**, with express orders able to seize resources, including modeling preemption cost (cleanup/restart).
- **Hazardous cap as a global semaphore/token pool (capacity 10)** spanning Packing and QC nodes — this is the most distinctive element: a **shared counter constraint across two activities**, which standard simulation templates lack. Each hazardous order acquires a token at Packing entry and releases it at QC exit.

### 4.4 Validation and Experimentation
- **Baseline validation:** Run the model on the historical period and confirm it **reproduces observed KPIs** (flow times, utilization, queue lengths, time-at-cap) within acceptable tolerance. This proves the ISCs are correctly captured before testing changes.
- **Scenario experiments:** Run each strategy individually and in combination, across normal and peak-season arrival rates.
- **Stress / peak testing:** Inject the high-volume seasonal arrival pattern to ensure strategies hold under pressure.
- **Compliance assertion:** The simulation must **never exceed the hazardous cap of 10** — this is a hard validation check, not a KPI to optimize.
- **Sensitivity analysis** on key levers (reservation count, batch max-wait, cap-sequencing rule) to find robust settings and avoid overfitting to historical patterns.
- **Output:** A comparative KPI table (mean/95th-percentile flow time, throughput, express SLA attainment, cold-queue length, batch dwell, hazardous waiting) per scenario, enabling data-driven selection.

## Section 5: Post-Implementation Monitoring

### 5.1 Monitoring Philosophy
Monitoring must track not just overall flow but the **health of each ISC**, because regression in a single constraint can silently degrade the whole system. I would deploy **near-real-time process mining dashboards** fed by the live event stream.

### 5.2 Dashboards and Their Metrics

**Dashboard A — End-to-End Performance (Executive view)**
- Mean and 95th-percentile order flow time (overall, and split by order type/hazardous/cold).
- Throughput (orders/hour, /day).
- SLA attainment % (especially express and express-hazardous).
- The **waiting-time decomposition** (within- vs. between-instance) trended over time — the headline indicator that ISC management is improving.

**Dashboard B — Shared Resource Health (Cold-Packing)**
- Live cold-station utilization and queue length.
- Contention waiting time trend (should fall vs. baseline).
- Express-preemption count at cold stations (target: near zero after reservation policy).

**Dashboard C — Batching Performance**
- Batch dwell/formation time per region.
- Batch fill rate and straggler-delay.
- Count of SLA-triggered early releases (confirms dual-trigger working).

**Dashboard D — Regulatory & Priority Compliance**
- **Real-time hazardous concurrency count with a hard alert if it approaches/breaches 10** (compliance is non-negotiable).
- Time-at-cap trend and hazardous-order waiting time.
- Express SLA attainment, including the at-risk express-hazardous segment.

### 5.3 Tracking ISC-Specific Improvement
For each constraint I define a **target band** validated in simulation, and monitor against it:
- Cold-packing: queue length and contention wait **down X%** vs. baseline.
- Batching: dwell time down while **route-efficiency maintained** (guard against over-fragmenting batches).
- Hazardous: **zero breaches** maintained *while* throughput up and waiting down.
- Priority: express SLA up *without* standard-order delay exceeding an agreed ceiling (monitor the trade-off ratio).

### 5.4 Continuous Improvement Loop
I would set up **conformance checking** to detect drift (e.g., new bottleneck migration, seasonal pattern shifts) and **automated alerts** when any ISC metric leaves its target band. Periodically (e.g., monthly), re-run the calibrated simulation with fresh data to **re-tune parameters** (reservation counts, batch thresholds), closing the loop between monitoring, analysis, and optimization.

---

## Summary

The core insight throughout is that this fulfillment center's problems are **not slow activities but constrained dependencies between orders**. My strategy therefore: (1) reconstructs system state to *measure* between-instance waiting and decompose delays; (2) explicitly analyzes how constraints *compound* one another; (3) proposes targeted, data-calibrated policies (cold-station reservation, dynamic batching, regulatory-aware scheduling) that respect interdependencies; (4) validates them in an ISC-aware digital twin before any rollout; and (5) sustains gains through ISC-specific monitoring with hard compliance guardrails. This ensures improvements are real, system-wide, and durable rather than merely shifting bottlenecks elsewhere.
