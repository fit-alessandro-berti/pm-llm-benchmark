## 3. Database Verification Strategies (PostgreSQL)

Below are targeted queries to validate whether these anomalies occur in production data. All queries assume `claim_events` contains one row per activity execution.

###  Query 1: Detect Premature Closures (C before E or P)
*Tests hypothesis: Claims are closed before proper evaluation/approval due to weak constraints or fast-track rejections.*
```sql
WITH first_occurrence AS (
  SELECT claim_id,
         MIN(timestamp) FILTER (WHERE activity = 'A') AS t_A,
         MIN(timestamp) FILTER (WHERE activity = 'E') AS t_E,
         MIN(timestamp) FILTER (WHERE activity = 'P') AS t_P,
         MIN(timestamp) FILTER (WHERE activity = 'C') AS t_C
  FROM claim_events
  GROUP BY claim_id
)
SELECT claim_id, t_A, t_E, t_P, t_C
FROM first_occurrence
WHERE t_C IS NOT NULL
  AND (
    t_E IS NULL OR t_P IS NULL OR 
    t_C < t_E OR t_C < t_P
  );
```
**Interpretation:** Returns claims where closure happened before evaluation/approval, or where those steps are entirely missing. Join with `claims` to check if these correlate with specific `claim_type` or low `claim_amount`.

###  Query 2: Detect Evaluation-Approval Loops
*Tests hypothesis: Claims undergo multiple E/P cycles due to iterative reviews or system workarounds.*
```sql
SELECT claim_id,
       COUNT(*) FILTER (WHERE activity = 'E') AS eval_count,
       COUNT(*) FILTER (WHERE activity = 'P') AS approve_count
FROM claim_events
GROUP BY claim_id
HAVING COUNT(*) FILTER (WHERE activity = 'E') > 1
    OR COUNT(*) FILTER (WHERE activity = 'P') > 1
ORDER BY eval_count DESC, approve_count DESC;
```
**Interpretation:** Flags claims with repeated evaluations or approvals. High counts suggest either legitimate iterative reviews or process inefficiencies (e.g., bouncing between departments).

###  Query 3: Detect Skipped Customer Notifications

```sql
SELECT ce.claim_id, c.claim_type, c.claim_amount
FROM claim_events ce

```
**Interpretation:** Identifies closed claims that never logged a notification step. Group results by `claim_type` or `claim_amount` to see if skips follow a business pattern (e.g., only auto_insurance or claims < $500).
