### 1. Analyzing Resource Behavior and Assignment Patterns

*   **Touch Time vs. Wait Time:** Differentiating the time between "Assigned" and "Work Start" (Wait Time/Queue Time) versus "Work Start" and "Work End" (Touch Time).
*   **First Contact Resolution (FCR) Rate for L1:** Calculating the percentage of tickets resolved by L1 without an "Escalate" or "Reassign" event.
*   **Ping-Pong/Handover Count:** The average number of distinct agents or tiers touching a ticket before "Ticket Closed."
*   **Skill Utilization Rate:** Cross-referencing "Agent Skills" with the final "Required Skill" noted at resolution to see if overqualified agents are doing basic work.

**Process Mining Techniques Applied:**
*   **Resource Handover Network (Social Network Analysis):** By tracking consecutive activities (e.g., Agent A05 -> Dispatcher -> Agent B12), we generate a handover matrix. This visualizes ping-ponging. Ideally, handovers should flow left-to-right (L1 -> L2 -> L3). If we see heavy bidirectional arrows (L2 -> L2, or L2 -> L1), we have dysfunctional assignment logic.
*   **Role/Organizational Discovery:** We will cluster agents based on the *actual activities* they perform and the categories they solve, rather than their official HR titles. This reveals if L2 specialists are inadvertently functioning as an L1 overflow pool.
*   **Skill-to-Task Alignment Analytics:** We will map the "Agent Skills" array against the "Required Skill" for completed tasks. For instance, in INC-1001, Agent B12 (App-CRM, DB-SQL) worked an App-CRM ticket but reassigned it because it required a different skill setup. This reveals gaps in initial ticket categorization.

To understand *why* these bottlenecks occur, we will use **Variant Analysis** and **Decision Mining**.

*   **Variant Analysis (Happy vs. Unhappy Paths):** We compare the process paths of tickets that met SLAs with zero reassignments versus P2/P3 tickets that breached SLAs with multiple handovers. Round-robin? ### 4. Developing Data-Driven Resource Assignment Strategies

Based on the empirical data derived from process mining, I recommend implementing the following three strategies:

#### Strategy 1: Predictive, Skill-Based Routing Matrix
*   **The Issue Addressed:** High reassignment rates and SLA breaches due to initial miscategorization and round-robin assignment.
*   **Process Mining Insight Used:** Historical correlation between ticket descriptions/categories at creation and the *actual* skills of the L2/L3 agent who ultimately resolved it without reassignment.
*   **Data Required:** Real-time access to ticket creation data, NLP for text parsing, and an updated, deeply granular Agent Skill Matrix (categorized by proficiency level: Novice, Competent, Expert).
*   **How it Works:** Instead of general "L1 Pool -> L2 Pool," the system predicts the required skill upon ticket creation. If a P2 ticket description heavily implies a Firewall issue, it is routed immediately to an L2 Firewall Expert, bypassing standard L1 triage if L1 historic FCR for this ticket type is <5%.
*   **Expected Benefit:** Drastic reduction in "ping-ponging" (reassignments), immediate drop in P2/P3 resolution times.

#### Strategy 2: Dynamic, Workload-Aware Assignment Logic
*   **The Issue Addressed:** Uneven workload distribution and excessive wait times in specialized queues (like Agent B12 waiting 25 minutes to start INC-1001).
*   **Process Mining Insight Used:** Analysis of active concurrent tickets per agent and standard activity execution times.
*   **Data Required:** Real-time agent status (Online/Offline/Busy), current queue depth, and estimated effort per ticket (derived from historical process mining averages).
*   **How it Works:** When dispatching a ticket, the system evaluates all agents with the required skill. It calculates a "readiness score" based on their current active tickets multiplied by the historical average completion time of those tickets. The ticket goes to the agent with the lowest projected time-to-availability, not just the next person in line.
*   **Expected Benefit:** Smoother workload distribution, elimination of silent bottlenecks, and improved agent morale.

#### Strategy 3: Time-Bound Escalation and "Shift-Left" Refinement
*   **The Issue Addressed:** L1 agents wasting SLA time on complex tickets; L2s doing basic tasks.
*   **Process Mining Insight Used:** Identifying specific ticket sub-categories where L1 spends >20 minutes but escalates 80% of the time, AND identifying L2 tickets resolved in <5 minutes.
*   **Data Required:** Activity durations mapped against categories and resolution outcomes.
*   **How it Works:** 
    1.  **Hard Escalation Triggers:** Implement system rules where, based on ticket category, if L1 has not resolved it in 'X' minutes (determined by process mining data), the ticket auto-escalates. L1 is instructed to stop work immediately.
    2.  **Shift-Left:** Take the L2 tasks historically resolved in <5 minutes, create knowledge base articles and scripts, and force-route them to L1.
*   **Expected Benefit:** Optimization of expensive L2/L3 resource time and preservation of SLA buffers for complex tickets.
