# Analysis of the Underfitting Process Tree Model

## 1. Issues with the Current Model

The current model uses a **PARALLEL (AND)** operator for the block containing `Create_Purchase_Order`, `Receive_Goods`, `Receive_Invoice`, `Match_Invoice_to_PO`, and `Pay_Invoice`. This is problematic because:

- **Parallel execution allows any interleaving** of these activities, which violates real-world business logic.
- **Unwanted behaviors allowed by the current model include:**
  - `Pay_Invoice` occurring before `Receive_Invoice` (paying before knowing the invoice details — impossible in practice).
  - `Match_Invoice_to_PO` occurring before `Create_Purchase_Order` (cannot match to a PO that doesn't exist).
  - `Match_Invoice_to_PO` occurring before `Receive_Invoice` (cannot match what hasn't been received).
  - `Receive_Goods` occurring before `Create_Purchase_Order` (cannot receive goods without a PO).
  - `Pay_Invoice` occurring before `Match_Invoice_to_PO` (payment should only happen after matching verification).

**Strict sequential dependencies that must be enforced:**
- `Create_Purchase_Order` must precede both `Receive_Goods` and `Receive_Invoice`.
- `Match_Invoice_to_PO` requires both `Receive_Goods` and `Receive_Invoice` to have completed.
- `Pay_Invoice` must occur after `Match_Invoice_to_PO`.

**Legitimate concurrency:**
- `Receive_Goods` and `Receive_Invoice` can realistically occur in any order (goods may arrive before or after the invoice). These two activities are genuinely parallel.

## 2. Corrected Process Tree Model

```python
from pm4py.objects.process_tree.obj import Operator, ProcessTree

# Define activities
create_req = ProcessTree(label="Create_Purchase_Requisition")
approve_req = ProcessTree(label="Approve_Purchase_Requisition")
create_po = ProcessTree(label="Create_Purchase_Order")
receive_goods = ProcessTree(label="Receive_Goods")
receive_invoice = ProcessTree(label="Receive_Invoice")
match_invoice = ProcessTree(label="Match_Invoice_to_PO")
pay_invoice = ProcessTree(label="Pay_Invoice")
close_case = ProcessTree(label="Close_Case")

# Parallel block only for Receive_Goods and Receive_Invoice
# (their relative order is not strictly fixed in real life)
receive_block = ProcessTree(operator=Operator.PARALLEL, children=[
    receive_goods,
    receive_invoice
])

# Root: strict sequence enforcing Procure-to-Pay order
root = ProcessTree(operator=Operator.SEQUENCE, children=[
    create_req,
    approve_req,
    create_po,
    receive_block,      # Goods and Invoice can be received in any order
    match_invoice,
    pay_invoice,
    close_case
])
```

### Structure Overview

```
SEQUENCE
 Create_Purchase_Requisition
 Approve_Purchase_Requisition
 Create_Purchase_Order
 PARALLEL
    Receive_Goods
    Receive_Invoice
 Match_Invoice_to_PO
 Pay_Invoice
 Close_Case
```

## 3. How the Updated Model Ensures Conformance

The corrected process tree enforces the domain logic through several refinements:

1. **Top-level SEQUENCE operator**: Forces the major phases of the process to occur in strict order. The requisition must be created and approved *before* any PO can be created, and the case can only be closed at the very end.

2. **`Create_Purchase_Order` placed before the receive block**: Guarantees that a PO exists before goods or an invoice can be received, modeling the real-world prerequisite that suppliers only deliver/invoice against an existing PO.

3. **Narrowly-scoped PARALLEL block for `Receive_Goods` and `Receive_Invoice`**: This reflects genuine real-world flexibility — goods may physically arrive before or after the invoice document. Parallelism is allowed only where it makes business sense.

4. **`Match_Invoice_to_PO` placed after the receive block**: Ensures three-way matching (PO + Goods Receipt + Invoice) only occurs once all three artifacts exist. This is a critical internal control in procurement.

5. **`Pay_Invoice` placed after `Match_Invoice_to_PO`**: Enforces the control principle that payment only occurs after invoice verification, preventing fraudulent or erroneous payments.

6. **`Close_Case` as the final step**: Guarantees that the case is only closed after all operational activities (including payment) are complete.

### Prevented Unwanted Behaviors

| Unwanted Behavior (previously allowed) | Prevented by |
|---|---|
| Paying before receiving invoice | Sequence enforces `Receive_Invoice`  `Match_Invoice_to_PO`  `Pay_Invoice` |
| Matching before PO exists | Sequence places `Create_Purchase_Order` before match |
| Receiving goods before PO creation | `Create_Purchase_Order` precedes the receive block |
| Closing case before payment | `Pay_Invoice` precedes `Close_Case` in sequence |
| Matching before goods received | Match occurs after the parallel receive block completes |

The resulting model is **precise** (rejects traces that violate P2P logic) while remaining **fit** for all legitimate execution variants of the process.
