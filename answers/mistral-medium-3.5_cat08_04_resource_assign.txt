# **Process Mining-Driven Optimization of Resource Assignment in TechSolve Solutions’ IT Service Desk**

## **1. Analyzing Resource Behavior and Assignment Patterns**

### **1.1 Performance and Behavior Metrics**
To assess agent and tier performance, we extract the following metrics from the event log:

- **Workload Distribution:**
  - Number of tickets assigned per agent/tier (e.g., L1 agents handle 60% of tickets, L2 30%, L3 10%).
  - Average concurrent ticket load per agent (e.g., Agent B12 handles 5 tickets simultaneously, while Agent A05 handles 2).
  - **Metric:** *Workload Imbalance Ratio* = (Max tickets/agent) / (Avg tickets/agent).

- **Processing Efficiency:**
  - **Mean/median resolution time per agent/tier** (e.g., L1 resolves P3 tickets in 30 mins, L2 in 2 hrs).
  - **First-Call Resolution (FCR) Rate for L1:** % of tickets resolved without escalation (target: >70%).
  - **Escalation Rate:** % of tickets escalated from L1L2 or L2L3 (high rates may indicate skill gaps or poor initial assignment).
  - **Reassignment Frequency:** Average number of reassignments per ticket (e.g., 1.5 reassignments for P2 Network tickets).

- **Skill Utilization:**
  - **Skill Match Rate:** % of tickets where the assigned agent’s skills *fully* match the required skill (e.g., only 40% of App-CRM tickets are assigned to agents with App-CRM skills).
  - **Overqualification Rate:** % of tickets handled by agents with *higher* skills than required (e.g., L3 agents resolving L1-level tasks).
  - **Skill Demand vs. Supply:** Heatmap of required skills (e.g., "Networking-Firewall" is in 25% of tickets but only 10% of agents have it).

### **1.2 Process Mining Techniques for Assignment Patterns**
- **Resource Interaction Analysis (Social Network Mining):**
  - Construct a **handover graph** where nodes are agents and edges represent ticket transfers (escalations/reassignments).
  - **Key Insight:** Identify "hub" agents (e.g., Agent B12 receives 40% of L2 escalations) or frequent reassignment loops (e.g., L1L2L1 for Software-OS tickets).
  - Compare with **intended assignment logic** (e.g., round-robin within L1). Deviations (e.g., 30% of L1 tickets go directly to L2) suggest rule bypassing.

- **Role Discovery:**
  - Use clustering (e.g., k-means on agent skill vectors) to detect *de facto* roles (e.g., some L2 agents specialize in Security-IAM, others in Database-SQL).
  - **Gap Analysis:** If the mined roles don’t align with official tier definitions, the tier system may need restructuring.

- **Variant Analysis:**
  - Group tickets by **assignment path** (e.g., "L1L2Resolved" vs. "L1L2L3ReassignedResolved").
  - **Insight:** Paths with >2 reassignments likely indicate poor initial matching.

- **Conformance Checking:**
  - Compare the *actual* assignment process (from logs) against the *ideal* process (e.g., "P1 tickets should go to L3 within 5 mins").
  - **Example:** If 20% of P2 tickets are assigned to L1 first (despite policy), compliance is low.

### **1.3 Skill Utilization Analysis**
- **Skill Mismatch Detection:**
  - For each ticket, check if the assigned agent’s skills (from `Agent Skills` column) include the `Required Skill`.
  - **Metric:** *Skill Mismatch Rate* = (Tickets with partial/no skill match) / (Total tickets).
  - **Example:** 60% of "Database-SQL" tickets are assigned to agents without SQL skills, leading to delays.

- **Specialist Overuse:**
  - Track time L2/L3 agents spend on L1-level tasks (e.g., password resets).
  - **Metric:** *Efficiency Loss* = (Hours spent by L2/L3 on L1 tasks) × (L2 hourly cost - L1 hourly cost).

- **Skill Bottlenecks:**
  - Identify skills with high demand but low supply (e.g., "Security-IAM" has a 3:1 ticket-to-agent ratio).
  - **Action:** Prioritize training or hiring for these skills.

## **2. Identifying Resource-Related Bottlenecks and Issues**

### **2.1 Key Problems and Quantification**
| **Issue**                          | **Detection Method**                                                                 | **Quantified Impact**                                                                 |
|-------------------------------------|--------------------------------------------------------------------------------------|--------------------------------------------------------------------------------------|
| **Skill Shortages**                 | Filter tickets where `Required Skill` is rare (e.g., <5 agents have it).            | 40% of "Networking-Firewall" tickets wait >2 hrs for assignment due to lack of L2 agents. |
| **Excessive Reassignments**        | Count tickets with >1 reassignment in the log.                                       | Each reassignment adds avg. 1.5 hrs to resolution time (SLA breach risk +30%).     |
| **Incorrect Initial Assignments**  | Compare `Ticket Category`/`Required Skill` with assigned `Agent Tier/Skills`.        | 25% of P2 tickets initially assigned to L1 are escalated within 10 mins.            |
| **Overloaded Agents**              | Agents with >80th percentile workload (e.g., Agent B12 handles 20% more tickets).   | Agent B12’s tickets have 2× longer resolution times than peers.                     |
| **Underutilized Agents**           | Agents with <20th percentile workload (e.g., Agent A07 handles 3 tickets/day).       | 15% of L1 agents are idle >50% of the time.                                          |
| **SLA Breach Correlation**          | Link SLA breaches to assignment paths (e.g., tickets escalated >1 time).           | 70% of SLA breaches occur after 2 reassignments.                                   |

### **2.2 Root Causes**
- **Round-Robin Flaws:** Ignores skills/workload, leading to mismatches (e.g., a Network ticket goes to an App specialist).
- **Poor Ticket Classification:** 30% of tickets are miscategorized (e.g., "Hardware" labeled as "Software"), causing wrong assignments.
- **Static Tiering:** L1 agents lack training to handle 40% of P3 tickets, forcing escalations.
- **No Real-Time Visibility:** Dispatchers assign tickets without knowing current agent workload/skills.
- **Skill Profile Gaps:** Agent skills in the system are outdated (e.g., Agent B12’s "DB-SQL" skill isn’t recorded).

## **3. Root Cause Analysis for Assignment Inefficiencies**

### **3.1 Deep Dive into Causes**
1. **Assignment Rule Deficiencies:**
   - Current rules (round-robin + manual escalation) don’t account for:
     - **Skill matching** (e.g., App-CRM tickets go to any L2, not just App-CRM experts).
     - **Workload balancing** (e.g., Agent B12 is overloaded because they’re the only Network-Firewall L2).
     - **Ticket complexity** (e.g., P2 tickets with "Database-SQL" are treated like generic P2s).

2. **Data Quality Issues:**
   - **Inaccurate Skill Tags:** 20% of agents’ skills are missing or incorrect (e.g., Agent A05 can handle basic SQL but isn’t tagged).
   - **Poor Ticket Categorization:** Natural Language Processing (NLP) on ticket descriptions reveals 35% are misclassified.

3. **Process Gaps:**
   - **No Predictive Routing:** Dispatchers manually assign tickets without AI/ML support.
   - **Lack of Feedback Loops:** No tracking of why tickets are reassigned (e.g., "wrong skill" vs. "agent unavailable").

4. **Organizational Factors:**
   - **Training Gaps:** L1 agents lack skills for 40% of P3 tickets, leading to unnecessary escalations.
   - **Incentive Misalignment:** L2/L3 agents are rewarded for resolving tickets quickly, not for mentoring L1.

### **3.2 Leveraging Process Mining for Root Causes**
- **Variant Analysis:**
  - Compare **smooth cases** (resolved in 1 assignment) vs. **chaotic cases** (multiple reassignments).
  - **Example Finding:** 80% of chaotic cases involve tickets where `Required Skill` wasn’t in the initial agent’s profile.

- **Decision Mining:**
  - Extract decision rules from logs (e.g., "IF Ticket.Priority = P2 AND Category = Network, THEN assign to L2").
  - **Insight:** Current rules are ad-hoc; 50% of P2 Network tickets go to L1 first (violating best practice).

- **Bottleneck Analysis:**
  - Use **performance spectrum analysis** to identify where delays occur (e.g., 60% of delays happen at L2 assignment for Security tickets).

## **4. Data-Driven Resource Assignment Strategies**

### **Strategy 1: Skill-Based Routing with Proficiency Weighting**
- **Issue Addressed:** Skill mismatches and specialist overuse.
- **How It Works:**
  - Assign tickets to agents whose skills *exactly match* the `Required Skill`, prioritizing:
    1. **Least loaded agent** with the skill.
    2. **Highest proficiency** (e.g., Agent B12 has "App-CRM: Expert" vs. Agent B15’s "App-CRM: Intermediate").
  - Fallback: If no exact match, assign to the closest skill (e.g., "Database-SQL"  agent with "DB-NoSQL").
- **Data Required:**
  - Updated `Agent Skills` (with proficiency levels).
  - Real-time `Agent Workload` (current ticket count).
- **Expected Benefits:**
  - Reduce skill mismatch rate from 60%  10%.
  - Decrease reassignment-induced delays by 40%.

### **Strategy 2: Predictive Assignment Using Historical Patterns**
- **Issue Addressed:** Poor initial categorization and reactive assignments.
- **How It Works:**
  - Train a **classifier** (e.g., Random Forest) on historical tickets to predict:
    - Required skill (from `Ticket Description`, `Category`, `Priority`).
    - Optimal tier (L1/L2/L3) based on past resolution paths.
  - **Example:** A ticket with "CRM login issue" + P3 is predicted to need "App-CRM" and L2.
  - Integrate with **NLP** to extract keywords from ticket descriptions (e.g., "firewall timeout"  "Networking-Firewall").
- **Data Required:**
  - Historical ticket attributes (`Description`, `Category`, `Required Skill`, `Resolution Path`).
  - Agent performance data (e.g., resolution time by skill).
- **Expected Benefits:**
  - Improve first-assignment accuracy by 30%.
  - Reduce L1L2 escalations by 25%.

### **Strategy 3: Dynamic Workload-Aware Assignment with Tier Fluidity**
- **Issue Addressed:** Static tiers and workload imbalance.
- **How It Works:**
  - **Real-Time Workload Balancing:**
    - Assign tickets to the *least busy* agent with the required skill.
    - Use a **queueing algorithm** (e.g., shortest job first for L1, priority-based for L2/L3).
  - **Tier Fluidity:**
    - Allow L1 agents to handle *some* L2 tasks if:
      - They have the required skill (e.g., Agent A05 has "Basic-Troubleshoot" + "App-CRM").
      - L2 queue is backlogged (>10 tickets).
    - Automatically promote/demote agents based on demand (e.g., during a Network outage, pull L3 agents to L2).
- **Data Required:**
  - Real-time `Agent Availability` (active/inactive, current load).
  - `Queue Lengths` per tier/skill.
  - `Agent Skill Profiles` (with cross-tier capabilities).
- **Expected Benefits:**
  - Reduce workload imbalance ratio from 2.5  1.2.
  - Cut resolution time for P2 tickets by 20%.

## **5. Simulation, Implementation, and Monitoring**

- **Approach:**

    - Current state: Round-robin + manual escalations.
    - Proposed state: Skill-based + predictive + dynamic workload routing.

  - **Outputs:**
