# POWL Models for the GlobalTeq Solutions Hiring Process

## Model 1: POWL Model with Potential Unfairness

This model includes an XOR branch after the skill assessment: candidates either go through the standard `CulturalFitCheck` or through the `CommunityAffiliationCheck`, which introduces the subtle bias point.

```python
import pm4py
from pm4py.objects.powl.obj import StrictPartialOrder, OperatorPOWL, Transition, SilentTransition
from pm4py.objects.process_tree.obj import Operator

# --- Activities ---
receive_application   = Transition(label="ReceiveApplication")
data_check            = Transition(label="DataCompletenessCheck")
request_more_info     = Transition(label="RequestMoreInfo")
skill_assessment      = Transition(label="SkillAssessment")
cultural_fit_check    = Transition(label="CulturalFitCheck")
community_affiliation = Transition(label="CommunityAffiliationCheck")  # bias point
managerial_review     = Transition(label="ManagerialReview")
final_decision        = Transition(label="FinalDecision")

# --- Loop for data completeness ---
# Execute DataCompletenessCheck; either exit, or do RequestMoreInfo
# and check again, until the application data is complete.
data_loop = OperatorPOWL(operator=Operator.LOOP,
                         children=[data_check, request_more_info])

# --- XOR branch: standard cultural fit OR affiliation-based evaluation ---
# This is where unfairness can be introduced: the CommunityAffiliationCheck
# branch grants a subtle score uplift to affiliated/local applicants.
cultural_xor = OperatorPOWL(operator=Operator.XOR,
                            children=[cultural_fit_check, community_affiliation])

# --- Root partial order (sequential workflow) ---
root_unfair = StrictPartialOrder(nodes=[receive_application,
                                        data_loop,
                                        skill_assessment,
                                        cultural_xor,
                                        managerial_review,
                                        final_decision])

root_unfair.order.add_edge(receive_application, data_loop)
root_unfair.order.add_edge(data_loop, skill_assessment)
root_unfair.order.add_edge(skill_assessment, cultural_xor)
root_unfair.order.add_edge(cultural_xor, managerial_review)
root_unfair.order.add_edge(managerial_review, final_decision)

pm4py.view_powl(root_unfair, format="svg")
```

**Structure summary:**

```
ReceiveApplication
  --> * ( DataCompletenessCheck, RequestMoreInfo )
  --> SkillAssessment
  --> X ( CulturalFitCheck, CommunityAffiliationCheck )   <-- BIAS POINT
  --> ManagerialReview
  --> FinalDecision
```

The XOR node `X ( CulturalFitCheck, CommunityAffiliationCheck )` is where the subtle unfairness arises: applicants routed through the `CommunityAffiliationCheck` path (e.g., members of the "Global Business Leaders Club" or local residents) receive an implicit advantage that other applicants cannot obtain.

## Model 2: POWL Model Without Unfairness

This model removes the discriminatory XOR branch entirely. Every applicant—regardless of affiliations—passes through the same single `CulturalFitCheck` activity, ensuring uniform evaluation.

```python
import pm4py
from pm4py.objects.powl.obj import StrictPartialOrder, OperatorPOWL, Transition, SilentTransition
from pm4py.objects.process_tree.obj import Operator

# --- Activities ---
receive_application = Transition(label="ReceiveApplication")
data_check          = Transition(label="DataCompletenessCheck")
request_more_info   = Transition(label="RequestMoreInfo")
skill_assessment    = Transition(label="SkillAssessment")
cultural_fit_check  = Transition(label="CulturalFitCheck")  # same path for everyone
managerial_review   = Transition(label="ManagerialReview")
final_decision      = Transition(label="FinalDecision")

# --- Loop for data completeness (kept, as it is fairness-neutral) ---
data_loop = OperatorPOWL(operator=Operator.LOOP,
                         children=[data_check, request_more_info])

# --- Root partial order (sequential workflow, no XOR for cultural fit) ---
root_fair = StrictPartialOrder(nodes=[receive_application,
                                      data_loop,
                                      skill_assessment,
                                      cultural_fit_check,
                                      managerial_review,
                                      final_decision])

root_fair.order.add_edge(receive_application, data_loop)
root_fair.order.add_edge(data_loop, skill_assessment)
root_fair.order.add_edge(skill_assessment, cultural_fit_check)
root_fair.order.add_edge(cultural_fit_check, managerial_review)
root_fair.order.add_edge(managerial_review, final_decision)

pm4py.view_powl(root_fair, format="svg")
```

**Structure summary:**

```
ReceiveApplication
  --> * ( DataCompletenessCheck, RequestMoreInfo )
  --> CulturalFitCheck        <-- single, uniform path for ALL applicants
  ... (after SkillAssessment)
  --> ManagerialReview
  --> FinalDecision
```

i.e.:

```
ReceiveApplication --> * ( DataCompletenessCheck, RequestMoreInfo )
                   --> SkillAssessment
                   --> CulturalFitCheck
                   --> ManagerialReview
                   --> FinalDecision
```

## Key Differences Between the Two Models

| Aspect | Model 1 (Unfair) | Model 2 (Fair) |
|---|---|---|
| Cultural evaluation | XOR: `X(CulturalFitCheck, CommunityAffiliationCheck)` | Single activity: `CulturalFitCheck` |
| Affiliation-based routing | Yes — affiliated/local applicants take a privileged path | No — affiliation data does not alter the control flow |
| Data completeness loop | `*(DataCompletenessCheck, RequestMoreInfo)` (neutral, retained) | Same (neutral, retained) |
| Bias surface | The XOR branch allows implicit score uplifts that bias the downstream `ManagerialReview` | No alternative evaluation path exists, so the review receives uniformly produced inputs |

The loop construct is retained in both models because re-requesting missing data applies equally to all applicants and is not a source of bias. The unfairness in Model 1 is structural: the mere existence of the alternative `CommunityAffiliationCheck` path encodes preferential treatment into the process design itself, which Model 2 eliminates by collapsing the XOR into a single, mandatory evaluation step.
