Here’s a clean DuckDB SQL query that does exactly that — keep only events from cases with ≤6 total events:

### Query
```sql
WITH case_counts AS (
    SELECT 
        case_id,
        COUNT(*) AS event_cnt
    FROM event_log
    GROUP BY case_id
    HAVING COUNT(*) <= 6
)
SELECT e.*
FROM event_log AS e
JOIN case_counts AS c
    USING (case_id);
```

### Why this works
- **Group + count**: The CTE `case_counts` groups by `case_id` and counts rows per case.
- **Filter early**: `HAVING COUNT(*) <= 6` drops any `case_id` with more than 6 events before we touch the main table again.
- **Return full events**: Joining back to `event_log` gives you every column for the remaining cases, not just the counts.

### Alternative using a window function
If you prefer a single scan without a self-join, DuckDB supports `QUALIFY`:
```sql
SELECT *
FROM event_log
QUALIFY COUNT(*) OVER (PARTITION BY case_id) <= 6;
```
This adds a window count per row, then `QUALIFY` filters rows where that count is ≤6. Same result, often faster on large tables because it avoids the join.

Both are standard DuckDB syntax and will return all `case_id`, `activity`, `timestamp`, and any other columns for cases with 6 or fewer events.
