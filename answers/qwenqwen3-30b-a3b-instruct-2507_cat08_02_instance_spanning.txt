## **1. Identifying Instance-Spanning Constraints and Their Impact**

To effectively address the complex dependencies between orders, we begin by **formally identifying and quantifying the impact** of each instance-spanning constraint using process mining techniques applied to the event log.

### **A. Constraint Identification via Process Mining Techniques**

We leverage **process discovery, conformance checking, and performance analysis** (using tools like ProM, Celonis, or UiPath Process Mining) on the event log to extract insights:

- **Shared Cold-Packing Stations (Resource Contention):**
  - **Technique:** Use *resource-based process models* and *resource utilization analysis*.
  - **Identification:** Create a resource-centric process model where each station (e.g., Station C2) is a node. Identify bottlenecks by analyzing **resource waiting times**, **resource utilization rates**, and **concurrent activity instances**.
  - **Quantification:** For each order requiring cold-packing, compute:
    - *Wait time before starting packing:* `Start Packing Timestamp - Complete Item Picking Timestamp`
    - *Total wait time due to resource contention:* Difference between actual start time and earliest possible start time (assuming no competition).
    - *Queue length at cold-packing stations:* Count of orders waiting for cold-packing at any given time.

- **Batching for Shipping (Inter-Order Delay):**
  - **Technique:** Apply *temporal analysis* and *batch detection algorithms* (e.g., time-window-based clustering, region-based grouping).
  - **Identification:** Group orders by `Destination Region` and analyze the time between the **completion of the last activity before labeling** and **Shipping Label Generation**.
  - **Quantification:** Compute:
    - *Batching delay:* `Shipping Label Gen. Timestamp - Max(Complete Quality Check Timestamp for all orders in batch)`
    - *Average batch size per region:* To assess batching efficiency.
    - *Wait time due to incomplete batch:* For orders that finish earlier but wait for others to join the batch.

- **Priority Handling (Express Order Interruption):**
  - **Technique:** Use *case correlation analysis* and *priority impact tracing*.
  - **Identification:** Flag all Express orders. For each such order, examine the timeline of standard orders currently using the same resource (e.g., station or staff).
  - **Quantification:**
    - *Preemption delay:* Duration between when a standard order begins an activity and when it is paused due to an express order.
    - *Total time lost per standard order due to preemption:* Sum of all pause durations.
    - *Express order acceleration:* `End-to-end time of express order - average end-to-end time of standard orders`.

- **Hazardous Material Limits (Regulatory Compliance):**
  - **Technique:** Apply *concurrent activity analysis* and *constraint validation*.
  - **Identification:** Track all orders with `Hazardous Material = TRUE` and their timestamps in `Packing` and `Quality Check` activities.
  - **Quantification:**
    - *Simultaneous hazardous orders:* Count how many hazardous orders are active in Packing or QC at the same time.
    - *Violation detection:* Flag any instance where >10 hazardous orders are active simultaneously.
    - *Waiting time due to limit:* Time an order waits in queue until the number of active hazardous orders drops below 10.

### **B. Differentiating Within-Instance vs. Between-Instance Waiting Time**

This distinction is critical for accurate diagnosis.

| **Factor Type** | **Definition** | **Detection Method** |
|------------------|----------------|------------------------|
| **Within-Instance Waiting** | Delays *within* a single order (e.g., picker idle between picking items) | Use *process duration analysis* per case; compute idle time between consecutive activities (e.g., `Packing Start - Packing Complete`). |
| **Between-Instance Waiting** | Delays caused by *other orders* (e.g., waiting for cold-packing station occupied by another order) | Use *resource contention modeling*; calculate the difference between actual start time and the earliest possible start time (based on available resources). |

> **Example:** If ORD-5001 starts packing at 10:20:10 but could have started at 10:15:45 (after item picking), the 4.5-minute gap is **between-instance** waiting. If it took 10 minutes to pick items, that’s **within-instance** waiting.

### **C. Key Metrics for Measuring Constraint Impact**

| Constraint | Primary Metric | Secondary Metric |
|----------|----------------|------------------|
| Cold-Packing Contention | Avg. wait time at cold-packing stations | % of cold-packing orders delayed >15 min |
| Batching Delays | Avg. batching delay per order | Batch formation time (min) |
| Priority Handling | Avg. preemption delay per standard order | Express order fulfillment rate vs. SLA |
| Hazardous Material Limits | Max concurrent hazardous orders | % of orders delayed due to limit |

## **2. Analyzing Constraint Interactions**

Understanding interactions is essential to avoid unintended consequences when optimizing one constraint.

### **A. Key Interactions Identified**

| **Interaction** | **Mechanism** | **Risk** |
|------------------|---------------|----------|
| **Express + Cold-Packing** | Express orders require cold-packing and may preempt standard orders, increasing queue and waiting time for *all* cold-packing orders. | Creates a "priority cascade" — more express orders  longer queues for cold-packing  more delays for standard orders. |
| **Batching + Hazardous Materials** | If multiple hazardous orders are destined for the same region, they may be batched together. However, if too many are in the same batch, they may exceed the 10-concurrent limit during packing/QC. | Batching increases risk of violating regulatory limits, potentially causing *additional* delays. |
| **Priority + Batching** | An express order might be delayed in batching if it arrives late, even though it should be expedited. | Batching logic may override priority, defeating the purpose of express handling. |
| **Hazardous + Cold-Packing** | Hazardous materials may require special handling *and* cold-packing. If both are required, the order faces double constraints. | High risk of bottlenecks at both specialized stations and compliance checks. |

### **B. Why Interaction Analysis Matters**

- **Optimization is not additive:** Fixing one constraint may worsen another (e.g., relaxing hazardous limits to reduce delays could violate regulations).
- **Synergies exist:** If we batch hazardous orders *across regions* instead of within regions, we reduce the risk of exceeding the 10-order limit in any one flow.
- **Dynamic trade-offs:** A strategy that prioritizes express orders must consider whether it increases cold-packing congestion or batch formation time.

> **Conclusion:** A holistic, constraint-aware optimization model must simulate these interactions to ensure **net positive impact**.

## **3. Developing Constraint-Aware Optimization Strategies**

We propose **three data-driven, constraint-aware strategies** that directly address the core challenges while respecting interdependencies.

---

### **Strategy 1: Dynamic Resource Allocation with Priority-Aware Scheduling for Cold-Packing Stations**

- **Constraint Addressed:** Shared Cold-Packing Stations + Priority Handling
- **Proposed Changes:**
  - Implement a **priority-based queue** for cold-packing stations:
    - Express orders (with cold-packing)  highest priority.
    - Standard cold-packing orders  second priority.
    - Non-cold-packing orders  lowest priority.
  - Use **real-time resource availability forecasting** (via process mining) to predict cold-packing station occupancy.
  - When an express order arrives, **preempt the lowest-priority standard order** currently using a station, **only if** it is not already in the final packing step.
- **Data/Analysis Leveraged:**
  - Historical data on cold-packing station utilization and express order arrival rates.
  - Predictive model to estimate station availability (e.g., using Markov chains or survival analysis).
- **Expected Outcomes:**
  - Reduce average cold-packing wait time for express orders by 30–50%.
  - Minimize knock-on delays to standard orders by limiting preemption to early-stage tasks.
  - Maintain compliance with hazardous material limits (since cold-packing is separate from hazardous handling).

---

### **Strategy 2: Adaptive Batching with Dynamic Trigger Rules**

- **Constraint Addressed:** Shipping Batching + Hazardous Material Limits
- **Proposed Changes:**
  - Replace static batch formation (e.g., "batch every 30 minutes") with **dynamic batch triggers**:
    - **Trigger 1 (Time + Capacity):** Form batch after 30 minutes **or** when 10 orders are ready, whichever comes first.
    - **Trigger 2 (Safety Check):** Before forming a batch, check:
      - Total number of **hazardous material orders** in the batch  5 (to leave room for others).
      - Total concurrent hazardous orders in facility  10.
    - If a batch would violate limits, delay formation until safe.
  - Allow **express orders to be batched immediately** (with no wait).
- **Data/Analysis Leveraged:**
  - Historical data on batching delays, batch sizes, and hazardous material distribution.
  - Clustering analysis to identify high-density hazardous regions.
- **Expected Outcomes:**
  - Reduce average batching delay by 25%.
  - Prevent violations of the 10-concurrent hazardous limit.
  - Improve express order throughput without compromising compliance.

---

### **Strategy 3: Bottleneck-Aware Order Routing and Process Decoupling**

- **Constraint Addressed:** All constraints (via process redesign)
- **Proposed Changes:**
  - **Decouple Quality Check from Packing** for hazardous materials:
    - Introduce a **pre-QC screening step** *after* item picking (for hazardous orders) to flag issues early.
    - Allow **non-hazardous orders** to proceed to packing while hazardous ones wait for pre-QC validation.
    - This reduces the load on QC staff and prevents hazardous orders from blocking standard ones.
  - **Re-route cold-packing orders dynamically** based on real-time availability:
    - Use a **centralized scheduler** (e.g., rule-based or ML-driven) to assign cold-packing stations based on:
      - Order priority.
      - Expected duration.
      - Current station load.
- **Data/Analysis Leveraged:**
  - Process mining to identify **bottleneck activities** and **resource contention hotspots**.
  - Predictive modeling of activity durations (e.g., using Weibull distribution or regression).
- **Expected Outcomes:**
  - Reduce average order throughput time by 15–20%.
  - Decrease queue lengths at critical stations.
  - Improve compliance by separating high-risk orders early.

## **4. Simulation and Validation**

Before implementation, we use **discrete-event simulation (DES)** to test the strategies.

### **A. Simulation Model Design**

- **Tool:** Use AnyLogic, Simio, or Python-based simulation (e.g., SimPy).
- **Model Components:**
  - **Entities:** Orders (with type, region, priority, hazardous flag).
  - **Resources:** Stations (standard, cold-packing, QC), staff.
  - **Processes:** Each activity with stochastic durations (based on log data).
  - **Constraints:** 
    - Max 5 cold-packing stations.
    - Max 10 concurrent hazardous orders in Packing/QC.
    - Batching logic with dynamic triggers.
    - Preemption rules for express orders.
- **Input Data:** Historical event log used to **calibrate** activity durations, arrival rates, and resource utilization.

### **B. Validation Focus Areas**

| Aspect | Simulation Focus |
|-------|------------------|
| **Resource Contention** | Track queue length at cold-packing stations; measure preemption frequency. |
| **Batching Delays** | Measure average time from last QC completion to label generation. |
| **Priority Interruptions** | Count number of preemptions; calculate average delay per standard order. |
| **Regulatory Limits** | Monitor concurrent hazardous orders; detect violations. |
| **Throughput & SLA Compliance** | Measure % of orders delivered within target time (e.g., 24h). |

### **C. Evaluation Method**

- Run **A/B testing** in simulation:
  - **Baseline:** Current process (as per event log).
  - **Scenario A:** Strategy 1 (Dynamic Resource Allocation).
  - **Scenario B:** Strategy 2 (Adaptive Batching).
  - **Scenario C:** Strategy 1 + 2 + 3 (Full Optimization).
- Compare KPIs: average end-to-end time, batching delay, preemption cost, compliance rate.

> **Outcome:** Select the strategy combination with the best **trade-off** between speed, compliance, and stability.
