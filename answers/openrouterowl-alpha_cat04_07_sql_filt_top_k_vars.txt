# DuckDB Query: Extracting Events from Top K Process Variants

## Understanding the Goal

We want to identify the most common **process variants** (ordered activity sequences) in an event log, pick the **top K** of them, and return **all events** belonging to cases that follow those top variants.

## The Query

```sql
WITH ordered_sequences AS (
    -- Step 1: Build the ordered activity sequence per case_id
    SELECT
        case_id,
        string_agg(activity, ' -> ' ORDER BY timestamp) AS variant
    FROM event_log
    GROUP BY case_id

    -- Alternative if you need true JSON list output:
    -- list(activity ORDER BY timestamp) AS activities_ordered
    -- (uncomment and use accordingly)
),

variant_counts AS (
    -- Step 2: Compute frequency of each variant
    SELECT
        variant,
        count(*) AS case_count
    FROM ordered_sequences
    GROUP BY variant
),

top_k_variants AS (
    -- Step 3: Identify the top K variants by frequency
    SELECT variant
    FROM variant_counts
    ORDER BY case_count DESC
    -- LIMIT K
),

top_k_cases AS (
    -- Step 4: Map top variants back to their original case_ids
    SELECT DISTINCT
        os.case_id
    FROM ordered_sequences os
    WHERE os.variant IN (SELECT variant FROM top_k_variants)
)

-- Step 5: Return all events from the qualifying cases
SELECT
    el.*
FROM event_log el
INNER JOIN top_k_cases tkc
    ON el.case_id = tkc.case_id
ORDER BY el.case_id, el.timestamp;
```

## Step-by-Step Explanation

### 1 `ordered_sequences` — Ordered Activity Aggregation per Case

```sql
SELECT
    case_id,
    string_agg(activity, ' -> ' ORDER BY timestamp) AS variant
FROM event_log
GROUP BY case_id
```

> - `string_agg(... ORDER BY timestamp)` concatenates activities **in chronological order**, separated by `->`, forming a variant fingerprint.
> - `GROUP BY case_id` ensures we get **one row per case**.

| case_id | variant                                   |
|---------|-------------------------------------------|
| A       | Register -> Approve -> Ship               |
| B       | Register -> Reject                         |
| C       | Register -> Approve -> Ship               |

---

### 2 `variant_counts` — Frequency of Each Variant

```sql
SELECT variant, count(*) AS case_count
FROM ordered_sequences
GROUP BY variant
```

| variant                          | case_count |
|----------------------------------|------------|
| Register -> Approve -> Ship      | 2          |
| Register -> Reject               | 1          |

---

### 3 `top_k_variants` — Select the Top K

```sql
SELECT variant
FROM variant_counts
ORDER BY case_count DESC
LIMIT K
> K = 1 keeps the most frequent variant: "Register -> Approve -> Ship"
```

---

### 4 `top_k_cases` — Trace Back to Original Case IDs

```sql
SELECT DISTINCT os.case_id
FROM ordered_sequences os
WHERE os.variant IN (SELECT variant FROM top_k_variants)
> Returns case_ids A and C (excluding B)
```

---

### 5 Final SELECT — Return All Matching Events

```sql
SELECT el.*
FROM event_log el
INNER JOIN top_k_cases tkc ON el.case_id = tkc.case_id
ORDER BY el.case_id, el.timestamp
```

## Notes & Alternatives

| Concern | Approach |
|---|---|
| **Sequence extraction** | Uses `string_agg` with `ORDER BY timestamp`. For JSON-list style, replace with `list(activity ORDER BY timestamp)` and use `IN` with a subquery on UNNEST'd lists. |
| **Tie-breaking at rank K** | `LIMIT K` with `ORDER BY case_count DESC` naturally handles ties by keeping arbitrary ones—add `variant` to ORDER BY to get deterministic results when there's a tie at the Kth position. |
| **Performance** | All CTEs are inlined; DuckDB optimizes pushdown. For giant logs, consider materializing `ordered_sequences` as |

 a temp table.> **Case B is entirely excluded** because its variant ("Register -> Reject") is not among the top K=1 variants. All events for cases A and C are returned in full.
