## Explanation of Constraints

The Log Skeleton captures the following logical relationships in the order fulfillment process:

**Equivalence Constraints**: These pairs of activities must occur the same number of times in any given case. For instance, if an order requires a quality check, it must also include a reservation; if a shipping label is generated, the order must also be packed.

**Always-Before Constraints**: These define the mandatory sequence dependencies. For example, picking items can only occur after reservation, and invoicing can only happen after dispatch. The Receive Order activity must precede all other activities in the process.

**Always-After Constraints**: These capture the reverse dependencies, ensuring that each activity is followed by its logical successors. For instance, once an order is dispatched, both invoicing and payment recording must eventually occur.

**Never-Together Constraints**: These prevent illogical combinations within the same case. For example, the initial order reception cannot coexist with the final dispatch, and picking activities cannot overlap with invoicing.

**Activity Frequency Constraints**: All activities in this process occur exactly once per order, reflecting the straightforward nature of a single order fulfillment cycle without iterations or conditional steps.

**Directly-Follows Constraints**: These specify immediate succession relationships between activities, representing the standard workflow where each activity triggers its direct successor without intervening activities.
