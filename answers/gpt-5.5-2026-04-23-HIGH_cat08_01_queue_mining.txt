## 1. Queue Identification and Characterization

### 1.1 Preparing the event log for queue mining

The clinicĺs event log contains both **START** and **COMPLETE** timestamps for each activity. The first step is to transform the log into an activity-instance format, where each activity occurrence has:

- Case ID / Visit ID
- Activity name
- Start timestamp
- Completion timestamp
- Resource, if available
- Patient attributes: patient type, urgency, specialty, appointment type, etc.

For each visit, activities are then ordered chronologically. The key distinction is:

- **Service time / processing time** = time spent actually performing an activity  
  Example:  
  `Registration service time = Registration COMPLETE - Registration START`

- **Waiting time / queue time** = time between becoming ready for the next activity and actually starting that next activity  
  Example:  
  `Wait for Nurse Assessment = Nurse Assessment START - Registration COMPLETE`

So, for a simple sequential path:

```text
Registration COMPLETE  Nurse Assessment START
```

the queue time is:

```text
Queue time before Nurse Assessment =
Nurse Assessment START - Registration COMPLETE
```

Similarly:

```text
Nurse Assessment COMPLETE  Doctor Consultation START
Doctor Consultation COMPLETE  ECG Test START
ECG Test COMPLETE  Check-out START
```

Each of those gaps represents a potential queue.

For the first activity, such as Registration, the waiting time can only be calculated if the clinic has an **arrival timestamp**, **check-in timestamp**, or **scheduled appointment timestamp**. If the first observed event is `Registration START`, then waiting before registration is not directly observable unless another timestamp is added.

### 1.2 Defining queues

A queue can be defined in two complementary ways:

#### A. Transition-based queue

This looks at waiting between two specific activities:

```text
Registration  Nurse Assessment
Nurse Assessment  Doctor Consultation
Doctor Consultation  ECG Test
ECG Test  Check-out
```

This is useful for identifying delays in handovers between process steps.

#### B. Activity-based queue

This looks at the wait before a target activity, regardless of the previous activity:

```text
Wait before Doctor Consultation
Wait before Blood Test
Wait before X-Ray
Wait before Check-out
```

This is useful when several paths lead to the same activity.

For example, patients may reach `Doctor Consultation` after:

- Registration only
- Nurse Assessment
- Diagnostic test
- Specialist Review

Aggregating all waits before Doctor Consultation helps determine whether the doctor stage itself is a bottleneck.

### 1.3 Handling more complex cases

In a multi-specialty clinic, not all flows are strictly sequential. Some activities may happen in parallel or be optional. For example:

```text
Blood Test and ECG may both be ordered after Doctor Consultation.
```

In those cases, the queue time should be calculated based on when the activity became enabled.

If an activity requires multiple prior steps, the enablement time is usually:

```text
Enablement time = latest completion time among required predecessor activities
```

Then:

```text
Queue time = Activity START - Enablement time
```

If the value is negative, that may indicate:

- Parallel execution
- Data quality issues
- Incorrect timestamps
- The activity began before the assumed predecessor was actually required

Those cases should be flagged and reviewed rather than blindly included.

### 1.4 Core queue metrics to calculate

For each queue, the clinic should calculate both central tendency and tail-risk measures. Important metrics include:

| Metric | Purpose |
|---|---|
| Average waiting time | Overall burden of the queue |
| Median waiting time | Typical patient experience, less affected by outliers |
| 75th / 90th / 95th percentile wait | Captures long-wait experience and patient dissatisfaction risk |
| Maximum waiting time | Identifies extreme cases |
| Standard deviation / variability | Shows consistency or instability of the queue |
| Number of patients entering the queue | Measures queue frequency |
| Percentage of cases experiencing the queue | Shows how common the queue is |
| Total waiting time burden | Sum of all waiting time in that queue |
| SLA violation rate | Percentage exceeding a defined threshold, e.g. wait > 30 minutes |
| Queue length over time | Number of patients waiting at different times of day |
| Time-of-day and day-of-week patterns | Identifies peak congestion periods |
| Wait by patient type | New vs. follow-up, urgent vs. normal, specialty-specific |
| Wait by resource | Wait associated with specific doctors, rooms, nurses, equipment |

The clinic should not only look at average waits. A queue with an average wait of 15 minutes but a 90th percentile of 75 minutes may be a major satisfaction problem because a significant minority of patients experience very long waits.

### 1.5 Reconstructing queue length over time

Using event data, approximate queue lengths can be reconstructed.

For a queue before a given activity:

- Patient enters the queue at the previous activityĺs completion time.
- Patient leaves the queue at the next activityĺs start time.

For example:

```text
Enter queue for Doctor Consultation = Nurse Assessment COMPLETE
Leave queue for Doctor Consultation = Doctor Consultation START
```

By adding `+1` at queue entry and `-1` at queue exit, the clinic can estimate how many patients were waiting at any point in time.

This is valuable because two queues may have the same average waiting time but very different operational causes. One may be consistently moderate, while another may have short periods of extreme congestion.

### 1.6 Identifying the most critical queues

The most critical queues should be selected using a weighted, data-driven approach. A queue should be prioritized if it has one or more of the following characteristics:

1. **High total waiting burden**
   - Example: The queue before Doctor Consultation contributes 40% of total patient waiting time across all visits.

2. **High frequency**
   - Example: 85% of all patients experience a wait before Nurse Assessment.

3. **Long tail waits**
   - Example: Median wait is 18 minutes, but 90th percentile wait is 70 minutes.

4. **High SLA violation rate**
   - Example: 35% of patients wait more than 30 minutes before seeing a doctor.

5. **Strong impact on total visit duration**
   - Queues that are highly correlated with long total visit times should be prioritized.

6. **Impact on clinically sensitive groups**
   - Urgent patients, elderly patients, patients needing time-sensitive diagnostics, or new patients with complex evaluations.

7. **High dissatisfaction association**
   - If patient satisfaction survey data is available, link survey responses to visit logs and identify which waits most strongly predict poor ratings.

A practical prioritization score could combine:

```text
Criticality Score =
Total waiting burden
+ SLA violation rate
+ 90th percentile wait
+ number of affected patients
+ impact on urgent/high-priority patients
+ impact on total visit duration
```

The output should be a ranked list such as:

1. Wait before Doctor Consultation - Cardiology
2. Wait before ECG Test
3. Wait before Check-out
4. Wait before Nurse Assessment
5. Wait before Registration

The exact ranking should come from the six-month event log rather than assumption.

## 2. Root Cause Analysis

Identifying a queue is only the first step. The more important question is why that queue occurs. Process mining can help distinguish between capacity problems, scheduling problems, handover problems, resource utilization issues, and process design issues.

### 2.1 Resource bottlenecks

A queue may occur because the required resource is highly utilized or unavailable.

Examples:

- Too few nurses available during the morning peak.
- Only one ECG room is available.
- Certain doctors consistently run behind schedule.
- Check-out clerks are also handling registration, creating competing demand.
- A diagnostic room is available, but the technician is not.

Using the event log, calculate for each resource:

```text
Busy time = sum of activity durations performed by resource
Available time = scheduled working time
Utilization = busy time / available time
```

High utilization, especially above 85ľ90%, often causes disproportionate increases in waiting time. In queueing systems, even small demand fluctuations can create long waits when utilization is near capacity.

Useful analyses:

- Utilization by resource and hour of day
- Number of patients waiting while resource is busy
- Service rate by resource
- Average service time by resource
- Wait time before activities handled by that resource
- Comparison between doctors, nurses, clerks, rooms, and equipment

Important distinction:

- **High wait + high utilization** suggests a true capacity bottleneck.
- **High wait + low utilization** suggests a coordination, handover, scheduling, or data-quality issue.

### 2.2 Activity dependencies and handovers

Some waits may be caused not by lack of capacity but by poor coordination between activities.

Examples:

- Nurse completes assessment, but doctor is not notified that the patient is ready.
- Doctor orders ECG, but order is not visible to the technician immediately.
- Diagnostic test is complete, but doctor review is delayed because results are not routed promptly.
- Patient physically moves between rooms without clear guidance.
- Staff batch handovers instead of passing patients continuously.

Process mining can identify these by analyzing:

- Long waits after completion of an activity despite apparent downstream resource availability.
- Repeated patterns of delay between specific activity pairs.
- Variability in handover times by department or resource.
- Differences between clinics or specialties using different coordination practices.

For example:

```text
Doctor Consultation COMPLETE  ECG Test START
```

If this queue is long even when ECG room utilization is moderate, the cause may be order release, room assignment, patient transport, or technician notification rather than ECG capacity.

### 2.3 Variability in service times

High variability in activity durations can create queues even if average capacity seems sufficient.

Examples:

- New patient consultations take much longer than follow-up consultations.
- Some doctors have highly variable consultation lengths.
- Registration for new patients takes longer due to insurance and demographic data capture.
- Diagnostic tests vary by patient condition or preparation requirements.
- Urgent add-ons disrupt the planned flow.

Metrics to examine:

- Average and median service time by activity.
- 90th percentile service time.
- Coefficient of variation.
- Service time by patient type.
- Service time by specialty.
- Service time by provider.
- Outlier cases.

If one specialty has a consultation time distribution like:

```text
Median: 18 minutes
90th percentile: 45 minutes
```

then standard 15-minute appointment slots are likely unrealistic for many patients.

### 2.4 Appointment scheduling policies

Long waits in outpatient clinics are often caused by appointment templates rather than insufficient total capacity.

Potential scheduling problems include:

- Too many patients booked at the top of the hour.
- Same slot lengths for new and follow-up patients.
- Overbooking to compensate for no-shows.
- Insufficient buffer slots for urgent patients.
- Diagnostic testing not synchronized with consultation schedules.
- Doctors scheduled for back-to-back appointments without recovery time.
- Multiple specialists sharing the same diagnostic resource at the same time.

Useful analyses:

- Appointment time versus actual arrival time.
- Appointment time versus start of consultation.
- Number of scheduled arrivals per 15-minute interval.
- No-show and late arrival rates.
- Wait time by appointment slot.
- Provider lateness at first appointment of the session.
- Schedule adherence across the day.

If the event log can be enriched with appointment data, the clinic can separate:

```text
Patient arrived early
Patient arrived on time but waited
Doctor started late
Appointment template overloaded
```

### 2.5 Patient arrival patterns

Actual patient arrivals may not match the appointment schedule.

Potential issues:

- Many patients arrive early, creating an artificial registration queue.
- Walk-ins cluster at specific times.
- Urgent patients arrive unpredictably.
- Morning peaks are much heavier than afternoon volumes.
- Patients are told to arrive 30 minutes early, causing congestion before capacity is available.
- Lab patients arrive in batches before fasting windows close.

Useful analyses:

- Arrival rate by 5-, 10-, or 15-minute interval.
- Registration queue length by time of day.
- Arrival distribution by patient type and urgency.
- Difference between scheduled and actual arrival time.
- Walk-in versus appointment-based demand.

### 2.6 Patient type, urgency, and specialty differences

Different patient groups may follow very different process variants.

Examples:

- New patients require longer registration, nurse assessment, and doctor consultation.
- Follow-up patients may not need nurse assessment.
- Cardiology patients often require ECG.
- Orthopedic patients may require X-ray.
- Urgent patients may bypass some queues but increase waits for normal patients.
- Patients needing multiple diagnostics have longer total visit durations.

Process mining can perform **variant analysis** to compare pathways such as:

```text
New Cardiology Patient:
Registration  Nurse Assessment  Doctor Consultation  ECG  Doctor Review  Check-out

Follow-up Dermatology Patient:
Registration  Doctor Consultation  Check-out
```

The goal is to avoid averaging together very different workflows. A queue that seems moderate overall may be severe for one subgroup.

### 2.7 Process mining techniques to support root cause analysis

Several process mining techniques are relevant:

#### A. Process discovery

Create the actual patient flow model from the event log. This reveals:

- Common pathways
- Optional activities
- Rework loops
- Unexpected variants
- Repeated diagnostic cycles
- Deviations from intended clinic flow

#### B. Performance-enhanced process maps

Overlay waiting times and service times on the discovered process map. This visually highlights bottlenecks.

For example:

```text
Registration -- 12 min wait --> Nurse Assessment -- 48 min wait --> Doctor Consultation
```

Thicker edges can represent higher volume, while color can represent longer waits.

#### C. Bottleneck analysis

Identify activities or transitions that contribute most to total visit duration. This includes both:

- Waiting time bottlenecks
- Service time bottlenecks

#### D. Resource analysis

Analyze workload, utilization, and handovers by staff, rooms, and equipment.

Questions include:

- Which resources are overloaded?
- Which resources have idle time while patients are waiting?
- Which handovers are slow?
- Are some providers consistently associated with longer waits?
- Are rooms or equipment constraining flow?

#### E. Variant analysis

Compare different patient paths:

- New vs. follow-up
- Urgent vs. normal
- Specialty A vs. specialty B
- Morning vs. afternoon
- Patients with diagnostics vs. without diagnostics

#### F. Conformance checking

Compare actual patient flows against the intended clinic process.

This can reveal:

- Steps occurring in the wrong order.
- Missing nurse assessments.
- Unnecessary repeated registrations.
- Diagnostics ordered after consultation when they could have been pre-ordered.
- Patients routed back to the doctor unnecessarily.

#### G. Predictive and statistical root cause analysis

Use regression, decision trees, random forests, or survival models to identify factors that predict long waits.

Candidate predictors:

- Time of day
- Day of week
- Doctor
- Specialty
- Patient type
- Urgency
- Prior activity
- Number of patients already waiting
- Resource utilization
- Appointment slot
- Diagnostic requirement

This helps move from descriptive findings to causal hypotheses.

## 3. Data-Driven Optimization Strategies

The best interventions should be selected after quantifying the actual bottlenecks. Below are five concrete strategies that are commonly effective in this type of clinic setting.

---

### Strategy 1: Align staffing and room capacity with observed demand peaks

#### Target queues

- Wait before Registration
- Wait before Nurse Assessment
- Wait before Diagnostic Tests
- Wait before Check-out

#### Root cause addressed

Demand-capacity mismatch by time of day.

For example:

- Registration queues may spike from 8:00ľ10:00.
- Nurse assessment queues may spike 20ľ30 minutes after registration peaks.
- Check-out queues may spike late morning and late afternoon.
- ECG or X-ray queues may spike when several specialties release patients to diagnostics simultaneously.

#### Data support

The event log can show:

- Patient arrival rate by 15-minute interval.
- Queue length over time.
- Staff utilization by time period.
- Waiting time by hour.
- Activities with high wait and high resource utilization.
- Whether staff idle time exists in other departments during peak queues.

If Registration has:

```text
Average wait: 18 minutes
90th percentile wait: 42 minutes
Peak queue length: 15 patients from 8:30ľ9:30
Clerk utilization during peak: 95%
Afternoon clerk utilization: 55%
```

then the problem is not necessarily total staffing but timing of staffing.

#### Intervention

Possible actions:

- Stagger clerk and nurse start times.
- Add a part-time or floating staff member during morning peaks.
- Cross-train check-out clerks to support registration during early peaks.
- Move lunch breaks away from peak diagnostic or check-out periods.
- Assign a dedicated ECG technician during high-volume cardiology sessions.
- Reserve specific rooms for predictable diagnostic demand.

#### Expected impact

If the queue is primarily caused by peak utilization, adding targeted capacity during peak intervals can produce large reductions without full-day staffing increases.

Potential expected outcomes, to be validated by simulation:

- 20ľ40% reduction in average registration wait during peak hours.
- 15ľ30% reduction in nurse assessment wait.
- Lower 90th percentile waits, which are most visible to dissatisfied patients.
- Reduced overall visit duration for morning patients.

---

### Strategy 2: Redesign appointment templates using actual service-time distributions

#### Target queues

- Wait before Doctor Consultation
- Wait before Nurse Assessment
- Wait before Diagnostic Tests

#### Root cause addressed

Poor appointment scheduling logic, especially when appointment slots do not reflect actual service time and routing needs.

Common issue:

```text
New patients and follow-up patients are both scheduled in 15-minute slots,
but new patient consultations actually take 25ľ35 minutes.
```

This causes doctors to fall behind early in the session, creating long downstream waits.

#### Data support

The event log can provide:

- Actual consultation duration by patient type.
- Actual consultation duration by specialty.
- Actual consultation duration by doctor.
- Delay accumulation across the day.
- Wait before doctor by appointment slot.
- Frequency of diagnostics after consultation by specialty and patient type.

Example finding:

```text
New cardiology patients:
Median doctor consultation: 28 minutes
90th percentile: 44 minutes

Follow-up cardiology patients:
Median doctor consultation: 12 minutes
90th percentile: 20 minutes
```

A single slot length for both groups is inappropriate.

#### Intervention

Redesign scheduling templates:

- Use longer slots for new or complex patients.
- Use shorter slots or grouped blocks for simple follow-ups.
- Stagger patient arrivals rather than booking several at the top of the hour.
- Reserve buffer slots for urgent add-ons.
- Avoid scheduling multiple diagnostic-heavy specialties to release patients to ECG/X-ray at the same time.
- Schedule first appointments only when the provider is actually available.
- Build specialty-specific templates based on historical routing patterns.

#### Expected impact

A better appointment template reduces the creation of queues before they form.

Potential expected outcomes:

- 15ľ35% reduction in wait before Doctor Consultation.
- Fewer extreme waits late in the morning or afternoon session.
- Improved provider schedule adherence.
- Lower probability of bottlenecks cascading into diagnostics and check-out.

---

### Strategy 3: Parallelize predictable diagnostic activities using protocol-based pre-orders

#### Target queues

- Wait after Doctor Consultation before diagnostic tests
- Wait before Doctor Review after diagnostics
- Total visit duration for diagnostic-heavy specialties

#### Root cause addressed

Unnecessary serial processing.

In many clinics, patients follow a sequence like:

```text
Registration  Nurse Assessment  Doctor Consultation  ECG  Doctor Review  Check-out
```

But for some specialties, certain diagnostics are highly predictable.

For example:

- Most new cardiology patients may need an ECG.
- Many orthopedic patients may need X-ray.
- Many diabetic follow-up patients may need blood work.

If diagnostics are almost always required, waiting until after the doctor consultation to order them creates avoidable delay.

#### Data support

Use variant analysis to identify high-probability diagnostic pathways.

Example:

```text
78% of new cardiology patients receive ECG.
Median wait from Doctor Consultation COMPLETE to ECG START: 24 minutes.
Median ECG service time: 8 minutes.
Median wait from ECG COMPLETE to Doctor Review START: 21 minutes.
```

This suggests substantial time may be saved by doing ECG earlier.

#### Intervention

Introduce protocol-based pre-orders where clinically appropriate:

- Nurse initiates ECG for eligible cardiology patients before doctor consultation.
- Blood tests are drawn before the doctor visit if required by standing protocol.
- X-ray is ordered prior to orthopedic consultation for specific complaint types.
- Diagnostic results are routed automatically to the doctorĺs worklist.
- Diagnostic slots are coordinated with appointment templates.

The key is that this must be clinically governed. Not all patients should receive automatic tests.

#### Expected impact

For eligible patients:

- Reduced post-consultation diagnostic waiting.
- Reduced need for the patient to return to the doctor after testing.
- Shorter total visit duration.

Potential expected outcomes:

- 15ľ25% reduction in total visit duration for eligible cardiology or orthopedic patients.
- 20ľ40 minute reduction for patients whose diagnostic test can be completed before the doctor is ready.
- Better doctor productivity because results are available at consultation time.

---

### Strategy 4: Introduce differentiated patient flows, such as fast-track follow-up and complex new-patient pathways

#### Target queues

- Registration
- Nurse Assessment
- Doctor Consultation
- Check-out

#### Root cause addressed

Heterogeneous patients are being processed through a single uniform flow.

New patients, follow-up patients, urgent patients, and diagnostic-heavy patients create different workloads. If all patients enter the same queues, simple visits are delayed by complex ones and complex visits may not receive sufficient planned capacity.

#### Data support

Process mining can compare:

- New vs. follow-up service times.
- New vs. follow-up queue times.
- Frequency of diagnostics by patient type.
- Check-out complexity by visit type.
- Total visit duration by variant.

Example finding:

```text
Follow-up dermatology visits:
Median total duration: 42 minutes
Median doctor service time: 8 minutes

New cardiology visits:
Median total duration: 145 minutes
Median doctor service time: 30 minutes
Diagnostics in 80% of visits
```

These should not be managed as the same operational process.

#### Intervention

Create differentiated pathways:

- Fast-track registration for follow-up patients with pre-verified demographics and insurance.
- E-check-in before arrival.
- Direct-to-room or direct-to-doctor process for selected low-complexity follow-ups.
- Separate nurse assessment protocols for new versus follow-up patients.
- Dedicated diagnostic-heavy appointment blocks.
- Express check-out for patients without payment, scheduling, or paperwork needs.
- Priority routing rules for urgent patients that are explicit and monitored.

#### Expected impact

Potential outcomes:

- Reduced registration and check-out waits for simple visits.
- Reduced congestion in shared queues.
- More predictable visit duration by patient type.
- Improved patient satisfaction because patients experience a flow appropriate to their needs.

Expected reduction may be:

- 20ľ50% reduction in registration time for pre-checked follow-up patients.
- 10ľ20% reduction in overall average clinic visit duration if a large share of visits are simple follow-ups.

---

### Strategy 5: Implement real-time queue visibility and handover coordination

#### Target queues

- Nurse Assessment  Doctor Consultation
- Doctor Consultation  Diagnostics
- Diagnostics  Doctor Review
- Diagnostics  Check-out
- Any queue with high wait but moderate resource utilization

#### Root cause addressed

Poor visibility and coordination rather than insufficient capacity.

Many outpatient queues occur because staff do not have real-time awareness of:

- Which patients are waiting.
- How long they have been waiting.
- Which rooms are available.
- Which diagnostic results are ready.
- Which patients need review before check-out.

#### Data support

The event log may show long waits where downstream resources are not fully utilized. For example:

```text
ECG room utilization: 48%
Median wait before ECG: 22 minutes
```

This suggests that capacity exists, but the handoff or triggering mechanism is weak.

#### Intervention

Implement operational coordination tools:

- Real-time dashboard showing queue length and longest waiting patient by stage.
- Alerts when a patient exceeds a wait threshold.
- Automatic notification when nurse assessment is complete.
- Automatic notification when diagnostic results are available.
- Clear room assignment and patient location tracking.
- Daily huddles using yesterdayĺs queue data.
- Escalation rules when wait exceeds threshold.

#### Expected impact

This is often lower-cost than adding staff.

Potential outcomes:

- 10ľ25% reduction in handover-related waits.
- Fewer extreme outlier waits.
- Faster response to bottlenecks during the clinic day.
- Better staff coordination and accountability.

## 4. Consideration of Trade-offs and Constraints

Optimization in healthcare must balance patient flow, cost, staff workload, fairness, and quality of care. Reducing waiting time should not mean rushing clinical care.

### 4.1 Risk of shifting the bottleneck

Adding capacity at one step may simply move the queue elsewhere.

Example:

- Adding more registration clerks may reduce registration wait.
- But more patients reach nurse assessment sooner.
- The nurse queue may then become worse.

Therefore, interventions should be tested using end-to-end process simulation, not isolated queue metrics.

### 4.2 Cost implications

Some interventions increase cost:

- Additional staff
- Overtime
- More rooms
- More equipment
- New software
- Staff training

The clinic should compare:

```text
Cost of intervention
versus
reduction in waiting time
versus
improvement in throughput and satisfaction
```

Targeted staffing during peak times is often more cost-effective than full-day staffing increases.

### 4.3 Staff workload and burnout

Improving throughput by pushing staff harder can be harmful.

Risks include:

- Increased burnout
- Reduced quality of communication
- More errors
- Lower staff satisfaction
- Higher turnover

Resource utilization should not be maximized to 100%. For healthcare systems with variability, some buffer capacity is necessary. A practical target is often to keep critical resources below approximately 80ľ85% sustained utilization, depending on variability and urgency mix.

### 4.4 Care quality and clinical appropriateness

Strategies like pre-ordering diagnostics must be clinically governed.

Risks include:

- Unnecessary testing
- Increased cost
- False positives
- Patient inconvenience
- Reduced physician judgment

Protocol-based diagnostics should be limited to cases where evidence and clinical leadership support them.

### 4.5 Fairness and priority rules

Prioritizing urgent patients is clinically appropriate, but it can increase waits for normal patients. The clinic should explicitly define priority rules and monitor their effects.

For example:

- Urgent patients should have shorter maximum waits.
- Normal patients should still have acceptable SLA targets.
- No patient group should experience systematically excessive waits.

### 4.6 Technology access and patient equity

E-check-in and digital pre-registration can reduce waits, but not all patients can use digital tools.

The clinic should maintain alternatives for:

- Elderly patients
- Patients with limited digital literacy
- Patients with language barriers
- Patients without smartphone or internet access

### 4.7 Balancing conflicting objectives

A balanced optimization approach should use multiple objectives:

1. Reduce average and 90th percentile waiting time.
2. Maintain or improve clinical quality.
3. Avoid excessive staff workload.
4. Control incremental cost.
5. Preserve equity across patient groups.
6. Maintain provider productivity.
7. Improve patient satisfaction.

A useful method is to evaluate each proposed intervention using a matrix:

| Criterion | Example Question |
|---|---|
| Wait reduction | How much queue time is reduced? |
| Cost | What is the incremental cost? |
| Feasibility | Can this be implemented quickly? |
| Quality impact | Does it affect clinical care? |
| Staff impact | Does it increase workload or stress? |
| Equity impact | Does it help or harm specific patient groups? |
| Bottleneck risk | Does it shift the queue elsewhere? |

Before full rollout, use:

- Pilot testing
- Discrete-event simulation
- Before-and-after comparison
- Staff feedback
- Patient feedback

## 5. Measuring Success

After interventions are deployed, the clinic should continuously monitor whether waiting times actually improve and whether improvements are sustained.

### 5.1 Core KPIs

#### Queue-specific KPIs

For each major queue:

- Average wait time
- Median wait time
- 90th percentile wait time
- 95th percentile wait time
- Maximum wait time
- Percentage of patients waiting more than threshold
- Queue length by time of day
- Number of patients affected
- Total waiting time burden

Examples:

```text
Wait before Doctor Consultation
Wait before Nurse Assessment
Wait before ECG
Wait before Check-out
```

#### End-to-end visit KPIs

- Total visit duration
- Median and 90th percentile visit duration
- Percentage of visits completed within target time
- Time from scheduled appointment to doctor start
- Time from arrival to check-out
- Time from doctor order to diagnostic completion
- Time from diagnostic completion to doctor review

#### Operational KPIs

- Patient throughput per session
- Resource utilization by staff, room, and equipment
- Overtime hours
- Staff idle time during clinic hours
- Number of patients in clinic at each time interval
- Appointment schedule adherence
- No-show rate
- Late arrival rate
- Provider start-time punctuality

#### Patient experience KPIs

- Patient satisfaction score
- Net Promoter Score, if used
- Complaints related to waiting
- Percentage of patients reporting visit as ôtoo longö
- Abandonment rate, if applicable
- Rescheduled or incomplete visits

#### Quality and safety KPIs

- Rate of missed or delayed diagnostic follow-up
- Rework loops, such as repeated registration or repeated assessment
- Unplanned return visits
- Clinical documentation completeness
- Diagnostic appropriateness if pre-order protocols are introduced

#### Equity KPIs

Monitor KPIs by:

- Patient type: new vs. follow-up
- Urgency
- Specialty
- Age group
- Language needs
- Insurance category, if relevant and appropriate
- Mobility or assistance needs

The clinic should ensure that improvements for one group do not worsen access or experience for another group.

### 5.2 Establishing baseline and targets

Before implementation, use the six-month log to define baseline performance.

Example targets:

```text
Reduce average wait before Doctor Consultation by 20%.
Reduce 90th percentile wait before Doctor Consultation from 75 minutes to 45 minutes.
Reduce total visit duration for cardiology new patients by 15%.
Reduce percentage of patients waiting over 30 minutes for Nurse Assessment from 28% to 10%.
Maintain provider utilization between 75% and 85%.
Maintain or improve patient satisfaction score.
```

Targets should include both average and percentile-based measures. Percentiles are especially important because patient dissatisfaction is often driven by extreme waits.

### 5.3 Ongoing process monitoring

The same event log structure can be used for continuous monitoring.

Recommended monitoring setup:

1. Extract START and COMPLETE events daily or weekly.
2. Recalculate queue times automatically.
3. Update dashboards by specialty, activity, patient type, and time of day.
4. Use alerts for excessive queues.
5. Compare current performance against baseline.
6. Monitor whether bottlenecks shift to other activities.
7. Review outlier cases weekly.
8. Re-run process discovery monthly or quarterly to detect process drift.

### 5.4 Statistical evaluation

To confirm that improvements are real, not random variation, use:

- Before-and-after comparisons
- Control charts
- Run charts
- Confidence intervals
- Difference-in-differences if piloting in one specialty first
- Case-mix adjustment to account for changes in patient volume or urgency
- Simulation to estimate expected performance under alternative staffing or scheduling scenarios

### 5.5 Sustained improvement cycle

The clinic should treat queue mining as an ongoing management capability, not a one-time project.

A practical cycle would be:

1. Measure current queues.
2. Identify critical bottlenecks.
3. Diagnose root causes.
4. Simulate or pilot interventions.
5. Implement targeted changes.
6. Monitor KPIs.
7. Adjust staffing, scheduling, and flow rules.
8. Repeat monthly or quarterly.

## Summary

Using process mining and queue mining, the clinic can convert raw timestamp data into a detailed understanding of where patients wait, why they wait, and which interventions are most likely to reduce visit duration. The core approach is to calculate waiting time between activity completion and the next activity start, characterize queues using average and percentile-based metrics, and prioritize queues based on total burden, frequency, long-tail waits, patient impact, and clinical importance.

Root cause analysis should combine process discovery, bottleneck analysis, resource utilization, variant analysis, scheduling analysis, and patient cohort comparisons. The most promising interventions are likely to include targeted resource allocation, redesigned appointment templates, protocol-based parallel diagnostics, differentiated patient flows, and real-time queue visibility.

Success should be measured using queue-specific, end-to-end, operational, patient experience, quality, and equity KPIs. Continuous monitoring using the same event log structure will allow the clinic to sustain improvements and prevent bottlenecks from reappearing or shifting elsewhere.
