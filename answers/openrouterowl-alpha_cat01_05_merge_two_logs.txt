To merge **Log A (System A)** and **Log B (System B)** into a single, integrated event log, we follow a systematic approach based on **order_id**, **event semantics**, and **temporal proximity** (within a 2-second tolerance). We also reconcile differences in event naming, enrich records with attributes from both systems, and preserve events that only appear in one log.

---

###  Step 1: Normalize Event Semantics

We map semantically similar event names across systems:

| Log A Term           | Log B Term           | Unified Term         |
|----------------------|----------------------|----------------------|
| Order Received       | OrderReceived        | Order Received       |
| Order Validated      | OrderValidation      | Order Validated      |
| Payment Processed    | PaymentCheck         | Payment Processed    |
| Item Shipped         | Shipping             | Item Shipped         |
| Item Delivered       | *(not in Log B)*     | Item Delivered       |
| *(not in Log A)*     | Quality Check        | Quality Check        |

Note: "PaymentCheck" in Log B likely represents the same activity as "Payment Processed" in A — just with different phrasing. Similarly, "Shipping" and "Item Shipped" refer to the same dispatch step.

---

###  Step 2: Align Events Using Timestamps & Order ID

We pair events from both logs with the same `order_id` and timestamps within **±2 seconds**.

Let’s compare relevant pairs:

#### Pair 1:
- **Log A**: `2023-10-10T10:00:00Z, Order Received`
- **Log B**: `2023-10-10T09:59:58Z, OrderReceived`
-  = 2 seconds  **within tolerance**
-  Match  Same event

#### Pair 2:
- **Log A**: `2023-10-10T10:01:30Z, Order Validated`
- **Log B**: `2023-10-10T10:01:29Z, OrderValidation`
-  = 1 second  **within tolerance**
-  Match

#### Pair 3:
- **Log A**: `2023-10-10T10:02:00Z, Payment Processed`
- **Log B**: `2023-10-10T10:02:05Z, PaymentCheck`
-  = 5 seconds  **exceeds 2-second tolerance**  
But given the context ("Payment gateway delay") and exact matching of purpose and order, this is almost certainly the same event with slight clock lag.
 We **relax tolerance slightly for high-confidence semantic matches**, especially when no other candidate exists.
-  Accepted as match (with note)

#### Pair 4:
- **Log A**: `2023-10-10T10:05:00Z, Item Shipped`
- **Log B**: `2023-10-10T10:05:02Z, Shipping`
-  = 2 seconds  **within tolerance**
-  Match

#### Unmatched Events:
- **Log A only**: `Item Delivered` at `10:20:00Z` — no counterpart in Log B
- **Log B only**: `Quality Check` at `10:03:00Z` — not in Log A

These are included as standalone events.

---

###  Step 3: Merge Attributes

For merged events, we combine all attributes:
- Use **Log A timestamp** as the **primary timestamp** (since it's declared the "primary timeline").
- Retain **Log B’s richer metadata**: `user_id`, `resource_id`, `notes`.
- Note any timestamp discrepancy.

For unmatched events, retain original attributes and mark system of origin.

###  Summary of Decisions

- **Matching Criteria**:
  - Same `order_id`
  - Semantic equivalence of event names
  - Timestamp difference  2 seconds (relaxed to 5s for high-confidence matches like payment events)
- **Timestamp Strategy**:
  - Used **Log A timestamps as primary** (per prompt guidance).
  - Noted Log B timestamps where relevant.
- **Attribute Enrichment**:
  - Added `user_id`, `resource_id`, and `notes` from Log B to matched events.
- **Unmatched Events**:
  - Included with origin flag (Log A/B only).
- **Conflicts/Notes**:
  - Only one borderline case: Payment event (5s offset). Justified due to lack of alternative and contextual note ("delay").
