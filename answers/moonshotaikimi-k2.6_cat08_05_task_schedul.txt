### 1. Analyzing Historical Scheduling Performance and Dynamics

To move beyond anecdotal evidence, we must first reconstruct the **ground truth** of shop floor execution from the MES event logs. The log structure is rich, capturing queue states, setup events, resource assignments, and disruptions, enabling a multi-dimensional performance analysis.

#### Reconstructing Actual Job Flows
Using **process discovery algorithms** (e.g., Inductive Miner or Heuristic Miner), we can reconstruct the actual process model from the event log. Unlike the theoretical routing plans, this reveals:
*   **Real routings:** Deviations from planned sequences (e.g., rework loops, skipped inspections, alternate machine usage).
*   **Concurrency and resource contention:** Where multiple jobs compete for the same machines simultaneously.
*   **Dotted Chart Analysis:** Plotting each Case ID on the y-axis against timestamps on the x-axis, with events color-coded by activity or resource. This visualization immediately exposes lead time magnitudes, queueing patterns, and the cascading impact of disruptions across parallel job streams.

#### Quantifying Performance via Process Mining Techniques

**Job Flow Times, Lead Times, and Makespan Distributions:**
Using a **case-centric performance analysis**, we compute the total cycle time for each job (from `Job Released` to final `Task End`). We analyze the distribution (not just the mean) of these times—often log-normal in job shops—to identify outliers. A **Cumulative Flow Diagram (CFD)** derived from queue entry and task completion events over time visualizes the relationship between WIP levels and lead times (applying Little’s Law).

**Task Waiting Times (Queue Times):**
**Queue Mining** is applied by measuring the interval between `Queue Entry` and the subsequent `Setup Start` or `Task Start` for each task-resource pair. These waiting times are aggregated by:
*   **Resource (Machine):** To identify congested work centers.
*   **Time of Day/Day of Week:** To detect systemic patterns (e.g., morning backlogs after shift changes).
*   **Priority Class:** To test if high-priority jobs actually receive faster queue resolution.

**Resource Utilization Analysis:**
A resource-centric view decomposes machine time into:
*   **Productive Time:** `Task Start` to `Task End` (using actual durations).
*   **Setup Time:** `Setup Start` to `Setup End`, linked to the preceding job via the log’s `Previous job` attribute or by sequencing events on the same resource.
*   **Idle Time:** Gaps between tasks where the machine is available but not processing.
*   **Downtime:** `Breakdown Start` to `Breakdown End`.
This yields precise utilization rates and setup ratios, distinguishing between capacity shortages and scheduling-induced inefficiency.

**Sequence-Dependent Setup Times (SDST):**
This is a critical and complex feature. We analyze the log using a **resource-sequential variant analysis**. For each machine resource (e.g., `CUT-01`), we extract the ordered sequence of jobs processed. We then construct a **setup transition matrix**:
$$S_{i,j,m} = \text{Average setup duration on machine } m \text{ when transitioning from job family } i \text{ to } j$$
Job families are not pre-defined; they are derived by clustering jobs based on attributes correlated with setup duration (e.g., material type, tooling changes inferred from operator notes or job IDs). If the log lacks explicit family codes, we use the matrix itself to discover families—jobs that frequently incur low setup times when sequenced together belong to the same implicit family. We also analyze the variance in these setup times to understand predictability.

**Schedule Adherence and Tardiness:**
For each case, we compare the timestamp of the final task completion (or a `Job Completed` event) against the `Order Due Date`. We compute:
*   **Tardiness:** $T_j = \max(0, C_j - d_j)$
*   **Lateness:** $L_j = C_j - d_j$
*   **On-Time Delivery (OTD) Rate:** Percentage of jobs with $T_j = 0$.
Process mining allows us to trace tardiness backward through the process model to identify *where* in the routing the fatal delays occurred (e.g., consistently at Heat Treatment).

**Impact of Disruptions:**
We filter the log for disruptive events (`Breakdown Start`, `Priority Change`). Using **variant analysis**, we compare two cohorts:
1.  Jobs whose processing on a critical machine was directly preceded by a breakdown.
2.  Jobs processed during stable periods.
By comparing their subsequent flow times and queue times, we quantify the **cascading delay multiplier** of disruptions. We also analyze post-disruption queue lengths to measure recovery time.

$$I_j = w_1 \cdot \text{CR}_j + w_2 \cdot \text{SetupEfficiency}_{j,m} + w_3 \cdot \text{DownstreamLoad}_j + w_4 \cdot \text{Slack}_j$$

*   **Critical Ratio ($\text{CR}_j$):** $(d_j - t) / \sum \text{remaining processing times}$. Uses historically mined task durations by (Machine, Operator, Job Complexity) to estimate remaining work accurately.
*   **Setup Efficiency ($\text{SetupEfficiency}_{j,m}$):** Derived from the historical setup transition matrix (Section 1). If selecting job $j$ next results in a setup time in the bottom quartile for machine $m$ (given the current machine state/job family), this term receives a bonus; a long setup incurs a penalty.
*   **Downstream Load ($\text{DownstreamLoad}_j$):** The current queue length and expected workload at the *next* machine in job $j$'s routing, mined in real-time from MES queue events. If the next machine is starved, $I_j$ is increased to pull work forward.
*   **Slack ($\text{Slack}_j$):** Time until due date minus expected remaining flow time.

**Process Mining Input:** Task duration distributions (for CR and Slack), setup transition matrices (for SetupEfficiency), real-time queue mining (for DownstreamLoad), and regression analysis on historical on-time jobs to optimize weights $w_1...w_4$.

**Pathologies Addressed:** Excessive setup on bottlenecks, high WIP caused by slow bottleneck turnover, throughput limitations.
**Expected Impact:** 20–35% reduction in setup time at bottlenecks; corresponding throughput increase; reduced WIP as jobs flow faster through the constraint.

**Evaluation Metrics:** Mean/median tardiness, % tardy jobs, average WIP by station, average flow time, bottleneck utilization, and setup time proportion. Statistical comparison (e.g., paired t-tests on simulation replications) determines the winning strategy with confidence.

#### Framework for Continuous Monitoring and Adaptation
Scheduling logic must evolve as the shop floor changes (new products, machine degradation, operator changes). We implement a **closed-loop Process Mining and Scheduling (PMS) framework**:

1.  **Online Process Mining:** The MES stream is fed continuously into a process mining engine. Dashboards track leading indicators: queue time trends per machine, tardiness rate (7-day rolling), and setup time efficiency (actual vs. historical matrix expectation).

2.  **Drift Detection:** Statistical Process Control (SPC) is applied to KPIs. If average flow time crosses control limits, or if the setup transition matrix changes significantly (detected via concept drift algorithms on the event stream), an alert is triggered.

3.  **Automated Parameter Adaptation:**
    *   For **CAMAD (Strategy 1)**, the weights $w_1...w_4$ are periodically re-tuned via regression on the most recent 500 completed jobs, ensuring the index remains aligned with current shop conditions.
    *   For **SOSB (Strategy 3)**, if the bottleneck shifts (detected via queue mining), the optimization target shifts to the new constraint.

4.  **Scheduled Audits:** Monthly, analysts run a full process mining diagnostic comparing the *scheduled* sequence (from the new system) vs. *actual* execution. Systematic deviations reveal either model inaccuracy or execution discipline issues, prompting model refinement.

By embedding process mining not just as a one-time diagnostic but as the **perceptual and adaptive nervous system** of the scheduling function, Precision Parts Inc. can transition from a reactive, chaotic job shop to a self-optimizing, resilient manufacturing system.
