# Business Process Explanation: Customer Request Handling

## Overall Purpose

This process describes how an organization receives, validates, and fulfills customer requests — distinguishing between **standard requests** (routine orders handled with predefined checks) and **custom requests** (requiring feasibility analysis and quotation). The process culminates in either an invoice and confirmation to the customer, or a rejection if a custom request is infeasible.

---

## Step-by-Step Walkthrough

### 1. Process Initiation
- **Start Event  Task A: "Receive Customer Request"**
  The process is triggered when a customer submits a request. Task A captures and registers the request, gathering the information needed for subsequent classification.

### 2. First Decision Point — Request Type (XOR Gateway)
The **exclusive (XOR) gateway** means *exactly one* path is taken, based on the request type:

#### Path 1: Standard Request
- **Task B1: "Perform Standard Validation"** — The request is checked against standard criteria (completeness, product availability rules, pricing conformity, etc.).
- **AND Gateway: "Run Parallel Checks"** — A **parallel (AND) split**, meaning *both* branches execute simultaneously:
  - **Task C1: "Credit Check"** — Verifies the customer's financial standing.
  - **Task C2: "Inventory Check"** — Confirms stock availability.
- **AND Join** — The process *waits until both checks complete* before continuing. This synchronization point is essential: delivery commitments cannot be made until both credit and inventory status are known.
- **Task D: "Calculate Delivery Date"** — Using the results of the parallel checks, a delivery date is determined.

#### Path 2: Custom Request
- **Task B2: "Perform Custom Feasibility Analysis"** — Because custom requests fall outside standard offerings, an analysis determines whether the customization can actually be produced/delivered.
- **XOR Gateway: "Is Customization Feasible?"**
  - **Yes  Task E1: "Prepare Custom Quotation"** — A tailored quote (pricing, timeline, specifications) is created.
  - **No  Task E2: "Send Rejection Notice"  End Event** — The customer is informed the request cannot be fulfilled, and the process **terminates here**. This is the only path that ends without an invoice.

---

### 3. Convergence of Paths
After **Task D** (standard path) or **Task E1** (custom path), the two branches **merge** into a common downstream flow. From this point on, both request types are treated the same way regarding approval and invoicing.

### 4. Approval Decision (XOR Gateway: "Is Approval Needed?")
- **No**  Proceed directly to **Task G: "Generate Final Invoice"**. (Typical for low-value or low-risk requests.)
- **Yes**  **Task F: "Obtain Manager Approval"** — A manager reviews the outcome (e.g., high-value orders, special pricing, tight delivery dates).

### 5. Approval Outcome (XOR Gateway: "Is Approval Granted?")
- **Granted**  **Task G: "Generate Final Invoice"**.
- **Denied**  **Task H: "Re-evaluate Conditions"** — This is the **rework/loop-back mechanism**:
  - For **custom requests**, the flow loops back to **Task E1** to revise the quotation (e.g., adjust price or scope).
  - For **standard requests**, the flow loops back to **Task D** to recalculate the delivery date (e.g., adjust terms that caused rejection).
  
  After rework, the process flows forward again through the approval gateway — meaning the loop can repeat until approval is granted (or no longer required).

### 6. Process Completion
- **Task G: "Generate Final Invoice"** — The formal billing document is created (reached either directly or after approval).
- **Task I: "Send Confirmation to Customer"** — The customer receives final confirmation of the order/quotation and invoice.
- **End Event** — The process concludes successfully.

---

## Summary of Gateway Semantics

| Gateway | Type | Meaning |
|---|---|---|
| Check Request Type | XOR (exclusive) | One path only: standard **or** custom |
| Run Parallel Checks | AND (parallel) | Both checks run concurrently; join waits for both |
| Is Customization Feasible? | XOR | Proceed with quote **or** reject and end |
| Is Approval Needed? | XOR | Manager review **or** straight to invoicing |
| Is Approval Granted? | XOR | Invoice **or** loop back for rework |

---

## Key Differences: Standard vs. Custom

| Aspect | Standard Request | Custom Request |
|---|---|---|
| Validation | Predefined standard validation (B1) | Feasibility analysis (B2) |
| Parallel checks | Yes — credit & inventory (C1/C2) | No |
| Key deliverable | Delivery date calculation (D) | Custom quotation (E1) |
| Early termination | Not possible | Yes — rejection if infeasible (E2) |
| Rework loop target | Back to Task D | Back to Task E1 |

---

## When Approvals and Rework Are Triggered

- **Approval** is triggered conditionally (business rules not specified in the model — typically value thresholds, risk, or non-standard terms). If not needed, the process goes straight to invoicing.
- **Rework (Task H)** is triggered *only* when approval is sought **and denied**. The process never terminates due to a denied approval — instead, conditions are re-evaluated and the relevant deliverable (delivery date or quotation) is reworked, creating an iterative refinement loop until the request becomes approvable.

In essence, this is a **classification  validation  approval  fulfillment** pattern with parallel execution for efficiency on the standard path, an early-exit safeguard on the custom path, and an iterative correction loop to recover from failed approvals rather than abandoning the request.
