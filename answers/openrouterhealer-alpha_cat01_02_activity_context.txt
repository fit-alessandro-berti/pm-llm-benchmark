# Event Log Analysis: Contextual Activity Renaming

---

## Step 1: Identifying Recurring Activities

After examining all three cases, I identified **four activities** that appear multiple times or carry contextual variation across cases:

| Activity | Cases Where It Appears | Recurrence |
|----------|------------------------|------------|
| **Verify Documents** | C001 (×2), C002, C003 (×2) | 5 total, with first/second occurrences in C001 & C003 |
| **Call Customer** | C001, C002 (×2), C003 | 4 total, with first/second occurrences in C002 |
| **Adjust Account** | C002, C003 | Different triggers (post-call vs. missing info) |
| **Initiate Request** | All three | Different channels (WebPortal vs. Branch) |

---

## Step 2: Contextual Analysis

### **Verify Documents** — The meaning shifts based on *what triggered it*

| Case | Occurrence | Trigger | Contextual Meaning |
|------|------------|---------|-------------------|
| C001 | 1st | Initial request submitted | Fresh intake — first look at raw documents |
| C001 | 2nd | After `Submit Additional` | Re-check — customer uploaded missing ID scan |
| C002 | Only | Initial request submitted | Standard verification, no issues found |
| C003 | 1st | Walk-in at branch | On-site physical ID inspection |
| C003 | 2nd | After `Adjust Account` | Re-check — account correction required doc review |

**Key insight:** The first occurrence is always *initial intake*. The second occurrence is always *re-verification*, but the reason for re-verification differs: C001 is triggered by *customer action* (uploading documents), while C003 is triggered by *internal action* (account correction).

---

### **Call Customer** — The purpose changes with position and intent

| Case | Occurrence | Trigger | Contextual Meaning |
|------|------------|---------|-------------------|
| C001 | Only | Verification incomplete | Proactive outreach — requesting missing documentation |
| C002 | 1st | Account discrepancy found | Investigative call — clarifying an issue |
| C002 | 2nd | After `Adjust Account` | Confirmation call — verifying changes were applied |
| C003 | Only | Customer left the branch | Operational call — returning customer to premises |

**Key insight:** In C002, both calls are customer-facing but serve opposite purposes: the first *investigates* a problem, the second *confirms* its resolution.

---

### **Adjust Account** — The driver differs across cases

| Case | Trigger | Contextual Meaning |
|------|---------|-------------------|
| C002 | After investigative call clarified discrepancy | Response to customer-reported issue |
| C003 | After missing information was discovered | Internal correction — self-identified gap |

**Key insight:** C002's adjustment is *customer-initiated* (triggered by the call's findings), while C003's is *agent-initiated* (triggered by internal discovery during processing).

---

### **Initiate Request** — Channel determines the nature of intake

| Case | Resource | Contextual Meaning |
|------|----------|-------------------|
| C001, C002 | WebPortal | Digital self-service submission |
| C003 | BranchReception | In-person walk-in at physical branch |

## Step 4: Justification Summary

| Original Activity | New Name(s) | Contextual Clues Used |
|-------------------|-------------|----------------------|
| **Verify Documents** | `Initial Document Verification` | First occurrence after request intake; no prior related activity |
| | `Re-Verify After Document Submission` | Follows `Submit Additional`; triggered by customer action |
| | `Re-Verify After Account Correction` | Follows `Adjust Account`; triggered by internal correction |
| | `On-Site Document Inspection` | Occurs at physical branch; involves face-to-face ID check |
| **Call Customer** | `Outreach for Missing Documentation` | Follows failed verification; context states "verification incomplete" |
| | `Account Discrepancy Inquiry Call` | Follows clean verification; context states "clarify account discrepancy" |
| | `Adjustment Confirmation Follow-Up` | Follows `Adjust Account`; context states "confirm adjustments made" |
| | `Customer Return Notification` | Context states customer "stepped out"; call is to bring them back |
| **Adjust Account** | `Post-Inquiry Account Adjustment` | Follows investigative call; customer-reported discrepancy drives it |
| | `Internal Information Correction` | Follows discovery of missing info; agent-initiated, not customer-initiated |
| **Initiate Request** | `Submit Online Request` | Resource = WebPortal; digital self-service channel |
| | `Submit Walk-In Request` | Resource = BranchReception; in-person physical channel |

---

### Design Rationale

The naming convention follows three guiding principles:

1. **Temporal position** — "Initial" vs. "Re-Verify" immediately signals whether this is a first pass or a repeat.
2. **Trigger source** — Names distinguish between customer-driven actions (document upload, call findings) and agent-driven discoveries (missing info, internal correction).
3. **Operational channel** — Branch activities get "On-Site" or "Walk-In" to distinguish from portal-based flows.

This approach ensures that even without the `Additional Context` column, a process analyst could infer the workflow stage and purpose from the activity name alone.
