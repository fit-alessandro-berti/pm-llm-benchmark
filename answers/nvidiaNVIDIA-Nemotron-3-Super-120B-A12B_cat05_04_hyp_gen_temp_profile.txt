### SQL Verification Approaches  

Below are reusable query templates. Replace placeholders (`<activity1>`, `<activity2>`, `<threshold_seconds>`, etc.) with the specific values you want to test.

#### 1. Flag claims where the time between two activities deviates beyond a Zscore threshold  

```sql
WITH paired AS (
    SELECT
        ce1.claim_id,
        ce1.activity   AS act1,
        ce1.timestamp  AS ts1,
        ce2.activity   AS act2,
        ce2.timestamp  AS ts2,
        EXTRACT(EPOCH FROM (ce2.timestamp - ce1.timestamp)) AS diff_sec
    FROM claim_events ce1
    JOIN claim_events ce2
      ON ce1.claim_id = ce2.claim_id
     AND ce1.activity = '<activity1>'
     AND ce2.activity = '<activity2>'
     AND ce2.timestamp > ce1.timestamp
)
SELECT
    claim_id,
    act1,
    act2,
    diff_sec,
    (diff_sec - <avg_sec>) / <stddev_sec> AS z_score
FROM paired
WHERE ABS((diff_sec - <avg_sec>) / <stddev_sec>) > <z_threshold>;   -- e.g., 3 for 3sigma
```

*Use this for each anomalous pair (RP, PN, AC, EN) with the corresponding `<avg_sec>` and `<stddev_sec>` from the temporal profile.*

#### 2. Examine whether specific adjusters or regions drive the deviations  

```sql
WITH paired AS (
    SELECT
        ce1.claim_id,
        ce1.activity   AS act1,
        ce1.timestamp  AS ts1,
        ce2.activity   AS act2,
        ce2.timestamp  AS ts2,
        EXTRACT(EPOCH FROM (ce2.timestamp - ce1.timestamp)) AS diff_sec,
        c.adjuster_id,
        a.region
    FROM claim_events ce1
    JOIN claim_events ce2
      ON ce1.claim_id = ce2.claim_id
     AND ce1.activity = '<activity1>'
     AND ce2.activity = '<activity2>'
     AND ce2.timestamp > ce1.timestamp
    JOIN claims c ON ce1.claim_id = c.claim_id
    JOIN adjusters a ON c.adjuster_id = a.adjuster_id
)
SELECT
    adjuster_id,
    region,
    COUNT(*) AS total_pairs,
    AVG(diff_sec) AS avg_diff_sec,
    STDDEV(diff_sec) AS stddev_diff_sec,
    PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY diff_sec) AS median_diff_sec
FROM paired
GROUP BY adjuster_id, region
HAVING AVG(diff_sec) < (<avg_sec> - <multiple> * <stddev_sec>)   -- or > for long delays
   OR AVG(diff_sec) > (<avg_sec> + <multiple> * <stddev_sec>);
```

*Adjust `<multiple>` (e.g., 2) to highlight adjuster/region groups whose average timing is unusually short or long.*

#### 3. Identify claims closed immediately after assignment (potential premature closure)  

```sql
SELECT
    ce_assign.claim_id,
    ce_assign.timestamp      AS assign_ts,
    ce_close.timestamp       AS close_ts,
    EXTRACT(EPOCH FROM (ce_close.timestamp - ce_assign.timestamp)) AS hours_between
FROM claim_events ce_assign
JOIN claim_events ce_close
  ON ce_assign.claim_id = ce_close.claim_id
 WHERE ce_assign.activity = 'A'
   AND ce_close.activity = 'C'
   AND ce_close.timestamp > ce_assign.timestamp
   AND EXTRACT(EPOCH FROM (ce_close.timestamp - ce_assign.timestamp)) < 7200;  -- < 2h
```

*Join with `claims` or `adjusters` to see if certain claim types, customers, or regions dominate these fast closures.*

#### 4. Detect unusually long approvaltonotification gaps  

```sql
SELECT
    ce_approve.claim_id,
    ce_approve.timestamp      AS approve_ts,
    ce_notify.timestamp       AS notify_ts,
    EXTRACT(EPOCH FROM (ce_notify.timestamp - ce_approve.timestamp)) / 86400 AS days_between
FROM claim_events ce_approve
JOIN claim_events ce_notify
  ON ce_approve.claim_id = ce_notify.claim_id
 WHERE ce_approve.activity = 'P'
   AND ce_notify.activity = 'N'
   AND ce_notify.timestamp > ce_approve.timestamp
   AND EXTRACT(EPOCH FROM (ce_notify.timestamp - ce_approve.timestamp)) > 604800 * 2;  -- > 14days
```

*Optionally filter by `claim_type`, `customer_id`, or `region` via joins to `claims` and `adjusters`.*
