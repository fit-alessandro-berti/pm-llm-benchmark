# 1) Analyzing Resource Behavior and Assignment Patterns

## 1.1 What to analyze from the event log

The event log is rich enough to assess both **process performance** and **resource behavior**. I would analyze it at three levels:

### A. Case / ticket level
For each incident:
- **End-to-end resolution time**
- **Waiting time between activities**
- **Number of handovers / reassignments**
- **Number of escalations**
- **SLA compliance** by priority
- **First-contact / first-call resolution** outcome for L1
- **Reopen rate** if available
- **Time spent in each tier**

### B. Resource / agent level
For each agent:
- **Number of tickets handled**
- **Workload over time** and concurrency
- **Average handling time** per activity type
- **Average waiting time before agent starts work**
- **Resolution rate**
- **Escalation rate**
- **Reassignment rate**
- **Case mix** by priority/category/required skill
- **Skill coverage** versus skills actually used
- **Queue time caused or experienced**

### C. Tier / team level
For L1, L2, L3:
- **Volume handled**
- **Throughput**
- **Average cycle time**
- **Queue backlog**
- **Escalation inflow and outflow**
- **Percent of work resolved at tier**
- **SLA breach rate by tier**
- **Load imbalance across agents in the same tier**

---

## 1.2 Metrics I would focus on

### Core metrics
- **Workload distribution**
  - Tickets per agent, per day/week
  - Gini coefficient or coefficient of variation for load imbalance
- **Processing time**
  - Start-to-end duration per activity
  - Average handling time by agent/tier
- **Waiting time**
  - Time between assignment and work start
  - Time between escalation and reassignment
- **First-call resolution rate (L1)**
  - % of tickets closed without escalation
- **Escalation rate**
  - % of L1 tickets sent to L2/L3
- **Reassignment frequency**
  - Average number of reassignments per ticket
- **SLA breach rate**
  - Overall and by priority (P1¢P4)
- **Skill fit rate**
  - % of assignments where the agentÆs skills match the required skill
- **Overqualification rate**
  - Highly specialized agents doing routine tasks
- **Underqualification / mismatch rate**
  - Tickets assigned to agents lacking the required skill

## 1.3 How process mining reveals actual assignment patterns

Process mining can uncover the real process as it happens, not how it is supposed to happen.

### A. Discovery
Using the event log, discover:
- The dominant process variants
- Typical paths for P1/P2/P3/P4 tickets
- Common handover chains like:
  - L1  L2
  - L1  L2  L2 reassign  L3
  - L1  L3 direct escalation

This shows whether actual assignments align with the intended tier-based logic.

### B. Conformance checking
Compare the discovered behavior against the intended rules:
- Are P2 incidents always handled by the right tier first?
- Are L1 agents escalating too early?
- Are L2 assignments sometimes bypassed?
- Are tickets routed to agents without the required skill?

You can quantify deviations such as:
- % of cases deviating from the desired path
- Average delay introduced by each deviation type

### C. Resource interaction analysis
This reveals:
- Which agents frequently work together on the same tickets
- Which handover patterns are most common
- Which agents are central in the escalation network

This is useful for identifying:
- Informal dependencies
- Knowledge silos
- Bottleneck specialists

### D. Social network analysis based on handovers
Construct a network where:
- Nodes = agents or tiers
- Edges = ticket handovers / reassignments

This helps identify:
- Overloaded ōhubö agents receiving too many escalations
- Agents who frequently send work to others
- Teams with high internal rework or poor routing

### E. Role discovery
Group agents by actual behavior rather than job title:
- ōGeneralist L1 resolversö
- ōNetwork escalation specialistsö
- ōFrequent reassignersö
- ōBacklog absorbersö
- ōHigh-volume low-complexity handlersö

This often reveals that actual operational roles differ from formal team structures.

## 1.4 Analyzing skill utilization across the agent pool

Since each ticket contains a **required skill** and each agent has documented skills, I would analyze skill utilization using:

### Skill fit matrix
A matrix of:
- Rows = agents
- Columns = skills
- Values =:
  - number of cases handled
  - average resolution time
  - success rate
  - escalation rate

### Measures of skill effectiveness
- **Skill match rate**
  - % of assignments where agent skill intersects required skill
- **Skill depth utilization**
  - Are advanced skills used on appropriate complexity tickets?
- **Skill underuse**
  - Specialized agents assigned to generic tickets too often
- **Skill mismatch**
  - Tickets needing Skill X assigned to agents without Skill X

### Examples of findings
- L2 agents with DB-SQL expertise spending 40% of time on basic software-app tickets
- L1 agents repeatedly assigned network incidents requiring firewall expertise
- L3 security experts handling low-complexity access requests that could be resolved by L1 with better scripts

This shows whether specialists are being used effectively or wasted on work below their expertise.

# 2) Identifying Resource-Related Bottlenecks and Issues

## 2.1 How to pinpoint issues

Using the metrics above, I would identify bottlenecks in several ways.

### A. Skill-specific bottlenecks
Look for:
- Long waiting times before assignment to certain required skills
- High queue lengths for specific categories like Network or Security
- Concentration of work on only one or two specialists

**Example**
If all Networking-Firewall tickets wait significantly longer before L2 assignment, that indicates a shortage or routing problem for that skill.

### B. Reassignment and escalation bottlenecks
Measure:
- Number of reassignments per case
- Delay added after each reassignment
- Escalation loops

If a ticket gets reassigned from B12 to B15 because a different skill is needed, that suggests poor initial skill matching.

### C. Initial assignment quality
Analyze:
- Accuracy of first assignment
- First-time-right rate
- Whether dispatcher/L1 assignment choices lead to later rework

A low first-time-right rate directly points to poor routing logic.

### D. Overloaded vs underutilized agents
For each agent:
- Compare handled volume to peer average
- Compare average queue time of assigned cases
- Compare idle time versus active time

This can expose:
- Overloaded specialists who become bottlenecks
- Underutilized agents who could absorb work

### E. SLA breach correlation
Join case attributes with operational behavior:
- Priority
- Category
- Required skill
- Number of handovers
- Assignment delays
- Agent skill match

Then test which factors are statistically associated with breaches.

---

## 2.2 Quantifying the impact

Useful quantifications include:

### Average delay caused by reassignment
For example:
- Tickets with 0 reassignments: average resolution time = 45 min
- Tickets with 1 reassignment: average resolution time = 70 min
- Tickets with 2+ reassignments: average resolution time = 110 min

Then estimate:
- **Incremental delay per reassignment**
- **Incremental SLA risk per reassignment**

### SLA breach rate by skill mismatch
Example:
- Skill-matched tickets: 8% breach rate
- Skill-mismatched tickets: 23% breach rate

This quantifies how much routing quality affects SLA outcomes.

### Time lost in queue due to specialist shortages
Example:
- Network tickets wait 35 min on average for assignment versus 8 min for software-app tickets

### Reassignment cost
Example:
- Each reassignment adds 12¢18 minutes on average, including waiting and redispatch time

### Overload impact
Example:
- Agents above 120% of normal workload have 2x higher average handling delay and 1.8x more escalations

# 3) Root Cause Analysis for Assignment Inefficiencies

## 3.1 Likely root causes

### A. Assignment rules are too simplistic
A pure round-robin approach ignores:
- Skills
- Ticket complexity
- Current workload
- Priority
- SLA urgency

Result:
- Tickets go to available agents, not best-fit agents

### B. Skill profiles are inaccurate or incomplete
Possible issues:
- Skills not updated after training
- Skills recorded too broadly
- Hidden expertise not captured in the system

Result:
- Dispatcher thinks an agent is suitable when they are not

### C. Poor initial categorization
If the ticket category or required skill is wrong at creation:
- Wrong tier gets the case
- Wrong specialist gets involved
- Rework increases

### D. Lack of real-time workload visibility
If dispatchers cannot see:
- agent queue length
- active cases
- skill availability

Then even a good rule cannot be applied well.

### E. L1 lacks empowerment or knowledge
If L1 agents are not trained or authorized enough:
- They escalate too aggressively
- They may avoid ownership
- FCR drops

---

## 3.2 How variant analysis helps

Compare two groups:
- **Smooth cases**: resolved with 0 reassignments and 0¢1 handovers
- **Problem cases**: multiple reassignments, long waits, SLA breaches

Then compare:
- ticket category
- priority
- required skill
- initial agent
- time of day
- agent workload
- assigned tier

This can reveal patterns like:
- P2 network incidents are more likely to be misrouted at peak hours
- Tickets assigned to certain L1 agents are more likely to escalate
- Skill mismatches are concentrated in specific categories

---

## 3.3 How decision mining helps

Decision mining can identify which attributes influence assignment decisions:
- priority
- category
- required skill
- channel
- time of day
- current queue length
- agent availability
- ticket wording/keywords

This helps answer:
- What factors are actually driving assignment?
- Are humans following intended rules or informal heuristics?
- Which attributes should be formalized into a routing algorithm?

# 4) Developing Data-Driven Resource Assignment Strategies

Below are four concrete strategies; any three would satisfy the requirement, but I recommend all four.

---

## Strategy 1: Skill-based routing with proficiency weighting

### Problem addressed
Tickets are assigned without enough regard to actual expertise.

### How it works
Route tickets to agents based on:
- exact skill match
- proficiency level in that skill
- historical resolution success for that skill
- current workload

Example:
- A Security-IAM ticket goes first to an L2/L3 agent with the strongest IAM success history
- If no specialist is free, route to the best available fallback agent

### Process mining input
- Skill-match rates
- Resolution times by skill-agent pair
- Reassignment patterns
- Escalation frequency by category

### Data required
- Agent skill matrix
- Historical ticket-to-agent assignments
- Resolution outcomes
- Ticket category / required skill
- SLA priority

### Expected benefits
- Fewer reassignments
- Better first-time-right rate
- Lower SLA breaches
- Better specialist utilization

---

## Strategy 2: Workload-aware dynamic assignment

### Problem addressed
Some agents are overloaded while others are underused.

### How it works
Assignment considers:
- current queue size
- active cases in progress
- predicted handling time
- remaining SLA slack
- agent utilization level

A lower-skilled but available agent may be preferred for simple tickets if they can resolve it faster than a busy specialist.

### Process mining input
- Load distributions over time
- Waiting times by agent/tier
- Queue bottlenecks
- Time-of-day demand patterns

### Data required
- Real-time workload state
- Historical handling-time predictions
- SLA remaining time
- Agent availability calendar

### Expected benefits
- More balanced workload
- Reduced queue times
- Less specialist overload
- Better SLA performance during peaks

---

## Strategy 3: Predictive ticket routing using classification models

### Problem addressed
Initial categorization and required skill identification are often inaccurate.

### How it works
Use historical logs and ticket text to predict:
- likely category
- required skill
- complexity
- probability of escalation
- best initial tier

Features can include:
- keywords from ticket description
- channel
- priority
- requester type
- historical patterns
- similar past tickets

### Process mining input
- Variant patterns by ticket type
- Misrouting history
- Escalation and rework patterns
- Successful resolution paths

### Data required
- Event log
- Ticket text / metadata
- Historical outcomes
- Classification labels from prior cases

### Expected benefits
- Better first assignment
- Fewer incorrect escalations
- Reduced resolution time
- More consistent routing decisions

---

## Strategy 4: Escalation optimization and L1 empowerment

### Problem addressed
L1 escalates too often or too early.

### How it works
Define escalation criteria based on:
- past resolution success
- ticket complexity
- knowledge articles
- time spent on attempt
- SLA remaining time

Also improve:
- L1 knowledge base
- guided troubleshooting scripts
- decision support for agents

### Process mining input
- L1 resolution success rates
- Cases where L1 could have resolved but escalated
- Average time before escalation
- Ticket classes with high L1 success potential

### Data required
- Historical L1 outcomes
- Knowledge article usage
- Case complexity labels
- Escalation timings

### Expected benefits
- Higher FCR
- Lower L2/L3 load
- Faster resolution for standard issues
- More effective use of specialists
