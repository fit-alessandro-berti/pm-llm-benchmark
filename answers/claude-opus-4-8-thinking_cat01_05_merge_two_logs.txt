## Reasoning & Decisions

### Matched & Merged Events

**Event 1 — Order Received  OrderReceived**
- Timestamps differ by exactly 2 s (10:00:00 vs 09:59:58). The B timestamp is *earlier*, consistent with the stated behavior that **Log A records "received" time** while System B captured the event at the warehouse source first. This directionality reinforces the match rather than undermines it.
- Names are clearly the same activity (only formatting/spacing differs).
- **Decision:** Merge. High confidence. Enriched with `user_id=u45`, `resource_id=r10`, and warehouse note from B.

**Event 2 — Order Validated  OrderValidation**
- Timestamps differ by 1 s (within tolerance). Names are semantically identical.
- **Decision:** Merge. High confidence. Enriched with validation actor `u12`, `r10`, and "Cross-check inventory" note.

**Event 3 — Payment Processed  PaymentCheck** *(judgment call)*
- Timestamps differ by **5 s**, which **exceeds the 2-second tolerance**, so a purely mechanical rule would keep these separate.
- However, several contextual signals support a match: same `order_id`, payment-related semantics, sequential position in the process, and—critically—**Log B's note explicitly says "Payment gateway delay,"** which directly explains the larger-than-usual offset.
- **Decision:** Merge, but flagged **Medium confidence**. I kept the explanatory note and B's distinct `resource_id=r99` (payment gateway resource) so a reviewer can audit the decision. If a strict tolerance policy is required, this is the one pair that should be split into two records.

**Event 5 — Item Shipped  Shipping**
- Timestamps differ by 2 s (within/at tolerance). "Item Shipped" and "Shipping" describe the same activity, corroborated by B's note "Package handed to courier."
- **Decision:** Merge. High confidence. Enriched with courier hand-off note and resource `r10`.

### Non-Matched (Single-Source) Events

**Event 4 — Quality Check (B only)**
- No corresponding event exists in Log A. It falls cleanly between Payment (10:02) and Shipping (10:05), so it is not a mis-timed duplicate of any A event.
- **Decision:** Retain as a standalone **B-origin** event. This is a genuine enrichment—System A simply doesn't track inspection steps.

**Event 6 — Item Delivered (A only)**
- No B event near 10:20:00 (B's log ends at Shipping). 
- **Decision:** Retain as a standalone **A-origin** event with empty enrichment fields (no user/resource data available).

## Conflict-Resolution Summary

1. **Timestamp conflicts:** Resolved by choosing Log A as the authoritative primary clock while preserving B's timestamp as an alternate. No data discarded.
2. **Naming variations:** Normalized to Log A's human-readable convention (e.g., `OrderValidation`  "Order Validated") to keep one consistent vocabulary, while the merge is documented per row.
3. **Attribute integration:** All B-only attributes (`user_id`, `resource_id`, `notes`) were carried into merged records, enriching A's sparse events.
4. **Tolerance edge cases:** Strict 2 s tolerance was applied for high-confidence merges; Event 3 was the sole exception, merged only because a domain note (gateway delay) plausibly justified the 5 s gap. This is explicitly flagged so it can be reverted under a stricter policy.

### Note / Recommendation
The single ambiguous decision is **Event 3**. Two valid outputs exist depending on policy:
- **Lenient (shown above):** one merged Payment event.
- **Strict (2 s only):** split into `Payment Processed` (A, 10:02:00) and `PaymentCheck` (B, 10:02:05) as separate rows.

I recommend the lenient merge given the corroborating "Payment gateway delay" note, but the strict alternative is trivial to switch to if required.
