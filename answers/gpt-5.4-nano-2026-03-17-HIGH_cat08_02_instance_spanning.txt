## 1) Identifying InstanceSpanning Constraints and Their Impact

Below is how I would *formally* identify each constraint in the event log and quantify its impact using process-mining/resource-mining techniques.

### A. Shared ColdPacking Stations (limited capacity; case competition)
**How to identify (log + mining approach)**
1. **Confirm the affected activities**: detect the “Packing” (or sub-step “Cold Packing”) events and the **resource IDs** that represent cold-capable stations (e.g., `Station C1..C5`).
2. **Build a resource timeline** for each cold station: for each event on that station, use `START/COMPLETE` timestamps to mark occupancy intervals.
3. **Compute queue/wait per case** at the cold step:
   - For each case `c`, let  
     - `t_ready(c)` = timestamp when the case becomes *ready* for the cold step (e.g., completion of preceding activity such as Quality Check or Packing substep start condition).
     - `t_start(c)` = timestamp when the cold-packing activity actually starts.
   - Then define **total waiting** before cold packing:  
     \[
     W_{\text{cold}}(c)= t\_{\text{start}}(c)-t\_{\text{ready}}(c)
     \]
4. **Decompose waiting into withincase vs betweencase**
   - Within-instance delays are typically caused by things like missing prerequisites, rework loops, or long preceding activity durations (those appear as long activity durations, not as “idle gaps”).
   - Between-instance delays for this constraint are the parts of \(W_{\text{cold}}\) caused by **resource unavailability**.
   - **Operational method**: for each case, compute the *earliest feasible start* given only its readiness time and the observed station occupancy pattern. The gap between feasibility and actual start can be attributed to competition/queueing.
   - Practically:
     - From the station timelines, infer the next time any cold station becomes free.
     - Let `t_free_next(c)` = earliest time a cold station can start that case after `t_ready(c)`.
     - Then:
       \[
       W_{\text{between}}(c)=t\_{\text{free\_next}}(c)-t\_{\text{ready}}(c)
       \]
       \[
       W_{\text{unexplained}}(c)=t\_{\text{start}}(c)-t\_{\text{free\_next}}(c)
       \]
     - In most fulfillment logs, the large “between” portion corresponds to cold-station contention.

**Metrics to quantify impact**
- **Average/median and distribution of queue waiting** at cold packing:
  - \( \text{Avg }W_{\text{cold}}, \text{P90 }W_{\text{cold}} \)
- **Cold station utilization** and effective capacity pressure:
  - utilization per station = fraction of time occupied
  - number of cases waiting at cold packing over time (queue length over time)
- **Bottleneck share**:
  - fraction of total end-to-end lead time attributable to cold-wait
- **SLA impact**:
  - % of orders missing deadlines due to cold-step queue growth

---

### B. Shipping Batches (batching creates crosscase dependency before Label Generation)
**How to identify**
1. Identify the activity **“Shipping Label Gen.”**
2. Detect **batch groups**:
   - If the log includes a batch identifier (e.g., `Batch B1`), use it directly.
   - If not, infer batching by checking whether multiple cases enter “Shipping Label Gen.” within the same tight time window *and* share a routing attribute like `Destination Region`.
3. For each case `c`, compute:
   - `t_ready_ship(c)` = completion time of the preceding step required before labeling (typically QC completion or Packing completion).
   - `t_batch_start(c)` = timestamp when labeling begins (or batch “release time”).
   - **Batch-induced waiting**:
     \[
     W_{\text{batch}}(c)= t\_{\text{batch\_start}}(c)-t\_{\text{ready\_ship}}(c)
     \]
4. **Differentiate within- vs between-instance**
   - If `t_ready_ship(c)` is earlier than the batch release time, the delay is *between-instance* (because labeling is gated by other cases and/or batch formation rules).
   - If the case is not ready (e.g., QC still running), that is within-instance (showing up as long activity duration or rework).

**Metrics**
- **Batch formation time**:
  - from first case ready in batch to batch release (or label start)
- **Batch waiting time distribution**:
  - median and P90 of \(W_{\text{batch}}\)
- **Batch size vs waiting tradeoff**:
  - correlation between batch size and waiting time
- **Region-level throughput**:
  - completed shipments per hour by region (to see if batching delays reduce throughput)

---

### C. Priority Handling (Express orders preempt standard work)
**How to identify**
1. Detect the resources/servers where priority matters (e.g., packing stations, QC staff).
2. Search the event stream for **preemption patterns**:
   - A standard activity on a resource is started, then before completion an express case starts on the *same resource*.
   - Because logs often record only starts/completes, you may infer preemption when:
     - standard’s COMPLETE occurs significantly later than expected, and
     - an express case START occurs in between on the same resource.
3. Quantify **impact on both classes**:
   - For express orders:
     - express waiting time before its priority step.
   - For standard orders:
     - delay attributable to express occupying the resource.

**Metrics**
- Express lead-time improvements:
  - Avg/P90 waiting time before priority step for Express
- **Preemption incidence rate**:
  - number of inferred preemptions per day/resource
- **Standard penalty**:
  - added delay on standards due to express interruptions (aggregate over affected cases)
- **Fairness / starvation checks**:
  - how often standards wait beyond thresholds while express queue grows

---

### D. Regulatory Compliance (global hazardous-material limit across the facility)
This is a classic **instance-spanning global capacity constraint**.

**How to identify**
1. Filter events/cases where `Hazardous Material = TRUE`.
2. For each timestamp, track how many hazardous orders are concurrently in:
   - “Packing” and “Quality Check” (both activities are included in the regulatory limit).
3. Compute concurrency over time:
   - Use event intervals: for each hazardous case, mark when it is in packing/QC (from START to COMPLETE).
4. Identify:
   - **Limit violations** (if any)
   - **Throttling evidence**: hazardous cases waiting because the global concurrency is already at 10.

**Metrics**
- **Maximum observed hazardous concurrency**:
  - Max concurrent hazardous in packing/QC
- **Time spent at/near the limit**:
  - % of time concurrency == 10
- **Hazardous waiting time** before starting packing/QC:
  - Similar to resource contention, but “server availability” is the global hazardous capacity.
- **Throughput reduction attributable to constraint**:
  - compare hazardous completion rate when concurrency saturates vs when it doesn’t
- **Compliance score**:
  - % of time windows violating the limit (should be 0 ideally)

**Key point: distinguishing within vs between**
- Within-instance for hazardous cases: long durations of packing/QC, rework, etc.
- Between-instance due to regulation:
  - hazardous case START is delayed while global hazardous concurrency is already at 10—this is purely cross-case.

## 2) Analyzing Constraint Interactions

Instance-spanning constraints rarely act independently. Key interactions I would explicitly test in analysis/simulation:

### Interaction 1: Express priority × Shared Cold Packing
- If express orders require cold packing, they may:
  - seize a cold station sooner than standards (queue discipline changes),
  - *increase* standard queueing at cold packing,
  - create a “priority inversion” effect if express repeatedly arrives.
- **What to look for in mining**:
  - express arrival rate vs cold-station waiting distribution for standards
  - whether express reduces express waiting but disproportionately increases standard waiting
  - preemption patterns on QC/picking that cascade into cold packing readiness timing

### Interaction 2: Express priority × Shipping batching
- Batching is region-based. If express orders are allowed to “jump” in labeling/dispatch preparation, they may:
  - be held waiting for a batch that is forming around standard orders,
  - or force the batch logic into smaller/fragmented batches, reducing route efficiency.
- **What to look for**:
  - express cases’ label-generation delay relative to their readiness time
  - how often express orders arrive late into a batch and become the *last* case to release it (long tails)

### Interaction 3: Hazardous global limit × Batching and region routing
- Hazardous orders likely have specific handling rules (may also affect readiness time).
- If hazardous orders to the same region complete packing/QC late, they can:
  - delay region batch releases (shipping label generation waits for them),
  - potentially cause batch formation to “stall” on a subset of cases.
- **What to look for**:
  - whether hazardous cases are overrepresented among “batch last released” orders
  - whether hazardous concurrency saturation correlates with larger batch waiting times in certain regions

### Interaction 4: Hazardous global limit × Priority handling
- If express hazardous orders arrive, the regulatory limit still caps concurrency, so express may:
  - wait at admission despite priority,
  - cause non-hazardous work to continue while hazardous is throttled.
- **Why this matters**:
  - you cannot “override” regulatory admission with scheduling alone; you need admission control + specialized lanes/time-slotting.

Understanding these interactions is crucial because many optimizations (e.g., strict preemption) can improve one KPI while worsening another (e.g., batch waiting, regulatory throttling, overall SLA).

## 3) ConstraintAware Optimization Strategies (at least 3)

### Strategy 1 — Predictive, PriorityAware Scheduling for Shared ColdPacking (and QC)  
**Primary constraints addressed:** Shared ColdPacking, Priority Handling (and partly Hazardous)  
**Proposed changes**
1. **Introduce a “cold-packing scheduler” queue** with explicit policies:
   - Express orders treated as high priority *but* served under a bounded preemption rule (e.g., preempt only at safe checkpoints or reserve a small fraction of cold-station capacity for express).
2. **Use predictive demand forecasting** from log history:
   - predict cold-packing arrivals by time-of-day/region and cold-station occupation.
   - pre-position staff or switch station modes earlier (reduce setup/idle).
3. **Separate lanes when feasible**:
   - If operations allow: “Express cold lane” using 1–2 stations (or time slices) to prevent express from repeatedly pausing standards.
   - Keep 1–2 stations for standard cold packing to avoid starvation and reduce tail delays.

**How data/analysis is leveraged**
- Use mining outputs:
  - \(W_{\text{cold}}\) distributions by order type (express vs standard)
  - inferred preemption rates and their effect on standards
  - cold-station utilization and queue length over time
- Identify times when contention is worst and reserve capacity accordingly.

**Expected outcomes**
- Reduced P90 waiting at cold packing for express and standards.
- Lower standard-order tail delays caused by uncontrolled priority interruptions.
- More stable utilization and better SLA performance.

---

### Strategy 2 — Batch Logic Redesign: “Release earlier” with region-aware dynamic batching  
**Primary constraints addressed:** Shipping Batches  
**Proposed changes**
1. **Change batch release triggers** from “wait for full batch” to **dynamic release**:
   - Use either:
     - a **time-based window** (e.g., release every N minutes), or
     - a **readiness threshold** (e.g., release when k% of cases for that region are ready), or
     - **dual threshold** (min batch size + max waiting time).
2. **Enable partial batch labeling / decouple label generation from physical loading**:
   - Generate labels per order when ready, while route-optimization is applied later at loading/dispatch.
   - (If label generation must be batched operationally, use smaller incremental batches.)
3. **Priority-aware batch composition**:
   - Prevent express orders from becoming the “last ready” bottleneck:
     - either exclude express orders from waiting for standard completions, or
     - guarantee express orders trigger an earlier release if they are ready and match region.

**How data/analysis is leveraged**
- Use \(W_{\text{batch}}\) distributions and batch formation time metrics.
- Identify typical batch “stall causes”:
  - which activities/orders (QC delays? hazardous throttling?) most often delay release.

**Expected outcomes**
- Lower batch-induced waiting \(W_{\text{batch}}\) and reduced end-to-end lead time tails.
- Improved SLA compliance without fully discarding batching benefits.

---

### Strategy 3 — Global Admission Control & Scheduling for Hazardous Orders  
**Primary constraints addressed:** Hazardous Material Limits  
**Proposed changes**
1. **Implement a system-level “hazardous admission controller”**:
   - before starting Packing or Quality Check for a hazardous case, check global concurrency (<10 allowed).
2. **Use an explicit hazardous scheduling policy** to minimize waiting:
   - e.g., earliest-ready-first for hazardous orders,
   - or group by expected duration to reduce time spent at concurrency saturation,
   - optionally time-slot hazardous capacity windows to avoid saturation peaks.
3. **Operational separation (if feasible)**:
   - reserve dedicated physical areas for hazardous processing while still respecting concurrency caps.
   - this reduces interference with normal flow and stabilizes throughput.

**How data/analysis is leveraged**
- Use mined concurrency timelines:
  - when concurrency saturates and how long it stays there
- Quantify hazardous waiting time due to regulatory throttling.
- Identify whether throttling is caused by:
  - long hazardous processing times (service time issue),
  - too many hazardous arrivals at once (arrival burst issue),
  - or both.

**Expected outcomes**
- No compliance violations; more predictable hazardous throughput.
- Reduced hazardous-case waiting, improving their contribution to batch delays and overall lead time.

---

### Strategy 4 (optional but often effective) — Reduce cross-step coupling by “decoupling readiness from completion”
**Primary constraints addressed:** all four, indirectly  
Example change:
- introduce intermediate buffers / asynchronous handoffs so that completion of one step doesn’t force tight synchronization with later batching or global constraints.
- This is especially helpful when QC delays (hazardous throttling) create downstream batch release delays.

## 4) Simulation and Validation (constraint-faithful testing)

### How to simulate
Use a **discrete-event simulation (DES)** (or a log-based simulation tool driven by process-mining results) with a resource + control logic model.

**What the simulation must capture (to respect instance-spanning constraints)**
1. **Resource contention**
   - 5 specialized cold-packing stations (capacity = 5), plus other resources for picking/packing/QC.
   - resource schedules inferred from logs: service-time distributions, queue disciplines.
2. **Batching logic**
   - region-based batcher model:
     - batch formation rules (time window, size threshold, release conditions)
     - batch release gates for “Shipping Label Generation”
3. **Priority & preemption**
   - priority queueing discipline at relevant servers:
     - preemptive-resume vs non-preemptive (match your actual operational behavior)
   - express arrival stream and priority behavior
4. **Global hazardous capacity**
   - a global counter representing “hazardous in Packing/QC concurrently  10”.
   - if at limit, hazardous jobs enter a holding state (queue) until admitted.
5. **Arrival process**
   - use log-derived inter-arrival times per order type and seasonality/peak periods.
6. **Routing attributes**
   - destination region affects batching grouping.
7. **Calibration**
   - tune distributions so baseline simulation reproduces observed KPIs (queue times, utilization, lead time).

### KPIs to compare (before/after strategies)
- End-to-end lead time distribution (median + P90)
- SLA miss rate by order type (express vs standard)
- Cold-packing queue waiting: mean/P90
- Shipping label batch waiting \(W_{\text{batch}}\)
- Express vs standard delay impact (fairness)
- Hazardous compliance:
  - max concurrent hazardous in packing/QC
  - any violations (should be zero)
- Throughput (orders/hour) and work-in-process (WIP)

### Why simulation is essential here
Many improvements trade off other constrained steps:
- priority improvements can worsen standard queue tails,
- partial batching can increase operational complexity but reduce waiting tails,
- hazardous admission policies can affect batching release timing.
DES is the safest way to test “system-level” outcomes.

## 5) Monitoring PostImplementation (continuous constraint management)

### A. Dashboards / metrics to track continuously

#### Shared ColdPacking (capacity pressure)
- Queue length over time and average waiting:
  - avg/P90 wait for cold packing
- Cold-station utilization
- Count of cases waiting > threshold (e.g., >30 minutes)
- Breakdown by order type (express vs standard; hazardous vs non-hazardous)

#### Shipping Batching
- Average and P90 batch formation time
- Batch release frequency and batch size distribution
- Batch waiting time \(W_{\text{batch}}\) distribution
- Share of batches released early vs full (depending on your new logic)

#### Priority Handling
- Express service time and waiting time
- **Impact metrics on standards**:
  - added waiting caused by express (aggregate and by resource)
- Preemption rate and “preemption storms” detection:
  - number of preemptions per hour per resource

#### Hazardous Material Limits (compliance + flow)
- Global hazardous concurrency in packing/QC:
  - max concurrency, time-at-limit %, and trend
- Hazardous waiting time before admission
- Throughput of hazardous orders per hour
- Compliance alerts: any breach events (should be 0)

---

### B. Process mining “monitoring mode”
Set up periodic replays against the discovered process model and constraint rules:
- **Conformance** (does behavior still match the intended process?)
- **Drift detection** (did arrival mix change? are waits increasing?)
- **Constraint violation detection** in near-real time:
  - global hazardous counter checks
  - batch release policy adherence (did labeling happen according to new rules?)
- **Root-cause drilldowns**:
  - use attribute slicing (region, order type, hazardous flag, cold-required flag)

- **Identification**: isolate each instance-spanning constraint by mapping inter-case dependencies into measurable queue/batch/preemption/concurrency effects.
- **Interaction understanding**: quantify how express priority and hazardous throttling ripple into batching and shared resources.
- **Optimization**: apply constraint-aware changes (priority-aware cold scheduling, dynamic batch release/decoupling, hazardous admission control).
- **Validation**: test with a DES model calibrated from the log so the interdependencies are preserved.
- **Monitoring**: track constraint-specific KPIs and cross-step end-to-end outcomes continuously.
