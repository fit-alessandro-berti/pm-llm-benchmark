# Comprehensive Event Log for Process Mining

Based on the provided ERP tables, I've integrated the data to create a comprehensive event log. Each event represents a key activity in the order-to-cash process, linked by `order_id` as the case identifier. I've made the following assumptions and design choices:

1. **Case ID**: Used `order_id` as the primary case identifier
2. **Activity granularity**: Created separate events for each meaningful activity occurrence
3. **Chronological ordering**: Events are sorted within each case by timestamp
4. **Resource enrichment**: Included resource information from optional tables where available
5. **Attributes**: Added relevant contextual attributes for each event type

## Key Assumptions and Notes

1. **Activity Definitions**:
   - "Item Picked" events are created for each line item in the Order Lines table
   - "Order Dispatched" and "Order Arrived" events are created for each delivery record
   - "Invoice Issued" events are created for each invoice record
   - "Payment Received" events are created for each payment record

2. **Data Gaps**:
   - Order 1004 has no delivery, invoice, or payment records in the provided data
   - Payment events have no resource information as none was provided in the Payments table

3. **Chronological Order**: Events within each case are ordered chronologically by timestamp. For cases with multiple deliveries/invoices, events are interleaved based on actual timestamps.

4. **Resource Enrichment**: Resource IDs were mapped to names and roles using the Resources table where available.

5. **Attribute Selection**: Additional attributes were selected to provide context relevant to each activity type while maintaining table readability.

This event log provides a complete trace of the order-to-cash process for all five orders, suitable for process mining analysis such as process discovery, conformance checking, and performance analysis.
