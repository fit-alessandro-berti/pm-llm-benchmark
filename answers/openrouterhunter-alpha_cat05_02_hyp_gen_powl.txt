# Analysis of Insurance Claim Handling Process Anomalies

## 1. Identified Anomalies in the POWL Model

Based on the provided POWL model, I've identified three primary anomalies:

### **Anomaly 1: Premature Claim Closure Path**
- **Structure**: Direct edge from `A` (Assign Adjuster) to `C` (Close Claim)
- **Issue**: Allows claims to be closed immediately after assignment, bypassing evaluation, approval, and customer notification
- **Business Impact**: Risk of closing valid claims without proper assessment, potentially leading to customer dissatisfaction and regulatory non-compliance

### **Anomaly 2: Evaluation-Approval Loop**
- **Structure**: `LOOP(E, P)` - creates a cycle where claims can be repeatedly evaluated and approved
- **Issue**: While re-evaluation might be legitimate in some cases (e.g., when additional information emerges), an unbounded loop could indicate:
  - Inconsistent evaluation criteria
  - Approval authority conflicts
  - Process inefficiencies
- **Business Impact**: Potential for claims to be stuck in endless review cycles or to receive contradictory decisions

### **Anomaly 3: Optional Customer Notification**
- **Structure**: `XOR(N, skip)` - allows skipping customer notification entirely
- **Issue**: Customer notification is a critical compliance and service component. Skipping it could violate:
  - Regulatory requirements (prompt notification mandates)
  - Service level agreements
  - Customer trust expectations
- **Business Impact**: Regulatory penalties, poor customer experience, potential legal liability

## 2. Hypotheses for Anomaly Origins

### **Hypothesis A: Partially Implemented Business Rule Changes**
- The direct `AC` path might have been introduced for specific claim types (e.g., clearly fraudulent claims) but became a general option due to incomplete system constraints
- The notification skip could be for sensitive cases (e.g., claims involving legal proceedings) that weren't properly restricted

### **Hypothesis B: System Migration/Integration Issues**
- Legacy system behaviors might have been carried forward without proper process redesign
- Different regional or departmental processes might have been merged without harmonization

### **Hypothesis C: Workaround for Process Bottlenecks**
- The evaluation-approval loop might have been created to handle complex claims requiring multiple expert reviews, but the loop lacks proper exit conditions
- The premature closure path could be a workaround for claim assignment errors or duplicate submissions

### **Hypothesis D: Inadequate Process Validation**
- The modeler might not have applied proper process mining validation techniques
- Testing might have focused on the "happy path" without exploring edge cases

## 3. Database Verification Queries

Here are SQL queries to investigate these anomalies in the actual event data:

### **Query 1: Identify Premature Claim Closures**
```sql
-- Find claims closed without evaluation or approval events
WITH claim_events_aggregated AS (
    SELECT 
        claim_id,
        STRING_AGG(activity, ', ' ORDER BY timestamp) AS activity_sequence,
        MIN(timestamp) AS first_event,
        MAX(timestamp) AS last_event,
        COUNT(*) AS event_count,
        BOOL_OR(activity = 'E') AS has_evaluation,
        BOOL_OR(activity = 'P') AS has_approval,
        BOOL_OR(activity = 'N') AS has_notification,
        BOOL_OR(activity = 'C') AS has_closure
    FROM claim_events
    GROUP BY claim_id
)
SELECT 
    c.claim_id,
    c.claim_type,
    c.claim_amount,
    c.submission_date,
    cea.activity_sequence,
    cea.event_count,
    cea.has_evaluation,
    cea.has_approval,
    cea.has_notification,
    CASE 
        WHEN cea.has_closure AND NOT cea.has_evaluation THEN 'Closed without evaluation'
        WHEN cea.has_closure AND NOT cea.has_approval THEN 'Closed without approval'
        WHEN cea.has_closure AND NOT cea.has_notification THEN 'Closed without notification'
        ELSE 'Normal process'
    END AS anomaly_type
FROM claims c
JOIN claim_events_aggregated cea ON c.claim_id = cea.claim_id
WHERE cea.has_closure = TRUE
AND (NOT cea.has_evaluation OR NOT cea.has_approval OR NOT cea.has_notification)
ORDER BY c.submission_date DESC;
```

### **Query 2: Detect Evaluation-Approval Loops**
```sql
-- Find claims with multiple evaluation-approval cycles
WITH event_sequences AS (
    SELECT 
        claim_id,
        activity,
        timestamp,
        LEAD(activity) OVER (PARTITION BY claim_id ORDER BY timestamp) AS next_activity
    FROM claim_events
)
SELECT 
    claim_id,
    COUNT(*) AS evaluation_approval_cycles,
    MIN(timestamp) AS first_cycle_start,
    MAX(timestamp) AS last_cycle_end,
    SUM(CASE WHEN activity = 'E' AND next_activity = 'P' THEN 1 ELSE 0 END) AS eval_to_approve_count,
    SUM(CASE WHEN activity = 'P' AND next_activity = 'E' THEN 1 ELSE 0 END) AS approve_to_reeval_count
FROM event_sequences
WHERE (activity = 'E' AND next_activity = 'P')
   OR (activity = 'P' AND next_activity = 'E')
GROUP BY claim_id
HAVING COUNT(*) > 1
ORDER BY evaluation_approval_cycles DESC;
```

### **Query 3: Analyze Notification Skip Patterns**
```sql
-- Investigate when and why notifications are skipped
SELECT 
    c.claim_type,
    a.specialization,
    a.region,
    COUNT(*) AS total_claims,
    SUM(CASE WHEN has_notification THEN 1 ELSE 0 END) AS notified_claims,
    SUM(CASE WHEN has_closure AND NOT has_notification THEN 1 ELSE 0 END) AS closed_without_notification,
    ROUND(100.0 * SUM(CASE WHEN has_closure AND NOT has_notification THEN 1 ELSE 0 END) / 
          NULLIF(SUM(CASE WHEN has_closure THEN 1 ELSE 0 END), 0), 2) AS notification_skip_rate_pct,
    AVG(EXTRACT(DAY FROM (closure_date - submission_date))) AS avg_days_to_closure
FROM (
    SELECT 
        ce.claim_id,
        c.claim_type,
        c.submission_date,
        BOOL_OR(activity = 'N') AS has_notification,
        BOOL_OR(activity = 'C') AS has_closure,
        MAX(CASE WHEN activity = 'C' THEN timestamp END) AS closure_timestamp
    FROM claim_events ce
    JOIN claims c ON ce.claim_id = c.claim_id
    GROUP BY ce.claim_id, c.claim_type, c.submission_date
) AS claim_summary
LEFT JOIN claim_events ce2 ON claim_summary.claim_id = ce2.claim_id AND ce2.activity = 'A'
LEFT JOIN adjusters a ON ce2.resource = a.adjuster_id::VARCHAR
GROUP BY c.claim_type, a.specialization, a.region
HAVING COUNT(*) > 10
ORDER BY notification_skip_rate_pct DESC;
```

### **Query 4: Process Flow Compliance Analysis**
```sql
-- Compare actual process flows against the ideal sequence
WITH actual_flows AS (
    SELECT 
        claim_id,
        STRING_AGG(activity, '' ORDER BY timestamp) AS actual_flow,
        BOOL_OR(activity = 'R') AS has_receive,
        BOOL_OR(activity = 'A') AS has_assign,
        BOOL_OR(activity = 'E') AS has_evaluate,
        BOOL_OR(activity = 'P') AS has_approve,
        BOOL_OR(activity = 'N') AS has_notify,
        BOOL_OR(activity = 'C') AS has_close
    FROM claim_events
    GROUP BY claim_id
)
SELECT 
    actual_flow,
    COUNT(*) AS frequency,
    ROUND(100.0 * COUNT(*) / SUM(COUNT(*)) OVER (), 2) AS percentage,
    CASE 
        WHEN actual_flow LIKE 'RAEPNC%' THEN 'Ideal flow'
        WHEN actual_flow LIKE 'RAC%' THEN 'Premature closure'
        WHEN actual_flow LIKE '%EPEP%' THEN 'Evaluation loop'
        WHEN actual_flow NOT LIKE '%N%' AND actual_flow LIKE '%C%' THEN 'Missing notification'
        ELSE 'Other deviation'
    END AS flow_category
FROM actual_flows
GROUP BY actual_flow
ORDER BY frequency DESC
LIMIT 20;
```

### **Query 5: Temporal Analysis of Anomalies**
```sql
-- Track anomaly trends over time
WITH monthly_stats AS (
    SELECT 
        DATE_TRUNC('month', c.submission_date) AS month,
        COUNT(*) AS total_claims,
        SUM(CASE WHEN ce.has_premature_closure THEN 1 ELSE 0 END) AS premature_closures,
        SUM(CASE WHEN ce.has_notification_skip THEN 1 ELSE 0 END) AS notification_skips,
        AVG(ce.evaluation_count) AS avg_evaluations_per_claim
    FROM claims c
    LEFT JOIN (
        SELECT 
            claim_id,
            BOOL_OR(activity = 'C' AND NOT EXISTS (
                SELECT 1 FROM claim_events ce2 
                WHERE ce2.claim_id = ce1.claim_id 
                AND ce2.activity IN ('E', 'P')
                AND ce2.timestamp < ce1.timestamp
            )) AS has_premature_closure,
            BOOL_OR(activity = 'C' AND NOT EXISTS (
                SELECT 1 FROM claim_events ce2 
                WHERE ce2.claim_id = ce1.claim_id 
                AND ce2.activity = 'N'
            )) AS has_notification_skip,
            COUNT(CASE WHEN activity = 'E' THEN 1 END) AS evaluation_count
        FROM claim_events ce1
        GROUP BY claim_id
    ) ce ON c.claim_id = ce.claim_id
    GROUP BY DATE_TRUNC('month', c.submission_date)
)
SELECT 
    month,
    total_claims,
    premature_closures,
    notification_skips,
    ROUND(100.0 * premature_closures / NULLIF(total_claims, 0), 2) AS premature_closure_rate,
    ROUND(100.0 * notification_skips / NULLIF(total_claims, 0), 2) AS notification_skip_rate,
    avg_evaluations_per_claim
FROM monthly_stats
ORDER BY month DESC;
```
