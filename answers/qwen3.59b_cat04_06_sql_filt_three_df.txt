Here is the DuckDB SQL query to solve the problem. It uses window functions to enforce strict sequential ordering within each case instance and filters based on the time gap before excluding those specific cases from the final result.

```sql
WITH RankedEvents AS (
    -- Assign a row number to every event within each case, ordered by timestamp
    SELECT 
        case_id, 
        activity, 
        timestamp,
        ROW_NUMBER() OVER (PARTITION BY case_id ORDER BY timestamp) as row_num
    FROM event_log
),
ProblematicCases AS (
    -- Identify cases containing the specific sequence with > 5 days delay
    SELECT DISTINCT r1.case_id
    FROM RankedEvents r1
    -- Join to find the immediate next event which is 'Create Order'
    JOIN RankedEvents r2 
        ON r1.case_id = r2.case_id 
        AND r2.row_num = r1.row_num + 1 
        AND r2.activity = 'Create Order'
    -- Join to find the immediate next event after that which is 'Confirm Order'
    JOIN RankedEvents r3 
        ON r2.case_id = r3.case_id 
        AND r3.row_num = r2.row_num + 1 
        AND r3.activity = 'Confirm Order'
    WHERE (r3.timestamp - r2.timestamp) > INTERVAL '5 days'
)
-- Return all events EXCEPT those belonging to the problematic cases identified above
SELECT case_id, activity, timestamp
FROM event_log
WHERE case_id NOT IN (SELECT case_id FROM ProblematicCases);
```

### Explanation of the Logic:

1.  **`RankedEvents` CTE**: 
    *   We use `ROW_NUMBER()` partitioned by `case_id` and ordered by `timestamp`. This creates a linear sequence number for every event within a specific process instance.
    *   This ensures that "direct succession" is checked using row offsets (`row_num + 1` and `row_num + 2`), guaranteeing no other events from the same case exist between the activities in the sequence.

2.  **`ProblematicCases` CTE**:
    *   We self-join the ranked events to connect an 'Approve Purchase Requisition' (start) -> 'Create Order' (middle) -> 'Confirm Order' (end).
    *   The join conditions `r2.row_num = r1.row_num + 1` and `r3.row_num = r2.row_num + 1` enforce the strict adjacency requirement.
    *   We filter where the time difference between 'Create Order' and 'Confirm Order' (`r3.timestamp - r2.timestamp`) is greater than 5 days (`INTERVAL '5 days'`).
    *   `SELECT DISTINCT case_id` ensures that if a case has multiple sequences, it is only flagged once for exclusion.

3.  **Final Selection**:
    *   We select all columns from the original `event_log` table but filter out any rows where `case_id` exists in the list of `ProblematicCases`. This effectively removes all events from cases that matched the negative criteria.
