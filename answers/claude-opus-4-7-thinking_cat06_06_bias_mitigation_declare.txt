## Rationale for Each Added Constraint

| Constraint Type | Added Rule | Bias-Mitigation Rationale |
|---|---|---|
| **existence** | `BiasMitigationCheck` must appear | Guarantees that every trace includes an explicit fairness-check step before closing. |
| **absence** | `Reject_DueToRace`, `Approve_DueToGender` must never occur | Forbids decisions whose label itself encodes an attribute-driven outcome. |
| **exactly_one** | Exactly one `FinalDecision` | Prevents re-opening cases to re-decide under different attribute information. |
| **responded_existence** | Any sensitive-attribute check implies `BiasMitigationCheck` exists | Knowledge of a protected attribute must always trigger a fairness review. |
| **coexistence** | `Approve_Minority` / `Reject_Minority` / `*_Senior`  `ManualReview` | Ensures decisions impacting protected demographic groups always go through a human ManualReview, avoiding automation-only discrimination. |
| **response** | `CheckApplicantRace/Gender/Age`  eventually `BiasMitigationCheck` | After seeing any sensitive attribute, the system must perform a bias check before finishing. |
| **precedence** | `BiasMitigationCheck`, `ManualReview` precede `FinalDecision` | No final decision can be issued without a prior fairness audit and review. |
| **succession** | `BiasMitigationCheck`  `FinalDecision` | Strengthens the above: mitigation is both required before and eventually followed by a decision. |
| **altprecedence** | `BiasMitigationCheck` must lie between sensitive check and `Approve`/`Reject` | Enforces a mitigation step between each sensitive exposure and subsequent decisions. |
| **chainprecedence** | `ManualReview` must immediately precede `*_Minority` decisions | The most sensitive decisions must be the direct result of human review瑈ot algorithmic shortcuts. |
| **noncoexistence** | `Reject` cannot coexist with a trace containing only race-check events | Rejects cannot be made in cases where the only recorded evaluation was a protected-attribute check. |
| **nonsuccession** | Sensitive checks cannot be eventually followed by `Approve`/`Reject` (without mitigation) | Blocks any path from a sensitive check to a decision that bypasses the mitigation workflow. |
| **nonchainsuccession** | Decisions cannot *directly* follow sensitive-attribute checks | Rules out immediate, attribute-driven reactions珳ecisions must never be one step away from a sensitive disclosure. |

## How These Constraints Reduce Bias

1. **Mandatory Fairness Checks:** `BiasMitigationCheck` is required to exist and to separate sensitive-attribute disclosures from decision steps, so protected attributes never directly drive outcomes.
2. **Human-in-the-loop for Protected Groups:** Coexistence and chain-precedence rules force `ManualReview` whenever decisions concern minority or senior applicants, preventing purely automated, potentially biased judgments.
3. **Elimination of Direct Attribute-to-Decision Paths:** Non-succession and non-chain-succession constraints cut off any trace in which a sensitive attribute check is (immediately or eventually) followed by a decision without a proper mitigation event.
4. **Label-Level Prohibitions:** Absence constraints ban activities whose labels imply the attribute caused the outcome, enforcing the principle that sensitive attributes must never be an explicit decision criterion.
5. **Decision Integrity:** `exactly_one` ensures decisions aren't renegotiated once new (possibly biased) information arrives, stabilizing the fairness guarantees of a trace.

Together, these constraints make biased sequences structurally impossible in any conforming trace.
