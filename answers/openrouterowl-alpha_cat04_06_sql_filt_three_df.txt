Here's a DuckDB SQL query that solves your problem step-by-step:

```sql
WITH ordered_events AS (
  -- Assign row numbers within each case to identify consecutive events
  SELECT 
    *,
    ROW_NUMBER() OVER (PARTITION BY case_id ORDER BY timestamp) as row_num
  FROM event_log
),

sequence_check AS (
  -- Find cases that contain the specific 3-activity sequence
  SELECT DISTINCT e1.case_id
  FROM ordered_events e1
  JOIN ordered_events e2 ON e1.case_id = e2.case_id AND e2.row_num = e1.row_num + 1
  JOIN ordered_events e3 ON e1.case_id = e3.case_id AND e3.row_num = e1.row_num + 2
  WHERE e1.activity = 'Approve Purchase Requisition'
    AND e2.activity = 'Create Order'
    AND e3.activity = 'Confirm Order'
),

timing_check AS (
  -- For cases with the sequence, check timing condition
  SELECT DISTINCT sc.case_id
  FROM sequence_check sc
  JOIN ordered_events e2 ON sc.case_id = e2.case_id AND e2.activity = 'Create Order'
  JOIN ordered_events e3 ON sc.case_id = e3.case_id AND e3.activity = 'Confirm Order'
  -- Ensure these are the consecutive events in the sequence
  WHERE e3.row_num = e2.row_num + 1
    AND EXTRACT(DAY FROM (e3.timestamp - e2.timestamp)) > 5
)

-- Return all events EXCEPT those from cases meeting both criteria
SELECT *
FROM event_log
WHERE case_id NOT IN (SELECT case_id FROM timing_check)
ORDER BY case_id, timestamp;
```

## How the Query Works:

### Step 1: `ordered_events` CTE
- Assigns sequential row numbers to events within each `case_id`, ordered by `timestamp`
- This ensures we can identify truly consecutive events (no gaps)

### Step 2: `sequence_check` CTE
- Uses self-joins to find cases where:
  - Event `n` = 'Approve Purchase Requisition'
  - Event `n+1` = 'Create Order'  
  - Event `n+2` = 'Confirm Order'
- The consecutive row numbers (`row_num`, `row_num+1`, `row_num+2`) guarantee there are no intervening events

### Step 3: `timing_check` CTE
- For cases identified in `sequence_check`, verifies that the time between 'Create Order' and 'Confirm Order' (which we already know are consecutive) exceeds 5 days
- Uses `EXTRACT(DAY FROM (timestamp - timestamp))` to get the day difference

### Step 4: Final Filter
- Returns all events from `event_log` **except** those belonging to cases that meet both:
  1. Have the exact consecutive sequence
  2. Have more than 5 days between 'Create Order' and 'Confirm Order'
- Results are ordered by `case_id` and `timestamp` for clarity

This approach ensures accurate identification of truly consecutive events and handles the exclusion logic correctly.
