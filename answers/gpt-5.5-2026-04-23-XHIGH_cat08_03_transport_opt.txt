# 1. Process Discovery and Conformance Checking

## 1.1 Preprocess and integrate GPS, scanner, dispatch, and maintenance data

Speedy Parcels should first create a unified, analysis-ready event log. Because the operation involves multiple related objects—vehicles, drivers, packages, stops, routes, and maintenance events—I would recommend both:

- A **classic event log** for common analyses, using **Vehicle-Day** as the main case ID, e.g., `V12-20241205`.
- An **object-centric event log** where one event can be linked to multiple objects, e.g., vehicle, driver, route, stop, package. This avoids distortion because one vehicle-day contains many packages, and one customer stop may involve multiple packages.

### Recommended event log structure

Each event should contain at minimum:

| Field | Example |
|---|---|
| Case ID | `V12-20241205` |
| Timestamp | `2024-12-05 08:25:10` |
| Activity | `Arrive Customer`, `Travel Leg`, `Delivery Failed`, `Unscheduled Stop` |
| Vehicle ID | `V12` |
| Driver ID | `D105` |
| Route ID | Planned route reference |
| Package ID | `P9876`, if relevant |
| Stop ID / Customer ID | Stop 1, customer location |
| Location | Lat/lon or geofence ID |
| Planned sequence number | Stop 1, Stop 2, etc. |
| Planned ETA / time window | From dispatch system |
| Actual duration attributes | Travel time, dwell time, idle time |
| Event source | GPS, scanner, dispatch, maintenance |
| Notes / reason codes | Customer not home, engine warning, etc. |

### Data integration steps

1. **Standardize timestamps and identifiers**
   - Synchronize clocks across GPS devices, scanners, dispatch, and maintenance systems.
   - Normalize time zones and daylight-saving changes.
   - Ensure vehicle IDs, driver IDs, package IDs, and route IDs are consistent.

2. **Transform GPS points into meaningful process events**
   GPS data is too granular to use directly as a process log. It needs abstraction:
   - Detect **Depart Depot** and **Arrive Depot** using depot geofences.
   - Detect **Arrive Customer** and **Depart Customer** using customer geofences and scanner events.
   - Detect **Travel Leg** between two stops.
   - Detect **Traffic Delay / Low Speed Segment** when speed remains below a threshold on the road network.
   - Detect **Unplanned Stop** when the vehicle is stationary outside depot/customer geofences for more than a defined duration.
   - Detect **Idle Time** when ignition is on but speed is zero.
   - Estimate **parking search time** from low-speed circling near the delivery address before the customer arrival scan.

3. **Integrate scanner events**
   Scanner events provide the actual delivery milestone sequence:
   - Scan at depot.
   - Arrive customer.
   - Delivery success.
   - Delivery failed.
   - Depart customer.

   These should be reconciled with GPS-based arrival/departure events. For example, if GPS shows the vehicle stopped at a customer location at 08:23 but the driver scanned “Arrive Customer” at 08:25, the two-minute difference may represent parking, walking, or delayed scanning.

4. **Add dispatch plan data**
   From the dispatch system, add:
   - Planned route.
   - Planned stop sequence.
   - Assigned packages.
   - Planned ETAs.
   - Customer-requested time windows.
   - Vehicle capacity.
   - Driver assignment.
   - Planned route distance and expected travel time.

5. **Link maintenance data**
   Maintenance logs should be connected to the vehicle timeline:
   - Scheduled maintenance periods.
   - Unscheduled repair events.
   - Engine warning or breakdown-related stops.
   - Vehicle unavailability.

6. **Enrich the log**
   Additional useful enrichment includes:
   - Weather data.
   - Road traffic data.
   - Day of week, season, holiday indicator.
   - Urban/suburban territory classification.
   - Package size, weight, or special handling requirements if available.
   - Fuel data from telematics, fuel cards, or estimated fuel models.

### Likely data challenges

Speedy Parcels should expect several data quality and modeling challenges:

- **Different data granularity:** GPS records frequent points, while scanner data only records milestones.
- **Missing scans:** Drivers may forget to scan arrival or departure.
- **GPS noise:** Urban areas may cause inaccurate coordinates due to tall buildings.
- **Ambiguous stops:** Multiple customers may be located in the same building or dense street.
- **Timestamp mismatches:** Scanner time and GPS time may not be perfectly synchronized.
- **Many-to-many relationships:** One vehicle serves many packages; one customer stop may include several deliveries.
- **Route changes after dispatch:** Dispatch plans may be modified during the day.
- **Data volume:** Six months of frequent GPS pings can be very large.
- **Distinguishing delay types:** A stationary vehicle could indicate traffic, parking, a break, a customer interaction, or a mechanical issue.

## 1.2 Discover the actual end-to-end delivery process

Once the event log is prepared, process discovery algorithms can reveal what actually happens in daily operations.

Useful techniques include:

- **Inductive Miner** for a clean, structured high-level process model.
- **Heuristics Miner** for noisy logistics data where activities may vary.
- **Fuzzy Miner** for highly variable “spaghetti” processes with many route variants.
- **Variant analysis** to identify common and exceptional route execution patterns.
- **Object-centric process mining** to analyze vehicles, drivers, packages, and maintenance events together.
- **GIS-based process visualization** to overlay routes, delays, and stop behavior on a map.

A discovered high-level delivery process might look like:

```text
Start Shift
 Route Assigned
 Load / Scan at Depot
 Depart Depot
 [Travel Leg  Arrive Customer  Delivery Success/Failed  Depart Customer] repeated many times
 Optional: Traffic Delay / Unplanned Stop / Maintenance Stop
 Arrive Depot
 End Shift
```

The discovered process model should show:

- How often routes start late.
- Whether drivers follow the planned sequence.
- How many stops are completed successfully.
- How often delivery failures occur.
- How frequently unscheduled stops happen.
- How much time is spent driving, waiting, idling, servicing customers, or dealing with exceptions.
- Which process variants lead to overtime or late deliveries.

For example, one high-performing variant may be:

```text
Depart Depot  Stop 1 Success  Stop 2 Success  ...  Return Depot on time
```

A problematic variant may be:

```text
Depart Depot late  Traffic Delay  Stop sequence deviation  Delivery Failed  Unplanned Stop  Return Depot late  Overtime
```

## 1.3 Conformance checking against planned routes

Conformance checking compares the planned process from the dispatch system with the actual process reconstructed from GPS and scanner events.

The planned route can be modeled as a case-specific expected trace:

```text
Depart Depot
 Visit Stop 1
 Visit Stop 2
 Visit Stop 3
 ...
 Visit Stop 35
 Return Depot
```

Each planned stop also has expected attributes:

- Planned ETA.
- Customer time window.
- Expected service duration.
- Expected travel time from previous stop.
- Planned route distance.
- Package assignment.
- Vehicle capacity constraints.

### Deviations to detect

Conformance checking should identify several types of deviations.

#### Sequence deviations

Examples:

- Driver visits Stop 5 before Stop 3.
- Driver skips a planned stop.
- Driver serves an unplanned stop.
- Driver returns to a previously visited area.
- Driver crosses into another territory.
- Driver returns to the depot before completing the route.

#### Timing deviations

Examples:

- Late depot departure.
- Arrival after the customer time window.
- Travel leg takes significantly longer than planned.
- Service time at customer exceeds expected duration.
- Route return occurs after planned shift end.
- Overtime is required.

#### Geographic deviations

Examples:

- Actual route distance is much longer than planned.
- Vehicle leaves the assigned service area.
- Driver takes repeated detours.
- Vehicle stops at locations not associated with customers, depot, or planned breaks.

#### Operational deviations

Examples:

- Unplanned maintenance stop.
- Long idle time with ignition on.
- Failed delivery attempt.
- Missing scanner events.
- Delivery marked successful far away from customer location.
- Package assigned to route but never attempted.

### Conformance metrics

Useful conformance metrics include:

- **Route sequence adherence score.**
- **Number of skipped planned stops.**
- **Number of unplanned stops.**
- **Actual vs. planned distance difference.**
- **Actual vs. planned route duration difference.**
- **ETA deviation per stop.**
- **Time window violation minutes.**
- **Conformance cost from alignment analysis**, where extra, missing, reordered, and late events receive penalties.

This allows Speedy Parcels to distinguish between acceptable deviations, such as dynamic rerouting to avoid congestion, and problematic deviations, such as avoidable detours or unauthorized stops.

# 2. Performance Analysis and Bottleneck Identification

## 2.1 Key performance indicators

The following KPIs are directly relevant to Speedy Parcels’ goals of improving punctuality and reducing operating costs.

| KPI | Calculation from event log | Purpose |
|---|---|---|
| **On-Time Successful Delivery Rate** | Successful deliveries within customer time window / total planned deliveries | Measures customer-facing punctuality |
| **On-Time Attempt Rate** | Delivery attempts within time window / total planned attempts | Separates operational punctuality from customer absence |
| **Failed Delivery Rate** | Failed delivery attempts / total delivery attempts | Measures re-delivery burden |
| **First-Attempt Success Rate** | Packages delivered successfully on first attempt / total packages attempted | Important for cost and customer satisfaction |
| **Average Stop Service Time** | `Depart Customer - Arrive Customer` | Measures customer stop efficiency |
| **Average Travel Time per Leg** | `Arrive at Stop N - Depart Stop N-1` | Measures travel efficiency |
| **Travel Time vs. Service Time Ratio** | Total travel time / total customer service time | Shows whether delays come mainly from driving or customer stops |
| **Route Duration** | `Arrive Depot - Depart Depot` or `End Shift - Start Shift` | Measures total route workload |
| **Depot Departure Delay** | Actual depot departure time - planned departure time | Detects start-of-day bottlenecks |
| **ETA Accuracy** | Actual arrival time - planned ETA per stop | Measures planning accuracy |
| **Time Window Violation Minutes** | Sum of minutes late beyond promised windows | Quantifies punctuality impact |
| **Distance per Package** | Actual distance traveled / packages delivered | Measures route efficiency |
| **Fuel Consumption per km/package** | Fuel used / km or packages delivered | Measures cost and environmental efficiency |
| **Idle Time Percentage** | Idle time / route duration | Strong driver of fuel cost |
| **Vehicle Capacity Utilization** | Loaded volume or package count / vehicle capacity | Measures asset utilization |
| **Vehicle Time Utilization** | Driving + service time / available shift time | Measures productivity |
| **Traffic Delay Duration** | Actual leg time - benchmark/free-flow/planned leg time | Quantifies congestion impact |
| **Unplanned Stop Frequency** | Count of unplanned stops per route/day | Detects operational disruption |
| **Maintenance Disruption Rate** | Routes with unscheduled vehicle issue / total routes | Measures reliability impact |
| **Overtime Rate** | Routes ending after planned shift / total routes | Measures labor cost pressure |

If fuel consumption is not directly available, it can be estimated using:

- Distance traveled.
- Vehicle type.
- Speed profile.
- Idle duration.
- Stop-and-go frequency.
- Historical fuel card or telematics data.

## 2.2 Process mining techniques for bottleneck identification

### Performance-annotated process maps

A process map can show each activity with:

- Average duration.
- Median duration.
- Waiting time.
- Frequency.
- Deviation rate.
- Number of late cases passing through that activity.

This will help identify whether the main bottlenecks are:

- Late route assignment.
- Late depot departure.
- Long travel legs.
- Long customer dwell times.
- Failed deliveries.
- Unplanned stops.
- Maintenance events.
- Return-to-depot delays.

### Variant analysis

Compare route variants such as:

- Routes completed on time.
- Routes with overtime.
- Routes with many failed deliveries.
- Routes with high fuel consumption.
- Routes involving vehicle breakdowns.
- Routes with many sequence deviations.

This reveals which behavioral patterns correlate with poor performance.

### Dotted charts and time-based views

Dotted charts can show event timing across many vehicle-days. They are useful for detecting:

- Morning departure congestion at the depot.
- Routes that consistently fall behind after a certain hour.
- Afternoon traffic waves.
- Long gaps between customer visits.
- End-of-day bunching and overtime.

### Geographic heatmaps

Using GPS and stop data, Speedy Parcels can create heatmaps for:

- Traffic delay hotspots.
- Long service-time locations.
- Failed delivery hotspots.
- Unplanned stop locations.
- High-idling zones.
- Parking difficulty areas.

This is especially valuable in dense urban delivery operations.

### Resource and driver analysis

Resource analysis can compare drivers on:

- Average service time.
- Idle time.
- Route adherence.
- Fuel efficiency.
- Scan compliance.
- Failed delivery documentation quality.
- Overtime frequency.

However, driver comparisons must be normalized for route difficulty, area, package count, customer type, traffic, and vehicle type.

### Impact quantification

For each bottleneck, quantify:

- Total minutes lost.
- Number of late deliveries caused or affected.
- Extra kilometers driven.
- Additional fuel cost.
- Additional labor cost from overtime.
- Re-delivery cost.
- Number of packages delayed.
- Customer satisfaction impact.

A useful decomposition is:

```text
Total Route Delay =
Depot Departure Delay
+ Travel Delay
+ Customer Service Delay
+ Failed Delivery/Re-delivery Delay
+ Unplanned Stop Delay
+ Maintenance Delay
+ Return-to-Depot Delay
```

This helps prioritize the bottlenecks with the largest operational and financial impact.

# 3. Root Cause Analysis for Inefficiencies

Identifying that a route is late is not enough. Speedy Parcels needs to understand why routes are late or costly.

## 3.1 Suboptimal route planning

### Possible root cause

Routes may be planned statically and may not reflect actual traffic, parking difficulty, customer availability, or service-time variability.

### Process mining validation

Look for:

- Frequent actual stop-sequence deviations.
- Repeated driver detours from the planned path.
- Actual distance much greater than planned distance.
- Backtracking between neighborhoods.
- Routes where drivers consistently reorder stops and perform better than the plan.
- Routes with unrealistic planned completion times.

### Analysis method

Compare planned and actual route sequences using conformance checking and sequence alignment. If high-performing drivers frequently deviate from the plan in similar ways, the planned sequence itself may be inefficient.

---

## 3.2 Inaccurate travel time estimations

### Possible root cause

The dispatch system may use average travel times that do not reflect time-of-day traffic, road restrictions, school zones, parking conditions, or seasonal congestion.

### Process mining validation

Analyze actual versus planned travel time by:

- Road segment.
- Time of day.
- Day of week.
- Territory.
- Weather condition.
- Urban versus suburban area.

For example:

```text
Planned Leg Time: 8 minutes
Actual Median Leg Time between 08:00 and 09:00: 17 minutes
```

This indicates that the planning model underestimates morning congestion.

---

## 3.3 Traffic congestion patterns

### Possible root cause

Certain routes may pass through predictable congestion hotspots during morning or afternoon peaks.

### Process mining validation

Use GPS-derived low-speed events and traffic-delay heatmaps to identify:

- Roads where vehicles repeatedly travel below expected speed.
- Time windows with recurring delay.
- Routes that pass through congested zones before late deliveries occur.
- Correlation between traffic delay events and missed time windows.

### Analysis method

Compare routes with similar package counts but different traffic exposure. If routes crossing specific congestion zones have higher lateness and fuel consumption, traffic is a likely root cause.

---

## 3.4 High variability in customer service time

### Possible root cause

Some customer stops take longer due to:

- Apartment buildings.
- Lack of parking.
- Security access.
- Signature requirements.
- Heavy or bulky parcels.
- Customer unavailability.
- Poor address quality.
- Driver searching for entrance or delivery point.

### Process mining validation

Measure dwell time:

```text
Customer Dwell Time = Depart Customer - Arrive Customer
```

Then segment by:

- Customer location.
- Building type.
- Package type.
- Driver.
- Time of day.
- Delivery outcome.
- Number of packages per stop.

Also separate the stop into subcomponents where possible:

```text
Parking/Search Time
+ Walking/Access Time
+ Customer Interaction Time
+ Scanner/Admin Time
```

Long and variable dwell times indicate that planning should use customer-specific or area-specific service-time estimates rather than one generic average.

---

## 3.5 Vehicle breakdowns and maintenance needs

### Possible root cause

Older or heavily used vehicles may be more likely to experience disruptions during routes.

### Process mining validation

Link vehicle maintenance logs to route performance:

- Routes with engine warning events.
- Unscheduled stops followed by repair logs.
- Vehicles with repeated downtime.
- Breakdown frequency per 1,000 km.
- Delay caused by maintenance-related route interruptions.
- Relationship between vehicle age, mileage, idling, and repair frequency.

If certain vehicles repeatedly appear in late-route variants, maintenance reliability is a root cause.

---

## 3.6 Driver behavior or skill differences

### Possible root cause

Differences may exist in:

- Route adherence.
- Parking choices.
- Scanner discipline.
- Eco-driving behavior.
- Idling habits.
- Customer interaction efficiency.
- Ability to recover from delays.

### Process mining validation

Use resource analysis, but control for route complexity. Compare drivers only after adjusting for:

- Number of stops.
- Route distance.
- Urban density.
- Traffic exposure.
- Package mix.
- Vehicle type.
- Time windows.

Look for patterns such as:

- Consistently longer service times.
- Frequent unnecessary sequence deviations.
- Higher idle time.
- More missing scans.
- Higher fuel consumption per kilometer.
- Higher late delivery rate on comparable routes.

The goal should be targeted coaching, not blame.

---

## 3.7 Failed delivery attempts and re-delivery loops

### Possible root cause

Failed deliveries may be caused by:

- Customers not home.
- Arrival outside preferred time.
- Poor communication of ETA.
- Inaccurate address data.
- Access restrictions.
- Time windows that are too broad or unrealistic.
- Deliveries attempted at historically low-success times.

### Process mining validation

Analyze package-level process variants:

```text
Out for Delivery  Delivery Failed  Return Depot  Replanned  Out for Delivery Again  Success
```

Quantify:

- Extra kilometers caused by re-delivery.
- Extra labor time.
- Delay to final delivery.
- Failed delivery hotspots.
- Failure rate by time of day.
- Failure rate by customer segment.
- Failure rate by driver or territory.

This reveals whether failed deliveries are primarily a customer availability issue, planning issue, address quality issue, or communication issue.

# 4. Data-Driven Optimization Strategies

Below are six concrete optimization strategies. Speedy Parcels should prioritize them based on quantified impact from the process mining analysis.

---

## Strategy 1: Implement dynamic routing and real-time ETA adjustment

### Targets

- Late deliveries caused by traffic congestion.
- Static route plans that do not adapt to disruptions.
- Inaccurate planned arrival times.
- Excess fuel usage from congestion and detours.

### Root cause addressed

The current routing may be too static and may not incorporate real-time congestion, historical route performance, or current route progress.

### Process mining support

Process mining can identify:

- Recurrent traffic-delay hotspots.
- Time periods where planned travel times are systematically wrong.
- Routes where actual driver deviations outperform the planned sequence.
- Stops most at risk of missing their time windows.

### Concrete actions

- Use historical actual travel times by road segment, day, and time.
- Integrate real-time traffic data into dispatch decisions.
- Re-optimize remaining stops during the route.
- Send updated ETAs to customers.
- Allow approved dynamic deviations while still tracking conformance against the updated plan.

### Expected KPI impact

- Higher On-Time Delivery Rate.
- Lower average travel delay.
- Lower overtime rate.
- Reduced fuel consumption per package.
- Improved ETA accuracy.
- Better customer satisfaction.

---

## Strategy 2: Redesign delivery territories and route sequences using historical performance data

### Targets

- Excess kilometers.
- Backtracking.
- Unbalanced route workloads.
- Routes that consistently require overtime.
- Poor vehicle utilization.

### Root cause addressed

Territories or routes may be designed based on geography alone rather than actual service time, traffic patterns, parking difficulty, and customer availability.

### Process mining support

Process mining can show:

- Actual versus planned route distance.
- Repeated sequence deviations.
- High-performing versus low-performing route variants.
- Areas with systematically long dwell times.
- Routes where package count appears reasonable but actual workload is too high.

### Concrete actions

- Re-cluster delivery territories based on actual historical workload, not only distance.
- Use route optimization with realistic service-time and travel-time distributions.
- Assign more difficult urban stops to smaller routes.
- Avoid combining multiple high-delay zones into one route.
- Balance workload across drivers and vehicles.
- Recalculate stop sequencing using actual historical performance.

### Expected KPI impact

- Reduced kilometers per package.
- Lower route duration variance.
- Higher vehicle utilization.
- Fewer overtime routes.
- Improved On-Time Delivery Rate.
- Lower fuel cost.

---

## Strategy 3: Improve customer time-window management and communication

### Targets

- Failed delivery attempts.
- Re-delivery cost.
- Customer-not-home events.
- Late or poorly timed attempts.

### Root cause addressed

Customers may not receive accurate ETAs or may be assigned delivery windows that do not match actual route feasibility or customer availability.

### Process mining support

Analyze:

- Failed delivery variants.
- Failure rate by time of day.
- Failure rate by customer area.
- Failure rate by planned versus actual arrival time.
- Packages requiring multiple attempts.
- Redelivery loops and their cost.

### Concrete actions

- Send proactive delivery notifications with accurate ETA.
- Offer customer self-service rescheduling.
- Introduce narrower but reliable delivery windows.
- Use parcel lockers or pickup points in high-failure areas.
- Prioritize stops with confirmed availability.
- Avoid attempting historically high-failure customers at low-success times.
- Flag risky addresses before dispatch.

### Expected KPI impact

- Higher First-Attempt Success Rate.
- Lower Failed Delivery Rate.
- Fewer re-delivery kilometers.
- Lower cost per delivered package.
- Shorter package cycle time.
- Improved customer satisfaction.

---

## Strategy 4: Develop predictive maintenance and smarter vehicle assignment

### Targets

- Unscheduled stops.
- Breakdowns during shifts.
- Maintenance-related delays.
- High repair costs.
- Poor vehicle availability.

### Root cause addressed

Maintenance may be reactive rather than predictive. Vehicles with higher breakdown risk may be assigned to long or time-critical routes.

### Process mining support

Link route performance with:

- Vehicle age.
- Mileage.
- Idle time.
- Engine warning events.
- Historical repair frequency.
- Unscheduled stop patterns.
- Maintenance downtime.

Identify vehicles that frequently appear in variants containing:

```text
Unscheduled Stop  Delay  Return Late  Repair
```

### Concrete actions

- Build a predictive maintenance model using usage and event history.
- Schedule maintenance before high-risk vehicles fail during routes.
- Assign high-risk vehicles to shorter or less time-critical routes until serviced.
- Reserve backup vehicles for critical routes.
- Monitor engine-warning events in real time.

### Expected KPI impact

- Lower unscheduled maintenance rate.
- Reduced route disruption.
- Higher vehicle availability.
- Lower repair cost.
- Fewer late deliveries caused by breakdowns.
- Less overtime from vehicle incidents.

---

## Strategy 5: Targeted driver coaching and standard operating procedures

### Targets

- High idle time.
- Long customer dwell time.
- Inefficient route deviations.
- Poor scanner compliance.
- High fuel consumption.
- Excessive stop-to-stop variability.

### Root cause addressed

Some inefficiencies may be caused by inconsistent operating practices, lack of training, or inefficient local decision-making.

### Process mining support

Resource analysis can identify drivers who, after controlling for route difficulty, have:

- Higher service times.
- Higher idle percentages.
- More missing scans.
- Higher fuel consumption per kilometer.
- More sequence deviations.
- More overtime.

Also identify high-performing drivers and extract best practices from their variants.

### Concrete actions

- Provide eco-driving training.
- Coach drivers on reducing idling.
- Train drivers on efficient parking and delivery procedures.
- Improve scanner discipline.
- Share route-specific best practices.
- Provide driver feedback dashboards.

### Expected KPI impact

- Lower fuel consumption.
- Shorter service time per stop.
- Better data quality.
- Lower idle time.
- Improved route adherence.
- Reduced overtime.

---

## Strategy 6: Improve depot dispatch and loading readiness

### Targets

- Late depot departures.
- Morning operational delays.
- Poor loading sequence.
- Drivers losing time searching for packages in the vehicle.

### Root cause addressed

Delivery routes may start late because packages are not staged, loaded, or scanned efficiently before departure.

### Process mining support

Analyze the pre-route process:

```text
Start Shift  Route Assigned  Scan at Depot  Load Vehicle  Depart Depot
```

Look for:

- Long waiting time between shift start and route assignment.
- Long loading duration.
- Late package availability.
- Rework due to missing or misassigned packages.
- Departure delay correlation with later missed time windows.

### Concrete actions

- Stage packages by route and stop sequence.
- Pre-load high-priority routes earlier.
- Use loading checklists and exception alerts.
- Assign routes before driver arrival where possible.
- Monitor depot readiness before shift start.

### Expected KPI impact

- Lower depot departure delay.
- Higher On-Time Delivery Rate.
- Less route compression later in the day.
- Lower overtime.
- Better driver productivity.

# 5. Operational Constraints and Continuous Monitoring

## 5.1 Accounting for operational constraints

Any optimization must respect real-world logistics constraints.

### Driver working hours

Dynamic routing and route redesign must consider:

- Legal shift limits.
- Planned breaks.
- Maximum overtime.
- Required return-to-depot time.
- Driver fatigue and safety.

The optimization engine should not simply add more stops to an already overloaded route.

### Vehicle capacity

Route plans must respect:

- Package count.
- Weight.
- Volume.
- Vehicle type.
- Loading order.
- Special handling requirements.

Dynamic reordering should also consider whether a package is physically accessible in the van.

### Customer time windows

Routing must prioritize:

- Narrow time windows.
- Premium delivery commitments.
- Customers with confirmed availability.
- Locations with restricted access hours.
- Business deliveries that close early.

### Vehicle availability and maintenance

The planning system should consider:

- Scheduled maintenance.
- Vehicle health risk.
- Vehicle type suitability.
- Backup vehicle availability.

### Geographic and regulatory constraints

Routes may also need to consider:

- Low-emission zones.
- Parking restrictions.
- Road closures.
- Weight or height restrictions.
- School zones.
- Urban access rules.

### Operational acceptance

Drivers and dispatchers must trust the recommendations. Therefore:

- Dynamic rerouting should be explainable.
- Dispatchers should see why a route is at risk.
- Drivers should receive clear, safe instructions.
- Approved route changes should be recorded as updated plans, not treated as unauthorized deviations.

After implementing improvements, Speedy Parcels should establish a continuous monitoring dashboard.

### Core dashboard views

#### 3. Cost and productivity dashboard

- Fuel consumption per km.
- Fuel consumption per package.
- Idle time percentage.
- Distance per package.
- Average route duration.
- Average stops per hour.
- Overtime rate.

#### 4. Bottleneck dashboard

Monitor:

- Longest travel legs.
- Highest-delay road segments.

#### 5. Geographic heatmaps

Display:

- Traffic delay hotspots.

#### 6. Maintenance monitoring

Track:

- Unscheduled stops by vehicle.

#### 7. Data quality dashboard

Monitor:

- Missing scanner events.
- GPS gaps.
- Timestamp inconsistencies.
- Delivery scans far from customer location.
- Unmatched packages or stops.

Poor data quality can create false process mining conclusions, so it should be monitored continuously.

---

## 5.3 Continuous improvement cycle

Speedy Parcels should use process mining as an ongoing management system, not a one-time analysis.

Recommended cycle:

1. **Establish baseline KPIs** using the past six months of data.
2. **Identify top bottlenecks** by cost and punctuality impact.
3. **Pilot optimization strategies** on selected territories or routes.
4. **Compare pilot versus control routes** using process mining dashboards.
5. **Quantify improvement** in on-time rate, fuel, distance, overtime, and failed deliveries.
6. **Roll out successful strategies** gradually.
7. **Continuously recalibrate** travel-time, service-time, and maintenance-risk models.
8. **Alert dispatchers in real time** when routes deviate or become at risk.
9. **Review weekly and monthly trends** with operations, dispatch, maintenance, and driver teams.

# Summary

Process mining can help Speedy Parcels move from anecdotal explanations of lateness and high cost to evidence-based operational improvement.

The most important analytical steps are:

1. Build a high-quality, integrated event log from GPS, scanner, dispatch, and maintenance data.
2. Discover the actual delivery process and its common variants.
3. Compare actual execution against planned routes through conformance checking.
4. Quantify bottlenecks using KPIs such as on-time delivery rate, travel delay, dwell time, failed delivery rate, fuel consumption, idle time, and maintenance disruption.
5. Validate root causes through variant analysis, spatial analysis, dwell-time analysis, resource analysis, and maintenance-event correlation.
6. Implement targeted improvements such as dynamic routing, territory redesign, customer communication, predictive maintenance, driver coaching, and depot loading optimization.
7. Monitor the operation continuously through process mining dashboards.

The expected business impact is improved delivery punctuality, lower fuel and maintenance cost, fewer failed deliveries, better vehicle utilization, reduced overtime, and more reliable customer service.
