        ('RO', 'CA'),

        ('RO', 'GSL'),

### Explanation of the modeling choices

- **equivalence**: In this strictly sequential happy-path process, every activity occurs exactly once per case, so each consecutive pair has equal occurrences. (Adjacent pairs are listed; equivalence is transitive across all activities.)

- **always_before**: Captures the strict ordering — e.g., `('RI','CA')` means *Reserve Items* requires *Check Availability* to have occurred earlier. Each downstream activity lists its key predecessors.

- **always_after**: The mirror constraint — e.g., `('RO','D')` means once an order is received, it must eventually be dispatched.

- **never_together**: Empty, because the described scenario is a single linear flow with no exclusive choices (e.g., no "cancel order" vs. "fulfill order" branch).

- **activ_freq**: Each activity occurs exactly once `{1}` in a normal fulfillment case.

- **directly_follows**: Encodes the immediate sequence:
  `RO  CA  RI  PI  QC  PO  GSL  D  IC  RP`.

> **Note:** This model assumes the "happy path" where every order is in stock, passes quality check, and is paid. If you wanted to model real-world variations (e.g., out-of-stock leading to back-order, or failed QC leading to re-picking), you would relax `activ_freq` (e.g., `{0,1}` for optional activities), add `never_together` pairs for exclusive branches, and remove some `directly_follows` constraints to allow alternative routings.
