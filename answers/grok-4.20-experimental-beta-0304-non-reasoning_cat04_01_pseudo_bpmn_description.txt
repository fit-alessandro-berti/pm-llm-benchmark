**Detailed Explanation of the Business Process**

This pseudo-BPMN diagram describes an **Order Handling / Quotation Process** that differentiates between **Standard** and **Custom** customer requests. The process is designed to validate incoming customer requests, perform the appropriate checks or analyses, optionally obtain managerial approval, generate an invoice/quotation, and finally confirm the order with the customer.

### 1. Start of the Process
- **Start Event  Task A: "Receive Customer Request"**  
  The process begins when a customer request (e.g., purchase order, quotation request, or service inquiry) is received. This task typically involves logging the request, capturing key details (customer information, requested items, quantities, special requirements, etc.), and creating a case or order record in the system.

### 2. Request Type Evaluation
- **Gateway (XOR): Check Request Type**  
  An **exclusive (XOR) gateway** that routes the process into one of two mutually exclusive paths based on whether the request is classified as **Standard** or **Custom**. Only one path is taken.

#### Path A: Standard Request
- **Task B1: "Perform Standard Validation"**  
  Basic validation of the order (checking data completeness, customer eligibility, order format, etc.).

- **Gateway (AND): "Run Parallel Checks"**  
  A **parallel (AND) gateway** that splits the flow into two concurrent tasks:
  - **Task C1: "Credit Check"** – Verifies the customer’s creditworthiness or payment reliability.
  - **Task C2: "Inventory Check"** – Checks whether the requested goods are in stock or can be procured in time.

- **AND Join (All Parallel Checks Completed)**  
  The process waits until **both** the Credit Check and Inventory Check are finished before proceeding.

- **Task D: "Calculate Delivery Date"**  
  Based on inventory availability, production/lead times, and logistics, the expected delivery date is calculated.

#### Path B: Custom Request
- **Task B2: "Perform Custom Feasibility Analysis"**  
  A more in-depth technical or engineering analysis to determine whether the customer’s special/custom requirements can be met (e.g., non-standard specifications, unique materials, special configurations).

- **Gateway (XOR): "Is Customization Feasible?"**  
  Another exclusive gateway with two possible outcomes:
  - **[Yes]**  **Task E1: "Prepare Custom Quotation"** (includes pricing, lead time, terms for the custom solution).
  - **[No]**  **Task E2: "Send Rejection Notice"**  **End Event**  
    The request is rejected early, and the customer is informed.

### 3. Convergence Point & Approval Decision (Common to Both Paths)
After either the Standard path (Task D) or the successful Custom path (Task E1) completes, the flows converge.

- **Gateway (XOR): "Is Approval Needed?"**  
  An exclusive gateway that decides whether managerial or financial approval is required. This decision is typically based on rules such as:
  - Order value (high-value orders need approval)
  - Special/custom nature of the request
  - Customer risk profile
  - Deviations from standard terms

  - **[If No Approval Needed]**  Directly to **Task G: "Generate Final Invoice"** (or formal quotation/confirmation).

  - **[If Yes]**  **Task F: "Obtain Manager Approval"**

    - **Gateway (XOR): "Is Approval Granted?"**
      - **[Yes]**  **Task G: "Generate Final Invoice"**
      - **[No]**  **Task H: "Re-evaluate Conditions"**

        - This task triggers a **loop back**:
          - For **Custom** requests  loops back to **Task E1** ("Prepare Custom Quotation") to adjust terms, pricing, or scope.
          - For **Standard** requests  loops back to **Task D** ("Calculate Delivery Date") to revise the delivery promise or other parameters.

The loop allows the organization to rework the proposal instead of outright rejecting the request when approval is denied.

### 4. Final Steps (Common to All Successful Paths)
- **Task I: "Send Confirmation to Customer"**  
  After the invoice/quotation is generated and all approvals are obtained, the customer receives official confirmation, which may include the final quotation, order acknowledgment, delivery schedule, and payment terms.

- **End Event**  
  The process terminates successfully (or after a rejection in the custom infeasible case).

---

### Summary of Key Differences Between Standard and Custom Paths

| Aspect                        | Standard Path                              | Custom Path                                      |
|-------------------------------|--------------------------------------------|--------------------------------------------------|
| Validation/Analysis           | Standard Validation + Parallel Credit & Inventory Checks | Detailed Custom Feasibility Analysis            |
| Parallel Work                 | Yes (Credit + Inventory)                   | No                                               |
| Decision after Analysis       | Always proceeds to delivery date calc.     | Feasibility decision (Yes  Quotation, No  Reject) |
| Typical Complexity            | Lower                                      | Higher (may involve engineering, R&D, special pricing) |
| Loop-back Target (if approval denied) | Task D (Calculate Delivery Date)      | Task E1 (Prepare Custom Quotation)              |

### Purpose of Gateways

- **XOR Gateways**: Used for **mutually exclusive decisions** (request type, feasibility, approval needed, approval granted). Only one outgoing path is taken.
- **AND Gateway**: Used to **split** into parallel independent checks (Credit + Inventory) and to **join** them, ensuring both complete before continuing.
- **Loop-back**: Implemented via the "Re-evaluate Conditions" task pointing back to earlier tasks, allowing iterative improvement of the proposal rather than immediate rejection.

This process ensures that standard orders are handled efficiently with parallel processing, while custom orders receive thorough feasibility scrutiny. It also incorporates a controlled rework loop to maximize the chance of closing the deal when initial approval is withheld.
