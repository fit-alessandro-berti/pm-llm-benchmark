# Optimizing Resource Assignment at TechSolve Solutions: A Process Mining Approach

## Introduction

TechSolve's challenges—SLA breaches, excessive reassignments, skill mismatches, and uneven workload—are classic symptoms of a resource allocation process that has grown organically without data-driven governance. The event log they've collected is rich enough to support a rigorous, resource-centric process mining analysis. Below I lay out a comprehensive methodology and concrete recommendations across the five requested dimensions.

A critical preprocessing note: the log mixes `START`/`COMPLETE` lifecycle transitions, which is excellent—it allows me to distinguish **processing time** (active work) from **waiting time** (queue/idle). I'll exploit this throughout, since the distinction is central to diagnosing assignment problems.

## 1. Analyzing Resource Behavior and Assignment Patterns

### 1.1 Agent- and Tier-Level Performance Metrics

Using the lifecycle-enriched log, I would compute the following metrics, sliced by agent, tier, priority, category, and required skill:

| Metric | Definition | What It Reveals |
|---|---|---|
| **Workload distribution** | Count of tickets handled / active hours per agent per period | Over- vs. under-utilization |
| **Active processing time** | `Work X End`  `Work X Start` | True effort per task type |
| **Waiting time (queue time)** | `Work X Start`  `Assign X` | Resource availability bottlenecks |
| **First-Contact Resolution (FCR) rate (L1)** | % L1 tickets closed without escalation | L1 effectiveness |
| **Escalation rate** | % tickets escalated per agent/category | Triage quality |
| **Reassignment rate** | Avg. `Reassign` events per case | Assignment accuracy |
| **Skill-match handling frequency** | Distribution of `Required Skill` vs. `Agent Skills` | Whether agents work in-skill |
| **SLA attainment** | % tickets meeting priority-based SLA targets | Outcome metric |

For example, from the snippet, INC-1001 shows a **30-minute queue delay** at L2 (`Assign L2` 09:40  `Work L2 Start` 10:05) — a quantifiable waiting-time signal. I'd aggregate these into agent "performance fingerprints" and tier-level dashboards.

### 1.2 Discovering *Actual* Assignment Patterns

The current intended logic is "round-robin within tiers + manual escalation." Process mining lets me reconstruct what is **actually** happening:

- **Handover-of-Work Social Network Analysis (SNA):** Build a directed graph where nodes = agents/dispatchers and edges = ticket handovers (e.g., A05  Dispatcher  B12  B15 for INC-1001). Edge weights = handover frequency. This exposes:
  - **Handover hubs** (dispatchers or "go-to" specialists receiving disproportionate flow).
  - **Ping-pong loops** (tickets bouncing between the same agents).
  - **Hidden informal routing** that bypasses official channels.

- **Role Discovery (organizational mining):** Cluster agents by the *activities and ticket types they actually perform* rather than their nominal tier. This often reveals that documented tiers don't match reality—e.g., an L2 agent functionally doing L1 work, or vice versa.

- **Resource interaction & subcontracting patterns:** Detect cases where work returns to a prior tier (rework loops), indicating misrouting.

- **Conformance checking:** Encode the intended model (L1  escalate  dispatcher assigns  L2/L3) as a reference and replay the log to measure **deviation frequency** — e.g., self-assignments (A02 in INC-1002 created, self-assigned, *and* worked the ticket), skipped triage steps, or direct assignments bypassing the dispatcher.

### 1.3 Skill Utilization Analysis

The standout opportunity. By joining `Required Skill` with `Agent Skills`:

- **Skill-mismatch ratio:** % of work items where the required skill is *not* in the agent's skill set (a hard failure  predicts reassignment).
- **Skill-overqualification ratio:** % of work items where a highly specialized agent (e.g., B12 with `App-CRM, DB-SQL`) handles low-complexity P3/P4 tasks that an L1 with broader basic skills could resolve. This directly addresses the "specialists doing junior work" complaint.
- **Skill demand vs. supply heatmap:** Map ticket volume per required skill against the count of available agents holding that skill. A skill with high demand and few holders (e.g., `Networking-Firewall`) becomes a structural bottleneck.

## 2. Identifying Resource-Related Bottlenecks and Issues

With the metrics above, I pinpoint and **quantify** specific problems:

**(a) Skill-scarcity bottlenecks.** If P2 Network tickets requiring `Networking-Firewall` consistently show long queue times before `Work L2 Start`, and only 2 agents hold that skill, this is a capacity bottleneck. *Quantify:* average queue time for skill-scarce vs. skill-abundant tickets; % of these tickets that breach SLA.

**(b) Reassignment-induced delay.** For INC-1001, the `Reassign` at 11:15 (B12  B15, "needs different skill DB") added rework. *Quantify:* 
- Average wall-clock delay added per reassignment = mean gap between `Reassign` and the next productive `Work Start`.
- Total throughput-time inflation: compare median resolution time for 0-reassignment cases vs. 1+, 2+ reassignment cases.

**(c) Incorrect initial assignment.** Dispatcher/L1 misclassification (assigning to an agent lacking the required skill) is detectable as a reassignment occurring shortly after the first `Work Start`. *Quantify:* % of reassignments traceable to skill mismatch at first assignment.

**(d) Over-/under-loaded agents.** Plot workload distribution; flag agents in the top/bottom deciles. Correlate overloaded agents with elevated queue times on their tickets.

**(e) SLA breach correlation.** This is the keystone analysis. Using the enriched log, run a correlation/feature-importance analysis: *Which factors predict SLA breach?* Candidate predictors: number of reassignments, initial skill mismatch (Y/N), queue time at L2, priority. Output something actionable like: "*68% of P2 breaches involved 1 reassignment*" or "*skill mismatch on first assignment increases breach probability by 3.2×*."

## 3. Root Cause Analysis for Assignment Inefficiencies

Symptoms (reassignments, breaches) are surface-level; root causes likely include:

- **Skill-blind assignment rules.** Round-robin ignores both skill and workload — the structural cause of mismatches and uneven loading.
- **Inaccurate/stale skill profiles.** If agents are reassigned despite a nominal skill match, the documented skills may be inaccurate or proficiency levels (novice vs. expert) aren't captured.
- **Poor upstream categorization.** If `Required Skill` is frequently corrected mid-case (as in INC-1001's DB pivot), initial triage/categorization is weak.
- **No real-time workload visibility.** Dispatchers can't see queue lengths, so they assign blindly.
- **Under-empowered L1.** Low FCR + high escalation rate on resolvable categories suggests training/tooling gaps.

### Techniques to Confirm Root Causes

- **Variant analysis:** Partition cases into "happy path" (0 reassignments, SLA met) vs. "problem variants" (multiple reassignments/escalations). Statistically compare their attributes — priority mix, category, initiating channel, first-assigned agent. Differences isolate the conditions producing poor assignments.

- **Decision mining:** At each decision point (Assign, Escalate, Reassign), train a decision tree using case attributes (priority, category, required skill, agent skills, current workload) as features and the *actual decision/outcome* as the target. This reveals the **implicit decision logic** actually in use and where it diverges from optimal. For instance, the tree might show escalation decisions driven by *which* L1 agent picked up the ticket rather than by genuine complexity — exposing inconsistent triage.

## 4. Data-Driven Resource Assignment Strategies

### Strategy 1: Proficiency-Weighted Skill-Based Routing

- **Issue addressed:** Skill mismatches and reassignments caused by skill-blind round-robin.
- **Leverages mining insight:** Skill-mismatch ratio and reassignment-by-mismatch analysis quantify the problem; mined historical resolution times per (agent × skill) establish proficiency scores.
- **Data required:** Validated skill matrix with proficiency levels (derived empirically from historical processing times and FCR per skill, not just self-reported), `Required Skill` per ticket, agent availability.
- **Implementation:** Route each ticket to the available agent whose skill set covers the required skill at sufficient proficiency, preferring the *lowest qualified* tier to avoid over-qualification (reserving specialists for hard tasks).
- **Expected benefits:** Sharp reduction in skill-mismatch reassignments; better specialist utilization; faster resolution.

### Strategy 2: Workload-Aware (Load-Balancing) Assignment

- **Issue addressed:** Uneven workload, overloaded agents, queue bottlenecks.
- **Leverages mining insight:** Workload distribution and queue-time analysis identify imbalance; SNA reveals handover hubs.
- **Data required:** Real-time queue length per agent, in-progress ticket count, agent availability/shift status.
- **Implementation:** Among skill-qualified candidates (from Strategy 1), assign to the one with the lowest current weighted load. Combine as a scoring function: `score = w1·skill_fit + w2·inverse_load + w3·SLA_urgency`.
- **Expected benefits:** Flattened workload, reduced queue waiting time, fewer SLA breaches from congestion.

### Strategy 3: Predictive Triage & Refined Escalation Criteria

- **Issue addressed:** Poor initial categorization, unnecessary escalations, low L1 FCR.
- **Leverages mining insight:** Decision mining identifies which ticket characteristics historically led to successful L1 resolution vs. genuine escalation need; variant analysis distinguishes truly complex tickets.
- **Data required:** Ticket text/keywords, category, priority, channel, plus historical resolution outcomes for model training.
- **Implementation:** A classification model predicts required skill *and* likely complexity/tier at creation, enabling **direct routing to the right tier** (skipping needless L1 touch for clearly L2/L3 tickets) and **data-driven escalation thresholds** (e.g., "this category + keyword pattern has 85% L1 FCR  keep at L1").
- **Expected benefits:** Higher FCR, fewer escalation hops, reduced specialist time on trivial tasks.

*(Optional Strategy 4 — Dynamic tier reallocation:* using demand-pattern mining to flex agents across skill queues during predictable peaks.)*

## 5. Simulation, Implementation, and Monitoring

### 5.1 Pre-Implementation Simulation

Before disrupting live operations, I'd build a **discrete-event simulation** seeded entirely from mined parameters:
- Process model and routing probabilities from discovery.
- Arrival patterns per category/priority from the log's timestamps.
- Processing-time distributions per (activity × skill × proficiency).
- Resource pool with real skill matrices and shift calendars.

I'd validate the baseline simulation against historical KPIs (resolution time, SLA rate) to confirm fidelity, then run **scenario experiments**: skill-based routing vs. workload-aware vs. combined, varying parameters (e.g., adding firewall-skilled agents). This produces quantified, risk-free forecasts—e.g., "combined Strategy 1+2 reduces P2 mean resolution time by 27% and SLA breaches by 40%"—and supports business-case decisions before any change.

### 5.2 Post-Implementation Monitoring

Stand up a **process mining dashboard** with continuous (ideally near-real-time) ingestion, tracking:

**Resource KPIs:**
- Workload distribution / Gini coefficient of load balance
- Skill-mismatch reassignment rate
- Average reassignments per case
- L1 FCR rate
- Specialist over-qualification ratio
- Queue/waiting time per skill

**Process & outcome views:**
- SLA attainment by priority (P1–P4)
- Conformance rate to the new routing logic
- Handover SNA (to confirm ping-pong loops have decreased)
- Resolution-time trend lines, before/after control charts

I'd configure **alerting** on threshold breaches (e.g., a skill queue exceeding target wait time) and run **periodic conformance checks** to catch drift back to old habits. A **continuous improvement loop** would feed monitoring insights back into recalibrating proficiency scores and the predictive triage model—closing the loop so the assignment logic stays adaptive as ticket patterns and the agent pool evolve.

## Summary

By exploiting the lifecycle-enriched event log, TechSolve can move from blind round-robin to **evidence-based, skill- and workload-aware routing**. The path is: (1) profile resources and discover real assignment patterns, (2) quantify reassignment/skill-mismatch/SLA linkages, (3) trace root causes via variant and decision mining, (4) deploy three complementary routing strategies, and (5) de-risk them through simulation and sustain them through continuous monitoring.
