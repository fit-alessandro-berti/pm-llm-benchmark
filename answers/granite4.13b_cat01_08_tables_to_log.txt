**Notes & Assumptions**

1. **Activity Naming**: Activities are derived from the lifecycle of each order as described in the tables (e.g., “Order Created” marks the start of a process instance; “Items Picked” reflects picking events for each line item).  
2. **Timestamp Mapping**: When multiple timestamps exist per activity (e.g., several items have `item_picking_date`), separate rows are created to preserve chronological granularity while still reflecting the overall order progression.  
3. **Resource Enrichment**: Warehouse staff (`picker_id`) and billing/paying personnel are included as “Resource/Actor” when available; otherwise, fields remain empty (denoted by “–”).  
4. **Assumptions**:
   - Each `delivery_dispatch_date` corresponds to the dispatch of the order’s first shipment.
   - The final delivery date (`delivery_arrival_date`) represents the last arrival for each order (or single shipment if only one entry exists).
   - Invoices and payments are linked by their respective IDs, ensuring that payment follows invoice issuance.  
5. **Data Completeness**: All timestamps present in the source tables have been used; any missing or ambiguous data points were left unpopulated to avoid speculative entries.

This event log is ready for import into a process mining tool, providing a clear chronological view of order lifecycle activities across all related systems.
