# Optimizing E-Commerce Fulfillment: A Constraint-Aware Process Mining Strategy

## 1. Identifying Instance-Spanning Constraints and Their Impact

Traditional process mining treats cases independently. For instance-spanning constraints (ISC), we must enrich the event log with **system-state features** computed at each event timestamp by scanning across all concurrently active cases.

### 1.1 Data Preparation
For every event `e` of case `c` at time `t`, compute:
- `active_cold_packing(t)` – count of cases currently in "Packing" with `RequiresColdPacking=TRUE`.
- `active_hazmat_packing_qc(t)` – count of cases currently in Packing or QC with `HazardousMaterial=TRUE`.
- `queue_length_by_activity_type(t)` – cases waiting to enter Packing/Cold-Packing/QC.
- `batch_membership(c)` – the batch ID, batch size, and first/last arrival-to-batch timestamps.
- `express_overlap(c)` – whether an express order competed for the same resource during this case's waiting period.

### 1.2 Techniques and Metrics per Constraint

**(a) Shared Cold-Packing Stations**
- *Technique:* Resource utilization analysis + queue mining. Plot the timeline of the 5 cold-packing stations. Whenever all 5 are simultaneously busy, any arriving perishable order experiences contention.
- *Metrics:*
  - **Cold-Pack Contention Waiting Time (CCWT)** = Packing start – Packing ready time, conditional on 5 stations being busy.
  - **Utilization rate** per station; **P95 queue length** for cold-pack.
  - **% of cold orders that waited > X min** due to contention.

**(b) Shipping Batches**
- *Technique:* Reconstruct batches from the shared resource identifier ("Batch B1" in the log) or cluster orders that share an identical "Shipping Label Gen." timestamp + Destination Region.
- *Metrics:*
  - **Batch Wait Time (BWT)** = Shipping Label timestamp – Quality Check complete timestamp, per case.
  - **Batch formation time** = last arrival – first arrival into batch.
  - **Batch fill efficiency** = actual batch size / target batch size.

**(c) Priority (Express) Handling**
- *Technique:* Detect preemption events—cases where a Standard order's activity has a START event, an inter-event gap, and resumes with the same resource just after an Express case seized that resource. Compare standard-order waiting times with and without a concurrent express arrival.
- *Metrics:*
  - **Preemption count** per resource/day.
  - **Standard Order Delay due to Priority (SODP)** = conditional waiting time premium when an Express order arrives concurrently.
  - **Express Expedite Gain** = reduction in express lead time versus standard.

**(d) Hazardous Material Regulatory Limit**
- *Technique:* Build a running counter of hazmat orders in PackingQC. Flag time windows where the counter = 10 (saturation). Any hazmat order whose Packing start was delayed while counter=10 is a constraint-caused delay.
- *Metrics:*
  - **Hazmat Saturation Time** = total minutes/day where counter = 10.
  - **Hazmat Queue Delay** = waiting time attributable to saturation.
  - **Throughput reduction** = hazmat orders/hour during saturation vs. non-saturation periods.

### 1.3 Differentiating Within-Instance vs. Between-Instance Waiting

Decompose each inter-activity time `t` into components:
- **Service time** = Complete  Start of activity (within-instance).
- **Handover/queue time** = Start of next – Complete of previous.
  - Classify handover time as **between-instance** if, at that moment, the required resource was occupied (cold-pack saturated, hazmat counter=10, resource seized by express order, or batch not yet full).
  - Otherwise, classify as **within-instance** (e.g., travel time, rework, staff break) — typically identified by no competing demand in the system-state features.

A decision-tree classifier or rule engine on system-state features at the moment waiting began produces a **waiting-time attribution breakdown** per case, which can then be aggregated.

## 2. Analyzing Constraint Interactions

The constraints do not act independently; they compound:

- **Express × Cold-Packing:** An express order requiring cold-packing can preempt a standard perishable order. Because there are only 5 cold stations, this creates a queue cascade where preempted standards re-enter the queue, amplifying CCWT for all subsequent perishable orders. Analysis: compute correlation between express arrivals and the P95 cold-pack queue depth.

- **Batching × Hazmat Limit:** If several hazmat orders destined for the same region converge on Packing/QC, the regulatory cap of 10 throttles how quickly the batch can be assembled. Conversely, releasing the batch forces hazmat orders to linger in QC-adjacent states, potentially keeping the counter near the cap. Analysis: cross-tabulate hazmat saturation windows with batches containing hazmat orders and measure batch formation time uplift.

- **Batching × Express:** Express orders need instant label generation, but batching delays shipping. If express orders were erroneously included in batches, BWT becomes a critical lead-time component; if they bypass batches, the batch loses optimization potential.

- **Priority × Hazmat:** An express hazmat order arriving when the counter is 9 will enter Packing, pushing it to 10 and blocking subsequent standard hazmat orders — priority handling effectively consumes regulatory quota.

**Why interactions matter:** Optimizing one constraint locally can worsen another. E.g., aggressively expediting express orders without considering hazmat quota can starve standard hazmat throughput. A global view — ideally via a **discrete-event simulation model or an LP/queueing network** — is required to reveal dominant bottlenecks and optimize end-to-end KPIs.

## 3. Constraint-Aware Optimization Strategies

### Strategy A: Predictive Dynamic Allocation for Cold-Packing with Priority-Aware Queuing
- **Targets:** Shared Cold-Packing + Express Priority interaction.
- **Change:** Implement a **reservation-based scheduler** for the 5 cold stations. Using historical arrival rates by hour/day and a short-horizon forecast (e.g., gradient-boosted model on order receipt patterns), reserve a rolling capacity buffer (e.g., 1 station) for predicted express perishables during peak windows. Use a **multi-class priority queue** with SLA-aware aging so standard orders do not starve.
- **Data leverage:** Historical CCWT distributions, arrival rate by hour, express share forecasts.
- **Outcome:** Lower average CCWT, reduced preemption events (because express orders find a free station more often), improved express SLA compliance without substantial standard penalty.

### Strategy B: Dynamic Batching with Time & Fill Thresholds
- **Targets:** Shipping Batch delays; secondarily Hazmat interaction.
- **Change:** Replace fixed batch rules with a **dual-trigger policy**: close a batch when either (i) size  target *or* (ii) oldest member's BWT  region-specific max (derived from delivery SLA  remaining transit time). Add a **hazmat-aware split rule**: if a regional batch contains >N hazmat orders, split or stagger labeling to avoid synchronizing hazmat Packing/QC surges.
- **Data leverage:** Historical inter-arrival times by region, empirical distribution of batch formation time, delivery route cost function.
- **Outcome:** Reduced P95 BWT, better batch-fill efficiency, and smoother hazmat counter trajectory.

### Strategy C: Hazmat-Aware Admission Control (Token Bucket)
- **Targets:** Hazmat regulatory limit; Priority/Hazmat interaction.
- **Change:** Implement a **token-based admission controller** at the entry of Packing. 10 hazmat tokens exist; a hazmat order cannot start Packing without a token (released at QC completion). Couple with **shadow scheduling**: for arriving hazmat orders, precompute expected token availability and release Item Picking in a staggered cadence — avoiding starving the system while maximizing legal concurrency. Reserve 1–2 tokens for express hazmat to uphold SLA without exceeding 10.
- **Data leverage:** Observed hazmat inter-arrival and service-time distributions; simulation-calibrated token-reserve sizes.
- **Outcome:** Full compliance maintained by design, increased effective hazmat throughput (fewer long saturation stalls), reduced variance in lead time for hazmat orders.

### (Optional) Strategy D: Decouple Quality Check from Packing for Hazmat
- **Targets:** Hazmat limit (counter is sum across Packing+QC).
- **Change:** Introduce a short **hazmat "holding area"** separating Packing completion from QC start with ventilated storage, so that an order can complete Packing and release its physical packing station while a dedicated hazmat QC slot opens. Keeps regulatory counter meaningful while decoupling physical queues.

## 4. Simulation and Validation

### 4.1 Simulation Approach
Build a **Discrete-Event Simulation (DES)** model calibrated directly from the process-mined event log:
- Activity duration distributions fit from START/COMPLETE pairs.
- Arrival processes fit per order type (Standard/Express, Cold/Hazmat) — potentially non-stationary Poisson with time-of-day intensities.
- Routing probabilities from the discovered process model (e.g., inductive miner).
- Resources modeled as **finite capacity pools**: Pickers, Standard Packing, Cold-Packing (cap=5), QC staff.

### 4.2 Capturing Instance-Spanning Behavior
Crucially, the simulator must implement **global state variables** that mirror the constraints:
- **Global hazmat counter** across Packing+QC with admission blocking at 10.
- **Priority queues with preemption logic** for express orders (and rules for whether to preempt mid-activity or only at start).
- **Batch assembly queues** per region with dynamic close triggers.
- **Cold-pack station semaphore** with reservation logic.

Validate the baseline model by checking that simulated KPIs (mean lead time, CCWT, BWT, hazmat saturation time) match the event log within tolerance (e.g., <10% on P50/P95). Use **conformance checking** between the simulated log and the real log as a validation step.

### 4.3 Experimental Design
- Run A/B scenarios: baseline vs. each strategy vs. combined strategies.
- Use **common random numbers** across scenarios for variance reduction.
- Stress-test under peak-season intensities (e.g., Black Friday arrival multipliers).
- Primary KPIs: end-to-end lead time (overall, per class), throughput, SLA compliance, resource utilization, regulatory violations (must be zero).
- Report confidence intervals (e.g., 30+ replications).

## 5. Post-Implementation Monitoring

### 5.1 Live Dashboards
A **process mining cockpit** fed from the live event stream with real-time state computations.

**Dashboard 1 — Resource Contention (Cold-Packing):**
- Live utilization of each of the 5 stations.
- Real-time queue length and CCWT distribution.
- Express preemption counter.
- Alert: CCWT P95 > threshold.

**Dashboard 2 — Batching Performance:**
- BWT distribution per region.
- Average batch size vs. target.
- Oldest-order-in-batch gauge (ensures SLA).
- Trend: batches closed by fill-trigger vs. time-trigger.

**Dashboard 3 — Regulatory Compliance & Hazmat Flow:**
- Live hazmat counter (Packing + QC), with red line at 10.
- Minutes per day at saturation (target: declining trend).
- Hazmat throughput per hour.
- Audit log: any violation = immediate alert.

**Dashboard 4 — Priority & SLA:**
- Express vs. Standard lead time percentiles.
- SODP (delay imposed on standards) trend — to detect over-prioritization.

### 5.2 Tracking ISC Management Effectiveness
Compare pre/post deployment on ISC-specific KPIs:
| Constraint | KPI | Target |
|---|---|---|
| Cold-pack | P95 CCWT |  30% |
| Batching | P95 BWT |  20%, fill ratio  |
| Priority | SODP (standard delay) |  pre-level |
| Hazmat | Saturation minutes/day |  while throughput ; violations = 0 |

### 5.3 Continuous Improvement Loop
- **Weekly conformance checks** between the re-designed model and actual log to detect drift.
- **Anomaly detection** on ISC state variables (e.g., unexplained saturation spikes).
- **Periodic re-simulation** with updated distributions to re-tune reservation/token parameters — the constraints are dynamic, so policies must be, too.

---

### Summary
The core insight is that **instance-spanning constraints require reasoning about the system state across cases, not just within a case**. By enriching the event log with global state features, quantifying between-instance waiting, modeling the interactions, validating via constraint-aware simulation, and monitoring via ISC-specific dashboards, the fulfillment center can measurably reduce lead times and increase throughput while guaranteeing regulatory and SLA compliance.
