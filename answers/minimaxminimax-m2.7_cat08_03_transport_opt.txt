## 1. Process Discovery and Conformance Checking

### 1.1 Data Preprocessing and Integration

The heterogeneous nature of Speedy Parcels' data sourcesŚGPS trackers, handheld scanners, dispatch systems, and maintenance logsŚrequires a structured preprocessing pipeline to create a unified event log suitable for process mining. My approach follows a **Multi-Perspective Event Log Construction** methodology that preserves the semantic richness of each source while establishing common identifiers for integration.

**Phase 1: Schema Standardization and Case Identification**

Each data source uses different primary keys and temporal granularities. I would establish a unified event schema with the following structure:

| Attribute | Source(s) | Description | Transformation Logic |
|-----------|-----------|-------------|---------------------|
| Case ID | All | Vehicle-Day identifier (e.g., V12-20241205) | Composite key from Vehicle ID + date |
| Timestamp | All | Exact event time | Normalized to UTC; GPS timestamps need clock-drift correction |
| Activity | Scanner, Dispatch, Vehicle | Semantic activity label | Mapped to canonical vocabulary (see below) |
| Resource | All | Driver ID, Vehicle ID | Direct from source |
| Location | GPS, Scanner | Lat/Lon coordinates | Retained as-is; geocoding for reverse-lookup |
| Speed | GPS | km/h | Direct from GPS; outliers flagged |
| Package ID | Scanner | Delivery identifier | Linked via stop sequence |
| Planned Route | Dispatch | Expected sequence | Stored as reference for conformance |

**Canonical Activity Vocabulary Development:**

From the sample data, I observe the following activity types that must be normalized:

- **Shift Activities:** Start Shift  Route Assigned  End Shift
- **Movement Activities:** Depart Depot, Arrive Depot, Depart Customer, Arrive Customer, Low Speed Detected (proxy for congestion), Unscheduled Stop
- **Delivery Activities:** Delivery Success, Delivery Failed
- **Maintenance Activities:** Maintenance Start, Maintenance End (from maintenance logs)
- **Idle Activities:** Idle periods between stops

**Phase 2: Data Fusion Strategies**

The integration presents several challenges that require careful handling:

| Challenge | Description | Resolution Approach |
|-----------|-------------|-------------------|
| **Temporal Alignment** | Different systems sample at different frequencies (GPS: ~30 sec, Scanner: event-based, Dispatch: static) | Interpolate GPS positions between scanner events; use scanner timestamps as ground truth for stop times |
| **Location Matching** | GPS coordinates may drift or be inaccurate near depots/addresses | Apply geofencing with 50m buffer for depot identification; snap to nearest known address for customer stops |
| **Stop Detection** | Distinguishing planned stops from unplanned (e.g., traffic vs. break) | Combine speed < 5 km/h threshold with scanner events; classify based on duration and proximity to scheduled customers |
| **Missing Links** | Some packages may not have GPS trails | Use temporal and spatial proximity to link; flag unlinkable records |
| **Unplanned Events** | Maintenance stops, traffic jams not in dispatch plan | Insert as deviation activities with special markers |

**Phase 3: Event Enrichment**

Additional derived attributes should be computed during preprocessing:

- **Stop Sequence Number:** Ordinal position within the case (derived from scanner events)
- **Time between Stops:** Calculated as elapsed time from previous Depart Customer to current Arrive Customer
- **Service Duration:** Elapsed time from Arrive Customer to Depart Customer
- **Cumulative Travel Distance:** Estimated from GPS track segments
- **Delay from Time Window:** For time-windowed deliveries, compute deviation from promised window
- **Planned vs. Actual Route Position:** Mapping discovered route steps to planned route sequence

### 1.3 Conformance Checking: Comparing Actual vs. Planned Processes

Conformance checking quantifies the alignment between discovered process behavior and the normative dispatch model. I recommend using **Token Replay** and **Alignment-Based Conformance** (Van der Aalst et al., 2012) techniques.

**Planned Process Model Construction:**

From the dispatch system, I would extract the expected process as a **block-structured workflow net**:

1. Start with departure from depot
2. For each assigned package in sequence:
   - Travel to customer location
   - Perform delivery attempt
   - Return to depot or continue
3. End shift upon return

**Deviation Types to Detect:**

| Deviation Category | Specific Patterns | Significance |
|-------------------|-------------------|--------------|
| **Sequence Deviations** | Skipped stops, out-of-order deliveries, revisiting locations | Route inefficiency, missed time windows |
| **Unplanned Activities** | Unscheduled stops, breaks, detours not in dispatch plan | Maintenance needs, driver behavior, traffic avoidance |
| **Timing Deviations** | Travel times significantly exceeding estimates, excessive dwell times | Traffic hotspots, parking difficulties, customer issues |
| **Missing Activities** | Stops not attempted, incomplete routes | Vehicle issues, driver fatigue, dispatch errors |
| **Extra Activities** | Additional stops beyond planned, repeated visits | Failed deliveries requiring re-attempt |
| **Resource Deviations** | Vehicle substitution, driver changes mid-route | Operational exceptions |

**Quantifying Conformance:**

For each case, I would compute:

- **Fitness:** Percentage of model-conforming behavior (0-1 scale)
- **Cost-based Deviation:** Weighted sum of deviations (e.g., per unplanned stop: -5 points; per missed time window: -10 points)
- **Trace Alignment:** Optimal alignment between planned and actual sequences with minimal cost

## 4. Data-Driven Optimization Strategies

Based on the process mining analysis framework outlined above, I propose the following **three distinct, high-impact optimization strategies** tailored to Speedy Parcels' context.

### Strategy 1: Dynamic Route Adjustment with Real-Time Traffic Integration and Predictive Bottleneck Avoidance

**Specific Inefficiency Targeted:**
Systematic traffic-related delays during rush hours, leading to missed time windows and overtime. Currently, routes are static and cannot adapt to traffic conditions.

**Root Cause Addressed:**
Static route planning using outdated travel time estimates; inability to anticipate and avoid traffic bottlenecks.

**Supporting Process Mining Insights:**

| Insight | Evidence | Implication |
|---------|----------|-------------|
| Traffic hotspots by time | Low-speed events cluster at 07:30-09:00 and 16:30-18:30 | Need time-dependent routing |
| Route segments with high delay variance | Some routes have 3x travel time variability | Identify unreliable segments |
| Sequence dependency | Late start compounds delays throughout day | Early intervention prevents cascade |

**Implementation Framework:**

1. **Real-Time Traffic Integration:**
   - Integrate GPS data stream with traffic API (TomTom, HERE, Google)
   - When actual travel time on segment exceeds estimate by >20%, trigger re-optimization

2. **Predictive Bottleneck Scoring:**
   - From historical analysis, score each route segment by likelihood of delay
   - Use **Markov chain models** of traffic progression to predict future congestion

3. **Dynamic Re-sequencing Algorithm:**
   - When delays exceed threshold, invoke VRP (Vehicle Routing Problem) solver
   - Optimize remaining stops considering: time windows, delay probability, driver hours remaining

4. **Driver Communication:**
   - Push updated route to driver's handheld device
   - Provide ETA updates to customers when delays occur

**Expected KPI Impacts:**

**Specific Inefficiency Targeted:**
Unscheduled vehicle breakdowns during shifts causing incomplete deliveries, emergency repairs, and high maintenance costs. The current preventive maintenance schedule is based on fixed intervals rather than actual vehicle stress.

**Root Cause Addressed:**
Maintenance gaps between scheduled service leading to in-field failures; lack of correlation between vehicle usage patterns and maintenance needs.

**Supporting Process Mining Insights:**

| Insight | Evidence | Implication |
|---------|----------|-------------|
| Unscheduled stop patterns | Engine warning events often precede returns | Warning signs detectable before breakdown |
| Vehicle-specific failure rates | Certain vehicles show clustering of issues | Individual vehicle risk scoring needed |
| Usage-failure correlation | High-mileage vehicles in certain zones break down more | Zone-specific maintenance schedules |
| Temporal clustering | Unscheduled stops cluster mid-week | Day-of-week maintenance timing |

**Implementation Framework:**

1. **Vehicle Health Scoring:**
   - From GPS data, calculate: cumulative idling time, harsh braking events, extended high-RPM periods
   - From scanner data: number of stops completed, total distance

2. **Early Warning Detection:**
   - Monitor for precursor events: engine warnings, extended crank times, coolant temperature spikes

3. **Dynamic Maintenance Scheduling:**
   - Instead of fixed 10,000 km intervals, trigger maintenance when:

     - Specific warning patterns detected

4. **Spare Vehicle Pool Optimization:**

5. **Route Position Optimization:**
