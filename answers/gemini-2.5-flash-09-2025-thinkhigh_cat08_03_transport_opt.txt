This comprehensive approach utilizes process mining techniques to transform Speedy Parcels' raw event data into actionable insights, focusing on improving delivery punctuality and reducing operational costs.

---

## 1. Process Discovery and Conformance Checking

The foundation of this analysis is creating a unified, trustworthy event log from the disparate data sources.

### Data Preprocessing and Integration

The primary challenge is the **heterogeneity and differing granularity** of the data: high-frequency, continuous GPS data versus discrete, event-based scanner logs and static dispatch plans.

**Integration Steps:**

1.  **Define the Case ID:** The primary Case ID must be the **Vehicle-Day** (e.g., `V12-20241205`). All events related to that day’s operational flow must link here. Package IDs can be secondary identifiers for sub-processes (delivery sequence).
2.  **Activity Mapping:** Standardize the event types into defined activities:
    * **Scanner/Dispatch:** Map directly to activities ('Arrive Customer', 'Delivery Success', 'Depart Customer', 'Start Shift').
    * **GPS Data:** Requires transformation. Segments of GPS data must be grouped and labeled:
        * `Speed > 5 km/h`: 'Traveling'
        * `Speed = 0 km/h` near a customer location (preceded by 'Traveling'): 'Searching/Service Time'
        * `Speed = 0 km/h` in an unplanned location for > 10 minutes: 'Unscheduled Stop' or 'Idle Time' (if ignition is on).
    * **Maintenance Logs:** Events must be slotted into the Case ID flow (e.g., `2024-12-05 11:05:00` becomes an activity: 'Unscheduled Maintenance Interruption').
3.  **Time Stamp Normalization:** Ensure all timestamps are in a single format and time zone for accurate sequencing.
4.  **Enrichment:** Use location coordinates to enrich activities (e.g., link GPS data proving the vehicle was parked to the 'Arrive Customer' scanner event, calculating travel distance/duration between confirmed stops).

**Challenges Encountered:**

*   **Granularity Mismatch:** Merging continuous GPS movement with discrete scanner events requires defining thresholds (e.g., when does 'Traveling' end and 'Service Time' begin?).
*   **Missing or Delayed Events:** Drivers may forget to scan. Process mining must infer activities (e.g., if a vehicle is parked at a customer location for 15 minutes, but no scan is recorded, the duration is still logged as 'Dwell Time at Stop').
*   **Handling Rework/Rerouting:** Failed deliveries result in complex loops and re-entry into the process. This must be modeled clearly as a rework variant.

### Process Discovery

Using algorithms like the Inductive Miner on the unified event log (Case ID, Activity, Timestamp), we can generate a **Descriptive Process Model** (Process Map or Petri Net) that visualizes the *actual* flow of a typical delivery day.

The discovered model will show:

1.  The actual sequence of activities from 'Start Shift' to 'End Shift'.
2.  The frequency of paths (e.g., 80% of deliveries follow the 'Arrive -> Success -> Depart' sequence, but 20% involve 'Arrive -> Failed -> Travel Back to Depot').
3.  The inclusion of unplanned or non-standard activities ('Unscheduled Stop', 'Engine Warning Light').

### Conformance Checking

The goal is to compare the discovered process models (what actually happened) against the ideal reference model derived from the **Dispatch System** (the planned route and schedule).

**Reference Model:** The dispatch system provides the intended sequence of stops and the expected time windows for each activity.

**Deviation Types to Look For:**

| Deviation Type        | Log Data Used           | Impact on Operations                                   |
| :-------------------- | :---------------------- | :----------------------------------------------------- |
| **Sequence Deviation** | Scanner/GPS vs. Planned | Driver visited Stops 5, 8, 6, 7 instead of 5, 6, 7, 8. |
| **Unplanned Activities** | GPS/Maintenance Logs    | Insertion of 'Unscheduled Stop' or 'Deviation for Fuel'. |
| **Timing Deviation**  | All Sources             | Planned travel time: 10 min. Actual travel time: 25 min. |
| **Constraint Violation** | Dispatch vs. Scanner    | Delivery arrived at 11:15, but customer time window was 8:00–10:00. (Root cause of failure). |

Conformance checking provides a compliance score for each route, quickly highlighting routes or drivers that frequently diverge from the plan, leading directly into root cause analysis.

## 2. Performance Analysis and Bottleneck Identification

Performance analysis focuses on quantifying delays, waste, and costs across the identified process flow.

### Key Performance Indicators (KPIs)

| KPI Category      | KPI Definition                                                 | Calculation Method from Event Log                                   |
| :---------------- | :------------------------------------------------------------- | :------------------------------------------------------------------ |
| **Punctuality**   | **On-Time Delivery (OTD) Rate**                                | (Deliveries completed within planned time window) / Total Deliveries |
|                   | **Delivery Success Rate (First Attempt)**                      | (Successful Deliveries) / (Total Delivery Attempts)                 |
| **Efficiency**    | **Average Service Time per Stop**                              | Average duration between 'Arrive Customer' and 'Depart Customer'.   |
|                   | **Travel Time Utilization Ratio**                              | Total Travel Time / (Total Travel Time + Total Idle Time)           |
| **Cost/Reliability** | **Unscheduled Stop Frequency**                                 | Count of 'Unscheduled Stop' or 'Engine Warning Light' events per 100 cases (vehicle-days). |
|                   | **Traffic Delay Impact (Time)**                                | Aggregate duration of GPS events tagged as 'Low Speed Detected' correlated with travel segments. |

### Bottleneck Identification Techniques

Process mining techniques allow us to transition from merely observing that a route is slow to identifying *where* the time is spent and *why*.

1.  **Throughput Time Analysis Overlay:**
    * Overlay the mean or median duration of activities onto the discovered process map.
    * **Bottleneck Example:** If the arc 'Depart Customer N' to 'Arrive Customer N+1' shows an average throughput time of 30 minutes (while planned was 15 minutes), the travel segment is the bottleneck. If the arc 'Arrive Customer' to 'Delivery Success' takes 8 minutes, the service activity is the bottleneck.

2.  **Resource-Based Analysis (Drivers/Vehicles):**
    * **Filtering:** Filter the event log by Driver ID or Vehicle ID. Compare throughput times.
    * **Insight:** If Driver D105 has an average service time 50% higher than the fleet average, the bottleneck is likely linked to driver behavior, training, or localized route knowledge (e.g., finding parking).

3.  **Geospatial and Temporal Clustering:**
    * Use GPS coordinates and timestamps correlated with high-duration travel segments.
    * **Insight:** Clusters of 'Low Speed Detected' events occurring consistently between 8:00 AM and 9:00 AM near a specific coordinate set pinpoint a predictable traffic hotspot, which is a major bottleneck impacting early-morning schedules.

4.  **Idle and Rework Analysis:**
    * High frequency of the 'Delivery Failed (Attempted)' activity indicates a bottleneck in the reliability/prediction phase, leading to expensive rework (the cost of the return trip and subsequent attempt).
    * High `Idle Time` recorded by GPS indicates wasted operational capacity, potentially due to poor scheduling, long depot loading times, or regulatory breaks taken at suboptimal times.

## 3. Root Cause Analysis for Inefficiencies

Identifying the true root causes requires correlating the performance bottlenecks with contextual data (attributes of the case).

### Validating Root Causes using Process Mining

| Potential Root Cause                               | Process Mining Validation Method                               |
| :------------------------------------------------- | :------------------------------------------------------------- |
| **Suboptimal Route Planning / Inaccurate ETAs**    | **Conformance & Performance Analysis:** High conformance failure rate (sequence/timing) coupled with high travel time deviation. Planned vs. Actual comparison validates if the static model is fundamentally flawed. |
| **Impact of Traffic Congestion**                   | **Correlation Analysis:** Correlate 'Low Speed Detected' event duration with specific route segments, time of day, and external traffic indices. Analyze if the longest travel arcs correspond precisely to known congestion points. |
| **High Variability in Service Time**               | **Variant Analysis (Dwell Time):** Calculate the Standard Deviation of 'Arrive Customer' to 'Depart Customer' time. Filter variants by customer type (e.g., business vs. residential) or package weight (from dispatch data). High variability suggests non-standard handling requirements (e.g., complex entry, signature requirements). |
| **Frequency of Vehicle Breakdowns**                | **Maintenance Log Linkage:** Use sequence analysis to see if specific GPS patterns (e.g., aggressive acceleration/braking frequency or high engine load during specific route segments) consistently *precede* unscheduled 'Maintenance Interruption' events, indicating vehicle misuse or underlying mechanical stress. |
| **Driver Behavior and Skill Differences**          | **Comparative Variant Analysis (Resource):** Compare the process map of top 10% efficient drivers (low idle, low service time variance, high conformance) against bottom 10%. Highlight differences in activity sequencing or use of unplanned stops. (e.g., inefficient drivers spend more time between scanning Pkg X and scanning Pkg Y at the same location). |
| **Issues Related to Failed Delivery Attempts**     | **Root Cause Mapping on Rework Loop:** Analyze the attributes of all 'Delivery Failed' cases. Correlate failures with constraint violations (e.g., 70% of failed deliveries were attempted outside the customer's requested time window). |

### Strategy 3: Usage-Informed Predictive Maintenance Scheduling

| Aspect | Detail |
| :--- | :--- |
| **Inefficiency Targeted** | High cost of unscheduled repairs and vehicle downtime during critical delivery hours. |
| **Root Cause Addressed** | Premature vehicle wear due to aggressive driver behavior (rapid acceleration/hard braking) or specific route conditions (rough roads) that traditional time/mileage-based maintenance misses. |
| **Process Mining Support** | Correlate specific GPS data metrics (e.g., speed variance, deceleration forces) logged during the ‘Traveling’ activity with subsequent ‘Maintenance’ activities. Driver variant analysis identifies which drivers contribute to higher vehicle wear. |
| **Proposal** | Implement a **Condition-Based Maintenance trigger**. If a vehicle’s operational profile (derived from process-mined GPS attributes) deviates from a healthy baseline for $N$ consecutive shifts (e.g., idle time increases by 20%, or average speed during ‘Travel’ drops by 10%), automatically flag the vehicle for priority inspection *before* a failure occurs. |
| **Expected Impacts** | Reduced Unscheduled Stop Frequency; Lower overall maintenance costs; Increased Vehicle Utilization Rate. |

## 5. Considering Operational Constraints and Monitoring

### Accounting for Operational Constraints

Any optimization must respect legal and contractual requirements:

*   **Driver Working Hours:** The process discovery provides the *actual* total shift duration and identifies non-productive time. Optimized routing must remain within maximum driving hours and mandate sufficient break duration (which can be enforced by adding mandatory 'Break' activities into the conformance model).
*   **Vehicle Capacity:** This is a key constraint in the dispatch system. The proposed routes must always respect the initial volume/weight limits. Process mining helps here by analyzing whether the time spent at the depot is inflated due to suboptimal loading procedures (e.g., long `Scan at Depot` to `Depart Depot` time correlates with high package count, indicating a loading bottleneck).
*   **Customer Time Windows:** This is the primary constraint for OTD. The Adaptive Route Calibration (Strategy 1) specifically targets meeting these windows by using accurate, process-mined travel times, ensuring the planned route remains feasible.

### Continuous Monitoring and Process Mining Dashboards

After implementation, a real-time process mining dashboard is crucial for monitoring effectiveness and detecting process drift.

**Key Metrics and Process Views to Track:**

1.  **Conformance Dashboard (Drift Detection):**
    * **Metric:** Route Adherence Score (Percentage of cases following the normative planned sequence).
    * **View:** Compare the currently dominant process map variant (the 'Actual') against the normative 'Planned' process. If new, inefficient variants (e.g., unscheduled refuel stops, excessive idling) begin to gain frequency, an alert is triggered, signaling process drift or non-compliance.

2.  **Performance and Efficiency Monitor:**
    * **Metric:** OTD Rate (Real-time tracking), Average Time per Stop Variance, Travel Time Deviation vs. Target.
    * **View:** Driver/Route Scorecards ranking performance metrics. This allows management to immediately identify if the new optimized routes are causing stress on specific drivers or if traffic changes have rendered the travel time assumptions inaccurate again.

3.  **Cost and Reliability View:**
    * **Metric:** Frequency of Unscheduled Stops, Rate of Failed Deliveries (first attempt).
    * **View:** Maintenance event overlay. Track the success of the predictive maintenance strategy by comparing the frequency of *scheduled* preventive actions triggered by the usage model against the diminishing frequency of *unscheduled* breakdowns.

Continuous monitoring ensures that the initial root causes remain resolved and provides the data necessary to feed continuous improvement cycles (re-calibrating the route models every quarter, for instance).
