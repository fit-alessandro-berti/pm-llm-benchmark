### Explanation of Constraints

**1. Always Before & Always After:**
These constraints define a strict linear sequence where each activity depends on its predecessor. The process follows:
`RO  CA  RI  PI  QC  PO  GSL  D  IC  RP`

The `always_before` set ensures that if an activity occurs, its required predecessor must have already occurred. For example, `('CA', 'RO')` means that when Check Availability occurs, Receive Order must have already happened.

The `always_after` set ensures that if an activity occurs, its required successor must occur at some later point. For example, `('RO', 'CA')` means that if Receive Order occurs, then Check Availability must eventually follow.

**2. Directly Follows:**
This is the strictest ordering constraint, specifying immediate succession. The process is designed as a single linear workflow with no branching or parallelism, making it suitable for a `directly_follows` specification.

**3. Activity Occurrences:**
Each activity is expected to occur exactly once in a typical successful order fulfillment. The `{1}` bound reflects that these are discrete, single-instance steps in the process.

**4. Equivalence & Never Together:**
- `equivalence` is empty because no two activities are guaranteed to always occur the same number of times (though in this linear case they all occur once, equivalence isn't needed).
- `never_together` is empty because all activities are part of the same successful flow and can coexist within a case.

### Process Context

This model represents an idealized, fully successful order fulfillment path. In practice, business processes often include exception handling (e.g., items out of stock, payment failures), which would require additional constraints or alternative paths. The current skeleton assumes every order is successfully processed from start to finish.

The constraints collectively ensure that:
- The process always starts with `RO` and ends with `RP`
- Activities occur in the prescribed order
- No steps are skipped
- Each step happens exactly once

This creates a deterministic, auditable workflow that can be used for compliance checking, performance monitoring, or conformance analysis against actual event logs.
