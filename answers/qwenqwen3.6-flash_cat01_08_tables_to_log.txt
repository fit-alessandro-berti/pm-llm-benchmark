###  Assumptions & Implementation Notes

1. **Case Identification**: `order_id` is used strictly as the `case_id`. When an order has multiple deliveries or invoices, they are treated as separate events within the same case, which is standard practice in process mining to capture parallel or split flows.
2. **Activity Derivation**: 
   - `Order Lines`  `Item Picked` (one event per line item to preserve granular picking data)
   - `Delivery`  `Order Dispatched` & `Order Delivered` (two events per delivery record)
   - `Invoice`  `Invoice Issued`
   - `Payments`  `Payment Received`
3. **Resources**: Mapped using the `Resources` table. For `Payment Received` events, no explicit actor is linked in the schema, so `Payment System` is used as a logical placeholder. In practice, this could be mapped to a `Finance Dept` or specific accounts payable clerk if available.
4. **Missing Data**: Order `1004` contains only `Order Created` and `Item Picked` events in the provided tables. No delivery, invoice, or payment records exist for it, so only the available events are included.
5. **Chronological Order**: The log is sorted globally by `timestamp` to satisfy process mining tool requirements. If your tool requires case-level sorting, you can easily reorder by `case_id` then `timestamp` without breaking traceability.
6. **Attributes Formatting**: Attributes are stored as `key=value` pairs separated by semicolons for compactness. Modern process mining platforms (e.g., Celonis, UiPath Process Mining, Disco) can parse these or you can split them into separate columns during ingestion.
