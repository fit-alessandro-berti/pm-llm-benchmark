## 1. Queue Identification and Characterization

### 1.1 Calculating Waiting Times from Event Logs
The event log contains **paired START/COMPLETE timestamps** for each activity, enabling precise calculation of two distinct time components per activity instance:

*   **Service Time (Processing Time):** `Timestamp(COMPLETE) - Timestamp(START)`  Actual time spent performing the activity (e.g., nurse triage, doctor consultation).
*   **Waiting Time (Queue Time):** `Timestamp(START_current) - Timestamp(COMPLETE_previous)`  Idle time between the end of the previous activity and the start of the next.

**Operational Definition of "Waiting Time" in this Context:**
*   **Inter-Activity Wait:** Time a patient spends in a physical waiting room or corridor between *Completion* of Activity A (e.g., Registration) and *Start* of Activity B (e.g., Nurse Assessment).
*   **Pre-First Activity Wait:** `Timestamp(START_First_Activity) - Scheduled_Appointment_Time` (if appointment time is available) or `Timestamp(START_First_Activity) - Timestamp(Arrival_Checkin)` (if arrival is logged). *Assumption: We derive this from Registration START vs. Appointment Time.*
*   **Post-Last Activity Wait:** Usually negligible (Check-out COMPLETE = Departure), but `Timestamp(Departure) - Timestamp(COMPLETE_Checkout)` if tracked.

**Algorithm:**
1.  Sort events by `Case ID` and `Timestamp`.
2.  For each case, pair START/COMPLETE events per activity to validate data integrity.
3.  Calculate `Service_Time` per activity instance.
4.  Calculate `Wait_Time` between `COMPLETE_(i)` and `START_(i+1)` for the sequential activity flow.

### 1.2 Key Queue Metrics (Calculated per Queue/Transition)
For every transition (e.g., *Registration  Nurse Assessment*, *Nurse Assessment  Cardio Consult*), we compute:

| Metric | Purpose |
| :--- | :--- |
| **Mean Waiting Time** | Baseline performance; input for cost-of-waiting calculations. |
| **Median Waiting Time** | Robust central tendency (less skewed by outliers). |
| **90th / 95th Percentile Waiting Time** | **Critical for patient experience.** Patients remember the "worst" waits; drives dissatisfaction. |
| **Max Waiting Time** | Identifies systemic failures/scheduling catastrophes. |
| **Queue Frequency (Volume)** | Number of cases passing through this transition per hour/day. High volume + moderate wait = High total system waste. |
| **% Cases Exceeding Threshold** | e.g., "% waits > 20 mins" (clinic SLA target). Actionable for management dashboards. |
| **Wait-to-Service Ratio** | `Total_Wait_Time / Total_Service_Time` per case. High ratio = "Hurry up and wait" process. |
| **Resource Utilization during Wait** | % Time target resource (Doctor/Room) was **Busy** vs **Idle** while patient waited. Distinguishes *resource starvation* from *coordination failure*. |

### 1.3 Identifying Critical Queues (Prioritization Matrix)
We cannot fix everything at once. We prioritize using a **Weighted Impact Score**:

$$ \text{Impact Score} = (W_1 \times \text{Avg Wait}) + (W_2 \times \text{90th Pct Wait}) + (W_3 \times \text{Volume}) + (W_4 \times \text{Wait/Service Ratio}) + (W_5 \times \text{Patient Type Weight}) $$

*   **Criteria Justification:**
    *   **High 90th Percentile + High Volume:** Affects the most patients severely (e.g., *Nurse Assessment  Doctor Consult* for Follow-ups).
    *   **High Wait-to-Service Ratio:** Indicates structural inefficiency (e.g., *Registration  Nurse* wait 15min for a 5min triage).
    *   **Patient Type/Urgency Weight:** Queues affecting **New Patients** (high anxiety, complex needs) or **Urgent** slots (clinical risk) get higher weights ($W_5$).
    *   **Bottleneck Propagation:** Queues early in the process (Registration) cascade delays downstream. Fixing the "First Mile" often yields highest ROI.

**Output:** A ranked "Top 5 Critical Queues" list (e.g., 1. NurseDoctor (Cardio), 2. RegistrationNurse (New Pts), 3. DoctorECG, 4. ECGCheck-out, 5. Blood DrawResults Review).

## 2. Root Cause Analysis

Moving beyond *where* queues exist to *why* they exist requires enriching queue metrics with process mining perspectives.

### 2.1 Potential Root Causes & Analytical Techniques

| Root Cause Category | Hypothesis | Process Mining Technique to Validate |
| :--- | :--- | :--- |
| **Resource Bottlenecks (Capacity)** | Specific doctors/rooms are overbooked relative to capacity. Nurse pool insufficient for peak arrival rates. | **Resource Utilization Heatmaps:** Plot % Utilization per Resource (Doctor/Room) per Hour of Day. <br>**Resource Queue Length:** Simulate queue length at resource level using `START/COMPLETE` as arrival/service signals. Identify resources where `Utilization > 85%` consistently. |
| **Scheduling Policy Mismatch** | Appointment slots (e.g., 15min) << Actual Service Time (e.g., 22min median). "Wave scheduling" (4 patients at 9:00) creates instant queue. | **Service Time Distribution Analysis:** Compare `Scheduled_Slot_Duration` vs. `Actual_Service_Time` (per Doctor/Specialty/Patient Type). <br>**Arrival Pattern Analysis:** Plot patient `Registration_START` density per 15-min bucket vs. Scheduled slots. Detect "Batching" artifacts. |
| **Variability (The "Variability Buffer" Effect)** | High Coefficient of Variation (CV) in Service Times (e.g., New Pt Cardio: 15–45 min) forces long buffers or causes spillover. | **Variant Analysis:** Cluster cases by "Happy Path" vs. "Complex Path" (e.g., +ECG, +Bloods). Compare Wait Times per Variant. <br>**CV Calculation:** $CV = \sigma_{service} / \mu_{service}$. High CV ($>0.5$) requires disproportionate capacity slack. |
| **Handoff/Coordination Failures** | Patient ready, Doctor ready, but Room not cleaned / Chart not pulled / Pager failed. "Invisible" waits. | **Conformance Checking / Petri Net Discovery:** Model shows parallel paths (Doctor ready || Room ready) but log shows sequentialization. <br>**Resource-Activity Matrix:** Check if `Doctor_START` correlates with `Room_Available` timestamp (if room sensors exist) or `Nurse_Handoff_Complete`. |
| **Patient-Type Segregation Failure** | New Patients (long service) mixed with Follow-ups (short service) in same Doctor queue  Follow-ups wait behind New Pts. | **Slicing/Dicing by Attributes:** Filter event log by `Patient_Type` (New vs Follow-up) and `Urgency`. Compare Wait Times & Service Times per segment. <br>**Social Network / Handover Analysis:** Who hands off to whom? Are New Pts always routed to Senior Drs causing backlog? |
| **Downstream Blocking (Output Constraint)** | Doctor finishes, but ECG room full / Lab backed up  Doctor cannot discharge patient  Next patient waits in Exam Room (occupying Doctor). | **Blocked Time Analysis:** Identify `Doctor_COMPLETE` where `Next_Activity_START` is delayed *and* `Doctor` resource is marked "Busy" (blocked) rather than "Free". <br>**End-to-End Throughput Time Correlation:** High correlation between `ECG_Wait` and `Doctor_Utilization_Drop`. |

### 2.2 Deep Dive Example: "Nurse Assessment  Doctor Consultation (Cardio)"
*   **Observation:** Avg Wait = 22 mins; 90th Pct = 45 mins.
*   **Resource Analysis:** Dr. Smith (Cardio) Utilization = 92% (10:00–12:00). Nurse Utilization = 65%.
*   **Scheduling:** Dr. Smith booked 4 New Pts / hour (15 min slots). Median Service Time New Pt = 28 mins.
*   **Variant Analysis:** 30% of Cardio cases require ECG *before* Doctor. These are scheduled *after* Doctor in error, forcing Doctor to wait for ECG results or patient to re-queue.
*   **Root Cause:** **Systemic Under-scheduling** (Demand > Capacity) exacerbated by **Incorrect Sequencing** (ECG should be pre-Doctor for 30% cohort) and **Lack of Segregation** (Follow-ups stuck behind New Pts).

## 3. Data-Driven Optimization Strategies

### Strategy 1: Dynamic, Specialty-Specific Appointment Slotting & "Fast Track" Lanes
*   **Targeted Queues:** *Registration  Nurse*, *Nurse  Doctor (All Specialties)*, *Doctor  Diagnostics*.
*   **Root Cause Addressed:** Scheduling Policy Mismatch (Fixed 15-min slots vs. Variable Reality) & Patient-Type Segregation Failure.
*   **Data Support:**
    *   Service Time Analysis: New Cardio = 28min (P80=35min); Follow-up Cardio = 12min (P80=18min); New Derm = 20min.
    *   Arrival Data: 60% of volume 09:00–11:00.
*   **Action:**
    1.  **Variable Slot Lengths:** Configure scheduling template: New Cardio = 30min; Follow-up Cardio = 15min; New Derm = 20min.
    2.  **Protected "Fast Track" Slots:** Reserve 2 slots/hour/doctor exclusively for Follow-ups / Low-Complexity (identified via `Patient_Type` & `Historical_Service_Time`).
    3.  **Demand Leveling:** Smooth appointment offers: "We have 9:15 or 10:45 for New Patient" instead of "9:00 or 9:15".
*   **Expected Impact:**
    *   Reduce *NurseDoctor* Wait for Follow-ups by **40-50%** (decoupled from New Pt overruns).
    *   Reduce Doctor Overtime / End-of-day backlog by **~30 mins/day**.
    *   Increase Doctor Utilization efficiency (less idle time waiting for late arrivals caused by upstream chaos).

### Strategy 2: Pre-Visit "Digital Triage & Prep" Protocol (Parallelizing Registration/Assessment)
*   **Targeted Queues:** *Registration  Nurse Assessment* (Avg Wait 18 mins, High Volume), *Nurse Assessment  Doctor* (Reduces Nurse Service Time).
*   **Root Cause Addressed:** Sequential Bottleneck at Front Desk; Nurse time wasted on administrative data collection (meds, history) instead of clinical assessment.
*   **Data Support:**
    *   Nurse Service Time breakdown (via sub-activity logging or time-motion sample): 40% Admin (Med Rec, History), 60% Clinical (Vitals, Triage).
    *   Registration Service Time: 6 mins. Patient arrives 10 mins early avg.
*   **Action:**
    1.  **Pre-Check-in Portal/SMS:** 24h prior, patient completes Demographics, Insurance, Med Recon, Chief Complaint.
    2.  **Kiosk/QR Arrival:** Patient scans QR  Auto-pulls pre-submitted data  Prints label/wristband  **Registration COMPLETE** triggered automatically (Service Time ~1 min).
    3.  **Nurse "Warm Handoff":** Nurse receives pre-populated chart. Nurse START = Vitals immediately. Nurse Service Time reduced by ~4 mins.
*   **Expected Impact:**
    *   Eliminate *Registration Queue* (Wait  **< 2 mins**).
    *   Reduce *Nurse Assessment Service Time* by **25-30%**  Increases Nurse Capacity  Reduces *NurseDoctor* Queue arrival rate variability.
    *   "Zero-Wait Registration" significantly boosts Net Promoter Score (NPS).

### Strategy 3: "Pull-Based" Diagnostic Scheduling & Point-of-Care Testing (POCT)
*   **Targeted Queues:** *Doctor Consultation  ECG/Bloods/X-Ray* (High Wait, High Blocking), *Diagnostics  Check-out / Doctor Review*.
*   **Root Cause Addressed:** Push-based flow (Doctor orders  Patient walks to Diagnostics  Waits in new queue); Downstream Blocking (Doctor waits for results).
*   **Data Support:**
    *   ECG Wait: Avg 25 mins (Tech Utilization 70% - **Coordination Failure**, not capacity).
    *   Blood Test  Result Review Loop: 40% of cases require Doctor Review *after* results. Current flow: Doctor Discharges  Patient waits 45min for Bloods  Patient waits 20min for Doctor Recall.
    *   Variant Analysis: "Cardio New Pt" variant *always* needs ECG.
*   **Action:**
    1.  **Anticipatory Ordering (Pull):** At **Registration/Pre-Checkin**, algorithm flags "Cardio New Pt"  Auto-schedules ECG **Time-Slot** *before* Doctor Consult (Parallel: Nurse Vitals + ECG happen simultaneously).
    2.  **POCT for Critical Path:** Deploy Point-of-Care Troponin/BNP/HbA1c in Nurse Station. Nurse draws blood  Result in 10 mins  Available at Doctor START.
    3.  **Synchronous "Results Review" Slot:** For complex cases needing Lab + Doctor, schedule a single 40-min block: 20min Doctor  15min Lab (POCT/Stat)  5min Doctor Review. Eliminates "Discharge-Requeue" loop.
*   **Expected Impact:**
    *   Reduce *DoctorECG* Wait from 25min  **< 5 mins** (Patient walks from Nurse to ECG room at appointed time).
    *   Eliminate **100% of "Doctor Recall" loops** for standard panels (saves ~20 mins/patient visit duration).
    *   Reduce Overall Visit Duration (Throughput Time) for Cardio New Pts by **35-40 mins**.

## 4. Consideration of Trade-offs and Constraints

| Strategy | Potential Negative Side-Effects / Trade-offs | Mitigation / Balancing Approach |
| :--- | :--- | :--- |
| **1. Variable Slotting / Fast Track** | **Revenue Risk:** Fewer "New Patient" slots/day (longer slots)  Lower New Patient Volume. <br>**Complexity:** Scheduling template complexity increases; Front desk training required. <br>**Equity:** Follow-ups get "better" access; New Patients wait longer for appointment availability. | *Financial Modeling:* Calculate Revenue/Slot. If New Pt Revenue >> Follow-up, ensure volume loss is offset by reduced No-Shows (better experience) & Higher Throughput (less overtime). <br>*Dynamic Templating:* Use ML to predict daily "New vs Follow-up" demand mix and auto-adjust template ratios weekly. <br>*Access KPI:* Monitor "Third Next Available Appointment" (TNAA) for New vs Follow-up; enforce max ratio (e.g., 1.5x). |
| **2. Digital Pre-Check-in / Kiosks** | **Digital Divide:** Elderly, low-literacy, non-English speakers excluded  Longer waits for vulnerable groups. <br>**Data Quality:** Patient-entered Med Rec errors  Clinical Risk. <br>**Cost:** Software licenses, Kiosk hardware, Integration (HL7/FHIR). | *Hybrid Model:* Maintain 1 "High-Touch" Registration Clerk for assisted check-in. <br>*Clinical Safety:* Nurse *must* verbally reconcile critical meds/allergies (High-Alert Meds) regardless of pre-entry. <br>*Phased ROI:* Start with SMS/Link (Low Cost). Kiosks only if volume justifies. Track "Assisted Check-in %" KPI. |
| **3. Pull-Based Diagnostics / POCT** | **Resource Lock-in:** Pre-booking ECG slots reduces flexibility for Urgent/Walk-ins. <br>**POCT Cost:** Per-test cost >> Central Lab. Reagent waste if patient No-Shows. <br>**Space/Workflow:** Nurse drawing blood + Vitals + ECG prep = Nurse Burnout Risk. | *Buffer Slots:* Keep 20% ECG slots "Open/Urgent" daily. <br>*Selective POCT:* Only for high-impact tests (Troponin, HbA1c, Strep) where result changes *immediate* disposition. Send routine Chem/Heme to Central Lab. <br>*Task Shifting:* Hire 1 Phlebotomist/Tech for "Pre-Visit Prep Zone" (Vitals + POCT + ECG)  Offloads Nurse. Cost offset by Nurse FTE reduction or throughput gain. |

### Balancing Conflicting Objectives: The "Iron Triangle" Framework
We optimize using a **Weighted Objective Function** monitored monthly:
$$ \text{Utility} = w_1(\text{Patient Wait}) + w_2(\text{Cost/Visit}) + w_3(\text{Clinical Quality/Safety}) + w_4(\text{Staff Satisfaction}) $$
*   **Governance:** Monthly "Process Performance Review" with Medical Director, Ops Manager, CFO, Nurse Lead.
*   **Guardrails:** Hard constraints (e.g., *Med Rec Accuracy > 99%*, *Staff Overtime < 5%*, *Urgent Wait < 15min*). Optimization only permitted within guardrails.

To sustain improvement, we implement a **Continuous Process Mining Loop**:

1.  **Automated Data Pipeline (Daily/Weekly):**
    *   ETL from EHR/Scheduling System  Standardized Event Log (XES/Parquet).
    *   *Critical:* Enrich with `Scheduled_Start_Time`, `Appointment_Type`, `No_Show_Flag`.
2.  **Automated KPI Calculation & Alerting (Python/PM4Py/Celonis/PowerBI):**
    *   Daily "Queue Health" Report: Top 3 queues exceeding thresholds (e.g., NurseDoctor Wait > 20min avg yesterday).
    *   **Control Charts (SPC):** Plot Weekly Avg Wait per Queue. Detect special cause variation (e.g., new doctor, flu season, IT outage) vs common cause.
3.  **Conformance Checking (Monthly):**
    *   Compare *Actual* Process Graph (Discovered) vs. *Target* Process Model (Designed).
    *   **Metric:** "Fitness" % and "Deviation Count" (e.g., "Doctor Consult started before Nurse Assessment Complete" = Safety/Process Violation).
4.  **Root Cause Drill-Down (Ad-hoc / Quarterly):**
    *   **Variant Analysis:** Compare "Happy Path" (Pre-checkin + Pull Diagnostics) vs "Legacy Path" performance.
    *   **Resource Profiling:** Identify specific resources drifting in performance (e.g., Dr. Jones Wait Times increasing 10% QoQ).
5.  **Feedback Loop to Scheduling:**
    *   Feed `Actual_Service_Time_Distributions` back into Scheduling Template Engine **Quarterly** (Dynamic Slotting).
    *   Feed `No_Show_Rates` per Slot Type/Time  Overbooking Algorithm.
