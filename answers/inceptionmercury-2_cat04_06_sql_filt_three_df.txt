**DuckDB SQL**

```sql
/* --------------------------------------------------------------
   1  Order the events inside each case
   2  Find cases that contain the threeevent pattern in **direct
       succession** and where the gap between “Create Order” and
       “Confirm Order” is > 5 days.
   3  Exclude every event that belongs to those cases.
   -------------------------------------------------------------- */

WITH ordered AS (
    /* give each event a row number that respects the case order */
    SELECT
        *,
        ROW_NUMBER() OVER (PARTITION BY case_id ORDER BY timestamp) AS rn
    FROM event_log
),

bad_cases AS (
    /* locate the three consecutive rows that match the pattern */
    SELECT DISTINCT o1.case_id
    FROM ordered o1
    JOIN ordered o2
          ON o2.case_id = o1.case_id
         AND o2.rn = o1.rn + 1          -- next event in the case
    JOIN ordered o3
          ON o3.case_id = o1.case_id
         AND o3.rn = o1.rn + 2          -- the event after that
    WHERE o1.activity = 'Approve Purchase Requisition'
      AND o2.activity = 'Create Order'
      AND o3.activity = 'Confirm Order'
      /* time elapsed between Create Order and Confirm Order > 5 days */
      AND DATEDIFF('day', o2.timestamp, o3.timestamp) > 5
)

SELECT *
FROM event_log
WHERE case_id NOT IN (SELECT case_id FROM bad_cases);
```

### How it works

| Step | What it does |
|------|--------------|
| **ordered** | Adds a `rn` (row number) per `case_id` ordered by `timestamp`. This lets us refer to “the next event” (`rn+1`) and “the event after that” (`rn+2`). |
| **bad_cases** | Joins the table to itself twice, aligning three consecutive rows (`o1`, `o2`, `o3`). The `WHERE` clause enforces the exact activity order and checks that the difference between the timestamps of `o2` (Create Order) and `o3` (Confirm Order) exceeds 5 days (`DATEDIFF`). `DISTINCT` ensures each offending `case_id` appears only once. |
| **final SELECT** | Returns every row from `event_log` whose `case_id` is **not** in the list of offending cases, i.e., all events from cases that do **not** satisfy both the sequenceandtiming condition. |

The query therefore yields **all events except those belonging to cases that contain the specified threestep sequence with a >5day gap between “Create Order” and “Confirm Order”.**
