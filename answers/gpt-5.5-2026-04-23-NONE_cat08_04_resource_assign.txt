## 1. Analyzing Resource Behavior and Assignment Patterns

### 1.1 Prepare the event log for resource-focused process mining

Before analysis, TechSolve should enrich and structure the event log so that each ticket can be analyzed from both a process and resource perspective.

At minimum, each event should include:

| Field | Purpose |
|---|---|
| Case ID / Ticket ID | Tracks the full lifecycle of one incident |
| Activity | Ticket created, assigned, work started, escalated, reassigned, resolved, closed |
| Timestamp | Enables waiting time, processing time, SLA analysis |
| Timestamp type | Start / complete events allow accurate duration calculations |
| Resource / Agent ID | Enables resource performance and workload analysis |
| Agent tier | L1, L2, L3 |
| Agent skills | Declared agent capabilities |
| Ticket priority | P1, P2, P3, P4 |
| Ticket category | Hardware, Network, Access Management, etc. |
| Required skill | Skill believed necessary to resolve the ticket |
| Assignment target | Agent or queue assigned to |
| SLA deadline / SLA status | Required to connect assignment patterns to breaches |
| Resolution outcome | Resolved by L1, escalated, reassigned, reopened, breached, etc. |

The event log should also distinguish between:

- Active work time.
- Queue/waiting time.
- Assignment delay.
- Reassignment delay.
- Escalation delay.
- Time waiting for customer/vendor, if applicable.

This separation is critical because SLA breaches may be caused less by “work effort” and more by waiting for the right person to pick up the ticket.

---

### 1.2 Analyze performance and behavior of agents and support tiers

The first step is to build a baseline of how work is actually distributed and executed across L1, L2, and L3.

#### A. Workload distribution

Metrics to analyze:

| Metric | Why it matters |
|---|---|
| Number of tickets handled per agent | Detects overloaded or underused agents |
| Number of activities performed per agent | Shows total process participation |
| Number of active work sessions per agent | Helps distinguish short administrative touches from real work |
| Tickets by priority per agent | Reveals whether critical tickets are concentrated on a few people |
| Tickets by category and required skill per agent | Shows specialization patterns |
| Average concurrent workload | Indicates whether agents are assigned too many open tickets |
| Queue assigned but not started | Reveals hidden backlog per agent or team |

Example analysis:

- Agent B08 may handle 35% of all Network/Firewall P2 incidents.
- Agent B12 may receive many App-CRM tickets but frequently reassigns tickets requiring Database-SQL.
- Some L1 agents may have many created tickets but few resolved tickets, indicating low first-call resolution effectiveness.

---

#### B. Processing time by agent, tier, priority, and ticket type

Processing time should be split into:

1. Time from ticket creation to first assignment.
2. Time from assignment to work start.
3. Active handling time.
4. Time from work end to escalation or resolution.
5. Total cycle time.
6. SLA remaining time at each handover.

Useful metrics:

| Metric | Example question answered |
|---|---|
| Median processing time per agent | Who resolves comparable tickets efficiently? |
| 90th percentile processing time | Which agents or queues create long-tail delays? |
| Processing time by ticket category | Are Network tickets slower than Software-App tickets? |
| Processing time by required skill | Are Database-SQL tickets waiting too long? |
| Queue time before L2/L3 work starts | Are specialists overloaded or unavailable? |
| Touch time vs. wait time ratio | Is the bottleneck effort or waiting? |

It is important to compare agents only on similar work types. For example, an L3 security specialist handling P1 security incidents should not be compared directly with an L1 agent handling password resets.

---

#### C. First-call resolution rate for L1

For L1, first-call resolution is a key metric.

Formula:

> First-call resolution rate = tickets resolved by L1 without escalation / tickets initially handled by L1

This should be segmented by:

- L1 agent.
- Ticket category.
- Required skill.
- Priority.
- Channel: phone, email, portal.
- Time of day or shift.
- Customer/business unit.

Example insights:

- L1 resolves 82% of Access Management password reset tickets.
- L1 resolves only 12% of App-CRM incidents, with most escalating to the same L2 team.
- Agent A02 escalates 70% of Network tickets after an average of 30 minutes of L1 work, suggesting either incorrect routing to L1 or lack of L1 network triage guidance.
- Some L1 agents may resolve categories that others escalate, indicating training opportunities.

---

#### D. Escalation and reassignment behavior

Key metrics:

| Metric | Purpose |
|---|---|
| Escalation rate from L1 to L2 | Detects ticket types unsuitable for L1 |
| Escalation rate from L2 to L3 | Detects complexity or misrouting |
| Reassignment count per ticket | Measures assignment quality |
| Reassignment count per agent/team | Identifies problematic queues or skill mismatches |
| Average delay per reassignment | Quantifies waste |
| Reassignment reason frequency | Separates valid from avoidable reassignments |
| Tickets with multiple handovers | Indicates process fragmentation |

Important distinctions:

- A valid escalation is where the ticket legitimately needs higher-tier expertise.
- An avoidable reassignment is where the ticket was routed to a person or team without the required skill, availability, or authority.
- A ping-pong ticket is repeatedly moved between resources or tiers without progress.

For example:

- Tickets with zero reassignment may have an average resolution time of 6 hours.
- Tickets with one reassignment may average 11 hours.
- Tickets with two or more reassignments may average 21 hours and breach SLA at three times the normal rate.

---

### 1.3 Use process mining to reveal actual assignment patterns

Process mining can discover the real incident handling process from the event log, rather than relying on documented procedures.

#### A. Process discovery

Using process discovery, TechSolve can compare the intended process:

> Create ticket  Assign L1  Work L1  Resolve  
> or  
> Create ticket  Assign L1  Work L1  Escalate L2  Work L2  Resolve

against the actual process, which may look more like:

> Create ticket  Assign L1  Work L1  Escalate L2  Assign L2  Reassign  Assign L2  Work L2  Escalate L3  Assign L3  Resolve

Process discovery will reveal:

- Frequent loops around reassignment.
- Cases where tickets skip L1.
- Tickets assigned directly to L3.
- Tickets returned from L2 to L1.
- Tickets assigned to queues but not started for long periods.
- Non-standard handling paths correlated with SLA breaches.

---

#### B. Conformance checking

Conformance checking compares actual event paths with the designed assignment rules.

Examples:

| Intended rule | Conformance question |
|---|---|
| P1 tickets should go directly to senior resolver group | Are P1 tickets being initially assigned to L1? |
| Network-Firewall tickets should be routed to certified network agents | Are they being assigned to general L2 resources? |
| L1 should attempt troubleshooting before escalation | Are L1 agents escalating immediately? |
| Tickets requiring Security-IAM should not be handled by non-IAM agents | Are there skill mismatches? |
| Reassignment should include a documented reason | Are reassignments properly justified? |

This helps separate policy violations from policy weaknesses. If everyone is following a flawed round-robin rule, the issue is design. If agents frequently bypass assignment rules, the issue may be governance, workload pressure, or lack of trust in the routing process.

---

#### C. Resource interaction analysis and handover networks

A social network analysis can be built from handovers between resources.

A handover occurs when one resource performs an activity on a ticket and the next activity is performed by another resource.

Examples:

- Agent A05  Agent B12.
- Agent B12  Agent B15.
- L1 Pool  L2 App Team.
- L2 Network Team  L3 Security Team.

A handover network can reveal:

- Which agents or teams are central bottlenecks.
- Which teams frequently pass work to each other.
- Whether certain L1 agents often escalate to the same L2 agent.
- Whether tickets bounce between teams.
- Whether informal routing patterns differ from official routing rules.

Useful network metrics:

| Metric | Meaning |
|---|---|
| Degree centrality | How many handover connections a resource has |
| Betweenness centrality | Whether a resource acts as a routing bridge or bottleneck |
| Handover frequency | Volume of transfers between two resources |
| Handover delay | Time lost between two resources |
| Loop frequency | How often work returns to a previous agent/team |

Example insight:

A handover network may reveal that almost all Database-SQL tickets eventually flow to Agent B15, regardless of initial assignment. This suggests the process should route these tickets to B15 or a qualified database group earlier, or train more agents in Database-SQL.

---

#### D. Role discovery

Role discovery groups agents by actual behavior rather than job title.

For example, the system may discover that:

- Some L1 agents effectively act as access management specialists.
- Some L2 agents mostly perform routing and triage rather than resolution.
- Some L3 specialists repeatedly handle routine L2 work.
- Certain agents are “informal experts” for specific applications.

This helps TechSolve identify:

- Actual skill clusters.
- Misalignment between formal tier and real work performed.
- Candidates for promotion or specialist designation.
- Tasks that could be shifted left to L1 or L2.

---

### 1.4 Analyze utilization of specific skills

Skill utilization analysis should compare three dimensions:

1. Ticket required skill.
2. Assigned agent skill profile.
3. Actual resolving agent.

For each ticket, classify the assignment as:

| Assignment type | Definition |
|---|---|
| Matched assignment | Assigned agent has the required skill |
| Partial match | Agent has related but not exact skill |
| Mismatch | Agent lacks required skill |
| Over-skilled assignment | Agent has much higher skill/tier than needed |
| Under-skilled assignment | Agent lacks skill or authority, likely causing escalation |
| Unknown | Required skill or agent skill profile missing |

Important metrics:

- Percentage of tickets assigned to agents with matching skills.
- Resolution time for skill-matched vs. mismatched tickets.
- Reassignment rate for skill-matched vs. mismatched tickets.
- SLA breach rate for skill-matched vs. mismatched tickets.
- Percentage of low-complexity tickets handled by L2/L3 specialists.
- Utilization rate of each skill group.
- Demand vs. capacity by skill and time period.

Example findings:

- Only 54% of Network-Firewall tickets are initially assigned to agents with that skill.
- Skill-mismatched P2 tickets breach SLA 2.4 times more often than matched tickets.
- L3 security specialists spend 28% of their time on routine access management tickets that could be handled by trained L1/L2 agents.
- Database-SQL tickets wait an average of 90 minutes before reaching a qualified agent.

This analysis directly supports skill-based routing, cross-training, and workload balancing.

## 2. Identifying Resource-Related Bottlenecks and Issues

### 2.1 Bottlenecks caused by insufficient availability of specific skills

The process mining analysis should identify where tickets wait longest and which resources or skills are associated with those waits.

Questions to answer:

- Which required skills have the longest assignment-to-start delay?
- Which skills have the highest backlog?
- Which agents with critical skills are consistently overloaded?
- Are delays caused by lack of capacity, poor routing, or both?
- Are bottlenecks worse during certain shifts, weekdays, or months?

Example analysis:

| Required Skill | Avg Queue Time Before Qualified Agent Starts | SLA Breach Rate | Interpretation |
|---|---:|---:|---|
| Networking-Firewall | 3.8 hours | 31% | Insufficient skilled availability or poor routing |
| Database-SQL | 4.5 hours | 37% | Severe bottleneck, likely too few qualified agents |
| App-CRM | 1.2 hours | 14% | Moderate issue |
| Security-IAM | 2.9 hours | 26% | Potential shift coverage issue |
| Basic-Troubleshoot | 0.4 hours | 5% | Adequate capacity |

If Database-SQL tickets all eventually go to two agents, and those agents have high utilization, the root problem may be skill capacity. If enough Database-SQL-skilled agents exist but tickets first go elsewhere, the root problem is assignment logic.

---

### 2.2 Delays introduced by reassignments and escalations

For every ticket, calculate:

- Number of reassignments.
- Number of escalations.
- Time between reassignment and next work start.
- Time spent before realizing reassignment was necessary.
- Whether the reassignment changed required skill.
- Whether the reassignment changed tier.
- Whether the ticket breached SLA.

Example metrics:

| Reassignment Count | Avg Resolution Time | SLA Breach Rate |
|---:|---:|---:|
| 0 | 5.5 hours | 8% |
| 1 | 9.8 hours | 17% |
| 2 | 16.4 hours | 34% |
| 3+ | 29.2 hours | 58% |

Average delay per reassignment can be calculated as:

> Reassignment delay = timestamp of next work start after reassignment  timestamp of reassignment/assignment event

Also calculate avoidable delay:

> Avoidable delay = time spent with agent lacking required skill + queue time before correct agent starts

Example:

- Average reassignment delay: 2.1 hours.
- Average active work before reassignment: 34 minutes.
- Average total loss per reassignment: 2.7 hours.
- 41% of P2 SLA breaches involved at least one skill-mismatch reassignment.
- Tickets with three or more handovers accounted for 12% of volume but 38% of SLA breaches.

---

### 2.3 Impact of incorrect initial assignments

Initial assignment quality can be measured by whether the first assigned agent or group could resolve or properly progress the ticket.

Metrics:

| Metric | Meaning |
|---|---|
| First-assignment match rate | Initial agent has required skill |
| First-assignment resolution rate | Ticket resolved by initially assigned agent/team |
| First-assignment escalation rate | Ticket escalated from first assignment |
| First-assignment reassignment rate | Ticket reassigned before meaningful work |
| Time lost before correct routing | Delay from first assignment to assignment to qualified agent |
| SLA risk introduced by initial assignment | SLA consumption before qualified work starts |

Segment by:

- Dispatcher.
- Channel.
- Category.
- Priority.
- Required skill.
- Time of day.
- L1 agent or resolver group.

Example findings:

- Web portal tickets with vague category “Software-App” are misrouted 38% of the time.
- Dispatcher-assigned P3 tickets have fewer reassignments than auto-round-robin assignments.
- Phone tickets self-assigned by L1 agents have high L1 engagement but also high escalation for Network-Firewall incidents.
- P2 Network tickets assigned initially to L1 consume an average of 22% of their SLA before escalation.

---

### 2.4 Identification of underperforming or overloaded agents/teams

This must be handled carefully and fairly. Agent-level analysis should be normalized by work type, priority, complexity, and shift.

Indicators of overload:

- High number of open tickets.
- High concurrent ticket count.
- Long assignment-to-start delay.
- High queue backlog.
- High after-hours work.
- Increasing processing time over the day/week.
- High percentage of critical or complex tickets.

Indicators of potential underutilization:

- Low ticket count.
- Low active work time.
- Low concurrency.
- High availability with few assignments.
- Skills in profile but rarely used.

Indicators of possible performance or training issues:

- High escalation rate compared to peers for similar tickets.
- High reassignment rate from the agent.
- Long processing time for standard ticket types.
- High reopen rate.
- High error or incomplete resolution rate.
- Low first-call resolution compared to peers.

Example:

- Agent B15 has high processing volume and long queue time for Database-SQL tickets, indicating overload, not poor performance.
- Agent A07 has a 72% escalation rate for Access Management tickets, while the L1 average is 28%, suggesting training or knowledge-base gaps.
- Agent C03 handles many P4 routine tasks despite being L3, indicating poor specialist utilization.

---

### 2.5 Correlation between assignment patterns and SLA breaches

Use statistical analysis and process mining filters to compare breached vs. non-breached cases.

Factors to test:

- Number of reassignments.
- Number of escalations.
- First assignment skill match.
- Time to first work start.
- Required skill.
- Ticket priority.
- Ticket category.
- Agent/tier involved.
- Queue length at assignment time.
- Workload of assigned agent.
- Time of day or shift.
- Channel.
- Reopen count.

Useful methods:

- Variant comparison.
- Regression analysis.
- Survival analysis / time-to-breach modeling.
- Decision trees.
- Process mining performance overlays.
- Predictive monitoring models.

Example quantified outputs:

| Factor | Effect |
|---|---|
| Initial skill mismatch | 2.3x higher odds of SLA breach |
| More than one reassignment | 3.1x higher odds of SLA breach |
| Assignment to agent with >8 open tickets | 1.8x higher odds of SLA breach |
| Database-SQL required skill | 2.6x higher odds of SLA breach |
| L2 queue time >2 hours | 4.0x higher odds of SLA breach |
| P2 ticket created after 4 PM | 1.7x higher odds of breach due to shift coverage |

These insights make the business case for assignment redesign.

## 3. Root Cause Analysis for Assignment Inefficiencies

### 3.1 Deficiencies in current assignment rules

The current model appears to combine round-robin within tiers and manual escalation decisions. This can cause problems if assignment ignores:

- Required technical skill.
- Agent proficiency.
- Current workload.
- SLA urgency.
- Availability or shift schedule.
- Ticket complexity.
- Historical success patterns.
- Whether the ticket is likely resolvable by L1.

Potential root causes:

- Round-robin assigns tickets equally by count but not by effort.
- Agents receive tickets they cannot resolve.
- P2/P3 tickets are not prioritized by SLA risk.
- Specialists receive routine work because they are “next in line.”
- Dispatchers rely on tribal knowledge instead of data.
- Escalations are based on subjective judgment, not predictive evidence.

Process mining can validate this by showing that tickets assigned through round-robin have higher reassignment and breach rates than skill-based manual assignments, or vice versa.

---

### 3.2 Inaccurate or incomplete agent skill profiles

Skill-based assignment depends on accurate skill data. Problems may include:

- Agent skills are outdated.
- Skill levels are binary rather than proficiency-based.
- Certifications are listed but not actually practiced.
- Informal expertise is not documented.
- Agents have skills but are not available for that type of work.
- Skill profiles do not distinguish between primary and backup skills.

Process mining can infer actual skills by examining:

- Which agents resolve which ticket types successfully.
- Resolution speed by category and required skill.
- Reopen rate by agent and ticket type.
- Frequency of reassignment away from an agent for certain skills.
- Historical success rate for similar tickets.

Example:

Agent B12 lists App-CRM and Database-SQL, but App-CRM tickets are resolved efficiently while Database-SQL tickets are frequently reassigned to B15. This suggests B12 may have basic SQL familiarity but not enough proficiency for advanced database incidents.

---

### 3.3 Poor initial ticket categorization or required skill identification

If the ticket category or required skill is wrong at creation, even a good routing algorithm will fail.

Issues may include:

- Customers choose incorrect web portal categories.
- Email tickets lack structured metadata.
- L1 agents misclassify symptoms.
- Required skill changes after investigation.
- Categories are too broad, such as “Software-App.”
- Multiple skills are required, but only one is captured.

Process mining can detect this by comparing:

- Initial category vs. final resolving team.
- Initial required skill vs. final required skill.
- Initial classification vs. reassignment reason.
- Ticket text keywords vs. assigned category.
- Categories with high reassignment rates.

Example:

If many tickets initially classified as Software-App ultimately require Database-SQL, TechSolve should improve classification rules or use natural language processing to identify database-related symptoms earlier.

---

### 3.4 Lack of real-time workload and availability visibility

Even when skill profiles are correct, assignment may fail if dispatchers cannot see:

- Current open ticket count.
- Queue backlog per agent.
- Active work in progress.
- Shift coverage.
- Agent absence.
- SLA risk of assigned tickets.
- Specialist capacity.
- Interruptions from P1 incidents.

Process mining can reveal workload problems retrospectively, but operational dashboards should expose them in real time.

Example root cause:

Agent B08 is qualified for Networking-Firewall, so tickets are assigned correctly by skill. However, B08 already has ten open P2 tickets, while B09 has the same skill and lower workload. This indicates the issue is not skill matching but lack of workload-aware balancing.

---

### 3.5 Insufficient L1 training or empowerment

Frequent escalation from L1 may have several causes:

- L1 lacks knowledge-base articles.
- L1 lacks access rights to execute fixes.
- L1 does not have diagnostic scripts or tools.
- L1 performance incentives encourage fast escalation.
- L1 agents do not know which issues they are expected to resolve.
- Some ticket types are incorrectly routed to L1 even though L1 cannot handle them.

Process mining can show:

- Which categories L1 successfully resolves.
- Which categories always escalate.
- Which L1 agents outperform peers on similar tickets.
- Whether escalation is immediate or after long unsuccessful work.
- Whether certain successful L1 patterns can be standardized.

Example:

If L1 frequently spends 25 minutes on Network-Firewall tickets and then escalates 90% of them, the better approach may be immediate skill-based routing to L2 Network, with L1 performing only specific triage steps.

---

### 3.6 Use variant analysis to compare smooth and problematic cases

Variant analysis compares different process paths.

Define “smooth assignment” cases as:

- No reassignment.
- Correct initial skill match.
- No unnecessary escalation.
- SLA met.
- Low queue time.
- Resolved by expected tier.

Define “problematic assignment” cases as:

- Two or more reassignments.
- Skill mismatch.
- SLA breached.
- Long queue time.
- Multiple tier changes.
- Reopen after resolution.

Compare the two groups by:

| Attribute | Smooth cases | Problematic cases |
|---|---:|---:|
| Initial category | Which categories dominate? | Which categories dominate? |
| Required skill | Which skills are stable? | Which skills change? |
| Channel | Phone/web/email distribution | Higher misclassification channel? |
| Initial agent/tier | Correct first assignment? | Wrong tier? |
| Time of day | Normal coverage? | Shift gaps? |
| Queue length at assignment | Low? | High? |
| Dispatcher | Consistent quality? | Higher misrouting? |

This identifies the conditions under which the assignment process breaks down.

---

### 3.7 Use decision mining to understand assignment decisions

Decision mining connects branching points in the process to case attributes and resource attributes.

Example decision points:

- Assign to L1 or directly to L2.
- Escalate from L1 to L2 or resolve at L1.
- Assign to App team or Database team.
- Reassign or continue with same agent.
- Escalate to L3 or retain in L2.

Decision mining can discover rules such as:

- If category = Network and priority = P2, tickets are usually escalated after L1.
- If required skill is blank, tickets have a high reassignment probability.
- If assigned agent has more than seven open tickets, work start is delayed.
- If ticket channel is email, categorization errors are more common.
- If initial category is Software-App and keywords include “database,” final resolver is Database-SQL group.

These learned rules can be transformed into improved assignment policies.

## 4. Developing Data-Driven Resource Assignment Strategies

Below are five concrete strategies. TechSolve should prioritize at least the first three, then phase in the others.

---

### Strategy 1: Implement skill-based routing with proficiency weighting

#### Assignment issue addressed

This addresses:

- Incorrect initial assignment.
- Excessive reassignments due to skill mismatch.
- Specialists receiving tickets too late.
- SLA breaches caused by routing delays.

#### How it uses process mining insights

Process mining identifies:

- Which required skills have high mismatch rates.
- Which agents actually resolve specific ticket types successfully.
- Which skills create bottlenecks.
- Which initial assignments lead to reassignments.
- Which agents have proven proficiency in specific categories.

Instead of assigning tickets round-robin within a tier, the routing engine should match ticket required skill to agent skill profile.

A simple scoring model:

> Assignment score = skill match score + proficiency score + historical success score + SLA priority score  overload penalty

Example:

For a P2 Network-Firewall ticket:

- Agent B08: Networking-Firewall proficiency 5, current workload high.
- Agent B09: Networking-Firewall proficiency 4, current workload low.
- Agent B10: General Network proficiency 3, no firewall specialization.

The algorithm may assign to B09 if the expected start time is faster and skill level is sufficient.

#### Data required

- Ticket category.
- Required skill.
- Ticket priority and SLA deadline.
- Agent skill profile.
- Skill proficiency level.
- Historical resolution rate by agent/skill.
- Reopen rate by agent/skill.
- Current availability.
- Current workload.
- Shift schedule.

#### Expected benefits

- Fewer reassignments.
- Faster time to qualified agent.
- Reduced SLA breaches for P2/P3 tickets.
- Better use of specialists.
- Improved customer experience.
- More transparent assignment decisions.

Example target improvements:

- Reduce skill-mismatch assignments by 40–60%.
- Reduce average reassignment count by 25–40%.
- Improve P2 SLA compliance by 10–20 percentage points.

---

### Strategy 2: Introduce workload-aware and SLA-aware assignment

#### Assignment issue addressed

This addresses:

- Uneven workload distribution.
- Overloaded specialists.
- Underutilized qualified agents.
- Long assignment-to-start delays.
- Tickets assigned to the right skill but wrong person due to capacity constraints.

#### How it uses process mining insights

Process mining reveals:

- Queue times by agent/team.
- Open ticket volume at assignment.
- Workload patterns by time of day.
- SLA breach correlation with overloaded resources.
- Actual throughput per resource.
- Historical average handling time by ticket type.

The assignment algorithm should not only ask:

> Who can solve this?

It should also ask:

> Who can solve this soon enough to meet SLA?

A workload-aware score may include:

- Current number of open tickets.
- Current active tickets.
- Remaining SLA time of existing tickets.
- Estimated effort of new ticket.
- Agent availability in current shift.
- Queue length of agent/team.
- Priority weighting.

Example rule:

Do not assign a P2 ticket to a qualified agent if their projected start time exceeds the latest safe start time for that SLA, unless no alternative exists.

#### Data required

- Real-time open ticket count per agent.
- Ticket status and queue position.
- Active vs. waiting tickets.
- Agent shift calendar.
- Estimated handling time by category/skill/priority.
- SLA deadline and remaining SLA time.
- Agent availability and absence data.

#### Expected benefits

- More balanced workload.
- Reduced queue time.
- Lower SLA breach risk.
- Better use of available skill capacity.
- Fewer emergency escalations.
- Lower burnout for key specialists.

Example target improvements:

- Reduce average assignment-to-start delay by 20–35%.
- Reduce tickets assigned to overloaded agents by 50%.
- Improve workload balance across similarly skilled agents.

---

### Strategy 3: Use predictive ticket classification and predictive assignment

#### Assignment issue addressed

This addresses:

- Poor initial categorization.
- Incorrect required skill identification.
- Delayed recognition of specialist needs.
- Unnecessary L1 handling of tickets historically not resolvable by L1.

#### How it uses process mining insights

Process mining shows historical patterns:

- Which ticket descriptions eventually required which skills.
- Which categories are frequently reclassified.
- Which keywords are associated with escalations.
- Which initial paths lead to SLA breaches.
- Which tickets are likely to be resolved by L1.

A predictive model can be trained using historical ticket data to predict:

- Required skill.
- Likely resolver group.
- Complexity.
- Probability of L1 resolution.
- Probability of escalation.
- SLA breach risk.
- Expected handling time.

Example:

If a ticket description contains “VPN,” “firewall,” “blocked port,” and category is Network, the model predicts Networking-Firewall with high confidence and routes directly to the appropriate L2 network specialist queue.

If a ticket description contains “CRM login error” and historical data shows 70% L1 resolution using password/session reset steps, route to trained L1 agents with the right knowledge article.

#### Data required

- Ticket short description and long description.
- Historical initial and final categories.
- Required skill at resolution.
- Final resolving agent/team.
- Escalation history.
- Reassignment history.
- Resolution codes.
- Knowledge-base articles used.
- SLA outcomes.
- Customer/channel metadata.

#### Expected benefits

- Better first assignment accuracy.
- Fewer reclassifications and reassignments.
- Faster routing to qualified resources.
- Improved L1 resolution for suitable tickets.
- Earlier identification of high-risk tickets.

Example target improvements:

- Increase first-assignment accuracy by 20–30%.
- Reduce category-related reassignments by 30–50%.
- Reduce time to qualified agent for misclassified ticket types.

---

### Strategy 4: Refine L1 escalation criteria and enable shift-left resolution

#### Assignment issue addressed

This addresses:

- Excessive escalations from L1.
- L2/L3 specialists handling routine work.
- L1 spending time on issues they cannot solve.
- Inconsistent first-call resolution across L1 agents.

#### How it uses process mining insights

Process mining identifies:

- Ticket types frequently resolved by high-performing L1 agents.
- Ticket types almost always escalated.
- L1 agents with unusually high or low escalation rates.
- Categories where L1 spends time but rarely resolves.
- Knowledge articles associated with successful resolution.
- Skills that could be moved from L2 to L1 through training or access changes.

Actions:

1. Create data-driven L1 resolution playbooks for high-volume ticket types.
2. Identify ticket types suitable for direct L2 routing.
3. Train L1 on categories where peer agents already succeed.
4. Give L1 limited permissions/tools for routine fixes.
5. Define escalation thresholds based on evidence.

Example escalation policy:

- Password reset / MFA issue: L1 handles unless account lock persists after defined steps.
- Network-Firewall P2: L1 performs triage checklist within 5 minutes, then routes to L2 Network.
- Database-SQL production incident: route directly to L2/L3 Database team; L1 only captures required diagnostics.

#### Data required

- L1 resolution rates by category and agent.
- L1 active handling time before escalation.
- Escalation reasons.
- Knowledge article usage.
- Required access/tools.
- Ticket complexity indicators.
- Reopen rates after L1 resolution.
- Customer satisfaction by resolution tier.

#### Expected benefits

- Increased L1 first-call resolution where appropriate.
- Reduced wasted L1 effort on unsuitable tickets.
- Lower L2/L3 workload for routine incidents.
- Faster escalation for genuinely complex tickets.
- More consistent service quality.

Example target improvements:

- Increase L1 first-call resolution for suitable categories by 10–25%.
- Reduce L2 handling of routine access tickets by 30%.
- Reduce average L1 time before unavoidable escalation.

### Strategy 5: Dynamic resource reallocation based on real-time demand

#### Assignment issue addressed

This addresses:

- Demand peaks by priority, category, or skill.
- Shift-based shortages.
- Specialists overloaded while adjacent teams are idle.
- Static tier boundaries that do not reflect real demand.

#### How it uses process mining insights

Process mining can reveal demand patterns:

- P2 network tickets spike Monday mornings.
- Access Management tickets spike after password policy changes.
- App-CRM tickets spike after application releases.
- Security-IAM demand increases during onboarding cycles.
- L2 database backlog grows after 3 PM due to limited coverage.

Dynamic reallocation may include:

- Temporarily assigning cross-trained L1 agents to access management queues.
- Moving L2 generalists to App-CRM backlog during peaks.
- Reserving L3 capacity for P1/P2 work.
- Creating rotating “rapid response” specialists for high-risk queues.
- Adjusting shift schedules based on demand heatmaps.

#### Data required

- Real-time ticket arrival rates.
- Backlog by category, skill, priority, and queue.
- Agent skill matrix.
- Cross-training eligibility.
- Shift schedules.
- SLA risk forecast.
- Historical demand by time period.
- Release/change calendar.

#### Expected benefits

- Better responsiveness during peaks.
- Reduced bottleneck queues.
- Improved SLA performance for high-priority tickets.
- Better specialist availability.
- More flexible staffing model.

## 5. Simulation, Implementation, and Monitoring

### 5.1 Use business process simulation before implementation

Business process simulation allows TechSolve to test proposed assignment strategies without disrupting operations.

The simulation should be built from mined process models and historical resource data.

#### A. Build the baseline simulation

The baseline should represent current reality, including:

- Ticket arrival patterns by time, priority, category, and channel.
- Current assignment logic.
- Current escalation probabilities.
- Current reassignment probabilities.
- Processing time distributions by activity, tier, skill, and resource.
- Agent availability and shift schedules.
- Queue discipline.
- SLA rules.
- Skill profiles.
- Workload constraints.

The baseline simulation should be calibrated so that it reproduces actual historical results, such as:

- Average resolution time.
- SLA breach rate.
- Reassignment rate.
- Queue time.
- Workload distribution.
- L1 resolution rate.

If the baseline model cannot reproduce reality, it should not be used for forecasting improvements.

---

#### B. Simulate alternative assignment strategies

Scenarios to test:

1. Current round-robin/manual model.
2. Skill-based routing only.
3. Skill-based plus workload-aware routing.
4. Predictive classification plus skill-based routing.
5. Direct L2 routing for selected ticket types.
6. Increased L1 empowerment for selected categories.
7. Cross-training additional agents in bottleneck skills.
8. Shift adjustment for high-demand periods.
9. Reserving specialist capacity for P1/P2 tickets.
10. Hybrid model combining multiple strategies.

For each scenario, compare:

| KPI | Baseline | Scenario A | Scenario B |
|---|---:|---:|---:|
| P2 SLA compliance | Current | Simulated | Simulated |
| P3 SLA compliance | Current | Simulated | Simulated |
| Avg resolution time | Current | Simulated | Simulated |
| Avg queue time | Current | Simulated | Simulated |
| Reassignment rate | Current | Simulated | Simulated |
| L1 first-call resolution | Current | Simulated | Simulated |
| Specialist utilization | Current | Simulated | Simulated |
| Workload balance | Current | Simulated | Simulated |

Simulation can help avoid unintended consequences. For example, routing all firewall tickets to the best firewall expert may reduce reassignments but overload that expert and worsen SLA performance. A workload-aware model may perform better.

### 5.2 Implementation plan

A practical rollout should be phased.

#### Phase 1: Diagnostic and data validation

- Clean and enrich the event log.
- Validate agent skill profiles.
- Define required skill taxonomy.
- Map SLA rules.
- Identify top bottleneck categories and skills.
- Build initial process mining dashboards.

#### Phase 2: Pilot improved routing

Start with a focused pilot, for example:

- P2/P3 Network-Firewall tickets.
- App-CRM and Database-SQL tickets.
- Access Management tickets suitable for L1 shift-left.

Implement:

- Skill-based routing.
- Workload visibility.
- Escalation rules.
- Dashboard monitoring.

Compare pilot group against control group or historical baseline.

#### Phase 3: Add predictive classification

- Train model on historical data.
- Validate predictions with resolver teams.
- Use recommendations in assistive mode first.
- Gradually automate high-confidence assignments.
- Require human approval for low-confidence predictions.

#### Phase 4: Scale and optimize

- Extend to more categories and skills.
- Introduce dynamic staffing recommendations.
- Update knowledge base and L1 training.
- Refine model based on feedback.
- Regularly review skill profiles and routing outcomes.

## Summary of Recommended Approach

TechSolve should use process mining to move from intuition-based resource assignment to evidence-based resource orchestration.

The core approach is:

1. **Discover the actual process** and compare it with intended assignment rules.
2. **Analyze resource behavior** by agent, tier, skill, workload, and ticket type.
3. **Quantify delays** caused by mismatches, reassignments, escalations, and overloaded queues.
4. **Identify root causes** using variant analysis, conformance checking, handover analysis, and decision mining.
5. **Implement improved routing**, starting with skill-based and workload-aware assignment.
6. **Enhance classification and escalation** using predictive analytics and L1 performance data.
7. **Test changes through simulation** before rollout.
8. **Monitor continuously** using process mining dashboards focused on SLA, skills, handovers, utilization, and assignment quality.

The expected outcomes are:

- Fewer unnecessary reassignments.
- Faster routing to qualified agents.
- Better workload balance.
- Improved P2/P3 SLA compliance.
- Higher L1 first-call resolution for appropriate tickets.
- Better utilization of L2/L3 specialists.
- More transparent and defensible assignment decisions.
