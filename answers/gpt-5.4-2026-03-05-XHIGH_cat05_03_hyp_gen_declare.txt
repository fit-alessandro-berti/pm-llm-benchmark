## 1) Anomalies in the model

| Issue | Why it is anomalous |
|---|---|
| `C` is mandatory, but `E` and `C` are declared non-coexistent | This makes evaluation incompatible with closure. In effect, a claim can be closed only if it was never evaluated, which is hard to reconcile with a normal claim-handling flow. |
| The model permits `R -> C` as a compliant path | Because only receipt is fixed at the start and closure is required somewhere, intermediate control steps can be skipped entirely. |
| The dependency between `A` and `E` is too weak | It only ties the two activities by eventual coexistence, not by order. That means evaluation is not guaranteed to happen after assignment. |
| No rule requires `P` or `N` | Approval and customer notification can be absent without violating the model. |
| No rule forces `C` to be the last step | A trace could continue after closing, even though closure should usually terminate the case. |
| `R` before `C` adds little beyond `R` being the first event | This constraint is mostly redundant and does not protect the missing middle of the process. |

### Net effect
The model is not necessarily unsatisfiable, but it is semantically misaligned: it favors fast closure paths while making evaluated cases unable to close.

---

## 2) Plausible hypotheses for why this happened

1. **Mixed process variants were merged into one model**  
   Low-risk or low-value claims may be auto-closed, while other claims go through manual evaluation. A single model over both variants can produce conflicting rules.

2. **Event labels are semantically inconsistent**  
   One activity code may mean different things in different systems or teams, leading to false non-coexistence or weak dependencies.

3. **Incomplete or biased event logging**  
   If evaluation or assignment events are missing from some systems, the mined constraints can incorrectly suggest that those steps do not coexist with closure.

4. **Policy changes were applied unevenly**  
   A newer “fast-track” closure policy may have been introduced, but older requirements around assignment/evaluation were not cleaned up.

5. **The model was mined from heterogeneous claim populations**  
   Different claim types, regions, or handling channels may follow different procedures, but the model was inferred globally.

6. **Operational pressure favored speed over control**  
   The process may have drifted toward closing claims quickly, with formal approval, notification, or evaluation becoming inconsistent in practice.

7. **Resource mapping is poor**  
   Assignment may exist operationally, but the event log may not clearly identify the assigned adjuster, making the `A`/`E` relationship look weaker than it really is.

## 3) SQL-based investigation strategies

Below are practical PostgreSQL queries to test whether these issues appear in the actual data.

---

### A. Quick case-level summary of the main anomalies

```sql
WITH case_flags AS (
    SELECT
        c.claim_id,
        c.claim_type,
        c.claim_amount,
        BOOL_OR(ce.activity = 'R') AS has_r,
        BOOL_OR(ce.activity = 'A') AS has_a,
        BOOL_OR(ce.activity = 'E') AS has_e,
        BOOL_OR(ce.activity = 'P') AS has_p,
        BOOL_OR(ce.activity = 'N') AS has_n,
        BOOL_OR(ce.activity = 'C') AS has_c
    FROM claims c
    LEFT JOIN claim_events ce
        ON ce.claim_id = c.claim_id
    GROUP BY c.claim_id, c.claim_type, c.claim_amount
)
SELECT
    COUNT(*) AS total_claims,
    COUNT(*) FILTER (WHERE has_c AND NOT has_e) AS closed_without_evaluation,
    COUNT(*) FILTER (WHERE has_c AND has_e) AS evaluation_and_close_coexist,
    COUNT(*) FILTER (WHERE has_e AND NOT has_a) AS evaluation_without_assignment,
    COUNT(*) FILTER (WHERE has_c AND NOT has_p) AS closed_without_approval,
    COUNT(*) FILTER (WHERE has_c AND NOT has_n) AS closed_without_notification
FROM case_flags;
```

**What it checks:**  
Overall prevalence of the most important suspicious patterns.

---

### B. Claims that were closed without any evaluation

```sql
SELECT
    c.claim_id,
    c.customer_id,
    c.claim_type,
    c.claim_amount,
    MIN(ce."timestamp") FILTER (WHERE ce.activity = 'C') AS close_ts
FROM claims c
JOIN claim_events ce
    ON ce.claim_id = c.claim_id
GROUP BY c.claim_id, c.customer_id, c.claim_type, c.claim_amount
HAVING COUNT(*) FILTER (WHERE ce.activity = 'C') > 0
   AND COUNT(*) FILTER (WHERE ce.activity = 'E') = 0
ORDER BY c.claim_id;
```

**What it checks:**  
Whether the data contains cases consistent with the weak/undesired closure path.

---

### C. Claims where evaluation and closure both occurred

```sql
SELECT
    ce.claim_id,
    COUNT(*) FILTER (WHERE ce.activity = 'E') AS eval_count,
    COUNT(*) FILTER (WHERE ce.activity = 'C') AS close_count,
    MIN(ce."timestamp") FILTER (WHERE ce.activity = 'E') AS first_eval_ts,
    MIN(ce."timestamp") FILTER (WHERE ce.activity = 'C') AS first_close_ts
FROM claim_events ce
GROUP BY ce.claim_id
HAVING COUNT(*) FILTER (WHERE ce.activity = 'E') > 0
   AND COUNT(*) FILTER (WHERE ce.activity = 'C') > 0
ORDER BY ce.claim_id;
```

**What it checks:**  
Whether real executions violate the non-coexistence rule between evaluation and closure.

---

### D. Evaluation events with no assignment at all

```sql
SELECT
    e.claim_id,
    e.event_id,
    e."timestamp" AS eval_ts,
    e.resource AS evaluator
FROM claim_events e
WHERE e.activity = 'E'
  AND NOT EXISTS (
      SELECT 1
      FROM claim_events a
      WHERE a.claim_id = e.claim_id
        AND a.activity = 'A'
  )
ORDER BY e.claim_id, e."timestamp", e.event_id;
```

**What it checks:**  
Whether evaluation can happen without any assignment event in the same case.

---

### E. Evaluation events not preceded by assignment

```sql
SELECT
    e.claim_id,
    e.event_id,
    e."timestamp" AS eval_ts,
    e.resource AS evaluator
FROM claim_events e
WHERE e.activity = 'E'
  AND NOT EXISTS (
      SELECT 1
      FROM claim_events a
      WHERE a.claim_id = e.claim_id
        AND a.activity = 'A'
        AND (
            a."timestamp" < e."timestamp"
            OR (a."timestamp" = e."timestamp" AND a.event_id < e.event_id)
        )
  )
ORDER BY e.claim_id, e."timestamp", e.event_id;
```

**What it checks:**  
Whether the intended ordering “assign before evaluate” is actually respected.

---

### F. Closed claims missing approval or notification

```sql
SELECT
    ce.claim_id,
    COUNT(*) FILTER (WHERE ce.activity = 'P') AS approve_count,
    COUNT(*) FILTER (WHERE ce.activity = 'N') AS notify_count,
    MIN(ce."timestamp") FILTER (WHERE ce.activity = 'C') AS close_ts
FROM claim_events ce
GROUP BY ce.claim_id
HAVING COUNT(*) FILTER (WHERE ce.activity = 'C') > 0
   AND (
       COUNT(*) FILTER (WHERE ce.activity = 'P') = 0
       OR COUNT(*) FILTER (WHERE ce.activity = 'N') = 0
   )
ORDER BY ce.claim_id;
```

**What it checks:**  
Whether approval or notification are being skipped before closure.

---

### G. Cases that do not start with receipt

```sql
WITH first_event AS (
    SELECT DISTINCT ON (claim_id)
        claim_id,
        event_id,
        activity,
        "timestamp"
    FROM claim_events
    ORDER BY claim_id, "timestamp", event_id
)
SELECT *
FROM first_event
WHERE activity <> 'R'
ORDER BY claim_id;
```

**What it checks:**  
Whether the claimed initialization rule is respected in real traces.

---

### H. Close events without a prior receipt

```sql
SELECT
    c.claim_id,
    c.event_id AS close_event_id,
    c."timestamp" AS close_ts
FROM claim_events c
WHERE c.activity = 'C'
  AND NOT EXISTS (
      SELECT 1
      FROM claim_events r
      WHERE r.claim_id = c.claim_id
        AND r.activity = 'R'
        AND (
            r."timestamp" < c."timestamp"
            OR (r."timestamp" = c."timestamp" AND r.event_id < c.event_id)
        )
  )
ORDER BY c.claim_id, c."timestamp", c.event_id;
```

**What it checks:**  
Whether closure ever happens without the required earlier receipt event.

---

### I. Cases with activity after closure

```sql
WITH last_close AS (
    SELECT DISTINCT ON (claim_id)
        claim_id,
        event_id AS close_event_id,
        "timestamp" AS close_ts
    FROM claim_events
    WHERE activity = 'C'
    ORDER BY claim_id, "timestamp" DESC, event_id DESC
)
SELECT
    ce.claim_id,
    ce.event_id,
    ce.activity,
    ce."timestamp",
    ce.resource
FROM claim_events ce
JOIN last_close lc
    ON lc.claim_id = ce.claim_id
WHERE ce."timestamp" > lc.close_ts
   OR (ce."timestamp" = lc.close_ts AND ce.event_id > lc.close_event_id)
ORDER BY ce.claim_id, ce."timestamp", ce.event_id;
```

**What it checks:**  
Whether claims continue to evolve after they have been closed.

---

### J. Frequent trace variants

```sql
WITH traces AS (
    SELECT
        claim_id,
        STRING_AGG(activity, '->' ORDER BY "timestamp", event_id) AS trace
    FROM claim_events
    GROUP BY claim_id
)
SELECT
    trace,
    COUNT(*) AS case_count
FROM traces
GROUP BY trace
ORDER BY case_count DESC, trace;
```

**What it checks:**  
Whether simplified paths like `R->C` or `R->A->C` are common in practice.

---

### K. Check whether evaluation is done by a known adjuster with matching specialization

```sql
SELECT
    e.claim_id,
    e.event_id,
    e."timestamp" AS eval_ts,
    e.resource AS evaluator,
    ad.adjuster_id,
    ad.specialization,
    c.claim_type
FROM claim_events e
JOIN claims c
    ON c.claim_id = e.claim_id
LEFT JOIN adjusters ad
    ON ad.name = e.resource
WHERE e.activity = 'E'
  AND (
      ad.adjuster_id IS NULL
      OR ad.specialization IS DISTINCT FROM split_part(c.claim_type, '_', 1)
  )
ORDER BY e.claim_id, e."timestamp", e.event_id;
```

**What it checks:**  
Whether evaluation is performed by a recognized adjuster and whether the adjuster’s specialization aligns with the claim type.

> If `resource` does not store the adjuster’s name directly, replace the join logic with the correct identifier mapping.

---

### L. Test whether the suspicious behavior is concentrated in certain claim types or amounts

```sql
WITH flags AS (
    SELECT
        c.claim_id,
        c.claim_type,
        c.claim_amount,
        BOOL_OR(ce.activity = 'E') AS has_e,
        BOOL_OR(ce.activity = 'C') AS has_c
    FROM claims c
    LEFT JOIN claim_events ce
        ON ce.claim_id = c.claim_id
    GROUP BY c.claim_id, c.claim_type, c.claim_amount
)
SELECT
    claim_type,
    CASE
        WHEN claim_amount < 1000 THEN '<1k'
        WHEN claim_amount < 5000 THEN '1k-5k'
        WHEN claim_amount < 20000 THEN '5k-20k'
        ELSE '20k+'
    END AS amount_band,
    COUNT(*) AS cases,
    COUNT(*) FILTER (WHERE has_c AND NOT has_e) AS closed_without_eval,
    ROUND(
        100.0 * COUNT(*) FILTER (WHERE has_c AND NOT has_e) / NULLIF(COUNT(*), 0),
        1
    ) AS pct_closed_without_eval
FROM flags
GROUP BY claim_type, amount_band
ORDER BY claim_type, amount_band;
```

**What it checks:**  
Whether the anomaly is really a hidden process variant, such as fast-track treatment for small claims.

## Suggested investigation sequence

1. Run the **summary query** first.
2. Quantify **closed-without-evaluation** and **evaluation-with-close** cases.
3. Check **assignment vs evaluation ordering**.
4. Inspect whether **approval/notification are often skipped**.
5. Use **trace variants** to see the dominant real paths.
6. Segment by **claim type / amount** to see if different variants are being conflated.
7. Use the **adjusters join** to detect resource-quality or specialization issues.
