# Analysis of Anomalies in the POWL Insurance Claim Handling Model

## 1. Identifying the Anomalies

Based on the POWL model definition, several structural anomalies deviate from the ideal process flow (R  A  E  P  N  C):

### Anomaly A: Loop Between Evaluation and Approval
```python
loop = OperatorPOWL(operator=Operator.LOOP, children=[E, P])
```
This creates a cycle where a claim can be **evaluated  approved  evaluated  approved** repeatedly. In a well-designed process, approval should be a terminal step after evaluation. This loop suggests claims can be repeatedly approved, which is logically inconsistent (you cannot "un-approve" and re-approve without an explicit rejection/re-submission step).

### Anomaly B: Optional Customer Notification (XOR with Skip)
```python
xor = OperatorPOWL(operator=Operator.XOR, children=[N, skip])
```
Customer notification is modeled as optional. In regulated insurance contexts, notifying customers of claim outcomes is typically a **mandatory compliance step**, not discretionary.

### Anomaly C: Partial Order Allows Premature Closure
```python
root.order.add_edge(A, C)   # Direct edge from A to C
# Missing: root.order.add_edge(xor, C)
```
The partial order connects `A  C` directly, and **does not enforce** `xor  C`. This means once the adjuster is assigned, the claim could theoretically be closed **without evaluation, approval, or customer notification**. This is a severe control failure.

---

## 2. Hypotheses for Why These Anomalies Exist

| # | Hypothesis | Rationale |
|---|-----------|-----------|
| H1 | **Business rule change partially implemented** | A new rule (e.g., "allow revisions for disputed claims") introduced the E鞛 loop but no exit conditions were formalized. |
| H2 | **Miscommunication between operations and compliance** | The notification-skip option may reflect an operational shortcut that was never reconciled with regulatory requirements. |
| H3 | **Technical workflow error / modeler misuse** | The extra `A  C` edge may be an accidental leftover from copy-paste editing or a debugging bypass that was never removed. |
| H4 | **Insufficient tool constraints** | The POWL modeler does not validate whether all terminal activities are constrained by upstream dependencies, allowing dangling partial orders. |
| H5 | **Fast-track closure policy for trivial claims** | Some low-value claims may be closed without full evaluation, and the model was relaxed rather than creating a separate sub-process. |

## 3. Database Verification Queries

### Query 1: Claims closed without evaluation or approval (tests Anomaly C, H3, H5)
```sql
SELECT c.claim_id, c.claim_type, c.claim_amount
FROM claims c
JOIN claim_events ce_close 
     ON c.claim_id = ce_close.claim_id AND ce_close.activity = 'C'
LEFT JOIN claim_events ce_eval 
     ON c.claim_id = ce_eval.claim_id AND ce_eval.activity = 'E'
LEFT JOIN claim_events ce_appr 
     ON c.claim_id = ce_appr.claim_id AND ce_appr.activity = 'P'
WHERE ce_eval.event_id IS NULL 
   OR ce_appr.event_id IS NULL;
```
**Purpose:** Find claims where `C` occurred without a preceding `E` or `P`.

---

### Query 2: Claims where 'C' occurred before 'E' or 'P' chronologically (tests Anomaly C)
```sql
SELECT c.claim_id,
       MIN(CASE WHEN ce.activity = 'C' THEN ce.timestamp END) AS close_time,
       MIN(CASE WHEN ce.activity = 'E' THEN ce.timestamp END) AS eval_time,
       MIN(CASE WHEN ce.activity = 'P' THEN ce.timestamp END) AS approve_time
FROM claims c
JOIN claim_events ce ON c.claim_id = ce.claim_id
GROUP BY c.claim_id
HAVING MIN(CASE WHEN ce.activity = 'C' THEN ce.timestamp END)
       < MIN(CASE WHEN ce.activity IN ('E','P') THEN ce.timestamp END);
```

---

### Query 3: Claims approved multiple times (tests Anomaly A, H1)
```sql
SELECT claim_id, COUNT(*) AS approval_count
FROM claim_events
WHERE activity = 'P'
GROUP BY claim_id
HAVING COUNT(*) > 1
ORDER BY approval_count DESC;
```
**Purpose:** Detect abuse of the E鞛 loop. Multiple approvals on the same claim indicate the anomaly is occurring in practice.

---

### Query 4: Evaluate/Approve cycles (sequence analysis)
```sql
WITH ordered AS (
  SELECT claim_id, activity, timestamp,
         LAG(activity) OVER (PARTITION BY claim_id ORDER BY timestamp) AS prev_activity
  FROM claim_events
  WHERE activity IN ('E','P')
)
SELECT claim_id, COUNT(*) AS loop_iterations
FROM ordered
WHERE prev_activity = 'P' AND activity = 'E'   -- re-evaluation after approval
GROUP BY claim_id
ORDER BY loop_iterations DESC;
```

---

### Query 5: Frequency of skipped customer notification (tests Anomaly B, H2)
```sql
SELECT 
  COUNT(DISTINCT c.claim_id) AS total_closed_claims,
  COUNT(DISTINCT CASE WHEN n.event_id IS NULL THEN c.claim_id END) AS without_notification,
  ROUND(100.0 * COUNT(DISTINCT CASE WHEN n.event_id IS NULL THEN c.claim_id END)
        / COUNT(DISTINCT c.claim_id), 2) AS pct_skipped
FROM claims c
JOIN claim_events ce_c ON c.claim_id = ce_c.claim_id AND ce_c.activity = 'C'
LEFT JOIN claim_events n ON c.claim_id = n.claim_id AND n.activity = 'N';
```

---

### Query 6: Skipped notification broken down by claim type / amount (tests H5)
```sql
SELECT c.claim_type,
       AVG(c.claim_amount) AS avg_amount,
       COUNT(*) FILTER (WHERE n.event_id IS NULL) AS skipped_notifications,
       COUNT(*) AS total_claims
FROM claims c
LEFT JOIN claim_events n ON c.claim_id = n.claim_id AND n.activity = 'N'
GROUP BY c.claim_type;
```
**Purpose:** If low-value/specific-type claims disproportionately skip `N`, this supports a "fast-track" hypothesis (H5).

---

### Query 7: Identify resources/adjusters involved in anomalous closures (tests H3)
```sql
SELECT ce.resource, a.name, a.region, COUNT(*) AS premature_closures
FROM claim_events ce
LEFT JOIN adjusters a ON a.name = ce.resource
WHERE ce.activity = 'C'
  AND ce.claim_id IN (
      SELECT claim_id FROM claim_events 
      GROUP BY claim_id
      HAVING BOOL_OR(activity='C') 
         AND NOT BOOL_OR(activity='E')
  )
GROUP BY ce.resource, a.name, a.region
ORDER BY premature_closures DESC;
```
**Purpose:** If anomalies cluster around a few resources/regions, this points toward local misuse rather than systemic modeler error.

## Summary

| Anomaly | Verification Query | What It Reveals |
|---------|-------------------|----------------|
| Loop EP | Q3, Q4 | Whether approvals repeat in practice |
| Optional N | Q5, Q6 | Whether notification is frequently skipped and under what conditions |
| Premature C | Q1, Q2, Q7 | Whether claims really close before evaluation/approval |

Combining these queries with the hypotheses lets the analyst distinguish between **model-only anomalies** (structural issues never exploited) and **real process deviations** that require intervention in policy, system controls, or training.
