### **1. Process Discovery and Conformance Checking**

**A. Data Preprocessing & Integration Strategy**

The first critical step is to create a single, coherent **event log** from the four data sources. The primary challenge is **entity resolution and event correlation**.

*   **Define the Case ID:** The most logical primary case is the **`Vehicle-ID + Date`** (e.g., `V12-20241205`). This represents a complete operational day for a vehicle. However, for package-level analysis, we can create secondary cases using `Package-ID`.
*   **Event Correlation & Timestamp Alignment:**
    *   **GPS & Vehicle Data:** These provide continuous streams. We need to **discretize** them into meaningful events (e.g., "Depart Depot," "Arrive Customer," "Low Speed Detected"). This can be done by defining rules: a stop > X minutes at a location with < Y km/h speed becomes an "Arrive" event; a sustained speed > 0 after a stop becomes a "Depart" event.
    *   **Scanner Data:** These are discrete, high-confidence milestone events. They are the **ground truth** for delivery activities ("Delivery Success," "Arrive Customer"). We must map these to the nearest corresponding GPS location and time.
    *   **Dispatch Data:** Provides the **planned process model**—the intended sequence of stops (`Stop 1 -> Stop 2 -> ... -> Stop N`) and their planned time windows.
    *   **Maintenance Logs:** These are exceptional events. We need to link them to the relevant `Vehicle-ID + Date` case, marking periods of inactivity or deviation.
*   **Integration Process:** Use a **data fusion layer**:
    1.  For each `Vehicle-ID + Date`, create a timeline.
    2.  Overlay GPS tracks to identify travel segments and stops.
    3.  **Anchor** this timeline with scanner events (they have precise timestamps and locations).
    4.  Use scanner events to label GPS stops: if a GPS stop at location L matches a scanner "Arrive Customer" event for package P, label that stop as `Service: [Customer Address]`.
    5.  Inject planned route data from dispatch as a reference sequence.
*   **Key Challenges:**
    *   **Data Granularity Mismatch:** GPS data is continuous; scanner data is event-based. Resolution requires intelligent interpolation and rule-setting.
    *   **Location Ambiguity:** GPS coordinates may not exactly match customer addresses. A tolerance radius (e.g., 50-100m) must be defined for "Arrive" matching.
    *   **Missing Data:** Scanner failures, GPS dropouts. Requires imputation strategies or case exclusion.
    *   **Event Naming Standardization:** Ensure "Depart Depot" from GPS rules means the same as "Depart Depot" from driver log.

**B. Process Discovery & Visualization**

Using the integrated event log, I would apply **process discovery algorithms** (like the **Heuristic Miner** or **Fuzzy Miner**) to visualize the *actual* as-is process.

*   **Expected Discovered Model Structure:**
    *   **Start:** `Start Shift` (Driver) -> `Route Assigned` (Dispatch) -> `Depart Depot` (Vehicle).
    *   **Main Body:** A **parallel or interleaved subnet** of:
        *   `Travel To Customer` (Vehicle moving)
        *   `Arrive Customer` (Scanner) -> `Service Time` (dwell) -> `Depart Customer` (Scanner)
        *   **AND** potentially: `Low Speed Detected` (Traffic), `Unscheduled Stop` (Breakdown/Fuel), `Delivery Failed` (Scanner).
    *   **End:** `Arrive Depot` -> `End Shift`.
*   **Visualization:** The model will show **loops** (e.g., re-visiting a failed delivery location), **parallel paths** (multiple stops in a row), and **splits/merges** indicating choice points (e.g., which customer to serve next when behind schedule).

**C. Conformance Checking Against Planned Routes**

This is where we quantify deviation from the ideal.

*   **Methodology:**
    1.  **Sequence Conformance:** For each vehicle-day case, compare the **sequence of visited locations** (derived from GPS/scanner) against the **planned sequence** from dispatch. Calculate a **sequence alignment score** (e.g., using the `conformance` plugin in ProM). High deviation indicates route flexibility or poor planning.
    2.  **Timing Conformance:** For each planned stop, compare the **actual arrival time** (from scanner/GPS) against its **planned time window**. Calculate **slack time** (early) or **delay** (late).
    3.  **Activity Conformance:** Check for **unplanned activities** (e.g., `Unscheduled Stop`, `Low Speed Detected` for > 10 mins) that are not in the planned model.
*   **Key Deviations to Hunt For:**
    *   **Sequence Deviations:** Skipping stops, re-ordering stops, adding new stops (e.g., for a failed re-delivery).
    *   **Timing Deviations:** Consistent delays at specific stops or times of day.
    *   **Unplanned Stops:** Frequency, duration, and location of stops not in the plan (e.g., traffic jams, vehicle issues, driver breaks).
    *   **Dwell Time Deviations:** Service time at a customer significantly longer than planned (indicating parking difficulty, complex delivery, customer interaction).

### **2. Performance Analysis and Bottleneck Identification**

**A. Key Performance Indicators (KPIs) & Calculation**

| KPI Category | Specific KPI | How to Calculate from Event Log |
| :--- | :--- | :--- |
| **Punctuality** | **On-Time Delivery Rate** | `% of deliveries where actual arrival time  planned time window end`. |
| | **Average Delay per Delivery** | `Mean(|actual_arrival - planned_window_midpoint|) for late deliveries`. |
| **Efficiency** | **Average Time per Delivery Stop** | `Total service time (dwell) / Total number of successful deliveries`. |
| | **Travel Time vs. Service Time Ratio** | `(Total travel time) / (Total service time)`. High ratio = too much driving, not enough delivering. |
| | **Fuel Consumption per km** | `Total fuel used (from maintenance/vehicle data) / Total distance traveled (from GPS)`. |
| | **Vehicle Utilization Rate** | `(Total active driving/service time) / (Total shift duration - scheduled breaks)`. |
| **Reliability** | **Failed Delivery Rate** | `Number of 'Delivery Failed' scans / Total delivery attempts`. |
| | **Frequency of Traffic Delays** | `Count of 'Low Speed Detected' events per km or per hour, mapped to known congestion zones`. |
| | **Unscheduled Stop Rate** | `Number of 'Unscheduled Stop' events (e.g., breakdown, warning light) per shift/day`. |

**B. Bottleneck Identification Techniques**

*   **Bottleneck Analysis (Process Level):** Use **process enhancement techniques** (like the **Dotted Chart** in ProM) to visualize the flow of multiple cases (vehicles) over time. Look for:
    *   **Clusters of Delays:** Do many vehicles get stuck in the same geographic area at the same time? (Traffic hotspot).
    *   **Resource Contention:** Are multiple vehicles arriving at the depot simultaneously, causing queue for parking/fueling?
*   **Variant Analysis:** Group vehicle-days into **performance-based variants**.
    *   **High-Performers:** High on-time rate, low fuel/km, short service times.
    *   **Low-Performers:** Opposite.
    *   **Compare:** What differs? Route sequence? Time of day? Driver? Vehicle type? This isolates the impact of controllable factors.
*   **Correlation Analysis:**
    *   **Traffic vs. Delay:** Correlate `Low Speed Detected` events with external traffic data (if available) or with known school/business rush hours.
    *   **Service Time vs. Customer Type:** Is service time longer at residential addresses vs. businesses? At addresses with stairs/garage?
    *   **Vehicle Age vs. Maintenance Stops:** Does older fleet have more `Unscheduled Stop` events?

### **3. Root Cause Analysis for Inefficiencies**

Process mining moves from *what* happened to *why*.

*   **Potential Root Causes & Validation Analyses:**
    1.  **Suboptimal Route Planning (Static):**
        *   *Evidence:* High sequence conformance deviation. Variant analysis shows low-performers have routes with more left turns, steep hills, or pass through known congestion zones.
        *   *Validation:* Simulate the *same* package set with a simple nearest-neighbor algorithm. Does it outperform the planned route?
    2.  **Inaccurate Travel Time Estimation:**
        *   *Evidence:* Systematic delays increasing with distance from depot, or at specific times (e.g., always 15 mins late on afternoon routes).
        *   *Validation:* Compare planned vs. actual travel time per segment. Calculate a **planning factor** (actual/planned). Is it >1.3 consistently for certain corridors?
    3.  **High Service Time Variability:**
        *   *Evidence:* Large standard deviation in `Average Time per Delivery Stop` across customers or drivers.
        *   *Validation:* Drill into `Delivery Failed` events. Is the re-delivery attempt faster? This suggests the initial service time was inflated by customer unavailability.
    4.  **Vehicle Breakdowns & Maintenance:**
        *   *Evidence:* `Unscheduled Stop` events correlate with specific vehicle IDs or high mileage days. These events cause cascading delays for that vehicle's entire route.
        *   *Validation:* Overlay maintenance logs. Do vehicles with recent minor repairs have more `Low Speed Detected` events?
    5.  **Driver Behavior Differences:**
        *   *Evidence:* Variant analysis by `Driver-ID`. Some drivers consistently have higher `Travel Time vs. Service Time Ratio` (more driving) or more `Low Speed Detected` events (aggressive acceleration/braking).
        *   *Validation:* Correlate with GPS speed patterns (harsh events) and fuel consumption data.
    6.  **Failed Delivery Impact:**
        *   *Evidence:* A `Delivery Failed` event creates a loop in the process model. The re-delivery attempt adds a full service cycle to the day.
        *   *Validation:* Calculate the **cost of a failed delivery**: `(Service time for 1st attempt + Service time for 2nd attempt + extra travel) / (packages per hour rate)**.
