**1. Process Discovery and Conformance Checking**  

**Data preprocessing & integration**  
- **Event extraction:** From each source create a homogeneous event record (timestamp, activity, case identifier, resource, and relevant data attributes).  
  *GPS:* Convert raw pings into higherlevel activities (e.g., *Depart Depot*, *Travel Segment*, *Arrive Customer Location*, *Low Speed Detected*, *Unscheduled Stop*, *Arrive Depot*). Use speed thresholds and geofencing around depot/customer addresses to label travel vs. service events.  
  *Scanner:* Directly yields milestone events (*Scan at Depot*, *Arrive Customer*, *Delivery Success/Failed*, *Depart Customer*). Attach the Package ID and outcome (success/fail).  
  *Dispatch:* Provides the planned route (ordered list of stops) and the intended time window per stop. Create a *Planned Route* artifact that can be joined to each vehicleday case.  
  *Maintenance:* Generates events (*Maintenance Start*, *Maintenance End*, *Unscheduled Repair*) with vehicle ID and downtime timestamps.  

- **Case definition:** Choose a case granularity that captures a full operational cycle – **VehicleDay** (e.g., V1220241205). All events whose timestamps fall between the first *Start Shift* and the last *End Shift* for that vehicle belong to the same case. This enables endtoend process discovery while preserving the link to planned routes and maintenance interventions.  

- **Data integration challenges**  
  *Temporal alignment:* GPS pings may be sampled every 510s, while scanner events are irregular. Need to aggregate GPS into meaningful activities and align timestamps to the nearest second.  
  *Spatial matching:* Geocode scannerrecorded addresses and match them to GPS traces to confirm arrival/departure; mismatches (e.g., GPS drift) must be filtered or corrected via mapmatching algorithms.  
  *Missing data:* Periods where GPS signal is lost (tunnels, dense urban canyons) create gaps; impute using last known speed or interpolate, but flag uncertain segments.  
  *Schema heterogeneity:* Different systems use different IDs (vehicle, driver, package). Build a crossreference table from dispatch logs to unify identifiers.  

- **Process discovery:** Apply an inductive miner (e.g., Inductive Miner infrequent) or a heuristic miner on the integrated event log to obtain a **process model** that shows the *asis* flow:  
  - Start Shift  (optional Depot Scan)  Travel Segment  Arrive Customer Location  (Service Time: Driver interaction, possible Parking Search)  Delivery Success/Failed  Depart Customer  (next stop)  …  Return to Depot  End Shift.  
  - Parallel branches for *Unscheduled Stop* (maintenance/warning) and *Low Speed Detected* (traffic jam) can appear as loops or deviations off the main travel path.  

- **Conformance checking:**  
  - Use the planned route from the dispatch system as a reference model (a strict sequence of planned stops with associated time windows).  
  - Align each trace (vehicleday) against this reference using tokenbased replay or alignment techniques (e.g., A* alignment).  
  - **Deviations to flag:**  
    *Sequence deviations:* Visiting stops outoforder, skipping a planned stop, or inserting extra stops (e.g., unscheduled maintenance, meals).  
    *Timing deviations:* Actual arrival time outside the promised window (early/late) – quantified as *window violation minutes*.  
    *Servicetime excess:* Dwell time at a stop significantly longer than the estimated service time (e.g., >2× expected).  
    *Unplanned travel:* Detours not explained by traffic (e.g., route deviation >500m from the planned polyline).  
    *Maintenance interference:* Occurrence of an *Unscheduled Stop* that splits a planned travel segment.  

These conformance metrics give a quantitative baseline of how often and why the real process diverges from the plan.


**2. Performance Analysis and Bottleneck Identification**  

**Relevant KPIs (derived from the event log)**  
| KPI | Calculation from log | Business relevance |
|-----|----------------------|--------------------|
| **OnTime Delivery Rate (OTDR)** | % of *Delivery Success* events where actual arrival time  end of customerrequested window (or within a agreed tolerance). | Direct measure of punctuality. |
| **Average Time per Stop** | Mean of (Depart Customer timestamp – Arrive Customer timestamp) across all successful stops. | Reflects service efficiency (parking, customer interaction). |
| **TravelTime vs. ServiceTime Ratio** | (Travel Segment durations) ÷ (Servicetime durations) per vehicleday. | High ratio  too much deadheading; low ratio  excessive service time. |
| **Fuel Consumption per km / per package** | If fuelsensor data unavailable, approximate using speedprofile models: fuel  (speed˛·dt) or use OBDderived metrics; then divide by total km or by number of delivered packages. | Links driving behavior to cost. |
| **Vehicle Utilization Rate** | (Total time vehicle is moving + serving) ÷ (Shift length – scheduled maintenance downtime). | Indicates how much of the driver’s shift is productive. |
| **Frequency/Duration of Traffic Delays** | Count and total duration of *Low Speed Detected* events where speed < threshold (e.g., 10km/h) and duration >30s. | Highlights congestion impact. |
| **FailedDelivery Rate** | % of stops where event = *Delivery Failed* (customer not home, access issue, etc.). | Drives redelivery costs and OTD. |
| **MaintenanceInduced Downtime** | (Unscheduled Stop duration) per vehicleday. | Connects reliability to cost. |

**Techniques for bottleneck detection**  
- **Performanceannotated process model:** overlay average durations and frequency on the discovered model (e.g., using *performance* or *dotted* chart in ProM). Hotspots appear as activities with high mean time or large variance.  
- **Sliceanddice analysis:** filter the log by timeofday buckets (e.g., 0709, 0912, 1215, 1518) or by driver/vehicle type and recompute KPIs to see where bottlenecks shift.  
- **Trace variant analysis:** group cases by variant (exact sequence of activities). Identify variants with high average total case time or low OTDR; compare them to the most frequent (presumably efficient) variant.  
- **Resourcecentric analysis:** compute KPIs per driver and per vehicle; outliers reveal skill or equipment issues.  
- **Spatial bottleneck mapping:** geocode the start/end of each *Travel Segment* and kerneldensity the accumulated delay time to visualize traffic hotspots.  

The output of these analyses pinpoints whether delays stem from (a) congested road segments, (b) excessive service time at certain address types (e.g., highrise apartments without secure access), (c) frequent unscheduled stops for specific vehicles, or (d) particular drivers whose stoptime variance is high.


**3. Root Cause Analysis for Inefficiencies**  

Having located *where* and *when* problems occur, we dig into *why* using complementary processmining lenses:

| Suspected root cause | Processmining technique to validate |
|----------------------|--------------------------------------|
| **Suboptimal route planning (static vs. dynamic)** | Compare planned route sequence (from dispatch) with the *actual* sequence of visited stops per variant. Compute extra distance/time incurred by deviations. If highperforming variants show systematic reordering (e.g., clustering by geography) that the static plan misses, this suggests a need for dynamic resequencing. |
| **Inaccurate traveltime estimations** | For each travel segment, compare the planned ETA (based on dispatch schedule + estimated segment time) with the actual timestamp. Calculate mean prediction error and its correlation with timeofday or trafficdelay events. Systematic underestimation indicates bad traveltime models. |
| **Impact of traffic congestion patterns** | Overlay *Low Speed Detected* events on a map; run a hotspot analysis (e.g., GetisOrd Gi*). Then compute the proportion of total delay time that occurs within these hotspots per vehicleday. A high proportion confirms congestion as a primary driver of lateness. |
| **High variability in service time** | For each stop type (e.g., singlefamily home, apartment building, business), compute the distribution of servicetime durations. Large variance or outliers point to addressspecific factors (parking availability, building access, customer readiness). |
| **Frequency/impact of vehicle breakdowns** | Link each *Unscheduled Stop* to subsequent KPI degradation (e.g., increase in traveltime ratio, missed windows in the following stops). Survivalanalysis style: compute hazard of a missed window given a prior unscheduled stop. |
| **Driver behavior or skill differences** | Compute perdriver metrics: average service time, fuel per km, number of failed deliveries, and variance in departurefromdepot timing. Use statistical tests (ANOVA) to see if certain drivers significantly deviate. |
| **Faileddelivery attempts causing redelivery** | Trace the lifecycle of a package: if a *Delivery Failed* event is followed later (same day or next day) by another *Arrive Customer*  *Delivery Success* for the same Package ID, count it as a redelivery. Measure extra distance/fuel and time incurred. Correlate with specific customer attributes (e.g., no safe drop, inaccurate contact info). |

By running these analyses, we can prioritize root causes (e.g., “30% of late arrivals stem from traffic hotspots on Main St between 08:0009:00; 20% from servicetime variance at apartment complexes; 15% from driverspecific parkingsearch inefficiency”).

**5. Considering Operational Constraints & Monitoring**  

**Accounting for constraints**  
- **Driver working hours:** Any resequencing or dynamic routing must respect maximum shift length and legally mandated breaks. The optimization engine treats the shift end time as a hard constraint; if reordering would cause overtime, it falls back to the closest feasible sequence.  
- **Vehicle capacity:** When clustering stops (e.g., for territory redesign), the algorithm checks cumulative package volume/weight against the van’s load limit before accepting a new stop sequence.  
- **Customer time windows:** Dynamic adjustments are allowed only if the resulting arrival time stays within the agreed window (or a prenegotiated tolerance buffer). If a window cannot be met without violating another constraint, the system flags the stop for proactive customer contact (e.g., offer alternative window).  

**Continuous monitoring plan**  
After deployment, run a **weekly processmining dashboard** that refreshes from the incoming event log. Key views:  

1. **Conformance Heatmap** – shows % of traces conforming to the planned route per day; trend over time to spot regressions.  
2. **KPI TimeSeries** – OTDR, Avg. Stop Time, Fuel/km, Unscheduled Stop Frequency displayed with control limits (e.g., +/- 2).  
3. **Variant Distribution** – Sankey diagram of the top 5 route variants; highlight any new variant appearing >5% of traffic (could indicate emerging detour or new traffic pattern).  
4. **Hotspot Map** – animated overlay of LowSpeed Detected events; allows operators to see if previously treated congestion zones have improved.  
5. **Driver Performance Scorecard** – radial chart per driver (fuel, service time, failed deliveries) to sustain coaching effects.  

Alerts are configured to fire when:  
- OTDR drops >2pp below target for two consecutive days.  
- Unscheduled stop frequency rises >20% weekoverweek.  
- A new variant with significantly higher total case time emerges (>15% slower than the baseline).  

By continuously feeding the dashboard with fresh event data, the company can verify that the optimization strategies are delivering the expected gains, quickly detect any degradation (e.g., due to road works, new construction, or seasonal demand shifts), and iterate the models (retrain routing engine, update servicetime standards, adjust maintenance thresholds) in a closedloop, datadriven fashion.
