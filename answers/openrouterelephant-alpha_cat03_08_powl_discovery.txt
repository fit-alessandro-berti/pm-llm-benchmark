**Goal:** Build a single POWL model that captures the patterns observed across all five cases in the event log, using:

- **Transitions** for visible activities (e.g., `OR`, `MS`, `QS`, etc.)
- **Silent transitions** (`tau`) where behavior is optional or skipped (e.g., `LB` may be missing)
- **Loops** (`*(A, B)`) for repeated patterns like `IQC  AS` and `TST  RT`
- **Choices** (`X(A, B)`) for mutually exclusive optional paths (e.g., `LB` vs. no `LB`, or simplified path in Case 4)
- **StrictPartialOrder** to enforce causal dependencies and allow concurrency where appropriate

---

##  Observed Patterns

### 1. **Mandatory Core Sequence (with internal variation)**
All cases follow a base sequence:
```
OR  MS  QS  AS  TST  PK  D
```
But with **variations**:
- **IQC** may occur **multiple times** between `AS` and `TST`
- **RT** may follow failed `TST`, forcing a return to `TST`
- **LB** is **optional** (missing in Case 2)
- **Case 4** skips `IQC`, `RT`, and `LB` entirely — a **simplified path**

### 2. **Repetition / Loops**
- **IQC  AS** repeats 1–3 times  suggests a **loop**: perform `AS`, then optionally redo `IQC`
  - But note: `IQC` may occur **before** first `AS`? No — first `AS` always precedes first `IQC` in all cases.
  - Actually: `AS  IQC  AS  (possibly IQC again)  AS  TST`
  - So correct loop: **AS  *(IQC, AS)**  TST**
  - Wait: After first `AS`, we see `IQC`, then `AS` again — so the loop body is `IQC  AS`, and it may repeat.

Thus:  
> After initial `AS`, we may repeat `IQC` then `AS` any number of times (0), then proceed to `TST`.

This is a **loop** where:
- Entry: `AS`
- Loop body: `IQC  AS` (repeated)
- Exit to `TST`

Modeled as:  
`*(AS, *(IQC, AS))`? Not quite — better:  
The pattern is: `AS` (initial), then optionally `(IQC  AS)*`, then `TST`.

So:  
`AS  *(IQC, AS)`  `TST`

But POWL * loop is defined as `* (A, B)` = execute A, then either exit or do B then A again.

So to model `(IQC  AS)*`, we can use:  
`* (IQC, AS)` — meaning: do `IQC`, then optionally `AS`, then repeat.

But we need the **first** `AS` **before** the loop.

So overall:  
`AS  *(IQC, AS)`  `TST`

Yes: first `AS`, then loop of `IQC  AS`, then exit to `TST`.

- **TST  RT  TST** when test fails  **loop**: `* (TST, RT)`
  - But note: `TST` always occurs at least once.
  - Then optionally `RT`, then back to `TST`, repeat.
  - So valid: `*(TST, RT)` — do `TST`, then optionally `RT`, then repeat.

But caution: In log, we see sequences like:
- `TST  RT  TST  RT  TST` (Case 5)
- `TST  RT  TST` (Case 1, 3)
- `TST` only (Case 2, 4)

So yes: `*(TST, RT)` captures this.

### 3. **Optional Activity: Labeling (LB)**
- Appears in Cases 1, 3, 5
- Missing in Case 2
- Always **after PK**, **before D**
- No variation in position  always optional but in fixed location

So: after `PK`, either `LB` or **skip** (silent), then `D`

 `X(PK  LB, PK  Silent)`? But `PK` must precede `D`.

Better: after `PK`, choose between `LB` then `D`, or directly `D`.

So: `X(LB, skip)` with both followed by `D`, but `PK` must come before.

Thus: `PK  *(X(LB, skip), D)`? No — `D` must follow.

Actually:  
`PK  X(LB, Silent)  D`

But `X` is a choice **between two models**, not a transition.

So define:
- Child1: `LB` then `D`
- Child2: `D` directly

But `D` is common — so better to use:
`PK  (LB  D) X ( D)`? Not directly supported.

Alternative: Use **silent transition** to skip `LB`:

After `PK`, do `X(LB, Silent)`, then `D`.

Yes — because `X` can have children that are **activities or operators**, including silent transitions.

So:
- `X(LB, Silent)` — either do `LB` or nothing (tau)
- Then `D`

This works.

### 4. **Optional Path in Case 4**
Case 4 skips:
- `IQC`, `RT`, `LB`

So after `QS  AS  TST  PK  (skip LB)  D`

This is already handled by:
- No loop for `IQC/AS` (loop may execute 0 times)
- Optional `LB` via `X(LB, Silent)`

So the same structure supports Case 4 if loops and choices are optional.

##  Final POWL Structure

We define the model as a **StrictPartialOrder** over a set of nodes, where nodes can be:
- Transitions (activities)
- SilentTransition
- OperatorPOWL (for XOR and LOOP)

We build the model step by step.

### Step 1: Define atomic transitions
```python
OR   = Transition(label="OR")
MS   = Transition(label="MS")
QS   = Transition(label="QS")
AS   = Transition(label="AS")
IQC  = Transition(label="IQC")
TST  = Transition(label="TST")
RT   = Transition(label="RT")
PK   = Transition(label="PK")
LB   = Transition(label="LB")
D    = Transition(label="D")
skip = SilentTransition()
```

### Step 2: Build inner loops and choices

#### Loop 1: IQC and AS (optional repetition after first AS)
> After initial `AS`, we may do `(IQC  AS)` zero or more times.

So define:
```python
loop_iqc_as = OperatorPOWL(operator=Operator.LOOP, children=[IQC, AS])
```
This means: do `IQC`, then optionally `AS`, then repeat.

But we need the **first** `AS` **before** the loop.

So the pattern becomes:
```
AS  loop_iqc_as  TST
```

#### Loop 2: TST and RT (re-test loop)
```python
loop_tst_rt = OperatorPOWL(operator=Operator.LOOP, children=[TST, RT])
```
This means: do `TST`, then optionally `RT`, then repeat.

But note: in all logs, `TST` always appears **at least once**, and `RT` only after a failure.

`*(TST, RT)` correctly models:
- One `TST`
- Then optionally `RT`, then back to `TST`, etc.

So final `TST` is followed by `PK`.

#### Choice for Labeling (LB or skip)
```python
choice_lb = OperatorPOWL(operator=Operator.XOR, children=[LB, skip])
```
Then follow with `D`.

But note: `D` must come after this choice.

So: `PK  choice_lb  D`

### Step 3: Assemble the full partial order

We now have the following high-level nodes:
1. `OR`
2. `MS`
3. `QS`
4. `AS`
5. `loop_iqc_as`  // represents *(IQC, AS) after first AS
6. `loop_tst_rt`  // represents *(TST, RT)
7. `PK`
8. `choice_lb`   // X(LB, skip)
9. `D`

But wait — the **first** `AS` is not part of the loop. So we need to include it **before** the loop.

So the full sequence of **macro-nodes** is:
- `OR`
- `MS`
- `QS`
- `AS` (initial)
- `loop_iqc_as` *(IQC, AS)*
- `loop_tst_rt` *(TST, RT)*
- `PK`
- `choice_lb` (X(LB, skip))
- `D`

But `loop_tst_rt` **includes** the first `TST`, so we must ensure we don’t duplicate.

Actually, the loop `*(TST, RT)` already includes the first `TST`, so we should **not** have a separate `TST` node.

Similarly, the first `AS` is **outside** the `*(IQC, AS)` loop.

So correct decomposition:

- `AS`  `*(IQC, AS)`  `*(TST, RT)`  `PK`  `X(LB, skip)`  `D`

But `*(TST, RT)` includes the first `TST`, so we don’t need a separate `TST`.

However, in the log, after `AS` comes `IQC` (if any), then `AS`, etc., and **only at the end** comes `TST`.

So the `*(IQC, AS)` loop **feeds into** `*(TST, RT)`.

Thus, the full chain is:

```
OR  MS  QS  AS  *(IQC, AS)  *(TST, RT)  PK  X(LB, skip)  D
```

All steps are **ordered** — each must complete before the next starts.

No concurrency (parallelism) is observed in the log — everything is sequential per case.

So the **partial order** has total order (a chain) of these 9 nodes.

##  Final POWL Model (Code Representation)

```python
import pm4py
from pm4py.objects.powl.obj import StrictPartialOrder, OperatorPOWL, Transition, SilentTransition
from pm4py.objects.process_tree.obj import Operator

# Atomic activities
OR   = Transition(label="OR")
MS   = Transition(label="MS")
QS   = Transition(label="QS")
AS   = Transition(label="AS")
IQC  = Transition(label="IQC")
TST  = Transition(label="TST")
RT   = Transition(label="RT")
PK   = Transition(label="PK")
LB   = Transition(label="LB")
D    = Transition(label="D")
skip = SilentTransition()

# Loop: *(IQC, AS)  -->  after AS, optionally repeat (IQC -> AS)
loop_iqc_as = OperatorPOWL(operator=Operator.LOOP, children=[IQC, AS])

# Loop: *(TST, RT)  -->  test, optionally re-test
loop_tst_rt = OperatorPOWL(operator=Operator.LOOP, children=[TST, RT])

# Choice: LB or skip (silent)
choice_lb = OperatorPOWL(operator=Operator.XOR, children=[LB, skip])

# Build the full workflow as a strict partial order (total order in this case)
root = StrictPartialOrder(nodes=[OR, MS, QS, AS, loop_iqc_as, loop_tst_rt, PK, choice_lb, D])

# Enforce sequential flow
root.order.add_edge(OR, MS)
root.order.add_edge(MS, QS)
root.order.add_edge(QS, AS)
root.order.add_edge(AS, loop_iqc_as)
root.order.add_edge(loop_iqc_as, loop_tst_rt)
root.order.add_edge(loop_tst_rt, PK)
root.order.add_edge(PK, choice_lb)
root.order.add_edge(choice_lb, D)
```

---

##  Explanation

- **Transitions** represent visible activities.
- **SilentTransition** is used for optional `LB` skipping.
- **LOOP** operators:
  - `*(IQC, AS)`: models repeated quality check and reassembly.
  - `*(TST, RT)`: models possible re-testing.
- **XOR** operator: models optional labeling (LB or skip).
- **StrictPartialOrder** with a chain of `add_edge` calls enforces the mandatory sequence, while allowing the internal structure of loops and choices.

This model **generalizes** all five cases:
- Case 2: loop `*(IQC, AS)` executes 0 times, `choice_lb` takes `skip`, so LB is omitted.
- Case 4: same, but also `loop_tst_rt` executes 0 times  skips `TST` and `RT`.
- Case 5: both loops execute multiple times, `LB` present.

Thus, the model is **concise, accurate, and general**.
