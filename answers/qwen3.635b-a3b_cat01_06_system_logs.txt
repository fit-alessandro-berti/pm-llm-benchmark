### Explanation of Design Choices

#### 1. Case Identification Logic
Events were grouped into **logical work units** rather than strict time windows or application boundaries. Each case represents a distinct document or task the user engaged with:
- `Case_Document1`: Captures all interactions with `Document1.docx`, even though the user paused it to handle emails and update a budget. Keeping these events in one case preserves process continuity and allows analysis of total work time, save frequency, and interruption patterns for that specific document.
- `Case_Email_Meeting`: Isolated as a self-contained task (opening, navigating, replying to, and sending an email).
- `Case_ReportDraft` & `Case_Budget`: Grouped by their respective files, as they represent distinct review and update tasks.
- `Case_QuarterlyReport`: Combines the initial focus at 08:59:50 and the final work session at 09:07:15. The ~7-minute gap is intentional, reflecting real-world interleaved workflows. In process mining, this is acceptable and often insightful for measuring task fragmentation.

#### 2. Activity Naming Logic
Raw OS-level events (`FOCUS`, `TYPING`, `CLICK`, `SCROLL`, `SWITCH`, `HIGHLIGHT`) were translated into **standardized, business-process-oriented activity names** using a consistent `<Verb>_<Object>` convention:
- `FOCUS`  `Open_Document` / `Open_Spreadsheet`
- `TYPING`  `Edit_Document` / `Update_Figures` / `Type_Reply`
- `SAVE`  `Save_Document` / `Save_Spreadsheet`
- `CLICK/SCROLL`  `Navigate_Email` / `Open_Email` / `Scroll_Document`
- `SWITCH`  `Switch_Application` (kept as a transition activity; can be filtered out during process discovery if focusing only on value-adding steps)
- `CLOSE`  `Close_Document`

This abstraction removes tool-specific noise while preserving the logical flow of the work.

#### 3. Coherent Narrative
The event log tells a clear story of a **quarterly reporting workflow**:
1. The user briefly opens the final `Quarterly_Report.docx`.
2. Shifts focus to drafting `Document1.docx`, typing content, and saving it.
3. Handles an urgent interruption: replies to an email about an upcoming meeting.
4. Reviews `Report_Draft.pdf` by scrolling and highlighting key findings.
5. Updates financial data in `Budget_2024.xlsx`, adding rows and saving.
6. Returns to `Document1.docx` to insert a budget reference, saves, and closes it.
7. Finally, compiles the executive summary in `Quarterly_Report.docx`, saves, and closes the file.

This sequence reflects a realistic, interrupt-driven knowledge worker session, making it highly suitable for process mining analysis (e.g., measuring cycle times, identifying bottlenecks, analyzing interruption impact, or discovering standard vs. variant paths).
