# Merged Event Log: Order Fulfillment Process

## Matching Strategy & Reasoning

Before presenting the merged log, here is how I approached the alignment:

### Matching Criteria
1. **Order ID** must match (all events here share `Order#1234`).
2. **Timestamp proximity** — events within ~2 seconds are candidates for merging.
3. **Semantic similarity of event names** — e.g., "Order Received"  "OrderReceived", "Payment Processed"  "PaymentCheck".
4. If no suitable counterpart exists, the event is kept standalone with its origin tagged.

### Match Decisions

| Log A Event | Log B Event | t | Decision |
|---|---|---|---|
| `Order Received` @ 10:00:00 | `OrderReceived` @ 09:59:58 | 2s |  Merge — same activity, name nearly identical |
| `Order Validated` @ 10:01:30 | `OrderValidation` @ 10:01:29 | 1s |  Merge — same activity |
| `Payment Processed` @ 10:02:00 | `PaymentCheck` @ 10:02:05 | 5s |  Merge (with note) — names semantically aligned; Log B notes "Payment gateway delay" explains the slight offset. Tolerance stretched because context (same order, same resource_id path, same activity phase) makes match highly likely |
| — | `Quality Check` @ 10:03:00 | — |  Log B only — no counterpart in Log A |
| `Item Shipped` @ 10:05:00 | `Shipping` @ 10:05:02 | 2s |  Merge — same activity |
| `Item Delivered` @ 10:20:00 | — | — |  Log A only — no counterpart in Log B |

### Conflict Resolution
- **Primary timestamp:** Log A (per problem statement — "primary timeline"), but Log B's timestamp is preserved as `timestamp_system_b` since it likely reflects the *actual start* of the event (Log A records *reception* time).
- **Event name:** Normalized to Log A's wording where merged; Log B's original name retained as `event_name_system_b`.
- **Payment event:** The 5-second gap is noted explicitly. I merged it because Log B's "Payment gateway delay" note explains the drift — excluding the merge would lose valuable context.

## Summary of Observations

1. **System B captures extra process steps** (Quality Check) that System A doesn't record — likely because System B is closer to the warehouse operations floor.
2. **System A captures final delivery** (Item Delivered) — likely because System B's scope ends when the package leaves the warehouse.
3. **System B timestamps precede System A's consistently** for matched events (by 1–2 seconds), supporting the hypothesis that System A logs *reception* time while System B logs *origination* time. For future integrations, treating Log B's timestamp as the "event start" and Log A's as "event recorded" may be more accurate.
4. **6 unique process events** result from merging 5 (Log A) + 5 (Log B) = 10 raw entries, with 4 merged pairs and 2 standalone events.
