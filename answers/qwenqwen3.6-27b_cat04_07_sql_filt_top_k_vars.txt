Here is a complete, production-ready DuckDB SQL query that fulfills all your requirements. It uses Common Table Expressions (CTEs) for clarity and step-by-step execution.

```sql
WITH case_variants AS (
    -- Step 1: Construct the ordered activity sequence per case
    SELECT 
        case_id,
        STRING_AGG(activity, ' -> ' ORDER BY timestamp) AS variant
    FROM event_log
    GROUP BY case_id
),
variant_counts AS (
    -- Step 2: Count how many cases follow each unique variant
    SELECT 
        variant,
        COUNT(case_id) AS case_count
    FROM case_variants
    GROUP BY variant
),
top_k_variants AS (
    -- Step 3: Identify the top K variants by frequency
    SELECT variant
    FROM variant_counts
    ORDER BY case_count DESC
    LIMIT 5  --  Replace 5 with your desired K value
),
target_cases AS (
    -- Map the top K variants back to their corresponding case_ids
    SELECT case_id
    FROM case_variants
    WHERE variant IN (SELECT variant FROM top_k_variants)
)
-- Step 4: Return all original events belonging to the filtered cases
SELECT el.*
FROM event_log el
INNER JOIN target_cases tc ON el.case_id = tc.case_id
ORDER BY el.case_id, el.timestamp;
```

###  How It Works
| CTE / Step | Purpose | DuckDB Feature Used |
|------------|---------|---------------------|
| `case_variants` | Orders events per `case_id` by `timestamp` and concatenates them into a single sequence string. | `STRING_AGG(... ORDER BY timestamp)` ensures deterministic chronological ordering. |
| `variant_counts` | Groups identical sequences and counts how many `case_id`s match each. | Standard `GROUP BY` + `COUNT()` aggregation. |
| `top_k_variants` | Ranks variants by frequency and keeps only the top `K`. | `ORDER BY case_count DESC LIMIT K` efficiently isolates the most common paths. |
| `target_cases` | Resolves which `case_id`s belong to those top `K` variants. | `IN` subquery filters cases without duplicating rows. |
| Final `SELECT` | Joins back to the raw event log to return all original events for the selected cases. | `INNER JOIN` preserves all columns and maintains original event granularity. |

###  DuckDB-Specific Tips
1. **Parameterizing `K`**: If you're running this in a script or application, replace `LIMIT 5` with `LIMIT :K` (DuckDB supports named parameters in Python/CLI) or use a parameter CTE:
   ```sql
   WITH params(K) AS (VALUES (5)),
   ...
   LIMIT (SELECT K FROM params)
   ```
2. **Delimiter Safety**: If your `activity` names might contain `' -> '`, switch to DuckDB's native array aggregation to avoid false matches:
   ```sql
   LIST(activity ORDER BY timestamp) AS variant
   ```
   Arrays are compared element-by-element in DuckDB, making them safer for exact sequence matching.
3. **Performance**: For large event logs, DuckDB's vectorized engine will process these CTEs efficiently. If memory becomes a constraint, consider adding `/*+ SET memory_limit='4GB' */` or partitioning by `case_id` beforehand.
