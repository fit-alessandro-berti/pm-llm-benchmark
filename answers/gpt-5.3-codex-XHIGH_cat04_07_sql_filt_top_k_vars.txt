Here is a DuckDB SQL pattern that does exactly what you described:

```sql
-- Set K here (top K variants)
WITH case_variants AS (
    SELECT
        case_id,
        string_agg(activity, ' -> ' ORDER BY "timestamp") AS variant
    FROM event_log
    GROUP BY case_id
),
variant_counts AS (
    SELECT
        variant,
        COUNT(*) AS case_count   -- number of cases following this variant
    FROM case_variants
    GROUP BY variant
),
top_k_variants AS (
    SELECT
        variant,
        case_count
    FROM variant_counts
    ORDER BY case_count DESC, variant
    LIMIT 5   -- <-- replace 5 with K
),
cases_in_top_k AS (
    SELECT
        cv.case_id,
        cv.variant
    FROM case_variants cv
    JOIN top_k_variants tk
      ON cv.variant = tk.variant
)
SELECT
    e.case_id,
    e.activity,
    e."timestamp",
    c.variant,
    tk.case_count AS variant_case_count
FROM event_log e
JOIN cases_in_top_k c
  ON e.case_id = c.case_id
JOIN top_k_variants tk
  ON c.variant = tk.variant
ORDER BY tk.case_count DESC, c.variant, e.case_id, e."timestamp";
```

This query:
1. Builds each case’s ordered activity sequence (`case_variants`).
2. Counts how many cases map to each sequence (`variant_counts`).
3. Picks the top K most frequent variants (`top_k_variants`).
4. Returns only events from cases in those top K variants.

If timestamps can tie within a case, add a secondary sort key in `string_agg(... ORDER BY ...)` (e.g., event_id) for deterministic ordering.
