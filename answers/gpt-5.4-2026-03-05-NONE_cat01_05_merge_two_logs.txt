## Matching approach

I used these rules:

- **Primary key for alignment:** `order_id`
- **Time tolerance:** up to **2 seconds** difference for likely same event
- **Name normalization / semantic mapping:**
  - `Order Received`  `OrderReceived`
  - `Order Validated`  `OrderValidation`
  - `Item Shipped`  `Shipping`
- **Borderline case:** `Payment Processed` vs `PaymentCheck`
  - Names are similar but not identical in meaning
  - Timestamp difference is **5 seconds**, which is outside the requested example tolerance
  - So I treat them as **separate events**
- Events with no good counterpart remain standalone, marked by source

For chronology, I use **System A timestamp as primary when a merge exists**, because Log A is described as the primary timeline. I also keep both timestamps where applicable.

## Event-by-event reasoning

1. **Order Received**
   - A: `2023-10-10T10:00:00Z, Order Received`
   - B: `2023-10-10T09:59:58Z, OrderReceived`
   - Same order, semantically same event, **2-second difference**
   - **Merged**

2. **Order Validated**
   - A: `2023-10-10T10:01:30Z, Order Validated`
   - B: `2023-10-10T10:01:29Z, OrderValidation`
   - Same order, semantically same event, **1-second difference**
   - **Merged**

3. **Payment**
   - A: `2023-10-10T10:02:00Z, Payment Processed`
   - B: `2023-10-10T10:02:05Z, PaymentCheck`
   - Same order, but:
     - time difference is **5 seconds**
     - event names suggest related but possibly distinct steps
   - **Left separate**

4. **Quality Check**
   - Only in B
   - **Included as standalone**

5. **Shipping**
   - A: `2023-10-10T10:05:00Z, Item Shipped`
   - B: `2023-10-10T10:05:02Z, Shipping`
   - Same order, semantically same event, **2-second difference**
   - **Merged**

6. **Item Delivered**
   - Only in A
   - **Included as standalone**

## Conflict resolution summary

- **Timestamp conflicts:** kept both timestamps; used **Log A timestamp as primary** for merged events.
- **Naming conflicts:** normalized to a unified label based on Log A wording when merge confidence was high.
- **Attribute enrichment:** retained System B’s richer fields (`user_id`, `resource_id`, `notes`) in merged records.
- **Uncertain payment match:** not merged because both **semantic mismatch** and **time gap beyond tolerance** reduce confidence.
