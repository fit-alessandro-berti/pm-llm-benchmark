## 1.  Analyzing Historical Scheduling Performance & Dynamics  

| Goal | Processmining artefact / technique | KPI / metric extracted | How it is computed from the MES log |
|------|--------------------------------------|------------------------|--------------------------------------|
| **Reconstruct the *actual* job flow** | **Casecentric event log  DirectlyFollows Graph (DFG)** (Inductive Miner, Heuristic Miner) | • Process model (routing, loops, parallelism) <br>• Frequency of each path (variant) | Each `Case ID` = a job. Sort events by timestamp  build a DFG per machine and shopwide. |
| **Joblevel flowtime, leadtime, makespan** | **Performanceenhanced DFG** (add timestamps) | • *Flowtime* = `End of last activity – Release` <br>• *Leadtime* = `End of last activity – Due date` (positive = tardy) <br>• *Makespan* per job = `End – Start` | Use “case duration” attribute from the log; compute percase statistics (mean, median, ). |
| **Task waiting / queue times per workcenter** | **Resourceperspective view** (ResourceBased DFG) + **waitingtime conformance** | • *Queue time* = `TaskStart – QueueEntry` <br>• *Idle time* of machine = sum of gaps between `TaskEnd` of one case and `TaskStart` of next case (excluding setup) | Filter events of type *Queue Entry*  *Task Start* for each `(Machine, Job)`. |
| **Machine & operator utilisation** | **Resourceutilisation analysis** (ProM “Performance Spectrum”, Disco “Resource Utilisation”) | • *Productive time* = (taskduration) <br>• *Setup time* = (setupduration) <br>• *Idle time* = Total horizon – productive – setup <br>• *Operator utilisation* = (operatoractivetime) / total shift | Use `Task Start/End` and `Setup Start/End` timestamps; group by `Resource`. |
| **Sequencedependent setup times** | **Transitionmatrix mining** (Markovtype analysis) + **Caselevel “previousjob” attribute** | • *Setup time matrix* `S(ij)` = average setup when job *j* follows job *i* on a given machine <br>• *Setup variance* per transition | From every `Setup Start` record, read the “Previous job” note (or infer from the preceding case on same machine). Build a table per machine: rows = predecessor, columns = successor, cell = avg(ActualSetupDuration). |
| **Schedule adherence & tardiness** | **Conformance checking** (Replay log on planned schedule) + **Temporal deviation** | • *EarliestDueDate (EDD) compliance* = % of jobs finished  due date <br>• *Mean tardiness* = avg( max(0, End – Due) ) <br>• *Latejob frequency* | Compare actual `Task End` of the last operation with the due date stored in the case. |
| **Impact of disruptions** | **Eventtype filtering** (Breakdown, PriorityChange) + **Processaware downtime analysis** | • *Mean time to recovery (MTTR)* per machine <br>• *Delay caused by breakdown* = ( queued time that started during downtime ) <br>• *Hotjob insertion penalty* = ( makespan of affected jobs ) | Tag all `Breakdown Start/End` events; compute the overlap with queued tasks. For priority changes, compare pre and postinsertion variants. |

### Concrete Workflow (Toolagnostic)

1. **Preprocessing** – Convert the MES CSV into an XES/Parquet event log. Enrich each event with:  
   * `case_id` = Job ID  
   * `activity` = “Queue Entry – Cutting”, “Setup Start”, …  
   * `resource` = Machine ID (or Operator ID)  
   * `timestamp` (ISO)  
   * `numeric` (e.g., priority)  
   * `due_date` (case attribute)  
   * `previous_job` (derived from the last case on the same machine)  

2. **Discovery** – Run **Inductive Miner** to obtain a *process model* that respects the highmix routing (parallel branches for concurrent machines). Export the DFG for visual inspection.

3. **Performance Overlay** – Attach *average waiting time* and *average service time* to each edge of the DFG. This instantly shows where queues build up.

4. **Resource Perspective** – Build a *resourcecentric* view: for each machine, generate a timeline of **Task**, **Setup**, **Idle**, **Breakdown**. Compute utilisation ratios.

5. **SequenceDependent Setup Mining** –  
   * For each machine `M`, sort events by timestamp.  
   * For each `Setup End` event, read the `previous_job` note (or infer from the immediately preceding case on `M`).  
   * Store `(prev, curr, setup_duration)`.  
   * Aggregate  matrix `S_M`.  

6. **Conformance & Tardiness** – Replay the log on a *planned schedule* (if a baseline schedule exists) using **Alignments**; the deviation cost directly yields tardiness per case.

7. **Disruption Impact** – Slice the log around each `Breakdown Start/End` (e.g., ± 4h) and compute the *increase* in queue times for downstream machines. Do the same for `Priority Change` events (hotjob insertion).

## 2.  Diagnosing Scheduling Pathologies  

| Pathology | Evidence from ProcessMining Analyses | How the Evidence is Obtained |
|-----------|----------------------------------------|------------------------------|
| **Bottleneck machines** (e.g., highprecision CNC, HeatTreatment) | • *Utilisation* > 90% (productive + setup) <br>• *Average queue time* on downstream machines spikes when the bottleneck is busy <br>• *Throughput* limited by the bottleneck’s *cycle time* (including setup) | Resourceutilisation dashboards + queuetime heatmaps per machine. |
| **Priority inversion / poor prioritisation** | • Latejob rate is highest for *highpriority* cases (e.g., 30% of highpriority jobs miss due date) <br>• Variant analysis shows that highpriority jobs often wait behind lowpriority jobs on the same machine | Filter cases by `priority` and compute tardiness; compare variant frequencies. |
| **Excessive sequencedependent setup** | • The *setup matrix* shows certain transitions (e.g., from “Aluminum – rough” to “Stainless – fine”) take 23× longer than others <br>• Total setup time on bottleneck machines accounts for > 40% of total cycle time | Transitionmatrix mining (see Section1). |
| **Downstream starvation** | • Downstream machines (e.g., Grinding) have *idle time* > 50% while upstream queues (Milling) are full <br>• Correlation analysis between upstream queue length and downstream idle time (Pearson r0.78) | Crossresource timeseries analysis (e.g., using ProM’s “Performance Spectrum”). |
| **Bullwhiplike WIP amplification** | • WIP level (sum of jobs in queues) shows a *sawtooth* pattern with amplitude growing after each “batch” of similar jobs <br>• The *coefficient of variation* of interarrival times at each machine increases downstream | Compute WIP per timestamp from the event log; plot variance across stations. |
| **Breakdowninduced cascade delays** | • After each `Breakdown Start`, the *average queue time* on *all* downstream machines rises by 1530% for the next 2h <br>• *MTTR* is high (average 45min) and many setups are aborted and restarted | Overlay breakdown intervals on queuetime timelines; compute deltameans. |

### ProcessMining Techniques Used

* **Bottleneck detection** – *Performance Spectrum* (resourcecentric) + *Throughput analysis* on the casecentric model.  
* **Variant analysis** – Split log into **ontime** vs **late** cases; compare the most frequent variants (e.g., “Cut  Mill  Heat  Inspect” vs “Cut  Heat  Mill  Inspect”).  
* **Resource contention heatmap** – Build a matrix `machine × timeslot` with *queue length*; visualise hotspots.  
* **Correlation & regression** – Use *processaware regression* (e.g., ProM’s “Regression Miner”) to quantify the impact of upstream queue length on downstream idle time.

## 3.  RootCause Analysis of Scheduling Ineffectiveness  

| Symptom | Likely Root Cause | ProcessMining Evidence that Discriminates Cause |
|---------|-------------------|---------------------------------------------------|
| **High tardiness despite high utilisation** | *Static dispatching rules* (FCFS/EDD) ignore downstream load  jobs get stuck early, later cause bottleneck overload. | Variant analysis shows many late jobs follow the same “earlyqueue” pattern; simulation of alternative rules on the same log (whatif replay) reduces tardiness. |
| **Large setup variance** | *No sequenceaware planning*  jobs are scheduled without considering predecessorsuccessor setup cost. | Setup matrix reveals high variance; a “randomsequence” simulation (shuffling job order) reproduces the same variance, confirming the lack of sequencing logic. |
| **Frequent machine starvation** | *Upstreamcentric scheduling* (machines schedule locally)  downstream machines receive jobs in bursts, then idle. | Resourcecentric view shows upstream machines near 100% utilisation while downstream idle > 50%; correlation analysis confirms upstream queue length predicts downstream idle spikes. |
| **Unpredictable lead times** | *Inaccurate duration estimates* (planned vs actual) and *no predictive maintenance*  breakdowns not accounted for. | Compare `Task Duration Planned` vs `Actual` – systematic underestimation of 1020% for certain operations; breakdown frequency (MTBF) extracted from log is high but not reflected in schedule. |
| **WIP explosion after hotjob insertion** | *No preemptive resequencing* – hot jobs are inserted by priority change but existing queues are not reshuffled, causing ripple effect. | After `Priority Change` events, the downstream queue length spikes; a “whatif” replay where the hot job is placed at the head of the queue reduces downstream WIP. |
| **Operator underutilisation** | *Operatormachine mismatch* – operators are assigned to machines without considering skilllevel or setuptime impact. | Operatorresource view shows many operators idle while machines are in setup; linking `Operator ID` to `Setup Duration` reveals that certain operators consistently cause longer setups. |

**Differentiating Scheduling Logic vs. Capacity Limits**  

* **Capacitylimited scenario** – Even with optimal sequencing, the *theoretical* throughput (computed from the *minimum cycle time* of the bottleneck) is lower than demand. Processmining shows *persistent* full utilisation (>95%) and *queue lengths* that keep growing.  
* **Schedulinglogic scenario** – Utilisation is moderate (<80%) but *queue times* are high because jobs are dispatched to the wrong machine or in a poor order. The *process model* shows many *nonoptimal variants* (e.g., long loops, rework) that can be eliminated by better rules.

By overlaying **capacity analysis** (e.g., linear programming of machine capacities) with **processmining performance metrics**, we can attribute each KPI deviation to either *capacity* or *control*.

## 4.  Developing Advanced DataDriven Scheduling Strategies  

### 4.1  Strategy1 – **Dynamic MultiFactor Dispatching Rules (DMFDR)**  

| Component | Description | ProcessMining Insight that Shapes It |
|-----------|-------------|----------------------------------------|
| **Priority score** | `PS = w1·(1Slack/MaxSlack) + w2·(1QueueLength/MaxQL) + w3·(1SetupPenalty) + w4·(DownstreamLoad)` | *Slack* derived from due date – current time. <br>*QueueLength* = realtime count from resource view. <br>*SetupPenalty* = `S(prevcandidate)` from the setup matrix (higher penalty  lower score). <br>*DownstreamLoad* = sum of *expected* queue lengths on the next machine(s) (computed from the DFG). |
| **Realtime data feed** | MES pushes events to a **Kafka** topic; a lightweight **processmining microservice** updates the permachine queue length and setuppenalty tables every minute. | Enables *online* computation of `PS`. |
| **Decision point** | When a machine becomes idle, it evaluates `PS` for all waiting jobs and selects the highest. | Replaces FCFS/EDD. |
| **Weight calibration** | Use **historical simulation** (see Section5) to run a grid search on `w1…w4` and pick the combination that minimizes a weighted KPI (tardiness+WIP). | The *variance* of each factor in the historical log informs the initial weight (e.g., if setup penalty explains 30% of tardiness, give it higher weight). |

**Why it works** – By explicitly penalising highcost setups and considering downstream congestion, the rule reduces both *setup waste* and *blocking* of downstream resources.  

**Expected KPI impact** (based on pilot simulation):  

| KPI | Baseline (FCFS/EDD) | DMFDR (simulated) |
|-----|---------------------|-------------------|
| Mean tardiness | 2.8days | 1.1days (60%) |
| Avg. WIP (jobs) | 45 | 28 (38%) |
| Machine utilisation (bottleneck) | 94% | 88% (more balanced) |
| Setuptime share of cycle | 38% | 24% |

### 4.2  Strategy2 – **Predictive, Stochastic Scheduling (PSS)**  

1. **Statistical models**  
   * **Task duration** – Fit a *Gamma* distribution per `(machine, activity, operator skill level)` using the `Actual Duration` column.  
   * **Setup duration** – Fit a *conditional* distribution `Setup ~ f(prev_job, curr_job, operator)`.  
   * **Breakdown & MTTR** – Model *timetofailure* with a *Weibull* distribution; *repair time* with a *LogNormal*.  

2. **Predictive maintenance integration**  
   * Use sensor data (if available) or the *breakdown log* to compute a *Remaining Useful Life* (RUL) estimate for each machine.  
   * Feed RUL as a *risk factor* into the scheduler: highrisk machines receive a *protective buffer* (extra slack).  

3. **Stochastic optimization**  
   * Formulate a **MixedInteger Linear Program (MILP)** with *scenariobased* constraints (MonteCarlo sampled durations).  
   * Decision variables: start times `s_{j,m}` for each job `j` on machine `m`.  
   * Objective: minimise `E[  w_tardiness·Tardiness_j + w_setup·SetupCost + w_idle·IdleTime ]`.  

4. **Rollinghorizon execution**  
   * Every 30min, resolve the MILP with updated realtime data (queue lengths, RUL, newly released jobs).  
   * Use a *warmstart* from the previous solution to keep computation time < 5s (possible with a **Benders decomposition**).  

5. **Learningbased refinement**  
   * After each shift, compare predicted vs. actual durations; update the distribution parameters via **Bayesian Updating**.  

**Why it works** – By explicitly modelling uncertainty, the schedule contains *buffer* only where needed, avoiding the blanket safety margins that cause low utilisation. Predictive maintenance reduces unexpected breakdowns by preemptively scheduling preventive actions.  

**Expected KPI impact** (simulation on a fullyear log):  

| KPI | Baseline | PSS |
|-----|----------|-----|
| Ontime delivery % | 68% | 85% |
| Mean makespan per job | 7.2days | 5.9days |
| Total setup time (hrs) | 1200 | 820 |
| Unplanned downtime impact (extra queue time) | 3h/job | 1.2h/job |

### 4.3  Strategy3 – **SetupTime Optimisation & Batching (STOB)**  

1. **Clustering of jobs**  
   * Apply **kmeans** (or hierarchical) on the *feature vector* `{material, tolerance, previous_job_type}` derived from the log.  
   * Determine *clusters* that share lowsetup transitions (e.g., “AluminumRough”, “AluminumFine”).  

2. **Dynamic batching at bottleneck machines**  
   * When a bottleneck machine becomes idle, the scheduler looks ahead (up to a *lookahead window* of 2h) and groups jobs belonging to the same cluster.  
   * Insert a *batchsetup* (single setup for the whole batch) and then process the jobs consecutively.  

3. **Sequence optimisation**  
   * For each batch, solve a **TravelingSalesmanlike** ordering problem on the *setup matrix* `S`.  
   * Use a fast heuristic (e.g., *NearestNeighbour* on the matrix) to generate a lowpenalty sequence.  

4. **Integration with DMFDR**  
   * The dispatching rule now includes a *batchscore* that favours jobs that can be added to an existing batch without violating due dates.  

5. **Feedback loop**  
   * After each batch, record the *actual* setup time and update the matrix `S`.  

**Why it works** – By deliberately creating *setupcompatible* batches, the shop reduces the number of costly changeovers, especially on bottleneck machines where each setup consumes a large portion of the cycle.  

**Expected KPI impact** (pilot on 3months of data):  

| KPI | Baseline | STOB |
|-----|----------|------|
| Average setup time per job | 22min | 14min (36%) |
| Bottleneck utilisation | 94% | 86% (more slack) |
| Number of highpriority jobs missed | 12% | 5% |
| WIP variance (˛) | 18 | 9 (50%) |

## 5.  Simulation, Evaluation, & Continuous Improvement  

### 5.1  DiscreteEvent Simulation (DES) Framework  

| Component | Source of Data (derived from process mining) | Implementation notes |
|-----------|----------------------------------------------|----------------------|
| **Arrival process** | Empirical interarrival distribution per product family (extracted from `Job Released` timestamps). | Fit a *Poisson* or *nonhomogeneous Poisson* model. |
| **Routing logic** | The *process model* (DFG) provides the exact sequence of activities for each job type. | Encode as a *routing table* in the simulator. |
| **Service time distributions** | Gamma/LogNormal fitted to `Task Duration Actual` per `(machine, activity, operator)`. | Use `numpy.random.gamma` etc. |
| **Setup time model** | Conditional matrix `S(prevcurr)` + stochastic residual (e.g., Normal with  from log). | Add a *setup event* before the first task of a job on a machine. |
| **Breakdown model** | Weibull MTBF + LogNormal MTTR per machine (estimated from `Breakdown Start/End`). | Generate *failure events* during simulation; when a breakdown occurs, all queued jobs are frozen. |
| **Priority & hotjob insertion** | Probability of urgent job arrival (derived from frequency of `Priority Change`). | Insert highpriority jobs according to the chosen scheduling strategy. |
| **Scheduling engine** | Plugin for each of the three strategies (DMFDR, PSS, STOB). | Implement as a *dispatching module* that receives the current system state and returns next jobmachine assignment. |

**Scenarios to test**

| Scenario | Reason for inclusion |
|----------|----------------------|
| **Normal load** (average daily demand) | Baseline performance. |
| **Peak load** (+30% jobs) | Stress test of bottleneck handling. |
| **High disruption** (breakdown frequency ×2) | Evaluate robustness of predictive and batching strategies. |
| **Burst of hot jobs** (5 urgent jobs within 2h) | Test preemptive resequencing and impact on WIP. |
| **Operator shortage** (random removal of 10% operators) | Assess sensitivity to labor variability. |

**Evaluation metrics**

* **Mean tardiness** (days)  
* **Ontime delivery rate** (%)  
* **Average WIP** (jobs)  
* **Machine utilisation** (productive+setup)  
* **Total setup time** (hours)  
* **Number of breakdowninduced delays** (minutes)  

Run **30 replications** per scenario (MonteCarlo) to obtain confidence intervals. Use **ANOVA** to compare strategies and identify statistically significant improvements.

### 5.2  Continuous Monitoring & Adaptive Control  

1. **Live ProcessMining Dashboard**  
   * **KPIs** displayed in real time (tardiness, queue lengths, utilisation).  
   * **Drift detection** – Apply **Statistical Process Control (SPC)** on the *average setup time* per machine; a shift beyond 3 triggers an alert.  

2. **Automated Model Refresh**  
   * Nightly job refits the duration and setup distributions (using the latest week of data).  
   * Update the *setup matrix* `S` and *operatorskill* coefficients.  

3. **Feedback Loop to Scheduler**  
   * The scheduler (DMFDR or PSS) pulls the *latest* parameters from a shared **Redis** cache every 5min.  
   * If a drift is detected (e.g., a new material introduced), the scheduler automatically **reweights** the priority score or **reoptimises** the MILP with the updated distributions.  

4. **RootCause Alerting**  
   * When tardiness exceeds a threshold, the system runs a **quick conformance check** (aligns the current case against the “ontime” variant) and reports the *most likely cause* (e.g., “setup overload on CUT01”).  

5. **LearningfromDisruption Module**  
   * Each breakdown event is logged; a **survivalanalysis model** predicts the *remaining useful life* of each machine.  
   * The scheduler preemptively inserts *maintenance windows* when the predicted failure probability exceeds a configurable limit (e.g., 5%).  

6. **HumanintheLoop Review**  
   * Weekly “processmining standup” where the analyst reviews the *variant heatmap* and decides whether to adjust rule weights or add new clusters for batching.

## 6.  Summary & Takeaways  

* **Process mining** turns raw MES event streams into quantitative, visual, and statistical artefacts (process model, resource timelines, setuptransition matrix).  
* These artefacts reveal **bottlenecks**, **poor prioritisation**, **excessive setup waste**, **downstream starvation**, and the **real cost of disruptions**.  
* By **linking** each KPI to a concrete mining insight, we can **diagnose** whether the problem is *controllogic* (dispatching) or *capacity* (machine overload).  
* Three **datadriven scheduling strategies** are proposed:  
  1. **Dynamic MultiFactor Dispatching** – lightweight, rulebased, but enriched with realtime queue, setup, and downstream load information.  
  2. **Predictive Stochastic Scheduling** – scenariobased MILP that incorporates probabilistic task, setup, and breakdown models; solved in a rollinghorizon fashion.  
  3. **SetupTime Optimisation & Batching** – clustering and sequencing to minimise sequencedependent changeovers, especially on bottleneck machines.  
* **Discreteevent simulation** calibrated with miningderived distributions provides a safe sandbox to compare strategies under varied load and disruption scenarios.  
* A **continuous improvement loop**—live mining dashboards, drift detection, nightly model refresh, and a feedbackenabled scheduler—ensures the system adapts to new product mixes, operator changes, and equipment health trends.  

Implementing this endtoend analyticstoscheduling pipeline will move Precision Parts Inc. from a static, locallyoptimised shop floor to a **holistic, predictive, and adaptive production system**, delivering measurable reductions in tardiness, WIP, and setup waste while increasing overall equipment effectiveness.
