# DuckDB SQL Solution: Top K Process Variants

Here is a query that follows the four steps in the prompt using a chain of CTEs:

```sql
WITH
-- 1) Build the ordered activity sequence for each case
case_sequences AS (
    SELECT
        case_id,
        array_agg(activity ORDER BY timestamp ASC) AS variant_sequence
    FROM event_log
    GROUP BY case_id
),

-- 2 & 3) Count cases per variant and rank them by frequency
variant_ranking AS (
    SELECT
        variant_sequence,
        COUNT(*) AS case_count,
        RANK() OVER (ORDER BY COUNT(*) DESC) AS rnk
    FROM case_sequences
    GROUP BY variant_sequence
),

-- Pick the top K variants (replace 5 with your K, or use a parameter)
top_variants AS (
    SELECT variant_sequence
    FROM variant_ranking
    WHERE rnk <= 5         -- <<< K = 5
),

-- Resolve which case_ids belong to those top K variants
top_cases AS (
    SELECT DISTINCT cs.case_id
    FROM case_sequences cs
    JOIN top_variants  tv USING (variant_sequence)
)

-- 4) Return every event whose case is in the top-K set
SELECT el.*
FROM   event_log el
JOIN   top_cases   tc ON el.case_id = tc.case_id
ORDER  BY el.case_id, el.timestamp;
```

## How it works, step by step

| CTE | Purpose |
|---|---|
| **`case_sequences`** | `array_agg(activity ORDER BY timestamp)` produces an ordered `LIST` per `case_id`. This is the canonical “trace” of the case. |
| **`variant_ranking`** | Groups by the full `LIST` (so two cases match only if their sequences are elementwise equal), counts cases per variant, and ranks them by frequency. |
| **`top_variants`** | Keeps the K highestranked sequences. |
| **`top_cases`** | Looks up the `case_id`s that produced those sequences. |
| **Final `SELECT`** | Joins back to the original `event_log` so you get the **raw event rows** (not aggregated), ordered for readability. |

## Things to consider

1. **Choosing K.** Replace the literal `5` with the K you need, or expose it via a session variable, e.g. `WHERE rnk <= $k`, and run `SET k = 5;` beforehand.
2. **Ties at the Kth boundary.** I used `RANK()`, which can return *more than* K variants when ties exist (e.g., the Kth place is shared). If you want **exactly** K variants, switch to `ROW_NUMBER()`; this arbitrarily breaks ties but guarantees K rows. If you want a strict, reproducible tiebreaker, append a secondary column:
   ```sql
   RANK() OVER (ORDER BY COUNT(*) DESC, variant_sequence)
   ```
3. **Why `array_agg`?** It is the most robust representation because it doesn’t depend on choosing a delimiter that might appear in activity names. DuckDB compares `LIST`s elementwise, so equality (`USING (variant_sequence)`) correctly identifies identical traces.
4. **Parameterizing the full query.** If you call this from Python (e.g., via `duckdb.execute`), use:
   ```python
   duckdb.execute("SET k = ?", [5])
   ```
   and reference `current_setting('k')` inside the query.
5. **Performance.** DuckDB executes the `array_agg` aggregation in a single pass; the join on the small `top_cases` set is efficient. For very large logs, ensure `timestamp` is indexed/clustered if possible.

The result is the subset of the original event log containing only the events that belong to cases in the most frequent K process variants.
