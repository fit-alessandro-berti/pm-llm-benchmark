## 1.  EndtoEnd Narrative of the Business Process  

| # | Symbol in the diagram | Activity / Decision (Purpose) |

| **2** | **TaskA – “Receive Customer Request”** | Capture the request details (customer ID, product(s), quantity, any special notes). This creates the **request record** that will be referenced throughout the workflow. |
| **3** | **XOR Gateway – “Check Request Type”** | **Exclusive choice** – the system looks at a field in the request record (e.g., `type = “Standard”` or `type = “Custom”`). Only **one** of the following branches will be taken. |
| **4** | **Standard path (left branch)** ||
|   4.1 | **TaskB1 – “Perform Standard Validation”** | Verify that the request complies with the company’s baseline rules (e.g., product codes exist, minimum order size, valid shipping region). This step filters out malformed or obviously invalid orders before any heavyweight checks are run. |
|   4.2 | **AND Gateway – “Run Parallel Checks”** | **Parallel split** – both subtasks below start simultaneously because they are independent. |
|   4.2.1 | **TaskC1 – “Credit Check”** | Query the creditrating service to see if the customer’s account is in good standing. |
|   4.2.2 | **TaskC2 – “Inventory Check”** | Verify stock availability for every lineitem in the request. |
|   4.3 | **AND Join – “All Parallel Checks Completed”** | **Synchronisation** – the process cannot continue until *both* the credit and inventory checks have finished. The join emits a single token that carries the combined result (e.g., `credit = OK`, `inventory = OK`). |
|   4.4 | **TaskD – “Calculate Delivery Date”** | Based on the inventory availability and the company’s leadtime matrix, compute an expected ship date and delivery window. This becomes the promise made to the customer. |
| **5** | **Custom path (right branch)** ||
|   5.1 | **TaskB2 – “Perform Custom Feasibility Analysis”** | A salesengineer or productspecialist evaluates the request against engineering, tooling, and cost constraints. |
|   5.2 | **XOR Gateway – “Is Customization Feasible?”** | **Exclusive choice** – if the answer is *Yes* the process follows the *yes* branch; if *No* it follows the *no* branch. |
|   5.2.1 (Yes) | **TaskE1 – “Prepare Custom Quotation”** | Generate a detailed pricequote that includes any special tooling, extra materials, or engineering hours. |
|   5.2.2 (No) | **TaskE2 – “Send Rejection Notice”** | Inform the customer that their request cannot be fulfilled and close the case. This branch terminates at an **End Event**. |
| **6** | **Convergence after Standard or Custom tasks** | After **TaskD** (Standard) or **TaskE1** (Custom) the workflow reaches the next XOR gateway. The two streams have already “merged” conceptually because only one token can be present – either the Standard token or the Custom token. |
| **7** | **XOR Gateway – “Is Approval Needed?”** | **Exclusive choice** – based on policy (e.g., order value > $10k, new customer, or a flagged credit risk) a Boolean flag `approvalRequired` is evaluated. |
|   7.1 (Yes) | **TaskF – “Obtain Manager Approval”** | Route the request to a manager (email, BPMN usertask, or inbox). The manager reviews the precalculated data (delivery date, quotation, credit status). |

| **8** | **TaskI – “Send Confirmation to Customer”** | Email or portal notification that the order is confirmed, includes the invoice and the promised delivery date. |
| **9** | **End Event** | The process terminates. |

## 2.  Gateway Semantics (Why they matter)

| Gateway type | Symbol in diagram | What it does | Typical usage in this process |
|--------------|-------------------|--------------|------------------------------|
| **XOR (exclusiveor)** | Diamond with a **X** | **Only one** outgoing branch is taken for a given token. The condition on each outgoing arrow is evaluated, and the first *true* (or the default) branch is selected. | *Check Request Type*, *Is Customization Feasible?*, *Is Approval Needed?*, *Is Approval Granted?*. |
| **AND (parallel)** | Diamond with a **+** | **All** outgoing branches are taken **simultaneously**. Tokens are created for each branch. A corresponding **AND join** later waits until **all** parallel tokens arrive before the workflow can proceed. | *Run Parallel Checks*  splits to Credit Check and Inventory Check. The **join** (implicit) ensures both checks are done before “Calculate Delivery Date”. |
| **Implicit XOR merge** | Not shown explicitly, but the gateway after the parallel block acts as a **merge** – when the parallel tokens arrive, the XOR gateway merges them back into a single flow. | Used after the parallel checks to continue with the next task. |
| **Loop back (rework)** | Arrow that goes from **TaskH** back to **TaskE1** or **TaskD** | Introduces a **rework loop**. The token reenters the earlier activity, allowing the system to incorporate manager feedback. | Guarantees that any needed adjustments (price, delivery date) are reflected before a final invoice is generated. |

## 3.  Divergence between **Standard** and **Custom** Tracks  

| Aspect | Standard request | Custom request |
|--------|------------------|----------------|
| **Initial validation** | A quick “Standard Validation” (B1) checks only basic compliance. | A deeper “Custom Feasibility Analysis” (B2) is required; it may involve engineering review, cost modeling, and timeline estimation. |
| **Parallel checks** | **Credit Check** + **Inventory Check** run in parallel because the product catalog is fixed and stock can be queried instantly. | No explicit parallel checks are needed; the feasibility step already covers feasibility of the requested customization. |
| **Outcome of feasibility** | Not applicable – there is no feasibility gate. | **Feasibility gate** (XOR) decides whether a quotation can be produced or the request must be rejected outright. |
| **Reevaluation loop** | If the manager rejects the delivery date or price, the loop goes back to **TaskD** (recalculate). | Loop goes back to **TaskE1** (reprepare quotation) so that the custom pricing reflects any required changes. |

## 4.  When & Why **Approvals** or **Rework** Are Triggered  

| Trigger | Condition (example) | Action taken |
|---------|----------------------|--------------|
| **Approval Needed** | - Order value > $10k<br>- Firsttime customer<br>- Credit check returned a warning flag | A manager is tasked to review the request (TaskF). |

| **Feasibility Failure** (Custom only) | Engineering or cost analysis finds the request impossible. | **TaskE2 – Send Rejection Notice** is invoked; the process ends with a negative End Event. |
| **Rework Loop** | After manager feedback, the request must be revised. | The loop back guarantees that any revised pricing or delivery estimate is reflected before a final invoice is issued. |

## 5.  Convergence & Loop Handling  

* **Single token after the initial XOR** – The “Check Request Type” gateway ensures that only **one** of the two major branches (Standard or Custom) is ever active.  
* **Synchronisation after parallel work** – The **AND join** (implicit after Credit Check + Inventory Check) guarantees that both checks finish before the next step, preventing a premature deliverydate calculation.  
* **Merge into the approval flow** – After either the deliverydate calculation (Standard) or the custom quotation (Custom), the workflow reaches the “Approval Needed” XOR. This merges the two streams into a single path, simplifying downstream logic.  
* **Rework loop** – The arrow from **TaskH** back to **TaskE1** (Custom) or **TaskD** (Standard) creates a *structured loop*. In practice you would normally add a **counter** or **timer** to avoid infinite cycles (e.g., *max 2 rework attempts*).

## 6.  Recommendations for a ProductionReady BPMN Model  

| Issue | Suggested Enhancement |
|-------|------------------------|
| **No explicit error handling** for failed credit/inventory queries. | Add a **Boundary Error Event** on the parallel block that routes to a “Handle Service Failure” task (e.g., retry, manual override). |
| **Potential infinite loop** if the manager repeatedly rejects. | Include a **Loop Counter** (e.g., `reworkCount  2`) and an **Escalation** path that routes to a senior manager after the limit is reached. |
| **No timeout on approvals** – a request could be stuck in the manager’s inbox forever. | Attach a **Timer Boundary Event** to TaskF that triggers an escalation or autorejects after X hours. |
| **Lack of audit data** – important decisions (approval, rejection) are not recorded. | Add **Data Objects** (e.g., `ApprovalRecord`, `RejectionReason`) that are created and stored at the decision points. |
| **No exception for “No Approval Needed” scenario** – if the rule for “approval needed” changes, the model must be updated manually. | Use a **Business Rule Task** that evaluates `approvalRequired` from a central policy table, making the rule changefriendly. |
| **Customer notification only after final invoice** – may be too late for highvalue custom quotes. | Add an **Intermediate Message Event** after the quotation is ready (TaskE1) to send a “Quote Ready” notification, then wait for the customer’s acceptance before proceeding to approval. |
