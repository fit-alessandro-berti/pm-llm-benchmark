```sql
WITH 
case_variants AS (
    SELECT 
        case_id,
        LIST(activity ORDER BY timestamp) AS activity_sequence
    FROM event_log
    GROUP BY case_id
),
variant_frequencies AS (
    SELECT 
        activity_sequence,
        COUNT(*) AS case_count
    FROM case_variants
    GROUP BY activity_sequence
),
top_k_variants AS (
    SELECT activity_sequence
    FROM variant_frequencies
    ORDER BY case_count DESC
    LIMIT ?
),
top_k_cases AS (
    SELECT cv.case_id
    FROM case_variants cv
    JOIN top_k_variants tv 
      ON cv.activity_sequence = tv.activity_sequence
)
SELECT 
    el.*
FROM event_log el
JOIN top_k_cases tc 
  ON el.case_id = tc.case_id
ORDER BY el.case_id, el.timestamp;
```

**Explanation**  
- `case_variants`: builds the exact ordered activity sequence per case using `LIST(... ORDER BY timestamp)`.  
- `variant_frequencies`: aggregates identical sequences and counts how many cases share each variant.  
- `top_k_variants`: selects the K most frequent sequences (K supplied as a query parameter).  
- `top_k_cases`: resolves the set of `case_id`s belonging to any of those top-K variants.  
- Final `SELECT`: returns every original row that belongs to a case in the top-K set, preserving the full event log structure.  

The query is fully deterministic with respect to ordering (timestamp within each case) and correctly filters the event log to only the desired process variants.
