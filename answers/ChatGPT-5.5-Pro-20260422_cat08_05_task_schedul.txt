## 1. Analyzing Historical Scheduling Performance and Dynamics

Process mining is a strong fit for Precision Parts because the MES log contains the key ingredients of an event log: a case identifier, activity name, timestamp, resource, and contextual attributes such as priority, due date, setup flag, planned duration, actual duration, and disruption notes. In process-mining terms, the analysis should not be limited to drawing an “average process map.” It should reconstruct the actual shop-floor execution from multiple perspectives: job flow, machine calendars, operator involvement, waiting behavior, setup behavior, and disruption response. Event logs are explicitly used in process mining to discover, monitor, and improve real processes; in manufacturing, the core process-mining tasks are discovery, conformance checking, and enhancement of process models. ([Springer][1])

### 1.1 Event-log reconstruction

I would first convert the MES records into two linked analytical views.

**Job-centric log**

Here, each `Case ID` is a job, for example `JOB-7001`. The activity sequence might be:

`Job Released → Queue Entry (Cutting) → Setup Start → Setup End → Task Start (Cutting) → Task End (Cutting) → Queue Entry (Milling) → … → Quality Inspection → Job Completed`

This view answers questions such as:

“Which routings do jobs actually follow?”
“Where do jobs wait?”
“Which variants are late?”
“How long does a job spend in queue, setup, processing, rework, inspection, or blocked states?”

**Resource-centric log**

Here, each machine or machine group becomes the analytical object. For `CUT-01`, for example, I would reconstruct a continuous timeline:

`Idle → Setup JOB-7001 → Process JOB-7001 → Idle / Breakdown / Process JOB-7002 → Setup JOB-7003 → …`

This view answers questions such as:

“Which jobs were processed in what order on each machine?”
“How much time was spent producing, setting up, idle, starved, blocked, or broken down?”
“Which job-to-job transitions caused long setups?”
“What was the real available capacity of each bottleneck machine?”

Because this is a high-mix job shop, I would also add a third **object-centric** view where jobs, machines, operators, tools, fixtures, material classes, and customer orders are linked. A pure job-centric model can hide machine contention; a pure machine-centric model can hide job lateness. The scheduling problem lives in the interaction between the two.

### 1.2 Actual job-flow and routing analysis

I would use process discovery techniques such as directly-follows graphs, performance-annotated process maps, Petri-net/BPMN discovery where helpful, and variant analysis. The objective is not merely to find the “happy path,” because high-mix manufacturing naturally has many legal variants. Instead, I would build stratified process maps by:

* product family or part family,
* routing pattern,
* priority class,
* due-date tightness,
* customer segment,
* machine group,
* on-time versus late jobs,
* normal jobs versus hot jobs,
* jobs affected versus not affected by breakdowns.

The output should reveal the actual route variants, rework loops, skipped steps, repeated inspections, excessive transfers, and unexpected machine substitutions. Tools such as PM4Py and commercial process-mining platforms support process discovery, conformance checking, variant analysis, performance analysis, bottleneck analysis, and throughput analysis, which are exactly the analytical modes needed here. ([processintelligence.solutions][2])

### 1.3 Flow time, lead time, and makespan metrics

For each job (j):

[
\text{Flow Time}_j = C_j - R_j
]

where (R_j) is the job release timestamp and (C_j) is the final completion timestamp.

If customer order entry is available:

[
\text{Customer Lead Time}_j = C_j - O_j
]

where (O_j) is order acceptance or order creation.

For each job, I would decompose total flow time into:

[
\text{Flow Time} =
\text{Queue Time}

* \text{Setup Time}
* \text{Processing Time}
* \text{Transfer / Inter-operation Time}
* \text{Rework Time}
* \text{Disruption Waiting Time}
  ]

This decomposition is crucial. Precision Parts may believe processing time is the problem, but in job shops the dominant component is often waiting before bottleneck resources.

For distributions, I would avoid averages alone and report:

* median, mean, 75th, 90th, and 95th percentile flow time,
* flow-time coefficient of variation,
* lead-time reliability by due-date tightness,
* flow time by routing family,
* flow time for on-time versus late jobs,
* flow time before and after breakdowns or hot-job insertions.

For horizon-level makespan, I would measure:

[
\text{Makespan}_{H} = \max(C_j) - \min(R_j)
]

for all jobs released or completed inside a defined planning window (H). This is useful for simulation and schedule comparison, but tardiness and flow-time reliability are more important for customer service.

### 1.4 Queue-time analysis at each work center

For each operation (o) of job (j) at machine (m):

[
\text{Pre-service Waiting Time}*{j,o,m}
= \text{Setup Start}*{j,o,m} - \text{Queue Entry}_{j,o,m}
]

If no setup is needed:

[
\text{Pre-service Waiting Time}*{j,o,m}
= \text{Task Start}*{j,o,m} - \text{Queue Entry}_{j,o,m}
]

I would calculate waiting-time distributions by machine, machine family, operation type, shift, weekday, priority class, and queue length at arrival. For each queue, I would also compute:

* average and percentile queue length,
* WIP area under the curve,
* time jobs spend waiting after becoming available,
* proportion of jobs waiting longer than a threshold,
* correlation between queue size and eventual tardiness,
* waiting time experienced by high-priority jobs versus normal jobs.

The most valuable visualization is a time-based queue heat map: machines on one axis, time on the other, color-coded by WIP or expected workload hours. This reveals whether WIP surges are isolated, recurring by shift, or caused by upstream release waves.

### 1.5 Resource utilization analysis

For each machine (m), I would reconstruct a complete calendar from event intervals:

[
\text{Available Time}_m =
\text{Scheduled Shift Time}

* \text{Planned Downtime}
  ]

Then classify the available time into:

* productive processing time,
* setup time,
* unplanned downtime,
* idle time due to no job available,
* idle time due to no operator/tool/material,
* blocked time if downstream constraints prevent progress,
* starvation time if upstream bottlenecks fail to feed the machine.

Core utilization metrics:

[
\text{Productive Utilization}_m =
\frac{\text{Processing Time}_m}{\text{Available Time}_m}
]

[
\text{Effective Load}_m =
\frac{\text{Processing Time}_m + \text{Setup Time}_m}{\text{Available Time}_m}
]

[
\text{Setup Share}_m =
\frac{\text{Setup Time}_m}{\text{Processing Time}_m + \text{Setup Time}_m}
]

[
\text{Downtime Share}_m =
\frac{\text{Breakdown Time}_m}{\text{Scheduled Shift Time}_m}
]

I would be careful with the interpretation. A machine with 92% effective load may look “efficient,” but if it is the bottleneck, that load level may be causing long queues, late jobs, and unstable flow. Conversely, a downstream machine with low utilization may not be overstaffed; it may be starved by an upstream bottleneck.

OEE-style measures can also be useful, but I would adapt them for job-shop scheduling. OEE-related bottleneck analysis is especially valuable when it considers not only the machine’s own performance but also upstream starvation and downstream blocking effects. ([ScienceDirect][3])

### 1.6 Sequence-dependent setup-time analysis

This is one of the highest-value analyses for Precision Parts.

For every machine, I would reconstruct the actual processing sequence:

[
(JOB_{a}, JOB_{b}, JOB_{c}, \ldots)_m
]

For each transition (JOB_i \rightarrow JOB_{i+1}), I would extract:

[
\text{Setup Time}*{i,i+1,m}
= \text{Setup End}*{i+1,m} - \text{Setup Start}_{i+1,m}
]

Then I would enrich each job with setup-relevant attributes:

* material type,
* alloy,
* thickness,
* geometry family,
* fixture type,
* tooling family,
* CNC program family,
* tolerance class,
* heat-treatment requirement,
* surface finish,
* customer-specific inspection requirement,
* operator,
* shift,
* machine,
* previous machine condition where available.

The result is a transition matrix:

[
S_m(a,b)
]

where (a) is the previous job family and (b) is the next job family on machine (m). For each transition family, I would estimate:

* median setup time,
* 75th and 90th percentile setup time,
* sample size,
* variability,
* operator effect,
* machine effect,
* shift effect,
* setup overrun probability.

For example:

| Machine |       Previous Family |          Next Family | Median Setup | P90 Setup | Count | Comment                        |
| ------- | --------------------: | -------------------: | -----------: | --------: | ----: | ------------------------------ |
| MILL-03 | Stainless / Fixture A | Aluminum / Fixture B |       74 min |   118 min |    42 | Major fixture + tooling change |
| MILL-03 |  Aluminum / Fixture B | Aluminum / Fixture B |       14 min |    25 min |    90 | Similar-family change          |
| CUT-01  |            Thin sheet |          Thick plate |       31 min |    50 min |    35 | Blade/feed-rate change         |

I would then compare actual sequences against lower-bound or optimized sequences to estimate avoidable setup time. Sequence-dependent setup scheduling is a difficult optimization problem; research has treated even single-machine versions with release dates, due dates, and sequence-dependent setups as NP-hard, so practical solutions usually need strong heuristics, rolling-horizon optimization, or simulation-supported dispatching rather than a simplistic global optimum promise. ([Springer][4])

### 1.7 Schedule adherence and tardiness

For each job:

[
\text{Lateness}_j = C_j - D_j
]

where (D_j) is the due date.

[
\text{Tardiness}_j = \max(0, C_j - D_j)
]

[
\text{Earliness}_j = \max(0, D_j - C_j)
]

I would report:

* percent of jobs tardy,
* mean tardiness,
* median tardiness,
* 90th percentile tardiness,
* maximum tardiness,
* weighted tardiness by priority or customer,
* high-priority tardiness rate,
* due-date adherence by product family,
* lateness distribution relative to promised dates,
* schedule adherence by operation: planned start versus actual start, planned end versus actual end.

Weighted tardiness is especially important:

[
\text{Weighted Tardiness}_j = w_j \cdot \max(0, C_j - D_j)
]

where (w_j) reflects priority, penalty cost, customer importance, or contractual risk.

I would also measure **schedule nervousness**:

* number of reschedules per job,
* number of machine assignment changes,
* number of priority changes,
* frequency of hot-job insertions,
* amount of work displaced by urgent jobs.

A shop can appear “responsive” because it constantly expedites, but that often creates hidden instability.

### 1.8 Impact of disruptions

For breakdowns, I would identify:

* machine affected,
* breakdown start and end,
* jobs interrupted,
* jobs waiting in that machine’s queue,
* downstream machines starved after the breakdown,
* jobs whose lateness increased after the event,
* recovery time until queue length returned to normal.

For hot jobs or priority changes, I would identify:

* jobs promoted to urgent,
* jobs displaced by the hot job,
* setup changes caused by insertion,
* additional tardiness imposed on other jobs,
* whether the hot job actually met its revised due date.

I would use event-window analysis:

[
\text{KPI before disruption}
\quad \text{versus} \quad
\text{KPI after disruption}
]

and compare affected jobs against a control group of similar jobs not exposed to the disruption. For major machines, I would also calculate the “tardiness shadow” of a breakdown: how long after the breakdown the delay continues to propagate through downstream queues.

## 2. Diagnosing Scheduling Pathologies

The performance analysis should produce specific, evidence-backed pathologies rather than generic statements such as “we have bottlenecks.” The goal is to identify where current dispatching rules create avoidable lateness, avoidable WIP, avoidable setup time, or avoidable resource starvation.

### 2.1 Bottleneck resources

A true bottleneck is not merely the machine with the highest utilization. I would identify bottlenecks using a combination of:

* high effective load,
* long and persistent queues,
* high WIP waiting before the resource,
* high setup share,
* high downtime impact,
* high proportion of late jobs passing through the resource,
* high sensitivity of system throughput to that resource,
* downstream starvation following that resource’s disruptions.

Evidence from process mining:

* performance-annotated process maps showing long waits before specific work centers,
* machine timelines showing near-continuous occupation,
* queue-length heat maps,
* jobs delayed immediately after visiting the suspected bottleneck,
* bottleneck-shift analysis over time,
* resource-centric directly-follows graphs showing excessive sequencing pressure.

Example diagnosis:

“MILL-03 is not just highly utilized; 68% of late jobs passed through MILL-03, the median pre-service wait at MILL-03 is 14.2 hours versus 2.8 hours elsewhere, and downstream grinding machines are starved after periods when MILL-03’s queue exceeds 20 jobs. Therefore, MILL-03 is a system bottleneck, not simply a busy machine.”

### 2.2 Poor prioritization and local dispatching conflict

The current rules use a local mix of FCFS and EDD. In a dynamic job shop, that can produce contradictions:

* FCFS can delay urgent jobs behind low-value jobs.
* EDD can select a near-due job with a long setup, causing multiple other jobs to become late.
* Local rules can optimize one machine while starving a downstream bottleneck.
* Priority changes can cause overreaction and schedule churn.

Evidence from process mining:

* queue snapshots showing high-priority or near-due jobs waiting behind low-priority jobs,
* comparison of actual dispatch sequence against EDD, critical ratio, slack, or weighted tardiness logic,
* late-job variants showing excessive waiting before the first bottleneck,
* time-to-start after priority escalation,
* number of times urgent jobs still missed due dates after being escalated.

A useful metric is **priority inversion time**:

[
\text{Priority Inversion Time}
==============================

\sum
\text{time a lower-priority job is processed while a higher-priority feasible job waits}
]

Another is **dispatch regret**: the estimated increase in weighted tardiness or setup time caused by the selected job versus the best feasible job under a more complete scoring rule.

### 2.3 Suboptimal sequencing and setup thrashing

If setup times are sequence-dependent, the order of jobs on each machine is itself a capacity decision. Poor sequencing consumes bottleneck capacity without producing parts.

Evidence from process mining:

* setup transition heat maps showing frequent high-setup transitions,
* high setup share at bottleneck machines,
* many single-job switches between families,
* repeated alternation such as `Family A → Family B → Family A → Family B`,
* planned setup time consistently lower than actual setup time,
* hot-job insertions causing large setups and cascading delay.

Metrics:

[
\text{Avoidable Setup Estimate}
===============================

## \text{Actual Setup Time}

\text{Best Feasible Rolling-Horizon Setup Time}
]

[
\text{Setup Thrash Index}
=========================

\frac{\text{Number of high-changeover transitions}}
{\text{Total transitions}}
]

A pathology exists if the bottleneck spends a large share of available capacity in setup, especially when many setups are between job families that could have been batched without violating due dates.

### 2.4 Starvation of downstream resources

Downstream starvation occurs when resources are idle not because demand is low, but because upstream scheduling decisions or bottlenecks fail to release work in a stable pattern.

Evidence from process mining:

* downstream idle intervals aligned with upstream bottleneck congestion,
* resource timelines showing alternating overload and starvation,
* WIP accumulation before one station while downstream stations wait,
* correlation between upstream queue spikes and downstream idle time with a lag,
* late jobs accumulating after the bottleneck rather than before it.

This is important because management may see a downstream machine at 45% utilization and assume there is excess capacity. In reality, that machine may be starved by upstream sequencing decisions.

### 2.5 Bullwhip-like WIP amplification

Even in a factory, scheduling variability can create a bullwhip-like pattern: small changes in release, priority, or machine selection create large WIP swings downstream.

Evidence from process mining:

* WIP oscillation by work center,
* bursts of queue entries after upstream batch completions,
* repeated stop-start patterns in machine timelines,
* high coefficient of variation in arrivals to downstream stations,
* frequent rescheduling after hot jobs or breakdowns,
* high WIP with no corresponding throughput increase.

Metrics:

[
\text{Arrival Variability}_{station}
====================================

CV(\text{interarrival times})
]

[
\text{WIP Amplification}_{station}
==================================

\frac{CV(\text{downstream WIP})}
{CV(\text{upstream completions})}
]

A high ratio indicates unstable flow caused by release and scheduling decisions, not necessarily by demand.

### 2.6 Variant analysis: on-time versus late jobs

I would create two process-mining cohorts:

* on-time jobs,
* late jobs.

Then compare:

* routing variants,
* number of operations,
* queue time at each station,
* setup exposure,
* number of priority changes,
* machine substitutions,
* operators used,
* rework loops,
* exposure to breakdowns,
* first operation waiting time,
* bottleneck waiting time.

This typically reveals patterns such as:

“Late jobs are not longer because their processing time is much higher; they are longer because they wait 3–5 times longer before milling and are more often interrupted by hot-job insertions.”

That distinction matters because the countermeasure is scheduling control, not necessarily faster machining.

## 3. Root Cause Analysis of Scheduling Ineffectiveness

### 3.1 Limitations of static local dispatching rules

The current system applies local rules at each work center. That is fundamentally mismatched to the problem. Precision Parts has:

* unique routings,
* sequence-dependent setups,
* shared bottleneck machines,
* urgent jobs,
* uncertain processing times,
* breakdowns,
* operator constraints,
* downstream congestion.

FCFS and EDD are simple, transparent, and easy to execute, but they ignore key system-wide effects:

* remaining routing after the current operation,
* current and future bottleneck load,
* setup penalty caused by the selected job,
* downstream starvation risk,
* job priority and penalty cost,
* probability of lateness,
* machine breakdown risk,
* opportunity cost of consuming bottleneck capacity.

So the root cause is not merely “bad rules.” It is that local dispatching rules are making decisions using an incomplete objective function.

### 3.2 Lack of real-time visibility

A scheduler must know the current state of the shop:

* which jobs are waiting,
* which machines are available,
* which setups would be required,
* which operators are available,
* which jobs are near lateness,
* which downstream queues are overloaded,
* which bottlenecks are starved,
* which machines are down,
* which hot jobs have entered.

If dispatchers are making decisions from stale queue lists or informal expediting signals, they will optimize locally and reactively. Process mining can convert raw MES events into a live digital operational view: current WIP, current queue length, estimated remaining time, actual machine state, and predicted due-date risk.

### 3.3 Inaccurate duration and setup-time estimates

The log contains planned and actual durations. I would calculate residuals:

[
\text{Processing Error}_{j,o}
=============================

## \text{Actual Processing Time}_{j,o}

\text{Planned Processing Time}_{j,o}
]

[
\text{Setup Error}_{j,o}
========================

## \text{Actual Setup Time}_{j,o}

\text{Planned Setup Time}_{j,o}
]

Then segment by machine, operator, material, part family, tolerance class, shift, and priority.

Common root causes include:

* planned times based on engineering standards but not actual shop performance,
* setup estimates not sequence-dependent,
* missing operator/tool/material delays,
* underestimated tail risk for complex jobs,
* hot jobs using optimistic durations,
* no uncertainty buffer for bottleneck operations.

If planned cutting is 60 minutes but actual cutting has a median of 66 minutes and a 90th percentile of 92 minutes, deterministic scheduling based on 60 minutes will systematically overcommit the machine.

### 3.4 Ineffective handling of sequence-dependent setups

If the planning system treats setup as a fixed add-on, or ignores the previous job entirely, it will repeatedly choose sequences that look good in a local due-date sense but are poor in capacity terms.

Example:

* Job A is due soon and requires a 90-minute setup after the current job.
* Job B is due slightly later and requires a 10-minute setup.
* Job C is same-family as Job B and requires a 5-minute setup after B.

Pure EDD may select A first. A setup-aware rule may run B and C first, then A, reducing total setup enough that all three finish earlier.

This is why sequence-dependent setup must be part of the dispatching score and the rolling-horizon optimizer. ATC-based dispatching rules have been studied specifically in settings with future ready times and sequence-dependent setups, which aligns closely with this shop-floor problem. ([IDEAS/RePEc][5])

### 3.5 Poor coordination between work centers

A local dispatcher may keep their own machine busy while creating problems elsewhere. Examples:

* Cutting releases too many jobs into milling, creating WIP.
* Milling optimizes its own due-date list but starves grinding.
* Heat treatment batches jobs too aggressively, delaying urgent inspection.
* Inspection prioritizes FCFS, causing completed urgent orders to miss shipment.

Process mining can expose this by showing queue propagation across stations, not just utilization at individual stations.

### 3.6 Inadequate disruption response

Breakdowns and hot jobs are unavoidable, but the response can be structured or chaotic.

A poor response pattern looks like this:

1. Machine breaks down.
2. Jobs accumulate.
3. Dispatchers manually expedite the loudest customer or highest-priority order.
4. The hot job creates additional setup.
5. Several medium-priority jobs become late.
6. Downstream resources alternate between starvation and overload.
7. The schedule is reworked repeatedly.

The root cause is not the breakdown alone; it is the absence of a disruption recovery policy that minimizes total weighted damage.

### 3.7 Distinguishing scheduling logic from capacity limitation and inherent variability

Process mining can help separate three different causes.

**Scheduling logic problem**

Evidence:

* high-priority jobs wait while lower-priority jobs run,
* avoidable high-setup transitions occur frequently,
* downstream bottlenecks are starved despite available upstream work,
* late jobs spent most of their time waiting, not processing,
* different dispatch decisions in similar historical states produced better outcomes.

Countermeasure: scheduling rule redesign, setup-aware sequencing, rolling-horizon optimization.

**Capacity limitation**

Evidence:

* bottleneck effective load is persistently near or above practical capacity,
* queues grow even under good sequencing,
* due-date misses occur across almost all priority classes,
* simulation shows lateness remains high even with improved dispatching,
* overtime or extra capacity sharply improves KPIs.

Countermeasure: capacity increase, overtime, alternate routing, outsourcing, debottlenecking, preventive maintenance.

**Inherent process variability**

Evidence:

* high variance in actual processing time even for similar jobs,
* long-tailed breakdown repair times,
* large operator or material effects,
* unpredictable rework loops,
* lateness concentrated after rare high-impact events.

Countermeasure: predictive buffers, robust scheduling, maintenance strategy, quality improvement, duration model improvement.

The most useful method is counterfactual simulation: keep the same historical arrivals, routings, processing-time distributions, setup-time distributions, and breakdown patterns, but replace the dispatching rule. If performance improves materially, scheduling logic is a major cause. If performance does not improve, capacity or variability is likely dominant.

## 4. Advanced Data-Driven Scheduling Strategies

I would not deploy one monolithic algorithm immediately. I would develop a layered scheduling architecture:

1. real-time data foundation from process mining,
2. dynamic dispatching for local decisions,
3. rolling-horizon predictive scheduling for system-wide coordination,
4. setup optimization at bottlenecks,
5. WIP-controlled release,
6. disruption recovery logic.

Below are four concrete strategies.

---

### Strategy 1: Dynamic multi-factor dispatching rules

#### Core logic

Replace FCFS/EDD with a dynamic scoring rule at each machine. Every time a machine becomes available, score each feasible waiting job (j) for machine (m):

[
\text{Score}_{j,m,t}
====================

\alpha_1 \cdot \text{Urgency}*{j,t}
+
\alpha_2 \cdot \text{Priority Weight}*{j}
+
\alpha_3 \cdot \text{Bottleneck Protection}*{j}
+
\alpha_4 \cdot \text{Downstream Need}*{j,t}
-------------------------------------------

## \alpha_5 \cdot \text{Estimated Setup}_{prev,j,m}

## \alpha_6 \cdot \text{Processing Burden}_{j,m}

\alpha_7 \cdot \text{Downstream Congestion}*{j,t}
+
\alpha_8 \cdot \text{Aging}*{j,t}
]

The job with the highest score is dispatched.

A practical version could combine:

* due-date slack,
* critical ratio,
* customer priority,
* remaining processing time,
* remaining bottleneck time,
* estimated setup time from the previous job,
* downstream queue load,
* risk of tardiness,
* time already spent waiting,
* hot-job status,
* machine eligibility,
* operator availability.

A setup-aware apparent-tardiness-cost style rule could be used as the starting point, then extended with downstream load and learned weights.

#### How process mining informs the rule

Process mining provides:

* actual processing-time distributions by machine/job family,
* actual setup-time matrix by previous/current family,
* historical due-date performance,
* bottleneck identification,
* queue-time sensitivity,
* downstream starvation patterns,
* priority inversion evidence,
* historical outcomes of similar dispatch choices.

Weights should not be guessed. They should be calibrated using historical replay and discrete-event simulation. For example, if the process-mining analysis shows that setup time at `MILL-03` is the dominant cause of late jobs, setup penalty receives a higher weight there. If `QC-01` frequently starves before shipment deadlines, downstream need gets more weight for operations feeding QC.

#### Pathologies addressed

This strategy addresses:

* poor prioritization,
* local FCFS/EDD conflicts,
* avoidable setup thrashing,
* downstream starvation,
* high-priority job delays,
* excessive queue time before bottlenecks.

#### Expected KPI impact

Expected improvements:

* lower weighted tardiness,
* fewer priority inversions,
* shorter bottleneck queues,
* reduced setup time at critical machines,
* improved high-priority service level,
* reduced lead-time variability.

The main risk is overfitting or creating opaque rules. To avoid this, I would keep the scoring function interpretable and show dispatchers the reason for each recommendation, for example:

“Recommended JOB-7005 because tardiness risk is 82%, setup is 12 minutes, and it feeds a currently starved inspection queue.”

---

### Strategy 2: Predictive rolling-horizon scheduling

#### Core logic

Move from static planned durations to probabilistic, continuously refreshed scheduling.

Every fixed interval, for example every 30–60 minutes, and after major events such as breakdowns or hot-job releases, the system recomputes a rolling-horizon schedule for the next one to three shifts. The optimizer considers:

* current WIP,
* actual machine states,
* actual queue contents,
* remaining operations,
* due dates,
* priorities,
* machine eligibility,
* operator availability,
* setup-state of each machine,
* predicted processing times,
* predicted setup times,
* breakdown risk,
* downstream congestion,
* freeze windows to avoid excessive schedule nervousness.

The schedule is not a rigid full-year plan. It is a short-horizon decision support system with a frozen near-term zone, a flexible middle zone, and a forecast long-term zone.

#### Predictive models

From the event log, I would train models for:

**Processing duration**

[
P(T_{j,o,m} \mid \text{job attributes}, \text{machine}, \text{operator}, \text{shift}, \text{priority}, \text{routing})
]

**Setup duration**

[
P(S_{prev,j,m} \mid \text{previous job family}, \text{current job family}, \text{machine}, \text{operator})
]

**Remaining flow time**

[
P(C_j - t \mid \text{current state of job } j)
]

**Tardiness probability**

[
P(C_j > D_j \mid \text{current shop state})
]

**Breakdown risk and repair time**

[
P(\text{failure in next } h \text{ hours} \mid \text{machine history})
]

Predictive process monitoring uses historical event-log data to predict future states or outcomes of ongoing cases, including deviations, violations, and delays; those predictions can support proactive actions such as reallocating resources or adjusting schedules. ([Springer][6])

#### Scheduling method

Depending on scale, I would use one of the following:

* constraint programming for detailed short-horizon schedules,
* mixed-integer optimization for bottleneck-focused scheduling,
* simulation-optimization for uncertain conditions,
* metaheuristics such as tabu search, large neighborhood search, or genetic algorithms,
* hybrid optimization plus dispatching rules for real-time feasibility.

The objective function should not be pure makespan. It should combine:

[
\min
\left[
\lambda_1 \cdot \text{Total Weighted Tardiness}
+
\lambda_2 \cdot \text{Setup Time}
+
\lambda_3 \cdot \text{WIP Waiting Cost}
+
\lambda_4 \cdot \text{Schedule Nervousness}
-------------------------------------------

\lambda_5 \cdot \text{Bottleneck Productive Time}
\right]
]

#### How process mining informs the strategy

Process mining supplies:

* realistic routing variants,
* actual duration distributions,
* actual setup behavior,
* breakdown frequencies and repair-time distributions,
* machine calendars,
* operator effects,
* conformance deviations,
* historical hot-job arrival patterns,
* queue-time distributions.

#### Pathologies addressed

This strategy addresses:

* unpredictable lead times,
* inaccurate completion promises,
* poor disruption response,
* bottleneck overload,
* schedule nervousness,
* hidden capacity loss from setup and downtime.

#### Expected KPI impact

Expected improvements:

* better delivery-date promises,
* earlier identification of jobs at risk,
* fewer emergency expediting decisions,
* lower high-priority tardiness,
* better bottleneck utilization,
* reduced variance in lead time,
* improved schedule stability.

---

### Strategy 3: Setup-time optimization through intelligent batching and sequencing

#### Core logic

This strategy focuses specifically on sequence-dependent setups, especially at bottleneck machines.

The idea is not to create huge batches that delay urgent orders. Instead, the system creates **due-date-constrained micro-batches** of similar jobs when doing so reduces setup time without materially increasing tardiness risk.

For each bottleneck machine, use the mined setup matrix (S_m(a,b)) to sequence jobs in a rolling horizon. A simplified objective:

[
\min
\sum S_m(i,i+1)
+
\beta \sum w_j T_j
]

where the first term minimizes setup time and the second term penalizes tardiness.

Jobs are grouped by setup family, for example:

* same material,
* same fixture,
* same tool package,
* same CNC program family,
* same tolerance class,
* same heat-treatment requirements.

But grouping is allowed only if slack permits.

A practical rule:

“Batch similar jobs only while the predicted tardiness probability of any job in the batch remains below a defined threshold.”

For urgent jobs, the system may deliberately accept a larger setup. For normal jobs, it should avoid unnecessary high-changeover transitions.

#### How process mining informs the strategy

Process mining provides:

* actual job-to-job setup times,
* high-cost transition pairs,
* setup variability by operator and shift,
* setup overruns,
* bottleneck setup share,
* historical cases where batching helped or hurt,
* due-date impact of delaying jobs for batching.

The output is a setup knowledge base:

[
\text{Previous State} + \text{Next Job Attributes}
\rightarrow
\text{Expected Setup Time}
]

#### Pathologies addressed

This strategy addresses:

* high setup share at bottlenecks,
* setup thrashing,
* capacity loss from poor sequence choices,
* excessive WIP caused by slow bottleneck turnover,
* late jobs caused by avoidable changeovers.

#### Expected KPI impact

Expected improvements:

* reduced total setup hours,
* increased effective bottleneck capacity,
* lower flow time through bottleneck work centers,
* fewer setup overruns,
* lower tardiness where bottleneck capacity is the limiting factor,
* smoother downstream flow.

The main tradeoff is that batching can increase waiting time for some jobs. That is why the batching rule must be due-date constrained and priority aware.

---

### Strategy 4: WIP-controlled release and bottleneck-oriented flow control

#### Core logic

Do not release every job to the floor as soon as possible. Use the process-mining evidence to regulate release based on bottleneck workload and due-date risk.

A job is released only when:

* its release will not overload the bottleneck beyond a threshold,
* required materials/tools are available,
* the job has enough slack to meet its due date,
* the release supports bottleneck continuity,
* the release does not violate WIP caps at key work centers.

This resembles a workload-control or drum-buffer-rope approach:

* the bottleneck is the drum,
* a protective buffer ensures it is never starved,
* release is controlled so upstream WIP does not explode,
* downstream resources are synchronized to bottleneck output.

#### How process mining informs the strategy

Process mining supplies:

* WIP-versus-flow-time relationships,
* bottleneck identity by time period and product family,
* queue thresholds after which lead time becomes unstable,
* release-to-completion distributions,
* arrival variability by work center,
* starvation/blocking patterns,
* load profiles by future routing.

#### Pathologies addressed

This strategy addresses:

* high WIP,
* unpredictable lead times,
* bullwhip-like queue amplification,
* overloaded bottlenecks,
* downstream starvation,
* poor release discipline.

#### Expected KPI impact

Expected improvements:

* lower average WIP,
* shorter and more reliable lead times,
* less congestion before bottlenecks,
* improved due-date predictability,
* less firefighting,
* better visibility of true capacity constraints.

This strategy may temporarily reduce the number of jobs physically on the shop floor, but that is usually beneficial. High WIP is not the same as high throughput.

## 5. Simulation, Evaluation, and Continuous Improvement

### 5.1 Discrete-event simulation using mined process data

Before live deployment, I would build a discrete-event simulation digital twin of the job shop. The simulation should be parameterized from process-mining outputs, not manually guessed averages. Recent manufacturing research explicitly connects event-centric process mining with fast generation of digital and simulation models for complex manufacturing systems. ([ScienceDirect][7])

The simulation model would include:

**Jobs**

* release patterns,
* due-date distributions,
* priority classes,
* hot-job arrival process,
* job families,
* routing variants,
* rework probabilities.

**Operations**

* machine eligibility,
* processing-time distributions,
* setup requirements,
* setup-time matrix,
* operator requirements,
* transfer times.

**Resources**

* machine calendars,
* operator calendars,
* planned downtime,
* breakdown frequency,
* repair-time distributions,
* tool/fixture constraints where relevant.

**Control logic**

* current FCFS/EDD baseline,
* dynamic dispatching strategy,
* predictive rolling-horizon strategy,
* setup-optimized sequencing,
* WIP-controlled release,
* combinations of the above.

### 5.2 Baseline validation

The simulation must first reproduce historical performance under the current rules.

Validation targets:

* flow-time distribution,
* tardiness percentage,
* weighted tardiness,
* queue-time distribution by machine,
* WIP distribution by work center,
* machine utilization,
* setup share,
* breakdown impact,
* throughput per week,
* high-priority service level.

If the simulated baseline does not match historical behavior, the model is not ready for strategy testing.

### 5.3 Strategy comparison

I would run controlled experiments comparing:

1. current FCFS/EDD baseline,
2. dynamic multi-factor dispatching,
3. predictive rolling-horizon scheduling,
4. setup-optimized sequencing,
5. WIP-controlled release,
6. combined strategy.

Use common random numbers across scenarios so differences are caused by scheduling logic, not random variation.

For each scenario, report:

* percent tardy,
* mean and P90 tardiness,
* total weighted tardiness,
* median and P90 flow time,
* WIP average and peak,
* queue time by bottleneck,
* setup hours,
* productive utilization,
* idle/starvation time,
* throughput,
* schedule nervousness,
* hot-job service level,
* impact on normal jobs after hot-job insertion.

The best strategy is unlikely to dominate every KPI. I would use a Pareto view: for example, one policy may minimize tardiness but create excessive setup time; another may minimize setup time but increase urgent-job risk. The final policy should reflect business priorities.

### 5.4 Stress-test scenarios

I would test at least the following scenarios.

**High load**

Demand increases by 10%, 20%, and 30%. This reveals whether the strategy degrades gracefully or collapses.

**Frequent breakdowns**

Increase breakdown frequency or repair duration for bottleneck machines. This tests resilience.

**Hot-job surge**

Inject urgent jobs with tight due dates. Measure whether normal jobs suffer excessive collateral lateness.

**Due-date compression**

Reduce promised lead times. This tests whether current delivery promises are feasible.

**Product-mix shift**

Increase jobs requiring heavy milling, heat treatment, or special inspection. This tests bottleneck migration.

**Setup-intensive mix**

Increase transitions between dissimilar job families. This tests setup-aware policies.

**Operator shortage**

Remove one skilled operator from bottleneck shifts. This tests whether the true constraint is machine or labor.

**Preventive-maintenance scenario**

Introduce planned downtime on bottleneck machines and compare against unplanned failure losses.

### 5.5 Deployment approach

I would not switch the whole shop at once. I would deploy in phases.

**Phase 1: Decision-support shadow mode**

The new scheduler recommends decisions, but dispatchers keep control. Compare recommended versus actual decisions and estimate lost opportunity.

**Phase 2: Bottleneck pilot**

Apply setup-aware dynamic dispatching to one bottleneck machine group, such as milling.

**Phase 3: Rolling-horizon scheduling**

Introduce system-wide schedule recommendations with freeze windows.

**Phase 4: WIP-controlled release**

Regulate release into the shop based on bottleneck workload and due-date risk.

**Phase 5: Automated exception handling**

Trigger rescheduling after breakdowns, hot-job arrivals, or severe predicted lateness.

### 5.6 Continuous monitoring and adaptation

Ongoing process mining should become the factory’s scheduling nervous system.

I would implement a live monitoring layer with dashboards for:

* current WIP by work center,
* predicted completion dates,
* tardiness risk by job,
* bottleneck queue hours,
* machine status,
* setup-state of each bottleneck,
* priority inversion alerts,
* breakdown recovery time,
* schedule adherence,
* actual versus predicted processing time,
* actual versus predicted setup time.

### 5.7 Drift detection

The system should automatically detect when assumptions become stale.

Examples:

**Processing-time drift**

Actual processing times for a family are now 20% higher than the trained model predicts.

**Setup-time drift**

A specific transition, such as aluminum to stainless on `MILL-03`, now takes much longer after tooling changes.

**Bottleneck drift**

The constraint moves from milling to heat treatment due to product-mix change.

**Priority drift**

Urgent orders increase from 5% to 18% of weekly volume.

**Breakdown drift**

Mean time between failures on `MILL-02` declines.

**Routing drift**

Jobs increasingly deviate from standard routing due to rework or alternate machine use.

When drift is detected, the system should trigger model retraining, scheduler-parameter recalibration, or a management review.

### 5.8 Closed-loop improvement framework

The continuous-improvement loop should operate as follows:

1. **Observe**
   Capture MES events in near real time.

2. **Mine**
   Update process maps, machine timelines, setup matrices, queue metrics, and tardiness diagnostics.

3. **Predict**
   Estimate remaining completion time, tardiness risk, bottleneck risk, and setup impact.

4. **Optimize**
   Generate dispatching and sequencing recommendations.

5. **Simulate**
   Test policy changes before applying them broadly.

6. **Deploy**
   Apply recommendations with human oversight and freeze-window discipline.

7. **Audit**
   Compare predicted versus actual outcomes.

8. **Adapt**
   Retrain models, tune weights, revise rules, and update scheduling policies.

This creates a practical bridge between process mining and scheduling. Process mining explains what actually happened; predictive modeling estimates what is likely to happen; scheduling optimization chooses what should happen next; simulation tests whether the policy is robust before the shop depends on it.

---

## Executive Summary

Precision Parts should use process mining to build a factual, time-stamped picture of how jobs and machines actually behave: where jobs wait, which machines constrain throughput, how setups depend on job sequence, how breakdowns propagate, and why late jobs differ from on-time jobs.

The likely root problem is not a single bad machine or a single bad rule. It is the combination of local dispatching, limited visibility, sequence-dependent setups, uncertain durations, bottleneck overload, and reactive disruption handling.

The recommended path is a layered scheduling strategy:

1. **Dynamic multi-factor dispatching** to replace FCFS/EDD with real-time, setup-aware, priority-aware, bottleneck-aware decisions.
2. **Predictive rolling-horizon scheduling** using mined duration, setup, queue, and breakdown models.
3. **Setup-time optimization** through intelligent sequencing and due-date-constrained micro-batching.
4. **WIP-controlled release** to reduce congestion and stabilize lead times.

Before deployment, these strategies should be tested in a process-mining-parameterized discrete-event simulation. After deployment, ongoing process mining should continuously monitor KPI drift, bottleneck shifts, setup changes, schedule adherence, and model accuracy.
