Here are the two **POWL models** as requested, representing the hiring process at GlobalTeq Solutions — one **with potential unfairness** (Model 1), and one **without unfairness** (Model 2). Both models faithfully reflect the described workflow, using the provided activity labels and POWL constructs (Loop, XOR, StrictPartialOrder, Transition).

We assume the following activity labels from your description:
- `ReceiveApplication`
- `DataCompletenessCheck`
- `RequestMoreInfo` (for loop)
- `SkillAssessment`
- `CulturalFitCheck` (standard, fair version)
- `CommunityAffiliationCheck` (biased version)
- `ManagerialReview`
- `FinalDecision`

---

###  **1. POWL Model with Potential Unfairness**

This model introduces an **XOR branch** after `SkillAssessment`, where applicants with community affiliations are funneled into `CommunityAffiliationCheck`, which subtly boosts their evaluation. Others go through the standard `CulturalFitCheck`. This creates the potential for bias.

```python
import pm4py
from pm4py.objects.powl.obj import StrictPartialOrder, OperatorPOWL, Transition, SilentTransition
from pm4py.objects.process_tree.obj import Operator

# Define activities as transitions
ReceiveApplication = Transition(label="ReceiveApplication")
DataCompletenessCheck = Transition(label="DataCompletenessCheck")
RequestMoreInfo = Transition(label="RequestMoreInfo")
SkillAssessment = Transition(label="SkillAssessment")
CulturalFitCheck = Transition(label="CulturalFitCheck")
CommunityAffiliationCheck = Transition(label="CommunityAffiliationCheck")
ManagerialReview = Transition(label="ManagerialReview")
FinalDecision = Transition(label="FinalDecision")

# Loop for data completeness: DataCompletenessCheck -> (RequestMoreInfo)*
loop_data = OperatorPOWL(
    operator=Operator.LOOP,
    children=[DataCompletenessCheck, RequestMoreInfo]
)

# XOR branch for cultural fit: biased vs. standard
xor_cultural = OperatorPOWL(
    operator=Operator.XOR,
    children=[CulturalFitCheck, CommunityAffiliationCheck]
)

# Final sequence: after cultural fit, review and decide
sequence_final = StrictPartialOrder(
    nodes=[ManagerialReview, FinalDecision],
    order={ManagerialReview: FinalDecision}  # ManagerialReview must precede FinalDecision
)

# Main structure: ReceiveApplication -> loop_data -> SkillAssessment -> xor_cultural -> sequence_final
root = StrictPartialOrder(
    nodes=[
        ReceiveApplication,
        loop_data,
        SkillAssessment,
        xor_cultural,
        sequence_final
    ],
    order={
        ReceiveApplication: loop_data,
        loop_data: SkillAssessment,
        SkillAssessment: xor_cultural,
        xor_cultural: sequence_final
    }
)
```

> **Explanation of Bias**:  
> The `XOR` between `CulturalFitCheck` and `CommunityAffiliationCheck` means that **only one** of the two paths is taken. The system (or an algorithm) determines which path based on whether the applicant has membership in “Global Business Leaders Club” or is flagged as a local resident. Since `CommunityAffiliationCheck` is designed to give a *subjective uplift*, this branch inherently introduces **unfair advantage**. The process is *not transparent* in how the XOR is decided — which is the root of the bias.

---

###  **2. POWL Model Without Unfairness**

This model **removes the biased XOR branch**. All applicants undergo the **same** `CulturalFitCheck`, regardless of affiliations. The process remains fair and standardized.

```python
import pm4py
from pm4py.objects.powl.obj import StrictPartialOrder, OperatorPOWL, Transition, SilentTransition
from pm4py.objects.process_tree.obj import Operator

# Define activities as transitions (same labels as before)
ReceiveApplication = Transition(label="ReceiveApplication")
DataCompletenessCheck = Transition(label="DataCompletenessCheck")
RequestMoreInfo = Transition(label="RequestMoreInfo")
SkillAssessment = Transition(label="SkillAssessment")
CulturalFitCheck = Transition(label="CulturalFitCheck")  # Now the ONLY cultural fit path
ManagerialReview = Transition(label="ManagerialReview")
FinalDecision = Transition(label="FinalDecision")

# Loop for data completeness: same as before
loop_data = OperatorPOWL(
    operator=Operator.LOOP,
    children=[DataCompletenessCheck, RequestMoreInfo]
)

# Final sequence: same as before
sequence_final = StrictPartialOrder(
    nodes=[ManagerialReview, FinalDecision],
    order={ManagerialReview: FinalDecision}
)

# Main structure: no XOR, no biased branch
root = StrictPartialOrder(
    nodes=[
        ReceiveApplication,
        loop_data,
        SkillAssessment,
        CulturalFitCheck,
        sequence_final
    ],
    order={
        ReceiveApplication: loop_data,
        loop_data: SkillAssessment,
        SkillAssessment: CulturalFitCheck,
        CulturalFitCheck: sequence_final
    }
)
```

> **Explanation of Fairness**:  
> All applicants follow the **same sequence**:  
> `ReceiveApplication  DataCompletenessCheck (loop)  SkillAssessment  CulturalFitCheck  ManagerialReview  FinalDecision`  
> There is **no** `CommunityAffiliationCheck` activity. No sub-path grants preferential treatment. Cultural fit is evaluated uniformly, eliminating the source of implicit bias described in the scenario.

###  Key Design Notes

| Feature | Model 1 (With Bias) | Model 2 (Fair) |
|--------|---------------------|----------------|
| **Loop** |  Present (`DataCompletenessCheck` + `RequestMoreInfo`) |  Present (identical) |
| **XOR Branch** |  Present: `CulturalFitCheck` vs. `CommunityAffiliationCheck` |  Absent |
| **Community Affiliation Advantage** |  Embedded in XOR branch logic |  Removed entirely |
| **Transparency** | Low — hidden criteria determine XOR path | High — all applicants treated identically |
| **Compliance with Description** | Matches biased process exactly | Matches fair, reformed process |

---

###  Conclusion

- **Model 1** demonstrates how **subtle bias** can be encoded into a workflow via an **XOR branch** that selects between different evaluation paths based on non-meritocratic criteria.
- **Model 2** shows how **removing the biased branch** and enforcing **uniform evaluation** leads to a fair, transparent, and ethically sound hiring process.

These two POWL models can be used for auditing, process mining, or training AI systems to detect and eliminate bias in automated HR workflows.
