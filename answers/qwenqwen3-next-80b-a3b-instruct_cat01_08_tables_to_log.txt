To produce a comprehensive **event log suitable for process mining**, we will:

1. Use `order_id` as the **Case ID**.
2. Derive **activities** from each relevant table, ensuring one row per event.
3. Use the **earliest available timestamp** for each activity (e.g., for multiple items picked in one order, each pick becomes its own event).
4. Enrich with **Resource/Actor** by joining with the `Resources` table.
5. Ensure **chronological order** within each case.
6. Include **additional attributes** like `customer_type`, `order_priority`, `item_description`, `carrier`, etc., where available and meaningful.
7. Handle **multiple deliveries/invoices/payments** per order by creating separate events.

###  Assumptions:
- **"Items Picked"** events are created per line item (not aggregated), since picking may happen at different times and by different staff.
- **"Order Dispatched"** and **"Order Arrived"** are separate events per delivery record.
- **"Invoice Issued"** is created per invoice (even if multiple invoices per order).
- **"Payment Received"** is created per payment (even if multiple payments per invoice).
- All timestamps are in UTC (`Z` suffix), so chronological ordering is valid.
- We assume **all steps** in the lifecycle are tracked (no missing steps).
- We enrich **Resource** using the `Resources` table via `picker_id`, `billing_person_id`, `sales_person_id`, `carrier_id` (carrier name used as resource).
