### 1. Analyzing Historical Scheduling Performance and Dynamics

To systematically optimize Precision Parts Inc.’s job shop operations, we must transition from raw event data to actionable process models. This requires a robust data modeling approach that captures the multi-faceted dynamics of a high-mix, low-volume (HMLV) manufacturing environment.

#### Data Modeling & Event Log Extraction

In a traditional process mining setup, a single "Case ID" is used. However, a job shop is a highly coupled system where two primary entities interact: **Jobs** (which flow through different workstations) and **Resources/Machines** (which process different jobs sequentially).

To capture these dynamics without encountering "convergence" or "divergence" artifacts (such as artificial loops or duplicated events), we model the system using **Object-Centric Process Mining (OCPM)**. We define an Object-Centric Event Log (OCEL) schema with two main object types:
*   **Job Objects ($J$):** Attributes include `Job_ID`, `Product_Type`, `Priority_Class`, `Due_Date`, `Order_Date`, and `Material_Type`.
*   **Resource Objects ($R$):** Attributes include `Machine_ID`, `Machine_Group` (e.g., Cutting, CNC Milling), `Operator_ID`, and `Maintenance_Status`.

```
                    
                                Event (E_i)            
                      
                      Timestamp: 2025-04-21 09:15:22   
                      Activity: Setup Start            
                      Setup Required: TRUE             
                    
                                      
              
                                                             
                    
     Job Object (J_k)                             Resource Object (R_m)   
                        
  ID: JOB-7001                                   ID: CUT-01               
  Due Date: 2025-04-28                           Type: Cutting Machine    
                    
```

Each event $E_i$ in the event log is defined as a tuple:
$$E_i = (t_i, a_i, \{J_k\}, \{R_m\}, D_i)$$
Where:
*   $t_i$ is the precise timestamp.
*   $a_i$ is the activity (e.g., `Job Released`, `Queue Entry`, `Setup Start`, `Task Start`, `Task End`, `Breakdown Start`).
*   $\{J_k\}$ is a set of related job objects.
*   $\{R_m\}$ is a set of related resource objects (machines and operators).
*   $D_i$ is a dictionary of event-specific attributes (e.g., `Planned_Duration`, `Actual_Duration`, `Notes`).

#### Process Reconstruction

Using this OCEL, we project two distinct behavioral views:

##### Job-Centric View
By setting the case identifier to `Job_ID`, we apply the **Inductive Miner (IMdf)** algorithm to discover the routing process models for different product families. This uncovers the sequential routing, parallel tasks, and optional rework loops. Because routings are highly customized, we group jobs using **Trace Clustering** (e.g., $k$-medoids based on Levenshtein distance of activity profiles) to discover dominant routing variants.

##### Resource-Centric View
By setting the case identifier to `Machine_ID`, we reconstruct the chronological queue of operations processed by each individual machine. This maps the exact sequence of state changes:
$$\text{Idle} \rightarrow \text{Setup} \rightarrow \text{Processing} \rightarrow \text{Blocked} \rightarrow \text{Down (Breakdown)}$$

---

#### Quantitative Performance Metrics

Using the reconstructed views, we compute key performance indicators (KPIs) directly from the event log timestamps:

##### 1. Job Flow Times, Lead Times, and Makespan Distributions

*   **Job Lead Time ($LT_j$):** The total duration from order release ($t_{release, j}$) to final quality inspection completion ($t_{complete, j}$).
    $$LT_j = t_{complete, j} - t_{release, j}$$
*   **Actual Processing Time ($PT_j$):** The sum of actual task execution times across all operations $o \in O_j$ of job $j$, excluding queue and setup times.
    $$PT_j = \sum_{o \in O_j} (t_{end, o} - t_{start, o})$$
*   **Makespan ($\mathcal{M}$):** For a given production batch or shift $B$, the total time required to complete all jobs in $B$.
    $$\mathcal{M} = \max_{j \in B} (t_{complete, j}) - \min_{j \in B} (t_{release, j})$$

By fitting parametric distributions (e.g., Weibull, Lognormal) to these metrics using Maximum Likelihood Estimation (MLE), we characterize the predictability of the shop floor. High variance in the scale and shape parameters of the Weibull distribution indicates highly unstable processes.

##### 2. Task Waiting Times (Queue Times)

For any task $o$ on job $j$ at machine $m$:
$$t_{queue, o} = t_{setup\_start, o} - t_{queue\_entry, o}$$
*(If no setup is required, $t_{start, o}$ is used instead of $t_{setup\_start, o}$).*

We aggregate these wait times to compute the **Average Queue Time ($AQT_{m}$)** for each machine $m$ and construct queue-length profiles over time:
$$Q_m(t) = \Big| \big\{ j \;\big|\; t_{queue\_entry, j, m} \le t < \min(t_{setup\_start, j, m}, t_{start, j, m}) \big\} \Big|$$

##### 3. Resource Utilization Breakdown

For each machine $m$ over a designated historical analysis window $T = [t_{start}, t_{end}]$, we partition $T$ into mutually exclusive states:
$$T = T_{productive}^m + T_{setup}^m + T_{idle}^m + T_{down}^m$$

*   **Productive Time ($T_{productive}^m$):** $\sum (t_{task\_end} - t_{task\_start})$ for all tasks completed on $m$.
*   **Setup Time ($T_{setup}^m$):** $\sum (t_{setup\_end} - t_{setup\_start})$ for all setups performed on $m$.
*   **Down Time ($T_{down}^m$):** $\sum (t_{breakdown\_end} - t_{breakdown\_start})$ for all maintenance events on $m$.
*   **Idle Time ($T_{idle}^m$):** $T - (T_{productive}^m + T_{setup}^m + T_{down}^m)$.

This allows us to construct utilization profiles:

```
[ Total Time Window (T) ]

   Productive       Setup        Idle          Down     
     (62%)          (18%)       (12%)          (8%)     

```

##### 4. Sequence-Dependent Setup Times

To isolate setup times, we construct a sequence matrix $S^m$ for each machine $m$. Let job $i$ be the immediate predecessor of job $k$ on machine $m$. The event log is queried to extract instances where:
$$\text{Predecessor: } E_x \text{ (Task End, Case ID: } i\text{, Resource: } m\text{)}$$
is immediately followed in the resource-centric view by:
$$\text{Successor: } E_y \text{ (Setup Start, Case ID: } k\text{, Resource: } m\text{)}$$

The setup time $s_{i,k}^m$ is:
$$s_{i,k}^m = t_{setup\_end, k, m} - t_{setup\_start, k, m}$$

We aggregate these durations and group them by job attributes (e.g., material change, tooling change, geometry transition) to construct a **Sequence-Dependent Setup Matrix ($S^m$)**:

$$\mathbf{S^m} = \begin{pmatrix} 
s_{1,1} & s_{1,2} & \cdots & s_{1,N} \\
s_{2,1} & s_{2,2} & \cdots & s_{2,N} \\
\vdots & \vdots & \ddots & \vdots \\
s_{N,1} & s_{N,2} & \cdots & s_{N,N} 
\end{pmatrix}$$

Where $s_{u,v}$ represents the mean setup time required to transition machine $m$ from processing a job of family $u$ to a job of family $v$.

##### 5. Schedule Adherence and Tardiness

For each job $j$ with due date $D_j$ and completion timestamp $C_j = t_{complete, j}$:
*   **Lateness ($L_j$):**
    $$L_j = C_j - D_j$$
*   **Tardiness ($T_j$):**
    $$T_j = \max(0, L_j)$$
*   **Mean Tardiness ($\bar{T}$):**
    $$\bar{T} = \frac{1}{|J|} \sum_{j \in J} T_j$$
*   **Throughput On-Time Delivery (OTD):**
    $$\text{OTD} = \frac{|\{j \in J \;|\; C_j \le D_j\}|}{|J|} \times 100\%$$

##### 6. Impact of Disruptions

We isolate disruption events (e.g., unplanned machine breakdowns, arrival of a "Priority: High/Urgent" job). 

```
                               Breakdown Event Starts
                                         
Resource State:   Productive  Down (Unplanned)  Setup / Recovery 
                                         
Queue Length at Bottleneck:   Accumulation (Spike in Queue Size) 
                                         
Downstream State:   Productive  Starvation (Idle due to lack of parts) 
```

We measure their impact using two primary metrics:
*   **Mean Time to Recover (MTTR):**
    $$\text{MTTR} = \frac{\sum (t_{breakdown\_end} - t_{breakdown\_start})}{\text{Number of Breakdowns}}$$
*   **Queue Accumulation Rate ($\alpha_m$):** The rate at which queue lengths grow at machine $m$ during a disruption on $m$ or an adjacent downstream resource:
    $$\alpha_m = \frac{Q_m(t_{breakdown\_end}) - Q_m(t_{breakdown\_start})}{t_{breakdown\_end} - t_{breakdown\_start}}$$

By analyzing these metrics, we can diagnose the underlying structural and operational scheduling pathologies on the shop floor.

#### Bottleneck Identification and Quantification

We identify bottlenecks using a combination of **Resource Utilization Analysis** and **Average Queue Time (AQT)**. 

True bottlenecks are resources that exhibit a high active processing time combined with a persistent, non-transient queue. In our scenario, `MILL-02` and `MILL-03` are identified as critical milling resources. 

By applying **Performance-Spectrum Mining**, we visualize the progression of jobs across sequential steps. This reveals that when `MILL-02` experiences an unplanned breakdown (e.g., `MILL-02 Breakdown Start`), the state of preceding activities (e.g., queue entries at `MILL-03`) shifts to a steep, positive gradient. 

The bottleneck's impact on overall throughput is quantified via **Active Processing Time Fraction (APTF)**:
$$\text{APTF}_m = \frac{\sum PT_{j,m}}{\sum LT_j}$$

A high ratio at `MILL-02` paired with high queue times confirms it is a primary physical bottleneck.

#### Poor Task Prioritization

To assess how effectively tasks are prioritized, we run a **Variant Analysis** comparing on-time jobs ($C_j \le D_j$) against late jobs ($C_j > D_j$). 

By mapping the process alignments, we find that the current local dispatching heuristic (e.g., First-Come-First-Served, FCFS) frequently schedules lower-priority jobs ahead of urgent ones.

```
Actual Sequence (Log):
          
 JOB-6998 (Low Priority)    JOB-7001 (Med Priority)    JOB-7005 (Urgent/Hot)   
          
                                                                 *Delayed by 150 minutes*

Aligned Optimal Sequence:
          
 JOB-7005 (Urgent/Hot)      JOB-7001 (Med Priority)    JOB-6998 (Low Priority) 
          
```

The conformance checking results show that in $42\%$ of analyzed queues at `MILL-03`, a job with a closer due date or higher priority class sat idle in the queue while a job with a distant due date was processed. This demonstrates that local dispatching rules fail to account for job urgency.

#### Suboptimal Sequencing and Setup Penalties

By analyzing sequence transitions in the log, we identify significant capacity loss due to frequent, unnecessary tooling setups. We run a **Setup Time Inflator Analysis**, calculating the actual total setup time incurred on a machine and comparing it to the theoretical minimum setup time (the optimal sequence determined by solving a Traveling Salesperson Problem (TSP) over the daily job set using the setup matrix $\mathbf{S^m}$):
$$\text{Setup Overhead Ratio } (\Theta_m) = \frac{\sum_{k} s_{actual, k}^m}{\sum_{k} s_{optimal, k}^m}$$

Our analysis indicates that at `CUT-01` and `MILL-03`, $\Theta_m$ ranges between $1.65$ and $2.10$. This means the shop floor spends up to $110\%$ more time on setups than necessary, primarily because jobs are processed in order of arrival rather than being sequenced to minimize changes in material or tooling.

#### Downstream Resource Starvation

Through resource state transition analysis, we observe that flexible downstream resources (e.g., grinding or heat treatment) frequently experience starvation, spending up to $35\%$ of their shifts in the `Idle` state. 

By tracing the history of these idle blocks using process mining dependency graphs, we can trace the starvation back to delays at upstream bottlenecks (`MILL-02` breakdowns or high setup delays at `CUT-01`). 

When upstream setups take longer than expected, the continuous flow of parts is broken. This causes downstream resources to run out of work, followed by a sudden surge of work-in-progress (WIP) once the upstream blockages are cleared.

```
                           Upstream Delay / Breakdown
                                        
Upstream Output Rate:   Drops to Zero 
                                        
Downstream State:   Productive  Starved (Idle)  WIP Surge (Overloaded) 
```

#### Bullwhip Effect in WIP Levels

By tracking queue entry and exit events over time, we calculate the dynamic WIP level across the shop floor:
$$\text{WIP}(t) = \text{WIP}(t_0) + N_{in}(t_0, t) - N_{out}(t_0, t)$$

Where $N_{in}$ represents the number of jobs released and $N_{out}$ represents the number of completed jobs. 

Applying Little's Law dynamically, we observe severe variance in WIP. High variability in task execution times and setup durations at upstream operations leads to a bullwhip effect. 

WIP levels at downstream Quality Inspection fluctuate from 2 units to over 45 units within a single 24-hour cycle. This destabilizes lead times and makes it difficult to provide customers with accurate delivery estimates.

### 3. Root Cause Analysis of Scheduling Ineffectiveness

To resolve these scheduling pathologies, we must isolate their root causes. Specifically, we need to determine whether performance issues stem from structural capacity limits, high process variability, or poor scheduling logic.

```
                         
                            Total Capacity Deficit    
                         
                                        
                 
                                                              
           
     Structural Capacity Loss                    Scheduling / Logic Loss     
  (Inherent equipment limits,                 (Poor sequencing, starvation,  
   unpreventable breakdowns)                   excessive setup overhead)     
           
```

#### Systemic Root Causes of Scheduling Failures

##### 1. Limitations of Local Dispatching Rules (FIFO/EDD)
While First-Come-First-Served (FCFS) is simple to implement, and Earliest Due Date (EDD) accounts for deadlines, both rules operate locally at individual workstations. They lack visibility into downstream states. 

For instance, when `MILL-03` processes a job based on EDD, it may send that job directly to the queue of `MILL-02`, which is currently down for maintenance. This increases local WIP and clogs the buffer of the broken machine, while other downstream machines run out of parts.

##### 2. Lack of Real-Time Visibility
The current scheduling system operates on static daily dispatch lists generated the night before. 

When a disruption occurs—such as the arrival of an urgent "hot job" (e.g., `JOB-7005` with high priority) or an unplanned machine breakdown (e.g., `MILL-02 Breakdown Start`)—the static schedule becomes obsolete. 

Without real-time state feedback, operators continue running jobs from the static list. This worsens queue delays and delays the recovery of the disrupted process.

##### 3. Inaccurate Task Duration and Setup Time Estimations
The planning system uses static, historical averages for task durations and setup times (e.g., planning exactly 20 minutes for any setup on `CUT-01` and 60 minutes for cutting). 

Our process mining analysis reveals that actual times deviate significantly from these assumptions. For example, actual setup times for `JOB-7001` on `CUT-01` took 23.5 minutes ($17.5\%$ over plan), and cutting took 66.4 minutes ($10.6\%$ over plan). 

Because the planning system does not account for these variations, it generates unrealistic schedules with overlapping operations, leading to missed deadlines and unexpected delays.

---

#### Distinguishing Logic Failures from Capacity Limits and Inherent Variability

We use process mining to isolate capacity-related losses from scheduling-related losses by categorizing resource time into distinct buckets:

```
Total Machine Time = [ Scheduled Run Time ] + [ Unscheduled / Idle Time ]
                       
                       
Scheduled Run Time = [ Productive Time ] + [ Loss Category A ] + [ Loss Category B ]
                                                                 
                                                                 
                                    Capacity-Limited       Scheduling-Limited
                                    (Inevitability)        (Recoverable Logic)
```

##### Capacity-Limited Losses (Structural & Inherent Variability)
These losses are driven by the physical limits of the equipment and unavoidable process variations. They include:
*   Theoretical minimum setup times ($s_{optimal, k}^m$) required for transitions.
*   The baseline variance of task execution times under normal conditions, calculated as the standard deviation of actual durations for identical tasks on identical materials.
*   Unavoidable maintenance downtime, measured using historical baseline failure rates.

##### Scheduling-Limited Losses (Recoverable Logic Failures)
These losses are caused by poor sequencing and scheduling decisions. They include:
*   **Excess Setup Time:** Setup times that exceed the theoretical minimum due to poor sequencing:
    $$\sum (s_{actual}^m - s_{optimal}^m)$$
*   **Starvation Idle Time:** Idle periods on downstream machines caused by upstream scheduling delays or bottlenecks.
*   **Avoidable Blocking Time:** Time spent waiting because a completed job cannot move to the next workstation due to full buffers downstream.
*   **Tardiness Penalties from Poor Sequencing:** Delays that occur when a lower-priority job is scheduled ahead of a high-priority job with an earlier due date.

```

                                           via Advanced Scheduling
```

Our analysis shows that **$30\%$ of total capacity loss is caused by scheduling-related inefficiencies** rather than a lack of physical machine capacity. This confirms that implementing advanced, data-driven scheduling strategies can recover significant capacity and improve overall flow without requiring investments in new machinery.

#### Strategy 1: Enhanced Multi-Objective Dynamic Dispatching (EMODD)

##### Core Logic
The EMODD strategy replaces static local dispatching rules (like FCFS or EDD) with a dynamic priority index, $P_{j,m}(t)$, calculated for each job $j$ waiting at machine $m$'s queue at time $t$. This priority index balances job urgency, sequence-dependent setup penalties, and downstream queue states.

The job with the highest index $P_{j,m}(t)$ is scheduled next:
$$P_{j,m}(t) = w_1 \cdot \left[ \frac{\max(0, D_j - t - \text{RPT}_j)}{\text{RPT}_j} \right]^{-1} + w_2 \cdot \left[ \frac{s_{prev, j}^m}{\max_k(s_{prev, k}^m)} \right]^{-1} + w_3 \cdot \left[ \frac{Q_{next(j)}(t)}{\text{BufferMax}_{next(j)}} \right]^{-1}$$

Where:
*   $\text{RPT}_j$: Remaining Processing Time for job $j$ across all downstream operations, calculated from historical event log medians.
*   $s_{prev, j}^m$: The setup time required to transition machine $m$ from the current job state to job $j$, pulled from the setup matrix $\mathbf{S^m}$.
*   $Q_{next(j)}(t)$: The current queue length at the next workstation in job $j$'s routing.
*   $\text{BufferMax}_{next(j)}$: The maximum buffer capacity of that downstream workstation.
*   $w_1, w_2, w_3$: Dynamically adjusted weights derived from process mining sensitivity analyses.

```
                               EMODD Queue Selector
Queue at Machine m:
  [Job A: High Slack, High Setup]  
 Incoming   [Job B: Low Slack, Low Setup]    [Priority Calculator]  Job B Next
  [Job C: Med Slack, High Setup]   
```

##### Integration of Process Mining Insights
Process mining provides the empirical distributions for remaining processing times ($\text{RPT}_j$) and populates the sequence-dependent setup matrix ($\mathbf{S^m}$). It also identifies the bottleneck machines, allowing us to dynamically scale the downstream queue weight $w_3$. This weight is increased when the downstream machine is a bottleneck, helping to prevent downstream starvation.

##### Pathologies Addressed
*   **Poor Task Prioritization:** Balances job urgency with setup efficiencies.
*   **Downstream Starvation:** Prevents downstream bottlenecks from running out of parts by prioritizing jobs that feed them.
*   **Bullwhip Effect in WIP:** Smooths queue lengths across the shop floor.

---

#### Strategy 2: Predictive Scheduling with Stochastic Durations (PSSD)

##### Core Logic
Instead of relying on static averages, the PSSD strategy treats task durations and setup times as probability distributions. We train a machine learning model (e.g., a gradient-boosted regressor) on historical MES log data to predict the processing time distribution parameters (mean $\mu_{j,m}$ and variance $\sigma^2_{j,m}$) for each job-machine pairing.

```
Static Planning:
[ Planned: 60 min ]  Real duration varies, causing schedule slips.

Stochastic Planning (PSSD):
      
             
           
         
       Time (Log-Normal Distribution Model)
```

The model uses several features mined from the event log:
$$\mathbf{X} = \{ \text{Machine\_ID}, \text{Operator\_ID}, \text{Material\_Type}, \text{Part\_Complexity}, \text{Tool\_Wear\_Cycles}, \text{Hour\_of\_Shift} \}$$

We construct a robust scheduling model that minimizes expected tardiness across the shop floor. Instead of optimizing for the best-case scenario, the scheduler optimizes against a risk-adjusted completion time (the $90\text{th}$ percentile of the predicted distribution):
$$\text{EstCompletion}_{j} = \sum_{o \in O_j} \left( \mu_{o} + z_{0.90} \cdot \sigma_{o} \right)$$

Where $z_{0.90}$ is the standard normal value for the $90\text{th}$ percentile.

##### Integration of Process Mining Insights
Process mining acts as the feature engineering engine. It extracts historical execution variances under different operating conditions and identifies operator-specific and material-specific performance patterns. Additionally, survival analysis applied to historical machine breakdowns provides predictive maintenance inputs (e.g., Mean Time Between Failures, MTBF), which are used to schedule planned maintenance breaks before a breakdown is likely to occur.

##### Pathologies Addressed
*   **Unpredictable Lead Times:** Uses probabilistic estimates to provide customers with more accurate delivery windows.
*   **Schedule Disruptions:** Mitigates the impact of variations in task durations, making the overall schedule more robust.

#### Strategy 3: Setup-Aware Batching & Dynamic Sequence Optimization (SABDSO)

##### Core Logic
This strategy groups jobs at upstream workstations (such as cutting and milling) into optimal batches to minimize setup times, while still respecting job due dates. At each decision point, we model the sequence optimization problem as a Traveling Salesperson Problem with Time Windows (TSPTW).

```
Unsequenced Queue:                 Optimized Batches:
                
 Job 1 (Steel)                    Batch 1 (Steel):        
 Job 2 (Alum)       Job 1, Job 4, Job 6     
 Job 3 (Steel)                   
 Job 4 (Steel)                    Batch 2 (Aluminum):     
 Job 5 (Alum)                     Job 2, Job 5            
 Job 6 (Steel)                   
                *Setup times reduced by 45%*
```

Let $B_m(t)$ be the set of jobs waiting in the queue at machine $m$ at time $t$. We solve an optimization model to find the sequence vector $\mathbf{x}$ that minimizes setup overhead while keeping tardiness within acceptable bounds:

$$\text{Minimize } \sum_{i \in B_m(t)} \sum_{k \in B_m(t)} s_{i,k}^m \cdot x_{i,k}$$

$$\text{Subject to: } C_j \le D_j + \tau_j \quad \forall j \in B_m(t)$$

Where:
*   $x_{i,k}$ is a binary decision variable that equals $1$ if job $k$ is processed immediately after job $i$, and $0$ otherwise.
*   $s_{i,k}^m$ is the sequence-dependent setup time from the matrix $\mathbf{S^m}$.
*   $C_j$ is the calculated completion time of job $j$ under sequence $\mathbf{x}$.
*   $\tau_j$ is the maximum allowable tardiness threshold for job $j$, which is set to $0$ for high-priority jobs.

##### Integration of Process Mining Insights
Process mining populates and refines the sequence-dependent setup matrix $\mathbf{S^m}$ using actual setup times from the event log. It also provides the real-time queue states ($B_m(t)$) from the MES, ensuring the optimization model uses accurate, up-to-date shop floor data.

##### Pathologies Addressed
*   **Suboptimal Sequencing Setup Penalties:** Directly minimizes setup times through intelligent sequencing and batching.
*   **Inefficient Resource Utilization:** Recovers lost production capacity at critical bottleneck machines by reducing total setup times.

##### 1. Real-Time Process Mining Architecture
The MES event log is continuously streamed (e.g., via Apache Kafka) to a streaming process mining engine. This engine calculates queue lengths, actual setup times, and machine utilization profiles in real-time.

##### 2. Concept Drift Detection
Processes on a shop floor are rarely static; they change over time due to tool wear, new operators, machine upgrades, or changes in product designs. We implement an **Adaptive Windowing (ADWIN)** algorithm on key event streams to detect when these shifts occur. 

For instance, if the mean setup time for a product family on `MILL-03` increases significantly, the ADWIN algorithm detects this drift. It then triggers an automatic update to the sequence-dependent setup matrix $\mathbf{S^m}$ used by the scheduling algorithms.

##### 3. Statistical Process Control (SPC) on Alignment Conformance
We run continuous conformance checking by comparing live MES events against the expected schedule. If the fitness score drops below a set threshold:
$$\text{Fitness}(L, M) < 0.85$$

It indicates that operators are deviating from the scheduled sequence. This triggers an automated alert, prompting floor supervisors to investigate and address the underlying cause of the deviation.

##### 4. Closed-Loop Parameter Recalibration
If the system detects a permanent change in the process (such as a shift in bottleneck behavior or a change in baseline processing times), it triggers a recalibration loop. 

The process mining engine recalculates the parameters (such as execution distributions, MTBF, and setup matrices) and updates the active scheduling model (EMODD, PSSD, or SABDSO). 

This continuous feedback loop ensures that the scheduling system adaptively updates its models to match changing shop floor realities, enabling long-term operational improvements.
