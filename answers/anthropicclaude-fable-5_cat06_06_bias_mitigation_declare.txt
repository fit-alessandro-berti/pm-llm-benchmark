## Rationale for Each Added Constraint

| Constraint | Activities | How It Mitigates Bias |
|---|---|---|
| `existence(BiasMitigationCheck)` | BiasMitigationCheck | Guarantees a fairness audit step occurs in **every** trace — no application can be processed end-to-end without it. |
| `absence(Reject_BasedOnDemographics)` | Reject_BasedOnDemographics | Outright forbids any activity that explicitly ties a rejection to demographic data. |
| `exactly_one(FinalDecision)` | FinalDecision | Prevents "decision shopping" — re-issuing decisions after a sensitive attribute becomes known. |
| `responded_existence(CheckApplicant*  BiasMitigationCheck)` | All sensitive checks | If a sensitive attribute is ever consulted, a bias check must exist somewhere in the trace (order-independent safety net). |
| `response(CheckApplicant*  BiasMitigationCheck)` | All sensitive checks | Strengthens the above: the bias check must occur **after** the sensitive attribute is disclosed, so the check actually audits the disclosure. |
| `precedence(BiasMitigationCheck  Approve/Reject)` | Decisions | No decision can be made unless a bias check has already happened — decisions are gated behind fairness verification. |
| `coexistence(Approve_Minority / Reject_Minority  ManualReview)` | Sensitive-demographic decisions | Any decision involving a protected group is always accompanied by a human-in-the-loop `ManualReview`, enforcing consistent extra scrutiny. |
| `precedence(ManualReview  *_Minority decisions)` | Sensitive-demographic decisions | The manual review must come **before** the decision, not as a retroactive rubber-stamp. |
| `chainprecedence(BiasMitigationCheck  Reject)` | Reject | The activity *immediately* before any rejection must be the bias check — the rejection cannot be the direct reaction to anything else (e.g., seeing race data). |
| `noncoexistence(SkipReview  Reject_Minority)` | SkipReview | Expedited paths that skip review can never appear in a trace where a minority applicant is rejected. |
| `nonchainsuccession(CheckApplicant*  Approve/Reject)` | All sensitive checks vs. decisions | Hard-blocks the most direct bias pattern: a decision occurring **immediately after** viewing a sensitive attribute. |
| `nonsuccession(CheckApplicantRace  Reject_BasedOnDemographics)` | Sensitive check vs. forbidden decision | Defense in depth: even eventual succession from race disclosure to a demographics-driven rejection is forbidden. |

## Summary

These constraints work together at three levels:

1. **Structural gating** (`precedence`, `chainprecedence`): Decisions are only reachable *through* fairness checkpoints.
2. **Temporal separation** (`nonchainsuccession`): Decisions can never be the immediate reaction to learning a sensitive attribute, breaking the direct causal/temporal link that signals biased decision-making.
3. **Accountability** (`coexistence`, `responded_existence`, `existence`): Every trace touching sensitive data or sensitive demographics carries an audit trail (`BiasMitigationCheck`, `ManualReview`), making biased shortcuts non-conformant and detectable during conformance checking.
