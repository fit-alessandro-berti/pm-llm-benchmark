To solve this, we will use a Common Table Expression (CTE) to identify the "bad" `case_id`s that meet both the sequence and timing criteria, and then perform a `LEFT JOIN` or `NOT IN` filter to exclude those cases from the final result.

We use the `LEAD()` window function to look ahead at the next two events in the sequence for each case.

```sql
WITH BadCases AS (
    -- Step 1: Identify cases that have the specific sequence
    -- Step 2: Filter those where the duration between event 2 and 3 > 5 days
    SELECT DISTINCT case_id
    FROM (
        SELECT 
            case_id,
            activity,
            timestamp,
            LEAD(activity, 1) OVER (PARTITION BY case_id ORDER BY timestamp) AS next_activity,
            LEAD(activity, 2) OVER (PARTITION BY case_id ORDER BY timestamp) AS next_next_activity,
            LEAD(timestamp, 1) OVER (PARTITION BY case_id ORDER BY timestamp) AS next_timestamp,
            LEAD(timestamp, 2) OVER (PARTITION BY case_id ORDER BY timestamp) AS next_next_timestamp
        FROM event_log
    ) AS sequences
    WHERE activity = 'Approve Purchase Requisition'
      AND next_activity = 'Create Order'
      AND next_next_activity = 'Confirm Order'
      AND next_next_timestamp - next_timestamp > INTERVAL '5 days'
)

-- Step 3: Return all events from cases that are NOT in the BadCases list
SELECT *
FROM event_log
WHERE case_id NOT IN (SELECT case_id FROM BadCases);
```

### Explanation:
1.  **Window Functions (`LEAD`)**: We partition by `case_id` and order by `timestamp`. This allows us to look at the current row's activity and the two subsequent activities within the same case.
2.  **Sequence Filtering**: We filter for the specific sequence: `'Approve Purchase Requisition'`  `'Create Order'`  `'Confirm Order'`.
3.  **Timing Condition**: We calculate the difference between the `timestamp` of the 3rd event (`next_next_timestamp`) and the 2nd event (`next_timestamp`). If it is greater than 5 days, the `case_id` is marked as a "bad case."
4.  **Exclusion**: Finally, we perform a standard `SELECT *` on the `event_log` table, excluding any `case_id` identified in our `BadCases` CTE. This ensures that even if a case has other valid events, if it triggered the "bad" sequence, the entire case is removed from the output.
