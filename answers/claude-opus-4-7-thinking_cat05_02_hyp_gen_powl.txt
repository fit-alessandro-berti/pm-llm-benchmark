# Analysis of Anomalies in the Insurance Claim POWL Model

## 1. Identified Anomalies

### Anomaly A: Loop Between Evaluate (E) and Approve (P)
```python
loop = OperatorPOWL(operator=Operator.LOOP, children=[E, P])
```
The loop structure `* (E, P)` allows the process to repeatedly Evaluate  Approve  Evaluate  Approve. In the ideal flow, evaluation and approval should each occur **exactly once**. This structure permits multiple approvals for a single claim, which undermines control integrity (a claim could be "re-approved" after partial evaluation).

### Anomaly B: XOR That May Skip Customer Notification
```python
xor = OperatorPOWL(operator=Operator.XOR, children=[N, skip])
```
The XOR between `N` (Notify Customer) and a silent transition means customers may never be notified. This violates the intended flow where notification is mandatory and creates customer-service and compliance risks.

### Anomaly C: Partial Ordering Allowing Premature Closure
```python
root.order.add_edge(A, C)   # direct edge from A to C
# No edge enforcing xor -> C
```
A direct ordering edge from `A` (Assign Adjuster) to `C` (Close Claim) is added, but **no edge** forces `xor  C`. Thus, `C` can occur **concurrently with** (or before) the loop and XOR. A claim could be closed before evaluation, approval, or customer notification.

---

## 2. Hypotheses on Why These Anomalies Exist

| # | Hypothesis | Rationale |
|---|-----------|-----------|
| H1 | **Partial implementation of a rework policy** | The loop may have been introduced to allow re-evaluation on appeal/escalation, but no guard conditions were added to limit its iterations. |
| H2 | **Cost/time optimization bypassing notifications** | Business may have allowed skipping notification for small-value or internal claims, implemented as a blanket XOR rather than conditional logic. |
| H3 | **Miscommunication between departments** | The claim-closing team may have negotiated an early "close" path (e.g., for withdrawn claims) that was modeled as an additional order edge rather than a distinct XOR branch. |
| H4 | **Modeling tool limitations / human error** | The modeler may have forgotten to add `xor  C`, inadvertently leaving `C` concurrent with the rest of the process. |
| H5 | **Legacy system integration issues** | External systems may close claims autonomously (e.g., on customer withdrawal), and the model was loosened to reflect observed reality rather than desired behavior. |

## 3. Database Verification Queries

### Q1. Claims Closed Without Evaluation or Approval (Anomaly C)
```sql
SELECT c.claim_id
FROM claims c
JOIN claim_events ce_close
  ON ce_close.claim_id = c.claim_id AND ce_close.activity = 'C'
WHERE NOT EXISTS (
    SELECT 1 FROM claim_events ce
    WHERE ce.claim_id = c.claim_id
      AND ce.activity = 'E'
      AND ce.timestamp < ce_close.timestamp
)
   OR NOT EXISTS (
    SELECT 1 FROM claim_events ce
    WHERE ce.claim_id = c.claim_id
      AND ce.activity = 'P'
      AND ce.timestamp < ce_close.timestamp
);
```

### Q2. Claims Approved More Than Once (Anomaly A – Loop Abuse)
```sql
SELECT claim_id, COUNT(*) AS approval_count
FROM claim_events
WHERE activity = 'P'
GROUP BY claim_id
HAVING COUNT(*) > 1
ORDER BY approval_count DESC;
```

### Q3. Evaluation-Approval Oscillation (Loop Depth)
```sql
SELECT claim_id,
       COUNT(*) FILTER (WHERE activity = 'E') AS evals,
       COUNT(*) FILTER (WHERE activity = 'P') AS approvals
FROM claim_events
WHERE activity IN ('E','P')
GROUP BY claim_id
HAVING COUNT(*) FILTER (WHERE activity = 'E') > 1
    OR COUNT(*) FILTER (WHERE activity = 'P') > 1;
```

### Q4. Claims Where Customer Notification Was Skipped (Anomaly B)
```sql
SELECT c.claim_id, c.claim_type, c.claim_amount
FROM claims c
WHERE EXISTS (
    SELECT 1 FROM claim_events ce
    WHERE ce.claim_id = c.claim_id AND ce.activity = 'C'
)
AND NOT EXISTS (
    SELECT 1 FROM claim_events ce
    WHERE ce.claim_id = c.claim_id AND ce.activity = 'N'
);
```

### Q5. Rate of Skipped Notifications per Claim Type (supports H2)
```sql
SELECT c.claim_type,
       COUNT(*) AS total_closed,
       COUNT(*) FILTER (
         WHERE NOT EXISTS (
           SELECT 1 FROM claim_events ce
           WHERE ce.claim_id = c.claim_id AND ce.activity = 'N'
         )
       ) AS skipped_notifications
FROM claims c
WHERE EXISTS (SELECT 1 FROM claim_events ce
              WHERE ce.claim_id = c.claim_id AND ce.activity = 'C')
GROUP BY c.claim_type;
```

### Q6. Claims Closed Before the XOR/Notification Stage
```sql
SELECT c.claim_id
FROM claims c
JOIN claim_events cl_close ON cl_close.claim_id = c.claim_id AND cl_close.activity = 'C'
LEFT JOIN claim_events cl_notify ON cl_notify.claim_id = c.claim_id AND cl_notify.activity = 'N'
WHERE cl_notify.timestamp IS NULL OR cl_close.timestamp < cl_notify.timestamp;
```

### Q7. Close Happening Before or Concurrent with Assignment (extreme Anomaly C case)
```sql
SELECT c.claim_id, a.timestamp AS assigned_at, cl.timestamp AS closed_at
FROM claim_events a
JOIN claim_events cl
  ON a.claim_id = cl.claim_id
WHERE a.activity = 'A'
  AND cl.activity = 'C'
  AND cl.timestamp <= a.timestamp;
```

### Q8. Resource Analysis – Who Performs Premature Closures? (supports H3/H5)
```sql
SELECT ce.resource, COUNT(DISTINCT ce.claim_id) AS premature_closures
FROM claim_events ce
WHERE ce.activity = 'C'
  AND NOT EXISTS (
      SELECT 1 FROM claim_events x
      WHERE x.claim_id = ce.claim_id
        AND x.activity = 'P'
        AND x.timestamp < ce.timestamp
  )
GROUP BY ce.resource
ORDER BY premature_closures DESC;
```

## Summary

By running Q1–Q8, we can quantify **how often** each anomaly materializes in actual process executions. If Q2/Q3 return many rows, the loop is being actively exploited (H1). If Q4/Q5 show high skip rates for a specific `claim_type`, H2 is likely. If Q7/Q8 reveal premature closures concentrated with particular resources, H3 or H5 are supported. This data-driven triangulation turns modeling anomalies into actionable process-improvement insights.
