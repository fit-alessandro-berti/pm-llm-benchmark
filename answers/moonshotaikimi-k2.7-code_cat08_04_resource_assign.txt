# Data-Driven Optimization of TechSolve’s Resource Assignment Using Process Mining

Below is a comprehensive plan for analyzing TechSolve’s incident-management event log to diagnose assignment inefficiencies and redesign resource allocation. Each section maps directly to the questions in the brief and is intended to be actionable for an ITSM process-improvement initiative.

---

## 1. Analyzing Resource Behavior and Assignment Patterns

### 1.1 Agent and Tier Performance Metrics

From the event log, derive a **resource perspective** by joining each activity to the agent/tier executing it and the ticket attributes at that moment. Key metrics:

| Metric | Definition / Calculation | Why It Matters |
|---|---|---|
| **Workload distribution** | Number of tickets handled, active work time, and queued time per agent/tier over a period. Visualized as histograms or Lorenz curves. | Reveals overloaded vs. underutilized agents. |
| **Activity processing time** | `Work L1 End`  `Work L1 Start`, `Work L2 End`  `Work L2 Start`, etc., per agent and ticket category/skill. | Highlights fast/slow agents and training gaps. |
| **First-contact (L1) resolution rate** | % of tickets resolved without escalation from L1. | Indicates L1 empowerment and initial assignment quality. |
| **Escalation/reassignment rate per agent/tier** | # escalations / # assignments; # reassignments / # assignments. | Pinpoints agents or dispatchers making poor routing decisions. |
| **Skill-specific throughput** | Tickets resolved by each skill cluster, by tier. | Shows whether skill capacity matches demand. |
| **Queue wait time** | `Work X Start`  `Assign X` timestamp. | Captures delay before work begins. |
| **Touch count** | Number of distinct agents or assignment events per ticket. | A high touch count often predicts SLA risk. |
| **Utilization** | Active work time / available working time per agent. | Identifies burnout risk and spare capacity. |

For example, compare Agent A02 and Agent A05: both are L1, but if A02 escalates 70 % of firewall tickets while A05 escalates 30 %, training or skill differences should be investigated.

### 1.2 Process Mining Techniques to Reveal Actual Assignment Patterns

| Technique | Application to TechSolve |
|---|---|
| **Resource interaction analysis** | Build a matrix of handovers (e.g., L1  Dispatcher  L2  L3). Count frequency and mean wait time per handover pair. |
| **Social network analysis (SNA)** | Create a handover graph where nodes = agents/roles and edges = ticket transfers. Edge thickness = frequency; edge color = average delay. This exposes “reassignment highways” and bottlenecks. |
| **Role discovery** | Cluster agents by the set of activities/skills they actually perform. Compare discovered roles against the formal L1/L2/L3 structure. You may find a shadow “L2 doing L1 triage” role. |
| **Dotted chart / timeline analysis** | Plot each ticket as a timeline of activities colored by agent/tier. Quickly visualizes queueing, parallel work, ping-pong reassignments, and SLA pressure points. |
| **Conformance checking** | Compare the actual event log against a target model (e.g., “L1 handles ticket  escalate only if required skill not in L1 skill set”). Deviations show where intended logic breaks down. |

**Comparison to intended logic:** The intended model is round-robin within tiers plus manual escalation. Process mining will likely reveal that the *actual* process is skill-agnostic, producing long reassignment chains and uneven queue lengths. The discovered model will show loops (Assign  Reassign  Reassign) that are not in the intended linear flow.

### 1.3 Skill Utilization and Skill-Level Mismatch

To evaluate whether specialized skills are used effectively:

1. **Skill-demand vs. skill-capacity matrix**  
   Rows = required skills (e.g., `App-CRM`, `Networking-Firewall`, `Database-SQL`). Columns = agents who possess the skill. Compare monthly demand for each skill against the available agent-hours of that skill.

2. **Skill-level utilization index**  
   For each ticket, compare the `Required Skill` complexity with the `Agent Tier` actually assigned. Flag cases where an L3 agent resolves basic password resets or where an L1 agent is assigned a firewall configuration ticket.

3. **Gini coefficient or standard deviation of workload per skill cluster**  
   A high Gini coefficient within a skill group means only a few specialists are carrying the load.

4. **Specialist “task-downgrading” rate**  
   % of an L2/L3 agent’s work that could be resolved by L1 based on historical L1 resolution patterns. If high, skills are being wasted.

## 2. Identifying Resource-Related Bottlenecks and Issues

### 2.1 Bottleneck Detection Approach

Using the mined model, overlay **performance data** (frequency, waiting time, processing time) on the process map. Bottlenecks appear as:

- Thick edges with long average waiting times between `Assign` and `Work Start`.
- Frequent loops around `Reassign` activities.
- High rework from L2 back to L1 or from L3 back to L2.

### 2.2 Specific Issues and Quantification

| Issue | Diagnostic Method | Quantification Examples |
|---|---|---|
| **Insufficient availability of specific skills** | Filter cases by `Required Skill`. Compute queue wait time (`Work Start`  `Assign`) per skill. | “Database-SQL tickets wait an average of 4.2 hours before an agent starts work vs. 0.8 hours for Access Management.” |
| **Reassignment delays** | For every `Reassign` activity, compute the time lost: cycle-time segment before and after reassignment. Compare tickets with 0, 1, 2+ reassignments. | “Each reassignment adds an average of 2.5 hours; 18 % of P2/P3 tickets are reassigned more than once.” |
| **Incorrect initial assignments** | Analyze first assignment event. If first assigned agent lacks the `Required Skill` or tier, flag as mismatch. | “34 % of P2 Network tickets are first assigned to L1 agents without `Networking-Firewall` skill, causing a 90 % escalation rate.” |
| **Overloaded / underutilized agents** | Utilization heatmap and workload distribution. | “Top 15 % of agents handle 42 % of all tickets; 8 agents sit below 45 % utilization.” |
| **SLA breach correlation** | Join assignment pattern features to SLA breach outcome. Use logistic regression or decision trees. | “Tickets reassigned at least once are 3.8× more likely to breach SLA; skill mismatch accounts for 28 % of P2 breaches.” |

### 2.3 Example Quantified Findings (Hypothetical, Derived from Log Patterns)

Based on the log snippet and typical patterns, an analysis might report:

- **Reassignment penalty:** 1.8 hours added per reassignment on average; 22 % of P2/P3 breaches involve two or more reassignments.
- **Skill bottleneck:** `Database-SQL` and `Security-IAM` tickets show queue wait times >3× the overall mean.
- **L1 escalation waste:** 25 % of escalated tickets were later resolved at L2 using only basic troubleshooting skills—suggesting L1 could have resolved them with better scripts or empowerment.
- **Workload imbalance:** Gini coefficient for ticket count per agent = 0.47, indicating a highly uneven distribution.

## 3. Root Cause Analysis for Assignment Inefficiencies

### 3.1 Potential Root Causes

| Root Cause | Evidence in the Log |
|---|---|
| **Round-robin ignores skills and workload** | Reassignment spikes immediately after `Assign L1/L2`; high queue wait times for scarce skills. |
| **Incomplete / stale agent skill profiles** | Agents assigned to tickets whose `Required Skill` they do not possess, or skilled agents not used for matching tickets. |
| **Poor initial categorization / skill identification** | Tickets changing `Required Skill` mid-life (e.g., `App-CRM`  `Database-SQL`) cause reassignment. |
| **No real-time visibility into workload** | Some agents receive multiple assignments while others are idle, visible in dotted charts. |
| **Over-cautious escalation criteria** | L1 agents escalate complex-looking tickets that peers resolve at L1, visible in variant comparison. |
| **Dispatcher judgment variability** | Different dispatchers produce different escalation/reassignment rates. |

### 3.2 Variant Analysis

Split cases into two populations:

1. **Smooth-flow cases:** 0 or 1 assignment, no reassignment, resolved within SLA.
2. **Troubled cases:** 2+ assignments, at least one reassignment, or SLA breach.

Compare them on:

- Ticket attributes: priority, category, required skill, channel.
- First assigned agent/tier and skill match.
- Dispatcher/agent identity.
- Time of day / day of week.
- Initial description keywords (if available).

**Insight example:** “Troubled cases are 5× more likely to start with an L1 agent lacking the required skill and 3× more likely to be created outside business hours.”

### 3.3 Decision Mining

Apply **decision mining** to the `Assign L1`, `Escalate L2`, and `Reassign` activities. Decision mining extracts the rules (or lack thereof) used by dispatchers/L1 agents. For example:

- If the rule “Priority = P2 AND Category = Network  assign to L2” appears only 40 % of the time, dispatcher discretion is inconsistent.
- If `Work L1 End` with note “Escalation needed” has no correlation with actual skill mismatch, L1 escalation training is the root cause.

## 4. Developing Data-Driven Resource Assignment Strategies

### Strategy 1: Skill- and Proficiency-Based Routing (SBR)

**Issue addressed:** Incorrect initial assignments, skill bottlenecks, specialists doing low-skill work.

**How it leverages process mining:**
- The skill-demand vs. capacity matrix identifies scarce skills.
- Agent processing-time analysis by skill reveals proficiency differences.

**Implementation:**
- Maintain a dynamic skill matrix: agent × skill × proficiency score (1–5).
- Route a ticket to the lowest-tier agent who possesses the required skill at sufficient proficiency.
- If multiple agents qualify, use secondary rules (workload, past resolution success for that ticket type).

**Data required:**
- `Required Skill` per ticket (reliable categorization).
- Agent skill profiles with proficiency levels.
- Historical resolution quality per skill/agent (e.g., rework, SLA).

**Expected benefits:**
- 25–40 % reduction in skill-mismatch reassignments.
- Better utilization of scarce specialists.
- Faster resolution for tickets requiring niche skills.

---

### Strategy 2: Workload-Aware and Availability-Based Dispatch

**Issue addressed:** Uneven workload distribution, queue delays, agent overload.

**How it leverages process mining:**
- Resource interaction and timeline analysis reveal queueing patterns.
- Utilization heatmaps identify overloaded agents.

**Implementation:**
- Build a real-time dispatch engine that considers:
  - Current open ticket count per agent.
  - Estimated remaining work per ticket.
  - Agent scheduled hours and availability status.
- Replace strict round-robin with a **weighted round-robin**: weight = available capacity and skill fit.

**Data required:**
- Real-time event stream (assignment, start, end, pause).
- Agent schedules and status.
- Estimated service times by ticket category/skill.

**Expected benefits:**
- More balanced workload (lower Gini coefficient).
- Reduced queue wait times, especially for P2/P3.
- Lower SLA breach rate due to overload.

---

### Strategy 3: Predictive Assignment and Escalation

**Issue addressed:** Unnecessary escalations, late identification of required skills, reassignments due to mis-categorization.

**How it leverages process mining:**
- Variant analysis identifies ticket characteristics associated with smooth resolution.
- Decision mining exposes when L1 could have resolved but escalated.

**Implementation:**
- Train a predictive model on historical tickets using features:
  - Channel, priority, category, description keywords, time of day.
  - Required skill, complexity indicator.
- At ticket creation, predict:
  - Required skill(s).
  - Probability of L1 resolution.
  - Expected resolution time.
- Route directly to the most appropriate tier/skill; provide L1 agents with a “likely escalation” warning and guided troubleshooting script.

**Data required:**
- Ticket description / keyword data.
- Historical feature-to-resolution outcomes.
- Feedback loop of actual assigned agent and resolution path.

**Expected benefits:**
- Higher first-contact resolution.
- Fewer “ping-pong” escalations.
- Proactive SLA risk flagging.

---

### Strategy 4: Dynamic Tier and Task Reallocation

**Issue addressed:** Structural skill bottlenecks and demand surges.

**How it leverages process mining:**
- Demand pattern analysis (by hour/day/skill) from the event log.
- Identification of low-skill tasks consumed by high-tier agents.

**Implementation:**
- Create a **flex-pool** of cross-trained agents who can shift between L1 and L2 tasks during demand spikes.
- Use simulation (see Section 5) to define thresholds for flex-pool activation.
- Reassign routine triage or password-reset tasks from L2/L3 back to L1 when L1 capacity is available.

**Data required:**
- Real-time demand by skill/tier.
- Cross-training records.
- Cost and SLA impact of tier-level reassignment.

**Expected benefits:**
- Absorb demand spikes without SLA breaches.
- Free specialists for complex work.
- Improve overall labor utilization.

## 5. Simulation, Implementation, and Monitoring

### 5.1 Business Process Simulation for What-If Analysis

Before deploying new assignment rules, build a **discrete-event simulation** calibrated from the mined event log:

1. **Model inputs from process mining:**
   - Arrival rates by priority, category, channel, and hour/day.
   - Service-time distributions per agent/tier/skill.
   - Resource pool definitions (agents, skills, schedules).
   - Current assignment logic (baseline) and proposed logic (scenarios).

2. **Scenarios to test:**
   - Baseline: current round-robin + manual escalation.
   - Skill-based routing only.
   - Workload-aware dispatch only.
   - Combined skill + workload + predictive routing.
   - Dynamic tier reallocation during peak hours.

3. **Simulation outputs:**
   - Average cycle time and queue wait time per priority.
   - SLA attainment rates.
   - Reassignment and escalation counts.
   - Agent utilization distributions.
   - Cost per ticket.

**Value:** Simulation estimates impact without disrupting live operations. For example, it might show that combining skill-based routing with workload awareness reduces P2 SLA breaches by 30 % while keeping utilization variance under 15 %.

### 5.2 Post-Implementation Monitoring with Process Mining Dashboards

After rollout, establish continuous process mining dashboards tracking:

| KPI / View | Purpose |
|---|---|
| **SLA attainment by priority and skill** | Direct measure of customer impact. |
| **Reassignment rate and reassignment delay** | Early warning of routing quality degradation. |
| **First-contact resolution rate** | Measures L1 empowerment and assignment accuracy. |
| **Queue wait time by required skill/tier** | Detects emerging skill bottlenecks. |
| **Agent utilization heatmap** | Ensures workload balance. |
| **Skill mismatch rate** | % of assignments where agent lacks required skill. |
| **Escalation appropriateness** | % of escalations later resolved by a peer at the lower tier. |
| **Conformance to new assignment rules** | Compare actual assignments vs. target logic. |

**Process views to maintain:**

1. **Live process map** filtered by priority and skill to see current flow and deviations.
2. **Handover social network** updated weekly to spot new reassignment highways.
3. **Variant explorer** comparing smooth vs. troubled tickets under the new rules.
4. **Resource timeline (dotted chart)** to validate real-time workload balancing.

**Continuous improvement loop:**
- Weekly review of dashboard KPIs.
- Monthly root-cause review of SLA breaches and reassignments.
- Quarterly retraining of predictive models and refresh of agent skill profiles.
- Periodic re-simulation as demand patterns or staffing change.
