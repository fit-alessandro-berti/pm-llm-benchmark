## Solution

The key idea is to use **window functions** (`LEAD`) partitioned by case to look at each event together with its two immediate successors. If an event is `'Approve Purchase Requisition'`, the next is `'Create Order'`, and the one after is `'Confirm Order'` — that's a direct succession. We then check the time gap between the 2nd and 3rd event of that sequence and exclude matching cases via an anti-join.

```sql
WITH ordered_events AS (
    SELECT
        case_id,
        activity,
        timestamp,
        LEAD(activity, 1) OVER (
            PARTITION BY case_id ORDER BY timestamp
        ) AS next_activity,
        LEAD(activity, 2) OVER (
            PARTITION BY case_id ORDER BY timestamp
        ) AS next_next_activity,
        LEAD(timestamp, 1) OVER (
            PARTITION BY case_id ORDER BY timestamp
        ) AS next_timestamp,
        LEAD(timestamp, 2) OVER (
            PARTITION BY case_id ORDER BY timestamp
        ) AS next_next_timestamp
    FROM event_log
),
violating_cases AS (
    SELECT DISTINCT case_id
    FROM ordered_events
    WHERE activity            = 'Approve Purchase Requisition'
      AND next_activity       = 'Create Order'
      AND next_next_activity  = 'Confirm Order'
      -- time between 'Create Order' and 'Confirm Order' > 5 days
      AND next_next_timestamp - next_timestamp > INTERVAL '5 days'
)
SELECT e.*
FROM event_log AS e
WHERE e.case_id NOT IN (SELECT case_id FROM violating_cases)
ORDER BY e.case_id, e.timestamp;
```

## How It Works

**Step 1 — `ordered_events` CTE:**
For every event, `LEAD(...)` fetches the activity and timestamp of the next and next-next events *within the same case*, ordered by `timestamp`. Because `LEAD` operates on adjacent rows of the partition, matching all three activities guarantees **direct succession** — no other event of that case can lie between them.

**Step 2 — `violating_cases` CTE:**
Filters for rows where the three activities line up exactly as:

```
'Approve Purchase Requisition'  'Create Order'  'Confirm Order'
```

and where the duration between the `'Create Order'` timestamp (`next_timestamp`) and the `'Confirm Order'` timestamp (`next_next_timestamp`) exceeds 5 days. `DISTINCT` ensures each case appears once, even if the pattern occurs multiple times.

**Step 3 — Final selection:**
An anti-join (`NOT IN`) removes *all* events of the violating cases, returning the complete event history only for cases that don't match the pattern + timing condition.

## Notes & Variations

- **Tie-breaking:** If two events in a case can share the same timestamp, make the ordering deterministic, e.g. `ORDER BY timestamp, rowid` or include an event-id column in the `ORDER BY`.
- **Anti-join alternative:** `NOT EXISTS` is equivalent and sometimes preferred (it's also null-safe):

```sql
SELECT e.*
FROM event_log AS e
WHERE NOT EXISTS (
    SELECT 1 FROM violating_cases v WHERE v.case_id = e.case_id
)
ORDER BY e.case_id, e.timestamp;
```

- **Interval check alternative:** `date_diff('day', next_timestamp, next_next_timestamp) > 5` also works in DuckDB, though the `INTERVAL` comparison handles partial days more precisely (e.g., 5 days and 3 hours counts as "more than 5 days").
