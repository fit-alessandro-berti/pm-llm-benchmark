### **1.1 Data Preprocessing & Integration**
**Approach to Unified Event Log Construction:**
To create a cohesive event log for process mining, I would **align all data sources along the *Case ID* (Vehicle-Day) and *Timestamp*** while standardizing attributes. The integration pipeline would involve:

- **Schema Harmonization:**
  - Map all events to a **standardized process mining schema**: `Case ID` (Vehicle-Day), `Timestamp`, `Activity` (normalized label), `Resource` (Vehicle ID + Driver ID), `Location`, `Package ID` (where applicable), and derived attributes (e.g., `Event Type Source`).
  - Example normalization:
    - GPS "Low Speed Detected"  `Traffic Delay` (Activity)
    - Scanner "Delivery Failed"  `Failed Delivery Attempt`
    - Maintenance "Unscheduled Stop"  `Breakdown Event`
    - Dispatch "Route Assigned"  `Route Planned`

- **Temporal Alignment & Event Correlation:**
  - **Upsample GPS data** to 1-second intervals (if not already) and **downsample to 10-30 second intervals** for efficiency, while preserving critical state changes (e.g., speed <5 km/h  `Idling` or `Stopped`).
  - **Link scanner events to GPS data** by timestamp and location (e.g., "Arrive Customer" at 50.81N, 6.12E at 08:25:10 matches GPS point at same location/time).
  - **Impute missing attributes**: For GPS events lacking activity labels, infer them using:
    - **Speed thresholds**: 0 km/h = `Stopped` (potential delivery/service), >0 = `Travel`.
    - **Location proximity**: Match GPS coordinates to planned stop locations from dispatch (using geospatial clustering with a 50m radius buffer).
    - **Sequence context**: E.g., a `Stopped` event between "Depart Customer" and next "Arrive Customer" = `Travel Delay`.

- **Case ID Generation & Hierarchical Modeling:**
  - Primary **Case ID = Vehicle-Day** (e.g., `V12-20241205`) for the *delivery route process*.
  - Secondary **sub-cases** for:
    - *Package-level* (e.g., `P9876-V12-20241205` for the delivery attempt flow: `Scan at Depot`  `Arrive Customer`  `Delivery Success/Failed`).
    - *Vehicle maintenance* (e.g., `V12-Maint-20241205` for unscheduled stops).
  - *Why*: Enables multi-level process mining (route-level vs. package-level).

- **Derived Attributes for Enrichment:**
  - **Distance traveled** between consecutive GPS points (Haversine formula).
  - **Fuel consumption estimate**: Use vehicle-specific fuel maps (e.g., 0.1L/km at 50 km/h, 0.15L/km at idle) applied to GPS speed/duration.
  - **Planned vs. Actual Sequence**: From dispatch, extract the *intended stop order* for each route; tag actual events with `Planned Position` (e.g., Stop 1, Stop 2) or `Deviation Flag`.

### **1.3 Conformance Checking: Comparing Actual vs. Planned**
**Approach:**
- **Align Actual and Planned Traces**:
  - The *planned trace* for a route is the sequence of stops from the dispatch system (e.g., `[Stop1, Stop2, ..., Stop35]`).
  - The *actual trace* is the sequence of delivery/service activities from the event log (e.g., `[Depart Depot, Arrive Stop1, Delivery Success, Depart Stop1, ...]`).
  - Use **sequence alignment algorithms** (e.g., Levenshtein distance) to map actual events to planned stops.

- **Conformance Metrics**:
  | **Metric**               | **Calculation**                                                                 | **Interpretation**                                                                 |
  |--------------------------|---------------------------------------------------------------------------------|-------------------------------------------------------------------------------------|
  | **Fitness**              | % of actual events that can be replayed in the planned model.                 | Low fitness = many unplanned activities (e.g., detours, breakdowns).              |
  | **Precision**            | % of planned behavior observed in reality.                                     | Low precision = planned routes are not followed (e.g., stops skipped/reordered).  |
  | **Escaping Edges**       | Number of transitions in actual log not allowed in planned model.              | Identifies sequence deviations (e.g., delivering Stop 3 before Stop 2).          |
  | **Missing/Extra Events** | Count of planned stops not visited or unplanned stops added.                   | Reveals skipped deliveries or unscheduled activities (e.g., maintenance).       |
  | **Timing Deviations**    | Mean absolute difference between planned and actual arrival times at each stop. | Quantifies schedule adherence (e.g., 15-minute average delay per stop).          |

**Key Deviations to Investigate:**
1. **Sequence Deviations**:
   - *Example*: Planned order: `[Stop1, Stop2, Stop3]`; Actual: `[Stop1, Stop3, Stop2]`.
   - *Root Cause*: Driver took a detour (e.g., due to traffic or personal preference).
   - *Impact*: Increased travel distance/time; potential missed time windows.

2. **Unplanned Stops/Activities**:
   - *Example*: `Unscheduled Stop` (maintenance) or `Traffic Delay` between Stop5 and Stop6.
   - *Root Cause*: Vehicle breakdown, congestion, or driver error.
   - *Impact*: Directly adds non-value-added time; may cause cascading delays.

3. **Timing Deviations**:
   - *Example*: Actual arrival at Stop10 is 45 minutes later than planned.
   - *Root Cause*: Cumulative delays from earlier stops, traffic, or long service times.
   - *Impact*: Reduced on-time delivery rate; may violate customer time windows.

4. **Failed Deliveries**:
   - *Example*: `Delivery Failed` at Stop5, requiring a re-delivery attempt later.
   - *Root Cause*: Customer not home, access issues, or incorrect address.
   - *Impact*: Doubles the cost (fuel, time) for that package; increases route variability.

**Tools for Conformance Checking:**
- **Token Replay** (in ProM): Simulate the planned model with actual traces to identify where tokens "escape" the model.
- **Alignments** (optimal alignment between actual and planned traces, considering costs for deviations).
- **Dashboard in Celonis**: Highlight deviations with color-coding (e.g., red for unplanned stops, yellow for timing delays).

- **Implementation**:
  - **Dynamic Time Window Slot Booking**:

  - **Proactive Notifications**:
    - Send **SMS alerts** 30 minutes before arrival with a **15-minute ETA window**.
    - Allow customers to **reschedule or provide delivery instructions** (e.g., "leave at doorstep").
  - **Predictive Failed Delivery Model**:
    - Use **logistic regression** to predict failed delivery risk based on:
      - Customer type (apartment vs. house).
      - Time of day.
      - Historical success rate for the address.
    - **Flag high-risk deliveries** for driver follow-up (e.g., call ahead).
  - **Incentivize Flexibility**:
    - Offer **discounts** for customers who choose off-peak time slots or allow "safe drop" locations.

## **5. Operational Constraints and Continuous Monitoring**

### **5.1 Accounting for Operational Constraints**
Each optimization strategy must respect **hard constraints** to ensure feasibility:

| **Constraint** | **Impact on Strategies** | **Mitigation Approach** |
|----------------|---------------------------|--------------------------|
| **Driver Working Hours (EU: 9h/day, 48h/week)** | Dynamic routing may extend route times. | Cap route duration at 8 hours; use **time-dependent vehicle routing** to respect breaks. |
| **Vehicle Capacity (Weight/Volume)** | Predictive maintenance may take vehicles offline. | Schedule maintenance during **low-demand days** (e.g., weekends) or use **backup vehicles**. |
| **Customer Time Windows** | Smart time windows must align with customer availability. | Use **customer preference data** (from past deliveries) to propose realistic slots. |
| **Traffic Regulations (e.g., no left turns, truck restrictions)** | Dynamic routing must comply with local laws. | Integrate **traffic rule layers** into the routing algorithm. |
| **Union Agreements (e.g., no performance-based pay)** | Gamification may require incentives. | Frame rewards as **team-based bonuses** or **safety awards** (e.g., for low idle time). |
| **Data Privacy (GDPR)** | Customer data used for notifications. | Anonymize customer data; obtain **explicit consent** for notifications. |

**Implementation Safeguards**:
- **Route Feasibility Checks**: Before deploying dynamic routes, validate that:
  - Total route time  8 hours (including breaks).
  - Total distance  200 km (vehicle range).
  - All time windows are met (use **constraint programming**).
- **Driver Buy-In**: Involve drivers in **pilot programs** to gather feedback and adjust thresholds (e.g., for idle time alerts).
- **Fallback Mechanisms**: If dynamic routing fails (e.g., no traffic data), revert to **static optimized routes**.

#### **B. Key Process Views for Sustainability**
1. **Process Model Evolution**:
   - **What to Track**: Monthly updates to the discovered process model (e.g., new variants, reduced loop frequencies for failed deliveries).
   - **Why**: Detect **process drift** (e.g., if a new traffic pattern emerges, the model will show more `Traffic Delay` events).
   - **Action**: Retrain dynamic routing models or adjust time windows.

2. **Variant Analysis Trends**:
   - **What to Track**: % of routes following the **most efficient variant** (e.g., no failed deliveries, minimal traffic delays).
   - **Why**: Measure adoption of optimizations (e.g., if dynamic routing increases the % of efficient variants).
   - **Action**: Investigate why some routes still follow inefficient variants (e.g., driver resistance, system errors).

3. **KPI Correlation Matrix**:
   - **What to Track**: Relationships between KPIs (e.g., does  ATPS correlate with  OTDR?).
   - **Why**: Identify **trade-offs** (e.g., rushing deliveries may reduce ATPS but increase failed deliveries).
   - **Action**: Adjust incentives to balance KPIs (e.g., reward both speed and accuracy).

4. **Anomaly Detection**:
   - **What to Track**: Outliers in KPIs (e.g., a route with OTDR = 50% when fleet average is 95%).
   - **Why**: Rapidly flag **emerging issues** (e.g., a new traffic jam, vehicle breakdown, or driver error).
   - **Action**: Automated alerts to dispatchers for intervention (e.g., reroute a struggling driver).

   - **Benchmark** against industry standards (e.g., Amazon’s last-mile OTDR = 98%).
