## **Process Mining Analysis & Optimization Strategy for TechSolve Solutions' IT Service Desk**

As a Process Improvement Consultant, I propose a structured, data-driven approach leveraging process mining to diagnose and resolve TechSolve's resource assignment inefficiencies. The event log is a rich source of truth to move beyond assumptions and reveal the real process dynamics.

### **1. Analyzing Resource Behavior and Assignment Patterns**

**A. Agent & Tier Performance Metrics:**
From the event log, I would first compute key performance indicators (KPIs) for each agent and support tier:
*   **Workload Distribution:** Number of active tickets per agent over time, queue lengths per tier.
*   **Activity Processing Times:** Average `Work L1 Start` to `Work L1 End` time per agent, and similarly for L2/L3. This identifies fast or slow performers.
*   **First-Call Resolution (FCR) Rate for L1:** Percentage of tickets where `Work L1 End` is followed directly by `Ticket Resolved`, without an `Escalate` activity. Low FCR suggests L1 skill or authority gaps.
*   **Ticket Type/Skill Specialization:** Correlation between agents and the `Required Skill` and `Ticket Category` they handle. Who handles most `App-CRM` tickets? Are they resolved by the same agent or bounced around?

**B. Revealing Actual Assignment Patterns via Process Mining:**
*   **Resource Interaction Analysis / Social Network Analysis (SNA):** I would create a "handover" network graph where nodes are agents and edges represent the frequency of ticket reassignments between them (e.g., Agent A05 handing off to Dispatcher for Agent B12). This visualizes the **actual collaboration and escalation pathways**, which may reveal:
    *   **Clusters:** Which agents frequently work together? Are there unintended silos or sub-teams?
    *   **Bottleneck Resources:** Which agents (like a central dispatcher or specific L2 specialist) are on the most reassignment paths?
    *   **Comparison to Intended Logic:** The intended process is L1 -> Dispatcher -> L2. The SNA might show tickets frequently going L1 -> Dispatcher -> L1 (reassignment within tier) or directly between L1 agents (informal handoffs), indicating a breakdown in the formal logic.

**C. Skill Utilization Analysis:**
*   **Skill-Task Matching Matrix:** I would cross-reference the `Agent Skills` with the `Required Skill` for each ticket they work on. This reveals:
    *   **Overqualified Work:** An L3 specialist with `Database-SQL` and `Security-IAM` skills spending time on tickets only requiring `Basic-Troubleshoot`.
    *   **Under-qualified Assignments:** An agent assigned a ticket requiring `Networking-Firewall` who lacks that skill, leading to reassignment.
    *   **Skill Pool Gaps:** High demand for a skill (e.g., `App-CRM`) but low supply of qualified agents, creating a systemic bottleneck.

Based on the analysis above, specific problems are quantified:

A high rate points to poor initial triage. **Impact:** Average `Timestamp` difference between `Assign L1` and the next `Reassign` or `Escalate` gives the direct delay introduced.
2.  **Unnecessary Tier Escalations:** Measure tickets escalated from L1 to L2/L3 but then resolved by an L2 agent with the same skill level as an L1 agent who previously handled it. This indicates L1 agents could have been empowered/trained.
3.  **Skill-Driven Bottlenecks:** Identify `Required Skill` values with the highest average queue time between `Escalate L2` and `Work L2 Start`. This points to critical skill shortages. **Impact:** Average resolution time for tickets with that skill vs. others.
4.  **Agent Overload/Underload:** Use workload metrics to identify agents consistently handling >X% more tickets than the tier average, leading to burnout and delays, versus agents with significantly lower utilization.
5.  **Correlation with SLA Breaches:** Join the event log with SLA breach flags. Perform root cause analysis to see if breaches cluster around tickets that experienced reassignment, were handled by overloaded agents, or involved a bottlenecked skill. **Quantify:** "Tickets with >2 reassignments have a 45% higher probability of breaching P2 SLA."

### **3. Root Cause Analysis for Assignment Inefficiencies**

The identified issues likely stem from several interconnected root causes:

*   **Overly Simplistic Assignment Rules:** The "round-robin within tiers" ignores skill requirements and real-time workload, as confirmed by the skill-mismatch and overload data.
*   **Static & Imprecise Skill Profiles:** Agent skill profiles may not reflect proficiency levels or recent upskilling. Decision mining on the reassignment decisions could show that dispatchers lack accurate data to make optimal choices.
*   **Poor Initial Triage at L1:** High escalation rates for certain categories indicate L1 agents may lack the scripts, knowledge, or permission to resolve them, as seen in FCR analysis.
*   **Lack of Real-Time Visibility:** Dispatchers cannot see agent queues or current tasks, leading to assignments that exacerbate bottlenecks.
*   **Variant Analysis for Root Causes:** Compare the process variants (sequences of activities) of "clean" tickets (minimal reassignment) to "chaotic" tickets. Analyze the conditions: Do chaotic tickets more often originate from phone calls? Involve a specific category? Get assigned to a newer agent? This pinpoints the specific factors leading to poor assignments.

### **4. Developing Data-Driven Resource Assignment Strategies**

**Strategy 1: Skill-Based Routing with Proficiency Weighting**
*   **Issue Addressed:** Initial skill-matched assignments and utilization of specialized skills.
*   **Leverages Process Mining:** The skill-task matching matrix and resolution time per skill-agent pair are used to build a **skill proficiency model**. An agent who resolves `App-CRM` tickets 30% faster than the average gets a higher proficiency score.
*   **Data Required:** `Agent Skills`, `Required Skill`, historical resolution times per agent-skill pair, current queue length per agent.
*   **Operation:** New tickets are routed to the available agent with the highest proficiency score for the `Required Skill`, not just any agent with that skill.
*   **Expected Benefits:** Reduced reassignments, lower resolution times, better leverage of expert knowledge.

**Strategy 2: Workload-Aware & Skill-Based Hybrid Algorithm**
*   **Issue Addressed:** Agent overload/underload and bottlenecks.
*   **Leverages Process Mining:** Real-time queue length data and historical processing times per agent-skill pair are fed into the model.
*   **Data Required:** Real-time agent availability, current number of active tickets per agent, estimated effort for ticket (based on category/skill).
*   **Operation:** The algorithm considers both **skill proficiency** and **current capacity**. It will prefer a slightly less proficient but underutilized agent over a highly proficient agent who is already overloaded, balancing efficiency with throughput.
*   **Expected Benefits:** More equitable workload distribution, reduced wait times in queues, improved SLA compliance.

**Strategy 3: Predictive Initial Triage & L1 Empowerment**
*   **Issue Addressed:** Unnecessary escalations and poor initial assignments.
*   **Leverages Process Mining:** Historical data on which tickets are frequently resolved at L1, and which are always escalated. Text mining on ticket descriptions (if available) can identify keywords that predict required skills or complexity.
*   **Data Required:** Ticket description text, historical outcomes (resolved at L1 vs. escalated), L1 agent success patterns.
*   **Operation:**
    1.  **Predictive Routing:** At ticket creation, a model predicts the most likely required skill and complexity, assigning it directly to a suitable L2 agent if complexity is high, bypassing futile L1 work.
    2.  **L1 Empowerment:** Create dynamic "cheat sheets" and decision trees for L1 agents based on historical successful resolutions for specific ticket types, increasing their FCR rate.
*   **Expected Benefits:** Drastic reduction in L1-L2 escalations and related delays, faster resolution for complex tickets, increased L1 effectiveness.
