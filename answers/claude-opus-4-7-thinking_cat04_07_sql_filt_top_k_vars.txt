```sql
WITH case_variants AS (
    SELECT 
        case_id,
        LIST(activity ORDER BY timestamp) AS variant
    FROM event_log
    GROUP BY case_id
),
variant_counts AS (
    SELECT 
        variant,
        COUNT(*) AS case_count
    FROM case_variants
    GROUP BY variant
),
top_k_variants AS (
    SELECT variant
    FROM variant_counts
    ORDER BY case_count DESC
    LIMIT 10  -- Replace 10 with desired K
),
top_k_cases AS (
    SELECT cv.case_id
    FROM case_variants cv
    JOIN top_k_variants tkv ON cv.variant = tkv.variant
)
SELECT el.*
FROM event_log el
JOIN top_k_cases tkc ON el.case_id = tkc.case_id
ORDER BY el.case_id, el.timestamp;
```

**Explanation of each step:**

1. **`case_variants`**: For each `case_id`, build the ordered sequence of activities using `LIST(activity ORDER BY timestamp)`, which is DuckDB's way of preserving order within an aggregation.

2. **`variant_counts`**: Group by the `variant` (list of activities) and count how many cases follow each unique variant.

3. **`top_k_variants`**: Sort variants by frequency in descending order and pick the top K (here, `K = 10` — change as needed).

4. **`top_k_cases`**: Identify the `case_id`s whose variant matches one of the top K variants.

5. **Final `SELECT`**: Return all original events from `event_log` that belong to these filtered cases, ordered by `case_id` and `timestamp` for readability.
