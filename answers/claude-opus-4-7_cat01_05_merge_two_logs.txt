# Merged Event Log: Order Fulfillment Process

## Matching Methodology

I matched events between Log A and Log B using the following criteria:
1. **Order ID** must be identical (all events here share `Order#1234`).
2. **Semantic equivalence** of event names (e.g., "Order Received"  "OrderReceived").
3. **Timestamp proximity** Ś within a ~2 second tolerance, extended slightly (to ~5s) where naming and sequence strongly suggest a match.
4. **Process logic** Ś events occurring in plausible sequence for fulfillment.

## Reasoning & Conflict Resolution

### Event 1: Order Received
- **Log A:** `10:00:00Z` / "Order Received"
- **Log B:** `09:59:58Z` / "OrderReceived" (2-second earlier)
- **Decision:** Merge. Naming is semantically identical; 2-second offset aligns with Log A's description of capturing *received* rather than *started* time. Log A timestamp chosen as primary (consistent with its designation as primary timeline); Log B's earlier timestamp preserved as secondary and likely reflects the actual origin.

### Event 2: Order Validated
- **Log A:** `10:01:30Z` / "Order Validated"
- **Log B:** `10:01:29Z` / "OrderValidation" (1 second earlier)
- **Decision:** Merge. Names are semantically equivalent; timestamp difference is within tolerance. Enriched with `u12`, `r10`, and inventory-check note.

### Event 3: Payment Processed
- **Log A:** `10:02:00Z` / "Payment Processed"
- **Log B:** `10:02:05Z` / "PaymentCheck" (5 seconds later)
- **Decision:** Merge (with note). Timestamp gap slightly exceeds the 2-second tolerance, but: (a) the note explicitly says "Payment gateway delay," explaining the offset; (b) no other candidate event exists for either side; (c) event types are semantically paired. This is a deliberate, justified extension of the tolerance.

### Event 4: Quality Check
- **Log B only:** `10:03:00Z` / "Quality Check"
- **Decision:** Keep as standalone. No counterpart in Log A. Log A likely does not track internal QC steps. Preserved with all Log B attributes; source flagged as `B only`.

### Event 5: Item Shipped
- **Log A:** `10:05:00Z` / "Item Shipped"
- **Log B:** `10:05:02Z` / "Shipping" (2 seconds later)
- **Decision:** Merge. Within tolerance; semantically equivalent. Enriched with courier handoff note.

### Event 6: Item Delivered
- **Log A only:** `10:20:00Z` / "Item Delivered"
- **Decision:** Keep as standalone. Log B has no delivery event Ś plausible since Log B appears warehouse/fulfillment-centered and may not receive downstream delivery confirmations. Preserved as `A only`.

## Summary of Conflicts & Resolutions

| Conflict Type | Resolution Strategy |
|---------------|----------------------|
| Timestamp offsets (2s) | Merged; used Log A as primary, retained Log B as secondary. |
| Timestamp offset >2s (Payment) | Merged based on corroborating note ("gateway delay") and absence of alternate matches. |
| Naming variations (e.g., "OrderReceived" vs. "Order Received") | Treated as equivalent after normalization (casing, spacing). |
| Log B-only event (Quality Check) | Included standalone; marked as B-origin. |
| Log A-only event (Item Delivered) | Included standalone; marked as A-origin. |
| Attribute enrichment | All Log B attributes (user_id, resource_id, notes) propagated to merged records. |

The final log preserves full auditability (both timestamps retained where relevant, origin tracked per event) while providing a unified, enriched view of the fulfillment process.
