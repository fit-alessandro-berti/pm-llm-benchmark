To solve this problem, we can use window functions (`LAG` and `LEAD`) to inspect the immediately preceding and succeeding events for each row within a case. This allows us to strictly enforce the "direct succession" requirement (i.e., no other events in between). 

Here is the DuckDB SQL query to achieve this:

```sql
WITH RankedEvents AS (
    SELECT 
        case_id,
        activity,
        timestamp,
        LAG(activity) OVER (PARTITION BY case_id ORDER BY timestamp) AS prev_activity,
        LEAD(activity) OVER (PARTITION BY case_id ORDER BY timestamp) AS next_activity,
        LEAD(timestamp) OVER (PARTITION BY case_id ORDER BY timestamp) AS next_timestamp
    FROM event_log
),
ExcludedCases AS (
    SELECT DISTINCT case_id
    FROM RankedEvents
    WHERE prev_activity = 'Approve Purchase Requisition'
      AND activity = 'Create Order'
      AND next_activity = 'Confirm Order'
      AND next_timestamp > timestamp + INTERVAL '5' DAY
)
SELECT e.*
FROM event_log e
WHERE NOT EXISTS (
    SELECT 1 
    FROM ExcludedCases ec 
    WHERE ec.case_id = e.case_id
)
ORDER BY e.case_id, e.timestamp;
```

### Explanation:
1. **`RankedEvents` CTE**: We partition the data by `case_id` and order it by `timestamp`. We then use `LAG` to grab the activity of the immediately preceding event and `LEAD` to grab both the activity and the timestamp of the immediately succeeding event. This guarantees we are checking for *direct succession* with no intervening events.
2. **`ExcludedCases` CTE**: We filter the ranked events to find the exact sequence: `'Approve Purchase Requisition'` $\rightarrow$ `'Create Order'` $\rightarrow$ `'Confirm Order'`. We simultaneously verify that the time elapsed between the `'Create Order'` event and the `'Confirm Order'` event is strictly greater than 5 days (`next_timestamp > timestamp + INTERVAL '5' DAY`). We use `DISTINCT` to ensure we only get a unique list of `case_id`s that violate the rule, even if the sequence occurs multiple times in the same case.
3. **Final Selection**: We query the original `event_log` table and use a `NOT EXISTS` clause to filter out any events belonging to the `case_id`s identified in the `ExcludedCases` CTE. Finally, we `ORDER BY` `case_id` and `timestamp` to ensure the output is properly sorted and readable.
