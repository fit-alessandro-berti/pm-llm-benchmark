## 1) Standard (normative) Hire-to-Retire logic (baseline)
A typical *Hire-to-Retire* “new employee lifecycle” would follow this general idea:

1. **Post job ad**
2. **Screen candidates**
3. **Conduct interviews**
4. **Make hiring decision** (choose a candidate / decide whether to hire)
5. **Onboard** the hired employee
6. **Add to payroll** (so the employee can be paid)
7. **Close the case** / finish the hiring matter

Usually, **Onboard comes before Payroll**, and **Close comes after the hiring/payroll steps**. Also, the “hiring decision” should be a gate that determines whether onboarding/payroll happen.

---

## 2) Anomalies in each model

### POWL Model 1
Edges:
- Post  Screen  
- Screen  Decide  
- Screen  Interview (parallel with Decide relative to Screen)  
- Decide  Onboard  
- Onboard  Payroll  
- Payroll  Close  

**Key anomalies**
1. **Interview is not properly sequenced before the hiring decision**
   - In a normative workflow, **interviews should occur before “Make_Hiring_Decision.”**
   - Here, `Screen  Decide` exists, so **Decide can happen immediately after screening**, even if interviews have not yet occurred (because `Screen  Interview` is separate and there is no constraint `Interview  Decide`).
   - That can produce executions like: *Post  Screen  Decide  Onboard …* while **Interview is still pending or never happens** (depending on execution semantics).

2. **Interview occurs in parallel with the decision rather than before it**
   - `Screen  Decide` and `Screen  Interview` together allow concurrency: interviews may happen while the decision is made, which is not typical for selection processes where the decision depends on interview outcomes.

**Severity**
- This is a **moderate-to-severe correctness issue**, because it can break the core dependency: *hiring decision should rely on interview results*.

---

### POWL Model 2
Edges:
- Post  Screen
- Post  Interview (so Interview can start right after posting, not after screening)
- Interview  Decide
- Decide  loop_onboarding
- loop_onboarding  xor_payroll
- xor_payroll  Close

Operators:
- `loop_onboarding = LOOP(Onboard, skip)`  
  (i.e., it can iterate the Onboard activity; exit via `skip`)
- `xor_payroll = XOR(Payroll, skip)`  
  (i.e., either run Payroll once **or** skip it)

**Key anomalies**
1. **Interview is allowed before screening**
   - There is an edge `Post  Interview` but only `Post  Screen` and **no edge `Screen  Interview`**.
   - So the process may execute *Post  Interview  Decide* while **Screening hasn’t happened** yet.

2. **Payroll is optional**
   - `xor_payroll = XOR(Payroll, skip)` allows **skipping “Add_to_Payroll”** entirely, yet the workflow still proceeds to `Close` (since `xor_payroll  Close`).
   - In hiring lifecycle logic, closing the case without adding to payroll is usually **not acceptable** if the case is truly about hiring/onboarding.

3. **Onboarding can repeat (loop)**
   - `LOOP(Onboard, skip)` allows **multiple onboardings** before payroll/close.
   - While re-onboarding or repeated onboarding steps can exist in real life, in a well-structured hire-to-retire case this is often not modeled as an unconstrained loop of the single coarse activity “Onboard_Employee” with no clear stopping condition tied to case completion. It can create **duplicate onboard executions**.

**Severity**
- This model has multiple substantial deviations:
  - (a) decision path can start without screening,
  - (b) payroll can be entirely omitted but the case can still close,
  - (c) onboarding can be repeated arbitrarily.
- Overall, these are **more severe** than Model 1 because they can fundamentally produce a “hired but never put on payroll / close anyway” type of outcome.

---

## 3) Which model aligns more with a normative Hire-to-Retire process?

### Choice: **Model 1 is closer to the normative process than Model 2.**

**Why Model 1 is better (despite its flaw):**
- It preserves the strongest core ordering constraints:
  - **Onboard  Payroll  Close** is correctly enforced.
  - It also makes sure **Onboard cannot happen before the hiring decision** (because `Decide  Onboard`).
- The main issue in Model 1 is that **Interview is not required before Decide**.
  - That can lead to decisions being made too early, but at least the process still follows the expected *post  screening  (decision/onboarding)  payroll  close* structure.

**Why Model 2 is worse:**
- It breaks multiple essential dependencies:
  - **Interview can happen before screening** (`Post  Interview` without `Screen  Interview`).
  - **Payroll can be skipped entirely** (`XOR(Payroll, skip)`), yet the process still reaches **Close**.
    - That directly violates integrity of the hiring lifecycle outcome.
  - **Onboarding can repeat** via a loop, which can create duplicate or inconsistent case behavior.

---

### Summary of anomaly impact
- **Model 1:** One major integrity issue (decision not constrained by interview), but lifecycle completion steps (onboardingpayrollclose) are structurally correct.
- **Model 2:** Multiple major integrity issues (screening dependency broken, payroll optional despite closure, onboarding repeat loop), making it less faithful to normative hire-to-retire case logic.

**Therefore, Model 1 aligns more closely with the normative version.**
