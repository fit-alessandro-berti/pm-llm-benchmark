### 1. Anomaly Identification and Hypothesis

#### Anomaly A: Late Shipment Confirmation (Case 1003)
*   **Observation:** For `case_id` 1003, the `Confirm Shipment` activity (event\_id 19) was marked with `late_confirmation=Y`.
*   **Hypothesis:** This suggests a delay in the logistics process, potentially leading to missed delivery windows, customer dissatisfaction, or failure to meet internal service level agreements (SLAs).
*   **Investigation Goal:** Quantify the frequency of late confirmations and identify the context (e.g., which logistics manager or system is responsible for the delay).

#### Anomaly B: Mismatch in Shipment Status (Case 1004)
*   **Observation:** For `case_id` 1004, the `Confirm Shipment` activity (event\_id 24) recorded `shipment_scheduled=N`.
*   **Hypothesis:** This indicates a failure in the logistics planning or scheduling stage. If the shipment was confirmed without being scheduled, it implies a gap in the internal planning workflow, which could lead to shipping errors or operational bottlenecks.
*   **Investigation Goal:** Identify all cases where the shipment scheduling status was marked as 'N' and correlate it with the actual shipment timing or resource.

#### Anomaly C: Process Timing and Efficiency (General)
*   **Observation:** Analyze the time elapsed between sequential mandatory steps (e.g., Register Order to Receive Payment).
*   **Hypothesis:** Significant time gaps between steps (e.g., Register Order to Credit Check) may indicate bottlenecks, manual delays, or systemic inefficiencies in the defined process flow.
*   **Investigation Goal:** Measure the duration of critical process steps to identify periods of unusual inactivity or excessive delays.

### 2. Proposed Investigative SQL Queries

The following queries are designed to validate the hypotheses and perform deeper, targeted investigations into the anomalies.

#### Query 1: Identifying Late Shipment Confirmations (Anomaly A)
This query focuses on the specific recorded failure in Case 1003 and attempts to find similar issues across the entire dataset.

```sql
SELECT
    oel.case_id,
    oel.activity,
    oel.timestamp AS event_time,
    oel.additional_info
FROM
    order_event_log oel
WHERE
    oel.additional_info LIKE '%late_confirmation=Y%'
ORDER BY
    oel.timestamp DESC;
```

#### Query 2: Identifying Scheduling Mismatches (Anomaly B)
This query isolates all instances where the shipment scheduling status was marked as 'N', highlighting potential planning failures.

```sql
SELECT
    oel.case_id,
    oel.activity,
    oel.timestamp,
    oel.additional_info
FROM
    order_event_log oel
WHERE
    oel.additional_info LIKE '%shipment_scheduled=N%'
ORDER BY
    oel.timestamp;
```

#### Query 3: Analyzing Process Duration and Bottlenecks (Anomaly C)
This query calculates the time difference between the start of the process (Register Order) and the completion of the process (Receive Payment) for each order, and also calculates the duration of specific key steps.

```sql
SELECT
    oel.case_id,
    oel.activity,
    oel.timestamp,
    -- Calculate time difference between consecutive events for flow analysis
    -- This requires complex window functions, but a simpler start/end check is more direct for flow analysis:
    -- (Requires self-joins or CTEs for accurate sequential duration, simplified here for aggregate context)
    CASE
        WHEN oel.activity = 'Register Order' THEN oel.timestamp
        WHEN oel.activity = 'Receive Payment' THEN oel.timestamp
        ELSE NULL
    END AS calculated_start_time,
    oel.timestamp AS calculated_end_time,
    EXTRACT(EPOCH FROM (oel.timestamp - COALESCE(CASE WHEN oel.activity = 'Register Order' THEN oel.timestamp ELSE NULL END))) AS duration_seconds
FROM
    order_event_log oel
WHERE
    oel.activity IN ('Register Order', 'Receive Payment')
ORDER BY
    oel.case_id, oel.timestamp;
```

#### Query 4: Identifying Resource Performance Correlation
This query attempts to correlate the time spent on specific high-value or critical steps with the specific roles involved (e.g., Credit Officers vs. Warehouse Clerks).

```sql
SELECT
    oel.case_id,
    r.role,
    r.department,
    oel.activity,
    AVG(EXTRACT(EPOCH FROM (oel.timestamp - LAG(oel.timestamp, 1, oel.timestamp) OVER (PARTITION BY oel.case_id ORDER BY oel.timestamp)))) AS avg_step_duration
FROM
    order_event_log oel
JOIN
    resources r ON oel.resource = r.resource_id
WHERE
    oel.activity IN ('Perform Credit Check', 'Validate Stock', 'Ship Goods')
GROUP BY
    oel.case_id, r.role, r.department, oel.activity
ORDER BY
    oel.case_id, avg_step_duration DESC;
```
