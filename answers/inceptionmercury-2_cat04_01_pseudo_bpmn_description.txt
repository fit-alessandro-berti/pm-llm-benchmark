### Overview of the Process  

The diagram models a **customerrequest handling workflow** that can be used by a salesorservice organization.  
Its purpose is to take an incoming request, validate it, determine whether any special handling (customization) is required, obtain any necessary internal approvals, produce the final invoice and finally notify the customer.  

The flow is expressed with BPMNstyle symbols:

| Symbol | Meaning in this model |
|--------|-----------------------|
| **Start Event** | The moment the request enters the system (e.g., via webform, email, or call). |
| **Task** | A unit of work performed by a person, system, or both. |
| **XOR (Exclusive) Gateway** | A decision point that routes the flow down **one** of several mutually exclusive branches based on a condition. |
| **AND (Parallel) Gateway** | A point where the flow **splits** into multiple branches that run **concurrently**; later a **Join** (also an AND) waits for **all** branches to finish before continuing. |
| **End Event** | The process terminates – the customer has been notified and all work is complete. |

Below is a stepbystep walkthrough of the process, the purpose of each element, and how the different paths interact.

---

## 1. Start  TaskA – “Receive Customer Request”

**Purpose:** Capture the request in the system (e.g., create a ticket, store the data).  
All subsequent actions depend on the information gathered here (customer ID, product, quantity, etc.).

---

## 2. XOR Gateway – “Check Request Type”

At this point the workflow decides **which** processing path to follow:

| Condition | Path |
|-----------|------|
| **Standard** request (no special configuration) |  **TaskB1** |
| **Custom** request (needs bespoke design, engineering, or special terms) |  **TaskB2** |

Only one branch can be taken, because a request cannot be both standard and custom at the same time.

---

## 3. StandardRequest Branch  

### 3.1 TaskB1 – “Perform Standard Validation”

Typical checks: completeness of the order, pricing rules, basic eligibility.  

### 3.2 AND Gateway – “Run Parallel Checks”

After the basic validation the process **splits** into two independent activities that can be executed at the same time:

| Parallel Task | What it does |
|---------------|--------------|
| **TaskC1 – Credit Check** | Queries the credit bureau or internal creditrisk system to ensure the customer can pay. |
| **TaskC2 – Inventory Check** | Looks up stock levels, reserving the required quantity if available. |

Because they are unrelated, running them in parallel shortens the overall lead time.

### 3.3 AND Join – “All Parallel Checks Completed”

The flow **waits** until **both** the credit and inventory checks finish (successfully or with a failure flag). Only then does it continue.

### 3.4 TaskD – “Calculate Delivery Date”

Using the results of the inventory check (availability) and possibly the credit check (payment terms), the system computes an expected delivery date and stores it on the order.

---

## 4. CustomRequest Branch  

### 4.1 TaskB2 – “Perform Custom Feasibility Analysis”

A more detailed assessment is required: engineering capacity, design constraints, cost estimation, regulatory compliance, etc.

### 4.2 XOR Gateway – “Is Customization Feasible?”

| Condition | Path |
|-----------|------|
| **Yes** (feasibility confirmed) |  **TaskE1 – Prepare Custom Quotation** |
| **No** (cannot be delivered as requested) |  **TaskE2 – Send Rejection Notice**  **End Event** |

Only one outcome is possible for each request.

### 4.3 TaskE1 – “Prepare Custom Quotation”

Creates a detailed quote (price, leadtime, terms) that reflects the custom work required.

### 4.4 TaskE2 – “Send Rejection Notice”

If the request is not feasible, the customer is informed and the process ends here (no further work).

---

## 5. Converging Point – “After Standard or Custom Path Tasks Completed”

Regardless of the branch taken, the flow now reaches a **common** point where the next decision is made. This is where the **standard** and **custom** paths **converge**.

---

## 6. XOR Gateway – “Is Approval Needed?”

The process asks: *Does this order require a higherlevel signoff?*  

Typical triggers:

* Large monetary value  
* Unusual terms (e.g., extended credit)  
* Custom work that exceeds a predefined cost threshold  

| Condition | Path |
|-----------|------|
| **Yes** (approval required) |  **TaskF – Obtain Manager Approval** |
| **No** (approval not needed) |  **TaskG – Generate Final Invoice** (bypass approval) |

---

## 7. Approval Path (when “Yes”)

### 7.1 TaskF – “Obtain Manager Approval”

The order (or quotation) is sent to a manager for review and signoff.

### 7.2 XOR Gateway – “Is Approval Granted?”

| Condition | Path |
|-----------|------|
| **Yes** |  **TaskG – Generate Final Invoice** |
| **No** |  **TaskH – Reevaluate Conditions** |

### 7.3 TaskG – “Generate Final Invoice”

Creates the official invoice (standard or custom) and stores it for downstream billing.

### 7.4 TaskH – “Reevaluate Conditions”

When approval is denied, the process must **rework** the request. The model shows two possible loops:

* **Standard Path Loop:** Return to **TaskD – Calculate Delivery Date** (e.g., adjust delivery schedule, renegotiate terms).  
* **Custom Path Loop:** Return to **TaskE1 – Prepare Custom Quotation** (e.g., modify price or scope).

The loop is **conditional**: the system knows which original branch the request came from and routes it back accordingly. This ensures that the reevaluation is performed on the appropriate artifact (delivery date vs. quotation).

After the reevaluation, the flow again reaches the “Is Approval Needed?” gateway, allowing the request to be resubmitted for approval.

---

## 8. PostInvoice  TaskI – “Send Confirmation to Customer”

Once the invoice is generated (whether directly or after a successful approval), a confirmation (email, portal notification, etc.) is sent to the customer, summarizing:

* Order details  
* Delivery date (standard) or expected completion date (custom)  
* Invoice number and payment instructions  

---

## 9. End Event

The process terminates after the customer has been notified. All required work items are now complete, and the order can move to the fulfillment or billing system.

---

### Summary of Differences Between **Standard** and **Custom** Paths  

| Aspect | Standard Request | Custom Request |
|--------|------------------|----------------|
| **Initial Validation** | Simple, rulebased checks (TaskB1) | Detailed feasibility study (TaskB2) |
| **Parallel Checks** | Credit & inventory run in parallel (C1, C2) | No parallel checks; feasibility analysis is a single, possibly longer activity |
| **Resulting Artifact** | Delivery date (TaskD) | Custom quotation (TaskE1) |
| **Rework Loop** | Returns to deliverydate calculation (D) if approval fails | Returns to quotation preparation (E1) if approval fails |
| **Path Convergence** | Both paths converge before the “Is Approval Needed?” decision |

---

### When Approvals or Rework Are Triggered  

| Trigger | What Happens |
|---------|--------------|
| **Approval Needed** (value, risk, custom cost) | Flow goes to TaskF  manager review. |
| **Approval Granted** | Invoice is generated (TaskG) and the process proceeds to customer confirmation. |
| **Approval Denied** | Reevaluation step (TaskH) is executed, then the flow loops back to the appropriate earlier task (D for standard, E1 for custom). This loop repeats until either approval is granted or the request is abandoned (e.g., by sending a rejection notice). |
| **Custom Infeasibility** (early XOR after B2) | Immediate rejection (TaskE2) and process ends – no approval step is reached. |

```
                       Join  Calculate Delivery Date

                         No  Send Rejection  End

   (Both branches)  Is Approval Needed? (XOR)

                               (then return to “Is Approval Needed?”)
```

---

## Key Takeaways  

1. **Decision points (XOR gateways)** ensure that only one mutually exclusive path is taken at a time (e.g., standard vs. custom, feasible vs. infeasible).  
2. **Parallel processing (AND gateway)** speeds up the standardrequest flow by allowing credit and inventory checks to run simultaneously.  
3. **Convergence** after the typespecific work enables a unified approval logic, simplifying governance and reporting.  
4. **Conditional looping** provides a structured way to handle rework without restarting the whole process—only the relevant segment (deliverydate or quotation) is revisited.  
5. **Endtoend visibility**: every request, whether standard or custom, ends with a final invoice and a customer confirmation, guaranteeing a consistent customer experience.
