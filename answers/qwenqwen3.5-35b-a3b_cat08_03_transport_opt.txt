# Process Mining Consultation Report: Speedy Parcels Optimization

**To:** Senior Management, Speedy Parcels
**From:** Process Mining Consultant
**Subject:** Strategic Analysis of Delivery Operations and Optimization Roadmap

## 1. Process Discovery and Conformance Checking

To apply process mining effectively, we must first transform raw telemetry and transactional data into a structured event log. The goal is to reconstruct the "As-Is" process and compare it against the "To-Be" (Dispatch Plan).

### Data Preprocessing and Integration
The primary challenge lies in data heterogeneity. GPS data is high-frequency telemetry (seconds), while Scanner data is event-based (minutes), and Dispatch data is static planning.
*   **Case ID Definition:** We will define the primary Case ID as `VehicleID + Date` (e.g., `V12-20241205`). This represents a single delivery shift. For granular analysis of packages, we will link `PackageID` as an attribute or create a nested trace.
*   **Activity Mapping:** We will map raw events to standard process activities:
    *   *GPS "Moving" + Location Change* $\rightarrow$ `Traveling`
    *   *Scanner "Arrive Customer"* $\rightarrow$ `Arrival at Stop`
    *   *Scanner "Delivery Success"* $\rightarrow$ `Service Completed`
    *   *GPS "Idle" + Engine On* $\rightarrow$ `Waiting/Dwell`
    *   *Maintenance Log "Unscheduled Repair"* $\rightarrow$ `Vehicle Breakdown`
*   **Challenges:**
    *   **Temporal Alignment:** GPS timestamps may drift relative to Scanner timestamps. We must synchronize clocks or allow a tolerance window (e.g., 30 seconds) to link "Arrive Customer" (Scanner) with "Speed 0" (GPS).
    *   **Geospatial Matching:** Matching GPS coordinates to specific Customer Addresses (from Dispatch) requires geocoding and proximity matching (e.g., GPS point within 50m of Planned Stop).
    *   **Noise:** GPS data often contains signal loss or jitter. We must filter out "ghost stops" where the vehicle stops for less than 2 minutes without a scanner event.

### Process Discovery
Using algorithms like the **Heuristic Miner** or **Inductive Miner**, we will generate a process map.
*   **Visualization:** This will show the flow from `Start Shift` $\rightarrow$ `Depart Depot` $\rightarrow$ [Loop: `Travel` $\rightarrow$ `Arrive` $\rightarrow$ `Service` $\rightarrow$ `Depart`] $\rightarrow$ `Return Depot` $\rightarrow$ `End Shift`.
*   **Handling Deviations:** The map will explicitly show parallel paths for `Delivery Success` vs. `Delivery Failed`. It will also highlight "unplanned" activities, such as `Unscheduled Stop` (e.g., the Engine Warning Light event in the snippet) or extended `Waiting` periods indicating traffic jams.

### Conformance Checking
We will compare the Discovered Process against the **Dispatch System Plan**.
*   **Method:** We will overlay the Planned Route (Sequence of Stops) with the Actual Route (Sequence of GPS/Scanner events).
*   **Deviations to Track:**
    *   **Sequence Deviations:** Did the driver skip a stop or visit Stop 5 before Stop 2? (e.g., To optimize traffic, a driver might reorder stops, which the plan doesn't reflect).
    *   **Timing Deviations:** Actual `Arrival` time vs. Planned `Time Window`.
    *   **Missing Activities:** Planned stops that show no GPS or Scanner activity (potential skipped deliveries).
    *   **Unplanned Stops:** GPS-detected stops not in the Dispatch plan (e.g., the 11:05:00 Engine Warning event).
    *   **Revisits:** Did the driver return to a location after leaving? (Indicates failed delivery or wrong address).

## 2. Performance Analysis and Bottleneck Identification

Process mining allows us to quantify performance across the entire process, not just at the end.

### Key Performance Indicators (KPIs)
We will calculate the following KPIs directly from the event log attributes:
1.  **On-Time Delivery Rate (OTD):** % of `Delivery Success` events occurring within the Customer-Requested Time Window.
2.  **Average Service Time:** Duration between `Arrive Customer` and `Depart Customer`. (High variance here suggests inconsistent parking/search times).
3.  **Travel vs. Service Ratio:** `Traveling` duration / `Service` duration. A high ratio indicates inefficient routing or excessive traffic.
4.  **Fuel Efficiency Proxy:** `Distance Traveled` (from GPS) / `Packages Delivered`.
5.  **Failed Delivery Rate:** % of `Delivery Failed` events requiring a re-attempt (tracked via `Case ID` continuity).
6.  **Vehicle Utilization:** Ratio of `Moving` time to `Shift Duration`. High idle time indicates inefficiency.

### Bottleneck Identification Techniques
*   **Performance Maps:** We will visualize the process map with a color gradient representing average duration per activity. "Red" areas will indicate where the process slows down (e.g., specific urban intersections or customer clusters).
*   **Duration Analysis:** We will analyze the distribution of `Service Time`. If the distribution is wide (e.g., 2 mins to 20 mins), it indicates a bottleneck in the "finding parking/entering building" sub-process.
*   **Resource Performance:** We will group data by `Driver ID` and `Vehicle ID`.
    *   *Example:* If Driver D105 has a significantly higher `Traveling` duration than Driver D106 on the same route, the bottleneck is likely Driver D105's driving behavior or route choice.
    *   *Example:* If Vehicle V12 has frequent `Unscheduled Stop` events compared to V13, the bottleneck is Vehicle Maintenance.
*   **Quantification:** We will calculate the **Delay Cost**. For every minute of delay in `Arrival`, we assign a cost based on overtime wages and fuel consumption, allowing us to prioritize bottlenecks by financial impact.

## 3. Root Cause Analysis for Inefficiencies

Identifying bottlenecks is step one; understanding *why* they occur is critical for optimization.

### Potential Root Causes & Analysis Techniques
1.  **Suboptimal Route Planning:**
    *   *Hypothesis:* Dispatch plans assume static travel times, ignoring real-world congestion.
    *   *PM Analysis:* **Variant Analysis.** Compare the duration of `Traveling` between the same two stops across different days. If duration varies significantly despite the same route, the root cause is traffic, not the route plan.
2.  **Inaccurate Travel Time Estimations:**
    *   *Hypothesis:* The system underestimates urban traffic.
    *   *PM Analysis:* Correlate `Speed (km/h)` from GPS with `Time of Day`. If average speed drops below 15km/h during peak hours, the estimation model is flawed.
3.  **High Variability in Service Time:**
    *   *Hypothesis:* Drivers spend too much time searching for parking or waiting for access codes.
    *   *PM Analysis:* Analyze the `Arrive` to `Depart` interval for "Delivery Success" vs. "Delivery Failed". If "Success" takes 10 mins and "Failed" takes 15 mins, the extra time is likely the "search/wait" phase.
4.  **Vehicle Breakdowns:**
    *   *Hypothesis:* Maintenance is reactive, causing mid-shift stoppages.
    *   *PM Analysis:* **Causal Discovery.** Correlate `Unscheduled Stop` events with specific Vehicle IDs and mileage. If a specific vehicle breaks down every 500km, the root cause is a maintenance interval issue.
5.  **Driver Behavior:**
    *   *Hypothesis:* Some drivers drive faster or take shortcuts that increase fuel burn.
    *   *PM Analysis:* Compare `Fuel Consumption` (estimated from GPS speed/load) against `Overtime` and `On-Time` metrics. A driver who finishes early but consumes 20% more fuel might be driving aggressively.

## 4. Data-Driven Optimization Strategies

Based on the insights derived above, I propose the following three strategies.

### Strategy 1: Dynamic Traffic-Aware Routing
*   **Target Inefficiency:** High `Traveling` duration and missed time windows due to congestion.
*   **Root Cause:** Static dispatch planning ignores real-time traffic patterns discovered in the GPS logs.
*   **Implementation:**
    *   Use historical GPS speed data to build a **Time-Dependent Travel Time Matrix**.
    *   Integrate this into the Dispatch System. Instead of assigning the shortest *distance* route, assign the route with the lowest *expected time* based on the hour of day.
    *   Enable **Real-Time Re-routing:** If GPS shows speed dropping below 5km/h (like the 08:10:55 event), the system alerts the driver or auto-adjusts the next stop sequence.
*   **Expected Impact:**
    *   **OTD Rate:** Increase by 10-15%.
    *   **Fuel Consumption:** Reduce by 5% (fewer stop-start cycles in traffic).

### Strategy 2: Predictive Maintenance & Vehicle Health Scoring
*   **Target Inefficiency:** Unscheduled stops (`Unscheduled Stop` events) and high repair costs.
*   **Root Cause:** Reactive maintenance leads to mid-shift breakdowns that disrupt the entire route (as seen with the Engine Warning Light at 11:05:00).
*   **Implementation:**
    *   Create a **Vehicle Health Score** using process mining data. Analyze correlations between `Idle Time`, `Engine RPM` (if available), and `Unscheduled Stops`.
    *   Shift from calendar-based maintenance to **Usage-Based Maintenance**. If Vehicle V12 shows a spike in idle time and warning lights, schedule service immediately rather than waiting for the next shift.
    *   Replace vehicles with consistently high `Unscheduled Stop` rates before they fail completely.
*   **Expected Impact:**
    *   **Vehicle Utilization:** Increase by reducing downtime.
    *   **Maintenance Costs:** Reduce by 20% (preventative vs. emergency repairs).

### Strategy 3: Failed Delivery Reduction & Time Window Optimization
*   **Target Inefficiency:** `Delivery Failed` events requiring re-delivery (rework cost).
*   **Root Cause:** Inaccurate customer time windows or lack of communication leading to "Customer Not Home."
*   **Implementation:**
    *   Analyze `Delivery Failed` events to identify patterns (e.g., specific neighborhoods, specific time windows, specific drivers).
    *   **Process Change:** Integrate the Scanner "Delivery Failed" event into a communication loop. If a failure is logged, trigger an automated SMS to the customer with a new time slot *before* the driver returns to the depot.
    *   **Route Optimization:** Cluster "Failed" locations. If a specific building has a 40% failure rate, update the dispatch algorithm to allocate more time or flag the address as "Access Difficult."
*   **Expected Impact:**
    *   **Cost per Package:** Reduce by 15% (eliminating re-delivery fuel and labor).
    *   **Customer Satisfaction:** Improve significantly by reducing failed attempts.

## 5. Considering Operational Constraints and Monitoring

### Operational Constraints
Any optimization must respect legal and operational boundaries:
*   **Driver Working Hours:** The routing engine must enforce maximum shift durations (e.g., 9 hours). If the optimized route exceeds this, the system must split the load or add a driver, even if it costs more.
*   **Vehicle Capacity:** The `Dispatch System` data must validate that the `Packages Assigned` do not exceed `Vehicle Capacity`. Process mining can reveal if drivers consistently overload vans (detected via GPS weight sensors or scan density).
*   **Customer Time Windows:** The "Dynamic Routing" strategy must prioritize strict adherence to time windows over speed if the customer SLA requires it.

### Continuous Monitoring Plan
To ensure sustainability, we will implement a **Process Mining Dashboard** (e.g., using Celonis, UiPath, or PowerBI with PM connectors).

**Key Metrics to Track:**
1.  **Process Compliance Rate:** % of shifts following the standard flow (e.g., no skipped `Depart Depot` events).
2.  **KPI Trend Lines:** Daily OTD, Fuel per km, and Failed Delivery rates.
3.  **Drift Detection:** Automated alerts if the "Average Service Time" increases by more than 10% compared to the baseline, indicating a new bottleneck (e.g., new construction, new driver).
4.  **Exception Heatmap:** A visual map highlighting "Red Zones" where delays consistently occur.

**Feedback Loop:**
*   **Monthly Review:** Process Mining team presents the top 3 bottlenecks to Operations Management.
*   **Quarterly Tuning:** Adjust the Dispatch algorithms based on the latest performance data (e.g., update travel time matrices).
*   **Driver Feedback:** Share anonymized performance insights with drivers to encourage best practices (e.g., "Driver X saves 10 mins on average by parking at Stop 3").

By implementing this structured process mining approach, Speedy Parcels will move from reactive problem-solving to proactive operational excellence, directly addressing the root causes of cost and punctuality issues.
