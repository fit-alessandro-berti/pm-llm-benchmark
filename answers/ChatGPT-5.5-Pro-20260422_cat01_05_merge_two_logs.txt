## Matching rules used

I used **Log A as the primary timeline**. For potential matches, I required:

1. Same `order_id`.
2. Equivalent or clearly related event name after normalization.
3. Timestamp difference within a **2-second tolerance**, treated as `<= 2 seconds`.

For merged events, I kept:

* `primary_timestamp` from **Log A**
* both original timestamps
* both original event names
* enriched attributes from Log B: `user_id`, `resource_id`, `notes`
* a match status explaining the merge

For unmatched events, I kept them as separate records and marked their origin as `System A only` or `System B only`.

## Reasoning by event

### 1. `Order Received` + `OrderReceived`

These were merged.

* Same `order_id`: `Order#1234`
* Event names are equivalent after normalization:

  * `Order Received`
  * `OrderReceived`
* Timestamp difference is exactly **2 seconds**
* Because the tolerance was treated as `<= 2 seconds`, this is a valid match.

The merged record uses Log A’s timestamp, `2023-10-10T10:00:00Z`, as the `primary_timestamp`, while preserving Log B’s earlier timestamp and richer metadata.

---

### 2. `Order Validated` + `OrderValidation`

These were merged.

* Same `order_id`
* Event names refer to the same validation activity
* Timestamp difference is **1 second**
* Log B contributes `user_id = u12`, `resource_id = r10`, and the note `"Cross-check inventory"`

This is a high-confidence match.

---

### 3. `Payment Processed` and `PaymentCheck`

These were **not merged** under the strict tolerance rule.

They are clearly related to the payment stage, but there are two reasons to keep them separate:

1. The timestamp difference is **5 seconds**, which is outside the 2-second tolerance.
2. The names may represent related but distinct lifecycle states:

   * `PaymentCheck` sounds like a verification/check step.
   * `Payment Processed` sounds like the completed payment outcome.

The Log B note, `"Payment gateway delay"`, also suggests the System B event may have been delayed or may reflect a separate gateway check. So I kept both events, while documenting the relationship.

A more aggressive merge policy could merge these as a low-confidence payment match, but with the stated 2-second tolerance, keeping them separate is safer.

---

### 4. `Quality Check`

This appears only in Log B.

There is no matching Log A event near `2023-10-10T10:03:00Z`, and no Log A event with a similar quality-inspection meaning. It is included as a standalone enriched event from System B.

---

### 5. `Item Shipped` + `Shipping`

These were merged.

* Same `order_id`
* Event names are semantically equivalent:

  * `Item Shipped`
  * `Shipping`
* Timestamp difference is **2 seconds**
* Log B contributes the warehouse/user/resource details and the note `"Package handed to courier"`

The merged record uses Log A’s timestamp, `2023-10-10T10:05:00Z`, as the primary timestamp.

---

### 6. `Item Delivered`

This appears only in Log A.

There is no corresponding delivery event in Log B, so it remains a System A-only record.

---

## Final interpretation

The integrated process timeline is:

1. Order received
2. Order validated
3. Payment processed in System A
4. Payment check recorded in System B, possibly delayed or separate
5. Quality check performed in System B
6. Item shipped
7. Item delivered

The final log preserves Log A’s primary ordering while enriching matched events with System B’s additional attributes. Borderline cases, especially the payment event, are retained separately to avoid incorrectly merging events that may represent distinct operational states.
