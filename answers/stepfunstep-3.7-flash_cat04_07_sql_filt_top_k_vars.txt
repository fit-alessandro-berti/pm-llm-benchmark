### DuckDB SQL Query
This query follows the required steps, with clear CTEs for readability and support for a configurable `top_k` parameter (replace `:top_k` with your desired number of top variants, e.g., `5`):
```sql
WITH
-- Step 1: Compute ordered activity sequence (variant) for each case
case_variants AS (
    SELECT
        case_id,
        -- Use string_agg for human-readable variant labels; use list_agg() for list-type variants
        string_agg(activity, '  ' ORDER BY timestamp) AS variant_sequence
        -- Alternative for list-type variants: list_agg(activity ORDER BY timestamp) AS variant_sequence
    FROM event_log
    GROUP BY case_id
),
-- Step 2: Count cases per variant and rank variants by frequency
variant_counts AS (
    SELECT
        variant_sequence,
        COUNT(*) AS case_count,
        -- Use ROW_NUMBER() for strict top K (no ties); use DENSE_RANK() to include ties for the Kth position
        ROW_NUMBER() OVER (ORDER BY case_count DESC) AS variant_rank
    FROM case_variants
    GROUP BY variant_sequence
),
-- Step 3: Filter to only top K variants
top_k_variants AS (
    SELECT variant_sequence
    FROM variant_counts
    WHERE variant_rank <= :top_k
),
-- Step 4: Get all case IDs belonging to top K variants
top_k_cases AS (
    SELECT cv.case_id
    FROM case_variants cv
    INNER JOIN top_k_variants tkv
        ON cv.variant_sequence = tkv.variant_sequence
)
-- Step 5: Return all original events for top K cases
SELECT
    el.*
FROM event_log el
INNER JOIN top_k_cases tkc
    ON el.case_id = tkc.case_id
-- Optional: Order events to match original process order per case
ORDER BY el.case_id, el.timestamp;
```
---
### Key Details & Customization
1. **Variant Definition**:
   - The `string_agg` function creates a human-readable activity sequence (e.g., `A  B  C`) for grouping. Replace it with `list_agg(activity ORDER BY timestamp)` if you prefer list-type variants for more compact storage.
   - If your `timestamp` column has duplicate values for the same case, add a secondary sort key to ensure deterministic variant ordering, e.g.:
     ```sql
     string_agg(activity, '  ' ORDER BY timestamp, activity) AS variant_sequence
     ```
2. **Top K Logic**:
   - Use `ROW_NUMBER()` for strict top K: if there is a tie for the Kth most frequent variant, only one will be included (arbitrary selection).
   - Use `DENSE_RANK()` instead if you want to include *all* variants tied for the Kth position (e.g., if K=2 and 3 variants tie for 2nd place, all 3 will be included).
3. **Parameter Setup**:
   - Replace `:top_k` with your desired integer (e.g., `5` for top 5 variants). In DuckDB clients like DBeaver or Python, you can also bind this as a query parameter.
---
### How It Works
1. The `case_variants` CTE first collapses each case into its unique ordered activity sequence, sorted by timestamp.
2. The `variant_counts` CTE aggregates these sequences to count how many cases follow each variant, then ranks variants by frequency (most frequent = rank 1).
3. We filter to only keep variants in the top K ranks, then map back to their associated `case_id`s.
4. Finally, we join back to the original `event_log` to return all raw events for cases in the top K variants, excluding all other cases.
