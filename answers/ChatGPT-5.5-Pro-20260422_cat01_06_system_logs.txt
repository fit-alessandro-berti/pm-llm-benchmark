## Case grouping logic

I treated each distinct work object as a separate process instance, because that gives an analyst-friendly event log where each case has a coherent business object.

| Case ID               | Interpreted Case                         | Rationale                                                                                                                               |
| --------------------- | ---------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------- |
| CASE-DOC1-001         | Drafting and finalizing `Document1.docx` | The user drafts text, saves, later returns after updating the budget, inserts a budget reference, saves again, and closes the document. |
| CASE-EMAIL-ANNUAL-001 | Handling the Annual Meeting email        | The user opens an email, reviews it, replies, writes confirmation text, and sends the reply.                                            |
| CASE-PDF-REPORT-001   | Reviewing `Report_Draft.pdf`             | The user opens the PDF, scrolls through it, and highlights “Key Findings.”                                                              |
| CASE-BUDGET-001       | Updating `Budget_2024.xlsx`              | The user focuses the spreadsheet, updates Q1 figures, inserts a Q2 row, and saves.                                                      |
| CASE-QR-001           | Editing `Quarterly_Report.docx`          | The user initially focuses the file, later resumes it, drafts the executive summary, saves, and closes it.                              |

## Activity naming logic

I normalized the raw system actions into higher-level activities:

* `FOCUS` and target-side `SWITCH` events became activities such as **Open / Focus Document**, **Open Email Inbox**, **Open / Focus PDF**, or **Resume Document Editing**.
* `TYPING` events became content-level activities, such as **Draft Document Content**, **Compose Email Reply**, **Update Spreadsheet Figures**, or **Draft Executive Summary**.
* `SCROLL` events were interpreted as review activities, such as **Review Email Content** or **Review PDF Content**.
* `CLICK` events were translated based on the action text, for example **Open Email**, **Start Email Reply**, and **Send Email Reply**.
* `SAVE` and `CLOSE` events were retained as standardized completion-oriented activities because they are meaningful milestones in document and spreadsheet work.
* `HIGHLIGHT` became **Annotate PDF**, since highlighting text in a PDF is a review/annotation action rather than just a low-level UI action.

Overall, the work session narrative is: the user briefly opens the quarterly report, drafts a new document, handles an Annual Meeting email, reviews a report PDF, updates the budget spreadsheet, returns to the document to add a budget reference, and finally edits and closes the quarterly report.
