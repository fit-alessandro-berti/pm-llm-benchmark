### Explanation of Case and Activity Logic

**Case Identification Strategy:**  
Cases are defined based on **document-centric workflows**. Each unique document or resource that the user interacts with in a continuous or logically connected manner forms a case. Specifically:

- **Case C1**: Focuses on `Quarterly_Report.docx`. Although initially opened early, it is only actively edited later (after working on other files), so all events related to this file are grouped into one case representing the "Quarterly Report Drafting" process.
- **Case C2**: Centers on `Document1.docx`, which is created, edited, saved, briefly left for email and spreadsheet work, then returned to for final edits before closing. This represents a complete drafting and refinement cycle for a new document.
- **Case C3**: Represents an email-handling task triggered by an inbox notification about the "Annual Meeting." It includes reading, replying, and sending—forming a self-contained communication workflow.
- **Case C4**: Involves reviewing and annotating `Report_Draft.pdf`. Though brief, it constitutes a distinct analytical activity.
- **Case C5**: Covers updating the `Budget_2024.xlsx` spreadsheet, including data entry and structural changes.

This approach ensures that each case reflects a coherent unit of work tied to a specific artifact (document, email thread, or file), aligning with best practices in process mining for user activity logs.

**Activity Naming Rationale:**  
Raw actions (e.g., TYPING, SCROLL, FOCUS) were abstracted into semantically meaningful activities that reflect user intent:
- Multiple `TYPING` events in the same context were retained individually if they represented distinct content additions (e.g., drafting vs. inserting references).
- `FOCUS` events that initiate interaction with a document are labeled as "Open Document" or "Open Spreadsheet" to denote case start.
- Transitions between applications (`SWITCH`) are included as contextual events but do not define new cases; instead, they are attributed to the target case if they lead into productive work.
- Actions like `HIGHLIGHT` or `SCROLL` are interpreted as "Annotate PDF" or "Review PDF" to convey purpose rather than mechanics.

This transformation yields a clean, analyst-friendly event log suitable for discovering workflows, identifying bottlenecks, or analyzing multitasking behavior across documents and applications.
