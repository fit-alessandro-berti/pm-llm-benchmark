## Rationale for Each Added Constraint

### 1. `existence`  `BiasMitigationCheck`
**Guarantees** that every trace contains at least one explicit fairness-control step. Without this, a process could legitimately skip any bias evaluation entirely.

### 2. `responded_existence`  `Check*  ManualReview`
Whenever a sensitive attribute (`Race`, `Gender`, `Age`) is inspected, a `ManualReview` **must also appear** somewhere in the trace. This ensures that sensitive data is never examined in isolation without human oversight.

### 3. `coexistence`  `*_Minority  ManualReview`
Any decision targeting a sensitive demographic group (`Reject_Minority`, `Approve_Minority`) must **co-occur** with a `ManualReview`. This enforces symmetric scrutiny: such decisions cannot exist without the review, and the constraint binds them together.

### 4. `response`  `Check*  BiasMitigationCheck` (eventually)
Once a sensitive attribute is read, the process is **obligated to eventually** run a `BiasMitigationCheck`. This closes any path where sensitive data influences flow without a downstream fairness control.

### 5. `precedence`  `BiasMitigationCheck` before `FinalDecision`; `ManualReview` before `*_Minority`
A `FinalDecision` is only valid if a `BiasMitigationCheck` **already happened** beforehand. Likewise, decisions about minority applicants require a prior `ManualReview`. This prevents premature decisions.

### 6. `altprecedence`  `BiasMitigationCheck` before each `Reject`
Stronger than ordinary precedence: between any two `Reject` events, a fresh `BiasMitigationCheck` must occur. This stops a single early check from "covering" multiple later rejections.

### 7. `chainresponse`  `CheckApplicantRace` immediately  `BiasMitigationCheck`
The strongest temporal link: inspecting race must be **immediately followed** by a bias check, leaving no room for a biased decision to slip in directly after viewing the attribute.

### 8. `chainprecedence`  `BiasMitigationCheck` immediately before `Reject`/`Approve`
Every `Approve`/`Reject` must be **directly preceded** by a `BiasMitigationCheck`, guaranteeing the most recent action before any decision is a fairness evaluation.

### 9. `nonsuccession`  `Check*  Reject` (eventually)
Forbids any path where viewing a sensitive attribute is **eventually followed** by a rejection. This is the core anti-discrimination rule: sensitive data cannot lead—through any route—to a raw `Reject`.

### 10. `nonchainsuccession`  `Check*  Reject/Approve` (immediately)
Forbids a sensitive-attribute check from being **immediately followed** by any decision, eliminating the most direct discriminatory pattern (e.g., "saw race  instantly rejected").

## How These Constraints Reduce Bias

| Mechanism | Effect |
|-----------|--------|
| **Mandatory controls** (`existence`, `responded_existence`) | A fairness step can never be skipped when sensitive data is involved. |
| **Ordering enforcement** (`precedence`, `chainprecedence`, `altprecedence`) | No decision can be made *before* a bias/manual review, preventing "snap" judgments. |
| **Immediate-adjacency bans** (`nonchainsuccession`) | Removes the most direct discriminatory pattern: attribute  decision. |
| **Eventual-path bans** (`nonsuccession`) | Blocks subtler routes where sensitive data still drives a negative outcome. |
| **Co-occurrence binding** (`coexistence`) | Sensitive-demographic decisions are inseparable from human review. |

Together, these constraints make it **structurally impossible** for the process to produce a decision that flows directly (or even indirectly) from a sensitive attribute without an intervening, auditable fairness control. This converts fairness from an informal expectation into a **verifiable property of the process model** that can be checked against every trace via conformance checking.

> **Note:** This is a defensive, fairness-by-design pattern. In a real deployment you would calibrate which `Check*`/decision activities exist in your event log, and validate the constraints against historical traces to confirm `support`/`confidence` of 1.0 actually holds (or relax them accordingly).
