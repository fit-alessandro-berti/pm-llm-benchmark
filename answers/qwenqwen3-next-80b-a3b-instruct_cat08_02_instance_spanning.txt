### **1. Identifying Instance-Spanning Constraints and Their Impact**

To identify and quantify the impact of instance-spanning constraints, we leverage **process mining techniques** — specifically **conformance checking**, **resource-based performance analysis**, **dependency mining**, and **attribute-based segmentation** — applied to the event log.

#### **a) Formal Identification of Constraints Using Process Mining**

- **Shared Cold-Packing Stations:**  
  We use **resource utilization heatmaps** and **resource conflict graphs**. For each Cold-Packing station (e.g., C1–C5), we extract timestamps of *START* and *COMPLETE* events. We then calculate **overlap intervals** across cases. If two orders (e.g., ORD-5002 and ORD-5007) have overlapping *Packing* activities on the same cold-packing station, we flag a **resource contention event**.  
  *Technique:* Use **Petri net-based resource modeling** or **directly-follows graphs with resource annotations** to visualize resource bottlenecks.

- **Shipping Batching:**  
  We group orders by *Destination Region* and analyze the **time gap between "Quality Check COMPLETE" and "Shipping Label Gen. COMPLETE"**. Orders that complete early but wait >30 minutes (based on historical SLA) for batch formation are flagged as **batching-delayed**. We then correlate this with the **size of the batch** and **time of day** (e.g., batch formed only at 14:00, 18:00).  
  *Technique:* **Attribute-based clustering** on *Destination Region* + **time-series analysis** of batch formation triggers.

- **Priority Handling (Express Orders):**  
  We identify **preemption events** by checking if a *Standard* order’s activity (e.g., Packing) was interrupted (i.e., *START*  *COMPLETE* with a large gap) and immediately followed by an *Express* order using the same resource.  
  *Technique:* **Temporal pattern mining** to detect “Resource Reassignment Pattern”:  
  `ORD-STD: Packing START  [GAP]  ORD-EXP: Packing START  ORD-STD: Packing COMPLETE`  
  We quantify **delayed standard order time** as the gap between original completion time (without interruption) and actual completion time.

- **Hazardous Material (HM) Limits:**  
  We aggregate the **number of concurrent HM orders** in *Packing* and *Quality Check* activities at any timestamp. We compute **violations** (when >10 HM orders active simultaneously) and **forced delays** (when an HM order waits because the limit was reached).  
  *Technique:* **State-based analysis** with **time-windowed aggregation** (e.g., every 5-minute interval) to count active HM cases per activity.

#### **b) Metrics for Impact Quantification**

| Constraint | Primary Metric | Secondary Metrics | Rationale |
|----------|----------------|-------------------|-----------|
| Shared Cold-Packing | **Avg. Waiting Time at Cold-Packing** | % of Express orders delayed due to cold-station contention | Direct measure of bottleneck cost |
| Shipping Batching | **Batch Formation Latency** (time from last order entry to batch closure) | % of orders waiting >1hr in “Ready for Shipping” state | Measures inefficiency in consolidation |
| Priority Handling | **Express-induced Delay to Standard Orders** | # of preemptions per hour, Avg. delay per preemption | Quantifies fairness and operational friction |
| HM Limits | **HM Throughput Reduction** | % of HM orders delayed due to regulatory cap, # of violations per day | Measures compliance cost and flow disruption |

#### **c) Differentiating Within-Instance vs. Between-Instance Delays**

- **Within-Instance Delays** are captured by **activity duration distributions** (e.g., average Packing time for Standard orders). If an order takes 45 minutes on Packing, but the median is 20 min  likely internal inefficiency (e.g., slow picker, complex item).
  
- **Between-Instance Delays** are identified by **waiting time anomalies**:
  - **Resource contention delay** = `Activity COMPLETE time of current order – Activity START time of current order – Median duration for same resource/activity type`
  - If the difference is >50% above median, and the resource was occupied by another case during that time  **between-instance delay**.
  
  *Example:* ORD-5001’s Packing took 8 minutes, but waited 40 minutes to start — resource was occupied by ORD-5002. This is **between-instance**, not within-instance.

  We use **process mining dashboards with control flow annotations** to highlight “waiting for resource” events as distinct from “long activity” events.

### **2. Analyzing Constraint Interactions**

The constraints do not operate in isolation — their **interactions amplify inefficiencies**:

- **Priority + Cold-Packing Interaction:**  
  An Express order requiring cold-packing (e.g., ORD-5002) preempts a Standard order using C2. This causes a **domino effect**: the displaced Standard order now waits longer, and if it was destined for the same region as other pending orders, it delays batch formation. This indirectly increases **batching latency**.

- **Batching + HM Limits Interaction:**  
  Suppose 8 HM orders are bound for the South region. If the batch for South is formed at 16:00 and already contains 5 HM orders, but 4 more HM orders arrive at 15:50, the system cannot include them without violating the 10-HM limit. Result: **batch is formed without them**, forcing them into the next batch  **double delay** (waiting for batch + waiting for HM slot).

- **HM Limits + Priority Interaction:**  
  An Express HM order arrives and triggers a preemption. But if all 10 HM slots are occupied, it must wait — even though it has priority — until an HM slot frees. This creates a **priority paradox**: high-priority orders are stuck due to regulatory constraints.

> **Why understanding interactions matters:**  
> Optimizing one constraint in isolation (e.g., adding cold-packing stations) may worsen another (e.g., if more Express orders are processed, HM limits are hit more often). Only by modeling interactions can we avoid **suboptimal local improvements** and achieve **global process efficiency**. Process mining reveals these **cross-constraint correlation patterns** — e.g., “When preemption rate > 12/hr, HM violations increase by 30%”.

### **3. Developing Constraint-Aware Optimization Strategies**

#### **Strategy 1: Dynamic Cold-Packing Resource Allocation with Priority Queues**

- **Addresses:** Shared Cold-Packing, Priority Handling  
- **Proposed Change:**  
  Introduce a **two-tier queue system** for cold-packing stations:  
  - **Tier 1 (Express):** Reserved 2 of 5 stations (C1, C2) exclusively for Express + Cold orders.  
  - **Tier 2 (Standard):** Remaining 3 stations (C3–C5) for Standard orders.  
  - **Overflow Rule:** If Tier 1 is idle and Tier 2 has a backlog of >5 orders, one Tier 1 station may be temporarily shared — but **only if no Express order is pending**.  
- **Data-Driven Leveraging:**  
  Use **predictive analytics** (e.g., LSTM model) on historical data to forecast Express order volume by hour/day. If forecast shows >8 Express orders/hour in West region, auto-allocate 3 stations to Tier 1.  
- **Expected Outcomes:**  
  - 40% reduction in Express cold-packing wait time  
  - 25% reduction in preemptions of Standard orders  
  - No increase in overall cold-packing utilization (avoids over-provisioning)

#### **Strategy 2: Adaptive Batching with Hazardous Material Constraints**

- **Addresses:** Shipping Batching, HM Limits  
- **Proposed Change:**  
  Replace fixed-time batching (e.g., every 4 hours) with **event-triggered batching** based on:  
  - **Region demand density** (e.g., if 10+ orders for South arrive in 1hr  trigger batch)  
  - **HM capacity buffer** (e.g., if HM orders in pending queue > 3, delay batch until 1 HM order completes in Packing/QC)  
  - **Batch size optimization** using **historical shipping cost vs. delay cost** (from log data)  
- **Data-Driven Leveraging:**  
  Apply **clustering** on destination regions and **time-to-batch** metrics to find optimal thresholds. Use **reinforcement learning** to simulate batch trigger rules and minimize total cost (delay cost + shipping cost).  
- **Expected Outcomes:**  
  - 30% reduction in average batch formation latency  
  - Elimination of HM violations due to proactive delay  
  - 15% improvement in shipping route density (fewer partial batches)

#### **Strategy 3: HM Activity Slot Reservation System with Real-Time Monitoring**

- **Addresses:** HM Limits, Priority Handling, Interactions  
- **Proposed Change:**  
  Implement a **centralized HM slot reservation system** that:  
  - Tracks real-time HM orders in Packing + QC (via event log stream)  
  - When an HM order enters Packing, it **reserves a slot** for 15 minutes (average processing time)  
  - If no slot available, order is **queued with estimated wait time** displayed to staff  
  - **Express HM orders** are given **priority in queue** but still subject to slot limit — no violation allowed  
  - If 10 slots are full, new HM orders (even Express) are **temporarily paused** at “Quality Check Ready” stage until slot frees  
- **Data-Driven Leveraging:**  
  Use **real-time process mining** (e.g., Camunda + Apache Flink) to monitor HM activity counts. Integrate with **digital twin dashboard** showing live slot usage.  
- **Expected Outcomes:**  
  - 100% compliance with HM regulations  
  - 20% reduction in HM order processing time (due to reduced queue juggling)  
  - Staff receive clear alerts  reduces confusion and compliance risk

### **4. Simulation and Validation**

Before implementation, we build a **discrete-event simulation (DES) model** using **AnyLogic** or **SimPy**, informed by process mining outputs.

#### **Simulation Model Design:**

- **Entities:** Each order is a process instance with attributes: Type, Region, HM flag, Cold requirement, Priority.
- **Resources:**  
  - 5 Cold-Packing stations (2 reserved for Express)  
  - 10 HM slots (shared across Packing + QC)  
  - Dynamic batch pools per region  
- **Rules Embedded:**  
  - Preemption logic for Express orders  
  - Batch formation triggers based on region volume + HM buffer  
  - HM slot reservation and release logic  
- **Input Data:**  
  - Activity durations from process mining (mean, std dev, percentiles)  
  - Arrival patterns (hourly order volume, seasonality)  
  - Constraint violation rates from historical log

#### **Validation Focus Areas:**

| Constraint | Simulation Focus |
|----------|------------------|
| Resource Contention | Measure cold-packing utilization, preemption frequency, queue length |
| Batching | Track batch size variance, time-to-batch distribution, % orders waiting >1hr |
| Priority Handling | Compare Express vs. Standard end-to-end times; measure delay caused to Standard |
| HM Limits | Count violations, track HM order throughput, measure forced queuing time |

#### **Validation Metrics:**
- **KPIs:** Avg. order cycle time, % on-time delivery, HM compliance rate, resource utilization rate  
- **Scenario Testing:**  
  - Baseline (current state)  
  - Strategy 1 only  
  - Strategy 2 only  
  - Strategy 3 only  
  - Combined (full solution)  
- **Outcome:** Select the strategy with **highest % improvement in on-time delivery** and **zero HM violations**.
