**Assumptions and design choices**  
- **Case ID**: `order_id` is used as the unique process instance identifier.  
- **Activities**:  
  - "Order Created"  derived from `Orders.order_date`.  
  - "Item Picked"  one event per line in `Order Lines`, using `item_picking_date`.  
  - "Order Dispatched" & "Order Arrived"  one event each per delivery record, using `delivery_dispatch_date` and `delivery_arrival_date`.  
  - "Invoice Issued"  from `Invoice.invoice_issue_date`.  
  - "Payment Received"  from `Payments.payment_date`.  
- **Resources**: IDs have been replaced with the humanreadable names from the Resources table. Carriers are treated as the actors for dispatch/arrival events. For payments, no internal resource is available, so the field is left blank.  
- **Chronology**: All events are sorted by timestamp. When timestamps coincide (e.g., `1002 Item Picked` and `1003 Order Created` both at `10:00`), the order shown does not imply a strict sequence but maintains logical grouping.  
- **Additional attributes**: Orderlevel data (customer, priority) are repeated for the "Order Created" event; item details (description, category) are included for picking events; delivery, invoice, and payment identifiers are attached to their respective events. Empty cells mean the attribute is not applicable for that event type.  

This event log is ready for process mining, providing a complete and chronologically consistent trace of each order from creation to payment.
