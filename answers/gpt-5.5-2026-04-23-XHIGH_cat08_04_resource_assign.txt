## 0. Preparation: Build an Analysis-Ready ITSM Event Log

Before analyzing resource assignment, I would enrich and structure TechSolveĺs event log so that resource behavior can be measured accurately.

Key preparation steps:

- **Define case ID:** Ticket ID / Incident ID.
- **Normalize activities:** `Ticket Created`, `Assign L1`, `Work L1 Start`, `Work L1 End`, `Escalate L2`, `Assign L2`, `Reassign`, `Resolve`, `Close`, `Reopen`, etc.
- **Pair START and COMPLETE events** to calculate actual work/processing time.
- **Separate waiting time from processing time:**
  - Waiting time: assignment to work start.
  - Processing time: work start to work end.
  - Transfer delay: reassignment/escalation to next work start.
- **Enrich with reference data:**
  - Agent roster, tier, team, shift calendar, vacation/absence.
  - Skill matrix with proficiency levels, e.g., basic/intermediate/expert.
  - SLA targets by priority.
  - Business hours calendar.
  - Ticket closure outcome, reopen flag, customer satisfaction if available.
- **Create derived flags:**
  - Skill match: required skill is present in assigned agentĺs profile.
  - Skill mismatch: required skill absent or proficiency too low.
  - Overqualification: highly specialized agent handling low-complexity/basic work.
  - Escalated case: L1 to L2/L3.
  - Reassigned case: one or more reassignments.
  - SLA breach: ticket exceeded target resolution time.

This enriched event log becomes the basis for resource, process, and performance mining.

# 1. Analyzing Resource Behavior and Assignment Patterns

## 1.1 Metrics for Individual Agents and Support Tiers

I would analyze resource performance at three levels:

1. **Individual agent**
2. **Support tier: L1, L2, L3**
3. **Skill group**, for example `Networking-Firewall`, `Database-SQL`, `App-CRM`, `Security-IAM`

### A. Workload Distribution

Key questions:

- Which agents handle the most tickets?
- Are certain agents consistently overloaded?
- Are some agents underutilized?
- Is workload balanced within each tier?

Metrics:

| Metric | Description |
|---|---|
| Number of tickets touched | Count of cases where the agent performed at least one activity |
| Number of tickets resolved | Count of tickets closed by the agent |
| Active WIP | Number of open tickets assigned to an agent at a given time |
| Queue size per agent/tier | Tickets waiting before work starts |
| Total processing time | Sum of actual work time from START to COMPLETE |
| Utilization rate | Processing time divided by available working time |
| Average daily/weekly workload | Workload by calendar period |
| Workload by priority | P1/P2/P3/P4 distribution per agent |
| Workload by skill | Required skill distribution handled by each agent |

A useful visualization would be a **resource workload heatmap**, showing agents on one axis and time periods on the other, with color intensity representing WIP, queue length, or utilization.

---

### B. Activity Processing Times

For each agent and tier, I would measure:

- Average, median, and 90th percentile work duration.
- Processing time by ticket priority.
- Processing time by ticket category.
- Processing time by required skill.
- Processing time by activity type, e.g., L1 troubleshooting, L2 analysis, L3 resolution.
- Difference between actual work time and waiting time.

Example metrics:

| Metric | Why It Matters |
|---|---|
| Average L1 handling time | Indicates whether L1 is spending too long before escalation |
| Average L2 resolution time by skill | Reveals specialist productivity and complexity |
| Waiting time before L2 work starts | Indicates queue bottlenecks |
| Processing time variance per agent | Identifies inconsistent handling or training needs |
| P90/P95 processing time | Captures outliers and chronic delays |

For example, if `Networking-Firewall` P2 tickets wait 3 hours before L2 work starts but only require 25 minutes of actual processing, the real bottleneck is resource availability, not technical complexity.

---

### C. First-Call Resolution Rate for L1

For L1, a critical metric is **first-call or first-contact resolution rate**.

Formula:

> **L1 FCR Rate = Tickets resolved by L1 without escalation / Total tickets initially handled by L1**

This should be analyzed by:

- Ticket category.
- Priority.
- Channel: phone, email, portal.
- Required skill.
- Individual L1 agent.
- Time of day or shift.
- Knowledge article availability.

This can reveal patterns such as:

- Some L1 agents consistently resolve certain categories better.
- Certain categories almost always require escalation.
- Web portal tickets may have poorer categorization and lower L1 resolution.
- L1 may spend excessive time on tickets that should be routed directly to L2.

---

### D. Reassignment and Escalation Metrics

I would calculate:

| Metric | Description |
|---|---|
| Average number of reassignments per ticket | Measures routing inefficiency |
| Percentage of tickets with at least one reassignment | Indicates assignment quality |
| Percentage of tickets with multiple reassignments | Detects problematic loops |
| Time lost per reassignment | Time between reassignment and next work start |
| Escalation rate from L1 to L2/L3 | Shows L1 effectiveness and triage accuracy |
| Escalation success rate | Whether the next tier resolves the ticket without further transfer |
| Incorrect initial assignment rate | Tickets initially assigned to an agent without required skill |
| Time to correct skill | Time from ticket creation to first work by a correctly skilled agent |

A particularly important measure is:

> **Time to first qualified agent = Timestamp of first work start by skill-matched agent - Ticket creation timestamp**

This measures how long the ticket takes to reach someone capable of resolving it.

---

### E. SLA Performance by Resource Pattern

I would segment SLA compliance by assignment pattern:

| Assignment Pattern | Expected Insight |
|---|---|
| L1 resolved directly | Usually fastest and lowest cost |
| L1 to L2 with no reassignment | Normal escalation path |
| L1 to L2 to L3 | More complex, higher SLA risk |
| L2 reassignment within same tier | Possible skill mismatch |
| Multiple reassignments | High risk of delay and SLA breach |
| Assigned to overloaded agent | Likely queue delay |
| Assigned to non-matching skill | Likely rework or transfer |

This helps determine which resource behaviors are most strongly associated with SLA breaches.

## 1.2 Process Mining Techniques to Reveal Actual Assignment Patterns

### A. Process Discovery

Using process discovery, I would generate actual process models for:

- All tickets.
- P1/P2/P3/P4 separately.
- Each category, such as Network, Software-App, Security.
- Each required skill.
- SLA-met versus SLA-breached cases.
- Tickets with zero reassignments versus multiple reassignments.

This would show the actual paths tickets follow, such as:

- `Ticket Created  Assign L1  Work L1  Resolve`
- `Ticket Created  Assign L1  Work L1  Escalate L2  Assign L2  Work L2  Resolve`
- `Ticket Created  Assign L1  Work L1  Escalate L2  Assign L2  Reassign  Assign L2  Work L2  Escalate L3  Resolve`

The goal is to compare the discovered reality with the intended operating model.

For example, management may believe the process is:

> L1 triage  L2 specialist  L3 only when needed

But process mining may reveal:

> L1 triage  L2 wrong specialist  L2 reassignment  L2 queue delay  L3 specialist  back to L2  resolution

That would indicate inefficient routing and unnecessary handoffs.

---

### B. Conformance Checking Against Intended Assignment Logic

I would define the intended rules, for example:

- All tickets start with L1 except predefined categories that bypass L1.
- P1/P2 tickets must be assigned within a defined time.
- Tickets requiring specialist skills must be routed to agents with those skills.
- L2 tickets should not be assigned to agents without the required skill.
- Reassignments should be exceptional, not routine.
- L3 should only handle advanced incidents.

Then I would use conformance checking to identify deviations:

| Intended Rule | Possible Deviation |
|---|---|
| L1 handles initial triage | Tickets bypass L1 unnecessarily |
| Required skill should match agent skill | Tickets assigned to unqualified agents |
| L2 resolves specialist cases | Tickets bounce among L2 agents |
| L3 handles complex cases | L3 works on low-priority/basic tickets |
| Round-robin balances load | Some agents receive far more work than others |

This comparison is essential because the documented process often differs significantly from the real process.

---

### C. Resource Interaction and Handover Analysis

Using handover-of-work analysis, I would build a **resource interaction network**.

Nodes:

- Agents, teams, or tiers.

Edges:

- Ticket transfers from one resource to another.

Edge weight:

- Number of handovers.
- Average delay introduced by the handover.
- SLA breach rate after the handover.

This helps identify:

- Agents who frequently transfer work.
- Agents who receive excessive escalations.
- Bottleneck specialists who become central nodes.
- Reassignment loops between specific agents or teams.
- Unofficial routing practices.

Example insight:

> 42% of `App-CRM` tickets handled by Agent B12 are reassigned to Agent B15, adding an average of 75 minutes before work resumes.

That would suggest B12 is frequently receiving tickets that actually require B15ĺs `Database-SQL` skill.

---

### D. Social Network Analysis

Social network analysis can reveal organizational patterns that are not visible in standard process maps.

Useful measures:

| Measure | Interpretation |
|---|---|
| Degree centrality | Agents heavily involved in handovers |
| Betweenness centrality | Agents who act as routing bridges or bottlenecks |
| Closeness centrality | Agents closely connected to many others |
| Handover frequency | How often tickets move between resources |
| Subcontracting relationship | One resource frequently does work before another |
| Working-together relationship | Agents often contribute to same tickets |

This may reveal that:

- A few specialists are acting as informal dispatchers.
- Certain L1 agents frequently escalate to the same L2 agents.
- Some teams operate in silos.
- Tickets are transferred repeatedly between Network and Security groups.
- Some L2 specialists are overloaded because everyone sends work to them directly.

---

### E. Role Discovery

Role discovery groups agents based on the activities and ticket types they actually handle.

This can identify:

- L1 agents who behave like L2 specialists.
- L2 specialists doing mostly L1-level tasks.
- L3 experts being used for routine tickets.
- Shadow experts who resolve many tickets in a specific category but are not formally recognized.
- Agents with documented skills they rarely use.
- Agents whose actual work does not match their assigned tier.

For example:

- Agent B12 is officially an L2 `App-CRM` and `DB-SQL` agent.
- Process mining shows B12 mostly handles generic application access requests.
- Meanwhile, Agent B15 handles nearly all true `Database-SQL` incidents.
- This suggests either the skill matrix is inaccurate or assignment rules are not precise enough.

# 2. Identifying Resource-Related Bottlenecks and Issues

## 2.1 Bottlenecks Caused by Insufficient Skill Availability

To identify skill bottlenecks, I would look for combinations of:

- High queue time before qualified work starts.
- High utilization of agents with that skill.
- High WIP/backlog for that skill.
- High SLA breach rate.
- High reassignment rate.
- Work concentrated in a small number of agents.

Example bottleneck indicators:

| Indicator | Interpretation |
|---|---|
| `Networking-Firewall` agents > 90% utilized | Capacity shortage or poor load balancing |
| P2 firewall tickets wait 2+ hours before L2 start | SLA risk caused by queue delay |
| Most firewall tickets assigned to two agents | Overdependence on limited specialists |
| High reassignment from generic Network to Firewall specialists | Initial skill identification is weak |

The key distinction is between:

- **Technical processing bottleneck:** Work itself takes a long time.
- **Resource availability bottleneck:** Ticket waits a long time before skilled work starts.

For SLA improvement, resource availability bottlenecks are often the easier and faster problem to fix.

---

## 2.2 Delays Introduced by Reassignments and Escalations

For every reassignment, I would calculate:

> **Reassignment delay = Next work start timestamp - Reassignment timestamp**

Also:

> **Total reassignment delay per ticket = Sum of all reassignment delays in the case**

Then compare cases:

- Tickets with no reassignment.
- Tickets with one reassignment.
- Tickets with two or more reassignments.

Example analysis output:

| Reassignments per Ticket | Avg Resolution Time | SLA Breach Rate |
|---:|---:|---:|
| 0 | 6 hours | 8% |
| 1 | 10 hours | 18% |
| 2+ | 18 hours | 42% |

This type of analysis would quantify how much delay is being created by routing inefficiency.

Important measures:

- Average added time per reassignment.
- Median added time per reassignment.
- P90 reassignment delay.
- Percentage of breaches involving reassignment.
- SLA breach probability by reassignment count.

---

## 2.3 Impact of Incorrect Initial Assignments

I would define an incorrect initial assignment as one where:

- The first assigned agent lacks the required skill.
- The ticket is reassigned soon after initial work starts.
- The initial agent records ôescalation neededö after minimal troubleshooting.
- The final resolving skill differs from the initially identified required skill.
- The same category repeatedly moves from one team to another.

Useful metrics:

| Metric | Description |
|---|---|
| First assignment skill-match rate | Whether the first handler had the needed skill |
| Time to correct skill | Time until first qualified agent starts |
| Incorrect initial assignment rate by dispatcher/L1 | Identifies where triage fails |
| Reassignment after first work | Indicates poor initial routing |
| SLA breach rate after incorrect assignment | Measures business impact |

Example finding:

> P2 Network tickets initially assigned to generic L2 Network agents but later reassigned to Firewall specialists breach SLA 2.3 times more often than tickets routed directly to Firewall specialists.

This would strongly support skill-based routing.

---

## 2.4 Identification of Overloaded or Underutilized Agents

I would combine workload, utilization, processing time, and case complexity.

### Overload indicators

- High WIP.
- High queue length.
- High utilization, e.g., above 85ľ90%.
- High number of P1/P2 tickets.
- Frequent after-hours work.
- Increasing backlog age.
- Higher SLA breach rate caused by waiting before work starts.

### Underutilization indicators

- Low utilization.
- Low assigned ticket count.
- Low complexity mix.
- Long idle periods during business hours.
- Skill profile indicates capability, but little matching work assigned.

Important caution:

An apparently ôslowö agent may simply be handling harder tickets. Therefore, performance should be normalized by:

- Ticket category.
- Priority.
- Required skill.
- Complexity.
- Number of affected users.
- Known problem/major incident linkage.
- Reopen rate.
- Escalation history.

The objective is not to blame individuals, but to identify capacity, training, routing, and process design issues.

## 2.5 Correlation Between Assignment Patterns and SLA Breaches

I would use statistical analysis and process mining performance overlays to quantify which assignment factors increase breach risk.

Potential predictors:

- Priority.
- Category.
- Required skill.
- Channel.
- Time of day.
- Initial assignment skill match.
- Agent workload at assignment time.
- Tier assigned.
- Number of handovers.
- Number of reassignments.

This allows TechSolve to prioritize the assignment issues with the greatest measurable SLA impact.

# 3. Root Cause Analysis for Assignment Inefficiencies

## 3.1 Potential Root Causes

### A. Current Assignment Rules Ignore Skill and Workload

If assignment is mostly round-robin within tiers, it may balance ticket count superficially but not actual workload.

Problems:

- A simple password reset and a complex firewall outage may count as one ticket each.
- Agents may receive tickets outside their skill area.
- Specialists may receive basic tickets because they are next in the rotation.
- An overloaded agent may receive new work while a capable agent is available.
- P2/P3 tickets may wait behind lower-priority work.

Root cause pattern:

> Assignment logic optimizes fairness of ticket count, not resolution capability or SLA risk.

---

### B. Inaccurate or Incomplete Agent Skill Profiles

The event log may show that documented skills do not match actual behavior.

Examples:

- Agents listed as `Database-SQL` rarely resolve database tickets.
- Agents not listed as `App-CRM` resolve many CRM incidents.
- Skill proficiency is missing, so beginner and expert agents are treated equally.
- Skill profiles are not updated after training or role changes.
- Some agents are unavailable for certain skills during specific shifts.

Root cause pattern:

> The assignment engine cannot route correctly because the underlying skill data is unreliable.

---

### C. Poor Initial Ticket Categorization or Skill Identification

If the ticket category or required skill is wrong at creation, even a good assignment engine will route incorrectly.

Common causes:

- Users choose wrong portal category.
- Email tickets lack structure.
- Phone tickets depend heavily on L1 interpretation.
- Required skill is inferred manually and inconsistently.
- Ticket descriptions are vague.
- Categories are too broad, e.g., ôSoftware-Appö instead of ôCRM integration issueö.
- Required skill changes after initial investigation.

Root cause pattern:

> The process is assigning based on inaccurate demand signals.

---

### D. Lack of Real-Time Workload and Availability Visibility

Manual dispatchers may not know:

- Current WIP per agent.
- Queue age per agent.
- Whether an agent is near shift end.
- Whether an agent is handling a major incident.
- Whether an agent has planned absence.
- Whether the agent is already overloaded with P1/P2 tickets.

Root cause pattern:

> Dispatch decisions are made without real-time capacity information.

---

### E. Insufficient L1 Training or Empowerment

High escalation from L1 can be caused by:

- Lack of knowledge articles.
- No access rights to perform common fixes.
- Limited diagnostic scripts.
- Fear of breaching SLA if they spend too long troubleshooting.
- Incentives based on speed of transfer rather than resolution quality.
- Weak feedback from L2/L3 after escalation.

Root cause pattern:

> L1 escalates tickets that could be resolved earlier with better training, tools, or authority.

---

### F. Specialists Used for Low-Complexity Work

This can happen when:

- Round-robin includes specialists in generic queues.
- There is no separation between routine and expert work.
- Ticket categorization is too coarse.
- L1 lacks authorization to complete basic tasks.
- L2/L3 agents are pulled into support tasks because of backlog pressure.

Root cause pattern:

> Scarce expert capacity is consumed by work that should be handled by lower tiers.

---

## 3.2 Variant Analysis to Find Root Causes

I would compare process variants such as:

### Smooth cases

- No reassignment.
- First assigned agent has required skill.
- Resolved within SLA.
- Few handovers.
- Short queue time.

### Problem cases

- Multiple reassignments.
- Skill mismatch at first assignment.
- Escalated more than once.
- Long waiting time.
- SLA breached.

Then I would compare their characteristics:

| Factor | Smooth Cases | Problem Cases |
|---|---|---|
| Initial skill match | High | Low |
| Average reassignments | Low | High |
| Time to first qualified agent | Short | Long |
| Assigned agent WIP | Moderate | High |
| Required skill | Common skills | Scarce/specialist skills |
| Ticket source | Phone/structured portal | Email/free text |
| Category accuracy | High | Low |
| Dispatcher/L1 involved | Certain agents | Other agents |
| Time of day | Core hours | Shift edges/off-hours |

This identifies the conditions under which assignment fails.

Example insight:

> P3 Software-App tickets from the web portal are usually resolved smoothly, but P2 Network tickets from phone calls after 3 PM frequently go through L1, generic Network L2, then Firewall L2, causing SLA breaches.

That points to both triage and shift coverage issues.

---

## 3.3 Decision Mining

Decision mining examines how choices are made at process decision points.

Relevant decision points:

- Assign to L1 or bypass L1.
- Choose specific L1 agent.
- Escalate to L2 or continue L1 troubleshooting.
- Choose L2 skill group.
- Choose specific L2 specialist.
- Escalate to L3.
- Reassign within same tier.

For each decision point, I would analyze which attributes influence the decision:

- Priority.
- Category.
- Required skill.
- Ticket text keywords.
- Channel.
- Current queue length.
- Agent tier.
- Agent skill profile.
- Agent workload.
- Time of day.
- Historical resolver patterns.

Decision mining can reveal poor decision logic such as:

- Assignment is mostly based on tier and availability, not skill.
- Dispatchers assign to the same familiar specialists.
- Certain L1 agents escalate too quickly.
- Certain categories are repeatedly misrouted.
- The required skill selected at assignment differs from the skill of the eventual resolver.
- Workload is not considered at all.

This provides evidence for redesigning the assignment rules.

# 4. Data-Driven Resource Assignment Strategies

Below are six concrete strategies TechSolve could implement. At minimum, I would prioritize the first three.

---

## Strategy 1: Skill-Based Routing with Proficiency Weighting

### Assignment issue addressed

This directly addresses:

- Skill mismatches.
- Unnecessary reassignments.
- Delays in reaching qualified agents.
- Specialists receiving incorrect work.
- P2/P3 SLA breaches caused by poor initial routing.

### How it uses process mining insights

The process mining analysis identifies:

- Which required skills have high reassignment rates.
- Which agents actually resolve specific ticket types.
- Where documented skills differ from actual performance.
- Which skills are bottlenecks.
- Which skill mismatches correlate with SLA breaches.

### Proposed logic

Instead of assigning purely by round-robin, route tickets to agents who have the required skill and sufficient proficiency.

Example scoring model:

> Assignment Score = Skill Match + Proficiency + Availability + SLA Urgency - Current WIP - Context Switching Penalty

A simplified routing rule:

1. Identify required skill from ticket category, description, configuration item, and historical patterns.
2. Find available agents with that skill.
3. Rank them by:
   - Proficiency level.
   - Current workload.
   - SLA urgency.
   - Historical success with similar tickets.
   - Shift availability.
4. Assign to the highest-ranked suitable agent or queue.
5. If no suitable agent is available, escalate to a skill queue with SLA-based prioritization.

### Data required

- Ticket category and required skill.
- Agent skill matrix.
- Proficiency levels.
- Historical resolution performance by skill.
- Agent availability and shift calendar.
- Current WIP and queue length.
- SLA targets by priority.

### Expected benefits

- Fewer incorrect assignments.
- Fewer reassignments.
- Shorter time to first qualified agent.
- Lower SLA breach rate for specialist tickets.
- Better use of scarce L2/L3 expertise.
- More predictable resolution times.

---

## Strategy 2: Workload-Aware and SLA-Aware Assignment

### Assignment issue addressed

This addresses:

- Uneven workload distribution.
- Overloaded agents.
- Underutilized agents.
- P2/P3 tickets waiting in the wrong queue.
- Assignments made without real-time capacity information.

### How it uses process mining insights

Process mining reveals:

- Queue delays by agent, tier, and skill.
- Utilization patterns.
- WIP thresholds associated with SLA breaches.
- Time-of-day demand peaks.
- Skill groups with chronic backlog.
- Agents who receive too much or too little work.

### Proposed logic

Assignment should consider real-time workload, not just agent rotation.

The system should evaluate:

- Current number of assigned open tickets.
- Current estimated effort already queued.
- Ticket priority and remaining SLA time.
- Agent availability during the expected handling window.
- Required skill.
- Expected processing time.

Example rule:

> For a P2 ticket requiring `Networking-Firewall`, assign to the available qualified agent with the lowest projected completion time, not simply the next firewall agent in rotation.

Projected completion time can be estimated as:

> Current queued effort for agent + predicted effort for new ticket

The assignment engine should also use:

- WIP limits per agent.
- Priority-based queue ordering.
- SLA risk alerts.
- Capacity reservation for P1/P2 incidents.
- Escalation if no qualified agent can start within the SLA-safe window.

### Data required

- Real-time ticket queue.
- Agent WIP.
- Agent calendars and shifts.
- Ticket priority and SLA deadline.
- Predicted handling time.
- Agent skill and proficiency.
- Historical queue and processing times.

### Expected benefits

- More balanced workload.
- Reduced waiting time.
- Fewer SLA breaches caused by queue delay.
- Better use of available capacity.
- Less burnout among overloaded agents.
- Improved transparency for dispatchers and managers.

## Strategy 3: Predictive Triage and Assignment Using Historical Data

### Assignment issue addressed

This addresses:

- Poor initial categorization.
- Wrong required skill identification.
- Excessive L1 handling before inevitable escalation.
- Tickets routed to the wrong tier or team.
- Manual triage inconsistency.

### How it uses process mining insights

The event log shows:

- Which ticket attributes predict escalation.
- Which categories are often misclassified.
- Which description keywords correlate with final resolving skill.
- Which tickets are usually solved by L1.
- Which tickets almost always require L2 or L3.
- Which routing paths lead to SLA success or failure.

### Proposed logic

Develop a predictive model that recommends:

- Likely required skill.
- Recommended tier.
- Likely resolver group.
- Expected resolution time.
- SLA breach risk.
- Whether L1 should attempt resolution or escalate immediately.
- Suggested knowledge article or diagnostic checklist.

Inputs could include:

- Ticket title and description.
- Category.
- Priority.
- Channel.
- Configuration item.
- User department/location.
- Historical incidents.
- Error codes.
- Affected application/service.
- Previous resolver skill.

Example:

A new ticket says:

> ôUsers cannot access CRM reports after database timeout error.ö

The model may predict:

- Category: Software-App / CRM.
- Required skill: `Database-SQL`.
- Recommended assignment: L2 Database/CRM specialist.
- L1 action: verify impact and collect logs, then route directly.

### Data required

- Historical ticket text.
- Historical categories and required skills.
- Final resolving agent/team/skill.
- Resolution codes.
- Escalation history.
- Processing and waiting times.
- SLA outcome.
- Knowledge base mappings.
- CMDB/service mappings.

### Expected benefits

- Improved first assignment accuracy.
- Shorter time to correct skill.
- Reduced manual dispatcher effort.
- Fewer avoidable L1/L2/L3 handoffs.
- Better SLA compliance for P2/P3.
- More consistent triage decisions.

## Strategy 4: Refined Escalation Criteria and L1 Empowerment

### Assignment issue addressed

This addresses:

- Excessive escalation from L1.
- L1 spending time on tickets they cannot resolve.
- L1 escalating tickets they could resolve with better guidance.
- L2/L3 specialists handling avoidable low-level work.

### How it uses process mining insights

Process mining can identify:

- Ticket types with high L1 resolution success.
- Ticket types with near-zero L1 resolution success.
- L1 agents with high first-call resolution rates.
- Categories where L1 troubleshooting time does not improve outcomes.
- Repeated escalations caused by missing L1 permissions or knowledge.

### Proposed logic

Create data-driven L1 handling rules:

1. **Resolve at L1** when historical L1 success is high.
   - Example: password resets, standard access requests, known software fixes.
2. **Use guided troubleshooting** when L1 can resolve with checklist support.
   - Example: common CRM client issues.
3. **Escalate immediately** when historical L1 resolution probability is very low.
   - Example: firewall rule changes, SQL performance incidents, IAM policy failures.
4. **Require diagnostic information before escalation** to reduce L2 rework.
   - Logs, screenshots, error code, affected user count, configuration item.

### Data required

- Historical L1 resolution rates by category/skill.
- Knowledge base article usage.
- L1 work duration before escalation.
- Escalation outcome.
- Resolution codes.
- L1 permissions and access rights.
- Training records.

### Expected benefits

- Higher L1 first-call resolution.
- Lower unnecessary L2/L3 workload.
- Faster escalation for truly complex tickets.
- Reduced specialist interruption.
- Improved ticket quality at escalation.
- Better customer experience.

## Strategy 5: Dynamic Skill-Based Capacity Management

### Assignment issue addressed

This addresses:

- Skill shortages at certain times.
- Uneven demand across shifts.
- Specialists overloaded during peak periods.
- Underused agents with relevant skills.
- Poor alignment between staffing and demand.

### How it uses process mining insights

The event log reveals:

- Demand patterns by hour, day, and month.
- Skills with recurring backlog.
- Periods with high SLA breach risk.
- Skills where demand exceeds supply.
- Agents with transferable skills.
- Seasonal incident patterns.

### Proposed logic

Use demand forecasts to adjust capacity dynamically.

Examples:

- Add more `Networking-Firewall` coverage during known peak periods.
- Temporarily move cross-trained L1 agents to handle access management tasks.
- Create a floating L2 pool for P2/P3 backlog reduction.
- Reserve expert capacity for P1/P2 incidents.
- Schedule specialists based on forecasted skill demand.
- Introduce ôswarmingö sessions for aging tickets before SLA breach.

### Data required

- Historical arrival rates by skill, priority, and time.
- Agent skill matrix.
- Agent calendars.
- Backlog and aging data.
- SLA risk forecast.
- Cross-training availability.
- Major incident calendar or planned change calendar.

### Expected benefits

- Better alignment of staffing with demand.
- Reduced queue time for scarce skills.
- Lower SLA breach rates during peaks.
- Reduced overload of key specialists.
- More flexible use of multi-skilled agents.

## Strategy 6: Continuous Skill Profile Updating Based on Actual Resolution Behavior

### Assignment issue addressed

This addresses:

- Outdated skill matrices.
- Incorrect proficiency assumptions.
- Hidden experts not formally recognized.
- Agents assigned based on stale profile data.

### How it uses process mining insights

Process mining shows actual work patterns:

- Which agents resolve which skills successfully.
- Average resolution time by agent and skill.
- Reopen rate after agent resolution.
- Escalation rate from agent to others.
- Frequency of specialist work performed.

### Proposed logic

Create a feedback loop that updates skill profiles using evidence.

For each agent-skill combination, calculate:

- Number of tickets resolved.
- Resolution success rate.
- Average resolution time.
- SLA performance.
- Reopen rate.
- Peer reassignment frequency.
- Recent training completion.

Then update proficiency indicators or recommend validation by team leads.

Example:

> Agent A07 has resolved 80 `Access Management` tickets in the last quarter with low reopen rate and strong SLA performance. Recommend adding `Security-IAM Basic` to profile.

### Data required

- Historical resolution records.
- Ticket required skill.
- Final resolver.
- Reopen data.
- SLA outcome.
- Training/certification records.
- Manager validation workflow.

### Expected benefits

- More accurate routing.
- Better use of hidden skills.
- Improved workforce planning.
- More targeted training.
- Reduced dependency on a few named specialists.

# 5. Simulation, Implementation, and Monitoring

## 5.1 Business Process Simulation Before Implementation

Before changing assignment logic, I would use business process simulation to test proposed strategies safely.

The simulation should be based on the mined process model and actual resource behavior.

### Simulation inputs

| Input | Source |
|---|---|
| Ticket arrival rates | Event log by hour/day/channel/priority |
| Ticket mix | Category, priority, required skill |
| Routing probabilities | Discovered process variants |
| Processing time distributions | START/COMPLETE event pairs |
| Waiting time distributions | Assignment to work start |
| Reassignment probabilities | Historical handover/reassignment data |
| Escalation probabilities | Historical L1/L2/L3 flows |
| Agent availability | Shift calendars and roster |
| Skill profiles | Agent skill matrix |
| SLA targets | ITSM policy |
| Queue discipline | FIFO, priority-based, SLA-risk-based, etc. |

### Scenarios to simulate

1. **Baseline**
   - Current round-robin/manual assignment.
2. **Skill-based routing**
   - Assign only to qualified agents where possible.
3. **Skill + workload-aware routing**
   - Assign to qualified agents with available capacity.
4. **Predictive triage**
   - Direct routing to likely resolver group.
5. **L1 empowerment**
   - Higher L1 resolution for selected categories.
6. **Dynamic staffing**
   - Shift capacity to high-demand skills during peaks.
7. **Combined future-state model**
   - Skill-based, workload-aware, predictive assignment with improved L1 rules.

### Simulation outputs

Compare each scenario on:

- SLA compliance by priority.
- Average and P90 resolution time.
- Time to first qualified agent.
- Queue time by skill.
- Resource utilization by agent/tier/skill.
- Number of reassignments.
- Escalation volume.
- L1 first-call resolution.
- Specialist workload.
- Backlog aging.
- Cost per ticket if cost data is available.

The simulation should be run multiple times using Monte Carlo methods to account for variability.

Example target comparison:

| KPI | Baseline | Skill-Based Routing | Skill + Workload-Aware |
|---|---:|---:|---:|
| P2 SLA compliance | 82% | 88% | 93% |
| P3 SLA compliance | 76% | 84% | 89% |
| Avg reassignments per ticket | 0.8 | 0.4 | 0.3 |
| Avg time to qualified agent | 2.1 hrs | 1.2 hrs | 0.8 hrs |
| L2 utilization variance | High | Medium | Low |

The goal is to quantify expected improvement and identify unintended side effects before rollout.
