### Logic used (case grouping + activity naming)

**Case identification (grouping into process instances):**
- I used an **artifact-centric approach**: each primary work object becomes a case.
  - Word documents  cases by filename (`Document1.docx`, `Quarterly_Report.docx`)
  - Excel workbook  case by filename (`Budget_2024.xlsx`)
  - PDF  case by filename (`Report_Draft.pdf`)
  - Email handling  one case for the specific email topic inferred from the click action (“Annual Meeting”).
- This yields coherent, analyst-friendly cases like “edit Document1”, “reply to Annual Meeting email”, etc., which is a common and effective case notion for desktop activity mining.

**Activity naming (low-level  higher-level):**
- Mapped raw actions to standardized business-readable activities:
  - `FOCUS` on a file window  **Open document / Open spreadsheet** (start/return to working on the artifact)
  - `TYPING`  **Draft content / Update spreadsheet data / Write reply** (specialized by application and context text)
  - `SAVE`  **Save document / Save spreadsheet**
  - `CLICK` with “Open Email/Reply/Send”  **Open email / Start reply / Send email**
  - `SCROLL`  **Review email content / Review PDF**
  - `HIGHLIGHT`  **Mark key text**
  - `CLOSE`  **Close document**
- I **did not keep** technical transitions like `SWITCH` as separate activities unless it meaningfully signaled opening a different artifact (e.g., switch into Acrobat treated as “Open PDF”).

This produces an event log where each case tells a clear story (create/edit/save/close; open/review/reply/send; review/highlight), suitable for discovery, conformance, and performance analysis.
