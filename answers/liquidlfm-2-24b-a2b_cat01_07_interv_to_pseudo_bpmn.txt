**BPMN-like Textual Process Model: Invoice Approval Process**

---

**Activities / Tasks:**  
1. **Receive Invoice** (Accounts Payable Clerk ñ Mary)  
2. **Validate PO Number** (Accounts Payable Clerk ñ Mary)  
   - If PO missing: **Request Missing PO Details** (to Supplier)  
   - If PO present: **Log Invoice in Accounting System**  
3. **Confirm Goods Received** (Purchasing Officer ñ Dan)  
   - If PO missing: **Ask Mary for PO**  
   - If goods not received: **Hold Invoice Until Receipt Confirmed**  
4. **Match Invoice to PO** (Accounts Payable Clerk ñ Mary)  
   - If mismatch: **Escalate to Department Manager ñ Rita**  
5. **Notify Manager for Approval** (System Automation)  
   - If approval needed: **Send Approval Request to Manager**  
6. **Manager Approval** (Department Manager ñ Carlos)  
   - If discrepancy: **Reject or Request Further Investigation**  
   - If all clear: **Approve Invoice in System**  
7. **Return to Purchasing for Payment Scheduling** (Accounts Payable Clerk ñ Mary)  
8. **Exception Handling** (Accounts Payable Supervisor ñ Rita)  
   - For unresolved discrepancies or missing PO: **Contact Supplier / Coordinate with Departments**  
   - For informal contracts: **Create Retrospective PO or Reject Invoice**  
9. **Supplier Communication** (Supplier Relationship Manager ñ Karen)  
   - For missing PO or pricing issues: **Provide Missing Information / Revised Invoice**  

---

**Sequence Flow:**  
1. Receive Invoice   
2. Validate PO Number   
   - If PO missing:  Request Missing PO Details  
   - If PO present:  Log Invoice in Accounting System   
3. Confirm Goods Received   
   - If goods not received:  Hold Invoice Until Receipt Confirmed   
4. Match Invoice to PO   
   - If mismatch:  Escalate to Manager  
   - If match:  Notify Manager for Approval   
5. Manager Approval   
   - If approved:  Return to Purchasing for Payment Scheduling  
   - If exception:  Escalate to Supervisor for Resolution   
6. Exception Handling   
   - Supplier Communication (if needed)   
7. Invoice Scheduled for Payment  

---

**Gateways / Decision Points:**  
- **PO Validation:**  
   - PO present?  Log & Match  
   - PO missing?  Request Details  
- **Goods Received Confirmation:**  
   - Received?  Match to PO  
   - Not Received?  Hold Invoice  
- **PO Match:**  
   - Match?  Manager Approval  
   - Discrepancy?  Escalate to Manager  
- **Approval Result:**  
   - Approved?  Payment Scheduling  
   - Discrepancy/Exception?  Escalate to Supervisor  
- **Exception Handling:**  
   - Supplier Contact Needed?  Supplier Communication  
   - Informal Contract?  Retrospective PO or Reject  

---

**Roles / Participants:**  
- **Accounts Payable Clerk (Mary)** ñ Initial receipt, PO validation, logging, goods confirmation, initial matching, escalation  
- **Purchasing Officer (Dan)** ñ PO verification, goods receipt confirmation, invoice confirmation  
- **Department Manager (Carlos)** ñ Final approval  
- **Accounts Payable Supervisor (Rita)** ñ Exception handling, supplier coordination  
- **Supplier Relationship Manager (Karen)** ñ Supplier communication for missing PO or pricing issues  
- **Supplier** ñ Provides missing information or revised invoice  
- **System Automation** ñ Sends approval requests  

---

**Artifacts / Inputs / Outputs:**  
- **Inputs:**  
   - Invoice (from Supplier or Purchasing)  
   - Purchase Order (PO)  
- **Artifacts:**  
   - System notifications/emails  
   - Approval records  
   - Supplier communications  
- **Outputs:**  
   - Approved Invoice (for payment)  
   - Rejected Invoice (with notes)  
   - Retrospective PO (if applicable)  

---

**Exceptions / Variations:**  
- PO missing  Supplier contact for details  
- Discrepancy in quantity/price  Escalation to Manager or Supplier  
- Informal contracts  Retrospective PO creation or rejection  
- Supplier unresponsive  Escalation to Supplier Relationship Manager  
- Discrepancy too large or suspicious  Manager rejects or holds for investigation  

---

**Visualization Notes:**  
- The process can be represented as a linear flow with parallel branches for exception handling.  
- Decision diamonds should be clearly marked for PO presence, goods receipt, PO match, and approval results.  
- Exception paths should loop back to the main approval flow after resolution.
