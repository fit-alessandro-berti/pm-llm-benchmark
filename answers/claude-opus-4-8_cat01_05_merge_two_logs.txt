# Integrated Event Log: Order Fulfillment Process

## Step 1: Matching Analysis

I aligned events using **order_id** (identical across all: Order#1234) and **timestamp proximity** (2 second tolerance), while reconciling naming differences semantically.

| Log A Event | Time A | Log B Event | Time B |  Time | Decision |
|---|---|---|---|---|---|
| Order Received | 10:00:00 | OrderReceived | 09:59:58 | 2s | **MATCH** (within tolerance, clear semantic match) |
| Order Validated | 10:01:30 | OrderValidation | 10:01:29 | 1s | **MATCH** (within tolerance) |
| Payment Processed | 10:02:00 | PaymentCheck | 10:02:05 | 5s | **MATCH** (exceeds tolerance, but strong semantic + sequence match — see note) |
| — | — | Quality Check | 10:03:00 | — | **B-ONLY** (no Log A counterpart) |
| Item Shipped | 10:05:00 | Shipping | 10:05:02 | 2s | **MATCH** (within tolerance) |
| Item Delivered | 10:20:00 | — | — | — | **A-ONLY** (no Log B counterpart) |

### Reasoning on Edge Cases

- **Order Received (2s gap):** Exactly at tolerance boundary. Log B timestamp is *earlier* (09:59:58), consistent with the stated assumption that Log A reflects *received* time (later) while Log B captures *start* time at the warehouse. Matched confidently.
- **Payment Processed  PaymentCheck (5s gap):** This **exceeds the 2-second tolerance**. However, I matched them because:
  - The semantic mapping is strong (payment activity).
  - The sequence position is identical (between validation and shipping).
  - Log B's note ("Payment gateway delay") explains the offset — a real processing delay rather than two distinct events.
  - This is a **lower-confidence match**, flagged accordingly. (A stricter, tolerance-only approach would leave these separate.)
- **Quality Check:** Exists only in Log B. No nearby Log A event (nearest are Payment at 10:02 and Ship at 10:05). Kept separate as a B-only enrichment event.
- **Item Delivered:** Exists only in Log A; Log B's recording ends at Shipping. Kept as A-only.

### Conflict Resolution Rules

- **Primary timestamp:** Log A is the designated "primary timeline," so I use Log A timestamps for sorting when a match exists; both timestamps are retained in the record.
- **Event naming:** Adopted Log A's human-readable name as canonical (`event_type`), retaining Log B's `event_name` as an alias.
- **Attributes:** All Log B attributes (`user_id`, `resource_id`, `notes`) merged into matched records.

## Step 3: Summary of Decisions

| Topic | Approach Taken |
|---|---|
| **Matching key** | order_id + timestamp proximity + semantic name mapping + sequence position |
| **Timestamp tolerance** | 2s default; relaxed once (Payment) with explicit justification & lower confidence flag |
| **Primary timestamp** | Log A (the designated primary timeline); Log B time retained as secondary |
| **Naming** | Log A name canonical; Log B name kept as `b_event_name` alias |
| **Attribute enrichment** | All Log B fields (user_id, resource_id, notes) folded into matched records |
| **Non-overlapping events** | Retained with `source` tag (`A-only` / `B-only`) — no data discarded |
| **Confidence tracking** | Added `match_confidence` to make uncertain merges auditable |

**Key takeaways:** 4 events merged (3 high-confidence, 1 medium), 2 events kept standalone. No source data was lost. The medium-confidence payment match and the two standalone events are the items most worth human review.
