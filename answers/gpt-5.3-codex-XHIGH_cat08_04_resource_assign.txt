Great scenario for **organizational process mining**.  
I’d treat this as: **(a) diagnose as-is assignment behavior, (b) quantify impact, (c) redesign routing logic, (d) simulate, then (e) monitor continuously.**

## 1) Analyze resource behavior and assignment patterns

### 1.1 Build an analysis-ready log (critical first step)
From your raw events, derive per-ticket/per-assignment features:

- **Queue wait** = `Work Start - Assign`
- **Touch time** = `Work End - Work Start`
- **Handover delay** = `Next Assign - Escalate/Reassign`
- **Reassignment count** per case
- **Escalation count** and tier path (L1L2L3 etc.)
- **Skill match flag** at assignment time: required skill in assigned agent skill set?
- **Workload-at-assignment**: agent’s active tickets when assigned
- **SLA remaining time** at each handover

Also do data checks: missing timestamps, inconsistent skill names, skill profile currency.

---

### 1.2 Metrics for agent/tier performance
Track metrics by **agent, tier, priority, category, skill**:

1. **Workload / utilization**
   - Tickets handled per day/week
   - Active WIP (concurrency)
   - Utilization % (touch time / available time)
   - Workload fairness (e.g., coefficient of variation or Gini across agents)

2. **Speed**
   - First-touch latency (Created  first Work Start)
   - Assignment wait (Assign  Work Start)
   - Touch time by activity/tier
   - End-to-end cycle time

3. **Quality**
   - **L1 First-Call Resolution (FCR)**  
     `Resolved in L1 without escalation / total L1-started cases`
   - Reopen rate
   - Reassignment frequency

4. **Assignment quality**
   - First-assignment-right rate
   - Skill mismatch rate
   - Avoidable escalation rate (cases escalated but historically similar cases solved at L1)

5. **SLA**
   - Breach rate by priority/category/required skill
   - Breach rate by path variant and by assignment pattern

> Important: normalize by case mix (P2 network is harder than P4 access).  
> Compare agents using “actual vs expected” based on ticket complexity.

---

### 1.3 Process mining techniques to reveal actual behavior
Use:

- **Process discovery** with performance overlay:
  - See real paths, loops, escalation chains, where time accumulates.
- **Variant analysis**:
  - Common “good” path vs reassignment-heavy path.
- **Handover social network analysis (SNA)**:
  - Nodes = agents/tiers, edges = handovers.
  - Edge weight = handover count; edge color = median delay.
  - Highlights bottleneck people/teams and ping-pong patterns.
- **Role discovery / org mining**:
  - Infer actual roles from performed activities and compare with official L1/L2/L3.
  - Detect “shadow specialists” or underused specialists.
- **Conformance checking against intended logic**:
  - Intended: round-robin + manual escalation.
  - Observed: e.g., repeated reassign loops, self-assignment bias, skill-unaware routing.

---

### 1.4 Skill utilization analysis
Create a **Skill Demand vs Skill Supply** view:

- Demand: tickets requiring `Networking-Firewall`, `DB-SQL`, etc. by hour/day/priority.
- Supply: available agent capacity per skill and tier.

Then compute:

- **Skill shortage indicators**: high queue wait + high specialist utilization
- **Underqualification assignments**: required skill absent at assignment
- **Overqualification usage**: L2/L3 time spent on basic tasks
- **Skill profile drift**: agents resolving tickets outside documented skills (profiles outdated)

## 2) Identify resource-related bottlenecks and quantify impact

Pinpoint issues with targeted diagnostics:

1. **Skill-specific bottlenecks**
   - Example: `Assign L2  Work L2 Start` for Firewall tickets is long.
   - Evidence: long waits + few agents repeatedly overloaded.

2. **Reassignment/escalation delay**
   - Compare cycle time by reassignment count (0 vs 1 vs 2+).
   - Quantify incremental delay:  
     ` delay per reassignment = median(cycle|k reassign) - median(cycle|0 reassign)`

3. **Incorrect initial assignments**
   - Flag tickets where first assignee lacks required skill or gets reassigned quickly.
   - Measure extra hops/time and breach impact.

4. **Overloaded vs underutilized resources**
   - Utilization heatmap by agent and shift.
   - Identify chronic overload (high WIP, long queue) and chronic idle capacity.

5. **Assignment pattern  SLA breach correlation**
   - Model breach probability using features like:
     - skill mismatch
     - reassignment count
     - first-touch delay
     - workload-at-assignment
   - Quantify effects, e.g., odds increase per reassignment.

Useful impact KPIs:

- `% breaches with skill mismatch`
- `% breaches with 2+ handovers`
- `Median SLA overrun by variant`
- `Top 10 delay-causing handover pairs`

## 3) Root cause analysis for assignment inefficiencies

### Likely root causes and how to validate with mining

- **Round-robin ignoring skills/workload**
  - Evidence: assignment decisions not correlated with skill fit or load.
- **Inaccurate/incomplete skill profiles**
  - Evidence: frequent reassign “needs different skill”; or hidden expertise not in profiles.
- **Poor initial categorization / skill requirement detection**
  - Evidence: early corrections + reassignments right after first touch.
- **No real-time workload visibility**
  - Evidence: assignments to overloaded agents while qualified idle agents exist.
- **L1 capability/training gaps**
  - Evidence: high escalation in categories where peer L1s solve directly.

### Variant analysis approach
Compare two cohorts:

- **Smooth cases**: 0–1 handover, on-time SLA
- **Problematic cases**: 2+ reassignments, SLA breach

Compare category, priority, channel, shift, assigned load, skill match, and assigned tier at intake.  
This isolates factors associated with bad assignment paths.

### Decision mining approach
At decision points (`Assign L1`, `Assign L2`, `Escalate`), build decision rules from observed behavior.

If discovered rules are like “next in queue/dispatcher preference” rather than “best skill + SLA risk + workload”, you’ve found rule design problems.

## 4) Data-driven strategies to optimize assignment (5 concrete options)

| Strategy | Issue Addressed | How It Uses Mining Insights | Data Required | Expected Benefit |
|---|---|---|---|---|
| **1) Skill- & proficiency-based routing** | Skill mismatches, reassignments | Uses agent-skill success history and mismatch hotspots to route to best-fit qualified agent | Required skill, agent skills + proficiency, historical resolution outcomes | Fewer reassignments, faster resolution, better P2/P3 SLA |
| **2) Workload- and SLA-aware dispatching** | Uneven load, delayed starts | Uses mined wait-time distributions and overload patterns to assign to least-delay qualified agent | Real-time WIP, queue lengths, calendars/availability, SLA deadlines | Lower queue time, balanced utilization, fewer breaches |
| **3) Predictive triage at intake (ML/NLP)** | Wrong initial categorization | Uses historical tickets to predict required skill/tier/complexity before first assignment | Ticket text, category, channel, past paths/outcomes | Higher first-assignment-right, fewer escalations |
| **4) Escalation redesign + L1 enablement** | Excessive escalations, specialists doing basic work | Uses variants where L1 resolved similar cases; codifies playbooks and escalation thresholds | L1 success patterns, escalation reasons, KB usage | Higher L1 FCR, reduced L2/L3 low-value workload |
| **5) Dynamic capacity by skill (flex staffing/cross-skilling)** | Skill bottlenecks by time/day | Uses demand seasonality and skill bottleneck maps to adjust shifts and temporary tier support | Arrival patterns, skill-demand forecasts, staffing rosters | Reduced peak-time backlogs, steadier SLA performance |

### Practical assignment scoring (for strategy 1+2)
Use a score like:

`Score(agent,ticket) = 0.45*SkillFit + 0.25*(1-Load) + 0.20*SLAUrgencyFit + 0.10*HistoricalSuccess`

- Hard constraint: no assignment if mandatory skill missing (except emergency fallback).
- Force reason code when overrides happen (great for later mining).

## 5) Simulation, implementation, and monitoring

### 5.1 Simulate before rollout (digital twin)
Build a **discrete-event simulation** from mined process behavior:

1. Mine as-is control flow (including escalation/reassignment probabilities)
2. Fit arrival patterns (hour/day/season), service-time distributions, and queue rules
3. Model resource calendars and skill constraints
4. Validate baseline model against actual KPIs (last quarter)
5. Run scenarios:
   - Baseline
   - Skill-based routing
   - + Workload/SLA aware
   - + Predictive triage
   - + Flex staffing
6. Compare outcome distributions (not single-point estimates)

Track in simulation:
- SLA compliance by P-level
- Median and P90 cycle time
- Reassignment rate
- Queue wait by required skill
- L2/L3 utilization and % time on low-complexity work

---

### 5.2 Post-implementation monitoring with process mining dashboards
Set up continuous dashboards with weekly/monthly governance.

**Core KPIs**
- First-assignment-right %
- Reassignments per 100 tickets
- Escalation rate by category/priority
- L1 FCR
- AssignWork Start wait by tier/skill
- SLA breach rate by priority/category/variant
- Utilization and workload fairness across agents
- Specialist time on low-complexity tickets
- Policy conformance violations (e.g., assignment without required skill)

**Essential process views**
- Process map heatmap (waiting-time hotspots)
- Variant explorer for breached cases
- Handover network (who hands off to whom + delay)
- Skill bottleneck heatmap (demand vs capacity)
- Pre/post cohort comparison (before vs after each rollout phase)

**Operating cadence**
- Daily: queue and SLA risk dashboard
- Weekly: assignment quality and bottleneck review
- Monthly: retrain triage model, refresh skill matrix, tune routing weights

If executed in this sequence, TechSolve can move from reactive dispatching to a **skill-fit, SLA-aware, workload-balanced assignment system** with measurable impact on P2/P3 breaches and reassignment churn.
