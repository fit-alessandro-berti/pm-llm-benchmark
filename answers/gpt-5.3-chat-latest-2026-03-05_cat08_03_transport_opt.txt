1. Process Discovery and Conformance Checking

Data preprocessing and integration  
The first step is constructing a unified event log where each event has at least: Case ID, Activity, Timestamp, Resource, and Attributes.

Steps:
- Case definition: Use ōVehicle-Dayö (e.g., V1220241205) as the primary case because the operational process is a delivery shift. A secondary case view can also be created per package.
- Event standardization: Map events from different systems into unified activities:
  - GPS  Depart Depot, Driving, Traffic Delay, Idle, Unscheduled Stop
  - Scanner  Arrive Customer, Delivery Success, Delivery Failed, Depart Customer
  - Dispatch  Route Assigned, Planned Stop Sequence
  - Maintenance  Maintenance Start, Maintenance End
- Timestamp alignment: Ensure consistent time zones and synchronize clocks between GPS devices, scanners, and backend systems.
- Enrichment:
  - Attach package attributes (priority, time window)
  - Attach driver, vehicle, and route attributes
  - Calculate derived attributes such as travel distance, dwell time, idle duration.

Challenges likely encountered:
- Heterogeneous data formats and schemas.
- Missing or inconsistent timestamps.
- GPS noise or excessive event granularity.
- Event correlation (linking scanner events to GPS positions).
- Multiple packages delivered at the same stop.
- Distinguishing legitimate stops vs. traffic-related halts.

After cleaning, the final event log might look like:
Case ID | Activity | Timestamp | Driver | Vehicle | Location | Package | Speed | Stop ID.

Process discovery  
Using algorithms such as:
- Inductive Miner
- Heuristic Miner
- Fuzzy Miner (for complex paths)

The discovered model would reveal the real operational flow:

Start Shift  
 Route Assigned  
 Depart Depot  
 Driving  
 Arrive Customer  
 Delivery Success / Delivery Failed  
 Depart Customer  
 Repeat deliveries  
 Optional deviations:
   - Traffic Delay
   - Idle
   - Unscheduled Stop
   - Maintenance  
 Return to Depot  
 End Shift

Enhancements:
- Overlay average durations between activities.
- Show frequency of different delivery sequences.
- Visualize loops (multiple stops).
- Highlight rare events like breakdowns.

Conformance checking with planned routes  
The dispatch system provides the planned model: planned sequence of stops and expected arrival windows.

Conformance techniques:
- Token-based replay
- Alignment-based conformance checking

Deviations to examine:
- Sequence deviations:
  - Stops visited in different order
  - Skipped stops
  - Unplanned stops
- Timing deviations:
  - Arrival outside delivery window
  - Excessive dwell time
  - Travel time significantly longer than expected
- Behavioral deviations:
  - Unauthorized detours
  - Idle time between stops
  - Maintenance events during delivery runs.

This comparison highlights operational inefficiencies and compliance issues with planning.

2. Performance Analysis and Bottleneck Identification

Key KPIs

On-Time Delivery Rate  
(number of deliveries completed within customer time window) / (total deliveries)

Average Time per Delivery Stop  
average(Depart Customer ¢ Arrive Customer)

Travel Time vs Service Time Ratio  
total driving time / total customer service time

Fuel Consumption per km or per package  
total fuel used / distance or packages delivered  
(can be estimated using speed, idle time, and vehicle telemetry)

Vehicle Utilization Rate  
time vehicle actively delivering / total shift duration

Traffic Delay Frequency  
count of ōLow Speedö or ōTraffic Delayö events per route or zone.

Average Idle Time  
sum of idle periods / shift.

Failed Delivery Rate  
failed deliveries / total attempts.

These metrics can be calculated directly from event timestamps and GPS-derived metrics.

Identifying bottlenecks

Process mining performance overlays help reveal where time accumulates.

Techniques:
- Throughput time analysis (time between activities)
- Waiting time analysis
- Queue analysis between stops
- Variant analysis by route or region.

Possible bottleneck patterns:
- Long travel time between specific zones
- High dwell time at particular customer types (e.g., offices)
- Repeated failed deliveries in residential areas during working hours
- Idle time caused by parking difficulties.

Segmentation analysis:
- By time of day
- By geographic cluster
- By driver
- By vehicle type
- By route.

Impact quantification:
Example:  
If traffic delays add 12 minutes per route and occur in 40% of routes, the annual cost impact can be calculated using fuel cost + overtime hours.

3. Root Cause Analysis for Inefficiencies

Suboptimal route planning  
Static routing may ignore real traffic conditions.

Validation method:
- Compare planned vs actual travel time between stops.
- Identify recurring detours or reordering of stops.

Inaccurate travel time estimations  
If dispatch assumes 10 minutes but logs show 18 minutes consistently, planning assumptions are flawed.

Analysis:
- Compare planned travel time vs actual travel time distributions.

Traffic congestion patterns  
Using GPS low-speed events and location clustering:
- Detect recurring congestion zones.
- Identify peak congestion times.

Service time variability at customers  
Dwell time = Depart Customer ¢ Arrive Customer.

Cluster analysis may reveal:
- Apartments vs houses
- Businesses vs residential
- Buildings without parking.

Vehicle breakdowns during routes  
Maintenance logs combined with route logs reveal:
- frequency of mid-route breakdowns
- vehicles with abnormal repair patterns.

Driver behavior differences  
Variant analysis across drivers:
- idle time differences
- delivery speed
- route deviations
- acceleration/harsh driving patterns.

Failed delivery attempts  
Process mining can show patterns such as:
Delivery Failed  Return later  Reattempt.

Correlations may reveal:
- time-of-day mismatch
- poor address data
- lack of customer notification.

4. Data-Driven Optimization Strategies

Strategy 1: Dynamic route optimization using real-time traffic data

Target inefficiency  
Traffic-related delays and inefficient stop ordering.

Root cause  
Static route planning ignores congestion patterns.

Process mining evidence  
Event logs show consistent delays in certain zones during specific hours.

Implementation
- Integrate real-time traffic APIs.
- Recalculate routes during the shift.
- Use historical travel-time models learned from logs.

Expected impact
- Reduced travel time
- Lower fuel consumption
- Higher on-time delivery rate.

Strategy 2: Delivery time-window optimization and customer communication

Target inefficiency  
High failed delivery rate.

Root cause  
Customers not present during delivery windows.

Process mining evidence  
High frequency of failed deliveries in residential zones during daytime.

Implementation
- Send pre-arrival notifications.
- Allow customers to reschedule delivery slots.
- Use lockers or pickup points.

Expected impact
- Reduced re-delivery trips
- Lower operational costs
- Higher customer satisfaction.

Strategy 3: Predictive maintenance and vehicle health monitoring

Target inefficiency  
Unscheduled stops and mid-route breakdowns.

Root cause  
Reactive maintenance strategy.

Process mining evidence  
Maintenance logs show repeated breakdowns after certain mileage or idle patterns.

Implementation
- Predict maintenance using usage patterns and engine alerts.
- Schedule servicing during non-delivery hours.

Expected impact
- Reduced downtime
- Fewer delivery disruptions
- Lower repair costs.

Strategy 4: Route territory redesign using historical performance

Target inefficiency  
Uneven workload across routes.

Root cause  
Territories designed without considering real delivery times.

Process mining evidence  
Throughput analysis shows some routes consistently exceeding shift time.

Implementation
- Recluster delivery zones using geographic and service-time data.
- Balance routes based on historical stop durations.

Expected impact
- Reduced overtime
- Improved vehicle utilization
- More balanced driver workload.

5. Operational Constraints and Continuous Monitoring

Accounting for operational constraints

Any optimization must respect:
- Driver working hour regulations
- Vehicle capacity constraints
- Customer delivery windows
- Depot departure and return schedules.

Optimization models should incorporate these constraints during route generation.

For example:
- Dynamic routing must ensure routes still end within driver shift limits.
- Territory balancing must respect vehicle capacity.

Continuous monitoring using process mining dashboards

Key dashboards should track:

Process performance
- On-time delivery rate
- Average route duration
- Travel vs service time ratio.

Operational efficiency
- Idle time per route
- Fuel consumption per km
- Vehicle utilization.

Reliability metrics
- Failed delivery rate
- Breakdown frequency
- Traffic delay hotspots.

Process views
- Updated process maps with performance overlays
- Route variant analysis
- Geographic heatmaps of delays.

Automated alerts could trigger when:
- Delivery time exceeds thresholds
- Idle time spikes
- Certain routes show repeated delays.

Continuous monitoring ensures improvements remain effective and helps detect new inefficiencies as delivery demand patterns evolve.
