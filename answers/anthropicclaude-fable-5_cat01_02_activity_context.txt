# Contextual Activity Disambiguation in the Support Workflow Log

## 1. Identifying Recurring Activities and Their Contextual Shifts

Three activities repeat within cases, each carrying distinct meanings:

### **Verify Documents** (recurs in C001 and C003)
| Occurrence | Context | Meaning |
|---|---|---|
| First occurrence (all cases) | Initial check after request submission | Standard initial verification |
| Second in C001 | After customer uploads missing ID scan | **Re-verification** triggered by new documents |
| Second in C003 | After internal account adjustment | **Re-verification** triggered by internal correction |

### **Call Customer** (recurs in C002; appears with different purposes across cases)
| Occurrence | Context | Meaning |
|---|---|---|
| C001 | Verification incomplete | Call to **request missing documents** |
| C002, first | Account discrepancy found | Call to **clarify a discrepancy** |
| C002, second | After account adjustment | **Follow-up/confirmation** call |
| C003 | Customer left the branch | Call to **recall customer to branch** (logistical, not informational) |

### **Initiate Request** (does not recur within a case, but differs across cases by channel)
- C001/C002: online portal; C003: branch walk-in. Channel-aware renaming improves clarity, though it is optional since there's no intra-case ambiguity.

## 4. Justification of Naming Choices

| New Name | Contextual Clues Used |
|---|---|
| **Verify Documents – Initial** | First occurrence in every case; immediately follows request initiation; no prior triggering event. |
| **Verify Documents – Re-check (New Docs)** | Occurs *after* "Submit Additional Documents"; context explicitly says "re-checks ... now that ID scan is provided." The trigger is customer-supplied material. |
| **Verify Documents – Re-check (Post-Adjustment)** | Occurs *after* "Adjust Account"; context says "re-verify docs after account adjustment." The trigger is an internal correction — a semantically different reason for re-verification than in C001. |
| **Call Customer – Request Missing Docs** | Follows a failed verification; context names the purpose (requesting missing documents). It's a *document-solicitation* call. |
| **Call Customer – Clarify Discrepancy** | Follows initial verification with a discrepancy noted; the call is *investigative*. |
| **Call Customer – Confirm Adjustment** | Occurs after "Adjust Account"; context says "follow-up call to confirm adjustments." This is a *confirmatory/closing* call, the second call in the same case. |
| **Call Customer – Recall to Branch** | Unique logistical purpose — the customer physically left; the call is neither investigative nor document-related. Notably performed by the BranchOfficer, not a FrontlineAgent. |
| **Initiate Request (Online / Branch Walk-in)** | Channel disambiguation based on Resource (WebPortal vs. BranchReception). Not strictly required (no intra-case repetition), but it makes inter-case variant analysis clearer. |
| **Submit Additional Documents** | Minor clarification of the truncated "Submit Additional" to make the object of submission explicit. |

### Key Disambiguation Principles Applied
1. **Positional context**: Whether the activity is a first occurrence or a repetition (initial vs. re-check).
2. **Triggering event**: What immediately precedes the activity (failed verification, account adjustment, new document upload).
3. **Purpose stated in the context field**: The free-text descriptions provide the semantic intent of each call.
4. **Resource analysis**: Different resources (BranchOfficer vs. FrontlineAgent; WebPortal vs. BranchReception) signal different process variants.

This renaming would allow process mining tools to correctly distinguish loops (genuine rework, e.g., re-verification) from semantically distinct steps that merely share a label — improving the accuracy of any discovered process model.
