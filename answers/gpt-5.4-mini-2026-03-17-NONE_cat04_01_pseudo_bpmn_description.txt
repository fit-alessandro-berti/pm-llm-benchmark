The pseudo-BPMN describes a customer request handling process with two main variants: a **standard request path** and a **custom request path**. Both paths begin the same way, but they diverge based on the request type and may later reconverge for approval, invoicing, and confirmation.

## Overall business process

The process starts when a customer request is received. The organization then classifies the request as either:

- **Standard**: a routine request that can be processed with predefined checks and pricing logic.
- **Custom**: a request that requires feasibility analysis because it may involve special requirements, customization, or exceptions.

After the request is evaluated, the process may require manager approval depending on business rules. If approval is not needed, the process continues directly. If approval is needed and granted, the final invoice is generated; if approval is denied, the request is reworked and sent back to an earlier step.

Finally, once the request is fully processed and approved, the customer is informed by sending a confirmation.

## Step-by-step explanation

### 1. Start Event
This marks the beginning of the process. It is triggered when a customer submits a request.

### 2. Task A: “Receive Customer Request”
This is the first operational task. The business captures the incoming request and logs the basic details needed for processing.

### 3. XOR Gateway: “Check Request Type”
An **XOR gateway** means **exclusive choice**: only one outgoing path is taken.

Here, the process decides whether the request is:

- **Standard**
- **Custom**

Only one of these paths proceeds.

---

## Standard request path

### 4. Task B1: “Perform Standard Validation”
For standard requests, the business performs routine validation. This may include checking required fields, verifying the request format, or confirming that the request fits predefined product/service rules.

### 5. AND Gateway: “Run Parallel Checks”
An **AND gateway** means **parallel execution**: all outgoing branches are activated simultaneously.

This gateway splits the process into two checks that happen in parallel:

- **Task C1: Credit Check**
- **Task C2: Inventory Check**

Both checks must complete before the process can continue.

### 6. Parallel tasks
- **Credit Check**: verifies the customer’s financial standing or payment eligibility.
- **Inventory Check**: confirms that the requested product or service is available.

### 7. Join after parallel checks
The note “All Parallel Checks Completed (Join)” indicates synchronization. The process waits until **both** the credit check and inventory check are finished. This is the convergence point after parallel execution.

### 8. Task D: “Calculate Delivery Date”
Once validation and both checks are complete, the organization calculates when delivery or fulfillment can occur.

This task is typical for standard requests because the process can often proceed using routine rules after validation.

---

## Custom request path

### 9. Task B2: “Perform Custom Feasibility Analysis”
For custom requests, the business cannot proceed with standard rules. Instead, it performs a feasibility analysis to determine whether the requested customization can realistically be delivered.

### 10. XOR Gateway: “Is Customization Feasible?”
This is another **exclusive decision gateway**. Based on the analysis, the process continues in one of two ways:

- **Yes**: the customization can be done
- **No**: the customization cannot be done

### 11. If Yes  Task E1: “Prepare Custom Quotation”
If the request is feasible, the business prepares a quotation tailored to the custom request. This may involve special pricing, additional lead time, or customized scope details.

### 12. If No  Task E2: “Send Rejection Notice”  End Event
If the request is not feasible, the business rejects it and informs the customer. The process ends immediately because there is no acceptable way to fulfill the request.

This is a clear termination branch.

## Convergence after standard or custom processing

The diagram says:

> “After Standard or Custom Path Tasks Completed”

This indicates that after the relevant path has been processed, the workflow reaches a common decision point. In BPMN terms, this suggests a convergence of the two main request-handling branches before moving to approval and invoicing.

Practically:

- Standard requests continue after **Task D**
- Custom feasible requests continue after **Task E1**
- Rejected custom requests end earlier and do not converge

---

## Approval decision

### 13. XOR Gateway: “Is Approval Needed?”
This gateway checks whether managerial approval is required before the process can continue.

Again, since this is XOR, only one path is taken:

- **Yes, approval is needed**
- **No, approval is not needed**

The approval requirement is likely based on business rules such as:

- high monetary value
- exceptional discount
- non-standard terms
- custom scope or risk

### 14. If Yes  Task F: “Obtain Manager Approval”
A manager reviews the request and decides whether it can proceed.

### 15. XOR Gateway: “Is Approval Granted?”
This gateway determines the outcome of the manager’s review:

- **If Yes**  proceed to invoicing
- **If No**  re-evaluation is required

### 16. If Yes  Task G: “Generate Final Invoice”
If approval is granted, the invoice is created and the request is considered ready for completion.

### 17. If No  Task H: “Re-evaluate Conditions”
If the manager rejects the request, the process is not necessarily terminated. Instead, the request is sent back for rework.

This indicates that the business may revise assumptions, pricing, feasibility, delivery date, or quotation details and then try again.

---

## Loop-back / rework behavior

The loop-back from **Task H: “Re-evaluate Conditions”** depends on which path the request came from:

- **Custom path**: loop back to **Task E1: “Prepare Custom Quotation”**
- **Standard path**: loop back to **Task D: “Calculate Delivery Date”**

This reflects that different parts of the process may need correction:

- For a **custom request**, the quotation may need to be adjusted and resubmitted.
- For a **standard request**, the delivery date or underlying assumptions may need to be recalculated.

This is a classic rework loop, allowing the business to make changes and re-enter the approval process.

---

## If approval is not needed

### 18. If No Approval Needed  Task G: “Generate Final Invoice”
If the request does not require managerial approval, the process skips the approval tasks and goes straight to invoice generation.

This means the request meets all conditions for automatic continuation.

---

## Final customer communication

### 19. Task I: “Send Confirmation to Customer”
After invoice generation, the process sends a confirmation to the customer. This may include:

- order confirmation
- invoice details
- expected delivery date
- next steps

This is the final operational step before the process ends.

### 20. End Event
The process concludes here. The customer request has either been fulfilled, rejected, or returned for rework earlier in the process.

# Meaning of the gateways

## XOR gateway
Used for decisions where only one branch can be taken.

Examples in this process:
- Request type: standard vs custom
- Feasibility: feasible vs not feasible
- Approval outcome: granted vs denied
- Approval needed: yes vs no

## AND gateway
Used for parallel execution where all branches must be completed.

Example:
- Credit check and inventory check run in parallel for standard requests.

The AND join waits until both complete before moving forward.

---

# How the paths differ

## Standard requests
These are handled with predefined rules:

1. Receive request
2. Validate
3. Run credit and inventory checks in parallel
4. Calculate delivery date
5. Possibly get approval
6. Generate invoice
7. Send confirmation

## Custom requests
These require analysis and may end earlier if infeasible:

1. Receive request
2. Perform feasibility analysis
3. If feasible, prepare custom quotation
4. Possibly get approval
5. Generate invoice
6. Send confirmation

If not feasible:
- send rejection notice
- end process

# Conditions for approvals and rework

## Approval is triggered when:
The business rules determine that managerial sign-off is necessary. The diagram does not specify exact criteria, but typical reasons include:

- custom requests with higher risk
- unusually large or expensive orders
- exceptions to standard policy
- special pricing or terms

## Rework is triggered when:
Approval is requested but **not granted**.

In that case:
- the process goes to **Re-evaluate Conditions**
- then loops back to the earlier task that needs revision

This allows the organization to adjust the quotation, feasibility assumptions, or delivery calculations and try approval again.
