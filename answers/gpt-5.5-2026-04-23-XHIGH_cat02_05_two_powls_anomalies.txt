A normative successful-hire path would normally be:

`Post_Job_Ad  Screen_Candidates  Conduct_Interviews  Make_Hiring_Decision  Onboard_Employee  Add_to_Payroll  Close_Case`

with `Close_Case` occurring only after all required hiring/onboarding/payroll work is complete.

## Model 1 analysis

Model 1 enforces:

`Post_Job_Ad  Screen_Candidates`

and then both `Conduct_Interviews` and `Make_Hiring_Decision` are enabled after screening. The downstream chain is:

`Make_Hiring_Decision  Onboard_Employee  Add_to_Payroll  Close_Case`

### Anomalies in Model 1

1. **Hiring decision is not dependent on interviews**  
   The model has:

   ```text
   Screen_Candidates  Make_Hiring_Decision
   Screen_Candidates  Conduct_Interviews
   ```

   but it does not have:

   ```text
   Conduct_Interviews  Make_Hiring_Decision
   ```

   Therefore, the hiring decision may be made before interviews are conducted.

2. **Onboarding, payroll, and closure may happen before interviews**  
   Because `Conduct_Interviews` is not ordered before `Make_Hiring_Decision`, the following trace is allowed:

   ```text
   Post_Job_Ad,
   Screen_Candidates,
   Make_Hiring_Decision,
   Onboard_Employee,
   Add_to_Payroll,
   Close_Case,
   Conduct_Interviews
   ```

   This is clearly abnormal: the case can be closed before interviews occur.

3. **However, all core activities are mandatory and mostly in the right chain**  
   Model 1 still requires posting, screening, interviews, decision, onboarding, payroll addition, and closure. The main issue is a missing precedence constraint around interviews.

## Model 2 analysis

Model 2 enforces:

```text
Post_Job_Ad  Conduct_Interviews  Make_Hiring_Decision  loop(Onboard_Employee)  XOR(Add_to_Payroll, skip)  Close_Case
```

It also has:

```text
Post_Job_Ad  Screen_Candidates
```

but screening is otherwise independent.

### Anomalies in Model 2

1. **Screening is not required before interviews or the hiring decision**  
   The model allows interviews immediately after the job is posted:

   ```text
   Post_Job_Ad,
   Conduct_Interviews,
   Make_Hiring_Decision,
   ...
   ```

   `Screen_Candidates` can occur later, even after the case is closed, because there is no edge from `Screen_Candidates` to `Conduct_Interviews`, `Make_Hiring_Decision`, or `Close_Case`.

2. **The case may close before screening is completed**  
   A possible trace is:

   ```text
   Post_Job_Ad,
   Conduct_Interviews,
   Make_Hiring_Decision,
   Onboard_Employee,
   Add_to_Payroll,
   Close_Case,
   Screen_Candidates
   ```

   This violates the expected recruitment logic.

3. **Onboarding may repeat indefinitely**  
   The loop:

   ```python
   loop_onboarding = OperatorPOWL(operator=Operator.LOOP, children=[Onboard, skip])
   ```

   means `Onboard_Employee` can be executed one or more times. Some rework during onboarding might be realistic, but repeatedly executing the full onboarding activity without an explicit reason or control condition is suspicious.

4. **Adding the employee to payroll is optional**  
   The XOR:

   ```python
   xor_payroll = OperatorPOWL(operator=Operator.XOR, children=[Payroll, skip])
   ```

   allows the process to skip `Add_to_Payroll` entirely and still close the case. For a successful hire, this is a serious anomaly: an onboarded employee should normally be added to payroll before the case is closed.

## Which model is closer to the normative process?

**Model 1 more closely aligns with the normative Hire-to-Retire process.**

Although Model 1 has a serious ordering problem — the hiring decision is not required to wait for interviews — it still preserves most of the expected structure:

```text
Post  Screen  Decision  Onboard  Payroll  Close
```

and all major activities are mandatory.

Model 2 has more fundamental integrity problems. It correctly places interviews before the hiring decision, but it fails to make screening a prerequisite, allows screening to occur after closure, permits repeated onboarding, and most severely, allows payroll addition to be skipped altogether.

Therefore, Model 1 is the better approximation of the standard process, while Model 2 contains broader and more damaging deviations from the expected Hire-to-Retire logic.
