**Case Identification Logic:**
I grouped events into five distinct cases based on logical work units:

1. **Case_001 (Quarterly_Report.docx):** Events related to editing and working with the quarterly report document
2. **Case_002 (Document1.docx):** All events related to creating, editing, and finalizing Document1.docx
3. **Case_003 (Email Communication):** The complete email handling workflow from opening to sending
4. **Case_004 (Report_Draft.pdf):** The PDF review and annotation session
5. **Case_005 (Budget_2024.xlsx):** The Excel budget editing session

Note that Case_002 events appear non-consecutively because the user switched between applications but returned to complete the same document. This represents a real work pattern where users multitask between documents.

**Activity Naming Logic:**
- **Raw action  Standardized activity:**
  - FOCUS/TYPING  "Open Document" or "Edit Content" (depending on context)
  - TYPING  "Edit Content" with the typed content as an attribute
  - SAVE  "Save Document"
  - CLOSE  "Close Document"
  - SWITCH  "Switch Application"
  - CLICK (opening email)  "Open Email"
  - SCROLL  "Review Content" or "Review Document"
  - CLICK (reply)  "Compose Reply"
  - CLICK (send)  "Send Email"
  - HIGHLIGHT  "Annotate Document"

**Temporal Considerations:**
I preserved the original timestamps and used them to maintain event sequence within cases. The CLOSE events serve as natural case endings, while the temporal gaps between some activities of the same case reflect real work interruption patterns.
