## 1) Analyzing Historical Scheduling Performance and Dynamics (via Process Mining)

### 1.1 Reconstructing the real job flow from MES event logs
**Goal:** turn raw MES events into an accurate, time-aligned representation of what *actually* happened: routing execution order, queueing/processing/setup/downtime segments, resource assignments, and disruptions.

**Core steps**
1. **Log preprocessing & normalization**
   - Map MES events to a consistent **event ontology**:
     - *Case* = job/case ID (`JOB-xxxx`)
     - *Activities* = task lifecycle states (Queue Entry, Setup Start/End, Task Start/End, etc.)
     - *Resources* = machine IDs and operator IDs (role-based: “Machine” vs “Operator”)
     - Add derived attributes: task type, work center, product family/type, priority class, due-date class.
   - Ensure **timestamp consistency** and correct for missing/late events (e.g., infer Task Start if only Task End exists).

2. **Derive “performance-relevant intervals”**
   For every `(case, task, machine)` occurrence, reconstruct:
   - **Setup interval**: from `Setup Start`  `Setup End`
   - **Processing interval**: from `Task Start`  `Task End`
   - **Queue interval**: `Task Start`  `Queue Entry` (or  previous end on that machine if queue entry missing)
   - **Inter-arrival / gap** at each machine: between end of one task and start of next (includes setup + idle)
   - **Downtime intervals**: from breakdown start/end (or maintenance periods inferred from “Resource Breakdown Start/End” patterns)

3. **Conformance / alignment to a discovered process model (optional but powerful)**
   - Use process mining to discover a **stochastic process model** (e.g., stochastic Petri net / Markov model / BPMN variants).
   - Then align each case trace to the model to detect:
     - missing events
     - repeated tasks
     - deviations caused by disruptions (e.g., urgent insertions, rework, rescheduling loops).

4. **Reconstruct “sequence-dependent setup context”**
   - For each machine, sort tasks by actual **Task Start** (or Setup Start).
   - Create a **machine transition log**:
     - previous job’s task type/family  next job’s task type/family (or previous activity class)
     - capture the setup duration between them.

### 1.2 Process mining techniques & metrics to quantify performance

#### A) Job flow times, lead times, makespan distributions
**Metrics (time perspective + distributions)**
- **Flow time / cycle time per job**: completion time of last task  release time
- **Lead time to due date**: completion time  due date (signed lateness)
- **Makespan**: max completion among jobs in the horizon
- **Distributional analysis**:
  - histogram / kernel density of cycle times
  - percentile performance (P50/P80/P90)
  - breakdown by job family, priority class, and routing variant

**Techniques**
- Use **time-series perspectives** (resource/time heatmaps) and **case duration distribution mining**.
- Split by **trace variant** (routing variant) and compare percentiles.

---

#### B) Task waiting times (queue times) at each work center/machine
**Metrics**
- For each `(work center / machine, task index)`:
  - mean/median/max queue time
  - 90th percentile queue time (often where lateness comes from)
- **Queue time decomposition**
  - waiting caused by:
    1) machine unavailability (breakdown)
    2) setup-induced delays
    3) contention for processing capacity

**Techniques**
- **Waiting time analysis** per resource using interval-based reconstruction.
- Build **directly-follows graphs** to see what precedes long queues (e.g., “jobs of family A before family B increase setups  longer queue”).

---

#### C) Resource utilization (productive, idle, setup, downtime)
**Metrics**
For each machine and operator:
- **Productive time**: processing intervals sum
- **Setup time**: setup intervals sum
- **Idle time**: uncovered time between processing completions and next processing start (excluding downtime)
- **Downtime**: breakdown / maintenance intervals sum
- **Utilization ratios**:
  - \( U_{proc} = \frac{T_{proc}}{T_{available}} \)
  - \( U_{setup} = \frac{T_{setup}}{T_{total}} \)
  - \( U_{downtime} = \frac{T_{down}}{T_{total}} \)

**Techniques**
- **Resource perspective** mining and “resource calendar reconstruction”.
- Identify time-varying utilization to expose:
  - starvation periods (low downstream WIP)
  - overload periods (high queue lengths at bottleneck machines)

---

#### D) Sequence-dependent setup times (quantify from the logs)
**Log-derived setup model**
You already have explicit `Setup Start/End` events in the snippet. Use that, but also validate gaps:

For each machine \(m\), build a dataset of transitions:
- previous job class: e.g., `(PrevTaskType, PrevProductFamily, PrevToolingState if available)`
- next job class: `(NextTaskType, NextProductFamily)`
- context: operator, shift, priority, due-date proximity (optional features)
- target: **setup duration** \(S\)

**Metrics & modeling approaches**
- **Empirical conditional distribution**:
  - \( P(S \mid \text{PrevClass} \rightarrow \text{NextClass}) \)
  - report mean/median + variance
- **Changeover cost matrix**: a learned “setup cost” table
- **Predictability**:
  - R˛ / error metrics of a regression model
- **Sequence-dependent impact**:
  - compute how much of total “non-processing” time comes from setup vs idle vs downtime.

**Techniques**
- **Next-event / directly-follows / transition mining** on a per-machine basis.
- Fit:
  - regression (GBDT/XGBoost)
  - categorical hazard/mixture models
  - or probabilistic sequence models (e.g., Markov with duration)

---

#### E) Schedule adherence and tardiness
**Metrics**
- **Tardiness**: \(T = \max(0, C - d)\)
- **Lateness**: \(L = C - d\) (signed)
- **Earliness rate** if relevant
- **Delay magnitude distribution**: quantiles of positive lateness
- **Due-date breach frequency**:
  - share of jobs late by priority class
  - late rate by routing variant and by “bottleneck crossing” point
- **Critical path decomposition**:
  - for each job, identify which task index had the largest contribution to lateness (via queue + setup accumulation)

**Techniques**
- Compare actual completion vs due date across:
  - trace variants
  - resource bottlenecks
  - different sequencing patterns at the key machines

---

#### F) Impact of disruptions (breakdowns, priority changes)
**Metrics**
- Breakdown frequency and duration per machine
- “Loss of work” estimation:
  - time lost while downstream queues grow
- Urgent/hot job insertion analysis:
  - measure how often inserting a hot job causes:
    - increased setup time for subsequent jobs
    - starvation downstream (if upstream preemption occurs)
    - increased queue times for jobs near due date

**Techniques**
- Event labeling:
  - tag cases/tasks as “affected by disruption” if their execution overlaps with downtime windows or after “priority change”.
- **Before/after analysis**:
  - KPI change in time windows preceding vs following breakdown/priority changes
- **Counterfactual sequencing comparisons**:
  - e.g., “jobs that were delayed after hot job insertion” vs “jobs inserted before” with matched controls.

## 2) Diagnosing Scheduling Pathologies (with evidence from process mining)

Below are common high-impact pathologies in job shops like yours, and *how* you’d prove them with log evidence.

### 2.1 Bottleneck overload and throughput constraints
**Pathology**
- One or a few machines (often with high setup + high contention) drive lateness.

**Evidence via process mining**
- Compute **throughput rate** by machine:
  - tasks completed per day/week
- Queue length / waiting time escalation:
  - P90 queue time spikes at bottleneck
- Use **resource capacity stress**:
  - fraction of time bottleneck is either processing or setup (low idle, high queue)
- **Waiting time decomposition**:
  - quantify how much of total job flow time is “explained” by waiting at specific machines.

**Bottleneck confirmation**
- Variant-based lateness:
  - if nearly all late jobs experience extreme waiting before a specific machine  likely bottleneck.

---

### 2.2 Poor task prioritization harming near-due or high-priority jobs
**Pathology**
- Local dispatching (FCFS/EDD mix) can invert global urgency, especially when setup costs and downstream blocking exist.

**Evidence**
- **On-time vs late trace comparison (variant analysis)**:
  - compare sequencing/order of jobs at each machine for on-time vs late groups
- **Priority stratification**:
  - late rate among “High urgency + near due date” vs “Medium/low”
- **Sequence disadvantage analysis**:
  - identify transitions at bottleneck where a high-priority job is preceded by expensive setup transitions or behind many jobs that should have been processed later.

---

### 2.3 Suboptimal sequencing increasing sequence-dependent setup time
**Pathology**
- Dispatching ignores changeover structure  frequent costly setups.

**Evidence**
- **Setup cost distribution by sequencing pattern**
  - show that certain job families followed by others have large setup times
- **Compare actual sequence cost vs best-known heuristic**
  - in hindsight, for each machine, compute “minimum setup path” over the observed jobs (or a rolling horizon estimate) and compare total setup duration.
- **Setup-time contribution to queue**
  - correlate setup duration at machine m with downstream waiting increase for tasks exiting m.

---

### 2.4 Starvation and propagation effects (blocked/unblocked dynamics)
**Pathology**
- Upstream machines keep processing while downstream is unavailable  WIP skyrockets (blocking).
- Or upstream starves because downstream jobs are stuck behind priority/setup.

**Evidence**
- Build **WIP trajectories** in time:
  - number of cases in each machine’s queue/intermediate buffer
- **Starvation windows**:
  - machine idle time that is caused by upstream depletion (infer via event causality and missing feeds)
- **Blocking detection**:
  - infer if processed parts cannot move to next machine (e.g., last task end occurs but next queue entry is delayed beyond typical transport/transfer time)

---

### 2.5 “Bullwhip” effects from scheduling variability
**Pathology**
- Small local rule randomness creates large variability in releases to downstream  WIP oscillations.

**Evidence**
- Cross-correlation between:
  - upstream processing variance and downstream WIP/queue variance
- Measure:
  - variance of inter-departure times per machine
- Show amplification from one work center to the next in WIP time series.

## 3) Root Cause Analysis: Why the current scheduling is ineffective

### 3.1 Scheduling logic limitations (static local rules vs dynamic shop-floor state)
Process mining can show **decision-context mismatch**:
- Local dispatching chooses next job without considering:
  - learned setup cost from last job
  - downstream load/blocked states
  - probabilistic processing times
  - expected completion relative to due dates

**How to distinguish**
- For each machine, take actual state snapshots and reconstruct:
  - what the local rule likely would pick
  - what alternative sequencing would have minimized setup/queue given the same observed arrivals
- If improvements mainly reduce setup + queue but not completion  local rule likely ignoring sequence-dependent setups.

---

### 3.2 Lack of real-time visibility and coordination between machines
**Evidence from logs**
- Long waiting times at machine m that do not correspond to “due urgency” or “available capacity”  indicates poor synchronization.
- High idle time at machine m during high WIP upstream (or vice versa)  coordination failure.

**Process mining support**
- Identify **mismatch periods**:
  - upstream machine processing while downstream machine queue is empty (or blocked)
- Use **causal performance traces**:
  - align job traces to resource availability windows.

---

### 3.3 Inaccurate task duration / setup time estimates
Even with good dispatching rules, inaccurate processing-time predictions produce poor schedules and tardiness.

**How to prove it**
- Compare planned vs actual durations:
  - setup and processing actual/expected ratio distributions
- Stratify error by:
  - job family, operator, shift, tool type, machine
- If error is systematically larger for certain classes, then the scheduler’s assumptions are wrong, not just sequencing.

---

### 3.4 Ineffective handling of sequence-dependent setups
**Evidence**
- If setup duration spikes occur in exactly those periods where due-date breaches increase, and setup spikes align with specific preceding job families, that’s strong causality.

**How process mining clarifies**
- Measure:
  - incremental effect: reroute sequencing *in a digital replay* and quantify reduction in setup and resulting lateness.
- This separates “capacity shortage” from “sequencing inefficiency”.

---

### 3.5 Distinguish “resource capacity limitation” vs “bad scheduling”
This is crucial. You can differentiate via two complementary analyses:

1. **Capacity stress metrics**
   - If bottleneck has utilization near 1 (or close), lateness may be unavoidable even with perfect sequencing.
2. **What-if scheduling replay**
   - Keep machine downtime and processing times fixed (capacity same),
   - but re-optimize sequencing decisions.
   - If KPIs improve dramatically  sequencing inefficiency is a major driver.
   - If KPIs barely improve  capacity limitation / variability / breakdown dominates.

Process mining provides the “fixed reality” (what downtime/processing actually happened), enabling clean separation.

## 4) Developing Advanced Data-Driven Scheduling Strategies ( 3)

Below are three distinct strategies, each directly tied to mined insights. They are compatible (you can implement them iteratively).

---

### Strategy 1 — Enhanced Multi-Factor Dynamic Dispatching (with predicted sequence-dependent setup and downstream pressure)
**Core logic (per work center / machine when a job needs to be selected)**
Compute a **score** for each candidate job \(j\) in the queue:
\[
Score(j) = w_1 \cdot \text{DueUrgency}(j) + w_2 \cdot \text{CostOfDelay}(j) - w_3 \cdot \widehat{Setup}(prev \rightarrow j) - w_4 \cdot \text{DownstreamPressure}(j)
\]
Where:
- **DueUrgency(j)**: function of (due date  current time) and job priority.
- **CostOfDelay(j)** (more advanced than EDD):
  - uses probabilistic expected remaining time:  
    \( \mathbb{E}[C_j \mid \text{current state}] \) and compare to due date
- **\(\widehat{Setup}(prev \rightarrow j)\)**:
  - learned from logs as conditional setup duration based on the previous job class on that machine
- **DownstreamPressure(j)**:
  - e.g., estimated expected waiting at next bottleneck(s) or current WIP level downstream
  - can be based on queue time forecasts from historical data

**How process mining informs it**
- Use Section 1’s setup transition mining to learn \(\widehat{Setup}\).
- Use Section 1’s waiting-time heatmaps to learn downstream pressure.
- Use Section 2’s on-time vs late analysis to fit weights \(w_i\) (e.g., via learning-to-rank on historical outcomes).

**Addresses key pathologies**
- Setup inefficiency (sequence-dependent)
- Priority inversion (by including due urgency + predicted completion)
- WIP explosion (via downstream pressure / lookahead)

**Expected KPI impact**
- Lower average and especially **P90 queue times**
- Reduced total setup time at bottleneck machines
- Reduced frequency of late jobs among urgent/near-due classes

---

### Strategy 2 — Predictive, Rolling-Horizon Scheduling using Probabilistic Processing + Breakdown Risk (Monte Carlo “digital twin” inside the scheduler)
**Core logic**
Instead of producing a single deterministic schedule, run a **rolling horizon** optimization each time the system state changes (job release, completion, breakdown, hot job insertion).

1. **Predictive models from logs**
   - Processing time distributions by `(task type, job family, operator, machine)`
   - Setup time distributions by sequence transition (from Strategy 1 mining)
   - Breakdown/downtime risk:
     - breakdown hazard or empirical downtime distribution per machine/age/shift (even if you only have historical averages, that still adds realism)

2. **Rolling optimization**
   - For a horizon (e.g., next 1–3 days or next N operations near bottleneck),
   - generate candidate schedules using constraint programming / MILP / heuristic search,
   - evaluate each candidate by **stochastic simulation** (Monte Carlo):
     - sample processing times, setup times, breakdown events
     - compute objective such as:
       - minimize expected weighted tardiness
       - minimize expected WIP (or penalize buffer overflow)
       - minimize probability of missing due dates

3. **Execute first-stage decisions**
   - implement only the first decision(s) (e.g., next assignment at bottleneck),
   - then reschedule when new events arrive.

**How process mining informs it**
- Stochastic models are directly fitted from empirical event-log distributions.
- Breakdowns and disruption insertions are modeled using mined downtime intervals and priority change patterns.

**Addresses key pathologies**
- Unpredictable lead times (schedule with uncertainty)
- Reactive failures after breakdowns/hot jobs (rolling horizon re-optimizes)
- Bottleneck starvation/overload (objective can explicitly penalize starving/overloading via queue state)

**Expected KPI impact**
- Improved **lead-time accuracy** for customer estimates (via quantiles)
- Reduced tardiness under disruptions
- Better WIP control due to stochastic planning and rescheduling

---

### Strategy 3 — Setup-Time Optimization via Sequence Planning / Family-Based “Changeover-Aware” Batching at Bottlenecks
This strategy focuses *specifically* on minimizing sequence-dependent setup costs, not only as a factor in dispatching.

**Core logic**
1. **Learn “family equivalence” and setup costs**
   - From logs, define product/task families (e.g., based on required tooling, material thickness, tolerance class).
   - Build a changeover cost matrix:
     - \(Cost(A \rightarrow B) = \mathbb{E}[Setup \mid A \rightarrow B]\)

2. **On bottleneck machines, plan a short sequence of families**
   - Decide an execution order for families rather than individual jobs each time.
   - Use an algorithm like:
     - dynamic programming / shortest-path ordering for a limited set of upcoming families
     - or an ILP: minimize sum of setup costs while ensuring due-date feasibility and max lateness constraints

3. **Batching / pooling with due-date constraints**
   - Batch jobs with similar setup requirements when:
     - urgent due dates are not violated beyond a risk threshold
     - or when the expected setup savings outweigh delay risk
   - Create a controlled tradeoff:
     - “setup consolidation” vs “priority urgency”

4. **Integrate sequence changes**
   - If a hot job arrives, compute marginal cost:
     - extra setup + delay impact if inserted now
     - vs alternative: finish current family block then switch

**How process mining informs it**
- Direct transition mining yields accurate setup costs.
- Use on-time/late analysis to determine acceptable tradeoff levels by priority class.

**Addresses key pathologies**
- Excessive setup frequency causing queue growth
- Lateness driven by expensive changeovers at bottlenecks
- Oscillatory sequencing from local rules

**Expected KPI impact**
- Reduced total setup minutes (often large in high-mix environments)
- Lower queue times around setup events
- Improved throughput stability and WIP profiles

## 5) Simulation, Evaluation, and Continuous Improvement

### 5.1 Simulation framework for rigorous comparison (digital twin)
**Discrete-event simulation (DES)** is essential before live deployment.

#### Parameterization from mined data
For each machine and task type, derive:
- processing time distributions (conditional on job family/task/operator)
- setup transition costs/distributions
- routing model (how jobs move through tasks)
- breakdown/downtime processes:
  - downtime duration distributions
  - breakdown frequency/hazard
  - downtime-triggered behavior (resume/restart rules if relevant)
- arrival/release processes:
  - historic job release times and priority change patterns
- job due dates distribution (already present)

#### Scheduling policy plug-in
Implement and simulate:
- Baseline: current local dispatch mix
- Strategy 1: enhanced scoring dispatching
- Strategy 2: rolling-horizon stochastic optimizer
- Strategy 3: setup-optimized sequence/family planning
- Combinations (e.g., Strategy 1 + 3 at bottlenecks)

#### What-if scenarios to test
At minimum:
1. **High utilization / overload**
   - scale job arrivals upward 10–30% and test robustness
2. **Frequent disruptions**
   - simulate breakdown rates at +1, +2
3. **Hot job injection**
   - increase frequency of priority changes; vary insertion timing
4. **Due-date tightness**
   - shift due dates earlier by X% to test sensitivity
5. **Setup volatility**
   - increase variance in setup duration (to test worst-case sequencing)

#### Evaluation metrics (statistical)
- lateness: mean/median + P90 tardiness
- probability of on-time completion per job class
- WIP: average/max and WIP oscillation amplitude
- machine KPIs: setup %, idle %, downtime %, throughput
- lead-time accuracy: calibration of predicted quantiles vs actual
Use multiple replications + confidence intervals; perform sensitivity analysis to weights/hyperparameters.

---

### 5.2 Continuous monitoring & adaptation using ongoing process mining
Treat scheduling as an **evolving control system**, not a one-time deployment.

#### Monitoring loop
1. **Ingest new MES logs continuously**
2. **Update performance dashboards**
   - queue time distributions per machine
   - setup times by transition
   - late-job rates by family/priority
   - breakdown KPIs per machine
3. **Drift detection / concept drift**
   - detect changes in:
     - processing time distributions (slower machines, new materials)
     - setup transition costs (new tooling practices)
     - breakdown frequencies (maintenance changes)
   - Use statistical distance metrics (e.g., KS test / PSI) or online drift detectors.

#### Automatic “re-learning” triggers
- if setup model error increases beyond threshold
- if P90 queue time at a bottleneck grows for 2–4 weeks
- if lateness rates rise sharply only for certain families (new routing/process introduced)

#### Governance
- keep an experiment registry:
  - which schedule logic version ran when
  - what parameter settings were active
- use A/B or staggered rollout by department/work center where possible.

- Root cause assessment then separates:
  - capacity/variability limits vs
  - scheduling-sequencing logic failures.
- The proposed strategies are **data-driven and adaptive**:
  - Strategy 1 improves decision rules with learned setup + downstream pressure.
  - Strategy 2 plans with probabilistic simulation and rolling re-optimization under uncertainty.
  - Strategy 3 targets setup optimization explicitly with family/sequence planning at bottlenecks.
- A **digital twin DES** validates improvements before deployment, and continuous process mining ensures ongoing performance under drift.
