## Logic and Assumptions

**Case identification**  
Each case is a distinct business object the user interacted with:
- `DOC-Document1` — drafting a Word document.
- `EML-AnnualMeeting` — handling an email thread.
- `PDF-ReportDraft` — reviewing a PDF.
- `XLS-Budget2024` — updating a budget spreadsheet.
- `DOC-QuarterlyReport` — editing the quarterly report.

Because users multitask, cases are interleaved in the global timestamp order (e.g., `Document1` is started, paused while email/PDF/Excel are handled, then resumed). Grouping by artifact keeps each case’s lifecycle coherent for process mining.

**Activity naming**  
Low-level system actions were mapped to standardized, application-aware process activities:
- `FOCUS` / target-side `SWITCH`  `Open …` or `Resume …`
- `TYPING`  `Edit Document`, `Update Spreadsheet`, or `Compose Email Reply`
- `SAVE`  `Save Document` / `Save Spreadsheet`
- `CLOSE`  `Close Document`
- `CLICK` (Open/Reply/Send)  `Open Email`, `Reply to Email`, `Send Email`
- `SCROLL` / `HIGHLIGHT`  `Read Email`, `Review PDF`, `Annotate PDF`

`SWITCH` events that represent *leaving* a case were not assigned to the source case, to avoid injecting “meta” transition noise. Instead, `SWITCH` events that bring focus *to* a new artifact were used as the `Open …` / `Resume …` event for the target case. A `Lifecycle` attribute was added to distinguish first opens from resumes and from completion events.
