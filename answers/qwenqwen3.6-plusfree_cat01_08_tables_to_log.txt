2. **Granularity of Picking Events**: Since each line item has a distinct `item_picking_date` and `picker_id`, I preserved them as separate `Item Picked` events rather than aggregating them. This maintains auditability and allows process mining algorithms to detect parallelism or resource contention.
3. **Delivery Split**: The `Delivery` table contains both dispatch and arrival timestamps. These were logically split into two distinct activities (`Order Dispatched` and `Order Arrived`) to accurately reflect the physical movement lifecycle.
4. **Resource Mapping for Payments**: The `Payments` table does not contain a specific employee ID. Therefore, the `payment_method` is used as the proxy Resource/Actor (e.g., `Credit Card System`). In a real ERP, this might map to a finance clerk or payment gateway API.
5. **Order 1004 Incompleteness**: This order only contains `Order Created` and `Item Picked` events. No delivery, invoice, or payment records exist in the provided dataset. This is common in real-world data (e.g., cancelled, on-hold, or B2B net-terms orders not yet invoiced). The log accurately reflects this truncated lifecycle.
6. **Chronological Integrity**: For Order 1002, note that `DLV-5003` arrives at `2024-01-08T10:00:00Z`, which precedes `INV-1002` issuance at `10:15:00Z`. The log strictly follows the timestamp order, which may indicate a process deviation (e.g., goods delivered before final invoice generation for backorders).
