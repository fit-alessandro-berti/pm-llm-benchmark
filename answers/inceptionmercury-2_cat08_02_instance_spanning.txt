## 1.Identifying InstanceSpanning Constraints & Their Impact  

| Constraint | How to Detect it in the Log (processmining artefacts) | Quantitative Impact Metrics | Separating *withininstance* vs *betweeninstance* waiting |
|------------|--------------------------------------------------------|----------------------------|----------------------------------------------------------|
| **Shared ColdPacking stations** (5stations, only orders with `Requires Cold Packing = TRUE` may use them) | • Build a **resourceusage view** (resourcecentric event log) – each `Packing` event is linked to a station ID (`Station C*`).  <br>• Apply **resourceconstrained replay** (e.g., “resourceaware token replay” in ProM/PM4Py) to compute **queue lengths** per station over time.  <br>• Use **conformance checking** to flag cases where a `Packing` start is delayed because the required station is busy. | • **ColdPacking waiting time** = `Start_Packing – End_ItemPicking` (only for orders with `Requires Cold Packing = TRUE`).  <br>• **Station utilisation** = (duration of Packing on station) / (simulation horizon).  <br>• **Queue length distribution** per station. | • **Withininstance**: compute the *minimum* possible start time assuming the station were free (i.e., `End_ItemPicking`).  <br>• **Betweeninstance**: the *extra* waiting = actual start – minimum start.  <br>• The extra part is the *intercase* delay caused by contention. |
| **Shippingbatch formation** (orders are held until a batch for a destination region is complete) | • Add a **batchID** attribute by grouping `Shipping Label Generation` events that share the same `Destination Region` and occur within the same “batch window”.  <br>• Use **caselevel performance** to compute the time between `Complete Quality Check` and `Complete Shipping Label Generation`.  <br>• Identify **batchwaiting** events where the case stays in the “waiting for batch” state. | • **Batchwaiting time** = `Start_ShippingLabel – End_QualityCheck`.  <br>• **Batch size** (orders per batch).  <br>• **Batch formation time** = time from first order entering QC to last order completing QC for that batch. | • **Withininstance**: the intrinsic processing time of QC  Shipping Label (normally negligible).  <br>• **Betweeninstance**: the *batchwaiting* component is the delay caused by waiting for other cases to join the batch. |
| **Expressorder priority (preemptive)** | • Flag all cases with `Order Type = Express`.  <br>• Build a **preemptive resource model**: for each resource (picker, packing station, QC staff) record whether an `Express` case started *while* a `Standard` case was already occupying the same resource.  <br>• Use **alignments with a preemptive model** (e.g., “preemptive token replay”) to capture interruptions. | • **Priorityinduced preempt time** = `Start_Express – (Standard_Standard` (if >0).  <br>• **Standardcase slowdown** = `(Standard_Completion – Standard_OriginalPlannedCompletion)`.  <br>• **ExpresstoStandard interruption count** per resource. | • **Withininstance**: normal processing time of the express case.  <br>• **Betweeninstance**: the *preempt* component (delay added to the standard case). |
| **Hazardousmaterial limit (10 concurrent in Packing+QC)** | • Create a **global counter** over time: every `Packing` or `Quality Check` event with `Hazardous Material = TRUE` increments the counter at its start and decrements at its completion.  <br>• Use **processmining “resourcecapacity” analysis** to locate periods where the counter exceeds 10 (capacity violation). | • **Regulatorywait time** = `Start_Packing/QC – End_PreviousActivity` for hazardous orders that are delayed because the limit is already reached.  <br>• **Violation frequency** (number of times >10).  <br>• **Average concurrent hazardous count**. | • **Withininstance**: the nominal processing time of Packing/QC for a hazardous order.  <br>• **Betweeninstance**: the extra waiting caused by the global limit being saturated. |

### Practical Extraction Steps (PM4Py / SQLstyle)

```python
# Example in PM4Py
log = pm4py.read_xes("warehouse_log.xes")
# 1) add case attributes (cold, hazardous, express, region)
log = pm4py.filter_variants_by_attribute_value(log, "requires_cold", True)
# 2) resourceconstrained replay
replay = pm4py.apply_conformance_check(log,
                                        petri_net,
                                        initial_marking,
                                        final_marking,
                                        parameters={pm4py.parameter.Variants.RESOURCE_AWARE_REPLAY: True})
# 3) compute waiting times
waiting = pm4py.get_waiting_times(log, activity="Packing", resource="Station")
```

A similar pipeline can be built in SQL (e.g., using window functions to compute `lag(end_timestamp)` per resource) or in a processmining platform (Celonis, Disco) that already provides “resource waiting time” and “batch waiting time” widgets.

## 2.Analyzing Constraint Interactions  

| Interaction | Why it Matters | How to Detect / Quantify |
|-------------|----------------|--------------------------|
| **Express+ColdPacking** | Express orders that need a cold station preempt standard cold orders, inflating the queue for that scarce resource and potentially causing a cascade of delays for many standard orders. | • Crosstabulate `Express` flag with `Requires Cold Packing`.  <br>• Compute **expressinduced queue growth**: (queue length) after each express start. |
| **Batching+Hazardous Limit** | If a batch is formed for a region that contains many hazardous orders, the batch may be forced to wait for the 10order ceiling, extending batch formation time dramatically. | • For each batch, count hazardous orders.  <br>• Correlate **batchformation time** with **hazardousorder count** (Pearson / Spearman). |
| **ColdPacking+Hazardous Limit** | Hazardous orders that also need cold packing occupy *both* a cold station and a slot in the hazardouslimit pool, creating a doublebottleneck. | • Identify cases with both flags.  <br>• Measure joint waiting time (max of coldpacking wait and hazardous wait). |
| **Express+Batching** | An express order may be forced to wait for a batch even though it should be expedited, negating its priority. | • Compare **express batchwaiting time** vs **standard batchwaiting time**.  <br>• Look for outliers where express orders wait > threshold. |
| **Priority Preemption+Regulatory Limit** | Preempting a standard hazardous order can push the concurrent hazardous count above 10, forcing the system to block further hazardous orders. | • Simulate the “preempt” event and recompute the hazardous counter.  <br>• Detect periods where the counter spikes after an express preemption. |

### Why Understanding Interactions Is Critical  

* **Hidden Cascades:** A change that appears beneficial for one constraint (e.g., giving express orders absolute priority) may worsen another (e.g., regulatory violations).  
* **Paretooptimal Tradeoffs:** By quantifying interaction effects we can plot a **constraintinteraction matrix** and identify “sweet spots” where a small sacrifice on one KPI yields a large gain on another.  
* **Policy Design:** Interaction insights allow us to design **compound rules** (e.g., “expresscold orders may preempt only if hazardous count <8”) that respect all constraints simultaneously.

## 3.ConstraintAware Optimization Strategies  

Below are three concrete, datadriven strategies. Each is described with the constraint(s) it tackles, the concrete change, the analytical foundation, and the expected benefit.

### Strategy1 – **Dynamic, Priorityaware Resource Allocation for ColdPacking**

| Aspect | Detail |
|--------|--------|
| **Primary Constraints** | Shared ColdPacking stations, Express priority, Hazardousmaterial limit |
| **Change** | Implement a **realtime scheduler** that (i) predicts the shortterm demand for each cold station using a slidingwindow forecast (e.g., ARIMA or LSTM on the last 30min of `Packing` start events), (ii) assigns a **priority score** = `·Express + ·Hazardous + ·TimeSinceArrival`. (iii) reserves a **preemptive slot** for the highestscoring order, but only if the **hazardouscount** after preemption stays 8 (leaving 2 slots for safety). |
| **Data / Analysis** | • Use the **resourceutilisation heatmap** to learn typical demand peaks.  <br>• Train a **binary classifier** (logistic regression) to predict whether a coldpacking request will cause a queue >5.  <br>• Feed the classifier into a **dispatch rule engine** (e.g., Drools) that decides whether to preempt a standard order. |
| **Expected Outcomes** | • **ColdPacking waiting time**  30% (average).  <br>• **Express order endtoend time**  20% (no longer blocked by standard orders).  <br>• **Hazardouslimit violations** remain <1% (because of the safety margin). |
| **Implementation Sketch** | ```python # forecast = model.predict(next_15min) # schedule = priority_queue(forecast, priority=score) # dispatch = scheduler.dispatch(schedule, max_hazardous=8) ``` |

### Strategy2 – **Adaptive, SizeBased Batching with “SoftDeadline” Triggers**

| Aspect | Detail |
|--------|--------|
| **Primary Constraints** | Shipping Batches, Hazardousmaterial limit, Express priority |
| **Change** | Replace the **fixedtime batch window** (e.g., “wait 15min”) with a **dynamic trigger**: <br>1. **Batch size target** = `k_region = ceil( avg_daily_orders_region / 4 )`. <br>2. **Soft deadline** = `t_soft = 10min` after the first order of the batch. <br>3. **Hard deadline** = `t_hard = 20min`. <br>4. If an **express** order arrives for that region, **override** the soft deadline and ship it immediately (bypass batch) **unless** the hazardous count would exceed 10; in that case, ship it in a *minibatch* of size 5. |
| **Data / Analysis** | • Compute **regionwise order arrival distribution** (Poisson fit).  <br>• Simulate different `k_region` values to find the point where **average batch waiting time** is minimized while keeping **batch size 3** (to preserve route optimisation).  <br>• Use **processmining “caseattribute clustering”** to identify “expressheavy” regions that need a different rule. |
| **Expected Outcomes** | • **Batchwaiting time**  40% for highvolume regions.  <br>• **Overall throughput**  12% (more orders leave the warehouse per hour).  <br>• **Express orders** meet SLA 95% of the time, while still preserving the cost benefit of batching for standard orders. |
| **Implementation Sketch** | ```sql SELECT region, COUNT(*) AS cnt FROM orders WHERE ts BETWEEN batch_start AND batch_start+INTERVAL '20 min' GROUP BY region; ``` (used to update `k_region` nightly). |

### Strategy3 – **RegulatoryCompliant “HazardousSlot Pool” with Parallel QC Lines**

| Aspect | Detail |
|--------|--------|
| **Primary Constraints** | Hazardousmaterial limit, Shared Packing, Priority handling |
| **Change** | 1. **Create a dedicated “Hazardous Slot Pool”** of 10 tokens that can be borrowed by either Packing or QC. 2. **Add a second QC line** (QCB) that is *only* used for hazardous orders, effectively **doubling the capacity** for the hazardous subprocess without increasing the overall QC staff count (staff can be crosstrained). 3. **Introduce a “preemptonlyifpoolavailable” rule**: an express hazardous order can preempt a standard hazardous order *only* when the pool has 2 free tokens, guaranteeing that the limit is never breached. |
| **Data / Analysis** | • Use **resourcepool simulation** to model token consumption.  <br>• Analyse **hazardous order arrival bursts** (e.g., using interarrival time distribution).  <br>• Run a **whatif analysis** in a discreteevent simulator to compute the required number of QC lines to keep the pool utilisation 80%. |
| **Expected Outcomes** | • **Regulatorywait time**  70% (most hazardous orders never wait).  <br>• **Overall utilisation** of QC staff  15% (more work done in parallel).  <br>• **Express hazardous orders** meet SLA without violating the 10order limit. |
| **Implementation Sketch** | In a BPMN model, replace the single `Quality Check` task with a **parallel gateway** leading to `QCStandard` and `QCHazardous`. Attach a **resourcepool token** to each branch and enforce the 10token limit via a **custom listener**. |

## 4.Simulation & Validation  

### 4.1Simulation Framework  

| Component | Description |
|-----------|-------------|
| **Process Model** | BPMN (or Petri net) enriched with **resource pools** (cold stations, QC lines), **batch nodes**, and **preemptive arcs** for express orders. |
| **EventLogDriven Parameters** | • Activity duration distributions (e.g., `Packing` – lognormal, parameters derived from the log).  <br>• Interarrival time distributions per order type & region.  <br>• Hazardousmaterial arrival burst profile. |
| **Constraint Engine** | • **ColdPacking capacity** = 5 stations.  <br>• **Hazardouslimit** = token pool of size 10.  <br>• **Batch trigger** = dynamic size/softdeadline (as per Strategy2).  <br>• **Preempt rule** = express can preempt only if pool safety 2. |
| **Simulation Tool** | AnyLogic, SimPy (Python) or the **ProcessMiningBased Simulator** (PM4Py’s `simulator` module). |

### 4.2What to Capture  

| KPI | How to Measure in Simulation |
|-----|------------------------------|
| **EndtoEnd Cycle Time** (per case) | Record `timestamp_last_event – timestamp_first_event`. |
| **ColdPacking Queue Length** | Track number of waiting orders per station at each simulation tick. |
| **Batch Formation Time** | Record `batch_completion_time – batch_start_time`. |
| **ExpressStandard Interruption Count** | Increment a counter each time a preemptive transition fires. |
| **HazardousPool Utilisation** | Compute `tokens_used / 10` over time; flag when >0.9. |
| **Throughput (orders/hr)** | Count completed `Shipping Label Generation` events per simulated hour. |
| **SLA Compliance** (express 2h, standard 4h) | Compute % of cases meeting target. |

### 4.3Validation Procedure  

1. **Baseline Run** – Simulate the *current* process (no optimization) using the extracted distributions. Verify that simulated KPIs match the **historical log** (within ±5%).  
2. **Scenario Runs** – Introduce each strategy individually, then in combination.  
3. **Statistical Comparison** – Use **paired ttests** or **bootstrap confidence intervals** on KPI differences.  
4. **ConstraintViolation Check** – Ensure that in every run the **hazardouspool** never exceeds 10 and that **express preemptions** never cause a deadlock.  
5. **Sensitivity Analysis** – Vary key parameters (e.g., number of cold stations, batchsize target) to see robustness.

| Dashboard | Visualisation | Insight |

| **ResourceUtilisation & Queue Heatmap** | Ganttstyle resource timeline with colourcoded waiting vs processing. | Spot stations that still become bottlenecks. |
| **BatchPerformance Matrix** | Heatmap: rows = region, columns = batchsize / formation time. | Identify regions where batch logic needs tuning. |

### 5.3Continuous Improvement Loop  

1. **Automated Log Ingestion** – Every night the new event log is imported into the processmining platform.  
2. **Metric Refresh** – Dashboards recompute KPIs automatically.  
3. **Alerting** – If any KPI exceeds its threshold (e.g., queue length >5), a **processengine alert** triggers a review of the scheduler parameters.  
4. **Periodic Retraining** – Forecast models for coldpacking demand and batchsize targets are retrained monthly with the latest data.  
5. **A/B Testing** – Deploy a **controlled pilot** where a subset of stations uses the new dynamic scheduler; compare KPIs against the control group before full rollout.

### TL;DR  

* **Identify** the four instancespanning constraints with resourceaware replay, batchwaiting analysis, preemptive alignments, and a global hazardouscounter.  
* **Quantify** waiting times and separate intracase from intercase delays.  
* **Analyse interactions** (expresscold, batchhazardous, etc.) to avoid hidden cascades.  
* **Deploy three datadriven strategies** – (1) dynamic, priorityaware coldstation scheduling, (2) adaptive, sizebased batching with soft deadlines, (3) a tokenpool + parallel QC line for hazardous orders.  
* **Validate** each change with a discreteevent simulation that respects all constraints.  
* **Monitor** continuously with a set of KPI dashboards and realtime resource counters, feeding the results back into forecast models for ongoing optimisation.  

By following this systematic, constraintaware processmining workflow, the fulfillment center can dramatically reduce waiting times, respect regulatory limits, and meet expressorder SLAs while preserving the cost benefits of batch shipping.
