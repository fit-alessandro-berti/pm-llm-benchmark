### Methodology & Assumptions
To create this log, the following steps and assumptions were applied:
1. **Case Identifier**: The `order_id` is used as the primary Case ID. Every event is traced back to an order.
2. **Payment-to-Order Mapping**: The `Payments` table does not contain an `order_id`. Payments were linked back to their respective cases by joining the `invoice_id` in the `Payments` table to the `invoice_id` in the `Invoice` table.
3. **Activity Granularity**: 
   - **Order Lines**: I generated a separate **"Item Picked"** event for every row in the `Order Lines` table. In process mining, this accurately reflects warehouse loops and parallel work. Context from the `Products` table was added to the attributes.
   - **Deliveries**: The `Delivery` table yields two distinct events per record: **"Order Dispatched"** and **"Order Arrived"**, mapping their respective timestamps to maintain chronological flow.
4. **Resources Mapping**: The `Resources` table was used to replace ID codes (like `SP-001` or `CR-01`) with clear actor names (e.g., "Alice Brown", "GlobalTransport") to provide better context. 
5. **Payment Actor**: Because no specific employee performs the "Payment Received" activity directly in the given data, the Resource was designated as *"System / Customer"*.
6. **Incomplete Cases**: Case `1004` only has "Order Created" and "Item Picked" events. This is natural in real-world ERP data (representing an order currently in progress) and is included.
7. **Sorting**: The log is grouped by **Case ID**, and then sorted chronologically by **Timestamp** so the lifecycle of each instance is easy to read.
