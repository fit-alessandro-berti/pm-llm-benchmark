To solve this, you can use a **Window Function** (`COUNT() OVER(...)`) or a **Common Table Expression (CTE)** with a subquery. Using a window function is generally more efficient in DuckDB as it avoids re-scanning the table multiple times.

Here is the SQL query:

```sql
WITH CaseCounts AS (
    SELECT 
        *,
        COUNT(*) OVER(PARTITION BY case_id) as event_count
    FROM event_log
)
SELECT 
    case_id, 
    activity, 
    timestamp
FROM CaseCounts
WHERE event_count <= 6;
```

### Explanation:
1.  **`COUNT(*) OVER(PARTITION BY case_id)`**: This calculates the total number of events for each `case_id` without collapsing the rows. It attaches the count as an extra column (`event_count`) to every row in the original dataset.
2.  **`WITH CaseCounts AS (...)`**: This CTE temporarily stores the table with the added event count.
3.  **`WHERE event_count <= 6`**: This filters out all rows belonging to cases that have more than six events, keeping only the full event history for cases that meet your criteria.

### Why this approach?
*   **Performance:** DuckDB handles window functions very efficiently. This approach scans the table to count the events and then filters them in a single pass.
*   **Readability:** Separating the calculation of the count from the final selection makes the logic clear and easy to maintain if you decide to change the threshold later.
