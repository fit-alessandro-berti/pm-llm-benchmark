### 1.5  SequenceDependent Setup Time Mining  

For each machine *m* we build a **pairwise setup matrix** `S_m(i,j)` where *i* is the *previous* jobĺs partfamily/tooling group and *j* is the *current* jobĺs group.  The value is the **average observed setup duration** (in minutes) computed from the log:

```sql
SELECT
  prev_job.part_family   AS from_group,
  curr_job.part_family   AS to_group,
  AVG(TIMESTAMPDIFF(MINUTE, setup_end, setup_start)) AS avg_setup
FROM events setup_start
JOIN events setup_end
  ON setup_start.case_id = setup_end.case_id
 AND setup_start.machine = setup_end.machine
 AND setup_start.activity = 'Setup Start'
 AND setup_end.activity   = 'Setup End'
JOIN jobs prev_job ON prev_job.case_id = (
    SELECT MAX(e2.case_id)
    FROM events e2
    WHERE e2.machine = setup_start.machine
      AND e2.activity = 'Task End'
      AND e2.timestamp < setup_start.timestamp
)
JOIN jobs curr_job ON curr_job.case_id = setup_start.case_id
GROUP BY prev_job.part_family, curr_job.part_family;
```

## 3.  RootCause Analysis of Scheduling Ineffectiveness  

### 3.1  Limitations of Static Dispatching Rules  

- **FCFS / EDD ignore downstream state**: A job arriving at a bottleneck first may block a highpriority job waiting behind it, causing cascading delays.  
- **No setup awareness**: Rules treat every job as independent; they do not consider that sequencing similar jobs together reduces changeover time.  
- **Lack of dynamism**: In a highmix environment, the ôaverageö job mix changes hourbyhour; static weights become stale.

### 3.2  Lack of RealTime Visibility  

PPIĺs MES logs events **after the fact** (e.g., *Task End*).  Dispatchers therefore make decisions with outdated information (queue lengths may have changed while a job was being set up).  Process mining can *reconstruct* a nearrealtime view by streaming events into a **live process dashboard** (using Kafka + Spark Streaming).

### 3.3  Inaccurate TaskDuration & SetupTime Estimations  

- **Planned durations** in the MES are often generic; they do not reflect operator skill, job complexity, or machine health.  The mined **actualtoplanned ratio** shows systematic bias (e.g., CNC Milling runs 12% longer on average).  
- **Setup times** are treated as a fixed ôsetup costö rather than a **pairwise function** of preceding and current job.

### 3.4  Ineffective Handling of SequenceDependent Setups  

Because the MES does not expose a **setup matrix**, schedulers cannot apply ôgroup technologyö logic.  The result is a **high variance** in daily setup hours, especially on bottleneck machines.

### 3.5  Poor Coordination Between Work Centres  

The current dispatch rule operates **locally** at each work centre.  There is no **global optimisation** that accounts for the *drumbufferrope* concept, leading to the observed starvation of downstream resources.

### 3.6  Inadequate Disruption Response  

When a breakdown occurs, the typical response is to **redispatch** jobs to the next available machine, but without **predicting the downstream impact**.  Hotjobs are inserted at the *head of the queue* regardless of their downstream routing, causing priority inversions and increased tardiness.

### 3.7  Differentiating SchedulingLogic Issues from Capacity Limitations  

Process mining allows us to **separate** the two:

- **Capacitydriven delay**: Queue times remain high even when jobs are perfectly sequenced (i.e., the machine is truly saturated).  This is visible in the utilization heatmap.  
- **Schedulinglogic delay**: Queue times are high but **variability** in waiting time is larger than expected for the load; variant analysis shows that ôpoor sequencingö jobs have significantly longer waits than ôgood sequencingö jobs.

A **regression model** can quantify the contribution of each factor (machine load, setup waste, priority inversion) to total flow time, giving a clear **rootcause pie chart**.

## 4.  Developing Advanced DataDriven Scheduling Strategies  

### 4.1  Strategy1 ľ MultiFactor Dynamic Dispatching (Weighted ATCPlus)  

**Core Logic**  
The **Apparent Tardiness Cost (ATC)** rule is a industryproven dynamic dispatch heuristic that balances processing time, due date, and workload.  We extend it with **setupaware** and **downstreamload** components:

\[
\text{Priority}_{j} = \max\left\{0,\; \frac{p_{j}}{p_{\text{avg}}}\right\}
  \times \exp\!\Bigl(-\frac{\max(0,\,d_{j}-t-\tau)}{k\,\tau}\Bigr)
  \times \exp\!\Bigl(-\frac{s_{ij}}{S_{\text{avg}}}\Bigr)
  \times \frac{1}{\text{DownstreamLoad}_{j}}
\]

Where  

- \(p_{j}\) = remaining processing time of job *j* (including all future operations).  
- \(p_{\text{avg}}\) = rolling average processing time across all jobs in the queue.  
- \(d_{j}\) = due date of job *j*.  
- \(t\) = current time.  
- \(\tau\) = lookahead window (e.g., 2h).  
- \(k\) = scaling factor (tuned via simulation).  
- \(s_{ij}\) = predicted setup time for job *j* after the job *i* currently on the machine (from the mined setup matrix).  
- \(S_{\text{avg}}\) = average setup time on this machine.  
- \(\text{DownstreamLoad}_{j}\) = sum of remaining processing times of job *j* on all downstream machines (computed from the jobĺs routing DFG).  

**How Process Mining Informs the Rule**  

| Input | Mined Source |
|-------|--------------|
| \(p_{\text{avg}}\) | Mean of *Actual Task Duration* per activity, updated weekly. |
| \(s_{ij}\) | Pairwise setup matrix `S_m(from_group, to_group)` for each machine *m*. |
| Downstream load | Routing DFG (probability of each downstream machine) + average processing times per downstream activity. |
| Scaling factor *k* | Calibrated by fitting the ATC rule to historical tardiness (optimised via simulation). |
| Lookahead window \(\tau\) | Determined from the *performance spectrum* ľ the 90thpercentile queue time at each machine. |

**Addressing Pathologies**  

- **Bottleneck overload**  The downstreamload factor penalises jobs that will further congest the bottleneck.  
- **Priority inversion**  The exponential duedate term quickly raises priority for jobs approaching their due date.  
- **Setup waste**  The setup term favours jobs that minimise changeover after the current job.  

**Expected KPI Impact**  

- **Tardiness reduction**: 3040% (based on simulation of ATCtype rules in similar job shops).  
- **WIP reduction**: 1520% (by smoothing flow through the bottleneck).  
- **Utilization**: +58% on bottleneck machines (more productive time due to reduced setup).  

### 4.2  Strategy2 ľ Predictive Scheduling with Stochastic Modeling  

**Core Logic**  

1. **Predict task durations** using a **GradientBoosted Regressor (GBR)** trained on historical events.  
   - Features: partfamily, material, operator ID, machine ID, timeofday, shift, average queue length at arrival, tool wear index (derived from breakdown frequency).  
2. **Predict setup times** using the same pairwise matrix (no regression needed ľ the matrix is deterministic).  
3. **Predict machine availability** using a **survival model** (Cox proportional hazards) on breakdown intervals to estimate *probability of failure* in the next 4hour window.  
4. **Generate a stochastic jobshop schedule**: For each job, sample a duration from the predicted distribution (e.g., lognormal with  = predicted,  = residual error).  Run a **MonteCarlo simulation** (1000 replications) to produce a **probability distribution of completion times**.  
5. **Schedule optimisation**: Use a **constraint programming (CP) solver** (or a hybrid CP + localsearch) to assign start times, respecting precedence, resource constraints, and the stochastic windows.  The objective is to **minimise expected weighted tardiness** while keeping the probability of tardiness below a threshold (e.g., 5%).  

**How Process Mining Informs the Model**  

- **Duration model**: All *Actual Task Duration* events feed the GBR training set.  
- **Residual variance**: Captures systematic bias (e.g., CNC Milling on MILL03 runs 12% longer).  
- **Breakdown model**: Interbreakdown times and maintenance logs feed the Cox model.  
- **Routing probabilities**: Derived from the DFG to define the stochastic network of operations.  

**Addressing Pathologies**  

- **Unpredictable lead times**  By providing a **confidence interval** for each jobĺs completion, customers receive realistic quotes.  
- **Disruption impact**  The survival model can trigger **preemptive rescheduling** when a machineĺs failure probability exceeds a threshold, reducing the ripple effect of breakdowns.  
- **Inefficient resource use**  By accounting for **toolwearderived degradation** (fewer breakdowns  higher effective capacity), the schedule can ôloadbalanceö more accurately.  

**Expected KPI Impact**  

- **Leadtime accuracy (within ▒10% of quoted)**: Improves from ~45% to >80%.  
- **Tardiness**: 2030% reduction (due to betterinformed slack allocation).  
- **Breakdowninduced delays**: 2535% reduction (preemptive rerouting).  

### 4.3  Strategy3 ľ SetupAware Batching & Sequence Optimisation at Bottleneck Machines  

**Core Logic**  

1. **Jobfamily clustering** ľ Using the **setup matrix** as a distance metric, perform **hierarchical agglomerative clustering** on jobs awaiting the bottleneck machine (e.g., MILL03).  The clustering cutoff is chosen so that **withincluster average setup time** is 10min while **betweencluster** is 20min.  
2. **Batch sequencing** ľ Within each cluster, order jobs to minimise total setup time using a **nearestneighbour heuristic** or a **smallscale TSP** (solve with Google ORTools).  The cluster order itself is determined by **duedate urgency** (earliest due date first).  
3. **Integration with overall schedule** ľ The batches are inserted into the global schedule as **macrooperations** with a deterministic duration (sum of runtimes + optimal intrabatch setup).  The remaining (nonbottleneck) machines still use the ATCPlus rule, but now receive jobs **pregrouped** to reduce changeovers.  
4. **Dynamic rebatching** ľ When a hotjob arrives, it is inserted into the **most compatible existing batch** (lowest additional setup cost).  If no compatible batch exists, a new singlejob batch is created but scheduled **after** the current batch to avoid disrupting the batch sequence.  

**How Process Mining Informs the Strategy**  

| Component | Mined Input |
|-----------|-------------|
| **Distance metric for clustering** | Pairwise setup matrix `S_m`. |
| **Batchsize decision** | Distribution of *Queue Length* vs. *Setup Saving* ľ choose the batch size where marginal setup saving 5min (to avoid excessive leadtime extension). |
| **Duedate urgency weighting** | Weighted tardiness analysis (historical) to set the weight of due date vs. setup saving. |
| **Dynamic rebatching threshold** | Frequency of hotjob insertions (Poisson rate) mined from *Priority Change* events. |

**Addressing Pathologies**  

- **Setup waste**  Directly tackled by clustering and intrabatch sequencing.  
- **Bottleneck overload**  By fixing a **batch size cap**, the algorithm prevents a single batch from monopolising the bottleneck, limiting the maximum wait for other jobs.  
- **WIP bulup**  Because jobs are released in batches, the **release rate** into downstream stations is smoother, dampening the bullwhip effect.  

**Expected KPI Impact**  

- **Setup time reduction**: 3045% on bottleneck machines (based on TSPoptimal vs. actual).  
- **Throughput increase**: 812% (more productive time on bottleneck).  
- **WIP reduction**: 1015% (controlled batch release).  

### 4.4  Hybrid Integrated Scheduling Engine  

In practice, the three strategies are **layered**:

1. **Macrolevel**: Predictive scheduling (Strategy2) generates a **stochastic master schedule** for the next 23 days, allocating time windows to job families.  
2. **Mesolevel**: Setupaware batching (Strategy3) builds **optimised batches** for each bottleneck within those windows.  
3. **Microlevel**: ATCPlus dispatching (Strategy1) is applied **locally** at each work centre to handle realtime arrivals, variability, and disruptions *within* the batch windows.

The layered architecture preserves **global optimisation** (Strategy2) while retaining **reactivity** (Strategy1) and **efficiency** (Strategy3).

## 5.  Simulation, Evaluation, and Continuous Improvement  

### 5.1  DiscreteEvent Simulation (DES) Model Construction  

**Toolset**: AnyLogic, Simio, or Pythonbased (SimPy) for rapid prototyping.  

**Model Entities & Parameters** (all sourced from process mining):

| Entity | Parameter | Source |
|--------|-----------|--------|
| **Job** | Routing (activity sequence), due date, priority, release time, job family | Event log ľ case attributes. |
| **Machine** | Cycletime distribution (lognormal), setuptime matrix, breakdown interarrival time (Weibull), repair time distribution | *Actual Task Duration* & *Setup* events; *Breakdown* events. |
| **Operator** | Skillbased speed factor, shift schedule | Operator ID + actual duration correlation. |
| **Queue** | Discipline (FCFS, ATCPlus, etc.) | Configurable dispatcher module. |
| **Disruption** | Hotjob arrival process (Poisson), breakdown schedule (nonstationary Poisson) | *Priority Change* and *Breakdown* events. |

**Validation**: Run the model for a **historical period** (e.g., 3months) and compare simulated KPIs (flow time, tardiness, utilization) against the actual mined KPIs.  A **mean absolute percentage error (MAPE)** <10% indicates a valid baseline.

### 5.4  Continuous Monitoring & ProcessMining Feedback Loop  

1. **RollingWindow Mining** (weekly or daily) ľ Rerun process discovery, bottleneck analysis, and KPI calculations on the most recent event slice.  
2. **KPI Drift Detection** ľ Implement **EWMA control charts** on:  
   - Mean queue time per machine.  
   - % Tardy jobs.  
   - Average setup time per machine.  
   - Downstream load forecast error (Strategy2).  

   When a control limit is breached (e.g., 3), an alert is raised.  

3. **Model Retraining Triggers** ľ  
   - If the **mean absolute error (MAE)** of the duration predictor exceeds a threshold (e.g., >15% of actual), automatically trigger a **refit** of the GBR model on the latest 3month window.  
   - If a **new partfamily** appears (identified by a previously unseen `Activity` pattern), a **new clustering** for batching is performed.  

4. **Dispatcher Weight Adaptation** ľ Use a **reinforcementlearning (RL) agent** (e.g., Deep QNetwork) that receives the same state representation as ATCPlus (queue lengths, due dates, setup times) and learns optimal weights for the priority function by maximising a reward defined as *ľWeighted Tardiness*.  The RL agent is updated weekly using the latest log data, providing a **selftuning** scheduler that evolves with changing product mix and equipment health.  

5. **DisruptionResponse Automation** ľ When a *Breakdown Start* event is received, the system automatically:  
   - Calculates the **impacted jobs** (any job currently processing on that machine).  
   - Invokes the **predictive model** to estimate delay.  
   - Executes a **fast localsearch reschedule** (Strategy1 with updated machine availability) and pushes revised duedate estimates to the MES/ERP within 60seconds.
