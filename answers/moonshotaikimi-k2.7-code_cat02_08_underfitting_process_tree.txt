### Invalid trace that the model accepts

```text
Receive_Order  Check_Inventory  Ship_Order  Authorize_Payment  Package_Items  Close_Order
```

In a more compact trace notation:

```text
<Receive_Order, Check_Inventory, Ship_Order, Authorize_Payment, Package_Items, Close_Order>
```

### Why this trace fits the model

The problematic part of the process tree is the **successful path**:

```python
successful_subtree = ProcessTree(operator=Operator.PARALLEL,
                                 children=[authorize_payment, package_items, ship_order])
```

Because the three activities are placed in a **parallel** block, the model allows them to occur in **any interleaving**, as long as each occurs exactly once. It does **not** enforce that packaging must come before shipping, or that payment authorization must come before shipping.

Therefore, the trace above is accepted as a valid “happy path” trace, even though it ships the order before payment has been authorized and before the items have been packaged.

### Why the trace is domain-invalid

In a real Order-to-Cash process, the physical and logical dependencies are:

1. **Authorize Payment** must happen before items leave the warehouse.
2. **Package Items** must happen before **Ship Order** — you cannot ship items that have not yet been packaged.

The trace:

```text
Ship_Order  Authorize_Payment  Package_Items
```

violates both of these rules:

- The order is **shipped before payment is authorized** — risking loss if payment fails.
- The order is **shipped before packaging** — which is operationally impossible in practice.

### What the model is missing

This is an example of **underfitting**: the parallel operator is too permissive. A correct model should enforce a stricter ordering on the successful branch, for example:

```text
Authorize_Payment  Package_Items  Ship_Order
```

rather than treating them as freely interchangeable.
