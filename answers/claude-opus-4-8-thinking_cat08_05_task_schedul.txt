# Data-Driven Scheduling Optimization for Precision Parts Inc.
## A Process Mining-Based Approach

---

## 1. Analyzing Historical Scheduling Performance and Dynamics

### 1.1 Reconstructing the Actual Process

The foundation of this analysis is **event log preparation and conformance to the process mining data model**. The MES log already contains the three essential ingredients: a **Case ID** (Job ID), an **Activity** (Task + lifecycle transition), and a **Timestamp**. Critically, this log uses a *lifecycle-extended* event model (Queue Entry  Setup Start  Setup End  Task Start  Task End), which is exactly what we need to decompose time into its constituent components.

**Preprocessing steps:**
- **Event abstraction & lifecycle pairing:** Pair Start/End events for each lifecycle phase (setup, processing, queueing) per (Case, Activity, Resource) tuple to derive durations.
- **Resource perspective enrichment:** Append Machine ID and Operator ID as separate resource dimensions, since utilization and setup behaviors differ between machine and operator.
- **Disruption event integration:** Treat Breakdown Start/End and Priority Change as overlay events on the timeline, joined to the affected resources/cases by timestamp windows.

**Process discovery:** Because routings are unique (high-mix), a standard Petri-net discovery (e.g., Inductive Miner) will produce a "spaghetti" model. I would therefore:
- Apply **trace clustering** (e.g., by routing signature, product family, or activity-sequence n-grams) to segment jobs into coherent **process variants/families** before discovery.
- Use the **Directly-Follows Graph (DFG)** with frequency and **performance** annotations as the primary visualization, since it tolerates high variability and directly exposes flow times on edges.
- Build a **resource-centric DFG** (machine handover network) to see how jobs flow *between machines*, revealing routing patterns and contention paths.

### 1.2 Specific Techniques and Metrics

**Flow times, lead times, and makespan:**
- **Lead time** = (Final Task End  Job Released) per case. Compute the full **distribution** (not just mean): median, P85, P95, and the coefficient of variation (CV). High CV is the quantitative signature of "unpredictable lead times."
- **Flow time vs. value-added time ratio:** Flow efficiency = (processing time) / total lead time. In job shops this is often <10%, immediately quantifying waiting waste.
- **Makespan** per planning window measured from the resource/machine timeline.

**Task waiting (queue) times:**
- Derived directly from `Task Start  Queue Entry` per task. Aggregate per machine to build a **queue-time heatmap** by work center and by time-of-day/week.
- Use **performance DFG edge metrics** to attribute waiting to specific machine handovers.

**Resource utilization decomposition:**
For each machine, partition its calendar timeline into:
- **Productive time** = (Task End  Task Start)
- **Setup time** = (Setup End  Setup Start)
- **Idle time** = available time  productive  setup  breakdown
- **Breakdown/down time** from resource events.

This yields an **Overall Equipment Effectiveness (OEE)**-style decomposition and a utilization profile per machine. For operators, the same applies (productive vs. idle vs. setup-attending).

**Sequence-dependent setup times — the analytical centerpiece:**
The `Notes` field already records "Previous job: JOB-6998." I would:
1. Construct a **transition matrix** of setup durations keyed by (previous job's attributes  current job's attributes). Since job-to-job specifics are too sparse, I'd abstract jobs to **setup-relevant attributes** (material type, geometry/tooling class, fixture, tolerance class).
2. Build an **N×N setup-time matrix S[i,j]** = expected setup time when a job of family *j* follows family *i*, per machine, with mean and variance estimated from the actual setup durations.
3. Statistically validate sequence dependence via **ANOVA / regression**: regress actual setup time on (predecessor family, successor family, machine) and confirm the predecessor term is significant. This separates genuine sequence dependence from random variation.

**Schedule adherence and tardiness:**
- **Lateness** = Completion  Due Date; **Tardiness** = max(0, Lateness).
- Metrics: % tardy jobs, mean tardiness, **total weighted tardiness** (weighted by Order Priority), and the tardiness *distribution*.
- **Planned vs. actual deviation:** systematically compare Planned vs. Actual durations (already in the log) to quantify estimation bias per activity/machine — actuals are consistently higher (23.5 vs 20; 66.4 vs 60), so estimation optimism is measurable.

**Impact of disruptions:**
- For each Breakdown event, define an **impact window** and measure: jobs queued/blocked at the affected and downstream machines, incremental waiting time, and tardiness attributable to the window (counterfactual against similar non-disrupted periods).
- For each Priority Change ("hot job"), trace the **schedule churn**: which jobs were displaced, how their completion times shifted, and the cascaded tardiness. This quantifies the cost of reactive expediting.

## 2. Diagnosing Scheduling Pathologies

Using the metrics above, I would surface and *evidence* the following pathologies:

### 2.1 Bottleneck identification and throughput impact
- **Evidence via process mining:** Combine high utilization + persistently long queues + jobs aging in queue. A machine that is simultaneously high-utilization and high-queue-time is a **true bottleneck**; high queue + low utilization signals a *blocking/starvation* problem instead.
- Quantify throughput impact by computing the **theoretical throughput ceiling** set by the bottleneck and comparing to actual, and by **drift analysis** of WIP accumulating upstream of the bottleneck.

### 2.2 Poor prioritization
- **Variant analysis comparing on-time vs. late jobs:** Split cases into on-time and tardy cohorts and compare their queue-time profiles. If High-priority or near-due jobs show *comparable or longer* queue times than low-priority jobs at shared machines, the local FCFS/EDD mix is failing to globally prioritize.
- Direct evidence: instances where a Medium job was processed ahead of a queued High/Urgent job (visible in resource timelines).

### 2.3 Suboptimal sequencing inflating setup
- Using the setup matrix S[i,j], compute **actual cumulative setup time** per machine vs. a **lower-bound** obtained by optimally re-sequencing the *same* job set (a TSP-like sequencing on the setup matrix). The gap = **avoidable setup waste** caused by sequencing decisions. This is powerful, defensible evidence.

### 2.4 Starvation of downstream resources
- Detect **idle periods on downstream machines that coincide with WIP starvation** (no upstream feed) while the upstream bottleneck was busy/down. Resource-centric DFG timelines reveal these "idle-while-starved" intervals.

### 2.5 WIP bullwhip / variability
- Track **WIP over time** per work center (count of jobs between Queue Entry and Task End). High amplitude oscillations correlated with scheduling decisions/disruptions evidence the bullwhip effect. Correlate WIP variance with batch/expedite events.

## 3. Root Cause Analysis

The diagnostics above describe *symptoms*; root cause analysis attributes them to *causes*.

| Pathology | Likely Root Cause | Process Mining Evidence |
|---|---|---|
| High tardiness for priority jobs | Static **local** dispatching (FCFS/EDD) lacks global, priority-aware view | Priority jobs queue behind lower-priority work at shared resources |
| Excess setup time | Rules ignore sequence-dependent setups | Actual vs. optimal-resequenced setup gap |
| Unpredictable lead times | Planning uses optimistic/static durations | Systematic Planned < Actual bias; high duration CV |
| Downstream starvation | No coordination / no look-ahead between centers | Idle-while-starved intervals downstream of bottleneck |
| Disruption fragility | No re-scheduling policy; reactive expediting | Large schedule churn & cascaded tardiness after each hot job |

**The crucial distinction: bad scheduling logic vs. genuine capacity limits vs. inherent variability.**

This is where process mining earns its value:
- **Capacity limitation** is indicated when the bottleneck machine runs at ~95–100% utilization *after removing setup and breakdown waste* — i.e., even a perfect schedule cannot meet demand. The fix is capacity (machines, shifts, outsourcing), not scheduling.
- **Scheduling logic problem** is indicated when the bottleneck's utilization is moderate but burdened by **avoidable setup** and **idle-while-jobs-wait** episodes — the same workload could fit with better sequencing. The "optimal-resequence gap" quantifies the *recoverable* capacity locked up by poor logic.
- **Inherent variability** is isolated by examining the **residual** duration variance after controlling for job family, operator, and setup type (via regression). Variance that remains unexplained is process noise that scheduling must *buffer* against (via safety time), not eliminate.

This decomposition prevents the common error of buying machines when the real problem is sequencing — or vice versa.

## 4. Advanced Data-Driven Scheduling Strategies

### Strategy 1 — Composite Dynamic Dispatching Rule (ATCS-Extended)

**Core logic:** Replace local FCFS/EDD with a **multi-factor priority index** computed in real time whenever a machine becomes free. Building on the **Apparent Tardiness Cost with Setups (ATCS)** rule, the priority of job *j* on machine *m* (last job was *i*) is:

```
(j) = (w_j / p_j) · exp( max(d_j  p_j  t , 0) / (k·p)) · exp( s_{ij} / (k·s))
```

where w_j = priority weight, p_j = remaining processing time, d_j = due date, t = current time, s_{ij} = sequence-dependent setup from the **mined setup matrix**, and a fourth multiplicative term for **downstream load** (favor jobs heading to currently-starved downstream machines).

**How process mining informs it:**
- p_j, s_{ij} use **mined actual** durations/distributions, not optimistic planned values.
- The scaling parameters k, k and the slack/look-ahead horizon are **tuned on historical data** (replaying the log) so the rule's weighting reflects this shop's real dynamics.
- The downstream-load term uses the **handover network** to anticipate starvation.

**Pathologies addressed:** poor prioritization (w_j term), excess setup (s_{ij} term), downstream starvation (load term), tardiness (due-date slack term).

**Expected KPI impact:** Lower weighted tardiness, reduced setup time, smoother downstream feed. Practical and deployable immediately as a real-time MES decision rule.

---

### Strategy 2 — Predictive / Stochastic Scheduling with Risk Buffers

**Core logic:** Generate a **rolling-horizon predictive schedule** using mined **probability distributions** rather than point estimates. Each task duration is modeled as a fitted distribution (e.g., lognormal) conditioned on job family, operator skill, and complexity. The scheduler:
- Computes **probabilistic completion-time estimates** (e.g., P85 finish) to give customers reliable due-date quotes.
- Inserts **dynamic safety time** at high-variance / bottleneck operations sized to the mined variance.
- Incorporates **predictive maintenance** signals: if breakdown inter-arrival times per machine are mined (survival/reliability analysis), planned buffers or pre-emptive maintenance windows are scheduled before high failure-risk periods.
- Proactively flags **predicted future bottlenecks** by forward-simulating the order book against mined capacities.

**How process mining informs it:** duration distributions, breakdown frequency/MTBF distributions, and routing probabilities all come directly from the log.

**Pathologies addressed:** unpredictable lead times (probabilistic quoting), disruption fragility (predictive buffering), proactive bottleneck management.

**Expected KPI impact:** Dramatically improved due-date reliability (quote accuracy), fewer surprise delays, modest WIP increase only where buffers are intentionally placed.

---

### Strategy 3 — Setup-Aware Batching & Bottleneck Sequence Optimization

**Core logic:** At the bottleneck machine(s), periodically (rolling window) solve a **sequencing optimization** that minimizes total setup while respecting due-date constraints. This is structurally a **prize-collecting / time-constrained TSP** on the mined setup matrix S[i,j], where:
- "Distance" = setup time between consecutive job families.
- A **due-date penalty** prevents excessive batching from making jobs late ("batching myopia").
- Jobs are **grouped into families** (mined from setup-relevant attributes) and sequenced to chain low-setup transitions.

A constrained **batch size cap** balances setup savings against tardiness — determined by analyzing the historical setup-vs-tardiness tradeoff curve.

**How process mining informs it:** the entire setup matrix, family clustering, and the empirical tradeoff frontier come from historical analysis (Section 1.2 / 2.3).

**Pathologies addressed:** avoidable setup waste, bottleneck capacity recovery (reclaimed setup time = free throughput at the constraint).

**Expected KPI impact:** Significant setup reduction at the bottleneck  higher effective throughput, lower WIP. Must be balanced against tardiness via the due-date constraint.

---

**Integration note:** These three are complementary, not competing. Strategy 3 sequences the *bottleneck*; Strategy 1 governs *non-bottleneck* real-time dispatching consistent with the bottleneck plan (Theory of Constraints "drum-buffer-rope" logic); Strategy 2 provides the *predictive layer* and customer quoting. The recommended deployment is a layered hybrid.

## 5. Simulation, Evaluation, and Continuous Improvement

### 5.1 Discrete-Event Simulation (DES) for Risk-Free Evaluation

I would build a **DES digital twin** of the shop floor, **parameterized entirely from process-mined data** — this is the rigorous, non-disruptive testbed before any live change.

**Simulation inputs (all mined):**
- Task duration **distributions** per (activity, machine, operator, family).
- **Routing probabilities** / job-family arrival patterns from variant analysis.
- **Sequence-dependent setup matrix** S[i,j].
- **Breakdown** inter-arrival and repair-time distributions per machine.
- **Order arrival** and priority/due-date generation reflecting historical demand.

**Validation:** First, replay the *baseline* (current rules) in the DES and confirm it reproduces historical KPIs (lead-time distribution, tardiness, utilization) within tolerance. This **face-validity check** is essential — only a validated model can be trusted for what-if analysis.

**Experiments — test each strategy (and the hybrid) under controlled scenarios:**
- **Nominal load** — baseline comparison.
- **High load / demand surge** — stress capacity, test robustness.
- **Frequent disruptions** — elevated breakdown rate + injected hot jobs to test reactive resilience.
- **High-mix volatility** — more diverse families to stress the setup-optimization logic.
- **Sensitivity / parameter tuning** — sweep k, k, buffer sizes, batch caps.

**Evaluation method:** Run **multiple replications** per scenario (common random numbers across strategies for paired comparison), report KPI distributions with **confidence intervals**, and test statistical significance of improvements. Evaluate the full KPI set jointly — tardiness, weighted tardiness, WIP, lead-time mean *and* variance, utilization — to expose tradeoffs (e.g., Strategy 3's batching may trade slight tardiness for setup gains).

### 5.2 Continuous Monitoring & Adaptation Framework

Scheduling logic must adapt because process mining models can **drift** (new products, machines, demand mix).

**Closed-loop framework:**
1. **Live KPI dashboard** fed by ongoing MES event streaming — the same metrics from Section 1, tracked with **statistical process control (SPC)** charts (control limits on tardiness, WIP, queue times).
2. **Concept-drift detection:** Continuously re-mine the setup matrix, duration distributions, and routing patterns on a rolling window. Statistical tests (e.g., distribution-shift / population-stability indices) flag when mined parameters diverge from those the live scheduler assumes.
3. **Conformance checking:** Compare the *actual* executed sequence against the *scheduled* sequence to measure **schedule adherence** — low conformance signals either a flawed schedule or operator override patterns worth investigating.
4. **Automated re-tuning triggers:** When drift or SPC violations are detected, trigger (a) re-estimation of scheduler parameters and (b) re-validation in the DES twin before pushing updated logic — preventing untested changes from going live.
5. **Periodic re-baselining & A/B review:** Quarterly comparison of realized vs. simulated-expected performance to confirm the strategies still deliver, and to feed lessons back into the model.

**This creates a self-correcting cycle:** mine  diagnose  design  simulate  deploy  monitor  re-mine — ensuring the scheduling system remains optimal as Precision Parts Inc.'s operating environment evolves.

---

### Summary
The approach moves Precision Parts Inc. from **reactive, local, static dispatching** to a **proactive, global, data-driven scheduling system**. Process mining provides the empirical evidence (bottlenecks, setup matrices, duration distributions, disruption impacts) to (1) precisely diagnose whether problems are *logic*, *capacity*, or *variability* driven, (2) design layered dynamic/predictive/setup-aware strategies grounded in real data, and (3) validate them risk-free in a mined digital twin before deployment — all wrapped in a continuous-improvement loop that keeps the system tuned to a changing shop floor.
