# 1) Analyzing Historical Scheduling Performance and Dynamics

The MES event log is rich enough to reconstruct the *actual* execution of each job, operation, machine, and disruption. The key is to turn raw timestamped events into a complete operational picture.

## 1.1 Reconstructing actual flow with process mining

### a) Event log preparation
I would first preprocess the MES logs into a clean event log with:
- **Case ID** = Job ID
- **Activity** = Job stage/event type
- **Timestamp**
- **Resource** = machine/operator
- **Attributes** = priority, due date, planned duration, actual duration, setup required, breakdown flag, etc.

Then I would standardize lifecycle transitions:
- Queue entry
- Setup start / end
- Processing start / end
- Move / transfer to next machine
- Breakdown start / end
- Priority changes

This allows building a **full case timeline** from release to completion.

### b) Process discovery
Use process mining discovery techniques to reconstruct actual flow:
- **Inductive Miner** or **Heuristic Miner** to discover the dominant process model
- **Variant analysis** to identify common routing patterns across job families
- **Directly-follows graphs (DFG)** to visualize task transitions and rework loops
- **Object-centric process mining** if jobs, machines, operators, and setups need to be analyzed jointly

For a job shop, the log will likely show many variants due to custom routings. That is useful: it exposes process complexity, routing flexibility, and points of delay.

### c) Conformance checking
Compare the discovered model to the “intended” routing or planned process:
- Identify deviations in task order
- Detect skipped, repeated, or reworked steps
- Measure schedule drift between planned and actual execution

This is important because the current scheduling rules may look acceptable locally but fail globally.

---

## 1.2 Metrics for performance analysis

### a) Job flow times, lead times, makespan distributions
For each job:
- **Flow time / lead time** = completion timestamp  release timestamp
- **Makespan** = start of first operation to end of last operation
- **Process segment times** = duration spent in queue, setup, processing, transfer, waiting for operator, waiting for machine

Analyze these using:
- Descriptive statistics: mean, median, percentiles, coefficient of variation
- Distribution fitting: lognormal, gamma, Weibull often fit manufacturing times
- Stratification by job family, priority, due date tightness, and routing complexity

Useful outputs:
- Lead time histograms
- Boxplots by product family
- CDFs for on-time completion probability
- Tail analysis for extreme delays

### b) Task waiting times at each work center
For each operation:
- **Queue time** = setup start  queue entry
- **Resource waiting time** = machine unavailable / operator unavailable / breakdown waiting
- **Inter-operation waiting time** = delay between consecutive operations for the same job

Use:
- Work center-specific queue time distributions
- Heatmaps of waiting time by machine and time of day
- Service-level metrics such as percent of operations starting within target waiting thresholds

This identifies where WIP accumulates.

### c) Resource utilization
For each machine/operator:
- **Productive time** = actual processing time
- **Setup time** = setup end  setup start
- **Idle time** = available time  occupied time
- **Blocked/starved time** if available from log or inferred from state transitions

Metrics:
- Utilization rate = productive + setup time / available time
- Effective utilization = productive time / available time
- Setup share of occupied time
- OEE-style analysis if quality/rework data exists

This shows whether machines are overburdened or underused.

---

## 1.3 Quantifying sequence-dependent setup times

This is a key issue in your scenario.

### Approach
For each machine, reconstruct the **sequence of jobs processed** and pair:
- Previous job characteristics
- Current job characteristics
- Setup duration between them

Then analyze setup duration as a function of:
- Material type
- Geometry class
- Tooling family
- Thickness / alloy
- Previous job family  current job family transition
- Operator / shift effects
- Machine type

### Techniques
- Transition matrix of job family pairs vs. average setup time
- Regression / tree models predicting setup time from previous-current job pair features
- Heatmaps of sequence-dependent setup matrices
- Clustering of jobs into setup-compatible families

Example:
- Transition from stainless steel milling to aluminum milling may have low setup
- Transition from heat-treated part to precision grinding may have high cleaning and calibration setup

This produces a **setup-time matrix** that can be directly used in scheduling.

---

## 1.4 Schedule adherence and tardiness

For each job and operation:
- **Due date tardiness** = completion time  due date, if positive
- **Earliness** = due date  completion time, if positive
- **Tardiness rate** = % of jobs late
- **Mean tardiness**, **median tardiness**, **95th percentile tardiness**
- **Weighted tardiness** if priorities or penalty weights exist

At operation level:
- Compare planned start/finish vs actual start/finish
- Measure schedule slippage per operation
- Track lateness propagation through routing

Useful analyses:
- On-time delivery by customer segment or priority class
- Tardiness by release-to-due-date slack
- Tardiness by route complexity and bottleneck machine usage

---

## 1.5 Impact of disruptions

Disruptions should be modeled as first-class events.

### Breakdown analysis
For each breakdown:
- Duration
- Frequency by machine
- Lost production time
- Queued jobs affected
- Delay propagation to downstream tasks

Metrics:
- Mean time between failures (MTBF)
- Mean time to repair (MTTR)
- Breakdown-induced delay per job
- Recovery time after disruption

### Priority change / hot job analysis
Measure:
- Preemption frequency
- Jobs delayed by urgent inserts
- WIP and tardiness impact of each insert
- Stability cost: how often schedules are rewritten

### Process mining methods
- **Event alignment** to compare pre-disruption and post-disruption trajectories
- **Temporal bottleneck analysis** around disturbance windows
- **Variant comparison** between disrupted and non-disrupted cases
- **Performance overlays** to show how cycle times change after a breakdown

# 2) Diagnosing Scheduling Pathologies

Process mining should reveal not just symptoms, but the structure of inefficiency.

## 2.1 Bottleneck resources
Likely bottlenecks are specialized CNCs, grinders, or heat treatment units.

### Evidence
- Highest queue times
- Highest utilization
- Longest waiting chains upstream
- Largest contribution to total lead time
- Most severe tardiness propagation

### Process mining tools
- Resource throughput analysis
- Time-series of queue length / WIP by machine
- Token-based performance analysis on discovered model
- Sankey/flow analysis showing accumulation before bottlenecks

If one machine has high utilization and persistent queues while downstream resources are starved, that machine is constraining system throughput.

---

## 2.2 Poor prioritization
The current FCFS/EDD mix may fail when:
- Hot jobs arrive late and are inserted poorly
- Due-date critical jobs are not expedited
- Similarity of jobs is ignored, causing avoidable setups

### Evidence
- High-priority jobs still experiencing long queue times
- Near-due jobs not receiving earlier service
- On-time jobs being delayed by low-urgency jobs

### Process mining diagnostics
- Variant analysis comparing high-priority vs low-priority job paths
- Scatterplots of priority vs queue delay
- Late-job trace clustering
- Waiting time distributions by order priority

If priority labels exist in the log, you can directly test whether urgency actually influenced execution order.

---

## 2.3 Suboptimal sequencing and setup inflation
The shop may be paying unnecessary setup penalties because scheduling ignores sequence dependence.

### Evidence
- Frequent transitions between setup-incompatible job families
- Large average setup times on certain machine-job transitions
- Setup time dominating occupancy on bottleneck machines
- Repeated “zig-zag” between dissimilar jobs

### Process mining methods
- Transition matrix of job families with setup cost overlay
- Analysis of machine-specific job sequences
- Counterfactual comparison: actual sequence vs grouped-by-family sequence cost
- Correlation between setup time and tardiness

This often reveals that local dispatching optimizes immediate due-date urgency but worsens global throughput.

---

## 2.4 Starvation and downstream underutilization
Upstream bottlenecks or poor release control can starve downstream machines.

### Evidence
- Idle periods at downstream resources despite backlog elsewhere
- High transfer delays between operations
- Jobs waiting between routing stages
- Uneven WIP distribution

### Process mining tools
- Machine idle/starved time segmentation
- Inter-operation waiting analysis
- Flow imbalance metrics by work center
- Time-window analysis of resource states

---

## 2.5 Bullwhip-like WIP amplification
In job shops, release variability and rescheduling can create unstable WIP waves.

### Evidence
- Sudden WIP surges after hot-job insertions or breakdowns
- Large variation in queue lengths over time
- High lead-time variability even when average load is stable

### Process mining methods
- Time-series of WIP and queue length
- Variability decomposition by source: release policy, routing complexity, disruption
- Comparison of stable vs disturbed periods

# 3) Root Cause Analysis of Scheduling Ineffectiveness

## 3.1 Likely root causes

### a) Static dispatching rules in a dynamic environment
FCFS/EDD alone cannot handle:
- Sequence-dependent setups
- Resource heterogeneity
- Dynamic priority changes
- Machine breakdowns
- Job-specific slack variability

These rules optimize locally and myopically.

### b) Lack of real-time visibility
Without full shop-floor state:
- Jobs are dispatched without knowing downstream congestion
- Machine queues are not balanced globally
- Operators may be assigned without considering future bottlenecks

### c) Poor duration estimation
If planning uses average task durations without context:
- Underestimated processing times lead to systematic tardiness
- Setup times vary by sequence, operator, and machine state
- Learning effects or fatigue may be ignored

### d) Ineffective setup handling
If setup is treated as constant or negligible:
- The schedule may look feasible on paper but fails in execution
- Sequence decisions at bottlenecks become suboptimal

### e) Weak disruption response
Breakdowns and hot jobs require dynamic rescheduling, not static fixed plans.

---

## 3.2 Distinguishing scheduling logic issues vs capacity/variability limits

Process mining helps separate these through comparison and decomposition.

### Indicators of scheduling logic failure
- High tardiness even when utilization is moderate
- Long waits concentrated in specific priority groups
- Large setup losses caused by poor sequencing
- Avoidable routing delays despite available capacity elsewhere

### Indicators of structural capacity limits
- Persistent bottlenecks at near-100% utilization
- Delays across all dispatching policies
- Queue growth even under optimized sequencing
- Performance degradation tied to insufficient machine count or skill coverage

### Indicators of process variability
- Wide spread in task durations for the same operation
- High breakdown frequency or repair variability
- Significant operator-dependent performance differences

### How process mining helps
- Compare performance across time windows, products, and resources
- Use regression / survival models to explain delay with covariates:
  - queue length
  - machine load
  - setup sequence
  - priority
  - disruption state
- If delays remain high after controlling for these factors, structural capacity may be the cause

In short: process mining provides the evidence base to say whether the issue is **bad policy**, **insufficient capacity**, or **unstable execution**.

# 4) Developing Advanced Data-Driven Scheduling Strategies

Below are three distinct strategies that are practical, dynamic, and informed by mined data.

---

## Strategy 1: Multi-criteria dynamic dispatching with sequence-aware scoring

### Core logic
At each machine, rank available jobs using a composite priority score such as:

**Score =**
- urgency factor
- slack factor
- remaining processing time
- downstream congestion factor
- estimated setup time from previous job to candidate job
- priority weight
- disruption sensitivity

Example conceptually:
- Prefer jobs with low slack and high priority
- Penalize jobs that create expensive setups
- Penalize jobs whose downstream route is congested
- Consider expected remaining flow time, not just local due date

### How process mining informs it
From logs, derive:
- Actual queue-time distributions by machine
- Setup-time transition matrix
- Average downstream waiting by route
- Tardiness contribution by job family and operation

These historical patterns define the weights and features of the dispatching score.

### How it addresses pathologies
- Reduces tardiness by considering urgency and global route risk
- Reduces setup waste by sequencing compatible jobs
- Reduces WIP by avoiding releases into already congested paths
- Improves fairness by not relying solely on FCFS

### Expected KPI impact
- Lower average and 95th percentile tardiness
- Reduced setup time
- Lower WIP
- Better machine utilization balance

---

## Strategy 2: Predictive scheduling using duration and disruption forecasting

### Core logic
Build predictive models for:
- Processing time
- Setup time
- Queue delay risk
- Breakdown likelihood
- Delay propagation risk

Then generate schedules using expected completion times with uncertainty bounds.

### Data-driven inputs from process mining
- Historical duration distributions by operation, machine, operator, shift, and job family
- Sequence-dependent setup prediction model
- Breakdown frequency and MTTR by machine
- Delay propagation patterns from prior disruptions

### Scheduling approach
Use predictive estimates to:
- Identify jobs likely to become late before they do
- Reserve capacity on bottleneck machines for high-risk jobs
- Create robust schedules with buffer placement
- Trigger proactive re-sequencing when predicted lateness exceeds threshold

### How it addresses pathologies
- Reduces surprises from inaccurate standard times
- Improves on-time delivery under variability
- Supports proactive re-planning instead of reactive firefighting
- Better handles machine failures and hot jobs

### Expected KPI impact
- Lower schedule instability
- Lower tardiness under disruptions
- More realistic promised dates
- Reduced overtime and expediting

---

## Strategy 3: Setup-time optimization through family batching and bottleneck sequencing

### Core logic
Treat sequence-dependent setups as a first-class optimization objective.

At bottleneck machines:
- Group jobs into setup-compatible families
- Sequence within families by due date/slack
- Minimize costly family transitions
- Use controlled batching where feasible

### How process mining informs it
- Identify high-cost transitions from sequence matrices
- Detect families with recurring setups
- Quantify total setup loss by machine and by transition pattern
- Find where setup time dominates capacity consumption

### Scheduling tactics
- **Family batching**: process similar jobs consecutively
- **Transition-aware sequencing**: choose next job to minimize incremental setup cost
- **Window-based batching**: allow a short waiting window to collect similar jobs if due-date risk is low
- **Bottleneck-first setup optimization**: focus on machines with largest marginal impact

### How it addresses pathologies
- Cuts unnecessary setups
- Increases effective capacity at bottleneck machines
- Reduces lead time and queue buildup
- Improves utilization efficiency

### Expected KPI impact
- Lower total setup time
- Higher throughput
- Lower lead time variability
- Fewer bottleneck-induced delays

# 5) Simulation, Evaluation, and Continuous Improvement

## 5.1 Discrete-event simulation for strategy testing

Before deployment, I would build a simulation model parameterized from process-mined data.

### Inputs from process mining
- Processing time distributions
- Setup time matrices
- Routing probabilities / variant frequencies
- Breakdown distributions and repair times
- Arrival patterns / release patterns
- Priority mix and hot-job arrival rate
- Operator assignment logic

### Scenarios to test
1. **Baseline normal load**
2. **High-load periods** such as end-of-month rush
3. **Frequent breakdown scenario**
4. **High hot-job insertion scenario**
5. **Machine downtime at bottleneck resource**
6. **Mixed priority stress test**
7. **Alternative workforce availability** such as reduced staffing shifts

### Comparison metrics
- Average and percentile tardiness
- On-time delivery rate
- Average flow time / lead time
- WIP levels
- Machine utilization and idle/starved time
- Total setup time
- Schedule stability / number of resequences

### Evaluation method
Compare:
- Current baseline dispatching
- Strategy 1
- Strategy 2
- Strategy 3
- Hybrid combinations

This yields evidence on which policy performs best under which conditions.

Track continuously:
- tardiness

Methods:
- Control charts
- Change-point detection

Automatically update:
- dispatching weights
- duration prediction models

### Continuous improvement loop
1. Capture events from MES
2. Mine actual performance
3. Detect anomalies and drift

5. Validate in simulation
6. Deploy updated policy

This creates a living scheduling system rather than a static rule set.
