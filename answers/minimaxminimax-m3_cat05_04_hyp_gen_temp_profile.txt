### Anomaly 2: Excessive Approval-to-Notification Delay (P  N)

- **Issue:** A week-long delay between approval and customer notification is operationally suspicious. The high STDEV (28% of mean) further suggests unpredictable, sporadic behavior rather than a stable process.

## 2. Hypotheses for the Anomalies

### Hypothesis A — Automated/Batch Processing
The RP rigid timing strongly suggests a **scheduled batch job** (e.g., nightly approval cycle) rather than a continuous workflow. Claims may be queued and approved at a fixed time, producing artificially low variance.

### Hypothesis B — Resource Bottleneck in Customer Communication
The PN 7-day delay with high variance points to a **manual notification backlog**. Notification may depend on a separate team or postal/email system that isn't synchronized with the approval workflow.

### Hypothesis C — Rejection or Auto-Close Path
The AC short window may reflect a **fast-track rejection path** — claims that are denied at intake (e.g., for missing documentation) and closed without formal evaluation. Alternatively, the `C` event may be misrecorded for inactive or duplicate claims.

### Hypothesis D — Missing Approval Step Logging
The EN 5-minute transition suggests the **approval activity (P) is either skipped or mislabeled** in the event log. The system may auto-notify on evaluation completion, ignoring the approval gate.

### Hypothesis E — Data Quality / Synthetic Timestamps
The low STDEV values on RP and EN hint at **artificially generated or rounded timestamps**, possibly from a test dataset, system clock drift, or a logging bug that defaults to round values.

### Hypothesis F — Process Fragmentation
The high PN variance combined with tight RP suggests **decoupled subsystems**: one for claim processing (tight) and another for customer communication (loose), each with its own SLAs.

## 3. Verification SQL Queries

### Query 1 — Identify Claims with Rigid RP Timing (Out-of-Range Z-Score)

```sql
WITH ordered_events AS (
    SELECT 
        ce.claim_id,
        ce.activity,
        ce.timestamp,
        LEAD(ce.timestamp) OVER (PARTITION BY ce.claim_id ORDER BY ce.timestamp) AS next_timestamp,
        LEAD(ce.activity) OVER (PARTITION BY ce.claim_id ORDER BY ce.timestamp) AS next_activity
    FROM claim_events ce
)
SELECT 
    claim_id,
    activity AS first_activity,
    next_activity,
    EXTRACT(EPOCH FROM (next_timestamp - timestamp)) AS actual_seconds,
    ABS(EXTRACT(EPOCH FROM (next_timestamp - timestamp)) - 90000) / 3600.0 AS deviation_hours
FROM ordered_events
WHERE activity = 'R' 
  AND next_activity = 'P'
  AND ABS(EXTRACT(EPOCH FROM (next_timestamp - timestamp)) - 90000) > 3 * 3600
ORDER BY deviation_hours DESC;
```

*Detects claims whose RP delay deviates by more than 3× the STDEV (1 hour).*

### Query 2 — Claims with Excessive Approval-to-Notification Delay (P  N)

```sql
WITH event_pairs AS (
    SELECT 
        ce.claim_id,
        ce.timestamp AS p_time,
        LEAD(ce.timestamp) OVER (PARTITION BY ce.claim_id ORDER BY ce.timestamp) AS next_time,
        LEAD(ce.activity) OVER (PARTITION BY ce.claim_id ORDER BY ce.timestamp) AS next_activity
    FROM claim_events ce
    WHERE ce.activity = 'P'
)
SELECT 
    ep.claim_id,
    c.claim_type,
    c.submission_date,
    EXTRACT(EPOCH FROM (ep.next_time - ep.p_time)) / 86400.0 AS delay_days,
    EXTRACT(EPOCH FROM (ep.next_time - ep.p_time)) / 604800.0 AS weeks_delay
FROM event_pairs ep
JOIN claims c ON c.claim_id = ep.claim_id
WHERE ep.next_activity = 'N'
  AND EXTRACT(EPOCH FROM (ep.next_time - ep.p_time)) > 604800 + 3 * 172800
ORDER BY delay_days DESC
LIMIT 100;
```

*Flags claims where PN exceeds the mean + 3·STDEV ( 13 days).*

### Query 3 — Premature Closure: Claims Closed Quickly After Assignment

```sql
WITH claim_sequences AS (
    SELECT 
        claim_id,
        MIN(CASE WHEN activity = 'A' THEN timestamp END) AS assign_time,
        MIN(CASE WHEN activity = 'C' THEN timestamp END) AS close_time,
        MAX(CASE WHEN activity = 'E' THEN 1 ELSE 0 END) AS has_evaluate,
        MAX(CASE WHEN activity = 'P' THEN 1 ELSE 0 END) AS has_approve
    FROM claim_events
    GROUP BY claim_id
)
SELECT 
    cs.claim_id,
    c.claim_type,
    c.claim_amount,
    cs.has_evaluate,
    cs.has_approve,
    EXTRACT(EPOCH FROM (cs.close_time - cs.assign_time)) / 60.0 AS minutes_to_close
FROM claim_sequences cs
JOIN claims c ON c.claim_id = cs.claim_id
WHERE cs.assign_time IS NOT NULL 
  AND cs.close_time IS NOT NULL
  AND EXTRACT(EPOCH FROM (cs.close_time - cs.assign_time)) < 7200
  AND (cs.has_evaluate = 0 OR cs.has_approve = 0)
ORDER BY minutes_to_close ASC;
```

*Identifies claims closed within 2 hours of assignment that skipped evaluation or approval.*

### Query 4 — Suspiciously Fast E  N Transitions

```sql
SELECT 
    ce1.claim_id,
    c.claim_type,
    EXTRACT(EPOCH FROM (ce2.timestamp - ce1.timestamp)) AS eval_to_notify_seconds,
    ce1.timestamp AS eval_time,
    ce2.timestamp AS notify_time
FROM claim_events ce1
JOIN claim_events ce2 
  ON ce1.claim_id = ce2.claim_id 
 AND ce2.activity = 'N'
 AND ce2.timestamp > ce1.timestamp
 AND ce2.timestamp = (
     SELECT MIN(ce3.timestamp) 
     FROM claim_events ce3 
     WHERE ce3.claim_id = ce1.claim_id 
       AND ce3.activity = 'N' 
       AND ce3.timestamp > ce1.timestamp
 )
JOIN claims c ON c.claim_id = ce1.claim_id
WHERE ce1.activity = 'E'
  AND EXTRACT(EPOCH FROM (ce2.timestamp - ce1.timestamp)) < 300 - 3 * 60
ORDER BY eval_to_notify_seconds ASC;
```

*Flags EN transitions faster than 2 minutes (mean  3·STDEV).*

### Query 5 — Correlate Anomalies with Adjusters and Specialization

```sql
WITH adjuster_assignments AS (
    SELECT 
        ce.claim_id,
        ce.resource AS adjuster_resource
    FROM claim_events ce
    WHERE ce.activity = 'A'
),
r_to_p_times AS (
    SELECT 
        ce.claim_id,
        EXTRACT(EPOCH FROM (
            LEAD(ce.timestamp) OVER (PARTITION BY ce.claim_id ORDER BY ce.timestamp) - ce.timestamp
        )) AS r_to_p_seconds
    FROM claim_events ce
    WHERE ce.activity = 'R'
)
SELECT 
    a.adjuster_id,
    a.name,
    a.specialization,
    a.region,
    COUNT(*) AS total_claims,
    AVG(rtp.r_to_p_seconds) / 3600.0 AS avg_r_to_p_hours,
    STDDEV(rtp.r_to_p_seconds) / 3600.0 AS stdev_r_to_p_hours
FROM r_to_p_times rtp
JOIN adjuster_assignments aa ON aa.claim_id = rtp.claim_id
JOIN adjusters a ON a.name = aa.adjuster_resource
GROUP BY a.adjuster_id, a.name, a.specialization, a.region
HAVING COUNT(*) > 5
ORDER BY stdev_r_to_p_hours ASC;
```

*Reveals whether particular adjusters or specializations drive the timing variance.*

### Query 6 — Anomaly Correlation by Claim Type and Region

```sql
WITH transitions AS (
    SELECT 
        ce.claim_id,
        ce.activity AS from_act,
        LEAD(ce.activity) OVER (PARTITION BY ce.claim_id ORDER BY ce.timestamp) AS to_act,
        EXTRACT(EPOCH FROM (
            LEAD(ce.timestamp) OVER (PARTITION BY ce.claim_id ORDER BY ce.timestamp) - ce.timestamp
        )) AS delta_seconds
    FROM claim_events ce
)
SELECT 
    c.claim_type,
    a.region,
    t.from_act,
    t.to_act,
    COUNT(*) AS occurrence_count,
    AVG(t.delta_seconds) AS avg_seconds,
    STDDEV(t.delta_seconds) AS stdev_seconds
FROM transitions t
JOIN claims c ON c.claim_id = t.claim_id
LEFT JOIN claim_events ce_a ON ce_a.claim_id = t.claim_id AND ce_a.activity = 'A'
LEFT JOIN adjusters a ON a.name = ce_a.resource
WHERE t.from_act IN ('R','A','E','P') AND t.to_act IN ('A','E','P','N','C')
GROUP BY c.claim_type, a.region, t.from_act, t.to_act
ORDER BY c.claim_type, a.region, t.from_act, t.to_act;
```

*Aggregates timing patterns by claim type and region to surface localized irregularities.*

### Query 7 — Detect Missing Intermediate Events (Process Conformance)

```sql
SELECT 
    c.claim_id,
    c.claim_type,
    c.claim_amount,
    STRING_AGG(ce.activity, ' -> ' ORDER BY ce.timestamp) AS event_sequence,
    (STRING_AGG(ce.activity, '' ORDER BY ce.timestamp) LIKE '%P%') AS has_approve,
    (STRING_AGG(ce.activity, '' ORDER BY ce.timestamp) LIKE '%E%') AS has_evaluate
FROM claims c
JOIN claim_events ce ON ce.claim_id = c.claim_id
GROUP BY c.claim_id, c.claim_type, c.claim_amount
HAVING (STRING_AGG(ce.activity, '' ORDER BY ce.timestamp) LIKE '%N%')
   AND ((STRING_AGG(ce.activity, '' ORDER BY ce.timestamp) NOT LIKE '%E%')
        OR (STRING_AGG(ce.activity, '' ORDER BY ce.timestamp) NOT LIKE '%P%'));
```

*Identifies claims where notification occurred without proper evaluation or approval being logged.*
