# DuckDB SQL Query: Filter Cases by Event Count

## Recommended Query (using a window function)

```sql
SELECT
    case_id,
    activity,
    timestamp
FROM (
    SELECT
        *,
        COUNT(*) OVER (PARTITION BY case_id) AS event_count
    FROM event_log
) AS sub
WHERE event_count <= 6;
```

### How it works
1. **Inner subquery** — A window function `COUNT(*) OVER (PARTITION BY case_id)` calculates the total number of events for each `case_id` and attaches that value to every row belonging to that case.
2. **Outer query** — Filters the rows so that only cases with **6 or fewer events** are kept. Rows from any case with more than 6 events are discarded.

This is efficient in DuckDB because the window function is computed in a single pass over the table.

---

## Alternative: Using a CTE (Common Table Expression)

```sql
WITH case_event_counts AS (
    SELECT
        case_id,
        COUNT(*) AS event_count
    FROM event_log
    GROUP BY case_id
)
SELECT
    el.case_id,
    el.activity,
    el.timestamp
FROM event_log AS el
JOIN case_event_counts AS cec
  ON el.case_id = cec.case_id
WHERE cec.event_count <= 6;
```

### How it works
1. **`case_event_counts` CTE** — Aggregates the event log to compute the count per case.
2. **`JOIN` + `WHERE`** — Joins this count back to the original event log so we can return the full event rows for the cases that pass the threshold.

---

## Alternative: Using a subquery with `IN`

```sql
SELECT *
FROM event_log
WHERE case_id IN (
    SELECT case_id
    FROM event_log
    GROUP BY case_id
    HAVING COUNT(*) <= 6
);
```

This is a more compact version of the CTE approach. It works well in DuckDB and is very easy to read, but the window-function version is typically the fastest for large event logs.
