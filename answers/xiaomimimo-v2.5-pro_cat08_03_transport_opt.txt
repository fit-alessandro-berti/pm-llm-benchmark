# Process Mining Analysis and Optimization Strategy for Speedy Parcels

## 1. Process Discovery and Conformance Checking

### Data Preprocessing and Integration
To create a unified event log, I would:
- **Map data sources to a common schema**: Create a master table where each event has a Case ID (e.g., `Vehicle-Day` as shown), Timestamp, Activity, and relevant attributes.
- **Handle timestamp alignment**: Synchronize times across systems (GPS, scanner, dispatch) with possible time-zone adjustments and latency offsets. Use linear interpolation for GPS points between scanner events.
- **Create composite keys**: For package-level tracking, we might need a hierarchical case structure (Vehicle-Day  Package ID).
- **Address challenges**:
  - *Missing or noisy GPS data*: Use map-matching algorithms to snap coordinates to road networks.
  - *Activity naming inconsistencies*: Standardize labels (e.g., "Delivery Failed (Attempted)" vs. "Failed Delivery").
  - *Case ID definition*: Since a "case" could be either a vehicle-day or a package journey, we may create two logs: one for the vehicle's daily process and another for each package's lifecycle.

### Process Discovery
Using algorithms like the **Heuristics Miner** (robust to noise) or **Inductive Miner** (ensures sound models):
1. Generate a process model for the **vehicle-day lifecycle**: From `Start Shift`  `Route Assigned`  `Depart Depot`  sequence of `(Travel | Delay | Delivery)`  `Arrive Depot`  `End Shift`.
2. Include all event types: GPS-derived travel segments, scanner-based customer interactions, dispatch events, and maintenance interruptions.
3. Visualize the model to show actual flows, including:
   - Common sequences (e.g., successful deliveries vs. failed attempts).
   - Loops (e.g., returning to a customer for re-delivery after a failed attempt).
   - Parallel paths (e.g., one vehicle handling multiple packages in a single stop).

### Conformance Checking
Compare the discovered model against the **planned route** from the dispatch system:
- **Token-based replay**: Simulate execution of the planned route using the event log. Identify mismatches.
- **Deviation detection**:
  - *Sequence deviations*: Driver visits stops in different order than planned.
  - *Timing deviations*: Actual service time per stop exceeds allocated time; travel times differ from estimates.
  - *Unplanned stops*: Maintenance breaks, extended idling, or unauthorized stops.
  - *Skipped activities*: Planned deliveries not attempted, or attempts outside planned windows.
- **Root causes of non-conformance**: Traffic congestion, inaccurate travel time estimation, poor parking availability, customer unavailability, or vehicle issues.

## 2. Performance Analysis and Bottleneck Identification

### Key Performance Indicators (KPIs)
From the event log, calculate:

1. **On-Time Delivery Rate**: % of deliveries completed within customer-requested window.  
   *Formula*: `(Count of 'Delivery Success' events within window) / Total 'Delivery Success' events`.

2. **Average Time per Delivery Stop**: From `Arrive Customer` to `Depart Customer`.  
   *Formula*: `Mean(Depart - Arrive)` for successful deliveries.

3. **Travel Time vs. Service Time Ratio**: Efficiency of movement vs. productive delivery.  
   *Formula*: `Total travel time / Total service time (at stops)`.

4. **Fuel Consumption per km/package**: If fuel data isn't directly logged, estimate via GPS speed/idling time models. Alternatively, use vehicle class average consumption.

5. **Vehicle Utilization Rate**: Percentage of shift time spent moving (not idle).  
   *Formula*: `(Total moving time) / (End Shift - Start Shift)`.

6. **Traffic Delay Impact**: Duration of `Low Speed Detected` events or actual vs. estimated travel times for segments.

7. **Failed Delivery Rate**: % of `Delivery Failed` events over total delivery attempts.  
   *Formula*: `(Failed attempts) / (Success + Failed attempts)`.

### Bottleneck Identification Techniques
- **Process Mining Analytics**:
  - *Dwell Time Analysis*: Calculate median/90th percentile time for each activity. Compare across routes, times of day, drivers.
  - *Bottleneck Detection*: Identify activities with high waiting times or resource contention (e.g., long queues at depot loading).
  - *Variant Analysis*: Compare performance of different execution paths (e.g., routes with early vs. late deliveries).
  - *Correlation Analysis*: Overlay traffic data, weather, or event density to see impact on delays.

- **Bottleneck Types**:
  - *Geographical*: Specific locations (e.g., downtown areas) with consistently high travel times.
  - *Temporal*: Peak hours causing congestion.
  - *Driver-related*: Some drivers have longer service times or deviations.
  - *Vehicle-related*: Certain vehicles more prone to breakdowns.
  - *Activity-related*: Finding parking, customer interaction.

- **Quantifying Impact**: Calculate average delay time per bottleneck occurrence, multiply by frequency to estimate total time/cost impact.

## 3. Root Cause Analysis for Inefficiencies

### Potential Root Causes
1. **Static Route Planning**: Dispatch plans optimal routes but real-time conditions (traffic, parking) cause deviations.
2. **Inaccurate Travel Time Estimation**: System underestimates travel times due to not accounting for traffic patterns or road conditions.
3. **High Traffic Congestion**: Specific corridors or times have consistent delays.
4. **Variable Service Times**: Some customers take longer due to package type (signature required, fragile) or location (apartment vs. house).
5. **Vehicle Reliability Issues**: Older vehicles break down more often during shifts.
6. **Driver Behavior**: Some drivers take longer routes or spend more time per stop.
7. **Failed Delivery Cascades**: Failed attempts require return trips, increasing cost and time.

### Process Mining Validation Techniques
- **Variant Analysis**: Compare process models of high-performing vs. low-performing drivers/routes. Identify activity sequences that lead to success.
- **Drill-down Analysis**: For bottleneck locations, examine GPS data to see if delays are due to speed, idling, or circling for parking.
- **Correlation Analysis**: Merge with external data (traffic APIs) to see if delays correlate with known congestion zones/times.
- **Resource Analysis**: Calculate per-driver/per-vehicle KPIs to isolate performance differences.
- **Root Cause of Failed Deliveries**: Analyze `Delivery Failed` events—do they cluster in certain areas, times, or with specific drivers? Check if customer availability windows are mismatched with planned delivery times.
