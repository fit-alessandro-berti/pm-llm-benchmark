# Comprehensive Process-Mining Strategy for Instance-Spanning Constraint Optimization

## 1. Identifying Instance-Spanning Constraints and Their Impact

### 1.1 Data preparation and process mining approach

The event log already contains **lifecycle transitions** (`START`/`COMPLETE`) and **case-level attributes** (Order Type, Destination Region, Requires Cold Packing, Hazardous Material). The first step is to transform the log into two complementary views:

| View | Purpose |
|------|---------|
| **Case-centric interval log** | End-to-end cycle time, waiting time between activities, and order attribute-based variants. |
| **Resource-centric interval log** | Who/what is busy when (Picker, Packing Station, QC Staff, Cold-Packing Station, Batch), enabling contention analysis. |

From these views, the following process mining techniques are applied:

- **Dotted chart / performance spectrum analysis**: Visualize cases over time to detect clusters of waiting at Packing, QC, and Shipping Label Generation.
- **Queue mining**: Reconstruct queues in front of each resource by ordering `START` events and matching them against the previous `COMPLETE` events of that resource.
- **Resource interaction / social network mining**: Detect handovers and contention among cases for the same station or staff.
- **Variant analysis**: Compare process variants (e.g., Standard vs. Express, Cold vs. Non-Cold, Hazardous vs. Non-Hazardous) to isolate the effect of each constraint.
- **Temporal case linkage**: Identify which orders were physically in the same batch, on the same cold-packing station, or in the hazardous concurrency window.

### 1.2 Constraint-specific identification and metrics

#### A. Shared Cold-Packing Stations (5 stations)

**Identification:**
- Filter cases where `Requires Cold Packing = TRUE`.
- Build an interval timeline for each cold-packing station (`Station C1`–`C5`) from `Packing START` to `Packing COMPLETE`.
- If a cold order’s `Packing START` occurs after its preceding activity `COMPLETE`, the gap (minus transport/setup) is **queue time for a cold station**.

**Metrics:**
- `Cold_Wait_Time = Packing_START  max(prev_activity_COMPLETE, station_available_time)`
- Average, median, p90/p95 cold-packing queue time.
- Cold-station utilization (`busy time / available time`) and queue length distribution per hour.
- Percentage of cold orders whose queue time exceeds the SLA threshold.
- Comparison of cold-order cycle time vs. non-cold standard orders.

#### B. Shipping Batching

**Identification:**
- Detect batch membership from the `Resource` field (e.g., `System (Batch B1)`) or by clustering `Shipping Label Gen.` events with the same destination region occurring within a short time window.
- For each order, the **ready-for-label** time is the `Quality Check COMPLETE` (or `Packing COMPLETE` if QC is skipped) timestamp.

**Metrics:**
- `Batch_Wait_Time = Shipping_Label_COMPLETE  Ready_For_Label_Time  Label_Service_Time`
- Batch formation time: `max(ready time in batch)  min(ready time in batch)`.
- Average batch size, number of batches per region per day.
- Orders delayed more than X minutes waiting for the last member of a batch.
- Cost–benefit: estimated transport savings per batch vs. inventory holding/waiting cost.

#### C. Priority Handling of Express Orders

**Identification:**
- Compare Express vs. Standard queue behavior at shared resources (Packing, QC, Cold-Packing).
- Look for explicit `SUSPEND`/`RESUME` lifecycle events, or infer preemption when:
  - A standard order’s `Packing START` is delayed after an express order arrives at the same station queue.
  - A standard order has a long inactive gap between `Packing START` and `Packing COMPLETE` while an express order uses the same station.

**Metrics:**
- Express on-time/in-SLA rate.
- Additional waiting time incurred by **standard orders** due to express preemption (`Preemption_Delay`).
- Number of preemption events per day and per station.
- Standard order cycle time deterioration caused by express handling.
- Queue reordering index: how often an Express order jumps ahead of older Standard orders.

#### D. Hazardous Material Regulatory Limit (max 10 concurrent in Packing/QC)

**Identification:**
- Create a combined interval log for all hazardous (`Hazardous Material = TRUE`) `Packing` and `Quality Check` events.
- For every timestamp, compute the number of hazardous orders simultaneously in Packing or QC.
- Identify periods where the count reaches or exceeds 10.

**Metrics:**
- Hazardous concurrency profile over time (avg, max, % of time at cap).
- `Hazmat_Wait_Time`: time a hazardous order waits because the facility is already at 10 concurrent hazardous orders.
- Number of hazardous orders blocked per day and average blocking duration.
- Throughput impact: compare completion rate of hazardous orders during peak vs. off-peak.
- Compliance risk: count of timestamps where concurrency exceeds 10 (violation events).

### 1.3 Differentiating within-instance vs. between-instance waiting time

Waiting time can be decomposed as follows:

```
Cycle_Time =  Service_Time(activity) 
           +  Within_Case_Wait_Time (e.g., transport, manual holds, rework) 
           +  Between_Instance_Wait_Time (resource contention, batching, preemption, regulatory cap)
```

Using the lifecycle data:

- **Service time (within-instance)**: `START`  `COMPLETE` while the resource is actively working on the case.
- **Queue time (between-instance)**: time between the completion of the previous activity and the `START` of the next activity, **minus** an expected transfer/setup time derived from non-congested periods or direct travel estimates.
- **Batch wait**: time between “ready for label” and label `COMPLETE`, not attributable to service.
- **Preemption delay**: inactive interruption gap inside an activity interval caused by another case taking over the resource.

**Technique:** Build a resource-state timeline. If a case is waiting while the required resource is busy with another case, the delay is **between-instance**. If the resource is idle but the case is still waiting (e.g., transport delay, missing information), it is classified as **within-instance or operational delay**. Sensitivity analysis using queue-mining algorithms helps separate the two causes.

## 2. Analyzing Constraint Interactions

The constraints do not operate independently. Key interactions include:

| Interaction | Mechanism | Impact |
|-------------|-----------|--------|
| **Express priority + Cold-Packing capacity** | An Express order needing cold-packing may preempt a Standard order at one of only 5 cold stations. | Short-term SLA gain for Express, but increased variability and longer tail waits for Standard cold orders; potential starvation of Standard orders during peak. |
| **Batching + Hazardous limits** | If several hazardous orders are destined for the same region, they are forced to wait both for each other to form a batch and for hazardous concurrency slots to open. | Double synchronization penalty; the last hazardous order in a batch may be ready but blocked by the 10-order cap while also holding back non-hazardous orders in the same batch. |
| **Hazardous limit + Cold-Packing / general Packing** | Hazardous concurrency counts across all Packing and QC stations, including cold-packing stations. | Even if cold-packing capacity is added, the hazardous cap may still throttle hazardous orders and indirectly occupy station time. |
| **Express priority + Batching** | Express orders are supposed to be fast, but if they are forced into region-based shipping batches, the batch wait can nullify the priority benefit. | Express service-level degradation; need for Express-specific batch rules. |
| **Batching + Cold-Packing queue** | Orders waiting for a batch occupy floor space and may block downstream buffers, indirectly congesting upstream packing stations. | Increased WIP and cycle-time inflation across the whole facility. |

**Why interactions matter:** Optimizing one constraint in isolation can worsen another. For example, adding cold-packing stations reduces cold wait but does not help if the hazardous cap is the true bottleneck. Similarly, always preempting Standard orders for Express may produce excellent Express SLAs but degrade overall throughput and destabilize batch formation. A holistic, constraint-aware optimization strategy must therefore consider these trade-offs simultaneously.

## 3. Developing Constraint-Aware Optimization Strategies

### Strategy 1: Predictive, Priority-Gated Scheduling for Cold-Packing Stations

**Primary constraints addressed:** Shared Cold-Packing Stations, Priority Handling, and interaction with Hazardous limits.

**Proposed changes:**
1. Introduce a **centralized cold-packing queue** with predictive dispatching.
2. Use historical and real-time data (picking completion rates, order attributes, time-of-day patterns) to forecast cold-packing demand in 15–30 minute windows.
3. Reserve one of the five cold stations as an **Express/fast-lane station** during peak hours, while limiting preemption of Standard orders:
   - Preempt a Standard order only if the Express order cannot be served within X minutes otherwise.
   - If a Standard order is interrupted, schedule its resumption before any new Standard order to minimize extra setup.
4. Before dispatching a hazardous cold order, check the **facility-wide hazardous concurrency counter**. If the cap is near 10, hold the order in a buffer rather than occupying the cold station, freeing the station for non-hazardous cold orders.

**Data leverage:**
- Process mining provides empirical distributions of cold-packing durations and queue lengths by hour and order type.
- Machine learning on upstream completion times predicts when each cold order will arrive at the queue.
- Conformance checking verifies whether the new dispatching rules are followed.

**Expected outcomes:**
- Reduced mean and p90 cold-packing wait time.
- Improved Express SLA attainment without excessive Standard-order starvation.
- Better utilization of scarce cold stations by avoiding blockages due to the hazardous cap.
- Lower WIP and floor congestion.

### Strategy 2: Dynamic, Service-Level-Aware Shipping Batch Formation

**Primary constraints addressed:** Shipping Batching, interaction with Express priority, and indirect interaction with Hazardous limits.

**Proposed changes:**
1. Replace fixed batch cutoffs with a **dynamic batch trigger**:
   - For each destination region, maintain a running batch.
   - Close the batch when either:
     - The batch reaches a target size (`N_orders`), or
     - The oldest waiting order has reached a maximum wait time (`T_max`).
2. Differentiate by service level:
   - **Express orders**: use a smaller `N_express` and shorter `T_max_express`, or bypass batching entirely and generate labels individually.
   - **Standard orders**: use larger batches to preserve route-optimization savings.
3. Separate hazardous orders into their own batch lane if the hazardous concurrency limit is saturated, preventing a hazardous order from delaying non-hazardous orders in the same batch.

**Data leverage:**
- Process mining identifies the historical distribution of arrival intervals per region and service level.
- Simulation/optimization finds the Pareto frontier between route-cost savings and batch-waiting cost.
- Historical batch-wait time distributions are used to set region-specific `N` and `T_max` values.

**Expected outcomes:**
- Shorter and more predictable waiting time at Shipping Label Generation.
- Express orders are no longer held back by Standard batch formation.
- Route efficiency is preserved for Standard orders while Express SLAs improve.
- Reduced synchronization penalty between batching and hazardous-cap constraints.

### Strategy 3: Real-Time Concurrency Control and Capacity/Staffing Rebalancing

**Primary constraints addressed:** Hazardous Material regulatory limit, Packing/QC contention, and interaction with Cold-Packing.

**Proposed changes:**
1. Implement a **real-time hazardous-occupancy dashboard and gate**:
   - Track the current number of hazardous orders in Packing and QC combined.
   - Release hazardous orders into Packing/QC only when the count is below 10, with a lookahead buffer (e.g., do not release if count will reach 9 with already-assigned orders).
2. Use **order release sequencing**:
   - During peak hazardous demand, temporarily prioritize non-hazardous orders in Packing/QC to keep overall throughput up.
   - Postpone low-priority hazardous orders upstream (after Item Picking) rather than letting them congest Packing/QC.
3. Add **overflow QC capacity** or flexible cross-trained QC staff during peak hazardous windows, identified from process mining.
4. If business case supports it, evaluate adding a sixth cold-packing station or converting one general station to dual-use cold-capable, while ensuring it still counts toward the hazardous cap.

**Data leverage:**
- Concurrency profiles from process mining identify peak hours and bottleneck activities (Packing vs. QC).
- Variant analysis shows whether hazardous orders spend disproportionately more time in QC than Packing, indicating where to add capacity.
- Simulation quantifies throughput gain from each additional QC staff member or station.

**Expected outcomes:**
- Zero violations of the 10-order hazardous limit.
- Reduced hazardous-order blocking time.
- Higher overall throughput because non-hazardous orders are not unnecessarily delayed.
- Informed capital decisions on whether extra cold-packing or QC capacity is justified.

### 4.2 Calibration and baseline validation

1. Run the DES in **AS-IS mode** using the discovered parameters.
2. Replay a representative week/month of historical arrivals through the simulator.
3. Compare simulated KPIs (cycle time, queue lengths, resource utilization, batch waits, hazardous concurrency peaks) with the actual event log.
4. Iterate until the model’s outputs are statistically close to reality (e.g., cycle time within ±5–10%).

### 4.3 Scenario testing

Simulate at least these scenarios:

1. **AS-IS baseline.**
2. **Strategy 1 only** — predictive cold-packing scheduling.
3. **Strategy 2 only** — dynamic batching.
4. **Strategy 3 only** — hazardous concurrency control + staffing.
5. **Combined strategies** — full integrated approach.
6. **Sensitivity analyses** — varying cold station count, batch size thresholds, hazardous cap, and Express arrival share.

### 4.4 Key simulation focus areas

To ensure the model accurately captures instance-spanning constraints:

- **Resource contention:** Model each station as an individual resource with a finite queue; track queue lengths and waiting times.
- **Preemption:** Implement explicit interrupt/resume logic for Express orders, measuring the added delay to preempted Standard orders.
- **Batching delays:** Model batch accumulation with size and time triggers; track each order’s wait-for-batch.
- **Regulatory limit:** Maintain a global counter of hazardous orders in Packing/QC; enforce a hard limit and record blocking events.
- **Stochastic variability:** Use distributions, not averages, to expose queue buildups during peak arrivals.
- **KPI evaluation:** End-to-end cycle time by order type, throughput, SLA attainment, resource utilization, average queue lengths, compliance violations, and cost per order.

## Summary

The fulfillment center’s performance is governed by four tightly coupled instance-spanning constraints: limited cold-packing stations, shipping batching, Express preemption, and a hazardous-material concurrency cap. A rigorous process mining approach—combining queue mining, resource timeline analysis, variant analysis, and temporal case linkage—can quantify each constraint’s impact and separate it from normal activity time. Because the constraints interact, optimization must be holistic:

1. **Predictive, priority-gated cold-packing scheduling** protects Express SLAs while controlling Standard-order starvation.
2. **Dynamic, service-level-aware batching** reduces shipping-label wait without sacrificing route efficiency.
3. **Real-time hazardous concurrency control and capacity rebalancing** maintains compliance and protects throughput.

These strategies should be validated through a calibrated discrete-event simulation that faithfully models resource contention, preemption, batch formation, and the hazardous cap. Finally, continuous process mining dashboards ensure that improvements are sustained and that the system adapts as demand patterns evolve.
