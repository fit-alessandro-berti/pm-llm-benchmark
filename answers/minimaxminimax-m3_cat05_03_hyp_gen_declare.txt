# Analysis of DECLARE Model Anomalies

## 1. Identified Anomalies

### Anomaly A: Contradictory Coexistence Constraint
The **`noncoexistence`** rule (`E` cannot coexist with `C`) directly conflicts with the **intended business flow** (R  A  E  P  N  C). Every properly processed claim should contain both an Evaluate and a Close event, but the model forbids them from occurring together in the same trace.

### Anomaly B: Conflict with Existence Constraint
The **`existence.C`** rule states that `C` (Close) must occur in every claim, while **`noncoexistence.E,C`** forbids `E` whenever `C` occurs. The logical consequence: **every claim must be closed, but no claim may ever be evaluated.** This creates a model where the only permitted path is R  A  C, completely bypassing evaluation.

### Anomaly C: Missing Mandatory Steps
There is no `existence`, `precedence`, or `responded_existence` rule for:
- `P` (Approve) — required in the intended flow
- `N` (Notify) — required in the intended flow

These steps can be silently omitted, yet the model reports full conformance.

### Anomaly D: Weak Precedence Chain
The **`precedence.C  R`** rule only enforces that `C` happens after `R`. It does **not** enforce `C` after `E`, `P`, or `N`. Therefore, even if the noncoexistence rule were removed, the model would still permit shortcut paths like R  A  C, skipping the evaluation, approval, and notification stages.

### Anomaly E: Unordered Responded Existence
The **`responded_existence.E  A`** rule guarantees that an Assign occurs whenever Evaluate occurs, but it does **not** enforce that Assign happens **before** Evaluate. Evaluate could theoretically occur before the adjuster is assigned, which violates business logic.

## 3. SQL-Based Verification Strategies

### V1. Find Claims Closed Without Any Evaluation
Detects the most critical anomaly: completed claims that bypass the Evaluate step.
```sql
SELECT c.claim_id, c.claim_type, c.submission_date
FROM claims c
JOIN claim_events ce_close ON c.claim_id = ce_close.claim_id
                            AND ce_close.activity = 'C'
WHERE NOT EXISTS (
    SELECT 1 FROM claim_events ce_eval
    WHERE ce_eval.claim_id = c.claim_id
      AND ce_eval.activity = 'E'
);
```

### V2. Detect Traces Containing Both Evaluate and Close (Violation of Noncoexistence)
Confirms whether the noncoexistence rule is empirically supported or violated in practice.
```sql
SELECT claim_id,
       COUNT(*) FILTER (WHERE activity = 'E') AS eval_count,
       COUNT(*) FILTER (WHERE activity = 'C') AS close_count
FROM claim_events
GROUP BY claim_id
HAVING COUNT(*) FILTER (WHERE activity = 'E') > 0
   AND COUNT(*) FILTER (WHERE activity = 'C') > 0;
```

### V3. Find Claims With Evaluation but No Adjuster Assignment
Tests the `responded_existence` rule in the reverse direction.
```sql
SELECT DISTINCT ce.claim_id
FROM claim_events ce
WHERE ce.activity = 'E'
  AND ce.claim_id NOT IN (
      SELECT claim_id FROM claim_events WHERE activity = 'A'
);
```

### V4. Verify Approve and Notify Steps Are Always Present
Reveals whether P and N are systematically skipped, supporting the missing-constraint anomaly.
```sql
SELECT c.claim_id,
       CASE WHEN EXISTS (SELECT 1 FROM claim_events WHERE claim_id = c.claim_id AND activity = 'P')
            THEN 'YES' ELSE 'NO' END AS has_approve,
       CASE WHEN EXISTS (SELECT 1 FROM claim_events WHERE claim_id = c.claim_id AND activity = 'N')
            THEN 'YES' ELSE 'NO' END AS has_notify
FROM claims c
ORDER BY has_approve, has_notify;
```

### V5. Check Temporal Ordering — Is Close Always After Evaluate?
Tests whether the intended ordering is respected, despite the model's weak `precedence` rule.
```sql
WITH order_check AS (
    SELECT claim_id,
           MIN(timestamp) FILTER (WHERE activity = 'E') AS eval_time,
           MIN(timestamp) FILTER (WHERE activity = 'C') AS close_time
    FROM claim_events
    GROUP BY claim_id
)
SELECT claim_id, eval_time, close_time
FROM order_check
WHERE eval_time IS NOT NULL
  AND close_time IS NOT NULL
  AND close_time < eval_time;
```

### V6. Check Whether Adjuster Specialization Matches Claim Type
A cross-table validation to detect if the assignment step uses proper resources.
```sql
SELECT ce.claim_id, c.claim_type, a.specialization
FROM claim_events ce
JOIN claims c ON ce.claim_id = c.claim_id
JOIN adjusters a ON ce.resource = a.adjuster_id::text
WHERE ce.activity = 'A'
  AND ((c.claim_type LIKE '%home%' AND a.specialization <> 'home')
    OR (c.claim_type LIKE '%auto%' AND a.specialization <> 'auto'));
```

### V7. Distribution of Activities per Claim (Process Variant Analysis)
A broad diagnostic to identify the actual dominant process variants and quantify anomalies.
```sql
SELECT claim_id,
       STRING_AGG(activity, '->' ORDER BY timestamp) AS trace,
       COUNT(*) AS event_count
FROM claim_events
GROUP BY claim_id
ORDER BY event_count;
```
