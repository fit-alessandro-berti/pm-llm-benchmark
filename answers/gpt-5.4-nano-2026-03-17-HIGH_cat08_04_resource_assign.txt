## 1) Analyzing Resource Behavior and Assignment Patterns

### 1.1 How to use the event log to analyze agent/tier performance
Using the fields in your event log (Case ID, Activity, Timestamp, Resource, Agent Tier, Agent Skills, Priority, Category, Required Skill, Timestamp Type), I would structure analysis around **three layers**: (a) workload & flow, (b) performance & outcomes, and (c) skill alignment.

#### A. Workload distribution & flow metrics
Compute per **agent** and per **tier** (L1/L2/L3):

- **Case handling volume**  
  - # of distinct tickets handled by each resource/tier
  - # of ôwork startsö by each resource/tier (robust even when agents reassign)
- **Work-in-progress (WIP) / queue indicators** (derived from gaps between ôAssign Lxö and ôWork Lx Startö)
  - Avg queue time to start work:  
    \[
    \text{queue\_time}(case, tier)=t(\text{Work Lx Start})-t(\text{Assign Lx})
    \]
- **Reassignment and escalation frequency**
  - Avg # of reassignments per case
  - % of cases with 1 reassignment within tier
  - % of cases escalated (L1L2, L2L3)
- **Handover counts between tiers**
  - Avg # of tier transitions per ticket (e.g., L1L2L3)
- **Throughput and cycle times**
  - Avg processing time per activity (e.g., ôWork L1 Startö  ôWork L1 Endö)
  - Avg total cycle time from ôTicket Createdö to ôCompleteö
  - SLA-relevant time components (first response time, time-in-queue, time-in-resolution)

> Why this matters: workload imbalance and queue time spikes are often the *immediate* causes of SLA breaches, and process mining helps you isolate where the time is spent (queue vs. actual work vs. waiting for rework).

#### B. Performance & outcome metrics
For each agent/tier:

- **Completion rate / resolution outcomes**
  - % of cases completed at that tier
  - % of cases escalated after first work attempt
- **First-call (or first-tier) resolution for L1**
  - Define: ôL1 resolvedö = case contains ôWork L1 Endö with outcome COMPLETE and no L2/L3 work starts
  - Metric:  
    - L1 First Resolution Rate = L1-resolved cases / all cases assigned to L1
- **Effectiveness by priority**
  - SLA breach rate broken down by (tier responsible at key steps, priority)

#### C. Skill utilization & skill alignment metrics
From ôAgent Skillsö, ôRequired Skillö, and ôTicket Category/Required Skillö:

- **Skill match rate**
  - For each work start at a tier:  
    - matched = required skill  agentĺs skills (or required skill mapped to category skill taxonomy)
- **Over/under-skilling**
  - **Under-skilled assignment**: required skill not in agent skills at the moment of work start
  - **Over-skilled assignment** (potentially costly): agent has the required skill but was routed to a tier/task that historically could be solved by less skilled agents (measured via ôhistorically successful tier for that skillö)
- **Specialist leakage**
  - % of cases that required ôNetworking-Firewallö where the work was handled at L2/L3 despite L1 being capable for that subset historically
- **Skill-to-tier proficiency curves**
  - For each required skill, measure the distribution of outcomes (resolved/completed/escalated) by tier and agent skill matching

> Deliverable: a ôSkill Capability Matrixö showing for each skill (or category) the best-performing tier(s) and the actual routing behavior observed.

### 1.2 Process mining techniques to reveal *actual* assignment patterns vs intended logic

#### A. Resource interaction analysis (handover/social network)
Build a graph where:

- Nodes = agents (or tiers)
- Directed edges = ôhandoverö events (e.g., from Agent A handling at L1 to Agent B at L2 for the same case)
- Edge weight = # of handovers, or avg time between handovers

This reveals:

- **Which agents/tier pairs frequently exchange cases**
- **Where loops happen** (e.g., repeated reassignments L1L2L1)
- **Whether escalation correlates with specific dispatcher patterns** rather than ticket needs

#### B. Role discovery / role mining
Apply role mining to discover frequent ôresource rolesö (behavioral roles), such as:
- ôL1 triage specialistsö
- ôFirewall escalatorsö
- ôSQL expertsö
- ôReassignment brokersö (agents frequently receiving escalated work but not finishing)

Role discovery often uncovers mismatch between:
- nominal tier expectations
- actual behavior patterns

#### C. Conformance checking (intended assignment logic vs reality)
If you can encode the ôintendedö rules (e.g., ôP2 Network should go to agents with Networking-Firewall skills at L1 if available; otherwise escalate to L2ö), then:
- Model the expected routing logic (as rules, BPMN, or constraints)
- Use **conformance checking** to quantify:
  - constraint violations (e.g., required skill not present)
  - unexpected reassignments
  - missing escalations (or late escalations)

Outputs:
- ôViolation rate by priority/categoryö
- ôTop violated rulesö

---

### 1.3 Analyze utilization of specific skills across the agent pool
Approach:

1. **Extract skill demands** per case:
   - required_skill (explicit) and/or categoryskill mapping
2. For each required_skill:
   - count demand volume
   - count how many tasks were actually handled by agents possessing that skill
3. Compare **demand vs supply**:
   - available capacity = # of productive work-hours per agent with that skill (derived from logs: work start/end and/or busy periods)
4. Identify ôcapacity gapsö:
   - If demand for ôDatabase-SQLö is high but SQL-capable agents show long queues or frequent non-matching assignments, you have supply imbalance

Deliverable: a **Skill Capacity & Utilization Dashboard**:
- skill demand trend
- skill matching rate
- queue time for cases requiring that skill
- SLA breach rate for that skill

## 2) Identifying Resource-Related Bottlenecks and Issues

Below are concrete bottleneck patterns process mining can identify.

### 2.1 Skill-capability bottlenecks (availability mismatch)
Pinpoint with:
- **Queue time to start work by required skill**
- **Escalation frequency for each required skill**
- **Time in L2/L3 waiting for specialized agents**

Quantify impact:
- For each skill s:
  - Avg queue time (by tier) vs SLA thresholds
  - SLA breach share among cases requiring s
- If ôNetworking-Firewallö shows high queue times at L2 and escalations from L1 repeat, then specialists are overloaded or routing is wrong.

### 2.2 Delays caused by reassignments/escalations
Compute per case:
- **Delay per reassignment**:
  \[
  \text{delay\_reassign}=\sum_{i=1}^{n} (t(\text{Assign})_{i+1}-t(\text{Work End})_{i})
  \]
- **Reassignment count vs cycle time distribution**
- Compare:
  - cases with 0 reassignments
  - cases with 1 reassign
  - cases with 2 reassignments

Then measure:
- SLA breach probability by reassignment bucket

Typical finding: even when resolution time might be similar, **queue/wait time inflates** due to rework loops.

### 2.3 Incorrect initial assignments by dispatchers/L1
Measure:
- **Initial assignment skill mismatch rate**:
  - mismatch at first work start at L1
- **Mismatch  escalation rate correlation**
  - For cases where L1 mismatch occurred, what % escalated vs resolved at L1?
- **Late correct routing**:
  - time from ôTicket Createdö until first work start by a skill-matching agent

This quantifies how much SLA pain is caused by routing error, not workload.

### 2.4 Underperforming or overloaded agents/teams
Use performance stratification:

- **Overload indicators**
  - high case volume + long queue time before start
  - high utilization but low completion at expected tier
- **Underutilization**
  - low throughput while queues exist elsewhere (capacity imbalance)

Quantify:
- correlation between agent utilization and queue time experienced by the cases they receive
- compare SLA breach rates for tickets routed to ôhigh-utilization but historically effectiveö agents vs ôhigh-utilization but historically ineffectiveö

### 2.5 Correlation between assignment patterns and SLA breaches
Use breakdowns:
- SLA breach rate by (tier responsible, reassign count, skill match status at first assignment)
- ôAttributionö model (simple but effective):
  - compute fraction of SLA breaches in:
    - skill-mismatch cases
    - multi-reassignment cases
    - queue-heavy cases (top decile queue time)

Even without full causal modeling, this provides a strong **data-driven prioritization**.

## 3) Root Cause Analysis for Assignment Inefficiencies

### 3.1 Likely root causes (mapped to what mining will reveal)

1. **Routing rules ignore skills/workload**
   - Symptom: round-robin within tiers with high mismatch rate
   - Mining evidence:
     - role discovery shows repeated assignment patterns unrelated to required_skill
     - conformance checking shows frequent rule violations

2. **Inaccurate or incomplete skill profiles**
   - Symptom: ôAgent has skill Xö but historical performance contradicts it
   - Mining evidence:
     - mismatch rate is high even when profiles claim capability
     - success outcomes inconsistent with stated skills

3. **Poor ticket categorization / wrong required skill identification**
   - Symptom: required_skill field disagrees with category keywords and historical successful skill mappings
   - Mining evidence:
     - cases assigned based on stated required_skill often trigger reassignments
     - decision mining finds patterns where dispatcher selects a skill that differs from eventual skill needed

4. **Lack of real-time visibility into workload/availability**
   - Symptom: high queue time for certain skills; overloaded agents receive more cases regardless
   - Mining evidence:
     - queue time spikes align with scheduling/assignment peaks
     - assignment correlates with static round-robin rather than current availability

5. **Insufficient empowerment/training of L1**
   - Symptom: L1 escalates frequently even for tasks that could be solved by L1
   - Mining evidence:
     - for each skill/category, L1 resolution rate is lower than expected given success history
     - escalation after L1 attempt is common for skills where L1 has matching skills and historically resolves faster

---

### 3.2 Variant analysis and decision mining

#### Variant analysis
Define variants as key routing patterns, e.g.:
- Variant A: L1 resolved (no reassignment, no escalation)
- Variant B: L1 match but escalated
- Variant C: L1 mismatch then escalated
- Variant D: multiple reassignments inside L2

Compare outcomes:
- SLA breach rate
- cycle time distribution
- queue time accumulation points

This identifies which variant patterns correlate with SLA violations.

#### Decision mining (why did dispatchers assign that agent?)
If the log captures dispatcher decisions as explicit assignments and includes required_skill/category/priority, use decision mining to:
- infer a decision model for assignment choices:
  - input features = (priority, category, required_skill, time-of-day, current agent tier load)
  - output = chosen agent/tier
- then compare ôdecision model predictionö vs actual needed skills and observed outcomes

Key deliverables:
- dominant decision factors actually used
- feature gaps (e.g., required_skill rarely used by dispatcher decision despite being available)
- suboptimal decision rules (e.g., ôchoose agent with smallest queueö not present)

## 4) Developing Data-Driven Resource Assignment Strategies (3+ concrete strategies)

Below are **four** strategies (you can implement in phases). Each includes issue addressed, mining leverage, data requirements, and expected benefits.

---

### Strategy 1: Skill-based routing with weighted proficiency + confidence
**What issue it addresses**
- Wrong initial assignments and repeated reassignments due to **skill mismatch**
- Specialists consuming work that L1 could handle (if routing doesnĺt respect capability/expected success tier)

**How it uses mining insights**
- Use the mined **SkillTier success map** and **agent skill match success rates**
- Build a routing score:
  - candidate agents = those whose skills cover required_skill
  - score also considers historical success probability and proficiency (derived from outcomes)

Example scoring concept:
\[
\text{score(agent)} = w_1\cdot P(\text{resolve at tier}|\text{agent, skill}) + w_2\cdot \text{proficiency} - w_3\cdot \text{predicted queue delay}
\]

**Data required**
- Current skill matrix (and corrections from mining)
- Historical success rates per (agent, skill, tier)
- Real-time availability (estimated from active cases or recent work patterns)

**Expected benefits**
- Higher first-tier resolution and fewer escalations
- Lower reassignment count
- Reduced SLA breach rate, especially for P2/P3 where timing is tight

---

### Strategy 2: Workload-aware assignment (queue-time minimization) with constraints
**What issue it addresses**
- Uneven workload distribution and SLA breaches driven by long queues
- Overloaded specialists while others underutilized

**How it uses mining insights**
- Use mined queue-time distributions by skill and tier
- Detect capacity gaps: ôskill s demanded heavily but served slowlyö
- Apply a constraint optimization:
  - constraint = skill coverage and tier rules
  - objective = minimize expected waiting time / maximize throughput

**Data required**
- Near-real-time WIP/queue estimates per agent/tier
- Capacity model from log-derived productivity:
  - expected service rate per agent/tier/skill
- Operational constraints:
  - max parallelism per agent (if any)
  - compliance rules (e.g., certain categories must be handled at L2)

**Expected benefits**
- Smoother workload distribution
- Reduced waiting time (big driver of SLA breach)
- Better utilization of underused agents

---

### Strategy 3: Predictive assignment for required skill and complexity (pre-routing intelligence)
**What issue it addresses**
- Incorrect categorization / wrong ôrequired skillö leading to late reassignments
- Wrong tier escalation decisions

**How it uses mining insights**
- Mining can reveal systematic patterns in cases where required_skill is later corrected (as seen in reassign/escalation)
- Train a classifier (or rules model) to infer required_skill and expected complexity from:
  - category, subcategory, keywords in description (if available)
  - priority, channel, historical resolution path

**Data required**
- Ticket text fields / enriched attributes (best-effort if description exists)
- Historical mapping from initial categorization to ôeventually needed skillö
- Feedback loop: confirm or correct the predicted skill after resolution

**Expected benefits**
- More accurate routing at the earliest stage
- Reduced time to first match with a skill-correct agent
- Fewer ôescalate then reassign againö cycles

---

### Strategy 4 (optional but strong): Refined escalation criteria using learned ôL1 capabilityö thresholds
**What issue it addresses**
- Excessive escalations because L1 either cannot solve or escalates too early/too often
- Specialists performing work that could be handled by L1

**How it uses mining insights**
- For each skill/category/priority, compute:
  - L1 resolution probability after first attempt
  - distribution of time-to-resolution for successful L1 cases
- Create escalation thresholds:
  - escalate only when L1 confidence is below threshold or expected time exceeds SLA window
  - allow ôsecond attemptö at L1 for certain skills where mining shows high success

**Data required**
- L1 attempt evidence (steps taken, diagnostics completed)Śif not available, use proxy: time spent at L1 + final outcome
- Historical success rates by skill at L1

**Expected benefits**
- Improved L1 first resolution rate
- Reduced specialist workload on Tier-appropriate tasks
- Lower escalations without sacrificing SLA

## 5) Simulation, Implementation, and Monitoring

### 5.1 How to evaluate strategies before implementation (business process simulation)
Use a **discrete-event simulation (DES)** or queueing-based simulation driven by the mined model:

1. **Control-flow model**
   - from process mining: typical paths (L1L1 resolve, L1L2, L2L3, reassign loops)
2. **Time distributions**
   - service times per tier/skill (from Work Start/End intervals)
   - queue time models per tier/skill and/or per agent pool
3. **Resource availability**
   - agent capacity and utilization patterns from historical data
4. **Decision policy**
   - plug in candidate assignment strategies:
     - round-robin baseline
     - skill-based routing
     - workload-aware routing
     - predictive skill routing
5. **Arrival process**
   - ticket arrival rates by priority and channel (time-of-day patterns)
6. **Outputs**
   - SLA breach rate per priority
   - average cycle time / time-in-queue
   - reassignment and escalation counts
   - agent utilization and fairness indices

This enables ôwhat-ifö comparisons without disrupting operations.

**Why process-mining-informed simulation is powerful here:** your event log already contains the real-world distributions of delays and reassignments; youĺre not guessing system behavior from scratch.

Continuously track resource-related KPIs and process views.

- **SLA breach rate** (and time-to-breach distribution)

#### Process views

- Variant dashboards: distribution of routing variants and their SLA outcomes

#### Governance / continuous improvement
- Weekly recalibration:
  - update skill proficiency from outcomes
  - refine escalation thresholds
  - adjust model features for predictive routing
- Alerting:
  - spike in mismatch rate or reassignments triggers investigation
