# **Process Mining Approach to Optimize TechSolve Solutions' Resource Assignment**

## **1. Analyzing Resource Behavior and Assignment Patterns**

### **Key Metrics for Agent and Tier Performance**
To evaluate agent and tier performance, we will extract and analyze the following metrics from the event log:

#### **Agent-Level Metrics:**
- **Workload Distribution:**
  - Total tickets handled per agent (volume).
  - Time spent per ticket (processing time, idle time between activities).
  - Number of concurrent tickets in progress (multitasking index).
- **Activity Processing Times:**
  - Average time to complete an activity (e.g., "Work L1 Start" to "Work L1 End").
  - Time spent in queue before assignment.
  - Time spent in rework due to reassignments.
- **First-Call Resolution (FCR) Rate (L1-Specific):**
  - Percentage of P2/P3 tickets resolved by L1 without escalation.
  - Breakdown by ticket category (e.g., Software-App vs. Network).
- **Skill Utilization:**
  - Frequency of handling tickets requiring their documented skills.
  - Percentage of tickets where agents are over/under-skilled (e.g., L1 handling P2 Network tickets requiring "Networking-Firewall").
- **Escalation & Reassignment Patterns:**
  - Number of escalations per agent (internal vs. external).
  - Average time between escalation and reassignment.
  - Reasons for reassignment (e.g., skill mismatch, workload, incorrect initial assignment).

#### **Tier-Level Metrics:**
- **SLA Compliance by Tier:**
  - Percentage of P1/P2/P3 tickets resolved within SLA for each tier.
  - Average resolution time per priority level.
- **Escalation Flow Analysis:**
  - Percentage of tickets escalated from L1  L2  L3.
  - Average number of handovers per ticket.
- **Skill Coverage:**
  - Distribution of required skills across tickets vs. available skills in each tier.
  - Frequency of "skill gaps" (e.g., no L2 agent with "Security-IAM" available when needed).

### **Process Mining Techniques for Assignment Pattern Analysis**
We will apply the following process mining techniques to uncover actual assignment behaviors:

#### **A. Resource Interaction & Social Network Analysis (SNA)**
- **Handover of Work Analysis:**
  - Visualize which agents frequently hand off tickets to each other (e.g., L1  L2, L2  L3).
  - Identify "hotspots" where tickets get stuck (e.g., certain L2 agents are bottlenecks).
- **Social Network of Assignments:**
  - Construct a network where nodes = agents, edges = ticket handovers.
  - Use centrality measures (e.g., betweenness centrality) to identify key agents who frequently escalate or reassign tickets.
  - Detect informal "expert networks" (e.g., L1 agents who always escalate to the same L2 agent).

#### **B. Conformance Checking**
- Compare the **actual** assignment flow against the **intended** process model:
  - **Intended Logic:** Round-robin within tiers, manual escalation based on skills.
  - **Actual Logic:** Are tickets being assigned based on availability, skills, or other factors?
  - Identify deviations (e.g., L1 agents escalating P3 tickets unnecessarily, L2 agents handling P4 tickets).

#### **C. Role Discovery & Clustering**
- Use **automatic role discovery** (e.g., clustering agents based on behavior) to identify:
  - **Specialists:** Agents who handle specific categories (e.g., only "Security-IAM" tickets).
  - **Generalists:** Agents who handle a wide variety of tickets.
  - **Escalators:** Agents who frequently reassign tickets (potential skill gaps or workload issues).

#### **D. Performance Analysis with Process Maps**
- **Performance-Enhanced Process Maps:**
  - Overlay processing times, waiting times, and SLA breaches on the process flow.
  - Highlight where delays occur (e.g., long queues before L2 assignments).
- **Variant Analysis:**
  - Compare cases with **few reassignments** vs. **many reassignments** to identify patterns in successful vs. problematic assignments.

### **Comparison to Intended Assignment Logic**
The event log likely reveals discrepancies between the **documented process** and **actual behavior**:
| **Intended Logic** | **Actual Observed Behavior** |
|--------------------|-----------------------------|
| Round-robin assignment within tiers | Some agents are overloaded while others are underutilized; dispatchers manually override assignments. |
| Escalation based on skill requirements | L1 agents escalate P3 tickets to L2 even when they could resolve them; L2 agents reassign tickets to other L2 agents due to workload. |
| Skill-based routing | Agents are assigned tickets outside their skill set (e.g., L1 handling Network tickets). |
| First-come-first-served for L1 | Dispatchers prioritize certain tickets (e.g., P2 over P3) based on implicit rules. |

## **2. Identifying Resource-Related Bottlenecks and Issues**

### **Bottleneck Detection**
Using the mined process data, we will identify the following bottlenecks:

#### **A. Skill-Related Bottlenecks**
- **Problem:** Lack of agents with specific required skills (e.g., "Security-IAM") leads to delays.
- **Detection:**
  - **Skill Coverage Analysis:** For each ticket, check if the required skill exists in the assigned agent’s profile.
  - **Queue Time Analysis:** Compare average wait times for tickets requiring rare skills vs. common skills.
  - **Reassignment Reasons:** If tickets are frequently reassigned due to "skill mismatch," this indicates a gap.

#### **B. Workload Imbalance Bottlenecks**
- **Problem:** Some agents/teams are overloaded while others are underutilized.
- **Detection:**
  - **Workload Heatmap:** Visualize agent workload over time (e.g., tickets per hour).
  - **Queue Length Analysis:** Identify tiers/agents with consistently long queues.
  - **Correlation with SLA Breaches:** Do overloaded agents have higher SLA breach rates?

#### **C. Escalation & Reassignment Bottlenecks**
- **Problem:** Excessive escalations/reassignments increase resolution time.
- **Detection:**
  - **Reassignment Frequency:** Calculate average number of handovers per ticket.
  - **Time Spent in Reassignment:** Measure delay between escalation and new assignment.
  - **Root Cause Analysis:** Are reassignments due to:
    - **Skill mismatch?** (Agent lacks required skill)
    - **Workload?** (Agent is too busy)
    - **Incorrect initial assignment?** (L1 assigned a P2 ticket to an L1 agent without the right skills)

#### **D. SLA Breach Correlation**
- **Problem:** P2/P3 SLA breaches are increasing.
- **Detection:**
  - **SLA Violation Timeline:** Overlay SLA breaches with assignment events (e.g., "Did a reassignment cause the breach?").
  - **Priority-Based Analysis:** Compare SLA compliance for P1 vs. P2 vs. P3 tickets.
  - **Agent-Specific SLA Impact:** Do certain agents/tiers contribute disproportionately to SLA breaches?

### **Quantifying Impact of Issues**
| **Issue** | **Metric** | **Expected Impact** |
|-----------|------------|---------------------|
| Skill mismatch | % of tickets reassigned due to skill gap | +15-30% resolution time increase |
| Workload imbalance | Standard deviation of agent workload | Higher variance  more SLA breaches |
| Excessive escalations | Avg. handovers per ticket | +20-40% resolution time |
| Incorrect L1 assignments | % of P2 tickets escalated unnecessarily | +10-25% L2 workload |
| Bottleneck agents | Agents with >2x avg. queue time | Delays propagate to downstream tickets |

## **3. Root Cause Analysis for Assignment Inefficiencies**

### **Potential Root Causes**
Based on the analysis, we hypothesize the following root causes:

| **Root Cause** | **Evidence from Process Mining** | **Impact** |
|----------------|----------------------------------|------------|
| **1. Poor Skill Matching in Initial Assignment** | - High % of tickets reassigned due to "skill mismatch" <br> - L1 agents handling tickets outside their skill set | Delays, unnecessary escalations |
| **2. Workload-Agnostic Assignment Logic** | - Round-robin assignment ignores queue lengths <br> - Some agents have 3x more tickets than peers | Overloaded agents  SLA breaches |
| **3. Inaccurate Agent Skill Profiles** | - Agents assigned tickets they lack skills for <br> - Frequent reassignments to "correct" skill level | Wasted time, frustration |
| **4. Lack of Real-Time Visibility** | - Dispatchers assign tickets without knowing agent availability <br> - No dynamic rebalancing | Tickets sit in queues unnecessarily |
| **5. Weak L1 Empowerment & Training** | - High % of P3 tickets escalated (could be resolved at L1) <br> - L1 agents lack confidence in handling certain categories | Increased L2/L3 workload |
| **6. Inflexible Tier Structure** | - L2 agents handling P4 tickets (below their skill level) <br> - No cross-tier flexibility (e.g., L1 agents temporarily assisting L2) | Underutilization of skills |

### **Advanced Root Cause Techniques**
- **Variant Analysis:**
  - Compare cases with **0 reassignments** vs. **3 reassignments** to identify key differences (e.g., skill match, workload, ticket category).
- **Decision Mining:**
  - Analyze the **conditions** leading to escalations (e.g., "If ticket priority = P2 AND category = Network, then escalate 70% of the time").
- **Predictive Modeling:**
  - Train a model to predict **escalation likelihood** based on ticket attributes (priority, category, initial agent skills).

## **4. Data-Driven Resource Assignment Strategies**

### **Strategy 1: Skill-Based Routing with Proficiency Weighting**
**Problem Addressed:**
- Skill mismatches causing unnecessary escalations/reassignments.
- Specialists (e.g., "Security-IAM") are underutilized or overburdened.

**Implementation:**
1. **Skill Proficiency Matrix:**
   - Assign a **proficiency score** (1-5) to each agent for each skill (e.g., Agent B12: "App-CRM" = 4, "DB-SQL" = 3).
2. **Dynamic Assignment Algorithm:**
   - For a new ticket, find the **best-fit agent** based on:
     - **Required skill match** (must have at least proficiency = 3).
     - **Workload balance** (avoid overloading agents with high proficiency).
     - **Tier constraints** (e.g., P1 tickets must go to L2/L3).
3. **Fallback Mechanism:**
   - If no agent meets the skill requirement, escalate to the next tier.

**Data Required:**
- Agent skill proficiency scores (from training records, historical performance).
- Real-time workload (queue length, active tickets).
- Ticket skill requirements (from categorization).

**Expected Benefits:**
- **Reduced reassignments** by 40-60%.
- **Faster resolution** for specialized tickets.
- **Better utilization of experts** (no over/under-assignment).

---

### **Strategy 2: Workload-Aware Assignment with Predictive Queuing**
**Problem Addressed:**
- Workload imbalance causing bottlenecks.
- Tickets sitting in queues due to dispatcher ignorance of agent availability.

**Implementation:**
1. **Real-Time Workload Dashboard:**
   - Track **active tickets per agent**, **time to next availability**, and **historical processing speed**.
2. **Predictive Assignment:**
   - Use a **queueing model** to estimate:
     - **Expected wait time** for each agent.
     - **Probability of SLA breach** if assigned to a given agent.
3. **Dynamic Tier Balancing:**
   - If L2 is overloaded, allow **L1 agents with sufficient skills** to handle certain P2 tickets (temporarily).

**Data Required:**
- Real-time agent availability (from ticket system).
- Historical processing times per agent/tier.
- SLA targets per priority level.

**Expected Benefits:**
- **Reduced queue times** by 30-50%.
- **More even workload distribution**.
- **Lower SLA breach rates** for P2/P3 tickets.

---

### **Strategy 3: Predictive Escalation & L1 Empowerment**
**Problem Addressed:**
- Excessive escalations from L1 due to lack of confidence/training.
- P3 tickets being escalated unnecessarily.

**Implementation:**
1. **Escalation Prediction Model:**
   - Train a **classification model** (e.g., Random Forest, XGBoost) to predict:
     - **Will this L1 ticket need escalation?** (Features: ticket category, priority, agent ID, historical FCR rate).
2. **L1 Empowerment Rules:**
   - If predicted escalation probability < 20%, **allow L1 to resolve**.
   - If > 50%, **auto-escalate to L2**.
3. **Training & Knowledge Base:**
   - Provide L1 agents with **guided troubleshooting steps** for common P3 categories (e.g., "Software-App" issues).

**Data Required:**
- Historical L1 resolution outcomes (escalated vs. resolved).
- Ticket attributes (category, priority, keywords in description).
- Agent performance data (FCR rate per category).

**Expected Benefits:**
- **Reduced L2 workload** by 20-30%.
- **Faster resolution for P3 tickets** (no unnecessary escalations).
- **Improved L1 agent confidence & skills**.
