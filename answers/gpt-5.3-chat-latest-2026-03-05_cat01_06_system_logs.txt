Explanation of the Transformation Logic

Case Identification  
Cases were defined by the main work object visible in the window title (document, email, PDF, or spreadsheet). Each unique artifact represents a logical work item:
- DOC_Document1.docx  writing a document  
- DOC_Quarterly_Report.docx  editing a report  
- EMAIL_Annual_Meeting  handling an email conversation  
- PDF_Report_Draft.pdf  reviewing a PDF report  
- XLS_Budget_2024.xlsx  updating a budget spreadsheet  

This object-centric grouping is common in desktop process mining because user work is typically centered around a specific file or message.

Activity Naming  
Low-level system actions were abstracted into higher-level activities:

- FOCUS  Open Document / Open Spreadsheet / Resume Editing  
- TYPING  Edit Document / Edit Spreadsheet / Write Email Reply  
- SAVE  Save Document / Save Spreadsheet  
- CLICK (open/reply/send)  Open Email, Start Email Reply, Send Email  
- SCROLL  Read Email or Review PDF  
- HIGHLIGHT  Highlight Content  
- CLOSE  Close Document  

This abstraction reduces noise and produces activities suitable for process discovery.

Switch Handling  
SWITCH events were interpreted as context changes between cases. Instead of keeping them as raw events, they were converted into:
- Open/Resume activities for the destination case.

Coherent Narrative  
The resulting log describes a multitasking knowledge-work session:
1. The user edits Document1.docx.
2. They pause to respond to an email about an annual meeting.
3. They briefly review a PDF report.
4. They update a budget spreadsheet.
5. They return to finish Document1.docx.
6. Finally, they edit and finalize Quarterly_Report.docx.

This structure produces a clean event log suitable for tools like ProM, Disco, or Celonis while preserving the story of the user𠏋 work across multiple documents.
