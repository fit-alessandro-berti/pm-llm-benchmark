### **1. POWL Model with Potential Unfairness**

This model includes an XOR branch after the skill assessment, where one path applies a community-based advantage.

```python
import pm4py
from pm4py.objects.powl.obj import StrictPartialOrder, OperatorPOWL, Transition, SilentTransition
from pm4py.objects.process_tree.obj import Operator

# Define activities
receive_app = Transition(label="ReceiveApplication")
data_check = Transition(label="DataCompletenessCheck")
request_info = Transition(label="RequestMoreInfo")
skill_assess = Transition(label="SkillAssessment")
cultural_fit = Transition(label="CulturalFitCheck")
community_check = Transition(label="CommunityAffiliationCheck")
manager_review = Transition(label="ManagerialReview")
final_decision = Transition(label="FinalDecision")

# Loop for data completeness
loop = OperatorPOWL(operator=Operator.LOOP, children=[data_check, request_info])

# XOR branching after skill assessment: standard cultural fit vs. community-based check
xor_branch = OperatorPOWL(
    operator=Operator.XOR,
    children=[cultural_fit, community_check]
)

# Path after XOR: both paths lead to ManagerialReview then FinalDecision
path_after_xor = StrictPartialOrder(
    nodes=[manager_review, final_decision],
    order=[(manager_review, final_decision)]
)

# Full workflow structure
unfair_model = StrictPartialOrder(
    nodes=[receive_app, loop, skill_assess, xor_branch, path_after_xor],
    order=[
        (receive_app, loop),
        (loop, skill_assess),
        (skill_assess, xor_branch),
        (xor_branch, path_after_xor)
    ]
)
```

### **2. POWL Model Without Unfairness**

This model removes the community-based branch and ensures all applicants undergo the same cultural fit evaluation.

```python
import pm4py
from pm4py.objects.powl.obj import StrictPartialOrder, OperatorPOWL, Transition, SilentTransition
from pm4py.objects.process_tree.obj import Operator

# Define activities
receive_app = Transition(label="ReceiveApplication")
data_check = Transition(label="DataCompletenessCheck")
request_info = Transition(label="RequestMoreInfo")
skill_assess = Transition(label="SkillAssessment")
cultural_fit = Transition(label="CulturalFitCheck")
manager_review = Transition(label="ManagerialReview")
final_decision = Transition(label="FinalDecision")

# Loop for data completeness
loop = OperatorPOWL(operator=Operator.LOOP, children=[data_check, request_info])

# Single path: SkillAssessment -> CulturalFitCheck -> ManagerialReview -> FinalDecision
path_after_skill = StrictPartialOrder(
    nodes=[cultural_fit, manager_review, final_decision],
    order=[
        (cultural_fit, manager_review),
        (manager_review, final_decision)
    ]
)

# Full workflow structure
fair_model = StrictPartialOrder(
    nodes=[receive_app, loop, skill_assess, path_after_skill],
    order=[
        (receive_app, loop),
        (loop, skill_assess),
        (skill_assess, path_after_skill)
    ]
)
```

### **Key Differences**
- **Unfair Model**:  
  - After `SkillAssessment`, an XOR branch splits into `CulturalFitCheck` (standard) or `CommunityAffiliationCheck` (biased).  
  - The biased branch (`CommunityAffiliationCheck`) implicitly advantages certain applicants, violating fairness.

- **Fair Model**:  
  - No XOR branch exists. All applicants proceed through the same `CulturalFitCheck` after `SkillAssessment`.  
  - Eliminates preferential treatment based on community affiliations.

Both models preserve the loop for data completeness (`DataCompletenessCheck`  `RequestMoreInfo`) and the sequential flow from `ReceiveApplication` to `FinalDecision`. The unfairness is isolated to the XOR branch in the first model.
