To solve this problem, we need to perform a three-step process: 
1. Use window functions (specifically `LEAD`) to detect the specific sequence of events.
2. Filter for cases that match the sequence and the time-duration condition.
3. Use a `NOT IN` or `EXCEPT` clause to filter the original log based on the `case_id`s identified.

Here is the DuckDB SQL query:

```sql
WITH ActivitySequences AS (
    -- Identify the sequence and calculate the time difference
    SELECT 
        case_id,
        activity,
        timestamp,
        LEAD(activity, 1) OVER (PARTITION BY case_id ORDER BY timestamp) as next_act,
        LEAD(activity, 2) OVER (PARTITION BY case_id ORDER BY timestamp) as next_next_act,
        LEAD(timestamp, 1) OVER (PARTITION BY case_id ORDER BY timestamp) as next_ts,
        LEAD(timestamp, 2) OVER (PARTITION BY case_id ORDER BY timestamp) as next_next_ts
    FROM event_log
),
FlaggedCases AS (
    -- Identify cases that meet BOTH the sequence criteria and the timing criteria
    SELECT DISTINCT case_id
    FROM ActivitySequences
    WHERE activity = 'Approve Purchase Requisition'
      AND next_act = 'Create Order'
      AND next_next_act = 'Confirm Order'
      AND (next_next_ts - next_ts) > INTERVAL '5 days'
)
-- Return all events from the event log, excluding the flagged cases
SELECT *
FROM event_log
WHERE case_id NOT IN (SELECT case_id FROM FlaggedCases);
```

### Explanation:
1.  **`ActivitySequences` CTE**: We use the `LEAD` window function to look ahead at the next two events for every row. This allows us to see the sequence `[current, next, next-next]` in a single row.
2.  **`FlaggedCases` CTE**: We filter for the specific three-activity sequence. We also calculate the duration between the second event ('Create Order') and the third event ('Confirm Order') using `next_next_ts - next_ts` and check if it exceeds an interval of 5 days. We use `DISTINCT` because one case might contain the sequence multiple times, but we only need to flag the `case_id` once.
3.  **Final `SELECT`**: We perform a simple filter on the original `event_log` table using `NOT IN` to remove all events associated with any `case_id` that was flagged by our criteria.
