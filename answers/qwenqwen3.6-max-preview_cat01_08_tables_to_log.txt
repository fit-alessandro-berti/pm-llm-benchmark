###  Assumptions & Data Transformation Notes
1. **Case Identifier**: `order_id` is used as the `Case ID` for all events, linking the entire lifecycle from order creation to payment.
2. **Activity Granularity**:
   - **Item Picking**: Each line item generates a separate `Item Picked` event because they have distinct timestamps and pickers. This preserves warehouse operational granularity.
   - **Split Deliveries**: Order `1002` was shipped in two batches. Each batch generates its own `Order Dispatched` and `Order Arrived` events to accurately reflect parallel/split logistics.
   - **Multiple Invoices/Payments**: Order `1002` has two invoices and two corresponding payments. Each is logged as a separate event linked back to the same case.
3. **Resource Mapping**: All actor IDs (`sales_person_id`, `picker_id`, `carrier_id`, `billing_person_id`) were resolved to human-readable names using the `Resources` table. Format: `Name (ID)`.
4. **Payment Actor**: The `Payments` table does not specify an internal employee. The resource is labeled `Customer / Payment System` to indicate an external or automated trigger.
5. **Incomplete Process (Order 1004)**: This case only contains `Order Created` and `Item Picked` events. No delivery, invoice, or payment records exist in the source data. This likely represents a work-in-progress order, a cancellation, or missing ERP data. It is included as-is to maintain data fidelity.
6. **Chronological Sorting**: Events are strictly sorted by `Case ID` and then by `Timestamp` to ensure correct sequence for process mining algorithms (e.g., Alpha Miner, Heuristic Miner, or Inductive Miner).
7. **Attribute Enrichment**: Product descriptions/categories and key reference IDs (delivery, invoice, payment) are consolidated into the `Other Attributes` column to keep the log wide-table compliant while preserving contextual data for filtering or performance analysis.
