# Temporal Process Analysis Report: Insurance Claim Handling

## 1. Anomaly Identification

Based on the temporal profile model derived from the `claim_events` log, several statistical irregularities indicate potential process deviations or data integrity issues. The following activity pairs exhibit significant deviations from expected process behavior:

*   **Receive (R) to Approve (P):**
    *   **Metric:** Average time  25 hours; Standard Deviation  1 hour.
    *   **Anomaly:** The extremely low standard deviation relative to the mean suggests a rigid, automated mechanism rather than human-driven review. In a manual process, variance should be higher due to workload fluctuations.
*   **Approve (P) to Notify (N):**
    *   **Metric:** Average time  7 days; Standard Deviation  2 days.
    *   **Anomaly:** This indicates a significant bottleneck. While the average delay is high, the high standard deviation implies inconsistency珦ome claims are notified quickly, while others wait weeks. This suggests a queueing issue or dependency on external resources (e.g., mail services, customer availability).
*   **Assign (A) to Close (C):**
    *   **Metric:** Average time  2 hours; Standard Deviation  1 hour.
    *   **Anomaly:** Closure occurring within 2 hours of assignment is unusually fast for a standard claim lifecycle. This often indicates claims are being closed prematurely without undergoing Evaluation (E) or Approval (P), potentially signaling system errors or fraud.
*   **Evaluate (E) to Notify (N):**
    *   **Metric:** Average time  5 minutes; Standard Deviation  1 minute.
    *   **Anomaly:** This transition is too rapid for a human-in-the-loop process. It strongly suggests that the Approval (P) step is being bypassed or that the system automatically triggers notification immediately upon evaluation completion without a distinct approval event.

## 2. Hypothesis Generation

The following hypotheses are proposed to explain the observed anomalies:

### H1: Automated Batch Processing for Low-Risk Claims
*   **Target Anomaly:** R to P (Low Variance)
*   **Reasoning:** Claims under a certain monetary threshold or of specific types (e.g., "auto_insurance") may be routed through an automated approval engine. This would explain the rigid timing (batch jobs run at specific intervals) and the low standard deviation.

### H2: Notification Backlog and Manual Intervention
*   **Target Anomaly:** P to N (Long Delay, High Variance)
*   **Reasoning:** While approval is system-driven, the notification step may rely on manual dispatch or a legacy email system that fails intermittently. High variance suggests that when the system fails, manual intervention is required, causing unpredictable delays.

### H3: Premature Closure or "Ghost" Claims
*   **Target Anomaly:** A to C (Very Short Duration)
*   **Reasoning:** Adjusters may be closing claims immediately after assignment to meet performance metrics (KPIs) without performing actual work. Alternatively, there could be a system bug where the 'Close' event is triggered erroneously upon 'Assign' for duplicate or test claims.

### H4: Missing Approval Logs (Process Skipping)
*   **Target Anomaly:** E to N (Extremely Short Duration)
*   **Reasoning:** The 'Approve' activity may not be logged in the `claim_events` table for certain claim types, even though it logically occurs. If evaluation triggers an auto-approval, the system might skip writing the 'P' event, directly jumping to 'N'.

## 3. SQL Verification Approaches

To validate these hypotheses, the following PostgreSQL queries are proposed. These queries utilize self-joins on the `claim_events` table to calculate time deltas between specific activities per claim.

### Query 1: Identify Premature Closures (A to C < 4 Hours)
*Purpose:* Verify Hypothesis H3 by finding claims closed immediately after assignment.

```sql
WITH first_events AS (
    SELECT 
        claim_id, 
        activity, 
        MIN(timestamp) as first_ts
    FROM claim_events
    GROUP BY claim_id, activity
)
SELECT 
    fe_assign.claim_id,
    fe_assign.first_ts as assign_time,
    fe_close.first_ts as close_time,
    EXTRACT(EPOCH FROM (fe_close.first_ts - fe_assign.first_ts)) / 3600.0 as hours_elapsed
FROM first_events fe_assign
JOIN first_events fe_close ON fe_assign.claim_id = fe_close.claim_id
WHERE fe_assign.activity = 'A' 
  AND fe_close.activity = 'C'
  AND EXTRACT(EPOCH FROM (fe_close.first_ts - fe_assign.first_ts)) < 14400 -- 4 hours in seconds
ORDER BY hours_elapsed ASC;
```

### Query 2: Detect Notification Delays (P to N > 10 Days)
*Purpose:* Verify Hypothesis H2 by identifying claims stuck between Approval and Notification.

```sql
WITH first_events AS (
    SELECT 
        claim_id, 
        activity, 
        MIN(timestamp) as first_ts
    FROM claim_events
    GROUP BY claim_id, activity
)
SELECT 
    fe_approve.claim_id,
    c.claim_type,
    a.region,
    fe_approve.first_ts as approve_time,
    fe_notify.first_ts as notify_time,
    EXTRACT(EPOCH FROM (fe_notify.first_ts - fe_approve.first_ts)) / 86400.0 as days_elapsed
FROM first_events fe_approve
JOIN first_events fe_notify ON fe_approve.claim_id = fe_notify.claim_id
JOIN claims c ON fe_approve.claim_id = c.claim_id
JOIN adjusters a ON c.customer_id = a.adjuster_id -- Note: Adjuster ID linkage may need verification based on actual FK
WHERE fe_approve.activity = 'P' 
  AND fe_notify.activity = 'N'
  AND EXTRACT(EPOCH FROM (fe_notify.first_ts - fe_approve.first_ts)) > 864000 -- 10 days in seconds
ORDER BY days_elapsed DESC;
```

### Query 3: Detect Missing Approval Steps (E to N without P)
*Purpose:* Verify Hypothesis H4 by finding claims where Evaluation is followed by Notification without an intervening Approval event.

```sql
WITH claim_activities AS (
    SELECT 
        claim_id,
        activity,
        timestamp
    FROM claim_events
    WHERE activity IN ('E', 'P', 'N')
),
eval_claims AS (
    SELECT claim_id, MIN(timestamp) as eval_time
    FROM claim_activities
    WHERE activity = 'E'
    GROUP BY claim_id
),
notify_claims AS (
    SELECT claim_id, MIN(timestamp) as notify_time
    FROM claim_activities
    WHERE activity = 'N'
    GROUP BY claim_id
),
approve_claims AS (
    SELECT claim_id, MIN(timestamp) as approve_time
    FROM claim_activities
    WHERE activity = 'P'
    GROUP BY claim_id
)
SELECT 
    e.claim_id,
    e.eval_time,
    n.notify_time,
    p.approve_time
FROM eval_claims e
JOIN notify_claims n ON e.claim_id = n.claim_id
LEFT JOIN approve_claims p ON e.claim_id = p.claim_id
WHERE n.notify_time < e.eval_time -- Sanity check: Notify must be after Eval
  AND (p.approve_time IS NULL OR p.approve_time > n.notify_time); -- No P event between E and N
```

### Query 4: Analyze R to P Variance by Claim Type
*Purpose:* Verify Hypothesis H1 by checking if the low variance in R to P is consistent across all claim types or specific to "auto_insurance".

```sql
WITH r_p_deltas AS (
    SELECT 
        c.claim_type,
        EXTRACT(EPOCH FROM (
            (SELECT MIN(timestamp) FROM claim_events e2 WHERE e2.claim_id = e1.claim_id AND e2.activity = 'P') 
            - 
            (SELECT MIN(timestamp) FROM claim_events e3 WHERE e3.claim_id = e1.claim_id AND e3.activity = 'R')
        )) as delta_seconds
    FROM claim_events e1
    JOIN claims c ON e1.claim_id = c.claim_id
    WHERE e1.activity = 'R'
)
SELECT 
    claim_type,
    COUNT(*) as claim_count,
    AVG(delta_seconds) as avg_seconds,
    STDDEV(delta_seconds) as stdev_seconds
FROM r_p_deltas
GROUP BY claim_type
HAVING STDDEV(delta_seconds) < 7200; -- Filter for unusually low standard deviation (2 hours)
```
