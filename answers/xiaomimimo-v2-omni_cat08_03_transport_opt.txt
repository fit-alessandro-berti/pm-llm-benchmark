# Process Mining Consultation for Speedy Parcels: A Data-Driven Optimization Plan

## 1. Process Discovery and Conformance Checking

### Data Integration and Preprocessing
To create a cohesive event log, we would:
- **Define a Unified Case Notion**: Use `Case ID (Vehicle-Day)` as the primary case identifier, representing a single vehicle's operational day. This allows us to track the entire delivery process from start shift to end shift.
- **Create a Harmonized Event Log**: Merge events from all sources (GPS, scanner, dispatch, maintenance) into a single timeline per case. This involves:
  - **Aligning Timestamps**: Synchronizing clocks across systems and handling different time granularities.
  - **Mapping Activities**: Translating raw data points (e.g., "Low Speed Detected" from GPS) into standardized process activities (e.g., "Travel Delay").
  - **Linking Entities**: Connecting Package IDs to Vehicle-Day cases for delivery-specific analysis.
- **Handle Challenges**:
  - **Data Quality**: Missing GPS points or scanner errors would require interpolation or filtering.
  - **Activity Standardization**: Defining when a "travel" activity starts/stops (e.g., speed > 0 for >1 minute).
  - **Event Correlation**: Linking a "Delivery Failed" event to the specific package and route position.

### Process Discovery Visualization
Using algorithms like **Heuristics Miner** or **Inductive Miner**, we would discover the actual process flow. A typical discovered model for a vehicle day might show:
- **Main Flow**: Depart Depot  Travel  Arrive Customer  Service Customer  Depart Customer  (repeat)  Return to Depot.
- **Deviations**: Branches for:
  - **Unscheduled Stops** (e.g., maintenance, traffic jams).
  - **Delivery Failures** leading to re-routing or return to depot.
  - **Long Travel Delays** as distinct activities.
- The visualization would highlight frequent alternative paths and their frequencies.

### Conformance Checking Against Planned Routes
We would compare the discovered model with the dispatch system's planned sequence:
- **Deviation Types to Detect**:
  1. **Sequence Deviations**: Visited customers in a different order than planned.
  2. **Extra Activities**: Unscheduled stops (e.g., "Engine Warning").
  3. **Missing Activities**: Skipped planned deliveries.
  4. **Timing Deviations**: Significant delays between planned and actual arrival times at stops.
- **Technique**: Use **token-based replay** or **alignment-based conformance checking** to quantify how much the actual process deviates from the ideal planned model and pinpoint where deviations occur.

## 2. Performance Analysis and Bottleneck Identification

### Key Performance Indicators (KPIs) and Calculation
- **On-Time Delivery Rate**: `(Deliveries within time window / Total Deliveries) * 100`. Calculated by comparing `Arrive Customer` timestamp with planned time window from dispatch data.
- **Average Time per Delivery Stop**: Average of (`Depart Customer` - `Arrive Customer`) timestamps per package. Indicates service efficiency.
- **Travel Time vs. Service Time Ratio**: Total time with `Speed > 0` vs. total `Service Time` (at customers). High ratios may indicate poor routing or traffic issues.
- **Fuel Consumption per km/package**: Integrate GPS-derived distance (`Location` changes) with vehicle fuel data (if available) or proxy via idle/stop durations. Divide by number of deliveries.
- **Vehicle Utilization Rate**: `(Time from Depart Depot to Return to Depot - Unscheduled Stop Time) / Total Shift Time`.
- **Frequency/Duration of Traffic Delays**: Count and total duration of activities like "Low Speed Detected" or "Traffic Jam" per route/time of day.
- **Rate of Failed Deliveries**: `(Delivery Failed events / Total Delivery Attempts) * 100`.

### Bottleneck Identification Techniques
- **Process Mining Techniques**:
  - **Bottleneck Analysis**: Identify activities with high waiting times (e.g., long `Arrive Customer` to `Depart Customer` intervals, indicating parking/search issues).
  - **Variant Analysis**: Compare process variants (e.g., routes with high vs. low on-time rates) to see differing paths or timings.
  - **Dotted Chart Analysis**: Visualize event timestamps to identify patterns in delays (e.g., clustered delays at certain times/locations).
- **Bottleneck Localization**:
  - **Route-Specific**: Aggregate KPIs per `Route ID` (from dispatch) to identify underperforming routes.
  - **Time-Based**: Analyze KPIs by hour of day to find peak congestion periods.
  - **Driver-Based**: Compare KPIs across `Driver ID` to identify training needs or best practices.
  - **Activity-Based**: Identify activities with high variability in duration (e.g., "Find Parking" inferred from slow speed before arrival).
- **Quantifying Impact**: Use **performance animations** on the process model to show time spent in each activity and link bottlenecks directly to delays and costs.

## 3. Root Cause Analysis for Inefficiencies

Beyond surface-level bottlenecks, we would investigate root causes:

1. **Suboptimal Route Planning**:
   - **Analysis**: Compare planned routes (dispatch) with discovered actual routes. Use **variant analysis** to see if drivers consistently alter sequences in specific areas.
   - **Data Support**: If many drivers deviate in the same way, the static plan may ignore practical constraints (e.g., one-way streets, parking availability).

2. **Inaccurate Travel Time Estimations**:
   - **Analysis**: Compare planned travel times (dispatch) with actual travel times (GPS between stops). Use **statistical comparison** to identify systematic over/under-estimation per zone or time of day.

3. **Impact of Traffic Congestion**:
   - **Analysis**: Correlate "Low Speed Detected" events with external traffic data (if available) or time/day patterns. Use **process mining with time-based filtering** to see if delays occur consistently on specific road segments.

4. **High Variability in Service Time**:
   - **Analysis**: Perform **statistical process control** on "Service Time" (`Arrive` to `Depart Customer`). Drill down by location type (urban vs. suburban), package type, or driver.

5. **Vehicle Breakdowns/Maintenance**:
   - **Analysis**: Analyze the `Unscheduled Stop` activity. Correlate with vehicle age, mileage (from GPS distance), and maintenance history. Use **predictive process monitoring** to identify patterns preceding breakdowns.

6. **Driver Behavior/Skill**:
   - **Analysis**: **Cluster drivers** based on KPIs (e.g., service time, deviation rate). Examine the process flows of top vs. bottom performers to identify behavioral differences (e.g., break compliance, route adherence).

7. **Failed Delivery Attempts**:
   - **Analysis**: Isolate process variants ending in "Delivery Failed". Analyze patterns: time of day, location, package type, or preceding activities (e.g., long travel delays causing missed windows).

## 4. Data-Driven Optimization Strategies

### Strategy 1: Dynamic Re-Routing Engine
- **Targets**: Traffic delays, suboptimal static routes.
- **Root Cause**: Inability to adapt to real-time conditions.
- **Data Support**: Process mining reveals recurring delay patterns (time/location). Conformance checking shows systematic deviations from planned routes.
- **Implementation**: Integrate process mining insights (common delay hotspots) with real-time GPS/traffic data to trigger route re-optimization mid-shift. Use historical performance to weight travel time predictions.
- **Expected Impact**:  On-Time Delivery Rate,  Travel Time,  Fuel Consumption.

### Strategy 2: Predictive Maintenance Scheduling
- **Targets**: Unscheduled stops due to breakdowns.
- **Root Cause**: Reactive maintenance based on failure rather than usage patterns.
- **Data Support**: Process mining identifies clusters of `Unscheduled Stop` events correlated with vehicle usage (mileage, idle time) and age.
- **Implementation**: Develop a predictive model using historical maintenance logs and vehicle activity patterns. Schedule maintenance before predicted failure points.
- **Expected Impact**:  Vehicle Utilization Rate,  Maintenance Costs,  On-Time Delivery Rate.

### Strategy 3: Delivery Window Optimization and Customer Communication
- **Targets**: High rate of failed deliveries and resulting re-routing.
- **Root Cause**: Inaccurate customer availability windows and poor communication.
- **Data Support**: Process variants ending in "Delivery Failed" show patterns (e.g., specific neighborhoods, time windows).
- **Implementation**: Use process mining insights to refine suggested delivery time windows during order placement. Implement automated notifications (SMS) with live tracking links based on actual route progress.
- **Expected Impact**:  Failed Delivery Rate,  On-Time Delivery Rate,  Service Time per Stop.

## 5. Considering Operational Constraints and Continuous Monitoring

### Accounting for Operational Constraints
- **Driver Hours**: All route optimizations would include hard constraints on maximum shift length and mandatory break times, derived from `Start Shift`/`End Shift` and break activity logs.
- **Vehicle Capacity**: Package assignments per vehicle would respect weight/volume limits from dispatch data, ensuring proposed routes don't exceed capacity.
- **Customer Time Windows**: Dynamic routing algorithms would treat customer-requested time windows as soft or hard constraints, learning from historical adherence data.

### Continuous Monitoring Plan
A **process mining dashboard** would track:
1. **Real-Time KPIs**: On-Time Delivery Rate, Failed Delivery Rate, Average Service Time, Live Vehicle Locations vs. Planned.
2. **Process Model Monitor**: Daily refreshed discovered process model to detect emerging deviations or new patterns.
3. **Alert System**: Threshold-based alerts for:
   - Spike in "Delivery Failed" events for a specific driver/zone.
   - Significant deviation of actual travel time from predicted (indicating need to update routing model).
   - Increase in unscheduled stop frequency.
4. **Weekly Performance Reviews**: Comparative variant analysis to see if optimization strategies are shifting process behavior toward desired outcomes.
5. **Feedback Loop**: Insights from monitoring would feed back into the dispatch system parameters and driver training programs, creating a cycle of continuous improvement.

By implementing this structured, data-driven approach, Speedy Parcels can move from reactive problem-solving to proactive process optimization, directly addressing their punctuality and cost challenges.
