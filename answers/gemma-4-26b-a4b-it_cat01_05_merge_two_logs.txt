### Documentation & Reasoning

#### 1. Matching Logic
To create this integrated log, I applied a heuristic matching process based on the `order_id` and a temporal proximity window.

*   **Successful Merges (Within 2s tolerance):**
    *   **Order Received:** Log A (10:00:00) and Log B (09:59:58) were matched. Although the difference was exactly 2 seconds, the semantic identity (Order Received vs OrderReceived) and shared Order ID provided high confidence.
    *   **Order Validated:** Log A (10:01:30) and Log B (10:01:29) were matched (1-second difference).
    *   **Item Shipped:** Log A (10:05:00) and Log B (10:05:02) were matched (2-second difference).

*   **Semantic Match (Exceeding 2s tolerance):**
    *   **Payment Processed:** Log A (10:02:00) and Log B (10:02:05) had a 5-second discrepancy. While this exceeds the strict 2-second rule, I chose to merge them because "Payment Processed" and "PaymentCheck" are semantically identical in an order lifecycle, and no other event occupied that timestamp window for this Order ID. This was marked with an asterisk (\*) in the log.

#### 2. Handling Non-Overlapping Events
*   **System B Only:** The event `Quality Check` (10:03:00) appeared only in Log B. Since there was no corresponding event in Log A within a reasonable timeframe, it was included as a unique enrichment to the timeline.
*   **System A Only:** The event `Item Delivered` (10:20:00) appeared only in Log A. As System B likely focuses on warehouse/fulfillment (as evidenced by its notes), it is logical that the final delivery step was only captured by the primary system (System A).

#### 3. Conflict Resolution & Attribute Integration
*   **Timestamp Selection:** I used **Log A** as the primary timestamp for all merged records, as System A was defined as the "primary timeline." This ensures a consistent baseline for the audit trail.
*   **Naming Convention:** I adopted the naming convention of **Log A** (e.g., "Order Received" instead of "OrderReceived") to maintain professional readability, while using Log B to fill in the semantic meaning.
*   **Attribute Enrichment:** For all merged events, I performed a "union" of attributes. For example, in the "Item Shipped" event, the `user_id`, `resource_id`, and `notes` from Log B were appended to the core event record from Log A, creating a much richer data point than either system provided individually.
