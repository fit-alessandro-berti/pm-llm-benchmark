**DuckDB SQL solution**

Below is a singlestatement query that fulfills every requirement in the prompt.  
It works for any integer`K` (the number of “top” variants you want) – just replace the placeholder `:K` with the desired value.

```sql
WITH ordered_events AS (
    -- 1  Order events inside each case by timestamp
    SELECT
        case_id,
        activity,
        timestamp,
        ROW_NUMBER() OVER (PARTITION BY case_id ORDER BY timestamp) AS rn
    FROM event_log
),

variant_strings AS (
    -- 2  Build a canonical string that represents the whole activity sequence of a case
    SELECT
        case_id,
        STRING_AGG(activity, '->' ORDER BY rn) AS variant   -- e.g. "A->B->C"
    FROM ordered_events
    GROUP BY case_id
),

variant_counts AS (
    -- 3  Count how many cases share each distinct sequence (i.e., each process variant)
    SELECT
        variant,
        COUNT(*) AS case_cnt
    FROM variant_strings
    GROUP BY variant
),

top_k_variants AS (
    -- 4  Keep only the K most frequent variants.
    --      If several variants tie for the Kth place they are all kept (standard “TOPK” semantics).
    SELECT
        variant,
        case_cnt
    FROM variant_counts
    ORDER BY case_cnt DESC, variant   -- deterministic tiebreak on the string itself
    LIMIT :K                         -- <-- replace :K with the desired topK value (e.g. 5)
),

top_k_cases AS (
    -- 5  Derive the list of case_ids that belong to those topK variants
    SELECT vs.case_id
    FROM variant_strings vs
    JOIN top_k_variants tkv ON vs.variant = tkv.variant
)

-- 6  Return **all** original events whose case_id is in the topK set
SELECT e.*
FROM event_log e
WHERE e.case_id IN (SELECT case_id FROM top_k_cases);
```

### How the query satisfies each bullet point

| Requirement | Where it happens in the query |
|-------------|------------------------------|
| **1. Extract ordered activity sequence per `case_id`** | `ordered_events` CTE uses `ROW_NUMBER()` to order events by `timestamp`, then `variant_strings` aggregates them with `STRING_AGG(... ORDER BY rn)` producing a canonical string like `"Start->A->B->End"` for each case. |
| **2. Group cases by their complete activity sequence** | The aggregation in `variant_strings` (`GROUP BY case_id`) yields one row per case, and the subsequent `variant_counts` groups those strings together (`GROUP BY variant`). |
| **3. Count cases per variant & pick topK variants** | `variant_counts` computes `case_cnt`. `top_k_variants` orders by that count (descending) and limits to `:K` rows, giving the *topK* most frequent process variants. |
| **4. Return only events belonging to cases in those topK variants** | `top_k_cases` extracts the `case_id`s from `variant_strings` that match a variant in `top_k_variants`. The final `SELECT` filters `event_log` with `WHERE case_id IN (…)` so every event of those cases is returned. |

### Usage notes

* **Parameter substitution** – In DuckDB you can bind the placeholder `:K` via a prepared statement or replace it directly, e.g. `LIMIT 5`.
* **Variant representation** – The delimiter `'->'` is arbitrary; any noncolliding string (e.g., `'|'`) works as long as it does not appear inside an activity name.
* **Tie handling** – Because we order by both frequency (`case_cnt DESC`) and the variant string itself, ties for the Kth place are deterministic but all tied variants will be included (which is the usual “topK” interpretation). If you need a strict cutoff at exactly `K` rows, drop the secondary ordering clause.

This query therefore delivers the exact set of events that belong to the most common process variants, as requested.
