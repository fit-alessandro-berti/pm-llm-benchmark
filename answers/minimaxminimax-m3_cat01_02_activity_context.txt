## 3. Justification of Renaming Choices

### **Verify Documents  "Initial Document Verification" / "Re-verify Documents"**

**Contextual clues that drove the rename:**
- In **C001**, the first instance precedes a customer call asking for missing documents, and the *Additional Context* says *"Agent checks identity documents"* (a first pass). The second instance explicitly says *"Agent **re-checks** documents now that ID scan is provided"* — the word "re-checks" and the trigger event (newly uploaded ID) clearly mark a *follow-up* verification.
- In **C003**, the same pattern emerges: the first verification is *Physical verification of ID*, the second is described as *"**Re-verify** docs after account adjustment."*
- Both second occurrences are triggered by a process event (new document or system correction) that necessitates a new review, not a duplicate execution. Renaming clarifies that the activity at position 2 is **conditional** on a prior defect being resolved.

### **Call Customer  "Initial Customer Call" / "Follow-up Customer Call"**

**Contextual clues that drove the rename:**
- In **C002**, the two calls are clearly different in intent. The first call (*"Outbound call to clarify account discrepancy"*) is an **information-gathering** activity that happens **before** any correction is made. The second call (*"Follow-up call to confirm adjustments made"*) occurs **after** the `Adjust Account` activity and exists only to **verify completion**.
- The same "Initial Customer Call" label is consistently applied to the *single* call in **C001** (request missing docs) and **C003** (request the customer to return) — both are first-contact, information-action calls before any resolution step. This keeps naming consistent across cases.
- The "Follow-up" prefix makes the causal dependency on a prior action (`Adjust Account`) visible in the activity name itself, which is a strong signal in process analysis.

### Why Other Activities Were Left Unchanged
- **Initiate Request, Approve Request, Adjust Account, Submit Additional** all occur exactly once per case, so they do not suffer from the same ambiguity. Their meaning is stable across the log.
- These renamings preserve backward compatibility — the original activity names are still recognizable within the new labels — while removing the ambiguity that prevents correct process discovery or conformance checking.
