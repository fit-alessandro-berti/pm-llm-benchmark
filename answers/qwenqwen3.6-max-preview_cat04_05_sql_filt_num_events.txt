###  Reference Solutions (For Grading)

**1. Optimal DuckDB Approach (`QUALIFY`)**
```sql
SELECT *
FROM event_log
QUALIFY COUNT(*) OVER (PARTITION BY case_id) <= 6
ORDER BY case_id, timestamp;
```
*Why it's ideal:* `QUALIFY` is DuckDB's native clause for filtering window function results. It's concise, readable, and avoids subqueries or joins.

**2. Standard SQL (CTE + JOIN)**
```sql
WITH valid_cases AS (
  SELECT case_id
  FROM event_log
  GROUP BY case_id
  HAVING COUNT(*) <= 6
)
SELECT e.*
FROM event_log e
INNER JOIN valid_cases v ON e.case_id = v.case_id
ORDER BY e.case_id, e.timestamp;
```

**3. Window Function + Subquery**
```sql
SELECT * EXCLUDE (evt_cnt)
FROM (
  SELECT *, COUNT(*) OVER (PARTITION BY case_id) AS evt_cnt
  FROM event_log
)
WHERE evt_cnt <= 6
ORDER BY case_id, timestamp;
```

All three are correct. Benchmark grading should accept any logically equivalent query.

---
###  Benchmark Evaluation Tips
- **Result Comparison:** Compare outputs as unordered sets, or enforce `ORDER BY case_id, timestamp` in the test harness.
- **Common Failure Modes to Catch:**
  - Using `LIMIT 6` (limits rows globally, not per case)
  - Filtering with `WHERE COUNT(*) <= 6` (invalid without `GROUP BY`/window)
  - Dropping columns or returning only aggregated results
  - Off-by-one errors (`< 6` instead of `<= 6`)
- **Hidden Test Cases to Include:**
  - Cases with exactly 6 events (boundary condition)
  - Cases with 0 or 1 event
  - Multiple cases interleaved in timestamp order
  - `NULL` or missing `case_id` values (decide if they should be excluded or grouped)
