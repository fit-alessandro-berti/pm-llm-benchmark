### 2. Proposed Investigation Queries

#### Query A: Detecting "Out of Order" Activities
This query looks for cases where `Confirm Shipment` (expected step 4) happens before `Perform Credit Check` (expected step 2).

```sql
SELECT 
    l1.case_id, 
    l1.timestamp AS confirm_shipment_time, 
    l2.timestamp AS credit_check_time
FROM order_event_log l1
JOIN order_event_log l2 ON l1.case_id = l2.case_id
WHERE l1.activity = 'Confirm Shipment' 
  AND l2.activity = 'Perform Credit Check'
  AND l1.timestamp < l2.timestamp;
```

#### Query B: Identifying Cases Missing Critical Activities
This query identifies `case_id`s that lack the mandatory `Perform Credit Check` step.

```sql
SELECT o.case_id, o.order_value, o.order_type
FROM orders o
WHERE NOT EXISTS (
    SELECT 1 
    FROM order_event_log l 
    WHERE l.case_id = o.case_id 
      AND l.activity = 'Perform Credit Check'
);
```

#### Query C: Identifying Process Inefficiency (Bottlenecks)
This query calculates the time elapsed between `Register Order` and `Receive Payment` to determine which orders are taking significantly longer than others, potentially highlighting process drag.

```sql
SELECT 
    reg.case_id, 
    pay.timestamp - reg.timestamp AS cycle_time
FROM order_event_log reg
JOIN order_event_log pay ON reg.case_id = pay.case_id
WHERE reg.activity = 'Register Order'
  AND pay.activity = 'Receive Payment'
ORDER BY cycle_time DESC;
```

#### Query D: Correlating Anomalies with Departments/Roles
To see if specific departments or resources are consistently involved in "out-of-order" events, we can join the logs with the `resources` table.

```sql
SELECT 
    r.department, 
    l.activity, 
    count(*) as anomaly_count
FROM order_event_log l
JOIN resources r ON l.resource = r.resource_id
WHERE l.activity IN ('Confirm Shipment', 'Ship Goods') 
  AND l.timestamp < (
      SELECT min(timestamp) 
      FROM order_event_log l2 
      WHERE l2.case_id = l.case_id 
      AND l2.activity = 'Perform Credit Check'
  )
GROUP BY r.department, l.activity;
```
