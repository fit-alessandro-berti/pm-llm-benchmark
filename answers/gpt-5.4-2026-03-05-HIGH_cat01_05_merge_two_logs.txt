Below is a merged, enriched event log using:

- **order_id** as the primary entity key
- **System A timestamp as the primary timestamp** when an A event exists
- **System B attributes added** when a confident match exists
- **event-name normalization + timing proximity** for matching

I used a **practical tolerance of about 2 seconds for strong matches**, and allowed **one semantic/manual match** for the payment event because the process position and naming strongly suggest it is the same business occurrence.

---

## Matching approach

### Normalized event mapping
- **Order Received**  **OrderReceived**
- **Order Validated**  **OrderValidation**
- **Payment Processed**  **PaymentCheck**
- **Item Shipped**  **Shipping**
- **Quality Check** has no counterpart in A
- **Item Delivered** has no counterpart in B

### Timestamp policy
- For merged events, I kept:
  - **primary_timestamp** = System A timestamp
  - **timestamp_a** and **timestamp_b** = both original timestamps
- For events that exist in only one log, I used that system’s timestamp as the primary timestamp.

## Reasoning by event

### 1. Order Received / OrderReceived
- Same `order_id`
- Timestamps differ by **2 seconds**
- Event names are clearly equivalent
- Merged as one event

**Why merged:** This is a straightforward naming variation with a negligible time offset.

---

### 2. Order Validated / OrderValidation
- Same `order_id`
- Timestamps differ by **1 second**
- Names are clearly equivalent
- Merged as one event

**Why merged:** Strong match on both time and semantics.

---

### 3. Payment Processed / PaymentCheck
- Same `order_id`
- Timestamps differ by **5 seconds**
- Names are related but not identical
- Both occur at the same stage of the process, between validation and later fulfillment steps
- B’s note says **“Payment gateway delay”**, which helps explain the timing mismatch

**Why merged:**  
This is the least certain merge, but still reasonable. Although **5 seconds exceeds the preferred 2-second tolerance**, there is:
- no competing payment event nearby,
- strong business-process alignment,
- a note in B that explains a delay.

So I merged it with **medium confidence** rather than leaving two separate payment-stage events.

**Conservative alternative:** If you want a strict rule of `<= 2 seconds only`, then these should be left as separate events.

---

### 4. Quality Check
- Present only in **System B**
- No corresponding A event near that time
- Kept as a standalone B-origin event

**Why separate:** This appears to be an additional operational event captured only by System B.

---

### 5. Item Shipped / Shipping
- Same `order_id`
- Timestamps differ by **2 seconds**
- Names are equivalent
- Merged as one event

**Why merged:** Same fulfillment step with very small timestamp offset.

---

### 6. Item Delivered
- Present only in **System A**
- No corresponding event in B

**Why separate:** B appears to be missing this later delivery event, so it is retained as A-only.

---

## Conflict-resolution choices

1. **Primary timeline**
   - I used **System A timestamps as primary** for matched events because A is the stated primary timeline.

2. **Preserving both timestamps**
   - I kept both `timestamp_a` and `timestamp_b` to retain source fidelity.

3. **Attribute enrichment**
   - For matched events, I added B’s richer attributes:
     - `user_id`
     - `resource_id`
     - `notes`

4. **Unmatched events**
   - Events with no confident counterpart were kept as standalone records with their origin marked as:
     - **A only**
     - **B only**
