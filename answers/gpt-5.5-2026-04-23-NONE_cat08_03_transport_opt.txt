## 1. Process Discovery and Conformance Checking

### 1.1 Data preprocessing and integration

Speedy Parcels’ data comes from multiple operational systems, each describing part of the delivery process. The first step is to create a unified event log suitable for process mining.

A process-mining-ready event log should contain, at minimum:

| Field | Purpose |
|---|---|
| Case ID | Defines the process instance, e.g. vehicle-day, route-day, package journey |
| Activity | The event name, e.g. Depart Depot, Arrive Customer, Delivery Success |
| Timestamp | Enables sequencing and duration analysis |
| Resource | Driver, vehicle, depot, route |
| Object attributes | Package ID, stop number, customer time window, planned sequence |
| Location | GPS coordinates, depot/customer geolocation |
| Performance attributes | Speed, distance, idle time, fuel estimate, delay reason |
| Planned attributes | Planned arrival time, planned departure time, planned route sequence |
| Maintenance attributes | Repair start/end, vehicle downtime, fault type |

Because Speedy Parcels has multiple perspectives, I would create several linked event logs:

1. **Vehicle-day log**
   - Case ID: `Vehicle ID + Date`, e.g. `V12-20241205`
   - Captures the operational day from start shift to end shift.
   - Useful for analyzing route execution, travel, idle time, depot time, maintenance interruptions, overtime.

2. **Package-level log**
   - Case ID: `Package ID`
   - Captures the parcel journey from depot scan to successful delivery, failed attempt, or re-delivery.
   - Useful for customer punctuality, failed deliveries, and reattempt analysis.

3. **Stop-level log**
   - Case ID: `Vehicle-Date-Stop`
   - Captures arrival, service, departure for each customer stop.
   - Useful for dwell time, parking time, customer interaction, stop efficiency.

4. **Vehicle maintenance log**
   - Case ID: `Vehicle ID` or `Vehicle-maintenance episode`
   - Useful for linking breakdowns and repairs to operational disruption and cost.

These logs should be connected through common object identifiers: vehicle, driver, package, route, stop, and date. This enables object-centric process mining, where one can analyze interactions between vehicles, drivers, routes, packages, and maintenance events.

---

### 1.2 Event normalization and enrichment

Raw GPS data is high-frequency and noisy. It needs to be transformed into meaningful process events.

Examples:

| Raw data | Derived event |
|---|---|
| Ignition on at depot | Vehicle Ready / Start Vehicle |
| GPS leaves depot geofence | Depart Depot |
| GPS enters customer geofence | Arrive Customer |
| Speed = 0 near customer for several minutes | Stop / Dwell |
| Scanner confirms package | Delivery Success |
| Scanner failed attempt | Delivery Failed |
| GPS idle away from known stop | Unplanned Stop |
| Low speed over defined duration | Traffic Delay |
| GPS enters depot geofence | Arrive Depot |
| Maintenance log overlaps shift | Vehicle Unavailable / Repair |

Additional enrichment should include:

- Planned route sequence from dispatch.
- Planned arrival and departure times per stop.
- Customer-requested time windows.
- Vehicle capacity and actual load.
- Driver assignment.
- Vehicle type, age, odometer, service history.
- Road segment or traffic zone.
- Weather and traffic data, if available.
- Fuel consumption estimates from telematics or modeled from speed, idling, distance, load, and vehicle type.

---

### 1.3 Key data integration challenges

Several challenges are likely.

#### Timestamp synchronization

Different systems may use different clocks, time zones, or timestamp precision. Scanner timestamps may lag real arrivals if drivers scan after parking or after walking to the customer.

Mitigation:

- Normalize all timestamps to a common time zone.
- Check clock drift between systems.
- Use GPS geofence arrival as physical arrival and scanner events as service milestones.

#### Event granularity differences

GPS provides continuous tracking; scanners provide sparse business events; dispatch provides planned events; maintenance logs may be manually entered.

Mitigation:

- Convert GPS streams into business-level events.
- Keep both raw and derived timestamps when possible.
- Define clear rules for derived events such as traffic delay, idle stop, and unplanned detour.

#### Missing or inconsistent identifiers

A package may be scanned without a route ID, or a vehicle may be reassigned during the day.

Mitigation:

- Use composite keys: vehicle, driver, date, location, time window.
- Reconstruct missing links through temporal and spatial proximity.
- Flag uncertain matches with a confidence score.

#### Location accuracy

Urban GPS can be noisy due to tall buildings, tunnels, parking structures, or signal loss.

Mitigation:

- Use map matching.
- Use customer/depot geofences.
- Apply smoothing and minimum-duration rules before creating stop events.

#### Ambiguous activity semantics

A vehicle stopped for 10 minutes may be delivering, waiting for parking, taking a break, stuck at a gate, or experiencing a mechanical issue.

Mitigation:

- Combine GPS, scanner, driver app status, maintenance logs, and location context.
- Classify stops into categories: planned customer stop, depot stop, break, traffic, maintenance, unplanned stop.

---

### 1.4 Process discovery

After creating the event log, process discovery algorithms can reveal the actual execution patterns.

Useful algorithms include:

- **Inductive Miner** for structured, readable process models.
- **Heuristics Miner** for noisy logistics data, emphasizing frequent behavior.
- **Fuzzy Miner** for highly variable processes, simplifying less frequent paths.
- **Object-centric process mining** for analyzing interactions between packages, vehicles, drivers, and routes.
- **Directly-Follows Graphs** for quick operational visualization.

A simplified discovered process for a vehicle-day might look like:

1. Start Shift  
2. Route Assigned  
3. Load / Scan at Depot  
4. Depart Depot  
5. Travel to Stop  
6. Arrive Customer  
7. Delivery Success or Delivery Failed  
8. Depart Customer  
9. Repeat stops  
10. Optional events:
   - Traffic Delay
   - Unplanned Stop
   - Refueling
   - Break
   - Maintenance Stop
   - Return to Depot for reload
   - Customer Not Available
11. Arrive Depot  
12. End Shift  

The discovered model would show the actual process, including loops, exceptions, and rework. For example:

- Repeated failed delivery attempts.
- Mid-route depot returns.
- Large numbers of unplanned stops.
- Long waiting or idling before customer arrival.
- Maintenance events during planned delivery windows.
- Deliveries performed out of sequence.

The process model should be analyzed at different abstraction levels:

#### High-level route process

Shows the overall operational day.

Example:

`Start Shift  Route Assigned  Depart Depot  Delivery Loop  Arrive Depot  End Shift`

#### Stop-level process

Shows service behavior at each customer.

Example:

`Arrive Customer  Search/Park/Idle  Scan Package  Delivery Success/Failed  Depart Customer`

#### Package-level process

Shows customer experience and rework.

Example:

`Depot Scan  Out for Delivery  Delivery Failed  Returned to Depot  Reassigned  Delivery Success`

### 1.5 Conformance checking against planned routes

Conformance checking compares what was planned with what actually happened.

The dispatch system provides the reference model:

- Planned stop sequence.
- Planned departure time.
- Planned arrival time per stop.
- Customer time windows.
- Planned vehicle and driver.
- Planned capacity usage.
- Planned route distance and duration.

The event log provides the actual execution.

Conformance checking would identify deviations such as:

#### Sequence deviations

Examples:

- Stop 7 delivered before Stop 3.
- Planned customer skipped.
- Extra unplanned stop inserted.
- Route reversed or partially reordered.

These deviations may be beneficial or harmful. For example, a driver may reorder stops to avoid congestion, but this may cause time-window violations.

#### Timing deviations

Examples:

- Depot departure 25 minutes later than planned.
- Arrival at customer outside requested time window.
- Travel time between Stop 4 and Stop 5 exceeded plan by 40%.
- Service time at Stop 9 was 15 minutes instead of planned 4 minutes.
- Return to depot after planned shift end.

#### Location deviations

Examples:

- Vehicle took a route significantly longer than planned.
- Vehicle entered known congestion zone despite alternative route.
- Driver stopped far from customer location, indicating parking difficulty.

#### Resource deviations

Examples:

- Different driver or vehicle used from the planned assignment.
- Vehicle with lower capacity assigned to a high-volume route.
- Older vehicle assigned to a long-distance route with higher breakdown risk.

#### Activity deviations

Examples:

- Maintenance stop occurred during delivery route.
- Customer stop recorded in GPS but no scanner event occurred.
- Delivery marked successful without physical proximity to customer geofence.
- Long idle event with no corresponding planned stop.

Conformance metrics would include:

- Fitness: How much of the planned route behavior is reflected in actual execution.
- Precision: Whether the actual process contains many unplanned behaviors.
- Generalization: Whether deviations are rare exceptions or common patterns.
- Time conformance: Difference between planned and actual timestamps.
- Sequence conformance: Difference between planned and actual stop order.
- Distance conformance: Difference between planned and actual kilometers.

## 2. Performance Analysis and Bottleneck Identification

### 2.1 Key performance indicators

Speedy Parcels should track KPIs aligned with punctuality, cost, productivity, and reliability.

---

### On-Time Delivery Rate

Measures the percentage of deliveries completed within the promised customer time window.

Formula:

`On-Time Delivery Rate = Deliveries completed within time window / Total attempted deliveries`

From the event log:

- Use `Delivery Success` timestamp.
- Compare with customer-requested time window from dispatch.
- Analyze by route, driver, vehicle, zone, customer type, day of week.

Useful breakdowns:

- Early deliveries.
- Late deliveries.
- Missed time-window deliveries.
- Failed deliveries within time window.
- Failed deliveries outside time window.

---

### Average Delay per Delivery

Measures how late deliveries are when they miss the planned or promised time.

Formula:

`Average Delay = Average(Actual delivery time - promised latest delivery time) for late deliveries`

This helps distinguish routes that are slightly late from routes that are severely late.

---

### Average Time per Delivery Stop

Measures productivity at the customer stop.

Formula:

`Average Stop Time = Depart Customer timestamp - Arrive Customer timestamp`

Can be broken down into:

- Parking/search time if GPS arrival differs from scanner arrival.
- Customer interaction time.
- Failed-delivery handling time.
- Building access time.

---

### Travel Time vs. Service Time Ratio

Distinguishes whether inefficiency comes from driving or at-stop activity.

Formula:

`Travel-to-Service Ratio = Total travel time / Total service time`

Where:

- Travel time = time between departure from one stop and arrival at next stop.
- Service time = time between customer arrival and departure.

Interpretation:

- High travel ratio: routes are geographically inefficient or traffic-heavy.
- High service ratio: customer stops are slow, parking is difficult, or delivery procedure is inefficient.

---

### Fuel Consumption per Kilometer

If fuel data is available:

`Fuel per km = Total fuel consumed / Total distance traveled`

If not directly available, estimate using:

- Distance.
- Speed profile.
- Idle time.
- Stop-start frequency.
- Vehicle type.
- Load.
- Traffic conditions.

This KPI identifies inefficient vehicles, aggressive driving, excessive idling, or congested routes.

---

### Fuel Consumption per Package

Formula:

`Fuel per package = Total fuel consumed / Packages successfully delivered`

This is highly relevant to last-mile profitability.

A route with low fuel per km may still be inefficient if delivery density is low.

---

### Vehicle Utilization Rate

Can be defined in several ways.

Capacity utilization:

`Capacity Utilization = Loaded packages or volume / Vehicle capacity`

Time utilization:

`Time Utilization = Active delivery time / Available shift time`

Distance utilization:

`Productive distance = Distance between planned delivery stops / Total distance`

Low utilization may indicate poor territory design, imbalanced routes, or excessive deadhead travel.

---

### Driver Productivity

Examples:

- Deliveries per hour.
- Successful deliveries per shift.
- Average service time per stop.
- Average delay per route.
- Overtime frequency.
- Failed delivery handling compliance.

This must be adjusted for route difficulty, package density, traffic, customer type, and vehicle type to avoid unfair comparisons.

---

### Failed Delivery Rate

Formula:

`Failed Delivery Rate = Failed delivery attempts / Total delivery attempts`

Subcategories:

- Customer not home.
- Wrong address.
- Access issue.
- Business closed.
- Driver unable to locate.
- Package not loaded.
- Time-window missed.

Failed deliveries increase cost because they create rework and re-delivery loops.

---

### Re-delivery Cost Rate

Formula:

`Re-delivery Rate = Packages requiring second or later attempt / Total packages`

Cost impact:

`Re-delivery Cost = Additional travel time + service time + fuel + capacity consumed + customer dissatisfaction`

---

### Depot Departure Delay

Formula:

`Depot Departure Delay = Actual depart depot time - planned depart depot time`

Late starts often cascade into late deliveries.

---

### Traffic Delay Duration and Frequency

Using GPS speed and map data:

- Identify low-speed segments below expected speed.
- Compare actual travel time to historical free-flow travel time.
- Calculate traffic delay per road segment, route, time of day.

Formula:

`Traffic Delay = Actual segment travel time - expected segment travel time`

---

### Unplanned Stop Rate

Formula:

`Unplanned Stop Rate = Number of stops not in planned route / Route`

Also measure duration and location.

Potential reasons:

- Driver breaks.
- Parking search.
- Refueling.
- Vehicle issue.
- Personal stops.
- Waiting.
- Loading error.
- Traffic blockage.

---

### Maintenance Disruption Rate

Formula:

`Maintenance disruption = Route time lost due to breakdowns or repairs / Total route time`

Also track:

- Number of breakdowns per 1,000 km.
- Unscheduled maintenance per vehicle.
- Deliveries affected by breakdown.
- Delay minutes caused by vehicle issue.

---

### Overtime Rate

Formula:

`Overtime Rate = Shifts exceeding planned working hours / Total shifts`

Also:

`Average overtime minutes per route`

This directly affects labor cost and may indicate unrealistic route plans.

---

## 2.2 Bottleneck identification using process mining

Process mining can identify bottlenecks through time, frequency, and variant analysis.

### Performance-annotated process maps

A discovered process map can be overlaid with:

- Average duration per activity.
- Waiting time between activities.
- Frequency of paths.
- Delay probability.
- Cost estimate.
- Rework loops.

For example, the edge:

`Depart Customer Stop 4  Arrive Customer Stop 5`

may show:

- Average planned travel time: 12 minutes.
- Actual average travel time: 26 minutes.
- High frequency of traffic delay events between 08:00 and 09:30.
- High fuel consumption due to stop-start driving.

This indicates a travel bottleneck.

---

### Variant analysis

Each vehicle-day route can be treated as a process variant.

Examples of variants:

- Standard route completed as planned.
- Route with early depot departure but many traffic delays.
- Route with multiple failed deliveries.
- Route with mid-shift maintenance stop.
- Route with frequent sequence deviations.
- Route requiring overtime.
- Route with return-to-depot reload.

By comparing high-performing and low-performing variants, Speedy Parcels can identify patterns associated with poor outcomes.

For example:

| Variant feature | High-performing routes | Low-performing routes |
|---|---:|---:|
| Depot departure delay | 4 min | 28 min |
| Failed deliveries | 2% | 11% |
| Average stop time | 5 min | 9 min |
| Traffic delay minutes | 18 min | 65 min |
| Sequence deviations | 3% | 27% |
| Overtime | 5% of routes | 48% of routes |

---

### Bottleneck categories

#### Route-specific bottlenecks

Some routes may consistently experience late deliveries or overtime.

Possible causes:

- Too many stops assigned.
- Poor geographic clustering.
- Insufficient time buffers.
- High-rise or gated buildings.
- Dense traffic corridors.
- Low parking availability.

Analysis:

- Compare planned vs actual route duration.
- Analyze average delay by route ID and geographic zone.
- Identify repeatedly late stop positions.

---

#### Time-of-day bottlenecks

Certain periods may cause recurring delays.

Examples:

- Morning congestion after depot departure.
- School-zone congestion around 08:00.
- Lunch-hour delivery access delays.
- Afternoon city-center restrictions.

Analysis:

- Heatmap of traffic delay by time and location.
- Travel-time distributions by hour.
- Compare routes leaving at 07:30 vs 08:00.

---

#### Driver-related bottlenecks

Some drivers may have longer stop times, more sequence deviations, or higher idling.

However, driver comparison must control for route complexity.

Analysis:

- Normalize by route density, traffic exposure, package count, and customer type.
- Compare drivers on the same route or zone.
- Identify best practices from consistently high performers.

---

#### Vehicle-related bottlenecks

Some vehicles may have higher fuel consumption, more idling, or more breakdowns.

Analysis:

- Fuel per km by vehicle.
- Unscheduled maintenance frequency.
- Delay minutes caused by vehicle issues.
- Compare vehicle age, mileage, model, load, and route type.

---

#### Activity-specific bottlenecks

Examples:

- Long depot loading time.
- Long parking/search time before scanner arrival.
- Long customer interaction.
- High failed-delivery handling time.
- Long return-to-depot unload process.

Analysis:

- Activity duration distributions.
- Waiting-time analysis between activities.
- Outlier detection at specific stops or customer types.

---

### Quantifying bottleneck impact

Each bottleneck should be translated into business impact.

Examples:

1. **Traffic hotspot impact**
   - Average delay: 12 minutes per route.
   - Affected routes: 40 per week.
   - Weekly lost time: 480 minutes.
   - Cost: driver wages + fuel + late delivery penalties.

2. **Failed delivery impact**
   - Failed delivery rate: 8%.
   - Average reattempt cost: 18 minutes and 4 km.
   - Monthly rework: 2,000 attempts.
   - Monthly cost: 600 labor hours plus fuel and capacity loss.

3. **Depot departure delay impact**
   - Average late departure: 20 minutes.
   - Correlation with late deliveries: routes leaving >15 minutes late have 2.5 times higher late-delivery rate.
   - Potential reduction: improve staging and loading.

4. **Maintenance impact**
   - Vehicles older than five years cause 70% of breakdown delay.
   - Average breakdown disruption: 95 minutes.
   - Late deliveries caused per breakdown: 12 packages.

## 3. Root Cause Analysis for Inefficiencies

Identifying bottlenecks is not enough. Speedy Parcels must understand why they occur.

---

### 3.1 Suboptimal route planning

#### Potential issue

Routes may be planned statically using average distances rather than actual historical travel times, parking conditions, customer service time, or time-window constraints.

Symptoms in the event log:

- Frequent sequence deviations by experienced drivers.
- Consistent lateness at certain stop positions.
- Planned distance much shorter than actual distance.
- Planned route crosses congestion zones at peak hours.
- Some routes regularly require overtime while others finish early.

Process mining validation:

- Compare planned sequence against actual sequence.
- Identify drivers who deviate but achieve better punctuality.
- Analyze planned vs actual travel time by segment.
- Examine whether high-performing variants use different stop ordering.
- Use conformance checking to distinguish harmful deviations from beneficial adaptations.

Example insight:

Drivers on Route R14 frequently deliver Stops 8–12 before Stops 4–7. These deviations reduce average route duration by 22 minutes because they avoid the city-center corridor after 08:30.

---

### 3.2 Inaccurate travel time estimations

#### Potential issue

Dispatch may underestimate travel time, especially during peak periods or in dense urban areas.

Symptoms:

- Actual travel time consistently exceeds planned travel time on specific legs.
- Delays occur even when drivers follow the planned sequence.
- Routes are punctual on low-traffic days but late on weekdays.
- Late deliveries cluster after known congestion segments.

Process mining validation:

- Compare actual vs planned travel time for each route segment.
- Build time-of-day travel-time profiles.
- Correlate low-speed GPS events with late deliveries.
- Analyze segment-level delay distributions, not just averages.

Example insight:

The segment between Zone A and Zone B is planned for 10 minutes but takes a median of 21 minutes between 08:00 and 09:00.

---

### 3.3 Traffic congestion patterns

#### Potential issue

Congestion may be predictable and recurring, but not incorporated into route planning.

Symptoms:

- Low-speed detected events at recurring locations.
- High idle or stop-start frequency.
- Higher fuel per km in specific corridors.
- Late delivery waves after traffic hotspots.

Process mining validation:

- Create geospatial heatmaps of traffic delay.
- Compare route variants that pass through hotspots versus those that avoid them.
- Analyze travel time by road segment, day of week, and hour.
- Connect traffic delay events to downstream late deliveries.

Example insight:

Routes departing at 07:45 hit a school-zone bottleneck at 08:10. Routes departing 15 minutes earlier avoid it and have 12% higher on-time delivery.

---

### 3.4 High variability in service time

#### Potential issue

The company may assume a uniform stop time, such as five minutes per delivery, but actual service time varies significantly.

Service time may depend on:

- Apartment building vs single-family home.
- Business vs residential customer.
- Signature required.
- Heavy package.
- Parking difficulty.
- Elevator access.
- Security gate.
- Customer availability.
- Driver familiarity with location.

Symptoms:

- Long dwell times at specific addresses or building types.
- High variance in stop duration.
- Routes with many apartment/business stops run late despite normal travel times.
- Scanner event delay after GPS arrival, indicating parking or access issues.

Process mining validation:

- Analyze dwell time distributions by customer type, address, building type, parcel size, delivery method.
- Compare GPS geofence arrival with scanner `Arrive Customer`.
- Identify stops where dwell time is consistently above expected.
- Use clustering to classify stop difficulty.

Example insight:

Apartment deliveries take a median of 8.5 minutes compared with 3.2 minutes for single-family homes. Routes with more than 40% apartment stops exceed planned duration by an average of 35 minutes.

---

### 3.5 Vehicle breakdowns and maintenance issues

#### Potential issue

Unscheduled repairs or vehicle health issues are disrupting routes and increasing costs.

Symptoms:

- Unscheduled stops linked to engine warnings.
- Maintenance logs overlapping delivery shifts.
- Specific vehicles repeatedly associated with delays.
- Older vehicles have higher fuel consumption and idle time.
- Vehicles returning early or requiring rescue transfers.

Process mining validation:

- Link maintenance events to route outcomes.
- Compare delay minutes and failed deliveries for routes with and without vehicle issues.
- Analyze breakdown frequency by vehicle age, mileage, route distance, load, and prior maintenance.
- Detect early warning patterns: increased idle, speed anomalies, repeated engine warnings.

Example insight:

Vans with more than 180,000 km have 3.4 times more unscheduled stops and 18% higher fuel per km.

---

### 3.6 Driver behavior or skill differences

#### Potential issue

Drivers may differ in scanning discipline, route adherence, parking strategy, idling behavior, speed management, or customer handling.

Symptoms:

- Some drivers have consistently long service times.
- Some drivers have high idling and fuel consumption.
- Some drivers frequently deviate from planned sequence.
- Some drivers have higher failed delivery rates.
- Some drivers consistently complete difficult routes efficiently.

Process mining validation:

- Compare drivers only after controlling for route difficulty.
- Analyze same route with different drivers.
- Identify process variants associated with high performance.
- Compare idling time, service time, scan timing, sequence adherence, and delivery success rate.
- Use social/resource mining to understand driver-route specialization.

Example insight:

Drivers familiar with downtown routes complete apartment deliveries 20% faster and have fewer failed deliveries, likely due to knowledge of parking and building access.

---

### 3.7 Failed delivery attempts and rework

#### Potential issue

Failed deliveries create costly rework and reduce customer satisfaction.

Root causes may include:

- Customer not home.
- Poorly chosen delivery time.
- Inaccurate address.
- Access restrictions.
- Business closed.
- Driver missed time window.
- Lack of customer notification.
- No safe-place or locker alternative.

Symptoms:

- Repeated failed attempts for the same customers or areas.
- Failed deliveries concentrated in residential areas during work hours.
- Business deliveries attempted after closing.
- Failed deliveries create return-to-depot loops.
- Reattempts consume capacity on future routes.

Process mining validation:

- Package-level process mining to identify re-delivery loops.
- Analyze failed delivery reasons by customer type, area, time of day.
- Compare failure rate against customer notifications.
- Measure additional cost of reattempts.
- Identify customers or addresses with repeated failures.

Example insight:

Residential deliveries between 10:00 and 14:00 have a failed delivery rate of 13%, while evening deliveries in the same zones have a rate of 4%.

## 4. Data-Driven Optimization Strategies

Below are concrete strategies based on likely process mining insights.

---

### Strategy 1: Dynamic routing using real-time and historical process insights

#### Inefficiency targeted

- Late deliveries caused by traffic congestion.
- Static routes that ignore time-of-day travel variation.
- Excessive fuel consumption from stop-start traffic.
- Drivers manually deviating from plans without system support.

#### Root cause addressed

Planned routes are likely based on static travel estimates and do not sufficiently account for real-world congestion, parking difficulty, and observed service times.

#### Process mining support

Process mining can reveal:

- Route segments where actual travel time consistently exceeds planned travel time.
- Time-of-day congestion patterns.
- Driver deviations that improve punctuality.
- Stop sequences associated with better on-time performance.
- Traffic hotspots causing downstream delays.

#### Recommended action

Implement a dynamic route optimization engine that:

- Uses historical travel-time distributions from GPS logs.
- Incorporates real-time traffic feeds.
- Adjusts sequence during the day when delays occur.
- Protects customer time windows.
- Factors in service-time predictions per stop type.
- Avoids known high-delay road segments during peak hours.
- Sends updated instructions to drivers via handheld devices.

The system should not blindly optimize for shortest distance. It should optimize for:

- On-time probability.
- Expected route duration.
- Fuel consumption.
- Driver working-hour compliance.
- Vehicle capacity.
- Customer priority.
- Re-delivery risk.

#### Expected KPI impact

- Increase On-Time Delivery Rate.
- Reduce average delay per late delivery.
- Reduce traffic delay minutes per route.
- Reduce fuel per package.
- Reduce overtime.
- Reduce unnecessary sequence deviations.

---

### Strategy 2: Redesign delivery territories and route sequences using historical performance

#### Inefficiency targeted

- Some routes consistently run late or require overtime.
- Uneven workload distribution across drivers.
- Excessive travel time between geographically dispersed stops.
- Low vehicle utilization on some routes and overload on others.

#### Root cause addressed

Territories and route assignments may not reflect actual delivery density, service-time complexity, congestion exposure, and vehicle capacity constraints.

#### Process mining support

Process mining can identify:

- Routes with persistent planned-vs-actual duration gaps.
- Geographic clusters with high service time or failure rates.
- Stops frequently delivered out of planned sequence.
- Routes with high travel-to-service ratios.
- Imbalanced route workloads.
- Areas where drivers finish early versus areas requiring overtime.

#### Recommended action

Rebalance territories and redesign route templates using:

- Historical actual travel times instead of theoretical distances.
- Stop-level service-time predictions.
- Delivery density.
- Customer time-window patterns.
- Vehicle capacity and package volume.
- Driver shift constraints.
- Known parking/access complexity.
- Failed-delivery probability.

For example:

- Split a route with many high-rise apartment deliveries into two smaller routes.
- Move business deliveries earlier in the day.
- Cluster residential evening-window deliveries together.
- Assign larger vehicles only where parking constraints allow.
- Reduce cross-town travel between low-density areas.

#### Expected KPI impact

- Reduce average route duration.
- Reduce overtime rate.
- Improve vehicle utilization.
- Improve deliveries per hour.
- Reduce travel time per package.
- Improve on-time delivery for chronically late routes.

---

### Strategy 3: Improve time-window management and customer communication

#### Inefficiency targeted

- Failed deliveries.
- Re-delivery loops.
- Missed customer time windows.
- Wasted travel and service time.
- Increased fuel per successful package.

#### Root cause addressed

Customers may not be available when deliveries are attempted, and planned windows may not reflect actual success probability. Customers may receive insufficient notification or have limited ability to redirect deliveries.

#### Process mining support

Package-level process mining can reveal:

- Failed delivery loops.
- Times and zones with high failure rates.
- Customer segments with repeated failed attempts.
- Difference in success rate by notification timing.
- Cost of failed deliveries in minutes, kilometers, and capacity.
- Business deliveries attempted after closing hours.
- Residential deliveries attempted during low-availability periods.

#### Recommended action

Introduce data-driven delivery appointment and notification policies:

- Predict failed-delivery risk for each package.
- Offer proactive rescheduling for high-risk deliveries.
- Send accurate ETA notifications based on actual route progress.
- Allow customers to choose alternative delivery windows.
- Promote safe-place, parcel locker, neighbor delivery, or pickup-point options.
- Prioritize business deliveries during business hours.
- Avoid repeated attempts at the same time of day after a failure.
- Use historical success probability by address and time window.

#### Expected KPI impact

- Reduce Failed Delivery Rate.
- Reduce re-delivery cost.
- Improve On-Time Delivery Rate.
- Increase successful deliveries per route.
- Reduce fuel consumption per successful package.
- Improve customer satisfaction.

---

### Strategy 4: Predictive maintenance based on operational event patterns

#### Inefficiency targeted

- Unscheduled stops.
- Mid-route breakdowns.
- Vehicle downtime.
- High fuel consumption from poorly performing vehicles.
- Delivery delays caused by vehicle faults.

#### Root cause addressed

Maintenance may be reactive or based only on fixed intervals, not actual usage intensity, route conditions, idling, stop-start cycles, or early warning signals.

#### Process mining support

By integrating maintenance logs with vehicle-day process data, Speedy Parcels can identify:

- Vehicles with repeated unscheduled stops.
- Patterns preceding breakdowns.
- Routes or driving conditions that increase wear.
- Relationship between vehicle age/mileage and late deliveries.
- Vehicles with abnormal fuel consumption.
- Maintenance events causing downstream route disruption.

#### Recommended action

Develop predictive maintenance triggers based on:

- Mileage and engine hours.
- Idle time.
- Stop-start frequency.
- Harsh acceleration/braking, if available.
- Engine warning events.
- Historical breakdown patterns.
- Fuel efficiency degradation.
- Repeated unscheduled stops.
- Time since last maintenance.

Schedule maintenance during low-demand periods and avoid assigning high-risk vehicles to long or time-critical routes.

#### Expected KPI impact

- Reduce maintenance-related route disruptions.
- Reduce unscheduled repair frequency.
- Reduce average vehicle downtime.
- Improve On-Time Delivery Rate.
- Reduce fuel per km.
- Reduce emergency repair costs.

---

### Strategy 5: Targeted driver coaching and best-practice transfer

#### Inefficiency targeted

- Long stop durations.
- Excessive idling.
- High fuel consumption.
- Unnecessary sequence deviations.
- Poor scanner compliance.
- High failed delivery handling time.

#### Root cause addressed

Drivers may vary in local knowledge, delivery technique, parking strategy, customer communication, scanning discipline, or eco-driving behavior.

#### Process mining support

Resource-based process mining can identify:

- Drivers with consistently low service times on comparable routes.
- Drivers with high on-time performance despite difficult routes.
- Drivers with high idle time or fuel consumption.
- Drivers with frequent unplanned stops.
- Drivers whose sequence deviations improve performance.
- Best-practice variants used by top performers.

#### Recommended action

Create targeted coaching programs:

- Eco-driving training for drivers with high fuel per km.
- Scanning compliance training for inconsistent event capture.
- Urban parking and building-access techniques.
- Route-specific knowledge sharing.
- Guidance on when route sequence deviations are beneficial.
- Reduced idling policies with feedback dashboards.
- Mentoring from high-performing drivers.

Importantly, coaching should be based on fair comparisons. Drivers should be benchmarked against similar routes, vehicle types, and delivery conditions.

#### Expected KPI impact

- Reduce average service time per stop.
- Reduce fuel consumption.
- Reduce idle time.
- Improve route completion rate.
- Reduce overtime.
- Improve data quality through better scanning compliance.

## 5. Operational Constraints and Continuous Monitoring

### 5.1 Accounting for operational constraints

Any optimization must respect real-world constraints.

---

### Driver working hours

Routing and scheduling must include:

- Maximum shift duration.
- Legally required breaks.
- Overtime limits.
- Depot start/end times.
- Driver availability.
- Skill or certification requirements.
- Fatigue and safety considerations.

Process mining can reveal whether planned routes are realistically executable within working hours. If a route regularly ends in overtime, it should be redesigned rather than expecting drivers to compensate.

---

### Vehicle capacity

Vehicle constraints include:

- Maximum parcel count.
- Weight capacity.
- Volume capacity.
- Refrigerated or special handling requirements, if applicable.
- Vehicle access restrictions in city centers.
- Parking constraints.
- Fuel or battery range for electric vehicles.

Optimization should avoid assigning too many bulky packages to small vans or sending large vans into areas where parking causes long service times.

---

### Customer time windows

Customer constraints include:

- Requested delivery windows.
- Business opening hours.
- Preferred customer availability.
- Priority service levels.
- Failed-delivery risk.
- Access restrictions.

Dynamic routing must preserve time-window feasibility. If delays make a time window impossible, the system should trigger proactive customer communication or reassignment.

---

### Depot and loading constraints

Depot operations also matter:

- Loading dock capacity.
- Package sorting readiness.
- Route staging sequence.
- Driver check-in time.
- Vehicle availability.
- Dispatch release time.

If late depot departure is a major bottleneck, route optimization alone will not solve the issue.

---

### Maintenance constraints

Vehicles must be scheduled to avoid operational disruption.

Consider:

- Maintenance windows.
- Spare vehicle availability.
- Vehicle risk score.
- Repair duration.
- Critical route assignments.
- Seasonal demand peaks.

---

## Summary of Recommended Approach

Speedy Parcels should use process mining to convert its GPS, scanner, dispatch, and maintenance data into a unified operational event log. This log will reveal how delivery routes are actually executed versus how they were planned.

The analysis should focus on:

1. Discovering actual route, stop, package, and maintenance processes.
2. Checking conformance against planned routes and time windows.
3. Measuring performance using KPIs such as on-time delivery, stop time, travel delay, fuel per package, failed delivery rate, overtime, and maintenance disruption.
4. Identifying bottlenecks by route, driver, vehicle, time of day, location, and activity.
5. Validating root causes through variant analysis, dwell-time analysis, traffic correlation, and package-level rework analysis.
6. Implementing data-driven improvements such as dynamic routing, territory redesign, better customer communication, predictive maintenance, and targeted driver coaching.
7. Continuously monitoring outcomes through process mining dashboards and real-time exception alerts.

The expected business benefits include higher delivery punctuality, lower fuel and labor costs, fewer failed deliveries, better vehicle reliability, improved driver productivity, and more resilient last-mile operations.
