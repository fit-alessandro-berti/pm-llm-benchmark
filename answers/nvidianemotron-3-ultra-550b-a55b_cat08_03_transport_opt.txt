## 1. Process Discovery and Conformance Checking

### 1.1 Data Preprocessing & Integration Strategy
The primary challenge is fusing heterogeneous data streams into a single, ordered event log where the **Case ID** represents a logical unit of work. For last-mile delivery, the optimal Case ID is **`Vehicle_ID + Operational_Date` (e.g., `V12-20241205`)**, representing a "Vehicle Day." However, to analyze package-level SLA adherence, a secondary log with **`Package_ID`** as Case ID must be derived.

**Integration Pipeline (ETL for Process Mining):**

| Source | Key Fields | Integration Logic & Challenges |
| :--- | :--- | :--- |
| **Dispatch System (Planned)** | `Route_ID`, `Sequence`, `Planned_Time_Window`, `Package_IDs`, `Vehicle_ID`, `Driver_ID` | **Baseline Model.** Defines the "Happy Path." *Challenge:* Plans change dynamically; need to capture *versioned* plans (Initial vs. Updated). |
| **Vehicle GPS (Contextual/Movement)** | `Timestamp`, `Lat/Lon`, `Speed`, `Ignition`, `Odometer` | **Spine of Reality.** High volume (1Hz–30s). *Challenge:* **Map Matching** (snapping coords to road network), **Stop Detection** (dwell time > X mins, speed=0), **Trip Segmentation** (Depot -> Stop 1 -> Stop 2... -> Depot). Requires interpolation for missing signals (tunnels/urban canyons). |
| **Driver Scanner (Transactional)** | `Timestamp`, `Event_Type` (Arrive, Success, Fail, Depart), `Package_ID`, `Location` | **Ground Truth for Service.** *Challenge:* **Clock Drift** (scanner time vs. GPS time vs. Server time). Requires time synchronization (NTP offset correction). **Missed Scans** (driver forgets "Arrive" but scans "Success"). |
| **Maintenance Logs (Disruptive)** | `Start_Time`, `End_Time`, `Type` (Scheduled/Unscheduled), `Vehicle_ID` | **Non-Delivery Context.** *Challenge:* Linking to Vehicle Day. An unscheduled repair at 11:05 AM splits the Vehicle Day into two sub-cases or adds a "Vehicle Unavailable" activity. |

**Critical Preprocessing Steps:**
1.  **Time Synchronization:** Align all timestamps to UTC using GPS time as the master clock.
2.  **Stop Detection Algorithm (GPS):** Apply DBSCAN or Time-Threshold clustering on zero-speed points. Enrich with POI (Point of Interest) data to label stops: *Customer Delivery, Depot, Fuel Station, Break, Unknown/Private*.
3.  **Event Correlation (The "Join"):** Fuzzy match Scanner events to GPS Stops using `Package_ID` -> `Planned_Stop_Location` + `Time_Window` +/- 15 mins. Resolve 1:N or N:1 matches (e.g., multiple packages dropped at one building).
4.  **Lifecycle Completion:** Ensure every `Depart Depot` has a matching `Arrive Depot`. Impute missing `End Shift` from last GPS movement + buffer.
5.  **Attribute Enrichment:** Add `Time_of_Day` (Peak/Off-Peak), `Day_of_Week`, `Weather`, `Traffic_Index` (external API), `Zone_Type` (Urban/Suburban/Rural).

### 1.2 Process Discovery: Visualizing the "As-Is"
Using the integrated `Vehicle-Day` log, I would apply the **Inductive Miner (IMf / IMd)** or **Heuristics Miner** (robust to noise/incompleteness typical in GPS data) to discover a Directly-Follows Graph (DFG) and BPMN model.

**Expected Discovered Model Structure (High-Level):**
*   **Start:** `Shift Start` -> `Vehicle Inspection` -> `Load Vehicle` -> `Depart Depot`.
*   **Core Loop (x35 Stops):**
    *   `Transit to Stop` (High variance duration).
    *   `Search Parking / Approach` (New activity discovered via GPS dwell < Scanner 'Arrive').
    *   `Arrive Customer` (Scanner).
    *   `Service Time` (`Delivery Success` / `Delivery Failed` / `Partial Delivery`).
    *   `Depart Customer`.
    *   *Deviation Branch:* `Unplanned Stop` (Break, Fuel, Vehicle Issue, Personal).
*   **Exception Handling:**
    *   `Delivery Failed` -> `Return to Depot (EOD)` OR `Re-attempt Later` (Loop back).
    *   `Vehicle Breakdown` -> `Roadside Assistance` -> `Repair` -> `Resume Route` / `Tow to Depot`.
*   **End:** `Arrive Depot` -> `Unload/Scan Returns` -> `Shift End`.

**Key Discovery Insights:** The model will likely reveal **"Invisible Activities"**—GPS dwells not matching scanner events (e.g., "Finding Parking," "Walking to Door," "Break at Roadside")—which are major hidden time consumers.

### 1.3 Conformance Checking: Plan vs. Reality
We align the **Planned Route (Dispatch)** vs. **Executed Route (Discovered Model/Trace)**.

**Technique:** **Token-based Replay** or **Alignments** (cost-based: Move-on-Log vs. Move-on-Model) using the Planned Sequence as the Reference Model.

**Deviation Taxonomy & Detection Logic:**

| Deviation Type | Process Mining Detection | Business Impact |
| :--- | :--- | :--- |
| **Sequence Deviation (Re-sequencing)** | Alignment shows `Skip` (planned stop visited later) or `Insert` (unplanned stop visited). Calculated via **Longest Common Subsequence (LCS)** between Planned Stop Sequence vs. Actual Stop Sequence. | Driver intuition vs. Algorithm; Traffic avoidance; Customer time window pressure. |
| **Unplanned Stops** | Activities in Log (`Unplanned Stop`, `Fuel`, `Breakdown`) not in Reference Model. Filter by `Location_Type != Customer`. | Lost productive time; Fuel theft risk; Unauthorized breaks. |
| **Missed Stops (Dropped Packages)** | Planned Stop ID exists in Dispatch but **no corresponding GPS Stop/Scanner Event** in Vehicle Day log. | Failed SLA; Cost of re-delivery; Customer churn. |
| **Time Window Violations** | `Arrive Customer` Timestamp > `Planned_Time_Window_End` (Hard) or > `Planned_Time_Window_Start` (Soft). | SLA Breach; Penalty costs. |
| **Excessive Dwell/Service Time** | `Duration(Arrive -> Depart)` > `Planned_Service_Time` (std: 3-5 min) + Threshold (e.g., > 10 min). | Bottleneck; Capacity erosion. |
| **Route Geometry Deviation** | Compare GPS Polyline vs. Planned Polyline (Frechet Distance / Hausdorff Distance). | Distance inflation (Fuel); Time inflation. |

## 2. Performance Analysis and Bottleneck Identification

### 2.1 Key Performance Indicators (KPIs) & Calculation Logic

| KPI Category | KPI Name | Calculation from Event Log | Target/Benchmark |
| :--- | :--- | :--- | :--- |
| **Punctuality** | **On-Time Delivery Rate (OTD %)** | `Count(Package_Delivered WHERE Actual_Time <= Promised_Time_Window_End) / Total_Delivered` | > 95% |
| | **Time Window Adherence (Early/Late %)** | Distribution of `(Actual_Arrival - Window_Start)`. | Minimize "Late" tail. |
| **Efficiency** | **Avg. Service Time / Stop** | `Mean( Depart_Customer_TS - Arrive_Customer_TS )` per `Stop_Type` (Residential/Business/Locker). | < 4 min (Resi), < 8 min (Biz) |
| | **Travel Time Ratio** | `Sum(Transit_Durations) / Sum(Service_Durations)` per Vehicle Day. | Benchmark per Zone (e.g., Urban 2.5, Suburban 4.0). |
| | **Planned vs. Actual Route Distance** | `Sum(Actual_GPS_Distance) / Sum(Planned_Dispatch_Distance)`. | < 1.10 (10% slack). |
| **Cost/Assets** | **Fuel Consumption / 100km** | `Total_Fuel_Liters (from fuel card/telematics) / Total_Actual_km`. Segment by `Vehicle_Type` & `Load_Factor`. | Manufacturer spec + 15%. |
| | **Vehicle Utilization Rate** | `Active_Delivery_Time / (Shift_End - Shift_Start)`. Active = Transit + Service. | > 75%. |
| | **Overtime Frequency / Duration** | `Count(Shift_End > Scheduled_Shift_End) / Total_Shifts`; `Mean(Overtime_Mins)`. | 0% / < 30 mins. |
| **Quality** | **First Attempt Delivery Rate (FADR)** | `Count(Success on 1st Attempt) / Total_Attempts`. | > 92%. |
| | **Failed Delivery Root Cause Mix** | `% Failed_Customer_Not_Home` vs `% Failed_Access_Issue` vs `% Failed_Damaged`. | Reduce "Not Home". |

### 2.2 Bottleneck Identification Techniques

1.  **Dotted Chart / Performance Spectrum (per Vehicle Day):**
    *   Visualize all Vehicle Days on X-axis (Start Time), Y-axis (Duration), Color (OTD Rate).
    *   *Insight:* Identify "Long Tail" shifts. Drill down: Are they specific drivers? Specific Routes (e.g., "Route 42 - City Center")? Specific Days (Monday peak)?

2.  **Bottleneck Analysis on DFG (Frequency & Performance):**
    *   **Waiting Time vs. Processing Time:** In the DFG edge `Transit -> Arrive`, high *Waiting Time* implies **Traffic/Parking**. High *Processing Time* on `Arrive -> Depart` implies **Customer Interaction/Packaging issues**.
    *   **Self-Loops:** High frequency on `Delivery Failed -> Re-attempt` or `Unplanned Stop -> Unplanned Stop`.

3.  **Sub-process / Cluster Comparison (Segmentation):**
    *   **Segment by:** `Hour_of_Day` (AM Peak vs Mid-day), `Zone_Density` (Urban Core vs. Suburban), `Driver_Tenure` (New vs. Veteran), `Vehicle_Age`.
    *   *Technique:* **Comparative Process Mining** (e.g., Disco/ProM "Compare" or Python `pm4py` filtering). Calculate **Relative Risk Ratios** for delays per segment.

4.  **Traffic Hotspot Correlation (Spatial Process Mining):**
    *   Map `Transit` edges (Origin Stop -> Dest Stop) onto GIS.
    *   Color edges by **Median Speed (km/h)** or **Travel Time Deviation (Actual - Planned)**.
    *   Identify "Red Arcs" recurring daily (e.g., "Bridge Crossing," "School Zone," "Construction Zone").

5.  **Resource Contention Analysis:**
    *   **Depot Bottleneck:** Analyze `Arrive Depot` -> `Depart Depot` (Next Day) or `Shift Start` -> `Depart Depot`. High variance here indicates **Yard Congestion** or **Loading Process Variability**.
    *   **Scanner/GPS Lag:** Analyze `Arrive Customer (Scanner)` vs `Stop Detected (GPS)`. Large gap = **Parking/Walking Time** (Major hidden cost in dense urban).

## 3. Root Cause Analysis for Inefficiencies

Moving beyond *symptoms* (late delivery) to *causes* requires multi-perspective analysis.

### 3.1 Hypothesis-Driven Validation via Process Mining

High **Levenshtein Distance** / Low LCS similarity. <br> **Temporal Analysis:** Plot `Sequence_Deviation_Count` by `Hour_of_Day`. | Drivers consistently re-sequence stops during Peak Hours (7-9am, 4-6pm) to avoid known bottlenecks. Planned route assumes free-flow speed. |
| **2. Inaccurate Travel Time Estimates** | **Regression Residual Analysis:** Model `Actual_Transit_Time` ~ `Planned_Transit_Time` + `Distance` + `Time_of_Day` + `Weather`. Analyze **Residuals (Error)**. | Systematic underestimation (Negative Residuals) for Urban segments >5km during Peak. High variance (Std Dev of Residual) for Suburban routes. |
| **3. Traffic Congestion Patterns** | **Spatial-Temporal Clustering:** Cluster `Transit` events by `Origin_Zone`, `Dest_Zone`, `Time_Bucket`. Calculate **Travel Time Index (TTI = Actual / Freeflow)**. | TTI > 1.5 consistently on "Main St Corridor" 08:00-09:30. GPS speed < 10km/h on specific arcs. |
| **4. High Service Time Variability** | **Box Plot / Violin Plot** of `Service_Duration` by: `Stop_Type` (Apt vs House vs Business), `Package_Count_Per_Stop`, `Driver_ID`, `Failed_vs_Success`. | Apartments with no concierge: Median 6min, P90 15min. Driver Behavior/Skill** | **Driver Profiling:** Cluster Drivers by: `Avg_Service_Time`, `Parking_Search_Time (GPS_Stop - Scanner_Arrive)`, `Sequence_Adherence`, `Idle_Time_Ratio`, `Harsh_Braking_Events (Telematics)`. | Cluster A: Fast service, high adherence, low idle (Veterans). Cluster B: High parking search, high idle, frequent re-sequencing (New hires / Disengaged). |

Calculate **Cost per Successful Delivery** = `(Cost_Att1 + Cost_Att2 + Admin) / 1`. | 12% of volume requires 2+ attempts. "Not Home" is 70% of fails. Re-delivery adds 15km & 40min per package. |

*   **Select:** Top 10% Vehicle-Days (High OTD, Low Overtime, Low km) vs. Bottom 10%.

## 4. Data-Driven Optimization Strategies

### Strategy 1: Dynamic Time-Window & Routing Engine Calibration (Target: Punctuality & Fuel)
*   **Targeted Inefficiency:** Systematic late arrivals due to static travel times; Excessive km from driver "hacks" (re-sequencing).
*   **Root Cause:** Dispatch uses static "Free-Flow + Fixed Buffer" travel times. Drivers deviate to meet windows, increasing distance.
*   **Process Mining Evidence:**
    *   Conformance: High Sequence Deviation during Peak Hours (LCS < 70%).
    *   Performance: `Actual_Transit_Time` correlates poorly with `Planned_Transit_Time` (R² < 0.4) but highly with `Time_of_Day` + `Traffic_Index`.
    *   Spatial: Identified "Red Arcs" where TTI > 1.6 daily.
*   **Proposal:**
    1.  **Build Historical Travel Time Matrix:** Use 6 months GPS `Transit` events to calculate **Percentile Travel Times (P50, P80, P95)** for every Zone-to-Zone pair per 30-min Time Bucket.
    2.  **Feed Dispatch Solver:** Replace static averages with **P85/P90 Historical Percentiles** as input costs for the VRP (Vehicle Routing Problem) solver.
    3.  **Dynamic Re-optimization:** Integrate Real-Time Traffic API (TomTom/Here) for "Next 2 Hours" adjustments. If `Predicted_Arrival` > `Window_End`, auto-suggest swap to Dispatcher.
*   **Expected KPI Impact:**
    *   OTD Rate: **+8-12%** (Realistic plans = achievable plans).
    *   Planned vs. Actual Distance: **Reduce Gap from 15% -> < 5%** (Drivers follow plan because it works).
    *   Overtime: **-20%** (Less end-of-day catch-up).

### Strategy 2: "Smart Delivery" Program to Eliminate Failed Attempts (Target: Cost per Parcel & Capacity)
*   **Targeted Inefficiency:** 12% First Attempt Failure Rate (FADR); Re-delivery costs ~$4-6/parcel; Wasted capacity.
*   **Root Cause:** "Customer Not Home" (70% of fails) driven by narrow/rigid time windows & lack of pre-notification. High variance in Service Time for "Difficult Access" stops.
*   **Process Mining Evidence:**
    *   Package Trace: `Fail_Not_Home` -> `Return_Depot` -> `Next_Day_Reload` loop adds 2 distinct Vehicle-Day activities per package.
    *   Variant Analysis: High performers use "Safe Drop / Locker / Neighbor" options (Scanner events: `Delivery Success - Alternative`); Low performers only attempt `Hand_To_Customer`.
    *   Correlation: Stops with `Planned_Window_Duration < 2hrs` have 3x Fail Rate.
*   **Proposal:**
    1.  **Predictive Failure Scoring:** Train ML model (XGBoost) on historical log: Features = `Customer_History_Fail_Rate`, `Address_Type`, `Time_Window_Width`, `Weather`, `Day_of_Week`. Output: `P(Fail)`.
    2.  **Proactive Intervention (Automated):**
        *   High `P(Fail)` + Residential -> Auto-send SMS/Email **"2hr Window + Locker Option / Safe Drop Consent Link"** at 06:00 AM.
        *   High `P(Fail)` + Business -> **Mandate Contact Phone Number** on manifest; Dispatcher calls 30min prior.
    3.  **Network Diversion:** Integrate Parcel Lockers / PUDO (Pick-up Drop-off) points as "Virtual Stops" in Routing. If `P(Fail) > 0.6`, route to Locker automatically.
*   **Expected KPI Impact:**
    *   FADR: **92% -> 97%**.
    *   Cost per Parcel: **-$0.40 to $0.60** (Eliminated 2nd attempt km + labor + sorting).
    *   Vehicle Utilization: **+3-5% Capacity Freed** (Fewer re-deliveries = more new parcels per van).

### Strategy 3: Predictive Maintenance & Vehicle Health Monitoring (Target: Unplanned Downtime & Maintenance Cost)
*   **Targeted Inefficiency:** Unscheduled stops (Breakdowns) causing 2.5hr avg delay; High Overtime; Safety Risk.
*   **Root Cause:** Reactive maintenance (Fix on Fail). No link between Telematics diagnostics (DTC codes, Battery Voltage, DPF Soot Levels) and Maintenance Logs.
*   **Process Mining Evidence:**
    *   Case Comparison: Vehicle-Days with `Unplanned_Stop_Breakdown` show **precursor signals** in GPS/Telematics 24-48h prior: `Engine_Load_High`, `Coolant_Temp_Variance`, `Battery_Voltage_Drop`, `Check_Engine_Light_On` (Event Type).
    *   Maintenance Log Join: 60% of breakdown vehicles had `Scheduled_Maintenance_Overdue > 30 days` or `Mileage_Since_Last_Service > Threshold`.
*   **Proposal:**
    1.  **Digital Twin Health Score:** Stream Telematics (OBD-II/CAN Bus) -> Cloud. Calculate daily **Vehicle Health Index (VHI)** using Survival Analysis (Cox Proportional Hazards) trained on historical `Breakdown_Event` as target.
    2.  **Dynamic Maintenance Scheduling:**
        *   `VHI < 0.3` (High Risk) -> **Flag for Next Depot Visit** (Insert `Unscheduled_Maintenance` activity into *next* day's Plan automatically).
        *   `VHI 0.3-0.6` -> **Prioritize in Weekly Workshop Plan**.
    3.  **Driver Feedback Loop:** Mobile App Alert: *"Vehicle V12: Coolant Temp Trending High. Limit Idling. Report to Depot on Return."*
*   **Expected KPI Impact:**
    *   Unscheduled Breakdowns: **-50-70%**.
    *   Overtime caused by Breakdowns: **Near Zero**.
    *   Maintenance Cost: **Shift from Corrective (3-5x cost) to Predictive (1x cost)**. Extended Vehicle Lifespan.

## 5. Operational Constraints & Continuous Monitoring Plan

### 5.1 Accounting for Constraints in Optimization
| Constraint | Strategy 1 (Routing) | Strategy 2 (Failed Delivery) | Strategy 3 (Maintenance) |
| :--- | :--- | :--- | :--- |
| **Driver Working Hours (EU/US Regs)** | Routing Solver **Hard Constraint**: `Route_Duration + Buffer <= Max_Shift (e.g., 10.5h)`. Overtime allowed only as *soft constraint with penalty cost*. | Failed delivery reduction **reduces** avg shift length, creating headroom. Predictive window mgmt avoids "Wait for Customer" idle time counting as work. | Maintenance insertion **must** respect `Min_Rest_Period (11h)`. If VHI flags vehicle for AM repair, solver assigns spare vehicle *previous night*. |
| **Vehicle Capacity (Volume/Weight)** | VRP Solver enforces `Sum(Package_Volume) <= Vehicle_Max_Volume`. Historical log used to calibrate `Avg_Package_Volume` per customer segment (B2C vs B2B). | Locker diversion **reduces** physical load on van (package removed at depot). Safe-drop reduces dwell but not volume. | Spare vehicle pool sizing based on `P(VHI < 0.3)` fleet-wide. Ensure spare capacity >= 95th percentile of simultaneous breakdowns. |
| **Customer Time Windows** | Solver uses **Soft Time Windows with Penalty Costs** calibrated from Process Mining: `Penalty = f(Window_Width, Customer_Tier, Historical_Fail_Cost)`. | **Core Lever:** Widens *effective* window via pre-notification ("We arrive 14:00-16:00, reply to change"). Converts Hard Window -> Soft Preference. | N/A (Background process). |

### 5.2 Continuous Monitoring & Governance Framework
**"Process Mining Control Tower" Dashboard Architecture:**

#### Layer 1: Executive KPI Dashboard (Daily/Weekly)
*   **North Star Metrics:** OTD %, Cost per Parcel, Fleet Utilization %, CO2 g/parcel.
*   **Trend Lines:** 13-week rolling average with Control Limits (SPC Charts). **Alert:** Rule = "2 out of 3 points > 2 Sigma".

#### Layer 2: Operational Process Views (Shift/Daily - For Dispatchers/Depot Managers)
*   **Conformance Monitor:** "Plan vs. Actual" for *Today's Routes*.
    *   View: Map with **Color-Coded Stops** (Green=On Time, Yellow=At Risk, Red=Late/Missed).
    *   Drill-down: Click Vehicle -> Show **Live Alignment** (Planned Next Stop vs. Actual GPS Heading).
*   **Bottleneck Heatmap:** Live "Parking Search Time" & "Traffic Delay" per Zone. Dispatcher reroutes proactively.
*   **Failed Delivery Tracker:** Real-time list of `Delivery Failed` scans today -> Auto-generate "Re-attempt Manifest" for tomorrow.

#### Layer 3: Deep-Dive Analytical Workbench (Weekly/Monthly - For Analysts/Continuous Improvement)
*   **Variant Explorer:** Compare "This Week" vs "Baseline (6mo avg)" process models. Highlight *new* activities (e.g., new "Construction Delay" stop).
*   **Root Cause Drill-Downs:** Pre-built notebooks for:
    *   *Driver Coaching:* "Top 5 Drivers by Excessive Idle/Parking Search" (Anonymized for privacy, used for training).
    *   *Route Engineering:* "Routes with >20% Distance Deviation" -> Candidate for Re-design.
    *   *Asset Health:* "Vehicle Health Index Trend" + Maintenance Backlog.

#### Governance Cadence:
1.  **Daily Stand-up (15 min):** Dispatchers review Layer 2 "At Risk" vehicles.
2.  **Weekly Process Review (60 min):** Ops Manager + Analyst review Layer 3. **Decision:** Approve Route Changes, Update Time Window Standards, Schedule Maintenance.
3.  **Monthly Strategic Review (90 min):** Leadership review Layer 1. **Decision:** Fleet Sizing, Technology Investment (Lockers/EVs), Contractor Performance.

#### Sustainability Mechanisms:
*   **Model Retraining:** Travel Time Matrix & Failure Prediction Model retrained **Monthly** (rolling 6-month window).
*   **Data Quality SLA:** Automated alerts if `Scanner_Event_Count / GPS_Stop_Count < 0.9` (Missing Scans) or `GPS_Gap > 10 mins` (Tracker Failure).
*   **Change Impact Measurement:** Every optimization (e.g., New Routing Params) treated as an **A/B Test** or **Interrupted Time Series Analysis** in the Event Log before full rollout.
