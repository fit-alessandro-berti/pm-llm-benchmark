# **Data-Driven Optimization of IT Service Desk Resource Assignment using Process Mining**

## **1. Analyzing Resource Behavior and Assignment Patterns**

To understand the current state, we will perform a multi-faceted analysis of the event log.

### **Key Metrics and Analysis Focus:**
- **Workload Distribution:** Calculate the number of tickets (by priority and category) handled per agent and per tier. Identify variance and imbalance.
- **Activity Processing Times:** Measure `Work L1 End - Work L1 Start` (handling time), `Assign L2 - Escalate L2` (escalation queue time), and `Assign L2 - Reassign` (reassignment delay). This will be segmented by agent, tier, ticket priority, and category.
- **First-Contact Resolution (FCR) Rate:** For L1, the percentage of tickets resolved without escalation. Low FCR indicates potential issues with L1 empowerment, skill, or initial ticket routing.
- **Skill Utilization & Mismatch:** Compare the `Required Skill` for a ticket against the `Agent Skills` of the assigned resource. Calculate the frequency of assignments where the agent's skills did not perfectly match the requirement, and the subsequent impact on resolution time and reassignment likelihood.

### **Process Mining Techniques for Assignment Pattern Discovery:**
- **Resource Interaction (Social Network) Analysis:** Model handovers between agents/tiers as a network. Nodes are agents/teams, edges represent ticket handovers (escalations, reassignments). Edge weight shows frequency. This will visually reveal:
    - **Frequent reassignment chains:** (e.g., Agent A -> Agent B -> Agent C), indicating unclear ownership or skill gaps.
    - **Central "hub" agents:** Who are overloaded with handovers?
    - **Departure from intended flow:** The model should show clean L1->L2->L3 paths; deviations (e.g., L1 directly to L3, or L2 back to L1) highlight process violations or bypasses.
- **Role Discovery (Organizational Mining):** Use algorithms to cluster agents based on the activities they perform and the ticket types they handle. This can reveal if the formal tiers (L1, L2, L3) align with actual behavioral roles. For instance, some L2 agents might consistently handle only simple escalations, acting as *de facto* senior L1s.
- **Comparative Analysis (Delta Analysis):** Contrast the **discovered process model** (showing all actual paths) with the **prescribed process model** (intended L1 -> L2 -> L3 escalation). The deviations (e.g., frequent reassignments within L2) are direct evidence of assignment inefficiency.

## **2. Identifying Resource-Related Bottlenecks and Issues**

Using the above analysis, we can pinpoint specific problems:

- **Skill-Based Bottlenecks:** By correlating `Required Skill` with queue times (`Assign - Escalate/Reassign`), we can identify which skills (e.g., `Networking-Firewall`, `Database-SQL`) have the longest queues and highest reassignment rates, signaling a shortage of specialists.
- **Cost of Reassignment:** Quantify the **"Reassignment Penalty."** Calculate the average added cycle time (`Timestamp of next assignment - Timestamp of reassignment`) for reassigned tickets versus direct assignments. For example, if reassignments add an average of 4 hours to P2 ticket resolution, this directly contributes to SLA risk.
- **Impact of Initial Assignment Errors:** Analyze tickets escalated by L1 within a short time (e.g., <10 minutes). This indicates a likely misrouting. We can calculate what percentage of SLA breaches (especially for P2/P3) had such rapid escalations.
- **Over/Under-Utilization:** Use the workload distribution data to identify agents consistently above the 90th percentile or below the 10th percentile in active work time. A high "queue time to work time" ratio for an agent indicates they are a bottleneck.
- **SLA Breach Correlation:** Perform a decision tree analysis on closed tickets. The target variable is `SLA Breach (Yes/No)`. Predictor variables include `Number of Reassignments`, `Initial Assignment Mismatch (Yes/No)`, `Assigned Agent's Tier`, and `Required Skill`. This will statistically show which resource factors are the strongest predictors of failure.

## **3. Root Cause Analysis for Assignment Inefficiencies**

The identified problems likely stem from several interrelated root causes:

1.  **Deficient Assignment Rules:** The current "round-robin within tier" is blind to **skill** and **real-time workload**, causing the mismatches and imbalances we see. This is the primary suspect.
2.  **Poor Ticket Triage & Skill Requirement Capture:** If the `Required Skill` field is often incorrect or `Uncategorized` at creation, any downstream assignment logic fails. We can analyze the completeness and accuracy of this field.
3.  **Inaccurate Agent Skill Profiles:** Agent skills in the system may be outdated or too broad (e.g., "Software-App" vs. "App-CRM"). The role discovery analysis might show agents performing work outside their documented skills, indicating the profiles are not trusted or used.
4.  **L1 Process & Empowerment Gaps:** A low L1 FCR rate, coupled with high volumes of rapid escalations, suggests L1 agents lack the tools, authority, or knowledge to resolve even moderately complex issues, overwhelming L2.

**Leveraging Process Mining for Root Cause:**
- **Variant Analysis:** Isolate the "best-case" variants (tickets with shortest cycle time, no reassignments) and "worst-case" variants (long cycle time, multiple reassignments). Compare their attributes: Did the worst cases start with a skill mismatch? Were they assigned to a consistently overloaded agent? This comparison isolates the factors causing delays.
- **Decision Mining:** Automatically discover the decision rules that were *actually* used for initial L1 assignment and escalation. We can model: "Given ticket Category='Network' and Priority='P2', what was the probability of it being assigned to an agent without the `Networking-Firewall` skill?" This exposes the flaws in the current heuristic.

## **4. Developing Data-Driven Resource Assignment Strategies**

Based on the insights, we propose three concrete strategies:

### **Strategy 1: Skill-Weighted, Workload-Aware Intelligent Routing**
*   **Issue Addressed:** Misrouted tickets and uneven workload.
*   **How it Leverages Insights:** Uses the discovered skill requirements and real-time workload data from the event log. Replaces round-robin with an algorithm that scores each available agent.
*   **Data Required:** Real-time agent availability status, current queue length per agent, ticket attributes (Category, Priority, Predicted Skill from NLP on description), and agent skill profiles with proficiency levels.
*   **Expected Benefits:** Drastic reduction in initial assignment errors and reassignments. Better load balancing. Shorter queue times for tickets as they go directly to a capable agent.
*   **Implementation:** `Agent Score = (Skill Match Score * Weight1) + (1 / Current Queue Length * Weight2)`. Assign to the highest score. Weights are tuned based on historical success.

### **Strategy 2: Predictive Escalation and L1 Empowerment Rules**
*   **Issue Addressed:** Low FCR and unnecessary escalations overwhelming L2.
*   **How it Leverages Insights:** Decision mining identifies ticket characteristics (e.g., specific error messages, simple network categories) that L1 resolved successfully in the past. Variant analysis shows what differentiates L1-resolved vs. L1-escalated tickets of the same type.
*   **Data Required:** Historical tickets with full text descriptions and final resolution notes. A knowledge base of L1-resolvable issue patterns.
*   **Expected Benefits:** Increased L1 FCR, reduced volume to L2/L3, freeing specialists for true complex work. Lower overall resolution time for the issues L1 keeps.
*   **Implementation:** Develop a classifier (e.g., based on ticket text) that suggests "Likely L1-Resolvable" with a confidence score. Integrate with a guided troubleshooting script for L1 agents, empowering them to handle a wider range of issues.

### **Strategy 3: Dynamic Tier Pooling and Specialization Adjustment**
*   **Issue Addressed:** Skill bottlenecks and rigid tier structures causing delays.
*   **How it Leverages Insights:** Social network analysis shows constant handoffs between specific L2 agents and a few L3 agents for a specific skill (e.g., `Database-SQL`). Role discovery might show a subset of L2 agents who only handle database tickets.
*   **Data Required:** Historical demand patterns for specific skills by time of day/week. Agent skill proficiency and cross-training potential.
*   **Expected Benefits:** More flexible resource allocation to meet fluctuating demand. Reduced wait times for specialized skills. Better career pathing for agents.
*   **Implementation:** Create "virtual pools" for high-demand skills (e.g., a "Database Pool") staffed by designated agents from L2 and L3. During peak demand, a dispatcher (or an automated rule) can pull from this pool directly, bypassing the rigid tier hierarchy. Use simulation to determine optimal pool sizes.

## **5. Simulation, Implementation, and Monitoring**

### **Pre-Implementation Simulation:**
We will build a **Business Process Simulation (BPS)** model calibrated with the mined process model, historical arrival rates (by priority/category), and the discovered resource performance metrics (handling times per skill/tier).
- **Experiment:** Run the simulation under the current "as-is" rules and then under each proposed assignment strategy (e.g., skill-based routing).
- **Evaluate:** Compare KPIs like **average cycle time, SLA compliance rate, resource utilization balance, and number of reassignments.** This provides a risk-free way to quantify expected improvement and fine-tune algorithm parameters (e.g., weights in Strategy 1) before go-live.

### **Post-Implementation Monitoring:**
Establish a **Process Mining Dashboard** with continuous monitoring of resource-centric KPIs:
1.  **Primary KPIs:**
    *   **FCR Rate by Tier**
    *   **Reassignment Rate** (per 100 tickets)
    *   **Average Queue Time by Skill**
    *   **SLA Compliance % by Priority**
    *   **Agent Utilization Gini Coefficient** (to measure workload equality)
2.  **Process Views:**
    *   **Live Social Network Graph:** To monitor handover patterns and detect emerging bottlenecks.
    *   **Resource Profile View:** For each agent, showing active tickets, queue, skill match rate, and personal FCR.
    *   **"Assignment Conformance" Dashboard:** Compares actual assignment decisions in real-time against the new "ideal" rules, flagging deviations for dispatcher review.

This closed-loop approach—**analyze, simulate, implement, monitor**—ensures that resource assignment becomes a continuously optimized, data-driven practice, directly addressing TechSolve's challenges of SLA breaches, reassignments, and skill underutilization.
