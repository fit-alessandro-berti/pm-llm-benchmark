### Reasoning & Conflict Resolution

**1. General Matching Strategy**
Events were aligned based on a combination of `order_id` (all matched Order#1234), semantic similarity in event names, and temporal proximity. Because Log A is the primary timeline, Log A's event names and timestamps were adopted as the "Unified Event Name" and "Primary Timestamp" respectively, while Log B provided the enriching attributes (`user_id`, `resource_id`, `notes`).

**2. Handling Timestamp Mismatches & Tolerance**
A timestamp tolerance of ** 2 seconds** was applied as the baseline for matching, with one noted exception:
*   **Order Received / OrderReceived:** Log A (10:00:00Z) vs Log B (09:59:58Z). Difference is exactly 2 seconds. Because the event names are a clear semantic match and no other competing events exist in that timeframe, these were confidently merged.
*   **Order Validated / OrderValidation:** Log A (10:01:30Z) vs Log B (10:01:29Z). Difference is 1 second. Confidently merged.
*   **Item Shipped / Shipping:** Log A (10:05:00Z) vs Log B (10:05:02Z). Difference is 2 seconds. Confidently merged.

**3. Exception Handling: The Payment Event**
*   **Payment Processed / PaymentCheck:** Log A (10:02:00Z) vs Log B (10:02:05Z). The timestamp difference here is **5 seconds**, which exceeds the 2-second tolerance. However, they were still merged for the following reasons:
    *   Semantic match: Both clearly refer to the payment step of the order.
    *   Contextual clue: Log B's note explicitly states *"Payment gateway delay"*, which logically explains why System B logged this event 5 seconds after System A received the confirmation. 

**4. Handling Non-Overlapping Events**
*   **Quality Check (10:03:00Z):** Appears only in Log B. There is no event in Log A within a reasonable window to match it. It is included as a standalone event, enriched with Log B's attributes, and logged with its native timestamp.
*   **Item Delivered (10:20:00Z):** Appears only in Log A. There is no corresponding event in Log B. It is included as a standalone event. Log B attributes (`user_id`, `resource_id`, `notes`) are left null/empty to indicate data absence.

**5. Attribute Integration**
For merged events, all attributes from both logs were combined. Because Log A lacks `user_id`, `resource_id`, and `notes`, these fields were populated entirely by Log B data. For events originating solely from Log A, these fields remain blank. Both timestamps are preserved in the final output for full data transparency, allowing analysts to see exactly when each system registered the event.
