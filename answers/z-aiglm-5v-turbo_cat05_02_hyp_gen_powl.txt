### 3. Verification via Database Queries

To confirm if these anomalies are occurring in real data, we can query the `claims` and `claim_events` tables. We assume the `activity` column in `claim_events` contains strings like `'Receive Claim'`, `'Close Claim'`, etc., matching the labels in the code (R, A, E, P, N, C).

#### Query A: Identify Prematurely Closed Claims (Skipping Eval & Approve)
This query finds claims that were closed but lack both an Evaluation and an Approval event in their history. This validates the `A -> C` shortcut anomaly.

```sql
SELECT 
    c.claim_id,
    c.customer_id,
    c.claim_amount,
    ce.timestamp AS closure_timestamp
FROM claims c
JOIN claim_events ce ON c.claim_id = ce.claim_id
WHERE ce.activity = 'Close Claim'
-- Ensure no Evaluation activity exists for this claim
AND NOT EXISTS (
    SELECT 1 FROM claim_events e 
    WHERE e.claim_id = c.claim_id AND e.activity = 'Evaluate Claim'
)
-- Ensure no Approval activity exists for this claim
AND NOT EXISTS (
    SELECT 1 FROM claim_events p 
    WHERE p.claim_id = c.claim_id AND p.activity = 'Approve Claim'
)
ORDER BY ce.timestamp DESC;
```
