## 1) Grouping logic used

I would group events **per CaseID**, ordered by timestamp, using these signals:

- **Primary signal: process purpose**
  - Do the events serve the same immediate goal?
- **Product-state transition**
  - Does the part move from raw material, to joined part, to finished part, to approved part?
- **Temporal proximity**
  - Events that happen back-to-back are likely part of the same stage.
- **Resource / station family**
  - Helpful as a secondary cue:
    - Operator A / robot / preheater = preparation
    - Operator B + welding tool = welding
    - Quality Sensor = inspection
    - Coating + drying units = finishing
    - Operator C = final QC
- **Inspection gates stay separate**
  - If an event validates the prior work and could trigger rework, it should usually be its own high-level step.

## 2) Proposed high-level activities

| High-level activity | Low-level events included | Why these belong together |
|---|---|---|
| **Material Intake & Preparation** | Retrieve raw metal sheet; Scan barcode; Place metal sheet on table; Align metal sheet; Preheat metal sheet | These are all preparatory actions on the same raw part. They establish **traceability** (barcode), **positioning/setup** (place, align), and **process readiness** (preheat) before any joining work starts. |
| **Welding** | Pick up welding tool; Weld corner A; Weld corner B | These form one continuous joining operation. Tool pickup is not a meaningful business step on its own; it is setup directly tied to the welding action. |
| **Weld Integrity Inspection** | Measure weld integrity | This is a distinct **in-process quality gate** immediately after welding. It validates the weld before downstream finishing begins, so it is better kept separate from welding itself. |
| **Protective Coating & Curing** | Apply protective coating; Dry coating | These are both part of the same finishing objective: protect the surface and complete the coating process. Drying is a natural continuation of coating, not a separate business phase. |
| **Final Visual Inspection** | Visual check | This is the final acceptance step after fabrication and finishing are complete. It represents end-of-line human QA. |

## 4) Rule-based mapping for the full log

A simple abstraction rule set for larger logs could be:

| If Activity looks like… | Map to… |
|---|---|
| Retrieve raw material / scan barcode / place on table / align / preheat | **Material Intake & Preparation** |
| Pick up welding tool / weld corner / weld seam / welding action | **Welding** |
| Measure weld integrity / sensor-based weld check | **Weld Integrity Inspection** |
| Apply coating / dry coating / cure coating | **Protective Coating & Curing** |
| Visual check / final check / final pass | **Final Visual Inspection** |

### Secondary signals from other attributes
Use these when activity names are ambiguous:

- **ToolID, WeldType**  likely **Welding**
- **IntegrityScore**  likely **Weld Integrity Inspection**
- **CoatingType** or drying temperature after coating  likely **Protective Coating & Curing**
- **Check: Passed/Failed**  likely **Final Visual Inspection** or QA stage

### Practical rule for extra low-level events
If the full log contains additional micro-events (machine checks, confirmations, minor adjustments), attach them to the **nearest functional stage** unless they form a clear **quality gate**.
