### Case Grouping and Activity Naming Logic

**Case Identification (Grouping):**
The raw system log represents a single, continuous user session without any explicit logout or "session end" markers (like closing the browser entirely). Therefore, **one Case ID (`CASE_001`)** was assigned to cover the entire timeline. The logical narrative is that of a **"Morning Work Session"** where the user is preparing a Quarterly Report. Although the user switches between different applications and documents (Word, Email, Acrobat, Excel), these are all sub-tasks contributing to the same overarching business goal. Splitting this into multiple cases would artificially fragment the workflow, whereas keeping it as one case allows for analysis of context switching behavior within a single work session.

**Activity Naming Strategy:**
The raw low-level actions (e.g., `FOCUS`, `TYPING`, `CLICK`) were mapped to standardized process mining activities to facilitate high-level analysis:
1.  **Context Switching:** Raw `SWITCH` events were renamed to `TASK_SWITCH`. This allows analysts to measure the frequency of context switching between applications, which is a key performance indicator (KPI) for productivity bottlenecks.
2.  **Document Interaction:** Raw `FOCUS` actions where the user begins working on a new document were standardized as `OPEN_DOCUMENT`. Subsequent typing events were grouped into `EDIT_CONTENT`, `UPDATE_BUDGET`, or `ANALYZE_DOCUMENT` depending on the application context to distinguish between drafting text, updating data, and reviewing documents.
3.  **State Persistence:** Raw `SAVE` actions were renamed to `COMMIT_WORK` to signify a state change in the business process where work is preserved.
4.  **Application Specifics:** Actions like `SCROLL`, `HIGHLIGHT`, and `CLICK` within email or PDF contexts were abstracted into `PROCESS_EMAIL_TASK` and `ANALYZE_DOCUMENT` respectively, removing noise from the raw system logs to focus on the *intent* of the activity rather than the mechanical action.
