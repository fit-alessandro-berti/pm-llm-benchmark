# Precision Parts Inc.: Process Mining-Driven Scheduling Transformation Strategy

## 1. Analyzing Historical Scheduling Performance and Dynamics

To move beyond anecdotal evidence, we must treat the MES event log as the "ground truth" of shop floor physics. Standard process discovery (e.g., Inductive Miner, Heuristics Miner) is insufficient here; we require **Performance-Enhanced Process Mining** combined with **Queueing Theory analytics** applied directly to the event timestamps.

### 1.1 Reconstructing Actual Flows & Execution Sequences
*   **Case-Level Reconstruction:** Correlate `Job Released` -> `Task Start/End` sequences per `Case ID`. Since jobs have unique routings, we cannot assume a single process model. We will use **Flexible Process Discovery** (e.g., Split Miner 2.0) to generate a *process map per job family* or a *global probabilistic automaton* capturing routing probabilities.
*   **Resource-Centric View (Gantt Charts from Logs):** Reconstruct the exact timeline for every `Resource (Machine ID)`. Sort events by `Timestamp` per resource. This reveals the *actual* sequence of jobs on CUT-01, MILL-03, etc., including gaps (idle), overlaps (data errors), and the critical `Setup Start/End` blocks. This is the primary input for Sequence-Dependent Setup Time (SDST) analysis.

### 1.2 Quantification Techniques & Metrics

| Performance Dimension | Process Mining Technique / Calculation | Key Metrics & Visualizations |
| :--- | :--- | :--- |
| **Job Flow/Lead Times** | **Throughput Time Analysis** (Case duration: `Job Released` to final `Task End` / `Job Completed`). | *Distribution:* Histograms/Boxplots per Job Family/Priority. *Percentiles:* P50, P85, P95 (critical for quoting). *Trend:* Control charts over time to detect drift. |
| **Task Waiting Times (Queue Times)** | **Event-to-Event Latency Analysis**. `Queue Entry (Milling)` -> `Setup Start (Milling)`. Requires explicit "Queue Entry" events (present in log). | *Avg/Max Queue Time per Work Center.* *Queue Length over Time* (Little’s Law reconstruction: $L = \lambda W$). *Heatmaps:* Hour-of-day vs. Day-of-week queue congestion. |
| **Resource Utilization (Productive/Idle/Setup)** | **Resource Profile Analysis** (Timestamp-based state machine per resource). States: `PROCESSING`, `SETUP`, `IDLE`, `BREAKDOWN`, `BLOCKED`. | *Utilization %:* $\frac{\sum(Processing+Setup)}{Total Time}$. *Setup Ratio:* $\frac{Setup Time}{Processing+Setup}$ (high ratio = batching opportunity). *OEE Decomposition:* Availability (Breakdowns), Performance (Speed losses), Quality (Rework loops mined from variants). |
| **Sequence-Dependent Setup Times (SDST)** | **Precedence Matrix Mining**. For each Machine $M$, extract ordered sequence $J_1, J_2, ... J_n$. For transition $J_i \to J_{i+1}$, calculate `Setup End` - `Setup Start`. Cluster jobs by `Setup Required` attributes (material, geometry, tooling). | *Setup Time Matrix:* $S_{ij}$ (Job $i$ followed by Job $j$). *Statistical Distribution:* Mean/StdDev per transition cluster. *Violin Plots:* Setup time variance for "Similar" vs. "Dissimilar" job pairs. *Regression:* Predict setup time based on feature distance (e.g., material hardness diff, tool change count). |
| **Schedule Adherence & Tardiness** | **Conformance Checking vs. Due Dates**. Align `Case Completion Timestamp` vs `Order Due Date` attribute. | *Tardiness:* $T_j = \max(0, C_j - d_j)$. *Tardiness Rate %.* *Weighted Tardiness* (inc. `Order Priority`). *Earliness:* $E_j = \max(0, d_j - C_j)$ (WIP cost). *Due Date Slack Consumption:* $\frac{Flow Time}{Allowable Lead Time}$. |
| **Disruption Impact** | **Root Cause Analysis / Comparative Variant Analysis**. Filter cases: "Affected by Breakdown" vs "Clean Run". Filter: "Priority Changed" vs "Static Priority". | *Delay Propagation:* Avg added lead time per breakdown hour. *Priority Inversion:* % of High-Priority jobs delayed by Low-Priority jobs already in queue (Non-preemptive FCFS pathology). *Recovery Time:* Time to return to baseline throughput post-disruption. |

## 2. Diagnosing Scheduling Pathologies

Using the metrics above, we move from *measurement* to *diagnosis*. We hypothesize specific pathologies and validate them via targeted mining.

### 2.1 Bottleneck Identification & Throughput Impact
*   **Technique:** **Bottleneck Analysis via Utilization & Queue Length Correlation**. Identify resources where Utilization > 85% *and* Avg Queue Time > Avg Processing Time.
*   **Evidence:** MILL-02/MILL-03 likely show "Starvation-Blocking" oscillations. Mined Gantt charts will show MILL-02 `BREAKDOWN` events causing immediate queue explosion at MILL-03 (blocking) and starvation at Grinding (downstream).
*   **Quantification:** **Theory of Constraints (ToC) Metrics**: *Constraint Utilization*, *Throughput Lost due to Constraint Downtime*. If MILL-02 is the drum, every minute of breakdown = lost revenue = $\text{Throughput Rate} \times \text{Price}$.

### 2.2 Poor Prioritization & Priority Inversion
*   **Technique:** **Variant Analysis: "On-Time" vs. "Late" Jobs**. Compare the *resource allocation sequences* for High-Priority jobs that met due dates vs. those that missed.
*   **Evidence:** Discovery of **"Priority Inversion" patterns**: High-Priority Job (JOB-7005) enters `Queue Entry (Milling)` but waits behind 3 Medium-Priority jobs because the local rule is FCFS/EDD *without preemption* or *lookahead*. The log shows `Priority Change` event at 11:30, but the job sits in queue for 4 hours while Medium jobs process.
*   **Metric:** *Priority Violation Index*: % of time a higher priority job waits while lower priority job processes on same resource.

### 2.3 Suboptimal Sequencing & Setup Inflation
*   **Technique:** **Setup Time Variant Analysis**. For the bottleneck machine (e.g., CUT-01), compare *Actual Sequence* vs. *Optimal Sequence* (Traveling Salesman Problem on Setup Matrix $S_{ij}$).
*   **Evidence:** Mined sequence: `Steel-A -> Aluminum-B -> Steel-C -> Titanium-D`. Setup times: 15min, 45min, 60min. Optimal cluster sequence: `Steel-A -> Steel-C -> Aluminum-B -> Titanium-D` (Setup: 15min, 10min, 30min).
*   **Quantification:** *Setup Efficiency Ratio* = $\frac{\text{Theoretical Min Setup Time (Clustered)}}{\text{Actual Setup Time}}$. If < 70%, massive capacity is wasted on non-value-added changeovers.

### 2.4 Starvation & Bullwhip in WIP
*   **Technique:** **WIP Level Reconstruction (Little’s Law)**. Count active `Case IDs` between `Job Released` and `Job Completed` per timestamp.
*   **Evidence:** **WIP Oscillations** correlate with "Hot Job" insertions. Release strategy ("Job Released" events) likely pushes large batches (Push System) rather than CONWIP/Kanban pull. High WIP -> Long Queues -> Long Lead Times -> Expediting -> More Hot Jobs -> More Variability (Bullwhip).
*   **Metric:** *WIP Variance / Mean WIP*. High coefficient of variation indicates instability.

## 3. Root Cause Analysis of Scheduling Ineffectiveness

We must distinguish **Scheduling Logic Failures** (controllable) from **Structural Capacity/Variability Limits** (require investment/buffer).

| Root Cause Category | Process Mining Diagnostic Approach | Differentiation Logic |
| :--- | :--- | :--- |
| **Static Dispatching Rules** | **Decision Point Analysis**: At every `Queue Entry` -> `Setup Start` transition, log the *attributes of all jobs waiting in queue at that exact timestamp* (reconstructed from event log). Simulate "What if we picked Job B instead of Job A?" using mined processing/setup times. | If simulated dynamic rule (e.g., ATC, COVERT) consistently outperforms actual history *without adding capacity*, the logic is the root cause. |
| **Lack of Real-Time Visibility** | **Information Lag Analysis**: Time between `Task End (Upstream)` and `Queue Entry (Downstream)` vs `Setup Start (Downstream)`. Large gaps indicate "Blind" downstream scheduling. | If downstream machine idles (`IDLE` state) while upstream job is already `Task End` (but not yet physically moved/logged), the issue is communication/visibility, not capacity. |
| **Inaccurate Estimations (Planned vs Actual)** | **Conformance Checking: Planned Duration vs Actual Duration**. Scatter plots per Task/Resource/Operator. | *Systematic Bias* (Planned << Actual) -> Estimation model failure. *High Variance* (Wide scatter) -> Inherent variability / poor setup definition. Scheduling based on "Planned" is guaranteed to fail if variance is high. |
| **Ineffective SDST Handling** | **Clustering Analysis on Setup Features**: Do jobs with same `Setup Required=TRUE` but different previous jobs have wildly different actual setup times? | If `Setup Duration` correlates strongly with *Previous Job Features* (mined from log), but scheduling assumes *Constant Setup*, the root cause is the scheduling model ignoring physics. |
| **Poor Inter-Work Center Coordination** | **Cross-Resource Correlation**: Correlation of `Queue Length (Milling)` vs `Utilization (Grinding)`. Negative correlation = Starvation. Positive correlation = Blocking. | If Milling is overloaded (Queue high) but Grinding is starved (Util low), the *Release Mechanism* or *Milling Sequencing* is pushing wrong job mix (e.g., all jobs needing Grinding stuck behind jobs needing Heat Treat). |
| **Disruption Response** | **Recovery Trajectory Analysis**: Post-breakdown (`Breakdown End`), how long until Utilization returns to mean? Does the scheduler re-optimize? | If schedule remains static (same sequence resumes) causing massive tardiness cascade -> Scheduling system is "Open Loop" (no feedback). |

**Key Differentiator:** Run a **"Perfect Information Simulation"**. Replay the *exact historical job arrivals, breakdowns, and actual processing times* using an *optimal offline scheduler* (e.g., CP-SAT solver). Compare KPIs (Tardiness, Makespan) against *Actual History*.
*   *Gap Small*: System is near structural limit (need capacity/buffer).
*   *Gap Large*: Scheduling logic is the primary lever.

## 4. Developing Advanced Data-Driven Scheduling Strategies

We propose three synergistic strategies forming a **Hierarchical Control Architecture**:
1.  **Strategic/Tactical:** Predictive Schedule Generation (Horizon: Days/Weeks).
2.  **Operational:** Setup-Aware Bottleneck Sequencing (Horizon: Shift/Hours).
3.  **Execution/Reactive:** Dynamic Multi-Attribute Dispatching (Horizon: Minutes/Real-time).

---

### Strategy 1: Predictive Probabilistic Scheduling (The "Digital Twin" Baseline)
**Core Logic:** Move from deterministic "Planned Durations" to **Stochastic Scheduling**. Generate a *robust baseline schedule* using distributions mined from logs, optimized for *expected* tardiness/WIP, not just makespan.

**Process Mining Inputs:**
*   **Task Duration Distributions:** Fit distributions (LogNormal, Weibull, Gamma) per `(Task, Resource, Job Family, Operator Skill)` tuple using `Task Duration (Actual)`. Capture correlation between Setup and Process times.
*   **Routing Probabilities:** For jobs with flexible routing (e.g., MILL-02 vs MILL-03), mine historical assignment probabilities and performance outcomes.
*   **Breakdown/Disruption Models:** Fit `Time Between Failures` (Exponential/Weibull) and `Time To Repair` (Lognormal) distributions per critical resource from `Breakdown Start/End` events.
*   **Arrival Process:** Model `Job Released` inter-arrival times and `Order Priority` mix.

**Algorithm:** **Sample Average Approximation (SAA) / Stochastic Programming** or **Simulation-Optimization Loop**.
1.  Generate $N$ scenarios sampling from duration/breakdown/arrival distributions.
2.  Solve deterministic FJSP-SDST for each scenario (using CP-SAT or MILP).
3.  Aggregate decisions (e.g., job release times, bottleneck sequence) via consensus or solve a master problem minimizing *CVaR (Conditional Value at Risk) of Tardiness*.

**Addressed Pathologies:** Unpredictable Lead Times (outputs prediction intervals), Inaccurate Estimations (uses reality distributions), Disruption Impact (builds protective capacity buffers *only where needed* at bottleneck).

**Expected KPI Impact:**
*   Quote Accuracy (Lead Time Prediction): +30-40% (P85 adherence).
*   WIP Reduction: 15-20% (Less safety time padding in release).
*   Tardiness: 25% reduction vs deterministic baseline.

---

### Strategy 2: Setup-Aware Bottleneck Sequencing & Intelligent Batching (The "Drum" Optimizer)
**Core Logic:** The system throughput is gated by the **Bottleneck Resource (The Drum)**. We implement a **Rolling Horizon Optimization** specifically for the bottleneck machine(s) (identified in Sec 2) that explicitly minimizes *Weighted Setup Time + Tardiness Cost*.

**Process Mining Inputs:**
*   **SDST Matrix $S_{ij}$:** Mined transition matrix (Sec 1.2), clustered by job attributes (Material, Tooling, Geometry).
*   **Job "Families" / Compatibility Graph:** Community detection on the setup matrix graph (jobs = nodes, low setup time = edges).
*   **Downstream Constraint Propagation:** Mined "Blocking Probability" – if Job A goes to Heat Treat (slow) and Job B goes to Grinding (fast), sequencing A then B might block the bottleneck if Heat Treat is full.

**Algorithm:** **Ant Colony Optimization (ACO) / Genetic Algorithm (GA) with Custom Heuristics** running every 30-60 mins (Rolling Horizon).
*   *State:* Current machine status, Queue contents (with priorities, due dates, downstream status), Remaining setup from current job.
*   *Objective:* $\min \sum (w_1 \cdot Setup_{ij} + w_2 \cdot Tardiness_j + w_3 \cdot Downstream_Blocking_Risk_j)$.
*   *Constraint:* Max batch size (to prevent WIP explosion), Max wait for High-Priority jobs (Preemption threshold).

**Addressed Pathologies:** Suboptimal Sequencing (Setup Inflation), Starvation of Downstream (explicit blocking risk term), High WIP (Batch size constraint).

**Expected KPI Impact:**
*   Bottleneck Setup Time Reduction: 30-50% (Free capacity = Free throughput).
*   Bottleneck Utilization: +5-10% effective capacity.
*   High-Priority Tardiness: Near zero (Hard constraint/High weight).

---

### Strategy 3: Context-Aware Dynamic Dispatching (The "Shield" / Execution Layer)
**Core Logic:** When the predictive schedule (Strat 1) meets reality (breakdowns, hot jobs), local dispatching rules at *non-bottleneck* centers must align with global optimum. We replace FCFS/EDD with a **Data-Driven Composite Priority Index (CPI)** calculated in real-time by the MES.

**Process Mining Inputs (Feature Engineering for CPI Weights):**
*   **SHAP Values / Feature Importance:** Train a Gradient Boosted Tree (XGBoost/LightGBM) to predict `Tardiness` or `Flow Time` using features available at `Queue Entry`: `Slack Time`, `Priority`, `Remaining Ops`, `Downstream Queue Length`, `Estimated Setup Time (from SDST matrix)`, `Bottleneck Status (Busy/Down)`.
*   **Learned Weights:** The model reveals: *When Bottleneck is Down, Downstream Queue Length weight $\uparrow$; When Slack Low, Slack Weight $\uparrow$*.

**The CPI Formula (Dynamic):**
$$CPI_j = \alpha \cdot \frac{1}{Slack_j} + \beta \cdot Priority_j + \gamma \cdot \frac{1}{Downstream\_Load_j} + \delta \cdot \frac{1}{Est\_Setup\_Time_{current \to j}} + \epsilon \cdot Bottleneck\_Urgency\_Signal$$
*Weights $(\alpha, \beta, \gamma, \delta, \epsilon)$ are updated weekly via retraining on latest 4 weeks of logs.*

**Addressed Pathologies:** Priority Inversion (Explicit Priority/Slack terms), Starvation (Downstream Load term), Setup Inflation at Non-Bottlenecks (Setup term), Disruption Response (Bottleneck Signal term - e.g., if MILL-02 breaks, prioritize jobs *not* needing MILL-02 or jobs feeding MILL-03 to keep it fed).

**Expected KPI Impact:**
*   Flow Time Variance Reduction: 20% (Smoother flow).
*   Operator Decision Support: Reduces "Firefighting" by 40% (Automated "Next Best Job" recommendation on tablet).
*   Hot Job Insertion Lead Time: Reduced by 50% (System dynamically clears path).

## 5. Simulation, Evaluation, and Continuous Improvement

### 5.1 Discrete-Event Simulation (DES) for Virtual Commissioning
**Tooling:** Python (SimPy / `job-shop-scheduling` libs) or Commercial (AnyLogic, FlexSim, Simio) – *Python preferred for CI/CD integration*.

**Model Parameterization (100% Data-Driven):**
*   **Entities:** Jobs (Attributes: Routing, Priority, Due Date, Family).
*   **Resources:** Machines (Capacity, Reliability: MTTF/MTTR from logs), Operators (Skills, Shift Calendars).
*   **Process Logic:** Routing probabilities, **SDST Logic** (Lookup $S_{ij}$ matrix), **Dispatching Logic** (Pluggable: Baseline FCFS/EDD vs. Strategy 3 CPI vs. Strategy 2 Bottleneck Sequence).
*   **Arrival Generator:** Non-homogeneous Poisson Process fitted to `Job Released` timestamps.

**Experimental Design (Scenario Testing):**
| Scenario | Description | Stress Test Target |
| :--- | :--- | :--- |
| **Baseline (Historical Replay)** | Feed *exact* historical arrival sequence & breakdowns. Validate sim output matches actual KPIs (Validation). | Model Validity. |
| **Nominal Load** | Average arrival rate, average breakdowns. | Steady-state KPI comparison (Strat 1, 2, 3 vs Baseline). |
| **High Load / Peak Season** | Arrival rate $\times 1.3$, Priority mix shift to High. | Throughput ceiling, Tardiness explosion point. |
| **Disruption Storm** | 3x Breakdown frequency, 5 "Hot Jobs" per day inserted randomly. | **Robustness/Resilience** of Strat 3 (Dynamic Dispatch) vs Strat 1 (Static Schedule). |
| **Bottleneck Shift** | Artificially slow down Grinding (new bottleneck). | Adaptability of Strat 2 (Bottleneck Optimizer detection). |
| **Estimation Error** | Simulate "Planned Durations" = Actual $\times 0.8$ (Optimistic) vs Actual $\times 1.2$ (Pessimistic). | Value of **Probabilistic Scheduling (Strat 1)** vs Deterministic. |

**Key Comparison Metrics:** *Mean Tardiness, % On-Time, Max WIP, Bottleneck Utilization, Setup Time %.* Use **Statistical Ranking (Friedman Test + Nemenyi post-hoc)** to prove significance.

### 5.2 Continuous Monitoring & Adaptive Control Framework
Deployment is not the end; it is the start of the **Process Mining Feedback Loop**.

**Architecture: "Process Mining Control Tower"**

1.  **Daily Automated Mining Pipeline (Airflow/Prefect):**

    *   Re-compute: SDST Matrices, Duration Distributions, Bottleneck Utilization, CPI Feature Importance.
    *   Detect **Concept Drift**: Kolmogorov-Smirnov test on Task Duration distributions vs. Model Training Set. Chi-square test on Routing Probabilities.

2.  **KPI Dashboard (Grafana/PowerBI) - Real-time & Daily:**
    *   *Leading Indicators:* Current Bottleneck Queue Length, "Schedule Deviation" (Actual Start vs Predictive Schedule Start), Hot Job Count.
    *   *Lagging Indicators:* Daily Tardiness, WIP Level, Setup Efficiency Ratio.

3.  **Automated Retraining & Rule Adjustment Triggers:**
    *   **Trigger A (Drift Detected):** KS-test $p < 0.01$ on Setup Times $\rightarrow$ Re-train SDST Matrix $\rightarrow$ Push new matrix to Strategy 2 Optimizer & Strategy 3 CPI.
    *   **Trigger B (KPI Breach):** Daily Tardiness > Threshold $\rightarrow$ Run "What-If" Simulation instantly (last 7 days data + current state) $\rightarrow$ Recommend temporary parameter shift (e.g., Increase $\alpha$ (Slack weight) in CPI, Reduce Batch Size Limit in Strat 2).
    *   **Trigger C (Structural Change):** New Machine Commissioned / New Job Family (>5% volume) $\rightarrow$ Full Strategy Re-optimization (Retrain Strat 1, Re-cluster Families for Strat 2).

4.  **Human-in-the-Loop Governance:**
    *   Weekly "Schedule Review" meeting: Analyst reviews *Auto-generated Insights* (e.g., "Setup time for Titanium->Steel increased 20% due to Tool Wear on CUT-01; Recommend PM").
    *   Quarterly "Strategy Tournament": Re-run DES with latest 3 months data. Champion/Challenger model deployment for CPI weights.
