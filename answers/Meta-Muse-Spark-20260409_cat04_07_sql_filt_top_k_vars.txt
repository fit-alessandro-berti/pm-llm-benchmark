To get the top K process variants and return all events for cases that follow those variants, you need 3 logical steps in DuckDB: 

1. Build the activity sequence per case using `list()` + `ORDER BY` in an aggregate
2. Find the top K sequences by frequency  
3. Semi-join back to the original log to keep only events from those cases

DuckDB makes step 1 easy because it supports `LIST(activity ORDER BY timestamp)` directly.

### DuckDB SQL Query

```sql
WITH 
-- 1. Get ordered activity sequence per case as an array
case_variants AS (
    SELECT 
        case_id,
        LIST(activity ORDER BY timestamp) AS variant_sequence
    FROM event_log
    GROUP BY case_id
),

-- 2. Count cases per variant and rank to find top K
variant_counts AS (
    SELECT 
        variant_sequence,
        COUNT(*) AS case_count,
        ROW_NUMBER() OVER (ORDER BY COUNT(*) DESC, variant_sequence) AS variant_rank
    FROM case_variants
    GROUP BY variant_sequence
),

-- 3. Identify case_ids that belong to top K variants
top_k_cases AS (
    SELECT cv.case_id
    FROM case_variants cv
    JOIN variant_counts vc 
      ON cv.variant_sequence = vc.variant_sequence
    WHERE vc.variant_rank <= ? -- replace ? with your K, e.g. 5
)

-- 4. Return all original events for those cases
SELECT el.*
FROM event_log el
JOIN top_k_cases tkc 
  ON el.case_id = tkc.case_id
ORDER BY el.case_id, el.timestamp;
```

### How it works

| CTE | Purpose |
| --- | --- |
| `case_variants` | For each `case_id`, build the **process variant** as an ordered array: `LIST(activity ORDER BY timestamp)`. DuckDB preserves order inside the aggregate. |
| `variant_counts` | Group by `variant_sequence`, count cases, and assign `ROW_NUMBER()` to rank by frequency. Added `variant_sequence` as tiebreaker for deterministic results. |
| `top_k_cases` | Keep only `case_id`s whose variant rank <= K. Replace `?` with your desired K. |
| Final `SELECT` | Join back to `event_log` to return every event from cases that match the top K variants. Final `ORDER BY` keeps events sorted within each case. |

### Notes

1. **Handling simultaneous timestamps**: If two activities in the same case have identical `timestamp`, the order inside `LIST(... ORDER BY timestamp)` is nondeterministic. Add a secondary sort like `ORDER BY timestamp, activity` or use a unique event id if you need stable ordering.
2. **Top K parameter**: DuckDB uses `?` for prepared statement parameters. If you’re running this ad-hoc, just replace `?` with an integer like `5`.  
3. **Performance**: DuckDB will push the `LIST()` aggregation and the join efficiently. For very large logs, consider creating `case_variants` as a temp table and indexing `case_id` if you reuse it.
