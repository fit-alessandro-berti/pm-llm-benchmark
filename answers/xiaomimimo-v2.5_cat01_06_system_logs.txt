## Explanation of Logic

### Case Identification
I grouped events into **5 distinct cases** based on logical units of work:

1. **DOC1**: Work on `Document1.docx` - a new Word document being created
2. **EMAIL-MEETING**: Email handling about an Annual Meeting in Google Chrome
3. **REVIEW-DRAFT**: Reviewing `Report_Draft.pdf` in Adobe Acrobat
4. **BUDGET-2024**: Editing `Budget_2024.xlsx` in Microsoft Excel
5. **REPORT-Q1**: Work on `Quarterly_Report.docx` - an existing quarterly report

**Key grouping principles:**
- Events are grouped by the **primary document/window** being worked on
- The user's workflow shows clear transitions between distinct tasks
- Each case represents a coherent work unit with beginning and end states

### Activity Naming
I translated raw system actions into **standardized, meaningful activities**:

| Raw Action | Standardized Activity | Rationale |
|------------|----------------------|-----------|
| FOCUS | Open Document/Spreadsheet | Indicates opening or focusing on a document |
| TYPING (first) | Draft Content | Initial content creation |
| TYPING (subsequent) | Edit Document/Spreadsheet | Modifications to existing content |
| SAVE | Save Document/Spreadsheet | Saving progress |
| CLICK (Open Email) | Open Email | Opening a specific email |
| CLICK (Reply) | Reply to Email | Initiating email reply |
| CLICK (Send) | Send Email | Completing email response |
| SCROLL | Read/Review Document | Navigating through document content |
| HIGHLIGHT | Annotate Document | Marking important content |
| CLOSE | Close Document | Ending work session on document |
| SWITCH | (excluded from log) | Transitional actions between cases |

- Which software tool is being used
- Which specific document/file is being worked on
- Application-specific process variations
