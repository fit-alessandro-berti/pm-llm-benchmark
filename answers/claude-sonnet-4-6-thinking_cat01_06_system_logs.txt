## Explanation of Logic

### Case Identification
Five distinct cases were identified, each centered on a **primary artifact** the user was working with. Cases are defined by document/task identity, not strict temporal contiguity — a case can be suspended and resumed:

| Case ID | Artifact | Description |
|------------|--------------------------|---------------------------------------------|
| **DOC-1** | Document1.docx | Drafting a new Word document (interleaved with other tasks) |
| **EMAIL-1** | Annual Meeting Email | Handling a complete email thread (open  read  reply  send) |
| **PDF-1** | Report_Draft.pdf | Reviewing and annotating a PDF report |
| **BUDGET-1** | Budget_2024.xlsx | Updating quarterly financial data in Excel |
| **QREPORT-1** | Quarterly_Report.docx | Working on the quarterly report (opened early, resumed later) |

**DOC-1** and **QREPORT-1** are both multi-segment cases: the user switches away to perform other tasks and returns later. The interleaving pattern (DOC-1  EMAIL-1  PDF-1  BUDGET-1  DOC-1  QREPORT-1) reflects a realistic knowledge-work session where information gathered from emails, PDFs, and spreadsheets feeds back into document writing.

### Activity Naming Decisions

| Raw Action | Context | Derived Activity Name |
|------------|---------|----------------------|
| `FOCUS` (first access) | Any app | **Open Document / Open Spreadsheet** |
| `FOCUS` (return visit) | Any app | **Resume Document** |
| `SWITCH` (arrival at new artifact) | Adobe Acrobat, Word | **Open PDF / Resume Editing** |
| `TYPING` | Microsoft Word | **Edit Document** |
| `TYPING` | Microsoft Excel | **Update Spreadsheet** |
| `TYPING` | Google Chrome (email) | **Compose Reply** |
| `SCROLL` | Google Chrome (email) | **Read Email** |
| `SCROLL` | Adobe Acrobat | **Review PDF** |
| `HIGHLIGHT` | Adobe Acrobat | **Annotate PDF** |
| `CLICK` (action-specific) | Google Chrome | **Open Email / Initiate Reply / Send Email** |
| `SAVE` | Word | **Save Document** |
| `SAVE` | Excel | **Save Spreadsheet** |
| `CLOSE` | Microsoft Word | **Close Document** |

### Key Design Decisions
1. **SWITCH events** were not treated as standalone activities. Instead, they were used to infer the **start of a new case context** (e.g., SWITCHAcrobat becomes *Open PDF* for PDF-1) or a **return to an existing case** (SWITCHWord becomes *Resume Editing* for DOC-1).
2. **Consecutive TYPING events** within the same window were kept as separate events with the same activity name, preserving temporal granularity for duration analysis.
3. **The initial brief FOCUS** on Quarterly_Report.docx (08:59:50, only 10 seconds before switching away) was retained as *Open Document* in QREPORT-1, indicating the case was started even if substantive work happened later.
