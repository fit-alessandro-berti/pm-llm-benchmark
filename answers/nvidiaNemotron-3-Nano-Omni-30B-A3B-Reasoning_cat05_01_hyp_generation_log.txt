### 2. Hypotheses for Anomalies

- **Case 1002**: The `Confirm Shipment` event occurs before `Register Order`, which violates the assumed process flow. This suggests either:
  - A **logging error** where timestamps are misaligned (e.g., system clock drift).
  - A **business exception** where logistics initiated shipment prematurely due to urgency (e.g., priority order).
  - **Data corruption** in the event log (e.g., incorrect event_id assignment or misordered entries).

- **Case 1004**: `Receive Payment` occurs before `Register Order`, which is highly unusual. Possible explanations:
  - **Pre-payment**: Customer paid before placing the order (e.g., deposit or full payment upfront).
  - **System integration issue**: Payment system and order system are out of sync.
  - **Fraud or duplicate record**: Payment recorded for a non-existent or duplicate order.
  - **Timestamp error**: Payment timestamp is inaccurate due to server time misconfiguration.

- **Case 1003**: Missing `Validate Stock` event. This could indicate:
  - **Process bypass**: Stock check skipped due to high demand or system outage.
  - **Automated process failure**: Stock validation step not triggered or failed silently.
  - **Manual override**: Warehouse staff shipped without verification.

- **Case 1004**: `Confirm Shipment` has `shipment_scheduled=N` but `Ship Goods` follows. This suggests:
  - **Inconsistent data**: The shipment scheduling status was not updated correctly.
  - **Process deviation**: Shipment initiated without formal scheduling (e.g., emergency or expedited handling).
  - **System bug**: Confirmation step failed to update the `shipment_scheduled` flag.

### 3. Recommended SQL Queries for Further Investigation

#### A. Investigate Event Order and Timestamps in Case 1002
```sql
SELECT 
    case_id,
    event_id,
    activity,
    timestamp,
    resource,
    additional_info
FROM order_event_log
WHERE case_id = 1002
ORDER BY timestamp;
```
> *Purpose*: Verify if the actual sequence of events matches the `event_id` order or if timestamps are misaligned.

#### B. Check for Pre-Payment or Out-of-Order Events in Case 1004
```sql
SELECT 
    o.case_id,
    o.activity,
    o.timestamp,
    r.role,
    r.department,
    o.additional_info
FROM order_event_log o
JOIN resources r ON o.resource = r.resource_id
WHERE o.case_id = 1004
ORDER BY o.timestamp;
```
> *Purpose*: Examine the full sequence of events and roles involved to detect anomalies in process flow.

#### C. Validate Stock Check Presence and Timing Across Cases
```sql
SELECT 
    case_id,
    MAX(CASE WHEN activity = 'Validate Stock' THEN timestamp END) AS stock_check_time,
    MAX(CASE WHEN activity = 'Ship Goods' THEN timestamp END) AS shipment_time
FROM order_event_log
WHERE activity IN ('Validate Stock', 'Ship Goods')
GROUP BY case_id
HAVING shipment_time < stock_check_time OR stock_check_time IS NULL;
```
> *Purpose*: Identify cases where shipment occurs before stock validation or where stock validation is missing.

#### D. Check Shipment Scheduling Status vs. Shipment Execution
```sql
SELECT 
    o.case_id,
    o.activity,
    o.timestamp,
    o.additional_info
FROM order_event_log o
WHERE o.activity = 'Confirm Shipment'
  AND o.additional_info LIKE '%shipment_scheduled=N%';
```
> *Purpose*: Find all instances where shipment confirmation indicates not scheduled, yet shipment occurs afterward.

#### E. Cross-Reference Orders and Event Logs for Payment-Order Mismatches
```sql
SELECT 
    o.case_id,
    o.activity,
    o.timestamp,
    ord.order_value,
    ord.order_type,
    o.additional_info
FROM order_event_log o
JOIN orders ord ON o.case_id = ord.case_id
WHERE o.activity = 'Receive Payment'
  AND o.case_id NOT IN (
    SELECT case_id FROM order_event_log WHERE activity = 'Register Order'
  );
```
> *Purpose*: Identify payment events occurring without a corresponding order registration.
