To achieve this in DuckDB, we can use Common Table Expressions (CTEs) to sequentially build the variants, rank them by frequency, and then filter the original event log. 

Below is the SQL query. Note that I have used a placeholder `:K` for the top K variants; you can replace this with a specific integer (e.g., `5`).

```sql
WITH case_variants AS (
    -- 1. Construct the ordered sequence of activities for each case
    SELECT 
        case_id, 
        list(activity ORDER BY timestamp) AS variant
    FROM event_log
    GROUP BY case_id
),
variant_counts AS (
    -- 2. Group by sequence to identify unique variants and their frequency
    SELECT 
        variant, 
        count(*) AS frequency
    FROM case_variants
    GROUP BY variant
),
top_k_variants AS (
    -- 3. Determine the top K variants by frequency
    SELECT variant
    FROM variant_counts
    ORDER BY frequency DESC
    LIMIT :K
),
filtered_cases AS (
    -- Identify the case_ids that belong to those top K variants
    SELECT cv.case_id
    FROM case_variants cv
    JOIN top_k_variants tkv ON cv.variant = tkv.variant
)
-- 4. Return all original events belonging only to those filtered cases
SELECT 
    el.case_id, 
    el.activity, 
    el.timestamp
FROM event_log el
JOIN filtered_cases fc ON el.case_id = fc.case_id
ORDER BY el.case_id, el.timestamp;
```

### Explanation of the Query:

1.  **`case_variants`**: We use the DuckDB `list()` aggregation function. By adding `ORDER BY timestamp` inside the `list()` function, we ensure that the activities are collected in the exact order they occurred for each `case_id`. This creates a "process variant" as a list object.
2.  **`variant_counts`**: We treat the list object as a single value to group by. This allows us to count how many distinct `case_id`s share the exact same sequence of activities.
3.  **`top_k_variants`**: We sort the unique variants by their frequency in descending order and use `LIMIT` to keep only the most frequent ones.
4.  **`filtered_cases`**: We join the list of top variants back to our `case_variants` table to retrieve a list of all `case_id`s that follow those specific patterns.
5.  **Final Select**: We perform a final join between the original `event_log` and the `filtered_cases` list. This effectively removes all events associated with "rare" variants, returning only the raw data for the most common processes.

### Complexity Analysis:
- **Time Complexity**: The most expensive operations are the aggregations and the joins. DuckDB's columnar execution engine handles the `list()` aggregation and joins very efficiently.
- **Space Complexity**: The `case_variants` CTE stores one row per case, making it significantly smaller than the original event log.
