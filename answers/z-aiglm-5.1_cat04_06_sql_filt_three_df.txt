To solve this problem, we can use DuckDB's window functions to identify the specific sequence of activities and calculate the time elapsed between them. Then, we can filter out the cases that match the timing condition and exclude all of their events from the final result.

Here is the step-by-step approach:
1. Use the `LEAD` window function to look ahead at the next two activities and their timestamps for each event, partitioned by `case_id` and ordered by `timestamp`. This allows us to check for direct succession.
2. Identify the rows where the current activity is 'Approve Purchase Requisition', the next is 'Create Order', and the one after that is 'Confirm Order'.
3. For these specific matching sequences, check if the time difference between 'Create Order' and 'Confirm Order' is strictly more than 5 days. We can use interval arithmetic for exact day/time comparisons.
4. Collect the `case_id`s of these sequences into a CTE (Common Table Expression) and exclude any events belonging to these cases in the final `SELECT`.

Here is the DuckDB SQL query:

```sql
WITH SequenceInfo AS (
    SELECT 
        case_id,
        activity,
        LEAD(activity, 1) OVER (PARTITION BY case_id ORDER BY timestamp) AS next_activity,
        LEAD(activity, 2) OVER (PARTITION BY case_id ORDER BY timestamp) AS next_next_activity,
        LEAD(timestamp, 1) OVER (PARTITION BY case_id ORDER BY timestamp) AS create_order_ts,
        LEAD(timestamp, 2) OVER (PARTITION BY case_id ORDER BY timestamp) AS confirm_order_ts
    FROM event_log
),
ExcludedCases AS (
    SELECT DISTINCT case_id
    FROM SequenceInfo
    WHERE activity = 'Approve Purchase Requisition'
      AND next_activity = 'Create Order'
      AND next_next_activity = 'Confirm Order'
      AND (confirm_order_ts - create_order_ts) > INTERVAL '5' DAY
)
SELECT e.*
FROM event_log e
WHERE NOT EXISTS (
    SELECT 1 
    FROM ExcludedCases ec 
    WHERE ec.case_id = e.case_id
);
```

### Explanation:
- **`SequenceInfo` CTE**: The `LEAD` functions look 1 and 2 steps ahead within each `case_id` ordered by `timestamp`. If an event is 'Approve Purchase Requisition', the `next_activity` will capture 'Create Order' and `next_next_activity` will capture 'Confirm Order'. This directly enforces the "no other event in between" rule. We also capture the respective timestamps of those future events.
- **`ExcludedCases` CTE**: We filter `SequenceInfo` to find exact pattern matches. We then enforce the 5-day interval condition using `(confirm_order_ts - create_order_ts) > INTERVAL '5' DAY`. Using interval math ensures that the comparison evaluates to true only if the duration is strictly greater than 5 full days (handling hours/minutes correctly). The `DISTINCT` keyword guarantees we collect each violating `case_id` exactly once.
- **Final `SELECT`**: We retrieve all columns from `event_log` but apply a `NOT EXISTS` filter to remove any events that belong to a `case_id` present in the `ExcludedCases` list.
