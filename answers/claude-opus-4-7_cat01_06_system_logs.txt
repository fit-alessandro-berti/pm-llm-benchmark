## Explanation of Logic

### Case Identification
I grouped events into cases based on the **artifact being worked on** (document, email, spreadsheet, PDF). Each unique artifact represents a logical unit of work:

- **C1 – Draft Document1 (`Document1.docx`)**: A single case even though the user switched away and came back. Process mining benefits from keeping the full lifecycle of a work item in one case, so I re-attached the 09:06 resumption to C1.
- **C2 – Handle Annual Meeting Email**: All interactions within the email thread (open  scroll/read  reply  send) belong together.
- **C3 – Review Report Draft PDF**: PDF review session in Adobe Acrobat.
- **C4 – Update Budget 2024**: Spreadsheet editing session.
- **C5 – Compile Quarterly Report**: The final editing session on `Quarterly_Report.docx`.

### Activity Name Standardization
Low-level OS actions were mapped to business-meaningful activities:

| Raw Action | Standardized Activity |
|------------|----------------------|
| FOCUS (first time on artifact) | Open Document / Open Spreadsheet / Open PDF |
| FOCUS (returning to artifact) | Resume Document |
| TYPING in doc/spreadsheet | Edit Content / Edit Data |
| TYPING in email reply | Compose Reply |
| SAVE | Save Document / Save Spreadsheet |
| SWITCH to email client | Open Email Client |
| CLICK "Open Email" | Open Email |
| SCROLL in email | Review Email |
| SCROLL in PDF | Review Document |
| CLICK "Reply" | Initiate Reply |
| CLICK "Send" | Send Email |
| HIGHLIGHT | Annotate/Highlight |
| CLOSE | Close Document |

### Narrative
The log tells a coherent story of a knowledge worker's morning: the user started drafting a new document, got interrupted to reply to a meeting email, briefly reviewed a PDF report, updated the budget spreadsheet, returned to finish and close the draft document, and finally worked on the quarterly report file. This case structure enables analysis of task-switching behavior, rework, and activity durations per artifact type.
