# Analysis and Optimization of Resource Assignment at TechSolve Solutions

## 1. Analyzing Resource Behavior and Assignment Patterns

To analyze TechSolve’s resource assignment practices, we would leverage the event log data to examine several key metrics:

- **Workload Distribution:** Measure the number of tickets assigned to each agent and tier to identify overloaded or underutilized agents. This helps detect imbalances where some agents handle significantly more tickets than others.
- **Processing Times:** Calculate the average time taken for each activity by agents across tiers. For instance, L1 agents should resolve P3 tickets quickly if they are suitable for their skill level; delays indicate potential mismatches.
- **First-Call Resolution Rate (FCRR):** Track the percentage of tickets resolved at the initial tier (typically L1). A low FCRR suggests that L1 agents may lack the skills or empowerment to resolve tickets, leading to unnecessary escalations.
- **Skill Utilization:** Analyze how often agents are assigned tickets matching their documented skills. For example, if an L2 agent with 'App-CRM' skills frequently handles 'Network' tickets, this indicates misassignment.

Process mining techniques would be crucial in revealing the actual process flow:

- **Resource Interaction Analysis:** By mapping handovers between agents, we can identify frequent reassignment patterns. For instance, if tickets frequently move from L1 to L2 and back, it may indicate that L1 agents are escalating prematurely.
- **Social Network Analysis:** Visualizing the network of ticket handovers can highlight bottlenecks (e.g., certain agents receiving many tickets, causing delays) and inefficient collaboration patterns.
- **Role Discovery:** This technique helps identify whether agents are operating outside their designated tiers due to workarounds for overloaded systems.

Comparing this data against the intended assignment logic (e.g., round-robin within tiers) can reveal discrepancies. For example, if the intended logic assumes that 80% of P3 tickets should be resolved at L1, but the log shows only 50% are resolved there, it signals a gap in L1 capabilities or misassignment.

## 2. Identifying Resource-Related Bottlenecks and Issues

Based on the analysis, several bottlenecks can be pinpointed:

- **Skill Shortages:** For example, if tickets requiring 'Networking-Firewall' skills frequently wait in queues for L2/L3 agents, this indicates a shortage of such specialists.
- **Frequent Reassignments:** Count the number of reassignments per ticket. Suppose an average ticket is reassigned 1.5 times, adding 30 minutes per reassignment. This could explain why SLAs for P2 tickets are breached—each reassignment introduces delays.
- **Initial Assignment Errors:** Analyze cases where tickets are initially assigned to agents lacking the required skills. For instance, P2 'Network' tickets assigned to L1 agents without networking skills may lead to immediate escalation, wasting time.
- **Overloaded vs. Underutilized Agents:** Identify agents consistently handling more than 20% above the average ticket volume, which can cause burnout and delays. Conversely, agents with consistently low ticket volumes may be underutilized due to skill mismatches or inefficient routing.
- **SLA Breach Correlation:** For P2 and P3 tickets, correlate breaches with the initial assignment tier and agent skills. Suppose 65% of breached P2 tickets were initially assigned to L1 agents without the required skills—this directly links misassignment to SLAs.

**Quantification of Impact:**
- Average delay per reassignment: If each reassignment adds 25 minutes (from log timestamps), a ticket reassigned twice incurs 50 minutes of extra time, likely causing SLA breaches for P2 tickets (which may have a 2-hour SLA).
- SLA breaches linked to skill mismatch: If 70% of P2 breaches involve tickets requiring skills not present in the initially assigned agent, this is a critical area for improvement.

## 3. Root Cause Analysis for Assignment Inefficiencies

Potential root causes include:

- **Deficient Assignment Rules:** The current round-robin approach ignores both agent skills and current workload, leading to misassignments and overload.
- **Inaccurate Skill Profiles:** Agent skill documentation may be outdated or exaggerated (e.g., an agent lists 'Database-SQL' but lacks proficiency, leading to reassignments).
- **Poor Ticket Categorization:** Tickets may be misclassified at creation, causing incorrect skill requirements to be flagged. For example, a 'Software-App' ticket might actually require 'Database-SQL' skills if the issue involves corrupted data.
- **Lack of Real-Time Visibility:** Dispatchers or L1 agents may not see real-time agent availability or workload, leading to poor assignment decisions.
- **Insufficient L1 Training:** L1 agents may lack the skills or confidence to resolve tickets that could be resolved at their level, leading to excessive escalations.

**Process Mining Techniques for Root Cause Analysis:**
- **Variant Analysis:** Compare cases with smooth flows (e.g., resolved at L1 without reassignment) vs. complex flows (multiple reassignments). For instance, variants where the initial category is 'Software-App' but the required skill is later changed to 'Database-SQL' might indicate categorization errors.
- **Decision Mining:** If the log includes escalation decisions, analyze the conditions under which L1 agents escalate. For example, if escalation decisions correlate with ticket age (e.g., escalating after only 5 minutes for a P3 ticket), it may indicate low L1 confidence or training gaps.

## 4. Developing Data-Driven Resource Assignment Strategies

### Strategy 1: Skill-Based Routing with Proficiency Weighting
**Issue Addressed:** L2/L3 specialists handling tasks that L1 agents could resolve.
**Data Leveraged:** Historical data on resolution rates by tier and skill. For example, if 65% of P3 'App-CRM' tickets are resolved by L1, these should default to L1 unless complex.
**Implementation:** Assign tickets to the lowest tier with sufficient aggregated proficiency. For example, use a scoring system where an L1 agent with a proficiency score of 8/10 for 'App-CRM' gets priority over an L2 agent with a score of 9/10.
**Data Required:** Agent skill proficiency scores (not just binary presence), historic resolution rates by skill/tier.
**Expected Benefits:** Reduce escalations by 30%, improve L1 FCRR from 50% to 75%, and free up L2/L3 for complex issues.

### Strategy 2: Workload-Aware Dynamic Assignment Algorithm
**Issue Addressed:** Uneven workload distribution, leading to overloaded and underutilized agents.
**Data Leveraged:** Real-time queue data and current agent workload from the event log. For example, if Agent X has 5 tickets in progress while Agent Y has only 1, prioritize assigning to Agent Y.
**Implementation:** Use a queue-based algorithm that assigns tickets to the least busy agent with the required skills. This can be integrated with the existing ticketing system.
**Data Required:** Real-time or near-real-time data on agent workload, skill matching, and ticket priority.
**Expected Benefits:** Reduce average resolution time by 20%, balance workload so that no agent is overloaded by more than 15% of the average, and improve agent satisfaction.

### Strategy 3: Predictive Assignment Using Ticket Description Analysis
**Issue Addressed:** Incorrect initial assignment due to poor categorization or mislabeled required skills.
**Data Leveraged:** Text of ticket descriptions and titles, along with historical skill requirements for similar tickets. For example, tickets containing "firewall" or "ACL" are 90% likely to need 'Networking-Firewall'.
**Implementation:** Use NLP to classify tickets at creation, automatically suggesting required skills. Then, route to agents with matching skills and available capacity.
**Data Required:** Historical ticket descriptions, required skills, and resolution paths. Initial training data could be the past year’s log.
**Expected Benefits:** Reduce reassignments by 40%, improve initial assignment accuracy to 90% for skill requirements, and decrease time-to-resolution for correctly routed tickets.

## 5. Simulation, Implementation, and Monitoring

**Simulation Approach:**
Before implementation, use a discrete-event simulation model based on the mined process and resource data. For example, simulate the impact of the workload-aware algorithm on resolution times by feeding historical ticket arrivals and agent characteristics into the model. Metrics to simulate include:
- Average resolution time per ticket
- SLA compliance rate
- Agent workload balance

**Implementation Plan:**
1. Pilot the strategies in one support category (e.g., 'Software-App') to validate the models.
2. Gradually roll out, starting with predictive assignment for ticket creation, followed by skill-based routing, and finally the dynamic algorithm.
3. Train agents and dispatchers on the new system to ensure buy-in.

**Monitoring Plan:**
- **KPIs to Track:**
  - Reassignment rate: Percentage of tickets reassigned at least once (target: reduce from 40% to 20%).
  - SLA compliance rate: Percentage of tickets resolved within SLA (target: improve from 75% to 90% for P2/P3).
  - Agent utilization rate: Percentage of time agents are actively working on tickets (target: maintain 70-80% without overloading).
- **Dashboards:**
  - Real-time view of agent workload heatmaps (traffic light system for over/under utilization).
  - Flow charts highlighting frequent reassignment paths.
  - Skill bottleneck alerts (e.g., if no available agents with 'Security-IAM' skills, trigger a notification).
- **Continuous Process Mining:**
  - Monthly reviews of the mined process to detect new bottlenecks or deviations from the expected flow.
  - Comparison of variants before and after implementation to ensure the strategies are working as intended.

By combining these strategies with continuous monitoring, TechSolve can systematically address inefficiencies, improve resource utilization, and achieve higher SLA compliance.
