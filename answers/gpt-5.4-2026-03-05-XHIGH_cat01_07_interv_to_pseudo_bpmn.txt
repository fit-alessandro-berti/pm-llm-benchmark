Below is a BPMN-like textual process model that reconciles the interview accounts.

## 1) Pools / Lanes

### External Pool
- **Supplier**

### Internal Pool: Invoice Approval Process
- **Accounts Payable Clerk / AP Team** (Mary)
- **Accounting System**
- **Purchasing Department / Purchasing Officer** (Dan)
- **Department Manager** (Carlos)
- **AP Supervisor** (Rita)
- **Supplier Relationship Manager** (Karen)
- **Optional ad hoc participant:** **Department Head** for retrospective PO cases

---

## 2) Main process flow

**Modeling note:** The interviews overlap on who performs the detailed PO comparison. To reconcile this, the model treats:
- **Purchasing** as doing the **business/receipt confirmation and initial PO comparison**
- **AP** as doing the **formal line-item/system match** before manager approval

### Core flow

| ID | BPMN-like element | Lane | Description | Next |
|---|---|---|---|---|
| S1 | Message Start Event | AP Clerk | **Invoice received** from Supplier (usually by email) | T1 |
| T1 | Task | AP Clerk | **Check invoice for valid PO number and basic completeness** | G1 |
| G1 | Exclusive Gateway | AP Clerk | **Valid PO present?** | Yes -> T2 / No -> EX1 |
| T2 | Task | AP Clerk + Accounting System | **Log invoice in accounting system** | T3 |
| T3 | Task | AP Clerk | **Forward invoice to Purchasing** for confirmation | T4 |
| T4 | Task | Purchasing | **Verify goods/services received and compare invoice to PO** (quantities, prices, delivery charges) | G2 |
| G2 | Exclusive Gateway | Purchasing | **Purchasing confirms invoice/receipt details?** | Yes -> T5 / No -> EX2 |
| T5 | Task | Purchasing | **Send confirmation back to AP** | T6 |
| T6 | Task | AP Clerk | **Match invoice line items against PO in the system** | G3 |
| G3 | Exclusive Gateway | AP Clerk | **Invoice matches PO in system?** | Yes -> T7 / No -> EX3 |
| T7 | Task | Accounting System | **Send automatic approval request / notification** to Department Manager | T8 |
| T8 | User Task | Department Manager | **Review invoice, PO, purchasing confirmation, and notes/budget context** | G4 |
| G4 | Exclusive Gateway | Department Manager | **Manager decision?** | Approve -> T9 / Hold-Investigate -> EX4 / Reject -> E2 |
| T9 | Task | AP Clerk | **Schedule invoice for payment** / add to next payment cycle | E1 |
| E1 | End Event | AP Clerk | **Invoice approved and scheduled for payment** | — |
| E2 | End Event | AP Clerk | **Invoice rejected / not paid** | — |

## 3) Exception / variation paths

These can be modeled as expanded exception branches or subprocesses.

### EX1. Missing or invalid PO
**Trigger:** G1 = No, or Purchasing later discovers the PO is missing/invalid.

1. **AP Clerk** requests missing PO/details or a revised invoice from the **Supplier**.
2. Invoice is placed **on hold**.
3. **Optional:** **Supplier Relationship Manager** contacts supplier to obtain PO or revised invoice.
4. **Gateway:** **Valid PO obtained?**
   - **Yes** -> return to **T1** (re-validate), then continue
   - **No** -> **AP Supervisor** reviews exception
5. **AP Supervisor** coordinates with **Department Head** to:
   - create a **retrospective PO**, or
   - decide the invoice should be **rejected**
6. **Gateway:** **Retrospective PO created?**
   - **Yes** -> return to **T1/T2**
   - **No** -> **E2 Rejected**

---

### EX2. Purchasing discrepancy / goods-receipt issue
**Trigger:** G2 = No

Typical reasons mentioned:
- extra units billed
- wrong pricing
- incorrect delivery charges
- goods/services receipt not confirmed
- missing information

1. **Purchasing** contacts the **Supplier directly** or asks **AP** to hold the invoice pending correction.
2. **Optional:** **Supplier Relationship Manager** checks supplier contract / agreed terms and contacts supplier.
3. **Gateway:** **Issue resolved at operational level?**
   - **Yes** -> **AP** updates/revalidates the invoice if needed, then return to **T4**
   - **No** -> escalate to **AP Supervisor**
4. **AP Supervisor** coordinates between supplier and internal departments.
5. **Gateway:** **Exception resolved?**
   - **Yes** -> return to **T4**
   - **No** -> **E2 Rejected**  
     *(Interviews also suggest invoices may stay on hold temporarily; rejection is the clearest terminal path described.)*

---

### EX3. AP match discrepancy
**Trigger:** G3 = No

Typical reasons mentioned:
- incorrect pricing
- missing items
- other mismatch between invoice and PO

1. **AP Clerk** emails **Purchasing** to clarify and places the invoice **on hold**.
2. **Optional:** **Supplier Relationship Manager** resolves pricing/contract issues with the supplier.
3. **Gateway:** **Resolved by AP/Purchasing?**
   - **Yes** -> return to **T6** (re-run AP/system match)
   - **No** -> escalate to **AP Supervisor**
4. **AP Supervisor** coordinates resolution; if needed, also handles missing-info / retrospective-PO issues.
5. **Gateway:** **Resolved?**
   - **Yes** -> return to **T6**, or to **T4** if Purchasing must reconfirm
   - **No** -> **E2 Rejected**

---

### EX4. Manager hold / investigation path
**Trigger:** G4 = Hold-Investigate

1. **Department Manager** asks **AP** to hold the invoice and investigate further.
2. **AP Clerk** investigates with **Purchasing**.
3. **Optional:** involve **Supplier Relationship Manager** (supplier/contract issue) and/or **AP Supervisor** (persistent exception).
4. **Gateway:** **Concern resolved?**
   - **Yes** -> return to **T6** as the preferred control point, then proceed through **T7-T8** again
   - **No** -> **E2 Rejected**

**Alternative variation:**  
Manager may **reject immediately** without an investigation loop.

---

## 4) Message flows

Key message/document exchanges:

- **Supplier -> AP:** invoice, revised invoice, missing PO/details
- **AP -> Supplier:** request for missing information / revised invoice
- **AP -> Purchasing:** invoice for confirmation
- **Purchasing -> AP:** confirmation, discrepancy notice, hold request
- **AP/Purchasing/Karen -> Supplier:** clarification on PO, pricing, terms
- **Accounting System -> Manager:** approval request / notification
- **Manager -> AP/System:** approve, reject, or hold/investigate
- **AP Supervisor <-> Department Head:** retrospective PO coordination (exception only)

- **Approval request / manager notification**

## 6) Ambiguities / assumptions noted

1. **Overlap between Purchasing check and AP match**  
   Dan describes a detailed comparison in Purchasing; Mary describes AP matching after Purchasing confirms. The model keeps both steps as separate controls.

2. **Missing PO may be found at more than one point**  
   AP usually checks first, but Purchasing may also detect it later. If that happens, the flow returns to **EX1**.

3. **“Hold” versus “Reject”**  
   Interviews mention both temporary holds and outright rejection. The model treats **hold** as an exception state and **reject** as the terminal end if unresolved.

4. **Scope end**  
   The interviews only describe the process up to **scheduling for payment**. Actual payment execution is outside scope.
