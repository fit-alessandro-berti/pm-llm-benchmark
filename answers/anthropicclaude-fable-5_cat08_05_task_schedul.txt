# Data-Driven Scheduling Transformation for Precision Parts Inc.
## A Process Mining-Based Analysis and Redesign Approach

---

## 1. Analyzing Historical Scheduling Performance and Dynamics

### 1.1 Log Preparation and Multi-Perspective Event Log Construction

Before any analysis, the MES log must be transformed into properly structured event logs. Critically, this scenario requires **multiple correlated views** of the same data:

- **Job-centric log (Case ID = JOB-xxxx):** Reconstructs each job's routing through the shop — the classical control-flow perspective.
- **Resource-centric log (Case ID = Machine ID):** Treats each machine as a "case," with events being the sequence of setups, processing tasks, idle intervals, and breakdowns it experiences. This view is **essential for sequence-dependent setup analysis and machine-level Gantt reconstruction** — it cannot be derived from the job view alone.
- **Operator-centric log:** Captures operator assignment patterns, multi-machine tending, and human-induced variability.

The MES log already provides lifecycle transitions (`Queue Entry`, `Setup Start/End`, `Task Start/End`), which is a gift: most process mining projects must infer waiting times, but here we can compute them **exactly** by pairing lifecycle events:

| Interval | Computation |
|---|---|
| Queue/waiting time | `Setup Start` (or `Task Start` if no setup)  `Queue Entry` |
| Setup time (actual) | `Setup End`  `Setup Start` |
| Processing time (actual) | `Task End`  `Task Start` |
| Transfer time | `Queue Entry (next op)`  `Task End (previous op)` |
| Flow time (lead time) | `Last Task End`  `Job Released` |
| Tardiness | `max(0, Completion Time  Due Date)` |

Data quality steps: deduplicate events, repair missing lifecycle pairs (e.g., a `Setup Start` without `Setup End` due to MES glitches), reconcile breakdown events (resource-level) with the jobs they interrupted (join on machine + time window), and snapshot **attribute changes over time** (e.g., JOB-7005's priority change must be modeled as a time-stamped attribute, not a static field, or all priority analyses will be biased).

### 1.2 Techniques and Metrics per Question Area

**(a) Flow times, lead times, makespan distributions**
- **Performance-annotated process discovery** (Inductive Miner / DFG with median + 90th percentile times on arcs) per product family — job shops have high routing variance, so cluster jobs by routing similarity (trace clustering on activity sequences) first to avoid spaghetti models.
- **Distributions, not averages:** Plot flow-time histograms and CDFs per product family and priority class. Compare planned vs. actual lead time; compute the **Flow Time Efficiency** = value-added (processing + necessary setup) time ÷ total flow time. In troubled job shops this is typically 5–15%, immediately quantifying the waiting-dominated nature of the problem.
- **Performance Spectrum analysis** (Denisov/van der Aalst): plot every job's segment-by-segment traversal over time. This reveals overtaking (priority jumps), batching patterns, congestion waves, and FIFO violations visually — extremely powerful for scheduling diagnosis.

**(b) Queue times per work center**
- From the lifecycle pairs above, compute waiting-time distributions **per machine, per shift, per day-of-week, per priority class**.
- Reconstruct **queue length over time** per machine: at any timestamp, queue length = count of jobs with `Queue Entry`  t and `Setup/Task Start` > t. This gives a WIP time series per station — the raw material for bottleneck and bullwhip analysis.
- Validate against **Little's Law** (WIP = throughput × flow time) per station; large deviations indicate non-stationarity or data issues.

**(c) Resource utilization decomposition**
Using the resource-centric log, build a **machine state timeline** partitioning each machine's calendar into:

`Productive (Task StartEnd) | Setup | Idle-Starved (no job in queue) | Idle-Blocked/Waiting-Operator (job in queue but no start) | Breakdown`

The distinction between **starved idle** (upstream problem) and **blocked idle** (operator/scheduling problem) is diagnostic gold and is computable by cross-referencing the queue reconstruction. Report utilization stacked-bar charts per machine; bottlenecks will show >85–90% productive+setup with persistent queues; starved machines will show idle-starved despite shop-wide WIP.

**(d) Sequence-dependent setup time mining**
This is where the resource-centric log shines:
1. For each machine, order all setup events chronologically; each setup event already references the *previous job* (see Notes field: "Previous job: JOB-6998").
2. Join job attributes (material, part family, dimensions, tooling/fixture codes) for both predecessor and successor jobs.
3. Build an empirical **setup-time matrix** S(family_i  family_j) per machine: mean, median, variance, and sample count of actual setup durations for each transition class.
4. Where part families aren't predefined, use **regression/decision trees** on (previous-job attributes, next-job attributes)  setup duration to *discover* which attribute changes drive setup time (e.g., material change costs 25 min, fixture change 40 min, same-family transition 5 min).
5. Quantify **avoidable setup time**: for each machine-day, compare actual total setup time against the setup time of the optimal (or greedily improved) sequence of the same jobs — a counterfactual lower bound. This monetizes the sequencing opportunity.

**(e) Schedule adherence and tardiness**
- **Tardiness metrics:** % tardy jobs, mean/max tardiness, total weighted tardiness (weight = priority), lateness distribution (including earliness — early completion inflates WIP/finished-goods inventory).
- **Conformance checking** against planned schedules (if MES stores planned start/end times): token replay or alignments to quantify where actual execution diverged — wrong sequence, late starts, duration overruns. Decompose tardiness root contribution: was a late job late because of (i) late release, (ii) queueing at a specific station, (iii) processing overruns, (iv) rework loops, or (v) breakdown? **Tardiness attribution per stage** = stage's excess time vs. on-time-job benchmark.
- **Planned vs. actual duration analysis:** the snippet shows systematic overruns (setup: 2023.5 min; cutting: 6066.4 min). Compute bias and variance of estimates per activity/machine/operator — biased estimates corrupt any due-date quoting and scheduling.

**(f) Disruption impact analysis**
- **Breakdowns:** for each `Breakdown Start/End`, measure MTBF/MTTR per machine, then quantify ripple effects: jobs queued at the broken machine (direct delay), jobs re-routed (detect routing deviations in that window), and downstream starvation. Compare KPIs in breakdown-affected time windows vs. matched normal windows (difference-in-differences style).
- **Hot jobs:** identify priority-change events; measure (i) the hot job's resulting flow time, and (ii) the **collateral delay** imposed on preempted/overtaken jobs (visible in the Performance Spectrum as overtaking lines). Quantify how many ordinary jobs became tardy in the wake of each hot-job insertion, including extra setups incurred by sequence interruption.

## 2. Diagnosing Scheduling Pathologies

### 2.1 Expected Pathologies and the Evidence Base

**P1 — Bottleneck resources constraining throughput.**
Evidence: machine state timelines showing one or two machines (likely Heat Treatment or a specific CNC mill) at near-saturation with long, persistent queues; queue-time Pareto (typically 70–80% of total job waiting time accrues at 1–2 stations); **blocking analysis** showing upstream machines idle-blocked and downstream machines idle-starved synchronized with bottleneck queue peaks. Quantify impact: estimated throughput loss = bottleneck unproductive time (setup + breakdown + starvation) × shop output rate, since lost bottleneck time is lost system throughput (Theory of Constraints logic).

**P2 — Poor prioritization: high-priority/near-due jobs delayed.**
Evidence: **variant analysis comparing on-time vs. tardy job cohorts** — same routings, but tardy jobs show 3–5× queue time at specific stations. Cross-tabulate queue discipline: if jobs with imminent due dates are regularly overtaken by FCFS-served jobs (visible as line-crossings in the Performance Spectrum and measurable as "slack-inversion rate" — fraction of dispatch decisions where a job with less slack waited while one with more slack was started), the local EDD/FCFS mix is demonstrably failing. Also check **priority dilution**: if >30% of jobs are flagged High, priority carries no information.

**P3 — Suboptimal sequencing inflating setup time.**
Evidence: from the setup matrix, compute actual vs. counterfactual setup time per machine (§1.2d). Identify "ping-pong" patterns: machine alternating between part families ABAB where AABB would have halved setups. Quantify: e.g., "MILL-03 spent 22% of available time in setup; greedy family-batching of the same historical workload would have required only 12% — a 10-point capacity recovery on a bottleneck."

**P4 — Downstream starvation from upstream decisions.**
Evidence: correlate (with lag) bottleneck output gaps with downstream idle-starved intervals. Cross-correlation of station-level WIP time series reveals propagation delays. Also detect cases where upstream stations processed jobs that subsequently sat for days in the bottleneck queue while starving the bottleneck of the *right* jobs — i.e., local FCFS upstream fed the bottleneck the wrong mix.

**P5 — Bullwhip / WIP oscillation.**
Evidence: WIP time series per station showing amplifying oscillations downstream; release-rate vs. WIP correlation showing uncontrolled release (jobs pushed onto the floor at order receipt regardless of congestion). Flow-time vs. WIP scatter (the operating curve) will show the shop operating far beyond the "knee," where additional WIP buys no throughput, only lead time.

**P6 — Disruption fragility.**
Evidence: KPI degradation half-life after each breakdown/hot job — how long until queue lengths and tardiness rates return to baseline. A fragile system shows multi-day ripple from single-hour breakdowns.

### 2.2 Methodological Note

The analytical mechanism throughout is **comparative**: on-time vs. tardy cohorts, disrupted vs. normal windows, actual vs. counterfactual sequences, planned vs. actual durations. Process mining's strength here is that all comparisons rest on *actual execution data*, not assumptions.

## 3. Root Cause Analysis of Scheduling Ineffectiveness

### 3.1 Candidate Root Causes and Diagnostic Tests

| Root Cause | Diagnostic Evidence from Process Mining |
|---|---|
| **Static, myopic dispatching rules** | Slack-inversion rate high; setup ping-pong; rules ignore downstream load (jobs dispatched into already-saturated next stations — measurable as queue-on-arrival at next op) |
| **No real-time shop-floor visibility** | Dispatch decisions uncorrelated with downstream queue states; long idle-blocked periods (machine free, job in queue, nobody dispatched it — coordination latency measurable as `Queue Entry  Start` gap even when machine was idle) |
| **Inaccurate duration/setup estimates** | Systematic planned-vs-actual bias (snippet shows +17.5% setup, +10.7% processing); if quoting/planning uses these biased values, every promise is structurally optimistic; variance unaccounted  no buffers |
| **Sequence-dependent setups ignored** | Setup matrix shows large transition-dependent variance, but observed sequences show no family clustering — the scheduler is blind to S(ij) |
| **Poor inter-station coordination** | Each station locally optimal but globally incoherent: upstream FCFS produces an arrival sequence at the bottleneck that maximizes its setups; transfer batches absent |
| **No disruption-response protocol** | Hot jobs inserted at queue heads everywhere (maximum collateral setups and delay) rather than only where critical; breakdowns trigger no re-routing despite alternative machines existing (visible as queues persisting at broken machines while alternates idle) |
| **Uncontrolled order release** | Release events uncorrelated with shop load  WIP explosion, which itself destroys the predictability of all downstream estimates (vicious cycle) |

### 3.2 Separating Scheduling Logic vs. Capacity vs. Inherent Variability

This distinction determines whether the fix is software (scheduling) or capital (machines) — and process mining can adjudicate:

1. **Capacity test:** Compute aggregate demonstrated capacity per work center (productive-rate during saturated periods × available hours) vs. aggregate workload over the year. If workload > ~85–90% of true capacity at a station *even with optimal setups*, it's a structural capacity problem; no scheduling rule fixes it. Use the counterfactual setup analysis to compute capacity *after* sequencing improvements — often the "capacity problem" disappears once avoidable setup is recovered.
2. **Scheduling-logic test:** Counterfactual replay — re-sequence historical queues at each machine under an alternative rule (e.g., critical-ratio with setup awareness) keeping arrivals fixed. If tardiness/setup drops substantially, the logic was the culprit. The slack-inversion and ping-pong metrics directly measure decision quality independent of load.
3. **Variability test:** Decompose flow-time variance: processing-time CV (inherent), arrival-pattern CV (release policy), breakdown contribution (MTBF/MTTR), and queue-discipline contribution. Kingman's equation logic applies: if processing CV is intrinsically high (true custom work), waiting time will be high at any utilization — the remedy is then buffer management and load control, not sequencing finesse.

Typically all three coexist; the value of the analysis is **quantified attribution** (e.g., "40% of tardiness attributable to sequencing decisions, 35% to uncontrolled release/WIP, 15% to breakdowns, 10% irreducible variability"), which prioritizes the intervention portfolio.

## 4. Advanced Data-Driven Scheduling Strategies

### Strategy 1 — Composite Dynamic Dispatching with Setup Awareness (ATCS-based, mined parameters)

**Core logic.** Replace local FCFS/EDD with a composite priority index evaluated at every dispatch decision. The foundation is the **Apparent Tardiness Cost with Setups (ATCS)** rule, extended with downstream-load awareness:

For each job *j* in the queue of machine *m* (last job processed: *l*):

**I(j) = (w_j / p_j) · exp(max(0, d_j  t  p_j  R_j) / (k·p)) · exp(s(lj) / (k·s)) · D(j)**

where:
- *w_j* = priority weight, *p_j* = expected processing time (from mined distribution, not nominal), *d_j* = due date, *t* = now
- **R_j = mined expected remaining downstream time** (remaining processing + setups + *current queue-state-dependent expected waiting* at downstream stations) — this is where process mining replaces guesswork: R_j comes from historical waiting-time models conditioned on real-time queue lengths
- **s(lj)** = expected sequence-dependent setup from the mined setup matrix
- **D(j)** = downstream-load damping factor: discounts jobs whose next station is heavily congested (avoid feeding a flooded buffer; prefer jobs feeding starved stations)
- k, k = scaling parameters **tuned via simulation** (§5) using mined p and s

**Differentiated policy at the bottleneck:** the bottleneck gets stronger setup-economy weighting (k smaller  favor same-family continuation) because bottleneck time is system throughput; non-bottlenecks weight due-date urgency more, since their capacity is cheap.

**Process mining linkage:** the setup matrix supplies s(lj); duration distributions supply unbiased p_j; queue/waiting models supply R_j; slack-inversion and ping-pong diagnostics validate the rule corrects the observed pathologies (P2, P3).

**Expected impact:** large reduction in weighted tardiness (urgency-aware), 5–10 points of bottleneck capacity recovered (setup-aware), reduced downstream starvation (load-aware). Implementable as an MES advisory screen at each station — low technical risk, fast deployment.

**Companion lever — Workload-Controlled Release (CONWIP/LUMS-style):** informed by the mined operating curve (P5), gate job release so that bottleneck-adjusted shop load stays at the curve's knee. Released-late-but-flowing-fast beats released-early-and-stagnating. This attacks WIP and lead-time variance directly and makes every downstream prediction more accurate.

### Strategy 2 — Predictive Scheduling with a Mined Digital Shadow

**Core logic.** Move from reactive dispatching to **rolling-horizon predictive scheduling**: every few hours (and on every disruption event), a finite-capacity scheduler generates a shop-wide schedule for the next 24–72 hours, using *probabilistic* inputs mined from the logs rather than nominal standards.

Components:
1. **Probabilistic duration models:** fit distributions (lognormal typically) for processing and setup times, conditioned on covariates found significant in the historical data — machine, operator, part family, job complexity attributes. Use quantiles (e.g., P70) rather than means for commitments, with risk-tiered buffering by job priority.
2. **Predictive breakdown/availability models:** from mined MTBF/MTTR per machine (and degradation signals if sensor data exists — e.g., processing-time drift on a machine preceding failures, detectable in the log itself), incorporate expected availability into the horizon: schedule less load on failure-prone machines before predicted maintenance windows; pre-position alternate routings.
3. **Online lead-time prediction:** a predictive monitoring model (gradient boosting or LSTM on running-case prefixes + shop state features: queue lengths, bottleneck load, WIP) predicts each open job's completion time and **tardiness risk** continuously. Jobs crossing a risk threshold trigger expediting *before* they are late — converting tardiness from a lagging surprise into a managed leading indicator. The same model powers **reliable due-date quoting** for new orders (quote = predicted flow time at current load + risk buffer), attacking the "unpredictable lead times" complaint at its root.
4. **Event-driven rescheduling policy:** define which disruptions warrant full reschedule (bottleneck breakdown, hot job touching the bottleneck) vs. local repair (right-shift / swap), to avoid schedule nervousness. Mined disruption-ripple analysis (§1.2f) calibrates these thresholds.

**Addresses:** inaccurate estimates, disruption fragility, unpredictable lead times, invisible emerging bottlenecks.
**Impact:** realistic schedules that hold; proactive expediting cuts tardiness; due-date quotes with stated confidence levels improve customer trust; predicted bottleneck shifts enable pre-emptive load balancing.

### Strategy 3 — Setup-Optimized Sequencing and Intelligent Family Batching at Bottlenecks

**Core logic.** Treat sequencing at setup-heavy machines (especially bottlenecks and batch processes like Heat Treatment) as a constrained optimization problem rather than a dispatching decision.

1. **Mine the setup structure:** from the setup matrix and attribute regression (§1.2d), derive a compact transition-cost model and identify natural **setup families** (clusters of jobs with near-zero intra-family setup). Identify which attribute changes are expensive (fixture > material > program-only).
2. **Rolling sequence optimization at the bottleneck:** every shift (or on significant queue change), solve a sequencing problem over the current queue + jobs predicted to arrive within the horizon (arrival predictions from Strategy 2): minimize  s() + · w_j·T_j — i.e., a setup-minimizing tour (asymmetric-TSP structure) *constrained by tardiness penalties*, so batching never blindly sacrifices urgent jobs. Solvable with heuristics/metaheuristics (savings + local search, or CP) in seconds for realistic queue sizes.
3. **Dynamic batch formation with a coverage rule:** continue a family run while compatible jobs are queued **unless** an out-of-family job's critical ratio falls below a threshold or family run-length exceeds a cap (caps tuned in simulation to prevent the classic batching pathology of starving non-family jobs). For true batch resources (Heat Treatment furnace), mine historical batch compositions and idle-fire incidents to set optimal "wait-to-fill vs. fire-now" rules.
4. **Coordinated upstream feeding:** instruct upstream stations (via Strategy 1's D(j) factor) to *prefer* releasing jobs belonging to the bottleneck's current/next family — global coherence replacing local myopia, directly fixing pathology P4.
5. **Hot-job insertion protocol:** insert urgent jobs at the **family-compatible position** in the bottleneck sequence where the marginal setup + collateral delay cost is lowest, rather than at queue head — mined hot-job collateral analysis (§1.2f) quantifies the savings.

**Addresses:** P3 (setup waste), P1 (bottleneck capacity), P4 (mis-sequenced feeding), hot-job collateral damage.
**Impact:** recovery of the quantified avoidable setup time (often 8–12% of bottleneck capacity  equivalent throughput gain shop-wide), shorter bottleneck queues  lower WIP and lead time, without the tardiness penalty of naive batching.

**Integration note:** the three strategies are layered, not alternatives — Strategy 3 governs the bottleneck sequence, Strategy 1 governs non-bottleneck dispatching coherently with it, Strategy 2 provides the predictive envelope (durations, arrivals, risk, quoting) that parameterizes both, and load-controlled release stabilizes the whole system.

## 5. Simulation, Evaluation, and Continuous Improvement

### 5.1 Process-Mining-Parameterized Discrete-Event Simulation (Digital Twin)

Build a DES model of the shop whose parameters are **mined, not assumed**:

- **Routings:** product-family routing variants and their probabilities from trace clustering.
- **Durations:** fitted processing/setup distributions per machine × family × (optionally operator), including the mined sequence-dependent setup matrix as a transition-cost lookup.
- **Arrivals:** order inter-arrival and mix patterns from release events, including seasonality; hot-job arrival rate and characteristics.
- **Disruptions:** MTBF/MTTR distributions per machine; repair-time variability.
- **Resources:** shift calendars, operator-machine qualification matrix mined from the operator dimension.
- **Baseline scheduling behavior:** encode the current FCFS/EDD mix as observed (including its measured deviations — e.g., informal expediting visible in slack inversions).

**Validation first:** replay the model against a held-out historical period and verify it reproduces the empirical KPI distributions (flow time, tardiness rate, station utilizations, queue lengths) within tolerance. Only a validated twin supports credible what-if comparison.

**Experimental design:**

| Dimension | Scenarios |
|---|---|
| Strategies | Baseline / S1 alone / S1+S3 / S1+S2+S3 / +load-controlled release |
| Load | Mined average; +15%, +30% demand; mix shifts toward setup-heavy families |
| Disruptions | Mined breakdown rates; 2× breakdown frequency; bottleneck-machine extended outage |
| Hot jobs | Mined rate; 2–3× hot-job rate |
| Robustness | Duration-estimate degradation (test sensitivity to forecast error); parameter sweeps for k, k, batching caps, WIP limits |

**Outputs:** full KPI distributions (not means) — weighted tardiness, % on-time, P90 lead time, WIP, bottleneck utilization, total setup hours, schedule nervousness (reschedule frequency/magnitude) — with replications and confidence intervals. Select the strategy configuration that **dominates under stress scenarios**, not just nominal conditions; robustness matters more than peak performance in a disruption-prone shop. The simulation also doubles as the **tuning engine** for all strategy parameters and as a training environment if reinforcement-learning-based dispatching is explored later.

### 5.2 Continuous Monitoring and Adaptation Framework

A closed-loop **Mine  Decide  Act  Monitor** cycle:

1. **Real-time KPI dashboard (daily/weekly):** tardiness funnel, station queue times, setup ratio at bottlenecks, WIP vs. target band, slack-inversion rate, schedule-adherence/conformance score, prediction accuracy (predicted vs. actual completion — the health metric of Strategy 2's models).
2. **Statistical drift detection:** control charts (CUSUM/EWMA) on key metrics; **concept-drift detection on the process itself** — new routing variants, shifting setup-matrix values (new tooling changes transition costs), processing-time distribution shifts (machine degradation, new operators), changing product mix. Alert when the mined parameters feeding the scheduler diverge from current reality.
3. **Automated model refresh:** retrain duration/setup/lead-time-prediction models on a rolling window when drift is confirmed; re-run simulation-based parameter tuning (k, k, WIP caps, batch limits) quarterly or on-trigger.
4. **Bottleneck migration watch:** the bottleneck *will* move once the current one is relieved; monthly utilization/queue Pareto re-ranking with automatic re-targeting of Strategy 3 to the new constraint.
5. **Decision audit loop:** continuously compare actual dispatch decisions against the recommended ones (conformance checking of the *scheduling policy itself*). Where operators systematically override, investigate — overrides often encode tacit knowledge (e.g., a setup the matrix underestimates) that should be fed back into the models, or they reveal training/trust gaps to address.
6. **Governance:** monthly performance review against the simulated business case (tardiness penalty reduction, capacity recovered, WIP carrying cost), with a documented change process for scheduling-logic modifications so the system improves deliberately rather than drifting.

### Closing Summary

The MES log makes Precision Parts' problems fully measurable: process mining converts a year of execution history into a quantified diagnosis (where waiting accrues, what sequencing costs in setups, how disruptions ripple, what prioritization failures cause tardiness) and into the *parameters* of the cure — setup matrices, duration distributions, queue models, disruption profiles. Layered on this foundation, setup-aware composite dispatching, predictive rolling-horizon scheduling with risk-based expediting and reliable quoting, and bottleneck-focused setup-optimized sequencing — all stress-tested in a mined-data digital twin and governed by a continuous drift-aware monitoring loop — directly attack tardiness, WIP, lead-time unpredictability, and capacity waste at their evidenced root causes.
