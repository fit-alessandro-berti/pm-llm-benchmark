## Assumptions Made

1. **Case identifier**
   - `order_id` is used as the **Case ID** for all events.

2. **Item picking granularity**
   - I created **one `Item Picked` event per order line**, because each line has its own timestamp and picker.

3. **Delivery granularity**
   - Since orders can have multiple shipment records, I created:
     - **`Shipment Dispatched`** per `delivery_dispatch_date`
     - **`Shipment Arrived`** per `delivery_arrival_date`

4. **Payments linked to orders**
   - `Payments` were linked to `Orders` via:
     - `Payments.invoice_id -> Invoice.invoice_id -> Invoice.order_id`

5. **Resource mapping**
   - `sales_person_id`, `picker_id`, `carrier_id`, and `billing_person_id` were mapped using the `Resources` table.
   - For **payments**, no responsible actor is present in the source data, so `Resource/Actor` is marked as **Not captured**.

6. **Missing downstream records**
   - **Order 1004** has no delivery, invoice, or payment records in the provided data, so its case appears **open/incomplete** in the event log.

7. **No synthetic events added**
   - I did **not** add inferred events such as “Order Completed” or “Picking Completed”; only events directly supported by the data were included.
