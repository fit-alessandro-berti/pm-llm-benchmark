# DuckDB SQL Query: Filter Cases with Six or Fewer Events

## Understanding the Problem

We need to:
1. **Count** the number of events per `case_id`
2. **Exclude** cases where the event count exceeds six
3. **Return all event columns** for the remaining (qualifying) cases

## The Query

```sql
SELECT el.*
FROM event_log el
WHERE el.case_id IN (
    SELECT case_id
    FROM event_log
    GROUP BY case_id
    HAVING COUNT(*) <= 6
);
```

## How It Works

| Step | Clause | Purpose |
|------|--------|---------|
| 1 | `GROUP BY case_id` | Groups all events by their process instance |
| 2 | `HAVING COUNT(*) <= 6` | Filters to only those cases with **at most** six events |
| 3 | `WHERE el.case_id IN (...)` | Uses the filtered case list to return **all columns/rows** for qualifying cases |

## Alternative Approaches

### Using a Window Function

```sql
SELECT *
FROM (
    SELECT
        *,
        COUNT(*) OVER (PARTITION BY case_id) AS case_event_count
    FROM event_log
) sub
WHERE case_event_count <= 6;
```

This approach uses `COUNT(*) OVER (PARTITION BY case_id)` to annotate **every row** with its case's total event count, then filters in the outer query. It avoids a self-join/subquery lookup but introduces an extra column (`case_event_count`) in the intermediate result.

### Using a CTE with JOIN

```sql
WITH valid_cases AS (
    SELECT case_id
    FROM event_log
    GROUP BY case_id
    HAVING COUNT(*) <= 6
)
SELECT el.*
FROM event_log el
INNER JOIN valid_cases vc
    ON el.case_id = vc.case_id;
```

This is functionally equivalent to the `IN`-based approach but uses an explicit **INNER JOIN**, which can sometimes improve readability and optimizer behavior on very large datasets.

## Why These Work Correctly

- **Grouping by `case_id`** ensures we evaluate the event count at the *case* level, not the individual event level.
- **`HAVING` (not `WHERE`)** is used because we're filtering on an *aggregate* (`COUNT(*)`) computed after grouping.
- **Returning `el.*`** ensures we get the complete event rows (including `activity`, `timestamp`, and any other columns) — not just the aggregated summary.

> **Recommendation:** For clarity and broad compatibility, the **first query** (using `IN` with a subquery) is the most straightforward. For large-scale event logs where performance is critical, the **window function** variant is often most efficient in DuckDB due to its columnar execution engine and single-pass evaluation.
