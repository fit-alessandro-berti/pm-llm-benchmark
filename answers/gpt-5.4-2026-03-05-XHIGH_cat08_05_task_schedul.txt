## Overall Approach

Precision Parts should treat this as a **process-mining-driven scheduling redesign** problem, not just a reporting exercise.

In this type of high-mix, low-volume job shop, the most valuable output from process mining is not a single ōprocess map.ö It is a **reconstructed operational state history** of the shop floor:

- which jobs were waiting where,
- which machine was processing, in setup, idle, or down,
- what sequence was actually executed on each machine,
- what alternatives existed at each dispatching decision,
- and how those decisions affected tardiness, WIP, and bottleneck behavior.

I would use the MES log to build that state history, diagnose loss mechanisms, and then design/test improved scheduling policies in a simulation-backed digital twin.

---

# 1. Analyzing Historical Scheduling Performance and Dynamics

## 1.1 Build the right process-mining data model first

A standard single-case event log is too limited for this environment. I would build an **object-centric event log** with at least these objects:

- **Job / Order**
- **Operation / Task**
- **Machine / Resource**
- **Operator**
- optionally **tooling / setup family / product family**

That matters because the same event belongs to multiple perspectives:
- `JOB-7001 Task Start (Cutting)` belongs to the job trace,
- to machine `CUT-01`,
- and to operator `OP-105`.

### Derived lifecycle states
From the raw events, I would reconstruct operation states such as:

- Released
- Waiting in queue
- Setup start / end
- Processing start / end
- Blocked / waiting for transfer if applicable
- Downstream waiting
- Completed

### Enriched attributes to derive from logs/master data
For each operation instance, I would derive:

- route position in the job
- remaining number of operations
- remaining expected work content
- current slack:  
  **slack = due date - current time - expected remaining processing/setup time**
- current queue length at the target machine
- previous job processed on the machine
- setup-family transition (e.g., material A  material B, tool family X  Y)
- machine eligibility set for flexible operations
- time since last breakdown / maintenance
- operator effect on processing/setup duration
- whether the job was exposed to a hot-job insertion or breakdown window

Because job shop routing variation can create ōspaghettiö models, I would also use **trace clustering / route-family clustering**:
- by product family,
- route signature,
- operation count,
- due-date tightness,
- priority class.

That avoids averaging together dissimilar jobs.

---

## 1.2 Reconstruct the actual job flow and machine execution sequence

I would reconstruct the process in **three synchronized views**.

### A. Job-centric view
For each job:
- release time,
- actual routing taken,
- timestamps for each queue/setup/process interval,
- final completion time,
- due-date performance.

Use:
- **Directly-follows graphs**
- **Inductive / Heuristics / Fuzzy mining**
- **performance overlays**
- **variant analysis** by route family

Goal: understand real end-to-end flow, deviations, loops, rework, and waiting structure.

### B. Machine-centric view
For each machine:
- exact sequence of jobs processed,
- setup intervals between jobs,
- idle periods,
- breakdown intervals,
- operator assignment,
- queue buildup before the machine.

This is crucial for sequence-dependent setup analysis. I would essentially reconstruct a **machine Gantt from logs**.

### C. State reconstruction over time
From queue entry/start/end and resource events, I would replay the log to reconstruct, for any time `t`:
- jobs waiting at each machine,
- machine status,
- WIP in system,
- candidate jobs competing for a machine,
- downstream queue conditions.

This enables analysis of **actual dispatching decisions under contention**, which is where the current policy likely fails.

## 1.3 Metrics and process-mining techniques

## 1.3.1 Job flow times, lead times, and makespan

### Metrics
For each job `j`:
- **Lead time / flow time**  
  `LT_j = Completion_j - Release_j`
- **Touch time**  
  sum of processing + setup
- **Flow efficiency**  
  `Touch time / Lead time`
- **Waiting share** of lead time
- **Makespan** for planning windows  
  e.g., daily or weekly:  
  `max completion - min release`

### Techniques
- Performance mining on job traces
- Distribution analysis by:
  - job family,
  - priority,
  - due-date tightness,
  - route length,
  - customer segment
- Performance spectrum / dotted charts to visualize spread

What I would look for:
- median vs 90th/95th percentile lead time,
- fat tails,
- unstable lead times for similar routings,
- flow factors far above 1, indicating excessive waiting.

---

## 1.3.2 Task waiting times at each work center / machine

### Metrics
For operation `(j,o)`:
- **Queue time**  
  `Queue time = Setup Start - Queue Entry`  
  or if no setup event, `Task Start - Queue Entry`
- Separate:
  - queue-before-setup,
  - setup time,
  - queue-after-setup if present,
  - move/transfer delay between operations
- Time-weighted queue length per machine
- Queue age distributions
- Percent of jobs waiting beyond threshold

### Techniques
- Queue state reconstruction from events
- Waiting-time heatmaps by machine, shift, weekday
- Queue-length time series
- Slice by priority and job family

This will show exactly where WIP accumulates and whether waiting is concentrated at a few resources or spread system-wide.

---

## 1.3.3 Resource utilization: productive, idle, setup, down

For each machine and operator, I would classify time into:

- **Processing**
- **Setup / changeover**
- **Idle but available**
- **Down / breakdown**
- **Off-shift / unavailable**
- optionally **starved** vs **blocked** if inferable

### Metrics
For machine `m`:
- **Productive utilization** = processing / available time
- **Occupancy** = (processing + setup) / available time
- **Setup ratio** = setup / occupied time
- **Idle ratio**
- **Downtime ratio**
- **Operator-matched utilization** if operator availability constrains machine usage

### Techniques
- Resource mining
- State timeline reconstruction
- Utilization heatmaps by shift/day
- Rolling-window utilization to detect **shifting bottlenecks**

Important: I would not rely on average utilization alone. In job shops, bottlenecks are often identified better by **persistent queue and active periods**, not just calendar utilization.

---

## 1.3.4 Sequence-dependent setup times

This is one of the highest-value analyses for this shop.

### How I would quantify it
For every setup event on machine `m`:
1. identify the **previous job** `i` on that machine,
2. identify the **current job** `j`,
3. calculate  
   `Setup_{i,j,m} = Setup End - Setup Start`

Then generalize beyond job IDs by mapping jobs into **setup-relevant attributes**, e.g.:
- material class,
- diameter/thickness range,
- tool family,
- fixture family,
- tolerance class,
- heat-treatment spec,
- program family.

This creates a **setup transition matrix / tensor**:
- `S[m, predecessor family, successor family]`

### Metrics
- mean, median, percentile setup time by transition
- asymmetry: `AB` vs `BA`
- setup variability by operator, shift, machine
- total setup hours by machine
- avoidable setup loss:
  - actual total setup time
  - versus setup-optimized sequence over the same job set / window

### Techniques
- Machine-centric sequence mining
- Transition heatmaps
- Regression / tree models for setup prediction from predecessor-successor attributes
- Counterfactual sequencing analysis

This analysis will directly reveal whether current sequencing is wasting capacity on bottlenecks.

---

## 1.3.5 Schedule adherence and tardiness

### Metrics
At the job level:
- **Lateness**  
  `L_j = Completion_j - DueDate_j`
- **Tardiness**  
  `T_j = max(0, L_j)`
- **Weighted tardiness** using priority weights
- % on-time
- tardiness by priority class
- tardiness distribution by route family and resource exposure

If planned machine sequence / planned starts exist:
- start deviation,
- finish deviation,
- sequence inversions,
- schedule stability / reschedule count.

If no formal detailed schedule exists:
- I would do **conformance to the intended dispatch logic** instead:
  - was the chosen job consistent with FCFS, EDD, or whatever rule was supposed to apply?
  - how often were there overrides?

### Techniques
- Conformance checking
- Temporal profile analysis
- Decision mining at dispatch points
- Cohort comparison: on-time vs late jobs

---

## 1.3.6 Impact of disruptions: breakdowns and priority changes

### Breakdown analysis
From breakdown start/end events:
- MTBF, MTTR by machine
- jobs directly affected:
  - processing on the failed machine,
  - queued for the failed machine,
  - rerouted to overloaded alternatives
- incremental delay caused by breakdown exposure
- recovery time until queue returns to normal

### Priority-change / hot-job analysis
For urgent jobs:
- time from priority change to first resource allocation
- jobs displaced by the urgent insertion
- extra setup induced by the insertion
- additional tardiness inflicted on displaced regular jobs  
  (ōexpedite taxö)

### Techniques
- Interrupted time-series analysis
- Before/after and matched-cohort analysis
- Resource contention analysis around disruption windows
- Process comparison of normal vs disrupted days/shifts

Using your snippet as an example, a `MILL-02` breakdown followed by a `JOB-7005` hot-job insertion is exactly the kind of ripple I would quantify:
- extra queue time added to milling jobs,
- overload shifted to alternate mills,
- added setup loss from re-sequencing,
- resulting weighted tardiness change.

# 2. Diagnosing Scheduling Pathologies

Based on the above analysis, I would expect the following pathologies to emerge.

## 2.1 Bottleneck resources Ś likely shifting, not fixed

### Evidence
- persistent queue buildup before certain machines
- long active periods where a machine never runs out of work
- large share of total system waiting attributable to a few resources
- late jobs disproportionately passing through the same machines
- bottleneck shifts by product mix, shift, or disruption state

### Process-mining methods
- bottleneck analysis on rolling windows
- queue accumulation analysis
- machine-state timelines
- on-time vs late variant comparison

In practice, the bottleneck may alternate between CNC milling, grinding, and inspection depending on mix and breakdowns.

---

## 2.2 Poor task prioritization and priority inversions

The current local FCFS/EDD mix likely causes cases where:
- high-priority jobs wait behind low-priority jobs,
- near-due jobs wait because local sequence was setup-convenient or simply FCFS,
- jobs with large downstream bottleneck exposure are released too late or too early.

### Evidence
At each dispatch point:
- compare actual chosen job with all jobs waiting
- compute which job had:
  - highest due-date risk,
  - highest weighted tardiness risk,
  - lowest setup penalty,
  - best global completion outcome

### Techniques
- decision mining
- contention-window analysis
- conformance to declared rule
- late vs on-time cohort comparison by queue experience

This gives objective proof of ōbad decisionsö rather than anecdotal planner frustration.

---

## 2.3 Setup-blind sequencing

A classic pathology in job shops is locally ōkeeping machines busyö while repeatedly alternating between dissimilar job families.

### Evidence
- high-frequency transitions between setup-incompatible jobs
- bottleneck machines spending excessive time in setup
- actual setup hours materially above setup-optimized counterfactual sequences
- urgent-job insertions causing repeated family breaks

### Techniques
- predecessor-successor setup heatmaps
- machine sequence mining
- counterfactual sequence optimization on historical windows

If total avoidable setup time on a bottleneck is large, that is equivalent to hidden lost capacity.

---

## 2.4 Upstream local optimization creating downstream starvation or WIP inflation

Current local rules often create one of two bad patterns:
- upstream work centers flood bottlenecks, causing WIP piles,
- or upstream decisions fail to feed the right mix, starving downstream resources.

### Evidence
- downstream machines idle while upstream work exists but not in the right readiness state
- upstream completion bursts followed by queue spikes at bottlenecks
- high WIP before bottlenecks and low WIP after them
- large waiting time after some upstream operations

### Techniques
- cross-resource flow synchronization analysis
- queue/WIP time series
- state reconstruction across stages
- correlation between upstream release bursts and downstream queue spikes

This is the job-shop version of a **bullwhip-like WIP amplification** effect.

---

## 2.5 Disruption amplification and schedule nervousness

A simple local dispatch environment usually reacts poorly to breakdowns and hot jobs:
- too much manual resequencing,
- too little buffer on bottlenecks,
- collateral damage to regular jobs,
- extra setups from constant reordering.

### Evidence
- long recovery times after breakdowns
- frequent sequence changes on bottleneck machines
- high expedite tax from urgent orders
- unstable completion-time promises

### Techniques
- disruption cohort analysis
- schedule stability metrics
- recovery-time measurement
- process comparison before/after hot-job injections

# 3. Root Cause Analysis of Scheduling Ineffectiveness

## 3.1 Likely root causes

### 1. Static local dispatching in a dynamic environment
FCFS/EDD at each work center is too myopic because it ignores:
- downstream bottleneck load,
- remaining route length,
- sequence-dependent setups,
- machine breakdown risk,
- job priority changes,
- flexible routing options.

### 2. Lack of real-time global visibility
A machine-level dispatcher cannot make good choices if it does not see:
- actual queue lengths elsewhere,
- alternate machine availability,
- current bottleneck status,
- hot-job disruptions,
- whether releasing one more job will only inflate WIP.

### 3. Poor time estimates
If planned durations or rough standards are biased:
- due dates become unrealistic,
- queue forecasts are wrong,
- slack calculations are meaningless.

The log already shows planned vs actual durations; I would quantify the bias and variance by job family, machine, operator, and setup transition.

### 4. Sequence-dependent setups not explicitly modeled
If setup is treated as ōfixedö or ignored in dispatching, the schedule will waste capacity precisely where capacity is scarcest.

### 5. Weak coordination between work centers
Local optimization often means:
- non-bottlenecks maximize their own utilization,
- bottlenecks get flooded,
- WIP rises,
- lead time becomes less predictable.

### 6. No robust response to breakdowns or urgent jobs
Without event-triggered rescheduling and protected capacity:
- each disruption causes shop-wide ripple effects.

### 7. Possibly unrealistic due dates
Process mining may show some jobs are effectively late at release because:
- promised due date minus release date is smaller than feasible processing + expected wait lower bound.

That is not a scheduling failure alone; it is a commercial/quoting issue too.

---

## 3.2 How process mining differentiates poor logic vs capacity shortage vs inherent variability

This distinction is critical.

## A. Signs of poor scheduling logic
- high waiting time despite moderate capacity utilization
- large avoidable setup loss
- repeated priority inversions at contention points
- WIP spikes caused by release/dispatch behavior
- significant improvement in simulation under better rules **without adding capacity**

## B. Signs of true capacity limitation
- offered load on specific resources persistently exceeds available capacity
- even near-optimal sequencing still leaves high tardiness
- bottleneck queues never drain under realistic demand
- simulation with improved rules helps only marginally

## C. Signs of inherent variability / disruption-driven instability
- average capacity is adequate, but tails are driven by:
  - high runtime variance,
  - breakdowns,
  - hot-job arrivals,
  - rework loops
- robust/predictive policies help more than deterministic ones

### Process-mining tools to support that distinction
- lead-time decomposition: wait vs process vs setup vs downtime
- offered-load vs capacity analysis by resource
- counterfactual sequencing analysis
- variant analysis controlling for route family and due tightness
- predictive error analysis of planned vs actual durations
- simulation under same demand/capacity but different scheduling logic

In short:  
**If better rules dramatically improve performance with the same machines, it is a scheduling problem.**  
**If not, it is a capacity problem.**  
**If performance is highly sensitive to variance and disruptions, it is a robustness problem.**

# 4. Developing Advanced Data-Driven Scheduling Strategies

These should be viewed as **layered**, not mutually exclusive.

---

## Strategy 1: Bottleneck-aware adaptive dispatching rule  
### (an ATCS-style dynamic rule with workload control)

## Core logic
Replace FCFS/EDD with a **dynamic composite priority score** recalculated whenever a machine becomes free.

A practical score for job `j` on machine `m` would combine:

- customer/job priority
- due-date risk / negative slack
- remaining route criticality
- predicted processing time
- **predicted sequence-dependent setup from current machine state**
- downstream bottleneck congestion
- queue age / anti-starvation term

Conceptually:

**Score = priority weight + due-risk + route criticality - setup penalty - downstream congestion + aging**

At bottlenecks, I would use an **ATCS-like rule** (Apparent Tardiness Cost with Setups) extended with:
- downstream load,
- remaining work,
- and hot-job handling.

For flexible operations, the rule would also choose the **best eligible machine**, not just the next available one.

### Strong addition: workload-controlled release
To reduce WIP, I would add **job release gating**:
- release new jobs based on projected bottleneck workload norms,
- not simply when paperwork/material is ready.

That turns the system from ōpush everythingö to controlled finite-capacity flow.

## How process mining informs it
- bottleneck identification from historical active periods
- setup transition matrix from machine sequences
- predictive process/setup times from historical logs
- downstream congestion patterns from queue reconstruction
- historical simulation to tune weights

## What pathologies it addresses
- poor prioritization
- excessive WIP
- upstream/downstream miscoordination
- setup-blind dispatching
- starvation/overfeeding of bottlenecks

## Expected KPI impact
- lower weighted tardiness
- lower average and percentile WIP
- shorter and more predictable lead times
- better bottleneck protection
- more balanced utilization  
  (note: not necessarily higher utilization everywhere, which is fine)

---

## Strategy 2: Predictive rolling-horizon stochastic scheduling

## Core logic
Move from reactive dispatching to a **rolling finite-capacity schedule** that is updated:
- every 30¢60 minutes, or
- on trigger events:
  - breakdown,
  - repair,
  - hot-job arrival,
  - queue threshold breach,
  - high predicted lateness risk.

The key difference: use **probabilistic durations**, not deterministic standards.

The scheduler would:
- predict task durations by job family, machine, operator, and setup state,
- predict setup times from predecessor-successor patterns,
- estimate breakdown risk / downtime exposure,
- generate a schedule for the next 1¢3 shifts,
- reserve buffer capacity on critical machines,
- reoptimize when conditions materially change.

It should also include a **schedule stability penalty** so the plan is not constantly reshuffled.

## How process mining informs it
From the historical log, build predictive models for:
- task duration distributions
- setup duration distributions
- breakdown frequency and repair durations
- hot-job arrival patterns
- rework probability if inspection failures exist
- machine-family performance differences
- operator effects

If predictive maintenance signals are available or derivable from event history, the schedule can:
- avoid assigning due-critical jobs to high-risk machines,
- or schedule preventive intervention in lower-impact windows.

## What pathologies it addresses
- unrealistic duration assumptions
- poor disruption handling
- unstable completion promises
- late discovery of bottlenecks
- chaotic response to hot jobs

## Expected KPI impact
- improved on-time delivery
- more reliable lead-time promises
- fewer last-minute expedites
- shorter recovery after breakdowns
- lower schedule nervousness

A major side benefit: customer promise dates become based on **completion probability**, not optimism.

---

## Strategy 3: Setup-time optimization through intelligent sequencing and family batching

## Core logic
Focus specifically on bottleneck and setup-heavy resources.

### Step 1: derive setup families
Using the mined setup analysis, define families driven by actual changeover behavior:
- material/tooling family,
- geometry family,
- tolerance/fixture family,
- surface-treatment family, etc.

### Step 2: optimize sequence in rolling windows
For each bottleneck machine, solve a short-horizon sequencing problem:

**Minimize:**  
weighted tardiness + lambda ū total sequence-dependent setup time

subject to:
- ready times,
- due dates,
- priority rules,
- machine eligibility,
- operator constraints if relevant,
- maximum wait limits for batching.

### Step 3: controlled batching
For compatible jobs, allow **micro-batching** or family runs, but with:
- max batch waiting time,
- max due-date deterioration,
- protected insertion slots for urgent jobs.

This is important: batching should reduce setup without creating huge delay for near-due jobs.

### Step 4: choose machine assignment with setup cost included
For flexible machines, assign jobs based on:
- expected finish time + setup impact,
not just earliest available machine.

## How process mining informs it
- setup matrices by predecessor-successor family
- operator-specific setup performance
- setup variability by shift/machine
- actual cost of family breaks caused by hot-job insertions
- identification of the few transitions that create most of the changeover loss

## What pathologies it addresses
- avoidable setup waste
- hidden capacity loss at bottlenecks
- excessive resequencing after urgent jobs
- overloaded critical resources

## Expected KPI impact
- reduced setup hours
- increased effective capacity on bottleneck machines
- reduced queue time and WIP upstream of bottlenecks
- better throughput and due-date performance

A likely side insight here: some transitions may be so expensive that the answer is not only better scheduling, but also **SMED-style setup reduction** projects.

---

## Recommended practical combination

The best real-world design is usually:

- **Strategy 2** at the planning / exception-management layer
- **Strategy 1** at the dispatch layer
- **Strategy 3** as a specialized optimizer for bottleneck/setup-heavy machines

That gives both responsiveness and discipline.

# 5. Simulation, Evaluation, and Continuous Improvement

## 5.1 Use a discrete-event simulation digital twin before deployment

Before changing live scheduling, I would build a **discrete-event simulation (DES)** of the job shop, parameterized directly from the mined log.

## Model inputs derived from process mining
- job arrival patterns by day/shift/season
- routing families and probabilities
- processing-time distributions by job-machine-operator combination
- sequence-dependent setup-time models
- machine eligibility and alternate-routing logic
- breakdown/repair distributions
- priority-change / hot-job arrival patterns
- operator calendars and skill constraints
- rework probabilities if inspection loops exist
- transport / handoff delays if measurable

## Validation
First, the DES must reproduce historical reality within tolerance:
- throughput,
- WIP distribution,
- queue times,
- utilization,
- tardiness,
- setup share,
- recovery after breakdowns.

If the model cannot mimic history, it should not be used for policy choice.

## What strategies/scenarios I would test

### Baseline
- current FCFS/EDD-like local dispatching

### Candidate policies
- Strategy 1 only
- Strategy 2 only
- Strategy 3 only
- combined strategy stack

### Stress scenarios
- high load / demand spike
- tighter due dates
- frequent bottleneck breakdowns
- surge in hot jobs
- product mix shift toward setup-heavy jobs
- operator shortage / skill mismatch
- lower estimate accuracy than expected
- temporary loss of a flexible machine

### Evaluation KPIs
- weighted tardiness
- % on-time by priority
- average and 95th percentile lead time
- average and 95th percentile WIP
- bottleneck queue length
- setup hours and setup ratio
- throughput
- schedule stability / nervousness
- expedite tax on regular jobs
- utilization balance across resources

### Experimental rigor
Use:
- multiple replications,
- common random numbers,
- confidence intervals / bootstrap comparisons,
- sensitivity analysis on rule weights, release thresholds, and batching windows.

That gives management defensible evidence before rollout.

## 5.2 Framework for continuous monitoring and adaptation

This should become a closed-loop system, not a one-time project.

## A. Online monitoring with ongoing process mining
Run near-real-time dashboards and alerts for:

### Outcome KPIs
- weighted tardiness
- on-time delivery by priority/customer
- lead-time distribution
- WIP by work center
- throughput

### Control KPIs
- queue age at bottlenecks
- setup ratio by machine
- bottleneck active period length
- breakdown recovery time
- number of hot-job insertions
- release load vs bottleneck norm
- starvation / idle-while-available time

### Model KPIs
- prediction error for task durations
- prediction error for setup durations
- breakdown-risk model quality
- schedule adherence / override rate

## B. Automatic drift detection
Use statistical change detection for:
- shifts in processing-time distributions
- shifts in setup matrix behavior
- route-mix changes
- new or shifting bottlenecks
- increased hot-job frequency
- deteriorating rule performance

Examples:
- if setup transition `Family A  B` suddenly worsens, recalibrate the setup model;
- if a new machine becomes the dominant queue source, retune dispatch weights and release logic.

## C. Adaptation loop
A practical cadence:
- daily: operational dashboard and exception handling
- weekly: KPI review and bottleneck drift analysis
- monthly: retrain predictive models and re-estimate setup matrices
- quarterly: re-run simulation and retune policy parameters

Any major policy change should go:
**live log -> detected issue -> simulation sandbox -> approved update -> controlled deployment**

## D. Human-in-the-loop learning
Planners will still override the system in some situations. That is useful data. Log:
- override reason,
- chosen sequence,
- resulting outcome.

Then mine those overrides to determine whether:
- the rule needs improvement,
- or the override was harmful.
