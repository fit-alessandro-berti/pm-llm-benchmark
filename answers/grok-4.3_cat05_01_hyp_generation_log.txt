**Anomalies and Undesirable Behaviors Identified**

**1. Case 1002 – Severe Process Violation (Credit & Stock Checks After Shipment)**
- **Anomaly**: `Perform Credit Check` and `Validate Stock` occur *after* both `Confirm Shipment` and `Ship Goods`.
- **Impact**: Goods were shipped without credit verification or stock validation.
- **Hypothesis**: 
  - Emergency/manual override for a priority order.
  - Logging error (timestamps recorded incorrectly).
  - System allowed logistics activities before mandatory checks were completed.

**2. Case 1003 – Missing Mandatory Step + Reversed Sequence**
- **Anomaly**: 
  - No `Validate Stock` activity recorded.
  - `Ship Goods` occurs *before* `Confirm Shipment`.
- **Hypothesis**:
  - Warehouse staff shipped without proper stock validation.
  - `Confirm Shipment` was logged retroactively after the goods had already left.
  - Possible process bypass for a specific customer or resource.

**3. Case 1004 – Critical Financial & Process Violation**
- **Anomaly**: 
  - `Receive Payment` occurs immediately after `Register Order` (before Invoice, Credit Check, or any logistics steps).
  - Missing `Perform Credit Check` and `Validate Stock` entirely.
  - Invoice issued *after* payment was received.
- **Hypothesis**:
  - Data entry error or fraud (payment recorded against an incomplete order).
  - Manual override by a high-privilege user.
  - System allowed payment processing before order validation.

**4. General Observations**
- Several cases skip mandatory control steps (`Validate Stock`, `Perform Credit Check`).
- Activity sequencing is inconsistent, particularly around logistics and finance activities.
- Some activities appear to be logged out of logical business order.

**SQL Queries to Investigate Further**

Here are targeted queries to explore these anomalies:

### 1. Get Full Activity Sequence Per Case (with ordering)
```sql
SELECT 
    case_id,
    activity,
    timestamp,
    resource,
    additional_info
FROM order_event_log
ORDER BY case_id, timestamp;
```

### 2. Identify Cases Missing Mandatory Activities
```sql
WITH mandatory_activities AS (
    SELECT unnest(ARRAY['Perform Credit Check', 'Validate Stock']) AS activity
)
SELECT 
    o.case_id,
    o.order_type,
    o.order_value,
    STRING_AGG(DISTINCT e.activity, ', ') AS activities_performed
FROM orders o
LEFT JOIN order_event_log e ON o.case_id = e.case_id
GROUP BY o.case_id, o.order_type, o.order_value
HAVING NOT EXISTS (
    SELECT 1 FROM order_event_log e2 
    WHERE e2.case_id = o.case_id 
    AND e2.activity = 'Perform Credit Check'
)
OR NOT EXISTS (
    SELECT 1 FROM order_event_log e3 
    WHERE e3.case_id = o.case_id 
    AND e3.activity = 'Validate Stock'
);
```

### 3. Detect Activity Order Violations (Credit Check after Shipment)
```sql
SELECT 
    a.case_id,
    a.activity AS earlier_activity,
    a.timestamp AS earlier_time,
    b.activity AS later_activity,
    b.timestamp AS later_time
FROM order_event_log a
JOIN order_event_log b ON a.case_id = b.case_id
WHERE a.activity = 'Ship Goods' 
  AND b.activity = 'Perform Credit Check'
  AND a.timestamp < b.timestamp;
```

### 4. Detect Cases Where Payment Occurs Too Early
```sql
SELECT 
    case_id,
    MIN(CASE WHEN activity = 'Register Order' THEN timestamp END) AS registration_time,
    MIN(CASE WHEN activity = 'Receive Payment' THEN timestamp END) AS payment_time,
    MIN(CASE WHEN activity = 'Issue Invoice' THEN timestamp END) AS invoice_time
FROM order_event_log
GROUP BY case_id
HAVING MIN(CASE WHEN activity = 'Receive Payment' THEN timestamp END) < 
       MIN(CASE WHEN activity = 'Issue Invoice' THEN timestamp END);
```

### 5. Analyze Resource Involvement in Anomalous Cases
```sql
SELECT 
    e.case_id,
    e.activity,
    e.resource,
    r.role,
    r.department
FROM order_event_log e
JOIN resources r ON e.resource = r.resource_id
WHERE e.case_id IN (1002, 1003, 1004)
ORDER BY e.case_id, e.timestamp;
```

### 6. Check for Skipped Steps by Comparing to Expected Flow
```sql
SELECT 
    case_id,
    COUNT(*) AS total_events,
    COUNT(DISTINCT activity) AS unique_activities,
    STRING_AGG(DISTINCT activity, ' | ' ORDER BY timestamp) AS activity_sequence
FROM order_event_log
GROUP BY case_id
ORDER BY case_id;
```

These queries can help surface whether the anomalies are isolated incidents, resource-specific patterns, or systemic issues.
