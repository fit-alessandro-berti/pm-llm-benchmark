## 2. Diagnosing Scheduling Pathologies

Based on the scenario characteristics, the following pathologies are highly probable and detectable via process mining:

### 2.2 Poor Prioritization and Preemption Inefficiency

**Pathology**: High-priority "hot jobs" (like JOB-7005 in the snippet) arriving mid-day may wait behind lower-priority jobs already in queue because FCFS dominates local decisions.

**Detection via Variant Analysis**:
1. Split the log into two subsets: **On-Time Jobs** vs. **Late Jobs**
2. Perform **variant comparison** (using techniques like differential process mining or decision point analysis):
   - At each queue entry decision point, analyze what factors predicted selection order
   - Identify decision points where Late Jobs experienced longer waits than On-Time Jobs with comparable characteristics
3. **Priority inversion events**: Explicitly count cases where a High Priority job entered a queue after a Low Priority job but exited after it

**Specific metric**: **Weighted Tardiness Concentration Ratio**—what percentage of total tardiness comes from top 20% priority jobs (should be near zero in well-functioning systems).

### 2.3 Suboptimal Sequencing Causing Setup Inflation

**Pathology**: With sequence-dependent setups, random or FCFS ordering can increase total setup time by 50-300% compared to optimized sequences.

**Analysis Approach**:

1. **Construct actual sequences**: For each machine, extract the daily/weekly processing sequence: $[J_1, J_2, J_3, ...]$

2. **Calculate realized setup burden**:
   $$\text{ActualSetupLoad}_r = \sum_{k=1}^{n-1} S_{r, \text{type}(J_k), \text{type}(J_{k+1})}$$

3. **Benchmark against optimal**: Solve a simplified Traveling Salesman Problem (TSP) for that day's job set on machine $r$ using the mined setup matrix to find theoretical minimum setup time. The ratio $\frac{\text{Actual}}{\text{Optimal}}$ reveals sequencing inefficiency.

4. **Pattern detection**: Identify "expensive transitions"—job-type pairs that frequently occur consecutively despite having high setup costs, indicating missed grouping opportunities.

**Visual evidence**: Sankey diagrams of job-type flows through bottleneck machines, weighted by setup time cost.

### 2.4 Downstream Starvation and Blocking Cascades

**Pathology**: Upstream batching or large transfer lots cause "lumpy" arrival patterns, starving downstream machines periodically.

**Detection Methods**:
- **Inter-arrival time analysis**: At each queue, compute CV˛ (coefficient of variation squared) of job arrivals. CV˛ >> 1 indicates high variability (starvation risk).
- **Block/Starve time correlation**: Cross-correlate idle time at Milling with busy time at Cutting. Negative lag correlation suggests Cutting output starvation causes Milling idle time.
- **WIP oscillation tracking**: Plot WIP levels at each buffer over time; synchronized spikes indicate push behavior without leveling.

### 2.5 Bullwhip Effect in WIP Amplification

**Pathology**: Local dispatching reacts to local queues without visibility of system-wide WIP, causing variability amplification.

**Measurement**: Calculate **WIP variance progression** along the routing. If Var(WIP_station_n) > Var(WIP_station_{n-1}), the bullwhip effect is present.

**Process mining technique**: **Case population analysis** tracking how many active cases exist at each stage over time, computing autocorrelation functions to identify cyclic buildup-and-release patterns (the "hockey stick" effect before shipping dates).

## 3. Root Cause Analysis of Scheduling Ineffectiveness

### 3.1 Differentiating Logic Failures vs. Capacity Constraints

Process mining enables this critical distinction through **controlled comparison**:

| Diagnostic Question | Process Mining Test | Interpretation |
|---------------------|---------------------|----------------|
| Is it capacity? | Check if bottleneck utilization > 95% even during low-WIP periods | Hard capacity constraint—needs capital or overtime |
| Is it logic? | Same utilization levels, but tardiness varies wildly by sequencing | Scheduling algorithm deficiency |
| Are estimates wrong? | Planned vs. Actual duration scatter by operation type | Estimation calibration needed |
| Is it variability? | Duration distribution kurtosis (heavy tails?) | Inherent process unpredictability |

**Conformance Checking**: Define an "oracle" schedule—what an omniscient scheduler with perfect foresight would achieve. Replay the log against this oracle. Large deviations indicate **controllable** losses (scheduling logic); small deviations indicate **uncontrollable** losses (capacity/variability).

### 3.2 Specific Root Causes Identified

**Root Cause 1: Myopic Local Optimization**
- Current rules optimize single-machine metrics (e.g., minimize local queue time) without considering:
  - Downstream congestion (releasing work to already-blocked stations)
  - Upstream readiness (starting jobs whose predecessors aren't complete)
  - System-wide due date proximity
  
**Evidence**: Cases completing early at non-critical stations but late overall due to bottleneck queuing.

**Root Cause 2: Sequence-Dependent Setup Blindness**
- FCFS treats all job transitions equally. When setup varies from 10 minutes (similar materials) to 120 minutes (complete changeover), ignoring this wastes effective capacity.
- **Quantification**: If average setup is 30 minutes but optimal sequencing achieves 15 minutes, the shop loses 50% of setup capacity—a hidden capacity drain.

**Root Cause 3: Static Rules in Stochastic Environment**
- Fixed dispatching cannot adapt to:
  - Dynamic arrival of hot jobs (reactive only, not proactive resequencing)
  - Machine breakdowns (no automatic reallocation)
  - Operator absences or skill variations
  
**Evidence**: High variance in daily throughput despite constant input rates.

**Root Cause 4: Lack of Predictive Visibility**
- No mechanism to predict that current WIP levels will inevitably cause tardiness 3 days from now (only realizing when due date passes).
- **Lead time estimation failure**: Historical data shows quoted lead times have poor accuracy (R˛ < 0.5 vs. actual).

**Root Cause 5: Disruption Response Fragility**
- When MILL-02 breaks down (as in snippet at 11:05:15), the system likely:
  - Continues feeding MILL-02 queue (building WIP)
  - Doesn't reroute eligible jobs to MILL-03 (if capable)
  - Doesn't adjust priorities elsewhere to compensate for lost capacity
  
**Evidence**: Spikes in WIP at broken-down machine's input buffer lasting beyond repair time.

## 4. Developing Advanced Data-Driven Scheduling Strategies

### Strategy 1: Adaptive Multi-Factor Composite Dispatching (AMCD)

**Core Logic**: Replace simple FCFS/EDD with a dynamic scoring function evaluated each time a machine becomes available:

$$\text{Priority Score}_{j,r,t} = w_1 \cdot \text{NORM(DDSlack}_{j,t}) + w_2 \cdot \text{NORM(PT)}_j + w_3 \cdot \text{NORM(Priority)}_j + w_4 \cdot \text{NORM(Congestion)}_{next(j)} - w_5 \cdot \text{NORM(SetUpSaved)}_{j,r,\text{last}(r)}$$

Where:
- **DDSlack**: Due Date minus expected remaining processing time (accounting for queue delays)
- **PT**: Remaining processing time (Shortest Processing Time component)
- **Priority**: Ordinal urgency (Hot > High > Medium > Low)
- **Congestion**: Estimated queue length at the next workstation in $j$'s routing (forward-looking)
- **SetUpSaved**: Setup time avoided if $j$ is processed next (vs. alternative candidate), calculated from the mined setup matrix

**How Process Mining Informs AMCD**:

1. **Weight optimization ($w_1$-$w_5$)**: Use historical data to perform regression analysis: which combination of factors best predicts on-time delivery? Train weights on 80% of log data, validate on 20%.

2. **Setup matrix construction**: From Section 1.2, derive $S_{r,i,j}$ with confidence intervals. The term $\text{SetUpSaved}$ uses expected value reduction.

3. **Congestion prediction**: Mine the correlation between current queue length at Station $X$ and expected delay for jobs routed there next. Build a lookup table: "If Milling has 5+ jobs, expect +4 hour delay."

4. **Dynamic adaptation**: Weights shift based on system state:
   - When WIP > threshold: Increase $w_2$ (SPT) to reduce congestion
   - When many near-due jobs: Increase $w_1$ (slack-based)
   - When bottleneck machine idle: Increase $w_4$ (feed the bottleneck)

**Addresses Pathologies**:
- Myopic optimization (includes forward-looking congestion)
- Setup blindness (explicit setup savings term)
- Poor prioritization (multi-factor including true slack)
- **Expected Impact**: 20-35% reduction in average tardiness, 15% WIP reduction through better flow balance.

### Strategy 2: Stochastic Predictive Rescheduling (SPR)

**Core Logic**: Move from reactive dispatching to **rolling-horizon predictive scheduling** with uncertainty quantification.

**Architecture**:

1. **Stochastic Activity Duration Models**:
   - For each (Operation Type, Machine, Operator Skill Level) combination, fit probability distributions to historical actual durations (not just means).
   - Example: CNC Milling on MILL-03 by Senior Operator follows LogNormal(=58min, =8min).
   - Incorporate **covariates**: Job complexity indicators (material hardness, tolerance requirements) extracted from order attributes to condition the distribution.

2. **Predictive Maintenance Integration**:
   - Mine breakdown patterns: MTBF (Mean Time Between Failures) per machine, conditional on usage intensity or recent maintenance.
   - Generate **availability forecasts**: Probability machine $r$ will be operational at future time $t$.

3. **Monte Carlo Schedule Projection**:
   Every T (e.g., 30 minutes), generate $N=1000$ simulated forward projections of the current shop floor state:
   - Sample task durations from fitted distributions
   - Sample potential breakdowns from reliability models
   - Apply proposed scheduling decisions
   - Record projected completion times and tardiness probabilities

4. **Decision Rule**: Choose the scheduling action (which job to start next on each free machine) that minimizes **expected total weighted tardiness** across the ensemble:
   $$\text{Action}^* = \arg\min_a \mathbb{E}\left[\sum_j w_j \cdot \max(0, C_j - d_j)\right]$$

**How Process Mining Enables SPR**:

- **Duration distributions**: Directly mined from `Task Duration (Actual)` column, stratified by relevant dimensions
- **Routing variability**: Some jobs deviate from standard routing (rework, alternate machines)—mine these probabilities
- **Correlation structures**: If one operation runs long, does the next typically run long too? (Learning curve or job difficulty effects)
- **Validation**: Back-test SPR on historical logs: "Given the state at 9AM on April 21st, what would SPR have scheduled, and how would that compare to actual outcomes?"

**Addresses Pathologies**:
- Unpredictable lead times (provides probabilistic estimates with confidence intervals)
- Disruption fragility (proactively schedules around predicted high-risk periods)
- Inaccurate estimations (uses empirical distributions, not guesses)
- **Expected Impact**: 40% improvement in due date accuracy (prediction error reduction), ability to quote reliable lead times to customers.

### Strategy 3: Setup-Aware Family Sequencing and Batching (SFSB)

**Core Logic**: Specifically target the setup time waste identified as a major pathology through intelligent job family grouping and bottleneck-focused sequencing.

**Components**:

**A. Job Family Clustering**:
- Apply clustering algorithms (e.g., agglomerative hierarchical clustering) on job characteristics (material type, tooling requirements, fixture types, color/cleaning needs) to define "setup families."
- **Objective**: Minimize intra-family setup time (ideally < 10 minutes) vs. inter-family (> 60 minutes).
- **Mining validation**: Verify clusters by checking if intra-family setup times in historical data are statistically significantly lower than inter-family (t-test or ANOVA on setup durations).

**B. Bottleneck-Prioritized Sequencing**:
- Identify the primary bottleneck machine $r^*$ (from Section 2.1).
- Apply **ATCS-like heuristics** or full **metaheuristic optimization** (Genetic Algorithm, Simulated Annealing) specifically for sequencing jobs on $r^*$ to minimize total setup time while respecting due dates.
- Formulate as: $\min \sum_k S_{r^*, j_k, j_{k+1}}$ subject to $\max(0, C_j - d_j) < \epsilon$ for critical jobs.

**C. Release Control (Input Regulation)**:
- Don't release jobs to the floor immediately upon order entry. Instead, **batch releases** by setup family to create "trains" of similar work flowing through the system.
- Use **CONWIP (Constant Work-in-Process)** cards limited by family to prevent excessive WIP of one type blocking others.

**D. Dynamic Setup Negotiation**:
- When machine $r$ finishes job $j$, and candidates $j_1, j_2, j_3$ are waiting:
  - Calculate setup times: $S_{r,j,j_1}$, $S_{r,j,j_2}$, $S_{r,j,j_3}$ from mined matrix
  - If $j_1$ has lowest setup but moderate due date pressure, select it
  - If $j_2$ has higher setup but is urgent (hot job), apply **setup cost vs. tardiness cost trade-off**:
    $$\text{Select } j_2 \text{ if } (S_{r,j,j_2} - S_{r,j,j_1}) \cdot \text{SetupCostRate} < \text{TardinessCost}_j$$

**How Process Mining Informs SFSB**:

- **Setup matrix derivation**: As detailed extensively above—the core input to this strategy.
- **Family definition**: Cluster analysis of job attributes correlated with setup time variation.
- **Bottleneck confirmation**: Resource utilization analysis identifying where sequencing yields highest ROI.
- **Batch size optimization**: Analyze trade-off between larger families (longer waits for first job, but faster processing once started) vs. smaller families (more responsive but more setup churn).

**Addresses Pathologies**:
- Suboptimal sequencing (directly optimizes the expensive dimension)
- Bottleneck throttling (frees up 20-40% of bottleneck capacity currently wasted on avoidable setups)
- High WIP (release control limits incoming work)
- **Expected Impact**: 25-40% reduction in total setup hours, 15-20% effective capacity increase at bottleneck, significant WIP reduction through smoother flow.
