**Assumptions & Notes**

1. **CaseID** is the `order_id`. All events belonging to an order share this identifier.
2. **Activity Naming** follows a natural language description of the ERP step (e.g., “Order Created”, “Items Picked”, “Order Dispatched”, etc.).
3. **Resources/Actors** are derived from:
   - Sales person (`sales_person_id`) for *OrderCreated*.
   - Warehouse picker (`picker_id`) for each *ItemsPicked* event (one row per lineitem pick, preserving individual timestamps).
   - Carrier (`carrier_id`) for both *OrderDispatched* and *OrderArrived* events; the same carrier is used for the paired dispatch/arrival pair.
   - Billing clerk (`billing_person_id`) for *InvoiceIssued*.
   - Payment record identifiers are shown as `payment_id` with method/reference in **Additional Attributes** (no dedicated resource column was provided).
4. **Chronology** respects the actual timestamps present in each table; where multiple deliveries exist for an order, they appear sequentially.
5. **Missing Data**: No explicit “Payment Received” timestamp conflicts arise because each payment links to a single invoice and its date is directly used.

This log can now be fed into processmining tools (e.g., Disco, ProM, Celonis) to analyze the endtoend order fulfillment performance, identify bottlenecks, or detect deviations.
