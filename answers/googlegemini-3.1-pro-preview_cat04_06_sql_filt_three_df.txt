Here is the DuckDB SQL query to achieve your goal. 

To determine cases that match the criteria (exact direct succession and specific time constraint), we can use the `LEAD` window function. This allows us to "look ahead" at the next chronological events within the exact same case. After identifying the cases that meet the sequence and time conditions, we use a `NOT IN` filter to exclude them from the final dataset.

```sql
WITH NextEvents AS (
    -- Retrieve the immediate next two activities and their timestamps for each event
    SELECT 
        case_id,
        activity AS step_1_act,
        LEAD(activity, 1) OVER w AS step_2_act,
        LEAD(activity, 2) OVER w AS step_3_act,
        LEAD(timestamp, 1) OVER w AS step_2_ts,
        LEAD(timestamp, 2) OVER w AS step_3_ts
    FROM event_log
    WINDOW w AS (PARTITION BY case_id ORDER BY timestamp ASC)
),
ExcludedCases AS (
    -- Filter down to the cases that exhibit the exact sequence and time constraint
    SELECT DISTINCT case_id
    FROM NextEvents
    WHERE step_1_act = 'Approve Purchase Requisition'
      AND step_2_act = 'Create Order'
      AND step_3_act = 'Confirm Order'
      AND (step_3_ts - step_2_ts) > INTERVAL 5 DAY
)
-- Return all events from the original log EXCEPT those belonging to the excluded cases
SELECT *
FROM event_log
WHERE case_id NOT IN (
    SELECT case_id 
    FROM ExcludedCases 
    WHERE case_id IS NOT NULL
)
ORDER BY case_id, timestamp ASC;
```

### Explanation of the logic:
1. **`NextEvents` CTE**: We partition the data by `case_id` and order by `timestamp` so that we only look at sequences strictly within a given process. By combining `LEAD(..., 1)` and `LEAD(..., 2)`, we represent groups of three distinct events in direct succession without any potential gaps from that case. 
2. **`ExcludedCases` CTE**: We filter those successions to uniquely identify `case_id`s where:
   - Event 1 is exactly `'Approve Purchase Requisition'`.
   - Event 2 is exactly `'Create Order'`.
   - Event 3 is exactly `'Confirm Order'`.
   - The time duration between Event 2 (`step_2_ts`) and Event 3 (`step_3_ts`) exceeds 5 days (`> INTERVAL 5 DAY`).
3. **`SELECT ... NOT IN`**: We grab all original events from `event_log` whose `case_id` did not match the sequence conditions found in `ExcludedCases`. We conclude by organizing the output primarily by `case_id` and `timestamp` so the resulting event tracks remain readable and strictly ordered.
