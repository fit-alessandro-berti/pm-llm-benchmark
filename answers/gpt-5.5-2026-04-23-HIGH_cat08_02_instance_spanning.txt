## 1. Identifying Instance-Spanning Constraints and Their Impact

### 1.1 Data preparation and process-mining approach

Because the main problems come from dependencies between orders, I would not analyze the log only as isolated order cases. I would create an enriched, object-centric view of the process with at least the following objects:

- **Order**: Case ID, order type, destination region, cold-packing flag, hazardous flag.
- **Resource**: picker, packing station, QC staff, labeling system.
- **Station type**: standard vs. cold-packing.
- **Shipping batch**: batch ID, region, batch creation time, batch release time.
- **Regulatory capacity token**: hazardous-material capacity, max 10 concurrent orders in Packing or QC.

From the start/complete timestamps, I would reconstruct activity intervals:

- `activity_start`
- `activity_complete`
- `activity_duration = complete - start`
- `ready_time_for_next_activity = completion time of previous activity`
- `queue_time = next_activity_start - ready_time_for_next_activity`

This allows separation of:

- **Processing time**: time spent actively in an activity.
- **Queue/waiting time**: time between readiness and the start of the next step.
- **Synchronization waiting time**: time spent waiting for a batch or other orders.
- **Resource-contention waiting time**: time spent waiting because another order occupies a required resource.

Traditional case-centric process mining is useful for discovering the main flow, but the instance-spanning constraints require **resource-centric**, **queue-centric**, and **object-centric process mining**.

---

### 1.2 Shared Cold-Packing Stations

#### Identification

Cold-packing constraints can be identified by filtering all orders where:

```text
Requires Cold Packing = TRUE
Activity = Packing
Resource = Cold-Packing Station C1-C5
```

For each cold-packing order:

```text
ready_for_packing = Item Picking complete timestamp
packing_start = Packing START timestamp
cold_packing_wait = packing_start - ready_for_packing
```

Then, reconstruct the occupancy timeline of each cold-packing station:

```text
Station C2 occupied from Packing START to Packing COMPLETE
```

At every point in time, calculate:

```text
number_of_busy_cold_stations(t)
```

If:

```text
number_of_busy_cold_stations(t) = 5
```

and a cold-packing order is waiting, then the waiting time is likely caused by **cold-station contention**.

#### Metrics

Key metrics:

- Average, median, P90, P95 waiting time before cold-packing.
- Cold-packing station utilization.
- Percentage of time all cold-packing stations are occupied.
- Cold-packing queue length over time.
- Number of orders delayed because all cold stations were busy.
- SLA breach rate for cold orders.
- Difference between express cold orders and standard cold orders:
  - Express cold-packing wait time.
  - Standard cold-packing wait time.
  - Standard delay caused by express prioritization.

Example metric:

```text
Cold-station contention delay =
sum of waiting time intervals where all cold stations were occupied
and the order was ready for Packing.
```

---

### 1.3 Shipping Batches

#### Identification

Shipping batching creates a synchronization point before Shipping Label Generation.

For each order:

```text
ready_for_label = Quality Check COMPLETE timestamp
label_generation_time = Shipping Label Generation timestamp
batch_wait_time = label_generation_time - ready_for_label
```

If batch IDs are recorded, such as `System (Batch B1)`, then each order can be linked directly to a batch. If batch IDs are not explicitly available, they can be inferred by clustering orders with:

- Same destination region.
- Similar label generation timestamp.
- Same shipment manifest or routing resource.
- Same carrier dispatch window.

For each batch:

```text
batch_start_time = first order in batch ready for label
batch_close_time = label generation time or batch release time
batch_formation_time = batch_close_time - batch_start_time
batch_size = number of orders in batch
```

#### Metrics

Key metrics:

- Average and P95 batch waiting time.
- Batch size distribution by region.
- Time from first order ready to batch release.
- Time from last order ready to batch release.
- Percentage of orders waiting more than threshold, e.g., 15 minutes, 30 minutes.
- SLA breach rate caused by batch waiting.
- Regional differences in batching delay.
- Frequency of underfilled batches.
- Frequency of late batches.

Important distinction:

```text
If an order completed QC at 10:35 but label generation occurred at 10:45 because the batch was waiting for other North-region orders, the 10-minute delay is not caused by the order itself. It is an inter-instance synchronization delay.
```

---

### 1.4 Priority Handling for Express Orders

#### Identification

Priority handling can be analyzed by comparing express and standard orders at each queue:

```text
Order Type = Express
```

For each activity, especially Packing and Quality Check:

```text
queue_time_standard
queue_time_express
activity_start_ordering
resource_allocation_ordering
```

I would check whether express orders systematically jump ahead of standard orders that were already waiting.

For example:

```text
Standard order ready for Packing at 10:20
Express order ready for Packing at 10:22
Express order starts Packing at 10:23
Standard order starts Packing at 10:31
```

This indicates priority overtaking.

If the process allows pausing standard orders, the log should ideally contain:

- SUSPEND event.
- RESUME event.
- Preemption reason.
- Resource reassignment.

If the current log only has START and COMPLETE events, preemptions may be inferred indirectly through:

- Unusually long activity durations for standard orders.
- Resource timelines showing that a station or staff member switched to an express order before the standard order completed.
- Overlapping intervals on the same resource, indicating possible pause/resume behavior not explicitly logged.

#### Metrics

Key metrics:

- Express order queue time by activity.
- Standard order queue time by activity.
- Number of priority overtakes.
- Number of standard-order interruptions.
- Total interruption time imposed on standard orders.
- Additional standard order cycle time attributable to express handling.
- Express SLA achievement rate.
- Standard SLA degradation during express peaks.
- Ratio:

```text
standard_delay_due_to_express / express_time_saved
```

This is important because excessive preemption may improve express performance while significantly damaging standard throughput.

---

### 1.5 Hazardous Material Regulatory Limit

#### Identification

For all hazardous orders:

```text
Hazardous Material = TRUE
Activity in {Packing, Quality Check}
```

Reconstruct active intervals:

```text
hazardous_order_active(t) = TRUE
if hazardous order is between START and COMPLETE of Packing or Quality Check
```

Then calculate:

```text
hazardous_concurrency(t) =
count of hazardous orders active in Packing or Quality Check at time t
```

The compliance rule is:

```text
hazardous_concurrency(t) <= 10
```

I would perform conformance checking against this rule to verify whether the facility ever violated the limit.

#### Metrics

Key metrics:

- Maximum hazardous concurrency observed.
- Percentage of time hazardous concurrency equals 10.
- Number of times the hazardous limit is reached.
- Queue length of hazardous orders waiting for Packing or QC.
- Hazardous order waiting time caused by the regulatory cap.
- Throughput reduction during hazardous-cap saturation.
- SLA breach rate for hazardous orders.
- Resource idle time while hazardous orders are waiting but cannot start due to the cap.

A particularly useful metric is:

```text
hazardous_cap_blocking_time =
time during which:
1. hazardous_concurrency = 10,
2. at least one hazardous order is ready for Packing or QC,
3. the relevant resource is otherwise available.
```

This isolates waiting caused by the regulatory limit rather than by lack of staff or stations.

---

### 1.6 Differentiating Within-Instance and Between-Instance Waiting

This is critical.

A delay can come from:

1. **Within-instance factors**
   - Long picking duration because items are hard to find.
   - Packing takes longer because the order has many items.
   - QC takes longer because the order fails inspection.
   - Label generation system is slow for that individual order.

2. **Between-instance factors**
   - Order waits because another order occupies the cold-packing station.
   - Order waits because the batch is not ready.
   - Standard order is delayed because an express order preempts it.
   - Hazardous order waits because the facility already has 10 hazardous orders in Packing/QC.

To separate these, I would decompose each order’s elapsed time into:

```text
End-to-end cycle time =
processing time
+ queue time
+ batching/synchronization time
+ interruption time
+ non-working/calendar time
```

For each waiting interval, I would replay the process state at that time:

| Question | Interpretation |
|---|---|
| Was the required resource unavailable? | Resource-contention delay |
| Were all cold-packing stations occupied? | Cold-packing contention |
| Was the hazardous limit already at 10? | Regulatory-cap delay |
| Was the order waiting for other orders in the same shipping batch? | Batch synchronization delay |
| Was an express order inserted ahead of a standard order? | Priority-induced delay |
| Was the order already being processed but activity duration was long? | Within-instance processing issue |
| Was the resource idle but the order still did not start? | Possible planning rule, missing data, or unmodeled constraint |

Where multiple constraints overlap, I would use interval-based attribution. For example, if a hazardous cold order is waiting and both the cold stations are full and the hazardous cap is reached, the waiting interval should be tagged as caused by multiple binding constraints.

## 2. Analyzing Constraint Interactions

The constraints should not be optimized independently because they interact strongly.

### 2.1 Express Orders and Cold-Packing Stations

An express order requiring cold-packing can disrupt the cold-packing queue.

Example:

- Five cold stations are occupied by standard cold orders.
- An express cold order arrives.
- A standard cold order may be paused or delayed.
- The express order starts quickly, but standard cold orders experience longer waits.

Potential effects:

- Express cold orders meet SLA.
- Standard cold orders suffer increased waiting.
- Cold-station utilization rises.
- Preemption overhead may reduce total throughput.
- Downstream QC and batching schedules become less predictable.

So, a policy that always prioritizes express orders may look good locally but may damage overall performance.

---

### 2.2 Batching and Hazardous Material Limits

Suppose several hazardous orders are going to the same region.

If they complete QC close together, they can form a shipping batch efficiently. However, the hazardous cap may delay some of them before or during Packing/QC.

This can cause:

- Batch formation delays.
- Some non-hazardous orders waiting for hazardous orders to complete QC.
- Hazardous orders missing batch cutoffs.
- More partially filled or delayed batches.

Therefore, batching rules should consider whether remaining orders in the batch are blocked by the hazardous cap.

---

### 2.3 Priority Orders and Batching

Express orders should generally not wait too long for a batch. If express orders are forced into the same batching logic as standard orders, their SLA may be lost after QC.

Possible interactions:

- Express orders reach Shipping Label Generation quickly but wait for standard orders in the same region.
- Standard batching improves route efficiency but hurts express delivery times.
- Creating separate express micro-batches may improve delivery time but increase shipping cost.

Therefore, batching logic should be priority-aware.

---

### 2.4 Hazardous Orders and Shared Packing Capacity

A hazardous cold order consumes two limited capacities simultaneously:

1. A cold-packing station.
2. One hazardous-material concurrency slot.

This creates a compound bottleneck.

If many hazardous cold orders arrive during a peak:

- Cold stations may be occupied.
- Hazardous limit may be saturated.
- QC may also be blocked.
- Downstream batches may be delayed.

This kind of interaction cannot be solved by only adding one cold station if the real binding constraint is the hazardous-material cap.

---

### 2.5 Why Interaction Analysis Matters

If each constraint is optimized separately, the company may simply move the bottleneck.

Examples:

- Adding cold-packing capacity may increase flow into QC, causing the hazardous cap to bind more often.
- Faster QC may increase waiting before batching.
- Aggressive express prioritization may cause standard orders to miss regional batches.
- Larger batch sizes may reduce shipping cost but increase cycle time and SLA violations.

Therefore, optimization should be done at the system level, using simulation and constraint-aware scheduling.

---

## 3. Developing Constraint-Aware Optimization Strategies

### Strategy 1: Rolling-Horizon Constraint-Aware Dispatching

#### Constraints addressed

- Cold-packing contention.
- Express priority.
- Hazardous-material limit.
- Downstream batching deadlines.

#### Proposed change

Introduce a central dispatching algorithm for Packing and QC that assigns orders to resources using a rolling time window, for example every 5 or 10 minutes.

Instead of using simple FIFO or manual dispatching, the system should score each waiting order based on:

- SLA deadline.
- Order type: express vs. standard.
- Whether cold-packing is required.
- Whether the order is hazardous.
- Current hazardous concurrency.
- Destination region and upcoming batch cutoff.
- Expected processing time.
- Current queue lengths.
- Risk of missing the next shipping batch.

A simplified priority score could be:

```text
priority_score =
SLA_urgency_weight
+ express_weight
+ batch_cutoff_weight
+ cold_station_wait_weight
+ hazardous_slot_weight
- preemption_penalty
```

The dispatcher should explicitly enforce:

```text
cold orders must use cold stations
hazardous concurrency <= 10
express orders receive priority, but not unlimited priority
```

#### Data used

The dispatching logic should be trained and calibrated using:

- Historical processing-time distributions.
- Arrival patterns by hour/day/season.
- Express order arrival rates.
- Cold-packing demand forecasts.
- Hazardous-order volume.
- Batch departure schedules.
- Historical SLA violations.

#### Expected outcomes

- Lower waiting time at cold-packing stations.
- Better use of scarce hazardous capacity.
- Fewer unnecessary express preemptions.
- Reduced standard-order starvation.
- Better alignment between Packing/QC completion and shipping batch deadlines.
- Improved overall SLA performance, not just express SLA.

---

### Strategy 2: Dynamic Cold-Packing Capacity and Reservation Policy

#### Constraints addressed

- Shared cold-packing stations.
- Express cold-order prioritization.
- Interaction between cold-packing and downstream QC/batching.

#### Proposed change

Introduce dynamic capacity management for cold-packing.

Possible actions:

1. **Convertible packing stations**
   - Equip one or two standard stations so they can temporarily operate as cold-packing stations during peaks.

2. **Time-based cold-station reservation**
   - Reserve a portion of cold-packing capacity during known express peaks.
   - Example: keep one cold station available for express orders only if the predicted express cold arrival rate is high.

3. **Anti-starvation rule**
   - A standard cold order cannot be bypassed indefinitely.
   - Example: if a standard cold order has waited more than 20 minutes, it receives elevated priority.

4. **Peak-season staffing adjustment**
   - Add trained staff to cold-packing stations during forecasted high-demand windows.

#### Data used

Use process mining to identify:

- Hours when all cold stations are occupied.
- Cold-packing queue peaks.
- Express cold arrival patterns.
- Standard cold-order starvation cases.
- Marginal benefit of adding one additional cold station.

A useful what-if analysis:

```text
How many SLA breaches would have been avoided
if one extra cold-packing station were available
during the top 10% busiest cold-packing periods?
```

#### Expected outcomes

- Reduced cold-packing queue time.
- Fewer express cold-order delays.
- Lower standard-order starvation.
- Higher cold-order throughput.
- Reduced need for disruptive preemption.

---

### Strategy 3: Dynamic, Priority-Aware Shipping Batch Formation

#### Constraints addressed

- Shipping batch synchronization.
- Express-order delay after QC.
- Regional batch waiting.
- Interaction with hazardous-order delays.

#### Proposed change

Replace static batching rules with dynamic batch triggers.

Instead of waiting until a fixed batch size is reached, use rules such as:

```text
Release batch when:
1. batch size reaches target threshold,
OR
2. oldest order in batch has waited more than X minutes,
OR
3. next carrier cutoff is approaching,
OR
4. an express order is close to SLA deadline.
```

Different batching policies should be used by order type:

- **Express orders**
  - Smaller batch threshold.
  - Shorter maximum waiting time.
  - Possible direct label generation without waiting for full batch.

- **Standard orders**
  - Larger batch threshold if SLA risk is low.
  - More cost-optimized batching.

- **Hazardous orders**
  - Batch rules should consider whether remaining expected orders are blocked by the hazardous cap.
  - Do not make non-hazardous orders wait excessively for hazardous orders that cannot enter Packing/QC yet.

#### Data used

Use historical data to estimate:

- Arrival rate by region and time of day.
- Typical batch formation time.
- Shipping cost per batch size.
- SLA risk as a function of waiting time.
- Probability that additional orders for a region will arrive soon.

A predictive batching model can estimate:

```text
Expected benefit of waiting 5 more minutes
versus
risk of SLA breach and downstream delay.
```

#### Expected outcomes

- Reduced waiting before Shipping Label Generation.
- Fewer orders missing delivery cutoffs because of batching.
- Better express-order performance.
- Improved balance between route efficiency and delivery speed.
- Less unnecessary waiting for orders blocked upstream by other constraints.

---

### Strategy 4: Hazardous-Material Token-Based WIP Control

#### Constraints addressed

- Regulatory limit of 10 hazardous orders in Packing/QC.
- Hazardous-order queueing.
- Interaction between hazardous limit, Packing, QC, and batching.

#### Proposed change

Implement a token-based control mechanism.

There are exactly 10 hazardous-processing tokens. A hazardous order must acquire a token before entering Packing and keeps it until QC is complete.

```text
Hazardous order can start Packing only if available_hazard_tokens > 0
```

However, rather than assigning tokens FIFO only, the token scheduler should consider:

- SLA urgency.
- Whether the order requires cold-packing.
- Upcoming batch cutoff.
- Expected Packing and QC duration.
- Whether QC capacity is available.
- Risk of blocking downstream flow.

Also consider reducing the time each hazardous order holds a token by:

- Prioritizing quick QC completion after hazardous Packing.
- Creating a dedicated hazardous QC lane.
- Avoiding gaps between hazardous Packing completion and QC start.
- Pre-staging documentation or compliance checks before Packing.

#### Data used

Use process mining to identify:

- How often the hazardous cap is saturated.
- Average hazardous token holding time.
- Waiting time before Packing caused by cap saturation.
- Waiting time between hazardous Packing and QC.
- Regions where hazardous orders often delay batches.

#### Expected outcomes

- Maintains full regulatory compliance.
- Reduces hazardous-order waiting time.
- Increases effective hazardous throughput without exceeding the limit.
- Reduces batch delays caused by hazardous orders.
- Makes regulatory capacity usage more predictable.

---

### Strategy 5: Controlled Preemption Policy for Express Orders

#### Constraints addressed

- Priority handling.
- Standard-order disruption.
- Cold-packing contention.
- Packing/QC resource stability.

#### Proposed change

Do not allow unlimited interruption of standard orders.

Introduce controlled preemption rules:

1. **Preempt only at safe checkpoints**
   - For example, after current packing subtask is complete.

2. **Preempt only if express SLA risk is high**
   - If express order can still meet SLA without preemption, keep current standard order running.

3. **Limit repeated preemptions**
   - A standard order cannot be paused more than once unless management override is required.

4. **Use express reserve capacity first**
   - If reserved express capacity is available, avoid preemption.

5. **Track preemption cost**
   - Include restart/setup time in the decision.

#### Data used

Analyze:

- How much time express orders save due to preemption.
- How much delay standard orders suffer.
- Restart/setup losses after interruption.
- SLA impact on both express and standard orders.

#### Expected outcomes

- Maintains strong express performance.
- Reduces unnecessary disruption to standard orders.
- Improves total throughput.
- Reduces variability in Packing and QC durations.
- Prevents standard-order starvation during express peaks.

---

## 4. Simulation and Validation

Before implementation, I would build a data-driven discrete-event simulation using the process-mining results.

### 4.1 Baseline simulation model

The simulation should reproduce the current process:

```text
Order Received
 Item Picking
 Packing
 Quality Check
 Shipping Label Generation
```

The model should include:

- Order arrivals by time of day, weekday, and season.
- Order attributes:
  - Express/standard.
  - Region.
  - Cold-packing requirement.
  - Hazardous-material flag.
- Resource pools:
  - Pickers.
  - Standard packing stations.
  - Cold-packing stations.
  - QC staff.
  - Labeling system.
- Resource calendars and shift schedules.
- Processing-time distributions by activity and order type.
- Queue rules.
- Preemption rules.
- Batch formation rules.
- Hazardous-material concurrency rule.

The model must explicitly enforce:

```text
maximum 5 cold-packing stations
maximum 10 hazardous orders in Packing/QC
batch synchronization before label generation
priority treatment for express orders
```

---

### 4.2 Calibration using the event log

The simulation parameters should be mined from the historical log:

- Arrival distributions.
- Activity duration distributions.
- Resource utilization.
- Routing probabilities.
- Batch sizes.
- Batch waiting times.
- Express-priority behavior.
- Hazardous concurrency patterns.
- Queue lengths and waiting times.

Then validate the baseline model by comparing simulated results with historical results:

| KPI | Historical | Simulated baseline |
|---|---:|---:|
| Average cycle time | From log | From simulation |
| P95 cycle time | From log | From simulation |
| Cold-packing wait | From log | From simulation |
| Batch waiting time | From log | From simulation |
| Express SLA rate | From log | From simulation |
| Standard SLA rate | From log | From simulation |
| Hazardous cap saturation | From log | From simulation |
| Resource utilization | From log | From simulation |

The baseline simulation must match the real process closely enough before testing changes.

---

### 4.3 Scenario testing

I would run several scenarios, such as:

1. Current process baseline.
2. Add one convertible cold-packing station during peak hours.
3. Introduce priority-aware dynamic batching.
4. Introduce hazardous token scheduling.
5. Introduce controlled express preemption.
6. Combine all recommended strategies.
7. Peak-season stress test with 20%, 40%, or 60% higher order volume.

Each simulation should be run multiple times to account for randomness.

---

### 4.4 Simulation focus areas

The model must accurately capture the instance-spanning constraints:

#### Resource contention

- Station occupancy.
- Staff occupancy.
- Queue discipline.
- Resource calendars.
- Utilization peaks.

#### Cold-packing limitation

- Cold orders can only use cold-packing stations.
- Cold station capacity is limited.
- Express cold orders may receive priority.

#### Batching

- Orders wait for regional batches.
- Batch release depends on size, time, priority, or cutoff.
- Express orders may use different batch rules.

#### Priority interruptions

- Express orders can overtake standard orders.
- If preemption is allowed, standard order remaining processing time must be stored.
- Restart/setup penalty should be modeled.

#### Hazardous-material cap

- No more than 10 hazardous orders can be in Packing/QC at the same time.
- Hazardous orders must wait if the cap is reached.
- Compliance must be hard-coded, not treated as optional.

---

### 4.5 KPIs to evaluate in simulation

Primary KPIs:

- End-to-end cycle time.
- P90/P95 cycle time.
- SLA achievement rate.
- Throughput per shift/day.
- Average WIP.
- Queue length by activity.
- Waiting time by constraint.
- Cold-packing utilization.
- Hazardous-cap saturation.
- Batch waiting time.
- Express vs. standard performance.
- Shipping cost impact from smaller or more frequent batches.

The best strategy is not necessarily the one with the lowest average cycle time. It should improve delivery reliability while respecting cost, capacity, priority, and compliance constraints.

## 5. Monitoring Post-Implementation

After implementation, I would set up continuous process-mining dashboards focused specifically on instance-spanning constraints.

---

### 5.1 Overall fulfillment dashboard

Core KPIs:

- Order volume by hour/day.
- End-to-end cycle time.
- Average, median, P90, P95 cycle time.
- SLA achievement rate.
- Throughput per shift.
- WIP by process stage.
- Rework or failed QC rate.
- Cycle time by:
  - Express vs. standard.
  - Cold vs. non-cold.
  - Hazardous vs. non-hazardous.
  - Region.
  - Batch type.

---

### 5.2 Cold-Packing Constraint Dashboard

Metrics:

- Cold-packing queue length over time.
- Average and P95 cold-packing waiting time.
- Cold station utilization.
- Percentage of time all cold stations are occupied.
- Number of cold orders waiting while all cold stations are busy.
- Express cold vs. standard cold waiting time.
- Standard cold orders bypassed by express orders.
- SLA breaches linked to cold-packing delay.

Useful alert:

```text
Trigger alert if:
cold-packing queue length > threshold
AND
all cold stations occupied for more than X minutes.
```

---

### 5.3 Shipping Batch Dashboard

Metrics:

- Batch formation time by region.
- Batch size by region.
- Waiting time from QC completion to label generation.
- Oldest order age in open batch.
- Percentage of batches released underfilled.
- Percentage of orders delayed beyond threshold due to batching.
- Express orders waiting in standard batches.
- Orders missing carrier cutoff due to batch delay.
- Shipping cost per order or per batch.

Useful alert:

```text
Trigger alert if:
oldest order in batch has waited more than X minutes
OR
express order in batch is within Y minutes of SLA risk.
```

---

### 5.4 Priority Handling Dashboard

Metrics:

- Express queue time by activity.
- Standard queue time by activity.
- Number of priority overtakes.
- Number of standard-order interruptions.
- Total interruption duration.
- Express time saved versus standard time lost.
- SLA rate for express orders.
- SLA impact on standard orders during express peaks.
- Starvation cases where standard orders wait beyond maximum allowed threshold.

Useful alert:

```text
Trigger alert if:
standard order has been overtaken more than N times
OR
standard wait time exceeds anti-starvation threshold.
```

---

### 5.5 Hazardous-Material Compliance Dashboard

Metrics:

- Real-time hazardous concurrency in Packing/QC.
- Maximum hazardous concurrency per shift.
- Percentage of time hazardous concurrency equals 10.
- Number of compliance violations.
- Hazardous queue length before Packing.
- Hazardous waiting time caused by cap saturation.
- Average hazardous token holding time.
- Time between hazardous Packing completion and QC start.
- SLA rate for hazardous orders.

Compliance alert:

```text
Immediate alert if hazardous_concurrency > 10.
```

Flow alert:

```text
Trigger alert if:
hazardous_concurrency = 10
AND
hazardous queue length > threshold
for more than X minutes.
```

---

### 5.6 Constraint Attribution Dashboard

A particularly useful dashboard would show the main cause of waiting time.

For each order, classify waiting time as:

- Waiting for standard packing station.
- Waiting for cold-packing station.
- Waiting for QC staff.
- Waiting for hazardous-material capacity.
- Waiting for shipping batch.
- Delayed by express priority.
- Other/unknown.

Then aggregate:

```text
Total delay minutes by constraint
```

This allows management to see whether the dominant bottleneck is shifting over time.

Example:

| Delay cause | Total delay minutes | % of total delay |
|---|---:|---:|
| Cold-packing contention | 4,200 | 28% |
| Batch waiting | 3,600 | 24% |
| Hazardous cap | 2,100 | 14% |
| Express preemption | 1,700 | 11% |
| QC resource shortage | 1,500 | 10% |
| Other | 1,900 | 13% |

This is more actionable than simply saying, “Packing is slow.”

---

### 5.7 Continuous Improvement Loop

The post-implementation monitoring should support a continuous improvement cycle:

1. Monitor KPIs daily.
2. Detect bottleneck shifts.
3. Compare performance before and after changes.
4. Recalibrate prediction models.
5. Update dispatching and batching rules.
6. Re-run simulations for new peak-season scenarios.
7. Validate that compliance remains intact.

The key is to manage the fulfillment center as a constrained, interacting system rather than as a sequence of independent order cases. The instance-spanning constraints — cold-packing capacity, batching, priority handling, and hazardous-material limits — must be measured, simulated, optimized, and monitored together.
