**1. Analyzing Historical Scheduling Performance and Dynamics**

Process mining reconstructs the actual job flows and task sequences directly from the MES event logs by treating each Case ID (Job ID) as a trace. The log structure—timestamped events for job release, queue entry, setup start/end, task start/end, resource breakdowns, priority changes, and completions—enables full trace extraction. Using algorithms such as the Inductive Miner or Heuristics Miner (via tools like ProM or PM4Py), one discovers the control-flow model per job family or globally, revealing the observed routing (e.g., Cutting  Milling  Heat Treatment) and deviations from any nominal process. Directly-follows graphs and dotted charts visualize temporal patterns, including parallel execution across machines and waiting periods.

Specific metrics and techniques include:

- **Job flow times, lead times, and makespan**: Compute cycle time per trace as the interval from "Job Released" to final "Task End." Aggregate into distributions (mean, median, 95th percentile, variance) using performance spectrum analysis. Makespan is derived for batches of jobs released in the same window. Variant analysis partitions traces into on-time vs. late cohorts to compare flow-time distributions.

- **Task waiting times (queue times)**: For each work center (e.g., CUT-01, MILL-03), calculate queue time as the delta between "Queue Entry" and "Setup Start" or "Task Start." Use bottleneck analysis plugins to rank machines by average and maximum queue time, producing heatmaps of contention periods.

- **Resource utilization**: For machines and operators, derive busy time from "Task Start" to "Task End" plus "Setup Start" to "Setup End." Idle time is total observation window minus busy/setup time minus breakdowns. Utilization = (productive + setup time) / (total time - breakdown time). Social network mining (handover-of-work) quantifies operator-machine pairings and their efficiency.

- **Sequence-dependent setup times**: Filter events where "Setup Required = TRUE." Group consecutive events on the same Resource (Machine ID). Extract the pair (previous Case ID's last task properties, current job properties) and compute actual setup duration (Setup End - Setup Start). This yields an empirical setup matrix: for each machine, a table or regression model of setup time as a function of job attributes (material, geometry, previous operation type). Statistical analysis (ANOVA, regression trees) identifies high-impact transitions.

- **Schedule adherence and tardiness**: For each completed job, compute tardiness = max(0, completion timestamp - Order Due Date). Measure schedule adherence via conformance checking: align actual traces against a normative model derived from due dates and planned durations. Metrics include percentage of tardy jobs, average tardiness, and maximum tardiness. Cumulative tardiness curves over time highlight systemic slippage.

- **Impact of disruptions**: Isolate "Breakdown Start/End" and "Priority Change" events. Use causal process mining or correlation analysis to link these to downstream waiting times, flow-time inflation, and tardiness spikes. For example, compute the conditional distribution of queue times immediately after breakdowns versus baseline periods. Process cubes slice the log by disruption presence to quantify KPI degradation.

These analyses produce both aggregate KPIs and granular, timestamped diagnostics (e.g., per-machine setup matrices and waiting-time distributions).

**2. Diagnosing Scheduling Pathologies**

The analyses typically reveal several interconnected pathologies in the current FCFS/EDD local dispatching regime:

- **Bottleneck resources**: Machines such as specialized CNC or Heat Treatment units show utilization >85% with queue times exceeding 2–3× processing time. Bottleneck analysis quantifies their impact via "critical path" contribution to overall flow time (often 40–60% of total lead time attributable to one or two resources).

- **Poor prioritization**: Variant analysis comparing late vs. on-time jobs shows that high-priority or near-due-date jobs frequently experience long queues behind low-priority work, especially after breakdowns. Conformance checking highlights frequent violations where EDD is overridden by local FCFS.

- **Suboptimal sequencing and setup inflation**: Setup matrix analysis reveals clusters where dissimilar jobs (different materials or geometries) are processed consecutively on bottleneck machines, inflating setup times by 50–200% versus best-case sequences. Process discovery often shows "setup-heavy" variants dominating.

- **Downstream starvation**: Performance spectra reveal periods where upstream bottlenecks cause downstream machines (e.g., Grinding) to idle for hours despite available work, indicating myopic local decisions that fail to consider global flow.

- **Bullwhip/WIP variability**: WIP accumulation between stations exhibits high variance; queue-length time series show amplification of upstream delays. This correlates strongly with disruption events.

Evidence is generated via:
- Bottleneck detection tools ranking resources by total waiting time and utilization.
- Variant comparison (Inductive Miner on filtered logs) showing structural differences between on-time and late traces.
- Resource contention periods identified by overlaying machine availability with job arrival rates.
- Causal nets or decision trees linking specific event patterns (e.g., breakdown followed by priority change) to KPI degradation.

**3. Root Cause Analysis of Scheduling Ineffectiveness**

Process mining differentiates root causes by contrasting behavior under varying load and disruption conditions:

- **Limitations of static dispatching**: Local rules ignore global state. Evidence appears in high queue-time variance even during low overall utilization periods, indicating myopic decisions rather than pure capacity shortages.

- **Lack of real-time visibility**: Logs show frequent "starvation" intervals where downstream queues are empty while upstream work is queued elsewhere. This points to absence of shared state rather than inherent variability.

- **Inaccurate duration/setup estimates**: Actual vs. planned duration analysis (via performance analysis) shows systematic underestimation, especially for setups after certain transitions. High deviation correlates with tardiness independent of machine load.

- **Ineffective sequence-dependent setup handling**: The mined setup matrix demonstrates that current rules never optimize transitions; setups are treated as fixed rather than sequence-dependent.

- **Poor coordination and disruption response**: Breakdown events trigger priority changes, but subsequent traces show cascading delays without compensatory rescheduling. Causal analysis (e.g., via conformance with deviation annotations) separates "capacity-constrained" periods (high utilization, moderate tardiness) from "scheduling-constrained" periods (moderate utilization, high tardiness and setup waste).

Process mining distinguishes these by filtering traces: if tardiness persists in low-load, low-disruption subsets, the cause is scheduling logic. If tardiness collapses when utilization drops below 70%, capacity is primary. This separation informs whether interventions should target rules versus capacity expansion.

**4. Developing Advanced Data-Driven Scheduling Strategies**

Three distinct strategies, each leveraging mined data:

**Strategy 1: Composite Dynamic Dispatching (Enhanced Multi-Factor Rules)**  
Core logic: At each work center, when a machine becomes free, rank queued jobs by a composite score:  
Score = w·(due date slack / remaining processing time) + w·(estimated sequence-dependent setup) + w·(downstream queue length / expected processing) + w·(priority weight) – w·(setup time from last job on this machine).  
Weights are tuned via offline optimization on historical traces. Process mining informs this by supplying the setup matrix (for w and w), empirical distributions for remaining times, and learned correlations between factors and tardiness (via regression on past traces).  

Addresses: Poor prioritization and suboptimal sequencing. Expected impact: 25–40% reduction in average tardiness, 15–30% lower setup waste, moderate WIP reduction through better flow.

**Strategy 2: Predictive Rolling-Horizon Scheduling**  
Core logic: Periodically (e.g., every 4–8 hours or on disruption), mine current system state from the live log, predict remaining and future task durations using ML models (random forests or gradient boosting trained on features: job attributes, machine, operator, time-of-day, prior setup type). Feed predictions into a forward simulation or lightweight MIP solver to generate a short-horizon schedule (next 24–72 hours) that minimizes a multi-objective function (weighted tardiness + setup + WIP). Update predictions with actuals.  

Process mining supplies: Training data (actual durations conditioned on context), routing probabilities, and breakdown frequency models. Predictive maintenance signals can be derived from resource event patterns.  

Addresses: Inaccurate estimates and lack of predictive capability. Expected impact: 20–35% lower lead-time variance, proactive bottleneck avoidance, 10–25% tardiness reduction.

**Strategy 3: Setup-Optimized Batching and Sequencing at Bottlenecks**  
Core logic: At identified bottlenecks, apply clustering (e.g., hierarchical or k-medoids on the mined setup similarity matrix) to form "setup families." Use a metaheuristic (genetic algorithm or simulated annealing) to sequence within and across families, minimizing total setup while respecting due dates and downstream loads. In extreme cases, temporarily hold jobs to form better batches if holding cost is offset by setup savings (threshold derived from historical cost analysis).  

Process mining directly provides the setup matrix and identifies high-setup-variance machines.  

Addresses: Sequence-dependent setup inflation and starvation. Expected impact: 30–50% reduction in total setup time at bottlenecks, improved utilization (5–15% effective capacity gain), lower WIP through smoother flow.

**5. Simulation, Evaluation, and Continuous Improvement**

Parameterize a discrete-event simulation (e.g., in AnyLogic, Simio, or custom Python/DEVS) using process-mined artifacts:
- Empirical or fitted distributions (lognormal/Weibull) for task durations per machine/job type.
- Sequence-dependent setup matrix (lookup or regression).
- Routing probabilities and job-mix distributions from trace variants.
- Breakdown inter-arrival and repair times from resource events.
- Priority and due-date distributions.

Test scenarios include: nominal load, +30% demand surge, high disruption rate (2× baseline breakdowns), and mixed (surge + disruptions). Run 50–100 replications per strategy. Compare against baseline (replay of historical dispatching rules) using KPIs: average/max tardiness, mean and variance of flow time, average WIP, machine utilization, total setup time, and throughput. Statistical tests (ANOVA, confidence intervals) identify superior strategies. Sensitivity analysis varies parameters to assess robustness.

For continuous improvement, implement an online process mining pipeline (e.g., via Celonis or custom streaming with Apache Kafka + PM4Py). Track leading indicators weekly: rolling tardiness, setup time per transition, queue-time distributions, and conformance scores. Deploy drift detection (e.g., statistical process control on KPI time series or concept-drift algorithms on process variants). When drift is detected (e.g., new job types increasing setup variance), trigger model retraining on recent logs and re-optimize strategy parameters (weights, batching clusters). A closed-loop dashboard visualizes these metrics alongside simulated "what-if" projections, enabling quarterly strategy reviews or automated rule updates. This creates an adaptive system that evolves with changing shop-floor dynamics.
