### 3. Database Verification Strategies

To test these hypotheses, write SQL queries against `claims`, `adjusters`, and `claim_events` that search for **event patterns** proving each anomaly. Focus on timestamps and activity sequences to detect deviations from the ideal flow (`R  A  (E,P)+  (N or skip)  C`).

#### **Verification for Anomaly 1: Uncontrolled Loop**  
**Hypothesis to test:** *Claims exhibit multiple evaluation-approval cycles (indicating unbounded looping) or zero cycles (indicating skipped evaluation).*  
**Query:**  
```sql
SELECT 
  c.claim_id,
  COUNT(CASE WHEN ce.activity IN ('E', 'P') THEN 1 END) AS ep_event_count,
  COUNT(CASE WHEN ce.activity = 'E' THEN 1 END) AS e_count,
  COUNT(CASE WHEN ce.activity = 'P' THEN 1 END) AS p_count
FROM claims c
JOIN claim_events ce ON c.claim_id = ce.claim_id
GROUP BY c.claim_id
HAVING 
  -- Case 1: Zero E/P events (skipped evaluation/approval entirely due to loop's * allowing 0 iterations)
  (e_count = 0 AND p_count = 0) 
  OR 
  -- Case 2: 2 E events (implies 1 full re-evaluation cycle; ideal = exactly 1 E and 1 P)
  (e_count >= 2)
ORDER BY ep_event_count DESC;
```  
**Why this works:**  
- Ideal claims have `e_count = 1` and `p_count = 1` (one evaluation-approval pair).  
- `e_count = 0` indicates **zero iterations** (loop skipped `E,P` entirely via `*`).  
- `e_count  2` indicates **2 evaluation events** (e.g., `E,P,E,P`  2 cycles), proving unbounded looping.  
- *Hypothesis support:* If results show many claims with `e_count=0` or `e_count2`, it confirms the loop was misimplemented as unbounded/zero-iteration instead of a single cycle.

#### **Verification for Anomaly 2: Non-Deterministic Notification Skip**  
**Hypothesis to test:** *Claims that reached the notification decision point (i.e., completed evaluation/approval) but skipped notification (`N`), with no correlation to claim attributes like low value.*  
**Query:**  
```sql
SELECT 
  c.claim_id,
  c.claim_amount,
  MAX(CASE WHEN ce.activity = 'N' THEN 1 ELSE 0 END) AS notified,
  MAX(CASE WHEN ce.activity IN ('E', 'P') THEN 1 ELSE 0 END) AS completed_eval_approve
FROM claims c
JOIN claim_events ce ON c.claim_id = ce.claim_id
GROUP BY c.claim_id, c.claim_amount
HAVING 
  -- Must have completed evaluation/approval (to have reached the xor point)
  completed_eval_approve = 1 
  AND 
  -- But no notification occurred (skip was taken via XOR)
  notified = 0
ORDER BY c.claim_amount;
```  
**Why this works:**  
- `completed_eval_approve = 1` ensures the claim passed through `A  loop` (so it reached the `xor` point; note: loop completion requires at least one `E` *and* one `P`, but we use `IN ('E','P')` as a proxy since zero iterations would fail this check—see Anomaly 1 query for strict loop completion).  
- `notified = 0` confirms the XOR chose `skip` instead of `N`.  
- **Hypothesis test:** If `claim_amount` shows **no pattern** (e.g., skipped notifications occur randomly across high/low values), it confirms the skip was *non-deterministic* (per the XOR). If skips **only** occur for `claim_amount < 500`, it would support a *correct* cost-saving rule—but the anomaly is that the XOR lacks this condition.  
- *Note:* To strictly verify loop completion, replace `completed_eval_approve` with `(e_count >= 1 AND p_count >= 1)` using subqueries (omitted here for brevity but implied in full implementation).

#### **Verification for Anomaly 3: Premature Closure**  
**Hypothesis to test:** *Claims where closure (`C`) occurs before any evaluation (`E`) or approval (`P`), proving bypass of the core assessment phase.*  
**Query:**  
```sql
SELECT 
  c.claim_id,
  MIN(CASE WHEN ce.activity = 'C' THEN ce.timestamp END) AS close_time,
  MIN(CASE WHEN ce.activity = 'E' THEN ce.timestamp END) AS first_eval_time,
  MIN(CASE WHEN ce.activity = 'P' THEN ce.timestamp END) AS first_approve_time
FROM claims c
JOIN claim_events ce ON c.claim_id = ce.claim_id
GROUP BY c.claim_id
HAVING 
  -- Closure happened BEFORE first evaluation (or approval)
  (close_time < first_eval_time OR close_time < first_approve_time)
  AND 
  -- Ensure closure and evaluation events exist (avoid NULL comparisons)
  close_time IS NOT NULL 
  AND (first_eval_time IS NOT NULL OR first_approve_time IS NOT NULL)
ORDER BY close_time;
```  
**Why this works:**  
- Ideal flow: `close_time` must be **after** both `first_eval_time` and `first_approve_time`.  
- If `close_time < first_eval_time`, closure occurred **before any evaluation** (e.g., `R  A  C`).  
- If `close_time < first_approve_time` but `> first_eval_time`, closure happened after evaluation but before approval (still invalid, but less severe).  
- **Hypothesis test:** Finding claims where `close_time` precedes `first_eval_time` confirms the `A  C` edge enabled premature closure. IfResults show such claims *only* for specific adjuster IDs or regions, it suggests a localized system error (e.g., legacy misconfiguration).
