# Optimizing Resource Assignment at TechSolve Solutions: A Process Mining Approach

---

## Section 1: Analyzing Resource Behavior and Assignment Patterns

### 1.1 Agent and Tier Performance Analysis

The event log's structure—with START/COMPLETE timestamp pairs, agent identifiers, tier labels, skill profiles, and ticket attributes—enables a rich, multi-dimensional resource analysis. I would build the following metric framework:

**Workload Distribution Metrics**
- **Tickets handled per agent per period** (daily/weekly), segmented by priority and category. Computed by counting "Work Lx Start" events per Resource ID.
- **Active working time per agent**: Sum of (Work End – Work Start) intervals per agent, distinguishing *touch time* from *calendar time*. This reveals whether "overloaded" agents are genuinely processing more work or simply holding tickets longer.
- **Concurrent ticket load**: At any timestamp, count tickets in an "assigned but not resolved" state per agent. Plotting this as a time series exposes chronic overload vs. underutilization and intraday/weekly demand-supply mismatches.
- **Workload Gini coefficient or coefficient of variation** within each tier — a single number quantifying how uneven the round-robin distribution actually is in practice (round-robin should produce near-uniform counts; deviation indicates manual cherry-picking or skill-driven concentration).

**Processing Performance Metrics**
- **Activity processing time per agent per ticket type**: Median and distribution of Work duration, stratified by Category × Required Skill × Priority. Comparing the same activity type across agents isolates agent-level efficiency differences from case-mix effects (critical: an agent handling only P1 Network tickets will look "slow" against an agent doing P4 password resets unless you control for ticket type).
- **First-Call/First-Touch Resolution (FCR) rate for L1**: Percentage of cases where the trace pattern is `Ticket Created  Assign L1  Work L1 Start  Work L1 End  Resolve/Close` with no Escalate event. Computed per L1 agent and per ticket category. This is the single most important L1 effectiveness metric.
- **Escalation rate by agent and by category**: Which L1 agents escalate the most? Which categories are essentially never resolved at L1 (suggesting they should bypass L1 entirely)?
- **Rework/bounce-back rate**: Cases where a ticket returns to a lower tier or to a previously-involved agent after escalation — indicating failed escalations or incomplete diagnosis.

**Queue and Responsiveness Metrics**
- **Pickup time**: (Work Start – Assign timestamp) per agent/tier. In the snippet, INC-1001 was assigned to L2 at 09:40 but work started at 10:05 — a 25-minute queue delay. Aggregating pickup times per tier/skill pool quantifies where work waits for people vs. where people wait for work.

### 1.2 Discovering Actual Assignment Patterns vs. Intended Logic

**Handover-of-Work Social Network Analysis (SNA)**
Construct a directed graph where nodes are agents (or agent-tier-skill groups) and edges represent handovers (consecutive activities on the same case performed by different resources, weighted by frequency and average handover delay). This reveals:
- The **real escalation topology**: Is L1L2L3 actually followed, or do tickets skip tiers, ping-pong between L2 agents (as INC-1001 does, B12B15 via Reassign), or flow backwards?
- **Hub agents**: Specialists who receive handovers from many sources — single points of failure and likely bottlenecks.
- **Broker patterns**: Agents whose primary contribution is forwarding rather than resolving (high in-degree and out-degree, low resolution count).

**Role Discovery / Organizational Mining**
Cluster agents by their *activity profiles* (which activities and ticket categories they actually execute) rather than their formal tier labels. Discrepancies between discovered roles and documented tiers are highly diagnostic:
- An L3 specialist whose discovered role profile resembles an L1 generalist  skill misallocation (the exact complaint TechSolve's specialists raise).
- L1 agents with profiles concentrated in one category  informal specialization the dispatcher could exploit formally.

**Conformance Against the Intended Assignment Logic**
Encode the intended rules as a normative model: e.g., "P3 Software-App tickets go to L1 first; escalation requires L1 attempt; L2 assignment must match Required Skill; round-robin within skill pool." Then run conformance checking:
- **Skill-match conformance**: For each Assign event, check whether the assigned agent's Skills attribute contains the ticket's Required Skill. The INC-1001 reassignment (B12 had App-CRM but the ticket actually needed Database-SQL) shows two failure modes to separate: (a) dispatcher assigned someone lacking the skill, vs. (b) the *required skill itself was misidentified* and only corrected mid-stream. The "Required Skill" attribute changing within a case (App-CRM  Database-SQL) is direct evidence of mode (b).
- **Tier-bypass and self-assignment analysis**: INC-1002 shows phone-channel self-assignment by Agent A02. Measure how often self-assignment occurs, by whom, and whether self-assigned tickets perform better or worse — agents may cherry-pick easy tickets.

### 1.3 Skill Utilization Analysis

Build a **skill supply–demand matrix**:

| | Demand (tickets requiring skill) | Supply (agent-hours with skill) | Utilization | Mismatch indicators |
|---|---|---|---|---|
| Networking-Firewall | from Required Skill counts × avg processing time | from agent rosters × availability | demand/supply ratio | queue time for this skill pool |

Key analyses:
- **Skill-pool queue times**: Average AssignWork Start delay per Required Skill. Skills with long queues despite available agents indicate routing failure; long queues with no idle capacity indicate genuine skill shortage.
- **Over-qualification rate**: Percentage of an agent's work time spent on tickets whose Required Skill is below their specialization (e.g., an L3 Database-SQL expert handling Basic-Troubleshoot tasks). Compute as: for each specialist, share of handled tickets where Required Skill  {skills any L1 possesses}. This directly quantifies the specialists' complaint.
- **Skill scarcity vs. skill concentration**: Skills held by only 1–2 agents create fragile dependencies — cross-reference with handover network hubs.

## Section 2: Identifying Resource-Related Bottlenecks and Issues

### 2.1 Pinpointing Specific Problems

**Skill-Availability Bottlenecks**
- Plot the **queue-time distribution per Required Skill per tier**. Overlay with agent availability calendars (inferred from work patterns in the log). A skill where median pickup time spikes during certain hours/days while demand is steady = capacity bottleneck; constant high pickup time = structural understaffing of that skill.
- **Bottleneck attribution**: Decompose total resolution time per case into: queue time (waiting for assignment/pickup), touch time (active work), and transfer time (between handovers). If queue time for `Networking-Firewall` tickets averages 4 hours while touch time is 40 minutes, the problem is availability/routing, not agent performance.

**Reassignment and Escalation Delay Analysis**
- **Reassignment frequency distribution**: Histogram of reassignment count per case. Identify the "ping-pong" tail (2 reassignments) and characterize those cases by category, priority, initial assignee, and dispatcher.
- **Cost per reassignment**: For each Reassign event, measure (next Work Start – Reassign timestamp) plus any duplicated diagnosis time. Aggregate: *"Each reassignment adds a median of X minutes/hours; cases with 2 reassignments take Y% longer end-to-end than zero-reassignment cases with the same priority/category."*
- **Unnecessary escalations**: Tickets escalated to L2/L3 but resolved with skills L1 nominally possesses, or escalated with near-zero L2 touch time (suggesting L1 could have resolved with better knowledge support). Quantify as % of escalations and the L2/L3 capacity they consume.

**Incorrect Initial Assignment Impact**
- Define "incorrect initial assignment" operationally: the first assigned agent neither resolves nor meaningfully progresses the ticket (immediate Reassign, or Required Skill mismatch at first assignment). Compare cycle times: cases with correct first assignment vs. incorrect. Attribute by assignment source (Dispatcher vs. self-assigned vs. system) to locate where the routing decision fails.

**Overloaded / Underperforming Agents**
- Cross-plot **workload (concurrent tickets, hours) vs. performance (median processing time, FCR)** per agent. Quadrants reveal: overloaded-and-slowing agents (capacity issue), low-load-low-output agents (utilization/engagement issue), and high-load-high-output agents (burnout risk and dependence risk).
- Check for **workload-induced degradation**: regress an agent's per-ticket processing time against their concurrent load at ticket start. A positive correlation quantifies congestion effects.

### 2.2 Correlating with SLA Breaches and Quantifying Impact

Enrich each case with a binary SLA-breach flag (computed from priority-specific SLA targets vs. CreatedResolved duration). Then:

- **Breach decomposition**: For each breached P2/P3 case, attribute breach time to phases: L1 queue, L1 work, escalation gap, L2 queue, L2 work, reassignment loops. Result: *"68% of P2 breach time accrues during L2 queue + reassignment phases."* (illustrative)
- **Statistical association**: Compare breach rates between cohorts — e.g., "Tickets with 1 skill-mismatched assignment breach SLA at 45% vs. 12% for skill-matched first assignments." Use logistic regression or a decision tree with features (reassignment count, first-assignment skill match, assigned agent load at assignment, escalation count, channel) to rank drivers of breach probability.
- **Capacity leakage quantification**: "Specialists spend N% of their logged work time on sub-specialty tickets, equivalent to X FTEs of specialist capacity lost" — connecting the over-qualification finding to a business number management understands.

## Section 3: Root Cause Analysis for Assignment Inefficiencies

### 3.1 Candidate Root Causes and How to Test Each

| Hypothesized Root Cause | Diagnostic Evidence from the Log |
|---|---|
| **Skill-blind round-robin assignment** | Assignment sequence per dispatcher shows rotation across agents regardless of Required Skill; first-assignment skill-match rate near random expectation (i.e.,  proportion of pool holding that skill). |
| **Inaccurate/incomplete skill profiles** | Agents repeatedly *successfully resolve* tickets requiring skills not in their documented profile (profile under-states ability), or repeatedly fail/reassign tickets matching their documented skills (profile over-states ability). Both are directly measurable. |
| **Poor initial categorization / skill identification** | The Required Skill attribute *changes during the case* (as in INC-1001: App-CRM  Database-SQL). Measure the skill-relabeling rate per category and per intake channel. High relabeling in web-portal tickets suggests poor intake forms; in phone tickets, poor L1 triage. |
| **No real-time workload visibility** | Assignments to agents already carrying high concurrent load while peers in the same skill pool sit idle — measurable by reconstructing each agent's load at every Assign timestamp and checking whether the dispatcher systematically ignored it. |
| **L1 under-empowerment / excessive escalation** | High escalation rates for ticket types that *some* L1 agents resolve routinely (proving L1 resolution is feasible); short L1 touch times before escalation ("escalate fast" behavior); variance in FCR across L1 agents for identical ticket types indicates a training/knowledge gap, not an inherent complexity barrier. |
| **Queue/priority discipline failures** | P2 tickets overtaken in queues by later-arriving P3/P4 tickets (compare Assign order vs. pickup order within a pool). |

### 3.2 Variant Analysis

Partition cases into cohorts:
- **"Smooth" variant**: created  assigned once  resolved (zero reassignments, 1 escalation).
- **"Turbulent" variant**: 2 reassignments or escalation loops.

Compare the cohorts on every available dimension: category, priority, channel, intake quality (skill relabeled?), dispatcher, time of day/day of week, first assignee's skill match and load. Statistical comparison (chi-square / effect sizes) surfaces what distinguishes turbulent cases. Typical findings in ITSM contexts: turbulent cases concentrate in (a) ambiguous categories (Software-App tickets that are really database issues), (b) specific intake channels, (c) periods when key skill holders are unavailable.

### 3.3 Decision Mining

Treat each assignment/escalation point as a decision node and mine the rules actually governing it:
- **At "Assign L2"**: Train a classifier with features (Category, Priority, Required Skill, time of day, available agents and their loads/skills) and target = chosen agent. If the learned model is essentially "next in rotation," round-robin is confirmed empirically. If the model has predictive power from skills but errors cluster on certain skill combinations, the dispatcher partially routes by skill but fails on ambiguous tickets.
- **At "Escalate vs. Resolve at L1"**: Mine which features predict escalation per L1 agent. If escalation depends more on *which agent* picked up the ticket than on ticket attributes, the root cause is agent capability/confidence variance — a training issue, not a process design issue.
- Critically, decision mining also tells you the **counterfactual potential**: e.g., "Tickets with feature set X are resolved at L1 87% of the time when handled by Agents A03/A07 but escalated 70% of the time by others"  these ticket types are L1-resolvable with knowledge transfer.

## Section 4: Data-Driven Resource Assignment Strategies

### Strategy 1: Skill- and Proficiency-Based Routing with Empirically Validated Skill Profiles

**Issue addressed**: Skill-blind round-robin causing mismatched assignments, reassignments, and the INC-1001 pattern (assigned to App-CRM agent when DB-SQL was needed).

**How process mining informs it**: Rather than trusting documented skill profiles (shown unreliable in Section 3), derive **empirical skill profiles** from the log: for each agent × skill, compute historical resolution success rate, median processing time, and rework rate. This yields a *proficiency score*, not just a binary skill flag. The skill-relabeling analysis also identifies which categories need a confirmation step before routing.

**Routing logic**: Assign each ticket to the agent in the appropriate tier whose empirical proficiency for the Required Skill is sufficient (not necessarily maximal — see Strategy 3), filtered by availability. Proficiency-weighted routing sends complex/P1-P2 tickets to high-proficiency agents and routine tickets to developing agents (which also builds skills).

**Data required**: Historical resolution outcomes per agent×skill (from the log); maintained skill taxonomy aligned with ticket Required Skill values; ongoing feedback loop updating proficiency scores from new resolutions.

**Expected benefits**: First-assignment skill-match rate  (target >90%); reassignment rate  30–50% for skill-mismatch-driven reassignments; direct reduction in the L2 queue+transfer phases that the breach decomposition identified as dominant.

### Strategy 2: Workload- and SLA-Aware Assignment Algorithm

**Issue addressed**: Uneven workload distribution; assignments to overloaded agents while peers idle; P2/P3 SLA breaches driven by queue time rather than work time.

**How process mining informs it**: The reconstructed load-at-assignment analysis quantifies how often current dispatching ignores load, and the congestion regression (Section 2) shows how processing time degrades with load — providing the parameters for the algorithm. SLA breach decomposition identifies which phase to protect.

**Logic**: Among skill-qualified agents, assign based on a composite score: current concurrent load, expected remaining work in queue (using mined per-ticket-type duration estimates), and **SLA urgency** of waiting tickets (time remaining to breach). Implement dynamic priority: as a P3 ticket approaches its SLA deadline, its effective queue priority rises — preventing the "P3 breach creep" TechSolve observes when P3s perpetually yield to fresh P2s.

**Data required**: Real-time queue state per agent (from the ITSM tool); mined duration distributions per ticket type×agent for load forecasting; SLA clocks per ticket.

**Expected benefits**: Workload Gini coefficient  (more even distribution); pickup-time variance ; P2/P3 SLA compliance  specifically by eliminating breach-time accrued in queues; reduced burnout risk for hub agents.

### Strategy 3: Tier-Skipping Predictive Triage + Specialist Shielding

**Issue addressed**: (a) Tickets that L1 essentially never resolves still consume an L1 cycle plus escalation delay; (b) specialists consumed by sub-specialty work.

**How process mining informs it**: The FCR-by-category analysis identifies ticket types with near-zero L1 resolution probability (e.g., if `Security-IAM` tickets are escalated 96% of the time, the L1 step is pure waste — adding ~30+ minutes per ticket as seen in INC-1001's L1 phase). Conversely, decision mining identifies ticket types resolvable at L1 by *some* agents — these should *never* reach specialists. The over-qualification analysis quantifies how much specialist capacity is reclaimable.

**Logic**: 
- Build a predictive triage model (features: category, channel, keywords from description, requester profile) that estimates required skill and L1-resolution probability at ticket creation.
- High L1-resolution probability  route to L1 (to the right L1 agent per Strategy 1).
- Near-zero L1 probability  **route directly to L2/L3 skill pool**, skipping the futile L1 cycle.
- Establish a "specialist shield": tickets predicted as routine are blocked from direct L3 assignment; specialists' queues only receive tickets matching their specialty tier.
- Pair this with **targeted L1 enablement**: the decision-mining finding that some L1 agents resolve ticket types others escalate defines a precise knowledge-transfer curriculum (knowledge articles, shadowing) to raise L1 FCR for those types.

**Data required**: Labeled historical tickets (category, text, outcome tier) for the triage model; ongoing prediction-accuracy monitoring; updated knowledge base mapped to ticket types.

**Expected benefits**: Eliminated dead L1 cycles (cycle-time reduction equal to L1 queue + L1 work + escalation gap for affected categories); specialist over-qualification rate reduced toward <10% of their time; effective specialist capacity gain of measurable FTEs; L1 FCR uplift for the targeted ticket types.

*(A complementary fourth lever: dynamic cross-tier flexing — using mined intraday demand patterns per skill to temporarily move multi-skilled L2 agents to surge categories. This addresses the temporal demand-supply mismatch revealed in Section 2.1.)*

## Section 5: Simulation, Implementation, and Monitoring

### 5.1 Pre-Implementation Simulation

Build a **discrete-event simulation (DES) model parameterized directly from the mined data** — this is the key advantage of process-mining-informed simulation over assumption-based simulation:

- **Arrival process**: Empirical inter-arrival distributions per category×priority×channel, including intraday/weekly seasonality mined from Ticket Created timestamps.
- **Processing time distributions**: Per activity × ticket type × agent (or agent proficiency class), fitted from Work Start/End pairs — not averages, full distributions, so tail behavior (the SLA-relevant part) is preserved.
- **Routing probabilities**: From the discovered process model (escalation rates, reassignment loops) for the *as-is* baseline.
- **Resource model**: Agent rosters, skill/proficiency matrices, shift calendars inferred from activity timestamps.

**Experiments**:
1. **Validate the baseline**: The as-is simulation must reproduce observed KPIs (cycle time distributions, SLA breach rates per priority, reassignment counts) within tolerance — otherwise simulation conclusions are not credible.
2. **Test each strategy individually and combined**: Replace the routing logic with skill-based routing (Strategy 1), workload-aware assignment (Strategy 2), tier-skipping (Strategy 3). Measure projected SLA compliance, cycle times, agent utilization spread, reassignment counts.
3. **Stress and sensitivity tests**: What if demand grows 20%? What if a key Networking-Firewall specialist leaves? What proficiency-score threshold optimizes the speed-vs-skill-development trade-off? What if the triage model's prediction accuracy is only 75% (misroutings reintroduce reassignments)?
4. **Identify negative side-effects pre-deployment**: e.g., strict skill routing may overload high-proficiency agents (utilization >90%  exploding queue times per queueing theory); simulation finds the balancing parameters before agents experience it.

### 5.2 Post-Implementation Monitoring Plan

Deploy a continuous process mining dashboard (live connection to the ITSM event stream), organized in three layers:

**Layer 1 — Outcome KPIs (weekly, management view)**
- SLA compliance rate per priority (P2/P3 trend specifically)
- End-to-end resolution time distribution (median + P90) per category
- L1 First-Call Resolution rate
- Reassignment rate per case; % cases with 2 reassignments

**Layer 2 — Assignment Quality KPIs (daily, operational view)**
- First-assignment skill-match rate (should trend toward >90%)
- Skill-relabel rate (triage accuracy proxy)
- Pickup time (Assign  Work Start) per tier and per skill pool
- Specialist over-qualification rate (% of L2/L3 time on routine work)
- Workload distribution: concurrent-load spread per tier (Gini/CV), with alerts on individual agents exceeding load thresholds
- Tier-skip routing accuracy: % of direct-to-L2 tickets that bounce back to L1 (over-routing) and % of L1 tickets escalated (under-routing)

**Layer 3 — Process Views (continuous conformance)**
- **Live handover network**: Watch for re-emergence of ping-pong patterns or new hub dependencies.
- **Conformance checking against the new assignment rules**: Flag every assignment that violates the deployed routing logic (manual overrides) — overrides aren't necessarily bad, but a rising override rate signals the algorithm is missing something dispatchers know; analyze override cases as a feedback loop.
- **Variant drift monitoring**: Alert when the share of turbulent variants rises above the post-implementation baseline.
- **Model refresh cadence**: Re-mine proficiency scores monthly; retrain the predictive triage model quarterly or on accuracy degradation; re-validate the simulation model semi-annually as the basis for further what-if planning.

**Governance**: Establish a monthly review where the process mining findings (especially override analysis and drift alerts) feed adjustments to routing rules, skill profiles, and L1 training priorities — turning the one-time analysis into a continuous improvement loop.

---

**Summary**: The event log already contains everything needed to (1) prove empirically that round-robin and manual escalation cause the SLA breaches via skill mismatches and queue accumulation, (2) quantify the cost of each reassignment and each unnecessary escalation, and (3) parameterize replacement routing logic and a simulation environment to de-risk its deployment. The proposed combination — empirically-validated skill routing, SLA/workload-aware dispatching, and predictive tier-skipping with specialist shielding — directly targets each of TechSolve's five stated pain points with measurable, monitorable interventions.
