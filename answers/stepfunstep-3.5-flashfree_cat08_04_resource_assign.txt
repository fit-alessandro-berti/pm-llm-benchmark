## 1. Analyzing Resource Behavior and Assignment Patterns

### Metrics for Agent/Tier Performance Analysis
From the event log, I would extract and calculate:

**Workload Distribution Metrics:**
- Cases handled per agent (absolute count and percentage of total)
- Active time vs. idle time (using `Work Start`/`Work End` events)
- Queue time before assignment (between `Ticket Created` and `Work Start`)
- Concurrent cases handled (overlapping work periods)

**Processing Efficiency Metrics:**
- Average handling time per agent/tier (duration between `Work Start` and `Work End`)
- First-call resolution rate for L1 (percentage of tickets resolved without escalation)
- Reassignment frequency per agent (count of `Reassign` events)
- Escalation rate from L1 to L2/L3 by priority/category

**Skill Utilization Metrics:**
- Percentage of time agents work on tickets matching their documented skills
- Skill coverage matrix: Which required skills are present/absent in each tier
- Specialization index: How often high-skill agents (L2/L3 with niche skills) handle basic vs. complex tickets

**SLA Impact Metrics:**
- SLA breach rate per agent/tier
- Correlation between reassignment events and SLA breaches
- Time spent in each tier relative to SLA targets by priority

### Process Mining Techniques for Pattern Discovery

**Resource Interaction Analysis:**
- Build a **handover network** showing which agents/tiers pass tickets to others
- Identify "hub" agents who receive many reassignments (potential bottlenecks)
- Detect circular reassignments (tickets bouncing between agents)
- Compare the **actual flow** (from event log) vs. the **intended flow** (L1  L2  L3 based on skill requirements)

**Social Network Analysis (SNA):**
- Create an agent-to-agent transfer matrix from `Reassign` and `Escalate` events
- Measure **betweenness centrality** to find agents who frequently mediate transfers
- Calculate **clustering coefficients** to detect teams/communities of practice
- Identify isolates (agents rarely involved in handovers) vs. overloaded connectors

**Role Discovery:**
- Use clustering on agent activity patterns to discover **actual roles** vs. declared tiers
- Identify "super-responders" who handle diverse ticket types beyond their tier
- Detect "specialist generalists" who frequently handle out-of-skill assignments

**Variant Analysis:**
- Compare process variants with 0, 1, 2+ reassignments
- Identify common paths leading to excessive reassignments
- Analyze which ticket attributes (priority/category/skill) correlate with smooth vs. problematic flows

## 3. Root Cause Analysis for Assignment Inefficiencies

**Primary Root Causes:**

1. **Static Assignment Rules:** Round-robin within tiers ignores:
   - Current agent workload (queue length, active cases)
   - Skill proficiency levels (binary "has/doesn't have" vs. graded expertise)
   - Ticket complexity variations within same category

2. **Incomplete Skill Profiling:**
   - Agent skills may be outdated (from log: L1 agent with `Basic-Troubleshoot` attempting `Networking-Firewall`)
   - No proficiency levels (e.g., `App-CRM: expert` vs. `App-CRM: basic`)
   - Missing secondary skills that could prevent escalations

3. **Poor Ticket Triage:**
   - L1/dispatchers assign based on **category** (e.g., "Software-App") not **required skill** (e.g., "App-CRM" vs. "Database-SQL")
   - In snippet: INC-1001 required `Database-SQL` but was initially assigned for `App-CRM`
   - No automated skill extraction from ticket description

4. **Lack of Real-Time Visibility:**
   - Dispatchers see agent tier but not:
     - Current workload (active cases, estimated completion times)
     - Skill utilization rates (is specialist already overloaded?)
     - Historical performance with similar tickets

5. **Escalation Culture:**
   - L1 agents escalate prematurely due to:
     - Insufficient training on borderline cases
     - No incentives for first-call resolution
     - Fear of SLA breach if they attempt complex tickets

**Variant Analysis Insights:**
Compare "clean" paths (L1  L2  resolve) vs. "problem" paths (multiple reassignments):
- **Clean paths:** Often have clear skill matches at each step, lower priority (P3-P4), simple categories
- **Problem paths:** 
  - High priority (P2) tickets misrouted initially
  - Tickets with **multiple required skills** (e.g., `App-CRM` + `DB-SQL`)
  - Assigned to agents with partial but not complete skill match
  - Often involve L2 agents who lack one sub-skill, requiring second escalation

**Decision Mining Findings:**
Mining decision points (assignment/reassignment events) reveals dispatcher heuristics:
- Priority: High priority  direct to senior agent (good for P1, bad for P2 that could be L1)
- Category: Hardware  always L2 (even if simple printer issue)
- Round-robin: Default when no clear rule
- **Missing decisions:** No check of agent's current workload, no skill match scoring, no historical success rate with ticket type

## 4. Developing Data-Driven Resource Assignment Strategies

### Strategy 1: Skill-Based Routing with Proficiency Weighting

**Addresses:** Skill mismatch, inappropriate escalations, specialist underutilization

**Implementation:**
1. Create a **skill matrix** with proficiency levels (1-5) for each agent-skill combination
2. Develop **skill-match score** for each ticket:
   ```
   Score = (proficiency_level × skill_importance_weight)
   ```
   (Importance weights derived from historical resolution times by skill)
3. Assign to agent with highest score **among available agents** (considering workload)

**Data Required:**
- Historical resolution times by (agent, skill, priority)
- Current agent availability (from calendar/system)
- Real-time workload (active cases, queue)

**Expected Benefits:**
- 30% reduction in first-try escalations from L1
- 25% fewer reassignments overall
- Better utilization: Specialists spend 80%+ time on high-skill tickets
- SLA improvement: P2 breaches reduced by 20%

### Strategy 2: Workload-Aware Dynamic Assignment

**Addresses:** Uneven distribution, bottlenecks, queue delays

**Implementation:**
1. Calculate **agent capacity score**:
   ```
   Capacity = 1 / (active_cases × estimated_remaining_time + queue_length × avg_handling_time)
   ```
2. At assignment, consider **both skill match (70% weight)** and **capacity score (30% weight)**
3. Implement **queue-based throttling**: If L2 queue > threshold, automatically route simpler L2 tickets to trained L1 agents

**Data Required:**
- Real-time case status tracking
- Accurate estimated resolution times (from historical data by ticket type)
- Agent shift schedules

**Expected Benefits:**
- 40% reduction in workload variance (Gini coefficient)
- 15% reduction in average queue time
- Prevents specialist overload while utilizing generalist capacity
- Faster response for high-priority tickets (specialists not tied up with low-skill work)

### Strategy 3: Predictive Skill Requirement Assignment

**Addresses:** Poor initial categorization, multi-skill tickets, dispatcher heuristics

**Implementation:**
1. Train a **classification model** on historical tickets to predict required skills from:
   - Ticket description text (NLP keywords)
   - Category + priority + channel
   - Requester department/history
2. Output: Probability scores for each required skill (e.g., 70% `App-CRM`, 20% `DB-SQL`, 10% `OS-WindowsServer`)
3. Use this to assign directly to agent with matching **skill combination**, not just primary category

**Data Required:**
- Historical ticket descriptions with final required skills
- Text mining infrastructure
- Continuous feedback loop: When reassignment occurs, label as "initial prediction error" to retrain model

**Expected Benefits:**
- 35% reduction in multi-step escalations for multi-skill tickets
- 50% fewer "wrong skill" reassignments
- Dispatcher workload reduced (automatic recommendation)
- More accurate SLA forecasting

The proposed three-pronged approach—**skill-weighted routing, workload awareness, and predictive triage**—directly addresses the root causes revealed through process mining. 

The key is moving from **static tier-based assignment** to **dynamic, data-informed matching** that considers the four critical dimensions: **ticket requirements, agent capabilities, current workload, and historical success patterns**. Simulation will validate impact before risky changes, while continuous process mining monitoring will ensure the system adapts to evolving ticket patterns and agent skill development.

By implementing these strategies, TechSolve can realistically achieve:
- 20-30% improvement in SLA compliance for P2/P3
- 40-60% reduction in unnecessary reassignments
- More equitable workload distribution
- Significant improvement in agent satisfaction (less frustration with mismatched assignments)
- Better leverage of existing skills without immediate hiring

The most critical first step is **skill matrix validation and enrichment**—without accurate skill data, even the best assignment algorithms will fail. Process mining provides the evidence to compellingly drive this data quality initiative alongside the technical improvements.
