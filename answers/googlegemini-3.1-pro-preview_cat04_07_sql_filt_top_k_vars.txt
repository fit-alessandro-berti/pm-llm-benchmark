Here is the DuckDB SQL query designed to accomplish your requirements. To make the steps clear and maintainability easy, the query sequence is broken down using Common Table Expressions (CTEs).

*Note: In the `TopVariants` CTE, I've used `LIMIT 5` as a placeholder. We recommend replacing `5` with whatever numeric value you wish to use for `K`.*

```sql
WITH CaseVariants AS (
    -- 1. Construct the sequence of activities for each case_id, 
    -- ordered by timestamp to define the process variant.
    SELECT 
        case_id, 
        list(activity ORDER BY timestamp ASC) AS variant
    FROM event_log
    GROUP BY case_id
),
TopVariants AS (
    -- 2 & 3. Group cases by their complete activity sequence, 
    -- count the frequency of each variant, and keep the top K.
    SELECT 
        variant
    FROM CaseVariants
    GROUP BY variant
    ORDER BY COUNT(case_id) DESC
    LIMIT 5  -- <--- Replace 5 with your desired value for K
),
TopKCases AS (
    -- Identify the specific case_ids that follow those top K variants.
    SELECT 
        case_id
    FROM CaseVariants
    WHERE variant IN (SELECT variant FROM TopVariants)
)
-- 4. Return all events from the original event log that belong 
-- only to cases part of the top K variants.
SELECT 
    e.*
FROM event_log e
INNER JOIN TopKCases tkc 
    ON e.case_id = tkc.case_id
ORDER BY 
    e.case_id, 
    e.timestamp ASC;
```

### How it works:
1. **`CaseVariants` CTE:** It aggregates activities together into an ordered list for each `case_id` utilizing DuckDB's `list()` function combined with `ORDER BY`. Because DuckDB natively supports lists, this perfectly defines a "process variant" sequence. *(If you strictly prefer storing variants as strings instead of lists, you can swap `list(...)` with `string_agg(activity, '->' ORDER BY timestamp ASC)`).*
2. **`TopVariants` CTE:** Groups the grouped logs from step 1 by exactly mapping them to the list array. It counts the respective variants and sorts by their occurrence via `ORDER BY COUNT(...) DESC`, truncating the result size to only the upper `K` variants (`LIMIT 5`).
3. **`TopKCases` CTE:** Cross-references the top variants back to the cases, identifying only those `case_id`s that executed those highly frequent paths.
4. **Final `SELECT`:** Grabs exclusively the filtered `case_id`s and performs an `INNER JOIN` back to our original `event_log` table, guaranteeing that *all* (and *only*) events from cases adhering to the dominant top K variants dictate the final tabular output. The final result set is ordered nicely by `case_id` and `timestamp`.
