# DuckDB Query: Top K Process Variants

Here's a complete solution using CTEs:

```sql
-- Set K as needed (here: top 5 variants)
WITH
-- Step 1: Build the ordered activity sequence per case
case_variants AS (
    SELECT
        case_id,
        STRING_AGG(activity, ' -> ' ORDER BY timestamp) AS variant
    FROM event_log
    GROUP BY case_id
),

-- Step 2 & 3: Count cases per variant and pick the top K
top_variants AS (
    SELECT
        variant,
        COUNT(*) AS case_count
    FROM case_variants
    GROUP BY variant
    ORDER BY case_count DESC
    LIMIT 5  -- <<< K
),

-- Step 4a: Identify all cases belonging to a top-K variant
top_cases AS (
    SELECT cv.case_id
    FROM case_variants cv
    INNER JOIN top_variants tv
        ON cv.variant = tv.variant
)

-- Step 4b: Return all original events of those cases
SELECT e.*
FROM event_log e
INNER JOIN top_cases tc
    ON e.case_id = tc.case_id
ORDER BY e.case_id, e.timestamp;
```

## How It Works

**1. `case_variants`** — DuckDB supports an `ORDER BY` clause *inside* aggregate functions, so `STRING_AGG(activity, ' -> ' ORDER BY timestamp)` collapses each case into its ordered activity sequence (e.g., `A -> B -> C`). This string serves as the variant signature.

**2. `top_variants`** — Groups cases by their variant string, counts how many cases follow each variant, and keeps only the top K via `ORDER BY case_count DESC LIMIT K`.

**3. `top_cases`** — Joins back to recover the `case_id`s whose variant survived the top-K cut.

**4. Final SELECT** — Joins the original `event_log` against the qualifying cases, returning every event (with all columns intact) belonging only to top-K-variant cases.

## Notes & Variations

- **Array-based variant (more robust):** If activity names might contain the separator string, use a list instead:
  ```sql
  LIST(activity ORDER BY timestamp) AS variant
  ```
  DuckDB can group and join on `LIST` values directly, avoiding string-collision issues.

- **Tie-breaking within a case:** If two events in the same case share a timestamp, the order is non-deterministic. Add a tiebreaker if available, e.g. `ORDER BY timestamp, activity` or a sequence/event-id column.

- **Ties at rank K:** `LIMIT K` cuts ties arbitrarily. To include all variants tied with the K-th one, replace `top_variants` with a windowed rank:
  ```sql
  SELECT variant FROM (
      SELECT variant,
             DENSE_RANK() OVER (ORDER BY COUNT(*) DESC) AS rnk
      FROM case_variants
      GROUP BY variant
  ) WHERE rnk <= 5
  ```

- **Parameterizing K:** With DuckDB's prepared statements, `LIMIT ?` works directly; in scripts you can also use `SET VARIABLE k = 5;` and `LIMIT getvariable('k')` (DuckDB  0.10).
