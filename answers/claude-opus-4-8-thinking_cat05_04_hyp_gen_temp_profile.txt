# Temporal Profile Anomaly Analysis for Claim Handling Process

## 1. Identified Anomalies in the Temporal Profile Model

After examining the activity-pair timing statistics, the following patterns stand out as irregular:

| Activity Pair | Avg Time | STDEV | Anomaly Type |
|---------------|----------|-------|--------------|
| **(R  P)** | ~25 hours | 1 hour | **Suspiciously low variance** — approvals cluster too tightly around a fixed interval |
| **(P  N)** | 7 days | 2 days | **Excessively long + highly variable** — notification lags far behind approval |
| **(A  C)** | 2 hours | 1 hour | **Implausibly fast closure** — claims close shortly after assignment, bypassing Evaluate/Approve |
| **(E  N)** | 5 minutes | 1 minute | **Too-rapid transition** — evaluation to notification is near-instant, suggesting skipped intermediate steps |

**Summary of concern:**
- **Unnaturally rigid timing** (RP): a near-constant ~25-hour gap implies an artificial or batched scheduling mechanism rather than organic work.
- **Long, unstable delays** (PN): a 7-day mean with ±2-day spread points to a backlog or inconsistent downstream handoff.
- **Short-circuit paths** (AC, EN): closures or notifications happening too quickly imply that mandatory steps may be skipped or logged out of order.

---

## 2. Hypotheses for the Anomalies

**A. (R  P) — Low variance, fixed ~25h interval**
- **H1 (Batch processing):** Approvals are executed by a scheduled batch job (e.g., nightly at a fixed time), producing a near-constant offset from receipt.
- **H2 (SLA-driven automation):** A rule auto-approves claims at a fixed deadline window, masking actual evaluation effort.
- **H3 (Backdated timestamps):** Approval timestamps may be system-generated rather than reflecting real human action.

**B. (P  N) — Long mean, high variance**
- **H1 (Notification backlog):** A queue/resource constraint delays customer notifications; spikes occur when the queue is flushed.
- **H2 (Manual notification step):** Notifications depend on staff availability, creating irregular, person-dependent delays.
- **H3 (External dependency):** Notifications wait on third-party channels (mail, partner systems) with variable turnaround.

**C. (A  C) — Fast closure after assignment**
- **H1 (Premature/auto closure):** Certain claim types (e.g., rejected or duplicate claims) skip Evaluate/Approve and are closed directly.
- **H2 (Logging gaps):** Intermediate events (E, P, N) are not recorded for some claims, collapsing the timeline.
- **H3 (Specific resource behavior):** A subset of adjusters or automated resources close claims without full workflow.

**D. (E  N) — Near-instant transition**
- **H1 (Automated notification trigger):** Notification fires automatically upon evaluation completion, skipping Approve.
- **H2 (Process skip):** The Approve step is bypassed for low-amount or low-risk claims.
- **H3 (Clock/sequencing issue):** Timestamps for E and N share a system-generated value.

## 3. SQL Verification Approaches

### 3.1 Helper: Pivot key events per claim
A reusable pattern to extract per-claim activity timestamps.

```sql
WITH event_times AS (
    SELECT
        claim_id,
        MIN(timestamp) FILTER (WHERE activity = 'R') AS t_receive,
        MIN(timestamp) FILTER (WHERE activity = 'A') AS t_assign,
        MIN(timestamp) FILTER (WHERE activity = 'E') AS t_evaluate,
        MIN(timestamp) FILTER (WHERE activity = 'P') AS t_approve,
        MIN(timestamp) FILTER (WHERE activity = 'N') AS t_notify,
        MIN(timestamp) FILTER (WHERE activity = 'C') AS t_close
    FROM claim_events
    GROUP BY claim_id
)
SELECT * FROM event_times;
```

---

### 3.2 (R  P): Detect rigid / clustered approval intervals

Check how tightly the receive-to-approve gap clusters; a very small STDEV confirms artificial scheduling.

```sql
WITH event_times AS (
    SELECT claim_id,
           MIN(timestamp) FILTER (WHERE activity='R') AS t_receive,
           MIN(timestamp) FILTER (WHERE activity='P') AS t_approve
    FROM claim_events GROUP BY claim_id
)
SELECT
    COUNT(*)                                          AS n_claims,
    AVG(EXTRACT(EPOCH FROM (t_approve - t_receive)))  AS avg_secs,
    STDDEV(EXTRACT(EPOCH FROM (t_approve - t_receive))) AS stdev_secs,
    MIN(EXTRACT(EPOCH FROM (t_approve - t_receive)))  AS min_secs,
    MAX(EXTRACT(EPOCH FROM (t_approve - t_receive)))  AS max_secs
FROM event_times
WHERE t_receive IS NOT NULL AND t_approve IS NOT NULL;
```

**Batch-hour check** — if approvals concentrate at a specific time of day, batching is likely:

```sql
SELECT EXTRACT(HOUR FROM timestamp) AS approve_hour,
       COUNT(*) AS approvals
FROM claim_events
WHERE activity = 'P'
GROUP BY approve_hour
ORDER BY approvals DESC;
```

### 3.3 (P  N): Identify excessive approve-to-notify delays and correlate

Flag claims exceeding the expected range (mean + Z·STDEV; here Z=2  604800 + 2·172800 = 950400 s  11 days).

```sql
WITH event_times AS (
    SELECT claim_id,
           MIN(timestamp) FILTER (WHERE activity='P') AS t_approve,
           MIN(timestamp) FILTER (WHERE activity='N') AS t_notify
    FROM claim_events GROUP BY claim_id
)
SELECT e.claim_id,
       c.claim_type,
       EXTRACT(EPOCH FROM (e.t_notify - e.t_approve)) AS delay_secs
FROM event_times e
JOIN claims c ON c.claim_id = e.claim_id
WHERE e.t_approve IS NOT NULL AND e.t_notify IS NOT NULL
  AND EXTRACT(EPOCH FROM (e.t_notify - e.t_approve)) > 950400
ORDER BY delay_secs DESC;
```

**Correlate delays with the resource performing notification** (tests backlog/manual-step hypotheses):

```sql
WITH approve AS (
    SELECT claim_id, MIN(timestamp) AS t_approve
    FROM claim_events WHERE activity='P' GROUP BY claim_id
),
notify AS (
    SELECT claim_id, MIN(timestamp) AS t_notify, MIN(resource) AS notify_resource
    FROM claim_events WHERE activity='N' GROUP BY claim_id
)
SELECT n.notify_resource,
       COUNT(*) AS n_claims,
       AVG(EXTRACT(EPOCH FROM (n.t_notify - a.t_approve)))/3600 AS avg_delay_hours
FROM approve a JOIN notify n ON a.claim_id = n.claim_id
GROUP BY n.notify_resource
ORDER BY avg_delay_hours DESC;
```

### 3.4 (A  C): Detect premature closures (skipped Evaluate/Approve)

Find claims closed quickly after assignment that lack E or P events.

```sql
WITH event_times AS (
    SELECT claim_id,
           MIN(timestamp) FILTER (WHERE activity='A') AS t_assign,
           MIN(timestamp) FILTER (WHERE activity='C') AS t_close,
           BOOL_OR(activity='E') AS has_evaluate,
           BOOL_OR(activity='P') AS has_approve
    FROM claim_events GROUP BY claim_id
)
SELECT e.claim_id,
       c.claim_type,
       EXTRACT(EPOCH FROM (e.t_close - e.t_assign)) AS assign_to_close_secs,
       e.has_evaluate,
       e.has_approve
FROM event_times e
JOIN claims c ON c.claim_id = e.claim_id
WHERE e.t_assign IS NOT NULL AND e.t_close IS NOT NULL
  AND EXTRACT(EPOCH FROM (e.t_close - e.t_assign)) < 7200      -- under 2h
  AND (e.has_evaluate = FALSE OR e.has_approve = FALSE)
ORDER BY assign_to_close_secs ASC;
```

**Segment premature closures by claim type and region** (via adjuster):

```sql
WITH fast_close AS (
    SELECT ce.claim_id,
           MIN(ce.timestamp) FILTER (WHERE ce.activity='A') AS t_assign,
           MIN(ce.timestamp) FILTER (WHERE ce.activity='C') AS t_close
    FROM claim_events ce
    GROUP BY ce.claim_id
    HAVING EXTRACT(EPOCH FROM (MIN(ce.timestamp) FILTER (WHERE ce.activity='C')
                             - MIN(ce.timestamp) FILTER (WHERE ce.activity='A'))) < 7200
),
assign_resource AS (
    SELECT claim_id, MIN(resource) AS adjuster_resource
    FROM claim_events WHERE activity='A' GROUP BY claim_id
)
SELECT c.claim_type,
       a.region,
       COUNT(*) AS fast_closures
FROM fast_close f
JOIN claims c        ON c.claim_id = f.claim_id
JOIN assign_resource ar ON ar.claim_id = f.claim_id
LEFT JOIN adjusters a ON a.adjuster_id::text = ar.adjuster_resource  -- adjust join to your resource encoding
GROUP BY c.claim_type, a.region
ORDER BY fast_closures DESC;
```

### 3.5 (E  N): Detect skipped Approve and near-instant notification

```sql
WITH event_times AS (
    SELECT claim_id,
           MIN(timestamp) FILTER (WHERE activity='E') AS t_evaluate,
           MIN(timestamp) FILTER (WHERE activity='N') AS t_notify,
           BOOL_OR(activity='P') AS has_approve
    FROM claim_events GROUP BY claim_id
)
SELECT e.claim_id,
       c.claim_type,
       c.claim_amount,
       EXTRACT(EPOCH FROM (e.t_notify - e.t_evaluate)) AS eval_to_notify_secs,
       e.has_approve
FROM event_times e
JOIN claims c ON c.claim_id = e.claim_id
WHERE e.t_evaluate IS NOT NULL AND e.t_notify IS NOT NULL
  AND EXTRACT(EPOCH FROM (e.t_notify - e.t_evaluate)) < 120   -- under 2 min
ORDER BY eval_to_notify_secs ASC;
```

**Test the "low-amount claims skip Approve" hypothesis:**

```sql
WITH event_times AS (
    SELECT claim_id, BOOL_OR(activity='P') AS has_approve
    FROM claim_events GROUP BY claim_id
)
SELECT (c.claim_amount < 1000) AS is_low_amount,
       e.has_approve,
       COUNT(*) AS n_claims
FROM event_times e
JOIN claims c ON c.claim_id = e.claim_id
GROUP BY is_low_amount, e.has_approve
ORDER BY is_low_amount, e.has_approve;
```

### 3.6 Generic deviation detector (any activity pair vs. profile)

Parameterize against the model's mean/STDEV with a configurable ZETA to surface all out-of-range transitions at once.

```sql
-- Example for (P,N) with mean=604800, stdev=172800, zeta=2
WITH pair AS (
    SELECT claim_id,
           MIN(timestamp) FILTER (WHERE activity='P') AS t1,
           MIN(timestamp) FILTER (WHERE activity='N') AS t2
    FROM claim_events GROUP BY claim_id
)
SELECT claim_id,
       EXTRACT(EPOCH FROM (t2 - t1)) AS observed_secs,
       ABS(EXTRACT(EPOCH FROM (t2 - t1)) - 604800) / 172800.0 AS z_score
FROM pair
WHERE t1 IS NOT NULL AND t2 IS NOT NULL
  AND ABS(EXTRACT(EPOCH FROM (t2 - t1)) - 604800) > 2 * 172800
ORDER BY z_score DESC;
```

## Interpretation Guidance

- If **3.2** shows STDEV near zero and **3.2 batch-hour** concentrates at one hour  confirms automated/batch approval (RP).
- If **3.3** delays cluster on specific `notify_resource` values  supports the manual/backlog hypothesis for PN.
- If **3.4** returns many claims missing E/P  confirms premature closure paths; the segmentation reveals whether specific types/regions drive it.
- If **3.5** shows skipped Approve concentrated among low `claim_amount` claims  confirms a fast-track rule for small claims (EN).

Together these queries move each anomaly from a statistical signal to an evidence-backed root cause tied to concrete claims, resources, and segments.
