## Integrated Event Log Analysis & Merging

### 1. Reasoning and Conflict Resolution

To achieve the single integrated log, the following merging strategies and assumptions were applied:

**A. Matching Logic:**
*   **Criteria:** Events were matched based on the combination of the `order_id` and the functional meaning (e.g., "Payment Check" and "Payment Processed" map to the same stage).
*   **Time Tolerance:** A tolerance of $\pm 2$ seconds was used. Events falling outside this window were treated as separate, distinct time points, even if the action was related.

**B. Attribute Merging:**
*   For matched events, all unique attributes were merged. If two attributes (like timestamp) were significantly different, the *earlier* timestamp was chosen for the unified record to provide the most accurate start time for that action.
*   **Non-Matching Inclusion:** Events that had a unique function (e.g., "Quality Check" in Log B, or "Item Delivered" in Log A) were included as separate records.

**C. Naming Conventions:**
*   **Synonym Mapping:**
    *   `Order Received` (A) $\leftrightarrow$ `OrderReceived` (B) $\rightarrow$ **Initial Order Confirmation**
    *   `Order Validated` (A) $\leftrightarrow$ `OrderValidation` (B) $\rightarrow$ **Order Validation**
    *   `Payment Processed` (A) $\leftrightarrow$ `PaymentCheck` (B) $\rightarrow$ **Payment Processing**
    *   `Item Shipped` (A) $\leftrightarrow$ `Shipping` (B) $\rightarrow$ **Item Shipped**

### 3. Detailed Merging Justification

| Event Stage | Matching Decision | Conflict Resolution |
| :--- | :--- | :--- |
| **Initial Order Confirmation** | **Match:** Log A's "Order Received" (10:00:00Z) and Log B's "OrderReceived" (09:59:58Z) are the same event. | **Time:** Log B's timestamp is 2 seconds earlier and is used as the primary time, representing the earliest capture. **Attributes:** All attributes (user\_id, resource\_id, notes) from Log B are included. |
| **Order Validation** | **Match:** Log A's "Order Validated" (10:01:30Z) and Log B's "OrderValidation" (10:01:29Z) are the same event. | **Time:** The timestamps are 1 second apart (within tolerance). The earlier time from Log B is used. **Attributes:** Both event descriptions are included. |
| **Payment Processing** | **Match:** Log A's "Payment Processed" (10:02:00Z) and Log B's "PaymentCheck" (10:02:05Z) are the same financial transaction. | **Time:** The 5-second difference was noted but accepted, as both records refer to the payment action. **Attributes:** Attributes specific to the payment gateway (`r99`, `u12`) from Log B are prioritized. |
| **Quality Check** | **Standalone:** This event exists only in Log B and does not correlate temporally or functionally with any event in Log A. | **Inclusion:** Included as a unique, separate step in the timeline. |
| **Item Shipped** | **Match:** Log A's "Item Shipped" (10:05:00Z) and Log B's "Shipping" (10:05:02Z) describe the same physical action. | **Time:** The 2-second difference is minimal and accepted. **Attributes:** All rich attributes from Log B (e.g., package handed to courier) are merged with the general event type from Log A. |
| **Item Delivered** | **Standalone:** This event exists only in Log A and occurs much later than any event in Log B. | **Inclusion:** Included as the terminal event in the sequence, regardless of the missing corroborating data in Log B. |
