# Comprehensive Process Mining Analysis for Speedy Parcels
## A Data-Driven Approach to Last-Mile Delivery Optimization

## Executive Summary

This analysis outlines a systematic process mining approach to address Speedy Parcels' challenges with delivery punctuality and operational costs. By integrating data from GPS trackers, driver scanners, dispatch systems, and maintenance logs, we can discover actual operational processes, identify bottlenecks, determine root causes of inefficiencies, and propose targeted optimization strategies backed by data-driven insights.

## 1. Process Discovery and Conformance Checking

### 1.1 Data Preprocessing and Integration

#### Integration Strategy

Creating a cohesive event log from four disparate data sources requires a carefully designed Extract-Transform-Load (ETL) pipeline with the following components:

**Case ID Definition:**
The case concept is critical for meaningful process analysis. I recommend a hierarchical case structure:
- **Primary Case Level:** Vehicle-Day (e.g., V12-20241205) for analyzing complete delivery routes
- **Secondary Case Level:** Package-Journey (e.g., P9876) for analyzing individual delivery lifecycles
- **Tertiary Level:** Driver-Shift for analyzing driver performance patterns

**Event Log Schema Design:**

```
Unified Event Log Schema:
 Case ID (Vehicle-Day)
 Activity (standardized activity name)
 Timestamp (normalized to UTC)
 Resource (Driver ID, Vehicle ID)
 Attributes:
    Location (Lat/Lon)
    Speed
    Package ID (if applicable)
    Planned vs. Actual flag
    Stop sequence number
    Time window (start, end)
    Event source (GPS/Scanner/Dispatch/Maintenance)
```

**Data Integration Process:**

| Source | Key Transformations | Linking Strategy |
|--------|-------------------|------------------|
| GPS Trackers | Aggregate high-frequency pings into meaningful events (stops, movements, idle periods); filter noise; calculate derived metrics (distance, idle duration) | Link via Vehicle ID and timestamp correlation |
| Scanner Data | Standardize activity names; validate timestamp sequences; handle missing scans | Link via Vehicle ID, Driver ID, Package ID |
| Dispatch System | Convert planned routes to comparable event format; extract planned timestamps and sequences | Link via Vehicle ID and date; use as reference model |
| Maintenance Logs | Convert to events with start/end activities; categorize by type (scheduled/unscheduled) | Link via Vehicle ID and timestamp ranges |

**GPS Data Processing Algorithm:**

```
For each vehicle-day:
1. Sort GPS pings by timestamp
2. Identify stop events: speed = 0 for > 2 minutes
3. Cluster consecutive stops by location (within 50m radius)
4. Classify stops:
   - Customer stop (matches scanner location within threshold)
   - Traffic stop (duration < 5 min, known congestion area)
   - Unplanned stop (no matching delivery, duration > threshold)
5. Calculate travel segments between stops
6. Merge with scanner events based on spatiotemporal proximity
```

#### Integration Challenges and Mitigation Strategies

| Challenge | Impact | Mitigation Strategy |
|-----------|--------|---------------------|
| **Timestamp Synchronization** | Different system clocks may be offset by seconds/minutes | Implement clock drift correction using depot events as synchronization points; cross-validate with scanner timestamps at known locations |
| **GPS Accuracy in Urban Canyons** | Location errors of 10-50m in dense urban areas | Apply Kalman filtering; use road network snapping; validate against known delivery addresses |
| **Missing Events** | Scanner battery failures, GPS signal loss, manual data entry gaps | Implement imputation algorithms using contextual data; flag imputed events for quality tracking; use GPS data to infer missing scanner events |
| **Granularity Mismatch** | GPS at 10-second intervals vs. discrete scanner events | Aggregate GPS to meaningful events; maintain raw data for detailed analysis when needed |
| **Activity Abstraction** | Converting continuous movement data to discrete activities | Define clear rules for activity detection (e.g., "Traffic Delay" = speed < 10 km/h for > 3 minutes on highway segments) |
| **Data Volume** | 6 months × 50+ vehicles × GPS pings every 10 seconds = billions of records | Implement data summarization at ETL stage; use columnar storage; pre-compute common aggregations |

### 1.2 Process Discovery Approach

#### Discovery Algorithm Selection

I recommend a **multi-algorithm approach** tailored to different analytical needs:

**1. Inductive Miner for Main Process Model:**
- Produces sound, block-structured process models
- Handles noise and infrequent behavior well
- Provides clear visualization of the core delivery process

**2. Fuzzy Miner for Complex Route Analysis:**
- Handles spaghetti processes common in logistics
- Allows filtering by frequency and significance
- Reveals patterns obscured by high variability

**3. Directly-Follows Graph (DFG) for Quick Insights:**
- Rapid initial exploration
- Easy to interpret frequency and performance overlays
- Suitable for stakeholder presentations

### 1.3 Conformance Checking Methodology

#### Reference Model Construction

From the dispatch system, extract planned routes as reference process models:
- Planned stop sequence
- Expected arrival times at each stop
- Planned travel durations between stops
- Customer time windows

#### Conformance Checking Techniques

**1. Token-Based Replay:**
- Replay actual event sequences against planned routes
- Identify fitness issues (missing/extra activities)
- Calculate conformance metrics per vehicle-day

**2. Alignment-Based Conformance:**
- Compute optimal alignments between actual and planned traces
- Quantify cost of deviations
- Distinguish between different deviation types

**3. Temporal Conformance:**
- Compare actual timestamps against planned timestamps
- Calculate time deltas at each checkpoint
- Identify where delays accumulate

#### Deviation Classification Framework

| Deviation Type | Detection Method | Example | Severity |
|---------------|------------------|---------|----------|
| **Sequence Deviation** | Stop visited out of planned order | Stops 5 and 6 swapped | Medium - may indicate driver optimization or planning error |
| **Unplanned Stop** | Stop location not in planned route | 15-minute stop at gas station | Low to High - depends on duration and impact |
| **Skipped Stop** | Planned stop not visited | Customer location bypassed | High - results in failed delivery |
| **Timing Deviation** | Actual vs. planned timestamp difference > threshold | Arrival 45 min later than planned | Variable - depends on time window impact |
| **Extended Dwell Time** | Service time significantly exceeds historical average | 20 min at stop vs. 5 min average | Medium - indicates service issues |
| **Route Deviation** | GPS path differs from planned routing | Different streets taken | Low to Medium - may indicate traffic avoidance |
| **Unauthorized Break** | Extended idle period not at customer or depot | 30 min stop at unknown location | Medium - impacts productivity |

#### Conformance Metrics

```
Fitness Score = 1 - (Cost of Deviations / Maximum Possible Cost)

Temporal Conformance Index = 1 - |Actual_Time - Planned_Time| / Total_Planned_Time

Sequence Conformance = Number of In-Order Stops / Total Planned Stops

On-Route Conformance = Distance on Planned Route / Total Distance Traveled
```

## 2. Performance Analysis and Bottleneck Identification

### 2.1 Key Performance Indicators (KPIs)

#### KPI Framework for Speedy Parcels

| KPI Category | KPI Name | Formula/Calculation from Event Log | Target Benchmark |
|--------------|----------|-----------------------------------|------------------|
| **Punctuality** | On-Time Delivery Rate | (Deliveries within customer time window / Total deliveries) × 100 | >95% |
| | Average Lateness | Mean(Actual_Delivery_Time - Window_End) for late deliveries | <15 min |
| | Time Window Hit Rate | (Deliveries within ±30 min of planned time / Total) × 100 | >90% |
| **Efficiency** | Stops per Hour | Total successful stops / (End_Shift - Start_Shift - Breaks) | >6 stops/hr |
| | Travel Time Ratio | Travel_Time / (Travel_Time + Service_Time) × 100 | <60% |
| | Average Service Time | Mean(Depart_Customer - Arrive_Customer) per delivery | <5 min |
| | First Attempt Success Rate | (First attempt successes / Total delivery attempts) × 100 | >90% |
| **Cost** | Fuel Consumption per Package | Total_Fuel / Packages_Delivered | <0.5L/pkg |
| | Distance per Package | Total_Distance / Packages_Delivered | <2 km/pkg |
| | Cost per Delivery | (Fuel + Labor + Vehicle) / Deliveries | Track trend |
| **Utilization** | Vehicle Utilization Rate | Active_Driving_Time / Shift_Duration × 100 | >70% |
| | Capacity Utilization | Packages_Loaded / Vehicle_Capacity × 100 | >85% |
| | Idle Time Percentage | Idle_Duration / Shift_Duration × 100 | <15% |
| **Reliability** | Failed Delivery Rate | Failed_Attempts / Total_Attempts × 100 | <8% |
| | Breakdown Frequency | Unscheduled_Maintenance_Events / Vehicle_Days | <0.02 |
| | MTBF (Mean Time Between Failures) | Operating_Hours / Number_of_Failures | >500 hrs |

#### KPI Calculation Examples from Event Log

**On-Time Delivery Rate:**
```sql
SELECT 
    COUNT(CASE WHEN delivery_timestamp <= time_window_end THEN 1 END) * 100.0 /
    COUNT(*) as OnTimeRate
FROM delivery_events
WHERE activity = 'Delivery_Success'
    AND date BETWEEN '2024-06-01' AND '2024-12-01'
```

**Travel Time vs. Service Time Ratio:**
```
For each Vehicle-Day:
1. Travel_Time = (duration where vehicle is moving and not at customer)
2. Service_Time = (Depart_Customer timestamp - Arrive_Customer timestamp)
3. Ratio = Travel_Time / (Travel_Time + Service_Time)
```

**Vehicle Utilization:**
```
For each Vehicle-Day:
1. Active_Time = Time from Depart_Depot to Arrive_Depot
2. Productive_Time = Active_Time - Idle_Periods - Unauthorized_Stops
3. Utilization = Productive_Time / (End_Shift - Start_Shift)
```

## 5. Operational Constraints and Continuous Monitoring

### 5.1 Accounting for Operational Constraints

#### Driver Working Hours Constraints

**Regulatory Requirements:**
- Maximum 10-hour shift duration
- Mandatory 30-minute break after 6 hours
- Weekly hour limits (48 hours average)

**Integration into Optimization:**

```
Route Planning Constraints:
1. Maximum planned route duration: 9.5 hours (buffer for delays)
2. Automatic break scheduling at appropriate stops
3. Real-time monitoring of accumulated hours
4. Alert system when approaching limits
5. Automatic route truncation if overtime risk detected

Dynamic Adjustment Rules:
- If current route projected to exceed 10 hours:
  - Re-prioritize remaining stops by time window urgency
  - Transfer lowest-priority stops to next-day or nearby vehicle
  - Notify dispatch for coverage decisions
```

#### Vehicle Capacity Constraints

**Physical Limits:**
- Volume capacity (cubic meters)
- Weight capacity (kg)
- Special handling requirements (fragile, temperature-controlled)

**Integration into Optimization:**

```
Loading Optimization:
1. Pre-route capacity validation
2. Loading sequence optimization (LIFO for route efficiency)
3. Real-time capacity tracking through scanner data
4. Mid-route pickup capability for returns/transfers

Constraint Handling:
- Vehicle assignment considers package mix and total volume
- Split shipments automatically allocated to appropriate vehicles
- Overflow packages flagged for next-day routing
```

#### Customer Time Window Constraints

**Types of Constraints:**
- Hard windows (business hours, perishables)
- Soft windows (customer preferences)
- Premium service windows (guaranteed slots)

**Integration into Optimization:**

```
Priority Framework:
1. Priority 1: Hard time windows (must deliver within window)
2. Priority 2: Premium service commitments
3. Priority 3: Historical failure risk addresses
4. Priority 4: Soft preferences

Dynamic Handling:
- Continuous time window feasibility checking
- Automatic re-sequencing when delays threaten windows
- Customer notification when window cannot be met
- Compensation protocol for premium service failures
```
