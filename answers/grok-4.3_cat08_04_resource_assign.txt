**1. Analyzing Resource Behavior and Assignment Patterns**

Using the event log, resource behavior analysis begins with performance and organizational mining. Filter the log by agent ID, tier, and timestamp to calculate per-agent metrics: average handling time (AHT) per activity (e.g., Work L1 Start to Work L1 End), workload distribution (number of active cases per agent at any timestamp), and first-call resolution (FCR) rate for L1 agents (cases resolved at L1 without escalation). Focus on ticket-type-specific metrics by grouping on Category, Required Skill, and Priority—e.g., L1 agents’ success rate on “Software-App / App-CRM” tickets versus escalation frequency. Tier-level views aggregate these to compare L1 pool performance against L2/L3 specialists.

Process mining reveals actual assignment patterns through resource interaction analysis and social network analysis. Discover handover graphs (e.g., using tools like ProM or Celonis) showing transitions from “Assign L1”  “Work L1 Start”  “Escalate L2”  “Assign L2,” highlighting frequent reassignments (e.g., Agent B12 to B15 in INC-1001). Role discovery algorithms cluster agents by behavior rather than declared tiers, exposing “shadow roles” where L1 agents perform L2-level work or L2 agents handle basic tickets. Compare these discovered models against the documented round-robin + manual escalation logic via conformance checking; deviations appear as high-frequency “Reassign” or “Escalate” activities not predicted by the intended rules. Social network metrics (betweenness centrality, handover frequency) quantify dispatcher influence and specialist overload.

Skill utilization analysis filters cases by Required Skill and matches against Agent Skills. Calculate the percentage of cases where an agent’s documented skills match the Required Skill at assignment time. Identify under-utilization by measuring how often L2/L3 agents (e.g., Agent B12 with App-CRM/DB-SQL) receive tickets below their level (e.g., basic Software-OS tickets) or perform “Work L1” activities after escalation. Skill-mismatch frequency is quantified by counting cases where Required Skill changes post-assignment (as in INC-1001).

**2. Identifying Resource-Related Bottlenecks and Issues**

Process mining performance analysis identifies bottlenecks by computing waiting times between activities (e.g., “Escalate L2” to “Assign L2” or “Work L2 Start”). In the log, frequent delays appear after reassignments, such as the 25-minute gap for INC-1001 between B12’s involvement and reassignment to B15. Bottlenecks are pinpointed where high-volume Required Skills (e.g., Networking-Firewall or Database-SQL) coincide with low agent availability, shown by queue-length traces or resource utilization heatmaps.

Frequent reassignments are quantified by counting “Reassign” events per case and correlating with resolution time. Cases with >1 reassignment show average delays of X minutes (derived from timestamp differences). Incorrect initial assignments are detected via conformance checking: cases routed to L1 despite Required Skill being “Networking-Firewall” (INC-1002) versus successful L1 resolutions. Overloaded agents are identified by resource performance dashboards showing high case counts and elevated AHT, while underutilized agents appear with low activity rates. Skill-mismatch impact is measured by comparing resolution times and SLA breach rates between matched and mismatched assignments.

Correlation with SLA breaches uses decision rules or regression on attributes: cases with reassignments or skill mismatches exhibit higher breach percentages. For example, P2/P3 tickets with escalation chains show breach rates 30-40% above direct-resolution cases. These issues are visualized in dotted charts and resource-centric process maps.

**3. Root Cause Analysis for Assignment Inefficiencies**

Potential root causes include rigid round-robin logic ignoring real-time workload and skills, incomplete agent skill profiles (agents listed with “Basic-Troubleshoot” but handling complex tickets), and poor initial categorization (dispatcher or L1 failing to map Category/Required Skill accurately). Lack of real-time availability data leads to queue buildup, while insufficient L1 empowerment causes premature escalations.

Variant analysis compares “happy path” cases (direct L1 resolution or single skilled assignment) against “problematic” variants (multiple reassignments/escalations). Decision mining on attributes (Priority, Category, Required Skill, Agent Tier at assignment) reveals rules triggering poor outcomes—e.g., “if Required Skill = Database-SQL and assigned to L1 tier  high reassignment probability.” Root causes are further validated by filtering on Notes (e.g., “Needs different skill”) and timestamp patterns showing queue-induced delays. This identifies whether issues stem from static rules, data quality in skill profiles, or missing real-time signals.

**4. Developing Data-Driven Resource Assignment Strategies**

**Strategy 1: Skill-Based Routing with Proficiency Weighting**  
Addresses skill mismatches and under-utilization of specialists. Mining insights show exact Required Skill values and successful resolution rates per skill-agent pair. Implement routing that matches ticket Required Skill to agents with that skill, weighted by historical success rate and proficiency (inferred from resolution speed).  
Data required: Agent skill matrix (updated from log), per-skill success rates, and current availability.  
Expected benefits: 25-35% reduction in reassignments, lower resolution time for complex tickets, and higher specialist utilization on matching work.

**Strategy 2: Workload-Aware Dynamic Assignment**  
Targets uneven workload and SLA breaches for P2/P3 tickets. Use mined resource utilization metrics (active cases, remaining handling time estimates) to route new tickets to the least-loaded qualified agent within the appropriate tier.  
Data required: Real-time queue lengths per agent/tier, estimated remaining time per open case (from performance mining), and agent availability flags.  
Expected benefits: Balanced utilization (reducing overload for agents like B12), fewer queue-induced delays, and improved SLA compliance for medium-priority tickets.

**Strategy 3: Predictive Initial Assignment Using Ticket Characteristics**  
Addresses excessive escalations and incorrect L1 assignments. Train a model on historical cases to predict Required Skill and complexity from Category, Priority, and free-text description (if available). Route directly to the optimal tier/agent at creation.  
Data required: Historical cases with outcome (resolved at L1 vs. escalated), ticket attributes, and resolution outcomes.  
Expected benefits: Higher L1 FCR rate, reduced escalations by 20-30%, and shorter overall cycle time by avoiding initial mismatches.

**5. Simulation, Implementation, and Monitoring**

Business process simulation (using tools like Arena or AnyLogic integrated with mined models) replays the discovered process with the current resource profiles, then tests each strategy by varying assignment rules. Input the event log statistics (activity durations, resource capacities, branching probabilities for escalations) and simulate 12-month runs under different policies. Compare KPIs such as average resolution time, SLA breach rate, and reassignment frequency between baseline and proposed scenarios to quantify impact before rollout.

Post-implementation, deploy process mining dashboards (e.g., in Celonis or Microsoft Power BI connected to the ITSM system) tracking:  
- Resource utilization rate per tier and skill  
- Reassignment rate and average delay per reassignment  
- FCR rate by L1 and by skill match  
- SLA compliance % segmented by Priority and assignment path  
- Skill utilization heatmaps  
- Handover network evolution  

Review weekly: set alerts for rising reassignment rates or declining skill-match percentages. Periodically re-mine the log to detect drift and refine the assignment algorithms.
