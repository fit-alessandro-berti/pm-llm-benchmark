Here is a DuckDB SQL query that satisfies all requirements:

```sql
WITH ordered_events AS (
  SELECT
    case_id,
    activity,
    timestamp,
    LAG(activity, 1) OVER (PARTITION BY case_id ORDER BY timestamp) AS prev_activity,
    LAG(activity, 2) OVER (PARTITION BY case_id ORDER BY timestamp) AS prev_prev_activity,
    LAG(timestamp, 1) OVER (PARTITION BY case_id ORDER BY timestamp) AS prev_timestamp
  FROM event_log
),

matching_sequences AS (
  SELECT DISTINCT
    case_id,
    timestamp AS confirm_timestamp,
    prev_timestamp AS create_order_timestamp
  FROM ordered_events
  WHERE
    activity = 'Confirm Order'
    AND prev_activity = 'Create Order'
    AND prev_prev_activity = 'Approve Purchase Requisition'
    AND (timestamp - prev_timestamp) > INTERVAL 5 DAY
)

SELECT
  e.case_id,
  e.activity,
  e.timestamp
FROM event_log e
WHERE e.case_id NOT IN (
  SELECT case_id
  FROM matching_sequences
)
ORDER BY e.case_id, e.timestamp;
```

---

### Explanation

1. **`ordered_events` CTE**  
   Adds two lagged activity columns and one lagged timestamp, partitioned by `case_id` and ordered by `timestamp`. This lets us detect direct succession of three activities within the same case.

2. **`matching_sequences` CTE**  
   Filters rows where the current activity is `'Confirm Order'`, the immediately preceding activity is `'Create Order'`, and the one before that is `'Approve Purchase Requisition'`.  
   It also checks that the time between `'Create Order'` (`prev_timestamp`) and `'Confirm Order'` (`timestamp`) exceeds 5 days.  
   Returns distinct `case_id`s (and timestamps) for cases meeting both criteria.

3. **Final query**  
   Selects all events from `event_log` excluding any `case_id` found in `matching_sequences`, then orders results by `case_id` and `timestamp`.
