## Explanation

### Case Identification Logic

I used a **document/artifact-centric approach** to define cases, where each unique document, email, or file represents a distinct case (work item). This approach makes intuitive sense because:

1. **Natural Work Boundaries**: Each document/file represents a discrete unit of work with a clear beginning and end
2. **Process Coherence**: All activities related to the same document form a logical workflow sequence
3. **Multitasking Support**: Users often switch between multiple documents; this approach correctly captures interleaved work patterns (e.g., CASE-D1 is interrupted but resumed later)

**Cases Identified:**
- **CASE-QR**: Quarterly_Report.docx (opened at start, worked on at end)
- **CASE-D1**: Document1.docx (main document with interrupted editing)
- **CASE-EM**: Email about Annual Meeting (email handling workflow)
- **CASE-PDF**: Report_Draft.pdf (PDF review session)
- **CASE-XL**: Budget_2024.xlsx (spreadsheet editing session)

### Activity Naming Logic

I transformed low-level system events into **higher-level, process-oriented activities**:

- **FOCUS**  "Open Document/PDF/Spreadsheet" (when first interaction with resource)
- **TYPING**  "Edit Document/Spreadsheet" or "Compose Reply" (context-dependent)
- **SAVE**  "Save Document/Spreadsheet"
- **CLOSE**  "Close Document"
- **CLICK**  Context-specific: "Open Email," "Reply to Email," "Send Email"
- **SCROLL**  "Review Email/PDF" (indicates reading/reviewing behavior)
- **HIGHLIGHT**  "Annotate PDF"
- **SWITCH**  Not included as standalone activity; used only for case context identification

This transformation creates **standardized, analyst-friendly activity names** that are consistent across similar operations while remaining specific enough to distinguish between different types of work (e.g., "Edit Document" vs. "Edit Spreadsheet").

The resulting event log supports standard process mining analyses such as process discovery, conformance checking, and performance analysis.
