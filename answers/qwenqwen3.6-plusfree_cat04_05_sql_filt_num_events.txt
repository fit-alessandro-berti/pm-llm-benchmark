Here is a clean, production-ready DuckDB SQL query that fulfills your requirements:

```sql
SELECT el.case_id, el.activity, el.timestamp
FROM event_log el
WHERE el.case_id IN (
    SELECT case_id
    FROM event_log
    GROUP BY case_id
    HAVING COUNT(*) <= 6
);
```

###  How It Works:
1. **Inner Subquery**: `GROUP BY case_id` groups all events by their case, and `COUNT(*)` calculates the total number of events per case.
2. **Filtering Logic**: `HAVING COUNT(*) <= 6` retains only those `case_id`s that have 6 or fewer events, effectively filtering out longer cases.
3. **Outer Query**: The `WHERE ... IN` clause matches the original `event_log` rows against the filtered list of valid `case_id`s, returning the full event details (`case_id`, `activity`, `timestamp`) for the remaining cases.

###  DuckDB-Optimized Alternative (Recommended)
DuckDB supports the `QUALIFY` clause, which allows you to filter directly on window functions in a single pass. This is often faster and more idiomatic in DuckDB:

```sql
SELECT case_id, activity, timestamp
FROM event_log
QUALIFY COUNT(*) OVER (PARTITION BY case_id) <= 6;
```

**Why `QUALIFY` shines here:**
- Executes in **one table scan** instead of two
- Avoids explicit `GROUP BY` + `JOIN`/`IN` overhead
- Fully leverages DuckDB's vectorized execution engine

###  Best Practices for Process Mining/Event Logs:
- If your table contains additional columns (e.g., `resource`, `cost`), replace the explicit column list with `SELECT el.*` or explicitly list them.
- For large event logs, ensure an index or sort order on `case_id` exists. DuckDB can use sorted data to accelerate window/partition operations.
- If you need deterministic ordering of returned events, append `ORDER BY case_id, timestamp` to the outer query.

Both queries are fully compliant with your specification. Use the standard `IN` version for maximum portability, or the `QUALIFY` version for optimal DuckDB performance.
