## Explanation of Transformation Logic

### Case Identification Strategy

Cases were defined based on **the primary document or task being worked on**, treating each distinct work object as a separate process instance:

- **CASE-001 — Editing Quarterly_Report.docx**: This case is split across two time segments. The user initially opens (focuses on) the quarterly report at 08:59:50, then switches away to work on other tasks, and returns at 09:07:15 to draft the executive summary, save, and close. This reflects a common real-world pattern of interrupted work on a single artifact.

- **CASE-002 — Editing Document1.docx**: Similarly, the user creates/opens Document1, drafts and saves content, switches away, then returns later (09:06:00) to add a budget cross-reference before final save and close. The return is clearly tied to the same document and was prompted by completing the budget spreadsheet work (CASE-005), suggesting the user needed budget data to finalize Document1.

- **CASE-003 — Handling Email about Annual Meeting**: All email-related actions (open inbox, read specific email, scroll, compose reply, type, send) form a coherent email-handling process from receipt to reply.

- **CASE-004 — Reviewing Report_Draft.pdf**: The PDF review session (open, scroll, highlight) represents a discrete review task on a specific document.

- **CASE-005 — Updating Budget_2024.xlsx**: The spreadsheet work (open, update figures, insert row, save) forms a self-contained budget update task.

### Activity Naming Logic

Raw low-level actions were abstracted into **meaningful business-level activities**:

| Raw Action | Context | Translated Activity |
|---|---|---|
| FOCUS (first occurrence on document) | Initial access | **Open Document / Open Spreadsheet** |
| FOCUS (return to previously opened doc) | Re-engagement | **Resume Editing** |
| TYPING (first substantive input) | Content creation | **Draft Content / Draft Executive Summary** |
| TYPING (subsequent input) | Content refinement | **Edit Content / Add Cross-Reference** |
| SWITCH (to new app) | Context change | Absorbed into the **Open** activity of the target case |
| SCROLL | Reading content | **Review Email Content / Review PDF Content** |
| CLICK (Open Email) | Email interaction | **Read Email** |
| CLICK (Reply) | Email composition | **Compose Reply** |
| CLICK (Send) | Email dispatch | **Send Reply** |
| HIGHLIGHT | Annotation | **Highlight Key Section** |
| SAVE | Persistence | **Save Document / Save Spreadsheet** |
| CLOSE | Completion | **Close Document** |

**SWITCH** events were not mapped as standalone activities since they represent transitions between cases rather than meaningful work steps. Instead, the switch timestamp was used as the start of the target case's activity when it represented the first interaction with a new task context.

### Key Design Decisions

1. **Interrupted cases**: CASE-001 and CASE-002 demonstrate that cases can span non-contiguous time intervals, reflecting realistic multitasking behavior. This is preferable for process mining because it preserves the true lifecycle of each work item.

2. **Typing granularity**: Multiple consecutive TYPING events on the same document were distinguished by semantic content (e.g., "Draft Content" vs. "Edit Content") rather than collapsed into a single event, preserving the analyst's ability to see the progression of work.

3. **Cross-case dependencies**: The sequence CASE-005  CASE-002 (budget update followed by adding a budget reference to Document1) suggests a dependency that an analyst could discover through process mining temporal analysis.
