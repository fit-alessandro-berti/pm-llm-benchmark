# Process Tree Analysis & Fix for Procure-to-Pay

## 1. Issues with the Current Model

The current model uses a single **PARALLEL (AND)** operator for the five activities `create_po`, `receive_goods`, `receive_invoice`, `match_invoice`, and `pay_invoice`. An AND-block allows every permutation of these activities, which violates several real-world domain constraints.

### Unwanted behaviors currently allowed

| # | Violation | Why it is wrong |
|---|-----------|-----------------|
| 1 | `pay_invoice` before `receive_goods` or `receive_invoice` | You cannot pay for goods/invoices that haven't arrived. |
| 2 | `pay_invoice` before `match_invoice` | Payment without 3-way matching breaks financial controls. |
| 3 | `match_invoice` before `receive_goods` or `receive_invoice` | You cannot match a document that does not yet exist. |
| 4 | `create_po` after `receive_goods` / `receive_invoice` | Goods and invoices always reference an existing PO. |
| 5 | `receive_goods` / `receive_invoice` before `create_po` | Same as above. |
| 6 | Any interleaving of `match_invoice` with goods/invoice arrival | The 3-way match requires **both** documents to be present. |

### What *should* be allowed in parallel?

Only **receiving goods** and **receiving the invoice** may occur in any order relative to each other (they are independent events triggered by external parties). Every other dependency is strictly sequential.

## 2. Corrected Process Tree

```python
from pm4py.objects.process_tree.obj import Operator, ProcessTree

# --- Define activities ---
create_req     = ProcessTree(label="Create_Purchase_Requisition")
approve_req    = ProcessTree(label="Approve_Purchase_Requisition")
create_po      = ProcessTree(label="Create_Purchase_Order")
receive_goods  = ProcessTree(label="Receive_Goods")
receive_invoice= ProcessTree(label="Receive_Invoice")
match_invoice  = ProcessTree(label="Match_Invoice_to_PO")
pay_invoice    = ProcessTree(label="Pay_Invoice")
close_case     = ProcessTree(label="Close_Case")

# --- Receive goods and receive invoice may occur in any order ---
receive_block = ProcessTree(
    operator=Operator.PARALLEL,
    children=[receive_goods, receive_invoice]
)

# --- Full process: a strict sequence with one nested parallel block ---
root = ProcessTree(
    operator=Operator.SEQUENCE,
    children=[
        create_req,      # 1
        approve_req,     # 2
        create_po,       # 3
        receive_block,   # 4 (goods || invoice)
        match_invoice,   # 5
        pay_invoice,     # 6
        close_case       # 7
    ]
)
```

## 3. Why the Fixed Model Conforms to the Domain

| Step | Enforced by | Domain rationale |
|------|-------------|------------------|
| 1  2 | outer SEQUENCE | A requisition must exist before it can be approved. |
| 2  3 | outer SEQUENCE | The PO is generated from an approved requisition. |
| 3  4 | outer SEQUENCE | A PO must exist before goods or invoices are received. |
| 4 (goods  invoice) | inner PARALLEL | The two receipts are independent — goods may arrive before the invoice, or vice versa. |
| 4  5 | outer SEQUENCE | The 3-way match requires **both** a goods receipt and an invoice; therefore `match_invoice` is placed *after* the parallel block. |
| 5  6 | outer SEQUENCE | Payment is released only after successful matching. |
| 6  7 | outer SEQUENCE | The case is closed only once the invoice is paid. |

### Behaviors that are now impossible

- Paying before matching.
- Matching before the goods receipt or the invoice has arrived.
- Creating a PO after goods/invoice arrival.
- Closing the case before payment.
- Any interleaving of `match_invoice` or `pay_invoice` with the receipt activities.

### Behaviors that remain allowed (intentionally)

The only flexibility preserved is the **permutation of `Receive_Goods` and `Receive_Invoice`**, which mirrors real-world procurement where suppliers and carriers operate on independent timelines. This keeps the model realistic without being over-restrictive.
