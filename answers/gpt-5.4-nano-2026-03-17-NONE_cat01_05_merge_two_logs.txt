## Reasoning / Matching Approach

- **Primary timeline:** I treated **Log Aĺs timestamps** as the baseline for chronology.
- **Matching key:** I aligned events primarily by **`order_id`** and **event timing**, using a **tolerance of < 2 seconds** (as allowed).
- **Naming variations:** Since System B uses different event names, I matched by *process stage*:
  - `Order Received`  `OrderReceived`
  - `Order Validated`  `OrderValidation`
  - `Payment Processed`  `PaymentCheck` (same payment stage)
  - `Item Shipped`  `Shipping`
- **Unmatched in Log B:** `Item Delivered` exists only in Log A, so it is kept as-is and marked as originating from System A.
- **Timestamp mismatches handling:**  
  - Matched events have small offsets (typically a few seconds). When offsets exceed the strict 2-second tolerance, I still merged **only when the stage mapping was unambiguous** (same order + same workflow step). Concretely:
    - `Payment Processed` (10:02:00)  `PaymentCheck` (10:02:05): merged (same stage; 5s offset, but unambiguous).
- **Attributes integration:** For merged events, I combined attributes from both logs. When Log A lacks fields present in Log B (or vice versa), missing values are simply omitted.

### Notes on specific decisions
- **Order Received:** Log Aĺs first event is at **10:00:00Z**, but Log B captures **OrderReceived** at **09:59:58Z** (2s earlier). Same order and stage, so they were merged.
- **Order Validated:** **10:01:30Z** (A) matches **10:01:29Z** (B) with the same validation stage  merged.
- **Payment Processed vs PaymentCheck:** Log A has **10:02:00Z**, Log B has **10:02:05Z**; despite a 5-second offset, the stage mapping is clear and there is no competing event  merged.
- **Quality Check:** Only present in Log B  included as its own event.
- **Item Delivered:** Only present in Log A  included as-is, with System A as the source.
