# Data-Driven Optimization of Resource Assignment at TechSolve Solutions
## A Process Mining Approach for IT Service Desk Optimization

---

## 1. Analyzing Resource Behavior and Assignment Patterns

### 1.1 Agent and Tier Performance Metrics

Using the event log's `Resource`, `Agent Tier`, `Agent Skills`, and `Timestamp` attributes, I would construct a multi-dimensional **resource performance profile** for each agent and tier. The core metrics include:

**Workload & Throughput Metrics:**
- **Tickets handled per agent** (total and per time window) to expose imbalances
- **Concurrent workload** (tickets simultaneously "in progress" per agent), computed from overlapping Work Start/End events
- **Tier-level throughput**: tickets resolved per L1/L2/L3 per day, segmented by priority and category

**Time-Based Metrics:**
- **Processing time** (Work Start  Work End) per agent, per category, per priority
- **Waiting time** (Assign  Work Start) — reveals queue-induced delays
- **Resolution time** (Ticket Created  Resolved) broken down by tier involvement
- **Touch time vs. lead time ratio** — identifies passive waiting in queues

**Quality Metrics:**
- **First-Call Resolution (FCR) rate for L1**: % of tickets completed at L1 without any `Escalate L2/L3` event
- **Reassignment rate per agent**: number of `Reassign` events initiated by the agent
- **Escalation rate** per L1 agent, normalized by ticket category and priority
- **Rework rate**: tickets returning to an earlier tier

**Skill-Match Metrics:**
- **Skill-match ratio**: % of tickets where `Required Skill`  `Agent Skills` of assigned agent
- **Skill breadth vs. depth usage**: do specialists work primarily on tickets matching their top skill?

### 1.2 Revealing Actual Assignment Patterns with Process Mining

Several complementary techniques expose the *real* behavior versus the intended logic:

- **Handover-of-Work Social Network Analysis**: Constructs a directed graph where nodes are agents and edges represent handovers (escalations, reassignments). Edge thickness shows frequency. This uncovers:
  - Hub agents who receive disproportionate reassignments (potential skill gaps or quality issues upstream)
  - Unintended "ping-pong" patterns between L2 agents
  - Clusters that bypass the prescribed L1L2L3 flow

- **Role Discovery / Organizational Mining**: Clusters agents by the activities they actually perform. If the clustering does not align with formal tier boundaries, it indicates role drift (e.g., L2 agents doing L1-level work).

- **Resource Interaction Analysis**: Quantifies collaboration patterns and hand-off delays between specific agent pairs. Combined with waiting time data, this surfaces queue formation at specific specialists.

- **Conformance Checking**: The intended model (L1  escalate if needed  L2  escalate if needed  L3) is checked against actual traces. Deviations such as **direct L1L3 escalations**, **L2L2 reassignments**, or **re-entries to L1** are quantified.

### 1.3 Skill Utilization Analysis

I would build a **skill-utilization matrix** by joining agent skill profiles with ticket `Required Skill`:

- **Skill coverage heatmap**: For each required skill, how many agents are certified vs. demand volume? Bottleneck skills appear as high demand/low supply.
- **Specialist-on-basic-task index**: For L3 agents, the % of time spent on tickets whose required skill is also held by many L1/L2 agents — this quantifies the management concern that specialists are underutilized.
- **Skill-mismatch trace analysis**: Identify traces like INC-1001 where a ticket is assigned to B12 (App-CRM, DB-SQL) but then reassigned because the *actual* need was DB-SQL — suggesting the initial skill identification was wrong, not the agent skill set.

## 2. Identifying Resource-Related Bottlenecks and Issues

### 2.1 Specific Resource Bottlenecks

Using dotted charts, performance sequence diagrams, and queuing analysis:

**Skill-Availability Bottlenecks:**
- Compute **waiting time distribution by required skill**. If `Networking-Firewall` tickets wait on average 45 min before Work Start while `OS-WindowsServer` tickets wait 8 min, the firewall skill pool is undersized.
- Correlate peak ticket inflow times with skill-specific availability gaps.

**Reassignment-Induced Delays:**
- Measure **delta between original assignment and final resolving agent**. In INC-1001, the reassignment added roughly 1h+ of rework and queue time.
- Compute: *Average delay per reassignment event* × *total reassignments per month* = lost productive hours.

**Incorrect Initial Assignment Impact:**
- Filter cases where the first-assigned L1 agent escalates within <10 minutes of `Work L1 Start` — this suggests the ticket never should have gone to L1 (misrouting at the dispatcher).
- Quantify: *% of P2/P3 tickets with escalation time < 10% of total resolution time* = candidates for direct L2 routing.

**Overloaded/Underperforming Agents:**
- Plot tickets-per-agent against average resolution time. Agents in the high-volume/high-time quadrant are overloaded; low-volume/low-time agents are underutilized.
- Gini coefficient on workload distribution quantifies inequality at each tier.

### 2.2 SLA Breach Correlation

Link each ticket's SLA breach flag (priority-based deadline vs. actual resolution time) to resource-related features. Example quantifications:
- "62% of P2 breaches involve 1 reassignment vs. 18% of on-time P2 tickets"
- "Breaches for firewall tickets correlate with being assigned when all 3 firewall specialists had >5 open tickets"
- "Average SLA breach margin is 2.3× higher for tickets experiencing skill mismatch at first L2 assignment"

These statistically grounded statements form the business case for intervention.

## 3. Root Cause Analysis for Assignment Inefficiencies

### 3.1 Candidate Root Causes

| Root Cause | How it Manifests in the Log |
|---|---|
| Round-robin ignores skills/workload | Assignments within a tier are evenly distributed regardless of agent skill match |
| Incomplete/stale skill profiles | Specialists never assigned tickets for skills they possess; recurring reassignments from same agent for same skill |
| Poor ticket categorization | `Required Skill` changes mid-case (as in INC-1001 from App-CRM to Database-SQL) |
| No real-time workload visibility | Assignments continue to agents already queued with many tickets |
| Under-trained L1 | High early-escalation rate for ticket types that historically had high L1 FCR |
| Manual dispatcher discretion | High variance in assignment choice for identical ticket characteristics |

### 3.2 Variant Analysis and Decision Mining

**Variant Analysis**: Segment cases into "smooth" (no reassignments, single-tier resolution or single escalation) versus "turbulent" (2 reassignments or cross-tier ping-pong). Compare them on:
- Ticket category & priority distribution
- Initial assignee characteristics
- Time-of-day of creation
- Dispatcher vs. self-assignment origin

Significant differences reveal predictive features of turbulence.

**Decision Mining**: At each decision point (e.g., "Assign L1"  which agent?; "Escalate"  to L2 or L3?), train an interpretable classifier (e.g., decision tree) using ticket attributes as features and outcomes (resolution time, SLA met, reassignment count) as the target. The resulting rules expose:
- Which ticket features predict a poor assignment
- Whether current routing rules align with what *should* drive the decision
- Feature importance: e.g., "category + priority + time-of-day" might explain 78% of assignment quality variance

## 4. Data-Driven Resource Assignment Strategies

### Strategy 1: Skill-Weighted, Workload-Aware Routing Algorithm

**Issue addressed**: Reassignments due to skill mismatch; delays from assigning to overloaded agents.

**Mining insight leveraged**: Skill-match ratio analysis (§1.3) + agent concurrent workload distributions (§2.1). For every (skill, agent) pair, process mining gives us an empirical **proficiency score** = *f*(past resolution speed, FCR rate, rework rate for that skill).

**Algorithm**: Assignment score = ** × Skill Proficiency  ** × Current Queue Length  ** × Predicted Processing Time. Pick the agent with max score subject to SLA deadline.

**Data required**: Real-time agent queue state; historical proficiency scores per (agent, skill, category); ticket priority and required skill.

**Expected benefit**: 30–50% reduction in reassignments; 15–25% reduction in waiting time; better utilization equality (lower Gini).

---

### Strategy 2: Predictive Triage with ML-Enhanced Skill Identification

**Issue addressed**: Mid-case skill changes (as in INC-1001) caused by inaccurate initial skill identification; inappropriate L1 routing when the ticket clearly needs L2/L3.

**Mining insight leveraged**: Decision mining (§3.2) revealing which ticket text patterns, categories, and channels historically correlate with specific required skills and complexity levels.

**Mechanism**: At `Ticket Created`, a classifier predicts (a) required skill(s) with confidence and (b) a **complexity score** predicting resolution tier. Tickets with high complexity probability skip L1; uncertain ones go to L1 with an attached "likely escalation target."

**Data required**: Historical ticket descriptions (NLP features); ground-truth resolving skill and tier; channel; priority; category.

**Expected benefit**: 20–40% fewer unnecessary L1 touches on high-complexity tickets; reduced mid-case skill reclassification; SLA improvement especially for P2.

---

### Strategy 3: Empowerment-Based Escalation Control

**Issue addressed**: Excessive L1 escalations consuming specialist bandwidth; specialists doing work L1 could handle.

**Mining insight leveraged**: FCR analysis per (L1 agent, category) and comparison of cases where L1 *did* resolve similar tickets vs. escalated them. Identifies "escalation-prone" agents and categories with training opportunity.

**Mechanism**: 
- Build **escalation-justification rules** from historical successful L1 resolutions (e.g., "For Software-App tickets with these symptom patterns, L1 resolved 73% of cases — require checklist completion before escalation").
- Target upskilling of L1 agents with below-peer-average FCR in specific categories.
- Introduce **"fast-path" L1 closure** for low-complexity categories identified via mining.

**Data required**: Categorized L1 resolutions with duration and quality outcomes; agent-level FCR metrics; ticket symptom attributes.

**Expected benefit**: 10–20% increase in L1 FCR; freed specialist capacity redirected to genuinely complex tickets; reduced overall resolution time.

---

### (Optional) Strategy 4: Dynamic Tier-Capacity Balancing

Using time-series analysis of ticket arrivals by skill/priority mined from the log, identify recurring demand patterns (e.g., Monday-morning Software-App spikes) and dynamically flex agent assignments — e.g., temporarily shift cross-trained L2 agents to high-demand skill pools.

## 5. Simulation, Implementation, and Monitoring

### 5.1 Pre-Implementation Simulation

Using the mined process model plus resource availability calendars and probability distributions for inter-arrival and processing times, I would build a **discrete-event simulation** (e.g., in BIMP, ProM's CPN Tools, or Simul8):

1. **Baseline calibration**: Simulate current routing and confirm output metrics (resolution times, SLA rates, reassignment counts) match observed log data within ~5% tolerance.
2. **Scenario simulation**: Replace the routing rule with each proposed strategy and simulate under identical demand conditions.
3. **Sensitivity analysis**: Vary demand +/ 30%, skill availability, and error in skill prediction to test robustness.
4. **Comparative KPIs**: Compare SLA compliance, mean/95th-percentile resolution time, reassignment count, agent utilization Gini, and specialist underutilization index.

This quantifies expected gains and risks without operational disruption.

### 5.2 Post-Implementation Monitoring Plan

Deploy a **live process mining dashboard** (e.g., Celonis, Disco, PM4Py + BI tool) refreshed daily/hourly with these views and KPIs:

**Resource-Centric KPIs:**
- Skill-match ratio at first assignment (target >90%)
- Reassignment rate per ticket (target continuous decline)
- Agent workload Gini coefficient (target <0.25)
- L1 FCR by category (track per-agent trend)
- Specialist underutilization index

**Process-Level KPIs:**
- SLA compliance by priority (P1/P2/P3/P4)
- Mean and 95th-percentile resolution time by category
- Escalation rate and escalation-accuracy rate (escalations not followed by another reassignment)
- Cycle-time distribution, filtered by variant

**Process Views:**
- Conformance dashboard comparing actual routing to the new designed model
- Handover social network — monitor for re-emergence of pathological patterns
- Variant explorer highlighting turbulent cases for root-cause drill-down
- Dotted chart of daily ticket flow by tier for bottleneck detection

**Governance loop**: A monthly review panel inspects drift in KPIs; the ML classifiers (Strategy 2) are retrained quarterly using the growing labeled log; proficiency scores (Strategy 1) update continuously. This establishes a **closed-loop, continuous improvement** framework where process mining doesn't just inform a one-time redesign but sustains optimal assignment over time.

---

**Summary**: By combining performance mining, social network analysis, decision mining, and variant analysis, TechSolve can move from opaque, rule-of-thumb assignment to transparent, skill-weighted, workload-aware routing — with measurable gains in SLA compliance, specialist productivity, and workload equity, validated through simulation and sustained via continuous monitoring.
