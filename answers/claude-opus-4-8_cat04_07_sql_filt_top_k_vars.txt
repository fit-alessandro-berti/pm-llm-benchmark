# Top-K Process Variants Query for DuckDB

Here's a complete SQL query that identifies the top K process variants and returns all events belonging to cases following those variants.

## Solution

```sql
WITH 
-- Step 1: Build the ordered activity sequence (variant) for each case
case_variants AS (
    SELECT
        case_id,
        string_agg(activity, ' -> ' ORDER BY timestamp) AS variant
    FROM event_log
    GROUP BY case_id
),

-- Step 2 & 3: Count cases per variant and rank them by frequency
variant_counts AS (
    SELECT
        variant,
        COUNT(*) AS case_count
    FROM case_variants
    GROUP BY variant
),

-- Rank variants; use case_count DESC, variant for deterministic tie-breaking
ranked_variants AS (
    SELECT
        variant,
        case_count,
        ROW_NUMBER() OVER (ORDER BY case_count DESC, variant ASC) AS variant_rank
    FROM variant_counts
),

-- Select only the top K variants (set K below)
top_k_variants AS (
    SELECT variant
    FROM ranked_variants
    WHERE variant_rank <= 5      -- <-- Set your K here
),

-- Step 4: Identify which case_ids belong to those top K variants
matching_cases AS (
    SELECT cv.case_id
    FROM case_variants cv
    JOIN top_k_variants tk
        ON cv.variant = tk.variant
)

-- Return all original events for the matching cases
SELECT el.*
FROM event_log el
JOIN matching_cases mc
    ON el.case_id = mc.case_id
ORDER BY el.case_id, el.timestamp;
```

## How It Works

| Step | CTE | Purpose |
|------|-----|---------|
| 1 | `case_variants` | Collapses each case into a single ordered sequence using `string_agg(... ORDER BY timestamp)` |
| 2 | `variant_counts` | Groups identical sequences and counts how many cases follow each variant |
| 3 | `ranked_variants` | Ranks variants by frequency (descending), with `variant` as a tie-breaker for determinism |
| — | `top_k_variants` | Keeps only the top K (filter `variant_rank <= K`) |
| 4 | `matching_cases` | Maps the surviving variants back to their `case_id`s |
| Final | — | Joins back to `event_log` to retrieve **all** original events for those cases |

## Key Design Notes

- **Deterministic ordering**: `string_agg(activity, ' -> ' ORDER BY timestamp)` guarantees the sequence respects timestamp order. The separator `' -> '` makes variants human-readable.

- **Tie-breaking**: `ROW_NUMBER()` returns exactly K variants. If you want *all* variants tied at the K-th frequency, replace it with `RANK()` (this may return more than K variants):

  ```sql
  RANK() OVER (ORDER BY case_count DESC) AS variant_rank
  ```

- **Parameterizing K**: In a prepared statement you can replace the literal:

  ```sql
  WHERE variant_rank <= ?
  ```

## Optional Enhancement — Handling Duplicate Timestamps

If multiple events share the same `timestamp` within a case, the order may be ambiguous. Add a stable secondary sort key (e.g., a row id or `activity`) if your table has one:

```sql
string_agg(activity, ' -> ' ORDER BY timestamp, event_id) AS variant
```

This ensures reproducible variant strings across runs.
