Below is a practical process-mining approach I would use for TechSolve.

---

## 1. Analyze Resource Behavior and Assignment Patterns

### 1.1 Start by preparing and enriching the event log
Before analyzing people and assignments, I would make the log analytically robust.

**Core log fields already available**
- Case ID / Ticket ID
- Activity
- Timestamp
- Resource
- Agent tier
- Agent skills
- Ticket priority
- Ticket category
- Required skill
- Lifecycle/timestamp type (`START`, `COMPLETE`)
- Notes / channel

**I would add or derive**
- SLA target and SLA due timestamp
- Resolution outcome (resolved, canceled, reopened)
- First assigned tier/resource
- Number of reassignments and escalations
- Waiting time vs active work time
- Time spent per tier
- Queue time between assignment and work start
- Whether the assigned resource had the required skill
- Skill proficiency level, if available
- Agent calendars/shifts/availability
- Historical workload at the moment of assignment
- Complexity proxies:
  - ticket age at assignment
  - description length / keyword signals
  - number of previous touches
  - number of categories or skill changes

This is important because most ITSM inefficiency is not just in **how long someone worked**, but in **how long the ticket waited for the right person**.

---

### 1.2 Analyze performance of individual agents and tiers

I would use **organizational mining + performance mining** to measure both **capacity** and **quality of assignment**.

#### A. Workload and capacity metrics
At both **agent** and **tier** level:

- **Tickets assigned per day/week/month**
- **Active handling time**
- **Queue length at assignment time**
- **Concurrent WIP** per agent
- **Utilization rate**
  - active work time / available time
- **Load distribution**
  - mean, variance, coefficient of variation
  - Gini coefficient across agents to quantify uneven load
- **Backlog by skill and tier**
- **Peak-hour pressure** by day/time/channel

This will show whether some L2/L3 agents are consistently saturated while others are idle.

#### B. Time and speed metrics
- **Time to first assignment**: `Created  Assign`
- **Time to first touch**: `Created  first Work Start`
- **Assignment-to-start delay**: `Assign  Work Start`
- **Active handling time**: `Work Start  Work End`
- **End-to-end resolution time**
- **Time spent in each tier**
- **Idle/waiting time between handoffs**
- **Reassignment-induced delay**
- **Escalation-induced delay**

For SLA issues, this decomposition matters:
- Is the delay caused by **slow assignment**?
- **Queueing after assignment**?
- **Long actual work**?
- **Rework due to wrong routing**?

#### C. Quality metrics
- **L1 first-call resolution rate (FCR)**
- **Escalation rate from L1 to L2/L3**
- **Reassignment rate**
- **Intra-tier reassignment rate** vs **cross-tier reassignment rate**
- **Reopen rate**
- **Assignment accuracy**
  - first assignment correct, no skill-related reassignment later
- **Bounce rate**
  - tickets that move more than once across agents or tiers

#### D. Case-mix-adjusted agent performance
I would avoid raw comparisons only. L3 agents often handle harder cases.

So I would segment or risk-adjust by:
- priority
- category
- required skill
- channel
- time of day
- complexity proxy
- whether ticket was already escalated

That avoids incorrectly labeling a specialist “slow” when they simply handle the hardest tickets.

---

### 1.3 Reveal actual assignment patterns with process mining

This is where process mining is especially valuable: it shows the **process as it truly runs**, not as designed.

#### A. Process discovery
I would discover the real control-flow of incident handling:
- Created  Assign L1  Work L1  Resolve
- Created  Assign L1  Work L1  Escalate L2  Assign L2  Work L2  Resolve
- Reassignment loops
- Cross-tier ping-pong
- Repeat touch patterns
- Direct self-assignment cases
- Dispatcher-heavy vs agent-driven routing patterns

This immediately exposes whether the “round-robin + manual escalation” model is actually what happens.

#### B. Resource interaction analysis / Social Network Analysis (SNA)
Using handover relations:
- **Handover-of-work network**
  - who passes tickets to whom
- **Subcontracting network**
  - which agents repeatedly pass work to the same specialists
- **Working-together network**
  - clusters of agents frequently touching same cases
- **Centrality analysis**
  - identifies bottleneck experts or informal dispatchers

Typical patterns to look for:
- A small number of L2/L3 agents becoming **handover hubs**
- Certain L1 agents escalating disproportionately to specific specialists
- Dispatcher behavior creating hidden specialization channels
- Reassignment loops around a few skills

#### C. Role discovery
Instead of assuming official tiers explain work, I would discover **actual roles** from behavior:
- some L1 agents may act like “mini app specialists”
- some L2 agents may spend most of their time on generic resets
- some L3 agents may be used as “queue drains” rather than true experts

This can reveal a mismatch between:
- **official org chart**
- **actual work performed**
- **actual expertise used**

#### D. Conformance checking against intended assignment logic
I would formalize the intended rules, for example:
- all tickets go to L1 first, unless defined exception
- round-robin within tier
- escalation based on complexity/skill gap
- L2/L3 only for skill-specific or high-complexity cases

Then compare actual event logs against this policy.

Key deviations:
- direct L2 assignments that bypass L1
- repeated reassignments within the same tier
- manual overrides
- tickets routed to agents without matching skills
- priority not influencing routing urgency
- specialists receiving low-complexity tickets

That gives a precise gap between **designed process** and **operating reality**.

---

### 1.4 Analyze skill utilization across the agent pool

This is critical for TechSolve because the problem is not just staffing; it is **matching ticket skill demand to agent skill supply**.

#### A. Build a demand-supply view
**Demand side**
- ticket volume by required skill
- demand by priority + required skill
- arrival pattern by hour/day/week
- average effort per skill type
- SLA risk by skill type

**Supply side**
- agents with each skill
- proficiency level
- shift coverage
- available hours by skill
- cross-skilled agents

#### B. Key skill utilization measures
- **Skill match rate**
  - % tickets first assigned to an agent with required skill
- **First-assignment skill accuracy**
  - % tickets with no later skill-related reassignment
- **Skill scarcity index**
  - demand hours / available skilled capacity
- **Specialist utilization**
  - active time of L2/L3 in their specialty
- **Specialist leakage**
  - % specialist time spent on work that lower tiers could resolve
- **Overqualification rate**
  - ticket assigned to higher skill/tier than necessary
- **Underqualification rate**
  - ticket assigned to agent lacking required skill/proficiency
- **Skill-change frequency**
  - how often required skill is corrected after initial work

#### C. What this reveals
You can answer questions like:
- Is `Networking-Firewall` genuinely under-capacity?
- Are `Database-SQL` specialists overloaded while `App-CRM` specialists are underused?
- Are L3 agents doing too many P3/P4 tasks?
- Are nominal skill tags misleading because agents have the skill on paper but still reassign cases?

That last point matters. In your snippet, a ticket is assigned to an L2 agent whose profile appears relevant, but later it is reassigned because another, more specific skill is needed. That often indicates one of three things:
1. skill profiles are too coarse,
2. required skill was identified too late,
3. proficiency depth matters more than binary skill presence.

## 2. Identify Resource-Related Bottlenecks and Issues

Once the above analysis is built, I would identify the specific assignment problems and quantify their impact.

---

### 2.1 Bottlenecks caused by insufficient availability of specific skills

I would isolate cases by **required skill** and measure:
- assignment-to-start waiting time
- total queue time
- backlog aging
- SLA breach rate
- utilization of agents with that skill
- time-of-day spikes

A skill bottleneck is likely when:
- tickets requiring a skill wait much longer than others,
- agents with that skill show very high utilization,
- breaches are concentrated in that skill group,
- reassignments toward that skill are frequent.

**Quantification examples**
- Average wait for `Networking-Firewall` tickets vs overall average
- P2 breach rate for `Database-SQL` tickets
- Share of breaches attributable to top 3 constrained skills
- Estimated capacity gap:
  - demand hours for skill minus available hours for skill

---

### 2.2 Delays from reassignments and escalations

Frequent reassignments are often one of the biggest hidden SLA killers.

I would measure:
- **tickets with 0 / 1 / 2+ reassignments**
- **average added waiting time per reassignment**
- **average added handling time per reassignment**
- **extra end-to-end duration caused by escalation loops**
- **breach probability by number of reassignments**

Useful derived measures:
- `Delay caused by reassignment = next Work Start - reassignment timestamp`
- `Rework burden = repeated active work on same ticket before correct resolver is found`

Also distinguish:
- **necessary escalations**  
  from  
- **avoidable reassignments due to poor initial routing**

A healthy process can have escalations. An unhealthy one has **late, repeated, or misdirected escalations**.

---

### 2.3 Impact of incorrect initial assignments

I would define **incorrect initial assignment** as one or more of:
- first assigned agent lacks required skill,
- ticket changes required skill after first handling,
- ticket gets reassigned within a short period for skill reasons,
- first handling ends with “wrong queue / wrong team / needs different skill”.

Then compare against correctly assigned cases:

- time to first resolution
- number of touches
- total queue time
- SLA compliance
- final tier reached
- number of reassignments

**Important KPI**
- **First-assignment accuracy rate**
  - % tickets reaching correct tier/skill on first routing decision

This metric is often more actionable than generic “resolution time”, because it isolates the assignment logic itself.

---

### 2.4 Identify overloaded, underutilized, or underperforming agents/teams

#### Overloaded
Signals:
- very high utilization
- high concurrent WIP
- long queue before start
- repeated ownership of urgent tickets
- overtime or out-of-hours work
- concentration of breach cases

#### Underutilized
Signals:
- low utilization despite compatible skill set
- low assignment volume
- few tickets in periods where similar agents are overloaded
- irregular use despite availability

#### Underperforming
This must be evaluated carefully and fairly.

I would compare agents only after controlling for complexity and ticket mix:
- adjusted handling time
- adjusted escalation rate
- adjusted reassignment rate
- success rate on assigned skill categories

This often reveals:
- true capability gaps,
- skill profile errors,
- training needs,
- or simply bad load balancing.

---

### 2.5 Correlate assignment patterns with SLA breaches

This is where process mining connects directly to business impact.

I would analyze SLA breaches by:
- first assigned tier
- first assigned resource
- required skill
- category
- priority
- number of reassignments
- assignment-to-start delay
- workload at assignment time
- whether assignment matched skill

#### Techniques
- cohort comparison
- regression / survival analysis
- decision rules for breach risk
- process variants ending in SLA breach vs SLA met

Questions to answer:
- How much does each reassignment increase breach probability?
- Do P2/P3 tickets breach mainly because they wait too long in queue, or because they are routed incorrectly?
- Which skill mismatches explain the largest share of breaches?

**Quantification examples**
- % of SLA breaches with at least one skill-related reassignment
- average breach risk for cases with incorrect first assignment vs correct first assignment
- extra median resolution time from intra-L2 reassignment
- % of P3 breaches occurring after waiting > X hours for a scarce skill

## 3. Root Cause Analysis for Assignment Inefficiencies

After locating the symptoms, I would use process mining to identify the likely causes.

---

### 3.1 Likely root causes to investigate

#### A. Current assignment rules ignore skill and workload
If assignment is mostly round-robin within tiers, it may distribute volume “fairly” but route work poorly.

Likely evidence:
- agents receive tickets outside strongest skills,
- frequent same-tier reassignments,
- long waits even while other qualified agents are free,
- no relationship between assignment choice and agent workload.

#### B. Skill profiles are incomplete, outdated, or too coarse
Binary labels like `DB-SQL` may hide meaningful differences:
- beginner vs expert
- read-only support vs performance tuning
- CRM functional skill vs technical integration skill

Likely evidence:
- tickets assigned to “skilled” agents still bounce quickly,
- certain agents frequently reassign despite nominal skill match,
- actual role discovery differs from HR skill records.

#### C. Poor initial categorization or required-skill identification
If ticket metadata is inaccurate, even a good routing engine will misroute.

Likely evidence:
- category/required skill changes after first touch,
- certain channels (phone/web/email) have higher misrouting,
- notes contain recurring keywords not reflected in structured fields,
- certain categories lead to unusually high reassignment rates.

#### D. No real-time view of queue/load/availability
Dispatchers may route to the “next person” rather than the “best available person”.

Likely evidence:
- tickets assigned to overloaded experts while other qualified agents are idle,
- assignment-to-start delay varies sharply by agent despite comparable skills,
- long queueing during predictable demand peaks.

#### E. L1 capability or empowerment is insufficient
If L1 lacks training, tools, or clear decision criteria, too many tickets escalate.

Likely evidence:
- high escalation rates for categories that historically could be resolved at L1,
- strong variation among L1 agents in FCR for similar ticket types,
- specialists spending large shares of time on repeatable routine tasks.

---

### 3.2 Use variant analysis to compare “good” and “bad” assignment paths

Variant analysis is very effective here.

#### Define “good” variants
For example:
- correct first assignment
- 0 reassignments
- resolved in 1 tier or 1 escalation
- SLA met
- short first-touch delay

#### Define “bad” variants
For example:
- 2+ reassignments
- cross-tier ping-pong
- required skill changes after work starts
- long queue before first touch
- SLA breach

Then compare the groups by:
- priority
- category
- required skill
- channel
- opening time/day
- initial assigned tier
- initial assigned agent
- agent workload at assignment time
- whether category or skill was later corrected

This reveals what differentiates smooth cases from problematic ones.

Example insight:
> “P3 Software-App tickets opened via web portal and initially assigned round-robin to generic L2 have a much higher chance of later DB-related reassignment than similar tickets pre-routed to CRM/DB specialists.”

That is directly actionable.

---

### 3.3 Use decision mining to expose poor assignment logic

Decision mining is ideal at these points:
- which tier is assigned first?
- which agent gets the ticket?
- escalate or continue?
- reassign within tier or escalate upward?

I would build decision models using historical features:
- priority
- category
- required skill
- channel
- keywords from description/notes
- queue lengths at decision time
- current agent workload
- time of day
- prior steps already executed

Then compare:
1. **what decisions were actually made**
2. **which decisions historically led to better outcomes**

This often reveals gaps such as:
- dispatchers over-weighting round-robin
- insufficient use of skill data
- decisions being driven by habit or informal networks
- escalation choices that do not reflect evidence from past successful resolutions

In short, decision mining lets TechSolve move from:
> “this is how people assign work today”
to
> “this is what the data says should have been assigned for better outcomes.”

## 4. Develop Data-Driven Resource Assignment Strategies

Below are five concrete strategies. TechSolve could implement them in phases.

---

### Strategy 1: Skill-based routing with proficiency weighting

**Issue addressed**
- Wrong first assignments
- Excessive reassignments
- Specialists receiving mismatched work
- Skill mismatch driving SLA breaches

**How it uses process mining insights**
- Uses discovered mismatch patterns, skill bottlenecks, and actual successful resolver profiles
- Incorporates role discovery to route to who actually resolves a skill category well, not just who is labeled that way

**How it would work**
1. Identify required skill at intake (or predicted skill, see Strategy 3)
2. Build eligible candidate pool based on:
   - skill match
   - proficiency level
   - tier suitability
   - historical success on similar tickets
3. Rank candidates by a score such as:
   - skill/proficiency match
   - historical resolution quality
   - current workload
   - SLA urgency
   - reserve penalty for scarce specialists

**Data required**
- ticket category/priority
- required skill or predicted skill
- agent skill matrix with proficiency
- historical resolution success by skill
- real-time workload and availability

**Expected benefits**
- higher first-assignment accuracy
- fewer reassignments
- reduced queueing from false starts
- better use of true specialists
- better SLA performance for P2/P3

**Important design note**
Do not use only binary skill flags. Use:
- proficiency,
- recency of use,
- historical effectiveness,
- whether the skill is scarce and should be protected.

---

### Strategy 2: Workload- and SLA-aware dispatching

**Issue addressed**
- Uneven workload
- Long assignment-to-start delays
- P2/P3 tickets breaching because work sits in the wrong queue

**How it uses process mining insights**
- Uses queue-time analysis, utilization patterns, arrival peaks, and delay hotspots by tier/skill
- Translates mined waiting-time behavior into dispatch rules

**How it would work**
For eligible agents, assign using a rule such as:
- prioritize tickets by SLA remaining time and business priority
- among suitable agents, favor:
  - shortest expected start time
  - lower current WIP
  - better match for predicted resolution time
- cap maximum concurrent WIP per agent where appropriate

This is much better than pure round-robin.

**Data required**
- real-time agent availability
- current queue lengths and WIP
- SLA due timestamp
- expected effort from historical cases
- skill eligibility

**Expected benefits**
- shorter time to first touch
- better balancing across agents
- fewer hidden backlogs
- reduced overload on “go-to” specialists
- fewer P2/P3 breaches caused by waiting rather than actual work

**Good operating principle**
Use **round-robin only after eligibility and load filters**, not as the main routing rule.

---

### Strategy 3: Predictive triage for required skill and complexity

**Issue addressed**
- Poor initial ticket categorization
- Late discovery of real skill need
- Manual dispatcher inconsistency

**How it uses process mining insights**
- Trains on historical event logs to predict:
  - likely required skill
  - likely tier
  - likely complexity
  - likely risk of reassignment or breach
- Uses variant analysis to learn which intake patterns lead to bad routing

**How it would work**
At ticket creation:
- use structured fields + ticket text + channel + user context
- predict:
  - required skill
  - whether L1 can likely resolve
  - escalation likelihood
  - expected resolution time
- send:
  - high-confidence tickets directly to the best queue
  - low-confidence tickets to expert triage

**Data required**
- historical incident descriptions and notes
- category, channel, priority
- final resolved skill/tier
- resolution time and reassignment history
- knowledge article linkage if available

**Expected benefits**
- better first routing
- fewer early reassignments
- lower dispatcher effort
- earlier identification of complex tickets
- faster path for specialized work

**Practical suggestion**
Make this initially a **recommendation engine**, not full automation:
- show top 3 recommended skills/queues
- show confidence score
- let dispatchers override
- log overrides for continuous learning

---

### Strategy 4: Refine L1 resolution scope and escalation rules

**Issue addressed**
- Specialists doing work that L1 could handle
- Excessive or premature escalations
- Low first-call resolution

**How it uses process mining insights**
- Identifies ticket types that strong L1 agents already resolve successfully
- Uses comparison of L1 success vs escalation patterns
- Finds categories where escalation is habitual rather than necessary

**How it would work**
- Define a list of ticket patterns with strong historical L1 resolution
- Build L1 playbooks / decision trees from those cases
- Require minimum diagnostic steps before escalation for specific categories
- Route certain categories to “L1+” agents with targeted secondary skills
- Create exception rules so truly urgent/high-risk tickets still escalate quickly

**Data required**
- L1 resolution history
- escalation reasons
- diagnostic steps taken
- knowledge base usage
- training data per agent
- category/skill-specific success rates

**Expected benefits**
- improved L1 FCR
- reduced load on L2/L3
- faster resolution for routine tickets
- better specialist focus on high-complexity incidents

**Important caution**
Do not force L1 to retain tickets that should escalate. The goal is **smarter escalation**, not escalation suppression.

---

### Strategy 5: Dynamic capacity reallocation / flex pools for scarce skills

**Issue addressed**
- Time-of-day bottlenecks
- Repeated overload in specific skill clusters
- Some agents idle while scarce-skill queues age

**How it uses process mining insights**
- Uses seasonality and demand patterns mined by skill, category, and time
- Identifies predictable spikes in P2/P3 demand and recurring scarce-skill bottlenecks

**How it would work**
- create flexible pools of cross-skilled agents
- temporarily shift selected L2/L3 or L1+ agents based on live demand
- pre-position capacity for known peak windows
- reserve scarce specialists for high-priority or high-complexity work

**Data required**
- demand forecasting by time and skill
- agent cross-skill map
- schedules/calendars
- SLA risk queue monitoring

**Expected benefits**
- reduced peak-hour queueing
- fewer breaches caused by demand surges
- better use of underutilized capacity
- more resilience without permanent headcount increase

## 5. Simulation, Implementation, and Monitoring

---

### 5.1 Use simulation before implementation

Because assignment changes affect queues, workloads, and SLA outcomes in non-linear ways, I would use **discrete-event simulation** based on the mined process.

#### Build an as-is simulation model from the event log
Calibrate:
- arrival rates by hour/day/channel
- case mix by priority/category/required skill
- actual routing probabilities
- service times by activity, tier, skill, and agent group
- queue disciplines
- calendars and shift patterns
- reassign/escalation probabilities
- SLA deadlines
- agent capacities

#### Test what-if scenarios
For example:
1. current state
2. skill-based routing
3. workload-aware routing
4. predictive triage
5. improved L1 scope
6. flex pool for scarce skills
7. combinations of the above

#### Evaluate outcomes
- SLA attainment by priority
- average and p90 resolution time
- first-touch delay
- queue size and aging by skill
- reassignments per ticket
- L1 FCR
- specialist utilization
- low-skill work handled by L2/L3
- agent workload balance

Simulation is especially valuable to answer:
- If we change routing, do we reduce reassignments but accidentally overload certain experts?
- What happens to P2/P3 if L1 retains more work?
- How many cross-skilled agents do we need to reduce P3 breaches?

#### Important simulation principle
Do not evaluate on averages alone. Track:
- p90/p95 delays,
- breach tails,
- peak periods,
- skill-specific bottlenecks.

That is where service desks usually fail.

---

### 5.2 Implementation approach

A practical rollout plan would be:

#### Phase 1: Baseline and transparency
- clean/enrich event log
- build dashboards
- identify top 5 routing pain points
- establish baseline KPIs

#### Phase 2: Shadow mode
- run new assignment logic in parallel
- compare recommended vs actual routing
- measure expected improvement without operational risk

#### Phase 3: Pilot
- choose one domain, e.g. `Software-App` or `Network`
- pilot with one region/team/shift
- use override logging and dispatcher feedback

#### Phase 4: Controlled rollout
- expand gradually
- tune thresholds and rules
- retrain predictive models regularly
- update skill profiles and governance

---

### 5.3 Monitor effectiveness using process mining dashboards

Post-implementation, I would create continuous process-mining dashboards with drill-down from SLA outcome to path to resource.

#### A. Assignment quality KPIs
- **First-assignment accuracy**
- **Skill match rate**
- **Reassignments per ticket**
- **Intra-tier vs inter-tier bounce rate**
- **% tickets routed to final resolver on first pass**
- **time lost to reassignment**

#### B. SLA and speed KPIs
- SLA attainment by priority/category/skill
- time to first assignment
- time to first touch
- assignment-to-start waiting time
- end-to-end resolution time
- time spent in each tier
- aging backlog by skill and priority

#### C. L1 effectiveness KPIs
- first-call resolution rate
- escalation rate
- escalation appropriateness
- % of historically L1-resolvable tickets actually resolved at L1
- repeat contacts/reopens after L1 resolution

#### D. Resource efficiency KPIs
- utilization by agent/tier
- active time vs waiting time
- WIP per agent
- load balance across agents
- overtime / after-hours handling
- throughput adjusted for complexity

#### E. Skill management KPIs
- demand vs capacity by skill
- specialist leakage
- scarce-skill queue time
- coverage gaps by shift/time window
- skill profile accuracy proxy:
  - tickets assigned on nominal skill but later reassigned

#### F. Process views to track continuously
- process map with performance overlays
- handover social network
- variant explorer:
  - smooth cases vs bounced cases
- conformance view to new routing rules
- heatmap of category  skill  tier transitions
- bottleneck view for queue aging by skill
- trend/drift detection:
  - are teams reverting to manual habits?

---

### 5.4 Governance cadence

I would monitor at three levels:

**Daily operational**
- queues, backlog, SLA risk, overload, scarce-skill hotspots

**Weekly tactical**
- reassignments, first-assignment accuracy, L1 escalation behavior, agent balance

**Monthly governance**
- strategy effectiveness
- skill profile updates
- training needs

## Final Recommendation

TechSolve’s issue is not just “too many tickets” or “not enough people.” It is very likely a **resource-allocation problem** involving:

- imperfect initial classification,
- assignment rules that ignore actual skill fit and live workload,
- excessive reassignments,
- specialist time wasted on work below their level,
- and weak visibility into where delays really occur.

Process mining is the right tool because it can reconstruct:
1. **how tickets actually move**,  
2. **who really handles what**,  
3. **where time is lost**, and  
4. **which routing decisions are most strongly associated with SLA success or failure**.

If TechSolve implements:
- **skill/proficiency-based routing**,
- **workload/SLA-aware dispatching**,
- **predictive triage**,
- and **better L1 escalation rules**,

it should be able to reduce reassignments, free specialist capacity, improve P2/P3 SLA performance, and create a more balanced service desk.
