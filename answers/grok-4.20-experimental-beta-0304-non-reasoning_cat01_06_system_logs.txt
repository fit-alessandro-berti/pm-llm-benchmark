**Explanation of Logic**

**Case Identification:**
- I identified **5 distinct cases** based on the primary document or task the user was working on. This creates a coherent, analyst-friendly log:
  - **Case 1**: Work on the main report (`Quarterly_Report.docx`)
  - **Case 2**: Work on the draft document (`Document1.docx`) — the longest and most fragmented case, with multiple switches
  - **Case 3**: Email handling (Annual Meeting email thread)
  - **Case 4**: PDF review (`Report_Draft.pdf`)
  - **Case 5**: Budget spreadsheet (`Budget_2024.xlsx`)

This grouping reflects real user work sessions: the user is clearly juggling a few related but distinct deliverables.

**Activity Naming:**
I replaced low-level actions (FOCUS, TYPING, SCROLL, CLICK, etc.) with higher-level, process-oriented activity names:
- `TYPING` + context  `Edit Document` / `Edit Spreadsheet` / `Write Email`
- `SCROLL` + `HIGHLIGHT`  `Review PDF` or `Read Email`
- `CLICK` actions interpreted based on context (Open Email, Reply to Email, Send Email)
- `FOCUS` / `SWITCH`  `Open Document` or `Switch Application` when they represent meaningful transitions

**Rationale for Design Choices:**
- The log now tells a clear story of a knowledge worker managing a quarterly reporting process involving document editing, email communication, PDF review, and budget updates.
- Cases are based on the primary artifact (document/file) rather than application or raw window title, which is more meaningful for process analysis.
- Activity names are standardized and consistent across applications where possible.
- Temporal order is fully preserved.

This event log is ready for import into tools like Celonis, Disco, ProM, or PM4Py.
