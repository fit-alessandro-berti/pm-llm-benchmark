### Executive Summary

TechSolve Solutions is suffering from "resource friction"—a disconnect between the demand (ticket complexity/skills required) and supply (agent availability/skill sets). By applying Process Mining to the provided event log, we can move from a reactive, rule-of-thumb assignment model (Round-Robin + Manual) to a predictive, skill-aware optimization model. This approach focuses on eliminating waste (reassignments), respecting SLAs, and maximizing the utility of high-cost L2/L3 resources.

---

### 1. Analyzing Resource Behavior and Assignment Patterns

To understand the current state, we must move beyond simple headcount and analyze the *digital footprint* of every interaction.

**A. Key Performance Metrics (Resource-Centric)**
Using the `Resource`, `Agent Tier`, and `Timestamp` columns, we would calculate:
*   **Load Balance Index:** Compare the standard deviation of active cases per agent within each Tier. High deviation confirms management’s suspicion of uneven distribution.
*   **True Utilization:** Distinguish between *busy time* (handling tickets) and *waste time* (gaps between assignment and start, often caused by multitasking or context switching).
*   **First Contact Resolution (FCR) Rate (L1 Specific):** Percentage of tickets resolved by L1 without `Escalate` or `Reassign` events. A low FCR rate indicates either a skills gap in L1 or poor initial triage.
*   **Specialist Leakage:** Measure the percentage of L2/L3 activities where `Ticket Category` is basic (e.g., Password Reset). This identifies expensive resources doing cheap work.

**B. Process Mining Techniques for Pattern Discovery**
*   **Social Network Analysis (SNA) – Handover of Work:**
    *   *Methodology:* Construct a network graph where nodes are Agents and edges are ticket transfers (`Assign`, `Escalate`, `Reassign`). Edge thickness represents frequency.
    *   *Insight:* This reveals "Super-Connectors" (agents who fix everyone else's mess) and "Black Holes" (agents who pick up tickets but immediately dump them). For example, if Agent B12 frequently hands off to B15, it suggests B12 may be a triage point rather than a resolver, or lacks specific sub-skills despite being in the correct Tier.
*   **Role Discovery & Hierarchy Mining:**
    *   *Methodology:* Algorithmically group agents based on the *actual* activities they perform (e.g., clustering those who do mostly `Work L1` vs. `Escalate L2`).
    *   *Insight:* We compare this "discovered" role structure against the formal hierarchy (L1/L2/L3). If we see a group of L1 agents performing L2 activities successfully, they are candidates for promotion or upskilling. Conversely, if L2 agents are performing L1 activities, the routing logic is broken.
*   **Skill Utilization Matrix:**
    *   Cross-reference `Agent Skills` with `Required Skill`.
    *   *Analysis:* Identify frequency of exact matches vs. partial matches. If agents without `App-CRM` are frequently assigned CRM tickets, we have found the root of reassignments.

### 2. Identifying Resource-Related Bottlenecks and Issues

By overlaying performance data on the process flow, we pinpoint specific operational failures.

**A. Pinpointing Specific Bottlenecks**
*   **The "Ping-Pong" Effect (Rework Loops):**
    *   Using the snippet example (**INC-1001**): Agent B12 held the ticket from 10:05 to 11:15 (1 hour 10 mins) before reassigning it because of a skill mismatch ("Needs different skill").
    *   *Quantification:* Calculate **"Bottleneck Waste Time"**: The duration between `Work L2 Start` (B12) and `Reassign`. This represents lost productive time for a high-value resource.
*   **Skill Bottlenecks (Queue Analysis):**
    *   Analyze the duration between `Assign L2` and `Work L2 Start`. If this duration spikes specifically for `Category: Security` or `Required Skill: Networking-Firewall`, there is a capacity shortage of specialists in that domain.
*   **Correlation with SLA Breaches:**
    *   Perform a decision-point analysis on breached tickets (P2/P3).
    *   *Hypothesis Testing:* Check if breached tickets statistically contain more `Reassign` events than non-breached tickets. We expect to find that every instance of reassignment adds a fixed overhead (e.g., 45 mins) that pushes P2/P3 tickets past their threshold.

**B. Quantifying Impact (Hypothetical Output)**
*   **Metric:** *Avg. Handling Time (AHT) Increase due to Reassignment.*
    *   *Finding:* Tickets with 0 reassignments avg 4 hours; Tickets with 1+ reassignments avg 9 hours. (125% inefficiency tax).
*   **Metric:** *Skill Match Accuracy.*
    *   *Finding:* Only 60% of assignments result in an exact match between `Required Skill` and primary `Agent Skills`.

### 3. Root Cause Analysis for Assignment Inefficiencies

We need to understand *why* the dispatcher or round-robin logic makes bad choices.

**A. Potential Root Causes**
*   **Deficient Routing Rules:** Round-robin ignores queue depth. If Agent A has 10 tickets and Agent B has 0, round-robin still sends the next ticket to Agent A (if it's their turn), causing delays.
*   **The "False Positive" Escalation:** L1 agents escalating because of missing Knowledge Base articles, not actual inability.
*   **Categorization Errors:** If `Ticket Category` is set incorrectly at creation (e.g., "Software" instead of "Access"), the skill-routing engine fails.

**B. Advanced Analysis Methods (The "How")**
*   **Variant Analysis:**
    *   Isolate the "Top 3 Most Frequent Process Variants."
        *   *Variant A (Happy Path):* Create -> L1 -> Resolve.
        *   *Variant B (The Leak):* Create -> L1 -> Escalate -> L2 -> Resolve.
        *   *Variant C (The Chaos):* Create -> L1 -> L2 -> Reassign -> L2 -> Resolve.
    *   Compare the attributes (Priority, Category) of Variant C against Variant A. If Variant C consists entirely of "App-CRM" tickets, we know CRM is our problem child category.
*   **Decision Mining:**
    *   Treat "Escalate" vs. "Resolve" as a decision outcome.
    *   *Model Inputs:* `Resource` (who is working?), `Ticket Priority`, `Time of Day`, `Text Description Length`.
    *   *Output:* The model might reveal that **Agent A05** resolves 90% of P3 tickets unless the text description exceeds 200 words. This suggests A05 is good at simple tasks but struggles with complex logging, implying a need for better triage or training.

### 4. Developing Data-Driven Resource Assignment Strategies

Based on the findings, here are three concrete strategies to optimize the desk.

#### Strategy 1: Implement "Proficiency-Weighted" Skill Routing
*   **Issue Addressed:** Skill mismatches and specialists handling basic tasks.
*   **Concept:** Move beyond binary "Has Skill Y/N." Assign a dynamic **Proficiency Score** (0-100) based on historical data:
    *   *Formula Proxy:* `(Speed Score * 0.4) + (Success Rate * 0.6)` for specific `(Agent, Skill)` pairs.
*   **Implementation Logic:**
    1.  When `INC-xxxx` arrives with `Required Skill: App-CRM`.
    2.  Filter available agents possessing `App-CRM`.
    3.  Rank them by Proficiency Score.
    4.  **Constraint:** If the top expert (Score 99) is busy, route to the next best (Score 85) *only if* their current workload < Threshold. Otherwise, queue for the expert.
*   **Expected Benefit:** Reduces reassignments (wrong person) and ensures experts are used only when necessary.

#### Strategy 2: Predictive "Tier-Jump" Triage (Complexity Detection)
*   **Issue Addressed:** Wasted L1 effort on tickets destined for L2 (The "Pass-Through" problem).
*   **Concept:** Use historical data to train a model that predicts the probability of L1 Resolution at the moment of ticket creation.
*   **Data Leveraged:** `Ticket Category`, Keywords from description (NLP), `Reporter ID` (repeat callers?), `Priority`.
*   **Implementation Logic:**
    *   If `P(Resolution by L1) < 15%`: Automatically tag as "Direct L2" and bypass L1 pool entirely.
    *   *Example:* Any ticket categorized as `Security-IAM` with Priority `P1` historically has a 98% escalation rate. The system should auto-route these to L3, saving L1 30 minutes of reading time.
*   **Expected Benefit:** Faster resolution for complex tickets; L1 focuses only on solvable incidents (increasing their morale and FCR).

#### Strategy 3: Workload-Aware Dynamic Queuing
*   **Issue Addressed:** Uneven distribution and SLA breaches caused by overloaded agents.
*   **Concept:** Replace static Round-Robin with **Shortest Processing Time (SPT)** or **Least Utilized** logic, adjusted for Skill.
*   **Implementation Logic:**
    *   Maintain a real-time "Capacity Vector" for every agent: `[Available Slots]` + `[Estimated Time to Clear Queue]`.
    *   Assignment Rule: Select the agent who can start the work *soonest* (considering their current queue) among those qualified.
    *   **Churn Prevention:** Lock a ticket to an agent for the first 15 minutes to prevent "cherry-picking" (agents closing/reopening tickets to game the queue).
*   **Expected Benefit:** Smooths workload distribution, eliminates "Idle Agent A" while "Overloaded Agent B" exists, directly reducing Wait Times (SLA component).
