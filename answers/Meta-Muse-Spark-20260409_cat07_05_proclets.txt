This visualization shows how **Proclets** work - a workflow modeling approach where you split a complex process into smaller, communicating pieces instead of one giant flowchart. 

The diagram has 3 parts: (a) the proclet interaction model, (b) a class diagram, and (c) an example message.

### (a) Two proclet classes connected through two channels
This is the main model. You have two independent "proclets" that represent different lifecycles:

**Left side: Lab visit proclet**
Models the clinical workflow with the patient
- **Roles involved**: doctor, lab assistant 
- **Steps**: Order entry → Take blood sample → Judge results → Either "Everything fine" or "Do tests again" → Send report
- **Key logic**: After judging results, there's a decision point. If results are fine, it ends. If not, it loops back to redo tests.

**Right side: Lab test proclet** 
Models the lab's technical workflow
- **Roles involved**: chemical analyst
- **Steps**: Perform test → Make report → Finish lab test. If asked to redo, it does "Perform test again" → "Provide updated report"

**How they talk: The two channels**
The proclets don't call each other directly. They communicate via message channels:
1. **Order system channel**: Lab visit sends a "Create" message to start a Lab test. Example in table (c): `Can you perform a HGB test for John?`
2. **HIS channel**: Lab test sends results back to Lab visit via "Receive result" and "Receive updated result" messages

The dashed boxes show the channel interfaces. The `1..1` and `1..*` notations are multiplicities - e.g. one Lab visit creates exactly one order message, but HIS can send multiple result messages back.

### (b) Class diagram 
This just formalizes the relationship: A **Lab visit** `requires` a **Lab test**. Multiplicity `1..1` to `1..*` means one visit requires one or more tests.

**The big idea**
Instead of modeling "patient comes in → test happens → report sent" as one massive flowchart, Proclets let you model the clinic workflow and lab workflow separately. They run independently and only sync via defined messages. This makes complex, distributed processes easier to design and change.

The looping between "Do tests again" and "Perform test again" shows how the two proclets coordinate a redo cycle without being tightly coupled.
