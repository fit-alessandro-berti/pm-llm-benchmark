Below is a process-mining–driven, data-first approach to diagnose and optimize TechSolve’s incident resource assignment logic across L1/L2/L3, using the year-long event log.

---

## 1) Analyzing Resource Behavior and Assignment Patterns

### 1.1 Build an event-log data model for process mining
First, normalize the event log into process-mining–ready attributes:

- **Case level**: `Ticket ID`, `Ticket Priority`, `Category`, `Required Skill` (or mapped skill), `Channel`, timestamps.
- **Activity level**: activities like `Assign L1`, `Work L1 Start/End`, `Escalate L2`, `Reassign`, etc.
- **Resource level**: `Agent ID`, `Agent Tier`, `Agent Skills`.
- **Timing**: for each activity compute:
  - **Cycle time**: `Work End - Work Start`
  - **Queue time**: `Work Start - previous completion`
  - **Handover delay**: time between assignment/reassign and actual work start.

Also derive:
- **Skill match indicator**: `Assigned skills  Required skill  `.
- **Skill under-qualification**: specialist assigned to ticket below their needed skill (if your policy expects it) or assigned without required skill.
- **Escalation count** per case and per tier.
- **Reassignment count** and “reassignment type” (within-tier vs cross-tier vs skill-driven vs random/unknown).

---

### 1.2 Metrics to analyze agent and tier behavior
Focus on workload, timing, and skill utilization.

#### A) Workload distribution & utilization
Per agent and per tier:
- **#Cases handled** (by Priority and Category)
- **Total work time** and **work time per case**
- **Average queue time before each “Work Start”**
- **Concurrency proxy** (if timestamps allow): overlap patterns of work intervals
- **Bottleneck potential**: high queue-time + high volume agents

Also analyze **load imbalance**:
- Gini coefficient / coefficient of variation for case counts across agents within each tier.
- % of tickets routed to top 10% busiest agents.

#### B) Timing performance (SLA-relevant)
Per tier and per priority:
- **Median/90th percentile** of:
  - L1 handling time (work duration)
  - time from **Ticket Created  first work start**
  - time from **first assignment  resolution**
- **Distribution shift over time**: do queues grow during specific weeks?

#### C) First-call resolution / escalation quality (especially L1)
Compute for L1:
- **First-call resolution rate**: cases where `L1 Work End = resolution` without escalation/reassignment.
- **Escalation rate** by L1 agent and by ticket type.
- **Escalation lead time**: time spent at L1 before escalation.

Compare:
- L1 agents that escalate quickly vs those that spend longer but resolve more—this distinguishes “speed vs effectiveness.”

#### D) Skill usage & skill effectiveness
- **Skill coverage rate**: % of cases where assigned agent had the required skill.
- **Skill mismatch rate**: % of cases where agent lacked the required skill (often leads to reassignments/escalations).
- **Specialist usage**:
  - How often do L2/L3 specialists work on tickets where a lower tier could have resolved them (a “over-skilling” indicator)?
  - Conversely, how often do L1 agents handle work requiring skills they don’t have (a “under-empowerment” indicator)?

#### E) Reassignment/escalation patterns
For each case, per tier:
- **#reassignments** and **#escalations**
- Average delay added per reassignment/escalation
- Reassignment frequency by:
  - Priority (P2/P3 vs P1/P4)
  - Category (Network/Security/etc.)
  - Channel (Phone/email/web)

---

### 1.3 Use process mining techniques to reveal *actual* assignment behavior

#### A) Resource interaction analysis (handover network)
Treat handovers as edges:
- Node = agent (or tier)
- Edge = “Agent X hands off to Agent Y” for reassignments/escalations

Metrics:
- **Edge frequencies** and **strong ties** (who repeatedly hands to whom)
- **Centrality** of certain agents/tier nodes (which ones become routing hubs)
- **Path analysis**: common sequences like `L1  L2  L2(reassign)  L3`

This will expose patterns such as:
- L1 consistently escalating to the same L2 agent even when skill mismatch occurs
- L2 specialist acting as a “catch-all” queue sink

#### B) Social Network Analysis (SNA) by tier/skill
Build separate networks:
- By tier (L1/L2/L3)
- By skill category needed (e.g., Network-Firewall)
- By ticket priority

If you see that certain skill areas repeatedly traverse multiple reassignments before completion, it indicates routing instability or poor initial assignment rules.

#### C) Role/variant discovery (intended vs actual logic)
Use **process variant discovery** and **role discovery**:
- Role = tier or agent group by skills
- Compare variants where tickets resolve smoothly vs those with many reassignments

Then contrast:
- **Expected path**: e.g., skill-based routing to correct tier/agent group
- **Observed path**: sequence frequencies of reassignments and skill mismatches

Concretely, you can discover “rule bypass” behavior such as:
- Dispatcher assigns via round-robin even when required skill is known  leads to reassign
- L1 tries basic troubleshoot even for highly specialized categories  triggers late escalation

---

### 1.4 Skill utilization analysis across the agent pool
Create a **Skill x Agent (or Skill x Tier) matrix**:
- Rows = required skills (App-CRM, Database-SQL, Networking-Firewall, Security-IAM, etc.)
- Columns = agents
- Values = number of cases where that skill was required and agent was assigned

Then compute:
- **Effective skill utilization**: cases completed without reassignment after assigning an agent who has the required skill.
- **Underutilization**: skills that exist in the agent profiles but are rarely used on cases that actually need them.
- **Misrouting causing specialist thrash**:
  - If a specialist is assigned frequently but still causes reassignments, the specialist’s “entry point” is wrong (e.g., assigned late after multiple wrong assignments).

## 2) Identifying Resource-Related Bottlenecks and Issues

### 2.1 Pinpoint specific resource-related bottlenecks
Use three complementary lenses:

#### A) Availability/queue bottlenecks for skill-critical work
For each required skill:
- Measure **queue time** before first work by any agent with that skill.
- Compare to SLA thresholds by priority (P2/P3 are key).

If queue times spike for certain skill areas (e.g., Database-SQL), you’ve found resource shortage or routing failure.

#### B) Delay inflation due to reassignments/escalations
Quantify the incremental overhead:
- For each reassignment event:
  - compute `delay = timestamp(next Work Start) - timestamp(current reassignment)`
- Aggregate delay by:
  - within-tier reassign vs cross-tier
  - skill mismatch vs skill match
  - priority level

This gives a direct measure like:
- “Cross-tier reassignments add +X minutes median delay”
- “Skill mismatch reassignments add +Y minutes and correlate with SLA breach rate”

#### C) Incorrect initial assignment impact (L1 and dispatcher)
Compare cases that start with:
- **skill-aligned L1 assignment** vs
- **skill-misaligned L1 assignment**

Compute:
- SLA breach rate difference
- expected escalation counts difference
- resolution time difference

Even if L1 eventually corrects the route, the damage may already be done (queue + delay).

#### D) Overloaded vs underutilized agents/teams
Identify:
- Agents with high load + high queue time (true bottlenecks)
- Agents with low load where their skills match incoming tickets (misrouting/underutilization)

This directly supports management’s uneven distribution concern.

---

### 2.2 Correlate resource patterns with SLA breaches
For each case:
- Label outcomes: `SLA breached? (P2/P3 threshold met?)`, `Resolved?`
- Add features:
  - skill match indicator at L1
  - #reassignments
  - #escalations
  - first queue time
  - assigned tier sequence
  - agent workload at assignment time (approx via moving average of prior workload)

Then compute:
- % SLA breaches attributable to:
  - skill mismatch (L1 or later)
  - excessive reassignments
  - queue time exceeding a threshold
- Use effect estimates (e.g., logistic regression or simple stratified comparisons) to quantify which factor matters most.

## 3) Root Cause Analysis for Assignment Inefficiencies

### 3.1 Candidate root causes (from process-mining evidence types)

#### 1) Assignment rules ignore skills/workload
Evidence to look for:
- High frequency of skill mismatch assignments at L1
- Round-robin-like behavior: similar agent selection regardless of required skill or current load
- Common patterns where the same mismatch occurs repeatedly across variants

#### 2) Inaccurate/incomplete agent skill profiles
Evidence:
- “Assigned agent lacked required skill” cases where business expects they should have had it.
- Skill-match indicator shows poor alignment even when it “should be known.”

Mitigation root cause:
- skill taxonomy mismatch (e.g., “Networking-Firewall” required but agent profile uses broader “Network”)
- outdated training records

#### 3) Poor ticket categorization / required-skill identification
Evidence:
- Required skill is often corrected only after escalation or after reassignments.
- A large portion of reassignments are driven by “needs different skill” (like your example reassign to DB-SQL).

#### 4) Lack of real-time visibility into workload/availability
Evidence:
- Reassignments happen when queue times were already large for the assigned tier.
- Agents chosen despite known backlog patterns.

#### 5) L1 empowerment/training gaps
Evidence:
- L1 agents frequently escalate after long troubleshooting durations even for categories they could handle.
- Alternatively, L1 escalates too quickly but to the wrong specialist, causing ping-pong.

---

### 3.2 Use variant analysis and decision mining
#### Variant analysis
- Segment cases into:
  - **Clean path**: 0 reassignments, 1 escalation
  - **Noisy path**: many reassignments/escalations
- Compare distributions by:
  - priority
  - category/required skill
  - initial assigned agent skills match rate
  - dispatcher assignment patterns

This tells you whether inefficiency is structural (routing logic) or situational (specific categories).

#### Decision mining (for dispatcher and tier routing choices)
Treat assignment actions as decisions:
- Inputs: ticket priority, category, required skill, channel, time of day, current queue metrics (if available)
- Decisions: which agent/tier to assign
- Outcomes: resolution time, SLA breach, #reassignments

You can discover:
- which decision rules correlate with “bad outcomes”
- whether observed decisions diverge from the intended logic

## 4) Developing Data-Driven Resource Assignment Strategies

Below are **four** concrete strategies (you asked for at least three). Each ties directly to mined insights and specifies what to implement and what data is needed.

---

### Strategy A — Skill-based routing with proficiency weighting (with tier guardrails)
**Addresses:** skill mismatch at assignment time; reduces unnecessary reassignments/escalations.

**Mining insight leveraged:**
- quantify skill mismatch rate at L1 and its correlation with SLA breaches and extra delays
- identify which skill-to-tier mappings historically succeed

**Approach:**
- Replace round-robin within tiers with a router:
  - Select agent(s) whose skills cover **required skill**
  - Weight by **historical success** for that skill + priority
  - Add tier guardrails: if required skill is App-CRM  strongly prefer L2 App-CRM capable agents unless evidence L1 resolves reliably

**Data required:**
- agent skill profile (clean taxonomy)
- mapping from ticket category/fields  required skills
- historical performance stats by (agent or skill group, skill, priority)
- optional confidence score when required skill is uncertain (see Strategy D)

**Expected benefits:**
- higher skill-match rate on first assignment
- fewer “needs different skill” reassignments
- lower queue + cycle times for P2/P3

---

### Strategy B — Workload-aware assignment (queue-length + capacity scheduling)
**Addresses:** overload hotspots and uneven workload distribution; reduces waiting time and cascading delays.

**Mining insight leveraged:**
- identify agents/tier states with high queue time and high volume
- measure effect of queue time on SLA breaches

**Approach:**
- At assignment time, compute a simple capacity score:
  - `effective workload = ongoing case count + expected processing time proxy + recent throughput`
- Assign to the agent among the skill-qualified candidates with:
  - **lowest expected start time** (shortest queue)
  - optionally a small penalty for high reassign history

**Data required:**
- near-real-time queue lengths (can be derived from event log continuously if integrated)
- recent throughput per agent (rolling window)
- expected service time per ticket type (from mined stats)

**Expected benefits:**
- reduced queue time variance
- better utilization balance
- fewer SLA breaches driven by backlog

---

### Strategy C — Predictive routing for “resolution likelihood” (category/keywords  skill + tier)
**Addresses:** incorrect or late skill requirement discovery; improves early triage.

**Mining insight leveraged:**
- some categories repeatedly cause escalations/reassignments
- titles/descriptions or categorization often correlate with required skill later

**Approach:**
- Build a predictive model from ticket attributes:
  - features: category, subcategory, keywords (from notes/description), channel
  - outputs: probability of required skill(s) and likely resolution tier
- Route based on highest-probability:
  - send to the most likely tier/skill group earlier (instead of waiting for L1 to “discover” it)

**Data required:**
- historical “required skill” label (or derived from eventual successful agent skills)
- ticket text fields (if available)
- mapping between final successful resolution and the skill needed

**Expected benefits:**
- fewer late escalations
- less L1 thrash on cases that clearly need specialists
- improved P2/P3 SLA compliance via earlier correct routing

---

### Strategy D — Reassignment and escalation criteria refinement (reduce thrashing)
**Addresses:** frequent reassignments within/between tiers; escalations that are too late or too frequent.

**Mining insight leveraged:**
- quantify incremental delay per reassignment and whether it tends to improve outcomes
- detect “reassignment loops” (e.g., L2  L2 reassignment repeatedly)

**Approach:**
- Define escalation rules based on mined thresholds:
  - time-in-stage limits by category/priority
  - escalation triggers based on missing evidence (e.g., after X minutes without certain diagnostics)
  - require “skill evidence” before cross-tier escalation (“why we escalated”)
- Introduce a “cooldown” or “one-reassign max” policy unless skill mismatch is detected.

**Data required:**
- diagnostic step outcomes at L1 (if logged)
- mapping of escalation reason codes
- mined thresholds by category/priority

**Expected benefits:**
- fewer unnecessary reassignments
- reduced delay inflation
- more consistent progression of cases through tiers

## 5) Simulation, Implementation, and Monitoring

### 5.1 Simulation to evaluate strategies before deployment
Use process-mining–informed simulation such as:
- **Discrete-event simulation (DES)** or **queueing simulation**
- Build a “digital twin” from:
  - empirical process variants (transition probabilities between L1/L2/L3)
  - empirical service times by ticket type/priority
  - empirical reassign/escalation delay distributions
  - resource constraints: number of agents per tier, skill coverage, and capacity

Then test policy variants:
1. Current policy baseline (round-robin + manual escalation)
2. Skill-based routing (Strategy A)
3. Skill-based + workload-aware (A+B)
4. Skill-based + predictive triage (A+C)
5. Plus escalation criteria improvements (D)

**Key outputs from simulation:**
- expected SLA breach rate for P2/P3
- average and 90th percentile lead times
- queue time distributions per skill group
- utilization and overload metrics per agent tier

This prevents “local improvements” that might shift bottlenecks elsewhere.

---

### 5.2 Monitoring plan post-implementation (process-mining dashboards)
Create dashboards that continuously compute process-mining KPIs.

#### Resource-related KPIs (must-have)
- **Skill match rate on first assignment** (L1 and L2 separately)
- **Reassignment count per case** (and within-tier vs cross-tier)
- **Escalations per case** and median time-to-escalate
- **Queue time distribution** by tier and required skill
- **SLA compliance rate** for P2 and P3
- **Agent workload balance** (e.g., coefficient of variation of case counts)
- **Over-skilling / under-skilling indicators**:
  - specialists working on tickets that could have stayed at L1 historically (to measure wasted expertise)
  - L1 assigned to tickets that required skills they didn’t have (to measure misrouting)

#### Process views (what to slice by)
- Funnel view: Created  assigned  first work start  resolved
- Sankey/variant diagram: top variants colored by SLA outcome
- Handover network: changes in reassignment edges after policy update
- Heatmaps:
  - required skill vs assigned agent tier
  - time of day vs queue time (if you see cyclic demand)

#### Control/alerting rules
- Alert when:
  - skill mismatch rate rises beyond a threshold
  - cross-tier reassignments increase
  - SLA breach rate spikes for specific skills/categories
