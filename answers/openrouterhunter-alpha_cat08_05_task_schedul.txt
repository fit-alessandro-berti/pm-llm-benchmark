# **Data-Driven Scheduling Optimization for Precision Parts Inc.**

## **1. Analyzing Historical Scheduling Performance and Dynamics**

### **Process Mining Approach**
I would begin by importing the MES event logs into a process mining framework (e.g., Celonis, ProM, or custom Python/PM4Py implementation) to reconstruct the actual execution flow rather than relying on assumed routings.

**Key analytical steps:**
- **Process Discovery**: Create a Directly Follows Graph (DFG) or Petri net model showing the actual sequence of activities per job type, revealing deviations from standard operating procedures
- **Variant Analysis**: Cluster jobs by their execution paths to identify common and exceptional routing patterns
- **Conformance Checking**: Compare actual execution against ideal scheduling models to quantify deviations

### **Specific Metrics and Techniques**

**a) Flow and Lead Time Analysis**
- **Process Mining Technique**: Case-level aggregation with timestamp analysis
- **Metrics**:
  - **Cycle Time Distribution**: Calculate from job release to completion (P50, P90, P95 percentiles)
  - **Touch Time vs. Wait Time**: Separate value-added processing time from non-value-added waiting
  - **Bottleneck Identification**: Apply critical path analysis through the process model to identify longest paths

**b) Task Waiting Time Analysis**
- **Process Mining Technique**: Resource-based process mining with queue analysis
- **Metrics**:
  - **Queue Time per Resource**: Average/median time jobs spend waiting before each machine
  - **Queue Length Dynamics**: Time-series analysis of WIP levels at each work center
  - **Starvation Metrics**: Percentage of time machines idle while downstream jobs wait

**c) Resource Utilization Analysis**
- **Process Mining Technique**: Resource-activity matrix and timeline analysis
- **Metrics**:
  - **Overall Equipment Effectiveness (OEE)**: Availability × Performance × Quality
  - **Setup Time Ratio**: Setup time as percentage of total machine time
  - **Operator Efficiency**: Task completion rates and variability by operator

**d) Sequence-Dependent Setup Time Analysis**
- **Process Mining Technique**: Pairwise transition analysis between consecutive jobs on same machine
- **Metrics**:
  - **Setup Time Matrix**: Build empirical matrix S[i,j] = average setup time when job type j follows job type i
  - **Setup Time Variability**: Coefficient of variation for each transition type
  - **Transition Frequency Analysis**: Identify most common job sequences and their associated setup times

**e) Schedule Adherence and Tardiness**
- **Process Mining Technique**: Delta analysis between planned and actual timelines
- **Metrics**:
  - **Tardiness Distribution**: Days late for each job, weighted by priority/value
  - **Schedule Compliance**: Percentage of tasks started within planned window
  - **Due Date Predictability**: Correlation between predicted and actual completion times

**f) Disruption Impact Analysis**
- **Process Mining Technique**: Disruption tagging and propagation analysis
- **Metrics**:
  - **Mean Time Between Failures (MTBF)** and **Mean Time To Repair (MTTR)** for each machine
  - **Disruption Propagation Radius**: Number of downstream jobs affected per disruption
  - **Priority Change Impact**: Additional delay caused by inserted hot jobs

## **2. Diagnosing Scheduling Pathologies**

### **Key Pathologies Identified via Process Mining**

**a) Bottleneck Resources and Their Impact**
- **Evidence**: Resource utilization heatmaps showing >90% utilization for specific machines (likely CNC mills or heat treatment furnaces)
- **Quantification**: Little's Law application: WIP = Throughput × Lead Time. Compare WIP levels upstream/downstream of suspected bottlenecks
- **Process Mining Tool**: Resource-activity matrix with bottleneck detection algorithms

**b) Poor Task Prioritization**
- **Evidence**: Correlation analysis between job priority/due date and actual processing sequence
- **Specific Finding**: Late jobs show patterns of being scheduled after early-due, low-priority jobs at bottleneck resources
- **Process Mining Tool**: Decision point analysis at each work center, comparing actual vs. optimal sequencing

**c) Suboptimal Sequencing and Setup Times**
- **Evidence**: Analysis of job transitions showing high-setup transitions occurring frequently
- **Quantification**: Compare actual total setup time vs. optimal (TSP-solved) sequence for same job mix
- **Process Mining Tool**: Transition matrix analysis with clustering to identify "setup-friendly" job families

**d) Starvation Effects**
- **Evidence**: Time-series correlation between upstream machine idle time and downstream queue length
- **Specific Finding**: Periodic starvation of grinding stations after heat treatment, indicating poor coordination
- **Process Mining Tool**: Cross-correlation analysis between machine status time-series

**e) Bullwhip Effect in WIP**
- **Evidence**: High coefficient of variation in daily WIP levels, amplifying from upstream to downstream stations
- **Quantification**: Spectral analysis of WIP time-series to identify oscillation frequencies
- **Process Mining Tool**: Process mining with timeline visualization of WIP movement

## **3. Root Cause Analysis of Scheduling Ineffectiveness**

### **Differentiating Root Causes**

**Process Mining-Based Differentiation Framework:**

| Root Cause Category | Process Mining Evidence | Discriminating Test |
|-------------------|------------------------|-------------------|
| **Poor Scheduling Logic** | Decision point analysis shows rules violated even when information available | Compare actual decisions vs. what simple rules would prescribe |
| **Resource Capacity Limitations** | Utilization >85% even under optimal sequencing | Compare makespan of actual sequence vs. optimal sequence for same job set |
| **Process Variability** | High coefficient of variation in task durations | Test if variability explains >30% of lead time variability via regression |
| **Inaccurate Estimates** | Systematic bias in planned vs. actual durations | Paired t-tests between planned and actual times by task type |

### **Specific Root Causes Identified**

**a) Static Rule Limitations**
- **Evidence**: Jobs with similar due dates but different routings receive identical priority treatment
- **Process Mining Insight**: Cluster analysis shows distinct job families requiring different scheduling approaches

**b) Lack of Real-Time Visibility**
- **Evidence**: Scheduling decisions made without knowledge of current queue lengths at next station
- **Process Mining Insight**: Cross-correlation shows scheduling decisions lag real shop floor status by 2-4 hours

**c) Inaccurate Duration Estimates**
- **Evidence**: 40-60% of tasks show >20% deviation from planned duration
- **Process Mining Insight**: Regression analysis reveals task duration depends on operator experience, job complexity, and time of day

**d) Ineffective Setup Handling**
- **Evidence**: Setup times constitute 15-25% of total machine time with high variability
- **Process Mining Insight**: Sequence analysis shows no correlation between job similarity and setup time optimization

**e) Poor Disruption Response**
- **Evidence**: After breakdowns, recovery takes 2-3× longer than repair time due to rescheduling inefficiencies
- **Process Mining Insight**: Control flow analysis shows lack of standardized recovery procedures

## **4. Advanced Data-Driven Scheduling Strategies**

### **Strategy 1: Multi-Factor Dynamic Dispatching with Setup Time Awareness**

**Core Logic**:
Implement a composite dispatching score at each machine:  
`Score =  × PriorityWeight +  × DueDateUrgency +  × SetupTimeSavings +  × DownstreamLoadBalance`

**Where**:
- `SetupTimeSavings` = `CurrentSetupTime` - `PotentialSetupTime` (based on historical transition matrix)
- `DownstreamLoadBalance` = Inverse of next machine's current queue length

**Process Mining Inputs**:
- Historical transition matrix for setup time prediction
- Real-time queue length data from MES
- Job priority and due date information

**Expected Impact**:
- 25-40% reduction in tardiness for high-priority jobs
- 15-20% reduction in total setup time
- Improved flow by reducing downstream starvation

### **Strategy 2: Predictive Scheduling with Digital Twin**

**Core Logic**:
Create a probabilistic simulation model that continuously updates with real-time data:

1. **Duration Prediction**: ML model (e.g., gradient boosting) predicting task duration based on:
   - Historical duration distributions per task type
   - Operator skill level (from historical performance)
   - Job complexity features (material, tolerances)
   - Time of day/shift effects

2. **Disruption Forecasting**: Predictive maintenance model estimating breakdown probability per machine based on:
   - Runtime hours
   - Historical failure patterns
   - Performance degradation signals

3. **Rolling Horizon Re-scheduling**: Every 2 hours, solve a stochastic optimization problem minimizing expected tardiness

**Process Mining Inputs**:
- Duration distributions and feature importance from historical data
- Breakdown patterns and lead/lag indicators
- Actual vs. planned variance analysis

**Expected Impact**:
- 30-50% improvement in due date prediction accuracy
- 20-30% reduction in WIP through better flow control
- Proactive bottleneck management

### **Strategy 3: Sequence-Optimized Batch Scheduling**

**Core Logic**:
At bottleneck machines, implement periodic batch scheduling (e.g., every 4-8 hours) solving a Traveling Salesman Problem (TSP) where:
- "Cities" = Jobs waiting in queue
- "Distance" = Setup time between jobs (from historical matrix)
- Objective: Minimize total setup time while respecting due dates via time window constraints

**Process Mining Inputs**:
- Complete setup time transition matrix
- Job due dates and slack times
- Machine-specific setup constraints

**Additional Enhancement**: Identify natural job families through clustering of setup time profiles and implement family-based scheduling rules.

**Expected Impact**:
- 40-60% reduction in setup time at bottleneck machines
- Increased effective capacity by 15-25%
- More predictable flow and lead times

3. **Stochastic Elements**: Task duration distributions, breakdown frequencies, setup time variability
4. **Scheduling Logic**: Plug-in architecture for testing different strategies

**Test Scenarios**:
- **Baseline**: Current FCFS/EDD rules
- **Stress Test 1**: 150% of normal job arrival rate
- **Stress Test 2**: Frequent breakdowns (2× historical rate)
- **Stress Test 3**: High proportion (30%) of urgent hot jobs
- **Stress Test 4**: Mixed scenario combining above

**Evaluation Metrics**:
- Primary: Tardiness (weighted by priority), WIP levels
- Secondary: Resource utilization balance, setup time ratio, lead time predictability

**Simulation Procedure**:
1. Warm-up period: 2 weeks simulated time
2. Measurement period: 8 weeks simulated time
3. 30 replications per scenario with different random seeds
4. Statistical comparison using ANOVA/Tukey tests
