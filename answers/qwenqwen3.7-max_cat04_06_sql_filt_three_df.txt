## Solution

Here is the DuckDB SQL query that satisfies all three requirements:

```sql
WITH ordered_events AS (
    -- Step 1: For each event, peek at the next two events within the same case
    SELECT
        case_id,
        activity,
        timestamp,
        LEAD(activity, 1)  OVER w AS next_activity,
        LEAD(activity, 2)  OVER w AS next_next_activity,
        LEAD(timestamp, 1) OVER w AS next_timestamp,
        LEAD(timestamp, 2) OVER w AS next_next_timestamp
    FROM event_log
    WINDOW w AS (PARTITION BY case_id ORDER BY timestamp)
),

matching_cases AS (
    -- Step 2: Identify cases that contain the exact 3-activity sequence
    --         AND where 'Create Order'  'Confirm Order' takes more than 5 days
    SELECT DISTINCT case_id
    FROM ordered_events
    WHERE activity            = 'Approve Purchase Requisition'
      AND next_activity       = 'Create Order'
      AND next_next_activity  = 'Confirm Order'
      AND (next_next_timestamp - next_timestamp) > INTERVAL '5 days'
)

-- Step 3: Return ALL events EXCEPT those belonging to the matching cases
SELECT e.*
FROM event_log e
WHERE NOT EXISTS (
    SELECT 1
    FROM matching_cases mc
    WHERE mc.case_id = e.case_id
)
ORDER BY e.case_id, e.timestamp;
```

---

### How It Works

| CTE / Clause | Purpose |
|---|---|
| **`ordered_events`** | Uses `LEAD()` window functions partitioned by `case_id` and ordered by `timestamp` to look ahead at the next two consecutive events. This guarantees **direct succession** — no other event from the same case can appear in between. |
| **`matching_cases`** | Filters for rows where the current event and the next two form the exact sequence `'Approve Purchase Requisition'  'Create Order'  'Confirm Order'`. It then checks that the elapsed time between `'Create Order'` (`next_timestamp`) and `'Confirm Order'` (`next_next_timestamp`) exceeds **5 days**. `DISTINCT` ensures each qualifying case appears only once. |
| **Final `SELECT`** | Uses `NOT EXISTS` to **exclude** every event belonging to a matched case, returning only events from "clean" cases. Results are ordered by `case_id` and `timestamp` for readability. |

### Key Design Decisions

- **`LEAD()` over self-joins** — More efficient and concise; a single scan with window functions avoids multiple joins on the same table.
- **`NOT EXISTS` over `NOT IN`** — Null-safe. If `case_id` were ever `NULL`, `NOT IN` would silently return no rows, whereas `NOT EXISTS` behaves correctly.
- **`INTERVAL '5 days'`** — DuckDB natively supports interval arithmetic on `TIMESTAMP` columns, making the time-difference check clean and readable.
- **Named `WINDOW w`** — Avoids repeating the `PARTITION BY ... ORDER BY` clause four times, keeping the query DRY.
