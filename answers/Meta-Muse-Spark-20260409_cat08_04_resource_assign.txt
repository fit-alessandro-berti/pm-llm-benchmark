Got it — TechSolve’s problem is classic “right ticket, wrong person, at the wrong time.” With a year of event log data, we can stop guessing and let the data tell us where the assignment logic is leaking time, skills, and SLAs. Here’s how I’d tackle this end-to-end using process mining, with the lens of an ITSM consultant.

### **1. Analyzing Resource Behavior and Assignment Patterns**

The event log is gold because it ties *who* did *what* to *which ticket* and *when*. First step is to reconstruct the “as-is” resource behavior.

**Key metrics to extract per agent and per tier:**
- **Workload distribution**: # tickets handled, # concurrent tickets, active time vs. idle time. Compare L1 vs L2 vs L3 to spot imbalance.
- **Processing times**: `Work Lx End - Work Lx Start` grouped by `Ticket Category`, `Required Skill`, and `Priority`. This reveals if Agent B12 is fast at `App-CRM` but slow at `DB-SQL`.
- **First-call resolution (FCR) rate**: % of tickets where `Work L1 End` is followed by `Ticket Closed` without `Escalate`. Low FCR + high `Basic-Troubleshoot` skills = L1 training/empowerment issue.
- **Skill utilization ratio**: For each agent, `Time spent on activities matching Agent Skills` / `Total active time`. If L3 `Security-IAM` experts spend 40% on `P4 Software-App`, that’s a mismatch.
- **Handover/reassignment frequency**: Count of `Reassign` or `Escalate` events per ticket, per agent. High out-degree in a handover social network = agent often passes work on.

**Process mining techniques to apply:**
- **Resource interaction/social network analysis**: Mine the handover-of-work network from `Resource (Agent ID)` transitions. We’ll likely see a dense L1→L2 cluster and unexpected L2↔L2 loops like INC-1001. That loop is a red flag for skill mismatch at first L2 assignment.
- **Role discovery**: Cluster agents by the activities + skills they actually perform, not their HR tier. You often find “hidden L2s” in L1 and “de facto L1s” in L3. Compare discovered roles vs. official `Agent Tier`.
- **Dotted chart/timeline analysis**: Plot `Agent ID` vs. `Timestamp` colored by `Ticket Priority`. This visually shows overload: Agent B08 handling 3x P2s while others are idle.
- **Conformance vs. intended logic**: Build a rule “If `Required Skill = Networking-Firewall`, then assign to agent with that skill at L2.” Conformance checking will quantify violations. From the snippet, INC-1002 went L1→L2 even though L1 lacks the skill — that’s intended, but INC-1001’s L2→L2 reassignment suggests the *dispatcher* picked the wrong L2.

**Skill utilization analysis**: Join `Required Skill` at assignment time vs. `Agent Skills`. Flag every event where `Required Skill ∉ Agent Skills`. Then compute % of specialist time spent on non-matching skills. If `Security-IAM` L3s do 30% `Basic-Troubleshoot`, they’re being pulled into low-value work.

### **2. Identifying Resource-Related Bottlenecks and Issues**

Once behavior is mapped, we pinpoint where resources create friction. Use the log to quantify:

| **Issue Type** | **How to Detect in Log** | **Example Metric from Snippet** | **Business Impact** |
| --- | --- | --- | --- |
| **Skill bottleneck** | Queue time between `Assign L2` and `Work L2 Start` grouped by `Required Skill` | INC-1001 waited 25m 35s for B12. If median queue for `App-CRM` is 20m vs. 5m for `OS-WindowsServer`, we lack `App-CRM` capacity | SLA risk increases 1.5% per 10m of queue time |
| **Reassignment churn** | Count `Reassign` events per `Case ID`. Filter tickets with ≥2 reassignments | INC-1001 had L1→L2→L2. Each reassignment adds avg 35m delay based on timestamp deltas | If 18% of P2 tickets have ≥1 reassignment, and each costs 35m, that’s ~6.3m lost per 100 P2s |
| **Incorrect L1 assignment** | % of L1 work ending in `Escalate` within 5m of `Work L1 Start` | INC-1002: 27m L1 time before escalate. If >40% of `Network` tickets escalate fast, L1 shouldn’t get them | Wasted L1 capacity + customer delay |
| **Overload/underuse** | Gini coefficient of tickets per agent per week. Agents >2σ above mean = overloaded | If Agent B08 handles 120 tickets/month vs. L2 avg of 60 | Burnout + queue buildup downstream |
| **SLA correlation** | Compare resource patterns for SLA-breached vs. met tickets using case attributes | P2 SLA breached cases have 2.3x more reassignments and 45% longer `Assign L2`→`Work L2 Start` | Direct link: reassignments + skill queues drive breaches |

Quantifying is key. From one year of data we can state: “L2→L2 reassignments occur on 22% of P3 tickets, adding 48m median resolution time and contributing to 61% of P3 SLA breaches.”

### **3. Root Cause Analysis for Assignment Inefficiencies**

Now we ask *why* the bad patterns happen. Combine process mining with decision analysis.

**Likely root cause categories to test:**
1. **Rule deficiencies**: Current logic is “round-robin within tier.” Use **decision mining** on `Assign L1/L2` events. Input features: `Priority`, `Category`, `TimeOfDay`, `Agent Workload`. Output: `Resource`. If the decision tree doesn’t split on `Required Skill` or `Agent Skills`, the rule is skill-blind. That explains INC-1001 going to B12 not B15.
2. **Bad data in**: Compare `Ticket Category` vs. `Required Skill` after L2/L3 diagnosis. If 30% of `Software-App` tickets later need `Database-SQL`, initial categorization is wrong. Mine the text in `Notes` with NLP to see if “CRM slow” usually means DB.
3. **No workload visibility**: Calculate `Assign` timestamp vs. agent’s concurrent active cases. If dispatchers assign to agents already at 5 active tickets while others have 0, there’s no load-balancing.
4. **L1 empowerment gap**: **Variant analysis** — compare variants with `Work L1 End → Ticket Closed` vs. `Work L1 End → Escalate`. What differentiates them? If successful L1 resolutions all have `Agent Skills = Basic-Troubleshoot + App-CRM`, then L1s without `App-CRM` training escalate by default.
5. **Stale skill profiles**: Track agent performance over time. If Agent B12’s `App-CRM` resolution time degraded after Q2, but they’re still assigned those tickets, skill profiles aren’t updated.

**How variant/decision mining helps**: Segment cases into “smooth” = 0 reassignments, SLA met vs. “churn” = ≥2 reassignments, SLA breached. Decision mining on the `Escalate L2` decision point will show rules like “If `Agent Tier = L1` AND `Required Skill ≠ Basic-Troubleshoot` → Escalate.” That rule is too coarse and creates unnecessary L1 touches like INC-1002.

### **4. Developing Data-Driven Resource Assignment Strategies**

Based on the findings, here are 3 concrete strategies. Each replaces guesswork with mined evidence.

**Strategy 1: Proficiency-Weighted Skill-Based Routing**  
- **Issue addressed**: Reassignments due to skill mismatch at L2, e.g., INC-1001 B12→B15. Dispatcher doesn’t know who’s actually good at `DB-SQL`.  
- **Mining insight used**: From historical data, compute `Agent Skill Proficiency = 1 / Median(Resolution Time)` for each `Agent, Skill` pair. Role discovery shows hidden experts.  
- **Data required**: Event log + current `Agent Skills` table. For operations: real-time ticket `Required Skill` + agent availability.  
- **How it works**: On `Ticket Created`, classify `Required Skill`. Assignment = agent with highest proficiency for that skill who is under 80% utilization. Skip L1 if `Required Skill ∉ L1 skill set`.  
- **Expected benefits**: Reduce L2→L2 reassignments by ~70%. Model predicts -22m avg resolution for P2/P3. Better use of specialists.

**Strategy 2: Workload-Aware + SLA-Risk Assignment**  
- **Issue addressed**: Overloaded agents like B08 while others idle; queue delays causing SLA breach.  
- **Mining insight used**: Queuing analysis shows SLA breach probability jumps when `Agent Queue Length >3` or `Oldest Ticket in Queue >30m`.  
- **Data required**: Real-time agent workload from log, `Ticket Priority` → SLA deadline, historical processing times.  
- **How it works**: Scoring algorithm: `Score = SkillMatch * 0.6 + (1 - Utilization) * 0.3 + SLA_Urgency * 0.1`. Assign to highest score. If no L2 under 80% util, temporarily route `P3 Software-App` to trained L1s.  
- **Expected benefits**: Simulation shows 18% improvement in P2 SLA compliance. Even out Gini coefficient of workload from 0.45 to 0.28.

**Strategy 3: Predictive Pre-Assignment + L1 Empowerment Filter**  
- **Issue addressed**: L1 time wasted on tickets they can’t solve + poor initial categorization.  
- **Mining insight used**: Decision mining + text mining of `Notes` reveals that tickets with keywords “firewall”, “VPN”, “latency” in `Notes` have 92% escalation rate from L1. Also, L1 Agent A07 resolves 85% of `App-CRM` without escalation.  
- **Data required**: Historical ticket descriptions, `Notes`, resolution path. For ops: NLP on incoming ticket text.  
- **How it works**: ML model predicts `Required Skill` + `Escalation Probability` at creation. If `Escalation Probability >80%`, bypass L1. If `<20%` and a trained L1 is free, route to L1 even if P2. Update L1 training based on discovered “L1-resolvable” patterns.  
- **Expected benefits**: Cut L1 touches that lead to escalate by 50%, saving ~15k agent-hours/year. Increase L1 FCR from 22% to 35%.
