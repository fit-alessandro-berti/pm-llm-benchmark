To solve this problem, I'll construct a DuckDB SQL query that follows these steps:

1. **Create activity sequences per case**: Use `array_agg` with `ORDER BY timestamp` to capture the ordered sequence of activities for each `case_id`
2. **Count cases per variant**: Group by the activity sequence and count occurrences
3. **Identify top K variants**: Use window functions to rank variants by frequency
4. **Filter events to top K cases**: Join back to get all events belonging to cases in the top K variants

Here's the complete solution:

```sql
WITH case_variants AS (
    -- Step 1: Extract ordered activity sequence for each case
    SELECT 
        case_id,
        array_agg(activity ORDER BY timestamp) AS activity_sequence
    FROM event_log
    GROUP BY case_id
),
variant_counts AS (
    -- Step 2: Count cases per variant
    SELECT 
        activity_sequence,
        COUNT(*) AS case_count
    FROM case_variants
    GROUP BY activity_sequence
),
ranked_variants AS (
    -- Step 3: Rank variants by frequency (top K)
    SELECT 
        activity_sequence,
        case_count,
        ROW_NUMBER() OVER (ORDER BY case_count DESC) AS variant_rank
    FROM variant_counts
)
-- Step 4: Return events from cases in top K variants only
SELECT e.*
FROM event_log e
JOIN case_variants cv ON e.case_id = cv.case_id
WHERE cv.activity_sequence IN (
    SELECT activity_sequence
    FROM ranked_variants
    WHERE variant_rank <= K  -- Replace K with your desired number of top variants
)
ORDER BY e.case_id, e.timestamp;
```

**Key aspects of this solution:**

- **Ordered aggregation**: `array_agg(activity ORDER BY timestamp)` preserves the temporal sequence of activities within each case
- **Variant identification**: The activity sequence serves as a unique identifier for each process variant
- **Efficient filtering**: The final join ensures only events from top K variant cases are returned
- **Result ordering**: Events are returned ordered by `case_id` and `timestamp` for readability

**Usage note**: Replace `K` in the query with your desired number of top variants (e.g., `WHERE variant_rank <= 5` for top 5 variants).
