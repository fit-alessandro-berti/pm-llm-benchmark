# Process Mining Analysis: Order Fulfillment Event Log

Based on the provided schema and sample event log data, I have analyzed the process flow against the assumed normal sequence. While Case 1001 adheres strictly to the standard operating procedure, Cases 1002, 1003, and 1004 exhibit significant deviations that warrant investigation.

## 1. Identified Anomalies

### A. Sequence Violations (Order of Operations)
The most critical anomalies involve activities occurring in the wrong order relative to the defined process flow.

*   **Shipping Before Risk Assessment (Case 1002):**
    *   **Observation:** `Ship Goods` (08:40) occurred before `Perform Credit Check` (09:10) and `Validate Stock` (09:15).
    *   **Risk:** Goods were dispatched before financial risk was assessed or inventory was confirmed. This exposes the company to bad debt or stockouts.
*   **Shipment Confirmation After Dispatch (Case 1003):**
    *   **Observation:** `Ship Goods` (09:10) occurred before `Confirm Shipment` (09:45).
    *   **Risk:** Logistics confirmation is a control step. Performing it post-facto suggests a lack of real-time oversight or manual backfilling of logs.
*   **Payment Before Invoicing (Case 1004):**
    *   **Observation:** `Receive Payment` (09:05) occurred 5 minutes after `Register Order` and 15 minutes before `Issue Invoice`.
    *   **Risk:** Standard accounting dictates an invoice precedes payment collection. This could indicate pre-authorization, system automation errors, or potential fraud (e.g., cash received without proper documentation).

### B. Missing Critical Activities
*   **Case 1003:** The activity `Validate Stock` is entirely absent from the log.
*   **Case 1004:** Both `Perform Credit Check` and `Validate Stock` are missing.
*   **Implication:** Controls designed to prevent financial loss and inventory errors were bypassed or not logged.

### C. Timing Anomalies
*   **Case 1004:** The time between `Register Order` and `Receive Payment` is only 5 minutes. While possible for pre-paid orders, combined with missing credit checks, it suggests an automated bypass or a "VIP" exception that isn't properly documented.

## 2. Hypotheses for Anomalies

1.  **Priority Order Bypass Policy:** Case 1002 is marked as `priority` in the `orders` table. There may be a business rule allowing priority orders to skip credit/stock checks to expedite delivery, but this creates a control gap.
2.  **System Integration Failure:** The missing activities (Stock Validation, Credit Check) in Cases 1003 and 1004 might be due to an integration failure between the Order Management System (OMS) and the Credit/Inventory systems, causing the steps to fail silently or not log.
3.  **Data Entry/Logging Latency:** In Case 1003, `Confirm Shipment` was logged after `Ship Goods`. This could indicate that the Logistics Manager confirmed the shipment in the system only after the physical goods had left the dock (retroactive logging).
4.  **Pre-Payment Configuration:** Case 1004 suggests a specific customer or order type allows immediate payment capture. If this is not a documented feature, it represents a deviation from the standard revenue recognition cycle.

## 3. Proposed SQL Investigation Queries

To validate these hypotheses across the full dataset, the following PostgreSQL queries are recommended.

### Query 1: Detect Shipping Before Credit Check
This query identifies all cases where goods were shipped before the credit check was completed, highlighting financial risk exposure.

```sql
SELECT 
    e_ship.case_id,
    o.order_type,
    e_ship.resource AS ship_resource,
    e_credit.resource AS credit_resource,
    e_ship.timestamp AS ship_time,
    e_credit.timestamp AS credit_time,
    (e_credit.timestamp - e_ship.timestamp) AS delay_hours
FROM order_event_log e_ship
JOIN order_event_log e_credit ON e_ship.case_id = e_credit.case_id
JOIN orders o ON e_ship.case_id = o.case_id
WHERE e_ship.activity = 'Ship Goods'
  AND e_credit.activity = 'Perform Credit Check'
  AND e_ship.timestamp < e_credit.timestamp;
```

### Query 2: Detect Payment Before Invoice
This query flags accounting anomalies where payment is recorded before an invoice is issued.

```sql
SELECT 
    e_pay.case_id,
    o.customer_id,
    o.order_value,
    e_pay.timestamp AS payment_time,
    e_inv.timestamp AS invoice_time,
    e_pay.resource AS payment_resource,
    e_inv.resource AS invoice_resource
FROM order_event_log e_pay
JOIN order_event_log e_inv ON e_pay.case_id = e_inv.case_id
JOIN orders o ON e_pay.case_id = o.case_id
WHERE e_pay.activity = 'Receive Payment'
  AND e_inv.activity = 'Issue Invoice'
  AND e_pay.timestamp < e_inv.timestamp;
```

### Query 3: Identify Cases with Missing Mandatory Activities
This query checks for cases that reached "Receive Payment" without logging "Perform Credit Check" or "Validate Stock".

```sql
SELECT 
    o.case_id,
    o.order_type,
    COUNT(DISTINCT e.activity) AS total_activities,
    CASE WHEN EXISTS (SELECT 1 FROM order_event_log WHERE case_id = o.case_id AND activity = 'Perform Credit Check') THEN 'Yes' ELSE 'No' END AS has_credit_check,
    CASE WHEN EXISTS (SELECT 1 FROM order_event_log WHERE case_id = o.case_id AND activity = 'Validate Stock') THEN 'Yes' ELSE 'No' END AS has_stock_validation
FROM orders o
JOIN order_event_log e ON o.case_id = e.case_id
WHERE e.activity = 'Receive Payment'
GROUP BY o.case_id, o.order_type
HAVING 
    NOT EXISTS (SELECT 1 FROM order_event_log WHERE case_id = o.case_id AND activity = 'Perform Credit Check')
    OR NOT EXISTS (SELECT 1 FROM order_event_log WHERE case_id = o.case_id AND activity = 'Validate Stock');
```

### Query 4: Correlate Anomalies with Resource Roles
This query investigates if specific departments or roles are responsible for the sequence violations (e.g., are specific Logistics Managers consistently shipping before confirmation?).

```sql
SELECT 
    r.department,
    r.role,
    e.activity,
    COUNT(*) AS violation_count

WHERE e.case_id IN (
    SELECT DISTINCT case_id
```
