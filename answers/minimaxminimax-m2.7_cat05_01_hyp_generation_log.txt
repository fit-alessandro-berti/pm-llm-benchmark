```sql
        WHEN e.activity IN ('Validate Stock','Ship Goods','Confirm Shipment')
             AND r.department <> 'Logistics' THEN 'Wrong department – should be Logistics'
        WHEN e.activity = 'Register Order' AND r.department <> 'Sales' THEN 'Wrong department – should be Sales'
        ELSE 'OK'
    END AS anomaly_flag
FROM order_event_log e
JOIN resources r ON e.resource = r.resource_id
WHERE NOT (                     -- filter only mismatches
    (e.activity IN ('Perform Credit Check','Issue Invoice','Receive Payment') AND r.department = 'Finance')
    OR (e.activity IN ('Validate Stock','Ship Goods','Confirm Shipment') AND r.department = 'Logistics')
    OR (e.activity = 'Register Order' AND r.department = 'Sales')
);
```

### 2.6  Valuerelated checks  

```sql
-- 13  Highvalue orders (>1500) missing a credit check
SELECT o.case_id,
       o.order_value
FROM orders o
WHERE o.order_value > 1500
  AND NOT EXISTS (
        SELECT 1
        FROM order_event_log e
        WHERE e.case_id = o.case_id
          AND e.activity = 'Perform Credit Check'
      );
```

```sql
-- 14  Payment amount (extracted from additional_info) does not match order_value
--       The amount is stored as e.g. 'amount=1250.00'
WITH pay AS (
    SELECT
        case_id,
        SUBSTRING(additional_info FROM 'amount=([0-9.]+)')::numeric AS paid_amt,
        additional_info
    FROM order_event_log
    WHERE activity = 'Receive Payment'
)
SELECT
    p.case_id,
    p.paid_amt,
    o.order_value,
    o.order_value - p.paid_amt AS difference
FROM pay p
JOIN orders o ON p.case_id = o.case_id
WHERE p.paid_amt <> o.order_value;
```

### 2.8  Endtoend steporder comparison (optional “processmining” style)

If you want to compare each case against the **canonical** order  
`Register Order  Perform Credit Check  Validate Stock  Confirm Shipment  Ship Goods  Issue Invoice  Receive Payment`, you can build a simple CTE that numbers the steps and then join to a static list:

```sql
-- 16  Compare actual step order vs. expected
WITH act AS (
    SELECT
        case_id,
        activity,
        ROW_NUMBER() OVER (PARTITION BY case_id ORDER BY timestamp) AS act_seq
    FROM order_event_log
),
expected AS (
    SELECT enumlabel AS activity, ROW_NUMBER() OVER (ORDER BY 1) AS exp_seq
    FROM unnest(ARRAY['Register Order',
                      'Perform Credit Check',
                      'Validate Stock',
                      'Confirm Shipment',
                      'Ship Goods',
                      'Issue Invoice',
                      'Receive Payment']) AS enumlabel
)
SELECT
    a.case_id,
    a.activity AS actual_activity,
    a.act_seq,
    e.activity AS expected_activity,
    e.exp_seq,
    CASE WHEN a.act_seq = e.exp_seq THEN 'Match' ELSE 'Mismatch' END AS status
FROM act a
JOIN expected e ON a.act_seq = e.exp_seq
ORDER BY a.case_id, a.act_seq;
```

Any rows where `status = 'Mismatch'` reveal a **process deviation** for further investigation.

## 3.  How to use these queries in practice  

1. **Run the “missing steps” queries first** – they give you a shortlist of `case_id`s that are definitely broken.  
2. **Run the “sequenceorder” queries** on that shortlist to see *where* the break occurs (e.g., before credit check, after shipping, etc.).  
3. **Crosscheck the “resourcedepartment” and “flagdriven” queries** to surface rootcause hints (wrong role, explicit flags).  
4. **Use the timing queries** to detect dataentry mistakes or systems that are “skipping” steps too fast.  
5. **Combine with `orders` and `resources`** when you need business context (order value, role, department).  

All queries are **parameterfree** – you can paste them straight into psql, pgAdmin, DBeaver, or any PostgreSQLcompatible tool.
