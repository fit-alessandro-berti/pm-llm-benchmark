## 1) Anomalies in the given POWL model

### A. “Loop” between **Evaluate (E)** and **Approve (P)**
- The model has `OperatorPOWL(operator=LOOP, children=[E, P])`.
- In practice this means the process may repeatedly execute evaluation/approval cycles (e.g., **E  P  E  P …**).
- This is not necessarily wrong (re-assessment can be legitimate), but it is an anomaly **if the business expects at most one approval** or if approvals occur without a meaningful preceding evaluation cycle.

**What to look for in logs:** multiple `P` events per claim, or `P` events that are not preceded by a corresponding `E`.

---

### B. XOR that can **skip Notify Customer (N)**
- The model defines `XOR(children=[N, skip])`.
- That means customer notification is optional: sometimes the process chooses the `skip` branch.

**What to look for in logs:** claims that reach `C` (close) but have **no `N`** event at all.

---

### C. Premature / out-of-order closing allowed by partial ordering
Edges added:

- `R -> A`
- `A -> loop`
- `loop -> xor`
- **and additionally:** `A -> C`
- There is **no** constraint like `xor -> C` or `loop -> C`.

Because `C` is only constrained to be after `A`, the model allows paths where:
- **Close (C) happens after A but before loop finishes** (i.e., before later `E/P` events).
- **Close (C) happens before Notify decision** (so `N` might be missing or occur after `C`).
- In general: `C` can occur “too early” relative to later `E`, `P`, or `N` events in the log.

**What to look for in logs:**
- `C` occurs at a timestamp earlier than the *last* `E` or `P` event for the same claim.
- `P` or `E` (or `N`) occurs **after** `C` for the same claim.

---

## 2) Hypotheses for why these anomalies exist

1. **Partial / evolving business rules**
   - Customer notification might have become optional (e.g., “only notify if approved / only notify for certain claim types”), but the modeler implemented it as a general XOR with skip.

2. **Modeling mis-constraint (missing edges in partial order)**
   - The tool/modeler likely intended `C` to be after the notification decision, but accidentally omitted edges like `xor -> C`.
   - The extra edge `A -> C` appears to be either:
     - an intended shortcut (“close after assignment”), or
     - a mistaken residual constraint added during refactoring.

3. **Misunderstanding of LOOP semantics**
   - The `LOOP(E, P)` construct might have been intended to mean something else (e.g., “E happens, then optionally re-do E if not approved”).
   - But the model could be enabling repeated `P` even when approvals should be single-shot.

4. **Technical / logging issues**
   - The system might log activity names inconsistently (e.g., “Approve” sometimes logged as `P` multiple times).
   - Or events might be recorded asynchronously (e.g., `C` logged before downstream `E/P/N` events because of UI/backend ordering).

5. **Department/process miscommunication**
   - Claims admin system (system events) might “close” a ticket while evaluation/notification still happens operationally in another service.

---

## 3) Verifying these hypotheses using the database (SQL)

Below I assume:
- `claim_events.activity` contains exactly: `'R','A','E','P','N','C'`
- Ordering is by `claim_events.timestamp` (and `event_id` as tiebreaker).

### 3.1 Claims closed **before evaluation**
```sql
WITH close_ev AS (
  SELECT claim_id, MIN(timestamp) AS close_ts
  FROM claim_events
  WHERE activity = 'C'
  GROUP BY claim_id
),
eval_ev AS (
  SELECT claim_id, MIN(timestamp) AS first_eval_ts
  FROM claim_events
  WHERE activity = 'E'
  GROUP BY claim_id
)
SELECT c.claim_id
FROM close_ev c
LEFT JOIN eval_ev e ON e.claim_id = c.claim_id
WHERE e.first_eval_ts IS NULL          -- no evaluation at all
   OR e.first_eval_ts > c.close_ts;  -- evaluation occurs after close
```

### 3.2 Claims closed **before approval**
```sql
WITH close_ev AS (
  SELECT claim_id, MIN(timestamp) AS close_ts
  FROM claim_events
  WHERE activity = 'C'
  GROUP BY claim_id
),
appr_ev AS (
  SELECT claim_id, MIN(timestamp) AS first_appr_ts
  FROM claim_events
  WHERE activity = 'P'
  GROUP BY claim_id
)
SELECT c.claim_id
FROM close_ev c
LEFT JOIN appr_ev p ON p.claim_id = c.claim_id
WHERE p.first_appr_ts IS NULL
   OR p.first_appr_ts > c.close_ts;
```

### 3.3 “Premature close”: events happen **after** `C`
This directly tests the partial-order anomaly.

```sql
WITH close_ev AS (
  SELECT claim_id, MIN(timestamp) AS close_ts
  FROM claim_events
  WHERE activity = 'C'
  GROUP BY claim_id
),
post_close AS (
  SELECT ce.claim_id,
         MAX(ce.timestamp) FILTER (WHERE ce.activity IN ('E','P','N')) AS last_post_close_ts,
         COUNT(*) FILTER (WHERE ce.activity IN ('E','P','N'))          AS num_post_close_events
  FROM close_ev c
  JOIN claim_events ce
    ON ce.claim_id = c.claim_id
   AND ce.timestamp > c.close_ts
   AND ce.activity IN ('E','P','N')
  GROUP BY ce.claim_id
)
SELECT *
FROM post_close
WHERE num_post_close_events > 0;
```

### 3.4 Customer notification skipped (no `N` before close)
Tests the XOR-with-skip anomaly.

```sql
WITH close_ev AS (
  SELECT claim_id, MIN(timestamp) AS close_ts
  FROM claim_events
  WHERE activity = 'C'
  GROUP BY claim_id
),
notify_ev AS (
  SELECT claim_id, MIN(timestamp) AS first_notify_ts
  FROM claim_events
  WHERE activity = 'N'
  GROUP BY claim_id
)
SELECT c.claim_id
FROM close_ev c
LEFT JOIN notify_ev n ON n.claim_id = c.claim_id
WHERE n.first_notify_ts IS NULL          -- never notified
   OR n.first_notify_ts > c.close_ts;  -- notified only after close
```

### 3.5 Multiple approvals per claim (loop anomaly)
```sql
SELECT
  claim_id,
  COUNT(*) FILTER (WHERE activity = 'P') AS num_approvals,
  COUNT(*) FILTER (WHERE activity = 'E') AS num_evaluations
FROM claim_events
GROUP BY claim_id
HAVING COUNT(*) FILTER (WHERE activity = 'P') > 1;
```

### 3.6 Approve without prior evaluation
If `P` is logged but there was no earlier `E` in that claim, that suggests a modeling/logging inconsistency.

```sql
WITH p_events AS (
  SELECT claim_id, timestamp
  FROM claim_events
  WHERE activity = 'P'
),
has_prior_eval AS (
  SELECT p.claim_id, p.timestamp,
         EXISTS (
           SELECT 1
           FROM claim_events e
           WHERE e.claim_id = p.claim_id
             AND e.activity = 'E'
             AND e.timestamp < p.timestamp
         ) AS has_eval_before
  FROM p_events p
)
SELECT *
FROM has_prior_eval
WHERE has_eval_before = FALSE;
```

### 3.7 Validate “C after notification decision” (missing constraint `xor -> C`)
Check whether `C` occurs before *any* `N` that happens at all (or before the last `N`/decision opportunity).

A simpler proxy: compare `close_ts` to the last `N` timestamp.
```sql
WITH close_ev AS (
  SELECT claim_id, MIN(timestamp) AS close_ts
  FROM claim_events
  WHERE activity = 'C'
  GROUP BY claim_id
),
last_n AS (
  SELECT claim_id, MAX(timestamp) AS last_notify_ts
  FROM claim_events
  WHERE activity = 'N'
  GROUP BY claim_id
)
SELECT c.claim_id
FROM close_ev c
JOIN last_n n ON n.claim_id = c.claim_id
WHERE n.last_notify_ts > c.close_ts;
```
(If this returns many cases, it supports “close is logged before notify completes”.)

### 3.8 Tie anomalies to time/business segmentation (often reveals root cause)
Example: see whether premature closes increased after a certain date.
```sql
SELECT
  date_trunc('month', cl.submission_date)::date AS submission_month,
  COUNT(*) FILTER (
    WHERE EXISTS (
      WITH close_ev AS (
        SELECT MIN(timestamp) AS close_ts
        FROM claim_events
        WHERE claim_id = cl.claim_id AND activity='C'
      )
      SELECT 1
      FROM claim_events ce, close_ev c
      WHERE ce.claim_id = cl.claim_id
        AND ce.activity IN ('E','P')
        AND ce.timestamp > c.close_ts
      LIMIT 1
    )
  ) AS num_premature_close_candidates,
  COUNT(*) AS total_claims
FROM claims cl
GROUP BY 1
ORDER BY 1;
```

## Summary of what you’ll learn
- **Premature closure**: if many claims have `E/P/N` timestamps after `C`, then the `A -> C` partial order and missing `xor/loop -> C` constraint are almost certainly problematic.
- **Skip notification**: if many closed claims have no `N`, the XOR-skip is matching reality (or the system sometimes fails to log `N`).
- **Loop anomalies**: if `P` happens multiple times or without prior `E`, then the LOOP semantics or event logging/mapping is inconsistent with intended behavior.
