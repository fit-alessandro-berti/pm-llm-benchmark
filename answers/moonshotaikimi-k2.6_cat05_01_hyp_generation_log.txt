*   **Sequence Violation:** `Confirm Shipment` (08:25) and `Ship Goods` (08:40) occur **before** `Perform Credit Check` (09:10) and `Validate Stock` (09:15).
*   **Risk Exposure:** Goods are shipped without verifying customer creditworthiness or stock availability.
*   **Evidence:** The `additional_info` flag `attempted_early=Y` on the `Confirm Shipment` event suggests the early shipment was deliberate or at least detected by the system.

### Case 1003 — Missing Control Gate & Inverted Shipment Logic
*   **Missing Activity:** `Validate Stock` is entirely absent from the trace.
*   **Sequence Violation:** `Ship Goods` (09:10) occurs **before** `Confirm Shipment` (09:45). The `late_confirmation=Y` flag confirms the confirmation was retroactive or delayed.
*   **Risk Exposure:** Shipment executed without stock validation and without prior logistical approval.

### Case 1004 — Financial Flow Inversion & Complete Bypass of Controls
*   **Extreme Sequence Violation:** `Receive Payment` (09:05) occurs only **5 minutes** after `Register Order` (09:00), and before `Issue Invoice` (09:20).
*   **Missing Activities:** Both `Perform Credit Check` and `Validate Stock` are absent.
*   **Policy Conflict:** `Confirm Shipment` contains `shipment_scheduled=N`, yet `Ship Goods` still occurs 25 minutes later.
*   **Process State:** The trace ends at `Ship Goods`; the case appears to have received payment upfront but lacks standard fulfillment controls.

## 2. Hypotheses for Root Causes

| Hypothesis | Description | Supporting Evidence |
|------------|-------------|---------------------|
| **H1: Expedite Override for Priority Orders** | Case 1002 is a `priority` order. Staff may be intentionally bypassing credit and stock checks to meet SLAs, using "attempted early" as a justification code. | `order_type = priority`; explicit `attempted_early=Y` flag. |
| **H2: System Detects but Does Not Prevent** | The presence of structured exception flags (`attempted_early`, `late_confirmation`, `shipment_scheduled=N`) implies the system logs violations but lacks hard enforcement (e.g., workflow engine blocks). | Flags exist on anomalous events, yet subsequent forbidden activities still execute. |
| **H3: Pre-Payment Data Misclassification** | Case 1004 may be a cash-in-advance or pre-paid order. The payment event could be a master-data import or a registration-time capture that is sequenced incorrectly in the log. | Payment occurs before invoice and within 5 minutes of registration; missing credit check is logical for pre-paid but suspicious for standard process. |
| **H4: Departmental Silos / Lack of Visibility** | Warehouse/Logistics staff may not have real-time visibility into whether Finance (credit) or Warehouse (stock) steps are complete, causing them to ship based on verbal or offline requests. | Shipping occurs before confirming steps in other departments; resources are from different departments with no apparent system hold. |
| **H5: Training Gaps or SOP Drift** | Case 1003’s missing `Validate Stock` suggests the step is optional in practice or that staff skip it when under pressure, relying on implicit stock knowledge. | Entire activity missing, not just reordered. |

## 3. Investigative SQL Queries

The following queries are designed to test the hypotheses above using PostgreSQL syntax. They require no prior assumptions about which `case_id`s are anomalous.

### Query A: Identify All Canonical Sequence Violations
Detects any case where an activity with a higher expected sequence number occurred earlier than an activity with a lower expected sequence number.

```sql
WITH expected_order AS (
  SELECT * FROM (VALUES
    ('Register Order', 1),
    ('Perform Credit Check', 2),
    ('Validate Stock', 3),
    ('Confirm Shipment', 4),
    ('Ship Goods', 5),
    ('Issue Invoice', 6),
    ('Receive Payment', 7)
  ) AS t(activity, seq_num)
),
event_positions AS (
  SELECT 
    oel.case_id,
    oel.activity,
    oel.timestamp,
    eo.seq_num
  FROM order_event_log oel
  JOIN expected_order eo ON oel.activity = eo.activity
)
SELECT 
  e1.case_id,
  e1.activity AS earlier_activity,
  e2.activity AS later_activity,
  e1.timestamp AS expected_later_time,
  e2.timestamp AS expected_earlier_time
FROM event_positions e1
JOIN event_positions e2 
  ON e1.case_id = e2.case_id 
 AND e1.seq_num < e2.seq_num
WHERE e1.timestamp > e2.timestamp
ORDER BY e1.case_id, e2.timestamp;
```

### Query B: Detect Missing Mandatory Activities Per Case
Identifies cases that skipped essential control gates entirely.

```sql
SELECT 
  o.case_id,
  o.order_type,
  o.order_value,
  MAX(CASE WHEN oel.activity = 'Perform Credit Check' THEN 1 ELSE 0 END) AS has_credit_check,
  MAX(CASE WHEN oel.activity = 'Validate Stock' THEN 1 ELSE 0 END) AS has_stock_validation,
  MAX(CASE WHEN oel.activity = 'Confirm Shipment' THEN 1 ELSE 0 END) AS has_confirm_shipment,
  MAX(CASE WHEN oel.activity = 'Issue Invoice' THEN 1 ELSE 0 END) AS has_invoice
FROM orders o
LEFT JOIN order_event_log oel ON o.case_id = oel.case_id
GROUP BY o.case_id, o.order_type, o.order_value
HAVING MAX(CASE WHEN oel.activity = 'Perform Credit Check' THEN 1 ELSE 0 END) = 0
    OR MAX(CASE WHEN oel.activity = 'Validate Stock' THEN 1 ELSE 0 END) = 0;
```

### Query C: Correlate Anomalies with Order Type and Resource Department
Tests **H1** (priority orders) and **H4** (departmental silos) by linking exception flags to order attributes and resource metadata.

```sql
WITH flagged_cases AS (
  SELECT 
    case_id,
    BOOL_OR(additional_info LIKE '%attempted_early=Y%') AS early_attempt,
    BOOL_OR(additional_info LIKE '%late_confirmation=Y%') AS late_confirmation,
    BOOL_OR(additional_info LIKE '%shipment_scheduled=N%') AS shipment_denied
  FROM order_event_log
  GROUP BY case_id
)
SELECT 
  o.order_type,
  r.department,
  COUNT(DISTINCT oel.case_id) AS cases_with_anomaly,
  SUM(CASE WHEN fc.early_attempt THEN 1 ELSE 0 END) AS early_attempts,
  SUM(CASE WHEN fc.late_confirmation THEN 1 ELSE 0 END) AS late_confirmations,
  SUM(CASE WHEN fc.shipment_denied THEN 1 ELSE 0 END) AS denied_but_proceeded
FROM order_event_log oel
JOIN flagged_cases fc ON oel.case_id = fc.case_id
JOIN orders o ON oel.case_id = o.case_id
JOIN resources r ON oel.resource = r.resource_id
WHERE fc.early_attempt OR fc.late_confirmation OR fc.shipment_denied
GROUP BY o.order_type, r.department
ORDER BY cases_with_anomaly DESC;
```

### Query D: Investigate Payment-Before-Invoice and Ultra-Fast Payment Cycles
Tests **H3** by quantifying how often payment precedes invoice or occurs suspiciously close to registration.

```sql
WITH case_times AS (
  SELECT 
    case_id,
    MIN(CASE WHEN activity = 'Register Order' THEN timestamp END) AS register_time,
    MIN(CASE WHEN activity = 'Receive Payment' THEN timestamp END) AS payment_time,
    MIN(CASE WHEN activity = 'Issue Invoice' THEN timestamp END) AS invoice_time,
    MIN(CASE WHEN activity = 'Perform Credit Check' THEN timestamp END) AS credit_time
  FROM order_event_log
  GROUP BY case_id
)
SELECT 
  ct.case_id,
  o.order_type,
  o.order_value,
  ct.register_time,
  ct.payment_time,
  ct.invoice_time,
  ROUND(EXTRACT(EPOCH FROM (ct.payment_time - ct.register_time))/60, 2) AS mins_reg_to_payment,
  CASE 
    WHEN ct.payment_time < ct.invoice_time THEN 'Payment Before Invoice'
    WHEN ct.credit_time IS NULL THEN 'Payment Without Credit Check'
    ELSE 'Normal'
  END AS anomaly_type
FROM case_times ct
JOIN orders o ON ct.case_id = o.case_id
WHERE ct.payment_time < ct.invoice_time 
   OR ct.credit_time IS NULL
ORDER BY mins_reg_to_payment;
```

### Query E: Audit "Ship Goods" Against Preceding Required Steps
Tests **H2** and **H4** by checking every shipment against its prerequisites (credit, stock, confirmation).

```sql
SELECT 
  ship.case_id,
  ship.timestamp AS ship_time,
  ship.resource AS ship_resource,
  confirm.timestamp AS confirm_time,
  stock.timestamp AS validate_time,
  credit.timestamp AS credit_time,
  CASE WHEN confirm.timestamp IS NULL THEN 'Missing Confirm'
       WHEN ship.timestamp < confirm.timestamp THEN 'Ship Before Confirm'
       ELSE 'OK' END AS confirm_status,
  CASE WHEN stock.timestamp IS NULL THEN 'Missing Validate'
       WHEN ship.timestamp < stock.timestamp THEN 'Ship Before Validate'
       ELSE 'OK' END AS validate_status,
  CASE WHEN credit.timestamp IS NULL THEN 'Missing Credit'
       WHEN ship.timestamp < credit.timestamp THEN 'Ship Before Credit'
       ELSE 'OK' END AS credit_status
FROM order_event_log ship
LEFT JOIN order_event_log confirm 
  ON ship.case_id = confirm.case_id AND confirm.activity = 'Confirm Shipment'
LEFT JOIN order_event_log stock 
  ON ship.case_id = stock.case_id AND stock.activity = 'Validate Stock'
LEFT JOIN order_event_log credit 
  ON ship.case_id = credit.case_id AND credit.activity = 'Perform Credit Check'
WHERE ship.activity = 'Ship Goods'
  AND (
    ship.timestamp < confirm.timestamp OR confirm.timestamp IS NULL
    OR ship.timestamp < stock.timestamp OR stock.timestamp IS NULL
    OR ship.timestamp < credit.timestamp OR credit.timestamp IS NULL
  )
ORDER BY ship.case_id;
```

### Query F: Identify Cases Where Shipment Was Denied but Executed Anyway
Directly tests **H2** (system enforcement gap) by finding negative confirmations that were overridden by downstream shipping.

```sql
SELECT 
  deny.case_id,
  deny.timestamp AS deny_timestamp,
  deny.additional_info AS denial_reason,
  ship.timestamp AS ship_timestamp,
  ship.resource AS ship_resource,
  ship.additional_info AS ship_evidence
FROM order_event_log deny
JOIN order_event_log ship 
  ON deny.case_id = ship.case_id 
 AND ship.timestamp > deny.timestamp
 AND ship.activity = 'Ship Goods'
WHERE deny.activity = 'Confirm Shipment'
  AND deny.additional_info LIKE '%shipment_scheduled=N%';
```

### Query G: Process Completion and Cycle Time Analysis
Identifies incomplete traces (e.g., Case 1004 ending at `Ship Goods`) and abnormally short or long cycle times.

```sql
WITH case_summary AS (
  SELECT 
    case_id,
    MIN(timestamp) AS process_start,
    MAX(timestamp) AS process_end,
    COUNT(DISTINCT activity) AS unique_activities,
    COUNT(*) AS total_events
  FROM order_event_log
  GROUP BY case_id
)
SELECT 
  cs.case_id,
  o.order_type,
  cs.unique_activities,
  cs.total_events,
  ROUND(EXTRACT(EPOCH FROM (cs.process_end - cs.process_start))/3600, 2) AS duration_hours,
  CASE WHEN NOT EXISTS (
    SELECT 1 FROM order_event_log oel2 
    WHERE oel2.case_id = cs.case_id AND oel2.activity = 'Receive Payment'
  ) THEN 'Incomplete' ELSE 'Complete' END AS completion_status
FROM case_summary cs
JOIN orders o ON cs.case_id = o.case_id
ORDER BY duration_hours;
```

### Query H: Resource-Level Anomaly Heatmap
Tests **H4** and **H5** by determining which roles most frequently appear in anomalous events.

```sql
WITH anomalous_events AS (
  SELECT oel.*
  FROM order_event_log oel
  WHERE 
    (oel.activity = 'Confirm Shipment' AND EXISTS (
      SELECT 1 FROM order_event_log oel2 
      WHERE oel2.case_id = oel.case_id 
        AND oel2.activity = 'Ship Goods' 
        AND oel2.timestamp < oel.timestamp
    ))
    OR oel.additional_info LIKE '%attempted_early=Y%'
    OR oel.additional_info LIKE '%late_confirmation=Y%'
    OR (oel.additional_info LIKE '%shipment_scheduled=N%' AND EXISTS (
      SELECT 1 FROM order_event_log oel2 
      WHERE oel2.case_id = oel.case_id AND oel2.activity = 'Ship Goods'
    ))
)
SELECT 
  r.department,
  r.role,
  ae.activity,
  COUNT(*) AS event_count,
  COUNT(DISTINCT ae.case_id) AS affected_cases
FROM anomalous_events ae
JOIN resources r ON ae.resource = r.resource_id
GROUP BY r.department, r.role, ae.activity
ORDER BY event_count DESC;
```

## 4. Recommended Next Steps (For the Analyst)

1.  **Run Query C** first. If `priority` orders show a disproportionate rate of `early_attempts`, this validates **H1** and suggests a policy review for expedited orders.
2.  **Run Query F**. If `shipment_scheduled=N` is frequently followed by `Ship Goods`, the system requires a hard-stop workflow rule rather than passive logging.
3.  **Run Query D**. If Case 1004 is not unique (i.e., multiple cases show payment before invoice with zero credit check), investigate whether a "pre-paid" order type exists but is being mapped to the standard process log incorrectly.
4.  **Run Query H**. If the `Logistics` department dominates the anomaly heatmap, targeted SOP training or a real-time dashboard showing Finance/Warehouse status is needed.
