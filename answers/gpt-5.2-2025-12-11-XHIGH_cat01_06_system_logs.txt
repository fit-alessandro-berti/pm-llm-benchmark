### Brief logic / rationale

**Case identification (grouping):**  
I used an **artifact-centric case notion**: each distinct work object becomes a case (Word document, Excel file, PDF, and the specific email thread). This yields coherent traces that match typical knowledge-work processes:
- `DOC:Document1.docx` (drafting/editing + saving + closing)
- `EMAIL:Annual Meeting` (read  reply  send)
- `PDF:Report_Draft.pdf` (review  annotate)
- `XLS:Budget_2024.xlsx` (update  save)
- `DOC:Quarterly_Report.docx` (draft  save  close)

**Activity naming (abstraction):**
- `FOCUS` / `SWITCH`  **Open/Resume** (window brought to foreground; “Open” for first activation in the case, “Resume” for later returns)
- `TYPING`  **Edit Document / Edit Spreadsheet / Compose Reply** (kept as a standardized edit step, with the typed snippet stored in `detail`)
- `SCROLL`  **Read Email / Review PDF**
- `HIGHLIGHT`  **Annotate PDF**
- `SAVE` / `CLOSE`  **Save / Close** activities

This produces an analyst-friendly log that tools can discover into separate process models per artifact type (documents, email handling, PDF review, spreadsheet updates), while still preserving timestamps and key context.
