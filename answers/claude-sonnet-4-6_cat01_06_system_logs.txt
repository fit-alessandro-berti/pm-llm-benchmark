## Explanation of Logic

### Case Identification Strategy

Cases were defined around **distinct work objects** — the specific file or task the user was actively engaged with. Each document, email thread, or artifact represents one coherent unit of work:

| Case ID | Work Object | Rationale |
|---------|-------------|-----------|
| CASE-001 | Document1.docx | A document drafted, saved, revisited for edits, and closed |
| CASE-002 | Email – Annual Meeting | A single email read-and-reply workflow |
| CASE-003 | Report_Draft.pdf | A PDF review-and-annotation task |
| CASE-004 | Budget_2024.xlsx | A spreadsheet update task |
| CASE-005 | Quarterly_Report.docx | A separate report authoring task |

> **Note on the initial FOCUS event (08:59:50):** The very first event is a FOCUS on `Quarterly_Report.docx`, but the user immediately switches to `Document1.docx` at 09:00:00 without taking any action on it. This was treated as an incidental window activation (not a true start of work), so CASE-005 begins only when the user intentionally returns to it at 09:07:15 with purposeful editing activity.

---

### Activity Naming Logic

Raw low-level system actions were translated into **business-meaningful activity names** using the following mapping rules:

| Raw Action | Context | Activity Name |
|------------|---------|---------------|
| `FOCUS` / `SWITCH`  new document | First interaction with artifact | **Open Document / Open Spreadsheet / Open PDF** |
| `TYPING` in Word/Excel | Content creation | **Draft Document Content / Update Spreadsheet Data** |
| `TYPING` in email reply | Email authoring | **Compose Reply** |
| `SCROLL` in PDF | Passive reading | **Review PDF Content** |
| `SCROLL` in email | Reading email body | **Read Email** |
| `HIGHLIGHT` | Annotation action | **Annotate PDF** |
| `CLICK  Open Email` | Email selection | **Open Email** |
| `CLICK  Reply` | Initiating reply | **Initiate Reply** |
| `CLICK  Send` | Sending message | **Send Email** |
| `SAVE` | Persistence checkpoint | **Save Document / Save Spreadsheet** |
| `CLOSE` | End of work unit | **Close Document** |
| `SWITCH` back to prior document | Re-engagement | **Resume Document Editing** |

### Handling Interruptions and Context Switches

CASE-001 (`Document1.docx`) demonstrates an **interrupted process**: the user saves the document, switches away to handle email (CASE-002) and Excel work (CASE-004), then **returns** to Document1.docx to add a budget reference before finally closing it. This is treated as a single continued case rather than two separate cases, because the work object is the same document and the user explicitly returns to complete it. The return event is labeled **Resume Document Editing** to make the interruption-and-return pattern visible to analysts.
