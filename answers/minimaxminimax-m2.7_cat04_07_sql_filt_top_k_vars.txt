Below is a **complete, readytorun** solution for DuckDB.  
It follows the four logical steps you asked for:

1. **Build the ordered activity list (the “variant”) for each case** – `LIST_AGG … ORDER BY timestamp`.  
2. **Count how many cases share the same variant** – a simple `GROUP BY` on the list.  
3. **Pick the topK variants** – `DENSE_RANK()` (or `ROW_NUMBER()`) over the frequency.  
4. **Return every original event that belongs to a case whose variant is in that topK set**.

You can paste the query into the DuckDB SQL editor, replace `:K` (or `?` if you prefer a positional parameter) with the desired number, and you’ll get the filtered event log.

```sql
-- 1  Build the ordered activity sequence for each case
WITH case_sequences AS (
    SELECT
        case_id,
        -- DuckDB’s ordered aggregate returns a LIST – works perfectly as a “variant”
        LIST_AGG(activity ORDER BY timestamp) AS activity_seq
    FROM event_log
    GROUP BY case_id
),

-- 2  Count how many cases share each variant and rank them
variant_counts AS (
    SELECT
        activity_seq,
        COUNT(*) AS variant_frequency,          -- how many cases follow this variant
        DENSE_RANK() OVER (ORDER BY COUNT(*) DESC) AS rnk   -- rank by frequency
    FROM case_sequences
    GROUP BY activity_seq
),

-- 3  Keep only the topK most frequent variants
top_k_variants AS (
    SELECT activity_seq
    FROM variant_counts
    WHERE rnk <= :K          -- replace :K with the number you want (e.g. 5)
)

-- 4  Return all original events whose case belongs to a topK variant
SELECT
    e.case_id,
    e.activity,
    e.timestamp
FROM event_log e
JOIN case_sequences cs
      ON e.case_id = cs.case_id
JOIN top_k_variants tk
      ON cs.activity_seq = tk.activity_seq
ORDER BY e.case_id, e.timestamp;   -- chronological order inside each case
```

### How it works (stepbystep)

| Step | What the CTE does | Why it matters |
|------|-------------------|----------------|
| **case_sequences** | For each `case_id` aggregates all `activity` values **ordered** by `timestamp`. The result is a list (e.g. `[A, B, C]`). | This is the exact *process variant* you defined – the ordered sequence of activities for a case. |
| **variant_counts** | Groups the rows from `case_sequences` on the list, counts the rows (`COUNT(*)`) and assigns a rank (`DENSE_RANK`) based on that count, highest first. | Gives you the frequency of each variant and lets you keep only the most frequent ones. |
| **top_k_variants** | Keeps only the variants whose rank K. | Guarantees you keep exactly the K mostcommon process variants. |
| **Final SELECT** | Joins the original `event_log` back to the filtered variant set (via `case_sequences`). The `ORDER BY case_id, timestamp` restores the original chronological ordering for every event that survived the filter. | You get *all* events from cases that follow one of the topK variants, in the correct order. |

### Optional – also see the variant frequency in the output

If you want to expose the frequency of the variant that each event belongs to, you can add a few columns:

```sql
SELECT
    e.case_id,
    e.activity,
    e.timestamp,
    vc.variant_frequency,
    vc.rnk   -- the rank (1 = most frequent)
FROM event_log e
JOIN case_sequences cs
      ON e.case_id = cs.case_id
JOIN variant_counts vc
      ON cs.activity_seq = vc.activity_seq
JOIN top_k_variants tk
      ON cs.activity_seq = tk.activity_seq
ORDER BY e.case_id, e.timestamp;
```
