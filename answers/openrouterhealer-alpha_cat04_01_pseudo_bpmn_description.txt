# Business Process Analysis: Customer Request Handling

## Overview

This process models how a company handles incoming customer requests from initial receipt through final confirmation. It's a **mixed-type workflow** — meaning it adapts its path depending on the nature of the request, then reconverges around approval and invoicing. Let me walk through it in detail.

---

## The Start: Receiving the Request

**Task A: "Receive Customer Request"**

This is the entry point. A customer submits a request — likely through a form, email, or sales channel. This task captures the request and passes it to the first decision point.

---

## The First Fork: Request Type Gateway

The **XOR Gateway (Exclusive Gateway)** here means "choose exactly one path." The process inspects the request type and routes accordingly:

### Path 1 — Standard Requests

| Step | Purpose |
|---|---|
| **Task B1: Perform Standard Validation** | Validates the request against predefined rules (completeness, eligibility, policy compliance). This is a routine, checklist-style check. |
| **AND Gateway: Run Parallel Checks** | An **inclusive/parallel gateway** — both checks run simultaneously to save time. |
| **Task C1: Credit Check** | Verifies the customer's creditworthiness (payment history, credit score, outstanding balances). |
| **Task C2: Inventory Check** | Confirms that the requested product(s) are in stock or can be sourced. |
| **Join: All Parallel Checks Completed** | The process **waits** until *both* C1 and C2 finish. This is a synchronization point — neither downstream step can proceed until both checks are done. |
| **Task D: Calculate Delivery Date** | Uses the results from both checks to determine a realistic delivery timeline. If credit fails or inventory is unavailable, this calculation will reflect that. |

### Path 2 — Custom Requests

| Step | Purpose |
|---|---|
| **Task B2: Perform Custom Feasibility Analysis** | Evaluates whether the customer's unique requirements can actually be met — considering technical constraints, resource availability, and cost. This is more complex than standard validation; it likely involves engineers or specialists. |
| **XOR Gateway: Is Customization Feasible?** | A go/no-go decision based on the feasibility analysis. |
| **Task E1: Prepare Custom Quotation** *(If Yes)* | Builds a tailored price quote with custom specifications, timelines, and terms. |
| **Task E2: Send Rejection Notice** *(If No)* | Politely informs the customer that the request cannot be fulfilled, then terminates the process for this request. This is one of two possible **end points** before the final confirmation stage. |

---

## Convergence: The Approval Gateway

After **either** the standard path (Task D) **or** the custom path (Task E1) completes, the two branches reconverge into a single flow:

**XOR Gateway: "Is Approval Needed?"**

This is where the process decides whether managerial oversight is required. Approval might be triggered by:

- **Amount thresholds** — requests exceeding a dollar value require sign-off.
- **Custom orders** — inherently need review.
- **Non-standard terms** — unusual payment or delivery conditions.
- **New customers** — no established trust relationship.

### If Approval IS Needed

**Task F: "Obtain Manager Approval"** — The request is routed to an authorized manager for review.

Then a **nested XOR Gateway: "Is Approval Granted?"**

| Outcome | Result |
|---|---|
| **Yes** | Proceeds to **Task G: Generate Final Invoice** — the approved order is formalized. |
| **No** | Routes to **Task H: "Re-evaluate Conditions"** — the manager has flagged concerns (price too high, terms unacceptable, risk too great). |

#### The Loop-Back: Rework Cycle

**Task H** triggers a **loop back** to earlier tasks depending on which path was taken:

- **Standard path**  loops back to **Task D** (recalculate delivery date, possibly with adjusted terms).
- **Custom path**  loops back to **Task E1** (revise the quotation, perhaps lowering price or adjusting scope).

This creates a **rework cycle** that continues until either:
- Approval is granted, or
- The request is ultimately rejected (though the model doesn't explicitly show this terminal rejection path — it's implied that repeated rejections would eventually end the process).

### If Approval Is NOT Needed

**Task G: Generate Final Invoice** — The request is approved by default (likely because it falls within authority limits), and the invoice is produced directly.

---

## Final Steps

**Task I: "Send Confirmation to Customer"**

Regardless of which path was taken, the customer receives:
- The finalized invoice
- Order confirmation
- Delivery timeline
- Any terms or conditions

**End Event** — The process concludes. The only other end point is **Task E2** for infeasible custom requests (early termination).

## Key Observations

1. **Adaptive routing**: The process is not one-size-fits-all. It dynamically selects validation depth based on request complexity.

2. **Parallelism for efficiency**: Credit and inventory checks run concurrently on standard requests — a smart optimization when checks are independent.

3. **Single approval chokepoint**: Both paths funnel through the same approval logic, ensuring governance consistency regardless of request type.

4. **Rework without full restart**: The loop-back to E1 or D (rather than all the way back to A) is a meaningful design choice — it preserves completed work and only revises what's necessary.

5. **One explicit dead-end**: Task E2 (Rejection Notice) is the only path that exits early without reaching Task I. All other paths converge on customer confirmation.

This is a well-structured, realistic process model for a mid-complexity order handling system. The main thing I'd flag for review is whether the rejection path in the rework cycle (if the manager denies approval repeatedly) should also explicitly route to a rejection notice rather than leaving it undefined.
