### **Notes and Assumptions**:

1. **Order Creation** is defined as the earliest `order_date` in the `Orders` table.
2. **Items Picked** events are derived from the `Order Lines` table. Since multiple items can be picked at different times, each line item generates its own event. The earliest `item_picking_date` per order is not used to merge events — instead, individual item picking times are preserved to reflect real process timing.
3. **Order Dispatched** is taken from the `delivery_dispatch_date` in the `Delivery` table. If an order has multiple deliveries (e.g., 1002), each dispatch is treated as a separate event.
4. **Order Arrived** uses `delivery_arrival_date`. For orders with multiple deliveries, each arrival is a separate event.
5. **Invoice Issued** uses `invoice_issue_date`. Multiple invoices per order (e.g., 1002) are treated as separate events.
6. **Payment Received** uses `payment_date`. Payments are linked to invoices via `invoice_id`. If an invoice has multiple payments, each is a separate event.
7. **Resource Assignment**:
   - **Sales Person**: Assigned via `sales_person_id`  linked to `Resources` table.
   - **Warehouse Staff**: Assigned via `picker_id`  linked to `Resources` table.
   - **Carrier**: Assigned via `carrier_id`  linked to `Resources` table.
   - **Billing Clerk**: Assigned via `billing_person_id`  linked to `Resources` table.
   - When multiple roles apply, the most relevant (e.g., warehouse staff for picking) is used.
8. **Chronological Order**:
   - Events are ordered by timestamp within each case.
   - For 1004, the dispatch date (2024-01-06) is earlier than the item picking date (2024-01-06T08:00:00Z), which may suggest a data inconsistency. However, since the delivery dispatch is recorded earlier, we retain it as-is, assuming the system may have pre-dated the dispatch before items were picked (e.g., partial shipment planning).
9. **Missing Events**: No "Order Cancelled" or "Order Rejected" events are present, as no such data exists. Process ends after payment.
