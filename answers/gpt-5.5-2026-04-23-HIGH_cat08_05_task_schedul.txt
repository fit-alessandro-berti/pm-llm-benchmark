## 1. Analyzing Historical Scheduling Performance and Dynamics

### 1.1 Build a process-mining-ready operational data model

Before mining, I would transform the MES event log into an analyzable operational history. In a high-mix job shop, the same log needs to be viewed from two perspectives:

1. **Job-centric view**: each case is a customer job, e.g., `JOB-7001`.
2. **Resource-centric view**: each machine/operator has its own sequence of activities, e.g., all work performed on `MILL-03`.

Because job-shop scheduling involves interactions between jobs, machines, setups, operators, and disruptions, I would use an **object-centric process mining** approach where the main objects are:

- Jobs/orders.
- Operations/tasks.
- Machines.
- Operators.
- Setup events.
- Breakdown/maintenance events.
- Priority-change events.

For each operation instance, I would reconstruct the lifecycle:

```text
Queue Entry  Setup Start  Setup End  Task Start  Task End
```

From this, we can derive:

- Queue time.
- Setup time.
- Processing time.
- Total operation sojourn time.
- Resource occupancy interval.
- Whether the operation was affected by breakdowns or interruptions.
- The previous job processed on the same machine.

This cleaned and enriched log becomes the foundation for process discovery, bottleneck analysis, conformance checking, predictive monitoring, and simulation.

---

### 1.2 Reconstruct the actual job flow and machine execution sequence

#### Job-flow reconstruction

For the job-centric perspective, I would apply:

- **Directly-Follows Graphs, DFGs**, to visualize the actual routing patterns.
- **Inductive Miner or Heuristics Miner**, to discover the dominant process model while filtering low-frequency noise.
- **Variant analysis**, to identify common and exceptional routings.
- **Trace clustering**, because in a high-mix environment one global model may become too complex. Jobs can be clustered by:
  - Product family.
  - Material.
  - Routing type.
  - Priority class.
  - Customer segment.
  - Complexity level.

This allows us to answer questions such as:

- Which routing variants are most common?
- Which variants have the longest lead times?
- Which operations are frequently skipped, repeated, or rerouted?
- Which job families are most prone to lateness?
- Where do jobs spend most of their time waiting?

#### Machine-sequence reconstruction

For each machine, I would build a resource-centric event log by ordering all tasks executed on that machine chronologically:

```text
MILL-03:
JOB-6988  JOB-6992  JOB-7001  JOB-7005  ...
```

For each consecutive pair of jobs on the same machine, we can extract:

- Previous job attributes.
- Next job attributes.
- Setup duration.
- Processing duration.
- Idle time between jobs.
- Whether there was a breakdown or operator unavailability between jobs.
- Whether the next job was selected despite another job having tighter due-date pressure.

This provides the evidence needed to analyze actual dispatching behavior, setup patterns, bottlenecks, starvation, and priority inversions.

---

### 1.3 Metrics and process mining techniques

#### A. Job flow times, lead times, and makespan distributions

For each job `j`:

- Release time: `Rj`
- Completion time: `Cj`
- Due date: `Dj`

Key metrics:

```text
Flow time / lead time = Cj - Rj
Lateness = Cj - Dj
Tardiness = max(0, Cj - Dj)
Earliness = max(0, Dj - Cj)
```

I would analyze:

- Mean, median, 75th, 90th, and 95th percentile lead time.
- Lead time by product family, priority, routing, customer, and due-date tightness.
- Lead time variance and coefficient of variation.
- Completion-time reliability.
- Distribution of remaining lead time after each operation.
- Makespan for production campaigns, weekly job sets, or customer order batches.

Useful process mining views:

- **Performance-annotated process maps** showing average and percentile waiting/processing times on each edge and activity.
- **Dotted charts** to visualize job progress over calendar time.
- **Cumulative flow diagrams** to show WIP accumulation by process stage.
- **Throughput time decomposition**, separating value-adding time from queue/setup/delay time.

---

#### B. Task waiting times at each work center or machine

For each operation:

```text
Queue time = Setup Start - Queue Entry
```

If no setup is required:

```text
Queue time = Task Start - Queue Entry
```

I would calculate:

- Average and percentile queue time per machine.
- Queue time by operation type.
- Queue time by priority level.
- Queue time by shift or day of week.
- Queue time before and after breakdowns.
- Queue time for on-time jobs versus late jobs.
- Queue time as a percentage of total job lead time.

Important questions:

- Which machines have the largest queues?
- Do high-priority jobs actually wait less?
- Are some queues caused by lack of machine capacity, lack of operators, or bad sequencing?
- Does WIP accumulate before a particular bottleneck?

A particularly useful analysis is to decompose total job lead time:

```text
Total lead time =
Queue time + Setup time + Processing time + Transport/wait-between-operations time + Disruption delay
```

In many job shops, the actual machining time is a small fraction of the full lead time. Process mining makes this visible.

---

#### C. Resource utilization: productive time, idle time, setup time, downtime

For each machine `m` over a calendar period:

```text
Calendar time = total scheduled time
Downtime = breakdown/maintenance time
Available time = calendar time - planned off-shift time - downtime
Productive time = sum of actual task processing times
Setup time = sum of setup durations
Busy time = productive time + setup time
Idle time = available time - busy time
```

Useful utilization metrics:

```text
Run utilization = productive time / available time
Busy utilization = (productive time + setup time) / available time
Setup ratio = setup time / busy time
Downtime ratio = downtime / calendar time
Idle ratio = idle time / available time
```

I would also classify idle time:

- **True idle**: no job waiting and machine available.
- **Starvation**: machine available but no upstream job has arrived.
- **Blocked**: machine cannot process because downstream constraints prevent movement.
- **Operator-constrained idle**: machine available but no qualified operator.
- **Breakdown-related unavailable time**.

For operators, similar metrics apply:

- Direct machining support time.
- Setup support time.
- Waiting/idle time.
- Time spent on multiple machines.
- Overtime.
- Skill-related bottlenecks.

This helps distinguish machine bottlenecks from labor bottlenecks.

---

#### D. Sequence-dependent setup time analysis

This is critical for Precision Parts Inc.

For each machine, I would reconstruct the exact sequence:

```text
Previous job i on machine m  Current job j on machine m
```

For the current job `j`:

```text
Setup time Sijm = Setup Endj - Setup Startj
```

Then enrich each pair with attributes of both jobs:

- Previous material type.
- Current material type.
- Previous part family.
- Current part family.
- Tooling family.
- Fixture type.
- Surface finish requirement.
- Heat treatment requirement.
- Tolerance class.
- Operator.
- Shift.
- Machine condition.
- Whether the job was urgent.

This allows construction of setup matrices such as:

| Previous Family | Current Family | Median Setup | 90th Percentile Setup | Count |
|---|---:|---:|---:|---:|
| Stainless | Stainless | 18 min | 30 min | 220 |
| Stainless | Aluminum | 55 min | 80 min | 140 |
| Aluminum | Aluminum | 12 min | 20 min | 310 |
| Aluminum | Titanium | 75 min | 110 min | 90 |

Techniques:

- **Resource-centric process mining** to reconstruct machine sequences.
- **Setup transition matrices** by material, family, tooling, or fixture.
- **Regression or machine learning models** to estimate setup time:
  
  ```text
  Expected setup time = f(machine, previous job attributes, current job attributes, operator, shift)
  ```

- **Heatmaps** to identify expensive setup transitions.
- **Outlier analysis** to identify abnormal setups.
- **Pareto analysis** to identify the transitions contributing most setup time.

This analysis directly supports setup-aware scheduling.

---

#### E. Schedule adherence and tardiness

For each job:

```text
Lateness = Completion time - Due date
Tardiness = max(0, Completion time - Due date)
```

Key metrics:

- Percentage of jobs completed late.
- Mean tardiness.
- Median tardiness.
- 90th percentile tardiness.
- Maximum tardiness.
- Weighted tardiness by priority or customer importance.
- Number of due-date violations per customer.
- Jobs completed early versus just-in-time versus late.
- Tardiness by routing variant, machine path, product family, or priority.

For operation-level adherence, compare actual starts/finishes with planned starts/finishes if a schedule exists:

```text
Start deviation = Actual Start - Planned Start
Finish deviation = Actual Finish - Planned Finish
```

This allows us to detect:

- Where schedules begin to slip.
- Which operations consume the most slack.
- Which work centers frequently cause downstream lateness.
- Whether urgent jobs are actually protected.

A useful visualization is a **slack consumption chart**:

```text
Job release  Operation 1  Operation 2  Operation 3  Completion
```

At each step, we measure how much due-date slack remains.

---

#### F. Impact of disruptions: breakdowns and priority changes

Breakdowns and hot jobs should not be treated as noise; they are central to the scheduling problem.

For each breakdown:

```text
Breakdown Start  Breakdown End
```

We measure:

- Mean time between failures, MTBF.
- Mean time to repair, MTTR.
- Downtime by machine.
- Jobs interrupted during processing.
- Jobs waiting in the affected queue.
- Downstream machines starved due to the breakdown.
- Additional tardiness attributable to the breakdown.

For priority changes or hot jobs:

- Time of priority change.
- Queue position before and after expediting.
- Jobs displaced by the hot job.
- Tardiness impact on displaced jobs.
- Whether the hot job still completed on time.
- Schedule nervousness caused by repeated priority changes.

Analytical techniques:

- **Event studies** around breakdowns: compare queue lengths and tardiness before/after the event.
- **Difference-in-differences analysis**: compare jobs exposed to a breakdown with similar jobs not exposed.
- **Causal process mining** to estimate the effect of priority changes and disruptions.
- **Survival analysis or hazard models** to estimate how disruptions increase the probability of tardiness.
- **Predictive process monitoring** to estimate delay risk after a disruption occurs.

---

## 2. Diagnosing Scheduling Pathologies

Once the historical behavior is reconstructed, the next step is to identify the scheduling pathologies causing missed due dates, excessive WIP, and unstable lead times.

---

### 2.1 Bottleneck resources

A bottleneck is not merely a busy machine; it is a resource whose capacity constrains overall throughput and creates system-wide waiting.

Evidence from process mining:

- Persistent high utilization, especially above 85¢90%.
- Large and growing queues.
- High queue-time contribution to total lead time.
- Jobs that pass through the resource have much higher tardiness.
- Downstream machines experience starvation after bottleneck delays.
- Cumulative flow diagrams show WIP piling up before the machine.

Metrics:

```text
Bottleneck queue contribution =
Queue time at machine m / total job lead time
```

Also:

```text
Load ratio = workload released to machine / available capacity
```

If, for example, `MILL-03` accounts for only 15% of processing time but 45% of total waiting time, it is likely a major bottleneck.

Process mining techniques:

- Bottleneck analysis with performance overlays.
- Resource utilization mining.
- Queue length time-series analysis.
- Critical-path analysis through job traces.
- Comparison of job variants that use versus do not use the suspected bottleneck.

---

### 2.2 Poor task prioritization

The current local FCFS/EDD rules may generate priority inversions. For example, an urgent job or a job close to due date may wait behind a low-priority job because that job arrived first at the queue.

Evidence:

- High-priority jobs still show long queue times.
- Jobs with little remaining slack wait behind jobs with large slack.
- Near-due jobs are delayed at non-bottleneck machines because local dispatching ignores downstream bottlenecks.
- Hot jobs cause excessive displacement but still complete late.
- EDD at one machine causes congestion at the next machine.

Process mining analyses:

- Compare queue time by priority level.
- Identify dispatching decisions where the selected job had lower urgency than other jobs in the queue.
- Analyze on-time versus late jobs using variant analysis.
- Mine decision rules implicitly used by operators or dispatchers.
- Compare actual task selection against idealized rules such as ATC, CR, EDD, or setup-aware dispatching.

A useful metric is the **priority inversion rate**:

```text
Priority inversion rate =
Number of times a less urgent job was processed while a more urgent feasible job waited
/ Total dispatch decisions
```

---

### 2.3 Suboptimal sequencing and excessive setup time

If sequence-dependent setup is ignored, machines may repeatedly switch between dissimilar jobs, wasting capacity.

Evidence:

- High setup-time percentage at bottleneck machines.
- Frequent transitions between incompatible product families.
- Setup times vary strongly depending on previous job attributes.
- The same workload could have been processed with fewer setup hours if grouped differently.
- Operators informally batch similar jobs, causing inconsistency and hidden WIP.

Process mining analyses:

- Setup transition matrices.
- Heatmaps of expensive transitions.
- Pareto charts of setup-loss contributors.
- Comparison of actual sequence versus setup-minimizing sequence.
- Analysis of setup time for on-time versus late jobs.

Example pathology:

```text
MILL-03 alternates:
Titanium  Aluminum  Titanium  Stainless  Aluminum

Instead of:
Aluminum  Aluminum  Stainless  Titanium  Titanium
```

The first sequence may protect FCFS fairness but causes excessive setup time and reduces bottleneck capacity.

---

### 2.4 Starvation of downstream resources

A downstream machine can appear underutilized not because demand is low, but because upstream scheduling fails to feed it consistently.

Evidence:

- Downstream resource has available time but no jobs ready.
- Upstream bottleneck queue is large while downstream machines are idle.
- Idle intervals downstream follow delayed completions upstream.
- Downstream utilization is volatile despite stable demand.

Process mining techniques:

- Resource-to-resource flow analysis.
- Time-aligned queue analysis.
- Cumulative flow diagrams between operations.
- Waiting-time propagation analysis.
- Inter-operation lag analysis.

This shows whether idle machines are genuinely over-capacity or are being starved by upstream instability.

---

### 2.5 WIP bullwhip and release instability

High WIP often arises from releasing too many jobs into the shop without considering bottleneck capacity.

Evidence:

- Large WIP swings by work center.
- Jobs released in large batches.
- Queue lengths increase faster than throughput.
- Due-date performance worsens during high-WIP periods.
- Expediting causes further instability.
- Jobs wait for long periods before bottleneck resources while downstream work centers are idle.

Process mining analyses:

- WIP time series by operation and work center.
- Cumulative input-output diagrams.
- LittleÆs Law validation:

```text
Average WIP  Throughput Rate ū Average Lead Time
```

- Correlation between WIP level and lead time.
- Correlation between queue length and tardiness.
- Variant comparison of jobs released during overloaded versus normal periods.

---

## 3. Root Cause Analysis of Scheduling Ineffectiveness

The diagnosed issues likely arise from several interacting root causes.

---

### 3.1 Static dispatching rules are inadequate for a dynamic job shop

FCFS and EDD are simple and locally understandable, but they ignore important factors:

- Remaining processing time.
- Downstream congestion.
- Bottleneck status.
- Setup dependency.
- Job priority.
- Hot-job insertion impact.
- Breakdown risk.
- Operator availability.
- Global due-date feasibility.

A job that looks urgent locally may not be the best next job globally. Conversely, a job that seems non-urgent locally may need to move now to avoid missing a downstream bottleneck slot.

Process mining can reveal this by comparing actual dispatching outcomes with counterfactual dispatching logic.

---

### 3.2 Lack of real-time shop floor visibility

If schedulers and operators do not have an accurate real-time view of:

- Machine status.
- Queue lengths.
- Job progress.
- Remaining work.
- Operator availability.
- Downstream congestion.
- Setup implications.
- Predicted completion risk.

Then local dispatching becomes reactive rather than proactive.

Process mining can support a live operational dashboard showing:

- Current WIP by operation.
- Jobs at risk of tardiness.
- Bottleneck queues.
- Machine availability.
- Setup opportunities.
- Predicted completion dates.
- Disruption impact.

---

### 3.3 Inaccurate processing and setup time estimates

If planned times are systematically wrong, schedules will be unrealistic.

Using the MES log, compare:

```text
Planned task duration versus actual task duration
Planned setup duration versus actual setup duration
```

Break down estimation error by:

- Machine.
- Operator.
- Part family.
- Material.
- Shift.
- Job complexity.
- Setup transition type.
- Priority status.

Metrics:

```text
Mean error = Actual - Planned
Absolute percentage error
Bias by operation type
Variance by job family
```

Findings might include:

- CNC milling on titanium parts is underestimated by 25%.
- Setups after stainless jobs are highly variable.
- Night-shift processing times differ from day-shift processing times.
- Hot jobs have shorter waits but higher disruption impact on others.

These insights should feed directly into predictive duration models.

---

### 3.4 Sequence-dependent setup is not being managed explicitly

If the schedule treats setup as fixed or negligible, it will underestimate capacity requirements and create unnecessary machine losses.

Process mining helps quantify:

- Which setup transitions are costly.
- Which machines are most setup-sensitive.
- Whether operators already perform informal batching.
- How much capacity could be recovered through better sequencing.
- Whether setup reduction would materially improve due-date performance.

---

### 3.5 Poor coordination between work centers

Local optimization can harm global performance.

Examples:

- Cutting releases work faster than milling can absorb it, creating WIP.
- Milling optimizes FCFS but starves grinding.
- Heat treatment batches jobs in a way that delays urgent orders.
- Inspection becomes overloaded at the end of the week due to upstream release patterns.

Process mining detects this through:

- Inter-operation waiting analysis.
- Cross-resource queue dynamics.
- Lead-time decomposition.
- Resource synchronization analysis.
- Bottleneck propagation analysis.

---

### 3.6 Weak disruption response

Breakdowns and hot jobs are inevitable, but the current system appears to respond reactively.

Problems may include:

- No rapid rescheduling after breakdowns.
- No estimation of downstream impact.
- No rules for inserting hot jobs without excessive schedule damage.
- No predictive warning that a machine is likely to fail.
- No reserve capacity or buffer policy for urgent work.

Process mining can show:

- How often breakdowns occur.
- Which machines create the largest disruption impact.
- How long recovery takes.
- Which jobs are displaced by hot jobs.
- Whether hot-job insertion improves service for urgent jobs at acceptable cost.

---

### 3.7 Distinguishing scheduling logic problems from capacity limitations

This distinction is essential. If demand exceeds capacity, no scheduling algorithm will solve tardiness alone.

I would use several tests.

#### A. Load versus capacity analysis

For each machine and time bucket:

```text
Required workload = sum of remaining processing + expected setup
Available capacity = machine calendar time - planned downtime - expected breakdown downtime
```

If:

```text
Required workload > Available capacity
```

for sustained periods, the issue is at least partly structural capacity shortage.

#### B. Utilization versus waiting-time relationship

- Very high utilization plus long queues suggests a capacity bottleneck.
- Moderate utilization plus long queues suggests poor scheduling, operator constraints, batching delays, or coordination problems.

#### C. Tardiness decomposition

For late jobs, decompose delay into:

```text
Delay from queueing
Delay from processing overrun
Delay from setup overrun
Delay from breakdown
Delay from waiting for operator
Delay from priority displacement
```

If queueing dominates, scheduling/release logic is likely a major cause.

#### D. Counterfactual simulation

Using the same historical arrivals and breakdowns, simulate alternative schedules:

- Current FCFS/EDD baseline.
- Setup-aware dispatching.
- Bottleneck-focused dispatching.
- Finite-capacity optimized scheduling.
- Perfect-information lower bound.

If alternative logic materially reduces tardiness without adding capacity, then scheduling logic is a major contributor.

#### E. Feasibility lower bounds

Compare due dates against theoretical minimum completion times:

```text
Minimum possible completion  release time + routing processing time + minimum setup + unavoidable queue/load constraints
```

If many due dates are infeasible even under ideal sequencing, capacity or due-date quoting is a root cause.

## 4. Developing Advanced Data-Driven Scheduling Strategies

I would not recommend one single rule as the final answer. Precision Parts needs a layered approach: better dispatching, predictive scheduling, setup optimization, and WIP control.

---

# Strategy 1: Dynamic multi-factor dispatching rules

### Core logic

Replace static FCFS/EDD rules with a dynamic priority index calculated whenever a machine becomes available.

For each candidate job `j` waiting at machine `m`, calculate a dispatch score based on:

- Due-date slack.
- Customer/order priority.
- Remaining processing time.
- Current operation processing time.
- Estimated sequence-dependent setup time from the previous job.
- Downstream machine congestion.
- Bottleneck urgency.
- Probability of tardiness.
- Whether the job is blocking other high-priority work.
- Operator availability.

A setup-aware dispatching index could be based on the **Apparent Tardiness Cost with Setups**, ATCS, extended for downstream load:

```text
Score(j,m) =
PriorityWeightj
ū exp(- max(Slackj, 0) / k1)
ū exp(- ExpectedSetupijm / k2)
ū 1 / ExpectedProcessingTimejm
ū DownstreamAdjustmentj
ū TardinessRiskj
```

Where:

```text
Slackj = DueDatej - CurrentTime - EstimatedRemainingProcessingTimej
```

The machine chooses the feasible job with the highest score.

### How process mining informs the rule

Process mining provides:

- Actual processing-time distributions by job type and machine.
- Setup-time estimates based on previous/current job attributes.
- Historical queue-time behavior by work center.
- Bottleneck identification.
- Tardiness-risk factors.
- Downstream congestion patterns.
- Historical evidence of priority inversions.

The weights in the rule should not be guessed. They should be tuned using historical replay and simulation.

### Pathologies addressed

This strategy addresses:

- Poor local prioritization.
- Excessive waiting for high-priority jobs.
- Ignoring sequence-dependent setup.
- Starvation of bottleneck or downstream resources.
- Overreaction to hot jobs.
- Poor due-date protection.

### Expected KPI impact

Expected improvements:

- Lower weighted tardiness.
- Better protection of urgent jobs.
- Reduced average queue time at constrained resources.
- Lower setup losses than FCFS/EDD.
- More predictable flow times.
- Better utilization of bottleneck machines.

This strategy is practical because it can be deployed incrementally at key work centers.

---

# Strategy 2: Predictive finite-capacity rolling-horizon scheduling

### Core logic

Instead of relying only on dispatch rules, build a rolling-horizon scheduler that creates a finite-capacity schedule for the next several shifts or days.

The scheduler would be rerun:

- At the start of each shift.
- When a breakdown occurs.
- When a hot job arrives.
- When a job finishes much earlier or later than expected.
- When WIP or queue thresholds are breached.

The optimization model should include:

- Job precedence constraints.
- Alternative eligible machines.
- Machine calendars.
- Operator skills and availability.
- Sequence-dependent setup times.
- Priority weights.
- Due dates.
- Current WIP.
- Machine breakdown status.
- Expected repair times.
- Release dates.
- No-preemption or limited-preemption rules.
- Schedule stability penalties to avoid excessive nervousness.

Possible solving approaches:

- Constraint programming, CP-SAT.
- Mixed-integer programming for smaller horizons.
- Metaheuristics such as tabu search, genetic algorithms, simulated annealing.
- Hybrid optimization plus dispatching.
- Rolling-horizon heuristic with bottleneck-first optimization.

Objective function:

```text
Minimize:
Weighted tardiness
+ Setup time
+ WIP holding cost
+ Machine idle cost at bottlenecks
+ Schedule change penalty
```

### Predictive models used

From the mined event log, build predictive models for:

#### Processing time

```text
ProcessingTime = f(machine, operation, material, part family, complexity, operator, shift)
```

Use not just point estimates but distributions or quantiles.

#### Setup time

```text
SetupTime = f(machine, previous job attributes, current job attributes, tooling, operator)
```

#### Remaining lead time

Using predictive process monitoring:

```text
RemainingTime = f(current operation, completed path, queue status, resource load, priority, routing)
```

#### Breakdown risk

If maintenance data are sufficient:

```text
FailureRisk = f(machine age, recent utilization, prior failures, operation type, time since last maintenance)
```

Even a simple machine-specific MTBF/MTTR model is useful for robust scheduling.

### How process mining informs the strategy

Process mining supplies:

- Actual routing variants.
- Empirical operation duration distributions.
- Setup transition models.
- Breakdown frequencies and repair durations.
- Current bottleneck structure.
- Delay propagation patterns.
- Realistic queue and WIP behavior.

### Pathologies addressed

This strategy addresses:

- Global coordination failures.
- Inaccurate duration assumptions.
- Lack of proactive bottleneck management.
- Poor disruption response.
- Unrealistic due-date promises.
- Local optimization that harms global flow.

### Expected KPI impact

Expected improvements:

- Lower tardiness and weighted tardiness.
- More realistic completion-date predictions.
- Reduced schedule infeasibility.
- Faster recovery after disruptions.
- Better machine and operator coordination.
- Lower lead-time variability.

This strategy is more complex than dispatching but provides the strongest global control.

---

# Strategy 3: Setup-time optimization through intelligent batching and sequencing

### Core logic

Use historical setup data to reduce avoidable setup losses, especially at bottleneck machines.

The key idea is not to batch blindly, but to batch intelligently within due-date and WIP constraints.

For each setup-sensitive machine:

1. Build a setup transition matrix.
2. Cluster jobs by setup similarity:
   - Material.
   - Tooling.
   - Fixture.
   - Part family.
   - Tolerance class.
3. Identify compatible batches.
4. Sequence jobs to minimize total setup while respecting due dates.
5. Limit batching delay so urgent jobs are not held too long.

The sequencing problem can be formulated as:

```text
Minimize:
Total sequence-dependent setup time
+ Weighted tardiness
+ Excess waiting from batching
```

This is similar to a traveling-salesman or shortest-path problem with time windows and due-date penalties.

### How process mining informs the strategy

Process mining provides:

- Actual setup durations by job-pair attributes.
- Machines where setup time materially reduces capacity.
- Expensive transitions to avoid.
- Setup-sensitive product families.
- Historical batching behavior.
- Tradeoff between setup reduction and waiting-time increase.

### Example

If `MILL-03` frequently alternates between aluminum and titanium jobs, and the setup matrix shows:

```text
Aluminum  Aluminum: 12 min
Titanium  Titanium: 18 min
Aluminum  Titanium: 75 min
Titanium  Aluminum: 70 min
```

Then the scheduler should avoid unnecessary alternation unless due-date pressure justifies it.

### Pathologies addressed

This strategy addresses:

- Excessive setup time.
- Bottleneck capacity loss.
- Erratic sequencing.
- Long queues caused by avoidable changeovers.
- Operators making informal, inconsistent batching decisions.

### Expected KPI impact

Expected improvements:

- Lower total setup time.
- Increased effective capacity at bottlenecks.
- Reduced queue time behind setup-sensitive machines.
- Improved throughput.
- Lower tardiness if setup savings outweigh batching delay.
- More stable machine schedules.

Important caveat: aggressive batching can increase WIP and delay urgent jobs. Therefore, batching must be constrained by due-date slack and WIP limits.

---

# Strategy 4: Bottleneck-centered WIP control and release scheduling

### Core logic

High WIP is often caused by releasing work into the shop faster than the bottleneck can process it. Therefore, scheduling should control not only which job runs next, but also when jobs are released.

Use a **drum-buffer-rope** or **CONWIP-like** policy:

- The bottleneck machine is the ōdrum.ö
- Jobs are released according to bottleneck capacity.
- A protective buffer ensures the bottleneck is never starved.
- WIP caps prevent uncontrolled queue growth.
- Upstream resources are synchronized to feed the bottleneck.
- Downstream resources prioritize jobs released from the bottleneck.

### How process mining informs the strategy

Process mining determines:

- The true bottleneck machines.
- Average and percentile queue times before bottlenecks.
- Required protective buffer size.
- WIP levels associated with stable versus unstable flow.
- Release patterns that lead to lateness.
- Lead-time distributions by job family.

### Pathologies addressed

This strategy addresses:

- Excessive WIP.
- Long and unpredictable lead times.
- Bottleneck overload.
- Downstream starvation.
- Release instability.
- WIP bullwhip effects.

### Expected KPI impact

Expected improvements:

- Lower WIP.
- Shorter and more predictable lead times.
- Better bottleneck utilization.
- Less firefighting.
- Improved due-date reliability.
- Clearer release priorities.

This strategy is especially important if the process mining analysis shows that many jobs spend most of their time waiting before a small number of constrained resources.

## 5. Simulation, Evaluation, and Continuous Improvement

### 5.1 Build a discrete-event simulation digital twin

Before live deployment, the proposed scheduling strategies should be tested in a discrete-event simulation model calibrated with process mining data.

The simulation should include:

#### Jobs

- Historical and stochastic job arrivals.
- Due dates.
- Priority levels.
- Routing variants.
- Product family and material.
- Operation precedence constraints.
- Hot-job arrivals and priority changes.

#### Resources

- Machines.
- Alternative machine eligibility.
- Operators and skill constraints.
- Calendars and shifts.
- Planned maintenance.
- Breakdowns and repair durations.

#### Operation behavior

- Processing-time distributions.
- Setup-time models.
- Queueing rules.
- Transport or move delays if relevant.
- Rework or inspection failure rates if present.

#### Disruptions

- Breakdown frequency and duration.
- Operator absence.
- Hot-job insertion.
- Emergency maintenance.
- Material delays.

The simulation should first reproduce the historical baseline. If the model cannot reproduce current WIP, utilization, lead time, and tardiness reasonably well, it should not yet be used for strategy evaluation.

### 5.2 Compare proposed scheduling strategies against the baseline

Strategies to compare:

1. Current FCFS/EDD baseline.
2. Dynamic multi-factor dispatching.
3. Predictive rolling-horizon finite-capacity scheduling.
4. Setup-aware batching/sequencing.
5. Bottleneck-centered WIP release control.
6. Hybrid strategy combining the best elements.

Use common random numbers and multiple replications to ensure statistically valid comparisons.

KPIs:

- Percentage of jobs tardy.
- Mean and 90th percentile tardiness.
- Weighted tardiness.
- Average lead time.
- Lead-time variability.
- WIP level.
- Queue time by machine.
- Bottleneck utilization.
- Setup time percentage.
- Throughput.
- Hot-job completion performance.
- Schedule nervousness.
- Operator overtime.
- Machine idle/starvation time.

The goal is not merely to maximize utilization. In job shops, excessive utilization at bottlenecks can explode lead times. The best strategy is usually one that balances due-date performance, WIP, setup efficiency, and robustness.

### 5.3 Scenarios to test

I would test the scheduling policies under multiple realistic stress scenarios.

#### Baseline replay

Use historical arrivals, routings, and breakdowns to compare each scheduling strategy against what actually happened.

#### High-load scenario

Increase order volume by 10%, 20%, and 30%.

Purpose:

- Determine when the system becomes capacity constrained.
- Identify which strategies degrade gracefully.

#### Frequent breakdown scenario

Increase breakdown frequency or MTTR on critical machines.

Purpose:

- Test robustness.
- Evaluate rescheduling speed.
- Quantify benefit of predictive maintenance or buffer capacity.

#### Hot-job surge scenario

Inject additional urgent jobs.

Purpose:

- Evaluate whether urgent work is protected without destroying normal-job performance.

#### Due-date compression scenario

Reduce customer lead times.

Purpose:

- Determine whether current due-date promises are feasible.
- Support better order promising.

#### Product-mix shift scenario

Increase jobs requiring setup-heavy transitions or bottleneck-heavy routings.

Purpose:

- Test whether setup-aware scheduling remains effective.

#### Operator constraint scenario

Reduce availability of skilled operators.

Purpose:

- Determine whether apparent machine bottlenecks are actually labor bottlenecks.

#### Maintenance policy scenario

Compare reactive maintenance versus planned preventive maintenance windows.

Purpose:

- Quantify whether planned downtime reduces total disruption.

### 5.4 Continuous monitoring and adaptation using process mining

After deployment, the scheduler should be managed as a continuously learning system.

#### Real-time process mining dashboard

Track:

- Current WIP by operation and work center.
- Queue lengths.
- Jobs at risk of lateness.
- Bottleneck machine status.
- Machine utilization.
- Setup time ratio.
- Breakdown status.
- Dispatching-rule adherence.
- Priority inversions.
- Predicted versus actual completion dates.
- Schedule nervousness.

#### KPI control charts

Use statistical process control for:

- Lead time.
- Queue time.
- Tardiness.
- Setup duration.
- Machine downtime.
- Bottleneck utilization.
- WIP levels.
- Prediction error.

This helps detect whether performance improvement is sustained or degrading.

#### Drift detection

Automatically detect changes in:

- Processing-time distributions.
- Setup-time distributions.
- Routing variants.
- Product mix.
- Breakdown frequency.
- Operator performance.
- Queue-time behavior.
- Hot-job frequency.
- Bottleneck location.

Techniques:

- CUSUM or EWMA control charts.
- Kolmogorov-Smirnov tests for distribution shifts.
- Population stability index for feature drift.
- Process variant drift detection.
- Conformance checking against expected routing and scheduling behavior.

#### Model recalibration

When drift is detected:

- Re-estimate processing-time models.
- Re-estimate setup matrices.
- Update bottleneck definitions.
- Retune dispatching-rule weights.
- Re-optimize WIP caps.
- Retrain tardiness prediction models.
- Revalidate the simulation model.

#### Closed-loop improvement cycle

A practical governance cycle would be:

```text
Mine actual performance 
Diagnose deviations 
Update predictive models 
Test scheduling changes in simulation 
Pilot on selected machines 
Monitor KPI impact 
Scale if successful
```

This creates a continuous improvement loop rather than a one-time scheduling project.

## Summary

Precision Parts Inc. should move from local, static dispatching to a data-driven scheduling architecture built on process mining.

The core steps are:

1. **Reconstruct actual job and machine behavior** from MES logs.
2. **Quantify lead time, waiting time, utilization, setup losses, tardiness, and disruption impact.**
3. **Diagnose bottlenecks, priority inversions, poor sequencing, starvation, WIP instability, and schedule nervousness.**
4. **Separate scheduling-logic failures from true capacity constraints.**
5. **Deploy improved scheduling strategies**, including:
   - Dynamic multi-factor dispatching.
   - Predictive rolling-horizon finite-capacity scheduling.
   - Setup-aware batching and sequencing.
   - Bottleneck-centered WIP release control.
6. **Validate all strategies through discrete-event simulation before deployment.**
7. **Use ongoing process mining for monitoring, drift detection, and continuous adaptation.**

The expected result is lower tardiness, reduced WIP, more predictable lead times, better bottleneck utilization, and a scheduling system that adapts to real shop-floor conditions rather than reacting after problems occur.
