### General Overview
This BPMN diagram illustrates a cross-departmental order fulfillment process. The process spans four distinct organizational pools (departments): **Sales**, **Production**, **Procurement**, and **Warehouse**. The workflow demonstrates how an order is processed, materials are sourced, the product is built, and finally shipped, utilizing message flows (dashed lines) to communicate between the separate pools.

---

### Step-by-Step Process Description

#### 1. Sales Pool
*   **Initiation:** The entire process begins in the Sales department with a Timer Start Event scheduled for **"Monday morning"**.
*   **Sub-process - Check customer's order:** The flow immediately enters this sub-process to validate the order.
    *   It starts by checking the order for completeness.
    *   Next, it checks if the customer is already in the database.
    *   An Exclusive Gateway (decision point) directs the flow based on the database check:
        *   If the **"Customer not in the database"**, a task is executed to **"Add customer to the database,"** which interacts with a data store (database).
        *   If the **"Customer in the database"**, it bypasses the addition step.
    *   Both paths merge, and the sub-process concludes.
*   **Broadcasting Order:** Once checked, the Sales team executes the **"Send order details"** task. This initiates two outgoing message flows containing the "Order details", triggering processes in both the Production and Procurement pools concurrently.
*   **Wait State:** Sales enters a wait state at an Intermediate Catching Message Event labeled **"Product sent"**, pausing further action until it hears back from the Warehouse.

#### 2. Procurement Pool (Parallel Activity)
*   **Trigger:** The process starts via a Message Start Event when **"Order details arrive"** from Sales.
*   **Material Compensation Subprocess:** The department executes an **"Order materials"** task. Notably, there is a compensation event attached to this task, linked to a **"Return material"** compensation task (which is triggered if the order needs to be canceled later).
*   **Quality Check:** After ordering, materials undergo a **"Check materials"** task. This task has an attached Error Boundary Event denoting exception handling.
    *   **Failure Path:** If the materials fail the check, the process flows through an error event to an **"Order cancelation"** Compensation End Event. This triggers the "Return material" task defined in the earlier subprocess.
    *   **Success Path:** If the check passes, the flow proceeds to an Intermediate Throwing Message Event. This sends a message signaling that the material is not faulty up to the Production pool.

#### 3. Production Pool (Parallel Activity)
*   **Trigger:** Starts via a Message Start Event when **"Order details arrive"** from Sales.
*   **Planning:** Production immediately begins to **"Plan production."**
*   **Synchronization Gateway:** After planning, the flow hits an AND-Join (Parallel Gateway). Production cannot proceed further until it satisfies two conditions:
    1.  The production planning is complete.
    2.  It receives the **"Material not faulty"** message from the Procurement department (via an Intermediate Catching Message Event).
*   **Manufacturing:** Once both conditions are met, the **"Manufacture product"** task is executed.
*   **Handoff:** Upon completion, the flow reaches an Intermediate Throwing Message Event labeled **"Product complete,"** which sends a message down to the Warehouse pool.

#### 4. Warehouse Pool
*   **Trigger:** Starts with a Message Start Event when it receives the **"Product complete"** message from Production.
*   **Shipping:** The Warehouse executes the **"Send product"** task to fulfill the order.
*   **Completion Notification:** Upon sending the physical product, the process hits a Message End Event labeled **"Product sent."** This sends a final message flow all the way back up to the waiting event in the Sales pool.

#### Finalizing in Sales
*   **Closing the Loop:** The Sales department's waiting event (**"Product sent"**) receives the message from the Warehouse.
*   **Invoicing:** The process resumes, and Sales executes the final task: **"Draft invoice."**
*   **End:** The entire end-to-end business process concludes at the final plain End Event in the Sales pool.
