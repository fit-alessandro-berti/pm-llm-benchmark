We translate raw system actions into higher-level process steps. Examples:

- `CLICK (Reply, Send)`  *Reply to Email / Send Email*  
- `SCROLL`  *Review Content*  
- `HIGHLIGHT`  *Highlight Content*  
- `CLOSE`  *Close Document*  
- `SWITCH`  Omit in favor of focusing on meaningful work (acts as navigation, not a process step).

### Step 4: Explanation of Transformation
- **Case Identification:** Each distinct *resource* (document, email, PDF) represents a single **case**. This way, the process log reflects the lifecycle of working on one artifact.  
- **Activity Mapping:** Low-level system events were abstracted into high-level user activities that align with *business process semantics*. For instance, *TYPING* is simplified to *Edit Content*, *FOCUS* on a resource becomes *Open Document*, and *CLICK  Send* becomes *Send Email*.  
- **Filtering Switches:** Navigation events (SWITCH/FOCUS shifts between apps) were suppressed unless they indicated actual case work (e.g., *first focus on a file* is mapped to *Open Document*).  
- **Narrative:**  
   - The user started by drafting content in *Document1.docx*,  
   - then handled an *Annual Meeting email*,  
   - reviewed a draft PDF report,  
   - updated *Budget_2024.xlsx*,  
   - returned to Document1, and finally worked on *Quarterly_Report.docx*.  

This structured log is now ready for import into standard process mining tools (e.g., Disco, ProM, Celonis), to discover how user work moves across tasks and documents.
