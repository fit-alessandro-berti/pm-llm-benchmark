## Identified Anomalies in Sample Data

### 1. **Case 1002: Process Skipping & Out-of-Order Execution**
- **Anomaly**: Credit check and stock validation occur AFTER shipment confirmation and goods shipping
- **Hypothesis**: System bypass for priority orders, urgent shipments, or process control failure
- **Query to investigate**:

```sql
-- Find priority orders that skip credit check or stock validation
SELECT 
    o.case_id,
    o.order_type,
    COUNT(DISTINCT e.activity) as unique_activities,
    BOOL_OR(e.activity = 'Perform Credit Check') as has_credit_check,
    BOOL_OR(e.activity = 'Validate Stock') as has_stock_validation,
    BOOL_OR(e.activity = 'Ship Goods') as has_shipment,
    MIN(e.timestamp) as first_activity,
    MAX(e.timestamp) as last_activity
FROM orders o
LEFT JOIN order_event_log e ON o.case_id = e.case_id
WHERE o.order_type = 'priority'
GROUP BY o.case_id, o.order_type
HAVING BOOL_OR(e.activity = 'Ship Goods') = true 
   AND (BOOL_OR(e.activity = 'Perform Credit Check') = false 
        OR BOOL_OR(e.activity = 'Validate Stock') = false);
```

### 2. **Case 1003: Missing Stock Validation**
- **Anomaly**: Entire "Validate Stock" step is missing from the process
- **Hypothesis**: Warehouse system malfunction, data entry error, or process bypass for standard orders
- **Query to investigate**:

```sql
-- Find all orders missing critical process steps
SELECT 
    o.case_id,
    o.order_type,
    o.order_value,
    CASE WHEN COUNT(CASE WHEN e.activity = 'Perform Credit Check' THEN 1 END) = 0 
         THEN 'Missing Credit Check' ELSE 'Has Credit Check' END as credit_check_status,
    CASE WHEN COUNT(CASE WHEN e.activity = 'Validate Stock' THEN 1 END) = 0 
         THEN 'Missing Stock Validation' ELSE 'Has Stock Validation' END as stock_validation_status,
    CASE WHEN COUNT(CASE WHEN e.activity = 'Confirm Shipment' THEN 1 END) = 0 
         THEN 'Missing Shipment Confirmation' ELSE 'Has Shipment Confirmation' END as shipment_confirmation_status

GROUP BY o.case_id, o.order_type, o.order_value
HAVING COUNT(CASE WHEN e.activity = 'Validate Stock' THEN 1 END) = 0
ORDER BY o.order_value DESC;
```

### 3. **Case 1004: Pre-payment & Process Reversal**
- **Anomaly**: Payment received before invoice issued, before shipment confirmed
- **Hypothesis**: Payment processing error, refund scenario, or process automation failure
- **Query to investigate**:

```sql
-- Find orders where payment occurs before invoicing
SELECT 
    e1.case_id,
    e1.activity as first_activity,
    e1.timestamp as first_timestamp,
    e2.activity as second_activity,
    e2.timestamp as second_timestamp,
    EXTRACT(EPOCH FROM (e1.timestamp - e2.timestamp))/60 as minutes_between
FROM order_event_log e1
JOIN order_event_log e2 ON e1.case_id = e2.case_id
WHERE e1.activity = 'Receive Payment'
  AND e2.activity = 'Issue Invoice'
  AND e1.timestamp < e2.timestamp
ORDER BY minutes_between DESC;
```

-- Find resources performing activities outside their department
SELECT

    COUNT(*) as activity_count,
    COUNT(DISTINCT e.case_id) as order_count

### 5. **Process Deviation Patterns**
```sql
-- Analyze process patterns by order type and value
WITH process_patterns AS (
    SELECT 
        o.case_id,
        o.order_type,
        o.order_value,
        ARRAY_AGG(e.activity ORDER BY e.timestamp) as process_sequence,
        COUNT(*) as step_count,
        BOOL_OR(e.activity = 'Perform Credit Check') as has_credit_check,
        BOOL_OR(e.activity = 'Validate Stock') as has_stock_validation
    FROM orders o
    JOIN order_event_log e ON o.case_id = e.case_id
    GROUP BY o.case_id, o.order_type, o.order_value
)
SELECT 
    order_type,
    CASE 
        WHEN order_value > 2000 THEN 'High Value'
        WHEN order_value > 1000 THEN 'Medium Value'
        ELSE 'Low Value'
    END as value_segment,
    COUNT(*) as order_count,
    AVG(step_count) as avg_steps,
    COUNT(CASE WHEN has_credit_check = false THEN 1 END) as missing_credit_check,
    COUNT(CASE WHEN has_stock_validation = false THEN 1 END) as missing_stock_validation,
    MODE() WITHIN GROUP (ORDER BY process_sequence) as most_common_pattern
FROM process_patterns
GROUP BY order_type, 
    CASE 
        WHEN order_value > 2000 THEN 'High Value'
        WHEN order_value > 1000 THEN 'Medium Value'
        ELSE 'Low Value'
    END
ORDER BY order_type, value_segment;
```
