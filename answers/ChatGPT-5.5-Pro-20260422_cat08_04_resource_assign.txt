## 1. Analyzing Resource Behavior and Assignment Patterns

Process mining should be used to reconstruct what *actually* happens to incidents, not what the documented procedure says should happen. The event log already contains the key ingredients: case ID, activity, timestamp, resource, tier, skills, ticket priority, category, required skill, and start/complete markers. This is exactly the type of event-log evidence process mining is designed to use for discovering, monitoring, and improving real operational processes, including organizational/resource behavior and simulation-based improvement. ([tf-pm.org][1])

### 1.1 Prepare an enriched incident-resource dataset

For each ticket, sort events by timestamp and derive these additional fields:

| Derived field                   |                                                       How to calculate it | Why it matters                                  |
| ------------------------------- | ------------------------------------------------------------------------: | ----------------------------------------------- |
| **End-to-end resolution time**  |           Ticket resolved/closed timestamp minus ticket created timestamp | Overall SLA performance                         |
| **Queue time after assignment** |                                         Work Start minus Assign timestamp | Detects waiting caused by resource availability |
| **Processing time**             |                                                 Work End minus Work Start | Measures actual effort, by agent/tier/skill     |
| **Escalation wait time**        |                             Next-tier Work Start minus Escalate timestamp | Quantifies delays between tiers                 |
| **Reassignment count**          |                    Count of Reassign or repeated Assign events per ticket | Measures routing instability                    |
| **First-assignment success**    | Ticket resolved without reassignment or escalation after first assignment | Core assignment-quality KPI                     |
| **Skill-match flag**            |           Required skill is present in assigned agent’s documented skills | Detects likely misrouting                       |
| **Tier-match flag**             |             Assigned tier is appropriate for priority/category/complexity | Detects overuse of L2/L3                        |
| **Workload at assignment time** |             Agent’s active WIP and queued tickets at assignment timestamp | Detects overload-based delays                   |
| **SLA remaining at assignment** |                                   SLA deadline minus assignment timestamp | Supports urgency-aware routing                  |

The analysis should distinguish **waiting time** from **working time**. In ITSM processes, SLA breaches often come less from long hands-on effort and more from long queueing, wrong handovers, or late escalation.

### 1.2 Agent- and tier-level performance metrics

For each agent and each tier, I would calculate:

| Area                       | Metrics                                                                                                                              |
| -------------------------- | ------------------------------------------------------------------------------------------------------------------------------------ |
| **Workload distribution**  | Number of tickets assigned, active WIP, queued WIP, utilization, idle time, overtime, workload Gini coefficient, backlog by priority |
| **Processing performance** | Mean, median, and P90 work duration by activity, priority, category, and required skill                                              |
| **Queue performance**      | Time from assignment to work start, time from escalation to next work start, time waiting for specialist skills                      |
| **Resolution quality**     | SLA breach rate, reopen rate, repeat-contact rate, tickets solved without reassignment                                               |
| **L1 effectiveness**       | First-call resolution rate, escalation rate, avoidable escalation rate, L1 handling time before escalation                           |
| **Escalation behavior**    | Escalations per 100 tickets, escalation timing, tier path patterns such as L1 → L2 → L1 or L2 → L2 reassignments                     |
| **Skill usage**            | Volume and success rate by required skill, skill-match rate, overqualification rate, specialist time spent on low-complexity work    |
| **Case mix**               | Distribution of priorities, categories, customer groups, channels, and required skills handled by each agent                         |

Raw agent performance should never be interpreted without controlling for case mix. An L2 network specialist handling mostly P2 firewall issues will naturally have longer durations and higher SLA risk than an L1 agent handling password resets. The fair comparison is “performance for similar ticket types under similar workload conditions.”

### 1.3 Discover actual assignment, escalation, and reassignment patterns

I would mine the process from several perspectives:

**Control-flow view:**
Discover the actual paths tickets follow, for example:

`Created → Assign L1 → Work L1 → Resolve`

versus

`Created → Assign L1 → Work L1 → Escalate L2 → Assign L2 → Wait → Reassign → Assign L2 → Work L2 → Resolve`

This reveals whether TechSolve’s practical process is close to the intended tiered model or dominated by loops, reassignments, and late escalations.

**Conformance view:**
Encode the intended assignment logic as rules and compare actual behavior against them. Examples:

| Intended rule                                                                              | Conformance question                                                     |
| ------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------ |
| P2 network firewall tickets should be routed to agents with Networking-Firewall capability | How often are they assigned to agents without that skill?                |
| L1 should handle basic troubleshooting and known-resolution cases                          | How often does L1 escalate cases that historically are L1-resolvable?    |
| L2/L3 specialists should focus on specialized incidents                                    | How much L2/L3 time is spent on low-complexity, repeatable work?         |
| Reassignments should be exceptional                                                        | Which categories or dispatchers produce repeated reassignments?          |
| Round-robin should distribute comparable work evenly                                       | Are some agents overloaded after controlling for shift and availability? |

**Resource interaction and handover analysis:**
Create a handover-of-work network where nodes are agents or teams and edges represent tickets moving from one resource to another. In process mining tools such as PM4Py, a handover-of-work network is essentially a directly-follows graph where the nodes are resources rather than activities, so it can show who passes work to whom. PM4Py also supports activity-based resource similarity, role discovery, subcontracting networks, and working-together networks. ([PM4Py][2])

This can reveal patterns such as:

* Agent A05 frequently escalates Software-App tickets to Agent B12.
* B12 frequently reassigns App-CRM tickets to B15 because the true need is Database-SQL.
* Certain dispatchers route most urgent cases to the same small group of trusted specialists.
* Some agents are “hidden hubs” even though the formal model says work is distributed evenly.
* Some teams operate as silos, causing ping-pong between application, database, and infrastructure groups.

**Role discovery:**
Cluster agents by their actual activity and ticket patterns. This may show that the formal L1/L2/L3 structure does not match reality. For example:

* Some L1 agents may behave like de facto access-management specialists.
* Some L2 agents may spend most of their time on basic password, setup, or application-support tasks.
* Some L3 agents may only be involved after repeated failed reassignments, indicating late expert engagement.

### 1.4 Analyze skill utilization

A skill-utilization matrix should be created with agents as rows and required skills as columns. For each agent-skill pair, calculate:

| Metric                               | Interpretation                            |
| ------------------------------------ | ----------------------------------------- |
| Tickets handled requiring that skill | Actual use of the skill                   |
| Median processing time               | Speed for that skill                      |
| Reassignment-after-agent rate        | Whether the agent was a good fit          |
| SLA success rate                     | Outcome quality                           |
| Documented skill present?            | Whether the skill profile is accurate     |
| Specialist-level task or basic task? | Whether scarce skills are being preserved |

Important skill metrics:

**Skill-match rate**
`Assignments where assigned agent has required skill / all assignments`

**Skill-mismatch penalty**
Difference in resolution time, queue time, reassignment rate, and SLA breach rate between skill-matched and skill-mismatched assignments.

**Specialist leakage**
Percentage of L2/L3 time spent on tickets that historical data shows could have been resolved by L1 or by a less specialized group.

**Scarce-skill utilization**
For each scarce skill, such as Networking-Firewall or Security-IAM:

`Demand hours for skill / available skilled-agent capacity hours`

**Proficiency estimate**
For each agent and skill, estimate proficiency using historical performance:

`Proficiency = function(success without reassignment, median handling time, SLA success, reopen rate, case complexity)`

This allows TechSolve to move beyond a binary “has skill / does not have skill” view toward a more realistic proficiency-based routing model.

## 2. Identifying Resource-Related Bottlenecks and Issues

### 2.1 Skill bottlenecks

A skill bottleneck exists when tickets requiring a specific skill experience long waits before the right agent starts work. This can be detected by comparing queue and processing times by required skill.

Example analysis:

| Required skill      | Queue time before specialist work | Processing time | Interpretation                                    |
| ------------------- | --------------------------------: | --------------: | ------------------------------------------------- |
| Networking-Firewall |                              High |        Moderate | Not enough available firewall-skilled agents      |
| Database-SQL        |                              High |            High | Possible capacity and complexity issue            |
| App-CRM             |                          Moderate |             Low | Potential routing issue, not technical difficulty |
| Security-IAM        |              High during mornings |             Low | Shift coverage mismatch                           |

The key distinction is important: if queue time is high but processing time is normal, the issue is probably **availability or routing**. If both are high, the issue may be **capacity, complexity, missing knowledge, or tool constraints**.

### 2.2 Reassignment delays

For every reassignment, calculate the delay introduced:

`Reassignment delay = next Work Start timestamp after reassignment − Reassign timestamp`

Also calculate the broader penalty:

`Adjusted reassignment penalty = additional resolution time for tickets with reassignment, compared with similar tickets without reassignment`

The comparison should control for priority, category, required skill, channel, time of day, and initial workload. Otherwise, reassignments may look bad simply because complex tickets are more likely to be reassigned.

Useful metrics:

| Metric                                | Purpose                                                                        |
| ------------------------------------- | ------------------------------------------------------------------------------ |
| Average delay per reassignment        | Direct waiting-time impact                                                     |
| P90 delay per reassignment            | Worst-case operational impact                                                  |
| Reassignments per ticket              | Routing stability                                                              |
| Reassignment loop rate                | Detects ping-pong between agents or teams                                      |
| Reassignment reason distribution      | Skill mismatch, workload, wrong category, unavailable agent, unclear ownership |
| SLA breach rate by reassignment count | Quantifies SLA risk escalation                                                 |

For example, the analysis might show that P3 tickets with zero reassignments rarely breach SLA, while P3 tickets with two or more reassignments breach frequently. The exact percentages should come from the full event log, not assumptions.

### 2.3 Incorrect initial assignment

Initial assignment quality is critical because early routing errors consume SLA time before the right person sees the ticket.

Measure:

| Metric                            | Definition                                                        |
| --------------------------------- | ----------------------------------------------------------------- |
| Initial skill mismatch rate       | First assigned agent lacks the required skill                     |
| Early bounce rate                 | Ticket is reassigned or escalated shortly after first work begins |
| L1 avoidable touch rate           | L1 worked the ticket but had no realistic chance of resolving it  |
| Wrong-category correction rate    | Required skill or category changed after work began               |
| Dispatcher-specific mismatch rate | Mismatch rate by dispatcher or assignment mechanism               |

For tickets like `INC-1001`, the sequence suggests the initial App-CRM requirement may have been incomplete because the ticket later needed Database-SQL. That is not only a resource-assignment problem; it may also be a **classification and diagnosis problem**.

### 2.4 Overloaded and underutilized agents

For each agent, reconstruct workload over time:

* Active assigned tickets.
* Tickets waiting in that agent’s queue.
* Tickets actively being worked.
* SLA exposure of the assigned backlog.
* Average age of assigned tickets.
* After-hours or overtime work.
* Number of P1/P2 tickets handled concurrently.

An overloaded specialist may appear slow, but the process-mining view may show that the real issue is excessive queue depth. Research on resource behavior in process mining emphasizes that human processing behavior can depend on workload, and that better resource characterization can improve work distribution and simulation realism. 

To avoid unfair conclusions, compare agents using case-mix-adjusted metrics:

`Adjusted performance = actual outcome − expected outcome for similar tickets`

This identifies:

* Truly overloaded agents.
* Underutilized agents with matching skills.
* Agents who are repeatedly assigned work outside their skill set.
* Agents whose documented skills may be inaccurate.
* Dispatchers or queues that concentrate work on a small number of trusted experts.

### 2.5 Correlation between assignment patterns and SLA breaches

Build a statistical model to quantify SLA breach drivers. A practical model could be:

`SLA breach probability = f(priority, category, required skill, channel, initial skill match, reassignment count, escalation count, queue time, workload at assignment, tier path, time of day, dispatcher, agent)`

This can be implemented using logistic regression, survival analysis, gradient-boosted trees, or interpretable decision trees. The output should answer questions such as:

* How much does an initial skill mismatch increase breach probability?
* How much SLA risk is added by each reassignment?
* Which required skills are most associated with P2/P3 breaches?
* Are breaches caused more by long work time or long queue time?
* Which tier transitions cause the most delay?
* Which assignment decisions are most predictive of successful resolution?

A useful executive view would decompose breached tickets into causes:

| SLA breach contributor              | Example quantification                                                        |
| ----------------------------------- | ----------------------------------------------------------------------------- |
| Waiting for specialist availability | % of breached tickets where specialist queue time exceeded threshold          |
| Initial skill mismatch              | % of breached tickets whose first assignment lacked required skill            |
| Multiple reassignments              | % of breached tickets with two or more reassignments                          |
| Late escalation                     | % of breached tickets where escalation occurred after consuming most SLA time |
| Overloaded agent queue              | % of breached tickets assigned to agents already above WIP threshold          |

## 3. Root Cause Analysis for Assignment Inefficiencies

### 3.1 Likely root causes

The event log should be used to validate or reject these hypotheses.

| Potential root cause                          | Evidence to look for in the event log                                                         |
| --------------------------------------------- | --------------------------------------------------------------------------------------------- |
| **Round-robin ignores skill**                 | Assignments are evenly distributed, but skill-mismatch and reassignment rates are high        |
| **Round-robin ignores workload**              | Agents receive new tickets while already above WIP threshold, while qualified agents are idle |
| **Manual dispatcher bias**                    | Certain agents receive disproportionate complex work from specific dispatchers                |
| **Incomplete skill profiles**                 | Agents repeatedly resolve tickets requiring skills not listed in their profiles               |
| **Inflated skill profiles**                   | Agents listed with a skill frequently reassign or breach tickets needing that skill           |
| **Poor ticket classification**                | Category or required skill changes after first work activity                                  |
| **Lack of real-time availability visibility** | Tickets are assigned to agents who are unavailable, overloaded, or near shift end             |
| **L1 training gaps**                          | High escalation rates for categories that other L1 agents resolve successfully                |
| **L1 empowerment gaps**                       | L1 escalates cases because they lack permissions, not because they lack knowledge             |
| **Overuse of specialists**                    | L2/L3 agents spend high time on historically L1-resolvable tickets                            |
| **Siloed technical teams**                    | Handover networks show repeated ping-pong between App, DB, Network, and Security teams        |

### 3.2 Variant analysis

Split cases into two groups:

**Smooth assignment variants**

* No reassignment.
* Resolved by first assigned agent or expected tier.
* SLA met.
* Low queue time.
* Skill match present at first assignment.

**Problematic variants**

* Multiple reassignments.
* Escalation loops.
* SLA breach or near breach.
* Skill mismatch.
* Long wait before specialist work.
* Late identification of true required skill.

Then compare the two groups by:

| Attribute                   | Question                                                                    |
| --------------------------- | --------------------------------------------------------------------------- |
| Priority                    | Are P2/P3 tickets more likely to follow problematic variants?               |
| Category                    | Which categories produce most reassignment loops?                           |
| Required skill              | Which skills are most often misidentified or unavailable?                   |
| Channel                     | Are phone tickets classified faster or less accurately than portal tickets? |
| Time of day                 | Are after-hours tickets more likely to be misrouted?                        |
| Initial agent               | Which agents produce high first-resolution versus high escalation?          |
| Dispatcher                  | Which dispatchers or assignment mechanisms create more mismatches?          |
| Workload at assignment      | Are bad variants caused by assigning to overloaded agents?                  |
| SLA remaining at escalation | Are escalations happening too late?                                         |

This allows TechSolve to identify the conditions under which assignment decisions go wrong, not merely count how often they go wrong.

### 3.3 Decision mining

Decision mining should be applied at key routing points:

1. **Assign L1**
2. **Escalate to L2**
3. **Assign L2**
4. **Reassign within L2**
5. **Escalate to L3**
6. **Resolve versus escalate**

At each decision point, extract the case attributes available at that moment:

* Priority.
* Category.
* Channel.
* Required skill.
* Description keywords.
* Customer or business unit.
* Current workload of candidate agents.
* Agent availability.
* Agent skill profile.
* Historical proficiency.
* SLA time remaining.
* Previous assignment attempts.
* Similar-ticket resolution history.

Then mine rules that explain successful and failed decisions. For example:

* “P2 Network tickets with Firewall keywords assigned to non-firewall L2 agents have high reassignment probability.”
* “P3 App-CRM tickets from web portal with database-error keywords are frequently reclassified to Database-SQL after first L2 work.”
* “Tickets assigned to agents with more than N active P2/P3 tickets experience materially higher queue delay.”
* “Certain L1 agents resolve Access Management tickets successfully, while others escalate the same category quickly.”

The goal is to convert informal dispatcher knowledge into explicit, testable, continuously improved routing rules.

## 4. Developing Data-Driven Resource Assignment Strategies

### Strategy 1: Skill-based routing with proficiency weighting

**Issue addressed:**
Skill mismatch, repeated reassignment, and late specialist engagement.

**How it uses process-mining insight:**
The mined log identifies which skills are actually required, which agents successfully resolve which skill types, and where skill mismatches produce rework or SLA breaches.

**Data required:**

* Required skill per ticket.
* Agent skill profile.
* Proficiency level by skill.
* Historical success rate by agent-skill-category.
* Agent availability and shift calendar.
* Current workload.
* SLA remaining.

**Assignment logic:**
Instead of assigning by simple round-robin, create a candidate pool of agents who match the required skill. Then rank them by predicted outcome.

Example scoring logic:

`Assignment score = skill match + proficiency + availability + SLA urgency + case familiarity − overload penalty − overqualification penalty`

The overqualification penalty matters. If a ticket is simple and historically L1-resolvable, the algorithm should avoid consuming scarce L2/L3 specialist capacity unless SLA risk requires it.

**Expected benefits:**

* Fewer skill-related reassignments.
* Faster specialist engagement for genuinely complex tickets.
* Reduced P2/P3 SLA breaches caused by incorrect routing.
* Better use of scarce L2/L3 capacity.
* More transparent assignment decisions.

---

### Strategy 2: Workload-aware and SLA-aware assignment

**Issue addressed:**
Uneven workload distribution, overloaded specialists, underused agents, and queue-related SLA breaches.

**How it uses process-mining insight:**
The event log shows actual queue times, active WIP, processing times, and how workload affects performance. This allows routing to consider not only “who can solve this?” but also “who can start and finish this before the SLA is at risk?”

**Data required:**

* Real-time active ticket count by agent.
* Queue length by agent and skill.
* Current shift availability.
* Historical processing-time distributions by skill and priority.
* SLA deadline and remaining time.
* Calendar constraints, breaks, on-call status, and planned absences.

**Assignment logic:**
For each eligible agent, estimate:

`Predicted completion time = expected queue delay + expected processing time`

Then assign the ticket to the qualified agent with the best predicted SLA outcome, not necessarily the lowest current ticket count.

For P2/P3 tickets, the algorithm should prioritize:

* Agents with matching skills.
* Agents with enough available capacity.
* Agents whose predicted completion time fits the SLA.
* Agents not already carrying excessive high-priority WIP.

**Expected benefits:**

* Lower queue time after assignment.
* More balanced utilization.
* Fewer tickets assigned to agents who cannot start soon.
* Better P2/P3 SLA compliance.
* Reduced reliance on manual dispatcher judgment.

---

### Strategy 3: Predictive triage and direct-to-skill routing

**Issue addressed:**
Poor initial classification, unnecessary L1 touches, and late discovery of the true required skill.

**How it uses process-mining insight:**
Historical variants show which ticket characteristics predict required skill, complexity, escalation likelihood, and successful resolver group.

**Data required:**

* Ticket title and description.
* Category and subcategory.
* Channel.
* Priority.
* Customer or service affected.
* Historical activity path.
* Final resolver group.
* Final required skill.
* Resolution code.
* Knowledge-base article used.
* Reassignment and SLA outcome.

**Assignment logic:**
Use a predictive model to estimate:

* Required skill.
* Likely resolver tier.
* Probability of L1 first-call resolution.
* Probability of reassignment.
* SLA breach risk.
* Whether the ticket should skip L1.

Examples:

| Model output                                                | Routing action                                            |
| ----------------------------------------------------------- | --------------------------------------------------------- |
| High confidence: L1-resolvable                              | Route to L1 agent with best performance for that category |
| High confidence: Networking-Firewall                        | Route directly to L2/L3 network specialist queue          |
| High confidence: Database-SQL hidden under App-CRM category | Add Database-SQL skill tag before assignment              |
| Low confidence                                              | Route to triage specialist or dispatcher review           |
| High SLA-breach risk                                        | Prioritize assignment and avoid overloaded queues         |

**Expected benefits:**

* Fewer incorrect initial assignments.
* Less wasted L1 effort on tickets L1 cannot resolve.
* Faster routing to the right specialist.
* Earlier detection of hidden skills such as Database-SQL behind application symptoms.
* Better SLA protection for P2/P3 tickets.

---

### Strategy 4: Data-driven L1 empowerment and escalation rules

**Issue addressed:**
Excessive escalation and L2/L3 specialists spending time on work that L1 could handle.

**How it uses process-mining insight:**
Analyze historical L1 successes to identify ticket types that are frequently resolved by strong L1 agents. Compare those with tickets escalated by other L1 agents.

**Data required:**

* L1 resolution history.
* L1 escalation history.
* Ticket category and required skill.
* Resolution codes.
* Knowledge-base usage.
* Agent permissions.
* Time spent before escalation.
* Reopen and customer-contact outcomes.

**Assignment logic:**

Create three routing classes:

| Class                          | Handling rule                                            |
| ------------------------------ | -------------------------------------------------------- |
| **Known L1-resolvable**        | Keep in L1; recommend playbook or knowledge article      |
| **L1-resolvable with support** | Assign to trained L1 agent or provide guided checklist   |
| **Not L1-resolvable**          | Skip unnecessary L1 work and route to correct specialist |

Escalation should also require structured evidence, such as:

* Troubleshooting steps completed.
* Error code captured.
* Skill required.
* Business impact confirmed.
* Logs attached.
* Suggested resolver group.

**Expected benefits:**

* Higher first-call resolution rate.
* Reduced L2/L3 workload on routine tickets.
* Better-quality escalations when escalation is necessary.
* Less back-and-forth between tiers.
* Improved L1 capability development.

---

### Strategy 5: Dynamic resource pooling and specialist “swarming”

**Issue addressed:**
Demand spikes, scarce-skill bottlenecks, and sequential reassignments for multi-skill incidents.

**How it uses process-mining insight:**
The mined log reveals demand patterns by time of day, day of week, priority, category, and required skill. It also shows recurring multi-skill paths such as App-CRM → Database-SQL or Network → Security.

**Data required:**

* Arrival forecasts by priority, category, skill, and time.
* Agent schedules.
* Cross-trained agent list.
* Real-time queue depth.
* SLA risk by queue.
* Multi-skill ticket patterns.
* Specialist availability.

**Assignment logic:**

Use dynamic thresholds:

* If Database-SQL queue exceeds threshold, temporarily route cross-trained agents to DB-related incidents.
* If P2 Networking-Firewall backlog exceeds threshold, reserve specialist capacity and defer lower-priority non-specialist work.
* If a ticket historically requires multiple skills, create a short “swarm” session with App, DB, and Network specialists rather than serially reassigning the ticket.

**Expected benefits:**

* Shorter P90 queue times for scarce skills.
* Fewer multi-hop reassignments.
* Better handling of demand surges.
* Faster diagnosis for complex cross-domain incidents.
* More resilient use of L2/L3 capacity.

---

### Strategy 6: Continuous skill-profile governance

**Issue addressed:**
Outdated, incomplete, or inaccurate agent skill records.

**How it uses process-mining insight:**
The event log shows what agents actually do successfully, not just what their profiles claim.

**Data required:**

* Documented skills.
* Tickets resolved by skill type.
* Success without reassignment.
* Resolution duration.
* Reopen rate.
* Training and certification records.
* Peer or manager validation.

**Assignment logic:**

Update the skill matrix periodically using evidence:

| Evidence                                                    | Skill-profile action                  |
| ----------------------------------------------------------- | ------------------------------------- |
| Agent repeatedly resolves Database-SQL tickets successfully | Add or raise Database-SQL proficiency |
| Agent frequently reassigns tickets requiring listed skill   | Review or downgrade proficiency       |
| Agent handles skill only with long delays or high reopens   | Target training                       |
| Agent is repeatedly selected for scarce skill               | Check overload and succession risk    |

**Expected benefits:**

* More accurate routing.
* Better training investment.
* Less dependence on informal dispatcher knowledge.
* Improved resilience when key specialists are absent.
* Clearer career development paths for agents.

## 5. Simulation, Implementation, and Monitoring

### 5.1 Use business process simulation before implementation

Before changing the assignment rules in production, TechSolve should build a simulation model using the mined process model and resource characteristics. Process simulation can compare proposed changes with the current process and model changes to resources, durations, wait times, process flow, working hours, and KPIs before implementation. ([Microsoft Learn][3])

The simulation should include:

| Simulation input           | Source from process mining                                        |
| -------------------------- | ----------------------------------------------------------------- |
| Ticket arrival rates       | Historical arrivals by hour, day, channel, priority, and category |
| Process variants           | Mined paths such as L1-only, L1→L2, L1→L2→L3, reassignment loops  |
| Branching probabilities    | Probability of escalation, reassignment, resolution, reopen       |
| Service-time distributions | Work duration by tier, agent, category, skill, and priority       |
| Queue-time distributions   | Assignment-to-start and escalation-to-start delays                |
| Resource calendars         | Shifts, weekends, on-call coverage, holidays                      |
| Skill matrix               | Agent skills and proficiency                                      |
| Workload effects           | Relationship between WIP and processing speed or delay            |
| SLA rules                  | Priority-specific SLA targets                                     |
| Assignment rules           | Current and proposed routing algorithms                           |

The first step is to simulate the current process and validate that the simulated baseline resembles actual historical performance. The model should reproduce:

* P2/P3 SLA breach rate.
* Mean and P90 resolution time.
* Mean and P90 queue time.
* Reassignment rate.
* Escalation rate.
* L1 first-call resolution.
* Specialist utilization.
* Backlog by skill.

Then test scenarios such as:

1. Current round-robin/manual assignment.
2. Skill-based routing only.
3. Workload-aware routing only.
4. Predictive triage plus direct-to-skill routing.
5. L1 empowerment rules.
6. Dynamic specialist pooling.
7. Combined future-state model.
8. Stress scenario with 10–30% higher P2/P3 demand.
9. Absence scenario where key specialists are unavailable.
10. After-hours scenario with reduced coverage.

Simulation outputs should include:

| KPI                              | Why it matters                                 |
| -------------------------------- | ---------------------------------------------- |
| P2/P3 SLA compliance             | Primary business target                        |
| Average and P90 resolution time  | Captures normal and worst-case performance     |
| Queue time by skill              | Identifies whether bottlenecks shift elsewhere |
| Reassignments per ticket         | Measures routing quality                       |
| L1 first-call resolution         | Measures L1 effectiveness                      |
| Specialist utilization           | Ensures L2/L3 are not overloaded               |
| Workload balance                 | Detects unfair or unsustainable allocation     |
| Overtime or after-hours workload | Captures hidden operational cost               |
| Cost per resolved ticket         | Supports business case                         |
| Customer-impact hours            | Connects operations to service impact          |

### 5.2 Implementation roadmap

A safe implementation should be phased.

**Phase 1: Baseline and data-quality correction**

* Standardize activity names.
* Validate START/COMPLETE timestamps.
* Clean agent skill profiles.
* Define SLA deadlines consistently.
* Add structured reason codes for reassignment and escalation.
* Ensure every assignment has resource, tier, priority, category, and required skill.

**Phase 2: Shadow-mode recommendations**

Run the new assignment logic in parallel without changing dispatcher behavior. For each new ticket, record:

* Current assignment decision.
* Recommended assignment decision.
* Reason for recommendation.
* Predicted SLA risk.
* Actual outcome.

This allows TechSolve to evaluate the algorithm before relying on it.

**Phase 3: Controlled pilot**

Start with one or two high-impact areas, such as:

* P2/P3 Network tickets requiring Networking-Firewall.
* P3 Software-App tickets with App-CRM and Database-SQL overlap.
* Access Management tickets with high L1 escalation volume.

Compare pilot queues against control queues using the same KPIs.

**Phase 4: Human-in-the-loop rollout**

Dispatchers and team leads should be able to override recommendations, but every override should require a reason code. Override analysis is valuable because it reveals missing data, exceptional constraints, or model weaknesses.

**Phase 5: Continuous improvement**

Update routing rules, predictive models, skill profiles, and escalation criteria using new event-log evidence.

### 5.3 Monitoring dashboards after implementation

The post-implementation process-mining dashboard should continuously track both process performance and resource assignment quality.

#### Dashboard 1: SLA and flow health

Track:

* P2/P3 SLA compliance.
* Tickets at risk of breach.
* SLA remaining by ticket.
* Average and P90 resolution time.
* Average and P90 queue time.
* SLA breaches by category, required skill, tier path, and dispatcher.
* Breach attribution: queue delay, reassignment, skill mismatch, late escalation, overload.

#### Dashboard 2: Assignment quality

Track:

* First-assignment success rate.
* Initial skill-match rate.

## Practical target state

TechSolve should move from a tier-first, round-robin/manual model to a **skill-, workload-, SLA-, and prediction-aware assignment model**.

The future-state assignment decision should answer five questions before assigning a ticket:

1. **What skill is likely required?**
2. **Which tier should handle it first?**
3. **Which available agents have proven proficiency for this work?**
4. **Who can start and complete it within SLA given current workload?**
5. **Would assigning this to a specialist waste scarce capacity that should be protected for harder work?**

That approach directly addresses TechSolve’s current problems: fewer reassignments, less queue delay, better L1/L2/L3 utilization, reduced specialist misuse, and improved P2/P3 SLA compliance.
