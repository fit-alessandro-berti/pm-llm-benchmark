### Explanation of Transformation Logic

#### 1. Case Identification Strategy: Artifact-Centric Approach
I grouped events into **Cases based on the specific digital artifact (file/email) being manipulated**, rather than the application or the entire session. This is the standard "object-centric" or "artifact-centric" approach in process mining for knowledge work, as it allows for analysis of the lifecycle of specific deliverables (e.g., "How long does it take to draft Document1?" or "What is the cycle time for the Annual Meeting email?").

*   **Doc1** (`Document1.docx`): The primary working document. The user opened it, edited, saved, suspended to do other tasks, resumed, edited again, saved, and closed.
*   **Email_AnnualMtg** (`Email - Inbox`): A distinct communication process instance triggered by opening a specific email thread.
*   **Report_PDF** (`Report_Draft.pdf`): A review/annotation process instance.
*   **Budget_XL** (`Budget_2024.xlsx`): A data update process instance.
*   **Quarterly_Report** (`Quarterly_Report.docx`): A separate document editing session. Note: The initial `FOCUS` at 08:59:50 was treated as noise/accidental focus (immediate switch away), so the case starts at the effective re-opening at 09:07:15.

#### 2. Activity Name Standardization
Raw system actions (verbs) were mapped to higher-level business activities (verb-noun pairs) to ensure consistency across applications:

| Raw Action(s) | Standardized Activity | Rationale |
| :--- | :--- | :--- |
| `FOCUS` (Initial) | **Open Document / Open Spreadsheet / Open Email Thread** | Represents the start of interaction with an artifact. |
| `TYPING` (Word) | **Edit Content** | Generic content creation. |
| `TYPING` (Excel) | **Edit Data** | Semantically distinct from text editing; implies cell modification. |
| `TYPING` (Chrome/Reply) | **Edit Content** | Part of the "Compose Reply" flow. |
| `SAVE` | **Save Document / Save Spreadsheet** | Explicit persistence action. |
| `SWITCH` (Source) | **Suspend Work** | Captures the interruption/context switch *from* the artifact's perspective. |
| `SWITCH` (Target) / `FOCUS` (Return) | **Resume Work** | Captures the continuation of a previously suspended case. |
| `CLICK` (Open/Reply/Send) | **Open Email Thread / Compose Reply / Send Email** | Maps UI clicks to mail-process milestones. |
| `SCROLL` | **Scroll Content** | Navigation/Review activity. |
| `HIGHLIGHT` | **Annotate Document** | Value-adding review action in PDF. |
| `CLOSE` / `SWITCH` (Final Away) | **Close Document / Close-Switch Away** | Termination of the artifact interaction. |

#### 3. Handling Interleaving (Concurrency)
The log shows heavy multitasking (e.g., working on Doc1 -> Email -> PDF -> Excel -> Doc1). By assigning a **Case ID per artifact**, the event log natively supports this interleaving. Standard process mining tools (like PM4Py, Celonis, Disco) can import this flat event log and:
*   Discover a **variant** for Doc1: `Open -> Edit* -> Save -> Suspend -> Resume -> Edit -> Save -> Close`.
*   Calculate **idle/waiting times** for Doc1 (the ~4 min gap between Suspend and Resume).
*   Analyze **resource (Application) utilization** per case.

#### 4. Attribute Enrichment
*   **`Details` column**: Preserves the semantic payload (typed text, highlighted text, click targets) which is crucial for root-cause analysis (e.g., seeing *what* was typed in the budget vs the report).
*   **`Application` column**: Acts as the "Resource" in process mining terms, enabling handover analysis (Word -> Chrome -> Adobe -> Excel -> Word).
