Below is a data-driven, process-mining-to-scheduling blueprint for Precision Parts Inc. It is designed for a high-mix, low-volume job shop with sequence-dependent setups, resource contention, and disruptions captured in MES event logs.

---

## 1) Analyzing Historical Scheduling Performance and Dynamics (using process mining)

### 1.1 Reconstruct the real shop-floor execution from MES logs
**Event-log reconstruction goals**
- Rebuild the **actual routing** of each job (case) across activities (Cutting  Milling  ů).
- Rebuild the **machine-centric execution traces** (what actually ran on CUT-01, MILL-03, etc., in what order).
- Identify **queueing periods** (time between ôqueue entryö and ôsetup startö or ôtask startö) and **processing periods** (setup + run).

**How to do it**
1. **Case model (job-centric)**
   - Sort events by timestamp per `Case ID`.
   - Map MES activity labels into a small canonical taxonomy:
     - `Job Released`, `Queue Entry (station/machine)`, `Setup Start/End`, `Task Start/End`, `Breakdown/Resumed`, `Priority Change`, `Complete`, etc.
   - Create derived attributes per task instance:
     - planned vs actual task duration
     - planned vs actual setup duration
     - waiting time(s)
     - number of interruptions

2. **Resource model (machine-centric)**
   - For each `Resource (Machine ID)` build a timeline of processing blocks:
     - Setup block duration
     - Processing block duration
     - Downtime blocks (breakdown)
   - Create ordered sequences of job-task identifiers processed on each machine.

3. **Conformance alignment (optional but powerful)**
   - If you have a known ôidealö process model (variant routing template), use **alignments** to locate deviations:
     - wrong/missing step
     - additional waits caused by changes/disruptions
     - rework loops (if present)

### 1.2 Techniques and metrics for the required KPIs

#### A) Job flow times, lead times, makespan distributions
**Definitions**
- **Flow time (per job):** from first task start to last task completion.
- **Lead time (per job):** from job released to job completion/delivery completion (if completion captured).
- **Makespan (per horizon):** distribution of completion times across jobs processed in a period; optionally per product family.

**Process mining techniques**
- **Time-annotation**: compute durations from event timestamps.
- **Variant analysis**: compare different routing variants and their lead-time distributions.
- **Distributional KPIs**:
  - mean/median and **tail metrics** (p90, p95) for flow/lead time (tail is what drives missed due dates).
  - makespan per period and per bottleneck load regime.

---

#### B) Task waiting times (queue times) at each work center/machine
**Definitions**
For each task instance on machine \(m\):
- Queue time = `Setup Start`  `Queue Entry`  
  (or if setup is not logged, Queue time = `Task Start`  `Queue Entry`)
- Setup time = `Setup End`  `Setup Start`
- Processing/run time = `Task End`  `Task Start`
- Total time in system on machine = Queue + Setup + Processing (+ interruption downtime if logged during setup/run)

**Process mining**
- **Bottleneck-aware queue mining**: compute queue-time distributions by:
  - machine
  - product family / routing type
  - job priority / due-date urgency band
  - time-of-day/day-of-week
- **Causal decomposition** (practical):
  - separate what portion of delay is queue vs setup vs processing deviation.

---

#### C) Resource utilization (productive, idle, setup)
**Definitions (per machine and optionally operator)**
- **Productive time**: setup + processing time during which the resource is actually engaged (excluding downtime).
- **Setup time share**: cumulative setup / total time in horizon.
- **Idle time**:
  - between tasks when machine is available but no job is processed (no queue-to-start event)
- **Breakdown/downtime**: durations from breakdown start to recovery/resume events.

**Metrics**
- utilization = productive / (productive + idle + downtime)
- OEE-like breakdown: productive/(productive+idle) vs downtime/(total)
- operator utilization similarly if operators are consistently logged.

**Process mining technique**
- **Timeline-based performance mining**: treat each resource as a ôsystemö and compute occupancy segments from event sequences.

---

#### D) Sequence-dependent setup times (SDST) quantification
This is central to scheduling improvement.

**How to mine SDST**
1. For each machine \(m\), build the sequence of tasks/jobbytes processed:
   \[
   \text{PrevJobTask} \rightarrow \text{NextJobTask}
   \]
2. For each transition, compute:
   - actual setup duration for the transition
   - planned vs actual setup duration
3. Define sequence state features:
   - attributes of previous and next jobs (product type, material grade, tool family, tolerance class, etc.)
   - operator used (if relevant)
   - machine condition (if logged)
4. Fit a model:
   - simplest: transition matrix (prev class  next class)
   - better: regression/ML for setup time conditioned on pair attributes:
     - \( \text{SetupTime} = f(\text{PrevAttrs}, \text{NextAttrs}, \text{Operator}, \text{TimeSinceLastSetup}, \dots)\)

**Key SDST metrics**
- average setup time by transition class
- setup-time coefficient of variation (variability)
- % of setups that are ôexpensiveö transitions
- ôsetup volatilityö over time (whether scheduling causes repeated expensive transitions)

---

#### E) Schedule adherence and tardiness
**Definitions**
- Tardiness \(T_j\) = completion time  due date
- EWT (evident work time?) optionally: lateness at each task milestone
- **Order-level**
  - # late jobs, mean/median tardiness, p90 lateness
  - ôdue-date adherence by stageö: whether lateness begins early or accumulates later

**Process mining techniques**
- **Due-date conformance analysis**:
  - for each job, compare planned vs actual completion
  - compute ôfirst violation timeö (first task/machine that causes downstream inability)
- **Variant vs tardiness**: compare routing variants and machine sequences between on-time and late jobs.

---

#### F) Impact of disruptions (breakdowns, priority changes) on KPIs
**How to isolate disruption impact**
1. Identify disruption intervals:
   - breakdown start/end per machine
   - hot job insertion events (priority changes)
2. Tag all task instances overlapping disruption windows.
3. Create disruption cohorts:
   - tasks started before vs during vs after disruption
   - jobs that entered system shortly before a breakdown
4. Use event-triggered analysis:
   - ôlead-time jumpö around disruptions
   - queue growth rates after breakdowns
   - measure whether priority changes systematically starve others (WIP/queue inflation for specific classes)

**Metrics**
- breakdown frequency and mean downtime
- disruption-induced queue backlog growth
- incremental tardiness attributable to breakdown windows (difference-in-differences style)

## 2) Diagnosing Scheduling Pathologies (and proving them)

Below are common pathologies for this job-shop setting and how process mining provides evidence.

### 2.1 Identify bottlenecks and quantify their throughput impact
**Evidence to mine**
- machines with:
  - highest blocking time (jobs waiting for downstream)
  - highest queue-time distributions
  - highest utilization near 100% productive (but with high variance due to setup/downtime)
- throughput-limited behavior:
  - changes in system throughput when the machine is down

**Process mining methods**
- **Bottleneck discovery**: correlation between machine queue growth and job completion delays.
- **Resource contention windows**:
  - detect periods where multiple upstream jobs simultaneously require the same bottleneck machine.
- **Causal-ish evidence**:
  - compare on-time vs late jobs: do late jobs disproportionately experience the bottleneckĺs queue/SDST transitions?

---

### 2.2 Poor prioritization (high-priority or near-due jobs delayed)
**What to look for**
- near-due jobs spending long times in queues while low-priority jobs get processed
- priority changes causing immediate disruption of local sequencing without global protection

**How to prove**
- **Comparison cohort analysis**
  - categorize jobs by priority + due-date urgency at queue entry (e.g., EDD urgency: time-to-due-date)
  - compare their realized waiting times and number of ôovertakesö on each machine
- **Sequence fairness metrics**
  - for each machine, compute how often a job with higher priority/urgency enters later but is processed earlier.
- **ôUrgency inversionö rate**
  - quantify the frequency and magnitude (time-to-due-date gap) of inversions.

---

### 2.3 Suboptimal sequencing increases total setup times
**What to look for**
- expensive SDST transitions happen frequently
- setups become dominant share of time at bottleneck machines
- high variability in setups causes downstream schedule instability

**How to prove**
- **Variant-by-setup decomposition**
  - for each job, compute total setup time it incurred across visits
  - compare late vs on-time job sequences
- **Transition entropy / costly transition rate**
  - measure whether the system repeatedly alternates between dissimilar job families.
- **Counterfactual analysis (lightweight)**
  - on each machine, simulate ôwhat if we only reorder within a limited lookahead window to reduce setup transitionsö
  - compare setup time and tardiness statistics using mined SDST model.

---

### 2.4 Starvation of downstream resources (blocking) or upstream starvation
**What to look for**
- WIP builds before a machine, while downstream sits idle
- job movement patterns indicate blocking due to machine capacity after certain steps

**How to prove**
- **Bottleneck propagation analysis**
  - identify where blocking time begins (task waiting due to downstream unavailability)
- **Littleĺs Law-like cross-check (empirical)**
  - WIP growth upstream vs throughput downstream for each station pair
- **Event-based blocking labels**
  - derive ôblockedö status if queue entry occurs but start is delayed while downstream machine is busy (requires mapping upstream/downstream relationships via routing).

---

### 2.5 Bullwhip effect in WIP due to scheduling variability
**What to look for**
- WIP fluctuates widely with little relation to stable demand
- queue length oscillations at machines after disruptions or hot job insertions

**How to prove**
- compute queue length time series per machine from event logs
- measure:
  - autocorrelation / periodicity
  - amplification of WIP variance downstream vs upstream
- **Disruption wave analysis**
  - after a breakdown or priority change, track how quickly queue spikes propagate across stations.

## 3) Root Cause Analysis of Scheduling Ineffectiveness

### 3.1 Likely root causes, and how process mining differentiates them

#### A) Limits of local static dispatching rules in a dynamic shop
**Symptom**
- Local FCFS/EDD decisions ignore SDST costs and downstream capacity/load, creating global instability.

**Mining-based diagnosis**
- For each machine event, reconstruct what local rule likely chose (if rule is known) and compare:
  - realized SDST cost vs alternative sequencing cost (using mined SDST model)
  - realized queue waiting for high-urgency jobs vs what global objective would prefer (e.g., minimizing expected earliness/lateness)
- Identify instances where greedy ôearliest due date nextö causes a costly SDST pattern.

---

#### B) Lack of real-time visibility / state-aware decisions
**Symptom**
- dispatching decisions appear uninformed about machine state (setup in progress, breakdown probability, queue explosion).

**Mining-based diagnosis**
- quantify time-to-start conditioned on:
  - predicted machine availability state (based on recent breakdown history)
  - queue length at dispatch moment
- show that decisions correlate poorly with actual availability states (e.g., jobs queued when machine likely to be down soon).

---

#### C) Inaccurate processing/setup time estimates
**Symptom**
- planned vs actual durations vary widely; schedules built on planned times miss due dates.

**Mining-based diagnosis**
- model estimation error:
  - (actual  planned) distributions per task type and per operator/machine
- correlate estimation error with tardiness:
  - if tardiness is high precisely when variability spikes, itĺs a planning-accuracy problem, not just dispatch order.

---

#### D) Ineffective handling of sequence-dependent setups
**Symptom**
- schedules treat setups as additive constants rather than sequence costs.

**Mining-based diagnosis**
- show that expensive transitions:
  - occur more often in late-job cohorts
  - increase waiting downstream by increasing cycle time variance
- measure whether reducing SDST would close the gap even holding same processing variability.

---

#### E) Poor coordination across work centers
**Symptom**
- local decisions create blocking/starvation patterns and high WIP.

**Mining-based diagnosis**
- compare on-time vs late jobs on **station-to-station timing**:
  - where the lateness ôstarts accumulatingö
- use **stagewise lateness decomposition**:
  - due date feasibility at each step vs observed completion.

---

#### F) Response failures to disruptions (breakdowns, hot jobs)
**Symptom**
- breakdowns/hot jobs cause uncontrolled schedule churn, WIP spikes, and lateness cascades.

**Mining-based diagnosis**
- measure the ôrecovery timeö and subsequent schedule stability:
  - how quickly queues return to baseline after breakdown end
- compare policies via retrospective ôwhat would have happened if we had rerouted/resequenced with interruption-aware objectives?ö

---

### 3.2 Distinguish scheduling logic issues vs capacity/variability limitations
A useful approach is **counterfactual decomposition**:

1. **Capacity-only** counterfactual:
   - hold routing and sequencing fixed, but replace processing/setup times with expected values (remove estimation error variability).
   - If tardiness remains high  likely capacity bottleneck.

2. **Scheduling-only** counterfactual:
   - keep timing distributions (variability) but improve sequencing using mined setup-aware models.
   - If tardiness drops significantly  sequencing/dispatch logic is the dominant driver.

Process mining supports both via simulation or ôreplayö with learned models.

## 4) Developing Advanced Data-Driven Scheduling Strategies (3)

### Strategy 1 Ś Setup-aware, downstream-aware Enhanced Dispatching (dynamic multi-factor rule)
**Core logic**
At each machine, when deciding which next job to process, score each candidate job \(j\) using a weighted expected-cost function:

\[
\text{Score}(j)=
w_1\cdot \text{EDD\_urgency}(j)
+w_2\cdot \mathbb{E}[W^{down}(j)]
+w_3\cdot \mathbb{E}[\text{SDST}(prev,j)]
+w_4\cdot \mathbb{E}[\text{Cycle}(j)]
+w_5\cdot \mathbb{I}(\text{job causes blocking risk})
\]

Where:
- **EDD_urgency**: e.g., \(\max(0, \text{DueDate}-\text{Now}-\text{expected remaining processing})\) or expected lateness.
- **SDST(prev,j)** from mined sequence-dependent setup model conditioned on the last processed job on that machine.
- **W^{down}(j)**: expected downstream waiting/blocking due to constrained downstream machines (learned from historical stage-to-stage delays).
- **Cycle(j)**: expected queue + setup + processing at current machine using mined distributions.
- **Blocking risk**: derived from empirical probability that downstream station is the limiting one and that this job would be blocked.

**How process mining informs it**
- Use mined SDST transition model to compute \(\mathbb{E}[\text{SDST}(prev,j)]\).
- Use cohort analysis to learn:
  - which job classes are ôstarvedö or ôovertakenö
  - which routing variants experience most downstream blocking
- Learn/estimate transition probabilities and remaining lead-time distributions by job class.

**Addresses identified pathologies**
- Reduces expensive SDST transitions.
- Improves near-due job handling without globally chaotic priority preemption.
- Mitigates starvation/blocking by accounting for downstream constraints.

**Expected impact**
- Lower tardiness (especially p90/p95).
- Reduced WIP variability (less queue oscillation).
- More stable machine utilization (less setup-driven cycle time explosion).

---

### Strategy 2 Ś Predictive Scheduling with probabilistic duration + bottleneck risk forecasting
**Core logic**
Build a rolling-horizon decision engine that uses learned distributions to predict:
- probability each candidate job will miss due date given current shop state
- probability of bottleneck saturation in near future
- expected completion times with uncertainty

A practical implementation:
1. **Remaining-time prediction model** per job class and routing:
   - Monte Carlo simulation using mined distributions for:
     - processing times
     - setup times (SDST-aware using last-job transition state)
     - breakdown downtime sampled from observed frequencies
     - rework/interruptions if present
2. Define objective for dispatch decisions:
   - minimize expected weighted tardiness:
     \[
     \min \sum_j \mathbb{E}[\alpha_j\cdot \max(0,C_j-D_j)]
     \]
3. Use **state-aware rolling rescheduling**:
   - periodically re-evaluate job scores when disruptions occur or when uncertainty grows.

**How process mining informs it**
- Extract empirical distributions by task type, machine, operator, and job attributes.
- Mine breakdown hazard rates (frequency) and downtime distributions.
- Mine probability of queue growth given current load regimes.

**Addresses identified pathologies**
- Improves lead-time estimates (uncertainty-aware).
- Prevents late-job creation by dispatching based on *expected* lateness rather than deterministic due-date ordering.
- Makes bottleneck risk explicit so scheduling reacts before queues blow up.

**Expected impact**
- Better due-date adherence (tail reduction).
- Reduced ôsurpriseö lead times.
- More robust schedules under disruption.

---

### Strategy 3 Ś Explicit Setup-Time Optimization via ôsequence planningö at bottleneck machines
**Core logic**
Treat setup minimization as a first-class planning layer at machines where SDST dominates (often the bottleneck).
A two-level approach:

**Level A: Sequence grouping (batching/ôcampaignö formation)**
- Cluster jobs into setup families using job attributes that drive setup changes (material, tolerance class, program/tool families).
- At a bottleneck machine, decide a short-term ôcampaignö sequence:
  - pick next campaign family that minimizes:
    - expected setup cost from current family
    - weighted tardiness impact of delaying other families
- Use a lookahead window of size \(L\) jobs or time horizon \(H\).

**Level B: Within-family scheduling**
- Process jobs within the same family using a tardiness-oriented rule (EDD/expected lateness), minimizing additional setups.

**How process mining informs it**
- From event logs, learn:
  - SDST transition costs between families
  - family composition effects on setup duration
  - trade-off between delaying a family and avoiding expensive transitions
- Mine whether setup reductions historically hurt due-date performance (to tune weights and avoid over-batching).

**Addresses identified pathologies**
- Directly targets ôsuboptimal sequencing increases total setup timesö.
- Reduces cycle-time variance caused by repeated high-cost transitions.
- Helps stabilize WIP.

**Expected impact**
- Lower setup share and fewer expensive transitions.
- Lower queue time variability  improved tardiness stability.
- Higher effective throughput at bottlenecks.

## 5) Simulation, Evaluation, and Continuous Improvement

### 5.1 Build a discrete-event simulation (DES) using process-mined parameters
**Why simulation is essential**
Scheduling changes interact with:
- SDST
- breakdowns
- routing variability
- queueing/blocking dynamics

**What to parameterize from logs**
For each machine/task:
- processing time distribution (by task type, machine, job family, operator)
- SDST model: expected setup time and variability conditioned on last job on machine
- breakdown model:
  - breakdown arrival rate (or hazard)
  - downtime distribution
- routing:
  - routing variants and transition probabilities if uncertain/rework exists
- priority change/hot-job arrival process:
  - model insertion times and urgency distributions from history

**Scenarios to test**
1. **Baseline replication** (sanity check):
   - reproduce observed KPIs with current dispatching approach.
2. **High load / peak demand**
   - stress queues and observe tail risks.
3. **Frequent disruptions**
   - increase breakdown frequency; generate multiple breakdowns in horizon.
4. **Hot job bursts**
   - simulate priority changes and new job arrivals at realistic rates.
5. **Due-date tightness regimes**
   - change due date distributions to test robustness.
6. **What-if sequencing windows**
   - vary lookahead L/H for strategy 1 and 3 to identify best trade-offs.

**Evaluation metrics (KPIs)**
- tardiness: mean, median, p90/p95; # late jobs
- WIP: average WIP, WIP variance, maximum WIP peaks
- lead time: median and tail; prediction error if strategy outputs ETAs
- machine utilization breakdown: productive vs idle vs setup vs downtime
- setup cost: mean setup time and costly transition frequency
- resilience: KPI degradation under disruptions

---

### 5.2 Continuous monitoring and adaptive improvement using ongoing process mining
**Operational loop**
1. **Daily/weekly replay mining**
   - recompute distributions for:
     - processing/setup times
     - breakdown rates
     - queue times and bottleneck status
2. **Drift detection**
   - detect changes in:
     - planned vs actual gaps
     - SDST transition patterns
     - breakdown frequency/severity
     - routing variant proportions
   - Use statistical drift tests (e.g., KL divergence for distributions, population stability index).
3. **Automated model refresh**
   - retrain SDST model and duration distributions when drift passes thresholds.
4. **Policy parameter tuning**
   - strategy weights (w1ůw5) updated using:
     - simulation-based optimization (e.g., Bayesian optimization on simulation surrogate)
     - or offline ôreplayö evaluation using historical state/decision logs.
5. **Exception analytics**
   - generate alerts when:
     - urgency inversions exceed threshold
     - bottleneck queue time distribution shifts
     - setup cost increases without utilization improvement

**Key idea**
Keep a living ôshop state modelö updated from MES: schedules should continuously re-align with reality.
