As a Process Mining Consultant, I approach the challenges at Speedy Parcels by treating their logistics network as a complex, highly variable business process. By integrating multi-source event data, we can move beyond intuition and expose the exact reality of last-mile delivery. Below is a comprehensive, five-part approach to analyzing and optimizing Speedy Parcels’ operations.

---

### 1. Process Discovery and Conformance Checking

**Data Preprocessing and Integration**
To create a cohesive event log, we must resolve the "multi-perspective" nature of the data (GPS, scanner, dispatch, maintenance) into a unified format: `(Case ID, Activity, Timestamp, Resource, Attributes)`.
*   **Case ID Definition:** We will use a dual-layer approach. The *Macro Case ID* is `Vehicle-Day` (e.g., V12-20241205) to analyze fleet routing. The *Micro Case ID* is `Package ID` to analyze the customer delivery lifecycle.
*   **Integration Logic:** We will fuse data using `Vehicle ID`, `Driver ID`, and time-based interpolation. For example, if a scanner logs "Arrive Customer" at 08:25:10, we enrich this event with the GPS coordinates from the nearest timestamp (e.g., 08:25:08) and the dispatch data for the planned stop.
*   **Challenges:** 
    *   *Clock Drift:* Different systems (GPS vs. handheld) may have unsynchronized clocks, requiring time-alignment algorithms.
    *   *High-Frequency Noise:* GPS logs every few seconds; this must be aggregated into meaningful states (e.g., grouping consecutive "Moving at 50km/h" points into a single "Traveling" activity) to prevent process maps from becoming unreadable.
    *   *Sparse Data:* GPS signals drop in tunnels or underground parking, creating logical gaps (e.g., Depart Depot -> magically Arrive Customer) that must be handled via interpolation or gap-annotation.

**Process Discovery**
Using an algorithm like the **Inductive Miner** (which is highly robust to noise, a certainty in GPS data), we will discover the end-to-end process. The resulting map will visualize the "Happy Path" (Depart Depot $\rightarrow$ Travel $\rightarrow$ Arrive $\rightarrow$ Success $\rightarrow$ Depart $\rightarrow$ Arrive Depot) alongside alternative paths like loops for "Failed Deliveries" or unexpected "Unscheduled Stops" (e.g., Engine Warning). 

**Conformance Checking**
We will use **Alignment-based Conformance Checking** to compare the actual event log (GPS/Scanner) against the planned model (Dispatch System). This calculates the minimal cost to "edit" the actual trace to match the planned trace. 
*   *Sequence Deviations:* Did the driver visit Stop 5 before Stop 3? 
*   *Unplanned Stops:* Are there GPS traces showing 0 speed and ignition on for $>5$ minutes where no scanner event occurred? (e.g., unauthorized breaks, searching for parking).
*   *Timing Deviations:* Did a planned 10-minute dwell time at a customer location stretch to 45 minutes?

### 2. Performance Analysis and Bottleneck Identification

**Key Performance Indicators (KPIs) Calculation**
*   **On-Time Delivery Rate:** `Count(Package Events where 'Delivery Success' Timestamp $\le$ Dispatch Time Window End)` / `Total Planned Deliveries`.
*   **Average Service Time per Stop:** `Timestamp('Depart Customer') - Timestamp('Arrive Customer')`. Isolated from travel time.
*   **Travel vs. Service Time Ratio:** `Total Duration where GPS Speed > 5km/h` / `Total Duration where GPS Speed = 0 and Ignition = On`.
*   **Proxy Fuel Consumption per km:** Calculated using `Distance (Haversine formula on GPS coordinates)` multiplied by a weight factor for `Duration where GPS Speed = 0 and Ignition = On` (idling burns fuel).
*   **Failed Delivery Rate:** `Count('Delivery Failed')` / `Total Delivery Attempts`.

**Bottleneck Identification Techniques**
*   **Animated Process Maps (Geographic):** By mapping the flow of the `Vehicle-Day` cases on a geographic information system (GIS) over time, we can physically *see* parcels piling up at specific intersections or depot loading bays.
*   **Time-Based Bottleneck Analysis:** We will analyze the "waiting time" between `Arrive Customer` and `Delivery Success`. If this is consistently high for specific apartment complexes, the bottleneck is "finding parking/access," not the delivery itself.
*   **Contextual Performance Dashboards:** Overlaying bottlenecks with data attributes. For example, grouping travel times by `Time of Day` might reveal that the bottleneck isn't a specific *route*, but a specific *time* (e.g., school zones between 8:00-9:00 AM). We quantify impact by calculating the delta between ideal travel time (free-flow speed) and actual travel time.

### 3. Root Cause Analysis for Inefficiencies

Knowing *where* the bottleneck is (e.g., long dwell times) is insufficient; we must know *why*. We will use **Decision Point Analysis** and **Trace Clustering** to isolate root causes:

*   **Suboptimal Route Planning (Static vs. Dynamic):** If conformance checking shows drivers frequently deviating from the dispatch sequence, we cluster the deviating traces. We may find that drivers deviate specifically to avoid a construction zone that the static dispatch software is unaware of.
*   **Inaccurate Travel Time Estimations:** By comparing Dispatch's `Planned Arrival` vs. Actual `Arrive Customer`, we can perform a root cause analysis on the estimation error. We might find that the dispatch system assumes an average speed of 30 km/h, but GPS data proves the actual average in suburban zones is 18 km/h due to speed bumps and stop signs.
*   **Impact of Traffic Congestion:** Correlating external traffic API data (if available) or GPS-derived "Low Speed Detected" events with delays. 
*   **High Variability in Service Time:** Using a decision tree within the process mining tool, we can determine what attributes most influence long dwell times. Root cause: "If Location Type = Gated Community AND Time of Day = After 5 PM, average dwell time increases by 15 minutes due to security check-in queues."
*   **Vehicle Breakdowns:** Analyzing traces containing `Unscheduled Stop` + `Engine Warning`. We can trace back to see if these vehicles had prolonged periods of "Low Speed Detected" (indicating engine strain) in the days prior.
*   **Driver Behavior Differences:** By clustering traces by `Driver ID` on the *exact same planned route*, we eliminate route complexity as a variable. If Driver A takes 6 hours and Driver B takes 8 hours on the same route, we can deep-dive into their GPS idle times (breaks) or travel speeds to identify behavioral root causes.
