# Optimizing E-commerce Fulfillment: A Constraint-Aware Process Mining Strategy

## 1. Identifying Instance-Spanning Constraints and Their Impact

Instance-spanning constraints (ISCs) cannot be detected by analyzing cases in isolation. They require a **cross-case, time-window-based analysis** of the event log. My approach would leverage the log's start/complete timestamps to reconstruct the state of the system at any point in time.

### 1.1 Detection Methodology per Constraint

**a) Shared Cold-Packing Stations (Resource Contention)**
- **Technique:** Build a *resource utilization timeline* for each of the 5 Cold-Packing stations. For every order where `Requires Cold Packing = TRUE`, compute the gap between `Packing START` and the `COMPLETE` of the preceding activity (Item Picking).
- **Detection logic:** For each cold-packing waiting period, count the number of *other* orders simultaneously occupying cold-packing stations. If all 5 were busy during the waiting interval, the delay is attributable to the ISC.
- **Metrics:**
  - *Cold-Pack Queue Waiting Time* = time between "ready for packing" and "packing start," conditional on all 5 stations being busy.
  - *Cold-Pack Utilization %* and *peak simultaneous demand*.
  - *Contention Frequency* = % of cold-pack orders delayed by saturation.

**b) Shipping Batches**
- **Technique:** Group cases by `Destination Region` and `Shipping Label Gen.` timestamp. Cases sharing the same batch ID (or clustered label-generation timestamps) form a batch.
- **Detection logic:** For each order, compute *Batch Waiting Time* = `Shipping Label Gen. COMPLETE`  `Quality Check COMPLETE`. Compare this to the batch's last-arriving order — earlier arrivals in a batch will show the largest wait.
- **Metrics:**
  - *Average Batch Wait Time* per region.
  - *Batch Size Distribution* and *time-to-fill-batch*.
  - *Batch-induced idle time* (orders QC-complete but waiting).

**c) Priority Order Handling (Preemption)**
- **Technique:** Identify interruptions by detecting non-contiguous START/COMPLETE pairs for the same (case, activity, resource) where an Express order's activity on the same resource occurs in between.
- **Detection logic:** A standard order's Packing START at t1, then paused (gap), while an Express order uses the same resource — classic preemption signature.
- **Metrics:**
  - *Preemption Count* per standard order.
  - *Preemption-induced delay* = sum of pause durations.
  - *Express turnaround time vs. Standard turnaround time* (comparative KPI).

**d) Hazardous Material Limits (Regulatory Cap)**
- **Technique:** Sliding-time-window analysis. At every event, count the number of hazardous orders currently in Packing OR Quality Check (using START/COMPLETE semantics).
- **Detection logic:** When the count hits 10, subsequent hazardous orders "ready to pack" must wait. Measure time between "ready" and "start" conditional on this saturation.
- **Metrics:**
  - *Time at regulatory ceiling* (% of time concurrent haz count = 10).
  - *Hazmat queue wait*, *throughput loss* vs. theoretical capacity.

### 1.2 Separating Within-Instance vs. Between-Instance Waiting Time

This is the analytical crux. For each transition between activities I decompose waiting time as:

```
Total Wait = Service Wait + Resource Contention Wait + Batch Wait + Regulatory Wait + Preemption Wait
```

- **Within-instance waiting** = delays driven only by the case's own state (e.g., long activity processing, missing info on the order itself).
- **Between-instance waiting** = delays caused when, at the moment the case was "ready," another case was:
  - holding the required resource (contention),
  - still being awaited to complete a batch,
  - occupying the regulatory slot, or
  - preempting via priority.

**Decomposition approach:** For every waiting interval, query the system state at start-of-wait. If all compatible resources are free, no hazmat saturation, no pending batch partners, and no express preemption  within-instance. Otherwise, attribute the wait to the dominant between-instance cause (or apportion if multiple). This yields a **constraint-attributed waiting time matrix** — the foundation for everything that follows.

## 2. Analyzing Constraint Interactions

Constraints rarely act independently; their **compounding effects** often produce the worst bottlenecks.

**Interaction A: Express + Cold-Packing**
An express order requiring cold packing will preempt a standard cold-pack order. Since only 5 cold-pack stations exist, the preempted order goes back into the queue for a *constrained* resource, potentially being leapfrogged again. This creates **starvation** for standard cold-pack orders during express surges.

**Interaction B: Batching + Hazmat Limits**
If many hazardous orders are destined for the same region, batching encourages them to be processed near-simultaneously (to catch the same shipping batch). This pushes against the 10-order hazmat cap, causing some to wait — which *then* causes them to miss the batch, cascading into a longer end-to-end delay.

**Interaction C: Express + Batching**
Express orders by design *shouldn't wait in batches*, but if routed into standard batching logic, the priority investment is wasted downstream. Conversely, forcing single-order shipping for every express order hurts logistics efficiency.

**Interaction D: Hazmat Limit + Cold-Packing**
A hazmat order that also needs cold packing consumes *two* scarce resource "tokens." During peak periods, these orders become systemic choke points.

**Why interactions matter:** Optimizing one constraint in isolation can worsen another. For example, increasing cold-pack throughput without adjusting hazmat scheduling may simply shift the queue downstream to the QC stage where the hazmat cap binds.

## 3. Constraint-Aware Optimization Strategies

### Strategy 1: Predictive Dynamic Scheduling for Cold-Packing with Priority-Aware Slotting

- **Constraints addressed:** Cold-Packing contention + Express priority (Interaction A).
- **Proposal:** Replace FIFO queueing at cold-pack stations with a scheduler that (a) forecasts incoming express cold-pack demand using the preceding hour's Item Picking completions and (b) reserves 1 of the 5 stations as an "Express fast lane" during predicted surge windows, releasing it to standard use otherwise.
- **Data leverage:** Train a short-horizon forecasting model on historical log arrivals by order type. Use queuing analysis from Section 1 to calibrate the fast-lane activation threshold.
- **Expected outcome:** Reduced express preemption of standard orders (fewer paused standard orders), lower average cold-pack waiting time for express, and minimal impact on standard throughput because the reserved slot reverts when not needed.

### Strategy 2: Dynamic, Deadline-Driven Batch Formation

- **Constraints addressed:** Shipping batch waiting + Express priority (Interaction C).
- **Proposal:** Move from "fill-then-ship" batching to a **hybrid trigger**: a batch ships when *either* (a) it reaches target size, *or* (b) the oldest order's cumulative wait exceeds a region-specific SLA threshold. Express orders trigger immediate label generation and are attached to the next available carrier pickup rather than waiting for batch formation.
- **Data leverage:** Analyze historical batch-fill times per region to derive the optimal SLA threshold where batching savings  delay cost. Simulate different size/time thresholds against historical demand.
- **Expected outcome:** Drops per-order batch wait significantly for low-volume regions; preserves logistical efficiency for high-volume ones; restores express value proposition.

### Strategy 3: Hazmat-Aware Work-In-Progress (WIP) Controller

- **Constraints addressed:** Hazmat regulatory cap + Batching (Interaction B).
- **Proposal:** Introduce a facility-level dispatcher that tracks the current hazmat WIP count in Packing+QC. When WIP reaches 8 (soft threshold), it (a) defers releasing new hazmat orders from Item Picking unless they are express, and (b) prioritizes the QC completion of hazmat orders already in-flight — effectively a pull-based CONWIP rule applied only to hazmat orders. Additionally, cluster hazmat orders bound for the same region into mini-batches that enter Packing in a staggered "pulsed" pattern matching the cap.
- **Data leverage:** Use historical distributions of concurrent hazmat count from Section 1 metrics to set the soft threshold; use regional hazmat demand patterns to time the pulses.
- **Expected outcome:** Near-zero regulatory breaches, reduced variance in hazmat throughput, and better alignment between hazmat readiness and regional shipping batches (mitigating Interaction B).

*(Additional candidate: cross-training packing staff so cold-pack stations can be flexibly configured during surges — a capacity-side lever.)*

## 4. Simulation and Validation

Before deployment, I would build a **Discrete-Event Simulation (DES)** model calibrated directly from the mined process model and log statistics (activity duration distributions, arrival patterns, routing probabilities, order-attribute mixes).

### Critical Modeling Requirements to Preserve ISC Fidelity

- **Resource pools with capacity:** Model the 5 cold-pack stations as a finite capacity pool with queueing, not as infinite resources.
- **Global counters for regulatory limits:** Implement a shared semaphore/counter that spans all Packing and QC activity instances — hazmat orders increment on start, decrement on QC complete, and cannot start Packing when counter = 10.
- **Batch entities:** Model batches as first-class objects with formation triggers (size, timeout). Orders join a batch entity and are released when the batch's condition fires.
- **Preemption logic:** Model express arrivals as events that can interrupt standard activity; track pause/resume semantics so throughput time reflects preemption correctly.
- **Stochastic attribute assignment:** Sample order type, hazmat flag, cold-pack flag, and destination from empirical joint distributions — not independently — so correlations (e.g., "express AND cold-pack") occur at realistic rates.

### Validation Approach

- **Baseline replay:** Simulate the *as-is* process on historical arrivals and verify simulated KPIs (waiting times, utilization, throughput) match the event log within tolerance. This validates model fidelity.
- **Counterfactual experiments:** For each strategy, run N replications under varying demand intensities (normal, peak, extreme peak). Compare KPIs: end-to-end cycle time (P50/P90), express SLA compliance, hazmat cap breach probability, cold-pack utilization, batch wait time.
- **Interaction testing:** Specifically stress-test scenarios designed around Interactions A–D to ensure no strategy worsens a different constraint.
- **Sensitivity analysis:** Vary policy parameters (fast-lane threshold, batch SLA, hazmat soft cap) to find robust operating points.

## 5. Post-Implementation Monitoring

I would deploy a **real-time process mining dashboard** (e.g., built on Celonis, Disco, or PM4Py + BI) with layered views.

### Key Metrics to Track

**Flow KPIs**
- End-to-end order cycle time (overall, by order type, by region).
- Throughput (orders/hour) and WIP.
- Express SLA compliance rate.

**Constraint-Specific KPIs**
- *Cold-Packing:* average and P90 queue wait, station utilization, simultaneous-demand distribution, fast-lane activation time.
- *Batching:* average batch wait by region, batch fill rate, batches shipped by size-trigger vs. SLA-trigger.
- *Priority:* number of preemptions/day, average preemption delay inflicted on standard orders.
- *Hazmat:* concurrent-hazmat-count time series with the 10-limit overlay, number of near-breaches (count  9), hazmat queue wait.

**Interaction KPIs**
- Cycle time for "express + cold-pack" orders (Interaction A health).
- Hazmat orders missing their intended region batch (Interaction B health).

### Dashboards

1. **Operational live view:** Current WIP, active resources, real-time hazmat counter (with alert when  9), cold-pack queue length, express orders in-flight.
2. **Constraint conformance view:** Conformance checking against the redesigned rules (e.g., "batch released when SLA threshold hit?"; "fast-lane activated when forecast exceeded threshold?").
3. **Trend view:** Weekly trends of all KPIs, with pre-/post-implementation comparison using statistical tests.
4. **Root-cause view:** The waiting-time decomposition from Section 1 maintained as a continuous metric, so any regression is immediately attributable to the specific ISC that caused it.

### Continuous Improvement Loop

Alerts trigger when (a) any hazmat breach occurs (zero tolerance), (b) cold-pack queue wait exceeds a rolling baseline by 2, or (c) express cycle time approaches SLA. Each alert feeds back into model retraining (for the predictive scheduler) and threshold recalibration (for batching and hazmat WIP). This closes the loop between process mining discovery, strategy execution, and adaptive governance — ensuring the instance-spanning constraints remain *managed*, not merely *observed*.
