The user wants a DuckDB SQL query to filter an event log table (`event_log`) based on case length.
Specifically: Keep cases with $\le 6$ events, discard cases with $> 6$ events.
Columns: `case_id`, `activity`, `timestamp`.
Output: All columns for the qualifying events.

Standard SQL approach:
1. Aggregate to find case sizes.
2. Filter case_ids where count <= 6.
3. Join back to original table (or use window function / IN / EXISTS subquery).

DuckDB supports window functions, CTEs, and semi-joins efficiently.

Option 1: Window Function (often very fast in DuckDB due to vectorized execution)
```sql
SELECT case_id, activity, timestamp
FROM (
    SELECT *, COUNT(*) OVER (PARTITION BY case_id) as case_size
    FROM event_log
)
WHERE case_size <= 6;
```

Option 2: CTE + IN / JOIN
```sql
WITH valid_cases AS (
    SELECT case_id
    FROM event_log
    GROUP BY case_id
    HAVING COUNT(*) <= 6
)
SELECT e.*
FROM event_log e
JOIN valid_cases v ON e.case_id = v.case_id;
```
Or using `IN`:
```sql
SELECT *
FROM event_log
WHERE case_id IN (
    SELECT case_id
    FROM event_log
    GROUP BY case_id
    HAVING COUNT(*) <= 6
);
```

DuckDB optimizes `IN` subqueries to semi-joins (mark joins) usually, making it very efficient. The Window function approach avoids a join but requires a sort/partition step. For "filter groups by aggregate", the `QUALIFY` clause (supported in DuckDB) is the most idiomatic and concise modern SQL way.

DuckDB supports `QUALIFY`!
```sql
SELECT case_id, activity, timestamp
FROM event_log
QUALIFY COUNT(*) OVER (PARTITION BY case_id) <= 6;
```
This is the cleanest, most "DuckDB-native" way. It handles the window function filtering automatically without a subquery.

Here is the most idiomatic and performant DuckDB query using the `QUALIFY` clause, which is designed specifically for filtering based on window function results without requiring a subquery or CTE.

```sql
SELECT 
    case_id, 
    activity, 
    timestamp
FROM event_log
QUALIFY COUNT(*) OVER (PARTITION BY case_id) <= 6;
```

### Why this is the best approach for DuckDB:
1.  **`QUALIFY` Clause**: This is a modern SQL standard (supported by DuckDB, Snowflake, BigQuery) that acts like `WHERE` but for window functions. It filters the result of the window aggregation (`COUNT(*) OVER ...`) directly.
2.  **Performance**: DuckDB's vectorized engine executes the window function (partitioned count) highly efficiently. `QUALIFY` avoids the overhead of materializing a full subquery/CTE and joining it back to the original table.
3.  **Readability**: The logic reads top-to-bottom: "Select from log, *qualify* where the case size is $\le 6$."

---

### Alternative: Standard SQL (CTE + Join/IN)
If you prefer standard SQL compatible with all dialects (or find the logic clearer separated), this is the canonical pattern. DuckDB optimizes the `IN` subquery into a highly efficient **Mark Join (Semi-Join)** automatically.

```sql
WITH case_sizes AS (
    SELECT case_id
    FROM event_log
    GROUP BY case_id
    HAVING COUNT(*) <= 6
)
SELECT e.case_id, e.activity, e.timestamp
FROM event_log e
WHERE e.case_id IN (SELECT case_id FROM case_sizes);
-- Alternatively: JOIN case_sizes cs ON e.case_id = cs.case_id
```
