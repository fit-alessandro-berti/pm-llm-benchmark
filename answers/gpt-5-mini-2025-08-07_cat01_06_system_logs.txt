Brief explanation of grouping and naming logic

- Case identification:
  - I created one case per logical work item: each Word/Excel/ PDF file and the specific email thread. That yields five cases:
    - CASE_DOC1 (Document1.docx)
    - CASE_QTR_REPORT (Quarterly_Report.docx)
    - CASE_XLS_BUDGET (Budget_2024.xlsx)
    - CASE_PDF_REPORT (Report_Draft.pdf)
    - CASE_EMAIL_ANNUAL (Email about Annual Meeting)
  - Rationale: The user’s interaction pattern shows focused sessions per file/email (open/focus  edit/read/annotate  save/close/send). Grouping by window/document gives coherent sequences useful for process analysis.

- Assigning SWITCH events:
  - SWITCH events contain both source and target windows. For process logs I assigned the SWITCH event to the target context (i.e., the activity of switching/starting work on that target case). This makes the case traces show when the user began working on that item.

- Activity naming:
  - I mapped low-level actions to standardized, semantically meaningful activities:
    - FOCUS / SWITCH (to target)  Open Document / Open Spreadsheet / Open Email Inbox / Open PDF
    - TYPING  Edit Document / Edit Spreadsheet / Compose Reply (for email)
    - SAVE  Save Document / Save Spreadsheet
    - CLICK (open email)  Open Email (specific message)
    - SCROLL  Read Email / Read PDF
    - HIGHLIGHT  Annotate PDF
    - CLICK (send)  Send Email
    - CLOSE  Close Document
  - Repeated low-level events (multiple TYPING lines) are kept as separate events (they represent distinct user interactions with timestamps), but they share the same activity label "Edit ..." so process miners can aggregate or analyze durations.

- Additional attributes:
  - I preserved application and window/file names so analysts can filter or group by resource or artifact. The core required attributes (case id, activity name, timestamp) are present for each event.

Notes / alternatives
- Another plausible approach would be to treat the entire session as a single "work session" case and include document-level subcases; however, per-document cases are more actionable for process mining focused on document-centric tasks (edit  save  close) and for email-handling metrics (time-to-reply, number of edits, etc.).
