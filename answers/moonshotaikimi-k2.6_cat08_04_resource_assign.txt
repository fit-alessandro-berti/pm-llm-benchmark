### 1. Analyzing Resource Behavior and Assignment Patterns

To move from anecdotal complaints to evidence-based optimization, we first reconstruct the **actual** behavior of agents, tiers, and dispatchers directly from the event log.

#### A. Agent and Tier Performance Metrics
Using the event log, we calculate a multi-dimensional performance profile for every agent and tier:

*   **Workload Distribution & Utilization:** Measure the ratio of *active work time* (sum of durations between `Work L* Start` and `Work L* End`) to *available time* within shifts. Aggregate by agent, tier, and skill domain. High variance in these ratios indicates uneven distribution.
*   **Processing Time by Activity and Skill:** Compute mean, median, and standard deviation of handling times per (`Activity`, `Ticket Category`, `Required Skill`, `Agent Tier`) tuple. Control charts (e.g., X-bar/R charts) identify agents who are statistical outliers—either unusually slow (potential training gap) or unusually fast (potential quality/rush risk).
*   **First-Call Resolution (FCR) Rate:** For L1, FCR is the percentage of tickets where the agent performs `Work L1 Start/End` and the case reaches `Resolved` without ever containing an `Escalate L2/L3` or `Reassign` activity. A low FCR suggests either inadequate L1 empowerment or poor initial routing.
*   **Skill Affinity and Handling Frequency:** Cross-tabulate each agent’s documented skills against the `Required Skill` of tickets they actually handle. This reveals whether agents are working within their specialty or being used as generalists.

#### B. Process Mining Techniques for Assignment Discovery
The *intended* process is L1  (escalate if needed)  L2/L3. Process mining reveals the *real* socio-technical system:

*   **Resource Interaction & Handover Analysis:** By treating each `Resource` handover (e.g., `Agent A05`  `Dispatcher`  `Agent B12`) as a transition, we discover the actual escalation topology. We compare this against the formal tier structure to find "backdoor" escalations (L1 directly messaging L3) or dispatchers acting as persistent bottlenecks.
*   **Social Network Analysis (SNA):** We construct a directed graph where nodes are agents and edges represent ticket handovers (reassignments or escalations), weighted by frequency and annotated by average delay. Dense clusters may indicate silos of knowledge; a high *betweenness centrality* score for a specific L2 agent reveals an unofficial bottleneck. Unexpected edges (e.g., frequent L3  L1 returns) indicate "ping-pong" behavior.
*   **Role Discovery:** Using unsupervised clustering on actual event sequences (e.g., k-means or heuristic mining on resource-activity profiles), we discover *emergent roles*. For example, an L2 agent might behaviorally function as an "L1 overflow" resource, or an L1 agent might secretly be handling L2-level Security tickets. Comparing discovered roles against formal job descriptions exposes structural misalignment.

#### C. Skill Utilization Effectiveness
We build a **Skill Utilization Matrix**:
*   **Rows:** Agents (with their documented skills).
*   **Columns:** `Required Skill` from tickets.
*   **Cells:** Count/time of tickets handled.

A high volume in cells where agent skill  required skill signals **skill mismatch** (leading to reassignments). Conversely, if a specialist with rare skills (e.g., `Security-IAM`) spends >30% of their time on generic `Basic-Troubleshoot` tickets, this signals **skill underutilization** and a poor return on specialized labor.

### 2. Identifying Resource-Related Bottlenecks and Issues

With behavioral baselines established, we isolate specific pathologies and quantify their damage.

#### A. Skill Availability Bottlenecks
We measure the **Queue Time per Required Skill**: the interval between `Ticket Created`/`Escalate` and the subsequent `Work L* Start`. If tickets requiring `Networking-Firewall` consistently wait 4 hours before an L2 agent begins work—while other skills wait 45 minutes—we have identified a resource contention bottleneck. This is exacerbated if SNA shows only 1–2 agents handle 80% of that skill’s volume.

#### B. The Reassignment Tax
For every `Reassign` activity in a case, we calculate:
1.  **Direct Delay:** Time between `Reassign` and the next `Work L* Start`.
2.  **Rework Waste:** Time already spent by the previous agent before reassignment (e.g., `Agent B12` worked for 70 minutes before realizing a DB skill was needed).

**Quantification:** Using a regression or ANOVA model on case variants, we determine that *each reassignment adds, on average, X hours to resolution time* and increases SLA breach probability by Y%. For example, if cases with zero reassignments average 6 hours and cases with 2 reassignments average 18 hours, the "reassignment tax" is 12 hours per excess reassignment.

#### C. Incorrect Initial Assignment Impact
We filter for tickets reassigned **before work starts** or **within the first 15 minutes** of `Work L1 Start`. A high frequency here indicates dispatcher or L1 self-assignment errors. We correlate these "fast fails" with specific dispatchers, shifts, or ticket categories (e.g., "Software-App" tickets are often misrouted to DB agents).

#### D. Overload and Underutilization
Using the utilization metrics from Section 1, we flag:
*   **Overloaded:** Agents consistently >85% utilized with growing backlogs (visible in dotted charts as dense, unbroken timelines). These agents are high-risk for SLA breaches.
*   **Underutilized:** Agents <40% utilized, often handling far fewer tickets than the team median. This may indicate skill gaps, schedule mismatches, or territorial ticket hoarding.

#### E. SLA Correlation
We define SLA targets by `Ticket Priority` (e.g., P2 = 8 hours). We then stratify SLA breach cases by assignment pattern:
*   `% of P2/P3 breaches involving 1 reassignment`
*   `% of breaches where the first assigned agent lacked the Required Skill`
*   `% of breaches occurring when assigned to an agent already handling >N concurrent tickets`

This directly ties resource assignment logic to the business pain point.

### 3. Root Cause Analysis for Assignment Inefficiencies

Metrics tell us *what* is broken; root cause analysis tells us *why*.

#### A. Systemic and Procedural Root Causes
*   **Deficient Assignment Rules:** Pure round-robin ignores `Required Skill`, agent proficiency, and real-time queue depth. The log will show reassignments clustering immediately after round-robin assignments.
*   **Stale or Inflated Skill Profiles:** If `Agent B12` is documented with `App-CRM` but the log shows they reassign all `App-CRM` tickets to others, their profile is outdated. Conversely, if agents lack documented secondary skills they demonstrably possess, the routing system cannot leverage them.
*   **Poor Upstream Categorization:** If tickets logged as `Software-App` frequently require `Database-SQL` (as seen in INC-1001), the root cause is an inaccurate initial categorization or missing mandatory `Required Skill` field at ticket creation.
*   **Lack of Real-Time Visibility:** Dispatchers assign tickets to agents who appear available in a static roster but are actually deep in a complex P1 case (not visible in the assignment UI).
*   **L1 Defensive Escalation:** L1 agents, measured on speed rather than resolution, may escalate tickets they are technically capable of solving to avoid risk. This is visible when certain L1 agents have near-zero FCR rates for ticket types that other L1 agents resolve successfully.

#### B. Variant and Decision Mining
*   **Variant Analysis:** We compare the "happy path" variant (e.g., `Create  Assign L1  Work L1  Resolved`) against the "reassignment spiral" variant (`Create  Assign L1  Work L1  Escalate  Assign L2  Reassign  Assign L2...`). By comparing attributes between these variant groups, we may find that the latter is 5x more likely when `Required Skill = Database-SQL` and `Priority = P3`, suggesting a structural gap in L1 capabilities for that combination.
*   **Decision Mining:** We train a decision tree using log attributes at the moment of assignment (`Ticket Category`, `Required Skill`, `Dispatcher ID`, `Time of Day`, `Agent Skills`, `Agent Current Load`) to predict the binary outcome: "Will this ticket be reassigned?" Decision mining might reveal rules such as: *"If Required Skill is NOT in Agent's skill set AND Priority = P2, then reassignment probability = 92%."* This proves that ignoring skill data in the routing rule is the primary driver of failure.

### 4. Developing Data-Driven Resource Assignment Strategies

Based on the mined evidence, we propose four distinct optimization strategies.

#### Strategy 1: Proficiency-Weighted, Skill-Based Routing (SBR)
*   **Issue Addressed:** Skill mismatch, frequent reassignments, and specialists performing generic work.
*   **Leveraging Insights:** Process mining quantified that X% of reassignments occur when the assigned agent lacks the `Required Skill`. SNA identified which agents are the de facto owners of rare skills.
*   **Data Required:** 
    *   Agent skill profiles enhanced with **proficiency scores** (1–5), validated against historical resolution times and FCR rates by skill.
    *   Real-time `Required Skill` field (mandatory at ticket creation or predicted via NLP).
*   **Implementation:** Replace round-robin with a routing engine that matches `Required Skill` to agent skills. Weight the match by proficiency *and* current queue depth. If `App-CRM` is required, the ticket routes to the proficient `App-CRM` agent with the shortest effective queue (current queue + estimated handling time), not simply the next available agent.
*   **Expected Benefits:** 40–60% reduction in skill-based reassignments; decreased resolution time by routing directly to qualified resources; improved specialist utilization by protecting their capacity for matched work.

#### Strategy 2: Predictive Tier and Complexity Assignment
*   **Issue Addressed:** Excessive or premature L1 escalations, and delays in reaching the correct tier.
*   **Leveraging Insights:** Variant analysis showed that tickets with certain attributes (e.g., `Category = Network`, `Required Skill = Networking-Firewall`, or description keywords like "VPN tunnel down") are escalated from L1 95% of the time. Decision mining revealed these cases consume L1 handling time with zero value.
*   **Data Required:** 
    *   Historical event log including ticket descriptions/keywords, resolution paths, and actual time spent per tier.
    *   Trained ML classifier (e.g., Random Forest or XGBoost) using ticket attributes to predict the minimum tier required for resolution and the probability of L1 success.
*   **Implementation:** At `Ticket Created`, the model assigns a **Complexity Score**. High-complexity tickets bypass L1 entirely and route directly to L2/L3 (skill-matched). Medium-complexity tickets route to L1 but with an "expert assist" prompt or knowledge article. Low-complexity tickets stay in standard L1.
*   **Expected Benefits:** Reduced "touches" per ticket; faster time-to-specialist for complex issues; increased L1 FCR by filtering out cases they cannot handle, allowing them to focus on resolvable work.

#### Strategy 3: Dynamic, Workload-Aware Dispatching
*   **Issue Addressed:** Uneven workload distribution and SLA breaches caused by agent overload.
*   **Leveraging Insights:** Dotted charts and queue analysis showed that some agents accumulate backlogs while peers are idle, because round-robin ignores work-in-progress (WIP).
*   **Data Required:** 
    *   Real-time agent status and WIP (tickets currently in `Work L* Start` without a corresponding `End`).
    *   Estimated handling time per ticket type (mined from historical data per `Category` × `Priority` × `Required Skill`).
    *   Agent shift schedules and break patterns.
*   **Implementation:** The dispatcher algorithm calculates an **Expected Availability Time (EAT)** for each qualified agent: `EAT = Current Time + (Sum of estimated handling times for all open tickets)`. The ticket is assigned to the agent with the earliest EAT who also meets the skill requirement. For P1/P2 tickets, this is overridden to favor the agent with the absolute lowest current queue to guarantee immediate attention.
*   **Expected Benefits:** Balanced utilization across the agent pool (reduced Gini coefficient); reduced queue time for high-priority tickets; fewer SLA breaches linked to congestion.

#### Strategy 4: Data-Driven Escalation Guardrails and Dynamic Floating Pools
*   **Issue Addressed:** Unnecessary reassignments after work has started, and skill bottlenecks during demand spikes.
*   **Leveraging Insights:** Analysis revealed many reassignments happen *after* an agent has already invested significant effort (e.g., B12’s 70 minutes). Temporal analysis also showed demand spikes for specific skills (e.g., `OS-WindowsServer` patches on Tuesdays) that create temporary bottlenecks.
*   **Data Required:** 
    *   Reassignment reason codes (captured in `Notes` or via mandatory dropdown at reassignment).
    *   Agent secondary/adjacent skills.
    *   Real-time demand signals (tickets created in last hour by skill).
*   **Implementation:** 
    1.  **Hard Guardrails:** The assignment system blocks or warns against assigning a ticket to an agent who lacks the `Required Skill` (or a closely correlated skill with high historical success).
    2.  **Floating Pools:** When queue depth for a specific skill exceeds a threshold (mined from the 90th percentile of historical queue lengths), the system automatically pulls qualified agents from underutilized pools or temporarily authorizes cross-trained agents to accept those tickets.
*   **Expected Benefits:** Near-elimination of post-start reassignments due to skill mismatch; reduction in wasted effort; absorption of demand spikes without SLA breaches.
