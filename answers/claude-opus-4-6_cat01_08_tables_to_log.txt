

# Process Mining Event Log Derivation

## Assumptions & Design Decisions

1. **Case ID**: `order_id` is used as the case identifier.
2. **Item Picking**: Each line item pick is recorded as a separate event ("Item Picked") since they occur at different times and are performed by different resources. This preserves granularity for process mining.
3. **Multiple Deliveries**: Orders with multiple shipments (e.g., order 1002) generate separate dispatch and arrival events per delivery, distinguished by `delivery_id`.
4. **Multiple Invoices**: Similarly, each invoice generates its own "Invoice Issued" event.
5. **Payments**: Each payment is linked back to its order via the invoice. Each payment generates a separate "Payment Received" event.
6. **Order 1004**: Has no delivery, invoice, or payment records — the process appears incomplete. Only "Order Created" and "Item Picked" events are included.
7. **Resource names** are resolved from the Resources table wherever possible.
8. **Chronological ordering** is maintained within each case.

### Key Observations for Process Mining

- **Order 1004** appears stalled after item picking — no delivery, invoicing, or payment recorded. This could indicate a blocked or cancelled order requiring further investigation.
- **Order 1002** exhibits a split-shipment pattern (two deliveries, two invoices, two payments), which is characteristic of backorder handling in wholesale/bulk orders.
- **Payment events** have no explicit resource/actor in the source data (the Payments table lacks a responsible person), so the Resource field is left blank ("—") for those events. If a system or department is responsible, it could be added.
- All timestamps maintain logical chronological order within each case: **Order Created  Item Picked  Order Dispatched  Order Arrived  Invoice Issued  Payment Received**.
