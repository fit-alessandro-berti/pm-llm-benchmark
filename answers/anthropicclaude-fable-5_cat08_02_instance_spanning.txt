# Optimizing Fulfillment Operations Under Instance-Spanning Constraints
## A Process Mining-Driven Strategy

---

## Section 1: Identifying Instance-Spanning Constraints and Their Impact

### 1.1 General Approach

Classical process mining treats cases as independent — but here, the bottlenecks live *between* cases. My approach is to augment standard case-centric analysis with **resource-centric, time-slice, and cross-case correlation analyses**. The event log already contains the necessary attributes (START/COMPLETE timestamps, resource IDs, Order Type, Region, Cold-Packing flag, Hazmat flag), which makes this feasible.

**Foundation step — log preparation:**
- Pair START/COMPLETE events to derive **activity service times** per case.
- Derive **waiting times** as the gap between completion of activity *n* and start of activity *n+1*.
- Construct a **resource occupancy timeline**: for every resource (e.g., Station C1–C5), build an interval log of which case occupied it and when.
- Construct **facility-level concurrency timelines**: at any point in time, how many cases are in each activity state, segmented by attributes (Hazmat = TRUE, etc.).

### 1.2 Constraint-Specific Detection and Quantification

**(a) Shared Cold-Packing Stations**

*Detection:* Filter cases with `Requires Cold Packing = TRUE`. Build the occupancy timeline for stations C1–C5. For each cold-packing case, compute the wait between "Quality-ready for Packing" (completion of Item Picking) and Packing START. Crucially, check whether **all 5 stations were occupied during that waiting interval** — if yes, the wait is attributable to resource contention, not, e.g., staff unavailability.

*Metrics:*
- **Contention-attributable waiting time** = portion of waiting time overlapping with full station utilization (all 5 busy).
- **Cold-station utilization rate** over time (hourly/daily heatmaps; peak vs. off-peak).
- **Queue length distribution** for cold stations (number of cold orders simultaneously waiting).
- **Blocking probability**: P(arriving cold order finds all stations busy), by time-of-day and season.

**(b) Shipping Batches**

*Detection:* Group cases by batch ID embedded in the resource field (e.g., "System (Batch B1)") or, if absent, infer batches by clustering Shipping Label Generation events that occur at (near-)identical timestamps for the same region. For each case, compute **batch-induced wait** = time between Quality Check COMPLETE and Shipping Label Generation, identifying the case as "first ready" vs. "last ready" in its batch.

*Metrics:*
- **Batch synchronization delay** per case = (batch release time)  (case ready time). The *last* order in a batch has ~zero delay; early finishers absorb the cost.
- **Batch size distribution** and **batch formation time** (first-ready to last-ready) per region.
- **Marginal delay analysis**: how much waiting does each additional order in a batch impose on the already-completed orders?
- Regional asymmetry: low-volume regions will show long batch waits (few orders  slow batch fill).

**(c) Priority Handling (Express Preemption)**

*Detection:* Look for **interruption signatures**: a standard order whose Packing activity at station X is "paused" (e.g., abnormally long Packing duration or a START–SUSPEND–RESUME pattern if recorded), coinciding with an Express order starting at the same station/resource. If suspend/resume events aren't logged, detect this statistically: standard orders whose activity duration significantly exceeds the norm *and* whose execution window overlaps with an express order on the same resource.

*Metrics:*
- **Preemption frequency** (interruptions per day/week, per station).
- **Preemption penalty** = excess duration of interrupted standard activities vs. uninterrupted baseline (including any rework/setup cost when resuming).
- **Express order benefit** = express waiting time with preemption vs. counterfactual queue position.
- **Knock-on delay**: downstream lateness of preempted standard orders (did preempted orders miss their delivery SLA more often?).

**(d) Hazardous Material Limits (max 10 concurrent in Packing + QC)**

*Detection:* Build a time series of `count(Hazmat orders concurrently in Packing or QC)`. Identify periods where the count = 10 (the cap is binding). Cross-reference: hazmat orders that completed Picking but show delayed Packing START *during cap-binding periods* are constrained by the regulation, not by resource availability.

*Metrics:*
- **Cap-binding time share**: % of operating hours where the limit of 10 was reached.
- **Regulation-attributable wait**: waiting time of hazmat orders during cap-binding intervals.
- **Effective hazmat throughput** during binding vs. non-binding periods.
- **Cap utilization profile**: distribution of concurrent hazmat count (if it rarely exceeds 6–7, the constraint is non-binding; if it sits at 10 daily, it's a structural bottleneck).

### 1.3 Differentiating Within-Instance vs. Between-Instance Waiting

This is the analytical core. I would decompose total case lead time as:

**Lead time = (service times) + (waiting times)**, and then partition each waiting interval using the **counterfactual availability test**:

| Waiting interval condition | Attribution |
|---|---|
| A qualified resource was idle during the wait, no batch/cap applied | Within-instance / organizational (e.g., handover, shift gap) |
| All eligible resources busy with *other cases* | **Between-instance: resource contention** |
| Case ready, batch not yet released | **Between-instance: batch synchronization** |
| Activity duration exceeded baseline, overlapping express activity on same resource | **Between-instance: preemption** |
| Hazmat case ready, concurrent hazmat count = 10 | **Between-instance: regulatory cap** |
| Long activity duration with no interruption | Within-instance (genuine processing time, complexity) |

This yields a **waiting-time attribution model** per case, which I'd aggregate into a Pareto chart: "X% of total lead-time waste is caused by cold-station contention, Y% by batching, Z% by hazmat cap..." This Pareto directly prioritizes the optimization portfolio in Section 3. Performance spectrum analysis and dotted charts (cases ordered by arrival, colored by attribution category) make the patterns visually explicit and reveal temporal clustering (e.g., peak-season contention).

## Section 2: Analyzing Constraint Interactions

Instance-spanning constraints rarely act in isolation; their interactions can amplify or mask each other.

### 2.1 Key Interaction Patterns

**Express × Cold-Packing (priority × scarce resource):** ORD-5002 in the snippet is exactly this case — Express *and* cold-packing. When express orders preempt at one of only 5 cold stations, the effect on standard cold orders is much harsher than at abundant standard stations: a preemption at a fully-utilized 5-station pool cascades — the preempted order rejoins a queue that may itself be blocked. Frequent express-cold arrivals can effectively shrink the cold pool's standard-order capacity to 3–4 stations during peaks. *Analysis:* measure standard cold-order waiting conditional on express cold-order arrival rate.

**Batching × Hazmat cap:** If several hazmat orders share a destination region, batching incentives push them to be processed close together in time — pushing the concurrent hazmat count toward the cap of 10. The cap then throttles them, delaying batch completion, which in turn delays *non-hazmat* orders waiting in the same batch. This is a **contagion effect**: a regulatory constraint on a subset of orders propagates delay to unconstrained orders via the batching mechanism. *Analysis:* compare batch formation times for batches containing 1 hazmat order vs. hazmat-free batches.

**Hazmat cap × Express priority:** If an express hazmat order arrives while the cap is binding, does it preempt? If yes, it pushes a standard hazmat order *out* (or delays it), inflating standard hazmat lead times. If no, express SLAs are violated. The current policy here is likely undefined — a discovery question for the log.

**Cold-Packing × Batching:** A region's batch can be held hostage by a single cold-packing order stuck in the cold-station queue. The batch wait of N1 completed standard orders is driven by contention experienced by one order — multiplying the effective cost of cold-station scarcity well beyond the cold orders themselves.

**Express × Batching:** Are express orders exempt from batching? If express orders are batched with standard orders, the entire express benefit gained upstream (via preemption) is destroyed at the batching step — a classic case of local optimization with no end-to-end gain. The log can verify this: do express orders show batch waits?

### 2.2 Why Interaction Understanding Is Crucial

1. **Avoiding bottleneck shifting:** Adding cold stations might simply move the queue to the hazmat cap or the batch wait. Optimization must target the *system*, not individual constraints.
2. **Correct attribution:** A naive analysis blames "batching" for delays whose root cause is cold-station contention upstream. Interaction analysis ensures we fix causes, not symptoms.
3. **Non-linear effects:** Constraint interactions create non-linear performance degradation in peak season (queue theory: utilization above ~85% makes waits explode, and coupled queues amplify this). Strategies must be evaluated under interaction-aware load scenarios, not averages.
4. **Policy consistency:** Express handling must be defined consistently across *all* constraints (preemption rules at cold stations, exemption from batching, treatment under hazmat cap), otherwise effort upstream is wasted downstream.

## Section 3: Constraint-Aware Optimization Strategies

### Strategy 1: Predictive Slot Reservation & Dynamic Allocation for Cold-Packing Stations

**Addresses:** Cold-station contention; Express × Cold interaction.

**Proposed changes:**
- Implement a **reservation-based scheduler** for the 5 cold stations. When an order with `Requires Cold Packing = TRUE` enters Item Picking, the system predicts its Packing-ready time (using a regression model trained on historical picking durations by item count, picker, time of day) and **reserves a cold-station slot**.
- Reserve **one station as a flexible "express lane"** during predicted express-heavy windows (mined from arrival patterns), eliminating preemption: express cold orders go to the reserved slot rather than interrupting in-flight standard work — removing the preemption penalty (setup/rework cost) entirely.
- Dynamically release the express reservation when forecast express demand is low (off-peak), restoring full 5-station capacity for standard orders.

**Data leverage:** Arrival-rate forecasting by hour/day/season from the 3-month log; service-time prediction models; mined preemption-penalty quantification justifies the trade-off (reserving capacity is worth it if penalty × frequency > idle cost of the reserved station).

**Expected outcomes:** Reduced contention-attributable waiting for cold orders; elimination of preemption penalties (which destroy capacity — every interruption wastes setup/teardown time); more predictable cold-order flow, which also stabilizes downstream batch formation (addressing the Cold × Batching interaction).

### Strategy 2: Dynamic, Constraint-Aware Batch Release Logic

**Addresses:** Batch synchronization delays; Batching × Hazmat and Batching × Cold interactions; Express × Batching.

**Proposed changes:**
- Replace static batching ("wait until N orders for region R") with a **dual-trigger policy**: release the batch when (a) target size N* is reached, OR (b) a maximum holding time T* expires for the earliest-ready order, OR (c) a predictive trigger fires — if the model forecasts that the next eligible order for region R won't be ready for > X minutes, release now (the marginal route-optimization benefit doesn't justify the wait).
- **Per-region calibration:** mine historical inter-completion times per region. High-volume regions (North) get larger N*, short T*; low-volume regions (e.g., remote) get small N* or no batching at all if the data shows batch waits dominate their lead times.
- **Express exemption:** express orders bypass batching entirely (individual label generation), or join only batches releasing within a tight window.
- **Hazmat decoupling:** never let a batch wait on a hazmat order delayed by the regulatory cap — if a batch member is cap-blocked, release the batch without it and roll the hazmat order into the next batch.

**Data leverage:** Optimal (N*, T*) per region computed from historical completion streams via trade-off analysis: shipping-cost savings per batch size vs. cumulative synchronization delay (the marginal-delay metric from Section 1). The predictive trigger uses the same ready-time prediction model as Strategy 1.

**Expected outcomes:** Sharp reduction in batch waits for early-finishing orders (often the largest single source of idle case time); express SLA protection; eliminates hazmat-cap delay contagion into unconstrained orders.

### Strategy 3: Hazmat Flow Smoothing via Token-Based Admission Control

**Addresses:** Hazmat cap; Hazmat × Batching interaction.

**Proposed changes:**
- Treat the 10-order cap as a **token pool**: a hazmat order may only *start* Packing if a token is available; the token is returned when QC completes. This makes compliance structurally guaranteed (no manual counting, no violations) while exposing the cap as a schedulable resource.
- Add an upstream **release-smoothing rule**: stagger the release of hazmat orders from Item Picking so that concurrent hazmat WIP rarely spikes to 10. Mined data tells us the typical Packing+QC sojourn time for hazmat orders; pacing releases to ~`10 / sojourn_time` per hour keeps utilization of the cap below the binding threshold.
- **Sequencing rule under contention:** when tokens are scarce, prioritize hazmat orders by (1) express flag, (2) batch criticality (is a batch waiting on this order? — though Strategy 2's decoupling reduces this), (3) due date.

**Data leverage:** The cap-binding time-share metric and concurrent-count distribution from Section 1 determine how aggressive smoothing must be. If the cap binds only 5% of the time, lightweight token control suffices; if it binds 40% of peak hours, the data supports a business case for process changes that shorten hazmat Packing+QC sojourn (e.g., dedicated hazmat QC staff to drain tokens faster — increasing effective throughput *under the same regulatory limit*).

**Expected outcomes:** Guaranteed compliance; reduced variance in hazmat lead times; higher effective hazmat throughput within an unchanged legal cap (the lever is shortening token holding time, not raising the limit); elimination of cap-induced batch delays in combination with Strategy 2.

### Strategy 4 (supplementary): Non-Preemptive Priority with Lookahead Scheduling

**Addresses:** Priority handling penalty on standard orders.

Replace hard preemption with **non-preemptive priority + short-job lookahead**: an express order jumps the *queue* but never interrupts an in-progress activity, *unless* the remaining processing time of the current standard order (predicted from elapsed time vs. mined duration distributions) exceeds a threshold. Mining shows the typical packing duration is short (~6–11 min in the snippet); letting it finish usually costs the express order only minutes while avoiding the full preemption penalty (interrupt + re-setup + context loss). This is justified directly by the quantified preemption penalty vs. express delay trade-off from Section 1.

## Section 4: Simulation and Validation

### 4.1 Approach: Data-Driven Discrete-Event Simulation (DES) / Digital Twin

Build a simulation model **parameterized directly from the mined log**, not from assumptions:

- **Control flow** from the discovered process model (Order Received  Picking  Packing  QC  Label Gen, with attribute-dependent routing).
- **Arrival processes**: non-stationary arrival rates per order type/attribute combination (Express/Standard × Cold × Hazmat × Region), fitted per hour/day/season — critical because the constraints only bite under realistic peak load.
- **Service-time distributions** per activity, per resource, mined from START/COMPLETE pairs (fit distributions, don't use means — variance drives queues).
- **Resource model**: explicit 5 cold stations, standard stations, pickers, QC staff with mined shift calendars.

**Validation first:** Replay the simulation against the actual 3-month log (baseline scenario) and verify it reproduces observed KPIs — lead-time distributions, cold-station queue lengths, batch waits, cap-binding frequency — within tolerance. Only a validated baseline gives credible what-if results.

### 4.2 Critical Modeling Aspects for Instance-Spanning Constraints

Most off-the-shelf simulation/process-mining tools assume case independence, so these must be explicitly engineered:

1. **Resource contention:** Model cold stations as a finite resource pool with explicit queue discipline (FIFO baseline vs. proposed reservation policy). Validate simulated blocking probability against the mined one.
2. **Batching:** Implement the batch-formation logic as a synchronization construct (orders accumulate in region buffers; release per trigger rule). Test current static rule vs. Strategy 2's dynamic (N*, T*, predictive) triggers.
3. **Preemption mechanics:** Model interruption explicitly *including the mined preemption penalty* (re-setup time, possible rework). A simulation that models preemption as a free pause will falsely favor preemptive policies.
4. **Regulatory cap:** Implement as a global semaphore (10 tokens spanning Packing + QC). This is a facility-level state variable — exactly the kind of cross-case mechanism standard case-based simulation omits.
5. **Constraint interactions:** Because all mechanisms live in one model, interaction effects (e.g., hazmat-blocked order delaying its batch) emerge naturally — this is the main reason simulation is indispensable here; analytical queueing models can't handle the coupled constraints.

### 4.3 Experimentation Plan

- **Scenarios:** Baseline; each strategy in isolation; combined strategy portfolio; capacity variants (6 vs. 5 cold stations) as a benchmarking reference.
- **Load conditions:** Average load, peak-season load (replayed from November-equivalent weeks), and stress tests (+20–30% volume) to check robustness.
- **KPIs:** End-to-end lead time (mean, P90, P95 — tails matter for SLAs), express SLA attainment, standard-order on-time rate, cold-station utilization and queue length, batch wait distribution, hazmat cap-binding share, **compliance violations (must be zero in all scenarios)**, throughput.
- **Statistical rigor:** Multiple replications with confidence intervals; verify improvements are significant, not noise.
- **Sensitivity analysis:** How sensitive is Strategy 2 to ready-time prediction error? At what express-volume share does the reserved cold lane (Strategy 1) become counterproductive?

## Section 5: Post-Implementation Monitoring

### 5.1 Dashboard Architecture

**Dashboard A — End-to-End Flow Performance (daily/weekly):**
- Lead time distribution (mean, P90) by order type, with pre/post-change baselines.
- **Waiting-time attribution Pareto** (the Section 1.3 decomposition, computed continuously): share of waiting caused by cold contention / batching / preemption / hazmat cap / within-instance factors. *This is the headline view: it shows directly whether instance-spanning waste is shrinking.*
- On-time delivery rate, segmented Express vs. Standard.

**Dashboard B — Cold-Packing Station Health (real-time/hourly):**
- Live and historical queue length for cold stations; blocking probability trend.
- Utilization heatmap (station × hour); reservation-policy adherence (Strategy 1): % of cold orders served via reserved slot vs. ad hoc; express-lane idle time (to detect over-reservation).

**Dashboard C — Batching Effectiveness:**
- Batch synchronization delay per case (target: declining trend), batch size distribution vs. calibrated N* per region, frequency of each release trigger (size / timeout / predictive) — a rising timeout share signals miscalibrated N*.
- Express batch-bypass compliance; count of batches released without cap-blocked hazmat orders (Strategy 2 decoupling working as designed).

**Dashboard D — Compliance & Hazmat Flow (real-time, with alerting):**
- Live concurrent hazmat count in Packing+QC with hard alert at 10 and soft alert at 8.
- **Compliance violations: zero-tolerance KPI**, audited from the event log (defensible evidence for regulators — a key process-mining by-product).
- Cap-binding time share trend; hazmat token holding time (Packing+QC sojourn) — the lever for effective throughput.

**Dashboard E — Priority Policy Monitoring:**
- Preemption/queue-jump frequency; preemption penalty incurred (should fall sharply under Strategy 4); express wait vs. SLA; standard-order delay attributable to express handling (fairness check — express gains must not silently degrade standard SLAs).

### 5.2 Monitoring Discipline

- **Continuous ETL** from execution systems into the process mining platform (daily refresh minimum; near-real-time for Dashboards B and D).
- **Statistical process control** on key metrics (control charts on batch wait, cold-queue length) to distinguish genuine drift from normal variation, with alerts on sustained excursions.
- **Drift detection:** Order mix changes (e.g., rising cold/hazmat share, regional shifts) invalidate calibrated parameters (N*, T*, reservation windows, smoothing rate). Monitor input distributions and trigger **periodic recalibration** — the prediction models and policy parameters should be re-fitted monthly and stress-checked before each peak season using the validated simulation model (closing the loop: the digital twin becomes a standing what-if capability).
- **Quarterly review:** Re-run the full waiting-time attribution analysis to confirm the constraint Pareto has shifted as predicted and identify the *next* binding constraint — because in a coupled system, relieving one constraint will expose another, and the optimization program must iterate accordingly.

---

**Summary:** The core analytical move is to stop treating cases as independent: decompose waiting time into within- vs. between-instance causes via counterfactual availability tests, quantify each constraint and its interactions, redesign the coordination policies (reservation, dynamic batching, token-based admission, non-preemptive priority) that govern *how cases share* the system, validate everything in an interaction-aware digital twin, and monitor with attribution-based dashboards that keep compliance provable while throughput improves.
