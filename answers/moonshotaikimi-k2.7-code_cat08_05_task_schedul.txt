# Data-Driven Scheduling Improvement for Precision Parts Inc.

This plan transforms the company’s MES event logs into actionable scheduling intelligence. The core idea is to move from **static, local dispatching rules** to **dynamic, predictive, globally aware scheduling** by combining process mining, stochastic modeling, discrete-event simulation, and closed-loop monitoring.

---

## 1. Analyzing Historical Scheduling Performance and Dynamics

### 1.1 Reconstructing the Actual Shop-Floor Flow
The MES log contains events such as *Job Released*, *Queue Entry*, *Setup Start/End*, *Task Start/End*, *Breakdown Start/End*, and *Priority Change*. I would first build a canonical event log where each row represents one event with at least: `Case ID`, `Activity`, `Timestamp`, `Resource`, and relevant attributes (priority, due date, setup flag, operator, job family, material).

Using this log:

- **Process Discovery:** Apply algorithms like the **Inductive Miner** or **Heuristics Miner** to reconstruct the real routings per job type. Because each job may have a unique routing, I would mine **process variants** (distinct sequences of activities per case) rather than assume a single process model.
- **Performance Spectrum:** Use a performance spectrum or dotted chart to visualize how jobs actually traverse machines over time, revealing congestion, long queues, and crossover behavior.
- **Event-Correlation:** Link queue-entry events to the subsequent setup/task events on the same resource, so that waiting time is computed as the interval from queue entry to task/setup start.

### 1.2 Key Metrics to Quantify

#### Job Flow Times, Lead Times, and Makespan
For each job `j`:

- **Flow time (manufacturing lead time):**  
  $$F_j = C_j - R_j$$
  where \(C_j\) = completion time of the last operation, \(R_j\) = release time.
- **Total planned vs. actual lead time:** compare the planned lead time from the ERP/MES with \(F_j\).
- **Makespan:** \(\max_j(C_j) - \min_j(R_j)\) over a given planning horizon.
- **Inter-operation time (task-level lead time):** for each operation \(i\) of job \(j\),  
  $$T_{ij} = \text{Task End}_{ij} - \text{Queue Entry}_{ij}$$  
  decomposed into *waiting*, *setup*, and *processing*.

#### Task Waiting Times (Queue Times)
For each task on each machine:

- **Queue time:** \(Q_{ij} = \text{Setup Start}_{ij} - \text{Queue Entry}_{ij}\)
- **Waiting-time distribution per work center/machine:** fit empirical distributions or parametric distributions (lognormal, gamma) to identify tail behavior.
- **Queue-length dynamics over time** per machine.

#### Resource Utilization
For each machine \(m\) and operator \(o\):

- **Productive utilization:**  
  $$\frac{\text{Task Processing Time}}{\text{Available Time}}$$
- **Setup utilization:**  
  $$\frac{\text{Setup Time}}{\text{Available Time}}$$
- **Idle/available time** and **starvation time** (machine ready but no job available).
- **Busy-period analysis:** identify machines with consistently long busy periods and high queue buildup.

#### Sequence-Dependent Setup Times
To quantify sequence dependence:

1. For every setup event, retrieve the previous job \(J_{prev}\) processed on the same machine and the current job \(J_{curr}\).
2. Enrich both jobs with attributes that drive setup: material, alloy, tooling required, dimensions, surface finish, machine program.
3. Build a **setup-time matrix** \(S(J_{prev}, J_{curr})\). Because exact pairs may be sparse, use clustering or regression:
   - **Regression/ML model:** predict setup time from features of the previous and current job.
   - **Association rules:** discover frequent job-type transitions with unusually high setup.
4. Visualize the matrix as a heatmap per bottleneck machine.

#### Schedule Adherence and Tardiness
For each job:

- **Tardiness:** \(T_j = \max(0, C_j - D_j)\)
- **Lateness:** \(L_j = C_j - D_j\)
- **Earliness:** \(E_j = \max(0, D_j - C_j)\)
- **On-time delivery rate:** fraction of jobs with \(C_j \leq D_j\).
- **Tardiness distribution** by priority, customer, and job family.

#### Impact of Disruptions
- **Breakdown impact:** For each breakdown event, measure the increase in queue time and tardiness for jobs routed to the affected machine during and after the breakdown. Use **survival/queueing analysis** to estimate how long the system takes to recover.
- **Hot-job impact:** Compare the flow times of jobs that were preempted or bumped by urgent jobs against similar jobs in stable periods (variant analysis).
- **Priority-change cascades:** Count how often a priority change causes another job to become tardy.

## 2. Diagnosing Scheduling Pathologies

Process mining provides concrete evidence for each pathology:

### 2.1 Bottleneck Resources
- **Active-period method:** A machine is a bottleneck if its busy periods consistently coincide with the longest queues and highest tardiness.
- **Utilization–queue scatter plot:** high utilization *and* long queues indicate a capacity-constrained bottleneck.
- **Critical-path analysis:** reconstruct the actual precedence network per job and identify which machine delays most often appear on the critical path of late jobs.

### 2.2 Poor Task Prioritization
- **Variant analysis:** compare process variants of on-time jobs vs. late jobs. If late jobs frequently waited behind lower-priority jobs at a work center, the dispatching rule is failing.
- **Slack analysis:** compute remaining slack at each queue entry. If jobs with negative or near-zero slack are not served first, FCFS is overriding due-date urgency.

### 2.3 Suboptimal Sequencing and Excessive Setups
- **Setup-time decomposition:** compare actual setup time per machine against a theoretical minimum obtained by solving a sequencing problem (e.g., TSP on setup matrix). The gap reveals sequencing inefficiency.
- **Transition heatmap:** identify repeated transitions between incompatible job families that could have been avoided.

### 2.4 Starvation of Downstream Resources
- **Resource synchronization view:** overlay the start/end times of upstream and downstream machines. Starvation appears as gaps where a downstream machine is idle while an upstream machine is still processing jobs that should feed it.
- **Handoff-time analysis:** measure intervals between completion at one machine and queue entry at the next; large variability suggests poor coordination.

### 2.5 Bullwhip Effect in WIP
- **WIP time-series:** compute WIP level per work center over time.
- **Autocorrelation and cross-correlation:** if WIP spikes upstream precede starvation downstream, scheduling variability is propagating.
- **Little’s Law check:** \(WIP = \text{Throughput} \times \text{Flow Time}\). Rising WIP without proportional throughput gain signals scheduling-induced congestion.

## 3. Root Cause Analysis of Scheduling Ineffectiveness

### 3.1 Limitations of Static Dispatching Rules
FCFS ignores due dates and setup; EDD ignores downstream load and setup. In a high-mix job shop, no single static rule is globally optimal.

### 3.2 Lack of Real-Time Visibility
The current rules act locally. A job may be released to a machine that is about to break down or that has a long queue, because the scheduler lacks a real-time global state.

### 3.3 Inaccurate Time Estimations
If planned durations are deterministic averages, high-variance tasks will systematically delay jobs. Process mining can fit actual duration distributions and reveal bias (e.g., planned durations underestimate setup by 20%).

### 3.4 Ineffective Handling of Sequence-Dependent Setups
Without a setup-aware sequencing rule, the shop incurs avoidable setup time. Batching by similarity is not done because the scheduler does not see the similarity structure in the data.

### 3.5 Poor Coordination Between Work Centers
Local dispatching at each work center optimizes locally but can create starvation, congestion, and missed handoffs globally.

### 3.6 Inadequate Response to Disruptions
Breakdowns and hot jobs are handled ad hoc. There is no proactive re-optimization.

### 3.7 Separating Scheduling Logic from Capacity/Variability
Process mining helps distinguish these causes:

- **Capacity issue:** Even a perfectly sequenced schedule (solved retrospectively) still has high tardiness. The bottleneck is simply overloaded.
- **Scheduling logic issue:** There is a large gap between actual performance and a retrospective optimized schedule (e.g., solved with deterministic actual times). This gap is “scheduling loss.”
- **Variability issue:** High coefficient of variation in task times causes queueing delays even under good sequencing; solution requires buffer/safety time or capacity flexibility, not just better sequencing.

## 4. Developing Advanced Data-Driven Scheduling Strategies

### Strategy 1: Dynamic Multi-Attribute Dispatching with Setup and Downstream Awareness

**Core Logic**
Replace static FCFS/EDD with a real-time **composite priority score** computed for each job waiting at a machine. When a machine becomes free, the job with the highest score is selected.

Example score:

\[
P_j = w_1 \cdot \text{PriorityWeight}_j + w_2 \cdot \frac{1}{\text{Slack}_j} + w_3 \cdot \frac{1}{\text{PredictedSetup}_{prev,j}} + w_4 \cdot \text{DownstreamLoad}_j + w_5 \cdot \text{CriticalRatio}_j
\]

where:

- \(\text{Slack}_j = D_j - t - \text{RemainingWork}_j\)
- \(\text{PredictedSetup}_{prev,j}\) = model-predicted setup time from the last processed job to job \(j\)
- \(\text{DownstreamLoad}_j\) = estimated future waiting time at the next required machines
- \(\text{CriticalRatio}_j = \frac{D_j - t}{\text{RemainingWork}_j}\)

**Use of Process Mining**
- Historical flow times and queue times inform the estimation of \(\text{RemainingWork}\) and \(\text{DownstreamLoad}\).
- Setup-time models from Section 1 populate \(\text{PredictedSetup}\).
- Variant analysis and simulation are used to **tune weights \(w_i\)** (via grid search or reinforcement learning) for the shop’s KPI mix.

**Addressed Pathologies**
- Tardiness (via slack and critical ratio)
- Setup waste (via setup-aware term)
- Starvation (via downstream-load term)
- WIP (via shorter effective lead times)

**Expected Impact**
- 15–30% reduction in tardiness
- 10–20% reduction in effective setup time
- More stable lead times

### Strategy 2: Predictive and Proactive Scheduling

**Core Logic**
Instead of scheduling with deterministic planned times, use **stochastic task-duration distributions** mined from logs. Generate schedules that are robust to variability and predict future bottlenecks before they occur.

Components:

1. **Predictive Task Durations:** Fit distributions (e.g., lognormal, gamma, or kernel density estimates) for each task type/machine/operator/material combination. Use these to estimate a job’s completion-time distribution.
2. **Remaining Flow-Time Prediction:** Use survival models or gradient-boosted regression to predict the probability that a job will be tardy given its current state.
3. **Predictive Maintenance Integration:** If breakdown data is available, derive time-to-failure distributions. Reserve protective capacity or schedule preventive maintenance during low-load windows.
4. **Proactive Bottleneck Prediction:** A rolling-horizon simulator runs continuously to forecast where queues will exceed thresholds. The scheduler then reroutes jobs to alternate machines or expedites high-priority jobs.

**Use of Process Mining**
- Actual duration distributions per task/resource.
- Breakdown and repair distributions.
- Queue buildup patterns and recovery times after disruptions.

**Addressed Pathologies**
- Unpredictable lead times
- Breakdown cascades
- Inefficient resource use from unplanned downtime

**Expected Impact**
- More reliable due-date quoting
- Reduced impact of disruptions by proactive rerouting
- Higher utilization through maintenance-window optimization

---

### Strategy 3: Setup-Time Optimization via Intelligent Sequencing and Batching

**Core Logic**
Minimize non-value-added setup time, especially on bottleneck machines, by exploiting job similarity.

Components:

1. **Job-Family Clustering:** Cluster jobs based on setup-relevant attributes (material, tooling, program, dimensions). Use clustering on historical setup times to discover natural “setup families.”
2. **Bottleneck Sequencing:** At each bottleneck machine, solve (periodically or continuously) a **sequence-dependent scheduling problem**:
   - Use the Apparent Tardiness Cost with Setups (**ATCS**) heuristic to balance due-date urgency and setup reduction.
   - For offline planning, use TSP/insertion heuristics or constraint programming to find low-setup sequences.
3. **State-Preserving Batching:** Where appropriate, group similar jobs into small batches so the machine stays in the same setup state, reducing per-job setup. The batch size is bounded to avoid excessive WIP and tardiness.

**Use of Process Mining**
- Setup-time matrix and setup-family clusters.
- Identification of bottleneck machines where setup optimization has highest ROI.
- Comparison of actual setup times against optimized sequences to estimate savings.

**Addressed Pathologies**
- Excessive setup times
- Bottleneck overload
- High WIP caused by poor flow

**Expected Impact**
- 20–40% reduction in setup time on bottleneck machines
- Higher effective capacity
- Reduced lead times and WIP

## 5. Simulation, Evaluation, and Continuous Improvement

### 5.1 Discrete-Event Simulation for Strategy Evaluation

I would build a **discrete-event simulation (DES)** model of the job shop, parameterized directly from the MES logs:

| Model Parameter | Source from Process Mining |
|---|---|
| Task processing time distributions | Fitted from actual durations |
| Setup time matrix/model | Historical setup events |
| Routing probabilities/variants | Process variants discovered from logs |
| Breakdown inter-arrival and repair distributions | Resource breakdown events |
| Job release distribution | Historical release timestamps |
| Priority and due-date distributions | Historical job attributes |

**Evaluation Scenarios**
1. **Baseline scenario:** current dispatching rules.
2. **High-load scenario:** 20–50% increase in job releases.
3. **Disruption-heavy scenario:** frequent breakdowns and hot jobs.
4. **Mix-variability scenario:** more heterogeneous job families.
5. **Seasonal demand scenario:** demand surges followed by lulls.

**KPIs Compared**
- Mean and 95th-percentile tardiness
- On-time delivery rate
- Average WIP and WIP profile
- Mean and variance of lead time
- Machine utilization and setup utilization
- Maximum queue length per machine

Run each scenario with sufficient replications, compute confidence intervals, and use paired comparisons to select the best strategy or hybrid.

### 5.2 Continuous Monitoring and Adaptation Framework

After deployment, implement a **closed-loop scheduling optimization cycle**:

1. **Real-Time Event Ingestion:** Stream MES events into a process mining platform.
2. **Live KPI Dashboards:** Track tardiness, WIP, queue lengths, utilization, and setup ratios.
3. **Drift Detection:**
   - **Concept drift:** detect when task duration distributions, setup patterns, or routing mix change (e.g., using statistical process control or change-point detection).
   - **Performance drift:** detect sustained deviation from baseline KPIs.
4. **Automatic Retraining:**
   - Retrain setup-time prediction models and duration distributions monthly or when drift is detected.
   - Re-tune dispatching-rule weights periodically using recent data.
5. **What-If Re-Optimization:** When a major disruption occurs, run a fast DES/optimization to reschedule remaining jobs.
6. **Scheduled Review:** Quarterly review of process variants, bottleneck maps, and scheduling loss to identify new improvement opportunities.
