#### Query 1: Sequence and Completeness Validator
This query pivots the event log to establish a timeline for each order. It flags cases where shipping happens before critical checks, payment happens before invoicing, or mandatory steps are missing entirely.

```sql
WITH EventTimeline AS (
    SELECT 
        case_id,
        MAX(CASE WHEN activity = 'Register Order' THEN timestamp END) AS t_register,
        MAX(CASE WHEN activity = 'Perform Credit Check' THEN timestamp END) AS t_credit,
        MAX(CASE WHEN activity = 'Validate Stock' THEN timestamp END) AS t_stock,
        MAX(CASE WHEN activity = 'Confirm Shipment' THEN timestamp END) AS t_confirm_ship,
        MAX(CASE WHEN activity = 'Ship Goods' THEN timestamp END) AS t_ship,
        MAX(CASE WHEN activity = 'Issue Invoice' THEN timestamp END) AS t_invoice,
        MAX(CASE WHEN activity = 'Receive Payment' THEN timestamp END) AS t_payment
    FROM order_event_log
    GROUP BY case_id
)
SELECT 
    case_id,
    -- Sequence Violations
    CASE WHEN t_ship < t_credit THEN 'Shipped before Credit Check' ELSE 'OK' END AS credit_seq_issue,
    CASE WHEN t_ship < t_stock THEN 'Shipped before Stock Validation' ELSE 'OK' END AS stock_seq_issue,
    CASE WHEN t_payment < t_invoice THEN 'Payment before Invoice' ELSE 'OK' END AS invoice_seq_issue,
    -- Missing Mandatory Steps
    CASE WHEN t_credit IS NULL THEN 'Missing Credit Check' ELSE 'OK' END AS missing_credit,
    CASE WHEN t_stock IS NULL THEN 'Missing Stock Validation' ELSE 'OK' END AS missing_stock
FROM EventTimeline
WHERE 
    t_ship < t_credit OR 
    t_ship < t_stock OR 
    t_payment < t_invoice OR 
    t_credit IS NULL OR 
    t_stock IS NULL
ORDER BY case_id;
```

#### Query 3: Segregation of Duties (SoD) Violation Check
This query ensures that activities are being performed by the correct department. For example, Finance should handle credit/invoicing, and Logistics should handle shipping. It flags any cross-departmental anomalies.

```sql
SELECT 
    e.case_id,
    e.activity,
    e.resource,
    r.department AS actual_department,
    CASE 
        WHEN e.activity = 'Register Order' AND r.department != 'Sales' THEN 'Violation'
        WHEN e.activity IN ('Perform Credit Check', 'Issue Invoice', 'Receive Payment') AND r.department != 'Finance' THEN 'Violation'
        WHEN e.activity IN ('Validate Stock', 'Confirm Shipment', 'Ship Goods') AND r.department != 'Logistics' THEN 'Violation'
        ELSE 'Compliant'
    END AS sod_status
FROM order_event_log e
JOIN resources r ON e.resource = r.resource_id
WHERE 
    (e.activity = 'Register Order' AND r.department != 'Sales') OR
    (e.activity IN ('Perform Credit Check', 'Issue Invoice', 'Receive Payment') AND r.department != 'Finance') OR
    (e.activity IN ('Validate Stock', 'Confirm Shipment', 'Ship Goods') AND r.department != 'Logistics')
ORDER BY e.case_id, e.timestamp;
```

#### Query 4: Abnormal Velocity Detector
This query identifies orders where the time between "Register Order" and "Receive Payment" is unrealistically short (e.g., under 15 minutes). This is a strong indicator of data entry errors, system glitches, or potential fraudulent pre-payment manipulation.

```sql
WITH EventTimeline AS (
    SELECT 
        case_id,
        MAX(CASE WHEN activity = 'Register Order' THEN timestamp END) AS t_register,
        MAX(CASE WHEN activity = 'Receive Payment' THEN timestamp END) AS t_payment
    FROM order_event_log
    GROUP BY case_id
)
SELECT 
    et.case_id,
    o.order_type,
    o.order_value,
    et.t_register,
    et.t_payment,
    ROUND(EXTRACT(EPOCH FROM (et.t_payment - et.t_register)) / 60, 2) AS minutes_to_payment
FROM EventTimeline et
JOIN orders o ON et.case_id = o.case_id
WHERE et.t_payment IS NOT NULL 
  AND et.t_register IS NOT NULL
  AND (et.t_payment - et.t_register) < INTERVAL '15 minutes'
ORDER BY minutes_to_payment ASC;
```
