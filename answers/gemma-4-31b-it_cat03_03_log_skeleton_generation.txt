Based on the scenario described for the order fulfillment process, here is the constructed Log Skeleton expressed as a Python dictionary.

```python
log_skeleton = {
    'equivalence': {
        ('RO', 'IC'),  # Every order received must be invoiced
        ('RO', 'D'),   # Every order received must eventually be dispatched
    },
    'always_before': {
        ('RO', 'CA'),  # Receive Order before Checking Availability
        ('CA', 'RI'),  # Check Availability before Reserving Items
        ('RI', 'PI'),  # Reserve Items before Picking Items
        ('PI', 'QC'),  # Pick Items before Quality Check
        ('QC', 'PO'),  # Quality Check before Packing Order
        ('PO', 'GSL'), # Pack Order before Generating Shipping Label
        ('GSL', 'D'),  # Generate Shipping Label before Dispatch
        ('IC', 'RP'),  # Invoice Customer before Recording Payment
    },
    'always_after': {
        ('CA', 'RO'),  # Checking Availability implies the order was received
        ('RI', 'CA'),  # Reserving implies availability was checked
        ('PI', 'RI'),  # Picking implies items were reserved
        ('QC', 'PI'),  # Quality Check implies items were picked
        ('PO', 'QC'),  # Packing implies quality check was performed
        ('GSL', 'PO'), # Shipping label implies packing is done
        ('D', 'GSL'),  # Dispatch implies a label was generated
        ('RP', 'IC'),  # Recording payment implies an invoice was sent
    },
    'never_together': set(),  # In this standard linear flow, all activities can co-exist in one case
    'activ_freq': {
        'RO': {1},   # Received once
        'CA': {1},   # Checked once
        'RI': {1},   # Reserved once
        'PI': {1},   # Picked once
        'QC': {1},   # Quality checked once
        'PO': {1},   # Packed once
        'GSL': {1},  # Label generated once
        'D': {1},    # Dispatched once
        'IC': {1},   # Invoiced once
        'RP': {0, 1}, # Payment may or may not have been recorded yet
    },
    'directly_follows': {
        ('RO', 'CA'),
        ('CA', 'RI'),
        ('RI', 'PI'),
        ('PI', 'QC'),
        ('QC', 'PO'),
        ('PO', 'GSL'),
        ('GSL', 'D'),
    }
}
```

### Logic Explanation:

1.  **Equivalence**: I have mapped `RO` (Receive Order) to `IC` (Invoice Customer) and `D` (Dispatch). In a successful business process, for every single order that enters the system, there should be exactly one invoice and one dispatch event.
2.  **Always Before / After**: These represent the logical dependencies of the supply chain. You cannot pick items before you reserve them, and you cannot dispatch a package before generating a label.
3.  **Never Together**: Since the scenario describes a standard sequential fulfillment process without alternative paths (like "Cancel Order" vs "Complete Order"), there are no mutually exclusive activities.
4.  **Activity Occurrences**: Most activities are set to `{1}` as they are mandatory and occur once per order. `RP` (Record Payment) is set to `{0, 1}` because, in real-world logs, a case might be captured after dispatch but before the customer has actually completed the payment.
5.  **Directly-Follows**: This captures the strict operational sequence from the moment the order is received until it is dispatched.
