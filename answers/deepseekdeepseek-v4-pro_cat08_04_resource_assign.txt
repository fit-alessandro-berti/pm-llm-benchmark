## 1. Analyzing Resource Behavior and Assignment Patterns

To understand the real-life execution of incident management, we will mine the event log from three perspectives: **performance metrics**, **social/organizational network analysis**, and **skill-utilization conformance**. This combined view reveals not only how individuals and tiers perform but also the hidden assignment networks that deviate from the intended logic.

### 1.1 Agent- and Tier-Level Performance Metrics

We compute a set of resource-centric KPIs for every agent and for aggregated tiers (L1, L2, L3). The event log allows us to distinguish “Work [Tier] Start/End” activities for precise processing times and to track ticket ownership over time.

| Metric group | Specific KPIs | Purpose |
|--------------|---------------|---------|
| **Workload & distribution** | – Number of tickets handled (completed, active, reassigned) per agent/tier <br> – Work-in-progress (WIP) counts over time (binned hourly) <br> – Arrival-to-assignment and assignment-to-work-start delays | Identify overloaded/underutilized agents and queues |
| **Processing efficiency** | – Median/mean activity processing time per agent for each ticket category <br> – Cycle time from “Work Start” to “Work End” <br> – Wait time between “Assign” and “Work Start” per tier | Detect slow performers, delays caused by queue buildup |
| **First-Call Resolution (FCR)** | – Percentage of L1-handled tickets that are resolved without escalation (no “Escalate L2” event) <br> – FCR broken down by ticket category and required skill | Quantify the effectiveness of L1 triage and training gaps |
| **Activity specialization** | – How often each agent works on tickets requiring a specific skill vs. their declared skills <br> – The distribution of ticket categories/priorities per agent | Uncover whether agents handle tasks aligned with their expertise |

Using simple aggregation and filtering on the log, we can, for example, see that L1 agent A02 frequently escalates “Networking-Firewall” tickets (FCR = 20% for this category) while another L1 agent handles the same category with 60% FCR—pointing to a skills imbalance within the tier.

### 1.2 Actual Assignment Networks via Organizational Mining

Process mining offers a suite of organizational analysis techniques that go far beyond static org charts.

- **Handover of Work / Social Network Analysis:** Treating reassignments and escalations as handovers between resources. We build a directed graph where nodes are agents (or pools), and edges represent the number of tickets transferred. The weights reveal:
  - *Unofficial hubs:* Certain agents (e.g., B12) repeatedly become “go-to” people for reassigned tickets even when they are not the intended assignees.
  - *Ping-pong patterns:* Two agents frequently sending tickets back and forth — a strong indicator of misassignment or unclear accountability.
  - *Escalation cascades:* A ticket like INC1001 jumps from L1 (Agent A05)  L2 (B12)  L2 (B15), showing a second intra-tier reassignment that adds delay.

- **Role Discovery (Clustering):** By applying clustering algorithms on the resource-activity matrix (which activities each agent performs), we can detect *de facto* roles that differ from formal job titles. For instance, an L1 agent who regularly performs “Database-SQL” tasks may effectively be operating as an L2 database specialist. Conversely, a designated L3 specialist who only performs password resets would be flagged as misutilized. This “discovered” role model is compared against the intended tier-based assignment logic to quantify the gap.

- **Conformance Checking on Assignment Rules:** We model the intended assignment flow (e.g., “L1 handles P3/P4, escalates P1/P2 immediately” or “tickets requiring `Networking-Firewall` go straight to L2 network team”). Replaying the log against this model exposes violations, such as a P2 network ticket staying too long in L1 or a low-priority software ticket being assigned directly to a highly paid L3 engineer.

### 1.3 Skill Utilization Analysis

We cross-reference the `Required Skill` field from the ticket with the `Agent Skills` attribute attached to every resource event. A mismatch occurs whenever the agent who starts working on the ticket does **not** list the required skill in their profile.

- A **skill utilization heatmap** (ticket required skill × agent declared skills) will spotlight:
  - *Underdeployment of specialists:* L3 network security experts spending 40% of their time on “Access Management” tickets that any L1 could resolve.
  - *Skill shortages:* A required skill like “Database-SQL” showing a high average wait time before work start because only two agents hold it, and both are perpetually overloaded.
  - *Overreliance on key individuals:* Single points of failure where one agent (e.g., B15 for Database-SQL) handles nearly all tickets needing that skill, causing delays when the agent is absent.

The analysis quantifies the **percentage of “work time” spent on tasks below an agent’s capability**, directly tying to resource waste and escalations delays.

## 2. Identifying Resource-Related Bottlenecks and Issues

Using the above performance and assignment mining results, we pinpoint concrete bottlenecks and quantify their operational impact.

### 2.1 Bottlenecks Caused by Skill Shortages

- **Detection:** For each required skill, calculate the average waiting time between the “Assign [Tier]” event and the “Work [Tier] Start” event. When this time exceeds a threshold (e.g., >30 min for P2), we flag a capacity bottleneck.
- **Example:** The log shows tickets needing “Networking-Firewall” wait three times longer than average before an L2 agent begins work, while 85% of these delays are due to only two agents having the skill.
- **Quantification:** The average delay per P2 network ticket is 47 minutes, directly contributing to 24% of all P2 SLA breaches.

### 2.2 Delays from Frequent Reassignments and Escalations

- **Detection:** We mine the number of assignment events per case. Cases with >1 reassignment after initial tier placement are considered “wasteful flow.”
- **Impact measurement:** By comparing the cycle time of straightforward cases (single assignment) versus cases with two or more reassignments, we can isolate the added time.
  - *Sample finding:* Cases like INC1001 experience an additional 1.6 hours of delay solely due to the reassignment loop (Agent B12  back to dispatcher  Agent B15).
- **Correlation with SLA:** Calculate the Pearson correlation between the number of handovers and the probability of SLA breach. In many service desks, each reassignment adds roughly 40–90 minutes of nonvalueadding wait time and increases breach risk by 15–20%.

### 2.3 Incorrect Initial Assignments

- **Root Identification:** For tickets that eventually required a reassignment, we trace back to the initial assigning resource (dispatcher or L1 agent). If 60% of reassignments originate from dispatcher decisions that ignored the required skill, the dispatcher rule engine is a prime culprit.
- **Quantification:** We measure the “firsttime right” assignment rate per agent role and per ticket category. For instance, tickets categorized as “SoftwareApp” but actually needing “DatabaseSQL” (as in INC1001) are misrouted 40% of the time, causing an average 2hour resolution delay.

### 2.4 Underperforming/Overloaded Agents

- **Workload imbalance:** Using case count and WIP distribution, we compute the Gini coefficient across agents within the same tier. A highly skewed distribution (e.g., top 20% of L2 agents handling 55% of all P2 tickets) signals an assignment rule that does not consider current queues.
- **Individual performance:** Agents with median processing times significantly above the tier average (e.g., +2 standard deviations) and simultaneous low FCR rates are flagged as underperforming. Overloaded agents are those whose WIP consistently exceeds capacity, leading to cascading queue delays for everyone.

### 2.5 Direct Link to SLA Breaches

We tag every case with its SLA outcome and run a logistic regression or decision tree to find which assignment characteristics most strongly predict breaches. The top predictors typically include:
- Number of reassignments,
- Wait time before first skilled work,
- Whether the initially assigned agent lacked the required skill.

This yields a concrete number: e.g., “73% of breached P2 tickets involved at least one skill mismatch, costing the organization an average 150 extra minutes per ticket.”

## 3. Root Cause Analysis for Assignment Inefficiencies

Process mining does not stop at symptoms; it enables structured root-cause discovery.

### 3.1 Variant Analysis: “Good vs. Bad” Flows

We split cases into two groups:
- **“Smooth” flow:** Tickets that have a single assignment or at most one direct escalation, with no reassignments.
- **“Turbulent” flow:** Tickets with multiple reassignments and escalations.

By comparing the distribution of attributes between the two groups (category, priority, initial assignee, hour of day, creation channel), we identify the conditions that breed poor assignments. For example:
- 80% of turbulent tickets are created via web portals where the userselected category is often inaccurate.
- Turbulent tickets are 3x more likely to be P2 priorities assigned during shiftchange hours, when dispatchers rush decisions.

### 3.2 Decision Mining on Assignment Choices

We apply decision mining on the exact moment a ticket is assigned or escalated. The event log provides context features: ticket priority, category, current queue length per agent, time of day, agent’s current WIP, and available skills. The decision tree learns what drives a **reassignment** or an **escalation**.

Potential discovered rules:
- *If ticket category = “Network” AND required skill = “Networking-Firewall” AND L2 queue length > 5  92% probability of reassignment to another L2 agent.* This indicates that dispatchers simply pass the ticket around when the specialized queue is full, rather than resolving the bottleneck.
- *If L1 agent skill set does NOT contain “DB-SQL” AND ticket key words contain “database”  85% chance of immediate escalation.* The system should proactively identify such tickets at creation.

### 3.3 Root Causes from the Analysis

The most probable root causes, confirmed by mining, are:

1. **Deficient assignment rules:** Current logic is roundrobin within tiers, ignoring skill requirements and realtime workload. It treats all L2 agents as interchangeable, causing mismatches like sending a firewall ticket to an App-CRM specialist.
2. **Incorrect / incomplete skill profiles:** The `Agent Skills` list is outdated; some agents have skills they never use, while others informally handle specialized tasks they aren’t officially tagged with, leading to blind spots in automatic routing.
3. **Poor initial categorization:** The ticket category “SoftwareApp” is too broad. It often hides the real required skill (e.g., DatabaseSQL), causing dispatchers to misroute.
4. **No realtime workload visibility:** Dispatchers lack a dashboard showing current WIP per agent; they assign based on static roles, not current capacity, so overloaded specialists receive even more work.
5. **L1 disempowerment:** L1 agents have minimal diagnostic scripts and authority; they escalate even when a simple knowledge article could resolve the issue. Decision mining shows that tickets with certain keywords (e.g., “password reset on CRM”) are escalated despite a 95% historical resolution rate by L1 when given proper instructions.

These root causes are validated by replaying the log against a “tobe” assignment model and observing where deviations cluster.

## 4. Developing Data-Driven Resource Assignment Strategies

Based on the insights above, I propose three distinct, impactful strategies that can be layered over time.

### Strategy 1: Skill-Based Routing with Proficiency-Aware Weighting

**Addressed issue:** Frequent misassignments and reassignments due to roundrobin ignoring skills.

**How it leverages process mining insights:**  
We first mine a **skilldemand baseline** from historical tickets: for each combination of ticket category, description keywords (extracted via NLP), and priority, we predict the required skill. Simultaneously, we build an enhanced agent skill matrix where each skill is weighted by proficiency (derived from past resolution speed and reassignment counts for that skill). For example, Agent B12 may have `App-CRM: 0.9` (solved 95% of CRM tickets without reassignment) but only `Database-SQL: 0.2`.

**Assignment logic:**  
When a new ticket arrives, its required skill is inferred using the mined prediction model. A scoring function for each available agent combines:
- Skill match score (exact match weighted by proficiency),
- Historical success rate for that ticket category,
- Currently active WIP (penalizing agents approaching capacity).

The system assigns the ticket to the agent with the highest score, not just any free resource.

**Data required:** Historical ticket attributes (category, keywords, required skill), agent skill-proficiency matrix, realtime WIP from the service desk tool.

**Expected benefits:**
- 40–60% reduction in reassignments by ensuring tickets reach the most competent available agent.
- Decrease in mean resolution time because the assigned agent possesses the right skills, reducing the need for consultation.
- Improved specialist morale as they no longer waste time on tasks outside their expertise.

---

### Strategy 2: Workload-Aware Pull Assignment with Dynamic Queue Balancing

**Addressed issue:** Uneven workload distribution and bottlenecks caused by blindly pushing tickets to overloaded agents.

**How it leverages process mining insights:**  
Using resource workload profiles mined from the log, we know the typical processing rate for each agent per ticket type (e.g., Agent B08 handles 2 network tickets per hour, while B10 handles 1.5). We also know the arrival pattern and queue buildup curves. This enables us to model the “time to availability” for each agent.

**Assignment logic:**  
Shift from a dispatcherpush model to a **guided pull system**. The system maintains a prioritized queue of unassigned tickets visible to all agents of the appropriate tier, with recommended assignments. An agent pulls the next ticket based on:
- Priority (ensuring SLA targets),
- A personal matching score (skill fit),
- The agent’s own current WIP.

Simultaneously, a background algorithm monitors queue lengths per skill group. If the “NetworkingFirewall” queue exceeds a threshold, the system can temporarily allow crosstrained L2 agents from other skills to pull those tickets (with appropriate support), dynamically rebalancing load.

**Data required:** Historical processing speeds per agent/skill, realtime WIP per agent, agent availability status, queue length metrics.

**Expected benefits:**
- Smoother workload distribution: Gini coefficient of case load across the tier drops by at least 30%.
- Waiting times for specialized skills decrease because no one agent is overloaded while another sits idle.
- SLA compliance improves as highpriority tickets are pulled immediately by the next free qualified resource, not queued behind a dispatcher’s manual action.

### Strategy 3: Predictive Assignment with Automated PreEscalation

**Addressed issue:** Poor initial categorization and excessive L1 handling of tickets that inevitably escalate, causing unnecessary delays and consumption of L1 capacity.

**How it leverages process mining insights:**  
We train a machine learning model on the entire event log to predict, at ticket creation, two things:
1. The **probability that L1 will be able to resolve** the ticket (FCR likelihood).
2. The **most likely required skill(s) and complexity level** (singleskill L2, multiskill L3, etc.).

The prediction uses features like ticket category, freetext description (TFIDF vectorized), priority, channel, and time of day. Decision mining earlier showed that certain patterns almost never resolve at L1 (e.g., P2 tickets mentioning “database corruption”  97% escalate).

**Assignment logic:**
- If the predicted L1 FCR probability is above a threshold (e.g., 70%), the ticket lands in the L1 queue, possibly accompanied by the top predicted solution steps from a knowledge base.
- If probability is low, the ticket is **automatically routed directly to the appropriate specialist pool** (L2/L3), bypassing L1 triage entirely. The required skill is attached, enabling skillbased routing (Strategy 1) to act immediately.

**Data required:** Ticket creation logs with resolution paths, text description for NLP, all attributes used in the predictive model; a feedback loop to update the model with outcomes.

**Expected benefits:**
- Drastic reduction in escalations from L1: tickets that would have bounced avoid L1 completely, saving 3090 minutes per case.
- L1 agents focus on truly resolvable issues, increasing their FCR and reducing their feeling of constant failure.
- Faster timetoresolution for complex tickets, directly improving P2 and P3 SLA adherence.
- Fewer reassignments because the specialist pool is targeted from the start.
