## 1. Highlevel Process Narrative  

| Phase | Business purpose |
|-------|-------------------|
| **Receive Customer Request** | Capture the incoming demand (e.g., a sales order, a support ticket, or a product inquiry). |
| **Determine Request Type** | Decide whether the request can be processed with the **standard catalogue** or requires a **custom configuration**. |
| **Standard Path** | Run a set of predefined validations and checks in parallel, generate a delivery date, possibly need a manager’s signoff, then produce the final invoice and send confirmation. |
| **Custom Path** | Conduct a feasibility analysis, check if the customisation is possible, produce a quotation or a rejection notice, then optionally need manager approval before the same finalinvoice step. |

## 2. Detailed Walkthrough of Each Element  

### 2.1 Start Event  **Task A: “Receive Customer Request”**  
* **Purpose:** Collect all inbound trigger data (order number, contact details, description of the work).  
* **Business relevance:** Marks the moment the customer enters the system; everything downstream is driven by this request.

### 2.2 XORGateway – **“Check Request Type”**  
* **Gateway type:** **Exclusive (XOR)** – exactly **one** of the outgoing flows is taken.  
* **Outcomes:**  
  * **Type = Standard**  proceed to the *Standard* subprocess (B1  …).  
  * **Type = Custom**  proceed to the *Custom* subprocess (B2  …).

### 2.3 **Standard Request Subprocess**  

| Element | Description |
|---------|-------------|
| **B1: “Perform Standard Validation”** | Verify that the request fits the predefined product/service bundles, pricing rules, and mandatory fields. |
| **Gateway (AND)** – **“Run Parallel Checks”** | Splits the flow into **two independent threads** that can execute simultaneously: <br>• **C1: “Credit Check”** – verify the customer’s creditworthiness. <br>• **C2: “Inventory Check”** – confirm product availability. |
| **Join (All Parallel Checks Completed)** | Waits until **both** C1 and C2 finish, then merges them back into a single stream. |
| **D: “Calculate Delivery Date”** | Uses the results of the credit and inventory checks (and possibly other standard parameters) to compute the earliest possible shipment date. |
| **XORGateway – “Is Approval Needed?”** | Checks whether the calculated delivery date or any other business rule triggers a **manager approval** step. |
| *If Yes*: <br>• **F: “Obtain Manager Approval”** – manager reviews the request/outcome and signs off. |
| *If No*: | Skips F directly to the next gateway. |
| **XORGateway – “Is Approval Granted?”** | After the manager has been asked (or skipped), a binary decision is made. |
| *If Yes*: <br>• **G: “Generate Final Invoice”** – the invoice is created and sent to the customer. |
| *If No*: <br>• **H: “Reevaluate Conditions”** – the request is sent back for a different decision point (e.g., a discount check, a different delivery window). The loop can either: <br>• Return to **E1** (only for custom path, see below) **or** <br>• Return to **D** (standard path) to recalculate the delivery date with adjusted parameters. |
| **If No Approval Needed** (i.e., the earlier XORgateway answered “No”)  directly to **G**. |

### 2.4 **Custom Request Subprocess**  

| Element | Description |
|---------|-------------|
| **B2: “Perform Custom Feasibility Analysis”** | Deepdive work to understand the bespoke requirements, potential technical constraints, and pricing impact. |
| **XORGateway – “Is Customization Feasible?”** | Decides whether the customisation can be delivered. |
| *If Yes*  **E1: “Prepare Custom Quotation”** – a detailed quote (including price, schedule, and scope) is generated. |
| *If No*  **E2: “Send Rejection Notice”**  **End Event** (process terminates). |
| **After the “Standard or Custom Path Tasks Completed”** gateway | A **coordinating** XORgateway that acknowledges *any* completion of the chosen path (standard or custom). It does **not** split; it simply waits for the branch that was taken to finish. |
| **XORGateway – “Is Approval Needed?”** | Identical decision point as in the standard branch, but now it sits **after** the customquotation step. It asks whether a manager signoff is required for the custom quotation or for any downstream contractual obligations. |
| *If Yes*: <br>• **F** (Obtain Manager Approval)  **XORGateway “Is Approval Granted?”**  **G** (Generate Final Invoice) or **H** (Reevaluate Conditions)  backloop to **E1** for another round of revision. |
| *If No*: | Directly to **G** (Generate Final Invoice). |

### 2.5 **Convergence & Final Steps**  

* Regardless of whether the request was **standard** or **custom**, once the invoice has been generated, the process reaches **Task I: “Send Confirmation to Customer”**.  
* **End Event** follows – the process instance is completed.

## 3. Meaning of the Gateways  

| Gateway | Type | Business meaning |
|---------|------|-------------------|
| **XOR (Check Request Type)** | Exclusive | Only **one** path can be taken – the request is either standard **or** custom. |
| **AND (Run Parallel Checks)** | Parallel (AND) | Two activities (**Credit Check** and **Inventory Check**) **must both finish** before the flow can continue. This models a coordination point where multiple independent validations are required simultaneously. |
| **Join (All Parallel Checks Completed)** | Synchronisation | Waits for the **parallel split** to converge; the next activity can start only after *both* checks are done. |
| **XOR (Is Approval Needed?)** | Exclusive decision | Determines whether a **manager approval** step is mandatory. |
| **XOR (Is Approval Granted?)** | Exclusive decision | Checks the **outcome** of the manager approval request. |
| **Coordinating XOR (After Standard or Custom Path Tasks Completed)** | Exclusive – simply marks that *whichever* subprocess finished, we move to the next coordination point. It does not introduce a split, only a decision point that can flow to either path. |

## 4. Convergence, Loopback, and ReWork  

| Situation | What Happens | Why It Matters |
|-----------|--------------|----------------|
| **Approval Denied (standard)** | Flow goes to **H: “Reevaluate Conditions”**  loops back to **D** (or optionally to **E1** for custom). | Allows the business to adjust parameters (e.g., price, delivery date) and rerun the calculation without restarting the entire process. |
| **Customisation Not Feasible** | Directly to **E2: “Send Rejection Notice”  End** | Terminates the instance early, saving resources and informing the customer promptly. |
| **Approval Granted  Generate Invoice  Loop Back** | If approval is granted but later the manager decides the invoice is unsuitable (e.g., a discount needs adjustment), the process can reenter **H** and **E1** (custom) or **D** (standard) to **recalculate**. | Provides a controlled mechanism for *rework* that respects the original decision points without forcing a full restart. |
| **After Approval Rejected** | If **Approval Granted?** answers **No**, the flow goes to **H** (or directly to **G** if “Approval Needed?” answered No). In the custom branch, looping back to **E1** is explicitly shown, meaning the custom quotation can be revised and reapproved. | Ensures that a custom offer can be refined (price, scope) before final invoicing. |

## 5. Standard vs. Custom Paths – How They Differ  

| Dimension | Standard Path | Custom Path |
|-----------|---------------|-------------|
| **Initial Validation** | Simple rulecheck (B1). | Feasibility analysis (B2) that may involve deeper technical review. |
| **Parallel Checks** | Credit + Inventory (both mandatory). | No explicit parallel checks; the feasibility step implicitly includes many subtasks but they are modeled sequentially. |
| **Quotation / Quote Generation** | Not shown explicitly; the final invoice is generated after validation and approval. | **E1** creates a **custom quotation** – a distinct activity that does not exist in the standard flow. |
| **Outcome of Feasibility Check** | N/A (standard always proceeds). | Binary **Yes/No** – “Yes” leads to quotation, “No” triggers a rejection notice and process end. |
| **Approval Trigger** | May be required based on delivery date or other thresholds. | Also may be required, but the decision point follows the custom quotation step. |
| **Rework Loop** | If approval is denied, the process loops back to **D** (recalculate delivery date) or to **E1** (if the request were custom). | If the feasibility analysis fails, the process ends; if approval is denied after a quotation, the process can loop back to **E1** to revise the quotation. |
| **Parallelism** | Explicit parallel split/join for **Credit** and **Inventory** checks. | No parallel split illustrated; the custom path is shown as a linear sequence (analysis  feasibility  quotation  approval  invoice). |
| **Business Typicality** | Represents routine, repeatable orders that follow a known process. | Represents bespoke, oneoff, or heavily customized requests that need extra scrutiny. |

## 6. Summary of Decisionpoint Logic  

1. **Requesttype decision (XOR)**  selects **Standard** or **Custom**.  
2. **Standard path** proceeds through:  
   - Validation  **AND** split  Credit & Inventory (parallel)  **Join**.  
   - Deliverydate calculation  **XOR** “Approval Needed?”  (optional) Manager approval  **XOR** “Approval Granted?”  **Invoice** or **Reevaluate** (loop).  
3. **Custom path** proceeds through:  
   - Feasibility analysis  **XOR** “Feasible?”  **Yes**  Quotation  **XOR** “Approval Needed?” (same managerapproval logic)  **Invoice** or **Reevaluate** (loop back to quotation).  
4. **Reevaluate** steps either recalculate the delivery date (standard) or generate a new quotation (custom).  
5. **Final confirmation** is always sent, then the process ends.

## 7. How Approvals & Rework Are Triggered – Condition Summary  

| Condition | Triggered Step | Resulting Action |
|-----------|----------------|------------------|
| **Manager approval required** (based on business rules, e.g., delivery date beyond a threshold, price exceeds discount limit) | **XOR “Is Approval Needed?”  Yes** | Execute **F: Obtain Manager Approval**. |
| **Manager approval granted** | **XOR “Is Approval Granted?”  Yes** | Proceed to **G: Generate Final Invoice**. |
| **Manager approval denied** | **XOR “Is Approval Granted?”  No** | Execute **H: Reevaluate Conditions**. |
| **Reevaluate Conditions (standard)** | Loop back to **D** | Recalculate delivery date (or any other parameter) with possibly different inputs. |
| **Reevaluate Conditions (custom)** | Loop back to **E1** | Revise the custom quotation and retrigger the feasibility/approval cycle. |
| **Customization not feasible** | **XOR “Is Customization Feasible?”  No** | Execute **E2: Send Rejection Notice  End**. |
| **No approval needed** (gateways answered **No** at the appropriate point) | Directly to **G** | Generate the final invoice and move to confirmation. |

### Bottom Line  

- The diagram captures a **twobranching, decisionrich workflow** that first distinguishes between **standard** and **custom** customer requests.  
- **Gateways** serve as **exclusive decision points** (XOR) or **synchronisation points** (AND/Join) that direct flow control and coordinate parallel activities.  
- **Parallel checks** (credit & inventory) run concurrently, after which the process converges before moving on.  
- **Approval steps** are inserted where business policy demands a manager’s signoff; if denied, the flow **reevaluates** conditions and can **loop back** to earlier calculation or quotation steps.  
- **Custom requests** introduce an extra feasibility analysis and quotation generation, and their rework loop returns to the quotation stage, whereas standard rework returns to the deliverydate calculation.  
- Once the invoice is generated, a single **confirmation step** closes the process, regardless of which path was taken.
