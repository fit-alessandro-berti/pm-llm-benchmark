*   **Pair:** `('R', 'P')`

*   **Anomaly:** The standard deviation is extremely low relative to the average (ratio  0.04). In a manual or semi-automated insurance process, approval times typically show higher variance due to claim complexity, adjuster workload, and decision-making time. A tight clustering around exactly 25 hours suggests a deterministic trigger rather than organic processing.

#### 2. Severe Notification Bottleneck (Approve  Notify)
*   **Pair:** `('P', 'N')`
*   **Metrics:** Average = 604,800 seconds (7 days), Standard Deviation = 172,800 seconds (2 days).
*   **Anomaly:** A seven-day delay between approval and customer notification is operationally excessive. The high standard deviation indicates inconsistent application of this delay; some notifications occur quickly while others lag significantly. This points to a systemic bottleneck, queue backlog, or integration failure between the claims engine and the notification service.

#### 3. Premature or Incomplete Closures (Assign  Close)
*   **Pair:** `('A', 'C')`
*   **Metrics:** Average = 7,200 seconds (2 hours), Standard Deviation = 3,600 seconds (1 hour).
*   **Anomaly:** The average time from assignment to closure is only two hours. Given the required process steps (Evaluate, Approve, Notify), a two-hour duration implies that intermediate steps are either skipped entirely or executed instantaneously. The high variance suggests that while some claims close rapidly, others may follow a normal path, or the data includes erroneously closed claims.

#### 4. Suspiciously Fast Evaluation-to-Notification (Evaluate  Notify)
*   **Pair:** `('E', 'N')`
*   **Metrics:** Average = 300 seconds (5 minutes), Standard Deviation = 60 seconds (1 minute).
*   **Anomaly:** Transitioning from evaluation to notification in five minutes is unrealistically fast for a process that includes approval and customer communication. This suggests that the "Evaluate" event is being immediately followed by "Notify," potentially bypassing the "Approve" step, or that automated approvals trigger instant notifications without human review.

#### 5. Inconsistent Evaluation-to-Closure Timing (Evaluate  Close)
*   **Pair:** `('E', 'C')`
*   **Metrics:** Average = 3,600 seconds (1 hour), Standard Deviation = 3,000 seconds (50 minutes).
*   **Anomaly:** The standard deviation is very high relative to the average (ratio  0.83), indicating highly erratic behavior. ### Hypotheses for Anomalies

1.  **Automated Low-Risk Processing (RP Rigid):** Claims falling below a certain threshold or type may be routed to an automated system that schedules approval exactly 25 hours after receipt based on a fixed rule or batch job, ignoring actual claim content.
2.  **Notification System Backlog (PN Delay):** The notification module may have a queue processing delay. The high variance could result from periodic batch processing of notifications or intermittent failures in the email/SMS gateway that are retried later.
3.  **Process Bypass or System Errors (AC / EN / EC):** Adjusters or system scripts may be closing claims without completing the full workflow. This could occur if the UI allows "Close" as a shortcut, or if a script erroneously marks claims as closed immediately after assignment or evaluation to clear the workload.
4.  **Instant Auto-Approval (EN):** There may be a logic flaw where approved claims trigger the notification event immediately upon evaluation completion, skipping the approval timestamp or merging the steps.
5.  **Resource or Region Constraints (PN Delay):** The delay may be concentrated in specific regions or among specific adjusters who handle manual correspondence or lack access to automated notification tools.

### SQL Verification Queries

The following PostgreSQL queries are designed to verify the anomalies and investigate root causes. They assume `claim_events` contains the activity logs and `claims` provides metadata.

#### 1. Verify Rigid RP Timing and Identify Outliers
This query isolates claims where the time between Receive and Approve falls within the suspicious low-variance window, and checks if they correlate with specific claim types.

```sql
WITH r_p_gaps AS (
    SELECT 
        r.claim_id,
        r.timestamp AS r_timestamp,
        p.timestamp AS p_timestamp,
        EXTRACT(EPOCH FROM (p.timestamp - r.timestamp)) AS duration_sec,
        c.claim_type
    FROM claim_events r
    JOIN claim_events p ON r.claim_id = p.claim_id
    JOIN claims c ON r.claim_id = c.claim_id
    WHERE r.activity = 'R' 
      AND p.activity = 'P' 
      AND r.timestamp < p.timestamp
)
SELECT 
    claim_id,
    claim_type,
    duration_sec,
    CASE 
        WHEN duration_sec BETWEEN 86400 AND 93600 THEN 'Rigid Cluster'
        ELSE 'Normal'
    END AS status
FROM r_p_gaps
WHERE duration_sec BETWEEN 86400 AND 93600
ORDER BY claim_type, duration_sec;
```

#### 2. Investigate PN Delays by Claim Type and Region
This query identifies claims with excessive notification delays and joins with claim and adjuster data to correlate delays with customer segments or adjuster regions.

```sql
WITH p_n_gaps AS (
    SELECT 
        p.claim_id,
        p.timestamp AS p_timestamp,
        n.timestamp AS n_timestamp,
        EXTRACT(EPOCH FROM (n.timestamp - p.timestamp)) AS delay_sec,
        p.resource AS adjuster_name
    FROM claim_events p
    JOIN claim_events n ON p.claim_id = n.claim_id
    WHERE p.activity = 'P' 
      AND n.activity = 'N' 
      AND p.timestamp < n.timestamp
)
SELECT 
    g.claim_id,
    c.claim_type,
    c.customer_id,
    adj.region,
    g.delay_sec,
    g.adjuster_name
FROM p_n_gaps g
JOIN claims c ON g.claim_id = c.claim_id
JOIN adjusters adj ON g.adjuster_name = adj.name
WHERE g.delay_sec > 259200 -- Delays > 3 days
ORDER BY g.delay_sec DESC;
```

#### 3. Detect Premature Closures (Missing Steps)
This query finds claims that were assigned and closed within the anomalous 2-hour window but lack intermediate activities (Evaluate, Approve, Notify), indicating process violations.

```sql
SELECT 
    ce.claim_id,
    MIN(CASE WHEN ce.activity = 'A' THEN ce.timestamp END) AS assign_time,
    MIN(CASE WHEN ce.activity = 'C' THEN ce.timestamp END) AS close_time,
    COUNT(DISTINCT ce.activity) AS distinct_activities,
    STRING_AGG(DISTINCT ce.activity, ', ' ORDER BY ce.activity) AS activities_present
FROM claim_events ce
WHERE ce.claim_id IN (
    SELECT ce_inner.claim_id
    FROM claim_events ce_inner
    WHERE ce_inner.activity IN ('A', 'C')
    GROUP BY ce_inner.claim_id
    HAVING 
        -- Only 2 distinct activities (A and C)
        COUNT(DISTINCT ce_inner.activity) = 2
        -- Duration between A and C is within the anomalous 2-hour window
        AND EXTRACT(EPOCH FROM 
            MAX(CASE WHEN ce_inner.activity = 'C' THEN ce_inner.timestamp END) -
            MIN(CASE WHEN ce_inner.activity = 'A' THEN ce_inner.timestamp END)
        ) < 7200
)
GROUP BY ce.claim_id;
```

#### 4. Analyze Adjuster Performance on Evaluation-to-Notification
This query checks if specific adjusters are causing the rapid EN transitions, which may indicate individual workflow shortcuts or system misconfigurations tied to specific resources.

```sql
WITH e_n_gaps AS (
    SELECT 
        e.claim_id,
        e.resource AS adjuster_name,
        EXTRACT(EPOCH FROM (n.timestamp - e.timestamp)) AS duration_sec
    FROM claim_events e
    JOIN claim_events n ON e.claim_id = n.claim_id
    WHERE e.activity = 'E' 
      AND n.activity = 'N' 
      AND e.timestamp < n.timestamp
)
SELECT 
    adjuster_name,
    COUNT(*) AS claim_count,
    AVG(duration_sec) AS avg_duration_sec,
    MIN(duration_sec) AS min_duration_sec,
    MAX(duration_sec) AS max_duration_sec
FROM e_n_gaps
GROUP BY adjuster_name
HAVING AVG(duration_sec) < 600 -- Average < 10 minutes
ORDER BY avg_duration_sec ASC;
```

#### 5. Correlate EC Variance with Claim Amount
This query investigates if the high variance in EC timing is driven by claim amount, suggesting that high-value claims might be handled differently or that low-value claims are being auto-closed.

```sql
WITH e_c_gaps AS (
    SELECT 
        e.claim_id,
        EXTRACT(EPOCH FROM (c.timestamp - e.timestamp)) AS duration_sec
    FROM claim_events e
    JOIN claim_events c ON e.claim_id = c.claim_id
    WHERE e.activity = 'E' 
      AND c.activity = 'C' 
      AND e.timestamp < c.timestamp
)
SELECT 
    CASE 
        WHEN cl.claim_amount < 1000 THEN 'Low (< 1000)'
        WHEN cl.claim_amount < 5000 THEN 'Medium (1000-5000)'
        ELSE 'High (> 5000)'
    END AS amount_bucket,
    COUNT(*) AS claim_count,
    AVG(ecg.duration_sec) AS avg_duration_sec,
    STDDEV(ecg.duration_sec) AS stddev_duration_sec
FROM e_c_gaps ecg
JOIN claims cl ON ecg.claim_id = cl.claim_id
GROUP BY amount_bucket
ORDER BY amount_bucket;
```
