To solve this in DuckDB, we can utilize `string_agg` (or `list`) to define the variants, a Common Table Expression (CTE) to identify the top K, and a semi-join to filter the original data.

Here is the SQL query. You can replace `:K` with your desired integer.

```sql
WITH CaseVariants AS (
    -- 1 & 2: Construct the sequence per case and group by it
    SELECT 
        case_id,
        string_agg(activity, ' -> ' ORDER BY timestamp) AS variant
    FROM event_log
    GROUP BY case_id
),
VariantCounts AS (
    -- 3: Count cases per variant and identify top K
    SELECT 
        variant,
        COUNT(case_id) as case_count
    FROM CaseVariants
    GROUP BY variant
    ORDER BY case_count DESC
    LIMIT :K
),
TopKCaseIds AS (
    -- Get list of case_ids that belong to the top K variants
    SELECT cv.case_id
    FROM CaseVariants cv
    JOIN VariantCounts vc ON cv.variant = vc.variant
)
-- 4: Return all events belonging to the filtered cases
SELECT e.*
FROM event_log e
JOIN TopKCaseIds t ON e.case_id = t.case_id
ORDER BY e.case_id, e.timestamp;
```

### Explanation of the approach:

1.  **`CaseVariants` (CTE):** We group the data by `case_id` and use the `string_agg` aggregate function. By providing an `ORDER BY` clause inside the `string_agg`, DuckDB ensures the sequence is constructed chronologically according to the `timestamp`.
2.  **`VariantCounts` (CTE):** We aggregate the variants created in the previous step to calculate the frequency of each unique sequence. We then use `LIMIT :K` to keep only the most frequent ones.
3.  **`TopKCaseIds` (CTE):** This acts as a filter mapping. We join the variant list back to the specific `case_ids` that constitute those variants.
4.  **Final `SELECT`:** We perform an inner join between the original `event_log` and the `TopKCaseIds`. This effectively filters the original log, discarding any events that do not belong to one of the top K process paths.

**Performance Note:** 
If your `event_log` table is extremely large, DuckDB's columnar nature will handle the grouping and string aggregation very efficiently. If you are dealing with millions of cases, ensure there is an index on `case_id` or that the data is sorted by `case_id` to speed up the final join.
