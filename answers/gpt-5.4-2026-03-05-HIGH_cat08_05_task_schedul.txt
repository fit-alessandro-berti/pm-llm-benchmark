## 1. Analyzing Historical Scheduling Performance and Dynamics

### 1.1 Build the right process-mining view first
For Precision Parts, I would not rely on a single “case = job” log only. The MES data should be transformed into a **multi-perspective / object-centric event log** because the same events involve:

- a **job**,
- a **task/operation**,
- a **machine**,
- often an **operator**,
- and sometimes a **disruption object** (breakdown, rush-order event).

That enables three complementary views:

1. **Job-centric traces**  
   Sequence of operations each job actually followed:
   `Release  Queue Cutting  Setup Cutting  Cutting  Queue Milling  ...  Inspection  Completion`

2. **Machine-centric traces**  
   Sequence of jobs processed on each machine:
   `JOB-6998  setup  JOB-7001  ...` on `CUT-01`

3. **Operator-centric traces**  
   Useful where setup time or process time depends materially on operator skill or shift.

This is important because the scheduling problem is not only “what path did the job follow?” but also “in what order did the machine process jobs?” and “what setup penalty did that sequence create?”

---

### 1.2 Reconstruct the actual shop-floor flow
I would use process discovery plus time-performance analysis to reconstruct reality.

#### Techniques
- **Inductive Miner / Heuristics Miner / directly-follows graphs**
  - On **job traces** to identify actual routing variants, rework loops, skipped steps, atypical paths.
- **Object-centric process mining**
  - To connect job progression with machine state changes and operator assignment.
- **Alignment-based conformance checking**
  - Compare actual execution against:
    - planned route sheets,
    - standard operation sequence,
    - target production plan / scheduled machine sequence if available.

#### Outputs
- Real routing maps by job family / customer / priority class
- Frequency of deviations from the nominal route
- Actual interleaving of jobs on shared machines
- Visualization of where jobs spend time: queue, setup, processing, waiting for downstream capacity

---

### 1.3 Quantify lead times, flow times, and makespan
#### Job-level metrics
For each job \(j\):

- **Lead time**  
  \[
  LT_j = CompletionTime_j - ReleaseTime_j
  \]

- **Operation flow time** for operation \(o\)  
  \[
  FT_{j,o} = TaskEnd_{j,o} - QueueEntry_{j,o}
  \]

- **Pure processing time**  
  \[
  PT_{j,o} = TaskEnd_{j,o} - TaskStart_{j,o}
  \]

- **Setup time**  
  \[
  ST_{j,o} = SetupEnd_{j,o} - SetupStart_{j,o}
  \]

- **Queue / waiting time**  
  \[
  WT_{j,o} = TaskStart_{j,o} - QueueEntry_{j,o}
  \]
  or, if setup is separate and resource reserved before processing:
  \[
  WT_{j,o} = SetupStart_{j,o} - QueueEntry_{j,o}
  \]

#### Shop-level metrics
- **Makespan** for a planning horizon \(H\):  
  \[
  Makespan_H = \max(CompletionTime) - \min(ReleaseTime)
  \]
- Distribution summaries:
  - mean / median,
  - percentiles (P50, P80, P95),
  - tail behavior for very late jobs,
  - by product family, priority, routing complexity, customer segment.

Process mining tools can overlay these times directly on the discovered model, so each activity or path shows median wait, median process time, and variability.

---

### 1.4 Quantify waiting times and queueing dynamics
The event log contains queue entry, setup start, and task start timestamps, so queueing can be measured precisely.

#### Metrics by machine / work center
- Average queue time
- P90/P95 queue time
- Queue length over time
- Queue aging distribution
- Number of jobs waiting > X hours/days
- Time spent waiting as % of total lead time

#### Useful analyses
- **Time-in-state analysis**:
  - waiting,
  - setup,
  - processing,
  - blocked after completion,
  - interrupted by breakdown.
- **Queue heatmaps by hour/day/shift**
- **WIP profiles over time** at each work center
- **Little’s Law sanity checks**:
  \[
  WIP \approx Throughput \times LeadTime
  \]

This helps distinguish chronic overload from poor local sequencing.

---

### 1.5 Quantify machine and operator utilization
For each machine \(m\), partition calendar time into:

- **Productive processing time**
- **Setup time**
- **Idle/starved time**: available but no job to process
- **Blocked time**: cannot proceed due to downstream issue, if relevant
- **Downtime**: breakdown / maintenance
- **Planned unavailable time**: off-shift, preventive maintenance

Then calculate:

- **Utilization**
  \[
  U_m = \frac{Processing + Setup}{AvailableTime}
  \]

- **Value-added utilization**
  \[
  VAU_m = \frac{Processing}{AvailableTime}
  \]

- **Setup ratio**
  \[
  SetupRatio_m = \frac{Setup}{Processing + Setup}
  \]

- **Downtime ratio**
  \[
  DowntimeRatio_m = \frac{BreakdownTime}{AvailableTime}
  \]

For operators, do similar analysis:
- operator load by shift,
- setup efficiency by operator,
- operator-machine eligibility constraints,
- impact of staffing shortages on bottlenecks.

A useful process-mining enhancement is **resource-performance profiling**:
- same task, same machine family, different operator  different actual duration distributions.

---

### 1.6 Analyze sequence-dependent setup times
This is a critical point in your scenario.

The machine-centric log should be converted into **transition records**:

For machine \(m\), whenever job \(i\) is followed by job \(j\), create a record:
\[
(i \rightarrow j, m, setup\_duration, attributes_i, attributes_j)
\]

Where attributes may include:
- material type,
- diameter/size range,
- tooling family,
- tolerance class,
- coating/heat-treatment requirement,
- fixture family,
- operator,
- shift.

Then estimate:

#### A. Transition setup matrix
For each machine or machine family:
\[
S_m(a,b) = E[setup\ time \mid previous\ family=a, next\ family=b]
\]

This gives a practical **setup-changeover matrix**.

#### B. Setup regression / predictive model
If job attributes are rich, model:
\[
SetupTime = f(previous\_job\_attributes,\ current\_job\_attributes,\ machine,\ operator,\ shift)
\]
using:
- gradient boosting,
- random forest,
- GAM,
- or even a simpler interpretable linear model.

#### C. Sequence inefficiency metrics
- actual cumulative setup hours per machine
- minimum feasible setup hours for the observed job set under optimized sequencing
- excess setup due to poor sequencing:
  \[
  ExcessSetup = ActualSetup - BestKnownSetup
  \]

This is often where job shops lose major capacity at bottlenecks.

---

### 1.7 Measure schedule adherence and tardiness
At the job level:

- **Lateness**
  \[
  L_j = Completion_j - DueDate_j
  \]
- **Tardiness**
  \[
  T_j = \max(0, L_j)
  \]
- **Earliness**
  \[
  E_j = \max(0, DueDate_j - Completion_j)
  \]

Important KPI variants:
- % jobs on time
- weighted tardiness by priority
- tardiness days by customer
- P95 tardiness
- urgent-job on-time rate

At operation level:
- deviation from planned start / finish if production schedule exists
- deviation from standard routing times
- **slack erosion** through the route:
  \[
  Slack_{j,t} = DueDate_j - t - RemainingExpectedWork_j
  \]

Using process mining, I would create **late-vs-on-time trace cohorts** and compare:
- where late jobs first diverge,
- which queues consume most slack,
- whether late jobs suffer more setup-related delay,
- whether hot jobs improve at the expense of others.

---

### 1.8 Measure disruption impact
The log explicitly includes:
- breakdown start/end,
- priority changes,
- hot-job release,
- possibly rework or maintenance.

I would do **event-centered impact analysis**:

#### Around breakdowns
For each breakdown event:
- queue growth before/after
- lost capacity on affected machine
- spillover to alternate machines
- increase in tardy jobs exposed to that machine
- recovery time to normal queue length

#### Around hot-job arrivals / priority changes
Measure:
- jobs displaced,
- additional setups caused,
- downstream ripple effects,
- increase in weighted tardiness of non-urgent work,
- schedule stability loss.

A good method is **matched comparison**:
- compare similar time windows with and without disruption,
- control for load level and mix.

This distinguishes true disruption impact from normal congestion.

## 2. Diagnosing Scheduling Pathologies

The goal is not just to say “performance is bad,” but to show *why*.

### 2.1 Bottleneck identification and impact quantification
Bottlenecks in a job shop can be **shifting**, not fixed. So I would use dynamic bottleneck analysis.

#### Evidence of a bottleneck machine
A resource is a likely bottleneck if it shows:
- very high busy + setup utilization,
- consistently long queues,
- queue time dominating upstream/downstream times,
- high number of tardy jobs passing through it,
- low starvation and high contention,
- throughput strongly correlated with its uptime.

#### Process-mining methods
- performance-annotated process map with queue times
- machine-state timeline analysis
- WIP accumulation before machine
- sensitivity of job lateness to waiting at that machine
- “criticality index”: % of late jobs that passed through / waited there

#### Quantifying impact
For machine \(m\):
- additional tardiness attributable to waiting at \(m\)
- marginal throughput gain if setup time or downtime at \(m\) is reduced
- fraction of total shop-floor WIP sitting upstream of \(m\)

---

### 2.2 Poor prioritization and dispatching failures
Local FCFS/EDD frequently fails in dynamic job shops because it ignores:
- remaining route length,
- bottleneck exposure,
- setup penalties,
- hot-job preemption effects,
- downstream congestion,
- stochastic duration.

#### Evidence to mine
- **priority inversions**: high-priority jobs queued behind lower-priority jobs without setup or feasibility justification
- jobs with low slack waiting behind jobs with ample slack
- urgent jobs receiving local priority but then blocking the entire flow downstream
- non-urgent jobs excessively expedited due to queue arrival order

#### Analytical approach
For each dispatch decision, reconstruct candidate queue and compute:
- actual chosen job,
- alternative jobs available,
- their due-date slack,
- predicted setup penalty,
- downstream bottleneck load.

Then quantify:
- how often actual dispatch differed from a better composite rule,
- expected tardiness loss from those choices.

This is very powerful because it turns vague criticism into evidence.

---

### 2.3 Suboptimal sequencing causing excess setup
This is often hidden under “busy machine” symptoms.

#### Evidence
- many short campaigns with frequent family switching
- actual machine job sequence causing high changeover time
- large setup overruns for specific transitions
- hot-job insertions splitting otherwise efficient sequences

#### Process-mining techniques
- machine-centric trace mining
- transition matrix analysis
- setup-pattern clustering
- comparison of actual sequence against:
  - due-date-only sequence,
  - family-batched sequence,
  - optimized sequence over same job set.

#### Example pathology
At a bottleneck lathe, jobs might alternate material/fixture families:
A  C  A  D  A  
instead of a tighter sequence:
A  A  A  C  D

The first pattern inflates setup hours, which further increases queueing and tardiness.

---

### 2.4 Downstream starvation and poor work-center coordination
Local rules can cause one center to maximize its own output while starving or flooding the next.

#### Symptoms
- downstream machines idle despite plenty of overall WIP
- upstream centers producing jobs not yet needed
- large queues before non-bottlenecks while true constraint waits on a feeder part
- poor synchronization for assembly/inspection convergence points

#### Process-mining evidence
- cross-resource state dependency:
  - downstream idle time correlated with late arrivals from specific upstream resource
- queue handoff analysis
- “time between predecessor completion and successor start”
- resource interaction graphs

This reveals whether work is being released and sequenced in a globally useful order.

---

### 2.5 WIP amplification / bullwhip-like scheduling behavior
In unstable shops, expediting and local rescheduling create oscillations:
- sudden release surges,
- queue explosions,
- then starvation after cleanup,
- then another expedite wave.

#### Evidence
- highly volatile WIP profiles by day/shift
- strong oscillation in queue lengths
- frequent priority changes
- increased setup fragmentation following rush interventions
- lead-time volatility disproportionate to demand volatility

#### Methods
- time-series analysis of WIP, queue length, release rate
- autocorrelation / cross-correlation between hot-job injections and queue spikes
- compare stable vs unstable periods through process variants

This gives hard evidence that the current policy is amplifying variability rather than absorbing it.

---

### 2.6 Comparing on-time vs late jobs through variant analysis
A very effective process-mining method here is to create cohorts:
- on-time jobs,
- late jobs,
- severely late jobs,
- urgent jobs,
- jobs touched by breakdowns,
- jobs requiring high-changeover paths.

Then compare them on:
- route variants,
- waiting-time profiles,
- number of priority changes,
- number of setups encountered,
- exposure to bottleneck machines,
- planned vs actual duration bias.

This often shows, for example:
- late jobs are not necessarily longer jobs,
- they are jobs that waited too long at 1–2 key bottlenecks,
- or jobs inserted during high setup fragmentation periods.

## 3. Root Cause Analysis of Scheduling Ineffectiveness

### 3.1 Static local dispatching rules are structurally inadequate
FCFS and pure EDD are attractive because they are simple, but in this environment they ignore essential realities:

- **sequence-dependent setups**
- **different routing lengths**
- **different bottleneck exposure**
- **operator constraints**
- **breakdowns**
- **hot-job arrivals**
- **downstream queue conditions**
- **actual stochastic processing-time behavior**

So even if every work center is “following the rule,” the plant can still perform badly because each local decision is myopic.

---

### 3.2 Lack of real-time visibility
If dispatching decisions are made without current knowledge of:
- machine uptime state,
- queue lengths,
- downstream congestion,
- current slack to due date,
- actual job progress,
- setup implications of the current machine state,

then priorities will be wrong in a large fraction of situations.

Process mining helps by reconstructing these state snapshots and showing what information was missing at each decision point.

---

### 3.3 Inaccurate duration and setup assumptions
A common root cause is that planned standards are not representative.

I would compare planned vs actual by:
- machine,
- operator,
- task type,
- material family,
- shift,
- job complexity class.

Likely findings:
- planned cutting times are unbiased, but milling is systematically underestimated
- setups after certain material/tool transitions are much longer than assumed
- breakdown-prone machines create hidden delay distributions
- urgent jobs disrupt normal setup campaigns and inflate actual durations further

If scheduling uses wrong time assumptions, even “correct” logic will generate infeasible schedules.

---

### 3.4 Poor handling of sequence-dependent setups
Many shops treat setup as a fixed constant or ignore it in dispatching. That is disastrous when setups are sequence-dependent.

Root causes may include:
- no setup-family coding in the scheduling logic,
- no transition-time matrix,
- no machine-sequence optimization at bottlenecks,
- hot-job insertion without quantifying setup penalty.

Process mining can prove this by showing:
- changeover patterns,
- excess transitions,
- setup hours lost specifically because of sequence choices.

---

### 3.5 Weak coordination across work centers
The current approach is likely optimizing each local queue independently. That causes:
- upstream overproduction into already congested downstream queues,
- feeder starvation of bottleneck machines,
- poor synchronization of long-route jobs,
- high WIP with no lead-time benefit.

This is a classic local-optimum/global-suboptimum problem.

---

### 3.6 Reactive disruption handling is too crude
Breakdowns and urgent orders are unavoidable; the weakness is in the response.

Likely issues:
- no event-triggered rescheduling,
- rush jobs force manual overrides,
- no protected capacity buffer at bottlenecks,
- no “frozen zone” vs “re-optimizable zone,” causing instability,
- no estimate of disruption propagation.

This results in frequent rescheduling noise with little actual improvement.

---

### 3.7 Distinguishing bad scheduling logic vs true capacity limitation vs natural variability
This distinction is critical. Process mining alone identifies patterns; combining it with controlled analysis tells you the cause.

#### A. Signs of bad scheduling logic
- high priority inversions
- high avoidable setup transitions
- downstream starvation despite available WIP
- high performance variation under similar load
- late jobs in periods where utilization is not near saturation
- substantial gap between actual setup hours and best-known sequencing

#### B. Signs of genuine capacity shortage
- one or more machines operate near practical maximum most of the time
- persistent queue buildup even under near-optimal sequencing
- lateness remains high in simulation even when scheduling is improved
- capacity demand on constraint consistently exceeds available effective hours

#### C. Signs of inherent variability dominating
- large coefficient of variation in process times or repair times
- lateness clustered around breakdown-heavy windows
- similar schedules produce different outcomes because of stochastic spread
- unstable routing/rework behavior

#### How to separate them
1. **Load-conditioned analysis**  
   Compare performance under similar demand/load levels.
2. **Counterfactual sequencing analysis**  
   Re-run historical queue states with better dispatch rules.
3. **Simulation under same capacity**  
   If performance improves materially, logic was part of the problem.
4. **Best-attainable envelope**  
   Estimate what performance is possible with current assets and realistic variability.

This avoids the common mistake of blaming scheduling for what is actually a structural capacity constraint—or vice versa.

## 4. Developing Advanced Data-Driven Scheduling Strategies

Below are three practical strategies informed directly by the mined evidence.

---

### Strategy 1: Dynamic, setup-aware, bottleneck-aware dispatching

#### Core logic
Replace static FCFS/EDD with a **dynamic composite dispatch score** recalculated whenever:
- a machine becomes free,
- a hot job arrives,
- a breakdown occurs,
- a priority changes.

For each candidate job \(j\) at machine \(m\), compute a score such as:

\[
Score_{j,m} =
w_1 \cdot PriorityWeight_j
+ w_2 \cdot Urgency_j
+ w_3 \cdot CriticalRatio_j
+ w_4 \cdot DownstreamNeed_j
- w_5 \cdot SetupPenalty_{prev\rightarrow j,m}
- w_6 \cdot PredictedTardinessRisk_j
+ w_7 \cdot Aging_j
\]

Possible terms:
- **PriorityWeight**: explicit business priority
- **Urgency / slack**:
  \[
  Slack_j = DueDate_j - now - RemainingExpectedWork_j
  \]
- **Critical ratio**:
  \[
  CR_j = \frac{DueDate_j - now}{RemainingExpectedWork_j}
  \]
- **DownstreamNeed**: reward jobs feeding starved bottlenecks or critical downstream steps
- **SetupPenalty**: estimated sequence-dependent changeover from current machine state
- **Aging**: prevents permanent neglect of long-waiting jobs

At bottlenecks, the score can be biased more heavily toward setup efficiency and global tardiness impact; at non-bottlenecks, more weight can be given to due-date slack and starvation prevention.

#### How process mining informs it
From the logs we derive:
- actual tardiness drivers,
- remaining-route risk patterns,
- setup transition matrix,
- downstream starvation relationships,
- realistic task-time distributions,
- machine-specific bottleneck criticality.

Weights can be tuned using:
- historical replay,
- simulation-based optimization,
- or supervised learning to predict which dispatch choice minimized future weighted tardiness.

#### Pathologies addressed
- poor prioritization
- high tardiness
- excessive setups
- lack of coordination between work centers
- starvation at critical downstream resources

#### Expected KPI impact
- lower weighted tardiness
- lower mean/P95 lead time
- fewer urgent-job failures
- reduced setup hours at bottlenecks
- better utilization of constrained resources
- lower WIP due to improved flow discipline

---

### Strategy 2: Predictive rolling-horizon scheduling with event-triggered rescheduling

#### Core logic
Instead of purely local dispatching, implement a **short-horizon predictive scheduler** that operates on a rolling basis.

Example cadence:
- every 30–60 minutes, or
- immediately after major events:
  - breakdown,
  - hot-job release,
  - priority change,
  - severe duration overrun.

The scheduler forecasts:
- expected completion times,
- tardiness risk,
- future queue growth,
- breakdown exposure,
- bottleneck congestion.

Then it generates a near-term schedule for key resources, especially bottlenecks, using:
- MILP / CP-SAT / heuristic optimization,
- robust buffers for stochastic durations,
- frozen near-term zone to avoid excessive schedule churn.

#### Process-mining inputs
- empirical task duration distributions by machine/operator/job family
- remaining-time prediction models
- setup-time prediction models
- routing probabilities / variant likelihoods
- breakdown frequency and repair-time distributions
- rework probability if present
- hot-job arrival patterns

A useful implementation is:
- **optimize the next 8–24 hours** for bottleneck resources,
- use dynamic dispatching for the rest,
- re-optimize when disruption occurs.

#### Predictive elements
- probability a job becomes tardy if not started within X hours
- predicted future queue at each constraint
- machine breakdown hazard based on recent history / condition signals if available
- ETA confidence intervals for customer commitment

#### Pathologies addressed
- unrealistic schedules based on average times
- poor reaction to disruptions
- missed due dates caused by hidden risk accumulation
- inability to provide reliable lead-time estimates

#### Expected KPI impact
- better schedule realism
- fewer surprises and less firefighting
- lower tardiness under disruptions
- improved due-date promise accuracy
- smoother queue profiles and lower lead-time variability

---

### Strategy 3: Sequence-dependent setup optimization through family-based campaigning

#### Core logic
For machines where setups matter most, especially bottlenecks, explicitly optimize sequence to reduce total changeover burden without sacrificing due dates excessively.

This strategy has two layers:

#### Layer A: Create setup families
Using historical setup analysis, cluster jobs into families based on similar setup requirements:
- material/tooling family,
- fixture family,
- diameter range,
- tolerance class,
- coating or heat-treatment class.

#### Layer B: Sequence jobs intelligently
At each bottleneck machine:
1. form a candidate set of jobs within a time fence,
2. estimate pairwise setup costs,
3. solve a sequencing problem that minimizes:
   \[
   \alpha \cdot TotalSetupTime + \beta \cdot WeightedTardiness
   \]
4. within a family campaign, dispatch mostly by due-date urgency.

This can be done with:
- ATCS-like rules,
- sequence-dependent single-machine optimization,
- tabu search / genetic algorithm / CP-SAT for short queues,
- or simple but effective campaign rules with caps.

#### Important design choice
Do **not** blindly batch by family, because that can hurt urgent due dates. Use a controlled policy:
- batch when due-date slack permits,
- break campaigns when tardiness risk crosses threshold,
- reserve limited “expedite slots” to absorb hot jobs.

#### How process mining informs it
- actual transition matrix
- family similarity definitions from setup behavior, not intuition alone
- identification of machines where setup compression matters most
- quantification of historical excess setup

#### Pathologies addressed
- suboptimal sequencing
- bottleneck overload caused by frequent changeovers
- WIP buildup upstream of setup-heavy resources
- instability caused by hot-job insertions

#### Expected KPI impact
- lower total setup time
- higher effective capacity on bottlenecks
- shorter queues before setup-heavy machines
- better throughput and lower tardiness
- more stable schedules

---

## Optional enhancement worth strong consideration: Bottleneck-driven release control
Although you asked for three strategies, I would strongly recommend adding a **WIP release policy** such as workload control / CONWIP / drum-buffer-rope logic.

Why: if too much work is released into the shop, better dispatching alone may not solve high WIP and lead-time volatility.

Core idea:
- release jobs based on bottleneck workload and due-date windows,
- keep a controlled queue before the constraint,
- avoid flooding the floor with work that cannot be processed soon.

This directly attacks:
- excessive WIP,
- lead-time unpredictability,
- bullwhip-like oscillation.

## 5. Simulation, Evaluation, and Continuous Improvement

### 5.1 Use discrete-event simulation before deployment
Before changing live scheduling, build a **discrete-event simulation (DES)** of the job shop. This becomes the testbed for strategy comparison.

### 5.2 Parameterize the simulation from process-mining results
Use mined data to populate the model with real behavior, not textbook assumptions:

#### Job arrivals and mix
- release patterns by day/shift
- job family mix
- due-date tightness distribution
- hot-job arrival frequency

#### Routing behavior
- actual routing variants by product family
- rework loops if present
- optional operations

#### Processing and setup times
- task duration distributions conditional on:
  - machine,
  - operator,
  - job family,
  - complexity,
  - shift
- sequence-dependent setup-time matrix or predictive model

#### Resource calendars
- machine availability calendars
- operator shifts and skill constraints
- planned downtime

#### Failures
- breakdown frequency (MTBF)
- repair time distributions (MTTR)
- resource-specific failure patterns

#### Dispatch and rescheduling logic
- baseline FCFS/EDD
- Strategy 1
- Strategy 2
- Strategy 3
- hybrids of the above

---

### 5.3 What scenarios to test
At minimum, test:

1. **Baseline average load**
2. **High-load period**  
   +15% / +25% release rate
3. **Frequent hot-job environment**
4. **Breakdown-heavy scenario**  
   especially on bottleneck machines
5. **Product-mix shift**  
   more setup-intensive jobs
6. **Tight due-date scenario**
7. **Operator shortage / skill loss**
8. **Long-tail processing-time variability scenario**

Also test mixed scenarios, because that is often where simple rules fail hardest.

---

### 5.4 How to compare strategies rigorously
For each strategy, evaluate:

- on-time delivery %
- weighted tardiness
- mean and P95 lead time
- total WIP and WIP by work center
- makespan for planning windows
- bottleneck queue time
- total setup hours
- machine utilization split:
  - processing,
  - setup,
  - downtime,
  - idle
- schedule stability:
  - number of resequencing actions,
  - number of jobs materially rescheduled,
  - nervousness due to hot jobs
- fairness / collateral damage to non-hot jobs

Use:
- multiple replications,
- common random numbers across strategies,
- confidence intervals,
- Pareto comparison (e.g., tardiness vs WIP vs schedule stability).

The best strategy is rarely the one with the lowest tardiness alone; it is the best trade-off.

---

### 5.5 Continuous monitoring and adaptation framework
Once a strategy is selected, process mining should remain in the loop continuously.

#### A. Live KPI monitoring
Dashboards refreshed daily or intra-shift for:
- on-time delivery
- weighted tardiness
- queue time per machine
- WIP by work center
- setup ratio at bottlenecks
- breakdown impact
- urgent-job service level
- schedule adherence

#### B. Online conformance monitoring
Compare actual decisions to intended scheduling policy:
- Was the recommended dispatch followed?
- If not, why?
- Are manual overrides improving or degrading performance?

This is crucial because execution drift can quietly undo the strategy.

#### C. Drift detection
Automatically detect changes in:
- processing time distributions
- setup transition times
- routing variants
- machine reliability
- hot-job frequency
- queueing structure

If, for example, setup times for a machine family suddenly rise, the setup model must be recalibrated.

#### D. Trigger-based model refresh
Re-estimate scheduling parameters when:
- tardiness exceeds threshold,
- bottleneck queue exceeds threshold,
- machine reliability degrades,
- product mix shifts materially.

#### E. Periodic strategy retuning
Monthly or quarterly:
- rerun historical replay,
- retrain predictive models,
- retune dispatch weights,
- update setup families,
- resimulate with recent data.

This creates a closed-loop improvement system rather than a one-time scheduling redesign.
