Anomalies in the temporal profile

1) Receive  Approve (R  P)  
Average  25 hours with very small STDEV (1 hour). For a process that normally includes Assign and Evaluate before approval, such tight timing is suspicious.

Possible reasons  
- Approvals might be executed in scheduled batches (e.g., daily automated approval job).  
- Some approvals could be pre-authorized by policy rules and triggered automatically at fixed times.  
- Event timestamps might be system-generated rather than reflecting the true execution time.  
- Missing intermediate events (A or E) could make approvals appear uniformly timed from reception.

Verification queries

Identify claims where approval happens without earlier evaluation or assignment:

SELECT p.claim_id
FROM claim_events p
LEFT JOIN claim_events e
  ON p.claim_id = e.claim_id AND e.activity = 'E'
LEFT JOIN claim_events a
  ON p.claim_id = a.claim_id AND a.activity = 'A'
WHERE p.activity = 'P'
AND (e.event_id IS NULL OR a.event_id IS NULL);

Check distribution of time between Receive and Approve:

SELECT
  percentile_cont(0.5) WITHIN GROUP (ORDER BY EXTRACT(EPOCH FROM (p.timestamp - r.timestamp))) AS median_seconds,
  MIN(EXTRACT(EPOCH FROM (p.timestamp - r.timestamp))) AS min_seconds,
  MAX(EXTRACT(EPOCH FROM (p.timestamp - r.timestamp))) AS max_seconds
FROM claim_events r
JOIN claim_events p
  ON r.claim_id = p.claim_id
WHERE r.activity = 'R'
AND p.activity = 'P';

Detect clustering of approvals around specific hours (possible batch job):

SELECT EXTRACT(HOUR FROM timestamp) AS hour_of_day, COUNT(*)
FROM claim_events
WHERE activity = 'P'
GROUP BY hour_of_day
ORDER BY hour_of_day;


2) Approve  Notify (P  N)  
Average  7 days with STDEV  2 days. This is unusually long compared with the rest of the process.

Possible reasons  
- Notification might depend on manual processing or document preparation.  
- Notifications may be queued and sent in weekly batches.  
- Communication may rely on external systems or mailing processes.  
- Some claims may be approved but intentionally delayed due to compliance or payment checks.

Verification queries

Measure the time between approval and notification for each claim:

SELECT
  p.claim_id,
  EXTRACT(EPOCH FROM (n.timestamp - p.timestamp)) AS seconds_delay
FROM claim_events p
JOIN claim_events n
  ON p.claim_id = n.claim_id
WHERE p.activity = 'P'
AND n.activity = 'N'
ORDER BY seconds_delay DESC;

Identify claims with extremely long delays (e.g., >10 days):

SELECT
  p.claim_id,
  p.timestamp AS approval_time,
  n.timestamp AS notify_time
FROM claim_events p
JOIN claim_events n
  ON p.claim_id = n.claim_id
WHERE p.activity = 'P'
AND n.activity = 'N'
AND n.timestamp - p.timestamp > INTERVAL '10 days';

Check if delays correlate with claim type:

SELECT
  c.claim_type,
  AVG(n.timestamp - p.timestamp) AS avg_delay
FROM claim_events p
JOIN claim_events n ON p.claim_id = n.claim_id
JOIN claims c ON c.claim_id = p.claim_id
WHERE p.activity = 'P'
AND n.activity = 'N'
GROUP BY c.claim_type;


3) Assign  Close (A  C)  
Average  2 hours. This suggests claims are sometimes closed shortly after assignment, possibly skipping evaluation or approval.

Possible reasons  
- Claims might be rejected immediately after assignment.  
- Adjusters may close duplicate or invalid claims without evaluation.  
- Missing event logging could hide intermediate activities.  
- Automated rules may close claims instantly if they fail eligibility checks.

Verification queries

Find claims closed after assignment without evaluation or approval:

SELECT a.claim_id
FROM claim_events a
JOIN claim_events c
  ON a.claim_id = c.claim_id
LEFT JOIN claim_events e
  ON a.claim_id = e.claim_id AND e.activity = 'E'
LEFT JOIN claim_events p
  ON a.claim_id = p.claim_id AND p.activity = 'P'
WHERE a.activity = 'A'
AND c.activity = 'C'
AND e.event_id IS NULL
AND p.event_id IS NULL;

Check which resources close claims quickly:

SELECT
  c.resource,
  COUNT(*) AS quick_closures
FROM claim_events a
JOIN claim_events c
  ON a.claim_id = c.claim_id
WHERE a.activity = 'A'
AND c.activity = 'C'
AND c.timestamp - a.timestamp < INTERVAL '3 hours'
GROUP BY c.resource
ORDER BY quick_closures DESC;


4) Evaluate  Notify (E  N)  
Average  5 minutes with very small STDEV. This suggests that notification often happens immediately after evaluation.

Possible reasons  
- Notification may be automatically triggered by the evaluation system.  
- Some claims might skip approval if evaluation directly determines the outcome.  
- Evaluation may include implicit approval logic not recorded as a separate event.  
- Notification events could be system-generated as soon as evaluation completes.

Verification queries

Check whether approval events are missing between evaluation and notification:

SELECT e.claim_id
FROM claim_events e
JOIN claim_events n
  ON e.claim_id = n.claim_id
LEFT JOIN claim_events p
  ON e.claim_id = p.claim_id AND p.activity = 'P'
WHERE e.activity = 'E'
AND n.activity = 'N'
AND p.event_id IS NULL;

Measure actual delays between evaluation and notification:

SELECT
  EXTRACT(EPOCH FROM (n.timestamp - e.timestamp)) AS seconds_delay
FROM claim_events e
JOIN claim_events n
  ON e.claim_id = n.claim_id
WHERE e.activity = 'E'
AND n.activity = 'N'
ORDER BY seconds_delay;

Analyze whether certain resources trigger immediate notifications:

SELECT
  e.resource,
  AVG(n.timestamp - e.timestamp) AS avg_delay
FROM claim_events e
JOIN claim_events n
  ON e.claim_id = n.claim_id
WHERE e.activity = 'E'
AND n.activity = 'N'
GROUP BY e.resource
ORDER BY avg_delay;
