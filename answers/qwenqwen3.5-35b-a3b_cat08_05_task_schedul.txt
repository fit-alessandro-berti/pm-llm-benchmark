# Strategic Optimization Plan for Precision Parts Inc.

**To:** Management Team, Precision Parts Inc.
**From:** Senior Operations Analyst
**Date:** October 26, 2023
**Subject:** Leveraging Process Mining for Advanced Scheduling Optimization

## 1. Analyzing Historical Scheduling Performance and Dynamics

To transition from reactive dispatching to proactive optimization, we must first establish a "Single Source of Truth" regarding shop floor reality using the available MES event logs. Process mining will serve as the diagnostic engine to reconstruct the actual process flow, distinct from the theoretical routing.

### Process Mining Techniques and Reconstruction
We will ingest the MES logs into a process mining tool (e.g., Celonis, Disco, or Python-based PM4Py) to construct a **Process Map**.
*   **Case Reconstruction:** The `Case ID` (Job ID) will be used to link all events chronologically. Since jobs move through multiple machines, we must ensure the `Activity/Task` field correctly identifies the operation (e.g., "Cutting", "Milling").
*   **Event Filtering:** We will filter for specific event types (`Task Start`, `Task End`, `Setup Start`, `Setup End`, `Resource Breakdown`) to calculate duration metrics accurately.

### Key Metrics and Analysis
*   **Job Flow Times & Makespan:**
    *   *Technique:* Calculate the duration from the first `Task Start` to the last `Task End` per `Case ID`.
    *   *Metric:* Distribution of Lead Time (e.g., P50, P90, P95). We will visualize this using a **Boxplot** to identify outliers where jobs take significantly longer than the median.
*   **Task Waiting Times (Queue Times):**
    *   *Technique:* Calculate the delta between `Task End` of the previous operation and `Task Start` of the current operation for the same `Case ID`.
    *   *Metric:* Average Queue Time per Work Center. High variance here indicates instability in upstream delivery.
*   **Resource Utilization:**
    *   *Technique:* Aggregate `Task Duration (Actual)` and `Setup Duration (Actual)` against the total shift time for each `Resource (Machine ID)`.
    *   *Metric:* Utilization % = (Active Time + Setup Time) / Available Time. Crucially, we will distinguish between **Value-Add Time** and **Non-Value-Add Time** (Setup/Wait).
*   **Sequence-Dependent Setup Times:**
    *   *Technique:* Extract the `Previous Job` attribute from the log (as seen in the snippet: `JOB-7001` follows `JOB-6998`). We will build a **Setup Transition Matrix**.
    *   *Metric:* Average Setup Time for specific transitions (e.g., Job Type A $\to$ Job Type B). This reveals if switching between specific part geometries incurs a penalty.
*   **Schedule Adherence & Tardiness:**
    *   *Technique:* Compare `Task End` timestamp against `Order Due Date`.
    *   *Metric:* Tardiness % = (Jobs completed after Due Date / Total Jobs). We will also calculate **Mean Absolute Deviation (MAD)** of completion times to measure predictability.
*   **Impact of Disruptions:**
    *   *Technique:* Correlate `Resource Breakdown Start` events with the subsequent `Task Start` times of jobs assigned to that machine.
    *   *Metric:* Delay propagation factor (e.g., 1 hour breakdown $\to$ X hours of downstream delay).

## 2. Diagnosing Scheduling Pathologies

Based on the metrics above, we anticipate identifying specific "pathologies" where the current system fails. Process mining allows us to move from anecdotal complaints to data-backed evidence.

### Identified Pathologies
1.  **The "Bottleneck Starvation" Phenomenon:**
    *   *Evidence:* High utilization on the bottleneck machine (e.g., CNC Milling) combined with high queue times *before* the bottleneck, but low utilization *after* it.
    *   *Mining Evidence:* **Bottleneck Analysis** will show that the `Resource` with the highest queue wait time correlates with the longest overall lead time.
2.  **Prioritization Failure:**
    *   *Evidence:* High-priority "Hot Jobs" are waiting behind Medium-priority jobs that arrived earlier (FCFS bias).
    *   *Mining Evidence:* **Variant Analysis** comparing `Order Priority` vs. `Lead Time`. We will look for cases where `Order Priority = High` but `Lead Time` exceeds the P90 of `Order Priority = Medium`.
3.  **Setup Time Inefficiency:**
    *   *Evidence:* The `Setup Duration (Actual)` fluctuates wildly (e.g., 10 min to 60 min) for the same machine, suggesting operators are not sequencing jobs to minimize changeover complexity.
    *   *Mining Evidence:* **Setup Transition Matrix** showing specific sequences (e.g., Large Part $\to$ Small Part) taking 3x longer than the reverse.
4.  **WIP Bullwhip Effect:**
    *   *Evidence:* Excessive WIP accumulating at non-bottleneck stations because the bottleneck is blocked or slow.
    *   *Mining Evidence:* **WIP Heatmaps** showing inventory build-up upstream of the identified bottleneck resources.
5.  **Disruption Amplification:**
    *   *Evidence:* A single breakdown causes a ripple effect where subsequent jobs are delayed by more than the breakdown duration due to lack of buffer management.
    *   *Mining Evidence:* **Event Correlation** showing the duration of the "recovery period" after a `Breakdown Start` event.

## 3. Root Cause Analysis of Scheduling Ineffectiveness

Having diagnosed the symptoms, we must isolate the root causes to ensure we fix the right problem. Process mining helps distinguish between *capacity constraints* and *scheduling logic failures*.

### Root Cause Differentiation
*   **Scheduling Logic vs. Capacity:**
    *   *Analysis:* If resource utilization is consistently >85% *and* lead times are increasing, the root cause is **Capacity Constraint** (need more machines). If utilization is <70% but lead times are high, the root cause is **Scheduling Logic** (poor sequencing/flow).
    *   *Finding:* We expect to find the latter. Bottlenecks are likely not fully utilized due to starvation caused by upstream scheduling delays.
*   **Static Rules vs. Dynamic Reality:**
    *   *Analysis:* The current mix of FCFS/EDD does not account for `Setup Required` or `Previous Job`.
    *   *Finding:* The lack of **Sequence-Dependent Setup Awareness** in the dispatching rule is a primary logic failure. The scheduler treats a 5-minute setup the same as a 45-minute setup.
*   **Estimation Bias:**
    *   *Analysis:* Compare `Task Duration (Planned)` vs. `Task Duration (Actual)` from the logs.
    *   *Finding:* If Planned is consistently lower than Actual (e.g., Actual is 120% of Planned), the root cause is **Inaccurate Standard Data**. Schedules are built on false premises, making them brittle.
*   **Reactive vs. Proactive:**
    *   *Analysis:* Analyze the time gap between `Breakdown Start` and the `Resource` becoming available again.
    *   *Finding:* If the schedule does not automatically re-optimize upon breakdown, the root cause is **Lack of Dynamic Rescheduling Logic**.

## 4. Developing Advanced Data-Driven Scheduling Strategies

We propose three distinct strategies that leverage the insights derived from process mining.

### Strategy 1: Dynamic Multi-Attribute Dispatching Rule (DMADR)
*   **Core Logic:** Replace FCFS/EDD with a weighted scoring system calculated at the moment a machine becomes free.
*   **Formula:** $Score = w_1(Due Date) + w_2(Setup Time) + w_3(Downstream Load) + w_4(Priority)$
*   **Process Mining Integration:**
    *   Use the **Setup Transition Matrix** to predict the setup time for the next job based on the *currently* running job.
    *   Use **Queue Length Analysis** to estimate downstream load (if the next machine has a long queue, delay the job).
*   **Addressing Pathologies:** Directly addresses **Prioritization Failure** and **Setup Inefficiency**. It ensures high-priority jobs aren't stuck behind low-priority ones, and it minimizes setup time by grouping similar jobs.
*   **Expected Impact:** 15-20% reduction in setup time; 10-15% improvement in On-Time Delivery (OTD).

### Strategy 2: Predictive Buffer & Capacity Planning
*   **Core Logic:** Move from deterministic scheduling (fixed times) to probabilistic scheduling (time buffers based on variance).
*   **Process Mining Integration:**
    *   Analyze the **Task Duration Distributions** (not just averages). If `Task Duration (Actual)` follows a Weibull distribution, use that to calculate a "Safety Buffer" for the schedule.
    *   Integrate **Breakdown Frequency** data to calculate the probability of machine failure during a specific time window.
*   **Addressing Pathologies:** Addresses **Unpredictable Lead Times** and **Disruption Amplification**. By building in statistically valid buffers, the schedule absorbs variability without collapsing.
*   **Expected Impact:** 25% increase in Lead Time Predictability (lower variance); reduced panic/expediting.

### Strategy 3: Bottleneck-Centric Sequence Optimization (Genetic Algorithm)
*   **Core Logic:** Focus scheduling optimization primarily on the bottleneck resource (e.g., `MILL-03`).
*   **Process Mining Integration:**
    *   Use the **Setup Transition Matrix** to create a Traveling Salesman Problem (TSP) variant. The goal is to sequence jobs on the bottleneck to minimize total setup time.
    *   Non-bottleneck resources will be scheduled to feed the bottleneck continuously (Theory of Constraints).
*   **Addressing Pathologies:** Addresses **Inefficient Resource Utilization** and **Bottleneck Starvation**. By maximizing the bottleneck's uptime and minimizing its setup, overall throughput increases.
*   **Expected Impact:** 10-15% increase in overall shop throughput; significant reduction in WIP at the bottleneck.

## 5. Simulation, Evaluation, and Continuous Improvement

Deploying these strategies directly to the shop floor is risky. We must validate them in a digital twin environment.

### Discrete Event Simulation (DES)
*   **Modeling:** Build a DES model (e.g., using AnyLogic, FlexSim, or Python SimPy) calibrated with process mining data.
*   **Parameterization:**
    *   *Durations:* Use the actual distributions mined (not averages).
    *   *Breakdowns:* Use the failure rates and repair times extracted from the logs.
    *   *Routing:* Use the actual routing probabilities (some jobs might loop back for rework).
*   **Testing Scenarios:**
    *   **Baseline:** Current FCFS/EDD rules.
    *   **Scenario A (High Load):** 120% of normal order volume.
    *   **Scenario B (Disruption):** Introduce breakdowns every 4 hours.
    *   **Scenario C (Hot Jobs):** Insert urgent orders randomly.
*   **Evaluation:** Compare KPIs (Tardiness, WIP, Utilization) across scenarios. Select the strategy that performs best under the "Stress Test" conditions.

### Continuous Improvement Framework
Once deployed, the system must evolve. We will implement a **Process Mining Feedback Loop**:
1.  **Real-Time Monitoring:** A dashboard displaying current KPIs (Lead Time, WIP, Bottleneck Load) updated hourly.
2.  **Drift Detection:** Automated alerts if `Actual Setup Time` deviates significantly from the `Predicted Setup Time` (indicating operator error or tooling issues).
3.  **Schedule Adjustment:** If the simulation predicts a bottleneck formation in the next shift, the system automatically adjusts the dispatching weights (Strategy 1) to push work away from that machine.
4.  **Retraining:** Every quarter, re-run the process mining analysis to update the Setup Matrix and Duration Distributions, ensuring the scheduling logic remains aligned with current shop floor conditions.
