# Event Abstraction: Low-Level to High-Level Process Mapping

## 1. Analysis Approach

I analyzed the event log using four grouping heuristics:

| Heuristic | Description |
|-----------|-------------|
| **Functional Cohesion** | Events serving a single manufacturing purpose (e.g., joining parts) |
| **Resource Continuity** | Consecutive events performed by the same resource/resource-type |
| **Temporal Proximity** | Events clustered tightly in time, then separated by a gap |
| **Logical Dependency** | Events that necessarily follow one another (pick tool  use tool) |

---

## 2. Proposed High-Level Activities

### **Stage 1: Material Preparation**
| Low-Level Activity | Resource | Rationale |
|--------------------|----------|-----------|
| Retrieve raw metal sheet | Operator A | Sourcing input material |
| Scan barcode | Operator A | Identification/traceability of material |
| Place metal sheet on table | Operator A | Staging for work |
| Align metal sheet | Robot Arm #2 | Positioning prerequisite |
| Preheat metal sheet | Heating Unit #1 | Conditioning material for welding |

**Rationale:** All events transform raw stock into a *ready-to-process* state. They share the purpose of intake, identification, and conditioning. They occur first and as a tight temporal cluster (08:00:05–08:00:20). The preheat step logically belongs here because it prepares the material *for* the subsequent welding, not as part of the weld itself.

---

### **Stage 2: Assembly / Welding**
| Low-Level Activity | Resource | Rationale |
|--------------------|----------|-----------|
| Pick up welding tool | Operator B | Tool acquisition precedes use |
| Weld corner A | Operator B | Core joining operation |
| Weld corner B | Operator B | Core joining operation |

**Rationale:** This is the value-adding transformation core. Grouped via **resource continuity** (all Operator B) and **logical dependency** (pick tool  weld). The "Pick up welding tool" is a setup action whose only purpose is to enable welding, so it is absorbed into this stage rather than treated separately.

---

### **Stage 3: In-Process Quality Verification**
| Low-Level Activity | Resource | Rationale |
|--------------------|----------|-----------|
| Measure weld integrity | Quality Sensor #1 | Verification of the just-completed weld |

**Rationale:** Distinct functional role (quality assurance), performed by a dedicated quality resource. Kept **separate from the final inspection** because it verifies an *intermediate* output (the weld) before further processing—an important distinction for process analysis (rework loops would branch here).

---

### **Stage 4: Surface Finishing**
| Low-Level Activity | Resource | Rationale |
|--------------------|----------|-----------|
| Apply protective coating | Coating Unit #3 | Surface treatment |
| Dry coating | Heating Unit #2 | Curing of the applied coating |

**Rationale:** Both events serve to finish/protect the surface. Grouped by **functional cohesion** (coating lifecycle) and **logical dependency** (apply  dry). Note: the drying uses a *Heating Unit* like preparation, but its **purpose** (cure coating) ties it to finishing, not preparation—illustrating why function trumps resource type here.

---

### **Stage 5: Final Quality Inspection**
| Low-Level Activity | Resource | Rationale |
|--------------------|----------|-----------|
| Visual check | Operator C | Final acceptance gate |

**Rationale:** The terminal verification confirming the *finished product* is acceptable. Separated from Stage 3 because it occurs after finishing and represents a final sign-off (different resource, different scope, end of sequence).

## 5. Generalization Rules (for the full log)

To apply consistently at scale, classify each low-level activity by **verb/intent**:

| Keyword Pattern |  Mapped Stage |
|-----------------|----------------|
| retrieve, scan, place, align, preheat | Material Preparation |
| pick up *tool*, weld, fasten, assemble, bolt | Assembly |
| measure, test, sensor reading (**mid-process**) | In-Process Quality Verification |
| coat, paint, dry, cure, polish | Surface Finishing |
| visual check, final inspect, sign-off (**last in case**) | Final Quality Inspection |

**Key disambiguation rule:** When the same resource type (e.g., Heating Unit) appears in multiple contexts, classify by the **operation's purpose and position in sequence**, not the machine. Quality events are split by **position** (mid-trace = in-process; end-of-trace = final).

---

## 6. Notes & Recommendations
- **Rework detection:** If `IntegrityScore` falls below threshold, expect a loop back to Assembly—worth modeling as a conditional branch.
- **Tool-pickup events** are treated as setup sub-steps; if your analysis needs setup-time metrics, retain them as a `subtype` attribute rather than discarding.
- This mapping enables **conformance checking** and **bottleneck analysis** at the meaningful stage level rather than noisy low-level events.
